## 1. Why Locks Exist

Locks prevent concurrent transactions from corrupting shared data. When T1 modifies a row, T2 must either wait or be told the row is unavailable.

---

## 2. Lock Types in PostgreSQL

```
TABLE-LEVEL LOCKS (from least to most restrictive):
  ACCESS SHARE:        SELECT  — many readers OK
  ROW SHARE:           SELECT FOR UPDATE — readers OK, heavy DDL blocked
  ROW EXCLUSIVE:       INSERT/UPDATE/DELETE — other writers blocked
  SHARE:               CREATE INDEX (non-concurrent) — blocks all writes
  SHARE ROW EXCLUSIVE: Limited DDL — blocks most other locks
  EXCLUSIVE:           Heavy DDL — blocks all except ACCESS SHARE
  ACCESS EXCLUSIVE:    DROP/TRUNCATE/FULL VACUUM — blocks everything

ROW-LEVEL LOCKS:
  FOR UPDATE:          Lock selected rows for update
  FOR NO KEY UPDATE:   Weaker row lock (for FK checks)
  FOR SHARE:           Lock for reading, block updates
  FOR KEY SHARE:       Weakest shared row lock

-- Check current locks:
SELECT pid, relation::regclass, mode, granted
FROM pg_locks
WHERE relation IS NOT NULL AND NOT granted;  -- show waiting locks
```

## 3. SELECT FOR UPDATE: Pessimistic Locking

```sql
BEGIN;

-- Lock the row immediately (before reading the value)
SELECT balance FROM accounts WHERE id = 1 FOR UPDATE;
-- Other transactions trying to read/update this row will WAIT

-- Safe to modify — no one else can change balance
UPDATE accounts SET balance = balance - 100 WHERE id = 1;

COMMIT;
-- Lock released, waiters can proceed

-- SKIP LOCKED: skip locked rows instead of waiting (queue processing)
SELECT id, payload FROM jobs
WHERE status = 'pending'
ORDER BY created_at
LIMIT 1
FOR UPDATE SKIP LOCKED;
-- Each worker gets a different job, no contention
```

## 4. Deadlocks

A deadlock occurs when two transactions each hold a lock the other needs:

```
T1: UPDATE accounts SET balance=... WHERE id=1;  (holds lock on row 1)
    UPDATE accounts SET balance=... WHERE id=2;  (waits for row 2)

T2: UPDATE accounts SET balance=... WHERE id=2;  (holds lock on row 2)
    UPDATE accounts SET balance=... WHERE id=1;  (waits for row 1)
→ Deadlock: each waits for the other forever
```

PostgreSQL detects deadlocks and aborts one transaction with: `ERROR: deadlock detected`

**Prevention strategies:**

```sql
-- STRATEGY 1: Always lock rows in the same order
-- Both T1 and T2 always lock by ascending id:
BEGIN;
SELECT * FROM accounts WHERE id IN (1, 2) ORDER BY id FOR UPDATE;
-- T1: locks row 1 first, then row 2
-- T2: locks row 1 first → waits for T1 → no deadlock
COMMIT;

-- STRATEGY 2: Lock all needed rows at once
BEGIN;
SELECT * FROM accounts WHERE id = ANY($1::int[]) FOR UPDATE;
-- Locks all rows in one statement = no interleaving possible

-- STRATEGY 3: Use advisory locks for application-level coordination
SELECT pg_advisory_xact_lock(user_id);  -- integer lock per user
-- Only one transaction per user_id at a time, no row-level deadlock
```

## 5. Lock Monitoring

```sql
-- Find long-running queries holding locks
SELECT
    pid,
    now() - query_start AS duration,
    query,
    state
FROM pg_stat_activity
WHERE state != 'idle'
  AND (now() - query_start) > INTERVAL '30 seconds'
ORDER BY duration DESC;

-- Find blocking relationships
SELECT
    blocked_locks.pid AS blocked_pid,
    blocking_locks.pid AS blocking_pid,
    blocked_activity.query AS blocked_query,
    blocking_activity.query AS blocking_query
FROM pg_catalog.pg_locks blocked_locks
JOIN pg_catalog.pg_stat_activity blocked_activity ON blocked_activity.pid = blocked_locks.pid
JOIN pg_catalog.pg_locks blocking_locks ON blocking_locks.locktype = blocked_locks.locktype
    AND blocking_locks.pid != blocked_locks.pid
JOIN pg_catalog.pg_stat_activity blocking_activity ON blocking_activity.pid = blocking_locks.pid
WHERE NOT blocked_locks.granted;

-- Kill a blocking query (last resort)
SELECT pg_terminate_backend(pid);
```

## 6. Senior Engineer Perspective

**The lock chain problem:** T1 holds a lock. T2 waits. T3 waits for T2. Now T4 arrives — it waits behind T3. A single long-running transaction creates a queue of blocked operations. Production symptom: requests start timing out after one slow query appears. Diagnosis: `pg_locks` showing a blocked chain.

**Statement timeouts prevent lock chains:**

```sql
-- Set at application level for all user-facing queries
SET statement_timeout = '5s';
-- Slow query killed after 5s → lock released → chain unblocked

-- Set at database level for a specific user
ALTER ROLE api_user SET statement_timeout = '10s';
```
