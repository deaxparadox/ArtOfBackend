## 1. Transaction Basics

A transaction groups multiple SQL operations into a single atomic unit.

```sql
-- Explicit transaction
BEGIN;

  UPDATE accounts SET balance = balance - 100 WHERE id = 1;
  UPDATE accounts SET balance = balance + 100 WHERE id = 2;
  INSERT INTO audit_log (action, user_id, amount) VALUES ('transfer', 1, 100);

COMMIT;  -- all three operations committed atomically

-- If anything goes wrong:
BEGIN;
  UPDATE accounts SET balance = balance - 100 WHERE id = 1;
  UPDATE accounts SET balance = balance + 100 WHERE id = 99;  -- id 99 doesn't exist
  -- PostgreSQL: ERROR: insert or update on table violates foreign key constraint
ROLLBACK;  -- ← all changes undone, even the first successful UPDATE
```

**Autocommit:** By default, every SQL statement is its own transaction (autocommit = on). Single statements are always atomic. For multi-statement atomicity: use explicit `BEGIN`/`COMMIT`.

## 2. Transaction Control Language (TCL)

```sql
BEGIN;               -- start a transaction (alias: START TRANSACTION)
BEGIN READ ONLY;     -- read-only transaction (no writes allowed)
COMMIT;              -- commit all changes
ROLLBACK;            -- undo all changes
SAVEPOINT name;      -- create a rollback point within the transaction
ROLLBACK TO name;    -- roll back to savepoint (partial undo)
RELEASE SAVEPOINT name;  -- remove a savepoint

-- Full example with savepoints
BEGIN;

  INSERT INTO orders (user_id, total) VALUES (1, 5000);
  SAVEPOINT after_order;  -- checkpoint

  INSERT INTO order_items (order_id, product_id) VALUES (lastval(), 999);
  -- ERROR: product 999 doesn't exist

  ROLLBACK TO after_order;  -- undo only the failed insert, keep the order

  -- Try a different product
  INSERT INTO order_items (order_id, product_id) VALUES (lastval(), 42);

COMMIT;  -- order + valid order item committed
```

## 3. Transactions in Python (asyncpg)

```python
import asyncpg

async def transfer_funds(pool, from_id: int, to_id: int, amount_cents: int):
    async with pool.acquire() as conn:
        async with conn.transaction():
            # Everything inside this block is one transaction
            # Auto-commits on exit, auto-rollbacks on exception

            # Check sufficient balance
            balance = await conn.fetchval(
                "SELECT balance FROM accounts WHERE id = $1 FOR UPDATE",
                from_id  # FOR UPDATE: lock this row to prevent concurrent modification
            )
            if balance < amount_cents:
                raise InsufficientFundsError("Insufficient balance")

            await conn.execute(
                "UPDATE accounts SET balance = balance - $1 WHERE id = $2",
                amount_cents, from_id
            )
            await conn.execute(
                "UPDATE accounts SET balance = balance + $1 WHERE id = $2",
                amount_cents, to_id
            )
            await conn.execute(
                "INSERT INTO transfers (from_id, to_id, amount) VALUES ($1, $2, $3)",
                from_id, to_id, amount_cents
            )
            # Context manager commits here on success
            # Any exception triggers automatic ROLLBACK

# SQLAlchemy async transaction
async with async_session() as session:
    async with session.begin():
        # session.begin() starts a transaction
        user = await session.get(User, user_id, with_for_update=True)
        user.balance -= amount_cents
        session.add(AuditLog(user_id=user_id, action="debit", amount=amount_cents))
        # Commits automatically at end of context manager
```

## 4. Common Transaction Patterns

### Read-Modify-Write (Critical to Get Right)

```sql
-- WRONG: race condition between read and write
-- Two concurrent requests both read balance=500
-- Both compute new_balance=600
-- Both write 600 (should be 700!)
SELECT balance FROM accounts WHERE id = 1;  -- read: 500
-- [other transaction modifies balance here]
UPDATE accounts SET balance = 600 WHERE id = 1;  -- write: lost update!

-- RIGHT OPTION 1: Atomic increment (no separate read needed)
UPDATE accounts
SET balance = balance + 100
WHERE id = 1;
-- The + 100 happens atomically in the database, no application read needed

-- RIGHT OPTION 2: Optimistic locking (check version hasn't changed)
BEGIN;
  SELECT balance, version FROM accounts WHERE id = 1;
  -- In application: compute new_balance
  UPDATE accounts
  SET balance = 600, version = version + 1
  WHERE id = 1 AND version = 5;  -- fails if version changed
  -- If 0 rows updated: conflict detected, retry
COMMIT;

-- RIGHT OPTION 3: Pessimistic locking (lock the row immediately)
BEGIN;
  SELECT balance FROM accounts WHERE id = 1 FOR UPDATE;  -- locks the row
  -- No other transaction can modify this row until we COMMIT/ROLLBACK
  UPDATE accounts SET balance = 600 WHERE id = 1;
COMMIT;
```

### Idempotent Transactions with Savepoints

```python
async def process_event_idempotent(conn, event_id: str, payload: dict):
    """Process event exactly once, even if called multiple times."""
    async with conn.transaction():
        # Try to insert idempotency record
        try:
            await conn.execute(
                "INSERT INTO processed_events (event_id) VALUES ($1)",
                event_id
            )
        except asyncpg.UniqueViolationError:
            # Already processed — return early (no error, no processing)
            return {"status": "already_processed"}

        # Process the event
        result = await do_actual_work(conn, payload)
        return {"status": "processed", "result": result}
```

## 5. Transaction Anti-Patterns

```python
# ANTI-PATTERN 1: Long-running transactions with user interaction
async def checkout_handler(session):
    async with session.begin():  # transaction starts here
        order = await create_order()
        payment_result = await stripe.charge()  # EXTERNAL API CALL inside transaction!
        # Stripe takes 500ms. Transaction holds locks for 500ms.
        # At scale: lock contention, connection pool starvation
        await update_order_status(order, payment_result)

# FIX: Do external calls OUTSIDE transactions
async def checkout_handler_correct(session):
    # 1. Create order (fast transaction)
    async with session.begin():
        order = await create_order()

    # 2. Call external API (outside transaction)
    payment_result = await stripe.charge()  # no lock held

    # 3. Update order (fast transaction)
    async with session.begin():
        await update_order_status(order, payment_result)

# ANTI-PATTERN 2: Catching ALL exceptions and continuing
async with session.begin():
    try:
        await risky_operation()
    except Exception:
        pass  # WRONG: transaction may be in a failed state
              # SQLAlchemy/asyncpg: any exception in a transaction aborts it
              # Must rollback before reusing the connection

# FIX: Use savepoints for partial rollback
async with session.begin():
    sp = await session.begin_nested()  # savepoint
    try:
        await risky_operation()
        await sp.commit()
    except Exception:
        await sp.rollback()  # rollback to savepoint, transaction continues
    await other_operation()  # this still runs
```

## 6. Senior Engineer Perspective

**The golden rule:** Keep transactions short. Long-running transactions hold locks, consume connection pool slots, and prevent autovacuum from cleaning dead rows (in PostgreSQL). A transaction that takes 5+ seconds is a production hazard.

**Transaction scope should match your consistency boundary.** If operations A and B must be atomic together, they belong in one transaction. If operation C can fail independently without affecting A and B, it belongs in a separate transaction (or a savepoint, if it's within the same logical unit).

**Never hold a transaction open across user input, network calls, or any external system call.** The pattern: do all external work first, then write to the database in a single fast transaction.
