## 1. The Most Important Database Debugging Tool

`EXPLAIN` shows the query execution plan — how PostgreSQL plans to execute a query. `EXPLAIN ANALYZE` actually executes it and shows real timing data. This is the tool that turns a 10-second query into a 5ms query.

**The rule: never optimize a query without looking at EXPLAIN ANALYZE first.**

---

## 2. EXPLAIN vs EXPLAIN ANALYZE

```sql
-- EXPLAIN: shows ESTIMATED plan (doesn't execute)
EXPLAIN SELECT * FROM orders WHERE user_id = 123;

-- EXPLAIN ANALYZE: executes and shows ACTUAL timing
EXPLAIN ANALYZE SELECT * FROM orders WHERE user_id = 123;

-- EXPLAIN (ANALYZE, BUFFERS): also shows memory/disk I/O
EXPLAIN (ANALYZE, BUFFERS, FORMAT JSON) SELECT * FROM orders WHERE user_id = 123;

-- WARNING: EXPLAIN ANALYZE actually runs the query
-- Don't use it on expensive UPDATE/DELETE without a transaction:
BEGIN;
  EXPLAIN ANALYZE DELETE FROM old_logs WHERE created_at < '2023-01-01';
ROLLBACK;  -- ← prevent the delete from actually happening
```

## 3. Reading EXPLAIN Output

```sql
EXPLAIN ANALYZE
SELECT u.name, COUNT(o.id) AS order_count
FROM users u
LEFT JOIN orders o ON u.id = o.user_id
GROUP BY u.id, u.name;

-- OUTPUT:
HashAggregate  (cost=5432.10..5532.10 rows=10000 width=40)
               (actual time=234.1..245.3 rows=10000 loops=1)
  Group Key: u.id, u.name
  ->  Hash Left Join  (cost=1234.5..4432.1 rows=200000 width=20)
                      (actual time=12.3..189.4 rows=200000 loops=1)
        Hash Cond: (o.user_id = u.id)
        ->  Seq Scan on orders o  (cost=0.0..2500.0 rows=200000 width=8)
                                   (actual time=0.1..89.2 rows=200000 loops=1)
        ->  Hash  (cost=620.0..620.0 rows=10000 width=20)
                  (actual time=11.8..11.8 rows=10000 loops=1)
              Buckets: 16384  Batches: 1  Memory Usage: 624kB
              ->  Seq Scan on users u  (cost=0.0..620.0 rows=10000 width=20)
                                       (actual time=0.1..8.4 rows=10000 loops=1)
Planning Time: 2.3 ms
Execution Time: 248.1 ms
```

**Reading the numbers:**

```
cost=5432.10..5532.10:
  First number:  cost to get first row
  Second number: total cost (arbitrary unit, lower is better)

rows=10000:
  Estimated rows — how many rows this node will produce

actual time=234.1..245.3:
  First: time to get first row (ms)
  Second: total time for this node (ms)

rows=10000 (actual):
  Actual rows produced — if very different from estimate: bad statistics

loops=1:
  How many times this node was executed (for nested loops: could be N)
  Multiply actual time by loops for total cost of this node
```

## 4. Warning Signs to Look For

```
1. SEQ SCAN on a large table:
   Seq Scan on orders (actual rows=2000000)
   → Missing index. Add an index on the filtered column.

2. ESTIMATED rows vs ACTUAL rows very different:
   (cost=... rows=100)   ← estimated 100
   (actual time=... rows=50000)  ← actual 50000
   → Out-of-date statistics. Run ANALYZE table_name.

3. NESTED LOOP with loops=N on large table:
   Nested Loop (loops=10000)
   →  Scan in inner loop (loops=10000)
   → N×M row join. Missing index on inner table's join column.

4. Large Hash/Sort operations using disk:
   Sort (... Disk: 1024kB)
   → Sort spilling to disk. Increase work_mem:
   SET work_mem = '256MB';

5. "Index Scan" when "Index Only Scan" expected:
   Index Scan using idx_orders_user_id on orders
   → Table heap access needed. Add INCLUDE columns to make it covering.
```

## 5. Key Query Plan Nodes

```
Seq Scan:           Full table scan. Good for small tables or >10% selectivity.
Index Scan:         Uses index + fetches from heap. Good for < 10% selectivity.
Index Only Scan:    Uses index, no heap access. Best case.
Bitmap Heap Scan:   Uses bitmap of pages from index. Good for 1-10% selectivity.
Nested Loop:        For each row in outer, find matching rows in inner.
                    O(n×m) without index. O(n×log m) with index on inner.
Hash Join:          Build hash table of smaller side, probe with larger side.
                    Good when both tables are large and neither is too small.
Merge Join:         Both sides sorted, merge together. Good for large sorted inputs.
HashAggregate:      GROUP BY using a hash table. Memory-intensive.
Sort:               ORDER BY or pre-sort for Merge Join. Can spill to disk.
Limit:              Stop after N rows. Can early-exit if inner node cooperates.
Gather:             Parallel workers join their results here.
```

## 6. Fixing Slow Queries: A Workflow

```sql
-- STEP 1: Find slow queries
SELECT query, calls, total_exec_time, mean_exec_time
FROM pg_stat_statements
ORDER BY mean_exec_time DESC
LIMIT 20;

-- STEP 2: EXPLAIN ANALYZE the worst offenders
EXPLAIN (ANALYZE, BUFFERS) <slow_query>;

-- STEP 3: Identify the bottleneck
-- Look for: Seq Scan on large tables, high rows×loops, disk usage

-- STEP 4: Fix
-- Missing index → CREATE INDEX CONCURRENTLY
-- Bad estimates → ANALYZE table_name
-- Sort to disk → SET work_mem = '64MB' (per session)
-- Suboptimal join → rephrase query, add index on join column

-- STEP 5: Verify fix
EXPLAIN (ANALYZE, BUFFERS) <same_query>;  -- should show improvement
```

## 7. Senior Engineer Perspective

**The one workflow to drill into your muscle memory:**

Any time a query takes >50ms, run EXPLAIN ANALYZE immediately. Look at: is it doing a Seq Scan it shouldn't? Is the estimated row count way off from actual? Are there nested loops with high loop counts?

90% of slow queries are fixed by adding a missing index or running ANALYZE to update statistics. The other 10% require query rewriting.

**Tools that make this easier:**

- **pganalyze:** automatic slow query detection and index recommendations
- **explain.depesz.com:** paste EXPLAIN output, get color-coded analysis
- **auto_explain:** PostgreSQL extension that logs slow query plans automatically
