## ORM Fundamentals (5.46)

An ORM (Object-Relational Mapper) provides an object-oriented interface to a relational database. You work with Python/Java/Node.js objects; the ORM generates SQL.

```
ORM VALUE PROPOSITION:
  ✓ Write Python/JS instead of SQL for most operations
  ✓ Database abstraction (switch PostgreSQL → MySQL with config change)
  ✓ Type safety (Pydantic/TypeScript types match DB columns)
  ✓ Migration generation from model changes
  ✓ Relationship loading (user.orders instead of JOIN)

ORM COST:
  ✗ Hides SQL → hard to optimize when queries are slow
  ✗ N+1 query problem (easy to create, subtle to detect)
  ✗ Complex queries are harder to express than raw SQL
  ✗ Performance overhead vs. raw SQL (small, but real)
  ✗ "Magic" behavior can surprise developers
```

## SQLAlchemy (Python) (5.47)

The most powerful Python ORM — two interfaces: Core (SQL abstraction) and ORM (object mapping).

```python
from sqlalchemy import Column, Integer, String, ForeignKey, select, func
from sqlalchemy.orm import DeclarativeBase, relationship, selectinload
from sqlalchemy.ext.asyncio import AsyncSession

class Base(DeclarativeBase):
    pass

class User(Base):
    __tablename__ = "users"

    id = Column(Integer, primary_key=True)
    email = Column(String(255), unique=True, nullable=False)
    name = Column(String(100), nullable=False)
    orders = relationship("Order", back_populates="user", lazy="raise")
    # lazy="raise": accessing .orders without explicit loading raises an error
    # Forces explicit: selectinload(User.orders) to prevent accidental N+1

class Order(Base):
    __tablename__ = "orders"

    id = Column(Integer, primary_key=True)
    user_id = Column(Integer, ForeignKey("users.id"), nullable=False)
    total_cents = Column(Integer, nullable=False)
    user = relationship("User", back_populates="orders")

# Async usage with SQLAlchemy 2.0
async def get_users_with_orders(session: AsyncSession):
    stmt = (
        select(User)
        .options(selectinload(User.orders))  # eager load orders (1 extra query, not N)
        .where(User.id.in_([1, 2, 3]))
    )
    result = await session.execute(stmt)
    return result.scalars().all()

# Core (SQL-level, for complex queries)
stmt = (
    select(User.id, User.email, func.count(Order.id).label("order_count"))
    .outerjoin(Order, User.id == Order.user_id)
    .group_by(User.id, User.email)
    .order_by(func.count(Order.id).desc())
    .limit(10)
)
result = await session.execute(stmt)
```

## Django ORM (5.48)

```python
# models.py — model = table definition + query interface
class User(models.Model):
    email = models.EmailField(unique=True)
    name = models.CharField(max_length=100)

    class Meta:
        indexes = [models.Index(fields=['email'])]

class Order(models.Model):
    user = models.ForeignKey(User, on_delete=models.CASCADE, related_name='orders')
    total_cents = models.IntegerField()
    status = models.CharField(max_length=20, choices=[...])

# QuerySets are lazy — evaluated only when iterated/sliced
users = User.objects.filter(is_active=True)  # no DB hit yet
users = users.order_by('-created_at')        # chaining, still no DB hit
users = users[:20]                           # limit, still no DB hit
for user in users:                           # DB hit happens HERE
    print(user.email)

# Prevent N+1: select_related (JOIN) and prefetch_related (separate query)
orders = Order.objects.select_related('user').filter(status='paid')
# Generates: SELECT orders.*, users.* FROM orders JOIN users ON ...

users = User.objects.prefetch_related('orders').all()
# Generates:
#   SELECT * FROM users;
#   SELECT * FROM orders WHERE user_id IN (1,2,3,...);  ← one query, not N

# Aggregate
from django.db.models import Sum, Count
stats = Order.objects.filter(status='paid').aggregate(
    total=Sum('total_cents'),
    count=Count('id'),
)
```

## Prisma (Node.js) (5.49)

```typescript
// schema.prisma
model User {
    id     Int    @id @default(autoincrement())
    email  String @unique
    name   String
    orders Order[]
}

model Order {
    id        Int    @id @default(autoincrement())
    user      User   @relation(fields: [userId], references: [id])
    userId    Int
    totalCents Int
    status    String @default("pending")
}

// Generated TypeScript client — fully type-safe
import { PrismaClient } from '@prisma/client';
const prisma = new PrismaClient();

// Find with eager loading (prevents N+1)
const users = await prisma.user.findMany({
    include: { orders: true },   // SELECT users + JOIN orders
    where: { orders: { some: { status: 'paid' } } },
    take: 10,
});

// Type-safe: users[0].orders[0].totalCents is number (not any)

// Raw SQL when ORM isn't enough
const result = await prisma.$queryRaw`
    SELECT u.email, COUNT(o.id)::int AS order_count
    FROM users u
    LEFT JOIN orders o ON u.id = o.user_id
    GROUP BY u.id, u.email
    ORDER BY order_count DESC
    LIMIT 10
`;
```

## The N+1 Problem (5.54): Critical

The most common ORM bug — N+1 queries instead of 1.

```python
# WRONG: N+1 (1 query for users, then 1 query per user for orders)
users = User.objects.all()          # 1 query: SELECT * FROM users
for user in users:
    orders = user.orders.all()      # 1 query PER user!
    total = sum(o.total_cents for o in orders)

# 100 users = 101 queries. 1000 users = 1001 queries.

# RIGHT: select_related / prefetch_related
users = User.objects.prefetch_related('orders').all()  # 2 queries total
for user in users:
    orders = user.orders.all()      # no DB hit! data already fetched
    total = sum(o.total_cents for o in orders)

# DETECTION: log all SQL queries in development
import logging
logging.getLogger('django.db.backends').setLevel(logging.DEBUG)

# Or use django-debug-toolbar to see query count per request

# RULE: Any > 5 queries for a single request is a red flag.
# Any loop that contains a DB query is an N+1.
```

## Raw Queries vs ORM (5.55)

```python
# Rule: use ORM for standard CRUD, raw SQL for complex analytics

# Raw SQL with asyncpg (fastest, no ORM overhead)
async def get_top_customers(conn, limit=10):
    return await conn.fetch("""
        SELECT
            u.email,
            COUNT(o.id) AS order_count,
            SUM(o.total_cents) AS total_spent
        FROM users u
        JOIN orders o ON u.id = o.user_id
        WHERE o.status = 'paid'
        GROUP BY u.id, u.email
        ORDER BY total_spent DESC
        LIMIT $1
    """, limit)

# SQLAlchemy Core (SQL abstraction without full ORM)
from sqlalchemy import text
result = await session.execute(
    text("SELECT * FROM complex_function(:param)"),
    {"param": "value"}
)

# The hybrid approach (best of both worlds):
# ORM for: CRUD, relationships, simple filters
# Raw SQL for: analytics, complex joins, bulk operations, anything ORM makes ugly
```

## ORM Comparison Summary

| ORM | Language | Type | Best For |
|---|---|---|---|
| SQLAlchemy | Python | Full ORM + Core | Most powerful Python ORM, async support |
| Django ORM | Python | Full ORM | Django apps, admin integration |
| Prisma | Node.js | Type-safe ORM | TypeScript first-class, schema-driven |
| TypeORM | Node.js | Full ORM | Decorator-based, Active Record/Data Mapper |
| Drizzle ORM | Node.js | SQL-first | Lightweight, TypeScript, performant |
| GORM | Go | Full ORM | Most popular Go ORM |
| Hibernate/JPA | Java | Full ORM | Enterprise Java, Spring Data |
