## 1. The Four Core SQL Operations (CRUD)

SQL's DML (Data Manipulation Language) has four operations: SELECT (read), INSERT (create), UPDATE (modify), DELETE (remove). Every database operation maps to one of these.

---

## 2. SELECT: Reading Data

SELECT is the most complex SQL operation and the one you'll use most. Master it completely.

```sql
-- Basic structure (every clause is optional except SELECT and FROM)
SELECT column1, column2, expression
FROM   table_name
WHERE  condition
GROUP BY grouping_column
HAVING condition_on_groups
ORDER BY sort_column [ASC|DESC]
LIMIT  n
OFFSET m;

-- SELECT all columns (use sparingly in production — explicit is better)
SELECT * FROM users;

-- SELECT specific columns (always do this for production queries)
SELECT id, email, name, created_at FROM users;

-- Expressions and aliases
SELECT
    id,
    email,
    upper(name) AS name_upper,
    extract(year FROM created_at) AS signup_year,
    total_cents / 100.0 AS total_dollars,
    CASE
        WHEN total_cents > 10000 THEN 'high_value'
        WHEN total_cents > 1000  THEN 'medium_value'
        ELSE 'low_value'
    END AS value_tier
FROM orders;
```

### WHERE: Filtering Rows

```sql
-- Comparison operators
WHERE age > 18
WHERE age >= 18
WHERE name = 'Alice'
WHERE name != 'Alice'
WHERE created_at BETWEEN '2024-01-01' AND '2024-12-31'

-- Pattern matching (LIKE)
WHERE email LIKE '%@gmail.com'    -- ends with @gmail.com
WHERE name  LIKE 'A%'             -- starts with A
WHERE email LIKE '%test%'         -- contains test
WHERE name  ILIKE 'alice'         -- case-insensitive (PostgreSQL)

-- NULL checks (NEVER use = NULL)
WHERE deleted_at IS NULL          -- not soft-deleted
WHERE middle_name IS NOT NULL

-- Set membership
WHERE status IN ('pending', 'processing')
WHERE id NOT IN (1, 2, 3)

-- Combining conditions
WHERE status = 'active' AND age >= 18
WHERE city = 'NYC' OR city = 'LA'
WHERE NOT (status = 'cancelled')

-- Subquery in WHERE
WHERE id IN (
    SELECT user_id FROM orders WHERE total_cents > 10000
)
```

### ORDER BY, LIMIT, OFFSET

```sql
-- Sorting
SELECT * FROM users ORDER BY created_at DESC;           -- newest first
SELECT * FROM users ORDER BY name ASC, id DESC;         -- multiple sort keys

-- Pagination (offset-based — see topic 2.18 for why cursor is better)
SELECT * FROM products ORDER BY id LIMIT 20 OFFSET 0;  -- page 1
SELECT * FROM products ORDER BY id LIMIT 20 OFFSET 20; -- page 2

-- NULLS FIRST / NULLS LAST (PostgreSQL)
SELECT * FROM users ORDER BY last_login DESC NULLS LAST;
```

### GROUP BY and Aggregate Functions

```sql
-- Count rows per group
SELECT status, COUNT(*) AS count
FROM orders
GROUP BY status;

-- Multiple aggregate functions
SELECT
    user_id,
    COUNT(*)          AS order_count,
    SUM(total_cents)  AS total_spent,
    AVG(total_cents)  AS avg_order,
    MIN(created_at)   AS first_order,
    MAX(created_at)   AS latest_order
FROM orders
GROUP BY user_id;

-- HAVING: filter groups (WHERE filters rows, HAVING filters groups)
SELECT user_id, COUNT(*) AS order_count
FROM orders
GROUP BY user_id
HAVING COUNT(*) > 5;       -- only users with more than 5 orders
```

## 3. INSERT: Creating Data

```sql
-- Single row insert
INSERT INTO users (email, name) VALUES ('alice@example.com', 'Alice');

-- Multiple rows at once (much more efficient)
INSERT INTO users (email, name) VALUES
    ('bob@example.com',   'Bob'),
    ('carol@example.com', 'Carol'),
    ('dave@example.com',  'Dave');

-- Insert and return generated values (PostgreSQL RETURNING)
INSERT INTO users (email, name)
VALUES ('eve@example.com', 'Eve')
RETURNING id, created_at;

-- Insert from SELECT (copy data between tables)
INSERT INTO archived_orders (id, user_id, total_cents, archived_at)
SELECT id, user_id, total_cents, NOW()
FROM orders
WHERE created_at < '2023-01-01';

-- Upsert: insert or update on conflict (PostgreSQL)
INSERT INTO user_settings (user_id, theme, notifications_enabled)
VALUES (123, 'dark', true)
ON CONFLICT (user_id) DO UPDATE SET
    theme = EXCLUDED.theme,
    notifications_enabled = EXCLUDED.notifications_enabled,
    updated_at = NOW();

-- Ignore on conflict (don't update if exists)
INSERT INTO processed_events (event_id, processed_at)
VALUES ('abc-123', NOW())
ON CONFLICT (event_id) DO NOTHING;
```

## 4. UPDATE: Modifying Data

```sql
-- Always use WHERE with UPDATE (without it: updates EVERY row)
UPDATE users
SET name = 'Alice Smith', updated_at = NOW()
WHERE id = 1;

-- Update multiple columns
UPDATE orders
SET
    status = 'shipped',
    shipped_at = NOW(),
    tracking_number = 'TRACK123'
WHERE id = 456;

-- Update based on another table (UPDATE ... FROM)
UPDATE orders
SET status = 'refunded'
FROM refund_requests
WHERE orders.id = refund_requests.order_id
  AND refund_requests.approved = true;

-- Conditional update (avoid fetching row then updating)
UPDATE users
SET login_count = login_count + 1,  -- atomic increment
    last_login = NOW()
WHERE id = 123;
-- Don't do: read login_count, add 1 in Python, write back
-- (race condition — two requests could both read 5, both write 6)

-- RETURNING (see what was updated)
UPDATE orders
SET status = 'cancelled'
WHERE user_id = 123 AND status = 'pending'
RETURNING id, status;
```

## 5. DELETE: Removing Data

```sql
-- Always use WHERE with DELETE (without it: deletes EVERY row)
DELETE FROM users WHERE id = 123;

-- Delete with subquery
DELETE FROM orders
WHERE user_id IN (
    SELECT id FROM users WHERE status = 'banned'
);

-- Soft delete (preferred over hard delete)
-- Instead of:
DELETE FROM posts WHERE id = 456;
-- Use:
UPDATE posts SET deleted_at = NOW() WHERE id = 456;
-- Then all queries add: WHERE deleted_at IS NULL

-- RETURNING (see what was deleted)
DELETE FROM sessions
WHERE expires_at < NOW()
RETURNING id, user_id;

-- TRUNCATE: fastest way to delete all rows (DDL operation)
TRUNCATE TABLE temp_imports;  -- no WHERE possible, resets sequences
-- Use only for temporary/staging tables you want completely cleared
```

## 6. Subqueries and CTEs

### Subqueries

```sql
-- Scalar subquery: returns one value
SELECT
    name,
    (SELECT COUNT(*) FROM orders WHERE orders.user_id = users.id) AS order_count
FROM users;
-- WARNING: this is a correlated subquery — executed once per row (N+1 problem)
-- Prefer JOIN or a lateral join instead

-- EXISTS: check if matching rows exist
SELECT * FROM users
WHERE EXISTS (
    SELECT 1 FROM orders WHERE orders.user_id = users.id AND status = 'pending'
);
```

### CTEs (Common Table Expressions) — WITH

```sql
-- CTEs: named temporary result sets, improve readability
WITH
high_value_users AS (
    SELECT user_id, SUM(total_cents) AS lifetime_value
    FROM orders
    WHERE status = 'paid'
    GROUP BY user_id
    HAVING SUM(total_cents) > 100000
),
user_details AS (
    SELECT u.id, u.email, u.name, hv.lifetime_value
    FROM users u
    JOIN high_value_users hv ON u.id = hv.user_id
)
SELECT * FROM user_details ORDER BY lifetime_value DESC;

-- Recursive CTE: for hierarchical data (org charts, categories)
WITH RECURSIVE category_tree AS (
    -- Base case: top-level categories
    SELECT id, name, parent_id, 1 AS depth
    FROM categories
    WHERE parent_id IS NULL

    UNION ALL

    -- Recursive case: children of current nodes
    SELECT c.id, c.name, c.parent_id, ct.depth + 1
    FROM categories c
    JOIN category_tree ct ON c.parent_id = ct.id
)
SELECT * FROM category_tree ORDER BY depth, name;
```

## 7. Window Functions

Window functions perform calculations across related rows without collapsing them into groups (unlike GROUP BY):

```sql
-- ROW_NUMBER: rank rows within a partition
SELECT
    user_id,
    id AS order_id,
    total_cents,
    ROW_NUMBER() OVER (PARTITION BY user_id ORDER BY created_at) AS order_sequence
FROM orders;
-- Returns: user 1's orders numbered 1,2,3...; user 2's orders numbered 1,2,3...

-- Running total
SELECT
    created_at::date AS date,
    daily_revenue,
    SUM(daily_revenue) OVER (ORDER BY created_at::date) AS running_total
FROM daily_revenue_view;

-- LAG/LEAD: compare with previous/next row
SELECT
    month,
    revenue,
    LAG(revenue) OVER (ORDER BY month) AS prev_month_revenue,
    revenue - LAG(revenue) OVER (ORDER BY month) AS month_over_month_change
FROM monthly_revenue;

-- RANK vs DENSE_RANK vs ROW_NUMBER
-- RANK: 1,2,2,4 (ties share rank, next rank skips)
-- DENSE_RANK: 1,2,2,3 (ties share rank, no skips)
-- ROW_NUMBER: 1,2,3,4 (always unique, arbitrary tie-breaking)
```

## 8. Senior Engineer Perspective

**What a junior gets wrong:**

Using `SELECT *` in production code. When you add a new column to a table, every `SELECT *` query now returns that column — potentially exposing data (password hashes, internal flags) or breaking code that expected a specific number of columns. Always select the exact columns you need.

**The SQL skills that matter most:**

1. **JOINs** — join conditions are where correctness lives (topic 5.3)
2. **Indexes** — determines if your query runs in 2ms or 2 minutes (topic 5.9)
3. **EXPLAIN ANALYZE** — reading query plans (topic 5.11)
4. **CTEs and window functions** — writing queries that don't require loading data into application memory

**The performance rule:**

Before writing application code to process data, ask: "Can SQL do this?" SQL operations happen inside the database (on fast disk or RAM), not over the network. Aggregating 1 million rows in SQL is fast. Fetching 1 million rows and aggregating in Python is slow (network transfer + memory).
