## 1. Stored Procedures and Functions

PostgreSQL functions (PL/pgSQL) run code inside the database:

```sql
-- Function: a reusable SQL computation
CREATE OR REPLACE FUNCTION user_lifetime_value(p_user_id BIGINT)
RETURNS NUMERIC AS $$
BEGIN
    RETURN (
        SELECT COALESCE(SUM(total_cents), 0) / 100.0
        FROM orders
        WHERE user_id = p_user_id AND status = 'paid'
    );
END;
$$ LANGUAGE plpgsql STABLE;  -- STABLE: same inputs = same output (cacheable)

-- Use in queries
SELECT id, email, user_lifetime_value(id) AS ltv
FROM users ORDER BY ltv DESC LIMIT 10;

-- Stored procedure (PostgreSQL 11+): can commit/rollback transactions
CREATE OR REPLACE PROCEDURE process_refund(order_id BIGINT)
LANGUAGE plpgsql AS $$
BEGIN
    UPDATE orders SET status = 'refunded' WHERE id = order_id;
    INSERT INTO refunds (order_id, amount_cents, processed_at)
    SELECT id, total_cents, NOW() FROM orders WHERE id = order_id;
    COMMIT;
END;
$$;

CALL process_refund(123);
```

**When to use stored procedures:** Complex calculations that run faster in the DB than fetching data, database-level business rules, scheduled jobs. **When NOT to use:** Business logic that belongs in application code (harder to test, deploy, and version control).

## 2. Triggers

Triggers execute automatically when DML events occur:

```sql
-- Trigger function
CREATE OR REPLACE FUNCTION update_updated_at()
RETURNS TRIGGER AS $$
BEGIN
    NEW.updated_at = NOW();
    RETURN NEW;
END;
$$ LANGUAGE plpgsql;

-- Create trigger on table
CREATE TRIGGER set_updated_at
    BEFORE UPDATE ON users
    FOR EACH ROW
    EXECUTE FUNCTION update_updated_at();

-- Audit trigger: log all changes
CREATE TABLE audit_log (
    id BIGSERIAL PRIMARY KEY,
    table_name TEXT,
    operation TEXT,
    old_data JSONB,
    new_data JSONB,
    changed_at TIMESTAMPTZ DEFAULT NOW(),
    changed_by TEXT DEFAULT current_user
);

CREATE OR REPLACE FUNCTION audit_changes()
RETURNS TRIGGER AS $$
BEGIN
    INSERT INTO audit_log (table_name, operation, old_data, new_data)
    VALUES (TG_TABLE_NAME, TG_OP,
            to_jsonb(OLD),  -- OLD: row before change (NULL for INSERT)
            to_jsonb(NEW)   -- NEW: row after change (NULL for DELETE)
    );
    RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER audit_orders
    AFTER INSERT OR UPDATE OR DELETE ON orders
    FOR EACH ROW EXECUTE FUNCTION audit_changes();
```

**The trigger trap:** Triggers are invisible in application code. A seemingly simple `UPDATE` can cascade into 5 other operations via triggers. Debug production bugs by checking what triggers exist: `\d+ tablename` in psql.

## 3. Views

A view is a saved SELECT query you can query like a table:

```sql
-- Simple view: encapsulates a complex query
CREATE VIEW active_orders AS
SELECT o.*, u.email, u.name
FROM orders o
JOIN users u ON o.user_id = u.id
WHERE o.status NOT IN ('cancelled', 'refunded')
  AND o.deleted_at IS NULL;

-- Query the view
SELECT * FROM active_orders WHERE user_id = 123;

-- Updatable views: simple views can be used for INSERT/UPDATE/DELETE
CREATE VIEW active_users AS
SELECT * FROM users WHERE deleted_at IS NULL;

UPDATE active_users SET name = 'Alice Smith' WHERE id = 1;
-- PostgreSQL translates this to an UPDATE on the users table
```

## 4. Materialized Views

A materialized view stores the query result on disk — it's a cached copy that must be refreshed:

```sql
-- Create materialized view (executes the query and stores results)
CREATE MATERIALIZED VIEW monthly_revenue AS
SELECT
    DATE_TRUNC('month', created_at) AS month,
    COUNT(*) AS order_count,
    SUM(total_cents) AS total_cents
FROM orders
WHERE status = 'paid'
GROUP BY DATE_TRUNC('month', created_at);

CREATE UNIQUE INDEX ON monthly_revenue (month);  -- required for CONCURRENTLY refresh

-- Query (fast — reads from stored data, not live orders table)
SELECT * FROM monthly_revenue ORDER BY month DESC LIMIT 12;

-- Refresh (update the stored data)
REFRESH MATERIALIZED VIEW CONCURRENTLY monthly_revenue;
-- CONCURRENTLY: doesn't block reads during refresh (requires unique index)
-- Without CONCURRENTLY: blocks all reads during refresh

-- Schedule refresh (pg_cron extension or application cron job)
SELECT cron.schedule('0 * * * *',  -- every hour
    $$REFRESH MATERIALIZED VIEW CONCURRENTLY monthly_revenue$$);
```

## 5. Senior Engineer Perspective

**Use database features pragmatically:**
- `updated_at` triggers: yes — enforced consistently, no way to forget it
- Audit triggers: yes — database-level guarantee of completeness
- Business logic triggers: no — hard to test, invisible to developers, difficult to debug

Materialized views are underused. For any dashboard or reporting query that runs frequently and doesn't need real-time data, a materialized view can reduce a 5-second query to 5ms. Refresh every 5-60 minutes depending on freshness needs.
