## NoSQL Philosophy (5.33)

NoSQL databases trade some ACID guarantees for better scalability, flexibility, or performance on specific access patterns.

```
BASE PROPERTIES (NoSQL):
  Basically Available:  System responds even during failures
  Soft state:          Data may change without input (replication)
  Eventually consistent: All replicas converge, given enough time

WHY NoSQL:
  - Schema flexibility (add fields without migrations)
  - Horizontal scalability (sharding built-in)
  - Access pattern optimization (key-value: O(1) lookup, document: single round-trip)
  - Specific data models (graph, time-series, document)

WHY NOT NoSQL:
  - Weak consistency = possible data loss/corruption
  - No ACID transactions (most NoSQL)
  - Less powerful querying (no arbitrary JOINs)
  - Schema freedom = data model inconsistency over time
```

## MongoDB (5.34): Document Database

MongoDB stores data as BSON documents (binary JSON). Collections = tables, Documents = rows.

```javascript
// MongoDB: flexible document model
db.users.insertOne({
    email: "alice@example.com",
    name: "Alice",
    address: {                    // nested document
        street: "123 Main St",
        city: "New York"
    },
    tags: ["admin", "premium"],   // array
    metadata: { source: "web", tier: "gold" }
});

// Query
db.users.find({ "address.city": "New York", tags: "admin" });
db.users.find({ "metadata.tier": { $in: ["gold", "platinum"] } });

// Aggregation pipeline (MongoDB's equivalent of SQL GROUP BY + joins)
db.orders.aggregate([
    { $match: { status: "paid" } },
    { $group: {
        _id: "$user_id",
        total: { $sum: "$amount" },
        count: { $sum: 1 }
    }},
    { $sort: { total: -1 } },
    { $limit: 10 }
]);

// Indexes
db.users.createIndex({ email: 1 }, { unique: true });
db.orders.createIndex({ user_id: 1, created_at: -1 });
db.products.createIndex({ name: "text", description: "text" });  // text search
```

**MongoDB's ACID evolution:** MongoDB 4.0+ supports multi-document ACID transactions. Previously only single-document operations were atomic.

**When MongoDB over PostgreSQL:**
- Truly flexible/evolving schema (document structure varies significantly between records)
- Deeply nested hierarchical data that maps naturally to documents
- Horizontal write scaling beyond what single-node PostgreSQL can handle
- The "JSON-first" data model genuinely fits your domain

**When NOT MongoDB:**
- You need complex JOINs across many collections
- Strong consistency and transactions are critical
- Your data is actually relational (user → orders → products is NOT a document use case)

## Redis (5.35): In-Memory Data Structure Store

Redis is not just a cache — it's a versatile in-memory data store with persistence options.

```python
import redis

r = redis.Redis(host='localhost', port=6379, decode_responses=True)

# STRING: most basic — cache any value
r.set("user:123", json.dumps({"id": 123, "name": "Alice"}))
r.setex("session:abc", 3600, "user_id=123")  # expires in 3600s
r.get("user:123")
r.incr("page_views:homepage")  # atomic increment

# HASH: like a dict (good for objects)
r.hset("user:123", mapping={"name": "Alice", "email": "alice@example.com", "age": "30"})
r.hget("user:123", "name")       # "Alice"
r.hgetall("user:123")            # all fields

# LIST: ordered list, push/pop both ends (queue or stack)
r.lpush("job_queue", "job:001", "job:002")
r.rpop("job_queue")  # FIFO queue (push left, pop right)
r.lrange("notifications:user:123", 0, 9)  # get 10 most recent

# SET: unordered unique elements
r.sadd("online_users", "user:1", "user:2", "user:3")
r.sismember("online_users", "user:1")  # True
r.smembers("online_users")
r.sinterstore("mutual_friends", "friends:alice", "friends:bob")

# SORTED SET: set with score (leaderboards, time-series indexes)
r.zadd("leaderboard", {"alice": 9500, "bob": 8200, "carol": 10100})
r.zrangebyscore("leaderboard", 8000, 10000, withscores=True)
r.zrank("leaderboard", "alice")   # rank (0-indexed)

# STREAMS: append-only log (like Kafka, lightweight)
r.xadd("events", {"type": "login", "user_id": "123"})
r.xread({"events": "0"}, count=10)  # read from beginning
```

**Redis persistence modes:**
- `RDB`: snapshots at intervals (fast restarts, possible data loss)
- `AOF`: log every write command (durable, slower)
- `RDB+AOF`: both (recommended for important Redis data)
- None: pure cache mode (fastest, data lost on restart)

**Redis Cluster (5.37):** Automatic sharding across nodes. Data divided into 16,384 hash slots, distributed across nodes. Reads and writes go to the shard owning the key's slot.

## DynamoDB (5.38): AWS Managed Key-Value / Document

DynamoDB is AWS's fully managed NoSQL database — unlimited scale with millisecond latency, but you must design your access patterns upfront.

```python
import boto3

dynamodb = boto3.resource("dynamodb")
table = dynamodb.Table("Orders")

# Put item
table.put_item(Item={
    "PK": "USER#123",              # Partition Key
    "SK": "ORDER#2024-06-15#456",  # Sort Key
    "status": "paid",
    "total_cents": 5000,
    "user_email": "alice@example.com",
})

# Query (must use PK; SK is optional range filter)
table.query(
    KeyConditionExpression=Key("PK").eq("USER#123") &
                           Key("SK").begins_with("ORDER#")
)

# Global Secondary Index (GSI): alternate access patterns
table.query(
    IndexName="StatusIndex",
    KeyConditionExpression=Key("status").eq("pending")
)
```

**DynamoDB's data model philosophy: single-table design.** Instead of multiple tables joined at query time, related entities co-exist in one table using creative key patterns. Learning curve is steep but queries are O(1) regardless of data size.

**DynamoDB's pricing:** Pay per request or provisioned capacity. Can be very cheap (serverless workloads) or expensive (high throughput). Always test cost before commitment.

## Cassandra (5.39): Wide-Column, Write-Optimized

Cassandra (Facebook, 2008) is designed for massive write throughput and availability across multiple data centers.

```
CASSANDRA DATA MODEL:
  - Wide-column store: rows can have different columns
  - Partitioned by partition key (determines which node stores the row)
  - Sorted within partition by clustering columns

CQL (Cassandra Query Language — similar to SQL):
```

```sql
-- Table definition with partition key and clustering column
CREATE TABLE order_events (
    order_id UUID,
    event_time TIMESTAMP,    -- clustering column (sorts within partition)
    event_type TEXT,
    payload TEXT,
    PRIMARY KEY (order_id, event_time)
) WITH CLUSTERING ORDER BY (event_time DESC);

-- Query MUST include partition key (no full table scans!)
SELECT * FROM order_events WHERE order_id = ?;
SELECT * FROM order_events WHERE order_id = ? AND event_time > ?;

-- Tunable consistency
-- QUORUM: majority of replicas must respond (strong consistency)
-- ONE: any one replica (eventual consistency, fastest)
SELECT * FROM orders WHERE order_id = ? CONSISTENCY QUORUM;
```

**When Cassandra:** Time-series data (IoT, metrics), write-heavy workloads (millions of events/second), multi-datacenter replication requirements.

**When not Cassandra:** You need flexible queries, complex aggregations, or don't have high write volume (Cassandra's operational complexity is high — you need dedicated expertise).

## Firestore / Firebase (5.40)

Google's managed document database, excellent for mobile/web real-time sync:

```javascript
import { doc, getDoc, setDoc, collection, query, where, onSnapshot } from 'firebase/firestore';

// Write
await setDoc(doc(db, 'users', 'user123'), { name: 'Alice', email: 'alice@example.com' });

// Real-time listener (WebSocket-based, offline support)
onSnapshot(query(collection(db, 'orders'), where('userId', '==', 'user123')),
    (snapshot) => {
        snapshot.docChanges().forEach(change => {
            console.log(change.type, change.doc.data());  // 'added', 'modified', 'removed'
        });
    }
);
```

**Firestore is excellent for:** Mobile/web apps with real-time requirements, offline capability, Google Cloud users, apps where Firebase Auth + Firestore + FCM (push notifications) is the full stack.

**Firestore's limitation:** Complex queries require pre-defined indexes. No arbitrary `OR` conditions, no aggregations without workarounds.

## Neo4j (5.43): Graph Database

Graph databases store nodes and relationships as first-class entities. Excel at traversal queries (friends-of-friends, recommendations, pathfinding).

```cypher
// Cypher query language
// Find users connected through mutual friends within 2 hops
MATCH (alice:User {name: 'Alice'})-[:FRIENDS_WITH*2]-(stranger:User)
WHERE NOT (alice)-[:FRIENDS_WITH]-(stranger)
RETURN stranger.name, COUNT(*) AS mutual_friends
ORDER BY mutual_friends DESC LIMIT 10;

// Shortest path
MATCH path = shortestPath(
    (alice:User {name: 'Alice'})-[:FOLLOWS*]-(target:User {name: 'Bob'})
)
RETURN length(path), [node in nodes(path) | node.name];
```

**Use graph databases for:** Social networks, recommendation engines, fraud detection (transaction patterns), knowledge graphs, network topology.

## InfluxDB and TimescaleDB (5.44-5.45)

```
TIME-SERIES DATABASES:
  Optimized for: metrics, sensor data, IoT, financial tick data
  Key feature: efficient storage + query of high-frequency time-stamped data

INFLUXDB:
  Purpose-built time-series, no SQL
  InfluxQL / Flux query languages
  Best for: IoT, metrics, monitoring
  Managed: InfluxDB Cloud

TIMESCALEDB (preferred for many use cases):
  PostgreSQL extension — full SQL + time-series optimizations
  Hypertables: automatically partitioned by time
  Continuous aggregates: auto-refreshing materialized views
  Compression: 20-200x compression for time-series data
  Best for: if you're already on PostgreSQL or need SQL analytics
```

```sql
-- TimescaleDB: create hypertable (auto-partitioned by time)
CREATE TABLE metrics (
    time TIMESTAMPTZ NOT NULL,
    device_id TEXT,
    temperature DOUBLE PRECISION,
    humidity DOUBLE PRECISION
);
SELECT create_hypertable('metrics', 'time');

-- Continuous aggregate: auto-refreshing hourly average
CREATE MATERIALIZED VIEW hourly_avg
WITH (timescaledb.continuous) AS
SELECT time_bucket('1 hour', time) AS hour,
       device_id,
       AVG(temperature) AS avg_temp
FROM metrics
GROUP BY 1, 2;
```

## NoSQL Selection Guide

```
KEY-VALUE (simple lookup by ID):
  Redis:      In-memory, caching, sessions, queues, real-time
  DynamoDB:   AWS, serverless, extreme scale, complex access patterns

DOCUMENT (JSON objects, flexible schema):
  MongoDB:    General-purpose, flexible schema, aggregations
  Firestore:  Mobile/web apps, real-time sync, Google ecosystem

WIDE-COLUMN (time-series, write-heavy):
  Cassandra:  Massive write throughput, multi-datacenter, IoT events

GRAPH (connected data, traversals):
  Neo4j:      Social networks, recommendations, fraud detection

TIME-SERIES (metrics, sensor data):
  TimescaleDB:  SQL + time-series on PostgreSQL
  InfluxDB:     Pure time-series, IoT, monitoring

VECTOR (AI similarity search):
  pgvector, Pinecone, Weaviate, Qdrant (see Phase 13)
```
