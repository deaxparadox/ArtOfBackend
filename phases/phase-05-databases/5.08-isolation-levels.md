## 1. The Problem: Concurrency Anomalies

When multiple transactions run concurrently, they can interfere in ways that produce incorrect results. Isolation levels control which anomalies are allowed in exchange for better performance.

**The four anomalies:**

```
DIRTY READ: Read a row that was modified by a transaction that hasn't committed yet.
  T1: UPDATE balance = 900 (not yet committed)
  T2: SELECT balance → 900  ← reads T1's uncommitted value
  T1: ROLLBACK
  T2 made decisions based on data that never existed.

NON-REPEATABLE READ: Read the same row twice, get different values.
  T1: SELECT balance → 1000
  T2: UPDATE balance = 900, COMMIT
  T1: SELECT balance → 900  ← different from first read!
  T1 made decisions assuming balance didn't change.

PHANTOM READ: Re-execute a query, get different rows.
  T1: SELECT COUNT(*) FROM orders WHERE user_id = 1 → 5
  T2: INSERT INTO orders (user_id, ...) VALUES (1, ...), COMMIT
  T1: SELECT COUNT(*) FROM orders WHERE user_id = 1 → 6
  T1 sees a "phantom" new row that didn't exist in its first read.

SERIALIZATION ANOMALY: Two transactions interact in a way that no serial order could produce.
  T1 reads A and writes B based on A
  T2 reads B and writes A based on B
  Result impossible if either ran first.
```

## 2. The Four Isolation Levels

```
LEVEL               DIRTY READ   NON-REPEATABLE   PHANTOM   SERIALIZATION
                                      READ           READ      ANOMALY
READ UNCOMMITTED      Allowed        Allowed        Allowed    Allowed
READ COMMITTED       Prevented       Allowed        Allowed    Allowed
REPEATABLE READ      Prevented      Prevented     Prevented*   Allowed
SERIALIZABLE         Prevented      Prevented      Prevented   Prevented

* PostgreSQL's REPEATABLE READ prevents phantoms too (better than SQL standard)

PRACTICAL USE:
  READ UNCOMMITTED: almost never used (not even supported in PostgreSQL)
  READ COMMITTED:   PostgreSQL default — good for most use cases
  REPEATABLE READ:  when you need consistent snapshot across multiple reads
  SERIALIZABLE:     financial transactions, inventory, any write skew protection needed
```

## 3. How PostgreSQL Implements Isolation: MVCC

PostgreSQL uses **MVCC (Multi-Version Concurrency Control)** rather than traditional locking:

```
MVCC CORE IDEA:
  Instead of one version of a row, keep multiple versions.
  Each version has: xmin (created by transaction), xmax (deleted by transaction)

  When T2 reads a row that T1 is modifying:
    LOCKING approach: T2 waits for T1 to finish (blocks)
    MVCC approach: T2 reads the old version (doesn't block)

  READ COMMITTED: each statement sees data committed before it started
  REPEATABLE READ: entire transaction sees data committed before the transaction started
  SERIALIZABLE: detects and prevents any non-serializable execution (SSI algorithm)
```

## 4. Setting Isolation Level

```sql
-- Per-transaction isolation
BEGIN TRANSACTION ISOLATION LEVEL REPEATABLE READ;
  SELECT balance FROM accounts WHERE id = 1;  -- snapshot taken here
  -- ... other operations ...
  SELECT balance FROM accounts WHERE id = 1;  -- same value as first read
COMMIT;

-- Per-session isolation (affects all subsequent transactions)
SET SESSION CHARACTERISTICS AS TRANSACTION ISOLATION LEVEL SERIALIZABLE;

-- Per-statement (inside a transaction)
SET TRANSACTION ISOLATION LEVEL READ COMMITTED;
```

```python
# asyncpg: set isolation level
async with pool.acquire() as conn:
    async with conn.transaction(isolation='serializable'):
        # This transaction uses serializable isolation
        balance = await conn.fetchval("SELECT balance FROM accounts WHERE id = $1", 1)
        await conn.execute("UPDATE accounts SET balance = $1 WHERE id = $2",
                           balance - 100, 1)

# SQLAlchemy: set isolation level
async with session.begin():
    await session.execute(text("SET TRANSACTION ISOLATION LEVEL REPEATABLE READ"))
    ...
```

## 5. Serialization Failures: Detecting and Retrying

At SERIALIZABLE isolation, PostgreSQL may abort a transaction with a serialization error when it detects a conflict that would make the result non-serializable. Your code MUST handle this:

```python
from asyncpg import SerializationFailureError
import asyncio

async def serializable_operation(pool, max_retries=3):
    for attempt in range(max_retries):
        try:
            async with pool.acquire() as conn:
                async with conn.transaction(isolation='serializable'):
                    # Your operations here
                    result = await conn.fetchval("SELECT COUNT(*) FROM orders")
                    await conn.execute("INSERT INTO order_summaries (count) VALUES ($1)", result)
                    return result

        except SerializationFailureError:
            if attempt == max_retries - 1:
                raise
            # Exponential backoff before retry
            await asyncio.sleep(0.1 * (2 ** attempt))

    raise Exception("Max retries exceeded")
```

## 6. Choosing the Right Isolation Level

```
USE READ COMMITTED (default) WHEN:
  ✓ Most application reads (user profiles, product listings)
  ✓ Simple writes (INSERT, UPDATE single row)
  ✓ You don't need consistency across multiple reads in one transaction
  ✓ Maximum concurrency is important

USE REPEATABLE READ WHEN:
  ✓ Transaction reads the same data multiple times and must see consistent values
  ✓ Generating a report that reads from multiple tables (consistent snapshot)
  ✓ Long-running read-only queries that must not see concurrent updates

USE SERIALIZABLE WHEN:
  ✓ Financial operations (debit/credit pairs)
  ✓ Inventory operations (check stock, then reserve)
  ✓ Any operation where "read then write" based on the read must be atomic
  ✓ When you'd otherwise need explicit SELECT ... FOR UPDATE

PERFORMANCE IMPACT:
  READ COMMITTED:  minimal overhead, highest concurrency
  REPEATABLE READ: snapshot overhead per transaction
  SERIALIZABLE:    conflict detection overhead, occasional transaction abort + retry
```

## 7. Senior Engineer Perspective

**The isolation level most engineers should use more:**

SERIALIZABLE. It's the only level where "if this works in a test with one user, it works correctly with 1,000 concurrent users." READ COMMITTED requires you to manually reason about all possible interleavings and add appropriate locks. SERIALIZABLE does that reasoning for you — at the cost of occasional retries.

**The real cost of READ COMMITTED:**

At READ COMMITTED, this sequence can produce wrong results:
```
T1: reads available_stock = 10 (10 items in warehouse)
T2: reads available_stock = 10
T1: reserves 3 items, updates available_stock = 7, commits
T2: reserves 3 items, updates available_stock = 7, commits  ← WRONG! Should be 4
```

With SERIALIZABLE: T2's transaction would be aborted and retried, reading the committed value of 7 from T1.
