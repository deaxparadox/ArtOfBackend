## 1. What Are Joins?

Joins combine rows from two or more tables based on a related column. This is the mechanism for querying across related data — the core operation that makes relational databases powerful.

```
USERS:                          ORDERS:
id  | name                      id  | user_id | amount
----+------                     ----+---------+-------
1   | Alice                     1   | 1       | 5000
2   | Bob                       2   | 1       | 2500
3   | Carol  ← no orders        3   | 2       | 7500
                                4   | 99      | 1000  ← no matching user
```

---

## 2. INNER JOIN: Only Rows That Match in Both Tables

```sql
SELECT u.name, o.id AS order_id, o.amount
FROM users u
INNER JOIN orders o ON u.id = o.user_id;

-- Result: only users with orders, only orders with users
-- name  | order_id | amount
-- Alice | 1        | 5000
-- Alice | 2        | 2500
-- Bob   | 3        | 7500
-- (Carol excluded — no orders)
-- (order 4 excluded — user 99 doesn't exist)
```

**Default JOIN is INNER JOIN** — you can write `JOIN` without `INNER`.

---

## 3. LEFT JOIN: All Rows from Left Table + Matched Rows from Right

```sql
SELECT u.name, o.id AS order_id, o.amount
FROM users u
LEFT JOIN orders o ON u.id = o.user_id;

-- Result: ALL users, NULL for orders where no match
-- name  | order_id | amount
-- Alice | 1        | 5000
-- Alice | 2        | 2500
-- Bob   | 3        | 7500
-- Carol | NULL     | NULL   ← Carol appears with NULLs

-- Find users WITH NO orders:
SELECT u.name
FROM users u
LEFT JOIN orders o ON u.id = o.user_id
WHERE o.id IS NULL;  -- ← only rows where no match occurred
-- Result: Carol
```

**LEFT JOIN is the most commonly used join type** for "include everything from the primary table."

---

## 4. RIGHT JOIN: Mirror Image of LEFT JOIN

```sql
SELECT u.name, o.id AS order_id, o.amount
FROM users u
RIGHT JOIN orders o ON u.id = o.user_id;

-- Result: ALL orders, NULL for users where no match
-- name  | order_id | amount
-- Alice | 1        | 5000
-- Alice | 2        | 2500
-- Bob   | 3        | 7500
-- NULL  | 4        | 1000   ← order 4 (user 99 not in users table)
```

RIGHT JOIN is rarely used in practice — the same result can always be written as a LEFT JOIN by swapping the table order. Most style guides recommend LEFT JOIN exclusively for clarity.

---

## 5. FULL OUTER JOIN: All Rows from Both Tables

```sql
SELECT u.name, o.id AS order_id, o.amount
FROM users u
FULL OUTER JOIN orders o ON u.id = o.user_id;

-- Result: everything from both sides, NULLs where no match
-- name  | order_id | amount
-- Alice | 1        | 5000
-- Alice | 2        | 2500
-- Bob   | 3        | 7500
-- Carol | NULL     | NULL   ← no orders
-- NULL  | 4        | 1000   ← orphan order
```

Use case: data reconciliation — finding rows that exist in one table but not the other (data import validation, detecting orphans).

---

## 6. CROSS JOIN: Cartesian Product (Every Combination)

```sql
SELECT u.name, p.name AS product_name
FROM users u
CROSS JOIN products p;

-- Result: every user × every product combination
-- 3 users × 5 products = 15 rows
-- WARNING: 1000 users × 1000 products = 1,000,000 rows
-- Always add a WHERE condition unless you truly want all combinations
```

Use case: generating test data, calculating all possible combinations (chess board, price matrices).

---

## 7. SELF JOIN: A Table Joining With Itself

Used for hierarchical data (employees and their managers, categories and parent categories):

```sql
-- Employee table with self-referential manager_id
SELECT
    e.name AS employee,
    m.name AS manager
FROM employees e
LEFT JOIN employees m ON e.manager_id = m.id;

-- Result:
-- employee | manager
-- Alice    | NULL     ← Alice is the CEO, no manager
-- Bob      | Alice
-- Carol    | Alice
-- Dave     | Bob
```

---

## 8. JOIN Conditions: ON vs USING vs NATURAL

```sql
-- ON: explicit condition (most flexible, clearest)
SELECT * FROM users u JOIN orders o ON u.id = o.user_id;

-- USING: when column names match (cleaner for same-name columns)
SELECT * FROM orders JOIN order_items USING (order_id);
-- Equivalent to: ON orders.order_id = order_items.order_id
-- Result: order_id appears once (not twice)

-- NATURAL JOIN: automatically joins on all columns with matching names
-- NEVER use this in production — fragile, behavior changes when columns are added
SELECT * FROM users NATURAL JOIN orders;  -- DON'T USE
```

## 9. Multiple Joins

```sql
-- Three-way join: users → orders → order_items → products
SELECT
    u.name,
    o.id AS order_id,
    p.name AS product_name,
    oi.quantity,
    oi.unit_price_cents
FROM users u
JOIN orders o ON u.id = o.user_id
JOIN order_items oi ON o.id = oi.order_id
JOIN products p ON oi.product_id = p.id
WHERE u.id = 123
  AND o.status = 'paid'
ORDER BY o.created_at DESC, p.name;
```

## 10. JOIN Performance Patterns

```sql
-- WRONG: Implicit join (old syntax, harder to read, easy to miss conditions)
SELECT u.name, o.id
FROM users u, orders o
WHERE u.id = o.user_id;  -- if you forget this WHERE: CROSS JOIN!

-- RIGHT: Explicit JOIN (modern, clear intent)
SELECT u.name, o.id
FROM users u
JOIN orders o ON u.id = o.user_id;

-- JOIN ON multiple conditions (compound join condition)
SELECT *
FROM shipments s
JOIN warehouses w ON s.warehouse_id = w.id
                  AND s.region = w.region;  -- second condition in JOIN, not WHERE

-- Filtering in JOIN vs WHERE:
-- Both produce same result for INNER JOIN, but:
-- For LEFT JOIN, filtering in JOIN condition preserves NULLs
-- Filtering in WHERE on right-table column turns LEFT JOIN into INNER JOIN

-- LEFT JOIN with filter on right table:
SELECT u.name, o.id
FROM users u
LEFT JOIN orders o ON u.id = o.user_id AND o.status = 'paid';
-- Carol appears with NULL (no paid orders) — Carol is preserved

-- LEFT JOIN with WHERE on right table:
SELECT u.name, o.id
FROM users u
LEFT JOIN orders o ON u.id = o.user_id
WHERE o.status = 'paid';
-- Carol disappears — WHERE eliminates the NULL row (became INNER JOIN behavior!)
```

## 11. Senior Engineer Perspective

**What a junior gets wrong:**

Not understanding the LEFT JOIN + WHERE NULL trap (shown above). Filtering on a column from the right table in the WHERE clause silently converts a LEFT JOIN to an INNER JOIN. This is one of the most common SQL bugs in production code — queries that appear to work in testing (when all rows have matches) but produce wrong results when some don't.

**The mental model for joins:**

Think of joins as building a "temporary super-table" by matching rows. INNER JOIN: only keep rows that matched in both. LEFT JOIN: keep all left rows, fill right-side columns with NULL if no match. Once you have this mental model, all join types are just variations on "which rows do I keep when there's no match?"

**Performance:** Joins on indexed foreign keys are fast. Joins on unindexed columns are table scans. For large tables, always ensure the join column has an index on both sides.
