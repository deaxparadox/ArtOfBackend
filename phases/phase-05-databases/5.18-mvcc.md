## 1. What Is MVCC?

MVCC (Multi-Version Concurrency Control) allows readers and writers to coexist without blocking each other. Instead of locking a row when it's read, the database keeps multiple versions of the row. Each transaction sees the version that was current when its snapshot was taken.

**The key insight:** Readers never block writers. Writers never block readers. Only writer-writer conflicts cause blocking.

---

## 2. How MVCC Works in PostgreSQL

```
EVERY ROW HAS HIDDEN COLUMNS:
  xmin: transaction ID that created this version
  xmax: transaction ID that deleted/updated this version (0 if current)
  ctid: physical location (page, offset) of this row version

EXAMPLE: User 1's balance changes from $500 to $400

Heap:
  (xmin=10, xmax=0,  balance=500)  ← created by T10, current version
  ↓ T15 runs UPDATE balance=400
  (xmin=10, xmax=15, balance=500)  ← now "dead" (deleted by T15)
  (xmin=15, xmax=0,  balance=400)  ← new version created by T15

T12 started before T15 committed:
  → Sees row with xmin=10, xmax=0 (T15 not in its snapshot)
  → Reads balance=500

T16 started after T15 committed:
  → T15 is in its snapshot (committed)
  → Sees row with xmin=15, xmax=0
  → Reads balance=400
```

## 3. Vacuum: Cleaning Up Dead Tuples

MVCC creates dead row versions. These accumulate and need cleanup:

```sql
-- Autovacuum runs automatically (check its progress)
SELECT schemaname, tablename, n_dead_tup, n_live_tup,
       last_autovacuum, last_autoanalyze
FROM pg_stat_user_tables
WHERE n_dead_tup > 1000
ORDER BY n_dead_tup DESC;

-- Manual vacuum (when autovacuum can't keep up)
VACUUM ANALYZE orders;         -- clean dead tuples + update statistics
VACUUM FULL orders;            -- rewrites entire table (blocks all access!)
                               -- ← only use during maintenance window
VACUUM (VERBOSE) orders;       -- verbose output to see what it's doing

-- Autovacuum configuration (per-table override for high-churn tables)
ALTER TABLE orders SET (
    autovacuum_vacuum_scale_factor = 0.01,   -- vacuum when 1% of rows are dead (default 20%)
    autovacuum_analyze_scale_factor = 0.005  -- analyze when 0.5% changed
);
```

**Transaction ID Wraparound (Critical):**

PostgreSQL uses 32-bit transaction IDs. With wraparound (every ~2 billion transactions), old transactions could appear "in the future." PostgreSQL runs emergency VACUUM to prevent this. Monitoring `pg_database` for `age(datfrozenxid)` approaching 2 billion is critical — at 1.5 billion, PostgreSQL refuses new connections to force VACUUM.

## 4. Long-Running Transactions and MVCC

```
PROBLEM:
  T1: BEGIN (takes snapshot at xid=1000)
  ... 2 hours pass ...
  T1 still running

  During these 2 hours:
  - Dead rows CAN'T be cleaned (vacuum skips them — T1 might need them)
  - Table bloat grows
  - Queries slow down

SYMPTOMS:
  - pg_stat_activity shows a query running for hours with state='idle in transaction'
  - Table size growing rapidly despite few writes
  - Vacuum running but not reducing table size

DETECTION:
  SELECT pid, now() - pg_stat_activity.query_start AS duration, query, state
  FROM pg_stat_activity
  WHERE state = 'idle in transaction'
    AND (now() - query_start) > INTERVAL '5 minutes';

PREVENTION:
  -- Set idle_in_transaction_session_timeout
  ALTER SYSTEM SET idle_in_transaction_session_timeout = '5min';
  -- Kills transactions that have been idle (in transaction) for 5+ minutes
  SELECT pg_reload_conf();
```

## 5. Senior Engineer Perspective

**MVCC is why PostgreSQL can be simultaneously the fastest and safest database for concurrent workloads.** Reads are always non-blocking. Writers only block other writers on the same row. This allows thousands of concurrent connections without the bottleneck that traditional locking databases had.

**The operational responsibility:** MVCC requires maintenance (VACUUM). A database with autovacuum disabled or misconfigured will bloat, slow down, and eventually stop accepting new transactions. Monitor `n_dead_tup`, watch for long-running transactions, and ensure autovacuum is running effectively for your write patterns.
