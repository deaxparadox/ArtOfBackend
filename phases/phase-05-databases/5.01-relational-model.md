## 1. What Is the Relational Model?

The relational model (Edgar Codd, 1970) is the theoretical foundation for all SQL databases. It organizes data into **relations** (tables) — two-dimensional structures with rows (tuples) and columns (attributes). Relationships between entities are expressed through **foreign keys**, not through physical pointers or nested structures.

**Why it became dominant:** Before the relational model, databases used hierarchical or network models where data was linked by physical pointers. Adding a new type of query often required restructuring the database. The relational model provided a query language (SQL) that could express any question about the data without knowing how it was physically stored.

**The real-world analogy:**

A spreadsheet. Each sheet is a table. Each row is one record. Each column is one attribute of that record. Foreign keys are like references between sheets — "Order row 5 references Customer row 12."

## 2. Core Concepts

### Tables (Relations)

A table represents a **set of entities** of the same type. Every row is one entity. Every column is one attribute.

```
USERS TABLE:
  id  | email              | name     | created_at
  ----+--------------------+----------+---------------------
  1   | alice@example.com  | Alice    | 2024-01-15 10:30:00
  2   | bob@example.com    | Bob      | 2024-02-20 14:15:00
  3   | carol@example.com  | Carol    | 2024-03-01 09:00:00

Rules:
  - Each row is unique (enforced by PRIMARY KEY)
  - Each column has a defined type (integer, text, timestamp, etc.)
  - Column order is irrelevant (SQL selects by name, not position)
  - Row order is irrelevant (use ORDER BY when order matters)
```

### Primary Keys

A **primary key** is a column (or combination of columns) that uniquely identifies each row. No two rows can have the same primary key value. No row can have NULL as a primary key.

```sql
CREATE TABLE users (
    id          BIGSERIAL PRIMARY KEY,  -- auto-incrementing integer
    email       VARCHAR(255) NOT NULL,
    created_at  TIMESTAMPTZ DEFAULT NOW()
);

-- Alternative: UUID primary key
CREATE TABLE products (
    id    UUID DEFAULT gen_random_uuid() PRIMARY KEY,
    name  TEXT NOT NULL
);
```

**Natural vs. surrogate keys:**
- **Natural key:** a real-world identifier that happens to be unique (email, SSN, ISBN)
  - Risk: natural identifiers change (person changes their email, ISBN gets reissued)
- **Surrogate key:** an artificial identifier (auto-incrementing id, UUID) with no real-world meaning
  - Preferred: stable, never changes, no business logic attached

### Foreign Keys

A **foreign key** is a column in one table that references the primary key of another table. This is how relationships are expressed.

```
ORDERS TABLE:
  id  | user_id | total_cents | status
  ----+---------+-------------+--------
  1   | 1       | 5000        | paid    ← user_id=1 references users.id=1
  2   | 1       | 2500        | pending ← same user, multiple orders
  3   | 2       | 7500        | paid    ← user_id=2 references users.id=2

user_id is a FOREIGN KEY → users(id)
```

```sql
CREATE TABLE orders (
    id          BIGSERIAL PRIMARY KEY,
    user_id     BIGINT NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    total_cents INTEGER NOT NULL CHECK (total_cents > 0),
    status      VARCHAR(20) NOT NULL DEFAULT 'pending',
    created_at  TIMESTAMPTZ DEFAULT NOW()
);
```

**ON DELETE behavior:**
- `CASCADE` — when user is deleted, all their orders are deleted too
- `RESTRICT` — prevent deleting a user if they have orders
- `SET NULL` — set user_id to NULL when user is deleted
- `NO ACTION` (default) — similar to RESTRICT, checked at end of transaction

## 3. Relationships Between Tables

### One-to-Many (Most Common)

One user has many orders. One order belongs to one user.

```
users (1) ──────< orders (many)
  id              user_id (FK)
```

Implemented by: foreign key on the "many" side pointing to the "one" side.

### Many-to-Many

A user can have many roles. A role can belong to many users.

```
users (many) >────< roles (many)
              |
              user_roles (junction table)
              user_id (FK) + role_id (FK)
```

```sql
CREATE TABLE user_roles (
    user_id  BIGINT REFERENCES users(id) ON DELETE CASCADE,
    role_id  BIGINT REFERENCES roles(id) ON DELETE CASCADE,
    PRIMARY KEY (user_id, role_id)  -- composite primary key
);
```

### One-to-One

A user has one profile. A profile belongs to one user.

```sql
CREATE TABLE profiles (
    user_id    BIGINT PRIMARY KEY REFERENCES users(id),  -- PK is also FK
    bio        TEXT,
    avatar_url TEXT
);
```

## 4. Constraints: Enforcing Data Integrity

```sql
CREATE TABLE products (
    id          BIGSERIAL PRIMARY KEY,
    name        VARCHAR(200) NOT NULL,                     -- required
    slug        VARCHAR(200) UNIQUE NOT NULL,              -- must be unique
    price_cents INTEGER NOT NULL CHECK (price_cents >= 0), -- must be positive
    category    VARCHAR(50) NOT NULL DEFAULT 'uncategorized',
    sku         VARCHAR(50),
    UNIQUE (category, sku),                                -- composite unique constraint
    CONSTRAINT valid_sku CHECK (sku IS NULL OR length(sku) >= 3)
);
```

**Constraint types:**
- `NOT NULL` — column must always have a value
- `UNIQUE` — no two rows can have the same value in this column
- `PRIMARY KEY` — NOT NULL + UNIQUE, identifies the row
- `FOREIGN KEY` — references another table's primary key
- `CHECK` — custom condition that must be true
- `DEFAULT` — value used when no value is provided on INSERT

Constraints are enforced at the database level, not just the application level. Even if your application has a bug that skips validation, the database will reject invalid data.

## 5. NULL: The Source of Much Confusion

NULL means "unknown" or "missing" — it is NOT the same as zero, empty string, or false.

```sql
-- NULL comparisons are tricky
SELECT * FROM users WHERE middle_name = NULL;    -- WRONG: returns 0 rows
SELECT * FROM users WHERE middle_name IS NULL;   -- RIGHT: uses IS NULL

SELECT * FROM users WHERE middle_name != 'Smith';  -- does NOT include NULLs
SELECT * FROM users WHERE middle_name != 'Smith' OR middle_name IS NULL;  -- includes NULLs

-- NULL in arithmetic
SELECT 5 + NULL;   -- NULL (unknown + anything = unknown)
SELECT 5 * NULL;   -- NULL

-- NULL in aggregations
SELECT AVG(score) FROM tests;  -- NULLs are EXCLUDED from aggregations
                                -- avg of (5, NULL, 3) = (5+3)/2 = 4, not (5+0+3)/3
```

**Production guidance:** Use `NOT NULL` constraints liberally. A NULL that means "unverified" or "optional" is fine. A NULL that means "0" or "false" is a design mistake.

## 6. Schema Design Principles

**Tables should represent one thing:**
```
GOOD:
  users table: user identity (id, email, name, password_hash)
  profiles table: user public info (bio, avatar, social links)
  addresses table: shipping addresses (street, city, country)

BAD:
  users table with 40 columns mixing identity, preferences,
  billing info, and shipping addresses
```

**Column naming conventions:**
```sql
-- Consistent, predictable naming
id              -- primary key (integer or UUID)
user_id         -- foreign key (table_name + _id)
created_at      -- when the row was created (TIMESTAMPTZ)
updated_at      -- when the row was last modified
deleted_at      -- for soft deletes (NULL = not deleted)
is_active       -- boolean flags start with is_ or has_
total_cents     -- store money as integers (cents, not dollars)
```

## 7. Real-World Context

### Why Instagram's database schema had ~20 tables at 1 billion users

Instagram's core schema was remarkably simple: users, media, comments, likes, follows, hashtags. Each table did one thing. The complexity was in query optimization and sharding — not schema complexity.

Large, complex schemas usually indicate missing abstraction (one giant table with mixed concerns) or premature optimization (pre-computing data that should be computed at query time).

### The cost of schema changes on large tables

Adding a `NOT NULL` column with no default to a 500 million row table requires touching every row. On PostgreSQL with a default value: PostgreSQL 11+ handles this without rewriting the table (it stores the default in catalog, adds it virtually). Without a default: the table must be rewritten row by row — this can lock the table for hours.

**Always design schema changes as non-blocking migrations:**
1. Add column as nullable (no lock, instant)
2. Backfill data in batches
3. Add NOT NULL constraint once data is populated

## 8. Senior Engineer Perspective

**What a junior gets wrong:**

Using strings where enums (or lookup tables) should be used. `status = "pending"` as a raw string means:
- Typos compile silently: `"pendng"` is stored without error
- Adding a new status requires a code search to find all usages
- No database-level validation

Better: `CHECK (status IN ('pending', 'paid', 'shipped', 'cancelled'))` or a dedicated `order_statuses` table.

**What a senior designs for:**

The schema is the hardest thing to change in a production system. Every schema change must be:
1. Backwards-compatible (old app code still works with new schema)
2. Non-blocking (no long-running locks on high-traffic tables)
3. Incremental (add then migrate then clean up)

"We need to rename this column" sounds simple. On a table with 100 million rows, queried 10,000 times per second, it is a multi-week migration project.
