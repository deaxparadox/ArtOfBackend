## 1. What Is a Database Migration?

A migration is a versioned, repeatable change to a database schema. Migrations allow you to:
- Track schema changes in version control (alongside your application code)
- Apply changes consistently across all environments (dev, staging, production)
- Roll back changes when something goes wrong
- Coordinate schema changes with application code deploys

## 2. Migration Tools

```python
# Alembic (SQLAlchemy ecosystem — Python)
alembic init migrations
alembic revision --autogenerate -m "add_phone_to_users"
alembic upgrade head       # apply all pending migrations
alembic downgrade -1       # rollback one migration
alembic history            # list all migrations and their status

# Django migrations
python manage.py makemigrations
python manage.py migrate
python manage.py showmigrations
python manage.py migrate myapp 0003  # migrate to specific version

# Flyway (Java/multi-language)
# Files named: V1__initial_schema.sql, V2__add_users.sql
flyway migrate
flyway info

# Liquibase (Java/multi-language)
# XML or YAML changesets
liquibase update

# Prisma (Node.js)
npx prisma migrate dev --name add_phone_to_users
npx prisma migrate deploy  # production

# Goose (Go)
goose postgres $DATABASE_URL up
goose postgres $DATABASE_URL down
```

## 3. Migration File Structure

```python
# Alembic migration example
"""add phone to users

Revision ID: 3a8b4c2d1e5f
Revises: 2a7b3c1d0e4f
Create Date: 2024-06-15 14:30:00
"""
from alembic import op
import sqlalchemy as sa

def upgrade() -> None:
    # Add column as nullable first (no table lock needed)
    op.add_column('users',
        sa.Column('phone', sa.String(20), nullable=True)
    )
    # Add index concurrently (no lock)
    op.create_index(
        'idx_users_phone', 'users', ['phone'],
        postgresql_concurrently=True  # non-locking!
    )

def downgrade() -> None:
    op.drop_index('idx_users_phone', table_name='users')
    op.drop_column('users', 'phone')
```

## 4. Zero-Downtime Migrations

Many schema changes can lock tables and block all reads/writes. Zero-downtime migration strategies avoid this.

### The Expand-Contract Pattern

```
PHASE 1 - EXPAND (backward compatible, can deploy without downtime):
  - Add new column (nullable)
  - Add new table
  - Add new index (CONCURRENTLY)
  - Backfill data in small batches

PHASE 2 - MIGRATE (update application code):
  - New code writes to BOTH old and new column/table
  - Reads from new column/table

PHASE 3 - CONTRACT (after all old code is gone):
  - Remove old column
  - Drop old index
  - Make new column NOT NULL (safe now — all rows have values)

Example: Rename column email → primary_email

Step 1 (expand):
  ALTER TABLE users ADD COLUMN primary_email VARCHAR(255);
  UPDATE users SET primary_email = email WHERE primary_email IS NULL;
  -- (Do in batches for large tables)

Step 2 (dual write):
  Deploy code that reads primary_email (falls back to email if null)
  Deploy code that writes to both columns

Step 3 (contract):
  ALTER TABLE users ALTER COLUMN primary_email SET NOT NULL;
  ALTER TABLE users DROP COLUMN email;
```

### Safe vs. Dangerous Operations

```
SAFE (no lock / brief lock):
  ✓ Add nullable column (PostgreSQL: instant metadata change)
  ✓ Add column with default (PostgreSQL 11+: metadata change, no rewrite)
  ✓ Add index CONCURRENTLY
  ✓ Drop index CONCURRENTLY
  ✓ Add table
  ✓ Add constraint NOT VALID + VALIDATE CONSTRAINT (separate steps)

DANGEROUS (long lock on high-traffic tables):
  ✗ Add NOT NULL column without default (rewrites table)
  ✗ Add column with volatile default (NOW(), random())
  ✗ Add index without CONCURRENTLY (full table lock)
  ✗ ALTER TYPE (changes column type — may rewrite table)
  ✗ Adding CHECK constraint (validates all existing rows)
  ✗ DROP COLUMN (marks column dead, deferred rewrite)
```

### Adding NOT NULL Constraint Safely

```sql
-- WRONG: locks table while validating all rows
ALTER TABLE users ALTER COLUMN phone SET NOT NULL;

-- RIGHT (PostgreSQL): add constraint NOT VALID, then validate separately
-- Step 1: Add constraint that only validates new rows (no lock on existing rows)
ALTER TABLE users
ADD CONSTRAINT users_phone_not_null
CHECK (phone IS NOT NULL) NOT VALID;
-- 'NOT VALID' = constraint only enforced for new rows, doesn't validate existing

-- Step 2 (background): Backfill NULLs
UPDATE users SET phone = '' WHERE phone IS NULL;

-- Step 3: Validate existing rows (takes ShareUpdateExclusiveLock, not full lock)
-- Allows concurrent reads and writes while validating
ALTER TABLE users VALIDATE CONSTRAINT users_phone_not_null;
```

## 5. Migration Best Practices

```
DO:
  ✓ Version every schema change as a migration file
  ✓ Include both upgrade() and downgrade()
  ✓ Test migrations on a copy of production data before deploying
  ✓ Run migrations separately from application deploys (before or after)
  ✓ Use CONCURRENTLY for indexes in production
  ✓ Backfill data in batches (not one giant UPDATE)
  ✓ Lock migrations table (prevent concurrent migration runs)

DON'T:
  ✗ Edit an already-applied migration (create a new one)
  ✗ Run migrations inside application startup (can timeout, cause race conditions)
  ✗ Drop columns or tables without verifying no code references them
  ✗ Rename columns without the expand-contract pattern
  ✗ Add NOT NULL without ensuring data exists first
```

---

## 6. Senior Engineer Perspective

**The most dangerous migration in production:** adding a NOT NULL column without `NOT VALID`. PostgreSQL must lock the table and verify every row. On a 100M row table, this takes minutes — during which every write to the table blocks. Always use the `NOT VALID` + `VALIDATE CONSTRAINT` pattern.

**The migration process in mature teams:**
1. Migration committed to version control with application code
2. CI runs migration against a staging database
3. Production deploy: migrate first, then deploy new code
4. Or: blue-green deploy with forward-compatible migration (works with both old and new code)
