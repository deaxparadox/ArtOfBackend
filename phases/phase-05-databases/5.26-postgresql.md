## PostgreSQL: The Gold Standard (5.26)

PostgreSQL (1996, UC Berkeley POSTGRES project) is the most feature-rich open-source relational database. It is the recommended default for any new backend project.

```sql
-- JSONB: store and query semi-structured data
CREATE TABLE events (
    id BIGSERIAL PRIMARY KEY,
    metadata JSONB NOT NULL DEFAULT '{}'
);

-- Insert JSON
INSERT INTO events (metadata) VALUES
('{"user_id": 123, "action": "login", "ip": "192.168.1.1", "tags": ["web", "mobile"]}');

-- Query JSON fields
SELECT * FROM events WHERE metadata->>'action' = 'login';
SELECT * FROM events WHERE metadata @> '{"user_id": 123}';  -- containment
SELECT * FROM events WHERE metadata->'tags' ? 'mobile';     -- array contains element
SELECT * FROM events WHERE (metadata->>'user_id')::int > 100;

-- Update JSON fields
UPDATE events SET metadata = metadata || '{"processed": true}'
WHERE id = 1;

-- GIN index on JSONB (for containment queries)
CREATE INDEX ON events USING GIN (metadata);
CREATE INDEX ON events USING GIN ((metadata->'tags'));

-- PostgreSQL-specific data types:
-- ARRAY: integer[], text[], uuid[]
-- UUID: native 128-bit UUID
-- INET/CIDR: IP addresses with subnet operations
-- RANGE: date ranges, integer ranges
-- HSTORE: key-value pairs (use JSONB instead)
-- TSVECTOR: full-text search vectors

-- Array operations
SELECT * FROM posts WHERE 'python' = ANY(tags);
SELECT * FROM posts WHERE tags @> ARRAY['python', 'backend'];
SELECT ARRAY_AGG(name ORDER BY name) FROM products;

-- EXCLUDE constraints (for ranges)
CREATE TABLE reservations (
    room_id INTEGER,
    during TSTZRANGE,
    EXCLUDE USING GIST (room_id WITH =, during WITH &&)
    -- prevents overlapping reservations for the same room
);
```

**PostgreSQL production checklist:**

```sql
-- Essential settings for production (postgresql.conf or ALTER SYSTEM)
shared_buffers = '256MB'        -- 25% of RAM for dedicated DB server
effective_cache_size = '1GB'    -- how much OS page cache to assume
wal_buffers = '16MB'            -- WAL buffer size
work_mem = '16MB'               -- per sort/hash operation (×concurrent operations)
maintenance_work_mem = '256MB'  -- for VACUUM, index builds
max_connections = 200           -- with PgBouncer, can be lower
random_page_cost = 1.1          -- for SSD storage (default 4.0 assumes HDD)
log_min_duration_statement = 1000  -- log queries taking > 1 second
track_io_timing = on            -- enable I/O timing in EXPLAIN ANALYZE
```

## MySQL / MariaDB (5.27)

MySQL (1995, now Oracle; MariaDB is the open-source fork) is the world's most widely deployed open-source database. Powers WordPress, Facebook (early), Wikipedia.

**Key differences from PostgreSQL:**

```sql
-- MySQL: different syntax and behaviors
-- No RETURNING clause (use LAST_INSERT_ID())
INSERT INTO users (email) VALUES ('alice@example.com');
SELECT LAST_INSERT_ID();  -- get the auto-increment ID

-- MySQL: case-insensitive string comparison by default (collation)
-- 'Alice' = 'alice' in most MySQL setups (can cause unique constraint surprises)
SELECT * FROM users WHERE email = 'Alice@Example.COM';  -- matches 'alice@example.com'

-- MySQL: InnoDB storage engine (use this, not MyISAM)
-- MyISAM: no foreign keys, no transactions — avoid

-- MySQL charset gotcha (critical!):
-- Use utf8mb4, NOT utf8:
-- MySQL's 'utf8' only supports 3-byte chars (no emoji!)
-- utf8mb4 supports full 4-byte UTF-8 (correct choice)
CREATE TABLE posts (content TEXT) CHARACTER SET utf8mb4 COLLATE utf8mb4_unicode_ci;

-- MySQL vs PostgreSQL key differences:
-- MySQL: no FULL OUTER JOIN (workaround: UNION of LEFT + RIGHT)
-- MySQL: LIMIT syntax: LIMIT offset, count  (reversed from PostgreSQL)
-- MySQL: AUTO_INCREMENT not SERIAL/BIGSERIAL
-- PostgreSQL: significantly more features (JSONB, arrays, window functions, CTEs with modifications)
```

**MariaDB:** Drop-in MySQL replacement with additional features and more active community development. Generally preferred over MySQL for new deployments.

## SQLite (5.28)

SQLite is an embedded, serverless database stored in a single file. The world's most widely deployed database (>1 trillion instances: every phone, browser, desktop app).

```python
import sqlite3

# SQLite: no server, no connection setup
conn = sqlite3.connect("myapp.db")  # creates file if not exists
conn.execute("CREATE TABLE IF NOT EXISTS users (id INTEGER PRIMARY KEY, email TEXT UNIQUE)")
conn.execute("INSERT INTO users (email) VALUES (?)", ("alice@example.com",))
conn.commit()

rows = conn.execute("SELECT * FROM users").fetchall()
```

**When SQLite is the right database:**
- Development and testing (no Docker needed)
- Desktop applications, mobile apps
- Edge/serverless: Cloudflare D1, Turso, LiteFS
- Single-user applications
- Embedded systems

**SQLite limitations:**
- Write concurrency: only one writer at a time (WAL mode: one writer + multiple readers)
- Not suitable for high write throughput (hundreds of writes/second)
- Network: no TCP server — app must be on same machine

**SQLite's modern revival:** Cloudflare D1 (SQLite at edge nodes), Turso (distributed SQLite), LiteFS (SQLite replication) are making SQLite viable for web applications at scale.

## CockroachDB (5.29)

```
COCKROACHDB: Distributed SQL, PostgreSQL-compatible
  - Horizontally scalable across nodes (PostgreSQL is single-node)
  - Automatic sharding and replication
  - Geo-distribution: route writes to nearest node
  - Strong consistency (serializable by default)
  - PostgreSQL wire protocol compatible (most PG drivers work)

USE WHEN:
  - Need true horizontal scalability for writes
  - Multi-region deployment with data locality requirements
  - PostgreSQL compatibility needed but scale exceeds single-node

LIMITATIONS:
  - No stored procedures (limited)
  - Some PG features not supported
  - Higher operational complexity
  - Slower single-node performance than PostgreSQL
```

## PlanetScale (5.30)

```
PLANETSCALE: MySQL-compatible, branching database model
  - Branches: create DB branches like Git branches
  - Schema changes via deploy requests (safe, reviewed)
  - Horizontal sharding built in (based on Vitess, YouTube's MySQL layer)
  - Serverless scaling

KEY DESIGN DECISION: No foreign keys
  - PlanetScale disables FK constraints (enables horizontal sharding)
  - You enforce referential integrity in application code
  - Trade-off: simpler sharding, less database-level safety

USE WHEN:
  - MySQL compatibility needed at scale
  - Need schema change safety net
  - Write scaling beyond single MySQL node
```

## Neon (5.31)

```
NEON: Serverless PostgreSQL
  - Separate storage from compute (storage on S3-compatible)
  - Scale compute to zero (no cost when idle)
  - Branching: create isolated DB branches for each PR
  - Full PostgreSQL compatibility

USE WHEN:
  - Startup or side project (cost: pay per query when idle)
  - Need DB branches for development workflow
  - Serverless/edge workloads

LIMITATION:
  - Cold start: waking from sleep takes ~1-3 seconds
  - Not for latency-sensitive production traffic (unless always warm)
```

---

## TiDB (5.32)

```
TIDB: Distributed HTAP (Hybrid Transactional/Analytical Processing)
  - Handles OLTP (transactions) AND OLAP (analytics) in one system
  - MySQL compatible
  - Row store for OLTP, column store for analytics
  - Horizontal scaling like CockroachDB

USE WHEN:
  - Need both real-time transactions AND real-time analytics on same data
  - MySQL compatibility needed
  - Currently rare for most applications
```

## SQL Database Selection Guide

```
New project, standard use case:           PostgreSQL
WordPress/PHP ecosystem:                  MySQL 8+ or MariaDB
Development/embedded/edge:                SQLite
High write scale + SQL:                   CockroachDB or TiDB
MySQL at write scale:                     PlanetScale
Serverless startup, free tier:            Neon (PostgreSQL) or PlanetScale
AWS-native, need Aurora compatibility:   Aurora PostgreSQL or Aurora MySQL
```
