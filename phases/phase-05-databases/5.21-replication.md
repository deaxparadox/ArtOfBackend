## Replication (5.21)

PostgreSQL replication sends WAL (Write-Ahead Log) changes from the primary to replicas.

```
STREAMING REPLICATION (physical):
  Primary writes WAL → streams to replica → replica replays WAL
  Result: byte-for-byte copy of the primary
  Use: high availability, read scaling

LOGICAL REPLICATION:
  Replicates at SQL statement level (INSERT, UPDATE, DELETE)
  Can replicate specific tables, to different schema structures
  Use: zero-downtime upgrades, migrations, CDC (Change Data Capture)

SYNCHRONOUS vs ASYNCHRONOUS:
  Synchronous: primary waits for replica to confirm WAL receipt before COMMIT
    → No data loss on primary failure, higher write latency
    → Configure: synchronous_standby_names = 'replica1'

  Asynchronous (default): primary commits without waiting for replica
    → Lower write latency, risk of losing last few transactions on failover
    → Replication lag: how far behind is the replica?
    → Monitor: SELECT now() - pg_last_xact_replay_timestamp() AS lag;
```

## Read Replicas (5.23)

```python
# Pattern: write to primary, read from replica
from sqlalchemy import create_engine

write_engine = create_engine("postgresql://primary-host/db")
read_engine  = create_engine("postgresql://replica-host/db")

async def get_user_list():
    async with read_engine.connect() as conn:  # reads from replica
        return await conn.execute(select(User))

async def create_user(data):
    async with write_engine.connect() as conn:  # writes to primary
        await conn.execute(insert(User).values(**data))
        await conn.commit()

# CRITICAL CAVEAT: replication lag!
# After creating a user, immediately reading from replica may return no result
# (replica may be 10-500ms behind)
# Solution: read your own writes → use primary for reads immediately after writes
```

```
REPLICATION LAG STRATEGIES:
  1. Route all writes + reads immediately after write to primary
  2. Add delay after write before redirecting to replica
  3. Track per-user "last write LSN", route to primary if replica hasn't caught up
  4. Use synchronous replication for critical reads (no lag, higher latency)
```

## Partitioning (5.24)

Table partitioning splits a large table into smaller physical pieces:

```sql
-- Range partitioning: by time (most common)
CREATE TABLE orders (
    id BIGSERIAL,
    user_id BIGINT,
    created_at TIMESTAMPTZ NOT NULL,
    total_cents INTEGER
) PARTITION BY RANGE (created_at);

CREATE TABLE orders_2024_q1 PARTITION OF orders
    FOR VALUES FROM ('2024-01-01') TO ('2024-04-01');
CREATE TABLE orders_2024_q2 PARTITION OF orders
    FOR VALUES FROM ('2024-04-01') TO ('2024-07-01');

-- Queries automatically route to relevant partition:
SELECT * FROM orders WHERE created_at BETWEEN '2024-01-01' AND '2024-04-01';
-- Only scans orders_2024_q1 (not all other partitions)

-- DROP old partitions instantly (vs DELETE which is slow):
DROP TABLE orders_2023_q1;  -- drops partition, instant, no vacuum needed

-- WHEN TO PARTITION:
-- Table > 100GB AND queries are partition-key filtered
-- Need fast deletion of old data (time-series, audit logs, events)
-- NOT for: tables < 10GB, tables without clear partition key
```

## Full-Text Search in PostgreSQL (5.25)

```sql
-- tsvector: stores document as a list of lexemes (normalized words)
-- tsquery: search query with operators

-- Create searchable column
ALTER TABLE posts ADD COLUMN search_vector tsvector;

-- Populate it
UPDATE posts SET search_vector = to_tsvector('english', title || ' ' || body);

-- Index it (GIN for fast full-text search)
CREATE INDEX idx_posts_search ON posts USING GIN (search_vector);

-- Search query
SELECT title, ts_rank(search_vector, query) AS rank
FROM posts, to_tsquery('english', 'backend & engineering') query
WHERE search_vector @@ query
ORDER BY rank DESC
LIMIT 20;

-- Auto-update via trigger
CREATE OR REPLACE FUNCTION posts_search_vector_trigger()
RETURNS trigger LANGUAGE plpgsql AS $$
BEGIN
    NEW.search_vector = to_tsvector('english',
        COALESCE(NEW.title, '') || ' ' || COALESCE(NEW.body, ''));
    RETURN NEW;
END;
$$;

CREATE TRIGGER posts_search_update
    BEFORE INSERT OR UPDATE ON posts
    FOR EACH ROW EXECUTE FUNCTION posts_search_vector_trigger();

-- For better full-text: use Elasticsearch (topic 13.2) or Typesense
-- PostgreSQL FTS is good for simple cases; external search engines for complex
```
