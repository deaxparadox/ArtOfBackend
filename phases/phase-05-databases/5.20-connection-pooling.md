## 1. Why Database Connections Are Expensive

Every PostgreSQL connection is a separate OS process (~5-10MB RAM each). Establishing a connection requires:
- TCP handshake
- TLS handshake (if SSL enabled)
- PostgreSQL authentication
- Session initialization

Total cost: 2-10ms per new connection.

**The math:** 100 requests/second × 5ms per new connection = 500ms of pure connection overhead per second. With a connection pool: 0ms overhead (reuse existing connections).

---

## 2. How Connection Pooling Works

```
WITHOUT POOL:
  Request 1 → open DB connection → query → close connection
  Request 2 → open DB connection → query → close connection
  (N connections opened and closed per second)

WITH POOL:
  Startup: pool opens 10 persistent connections to DB
  Request 1 → acquire connection from pool → query → return to pool
  Request 2 → acquire connection from pool → query → return to pool
  (0 connection overhead)
```

## 3. Application-Level Pools (asyncpg, SQLAlchemy, HikariCP)

Built into the database driver/ORM:

```python
# asyncpg connection pool
import asyncpg

pool = await asyncpg.create_pool(
    dsn="postgresql://user:pass@localhost/db",
    min_size=5,      # minimum idle connections
    max_size=20,     # maximum connections (concurrent queries)
    command_timeout=30,  # query timeout
)

async def get_user(user_id: int):
    async with pool.acquire() as conn:  # take connection from pool
        return await conn.fetchrow("SELECT * FROM users WHERE id=$1", user_id)
    # connection automatically returned to pool here

# SQLAlchemy async pool
from sqlalchemy.ext.asyncio import create_async_engine

engine = create_async_engine(
    "postgresql+asyncpg://user:pass@localhost/db",
    pool_size=10,           # max persistent connections
    max_overflow=5,         # max extra connections when pool is full
    pool_timeout=30,        # wait max 30s for a connection
    pool_pre_ping=True,     # verify connection health before use
    pool_recycle=3600,      # recreate connections older than 1h (avoid stale connections)
)
```

**Pool sizing formula (Little's Law):**
```
pool_size = expected_concurrent_queries × avg_query_duration

Example: 100 req/s, 20ms avg query, 50% have DB queries
  = 50 concurrent queries × 0.02s = 1 connection... too low
  
Real example: 1000 req/s, 50ms avg query, 80% DB queries
  = 800 concurrent × 0.05s = 40 connections needed
  
Rule of thumb: pool_size = 2 × num_CPU_cores on DB server
  For 4-core DB: pool_size=8-16
```

## 4. PgBouncer: Server-Side Connection Pooling

PgBouncer sits between your application and PostgreSQL, pooling connections at the database level. Useful when you have many application instances (20 app servers × 20 pool connections = 400 connections to PG, potentially overwhelming it).

```
WITHOUT PgBouncer:
  App Server 1 (pool: 20) ─────┐
  App Server 2 (pool: 20) ──────┤─→ PostgreSQL (400 connections!)
  ...                           │
  App Server 20 (pool: 20) ────┘

WITH PgBouncer:
  App Server 1 (pool: 20) ─────┐
  App Server 2 (pool: 20) ──────┤─→ PgBouncer ─→ PostgreSQL (20-50 connections)
  ...                           │
  App Server 20 (pool: 20) ────┘
```

```ini
# pgbouncer.ini
[databases]
mydb = host=localhost port=5432 dbname=mydb

[pgbouncer]
listen_port = 6432
listen_addr = *
auth_type = md5
auth_file = /etc/pgbouncer/userlist.txt

pool_mode = transaction   # session | transaction | statement
                          # transaction: connection returned after each transaction
                          # (most efficient, but some session features don't work)
max_client_conn = 1000    # max connections from apps
default_pool_size = 25    # connections per database/user pair to PostgreSQL
min_pool_size = 5
server_idle_timeout = 600 # close idle server connections after 10 min
```

**Pool modes:**
- `session`: one server connection per client session (least efficient, most compatible)
- `transaction`: connection per transaction (most efficient — 100+ clients per server connection)
- `statement`: connection per statement (rarely used, breaks multi-statement transactions)

## 5. HikariCP (Java's Standard Pool)

```java
// Spring Boot application.yml
spring:
  datasource:
    hikari:
      maximum-pool-size: 20
      minimum-idle: 5
      connection-timeout: 30000       # 30s wait for connection
      idle-timeout: 600000            # 10m before idle connections closed
      max-lifetime: 1800000           # 30m max connection lifetime
      validation-timeout: 5000        # 5s to validate connection health
      leak-detection-threshold: 60000 # 60s: log if connection held this long
```

## 6. Senior Engineer Perspective

**The most common production database issue:** Too many connections. PostgreSQL's default `max_connections = 100`. With 20 app servers each with 20-connection pools = 400 connections. PostgreSQL rejects excess connections with: `FATAL: remaining connection slots are reserved for non-replication superuser connections`.

**The solution hierarchy:**
1. Reduce pool size per instance
2. Add PgBouncer in transaction mode (100 client connections → 5-10 server connections)
3. For AWS RDS: use RDS Proxy (managed PgBouncer)
4. For very high connection counts: increase `max_connections` (costs RAM)

**The connection starvation trap:** Pool size = 20, query runs for 5s, 20 concurrent requests = all 20 connections busy. Request 21 waits. At 30s timeout: error. This is invisible until load testing. Always load test and monitor pool exhaustion metrics.
