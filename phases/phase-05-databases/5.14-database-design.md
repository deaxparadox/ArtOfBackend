## 1. Design Principles for Production Schemas

Good database design decisions survive years of schema evolution. Bad ones cause painful migrations.

---

## 2. Primary Key Selection

```sql
-- OPTION 1: Auto-incrementing integer (BIGSERIAL / IDENTITY)
CREATE TABLE users (
    id BIGSERIAL PRIMARY KEY,  -- PostgreSQL: generates 64-bit integer
    -- or: id BIGINT GENERATED ALWAYS AS IDENTITY
    email VARCHAR(255) UNIQUE NOT NULL
);
-- Pros: small (8 bytes), fast comparisons, sortable, sequential I/O for inserts
-- Cons: predictable (exposes row count), single-server only (sharding issues)

-- OPTION 2: UUID v4 (random)
CREATE TABLE events (
    id UUID DEFAULT gen_random_uuid() PRIMARY KEY,
    ...
);
-- Pros: globally unique (safe for distributed systems), non-predictable
-- Cons: 16 bytes, random inserts cause B-tree fragmentation, not sortable by time

-- OPTION 3: ULID (Universally Unique Lexicographically Sortable Identifier)
-- Timestamp prefix + random suffix
-- Pros: globally unique + sortable by creation time + URL-safe string
-- Cons: requires extension or application-level generation

-- OPTION 4: UUID v7 (PostgreSQL 17+ or pg_uuidv7 extension)
-- Time-ordered UUID — sortable like ULID but standard UUID format
CREATE TABLE orders (
    id UUID DEFAULT gen_uuidv7() PRIMARY KEY,
    ...
);
-- Best of both worlds: UUID compatibility + time ordering + B-tree friendly

-- RECOMMENDATION:
-- Internal tables with high write throughput: BIGSERIAL (fast sequential inserts)
-- Public-facing IDs: UUID v7 (non-predictable, globally unique, sortable)
-- Distributed systems: UUID v4 or UUID v7
```

## 3. Foreign Key Design

```sql
-- Always define FK constraint (prevents orphan rows)
CREATE TABLE orders (
    id          BIGSERIAL PRIMARY KEY,
    user_id     BIGINT NOT NULL,
    CONSTRAINT fk_orders_user FOREIGN KEY (user_id) REFERENCES users(id)
        ON DELETE RESTRICT   -- default: can't delete user who has orders
        ON UPDATE CASCADE    -- if user.id changes, order.user_id updates too
);

-- ON DELETE CASCADE vs RESTRICT vs SET NULL:
-- CASCADE: delete all orders when user is deleted
--   → Use for: truly dependent data (order_items when order is deleted)
-- RESTRICT: prevent deletion if children exist
--   → Use for: critical data that shouldn't orphan (can't delete user with orders)
-- SET NULL: nullify the FK when parent is deleted
--   → Use for: optional references (deleted_by_id, assigned_to_id)
-- SET DEFAULT: set to default value when parent is deleted

-- CRITICAL: always index foreign keys
CREATE INDEX idx_orders_user_id ON orders (user_id);
-- Missing FK index = full table scan on every join
```

## 4. Column Design Best Practices

```sql
CREATE TABLE orders (
    -- IDs
    id          BIGSERIAL PRIMARY KEY,
    user_id     BIGINT NOT NULL REFERENCES users(id),

    -- Monetary values: always integers (cents), never floats
    total_cents INTEGER NOT NULL CHECK (total_cents >= 0),
    tax_cents   INTEGER NOT NULL DEFAULT 0,

    -- Status: enum or check constraint (never raw string without validation)
    status      VARCHAR(20) NOT NULL DEFAULT 'pending'
                CHECK (status IN ('pending', 'paid', 'shipped', 'delivered', 'cancelled')),
    -- Alternative: PostgreSQL ENUM type
    -- status order_status NOT NULL DEFAULT 'pending',

    -- Timestamps: always TIMESTAMPTZ (with timezone)
    created_at  TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    updated_at  TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    shipped_at  TIMESTAMPTZ,  -- nullable: NULL until shipped

    -- Soft delete: prefer over hard delete
    deleted_at  TIMESTAMPTZ,  -- NULL = not deleted; timestamp = when deleted

    -- Metadata/configuration: JSONB for flexible schemas
    metadata    JSONB NOT NULL DEFAULT '{}',

    -- Never NULL where you mean "empty":
    -- name VARCHAR(100) NOT NULL DEFAULT '',  ← empty string, not NULL
    -- tags TEXT[] NOT NULL DEFAULT '{}',      ← empty array, not NULL

    CONSTRAINT unique_order_reference UNIQUE (user_id, reference_number)
);

-- Auto-update updated_at
CREATE OR REPLACE FUNCTION set_updated_at()
RETURNS TRIGGER AS $$
BEGIN
    NEW.updated_at = NOW();
    RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER set_orders_updated_at
    BEFORE UPDATE ON orders
    FOR EACH ROW EXECUTE FUNCTION set_updated_at();
```

## 5. Soft Deletes

```sql
-- Hard delete: row is gone forever (bad for audit trails, FK integrity)
DELETE FROM users WHERE id = 123;

-- Soft delete: set deleted_at timestamp (preferred)
UPDATE users SET deleted_at = NOW() WHERE id = 123;

-- All queries must include: WHERE deleted_at IS NULL
-- (or use a view that adds this automatically)

CREATE VIEW active_users AS
    SELECT * FROM users WHERE deleted_at IS NULL;

-- Partial index for performance:
CREATE INDEX idx_users_active ON users (email)
WHERE deleted_at IS NULL;  -- only index active users

-- Unique constraint considering soft deletes:
-- Want: unique email among ACTIVE users, but allow re-registration after deletion
CREATE UNIQUE INDEX idx_users_active_email ON users (email)
WHERE deleted_at IS NULL;
```

## 6. Senior Engineer Perspective

**Schema design decisions have long tails.** Choosing `INT` instead of `BIGINT` for a primary key: if your table grows beyond 2 billion rows (2^31), you overflow. Instagram hit this with their photo IDs. Migration is painful. Always use BIGINT.

**The schema review checklist:**

```
□ Every table has a primary key
□ Every FK column has an index
□ Monetary values are integers (cents)
□ Timestamps are TIMESTAMPTZ not TIMESTAMP
□ String columns have appropriate length limits
□ Nullable columns are intentionally nullable (not "we'll figure it out later")
□ Status/type columns have CHECK constraints or ENUM
□ ON DELETE behavior is explicitly chosen (not left as default)
□ created_at, updated_at on every table
□ Soft delete strategy defined if needed
```
