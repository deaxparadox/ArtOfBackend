## 1. Subqueries

A subquery is a SELECT statement nested inside another SQL statement. The result can be used as a value, a set of values, or a table.

```sql
-- SCALAR SUBQUERY: returns a single value
SELECT
    name,
    (SELECT COUNT(*) FROM orders WHERE user_id = users.id) AS order_count
FROM users;
-- WARNING: correlated scalar subquery runs once per row = N+1 problem
-- Much better with JOIN or window function

-- TABLE SUBQUERY (derived table): use a query result as a table
SELECT u.email, t.total_spent
FROM users u
JOIN (
    SELECT user_id, SUM(total_cents) AS total_spent
    FROM orders WHERE status = 'paid'
    GROUP BY user_id
) t ON u.id = t.user_id;

-- IN SUBQUERY: check membership
SELECT * FROM users
WHERE id IN (
    SELECT DISTINCT user_id FROM orders WHERE status = 'pending'
);

-- NOT IN SUBQUERY: check non-membership (CAREFUL with NULLs)
SELECT * FROM users
WHERE id NOT IN (
    SELECT user_id FROM orders WHERE status = 'paid'
);
-- WARNING: if any user_id in subquery is NULL, NOT IN returns no rows!
-- Use NOT EXISTS instead:
SELECT * FROM users u
WHERE NOT EXISTS (
    SELECT 1 FROM orders o WHERE o.user_id = u.id AND o.status = 'paid'
);

-- EXISTS: check if any matching row exists (stops at first match, efficient)
SELECT * FROM users u
WHERE EXISTS (
    SELECT 1 FROM orders o WHERE o.user_id = u.id AND o.amount > 10000
);
```

## 2. CTEs (Common Table Expressions)

CTEs are named temporary result sets defined with `WITH`. They make complex queries more readable.

```sql
-- Simple CTE: improve readability
WITH recent_orders AS (
    SELECT * FROM orders
    WHERE created_at >= NOW() - INTERVAL '30 days'
)
SELECT
    u.email,
    COUNT(o.id) AS recent_order_count
FROM users u
LEFT JOIN recent_orders o ON u.id = o.user_id
GROUP BY u.id, u.email;

-- Multiple CTEs (comma-separated)
WITH
paid_orders AS (
    SELECT user_id, SUM(total_cents) AS paid_total
    FROM orders WHERE status = 'paid'
    GROUP BY user_id
),
high_value AS (
    SELECT user_id FROM paid_orders WHERE paid_total > 100000
),
user_info AS (
    SELECT u.id, u.email, u.created_at
    FROM users u WHERE u.is_active = true
)
SELECT ui.email, po.paid_total
FROM user_info ui
JOIN paid_orders po ON ui.id = po.user_id
WHERE ui.id IN (SELECT user_id FROM high_value)
ORDER BY po.paid_total DESC;

-- CTEs are materialized in PostgreSQL by default (run once, result cached)
-- This can be faster OR slower than equivalent subquery — test both

-- Disable materialization (PostgreSQL 12+):
WITH orders AS NOT MATERIALIZED (
    SELECT * FROM orders WHERE status = 'paid'
)
SELECT ...
```

## 3. Recursive CTEs

For hierarchical data (categories, org charts, bill of materials):

```sql
-- Traverse category hierarchy
WITH RECURSIVE cat_tree AS (
    -- Anchor: start with root categories (no parent)
    SELECT id, name, parent_id, ARRAY[name] AS path, 0 AS depth
    FROM categories
    WHERE parent_id IS NULL

    UNION ALL

    -- Recursive: find children of current nodes
    SELECT
        c.id, c.name, c.parent_id,
        ct.path || c.name,  -- append to path array
        ct.depth + 1
    FROM categories c
    JOIN cat_tree ct ON c.parent_id = ct.id
    WHERE ct.depth < 10  -- max depth guard (prevent infinite loops)
)
SELECT
    id,
    name,
    array_to_string(path, ' > ') AS full_path,
    depth
FROM cat_tree
ORDER BY path;
-- Result: Electronics > Computers > Laptops
--         Electronics > Phones
--         Clothing > Men > Shirts

-- Find all ancestors of a category (walk UP the tree)
WITH RECURSIVE ancestors AS (
    SELECT id, name, parent_id
    FROM categories
    WHERE id = 42  -- start at specific category

    UNION ALL

    SELECT c.id, c.name, c.parent_id
    FROM categories c
    JOIN ancestors a ON c.id = a.parent_id
)
SELECT * FROM ancestors;
```

## 4. Lateral Joins (Advanced)

`LATERAL` allows a subquery in FROM to reference columns from preceding tables — like a correlated subquery but used as a table:

```sql
-- Get the 3 most recent orders per user (WITHOUT lateral: complex workaround needed)
SELECT u.name, o.id, o.total_cents, o.created_at
FROM users u
JOIN LATERAL (
    SELECT * FROM orders
    WHERE user_id = u.id      -- ← reference to outer query (lateral!)
    ORDER BY created_at DESC
    LIMIT 3
) o ON true;  -- always true — we want all users even if fewer than 3 orders
```

## 5. Subquery vs. CTE vs. Window Function: When to Use Each

```
USE SUBQUERY WHEN:
  - Simple one-time filter or scalar value
  - Quick to read in context
  - No need to reuse the result

USE CTE WHEN:
  - Complex multi-step logic
  - Need to reuse the same intermediate result
  - Recursive data traversal
  - Readability is more important than marginal performance

USE WINDOW FUNCTION WHEN:
  - Need row-level calculations + keeping all original rows
  - Running totals, moving averages, rankings
  - Much faster than equivalent correlated subquery

USE JOIN INSTEAD OF SUBQUERY WHEN:
  - Correlated subqueries (N+1 problem)
  - IN (SELECT ...) when result set is large
  - EXISTS can often be replaced by LEFT JOIN ... IS NULL
```

## 6. Senior Engineer Perspective

**The N+1 problem in pure SQL:**

A correlated scalar subquery in the SELECT clause executes once per row — the SQL equivalent of the N+1 problem:

```sql
-- N+1 equivalent in SQL (avoid):
SELECT
    u.name,
    (SELECT COUNT(*) FROM orders WHERE user_id = u.id) AS order_count  -- 1 query per user
FROM users;

-- Correct approach (1 query):
SELECT u.name, COALESCE(o.order_count, 0) AS order_count
FROM users u
LEFT JOIN (
    SELECT user_id, COUNT(*) AS order_count
    FROM orders GROUP BY user_id
) o ON u.id = o.user_id;
```

Understanding that correlated subqueries in SELECT are O(n) is fundamental to writing efficient SQL. The query planner may optimize some cases, but it's better to write the explicit JOIN version.
