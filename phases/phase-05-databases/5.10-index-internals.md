## 1. The B-tree Structure

A B-tree index is a balanced tree where:
- All leaf nodes are at the same depth
- Each node contains multiple keys (not just one like a binary tree)
- Leaf nodes are linked in sorted order (makes range scans fast)

```
B-TREE INDEX on users.email (simplified):

                   [N, T]
                  /   |   \
              [A-M] [N-S] [T-Z]
             /    \         \
       [A-G][H-M]          [T, V, Z]
        |
    Leaf nodes (sorted, linked):
    [alice] → [bob] → [carol] → [dave] → ...
    (each leaf points to table row's physical location)
```

**Tree height:** log₂(n) / log₂(branching factor). With branching factor ~100 and 1M rows: height = ~3 levels. Finding any row = 3 disk reads.

**Why balanced trees:** Insertions and deletions maintain balance via node splitting and merging, keeping height constant (O(log n)) regardless of insert order.

## 2. Database Pages: The Unit of I/O

PostgreSQL (and all relational databases) read/write data in fixed-size pages (default: 8KB):

```
DATABASE PAGE (8KB):
┌─────────────────────────────────────────────┐
│ Page Header (24 bytes)                       │
│   LSN, checksum, flags                       │
├─────────────────────────────────────────────┤
│ Item IDs (array of offsets to rows)          │
├─────────────────────────────────────────────┤
│ Free space                                   │
├─────────────────────────────────────────────┤
│ Tuples (rows): each row is a heap tuple      │
│   Header: xmin, xmax, ctid                  │
│   Data: column values                        │
└─────────────────────────────────────────────┘

IMPLICATION:
  1 row = 1 page read (even if the row is 50 bytes and the page is 8KB)
  Reading 100 rows spread across 100 pages = 100 page reads
  Reading 100 rows on the same 2 pages = 2 page reads

  → Dense data (filled pages) = better I/O efficiency
  → Sequential scans read pages in order (fast, prefetchable)
  → Random index seeks read pages randomly (slow if not cached)
```

## 3. Sequential Scan vs. Index Scan

```
TABLE: orders (1 million rows)
QUERY: SELECT * FROM orders WHERE status = 'pending'

SEQUENTIAL SCAN (no index on status):
  Read all 1M rows (100MB of pages) sequentially
  Filter rows where status = 'pending'
  Return matching rows
  Time: ~500ms (sequential I/O, OS can prefetch pages)
  Use when: returning > 5-10% of rows (index overhead > full scan)

INDEX SCAN (B-tree on status):
  Follow B-tree to 'pending' entries
  For each match: read the corresponding table page (random I/O)
  Time: O(k × random_I/O) where k = matching rows
  Use when: returning < 5-10% of rows

INDEX ONLY SCAN (covering index):
  Follow B-tree to 'pending' entries
  Return data directly from index (no table access)
  Time: O(log n + k)  — fastest possible
  Use when: index covers all needed columns (INCLUDE clause)

BITMAP INDEX SCAN (middle ground):
  Build bitmap of matching page locations
  Sort locations for sequential I/O
  Read pages in order (efficient even for larger result sets)
  Time: between Index Scan and Seq Scan
  Use when: 1-10% of rows match
```

## 4. The Cost Model

PostgreSQL's query planner uses a cost model to decide between scan strategies:

```sql
-- Check table statistics
SELECT relname, reltuples AS estimated_rows, relpages AS page_count
FROM pg_class WHERE relname = 'orders';

-- The planner estimates:
-- seq_scan cost = relpages × seq_page_cost  (default: 1.0)
-- index_scan cost = pages_from_index × random_page_cost  (default: 4.0)
--                 + matching_rows × cpu_tuple_cost

-- random_page_cost = 4.0 (assumes HDD-like random I/O)
-- For SSDs: lower this to improve index preference
ALTER SYSTEM SET random_page_cost = 1.1;  -- SSDs have near-sequential I/O
SELECT pg_reload_conf();
```

**This matters:** If you're on SSD storage (NVMe) and PostgreSQL still chooses sequential scans over index scans, lower `random_page_cost`. The default (4.0) was designed for spinning HDDs.

## 5. MVCC and Index Overhead

PostgreSQL's MVCC keeps multiple versions of rows. Dead tuples (old versions) stay in the heap and indexes until VACUUM cleans them:

```
Effect on indexes:
  - Index entries point to specific heap locations (ctid: page + offset)
  - When a row is updated: new version inserted, old version marked dead (xmax set)
  - Old index entry still points to dead version
  - SELECT: index scan → find entry → read heap tuple → check visibility
  - VACUUM: removes dead heap tuples AND dead index entries

HOT UPDATE optimization:
  If updated columns are NOT indexed AND there's free space on the same page:
  PostgreSQL uses HOT (Heap Only Tuple) update — no index update needed
  → Significant performance gain for high-update tables with many indexes
  → Keep table fill factor < 100 to leave room for HOT updates:
  ALTER TABLE orders SET (fillfactor = 90);  -- leave 10% free per page
```

## 6. Senior Engineer Perspective

**When the query planner makes wrong choices:**

```sql
-- Force a specific plan (diagnostic only, not production)
SET enable_seqscan = off;  -- force index scans
SET enable_indexscan = off;  -- force sequential scans

-- Better: update statistics (planner uses stale stats)
ANALYZE orders;  -- update statistics for this table
-- Autovacuum does this periodically, but after large data changes: run manually

-- Force a specific index
SELECT * FROM orders WHERE status = 'pending';  -- wrong plan

-- Diagnostic: what plan WOULD it choose for a specific index?
SET enable_seqscan = off;
EXPLAIN SELECT * FROM orders WHERE status = 'pending';
```

The query planner makes decisions based on statistics. Fresh statistics → better plans. After bulk data loads, run `ANALYZE table_name` before relying on query plans.
