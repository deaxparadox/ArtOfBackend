## 1. What Is Denormalization?

Denormalization is the deliberate introduction of redundancy into a schema to improve read performance at the cost of write complexity and storage.

**The trade-off:** Normalized = one source of truth, complex reads. Denormalized = redundant data, simpler/faster reads, complex writes.

---

## 2. When Denormalization Makes Sense

```
DENORMALIZE WHEN:
  ✓ A query joins 5+ tables and runs 1000×/second
  ✓ Read latency is critical (user-facing, real-time)
  ✓ Write frequency is low compared to read frequency
  ✓ Data changes infrequently (category names, user names)
  ✓ Reporting/analytics where joins are expensive

KEEP NORMALIZED WHEN:
  ✓ Data changes frequently
  ✓ Consistency is critical (financial, medical)
  ✓ Write performance matters as much as read
  ✓ The join cost is acceptable
```

## 3. Common Denormalization Patterns

### Cached/Derived Columns

```sql
-- NORMALIZED: must JOIN to count orders
SELECT u.id, u.name, COUNT(o.id) AS order_count
FROM users u LEFT JOIN orders o ON u.id = o.user_id
GROUP BY u.id, u.name;

-- DENORMALIZED: store count on users table
ALTER TABLE users ADD COLUMN order_count INTEGER DEFAULT 0;

-- Keep count updated via trigger
CREATE OR REPLACE FUNCTION update_user_order_count()
RETURNS TRIGGER AS $$
BEGIN
    IF TG_OP = 'INSERT' THEN
        UPDATE users SET order_count = order_count + 1 WHERE id = NEW.user_id;
    ELSIF TG_OP = 'DELETE' THEN
        UPDATE users SET order_count = order_count - 1 WHERE id = OLD.user_id;
    END IF;
    RETURN NULL;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER update_order_count
AFTER INSERT OR DELETE ON orders
FOR EACH ROW EXECUTE FUNCTION update_user_order_count();

-- Now the read is instant:
SELECT id, name, order_count FROM users WHERE id = 123;
```

### Flattening Joins with Redundant Columns

```sql
-- Store username alongside orders to avoid joining users table for order lists
ALTER TABLE orders ADD COLUMN user_email VARCHAR(255);
-- Populated at order creation, accepted as stale if user changes email
```

### Materialized Views

```sql
-- Store expensive aggregation results
CREATE MATERIALIZED VIEW user_order_stats AS
SELECT
    u.id AS user_id,
    u.email,
    COUNT(o.id) AS total_orders,
    COALESCE(SUM(o.total_cents), 0) AS lifetime_value_cents,
    MAX(o.created_at) AS last_order_at
FROM users u
LEFT JOIN orders o ON u.id = o.user_id AND o.status = 'paid'
GROUP BY u.id, u.email;

CREATE UNIQUE INDEX ON user_order_stats (user_id);

-- Query the materialized view (fast, no joins)
SELECT * FROM user_order_stats WHERE user_id = 123;

-- Refresh when data changes (can be scheduled or triggered)
REFRESH MATERIALIZED VIEW CONCURRENTLY user_order_stats;
-- CONCURRENTLY: doesn't lock the view during refresh
-- Requires a unique index on the view
```

## 4. The Consistency Problem

```
DENORMALIZATION CONSISTENCY RISK:

users.order_count = 5
orders table has 6 rows for this user
→ Inconsistent state!

CAUSES:
  - Bug in application code that increments the counter
  - Direct database modification that bypasses the trigger
  - Failed trigger

MITIGATION:
  - Periodic reconciliation jobs (compare computed vs stored)
  - Triggers (database-level enforcement)
  - Application-level consistency checks in tests

RULE: Never trust a denormalized value for critical decisions.
  Use it for display/caching, verify with the normalized truth for operations.
```

---

## 5. Senior Engineer Perspective

**The pattern that scales:** Use normalized storage as the source of truth, and maintain denormalized "read models" (materialized views, cached columns, Redis summaries) as performance optimization layers. The read models can be stale. The normalized data is always authoritative. This is the "read model" pattern from CQRS.
