## 1. What Is ACID?

ACID is the set of four properties that guarantee database transactions are processed reliably. Every relational database (PostgreSQL, MySQL, SQLite) provides ACID guarantees. Understanding each property tells you exactly what failures your database will and won't protect you from.

---

## 2. Atomicity: All or Nothing

**A transaction is treated as a single unit.** Either ALL operations in the transaction succeed, or NONE of them do. There is no partial completion.

```
BANK TRANSFER: Move $100 from Account A to Account B

Without atomicity:
  1. Deduct $100 from Account A ← succeeds
  2. [SERVER CRASHES]
  3. Add $100 to Account B     ← never runs
  Result: $100 vanished from the system.

With atomicity:
  BEGIN;
    1. Deduct $100 from Account A
    2. Add $100 to Account B
  COMMIT;
  — If any step fails, ROLLBACK undoes ALL steps —
  Result: either both happen or neither happens. $100 is never lost.
```

**How it's implemented:** The database writes changes to a write-ahead log (WAL) before applying them to actual data pages. If a crash occurs before COMMIT, the uncommitted changes are rolled back on recovery using the WAL.

---

## 3. Consistency: Valid State to Valid State

**A transaction brings the database from one valid state to another valid state.** All constraints, triggers, and rules must be satisfied before COMMIT.

```sql
-- Consistency in action: constraints enforced atomically

-- Table constraint: account_balance must be >= 0
ALTER TABLE accounts ADD CONSTRAINT non_negative CHECK (balance >= 0);

BEGIN;
    UPDATE accounts SET balance = balance - 10000 WHERE id = 1;
    -- If balance would go negative: transaction aborted here
    -- "ERROR: new row for relation 'accounts' violates check constraint"
COMMIT;
-- Result: if balance would go negative, the whole transaction fails.
-- Consistency ensured at the database level, not the application level.
```

Consistency covers:
- Column constraints (NOT NULL, CHECK, UNIQUE)
- Foreign key constraints
- Custom triggers
- Application-defined invariants enforced by the transaction logic

---

## 4. Isolation: Concurrent Transactions Don't Interfere

**Multiple concurrent transactions execute as if they were running serially.** The effects of a transaction are not visible to other transactions until it commits.

Without isolation:
```
T1 reads user balance: $500
T2 reads user balance: $500
T1 adds $100: balance = $600, commits
T2 adds $100: balance = $600, commits  ← should be $700!
```

With isolation (serializable): one of these transactions would see the other's changes or fail with a serialization error.

**Isolation levels are a spectrum** (see topic 5.8):
- Read Uncommitted → Serializable
- More isolation = more correctness, less concurrency
- Less isolation = faster, more anomalies possible

---

## 5. Durability: Committed Means Committed

**Once a transaction commits, it remains committed even if the system crashes.**

How: PostgreSQL writes all changes to the WAL (Write-Ahead Log) before acknowledging COMMIT to the client. On crash recovery, PostgreSQL replays the WAL to restore committed transactions.

```
Client: COMMIT;
DB writes WAL entry to disk: "transaction 12345 committed"
DB acknowledges: "COMMIT"
Server crashes immediately after.

On restart:
DB reads WAL: "transaction 12345 committed"
DB replays any data pages not yet written
Data is fully intact.
```

**Durability vs. performance trade-off:**

```sql
-- Full durability (default, correct for production)
-- synchronous_commit = on
-- Each COMMIT waits for WAL write to disk before acknowledging

-- Faster but less durable (for bulk imports, test data)
SET LOCAL synchronous_commit = off;
-- Acknowledge COMMIT before WAL hits disk
-- Risk: if server crashes in next ~200ms, committed transactions can be lost
-- Never use in production for user-visible data
```

## 6. BASE: The NoSQL Alternative

NoSQL databases often trade ACID for availability and performance. The alternative is **BASE**:

```
BASE:
  Basically Available  — system responds even during failures
  Soft state          — state may change without input (eventual sync)
  Eventually consistent — all replicas will eventually converge

vs ACID:
  Atomicity          — all or nothing transactions
  Consistency        — always in valid state
  Isolation          — concurrent transactions don't interfere
  Durability         — committed data survives crashes

WHEN BASE IS ACCEPTABLE:
  - Social media likes (slightly stale count is fine)
  - Analytics counters (approximate values OK)
  - Cache stores (losing cached data is recoverable)
  - Shopping cart items (eventual sync acceptable)

WHEN ACID IS REQUIRED:
  - Financial transactions (money must never be double-counted)
  - Inventory management (can't oversell)
  - User authentication (security-critical state)
  - Any operation where partial completion is catastrophic
```

## 7. Real-World Context

### The Stripe use case

Stripe's payment system requires all ACID properties:
- **Atomicity:** charge the card and create the payment record — both or neither
- **Consistency:** payment amounts must be positive, account IDs must exist
- **Isolation:** two concurrent refund requests can't both succeed if only $50 remains
- **Durability:** a confirmed charge must survive any server failure

### PostgreSQL's ACID implementation

- **Atomicity:** MVCC + WAL. If transaction aborts, its changes were in a private version, never visible to others.
- **Consistency:** constraint checking at COMMIT time.
- **Isolation:** MVCC (each transaction sees a snapshot). Row-level locks for writes.
- **Durability:** WAL. Data changes are logged before applied. `fsync` calls ensure WAL is on disk.

## 8. Senior Engineer Perspective

**What a junior gets wrong:**

Assuming ACID handles all correctness. ACID protects against machine failures and transaction atomicity. It does NOT protect against:
- Logic bugs in your transaction (debiting the wrong account atomically)
- Application-level race conditions if you don't use transactions properly
- Selecting the wrong isolation level for your use case

**The question to ask for every multi-step operation:**

"If this fails halfway through, what is the database state?" If the answer is "corrupted/inconsistent," that operation belongs in a transaction.

A surprisingly common bug: reading data, processing it in application code, then writing it back in separate round trips without a transaction. The database sees three separate operations. Another transaction can modify the data between the read and the write. Atomicity only helps if you use transactions.
