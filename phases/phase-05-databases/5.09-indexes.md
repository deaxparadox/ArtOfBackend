## 1. What Is an Index?

An index is a data structure that allows the database to find rows matching a condition without scanning every row in the table (a sequential scan).

**The real-world analogy:**

A book's index at the back. Instead of reading every page to find mentions of "PostgreSQL," you look up "PostgreSQL" in the index which tells you exactly which pages to go to. The index occupies extra space but makes lookups vastly faster.

**Without an index on users.email:**
```
SELECT * FROM users WHERE email = 'alice@example.com';
→ Database reads every row: row 1, row 2... row 1,000,000
→ O(n) time: 1M rows = ~100ms
```

**With an index on users.email:**
```
SELECT * FROM users WHERE email = 'alice@example.com';
→ Database follows the B-tree to the exact location: ~3 I/O operations
→ O(log n) time: 1M rows = ~0.1ms
→ 1000× faster
```

## 2. B-tree Index (The Default)

The B-tree (balanced tree) is the default and most common index type. It supports:
- Equality: `WHERE email = 'alice@example.com'`
- Range: `WHERE age > 18`, `WHERE created_at BETWEEN x AND y`
- Prefix: `WHERE name LIKE 'Alice%'` (but NOT `LIKE '%alice%'`)
- Sorting: `ORDER BY created_at` (can use the index ordering)
- NULL checks: `WHERE deleted_at IS NULL`

```sql
-- Create a B-tree index (default)
CREATE INDEX idx_users_email ON users (email);

-- Multi-column (composite) index
CREATE INDEX idx_orders_user_status ON orders (user_id, status);

-- DESC index (for ORDER BY col DESC queries)
CREATE INDEX idx_orders_created_desc ON orders (created_at DESC);

-- Unique index (also enforces uniqueness constraint)
CREATE UNIQUE INDEX idx_users_email_unique ON users (email);
-- Equivalent to: ALTER TABLE users ADD CONSTRAINT ... UNIQUE (email)
```

**Composite index column order matters:**

```sql
CREATE INDEX idx_orders_user_status ON orders (user_id, status);

-- This index is used by:
WHERE user_id = 1                       ✓ (leftmost prefix)
WHERE user_id = 1 AND status = 'paid'   ✓ (full index)
WHERE user_id = 1 ORDER BY status       ✓ (leftmost prefix + sort)

-- This index is NOT efficiently used by:
WHERE status = 'paid'                   ✗ (skips first column)
ORDER BY status                         ✗ (can't use without user_id filter)
```

**Rule: queries filter by columns in left-to-right order of the composite index.**

## 3. When Indexes Are NOT Used

```sql
-- Function on indexed column: index NOT used
WHERE UPPER(email) = 'ALICE@EXAMPLE.COM'  -- index on email, not UPPER(email)
-- Fix: CREATE INDEX ON users (UPPER(email))  OR use ILIKE

-- Implicit type cast: index may not be used
WHERE id = '123'  -- id is integer, comparing to string → cast may prevent index use
-- Fix: WHERE id = 123

-- OR conditions spanning different columns:
WHERE user_id = 1 OR status = 'paid'  -- may cause full scan
-- Fix: UNION of two indexed queries

-- NOT IN with large sets: full scan often faster
WHERE status NOT IN ('pending', 'processing', 'cancelled', ...)

-- Low cardinality columns (wrong direction):
WHERE active = true  -- if 99% of rows are active, index is useless
                     -- database may choose full scan (cheaper)

-- Leading wildcard:
WHERE name LIKE '%alice'  -- B-tree can't help (no left anchor)
-- Fix: use pg_trgm extension + GIN index for arbitrary substring search
```

## 4. Partial Indexes: Index Only What You Query

A partial index only indexes rows matching a WHERE condition. Smaller, faster, often more useful than full indexes.

```sql
-- Index only unshipped orders (much smaller than indexing all orders)
CREATE INDEX idx_orders_unshipped
ON orders (created_at)
WHERE status IN ('pending', 'processing');
-- Queries like: WHERE status = 'pending' AND created_at > x use this index
-- 99% of old 'shipped' orders are not in the index

-- Index only active users
CREATE INDEX idx_active_users_email
ON users (email)
WHERE is_active = true;
-- Much smaller than indexing all users if most are inactive

-- Index only non-null values
CREATE INDEX idx_orders_tracking
ON orders (tracking_number)
WHERE tracking_number IS NOT NULL;
-- Don't waste space indexing NULLs
```

## 5. Expression Indexes: Index Computed Values

```sql
-- Case-insensitive email search
CREATE INDEX idx_users_email_lower
ON users (lower(email));
-- Now: WHERE lower(email) = lower($1) uses the index

-- Extract year from timestamp
CREATE INDEX idx_orders_year
ON orders (extract(year FROM created_at));
-- Now: WHERE extract(year FROM created_at) = 2024 uses the index

-- JSON field index (PostgreSQL)
CREATE INDEX idx_metadata_source
ON events ((metadata->>'source'));
-- Now: WHERE metadata->>'source' = 'web' uses the index
```

## 6. Covering Indexes: Index-Only Scans

A covering index includes all columns needed by a query, so the database doesn't need to touch the main table at all.

```sql
-- Query: SELECT email, name FROM users WHERE status = 'active' ORDER BY created_at
CREATE INDEX idx_active_users_covering
ON users (status, created_at) INCLUDE (email, name);
-- INCLUDE: store these columns IN the index (not used for searching/sorting)
-- Result: the query is answered entirely from the index — no table access

-- Index-only scan is shown in EXPLAIN output:
-- Index Only Scan using idx_active_users_covering on users
-- (not "Index Scan" which still hits the table for each row)
```

## 7. Other Index Types (PostgreSQL)

### Hash Index

```sql
CREATE INDEX idx_users_email_hash ON users USING HASH (email);
-- Faster than B-tree for equality only (=)
-- Does NOT support range, sorting, or LIKE
-- Rarely used — B-tree handles equality well enough
```

### GIN (Generalized Inverted Index)

Used for: full-text search, arrays, JSONB, `tsvector`.

```sql
-- Full-text search index
CREATE INDEX idx_posts_content_fts
ON posts USING GIN (to_tsvector('english', content));
-- Now: WHERE to_tsvector('english', content) @@ to_tsquery('english', 'backend')

-- JSONB index (any key in the JSON document)
CREATE INDEX idx_events_metadata
ON events USING GIN (metadata);
-- Now: WHERE metadata @> '{"source": "web"}'  (containment check)

-- Array index
CREATE INDEX idx_posts_tags
ON posts USING GIN (tags);
-- Now: WHERE tags @> ARRAY['python', 'backend']  (array contains)
```

### GiST (Generalized Search Tree)

Used for: geometric types, full-text search, range types, PostGIS.

```sql
-- Range overlap queries
CREATE INDEX idx_reservations_period
ON reservations USING GIST (daterange(start_date, end_date));
-- Now: WHERE daterange(start_date, end_date) && daterange($1, $2)

-- PostGIS spatial index
CREATE INDEX idx_locations_geom
ON locations USING GIST (coordinates);
-- Enables efficient: ST_Within, ST_DWithin, ST_Intersects queries
```

## 8. Index Maintenance

Indexes are NOT free — they have costs:

```
WRITE OVERHEAD:
  Every INSERT: update all indexes for the new row
  Every UPDATE: update indexes for changed columns
  Every DELETE: mark rows in indexes as dead
  
  A table with 10 indexes: each write touches 10 index structures.
  
STORAGE:
  Indexes take disk space (typically 10-30% of table size per index)

BLOAT:
  Indexes accumulate dead entries over time (from UPDATEs and DELETEs)
  PostgreSQL autovacuum cleans these, but high-churn tables need monitoring

MONITORING:
  -- Check index usage (has this index ever been used?)
  SELECT schemaname, tablename, indexname, idx_scan, idx_tup_read
  FROM pg_stat_user_indexes
  WHERE idx_scan = 0;  -- never-used indexes

  -- Check index bloat
  SELECT indexrelname, pg_size_pretty(pg_relation_size(indexrelid)) AS size
  FROM pg_stat_user_indexes
  ORDER BY pg_relation_size(indexrelid) DESC;
```

## 9. Index Creation Strategy

```sql
-- CREATE INDEX CONCURRENTLY: build index without locking the table
-- (slower, but doesn't block reads/writes — use this in production!)
CREATE INDEX CONCURRENTLY idx_orders_created ON orders (created_at);

-- DROP INDEX CONCURRENTLY: remove index without locking
DROP INDEX CONCURRENTLY idx_orders_created;

-- Regular CREATE INDEX: LOCKS the table (use only on new/empty tables, or during maintenance window)
CREATE INDEX idx_orders_created ON orders (created_at);
```

## 10. Senior Engineer Perspective

**The indexing strategy:**

1. Don't add indexes preemptively. Add them when you have slow queries.
2. Use EXPLAIN ANALYZE to identify what's slow (topic 5.11).
3. Add the minimum index that fixes the problem.
4. Monitor index usage — remove unused indexes (they hurt write performance).

**The most important indexes to always have:**

- Primary key (automatic)
- Every foreign key column (NOT automatic in PostgreSQL — add manually)
- Every column used in a WHERE clause in frequently-executed queries
- Every column used in an ORDER BY on large-table queries

**Missing foreign key indexes is a major performance issue:**

```sql
-- PostgreSQL does NOT automatically index foreign keys
CREATE TABLE orders (
    user_id BIGINT REFERENCES users(id)  -- FK created, index NOT created
);

-- This query will do a SEQUENTIAL SCAN:
SELECT * FROM orders WHERE user_id = 123;

-- Add the index:
CREATE INDEX ON orders (user_id);  -- now it's O(log n)
```

This is one of the most common performance issues in PostgreSQL production databases.
