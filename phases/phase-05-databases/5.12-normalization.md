## 1. What Is Normalization?

Normalization is the process of organizing a database schema to reduce data redundancy and improve data integrity. Each "normal form" eliminates a specific type of redundancy/anomaly.

**Why it matters:** Redundant data leads to update anomalies — when the same fact is stored in multiple places, updates can leave the database in an inconsistent state. Normalization ensures each fact is stored exactly once.

---

## 2. The Normal Forms

### First Normal Form (1NF): Atomic Values, No Repeating Groups

**Rule:** Every column must contain atomic (indivisible) values. No arrays, no comma-separated lists, no repeating column groups.

```
VIOLATES 1NF:
  order_id | products                        | customer
  ---------+---------------------------------+---------
  1        | "Widget, Gadget, Doohickey"     | Alice     ← comma-separated in one column
  2        | "Widget"                         | Bob

  order_id | product_1 | product_2 | product_3    ← repeating column groups
  ---------+-----------+-----------+-----------
  1        | Widget    | Gadget    | Doohickey
  2        | Widget    | NULL      | NULL

SATISFIES 1NF:
  order_items:
  order_id | product_name | customer
  ---------+--------------+---------
  1        | Widget       | Alice
  1        | Gadget       | Alice
  1        | Doohickey    | Alice
  2        | Widget       | Bob
```

### Second Normal Form (2NF): No Partial Dependencies

**Rule:** Every non-key column must depend on the WHOLE primary key (applies only to tables with composite keys).

```
VIOLATES 2NF (composite PK: order_id + product_id):
  order_id | product_id | quantity | customer_name | product_price
  ---------+------------+----------+---------------+--------------
  1        | 101        | 2        | Alice         | $10.00

  customer_name depends only on order_id (partial dependency)
  product_price depends only on product_id (partial dependency)

FIX: Decompose into tables where non-key columns depend on the FULL key:
  orders: order_id, customer_name
  products: product_id, product_price
  order_items: order_id, product_id, quantity
```

### Third Normal Form (3NF): No Transitive Dependencies

**Rule:** No non-key column should depend on another non-key column (no transitive dependencies).

```
VIOLATES 3NF:
  employee_id | department_id | department_name | department_budget
              department_name depends on department_id, not directly on employee_id

FIX:
  employees: employee_id, department_id
  departments: department_id, department_name, department_budget
```

### Boyce-Codd Normal Form (BCNF): Every Determinant Is a Candidate Key

BCNF is a stronger form of 3NF. Most databases that are in 3NF are also in BCNF. The rare case where 3NF ≠ BCNF involves tables where a non-key column determines part of a composite key.

## 3. Practical Normalization Guidelines

```
SIGNS YOUR SCHEMA IS NOT NORMALIZED:
  - Updating a customer's address requires changing multiple rows
  - Inserting a new product requires adding it to every order it appears in
  - Deleting an order deletes the only record of a product existing
  - The same phone number appears in 47 rows and they're all slightly different

PRACTICAL RULE OF THUMB:
  1NF: no JSON arrays, no comma-separated values in a column
  2NF: no column describes something other than the primary key
  3NF: each fact stored once (no redundancy)

  "3NF" is the practical target for OLTP databases.
  BCNF: handle only if 3NF still has anomalies.

WHEN NOT TO NORMALIZE:
  Reporting/analytics: denormalized star/snowflake schemas are faster to query
  High-read, low-write: denormalized data avoids joins (see topic 5.13)
  Caching/materialized views: pre-computed denormalized views for speed
```

## 4. Real Example: Normalizing an Order System

```sql
-- UNNORMALIZED (stores everything in one table):
CREATE TABLE orders_bad (
    order_id     INTEGER,
    customer_id  INTEGER,
    customer_email TEXT,    -- duplicated across all of customer's orders
    customer_name  TEXT,    -- duplicated
    product_ids    TEXT,    -- "101,102,103" — violates 1NF
    product_names  TEXT,    -- "Widget,Gadget,Doohickey" — redundant
    total_cents    INTEGER
);

-- NORMALIZED (3NF):
CREATE TABLE customers (
    id    BIGSERIAL PRIMARY KEY,
    email VARCHAR(255) UNIQUE NOT NULL,
    name  VARCHAR(100) NOT NULL
);

CREATE TABLE products (
    id          BIGSERIAL PRIMARY KEY,
    name        VARCHAR(200) NOT NULL,
    price_cents INTEGER NOT NULL
);

CREATE TABLE orders (
    id          BIGSERIAL PRIMARY KEY,
    customer_id BIGINT REFERENCES customers(id),
    created_at  TIMESTAMPTZ DEFAULT NOW()
);

CREATE TABLE order_items (
    id          BIGSERIAL PRIMARY KEY,
    order_id    BIGINT REFERENCES orders(id),
    product_id  BIGINT REFERENCES products(id),
    quantity    INTEGER NOT NULL,
    unit_price_cents INTEGER NOT NULL  -- snapshot price at time of order
);
-- Note: unit_price_cents is stored here, not fetched from products.price_cents
-- because product price can change after the order is placed
```

## 5. Senior Engineer Perspective

Normalization is a starting point, not a goal. Production schemas are almost always partially denormalized for performance. The common pattern:

1. Start normalized (3NF)
2. Measure which queries are slow
3. Selectively denormalize the slow ones (add redundant data where the join is expensive)
4. Keep the normalized version as the source of truth, update denormalized data via triggers or application logic

The worst production schemas are those that started denormalized — they're inconsistent by design and progressively harder to fix as the application grows.
