## Authentication vs. Authorization (8.1)

```
AUTHENTICATION: "Who are you?"
  → Verify the identity of the caller
  → Login, token validation, API key verification
  
AUTHORIZATION: "What are you allowed to do?"
  → Verify the caller has permission for this action
  → Role checks, ownership verification, policy evaluation

COMMON CONFUSION:
  401 Unauthorized → actually means UNAUTHENTICATED (who are you?)
  403 Forbidden    → actually means UNAUTHORIZED (you don't have permission)
  
  The HTTP standard used the wrong word for 401.
  → 401: no valid credentials / credentials invalid
  → 403: authenticated but not allowed to perform this action
```

## Password Hashing (8.2)

```python
# WRONG: store plaintext passwords (catastrophic)
user.password = "mypassword123"  # if DB is breached: all passwords exposed

# WRONG: use MD5 or SHA256 (fast hashes)
import hashlib
user.password = hashlib.md5("mypassword123".encode()).hexdigest()
# MD5 is reversible via rainbow tables; can compute 10 billion/second on GPU

# RIGHT: use bcrypt, Argon2, or scrypt (deliberately slow hashes)
# These are designed to be slow — makes brute-force attacks infeasible

# BCRYPT (most widely deployed)
import bcrypt

# Hash a password (cost factor 12 = ~250ms)
password = "mypassword123"
hashed = bcrypt.hashpw(password.encode(), bcrypt.gensalt(rounds=12))
# $2b$12$... — includes salt and cost factor in the hash

# Verify (always use constant-time comparison)
def verify_password(plain: str, hashed: bytes) -> bool:
    return bcrypt.checkpw(plain.encode(), hashed)

# ARGON2 (recommended for new projects — winner of Password Hashing Competition)
from argon2 import PasswordHasher
from argon2.exceptions import VerifyMismatchError

ph = PasswordHasher(
    time_cost=2,      # iterations
    memory_cost=65536, # 64MB memory
    parallelism=2,    # parallel threads
)

hashed = ph.hash("mypassword123")
# $argon2id$v=19$m=65536,t=2,p=2$...

def verify_argon2(plain: str, hashed: str) -> bool:
    try:
        return ph.verify(hashed, plain)
    except VerifyMismatchError:
        return False

# SCRYPT (Node.js standard)
import hashlib, os

def hash_scrypt(password: str) -> str:
    salt = os.urandom(32)
    key = hashlib.scrypt(password.encode(), salt=salt, n=16384, r=8, p=1)
    return salt.hex() + ":" + key.hex()
```

**Cost factor selection:**
- bcrypt rounds=12: ~250ms on modern hardware (good for login — slow enough to prevent brute force)
- rounds=10: ~60ms (minimum acceptable for production)
- Argon2: configure so hashing takes 100-500ms on your hardware
- Never below 50ms — too fast = too easy to brute force

## Sessions (8.3)

Server-side sessions store user state in the server (or shared Redis):

```python
from fastapi import FastAPI, Response, Cookie, Depends, HTTPException
import redis.asyncio as aioredis
import secrets, json

app = FastAPI()
redis = aioredis.from_url("redis://localhost:6379")
SESSION_TTL = 86400  # 24 hours

@app.post("/auth/login")
async def login(credentials: LoginRequest, response: Response):
    user = await authenticate(credentials.email, credentials.password)
    if not user:
        raise HTTPException(401, "Invalid credentials")

    # Generate opaque session ID (never put sensitive data in the cookie itself)
    session_id = secrets.token_urlsafe(32)  # 256 bits of entropy

    # Store session data server-side
    session_data = {
        "user_id": user.id,
        "email": user.email,
        "roles": user.roles,
        "created_at": datetime.utcnow().isoformat(),
    }
    await redis.setex(f"session:{session_id}", SESSION_TTL, json.dumps(session_data))

    # Set secure cookie
    response.set_cookie(
        key="session_id",
        value=session_id,
        max_age=SESSION_TTL,
        httponly=True,   # no JS access
        secure=True,     # HTTPS only
        samesite="lax",  # CSRF protection
    )
    return {"message": "Logged in"}

async def get_current_user(
    session_id: str | None = Cookie(default=None)
) -> dict:
    if not session_id:
        raise HTTPException(401, "Not authenticated")

    data = await redis.get(f"session:{session_id}")
    if not data:
        raise HTTPException(401, "Session expired")

    return json.loads(data)

@app.post("/auth/logout")
async def logout(response: Response, session: dict = Depends(get_current_user),
                 session_id: str = Cookie()):
    await redis.delete(f"session:{session_id}")  # invalidate server-side
    response.delete_cookie("session_id")
    return {"message": "Logged out"}
```

## JWT (8.4): JSON Web Tokens

JWT is a compact, URL-safe token containing claims (data) signed by the server. Stateless — no server-side session storage needed.

```
JWT STRUCTURE:
  eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9
  .
  eyJzdWIiOiIxMjMiLCJlbWFpbCI6ImFsaWNlQGV4YW1wbGUuY29tIiwiZXhwIjoxNzE4NDUwMjAwfQ
  .
  SflKxwRJSMeKKF2QT4fwpMeJf36POk6yJV_adQssw5c

  Part 1: Header (Base64url)
    {"alg":"HS256","typ":"JWT"}
    
  Part 2: Payload (Base64url) — the claims
    {"sub":"123","email":"alice@example.com","exp":1718450200,"roles":["admin"]}
    
  Part 3: Signature
    HMAC-SHA256(Base64url(header) + "." + Base64url(payload), secret_key)
```

```python
from datetime import datetime, timezone, timedelta
import jwt

SECRET_KEY = "your-256-bit-secret"  # use secrets.token_urlsafe(32) in production

def create_access_token(user_id: int, email: str, roles: list[str]) -> str:
    payload = {
        "sub": str(user_id),   # subject (user ID)
        "email": email,
        "roles": roles,
        "iat": datetime.now(timezone.utc),               # issued at
        "exp": datetime.now(timezone.utc) + timedelta(minutes=15),  # expires in 15 min
        "jti": secrets.token_hex(16),                    # JWT ID (for revocation)
    }
    return jwt.encode(payload, SECRET_KEY, algorithm="HS256")

def verify_access_token(token: str) -> dict:
    try:
        payload = jwt.decode(
            token,
            SECRET_KEY,
            algorithms=["HS256"],
            options={"require": ["sub", "exp", "iat"]}
        )
        return payload
    except jwt.ExpiredSignatureError:
        raise HTTPException(401, "Token expired")
    except jwt.InvalidTokenError:
        raise HTTPException(401, "Invalid token")
```

## JWT Pitfalls (8.5): Critical Security Issues

```python
# PITFALL 1: alg:none exploit (verify algorithm)
# Attacker creates token with "alg": "none" — no signature needed!
# jwt.decode("eyJ...", algorithms=["none"])  # ALWAYS verify with expected algorithm

# RIGHT: always specify expected algorithm
jwt.decode(token, key, algorithms=["HS256"])  # never ["none"] or []

# PITFALL 2: HS256 vs RS256
# HS256: symmetric — same secret for signing AND verifying
#   → All services that verify tokens MUST share the secret
#   → If secret leaks: attacker can forge any token
# RS256: asymmetric — private key signs, public key verifies
#   → Share public key widely; keep private key on auth server only
#   → Better for microservices (verify without the signing key)

# For production with multiple services: use RS256
from cryptography.hazmat.primitives import serialization

with open("private.pem", "rb") as f:
    private_key = serialization.load_pem_private_key(f.read(), password=None)

token = jwt.encode(payload, private_key, algorithm="RS256")
# Verify (public key only — can be shared)
jwt.decode(token, public_key, algorithms=["RS256"])

# PITFALL 3: Long expiry = can't revoke
# 7-day JWT that's stolen can be used for 7 days with no way to invalidate.
# Solution: short-lived access tokens (15 min) + long-lived refresh tokens (in Redis)
# Revoke refresh token → user must re-authenticate → 15 min max exposure

# PITFALL 4: Sensitive data in payload
# JWT payload is Base64-encoded, NOT encrypted!
# Anyone can decode and read the payload.
# NEVER put: passwords, credit card numbers, sensitive PII in JWT payload.
# Fine to include: user ID, email, roles, permissions.
```

## OAuth 2.0 (8.6): Delegated Authorization

OAuth 2.0 allows users to grant third-party apps access to their data without sharing passwords.

```
OAUTH 2.0 FLOWS:

AUTHORIZATION CODE (for web apps and mobile):
  1. User clicks "Login with Google"
  2. App redirects to Google: GET accounts.google.com/o/oauth2/auth
     ?client_id=...&redirect_uri=...&scope=email&response_type=code
  3. User logs in to Google, approves permissions
  4. Google redirects back: GET /callback?code=xyz
  5. App exchanges code for token: POST accounts.google.com/o/oauth2/token
     {code: xyz, client_secret: ...}
  6. Google returns: {access_token: ..., refresh_token: ..., expires_in: 3600}
  7. App uses access_token to call Google APIs

WHY USE A CODE?
  The code is short-lived and single-use.
  The token is exchanged server-to-server (never exposed in browser URL).
  If someone captures the redirect URL, the code is useless (already exchanged).

CLIENT CREDENTIALS (for server-to-server):
  No user involved.
  Your service → POST /token {client_id, client_secret, grant_type=client_credentials}
  Receive: access_token
  Use for: microservice-to-microservice auth, batch jobs

DEVICE FLOW (for devices without browser):
  Smart TVs, CLIs.
  Device shows a code → user types it at accounts.google.com/device
  Device polls for token until user approves.
```

```python
# Building OAuth2 in FastAPI (provider side)
from fastapi.security import OAuth2PasswordBearer, OAuth2PasswordRequestForm

oauth2_scheme = OAuth2PasswordBearer(tokenUrl="/auth/token")

@app.post("/auth/token")
async def login_for_access_token(form_data: OAuth2PasswordRequestForm = Depends()):
    user = await authenticate(form_data.username, form_data.password)
    if not user:
        raise HTTPException(401, detail="Incorrect credentials",
                          headers={"WWW-Authenticate": "Bearer"})

    access_token = create_access_token(user.id, user.email, user.roles)
    refresh_token = await create_refresh_token(user.id)  # stored in Redis

    return {
        "access_token": access_token,
        "token_type": "bearer",
        "expires_in": 900,  # 15 minutes
        "refresh_token": refresh_token,
    }

async def get_current_user(token: str = Depends(oauth2_scheme)) -> dict:
    return verify_access_token(token)
```

## OAuth 2.0 + PKCE (8.7)

PKCE (Proof Key for Code Exchange) prevents authorization code interception in mobile apps and SPAs:

```
PKCE FLOW:
  1. Client generates: code_verifier (random 43-128 char string)
  2. Client computes: code_challenge = BASE64URL(SHA256(code_verifier))
  3. Authorization request includes code_challenge
  4. Token exchange includes code_verifier
  5. Server: verifies SHA256(code_verifier) == code_challenge
  
  → Even if attacker intercepts the code, they can't exchange it
    (they don't have code_verifier, only client does)
    
  Required for: all mobile apps, all SPAs (browser-based apps)
  Optional for: server-side web apps (but recommended)
```

## OpenID Connect (8.8)

OIDC is an identity layer on top of OAuth 2.0. OAuth 2.0 gives you authorization (access tokens). OIDC adds authentication (ID tokens + user info endpoint).

```
OAUTH 2.0: "You are authorized to access this resource"
OIDC:      "You are authenticated as Alice (alice@example.com)"

OIDC ADDITIONS:
  ID Token: JWT with user identity claims
    {sub: "123", email: "alice@example.com", name: "Alice", ...}
  
  /userinfo endpoint: fetch user profile with access_token
  
  Discovery document: /.well-known/openid-configuration
    Lists all endpoints, supported algorithms, etc.
    Used by libraries to auto-configure

STANDARD CLAIMS:
  sub:    Subject (user ID — stable, unique per provider)
  email:  Email address
  name:   Full name
  picture: Avatar URL
  aud:    Audience (your app's client_id)
  iss:    Issuer (identity provider)
  iat/exp: Issued at / expires
```

## Token Refresh Pattern (8.14)

```python
# Access token: short-lived (15 min), stateless JWT
# Refresh token: long-lived (7-30 days), stored in Redis (can be revoked)

async def create_refresh_token(user_id: int) -> str:
    token = secrets.token_urlsafe(32)
    # Store in Redis with 30-day TTL
    await redis.setex(
        f"refresh:{token}",
        30 * 86400,
        json.dumps({"user_id": user_id})
    )
    return token

@app.post("/auth/refresh")
async def refresh_access_token(refresh_token: str = Cookie()):
    # Look up refresh token
    data = await redis.get(f"refresh:{refresh_token}")
    if not data:
        raise HTTPException(401, "Invalid or expired refresh token")

    data = json.loads(data)
    user = await db.get_user(data["user_id"])

    # Rotate refresh token (each use gets a new one)
    await redis.delete(f"refresh:{refresh_token}")
    new_refresh_token = await create_refresh_token(user.id)

    new_access_token = create_access_token(user.id, user.email, user.roles)

    return {
        "access_token": new_access_token,
        "expires_in": 900,
        "new_refresh_token": new_refresh_token,
    }
```

## Managed Auth Platforms (8.15)

Building auth from scratch is complex and error-prone. Managed solutions handle the hard parts:

| Platform | Best For | Notes |
|---|---|---|
| **Auth0** | Enterprise, complex rules | Most features, expensive at scale |
| **Clerk** | Developer UX, startups | Beautiful UI components, easy setup |
| **Supabase Auth** | Supabase users | Free tier, PostgreSQL-based |
| **Firebase Auth** | GCP/Firebase ecosystem | Google's, mobile-friendly |
| **Cognito (AWS)** | AWS-native apps | Complex UX, but deeply integrated |
| **Keycloak** | Self-hosted, enterprise | Open source, full-featured SSO |

**The build-vs-buy decision:**

If auth is not your core product: use a managed service. Implementing OAuth 2.0, SAML, MFA, session management, token rotation, and social login correctly takes weeks and requires ongoing security maintenance. Auth0 or Clerk costs money but saves engineering time and security risk.
