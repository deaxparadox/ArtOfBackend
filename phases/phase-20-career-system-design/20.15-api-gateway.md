## Why This System Is Hard

An API gateway is a stateless proxy that must enforce auth, rate limiting, routing, and observability on every single request, adding as little latency as possible. The hard problems are:

1. **Latency overhead must be minimal** — Every millisecond you add is paid by every user on every call. A 10ms gateway overhead on 1M requests/second = 10 seconds of cumulative extra wait per second. The gateway itself must be fast.
2. **Auth at the gateway without a network hop** — Naively, validating a token means calling the auth service on every request. That adds 5-20ms and creates a single point of failure. You need local validation.
3. **Distributed rate limiting** — With 10 gateway nodes, a per-node counter lets clients send 10x their limit. You need a shared atomic counter store. But that shared store becomes the bottleneck.
4. **Zero-downtime config updates** — Routes, rate limits, and auth rules change constantly. How do you push new config to 10 stateless gateway nodes without a restart that drops connections?
5. **Health checking and automatic failover** — Upstream services go down. The gateway must detect this and stop routing traffic to dead backends, then resume when they recover.

## Requirements Clarification

**Functional requirements:**
- Route incoming HTTP requests to the correct upstream microservice
- Authenticate and authorize every request (JWT validation)
- Enforce per-client rate limits
- Support request/response header transformation
- Provide observability: access logs, metrics, distributed tracing headers
- Health check upstream services and route around unhealthy backends

**Non-functional requirements:**
- Gateway overhead: < 5ms at p99 (the gateway cannot be the bottleneck)
- Availability: 99.99% (4 nines — gateway down = entire platform down)
- Throughput: 1M requests/second across the cluster
- Stateless nodes: any gateway node can serve any request
- Config updates propagate to all nodes within 30 seconds, no restarts

**Out of scope (clarify explicitly):**
- WebSocket proxying
- gRPC transcoding
- GraphQL federation

## Deep Dive: The Hard Problems

### 1. Authentication Without a Network Round-Trip

**Naive approach:**
```
Every request -> call Auth Service -> validate token -> allow or deny
Added latency: ~15ms average (local network + auth service processing)
```

15ms doesn't sound like much. At 35,000 RPS, that is 525,000 ms of added wait per second. The auth service itself becomes the bottleneck.

**Solution: Asymmetric JWT validation (local crypto only)**

Auth service signs tokens with its private key. Gateway validates using the public key — a purely local CPU operation.

```
Auth Service                     API Gateway
     |                                |
     | Sign JWT with RSA private key  |
     | Include:                       |
     |   exp (expiry timestamp)       |
     |   iss (issuer: "auth.company") |
     |   aud (audience: "api")        |
     |   sub (user ID)                |
     |   roles: ["user", "premium"]   |
     |   kid: "key-2024-01"           |
     |                                |
     |                                | 1. Extract JWT from header
     |                                | 2. Decode header -> get kid
     |                                | 3. Look up public key from
     |                                |    JWKS cache by kid
     |                                | 4. crypto.verify(sig, pubkey)
     |                                |    -> pure CPU, no network
     |                                | 5. Check exp > now()
     |                                | 6. Check iss, aud
     |                                | 7. Extract user_id, roles
     |                                |    -> inject as request headers
```

**JWKS (JSON Web Key Set) caching:**

```python
class JWKSCache:
    def __init__(self, jwks_url: str, ttl_seconds: int = 300):
        self.jwks_url = jwks_url
        self.cache: dict[str, PublicKey] = {}
        self.last_refresh = 0
        self.ttl = ttl_seconds

    def get_public_key(self, kid: str) -> PublicKey:
        if time.time() - self.last_refresh > self.ttl:
            self.refresh()

        if kid not in self.cache:
            # Key not found even after refresh -> invalid kid
            self.refresh()  # Force refresh in case key is new
            if kid not in self.cache:
                raise InvalidToken(f"Unknown key ID: {kid}")

        return self.cache[kid]

    def refresh(self):
        response = requests.get(self.jwks_url, timeout=2.0)
        keys = response.json()["keys"]
        self.cache = {k["kid"]: parse_public_key(k) for k in keys}
        self.last_refresh = time.time()
```

**Token revocation (logout before expiry):**

The weakness of local JWT validation: you cannot revoke a token until it expires. A user logs out, but their token is valid for another hour.

Solution: A small Redis blocklist for explicitly revoked tokens. Check `jti` (JWT ID) after signature validation:

```python
def validate_jwt(token: str, jwks_cache: JWKSCache, redis: Redis) -> dict:
    claims = jwt.decode_header(token)
    public_key = jwks_cache.get_public_key(claims["kid"])

    # Local crypto - fast
    payload = jwt.decode(token, public_key, algorithms=["RS256"],
                         audience="api", issuer="auth.company")

    # One Redis lookup for revocation check
    if redis.sismember("revoked_tokens", payload["jti"]):
        raise TokenRevoked()

    return payload
```

The blocklist stays small: only tokens explicitly revoked before expiry. Most tokens expire naturally and never need blocklist entries. Typical size: tens of thousands of entries.

### 2. Distributed Rate Limiting

**Why per-node counters fail:**

```
10 gateway nodes, client limit: 1,000 req/min

Per-node counter behavior:
  Node 1: sees 100 requests -> 100 < 1000 -> allow
  Node 2: sees 100 requests -> 100 < 1000 -> allow
  ...
  Node 10: sees 100 requests -> 100 < 1000 -> allow
  Total: 1,000 requests allowed

Next round:
  Client sends 200 requests/node
  Each node: 200 < 1000 -> allow
  Total: 2,000 requests allowed -- 2x the limit!
```

**Solution: Shared Redis counter**

Four algorithm choices — each with different trade-offs:

```
Fixed Window Counter:
  key = "rl:{client_id}:{current_second}"
  INCR key; EXPIRE key 1
  Simple but has boundary problem:
    Client sends 1000 at 00:00:59.9 and 1000 at 00:01:00.1
    = 2000 requests in 200ms window, each window saw only 1000

Sliding Window Log:
  Store sorted set of request timestamps
  ZADD "rl:{client_id}" {timestamp} {request_id}
  ZREMRANGEBYSCORE ... 0 (now - window)
  ZCARD ... -> current count
  Exact but high memory: O(N) per client where N = requests in window

Sliding Window Counter (recommended):
  Use two fixed windows, interpolate
  rate = current_count + prev_count * (1 - position_in_window)

Token Bucket:
  More complex state but allows burst then steady rate
  Redis doesn't have native "refill over time" -> need Lua script
```

**Sliding window counter with Redis Lua:**

```lua
-- sliding_window_rate_limit.lua
local curr_key  = KEYS[1]  -- "rl:client123:1718000"
local prev_key  = KEYS[2]  -- "rl:client123:1717999"
local limit     = tonumber(ARGV[1])  -- 1000
local window    = tonumber(ARGV[2])  -- 60 (seconds)
local now_frac  = tonumber(ARGV[3])  -- 0.0 to 1.0 position in window

local curr_count = tonumber(redis.call('GET', curr_key) or '0')
local prev_count = tonumber(redis.call('GET', prev_key) or '0')

-- Weighted estimate of requests in the past 60 seconds
local estimated = curr_count + prev_count * (1.0 - now_frac)

if estimated >= limit then
    return {0, math.floor(estimated), limit}  -- rejected
end

-- Allow the request, increment current window counter
local new_count = redis.call('INCR', curr_key)
if new_count == 1 then
    redis.call('EXPIRE', curr_key, window * 2)  -- TTL = 2 windows
end

return {1, math.floor(estimated) + 1, limit}  -- allowed
```

```python
def check_rate_limit(client_id: str, limit: int, window: int = 60) -> bool:
    now = time.time()
    window_start = int(now / window) * window  # floor to window boundary
    position = (now % window) / window         # 0.0 to 1.0

    curr_key = f"rl:{client_id}:{window_start}"
    prev_key = f"rl:{client_id}:{window_start - window}"

    result = redis.eval(LUA_SCRIPT, 2, curr_key, prev_key,
                        limit, window, position)

    allowed, current_count, limit = result
    return bool(allowed)
```

### 3. Request Routing: Trie-Based Prefix Matching

Naive approach: iterate all routes in order, regex match each one.
At 35,000 RPS with 1,000 routes, that is 35 million regex operations per second.

**Solution: Prefix trie for O(path_length) lookup**

```
Routes:
  /api/v2/orders
  /api/v2/orders/{id}
  /api/v2/users
  /api/v1/orders

Trie structure:
  /
  └── api/
      ├── v2/
      │   ├── orders/ --> route: order-service-v2
      │   │   └── {id} --> route: order-service-v2 (single item)
      │   └── users/  --> route: user-service-v2
      └── v1/
          └── orders/ --> route: order-service-v1
```

```python
class RouteTrie:
    def __init__(self):
        self.root = TrieNode()

    def insert(self, path: str, route: Route):
        segments = path.strip('/').split('/')
        node = self.root
        for seg in segments:
            if seg not in node.children:
                node.children[seg] = TrieNode()
            node = node.children[seg]
        node.route = route

    def match(self, path: str) -> Optional[Route]:
        segments = path.strip('/').split('/')
        node = self.root
        params = {}

        for seg in segments:
            if seg in node.children:
                node = node.children[seg]
            elif '{id}' in node.children:    # wildcard segment
                params['id'] = seg
                node = node.children['{id}']
            else:
                return None  # no match

        return node.route
```

Lookup is O(path depth), typically 3-5 segments. No regex. No iteration.

### 4. Zero-Downtime Config Updates

Routes and policies change constantly (new service deployed, rate limit adjusted, A/B test configuration).

**Solution: etcd watch with atomic in-memory swap**

```
Config stored in etcd:
  /gateway/config/routes    -> JSON route table
  /gateway/config/plugins   -> plugin settings
  /gateway/config/version   -> monotonic version number

Each gateway node:
  1. On startup: load full config from etcd, build in-memory trie
  2. Start watch on /gateway/config/version
  3. When version changes:
     a. Fetch new config
     b. Build NEW trie in background (no lock needed)
     c. Atomic pointer swap: old_trie = new_trie  (one instruction)
     d. GC collects old trie
```

```python
class ConfigManager:
    def __init__(self, etcd_client):
        self.etcd = etcd_client
        self._current_config = None
        self._config_lock = threading.RLock()

    def start_watch(self):
        threading.Thread(target=self._watch_loop, daemon=True).start()

    def _watch_loop(self):
        for event in self.etcd.watch("/gateway/config/version"):
            new_config = self._build_config(
                self.etcd.get("/gateway/config/routes")
            )
            # Atomic swap - requests in flight use old config until this line
            with self._config_lock:
                self._current_config = new_config
            # Old config object is now unreferenced, GC will collect it

    def get_route(self, path: str) -> Optional[Route]:
        # Read the current config reference (atomic on 64-bit systems)
        config = self._current_config
        return config.trie.match(path)
```

Requests in flight see either the old or new config — both are valid during the brief transition. No request sees a partial/inconsistent config state.

### 5. Circuit Breaker: Route Around Unhealthy Backends

```
Circuit breaker states:

  +----------+     5 failures     +--------+     30 sec timeout
  |  CLOSED  |-----------------> |  OPEN  |-------------------+
  | (normal) |                   |(reject)|                   |
  +----------+                   +--------+                   |
       ^                              ^                        v
       |                              |            +------------+
       | 3 successes                  | probe      | HALF-OPEN  |
       +------------------------------|------------|  (1 probe) |
                                      |            +------------+
                                      | probe fails
```

```python
class CircuitBreaker:
    def __init__(self, service: str,
                 failure_threshold: int = 5,
                 recovery_timeout: float = 30.0,
                 success_threshold: int = 3):
        self.service = service
        self.state = "closed"
        self.failure_count = 0
        self.success_count = 0
        self.last_failure_time = 0.0
        self.failure_threshold = failure_threshold
        self.recovery_timeout = recovery_timeout
        self.success_threshold = success_threshold

    def call(self, fn: Callable, *args, **kwargs):
        if self.state == "open":
            if time.time() - self.last_failure_time > self.recovery_timeout:
                self.state = "half_open"
                self.success_count = 0
            else:
                raise CircuitOpen(f"Service {self.service} is unavailable")

        try:
            result = fn(*args, **kwargs)
            self._on_success()
            return result
        except (TimeoutError, ConnectionError, HTTP5xxError) as e:
            self._on_failure()
            raise

    def _on_success(self):
        if self.state == "half_open":
            self.success_count += 1
            if self.success_count >= self.success_threshold:
                self.state = "closed"
                self.failure_count = 0
        else:
            self.failure_count = 0  # reset on success in closed state

    def _on_failure(self):
        self.failure_count += 1
        self.last_failure_time = time.time()
        if self.failure_count >= self.failure_threshold:
            self.state = "open"
```

### 6. Plugin Architecture: Cross-Cutting Concerns as Composable Units

Each cross-cutting concern (auth, rate limit, logging) is a plugin. Plugins form an ordered pipeline.

```python
class Plugin(ABC):
    @abstractmethod
    def process(self, request: Request, context: Context) -> Optional[Response]:
        """Return Response to short-circuit, None to continue pipeline"""
        pass

class JWTAuthPlugin(Plugin):
    def process(self, request, context):
        token = request.headers.get("Authorization", "").replace("Bearer ", "")
        if not token:
            return Response(401, {"error": "missing token"})
        try:
            claims = validate_jwt(token)
            context.user_id = claims["sub"]
            context.roles = claims["roles"]
        except InvalidToken:
            return Response(401, {"error": "invalid token"})
        return None  # continue to next plugin

class RateLimitPlugin(Plugin):
    def process(self, request, context):
        client_id = context.user_id or request.client_ip
        allowed = check_rate_limit(client_id, limit=1000)
        if not allowed:
            return Response(429, {"error": "rate limit exceeded"},
                           headers={"Retry-After": "30"})
        return None

class RequestIDPlugin(Plugin):
    def process(self, request, context):
        request_id = request.headers.get("X-Request-ID") or str(uuid4())
        context.request_id = request_id
        request.headers["X-Request-ID"] = request_id  # propagate downstream
        return None

# Pipeline executor
class GatewayPipeline:
    def __init__(self, plugins: list[Plugin]):
        self.plugins = plugins

    def execute(self, request: Request) -> Response:
        context = Context()
        for plugin in self.plugins:
            result = plugin.process(request, context)
            if result is not None:
                return result  # short-circuit (auth failure, rate limit, etc.)
        # All plugins passed, route to upstream
        return proxy_to_upstream(request, context)
```

Plugin order matters: always auth before rate limiting (so rate limits are per-authenticated-user, not per-IP).

## Trade-offs and Failure Modes

### Trade-off: Inline gateway vs. sidecar proxy

| Inline API Gateway | Sidecar (Service Mesh - Istio/Envoy) |
|---|---|
| One chokepoint for all traffic | Distributed per-pod proxy |
| Easier to reason about | No single point of failure |
| Simpler operational model | More complex to configure |
| All traffic through one place | mTLS between all services |

**Recommendation**: Start with an inline API gateway. Move to service mesh only when you have 100+ microservices and need mTLS everywhere.

### Trade-off: Where to store rate limit state

| In-process (per node) | Redis (shared) |
|---|---|
| Zero latency | ~1ms per request |
| 10x limit bypass possible | Accurate global limits |
| Simple | Redis becomes critical path |
| Works without Redis | Need Redis HA cluster |

**Production choice**: Redis for the shared counter, but degrade gracefully if Redis is down — allow traffic and alert, rather than blocking all traffic because rate limiting is unavailable.

### Failure Mode: Redis down, rate limiting unavailable

```python
def check_rate_limit_with_fallback(client_id: str, limit: int) -> bool:
    try:
        return check_rate_limit(client_id, limit)
    except RedisConnectionError:
        # Redis is down: allow traffic, alert loudly
        metrics.increment("rate_limit.redis_unavailable")
        alert_on_call("Rate limiting unavailable - Redis down")
        return True  # fail open (allow) - don't take down the platform
```

This is the right trade-off: a brief period of slightly elevated traffic is preferable to a complete outage for all customers.

## Senior Engineer Perspective

**What interviewers want to hear:**

1. **Why is JWT validation local (no RPC)?** Because 35,000 RPS means you cannot afford a network call per request. Asymmetric crypto is the answer. Know this cold.

2. **How do you handle Redis being down for rate limiting?** Fail open — allow traffic and alert. Don't take the platform down because a rate limiting store is unavailable. This is called "graceful degradation."

3. **How do routes get updated without restart?** etcd watch + atomic in-memory config swap. New config is built in the background, then swapped atomically. Requests in flight use old or new config — both are valid.

**Real-world API gateways:**
- **AWS API Gateway**: Managed, serverless pricing, integrates with Lambda/IAM. Gets expensive at high volume ($3.50 per million requests).
- **Kong**: Open-source, Nginx-based, Lua plugin ecosystem. Most widely deployed self-hosted gateway.
- **NGINX + OpenResty**: Maximum performance, Lua-scriptable. Used by Cloudflare, Fastly. Operationally complex.
- **Envoy**: L7 proxy built for service meshes. Used by Istio. Best observability, steep config learning curve.
- **Traefik**: Cloud-native, auto-discovers services from Docker/Kubernetes labels. Popular in smaller deployments.

**The key insight to land the interview:**

> "The gateway must be dumb and fast. Move all stateful logic to backing stores: rate limit counters in Redis, route config in etcd, JWT public keys in JWKS cache. The gateway's job is to be a thin, stateless enforcement layer. Every millisecond of added latency is multiplied across every request from every user."
