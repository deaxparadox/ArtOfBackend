## The Key Architectural Insight

The central engineering challenge in a booking system is **preventing double-booking**: two users must not be able to book the same property for the same nights simultaneously. This is a concurrency problem — a race condition at the database level — and solving it correctly under high load is the heart of this design.

The second hard problem is **search**: users want to filter by location, dates, and price simultaneously. Geospatial queries combined with availability checks are expensive, and the two data sources (property locations and availability calendars) live in different systems optimized for different access patterns.

## Section 1: Requirements Clarification

| Question | Why it matters |
|---|---|
| Scope: listing search, booking, payments, or all three? | Scope the design — covering all three is 3 separate systems |
| Instant booking or request-to-book? | Instant booking requires a reliable atomic hold; request-to-book is async |
| How far in advance can bookings be made? | Determines calendar size (1 year vs. 2 years of dates per property) |
| Can hosts block dates manually? | Yes -- host availability management is a feature, not an edge case |
| Expected scale: properties, bookings/day? | Drives concurrency and database sizing |
| Dynamic pricing or fixed price per property? | Dynamic pricing is a complex subsystem requiring its own design |

**Scope for this design:** Property listing + availability search + instant booking with payment. 10 million properties, ~3 million bookings/day, strong consistency for double-booking prevention, dynamic pricing support.

## Section 4: Deep Dive — The Hard Problems

### 4.1 Preventing Double-Booking

Alice and Bob both find property "villa-123" available on July 4th. Both click "Book." Both requests arrive at the server simultaneously. Only one should succeed.

**The naive approach (broken):**
```sql
-- Thread 1 (Alice):
SELECT COUNT(*) FROM reservations
WHERE property_id = 'villa-123'
  AND check_in <= '2024-07-04'
  AND check_out > '2024-07-04';
-- Returns 0 (no booking yet)

-- Thread 2 (Bob) executes the same SELECT here, ALSO gets 0

-- Thread 1 (Alice): INSERT INTO reservations ...  -- succeeds
-- Thread 2 (Bob):   INSERT INTO reservations ...  -- ALSO succeeds!
-- Result: property is double-booked for July 4th
```

The SELECT and INSERT are not atomic. This is a Time-of-Check to Time-of-Use (TOCTOU) race condition.

**Solution 1: Pessimistic Locking (SELECT FOR UPDATE)**

```sql
BEGIN;

-- Acquire an exclusive row-level lock on this property.
-- Bob's identical transaction BLOCKS here until Alice commits.
SELECT id FROM properties
WHERE property_id = 'villa-123'
FOR UPDATE;

-- Now check availability -- no race is possible, we hold the lock
SELECT COUNT(*) FROM reservations
WHERE property_id = 'villa-123'
  AND daterange(check_in, check_out) && daterange('2024-07-04', '2024-07-05');
-- Returns 0 for Alice, returns 1 for Bob (after Alice commits)

-- If available: insert the reservation
INSERT INTO reservations (property_id, guest_id, check_in, check_out, status)
VALUES ('villa-123', 'alice-456', '2024-07-04', '2024-07-05', 'confirmed');

COMMIT;
-- Alice's lock is released. Bob's blocked SELECT FOR UPDATE proceeds,
-- finds Alice's booking, and returns 0 rows inserted to Bob.
```

**Solution 2: Calendar Table with Per-Date Rows (most scalable)**

Create one row per (property, date):

```sql
CREATE TABLE availability_calendar (
    property_id     UUID    NOT NULL,
    date            DATE    NOT NULL,
    status          TEXT    NOT NULL DEFAULT 'available',  -- available/held/booked
    reservation_id  UUID,
    held_until      TIMESTAMPTZ,
    PRIMARY KEY (property_id, date)
);
```

Booking July 4-6 means atomically updating 3 rows inside one transaction:

```sql
BEGIN;

UPDATE availability_calendar
SET    status = 'booked', reservation_id = 'res-xyz'
WHERE  property_id = 'villa-123'
  AND  date IN ('2024-07-04', '2024-07-05', '2024-07-06')
  AND  status = 'available';

-- Check that ALL 3 rows were updated
-- (if only 1 or 2 updated, someone else held a date -- partial availability = conflict)
GET DIAGNOSTICS rows_updated = ROW_COUNT;
IF rows_updated != 3 THEN
    ROLLBACK;
    RAISE EXCEPTION 'property not fully available for requested dates';
END IF;

INSERT INTO reservations (...) VALUES (...);

COMMIT;
```

Because `(property_id, date)` is a PRIMARY KEY, two concurrent transactions cannot both update the same row simultaneously — the database serializes them. One wins, one gets a stale read and the `WHERE status = 'available'` clause filters it out.

**Solution 3: PostgreSQL Range Exclusion Constraint (declarative)**

```sql
CREATE TABLE reservations (
    reservation_id  UUID        PRIMARY KEY DEFAULT gen_random_uuid(),
    property_id     UUID        NOT NULL,
    guest_id        UUID        NOT NULL,
    check_in        DATE        NOT NULL,
    check_out       DATE        NOT NULL,
    status          TEXT        NOT NULL DEFAULT 'confirmed',
    -- Declarative no-overlap: PostgreSQL rejects overlapping date ranges
    -- for the same property_id at the DB constraint level
    CONSTRAINT no_double_booking EXCLUDE USING gist (
        property_id WITH =,
        daterange(check_in, check_out, '[)') WITH &&
    )
);
```

The `EXCLUDE USING gist` constraint tells PostgreSQL: "for any two rows with the same `property_id`, their date ranges must not overlap." Any INSERT that would create an overlap raises a constraint violation — atomically enforced by the database. This is the most elegant solution: no application-level locking code required.

**Comparison:**

| Approach | Pros | Cons |
|---|---|---|
| `SELECT FOR UPDATE` | Simple, works on any DB | Locks entire property row, reduces throughput |
| Calendar table + conditional UPDATE | Scalable, per-date granularity | Requires updating N rows per booking |
| Exclusion constraint (PostgreSQL) | Declarative, DB-enforced, elegant | PostgreSQL-specific, requires GiST index |

### 4.2 Two-Phase Booking: Hold → Confirm

For instant booking with payment, you need a hold to prevent the property from appearing available while someone is actively paying:

```
Phase 1: HOLD (optimistic reservation)
  - Atomically set calendar dates to 'held' with expiry = NOW() + 10 minutes
  - Create reservation record with status = 'pending_payment'
  - Return checkout page with timer: "complete payment within 10 minutes"

Phase 2: CONFIRM (after payment succeeds)
  - Payment service charges card
  - On success:  UPDATE calendar SET status = 'booked'
                 UPDATE reservation SET status = 'confirmed'
                 Publish booking_confirmed event to Kafka
  - On failure:  UPDATE calendar SET status = 'available'
                 UPDATE reservation SET status = 'failed'

Background cleanup (runs every 60 seconds):
  UPDATE availability_calendar
  SET    status = 'available', reservation_id = NULL, held_until = NULL
  WHERE  status = 'held'
    AND  held_until < NOW();

  UPDATE reservations
  SET    status = 'expired'
  WHERE  status = 'pending_payment'
    AND  created_at < NOW() - INTERVAL '10 minutes';
```

This prevents the property from appearing available during checkout without permanently blocking it if payment fails or the user abandons checkout.

**Reservation state machine:**
```
                   +-----------------+
     user initiates|    PENDING      | user abandons / payment times out
   ---------------->(hold placed,    +---------------------------------> EXPIRED
                   | 10min window)   |
                   +-------+---------+
                           |
                           | payment succeeds
                           v
                   +-----------------+
                   |   CONFIRMED     | host or guest cancels
                   |                 +-----------------------------------> CANCELLED
                   +-------+---------+
                           |
                           | check-in date arrives
                           v
                   +-----------------+
                   |   CHECKED_IN    |
                   +-------+---------+
                           |
                           | check-out date arrives
                           v
                   +-----------------+
                   |   COMPLETED     |
                   +-----------------+
```

State transitions are strictly validated — a COMPLETED booking cannot become CONFIRMED. This prevents bugs from async events (webhooks, retries) arriving out of order.

### 4.3 Geospatial Search + Availability Combined

Users search: "2-bedroom near San Francisco, available July 4-7, under $200/night."

This combines three orthogonal filters across different data stores:
1. **Geospatial:** within X km of lat/lng
2. **Availability:** no bookings for those dates
3. **Price:** within budget

**Why this is hard:** Geo and availability live in different systems. You cannot efficiently JOIN them in real time for 50,000 searches/second.

**Approach: Two-pass with Redis bitmap availability cache**

```
Pass 1: Elasticsearch geospatial + attribute query
  GET /properties/_search
  {
    "query": {
      "bool": {
        "must": [
          { "geo_distance": { "distance": "50km", "location": { "lat": 37.7, "lon": -122.4 } } },
          { "range": { "bedrooms": { "gte": 2 } } },
          { "range": { "base_price": { "lte": 200 } } }
        ]
      }
    },
    "size": 1000
  }
  Result: [property_id_1, property_id_2, ..., property_id_1000]

Pass 2: Redis bitmap availability check (sub-millisecond)
  For each property_id, check: are July 4, 5, 6 ALL available?

  Key format: avail:{property_id}:{YYYYMM}
  Value:      bitmap (bit N = day N of the month, 1=available, 0=unavailable)

  GETBIT avail:prop-1:202407 4   --> 1
  GETBIT avail:prop-1:202407 5   --> 1
  GETBIT avail:prop-1:202407 6   --> 1
  All 3 = 1 --> property is available

  With pipelining: check 1000 properties x 3 days = 3000 GETBIT ops in ~5ms

Pass 3: Filter, sort by relevance/price, paginate
  Return top 20 available properties
```

**Redis bitmap storage math:**
- 1 bitmap per property per month = 31 bits ~ 4 bytes
- 12 months per property = 48 bytes
- 10 million properties = **480 MB** -- fits entirely in Redis RAM

Bitmap updates on booking: when a date is booked, `SETBIT avail:{property_id}:{YYYYMM} {day} 0`. On cancellation, `SETBIT ... 1`. The bitmap is the read-optimized view of the calendar table.

### 4.4 Dynamic Pricing

```sql
CREATE TABLE pricing_rules (
    property_id     UUID        NOT NULL REFERENCES properties(property_id),
    base_price      DECIMAL(10,2) NOT NULL,          -- host's set minimum
    weekend_mult    DECIMAL(4,2) NOT NULL DEFAULT 1.0, -- e.g., 1.2 = +20% Fri/Sat
    cleaning_fee    DECIMAL(10,2) NOT NULL DEFAULT 0,
    min_price       DECIMAL(10,2) NOT NULL,
    max_price       DECIMAL(10,2),
    smart_pricing   BOOLEAN NOT NULL DEFAULT FALSE,  -- opt-in to ML pricing
    PRIMARY KEY (property_id)
);

CREATE TABLE demand_factors (
    region          VARCHAR(64) NOT NULL,
    date            DATE        NOT NULL,
    demand_mult     DECIMAL(4,2) NOT NULL DEFAULT 1.0,  -- ML model output: 0.8-2.0
    PRIMARY KEY (region, date)
);
```

**Price calculation per night:**
```
base = pricing_rules.base_price
day_mult = 1.2 if Friday/Saturday else 1.0
demand_mult = demand_factors.demand_mult for (property.region, date)
final_price = CLAMP(base * day_mult * demand_mult, min_price, max_price)
```

A background ML pipeline updates `demand_factors` nightly based on search volume, booking conversion rates, and comparable property prices in the region. Hosts see their "suggested price" update dynamically; they can override with manual min/max constraints.

);

    USING gist (ST_MakePoint(lng, lat)::geography);

    property_id     UUID          NOT NULL REFERENCES properties(property_id),

    status          TEXT          NOT NULL DEFAULT 'available',

    PRIMARY KEY (property_id, date)
);

CREATE TABLE reservations (
    reservation_id  UUID          PRIMARY KEY DEFAULT gen_random_uuid(),
    property_id     UUID          NOT NULL REFERENCES properties(property_id),
    guest_id        UUID          NOT NULL REFERENCES users(user_id),

    check_out       DATE          NOT NULL,

    )

## Section 6: Trade-offs and Failure Modes

| Scenario | Impact | Mitigation |
|---|---|---|
| Payment succeeds, DB write fails | Guest charged, no booking created | Idempotency key on payment; booking service listens to payment webhook to confirm |
| Calendar table hotspot on popular property | All booking attempts serialize | Expected and correct -- per-property serialization IS the requirement |
| Elasticsearch out of sync with DB | Search shows wrong availability | Availability always re-verified at booking time; ES is eventually consistent |
| Host manually blocks dates while guest is in checkout | Host action races with guest hold | Hold sets `status = 'held'` first; host action only succeeds against `status = 'available'` |
| Double charge on network timeout retry | Guest charged twice | `idempotency_key` on payment API call -- same key = same charge, never duplicated |
| Background cleanup job fails | Expired holds never released | Monitor held_count; alert if holds older than 15 minutes exist |
| Redis bitmap cache inconsistent with DB | Wrong availability shown to users | Redis is rebuilt from DB on startup; cache-aside pattern on booking confirms |

## Section 7: Senior Engineer Perspective

Double-booking prevention is a solved problem, but many candidates get it wrong under interview pressure. The common mistakes:

**1. Application-level checks are never sufficient.** Any solution that reads availability and then writes a booking in two separate SQL statements is broken under concurrency. The database must enforce the invariant through transactions, locks, or constraints. No application code can substitute for database-level atomicity.

**2. The PostgreSQL exclusion constraint is the elegant answer.** It expresses the no-overlap invariant declaratively — no lock management, no application code, the DB rejects overlapping inserts with a constraint error. Know it, name it, explain when to use it.

**3. The hold expiry job is operationally critical.** At 3 million bookings/day with a 5% checkout abandonment rate, you have ~150,000 expired holds per day to clean up. If the cleanup job falls behind, properties appear unavailable when they are not. Monitor hold age as a business metric.

**4. Search and booking are separate scaling problems with conflicting requirements.** Search is read-heavy, can be served from Elasticsearch with Redis caches, and can tolerate eventual consistency. Booking is write-critical and must be strongly consistent. Conflating them leads to either an over-engineered search or an under-consistent booking system.

**5. Idempotency keys on payments are non-negotiable.** Network timeouts cause clients to retry. Without idempotency keys, retries create duplicate charges. The payment service must be idempotent by design, not by hope.

> **The interview signal:** The candidate who says "I'd use SELECT FOR UPDATE" understands concurrency. The candidate who also mentions the PostgreSQL exclusion constraint, explains LWW vs. per-date calendar atomicity trade-offs, and asks "what's the expected booking abandonment rate so we know how aggressively to run the hold cleanup?" -- that candidate demonstrates production thinking, not just interview pattern-matching.
