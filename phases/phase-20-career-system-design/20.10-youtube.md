# 20.10 — Design YouTube / Video Streaming

## Why This Problem Is Hard

Video is the most bandwidth-intensive workload on the internet. YouTube alone accounts for ~11% of all internet traffic. The design challenges are:

- **Upload pipeline**: A raw 4K video at 100 Mbps is gigabytes per minute. Accepting it, storing it, encoding it into multiple resolutions, and making it available — all without blocking the uploader — requires an async pipeline with dozens of components.
- **Encoding at scale**: YouTube ingests 500 hours of video per minute. Encoding is CPU-intensive. You need a distributed encoding fleet.
- **Delivery at global scale**: Serving a 4K stream to 2 billion users requires CDN edge nodes on every continent. You cannot serve video from a single origin.
- **Adaptive bitrate**: A user on a mobile phone on a train should not buffer. The player must automatically switch quality without interrupting playback.
- **View count accuracy**: Writing a DB row on every view would destroy the database at YouTube's scale. You need approximate counters with eventual consistency.

## Capacity Estimation

```
Uploads per day:          720,000 videos
Average raw video size:   1 GB (mix of short and long)
Raw storage per day:      720,000 * 1 GB = 720 TB/day

After encoding (6 resolutions + original):
  Encoded storage:        ~3x raw (multiple bitrates) = 2.16 PB/day
  Total storage per year: ~780 PB  (requires tiered object storage with cold tier)

Views per day:            5 billion
Average view duration:    7 minutes
Average video bitrate:    2 Mbps (720p)
Bandwidth per view:       2 Mbps * 7 * 60 = 840 Mb = 105 MB
Total egress per day:     5B * 105 MB = 525 PB/day
CDN egress per second:    525 PB / 86400 = ~6 TB/s  (massive CDN required)

View count writes:
  Without Redis:          5B writes/day = 57,870 writes/sec  (too high for OLTP DB)
  With Redis batching:    ~57 writes/sec (flush every 1000 views)
```

## Data Model

```sql
-- Core video record
CREATE TABLE videos (
    id          BIGSERIAL PRIMARY KEY,              -- Snowflake ID (URL-safe base62 externally)
    user_id     BIGINT NOT NULL,
    title       VARCHAR(500) NOT NULL,
    description TEXT,
    status      VARCHAR(20) NOT NULL DEFAULT 'processing'
                    CHECK (status IN ('processing', 'ready', 'failed', 'deleted')),
    duration_sec INT,                               -- null until encoding completes
    view_count  BIGINT NOT NULL DEFAULT 0,          -- eventually consistent, flushed from Redis
    created_at  TIMESTAMP NOT NULL DEFAULT now(),
    published_at TIMESTAMP                          -- null until status = ready
);

CREATE INDEX idx_videos_user ON videos(user_id, created_at DESC);
CREATE INDEX idx_videos_status ON videos(status) WHERE status = 'processing';

-- One row per resolution per video
CREATE TABLE video_versions (
    video_id        BIGINT NOT NULL REFERENCES videos(id),
    resolution      VARCHAR(10) NOT NULL,   -- '360p', '720p', '1080p', '4k'
    storage_path    TEXT NOT NULL,          -- s3://bucket/videos/{video_id}/{resolution}/
    bitrate_kbps    INT NOT NULL,
    file_size_bytes BIGINT,
    manifest_path   TEXT,                   -- HLS .m3u8 or DASH .mpd path
    encoded_at      TIMESTAMP,
    PRIMARY KEY (video_id, resolution)
);

-- Upload session tracking for resumable uploads
CREATE TABLE upload_sessions (
    session_id      UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id         BIGINT NOT NULL,
    video_id        BIGINT NOT NULL REFERENCES videos(id),
    total_size      BIGINT NOT NULL,
    chunks_total    INT NOT NULL,
    chunks_uploaded INT NOT NULL DEFAULT 0,
    storage_prefix  TEXT NOT NULL,          -- where chunks are temporarily stored
    expires_at      TIMESTAMP NOT NULL,     -- session expires after 24 hours of inactivity
    created_at      TIMESTAMP NOT NULL DEFAULT now()
);
```

## Deep Dive: The Four Hard Problems

### 1. Resumable Chunked Uploads

A 10 GB file cannot be uploaded in a single HTTP request. Network drops at 9 GB mean starting over. Resumable uploads split the file client-side:

```
Client-side algorithm:
1. POST /uploads/init  { filename, size, content_type }
   -> Server creates upload_session, returns { session_id, chunk_size: 10MB }

2. For each chunk i (0-indexed):
   GET /uploads/{session_id}/status
   -> Server returns { chunks_uploaded: [0, 1, 2, ...] }

   If chunk i not yet uploaded:
   PUT /uploads/{session_id}/chunks/{i}
       Body: bytes[i*chunk_size : (i+1)*chunk_size]
   -> Server writes chunk to s3://tmp/{session_id}/chunk_{i}
   -> Server atomically increments chunks_uploaded counter

3. POST /uploads/{session_id}/complete
   -> Server copies all chunks to final object
   -> Server deletes temp chunks
   -> Server enqueues video for encoding
```

Pre-signed S3 URLs can be issued per-chunk, allowing the client to upload directly to object storage without routing through the application server — this reduces server load and latency.

### 2. Encoding Pipeline

Raw video arrives as a single large file. The encoding pipeline:

```
1. Encoding Queue receives:
   { video_id, source_path, target_resolutions: [360p, 720p, 1080p, 4k] }

2. A Coordinator worker picks up the job, splits it into per-resolution sub-jobs.
   Each sub-job is enqueued independently, allowing parallel execution.

3. Encoder workers (one per resolution per video) run FFmpeg:

   ffmpeg -i input.mp4 \
     -vf scale=-2:720 \
     -c:v libx264 -crf 23 -preset fast \
     -c:a aac -b:a 128k \
     -hls_time 6 -hls_playlist_type vod \
     output_720p/index.m3u8

   This produces HLS segments (6-second .ts files) and a manifest (.m3u8).

4. On success, each encoder writes to video_versions and emits a completion event.

5. Coordinator listens for completion events. When all resolutions are done:
   UPDATE videos SET status='ready', duration_sec=X, published_at=now()
   WHERE id = ?
   Triggers notification to uploader.

6. On failure, the job is retried up to 3 times. After 3 failures:
   UPDATE videos SET status='failed'
   Alert is sent to the uploader.
```

Key operational decision: encoders are stateless workers. They pull work from a queue, write output to object storage, and exit. You can scale the encoder fleet up and down based on queue depth. During peak upload hours, auto-scale to 1000 concurrent encoders.

### 3. Adaptive Bitrate Streaming (ABR)

ABR is why YouTube rarely buffers even on poor connections. Instead of one video file, the encoded output is:

```
index_master.m3u8  (master playlist, lists all quality levels)
  -> index_360p.m3u8  (360p playlist: lists 6-second segment files)
       segment_000.ts
       segment_001.ts
       ...
  -> index_720p.m3u8
       segment_000.ts
       ...
  -> index_1080p.m3u8
       ...
```

HLS (HTTP Live Streaming) is Apple's protocol; DASH (Dynamic Adaptive Streaming over HTTP) is the MPEG standard. Both work the same way: the player downloads a manifest, picks a quality, and downloads sequential segments. After each segment, the player measures download speed and switches quality up or down.

```
Player algorithm:

buffer_seconds = len(downloaded_but_not_played_segments) * 6

if buffer_seconds < 10:
    switch_down()   # reduce quality to fill buffer faster
elif buffer_seconds > 30 and current_speed > next_quality_bitrate * 1.5:
    switch_up()     # upgrade quality for better experience
```

This is why YouTube quality changes happen smoothly at segment boundaries (every 6 seconds) rather than mid-playback.

### 4. View Count at Scale

Naive approach: `UPDATE videos SET view_count = view_count + 1 WHERE id = ?`

At 57,870 views/sec across potentially 1 million active videos, this is:
- 57,870 write transactions per second
- Row-level locks on the `view_count` column cause contention on hot videos

Redis-based batching:

```python
def record_view(video_id: str):
    # O(1), no lock contention, runs in memory
    redis.incr(f"views:{video_id}")

# Background job runs every 60 seconds
def flush_views_to_db():
    for key in redis.scan_match("views:*"):
        video_id = key.split(":")[1]
        count = redis.getdel(key)  # atomic get-and-delete
        db.execute(
            "UPDATE videos SET view_count = view_count + %s WHERE id = %s",
            [count, video_id]
        )
```

This reduces DB writes from 57,870/sec to roughly 1 write per video per minute. View counts are eventually consistent with up to 60 seconds of lag — acceptable for a metric users interpret as approximate anyway.

For analytics (how many unique views, watch duration histograms), send view events to Kafka and process with a streaming framework (Flink/Spark). Never aggregate analytics in the OLTP database.

## CDN Strategy

Not all videos are equal. The distribution follows a power law:

```
Top 1% of videos  = ~90% of all views  (viral, popular, recent)
Bottom 99% of videos = 10% of views    (long-tail, niche, old)
```

**Push strategy** for popular videos:
When a video crosses a popularity threshold (e.g., 10K views in 1 hour), proactively push all its segments to all CDN edge nodes globally. Use a background job that monitors view velocity and triggers CDN prewarming.

**Pull strategy** for long-tail:
CDN fetches from origin on first request per edge location. Uses LRU eviction to keep edge storage manageable. Most long-tail videos are only requested in 1-2 geographies; no reason to push globally.

**Signed URLs for private/paid content:**

```
CDN generates:
  https://cdn.example.com/video/abc123/720p/seg_001.ts
    ?Expires=1718003600
    &Signature=HMAC(secret, url + expiry)
    &KeyPairId=CDN_KEY_1

Client requests a new signed URL from the API server every 30 minutes.
The CDN validates the signature without calling the origin.
```

## Trade-offs Table

| Decision | Option A | Option B | Notes |
|---|---|---|---|
| Upload method | Single multipart POST | Chunked resumable upload | Resumable wins for large files |
| Encoding | Synchronous (blocking) | Async queue-based | Async always; encoding is too slow to block |
| Encoding parallelism | Sequential resolutions | Fan-out per resolution | Fan-out reduces time-to-ready by 4x |
| Streaming protocol | HLS | DASH | HLS for Apple devices; DASH for Android; serve both |
| View count | Direct DB write | Redis buffer + batch flush | Redis buffer avoids write contention |
| CDN | Pull only | Push for popular + pull for long-tail | Hybrid reduces origin load for viral content |
| Storage tier | Hot (SSD) for all | Hot + cold archival | Tiered storage; most videos are rarely watched |

## Senior Engineer Perspective

**What juniors get wrong:**

1. Proposing a synchronous encoding step during upload. Encoding a 1-hour video takes minutes. The upload handler cannot block the HTTP request for minutes.
2. Storing video files in a relational database as BLOBs. Object storage (S3-compatible) is purpose-built for this: cheap, durable, infinitely scalable, CDN-integrable.
3. Not separating the manifest from the segments. The manifest (.m3u8) is tiny and changes frequently (for live streams) or never (for VOD). Segments are large and immutable once written. Apply different cache TTLs: segments can be cached forever (content-addressable), manifests need shorter TTLs.

**The real cost is egress, not storage:**
Storing 1 PB on S3 costs roughly $23,000/month. Transferring 1 PB out via AWS CloudFront costs roughly $85,000/month. At YouTube's 525 PB/day, the CDN bill at public cloud rates would be enormous. This is why YouTube operates its own CDN infrastructure (Google's global fiber and edge pop network). At smaller scale (under 1 PB/month), managed CDN (CloudFront, Fastly, Cloudflare) is cost-effective.

**Encoding optimization — scene detection:**
FFmpeg supports scene-cut detection to place segment boundaries at scene changes rather than fixed 6-second intervals. This improves compression efficiency (frames within a segment are more temporally similar) and improves subjective quality at the same bitrate. Netflix encodes each title multiple times with different quality targets per scene complexity.

**The thumbnail pipeline:**
When encoding completes, generate N thumbnail candidates (e.g., every 10 seconds), run them through a quality classifier (sharpness, face detection, brightness), select the best N candidates, and store them in object storage. The uploader can then choose a thumbnail from the candidates or upload a custom one.
