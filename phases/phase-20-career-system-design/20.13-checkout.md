## Why This System Is Hard

Checkout looks like a form submission. It is actually a distributed transaction that must coordinate two external systems — inventory and payment — neither of which shares a database. The hard problems are:

1. **Inventory + payment atomicity** — You cannot deduct inventory and charge the card in a single ACID transaction. They live in different services. What happens if inventory reserves successfully but payment fails? You must compensate.
2. **Overselling** — Two users simultaneously buy the last item. Both see stock = 1. Both proceed. You ship to neither and refund both. This is catastrophic at scale (think flash sales).
3. **Idempotency** — A user double-taps "Place Order" due to a slow network. You must not charge them twice or create two orders.
4. **Payment + inventory race** — The reservation TTL expires during payment processing. You must handle the case where the hold releases while the card is being charged.

---

## Requirements Clarification

**Functional requirements:**
- User can add items to a cart
- Checkout reserves inventory and initiates payment
- On payment success: decrement inventory, create order, send confirmation
- On payment failure: release inventory reservation
- Support flash sales (100K users competing for 500 units simultaneously)
- Idempotent checkout: submitting twice produces one order

**Non-functional requirements:**
- Never oversell — if stock = 0, no order should succeed
- Checkout latency: < 500ms for reservation + payment initiation
- 50,000 concurrent checkouts at peak (Black Friday)
- Flash sale peak: 500,000 checkouts/minute = 8,333/second
- Availability: 99.95% (checkout downtime = direct revenue loss)

## Data Model

```sql
-- Inventory: available and reserved tracked separately
CREATE TABLE inventory (
  sku         TEXT PRIMARY KEY,
  product_id  UUID NOT NULL,
  available   INTEGER NOT NULL DEFAULT 0 CHECK (available >= 0),
  reserved    INTEGER NOT NULL DEFAULT 0 CHECK (reserved >= 0),
  version     INTEGER NOT NULL DEFAULT 0,  -- for optimistic locking
  updated_at  TIMESTAMPTZ DEFAULT NOW()
);

-- Soft reservations: the hold during payment window
CREATE TABLE reservations (
  id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  order_id    UUID,                         -- null until order confirmed
  checkout_id UUID NOT NULL,                -- idempotency: one per checkout
  sku         TEXT NOT NULL REFERENCES inventory(sku),
  quantity    INTEGER NOT NULL,
  status      TEXT NOT NULL DEFAULT 'held'
                CHECK (status IN ('held', 'confirmed', 'released', 'expired')),
  expires_at  TIMESTAMPTZ NOT NULL DEFAULT NOW() + INTERVAL '10 minutes',
  created_at  TIMESTAMPTZ DEFAULT NOW()
);

CREATE INDEX reservations_expires_idx
  ON reservations(expires_at)
  WHERE status = 'held';

CREATE UNIQUE INDEX reservations_checkout_sku_idx
  ON reservations(checkout_id, sku)
  WHERE status = 'held';  -- one held reservation per sku per checkout

-- Orders: immutable snapshot of what was purchased
CREATE TABLE orders (
  id           UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id      UUID NOT NULL,
  checkout_id  UUID UNIQUE NOT NULL,  -- idempotency key
  status       TEXT NOT NULL DEFAULT 'pending'
                 CHECK (status IN ('pending', 'paid', 'cancelled', 'refunded')),
  total_cents  INTEGER NOT NULL,
  currency     CHAR(3) NOT NULL DEFAULT 'USD',
  items        JSONB NOT NULL,  -- snapshot at checkout time (immutable)
  payment_id   TEXT,            -- external payment provider's ID
  created_at   TIMESTAMPTZ DEFAULT NOW()
);
```

## Deep Dive: The Hard Problems

### 1. Inventory Reservation + Payment Atomicity

The core problem: these are two separate systems. No distributed transaction can span them. You must use the **Saga pattern**.

**Saga: reserve inventory -> charge payment -> confirm order**

```
Step 1: Reserve inventory (Inventory Service)
  - Decrement available, increment reserved
  - Create reservation record with TTL
  - Returns: reservation_id

Step 2: Charge payment (Payment Service)
  - Call Stripe/Braintree with card token
  - Returns: payment_id OR error

Step 3a: SUCCESS
  - Confirm reservation (decrement reserved, reservation status -> confirmed)
  - Create order record
  - Emit order.created event

Step 3b: FAILURE (compensating transaction)
  - Release reservation (increment available, decrement reserved)
  - reservation status -> released
  - No order created
  - Emit payment.failed event -> notify user
```

```python
class CheckoutSaga:
    def execute(self, checkout_id: str, cart: Cart, payment_method: str):
        reservation_ids = []

        try:
            # Step 1: Reserve inventory for all items
            for item in cart.items:
                res_id = inventory_service.reserve(
                    checkout_id=checkout_id,
                    sku=item.sku,
                    quantity=item.quantity
                )
                reservation_ids.append(res_id)

            # Step 2: Charge payment
            payment_id = payment_service.charge(
                amount_cents=cart.total_cents,
                payment_method=payment_method,
                idempotency_key=f"checkout_{checkout_id}"
            )

            # Step 3a: Confirm all reservations
            for res_id in reservation_ids:
                inventory_service.confirm(res_id)

            order = order_service.create(
                checkout_id=checkout_id,
                cart=cart,
                payment_id=payment_id
            )
            return order

        except PaymentDeclined as e:
            # Compensate: release all reservations
            for res_id in reservation_ids:
                inventory_service.release(res_id)
            raise CheckoutFailed(f"Payment declined: {e}")

        except InventoryInsufficient as e:
            # Release any partial reservations
            for res_id in reservation_ids:
                inventory_service.release(res_id)
            raise CheckoutFailed(f"Item out of stock: {e}")
```

### 2. Overselling: Race Condition When Two Users Buy the Last Item

**The scenario:**

```
Time: 10:00:00.001   User A reads inventory: sku=SHOE, available=1
Time: 10:00:00.002   User B reads inventory: sku=SHOE, available=1
Time: 10:00:00.010   User A calls reserve(SHOE, qty=1) -> succeeds
Time: 10:00:00.011   User B calls reserve(SHOE, qty=1) -> ??? also succeeds?
```

The naive implementation reads then writes in two separate statements with a gap. Both users win. You ship nothing and refund both.

**Solution A: Pessimistic locking (SELECT FOR UPDATE)**

```sql
BEGIN;
-- Lock the row before reading
SELECT available FROM inventory
WHERE sku = 'SHOE-42' FOR UPDATE;
-- Other transactions block here until we COMMIT

UPDATE inventory
SET available = available - 1,
    reserved  = reserved  + 1
WHERE sku = 'SHOE-42'
  AND available >= 1;  -- guard condition

COMMIT;
-- Returns 1 row = success; 0 rows = out of stock
```

Works well for moderate concurrency. Under extreme load (flash sales), the `FOR UPDATE` lock creates a serial bottleneck: all 100,000 checkout threads queue behind the lock.

**Solution B: Optimistic locking (version column)**

```sql
-- No lock on read
SELECT available, version FROM inventory WHERE sku = 'SHOE-42';
-- Returns: available=1, version=42

-- Write includes version check
UPDATE inventory
SET available = available - 1,
    reserved  = reserved  + 1,
    version   = version   + 1
WHERE sku = 'SHOE-42'
  AND version = 42        -- only updates if no one else changed it
  AND available >= 1;

-- 0 rows updated = conflict; retry with fresh read
```

Better concurrency under moderate load. Under extreme flash sale load, retry storm: 99,999 users all retrying after the first one wins.

**Solution C: Redis atomic decrement (for flash sales)**

The flash sale problem: 100,000 users, 500 units. A DB lock serializes all 100,000 through one thread.

Redis processes commands on a single thread — every command is atomic. Use a Lua script to atomically check and decrement:

```lua
-- Called via redis.eval(), runs atomically
local key = KEYS[1]             -- "inventory:SHOE-42"
local requested = tonumber(ARGV[1])

local available = tonumber(redis.call('GET', key) or '0')
if available < requested then
    return -1  -- out of stock
end

local new_count = available - requested
redis.call('SET', key, new_count)
return new_count  -- new available count
```

```python
def reserve_flash_sale(sku: str, quantity: int) -> Optional[str]:
    result = redis.eval(LUA_SCRIPT, 1, f"inventory:{sku}", quantity)
    if result == -1:
        raise InventoryInsufficient(sku)

    # Async: write reservation to Postgres (durable record)
    reservation_id = str(uuid4())
    task_queue.enqueue(write_reservation_to_db, reservation_id, sku, quantity)
    return reservation_id
```

Redis handles 100,000 concurrent requests in ~1ms total. The first 500 win immediately. The remaining 99,500 get an instant "out of stock" response. No DB lock contention.

### 3. Idempotency: Retry-Safe Checkout

User taps "Place Order." Network times out. Client retries. Should this create a second order? No.

**Solution: Checkout ID as idempotency key**

The client generates a UUID (checkout_id) before calling the API. The API uses this as the idempotency key.

```python
@app.post("/checkout")
def checkout(req: CheckoutRequest):
    # checkout_id comes from client (UUID generated before first call)
    checkout_id = req.checkout_id

    # Check if this checkout was already processed
    existing = db.query(
        "SELECT id, status FROM orders WHERE checkout_id = %s",
        (checkout_id,)
    )
    if existing:
        return {"order_id": existing.id, "status": existing.status}

    # Proceed with new checkout
    order = checkout_saga.execute(checkout_id, req.cart, req.payment_method)
    return {"order_id": order.id, "status": order.status}
```

The `UNIQUE` constraint on `orders.checkout_id` enforces this at the database level too. Concurrent duplicate requests: the first one creates the order, the second one hits the UNIQUE constraint and returns the existing order.

### 4. Reservation TTL Expiry During Payment

Payment takes 800ms. User's reservation TTL is 10 minutes. Normally fine. But what if:
- Stripe is slow (30 seconds)
- User opens cart, waits 9 minutes, then checks out

The reservation may expire while payment is processing.

**Solutions:**

```
Option A: Extend TTL when payment starts
  - When checkout_saga.execute() begins, extend reservation expiry by +5 minutes
  - Ensure: if payment finishes in 3 minutes, reservation is still valid

Option B: Longer initial TTL (15-30 minutes)
  - Simple, but holds stock longer
  - Use for normal items; shorter TTL for flash sale items (1-2 minutes)

Option C: Accept the race, handle in compensation
  - After payment succeeds, confirm reservation
  - If reservation is already expired, try to re-reserve
  - If stock gone, refund payment and cancel order
  - Complex, but handles rare edge case cleanly

Recommended: Option A for normal checkout, Option C as fallback for edge cases
```

### 5. Payment Circuit Breaker

Payment providers (Stripe, Braintree) have occasional outages. Without protection:

```
Stripe is slow (5s response time)
-> 833 checkouts/sec * 5s per checkout = 4,150 concurrent checkout threads stuck
-> Thread pool exhausted
-> All other APIs starved
-> Site appears down even though Stripe is "only" slow
```

```python
class PaymentService:
    def __init__(self):
        self.circuit = CircuitBreaker(
            failure_threshold=10,    # 10 failures -> open
            recovery_timeout=30,     # try again after 30 seconds
            success_threshold=3      # 3 successes -> close
        )

    def charge(self, amount_cents: int, payment_method: str, **kwargs):
        if self.circuit.state == 'open':
            # Try fallback provider (Braintree) before failing fast
            return self.fallback_provider.charge(amount_cents, payment_method)

        try:
            with self.circuit:
                return stripe.charge(amount_cents, payment_method, timeout=3.0)
        except stripe.Timeout:
            self.circuit.record_failure()
            raise PaymentProviderUnavailable()
```

## Trade-offs and Failure Modes

### Trade-off: Eager vs. lazy inventory reservation

| Eager (reserve on checkout start) | Lazy (reserve on payment confirm) |
|---|---|
| Prevents overselling | Risk of overselling |
| Holds stock during payment window | Stock available longer |
| Requires TTL management | Simpler implementation |
| Best for scarce items | OK for abundant inventory |

**Recommendation**: Eager reservation. The TTL management complexity is worth it to prevent overselling.

### Failure Mode: Payment succeeds but confirmation fails

```
1. Payment charged -> SUCCESS (Stripe returns payment_id)
2. Checkout service crashes before calling inventory.confirm()
3. Inventory reservation expires 10 minutes later
4. Stock appears to go back up
5. Another customer buys the same unit
6. You ship to customer 1 (paid) and need to refund customer 2
```

**Solution**: The Outbox pattern. Write payment result + confirmation intent to the DB atomically before any external calls. A background process reads the outbox and sends the confirmation:

```sql
-- Outbox table
CREATE TABLE checkout_outbox (
  id           UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  event_type   TEXT NOT NULL,  -- 'confirm_reservation', 'release_reservation'
  payload      JSONB NOT NULL,
  processed_at TIMESTAMPTZ,
  created_at   TIMESTAMPTZ DEFAULT NOW()
);

-- In the checkout saga, after payment succeeds:
BEGIN;
  INSERT INTO orders (...) VALUES (...);
  INSERT INTO checkout_outbox (event_type, payload)
    VALUES ('confirm_reservation', '{"reservation_ids": [...]}');
COMMIT;
-- If we crash after this commit, the outbox processor will still confirm
```

## Senior Engineer Perspective

**Three questions that reveal depth:**

1. **What happens if Inventory Service is down during checkout?**
   - Option A: Fail fast. Return 503. User retries. Clean, but frustrating.
   - Option B: Circuit breaker + allow checkout with "soft hold" in cache. Risk slight overselling on rare items.
   - The right answer is business-specific. A $10 t-shirt allows option B. A $1,000 limited sneaker requires option A.

2. **How do you handle a flash sale without the Redis approach?**
   - Pre-generate a fixed pool of "purchase tokens" as rows in a DB table
   - Each checkout claims one token with `SELECT FOR UPDATE SKIP LOCKED`
   - 500 tokens = 500 possible sales. Token exhaustion = sold out.
   - This is the queue-based approach used by ticketing systems (Ticketmaster uses a virtual queue).

3. **Price changes mid-checkout?**
   - Orders snapshot price at checkout time (the `items JSONB` column)
   - Orders are immutable records of historical fact
   - Never live-link order items to the product catalog
   - This also matters for returns/refunds: you need the original purchase price

**The mental model for checkout design:**

> "Checkout is a negotiation between two scarce resources: inventory and payment authorization. Both can be revoked independently. Your job is to coordinate them without a distributed transaction, using Saga + compensation + idempotency."
