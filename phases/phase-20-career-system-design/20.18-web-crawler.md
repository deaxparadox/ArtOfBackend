## The Key Architectural Insight

Building a crawler that *fetches pages* is easy. Building one that fetches **billions of pages responsibly** is the real problem. The two hard constraints are:

1. **Politeness** — you cannot hammer a single domain with thousands of requests per second. Servers will ban you, you will violate robots.txt, and you will be a bad internet citizen.
2. **Deduplication** — the web is full of duplicate content, query-string variants, and redirect loops. Without aggressive dedup, you waste storage, bandwidth, and CPU re-crawling the same content forever.

Everything else — the fetching, parsing, and storage — is comparatively straightforward.

## Section 1: Requirements Clarification

Ask these in an interview before drawing anything:

| Question | Why it matters |
|---|---|
| What is the scope? (entire web vs. a domain vs. news sites) | Determines scale dramatically — full web = billions of URLs |
| How fresh must the data be? | Drives recrawl frequency |
| Are we storing raw HTML, extracted text, or structured data? | Determines storage tier |
| Do we need JavaScript rendering? | A JS-heavy crawl needs headless browsers — 10x more expensive |
| Should we respect robots.txt? | Almost always yes; affects design of the fetcher |
| What's the target throughput? (pages/sec) | Drives worker count and network design |

**Scope for this design:** General-purpose web crawler, respecting robots.txt, targeting ~1 billion pages, storing raw HTML + extracted links.

## Section 2: Capacity Estimation

| Parameter | Estimate |
|---|---|
| Pages to crawl | 1 billion |
| Average HTML page size | 100 KB |
| Crawl target | complete in 4 weeks |
| Throughput needed | 1B / (4 * 7 * 86400) ≈ **~400 pages/sec** |
| Total storage (HTML) | 1B * 100 KB = **100 TB** |
| URL storage (500 bytes/URL) | 1B * 500 bytes = **500 GB** |
| DNS lookups | 1 billion unique domains? No — web is long-tail, ~100M unique domains |

400 pages/second with ~100 ms average fetch time means you need **~40 concurrent fetcher workers minimum**, assuming network is the bottleneck.

## Section 4: Core Components

### 4.1 URL Frontier

The URL Frontier is the heart of the crawler. It manages *what to crawl next* and *when*.

Two-tier design:
- **Front queues (priority queues):** URLs sorted by priority score (like PageRank, freshness, domain importance). High-priority URLs get crawled sooner.
- **Back queues (per-domain queues):** Each domain gets its own FIFO queue. A scheduler maps front queues to back queues, enforcing one-at-a-time access per domain.

```
Front Queues                Back Queues
(by priority)               (by domain)

[Priority 1] ----\          [queue: nytimes.com]  --> Fetcher A
[Priority 2] -----+-------> [queue: reddit.com]   --> Fetcher B
[Priority 3] ----/          [queue: bbc.co.uk]    --> Fetcher C
                            [queue: github.com]   --> Fetcher D
```

**Politeness contract:** A fetcher picks up a back queue (domain), fetches one page, then waits `crawl-delay` seconds before picking up that queue again. This is enforced by a heap keyed on `(next_allowed_fetch_time, domain)`.

### 4.2 Fetcher Workers

Each fetcher worker:
1. Picks the next domain from the heap (earliest `next_allowed_time`)
2. Checks robots.txt for the URL path
3. Issues HTTP GET with a proper `User-Agent: Googlebot` (or your bot name)
4. Handles redirects (follow up to N hops, detect redirect cycles)
5. Handles status codes: 200 → parse, 301/302 → follow, 429 → back-off, 404 → discard, 5xx → retry with exponential back-off
6. Returns raw HTML + metadata to the parser

**DNS caching:** Resolve each domain once, cache with TTL. Without this, DNS becomes a bottleneck at scale. Use a local DNS cache per worker cluster.

### 4.3 HTML Parser

- Extracts all `<a href>` links from the page
- Normalizes URLs (see deduplication below)
- Extracts metadata: title, description, language, last-modified headers
- Sends extracted links to the Dedup Filter
- Sends raw HTML + metadata to Content Storage

### 4.4 Deduplicator

Two levels of deduplication:

**URL-level dedup (have we visited this URL?):**
- Bloom filter: probabilistic, allows false positives (miss re-crawling a page) but never false negatives (never miss a new page) — acceptable trade-off
- At 1 billion URLs, a Bloom filter with 10 bits per element = **10 GB in memory** — fits on one machine
- Backed by a persistent store (Redis or RocksDB) for crash recovery

**Content-level dedup (is this page a near-duplicate?):**
- SimHash: generates a 64-bit fingerprint where similar pages have fingerprints with few bit differences (Hamming distance)
- Store all fingerprints; for a new page, check if any stored fingerprint has Hamming distance < threshold (e.g., < 3 bits)
- Catches mirror sites, paginated views of the same content, print-friendly vs. normal versions

### 4.5 Content Storage

- Raw HTML: object storage (S3, GCS, HDFS) — cheap, append-friendly
- Extracted metadata: key-value store or document DB (Cassandra, Elasticsearch)
- URL graph (what links to what): graph database or edge list in column store

## Section 5: The Hard Problem — Politeness + Deduplication

### Politeness in Depth

**robots.txt:** Every domain publishes rules like:
```
User-agent: *
Disallow: /private/
Crawl-delay: 10
```

Your crawler must:
1. Fetch and parse robots.txt before crawling any path on a domain
2. Cache robots.txt per domain (re-fetch after TTL, e.g., 24 hours)
3. Respect `Crawl-delay` — many sites set this to 10-30 seconds
4. Honor `Disallow` rules — pattern matching on URL paths

**Per-domain rate limiting:** Even without `Crawl-delay`, enforce your own limits (e.g., 1 request/second per domain maximum). Use a token bucket per domain.

**Back-off on 429 (Too Many Requests):** Exponential back-off: wait 1s → 2s → 4s → 8s → give up and re-queue the domain for later. Check `Retry-After` header if present.

**IP rotation caution:** Some crawlers rotate IPs to evade rate limits. This is against the spirit of robots.txt and terms of service for most sites. A legitimate crawler identifies itself and respects limits.

### URL Normalization (foundation of dedup)

Before inserting any URL into the frontier or the Bloom filter, normalize it:
1. Lowercase the scheme and host: `HTTP://EXAMPLE.COM` → `http://example.com`
2. Remove default ports: `http://example.com:80/` → `http://example.com/`
3. Remove fragments: `http://example.com/page#section` → `http://example.com/page`
4. Sort query parameters: `?b=2&a=1` → `?a=1&b=2`
5. Resolve relative URLs: `../about` → absolute form
6. Remove tracking parameters: `utm_source`, `fbclid`, `gclid` — these create millions of duplicate URLs

### Bloom Filter Details

A Bloom filter answers "have I seen this URL before?" in O(1) time and O(1) space per element.

Trade-off: ~1% false positive rate at 10 bits/element. This means ~1% of new URLs are mistakenly skipped. For a web crawler, that is completely acceptable.

```
Insert URL:    hash(url) → k bit positions → set bits to 1
Query URL:     hash(url) → k bit positions → if ALL bits are 1, "probably seen"
                                            → if ANY bit is 0, "definitely not seen"
```

At 1 billion URLs with 10 bits per URL: 10 billion bits = **1.25 GB** — trivial.

### SimHash for Near-Duplicate Content

```python
def simhash(text: str) -> int:
    v = [0] * 64
    for token in tokenize(text):
        h = hash(token)  # 64-bit hash
        for i in range(64):
            if h & (1 << i):
                v[i] += 1
            else:
                v[i] -= 1
    return sum(1 << i for i in range(64) if v[i] > 0)

def hamming_distance(a: int, b: int) -> int:
    return bin(a ^ b).count('1')
```

Two pages with hamming distance < 3 are near-duplicates. Keep the canonical version (higher PageRank or earlier crawl date).

## Section 6: Scale and Trade-offs

### Distributed Crawl with Consistent Hashing

To scale beyond one machine:
- Use **consistent hashing** to assign domains to fetcher workers: `hash(domain) % N` assigns `example.com` to worker 3 always
- This ensures one worker owns all politeness state for a domain
- Avoids two workers both hitting `example.com` simultaneously

### Recrawl Scheduling

Not all pages need equal freshness:
- News sites: recrawl every hour
- Wikipedia: recrawl weekly
- Static company pages: recrawl monthly

Track: `last_crawled`, `last_modified`, `change_frequency` (from sitemap.xml). Use a priority score that increases as pages age past their expected change frequency.

### Real-World Examples

| System | Key design choice |
|---|---|
| **Googlebot** | Dedicated crawler infrastructure, respects robots.txt, has a massive URL frontier using Bigtable |
| **Common Crawl** | Open dataset, crawls ~3 billion pages/month, stores raw WARC files in S3 |
| **Scrapy** | Open-source framework; uses `DOWNLOAD_DELAY` and `AUTOTHROTTLE` for politeness |

## Senior Engineer Perspective

In an interview, most candidates describe the fetching and storage. The ones who stand out go deep on:

1. **The back-queue design** — "how do you ensure you never send two requests to the same domain at the same time across 100 workers?" This requires consistent hashing + per-domain locking.

2. **robots.txt caching invalidation** — if you cache robots.txt for 24 hours but a site updates it, you might violate their rules for a day. What's the right TTL? How do you handle per-path rules efficiently?

3. **The URL trap problem** — dynamic sites generate infinite URLs (calendars, infinite scroll, session tokens in URLs). You need heuristics to detect and avoid these traps: max URL depth, max query parameters, cycle detection.

4. **Politeness is a product decision** — a search engine that respects crawl delays builds trust and avoids lawsuits. A scraper that ignores them gets IP-banned and may face legal action under the CFAA (Computer Fraud and Abuse Act in the US).

> **The interview signal:** Candidates who say "politeness just means checking robots.txt" miss the depth. Politeness is a continuous concern: rate limits, back-off, robots.txt TTL, respecting Crawl-delay, not crawling during peak traffic hours for small sites. It is a first-class system requirement, not an afterthought.
