## Why This System Is Hard

Uber is deceptively complex. On the surface it looks like "match a rider to a driver". The real challenge is:

1. **Real-time location at massive scale** — 1 million active drivers each pushing GPS updates every 5 seconds = 200,000 writes/second continuously
2. **Geospatial proximity queries** — "Find me all available drivers within 5km" on live data, sub-second latency
3. **Matching under contention** — Multiple riders may request the same nearest driver simultaneously
4. **State synchronization** — Driver state changes (available → on_trip) must be globally consistent before another rider claims the same driver
5. **Live tracking** — Rider watches driver move on the map in real-time, requiring a persistent push channel

No single traditional database handles all of this well. The solution is a set of purpose-built subsystems.

---

## Requirements Clarification

**Functional requirements:**
- Rider can request a ride from current location to a destination
- System matches rider to nearest available driver
- Rider sees driver location on map in real-time until pickup
- Driver can set status: available or offline
- Rider can see estimated time of arrival (ETA)
- Trip is recorded with fare at end

**Out of scope (for this design):**
- Fare calculation algorithm
- Surge pricing
- Driver payments/payouts
- Ratings system

**Non-functional requirements:**
- Ride match latency: < 2 seconds
- Driver location update lag on rider's screen: < 3 seconds
- 99.99% uptime (missing a match = lost revenue)
- Global deployment (but design one region first)

## Capacity Estimation

```
Active drivers:     1,000,000
Location update:    every 5 seconds
Write throughput:   1,000,000 / 5 = 200,000 writes/second

Location payload:   { driver_id, lat, lng, timestamp } = ~50 bytes
Write bandwidth:    200,000 × 50 bytes = 10 MB/second

Active riders:      500,000 (riders = ~50% of drivers at peak)
Ride requests:      ~50,000/minute = ~833/second

Live tracking:      500,000 riders × 1 WebSocket connection
                    = 500,000 persistent connections

Storage (location history, 30 days):
  200,000 writes/sec × 86,400 sec/day × 30 days × 50 bytes
  = ~26 TB raw (compress to ~6 TB)

Trip records:       ~5M trips/day × 500 bytes = 2.5 GB/day
```

## Data Model

### Users table (PostgreSQL)
```sql
CREATE TABLE users (
  id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  email       TEXT UNIQUE NOT NULL,
  phone       TEXT UNIQUE NOT NULL,
  role        TEXT NOT NULL CHECK (role IN ('rider', 'driver')),
  created_at  TIMESTAMPTZ DEFAULT NOW()
);
```

### Driver state (Redis hash + sorted set)
```
HASH  driver:{id}
  → status: "available" | "on_trip" | "offline"
  → current_trip_id: UUID or null
  → last_seen: Unix timestamp

GEO   driver_locations      ← Redis GEOADD/GEORADIUS
  → driver_id → (lng, lat)
  
Only available drivers are in the GEO index.
When driver goes on_trip, remove from GEO index.
```

### Trips table (PostgreSQL)
```sql
CREATE TABLE trips (
  id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  rider_id        UUID REFERENCES users(id),
  driver_id       UUID REFERENCES users(id),
  status          TEXT NOT NULL CHECK (status IN (
                    'requested', 'matched', 'driver_arriving',
                    'in_progress', 'completed', 'cancelled'
                  )),
  pickup_lat      DOUBLE PRECISION NOT NULL,
  pickup_lng      DOUBLE PRECISION NOT NULL,
  dropoff_lat     DOUBLE PRECISION,
  dropoff_lng     DOUBLE PRECISION,
  requested_at    TIMESTAMPTZ DEFAULT NOW(),
  matched_at      TIMESTAMPTZ,
  started_at      TIMESTAMPTZ,
  completed_at    TIMESTAMPTZ,
  fare_cents      INTEGER
);

CREATE INDEX trips_rider_id_idx   ON trips(rider_id);
CREATE INDEX trips_driver_id_idx  ON trips(driver_id);
CREATE INDEX trips_status_idx     ON trips(status);
```

### Location history (Cassandra — write-optimized)
```
TABLE driver_location_history
  driver_id   UUID
  ts          TIMESTAMP
  lat         DOUBLE
  lng         DOUBLE
  PRIMARY KEY ((driver_id), ts)
  WITH CLUSTERING ORDER BY (ts DESC)
  AND default_time_to_live = 2592000   -- 30 days TTL
```

## Deep Dive: The Hard Problems

### 1. Real-Time Location at 200K Writes/Second

**Naive approach**: Store every GPS update in PostgreSQL.
**Problem**: 200K writes/second will saturate any single relational DB.

**Solution: Two-tier storage**

```
Driver App
    │
    │  POST /location  {lat, lng}
    │  every 5 seconds
    ▼
Location Service
    │
    ├──► Redis GEOADD driver_locations <lng> <lat> <driver_id>
    │        ↑ Hot data: current position (O(log N) per update)
    │        ↑ Supports GEORADIUS queries
    │
    └──► Kafka topic: driver-locations
             │
             ▼
         Cassandra consumer (async, batched)
             ↑ Cold data: historical path for replay/analysis
```

Redis `GEOADD` uses a sorted set with geohash score — O(log N) for insert and radius queries.

For 1M drivers:
- Redis memory: 1M × ~50 bytes = 50 MB (trivial)
- GEORADIUS query: typically < 1 ms
- Redis cluster with 3 shards handles 200K ops/sec easily

### 2. Geospatial Proximity: Find Drivers Within 5km

**Geohashing** — divide the world into a grid of cells, each cell has a string ID:

```
World grid (simplified):
┌───────┬───────┬───────┐
│ 9q8y  │ 9q8z  │ 9q9p  │
│       │       │       │
├───────┼───────┼───────┤
│ 9q8v  │ 9q8w  │ 9q8x  │  ← Rider is in 9q8w
│       │  [R]  │       │
├───────┼───────┼───────┤
│ 9q8t  │ 9q8u  │ 9q8v  │
│       │       │       │
└───────┴───────┴───────┘

To find nearby drivers:
1. Hash rider's lat/lng → geohash "9q8w"
2. Get 8 neighboring cells: 9q8v, 9q8x, 9q8y, 9q8z, 9q8t, 9q8u, 9q9p...
3. Look up all available drivers in those cells
4. Filter by exact Haversine distance
```

Redis `GEORADIUS` does this natively:

```python
# Find all available drivers within 5km
drivers = redis.georadius(
    "driver_locations",
    lng=rider_lng,
    lat=rider_lat,
    radius=5,
    unit="km",
    withcoord=True,
    withdist=True,
    sort="ASC",
    count=20  # top 20 nearest
)
```

### 3. Matching Service: Claim a Driver Atomically

Problem: Rider A and Rider B both see Driver X as nearest. Both try to claim Driver X at the same time.

**Solution: Optimistic locking with Redis atomic operations**

```python
def match_driver_to_rider(rider_id: str, rider_lat: float, rider_lng: float):
    candidates = redis.georadius(
        "driver_locations",
        lng=rider_lng, lat=rider_lat,
        radius=5, unit="km", sort="ASC", count=10
    )

    for driver_id, distance in candidates:
        # Atomic compare-and-set
        # SET driver:{id}:status "on_trip" NX  ← only if not already set
        claimed = redis.set(
            f"driver:{driver_id}:lock",
            rider_id,
            nx=True,        # Only set if not exists
            ex=30           # Auto-expire in 30s if something goes wrong
        )
        if claimed:
            # We won the race. Commit the match.
            create_trip(rider_id, driver_id)
            remove_driver_from_geo_index(driver_id)
            notify_driver(driver_id, rider_id)
            return driver_id

    return None  # No driver available — retry after backoff
```

The `NX` flag is the key: Redis guarantees atomicity, so only one caller can set the lock.

### 4. Driver State Machine

```
                    Driver opens app
                          │
                          ▼
                      ┌──────────┐
                ┌────►│ OFFLINE  │◄────┐
                │     └──────────┘     │
           Goes │          │           │ Driver
          offline│     Goes │ online   │ logs out
                │          ▼           │
                │     ┌──────────┐     │
                │     │AVAILABLE │─────┘
                │     └──────────┘
                │          │ ▲
          Trip  │   Match  │ │ Trip ends /
         cancelled│  found  │ │ rider cancels
                │          ▼ │
                │     ┌──────────┐
                └─────│ ON_TRIP  │
                      └──────────┘

State is stored in Redis:
  HSET driver:{id} status "available"
  HSET driver:{id} current_trip_id ""

Transitions are atomic Redis + PostgreSQL writes (dual-write on state change).
```

### 5. Live Location Updates to Rider: WebSocket vs SSE

Once matched, the rider needs to see the driver moving on the map.

```
Driver App                    Server                    Rider App
     │                          │                           │
     │ POST /location           │                           │
     │ {lat:37.1, lng:-122.4}  │                           │
     │─────────────────────────►│                           │
     │                          │                           │
     │                          │  Publish to Pub/Sub       │
     │                          │  channel: trip:{trip_id}  │
     │                          │                           │
     │                          │◄──────────────────────────│
     │                          │  WebSocket connection     │
     │                          │  (persistent, open)       │
     │                          │                           │
     │                          │  Push location event ─────►
     │                          │  {lat, lng, eta}          │
```

**WebSocket** is preferred over polling because:
- Polling at 1-second intervals × 500K active trips = 500K HTTP requests/second just for tracking
- WebSocket: server pushes on each driver update → far fewer server-initiated events

**Pub/Sub architecture for WebSocket at scale:**

```
Multiple WebSocket servers (stateless):

  WS Server 1  WS Server 2  WS Server 3
      │             │             │
      └──────┬──────┘             │
             │  Redis Pub/Sub     │
             │  channel:          │
             │  trip:{id}         │
             └────────────────────┘

When driver location arrives:
  1. Location Service publishes to Redis channel trip:{trip_id}
  2. All WS servers subscribed to that channel receive it
  3. The WS server that has the rider's connection pushes it out

Result: WS servers are stateless. Any server can serve any rider.
```

## Trade-offs and Failure Modes

### Trade-off 1: Redis for location vs. PostGIS

| Redis GEORADIUS | PostGIS |
|---|---|
| In-memory, microsecond reads | Disk-backed, millisecond reads |
| No complex queries | Full SQL + spatial joins |
| Limited history | Full history queryable |
| Easier to scale writes | Needs sharding/partitioning |

**Decision**: Redis for live matching (speed critical), PostGIS/Cassandra for analytics and history.

### Trade-off 2: Matching granularity

Fine geohash cells → more accurate but more cells to check per query.
Coarse cells → faster lookup but may include drivers 20km away.

**Uber's approach**: Dynamic geohash precision based on driver density. Dense cities use finer cells.

### Failure Mode 1: Location Service goes down

Drivers stop being locatable. Fallback: Match service reads from Cassandra's last-known location (stale by seconds, not minutes).

### Failure Mode 2: Driver accepts but app crashes before starting trip

Trip stays in `matched` state. Solution: Timeout. If driver doesn't confirm within 30 seconds, auto-cancel and re-match. The 30-second lock key in Redis expires automatically.

### Failure Mode 3: Duplicate ride requests

Rider taps "Request" twice (slow network, double-tap). Solution: Idempotency key = (rider_id + timestamp rounded to 5 seconds). Same key → same trip, don't create duplicate.

## Senior Engineer Perspective

**What interviewers want to see:**

1. You immediately identified the scale problem: 200K writes/second is not a PostgreSQL problem, it's a Redis problem.

2. You know geohashing is how Uber actually does it (confirmed in their engineering blog). Not a naive SQL `WHERE dist < 5` query.

3. You handle the race condition. Two riders → one driver is the classic distributed systems problem. You solved it with Redis NX atomic CAS.

4. You know WebSockets need a pub/sub backing (Redis or Kafka) because WS connections are stateful but servers should be stateless.

5. You separated the data by access pattern: Redis (hot live), Cassandra (warm history), PostgreSQL (cold trips/users).

**What separates senior answers:**

- Mention that Redis GEO index only contains **available** drivers. When a driver goes on_trip, remove them from the index immediately. Do not filter — filtering at query time is wasteful at 1M drivers.
- Driver heartbeat: if a driver stops sending GPS updates for 60s, mark them offline automatically. Prevents "ghost drivers" in the index.
- ETA calculation happens separately (call a maps/routing service like Google Maps API or an internal OSRM instance — not your job in this interview).

**Real Uber architecture (public knowledge):**
- They use a custom service called "Supply Service" backed by their distributed datastore "Ringpop" (now Cadence/Temporal for orchestration)
- They wrote a custom geospatial library "H3" (hexagonal hierarchical geospatial indexing) — open sourced in 2018
- H3 is superior to traditional geohash: hexagons have more uniform neighbor distances than rectangles
