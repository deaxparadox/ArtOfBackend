## System Design Interview Framework (20.1)

```
THE FRAMEWORK (use this for every system design question):

STEP 1: CLARIFY REQUIREMENTS (5 min)
  Functional: What does the system do?
    - Core features only (not every edge case)
    - "What are the top 3 things users must be able to do?"
  
  Non-functional: How should it do it?
    - Scale: how many users, how many requests/second?
    - Availability: 99.9%? 99.99%?
    - Consistency requirements: can we accept eventual consistency?
    - Latency: p99 < 100ms? < 1s?
    - Data size: how many records? How much storage?

STEP 2: CAPACITY ESTIMATION (5 min)
  Back-of-envelope math. Show your reasoning.
  10M users × 1 request/day = 115 req/s
  × 1KB average response = 115 KB/s → ~100 GB/day storage

STEP 3: HIGH-LEVEL DESIGN (10 min)
  Draw boxes: client, CDN, load balancer, app servers, DB, cache
  Identify the core components
  Show data flow for the main use case

STEP 4: DEEP DIVE (15-20 min)
  Pick the most interesting/complex component
  Explain the tradeoffs
  Address scaling, failure modes

STEP 5: SCALABILITY & EDGE CASES (5 min)
  Where are the bottlenecks?
  How would you handle 10x traffic?
  What happens when X fails?
```

## Capacity Estimation (20.2): The Math Every Engineer Needs

```
NUMBERS TO MEMORIZE:

TIME:
  86,400 seconds/day
  3 million seconds/month
  
STORAGE:
  1 char = 1 byte
  1 UUID = 16 bytes
  1 photo (compressed) = 100KB-1MB
  1 tweet = ~280 bytes
  
THROUGHPUT:
  1 server, simple API: ~10,000 req/s
  1 PostgreSQL (optimized): ~10,000 queries/s
  1 Redis: ~100,000 ops/s

CALCULATION EXAMPLE: Design Twitter

  Active users: 100M daily
  Tweets per user per day: 0.1 (most users don't tweet daily)
  Reads per user per day: 50 (reading feed)
  
  Write QPS: 100M × 0.1 / 86400 = ~115 writes/s
  Read QPS: 100M × 50 / 86400 = ~58,000 reads/s  (500:1 read:write ratio)
  
  Storage:
  Tweet size: 140 chars + metadata = ~500 bytes
  Tweets per day: 10M
  Storage per day: 10M × 500 bytes = 5GB/day
  5 years: 5GB × 365 × 5 = ~9TB
  
  Conclusion:
  - Read-heavy system → cache aggressively
  - Write QPS is manageable, read QPS needs sharding
  - Storage is manageable with compression
```

## Design: URL Shortener (20.3)

```
REQUIREMENTS:
  Functional:
    - Given a long URL, return a short URL
    - Given a short URL, redirect to the long URL
    - Custom aliases (optional)
  
  Non-functional:
    - 100M URLs stored
    - 1000 writes/s, 10,000 reads/s
    - Low latency for redirects (< 10ms)
    - High availability (99.99%)

CORE ALGORITHM: SHORT CODE GENERATION
  Option 1: Random ID
    base62 (a-z, A-Z, 0-9): 62^7 = 3.5 trillion unique 7-char codes
    Generate randomly, check for collision in DB
  
  Option 2: Counter (global atomic counter)
    Convert counter to base62 string
    Pro: no collisions by design
    Con: revealing patterns, needs distributed counter (Redis INCR)
  
  Option 3: MD5 hash of long URL, take first 7 chars
    Deterministic (same URL → same short code, no duplicates)
    Collision probability: extremely low

STORAGE:
  Table: short_urls(code VARCHAR(7) PK, long_url TEXT, created_at, user_id, click_count)
  Cache: Redis "short:abc123" → "https://..." TTL=24h (cache redirect lookups)
  
  Hot path for redirects: Redis → DB (99% hit rate means DB rarely needed)

REDIRECT:
  GET /{code}
  1. Check Redis cache
  2. If miss: query DB, populate cache
  3. Return 301 (permanent redirect, browser caches) or 302 (temporary, track clicks)
  
  301 vs 302: use 302 if you want to track every click; 301 if you want browser caching.

SCALING:
  App servers: stateless, horizontally scalable
  DB: single-region read replica for read scaling
  Cache: Redis cluster
  Global: multi-region with geo-DNS for low latency worldwide
```

## Design: Rate Limiter (20.4)

```
ALGORITHMS (see topic 2.17 for implementation details):
  Fixed window, sliding window, token bucket

DISTRIBUTED RATE LIMITING:
  Challenge: if 10 app servers, rate limit must be shared across all
  
  Solution: Redis as shared counter
  app-server-1: INCR "rate:user:123:minute:14:30" → 47
  app-server-2: INCR "rate:user:123:minute:14:30" → 48
  Both see the same count via Redis
  
  Atomic: use Redis INCR (atomic increment), not GET + SET
  TTL: Redis automatically expires old windows

HEADERS:
  X-RateLimit-Limit: 1000
  X-RateLimit-Remaining: 953
  X-RateLimit-Reset: 1718450260  (Unix timestamp)
  Retry-After: 60  (on 429 response)

TIERS:
  Anonymous: 60/hour (IP-based)
  Free tier: 1000/hour (user ID-based)
  Paid tier: 10000/hour
  Enterprise: custom
```

## Design: Notification System (20.5)

```
REQUIREMENTS:
  Send notifications via: email, SMS, push (mobile), in-app

ARCHITECTURE:
  Notification Service (REST API)
        ↓
  Message Queue (Kafka/SQS)
  ├── Email Queue → Email Workers → SendGrid
  ├── SMS Queue → SMS Workers → Twilio
  ├── Push Queue → Push Workers → FCM/APNs
  └── In-App Queue → In-App Workers → WebSocket/SSE

WHY QUEUES?
  - Decouple: ordering service doesn't know about email/SMS
  - Retry: failed notifications retried by queue
  - Rate limiting: SMS/email providers have rate limits; queue absorbs bursts
  - Scale: add more workers independently per channel

DEDUPLICATION:
  User may trigger same notification twice (retry, bug)
  Store notification_id in DB before sending
  INSERT ... ON CONFLICT DO NOTHING (idempotent)
  
PRIORITY:
  OTP codes (login 2FA): high priority, immediate
  Order updates: medium priority
  Marketing: low priority, batched
  
FAILURE HANDLING:
  Email fails → retry 3 times → dead letter queue → alert
  User opted out → check preferences before sending
  User's device token expired → remove from FCM registry
```

## Design: Chat System (20.7)

```
REQUIREMENTS:
  1-on-1 and group chat
  Message delivery: at-least-once (no lost messages)
  Read receipts
  Online presence
  Message history

REAL-TIME MESSAGING:
  WebSockets for live messages
  Each user: persistent WebSocket to chat server
  Message flow:
    Alice sends → Chat Server A (Alice's server)
    Chat Server A → Message Queue (Kafka)
    Chat Server B (Bob's server) consumes → Bob's WebSocket
  
  Multi-server WebSockets need shared state → Redis Pub/Sub or Kafka
  When Bob is offline: store in inbox DB, deliver on reconnect

STORAGE:
  Messages: {id, room_id, sender_id, content, timestamp, type}
  Cassandra (time-series, high write, partition by room_id + timestamp)
  OR PostgreSQL with partitioning by month
  
  Message IDs: Snowflake IDs (time-ordered, globally unique)
  
PRESENCE:
  Redis: SET "user:123:online" 1 EX 60 (1-minute TTL, heartbeat refreshes)
  Or: "last_seen_at" timestamp updated on every activity

READ RECEIPTS:
  Store per-user-per-message-delivery-status
  Scale challenge: group of 1000 members × 1M messages = 1B rows
  Optimization: store only "last_read_message_id" per user per room

HISTORY:
  Hot: last 30 days in Cassandra/PostgreSQL
  Cold: older messages in S3 (compressed JSON, searchable via Elasticsearch)
```

## Design: YouTube / Video Streaming (20.10)

```
VIDEO UPLOAD PIPELINE:
  User uploads raw video → S3 (raw bucket)
  Message Queue → Video Processing Workers
  Workers: transcode to multiple resolutions (360p, 720p, 1080p, 4K)
  Workers: extract thumbnail frames
  Workers: generate subtitles (ML)
  Output stored in S3 (processed bucket)
  CDN configured to serve from S3

TRANSCODING:
  Raw video → FFmpeg → multiple resolutions + formats (HLS segments)
  HLS: HTTP Live Streaming (Apple's format, widely supported)
  Segments: 6-10 second chunks
  Client: downloads segments as needed (adaptive bitrate)
  
STORAGE:
  Video metadata: PostgreSQL (title, description, user_id, views)
  Raw + processed videos: S3 (object storage, cheap, durable)
  Comments: PostgreSQL (write-heavy for popular videos → sharding)
  View counts: Redis counter, batch-flush to PostgreSQL hourly

CDN STRATEGY:
  Popular videos (top 20%): cached at edge nodes globally
  Long-tail (80%): cache only in regional nodes
  Video starts at ~5-10MB/min → CDN absorbs most traffic

RECOMMENDATIONS:
  ML model: collaborative filtering + content-based
  Precomputed: recommendation job runs hourly, stores in Redis
  Real-time signals: watch history, likes stored in Kafka stream
```

## The Senior/Principal Engineer Mindset (20.20)

```
WHAT THEY THINK ABOUT IN DESIGN:

1. FAILURE MODES FIRST
   "What happens when this component fails?"
   "What is the blast radius?"
   "Can we degrade gracefully instead of failing completely?"

2. OPERATIONAL SIMPLICITY
   "Will on-call understand this at 3am?"
   "How do we debug this in production?"
   "What does the runbook look like?"

3. DATA CORRECTNESS
   "Can this data be corrupted or lost?"
   "How do we detect and recover from corruption?"
   "What are the consistency requirements?"

4. SCALE ASSUMPTIONS
   "At 10x current load, where does this break?"
   "Which components need to be replaced first?"
   "What's the cost curve as we scale?"

5. EVOLUTION
   "Can we change this later without a painful migration?"
   "What decisions are we locking ourselves into?"
   "What will this look like in 2 years?"

THE INTERVIEW SIGNAL THAT GETS SENIOR ROLES:
  Not: knowing the perfect architecture
  But: demonstrating that you understand trade-offs deeply
  
  "We could use Kafka here, which would give us X benefit,
   but the operational overhead is Y. Given our team size,
   SQS is a better fit now. If we hit Z scale, we'd migrate to Kafka."
  
  This shows: you know multiple options, you understand the trade-offs,
  you're pragmatic about what the team can sustain.
  That's what senior engineers actually do.
```
