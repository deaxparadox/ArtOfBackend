## Why This System Is Hard

Payments are not just API calls to a bank. The hard problems are:

1. **Exactly-once money movement** — Networks time out. Servers crash. Users click "Pay" twice. You must guarantee each payment executes exactly once, even when requests retry. Double charging is a legal violation, not just a bug.
2. **Immutable audit trail** — Every state change must be permanently recorded. Regulators can demand 7 years of transaction history. Records can never be deleted or retroactively altered.
3. **PCI-DSS compliance** — You are not allowed to store raw card numbers in your main database. A breach that exposes card data is a $5-50K fine per card, plus potential loss of ability to process payments.
4. **Distributed transaction across two systems** — Your internal ledger and the external bank/PSP must stay in sync. What happens when you charge the card successfully but then crash before writing to your DB?
5. **The ledger must always balance** — Total debits must equal total credits at all times. A rounding error or lost write is a financial discrepancy. These must be detected within hours, not months.

## Requirements Clarification

**Functional requirements:**
- Accept card payments (one-time charges and refunds)
- Maintain an internal ledger (accounting record)
- Expose payment status to merchants and users
- Support multiple PSPs (Stripe, Braintree) with failover
- Nightly reconciliation against PSP settlement files

**Non-functional requirements:**
- Exactly-once semantics: same payment request never charges twice
- 99.999% durability of payment records
- PCI-DSS compliant (no raw card data stored)
- Full audit trail for every state transition
- 500 payments/second peak throughput

**Out of scope:**
- Card tokenization (handled by PSP's JavaScript SDK in browser)
- Fraud detection (separate ML service)
- Tax calculation
- Subscription billing

## Data Model

```sql
-- Primary payment record
CREATE TABLE payments (
  id                UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id           UUID NOT NULL,
  merchant_id       UUID NOT NULL,
  amount_cents      BIGINT NOT NULL,       -- always store as integer cents
  currency          CHAR(3) NOT NULL,      -- 'USD', 'EUR', 'GBP'
  status            TEXT NOT NULL DEFAULT 'pending'
                      CHECK (status IN (
                        'pending', 'processing', 'succeeded',
                        'failed', 'refunded', 'disputed'
                      )),
  psp_provider      TEXT,                  -- 'stripe', 'braintree'
  psp_charge_id     TEXT UNIQUE,           -- PSP's external ID (for reconciliation)
  idempotency_key   TEXT UNIQUE NOT NULL,
  failure_code      TEXT,                  -- 'card_declined', 'insufficient_funds'
  failure_message   TEXT,
  metadata          JSONB,                 -- merchant-supplied key-value pairs
  created_at        TIMESTAMPTZ DEFAULT NOW(),
  updated_at        TIMESTAMPTZ DEFAULT NOW()
) PARTITION BY RANGE (created_at);

-- Monthly partitions
CREATE TABLE payments_2024_01 PARTITION OF payments
  FOR VALUES FROM ('2024-01-01') TO ('2024-02-01');

-- Indexes on base table (inherited by partitions)
CREATE INDEX payments_user_id_idx    ON payments(user_id, created_at DESC);
CREATE INDEX payments_merchant_id_idx ON payments(merchant_id, created_at DESC);
CREATE INDEX payments_psp_charge_idx ON payments(psp_charge_id);

-- Double-entry ledger: NEVER update or delete rows
-- Every payment creates 2+ entries that sum to zero
CREATE TABLE ledger_entries (
  id            UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  payment_id    UUID NOT NULL,            -- no FK to avoid cross-partition issues
  account_id    UUID NOT NULL,            -- user, merchant, platform_fees, etc.
  account_type  TEXT NOT NULL,            -- 'user', 'merchant', 'platform'
  entry_type    TEXT NOT NULL CHECK (entry_type IN ('debit', 'credit')),
  amount_cents  BIGINT NOT NULL CHECK (amount_cents > 0),
  currency      CHAR(3) NOT NULL,
  description   TEXT NOT NULL,
  created_at    TIMESTAMPTZ DEFAULT NOW()
) PARTITION BY RANGE (created_at);

-- Example entries for a $100 payment with 3% fee:
-- DEBIT  user_account         10000  "Payment for order #1234"
-- CREDIT merchant_account      9700  "Payment received for order #1234"
-- CREDIT platform_fees_account  300  "Processing fee 3%"
-- Check: 10000 DEBIT = 9700 + 300 CREDIT = 10000 CREDIT  OK

-- Idempotency key store (with full response caching)
CREATE TABLE idempotency_keys (
  idempotency_key  TEXT PRIMARY KEY,
  payment_id       UUID NOT NULL,
  request_hash     TEXT NOT NULL,     -- SHA-256 of serialized request body
  http_status      INTEGER,           -- response HTTP status code
  response_body    JSONB,             -- full response body
  created_at       TIMESTAMPTZ DEFAULT NOW(),
  expires_at       TIMESTAMPTZ DEFAULT NOW() + INTERVAL '24 hours'
);

-- Payment audit log (immutable record of every state change)
CREATE TABLE payment_events (
  id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  payment_id  UUID NOT NULL,
  event_type  TEXT NOT NULL,    -- 'payment.created', 'payment.succeeded', etc.
  actor       TEXT NOT NULL,    -- 'user', 'system', 'webhook'
  data        JSONB NOT NULL,   -- full event payload snapshot
  created_at  TIMESTAMPTZ DEFAULT NOW()
);
```

## Deep Dive: The Hard Problems

### 1. Exactly-Once Money Movement with Idempotency Keys

**The double-charge failure scenario:**

```
t=0ms    Client sends POST /payments
t=50ms   Payment service calls Stripe
t=250ms  Stripe charges the card SUCCESSFULLY
t=255ms  Network packet with Stripe's 200 OK is lost
t=5000ms Client timeout fires, client retries
t=5050ms Payment service calls Stripe again
t=5300ms Stripe charges the card A SECOND TIME
```

Result: customer charged twice. This is not hypothetical — it happens in every payment system without idempotency.

**The fix: idempotency key stored before calling PSP**

```python
def process_payment(idempotency_key: str, request: PaymentRequest) -> Payment:
    request_hash = sha256(json.dumps(request.dict(), sort_keys=True))

    # Step 1: Check if this key was already processed
    existing = db.query(
        "SELECT payment_id, http_status, response_body FROM idempotency_keys "
        "WHERE idempotency_key = %s AND expires_at > NOW()",
        (idempotency_key,)
    )
    if existing:
        if existing.request_hash != request_hash:
            # Same key, different request body = developer error
            raise IdempotencyConflict("Key reused with different request body")
        # Return the cached response - no new charge
        return existing.response_body

    # Step 2: Lock the key (atomic INSERT)
    try:
        db.execute("""
            INSERT INTO idempotency_keys
              (idempotency_key, payment_id, request_hash)
            VALUES (%s, %s, %s)
        """, (idempotency_key, new_uuid(), request_hash))
    except UniqueViolation:
        # Race condition: another request with same key beat us
        # Fetch and return their result
        return wait_for_result(idempotency_key)

    # Step 3: Create payment record in PENDING state
    payment = db.insert_payment(
        idempotency_key=idempotency_key,
        amount_cents=request.amount_cents,
        status='pending'
    )
    emit_event(payment.id, 'payment.created', request.dict())

    # Step 4: Call PSP
    try:
        psp_response = stripe.charge(
            amount=request.amount_cents,
            currency=request.currency,
            payment_method=request.payment_method_token,
            idempotency_key=idempotency_key  # Pass our key to Stripe too
        )
        status = 'succeeded'
        psp_charge_id = psp_response.id

    except stripe.CardDeclined as e:
        status = 'failed'
        failure_code = e.code
        psp_charge_id = None

    # Step 5: Write ledger (if succeeded)
    if status == 'succeeded':
        write_ledger_entries(payment.id, request)

    # Step 6: Update payment record
    db.execute("""
        UPDATE payments
        SET status = %s, psp_charge_id = %s, updated_at = NOW()
        WHERE id = %s
    """, (status, psp_charge_id, payment.id))

    # Step 7: Cache the response in idempotency_keys table
    response = build_response(payment, status)
    db.execute("""
        UPDATE idempotency_keys
        SET http_status = %s, response_body = %s
        WHERE idempotency_key = %s
    """, (200 if status == 'succeeded' else 402, response, idempotency_key))

    emit_event(payment.id, f'payment.{status}', response)
    return response
```

**Why we pass the idempotency key to Stripe as well:**
Stripe stores idempotency keys for 24 hours. If we crash after calling Stripe but before writing to our DB, when we retry, Stripe returns the same result without charging again. This closes the gap between "Stripe charged" and "we wrote the result."

### 2. Double-Entry Bookkeeping

Every financial movement has two sides. This is not just accounting philosophy — it is a consistency check built into the data model.

```
Payment: User pays $100, merchant gets $97, platform keeps $3

ledger_entries:
  +----------------------------------+----------+---------+----------+
  | account                          | type     | amount  | balance  |
  +----------------------------------+----------+---------+----------+
  | user_acct_123                    | DEBIT    | $100.00 | -$100.00 |
  | merchant_acct_456                | CREDIT   | $97.00  | +$97.00  |
  | platform_fees_acct               | CREDIT   | $3.00   | +$3.00   |
  +----------------------------------+----------+---------+----------+
  Sum of all entries: -100 + 97 + 3 = 0  BALANCED

Refund: Same payment refunded

  +----------------------------------+----------+---------+
  | user_acct_123                    | CREDIT   | $100.00 |
  | merchant_acct_456                | DEBIT    | $97.00  |
  | platform_fees_acct               | DEBIT    | $3.00   |
  +----------------------------------+----------+---------+
  Sum: +100 - 97 - 3 = 0  BALANCED
```

Invariant check (run every hour):
```sql
SELECT SUM(
  CASE WHEN entry_type = 'credit' THEN amount_cents
       ELSE -amount_cents
  END
) as balance
FROM ledger_entries
WHERE created_at >= NOW() - INTERVAL '1 hour';
-- Should always equal 0
-- If not 0: ALERT ON CALL IMMEDIATELY
```

### 3. PCI-DSS Compliance: Card Data Isolation

PCI-DSS Level 1 (required for >6M transactions/year) prohibits storing raw card data in your main systems.

```
What you CANNOT store:
  - Full card number (PAN): 4111 1111 1111 1111
  - CVV/CVC code
  - Full magnetic stripe data

What you CAN store:
  - Last 4 digits: "1111"
  - Card brand: "Visa"
  - Expiry month/year
  - Payment method TOKEN (opaque string from Stripe)

Architecture:
  Browser ──────────────────────────────► Stripe.js (CDN)
                                              |
                                    Card data never touches
                                    your servers at all
                                              |
                                     Stripe tokenizes card
                                     Returns: pm_xxx_token
                                              |
  Browser ─────────────────────────────────► Your API
  POST /checkout { payment_method_token: "pm_xxx" }
                                              |
  Your server stores only: "pm_xxx"
  Your server never sees card number
```

**Token Vault for stored cards:**
When a user wants to "save card for later," Stripe stores the card and gives you a `customer_id`. You store `stripe_customer_id` in your users table. To charge later: `stripe.charge(customer=customer_id)`. The card never leaves Stripe's PCI-compliant vault.

### 4. Stripe's Architecture: Immutable Event Log as Source of Truth

Stripe's public engineering posts reveal their core insight:

```
Most systems: Database is source of truth, events are side effects
Stripe:       Event log IS the source of truth, state is derived

Every operation writes to an append-only event log:
  payment.created { id, amount, user_id, timestamp }
  payment.processing { id, psp_call_started }
  payment.succeeded { id, psp_charge_id, timestamp }
  payment.refunded { id, refund_id, amount, timestamp }

Current state is derived by replaying the event log.
You can audit any point in time: "what was the state of payment X at 3:45 PM?"
You can replay events to rebuild any derived view.
You can add new projections/views without changing the event log.
```

This is event sourcing applied to financial data. The ledger_entries table is the closest approximation in a traditional SQL system.

### 5. PSP Failover: Multiple Payment Providers

```python
class PaymentRouter:
    def __init__(self):
        self.primary = StripeClient()
        self.fallback = BraintreeClient()
        self.circuit_breaker = CircuitBreaker(failure_threshold=5, timeout=30)

    def charge(self, amount_cents: int, payment_method: str, **kwargs) -> PSPResponse:
        # Normalize payment method to PSP-specific token
        if payment_method.startswith('pm_'):      # Stripe token
            provider = self.primary
        elif payment_method.startswith('bt_'):    # Braintree token
            provider = self.fallback
        else:
            # Unknown token type - try primary
            provider = self.primary

        if self.circuit_breaker.is_open_for(provider):
            # Primary is unhealthy, use fallback
            provider = self.fallback

        try:
            return provider.charge(amount_cents, payment_method, **kwargs)
        except ProviderUnavailable:
            self.circuit_breaker.record_failure(provider)
            if provider == self.primary:
                # Failover to secondary
                return self.fallback.charge(amount_cents, payment_method, **kwargs)
            raise
```

**Complication with multi-provider**: Reconciliation must be done separately per PSP. Stripe settlement file + Braintree settlement file, merged into one internal view.

### 6. Nightly Reconciliation

```python
def nightly_reconciliation(date: datetime.date):
    # Download PSP settlement file
    stripe_settlements = stripe_api.download_settlement(date)

    discrepancies = []

    for psp_record in stripe_settlements:
        internal = db.query("""
            SELECT id, status, amount_cents
            FROM payments
            WHERE psp_charge_id = %s
        """, (psp_record.charge_id,))

        if not internal:
            discrepancies.append({
                'type': 'missing_internal_record',
                'psp_charge_id': psp_record.charge_id,
                'psp_amount': psp_record.amount
            })
        elif internal.status != 'succeeded':
            discrepancies.append({
                'type': 'status_mismatch',
                'payment_id': internal.id,
                'internal_status': internal.status,
                'psp_status': psp_record.status
            })
        elif internal.amount_cents != psp_record.amount_cents:
            discrepancies.append({
                'type': 'amount_mismatch',
                'payment_id': internal.id,
                'internal_amount': internal.amount_cents,
                'psp_amount': psp_record.amount_cents
            })

    if discrepancies:
        write_to_discrepancy_table(discrepancies)
        alert_finance_team(f"{len(discrepancies)} discrepancies found")

    # Expected: < 0.01% discrepancy rate
    # Actual discrepancies require manual review
```

## Trade-offs and Failure Modes

### Trade-off: Synchronous vs. async ledger writes

| Synchronous (write ledger in same transaction as payment) | Async (Kafka event -> ledger writer) |
|---|---|
| Guaranteed consistency | Eventual consistency |
| Simpler code | Higher throughput |
| Ledger write failure blocks payment | Ledger write retried separately |
| Correct for payment system | Acceptable for analytics ledger |

**Recommendation**: Write the core ledger synchronously in the same DB transaction as the payment status update. Use async Kafka only for downstream consumers (analytics, email notifications).

### Trade-off: Integer cents vs. NUMERIC for amounts

Never use FLOAT for money:
```python
>>> 0.1 + 0.2
0.30000000000000004  # This is wrong
```

Always use `BIGINT` (cents) or `NUMERIC(12,2)`:
```python
# Correct
amount_cents = 1000   # = $10.00
# Never
amount = 10.00       # floating point hazard
```

### Failure Mode: PSP webhook delivered multiple times

Stripe retries webhooks for up to 3 days. Your webhook handler may receive `charge.succeeded` twice.

```python
@app.post("/webhooks/stripe")
def stripe_webhook(event: StripeEvent):
    # Idempotent: check if we've already processed this event
    already_processed = db.query(
        "SELECT 1 FROM payment_events WHERE data->>'stripe_event_id' = %s",
        (event.id,)
    )
    if already_processed:
        return {"status": "already_processed"}

    # Process the event
    if event.type == 'charge.succeeded':
        handle_charge_succeeded(event.data.object)

    # Record the event (idempotency)
    db.insert_payment_event(
        payment_id=event.data.object.metadata.payment_id,
        event_type=event.type,
        data={'stripe_event_id': event.id, **event.data}
    )
```

## Senior Engineer Perspective

**The key insight:**

> "In most systems, a bug means users see wrong data. In a payment system, a bug means you owe someone money or are in legal violation. Every engineering decision must be viewed through this lens."

**Questions that reveal depth:**

1. **PSP says succeeded, your DB says pending — which is right?**
   The PSP is right. Money moved. You must reconcile. The nightly job catches this. Emergency: if discovered during business hours, manually update the DB and write compensating ledger entries. Root cause: the response was lost after PSP charged. Fix: more robust webhook processing + tighter idempotency key TTL matching.

2. **How do you test payment flows without real money?**
   PSPs provide sandbox/test environments. Critically, your idempotency and ledger logic must be testable with a mock PSP. The PSP is an external dependency — your correctness must hold regardless of what the PSP does. Use dependency injection to swap real Stripe for a test double.

3. **User disputes a charge (chargeback). How does your system handle it?**
   PSP sends a `charge.dispute.created` webhook. You create a new payment event, update the payment status to `disputed`, write ledger entries reversing the credit to the merchant (provisional hold), and notify the merchant to submit evidence. This is a whole workflow — flag it as "out of scope for this interview but mention it exists."

**Real-world context:**
- **Stripe**: Idempotency keys were their killer feature when launched in 2012. Their public engineering blog has detailed posts on exactly this design.
- **Airbnb**: Published a blog post on their payment ledger migration — moving from mutable rows to immutable event sourcing solved their reconciliation problems.
- **Uber**: Their internal "money" service handles ~$50M/day in driver payouts using exactly these patterns.
