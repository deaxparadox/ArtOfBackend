# 20.03 — Design a URL Shortener (TinyURL)

← Previous: [20.02 — Capacity Estimation](./20.02-capacity-estimation.md) | [Phase 20 Index](./README.md) | Next: [20.04 — Rate Limiter](./20.04-rate-limiter.md) →

---

## What Makes This System Hard

On the surface, a URL shortener looks trivial: store a mapping from short code to long URL. The
real challenges are:

1. **ID generation at scale without coordination.** At 1,200 writes/sec across 10 app servers,
   how do you generate 7-character codes that are globally unique, non-predictable, and collision-free
   without every server hitting a single counter?

2. **Read latency is load-bearing.** The redirect is in the critical path of a page load. A 50ms
   redirect is a 50ms penalty on every link click. Cache behavior matters enormously.

3. **Abuse surface.** URL shorteners are the primary vector for phishing links. Any design must
   account for rate limiting, safe-browsing checks, and instant deactivation.

The key insight that separates senior from junior answers: **this is primarily a caching and ID
generation problem, not a database problem.** The database is barely touched on reads if you cache
correctly.

---

## 1. Requirements Clarification

### Functional Requirements
- Given a long URL, return a unique short URL (e.g., `https://tny.io/aB3xZ9f`)
- Given a short URL, redirect the client to the original long URL
- Optional: custom aliases (user chooses their own short code)
- Optional: link expiry (URL stops working after a set date)
- Optional: click analytics (count, geo, device)

### Non-Functional Requirements
- **Read-heavy**: 100:1 read-to-write ratio — optimize hard for reads
- **Redirect latency**: < 10ms p99 (in the critical path of every page load that uses the link)
- **Availability**: 99.99% — a broken short link is visible user-facing damage
- **Non-enumerable IDs**: `abc124` should not be the obvious successor to `abc123`

### Out of Scope (for first pass)
- User dashboards, link management UI
- A/B testing on redirects
- Real-time analytics streams

## 2. Capacity Estimation

```
Assumptions:
  Write QPS:    100 million new URLs/day  →  ~1,200 writes/sec
  Read QPS:     10 billion redirects/day  →  ~115,000 reads/sec
  Peak factor:  3x average               →  ~345,000 reads/sec peak

Storage per URL row:
  short_code:   7 bytes
  long_url:     200 bytes (average)
  metadata:     ~100 bytes (timestamps, user_id, expiry, flags)
  Total:        ~307 bytes/row

Storage at 5 years:
  100M URLs/day × 365 × 5 = 182 billion rows (theoretical max churn)
  Realistic active URLs:    ~1 billion
  1B × 307 bytes            = ~300 GB  (single large PostgreSQL instance)

Cache sizing (Pareto principle: 20% of URLs = 80% of traffic):
  200M URLs × 307 bytes     = ~60 GB  →  fits in a Redis cluster
  Cache hit rate target:    95%+  →  only 5% of reads touch the DB

Bandwidth:
  Read: 115,000 req/sec × 300 bytes response = ~35 MB/s outbound
```

## 3. High-Level Architecture

```
                            ┌──────────────┐
                            │    Client    │
                            └──────┬───────┘
                                   │  HTTP GET/POST
                       ┌───────────▼───────────┐
                       │      Load Balancer     │
                       │   (L7, sticky by IP)   │
                       └───────────┬───────────┘
                                   │
             ┌─────────────────────┼──────────────────────┐
             │                     │                      │
    ┌────────▼────────┐  ┌─────────▼────────┐  ┌─────────▼────────┐
    │   App Server 1  │  │   App Server 2   │  │   App Server 3   │
    │  (stateless)    │  │   (stateless)    │  │   (stateless)    │
    └────────┬────────┘  └─────────┬────────┘  └─────────┬────────┘
             └─────────────────────┼──────────────────────┘
                                   │
                       ┌───────────▼───────────┐
                       │   Redis Cache Cluster  │
                       │  short_code → long_url │
                       │  TTL = expiry or 24h   │
                       └───────────┬───────────┘
                                   │ cache miss
                       ┌───────────▼───────────┐
                       │  PostgreSQL Primary    │
                       │  + Read Replicas       │
                       └───────────────────────┘

Write Path:
  Client POST /shorten
    → App Server
    → Generate short_code (see Section 5)
    → INSERT into PostgreSQL
    → SET in Redis with TTL
    → Return short URL

Read Path (redirect):
  Client GET /aB3xZ9f
    → App Server
    → Redis GET aB3xZ9f  →  HIT: return 302 redirect
                          →  MISS: SELECT from DB → SET in Redis → return 302 redirect

Analytics Path (async, non-blocking):
  App Server → Kafka topic "click_events"
    → Stream consumer → ClickHouse analytics DB
```

## 4. Data Model

```sql
-- Primary mapping table
CREATE TABLE urls (
    short_code   CHAR(7)      PRIMARY KEY,
    long_url     TEXT         NOT NULL,
    user_id      BIGINT,                          -- NULL for anonymous
    created_at   TIMESTAMPTZ  NOT NULL DEFAULT NOW(),
    expires_at   TIMESTAMPTZ,                     -- NULL = never expires
    click_count  BIGINT       NOT NULL DEFAULT 0, -- approximate, async-updated
    is_active    BOOLEAN      NOT NULL DEFAULT TRUE
);

CREATE INDEX idx_urls_user_id     ON urls(user_id);
CREATE INDEX idx_urls_expires_at  ON urls(expires_at) WHERE expires_at IS NOT NULL;
CREATE INDEX idx_urls_is_active   ON urls(is_active)  WHERE is_active = FALSE;

-- Async click events (written to Kafka, landed here by consumer)
CREATE TABLE click_events (
    id           BIGSERIAL    PRIMARY KEY,
    short_code   CHAR(7)      NOT NULL REFERENCES urls(short_code),
    clicked_at   TIMESTAMPTZ  NOT NULL DEFAULT NOW(),
    ip_hash      CHAR(64),    -- SHA-256 of IP for privacy
    country      CHAR(2),
    device_type  TEXT,        -- 'mobile' | 'desktop' | 'bot'
    referrer     TEXT
);

-- Partition click_events by month for query performance
-- e.g., CREATE TABLE click_events_2024_01 PARTITION OF click_events
--       FOR VALUES FROM ('2024-01-01') TO ('2024-02-01');
```

**Sharding key**: `short_code`. It distributes evenly because it is pseudo-random. Never shard on
`user_id` alone — hot users would overwhelm a single shard.

## 5. Deep Dive: Unique Short Code Generation at Scale

### Why This Is the Core Problem

Any approach that requires a round-trip to a single central store for every write creates a
bottleneck at 1,200 writes/sec. Any approach that allows collisions requires a retry loop that
becomes unreliable as the namespace fills. The right design balances uniqueness, speed, and
unpredictability.

### Option A: Hash of Long URL (Do Not Use as Primary Strategy)

```python
import hashlib

def generate_code(long_url: str) -> str:
    digest = hashlib.md5(long_url.encode()).hexdigest()
    return digest[:7]
```

Problems:
- Same URL always generates the same code — breaks if a user shortens the same URL twice wanting
  separate tracking links
- Collisions are possible (birthday problem over the 7-char prefix space)
- Every write requires a read-before-write to check for collision

### Option B: Counter + Base62 Encoding (Good, Needs Coordination)

```python
ALPHABET = "0123456789ABCDEFGHIJKLMNOPQRSTUVWXYZabcdefghijklmnopqrstuvwxyz"

def base62_encode(n: int) -> str:
    """Encode a non-negative integer to base62 string."""
    if n == 0:
        return ALPHABET[0]
    result = []
    while n:
        n, remainder = divmod(n, 62)
        result.append(ALPHABET[remainder])
    return ''.join(reversed(result))

# ID 1,000,000 → "4c92"
# ID 3,500,000,000 → "7n42DGM" (7 chars)
print(base62_encode(1_000_000))       # "4c92"
print(base62_encode(3_500_000_000))   # "7n42DGM"
```

62^7 = 3,521,614,606,208 possible codes — enough for billions of URLs.

Distributed counter mitigation: pre-allocate ID ranges per server.

```
Central counter (Redis INCR):  current = 50,000,000

Server A calls: INCRBY counter 1000  → gets range [50,000,001 – 51,000,000]
Server B calls: INCRBY counter 1000  → gets range [51,000,001 – 52,000,000]

Each server uses its local range with no further coordination.
When range is exhausted, it fetches a new batch.
```

Pros: zero collisions, O(1) generation, predictable lengths.
Con: IDs are sequential within a batch — slight information leakage about creation order.

### Option C: Random Base62 (Simple, Works at Moderate Scale)

```python
import secrets

ALPHABET = "0123456789ABCDEFGHIJKLMNOPQRSTUVWXYZabcdefghijklmnopqrstuvwxyz"

def generate_code(length: int = 7) -> str:
    return ''.join(secrets.choice(ALPHABET) for _ in range(length))

def create_short_url(long_url: str, db, max_attempts: int = 5) -> str:
    for attempt in range(max_attempts):
        code = generate_code()
        # Use INSERT ... ON CONFLICT DO NOTHING and check rows affected
        inserted = db.execute(
            "INSERT INTO urls(short_code, long_url) VALUES (%s, %s) ON CONFLICT DO NOTHING",
            (code, long_url)
        ).rowcount
        if inserted == 1:
            return code
    raise RuntimeError(f"Failed to generate unique code after {max_attempts} attempts")
```

Collision probability at N URLs: approximately N / 62^7.
At 1 billion URLs: 1B / 3.5T = 0.028% per attempt — five retries is more than enough.

### Option D: Snowflake-style ID (Best for Hyperscale)

```
64-bit integer layout:
  [41 bits: millisecond timestamp] [10 bits: machine ID] [12 bits: sequence]

Encode to base62 → ~10 characters, globally unique, no coordination, monotonically increasing.
```

Used by Twitter. The slight length increase (10 vs 7 chars) is the trade-off.

### Decision Matrix

| Approach        | Collisions | Coordination | Predictable | Use When              |
|-----------------|------------|--------------|-------------|-----------------------|
| Hash of URL     | Possible   | None needed  | Yes         | Never (dedup only)    |
| Random Base62   | Retry loop | None         | No          | < 500M URLs           |
| Range Counter   | Zero       | Batch fetch  | Slight      | 500M – 10B URLs       |
| Snowflake ID    | Zero       | None         | Timestamp   | > 10B URLs            |

## 6. 301 vs 302 Redirect

```
301 Permanent Redirect:
  - Browser caches the redirect indefinitely
  - Subsequent clicks go directly to destination — never touch your servers
  - Result: you lose all analytics after the first click
  - Use case: pure link shortening, SEO pass-through

302 Temporary Redirect:
  - Browser re-requests your server on every click
  - You log every click: timestamps, geo, device, referrer
  - Result: accurate analytics but higher server load
  - Use case: any system that monetizes or measures link clicks

Decision rule: if analytics is in scope, use 302. If not, 301 saves significant server load.
```

## 7. Trade-offs Summary

| Concern               | Solution                        | Trade-off                                    |
|-----------------------|---------------------------------|----------------------------------------------|
| Read throughput       | Redis cache, 95%+ hit rate      | Stale cache after URL deactivation           |
| ID uniqueness         | Range counter or Snowflake      | Counter: slight order leakage                |
| Write throughput      | Shard PostgreSQL by short_code  | Cross-shard queries for "list user's URLs"   |
| Analytics             | Kafka → ClickHouse async        | Eventual consistency; counts lag by seconds  |
| Availability          | Multi-region replicas           | Potential split-brain on writes              |
| Abuse / phishing      | Rate limit + Safe Browsing API  | Latency added to write path                  |

## 8. Senior Engineer Perspective

**The redirect is latency-critical in a way most systems are not.** A slow database query in an
admin panel is annoying. A slow redirect is a visible delay every time someone opens a link. Cache
aggressively. Pre-warm caches for newly created links. Set Redis TTLs equal to URL expiry so the
cache auto-evicts on schedule.

**ID generation is the one irreversible decision.** If you start with sequential IDs and later
decide they leak too much information, you cannot migrate without breaking all existing URLs. Decide
on your ID strategy before writing any code, and document the trade-offs explicitly.

**Analytics are almost always harder than the core system.** "Just count clicks" sounds simple until
the product team asks for clicks by country by device by referrer by day. Design the analytics path
as a separate system from the start: async events, a streaming pipeline, a columnar store. Do not
bolt it onto the main read path.

**301 vs 302 is a business decision, not a technical one.** Engineers often pick 301 without
thinking about it. If the company's business model involves measuring link performance (marketing
campaigns, A/B tests, affiliate tracking), 302 is non-negotiable. Ask the product team before
implementing.

**Abuse is not an edge case.** URL shorteners are the primary mechanism for disguising phishing
URLs. Any production system needs: rate limiting on creation (by IP and by user), integration with
Google Safe Browsing API on write, the ability to deactivate a URL and have that propagation reach
every edge within seconds (which requires cache invalidation to work reliably).
