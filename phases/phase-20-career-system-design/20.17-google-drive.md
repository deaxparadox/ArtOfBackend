## The Key Architectural Insight

Designing file storage sounds simple: store bytes, return bytes. The hard part is **synchronization**. Users have the same files on a laptop, a phone, and a desktop. When they edit a file on the laptop, the phone and desktop must reflect the change — fast, reliably, and without corrupting the file if two devices edit it simultaneously.

The second hard problem is **scale**: users store large files (videos, backups, disk images). You cannot load a 10 GB file into memory to process it. You must split large files into chunks, handle partial uploads, and deduplicate chunks across users. This is where the real engineering lives.

---

## Section 1: Requirements Clarification

| Question | Why it matters |
|---|---|
| Max file size? | Determines chunking strategy and upload protocol |
| How many devices per user? | More devices = more sync complexity |
| Is real-time collaborative editing needed? (like Google Docs) | Live editing requires OT/CRDT — a completely different design |
| Offline mode required? | Files must be downloadable and synced on reconnect |
| Sharing model? | Per-file ACLs, link sharing, team drives |
| Version history? | How many versions, how long retained? |

**Scope for this design:** Dropbox-style file sync (not real-time collaborative editing). Up to 5 devices per user, files up to 5 GB, 30-day version history, sharing via links and ACLs.

## Section 2: Capacity Estimation

| Parameter | Estimate |
|---|---|
| Total users | 500 million |
| Active daily users | 100 million |
| Average storage per user | 10 GB |
| Total storage | 500M * 10 GB = **5 EB** |
| Daily file changes | 100M users * 2 changes/day = **200M changes/day** |
| Average file size at upload | 1 MB (mix of docs, images, code) |
| Daily upload volume | 200M * 1MB = **200 TB/day** |
| Peak upload bandwidth | 200 TB / 86400s ≈ **2.3 GB/sec** |
| File metadata entries | 500M users * 1000 files avg = **500 billion rows** |

The metadata table (file tree) is enormous. At 500 billion rows, it cannot live in a single relational database — you need horizontal sharding, likely by `user_id`.

## Section 4: Deep Dive — The Hard Problems

### 4.1 Block-Level Deduplication

**Why split files into blocks?** Three reasons:

1. **Resumable uploads:** If a 2 GB upload fails at 80%, resume from the last successful block — not from scratch.
2. **Delta sync:** If a user edits line 5 of a 100 MB document, only the blocks containing that line change. Upload 4 MB instead of 100 MB.
3. **Cross-user deduplication:** If 1 million users upload the same Ubuntu ISO, store one copy globally. Only metadata differs per user.

**Block split strategy (fixed-size):**
```
File: video.mp4 (400 MB)
  Block 1:  bytes         0 to 4,194,303  (4MB)  SHA256: aabbcc...
  Block 2:  bytes 4,194,304 to 8,388,607  (4MB)  SHA256: ddeeff...
  Block 3:  bytes 8,388,608 to 12,582,911 (4MB)  SHA256: 112233...
  ...
  Block 100: bytes ~396MB to 400MB        (4MB)  SHA256: 445566...
```

Fixed 4 MB blocks are simple but have a flaw: insert a byte at position 0 and every block's content shifts — all 100 blocks appear changed, defeating delta sync.

**Content-Defined Chunking (CDC)** solves this: use a rolling hash (Rabin fingerprint or Gear hash) to find chunk boundaries based on content patterns. Boundaries are stable across insertions — only the blocks near the edit point actually change.

**Dedup check protocol:**
```python
def upload_file(file_path: str, user_id: str):
    blocks = split_into_blocks(file_path)   # [(hash, data), ...]
    hashes = [b.hash for b in blocks]

    # One API call: "which of these hashes do you NOT have?"
    missing_hashes = api.check_blocks(hashes)   # server checks dedup table

    # Upload only the blocks the server doesn't already have
    for block in blocks:
        if block.hash in missing_hashes:
            api.upload_block(block.hash, block.data)

    # Always update metadata: file now maps to these ordered blocks
    api.update_file_metadata(user_id, file_path, hashes)
```

In practice, dedup rates for popular file types (OS images, video files, common libraries) are 30-50%. Dropbox reported enormous storage savings from cross-user dedup before switching to per-user encryption.

### 4.2 Metadata Model

The file tree is pure metadata — not content. Content lives in object storage, addressed by block hash.

```sql
-- File tree (sharded horizontally by user_id)
CREATE TABLE files (
    user_id         BIGINT        NOT NULL,
    file_id         UUID          NOT NULL DEFAULT gen_random_uuid(),
    parent_id       UUID,                          -- NULL = root directory
    name            VARCHAR(1024) NOT NULL,
    is_folder       BOOLEAN       NOT NULL DEFAULT FALSE,
    current_rev_id  UUID,                          -- FK to revisions
    created_at      TIMESTAMPTZ   NOT NULL DEFAULT NOW(),
    updated_at      TIMESTAMPTZ   NOT NULL DEFAULT NOW(),
    deleted_at      TIMESTAMPTZ,                   -- soft delete
    PRIMARY KEY (user_id, file_id)
);

-- Version history (every save creates a new revision)
CREATE TABLE revisions (
    revision_id     UUID          PRIMARY KEY DEFAULT gen_random_uuid(),
    file_id         UUID          NOT NULL,
    user_id         BIGINT        NOT NULL,
    block_hashes    TEXT[]        NOT NULL,        -- ordered list of block SHA256s
    size_bytes      BIGINT        NOT NULL,
    content_hash    VARCHAR(64)   NOT NULL,        -- SHA256 of full file (quick compare)
    created_at      TIMESTAMPTZ   NOT NULL DEFAULT NOW(),
    device_id       VARCHAR(128)  NOT NULL         -- which device made this change
);

-- Global block deduplication index
CREATE TABLE blocks (
    block_hash      VARCHAR(64)   PRIMARY KEY,     -- SHA256 of block content
    size_bytes      INT           NOT NULL,
    storage_key     VARCHAR(512)  NOT NULL,         -- path in S3/GCS
    reference_count INT           NOT NULL DEFAULT 1,
    created_at      TIMESTAMPTZ   NOT NULL DEFAULT NOW()
);
```

The `blocks` table is a global dedup index. `reference_count` tracks how many file revisions reference each block. When it reaches 0, the block is eligible for garbage collection from object storage.

### 4.3 Change Notification — How Other Devices Know

When you save a file on your laptop, your phone and desktop must discover the change. Two mechanisms:

**Long polling (Dropbox's original approach):**
```
Device opens HTTP request:
  GET /api/changes?since=cursor_12345&timeout=30s

Server holds the connection open up to 30 seconds.
  - If a change occurs:  respond immediately with change summary
  - If no change in 30s: respond with "no change", client re-opens immediately

Pros: Works through firewalls and proxies, simple to implement
Cons: Server holds one open connection per device -- C10K problem at scale
     (1M active devices = 1M open connections, needs async server like Nginx/Node)
```

**WebSocket (modern approach):**
```
Device opens persistent WebSocket:
  wss://sync.example.com/ws?token=abc&cursor=12345

Server pushes events as they arrive:
  {"type": "file_changed", "file_id": "xyz", "new_rev": "rev_abc456"}

Client receives event --> asks "what blocks changed since my cursor?"
                      --> downloads only missing blocks
                      --> updates local file and cursor

Pros: Sub-second latency, lower server overhead at scale vs. long-poll
Cons: Stateful connections, harder to load balance (need sticky sessions
      or a pub/sub backplane like Redis Pub/Sub between WS servers)
```

**Change propagation pipeline:**
```
1. Client A uploads new revision --> Metadata Service
2. Metadata Service writes new revision to DB
3. Metadata Service publishes to Kafka:
   { "user_id": 123, "file_id": "abc", "rev_id": "def", "source_device": "laptop-1" }
4. Notification Service consumes Kafka event
5. Looks up all device connections for user 123
6. Pushes notification to all devices EXCEPT the source device
7. Each device receives notification --> calls "get changes since cursor X"
8. Downloads missing blocks --> applies to local copy --> updates cursor
```

The **cursor** is a logical position in the user's change stream (like a Kafka offset per user). Devices reconnecting after being offline query "give me all changes since cursor X" and replay missed events.

### 4.4 Conflict Resolution

Two devices edit the same file simultaneously while offline:

```
Timeline:
  Device A goes offline. Edits report.docx at 10:00 AM.
  Device B goes offline. Edits report.docx at 10:05 AM.
  Both come online at 11:00 AM and try to upload their version.

  Both changes are based on the same parent revision.
  This is a CONFLICT -- same parent, two divergent children.
```

**Last-Write-Wins (LWW) with conflict copy (Dropbox's approach):**
- Device B's revision is "newer" by wall-clock time -- it wins as the "canonical" version
- Device A's revision is saved as `report (conflicted copy 2024-01-15, Alice's MacBook).docx`
- Both files are visible in the folder -- no data is lost
- User resolves manually

Simple, never destroys data, and users understand it. The downside: silently discards one set of changes as "the other file."

**Three-way merge (for text-based files):**
Compare Device A's changes and Device B's changes against the common ancestor revision. If changes touch different lines, merge automatically. If the same line changed in both -- flag as a conflict requiring manual resolution (exactly like `git merge`).

**In practice:** For a general-purpose file sync system, LWW with a conflict copy is the correct default. Reserve three-way merge for specific file types (text, code) where it adds value without risk.

### 4.5 Chunked Upload Protocol

For large files, a single HTTP POST is fragile. Use a resumable upload protocol:

```
Step 1: Initialize upload session
  POST /api/upload/init
  Body: { "filename": "video.mp4", "total_size": 4294967296, "block_count": 1024 }
  Response: { "upload_id": "sess_abc123", "expires_at": "2024-01-22T10:00:00Z" }

Step 2: Upload blocks (parallelized, up to 8 concurrent per client)
  PUT /api/upload/sess_abc123/block/47
  Headers: Content-SHA256: aabbccdd...
  Body: [4MB binary data]
  Response: { "block_index": 47, "status": "received" }

Step 3: Query progress (on reconnect after failure)
  GET /api/upload/sess_abc123/status
  Response: { "uploaded_blocks": [0,1,2,5,6,47], "missing": [3,4,7,...,1023] }

Step 4: Re-upload only missing blocks (resume from failure)

Step 5: Finalize (only after all blocks confirmed)
  POST /api/upload/sess_abc123/finalize
  Body: { "block_hashes": ["aabbcc", "ddeeff", ...] }
  Response: { "file_id": "file_xyz", "revision_id": "rev_001" }
```

The server stores a partial upload manifest. If the client crashes, it reconnects, queries the manifest, and continues from where it left off. The manifest expires after 7 days to prevent leaked resources.

## Section 5: Trade-offs and Failure Modes

| Scenario | Impact | Mitigation |
|---|---|---|
| Object storage (S3) outage | No new uploads or downloads | Multi-region replication; read from replica during primary outage |
| Metadata DB shard overloaded | File tree operations slow | Read replicas for listing; shard more finely (by hash of user_id) |
| Notification service down | Devices don't learn about changes | Devices fall back to polling every 30s; reconnect WebSocket on drop |
| Block referenced by multiple users is deleted | Data loss for other users | Reference counting -- GC only when ref_count reaches 0 |
| Large upload times out at 95% | Wasted work | Resumable upload sessions -- query progress on reconnect and continue |
| Two clients upload the same block simultaneously | Duplicate S3 write | Idempotent S3 PUTs by hash key -- safe to write twice, last writer wins |
| Cross-user dedup reveals file hash matches | Privacy exposure | Per-user encryption (post-2011 Dropbox) -- breaks cross-user dedup entirely |

## Section 6: Senior Engineer Perspective

The most common interview mistake: designing Google Drive as "upload to S3, store metadata in Postgres." Correct but shallow. What separates strong candidates:

**1. Block-level dedup is the core insight.** Not just for storage efficiency, but for delta sync. A user editing a 10 GB video file should upload 4 MB, not 10 GB. This changes the UX from "sync takes hours" to "sync takes seconds." It is the product's core value proposition.

**2. The cursor-based change feed must survive offline reconnects.** What happens when a device is offline for 3 days? The change feed must be durable enough to replay 3 days of changes. If you use a "latest state only" approach, you miss intermediate revisions and break version history. Design the cursor explicitly.

**3. Cross-user dedup creates privacy and legal exposure.** Storing one copy of a file shared by 1 million users means a single deletion affects everyone. It also means you can confirm that specific users had specific files. Dropbox switched to per-user encryption after a 2011 security incident, which broke cross-user dedup but was the right security decision. Know this trade-off.

**4. CDC vs. fixed-size chunking is a real architectural choice.** Fixed 4 MB blocks are easy to explain in an interview. Content-defined chunking is the production-grade answer for delta sync. Mention both and explain the trade-off.

> **The interview signal:** The best candidates start with: "Before I design anything — is this Dropbox-style file sync, or Google Docs-style real-time co-editing? Because those are completely different systems." That single question signals senior thinking. File sync is an eventually consistent sync problem. Real-time co-editing is a distributed consensus problem (OT/CRDT). They share almost nothing architecturally.
