## What Makes This System Hard

Sending a single notification is trivial — make an HTTP call to FCM or SendGrid. The hard problems
emerge at scale:

1. **Multi-channel fan-out**: A single event (order shipped) must trigger push, email, and SMS
   through different third-party providers, each with different APIs, rate limits, and failure modes.

2. **Reliability with deduplication**: If your notification service crashes after calling FCM but
   before marking the notification as sent, you will send the notification again on retry. Users
   who receive the same OTP twice or the same "your card was charged" notification twice lose trust.

3. **Priority and quiet hours**: An OTP must arrive in under 5 seconds. A marketing email can wait
   30 minutes. Sending a marketing push at 2am gets your app uninstalled. These are not the same
   system, and conflating them causes either OTPs timing out or users rage-uninstalling.

The key architectural insight is: use the outbox pattern to guarantee at-least-once delivery, use
idempotency keys to make at-least-once behave like exactly-once, and use separate queues per
channel and per priority class.

## 1. Requirements Clarification

### Functional Requirements
- Send notifications via multiple channels: push (FCM for Android, APNs for iOS), email (SendGrid),
  SMS (Twilio), and in-app inbox
- Triggered by: user actions (via API call), scheduled campaigns, and system events
- User preferences: opt out of channels and categories (marketing, transactional, social)
- Delivery tracking: sent, delivered, failed, clicked
- Priority tiers: transactional (OTP, payment alert) vs promotional (weekly digest, sale)
- Scheduled delivery: send at a specific future time or in the user's local timezone

### Non-Functional Requirements
- **Throughput**: 10 million notifications/day, burst of 1 million/hour during campaigns
- **Transactional SLA**: OTPs and payment alerts delivered in under 5 seconds
- **Promotional SLA**: marketing content can take up to 30 minutes
- **Reliability**: transactional delivery at 99.9%; promotional at best-effort
- **Idempotency**: the same logical event must not produce duplicate notifications

### Clarifying Questions to Ask in an Interview
- Are transactional and promotional notifications from the same code path?
- Do we need push read receipts (delivered vs opened)?
- What is the expected notification volume per user per day?
- Do we need to support per-user quiet hours and DND windows?

## 5. Deep Dive: The Hard Problems

### Problem 1 — Deduplication via Idempotency Keys

Any distributed system that guarantees delivery must use at-least-once semantics. A worker can
call FCM, receive no response (network timeout), and then retry — leading to a duplicate delivery.
Idempotency keys convert at-least-once into effectively-once.

```python
import hashlib
import time

def build_idempotency_key(user_id: int, event_type: str,
                           entity_id: str, window_sec: int = 3600) -> str:
    """
    Key is stable within a time window.
    Two calls with same (user, event, entity) within window_sec
    seconds produce the same key → deduplication.
    """
    window = int(time.time()) // window_sec
    raw = f"{user_id}:{event_type}:{entity_id}:{window}"
    return hashlib.sha256(raw.encode()).hexdigest()

def send_notification(user_id: int, event_type: str,
                      entity_id: str, payload: dict) -> str:
    key = build_idempotency_key(user_id, event_type, entity_id)

    # INSERT ... ON CONFLICT DO NOTHING is atomic
    # If the key already exists, rowcount == 0  →  duplicate, skip
    result = db.execute(
        """
        INSERT INTO notifications (idempotency_key, user_id, channel, body, payload)
        VALUES (%s, %s, %s, %s, %s)
        ON CONFLICT (idempotency_key) DO NOTHING
        """,
        (key, user_id, 'push', payload.get('body', ''), payload)
    )

    if result.rowcount == 0:
        # Already exists — return the existing notification ID
        existing = db.fetchone(
            "SELECT id FROM notifications WHERE idempotency_key = %s", (key,)
        )
        return existing['id']

    new_id = result.lastrowid
    queue.enqueue('notifications', notification_id=new_id)
    return new_id
```

### Problem 2 — Multi-Channel Routing with User Preferences

```python
def route_notification(notification_id: int) -> list[str]:
    """
    Read user preferences and route to the right channel queues.
    Returns list of channel names the notification was routed to.
    """
    notif = db.get_notification(notification_id)
    prefs = preference_cache.get(notif.user_id)  # Redis, 5-min TTL

    channels_sent = []

    for channel in ['push', 'email', 'sms']:
        pref = prefs.get((channel, notif.category))

        # Skip if user opted out
        if pref and not pref.is_enabled:
            continue

        # Skip if in quiet hours (only for non-transactional)
        if notif.priority != 'high' and pref and is_quiet_hours(pref):
            # Schedule for quiet_end time instead of dropping
            schedule_for_quiet_end(notification_id, channel, pref)
            continue

        # Enqueue to channel-specific queue
        queue_name = f"notifications.{channel}.{notif.priority}"
        queue.enqueue(queue_name, {
            'notification_id': notification_id,
            'user_id': notif.user_id,
            'channel': channel
        })
        channels_sent.append(channel)

    return channels_sent
```

### Problem 3 — Handling Stale Device Tokens

FCM and APNs return error codes when a device token is no longer valid (app uninstalled, token
rotated). You must process these errors and clean up your token registry.

```python
def send_push_fcm(user_id: int, tokens: list[str], payload: dict) -> None:
    """
    FCM batch send: up to 500 tokens per call.
    Process responses and clean up invalid tokens.
    """
    response = fcm_client.send_multicast({
        'registration_ids': tokens,
        'notification': {'title': payload['title'], 'body': payload['body']},
        'data': payload.get('data', {})
    })

    for i, result in enumerate(response['results']):
        token = tokens[i]

        if 'message_id' in result:
            # Success — optionally update last_seen
            pass

        elif 'error' in result:
            error = result['error']
            if error in ('NotRegistered', 'InvalidRegistration'):
                # Token is dead — deactivate immediately
                db.execute(
                    "UPDATE device_tokens SET is_active = FALSE WHERE token = %s",
                    (token,)
                )
            elif error == 'MismatchSenderId':
                # Token belongs to a different project — delete
                db.execute(
                    "DELETE FROM device_tokens WHERE token = %s", (token,)
                )
            # Other errors (Unavailable, InternalServerError) → retry the batch
```

### Problem 4 — Retry Strategy with Dead Letter Queue

```
Retry schedule for HIGH priority (transactional):
  Attempt 1:  immediate
  Attempt 2:  30 seconds
  Attempt 3:  2 minutes
  → After 3 failures: move to DLQ, page on-call

Retry schedule for NORMAL priority (social, activity):
  Attempt 1:  immediate
  Attempt 2:  1 minute
  Attempt 3:  5 minutes
  Attempt 4:  30 minutes
  → After 4 failures: move to DLQ, alert (no page)

Retry schedule for LOW priority (marketing):
  Attempt 1:  immediate
  Attempt 2:  10 minutes
  Attempt 3:  1 hour
  → After 3 failures: discard silently (user missed a promo, not critical)
  → Promotional notifications older than 24 hours: discard even if retrying
```

## 6. Outbox Pattern — Guaranteed Delivery

The outbox pattern prevents the "partial write" problem: the service processes a payment, wants
to send a notification, but crashes between the DB commit and the queue publish. Without the
outbox, the notification is lost silently.

```
Without outbox (broken):
  BEGIN
    UPDATE orders SET status = 'paid'  -- succeeds
    queue.publish('order_paid', ...)   -- crashes here
  COMMIT
  Result: order is marked paid, notification never sent

With outbox (correct):
  BEGIN
    UPDATE orders SET status = 'paid'
    INSERT INTO notification_outbox (payload) VALUES (...)  -- same transaction
  COMMIT
  
  Separate outbox poller (runs every 100ms):
    SELECT * FROM notification_outbox WHERE processed = FALSE LIMIT 100
    FOR EACH row:
      queue.publish('notifications', row.payload)
      UPDATE notification_outbox SET processed = TRUE WHERE id = row.id
```

The outbox and the business record commit atomically in the same database transaction. The queue
publish happens separately. If the poller crashes after publishing but before marking the row, it
will re-publish on restart — which is why idempotency keys on the consumer side are essential.

## 7. Trade-offs Summary

| Concern                   | Solution                             | Trade-off                                        |
|---------------------------|--------------------------------------|--------------------------------------------------|
| Duplicate delivery        | Idempotency keys + ON CONFLICT       | Write amplification on every notification insert |
| Channel fan-out           | Separate queue per channel           | More queues to operate; separate monitoring      |
| Priority isolation        | Separate queue per priority tier     | More infrastructure complexity                   |
| Provider outage           | Fallback channel (push → SMS)        | May surprise users; add logic for opt-in only    |
| Stale tokens              | Process FCM/APNs error responses     | Token cleanup is async; stale tokens persist briefly |
| Quiet hours               | Per-user scheduled delivery          | Scheduled jobs accumulate; need expiry           |
| Preference lookup latency | Redis cache, 5-minute TTL            | Stale preferences for up to 5 minutes            |

## 8. Senior Engineer Perspective

**Transactional and promotional are not the same system.** They have different SLAs, different
compliance requirements, and different failure modes. An OTP that fails to deliver in 5 seconds
is a security incident. A marketing email that arrives 30 minutes late is irrelevant. Put them on
separate queues with separate workers and separate alerting thresholds from the beginning.

**Stale device tokens are not an edge case — they are the normal state.** Expect 20-40% of your
push token list to be invalid at any given time. Apps get uninstalled. Users upgrade phones. APNs
rotates tokens. Build the error-response cleanup loop before launch. If you don't, you will watch
FCM report 35% error rates and assume your notification code is broken when the tokens are simply
dead.

**Provider abstraction is worth the upfront cost.** Build a thin interface with a common contract
(send notification → return result). Implement it for FCM, APNs, and SendGrid separately. This
lets you A/B test providers, fail over between providers during an outage, and add new providers
without rewriting workers. Without this abstraction, changing your email provider requires
touching every piece of code that calls the provider directly.

**Frequency capping is a retention feature, not a technical one.** "Do not send more than 3
marketing notifications per user per week" sounds like a rule, but it is actually what prevents
users from uninstalling your app. Implement it as a Redis counter (key per user per category per
week, TTL = 7 days). It is simple to build and critically important to maintain user trust.

**The outbox pattern is the correct foundation.** Any system that tries to publish to a message
queue and commit to a database separately will eventually lose messages or send duplicates under
failure conditions. The outbox pattern makes delivery a property of the database transaction,
not a prayer about the network being available at the exact right moment.
