## What Makes This System Hard

Rate limiting a single server is trivial: keep a counter in memory, reject when it exceeds the
limit. The hard problem is **distributed rate limiting**: when 10 app servers can each handle any
user's request, how do you enforce a global limit of "100 requests per minute per user" without
every request paying a round-trip to a central bottleneck, and without that bottleneck becoming a
single point of failure?

The secondary challenge is **algorithm selection**. Different algorithms have different fairness
guarantees, burst behaviors, and implementation complexity. A fixed window counter is simple but
allows up to 2x the stated limit at window boundaries. A sliding window log is accurate but
stores every request timestamp. A token bucket allows controlled bursts. Choosing the wrong
algorithm for your traffic pattern causes silent over-admission or unfair client rejections.

---

## 1. Requirements Clarification

### Functional Requirements
- Limit requests per client (user ID, API key, or IP) to N requests per time window
- Support per-endpoint limits (e.g., `/login` is stricter than `/search`)
- Return HTTP 429 Too Many Requests when limit is exceeded
- Include rate limit metadata in every response via standard headers
- Optional: tiered limits (free vs paid users get different allowances)
- Optional: configurable rules without requiring a deployment

### Non-Functional Requirements
- **Low overhead**: rate limit check must add less than 1ms to request latency
- **Accuracy**: must not admit more than ~5% above the stated limit
- **Availability**: the rate limiter must not be a harder dependency than the API itself
- **Distributed**: consistent enforcement across all app servers sharing no local state

### Clarifying Questions to Ask in an Interview
- Is this for a public API or internal service-to-service calls?
- Hard guarantees (never exceed limit) or soft (small overage acceptable)?
- Rate limit by user ID, IP, API key, or a combination?
- What happens when the backing store (Redis) is unavailable — fail open or closed?
- Do different endpoints have different limits, or a single global policy?

### Out of Scope (for first pass)
- Rate limiting at the DNS or anycast layer
- Adaptive limits that change based on server health

## 2. Capacity Estimation

```
Assumptions:
  Total API QPS:               100,000 req/sec
  Distinct rate-limit keys:    1 million  (user + endpoint combinations active in a window)
  Redis ops per request:       2  (INCR + EXPIRE on first request in window)
  Redis throughput required:   200,000 ops/sec  →  within a single Redis node (500K ops/sec)

Memory per counter key:
  Key string:   ~40 bytes  ("rate:u:123456:search:28356")
  Int64 value:  8 bytes
  TTL metadata: ~8 bytes
  Total:        ~56 bytes per active key

Total memory for 1M active keys:
  1,000,000 × 56 bytes = 56 MB  →  negligible

Sliding window log memory (worst case):
  100 req/min limit × 1M active users = 100M timestamp entries
  Each timestamp: 8 bytes
  Total: 800 MB  →  manageable, but 14x more than a counter approach
```

## 3. Rate Limiting Algorithms

### Algorithm 1: Fixed Window Counter

The simplest approach. Divide time into fixed windows (every 60 seconds). Track a counter per
window. Reset counter when the window advances.

```
Limit: 5 requests per 60-second window

Timeline:
  Window A  [0s – 60s]      Window B  [60s – 120s]
  |--- 3 requests ---|      |--- 4 requests ---|
                   ^
         window boundary

Problem — boundary burst:
  t=59s:  3 requests arrive in window A  →  counter = 3, allowed (3 < 5)
  t=61s:  5 requests arrive in window B  →  counter = 5, allowed (5 <= 5)
  
  Result: 8 requests in 2 seconds, despite a limit of 5/minute.
  User exploited the window reset by straddling the boundary.
```

```python
import time
import redis

r = redis.Redis()

def is_allowed_fixed_window(user_id: str, endpoint: str,
                             limit: int, window_sec: int) -> bool:
    now = int(time.time())
    window_start = (now // window_sec) * window_sec
    key = f"rate:{user_id}:{endpoint}:{window_start}"

    pipe = r.pipeline()
    pipe.incr(key)
    pipe.expire(key, window_sec)
    results = pipe.execute()

    count = results[0]
    return count <= limit
```

Pros: O(1) time and space, two Redis ops, dead simple.
Cons: The boundary burst allows up to 2x the configured limit for a brief period.

### Algorithm 2: Sliding Window Log

Store a timestamp for every request in a sorted set. On each new request, remove timestamps
older than one window ago and count what remains.

```
Limit: 5 requests per 60-second window

Timeline:
  t=0s:   request 1  →  log=[0]             count=1  ALLOW
  t=15s:  request 2  →  log=[0, 15]         count=2  ALLOW
  t=30s:  request 3  →  log=[0, 15, 30]     count=3  ALLOW
  t=45s:  request 4  →  log=[0, 15, 30, 45] count=4  ALLOW
  t=55s:  request 5  →  log=[0..55]         count=5  ALLOW
  t=58s:  request 6  →  log=[0..55, 58]     count=6  DENY  (6 > 5)
  t=65s:  request 7  →  remove t=0 (65-60=5, so 0 < 5 is expired)
                         log=[15..55, 65]   count=5  ALLOW
```

No boundary burst. The window is always exactly the past 60 seconds.

```python
def is_allowed_sliding_log(user_id: str, endpoint: str,
                            limit: int, window_sec: int) -> bool:
    key = f"rate:log:{user_id}:{endpoint}"
    now = time.time()
    window_start = now - window_sec

    pipe = r.pipeline()
    # Remove all timestamps older than the window start
    pipe.zremrangebyscore(key, 0, window_start)
    # Count remaining (all within the last window_sec seconds)
    pipe.zcard(key)
    # Add current request timestamp as both score and member
    # Use str(now) + unique suffix to handle identical timestamps
    pipe.zadd(key, {f"{now}": now})
    pipe.expire(key, window_sec + 1)
    results = pipe.execute()

    count_before_add = results[1]
    return count_before_add < limit
```

Pros: perfectly accurate, no boundary burst.
Cons: memory grows with (limit × number of active users). At 100 req/min limit and 1M users,
stores 100M sorted set entries.

### Algorithm 3: Sliding Window Counter (Hybrid — Best Practical Choice)

Approximates the sliding window using two fixed window counters with a weighted interpolation.
Constant memory. Accuracy error is bounded at less than 1% for typical traffic patterns.

```
Limit: 100 requests per 60-second window

At t=75s (15s into the second window):
  Previous window [0–60s]:   80 requests  (complete)
  Current window  [60–120s]: 30 requests  (15/60 = 25% elapsed)

Overlap fraction of previous window still "in" the sliding window:
  remaining fraction = 1 - (15/60) = 0.75

Estimated count in the sliding [15s, 75s] window:
  = (previous_count × remaining_fraction) + current_count
  = (80 × 0.75) + 30
  = 60 + 30
  = 90  →  ALLOW (90 < 100)
```

```python
def is_allowed_sliding_counter(user_id: str, endpoint: str,
                                limit: int, window_sec: int) -> bool:
    now = time.time()
    current_window = int(now // window_sec)
    previous_window = current_window - 1
    elapsed_in_window = now % window_sec
    elapsed_fraction = elapsed_in_window / window_sec

    key_curr = f"rate:{user_id}:{endpoint}:{current_window}"
    key_prev = f"rate:{user_id}:{endpoint}:{previous_window}"

    pipe = r.pipeline()
    pipe.get(key_prev)
    pipe.get(key_curr)
    pipe.incr(key_curr)
    pipe.expire(key_curr, window_sec * 2)
    results = pipe.execute()

    prev_count = int(results[0] or 0)
    curr_count = int(results[1] or 0)
    # Note: curr_count is before the increment — we check before admitting

    remaining_prev_fraction = 1.0 - elapsed_fraction
    estimated = (prev_count * remaining_prev_fraction) + curr_count
    return estimated < limit
```

This is the algorithm Cloudflare uses at their edge. Two INCR operations, O(1) memory,
error bound less than 1% for smooth traffic. The right default choice for most systems.

### Algorithm 4: Token Bucket

Each user has a bucket that holds up to C tokens. Tokens refill at a constant rate R per second.
Each request consumes one token. If the bucket is empty, reject.

```
Bucket capacity: 10 tokens
Refill rate:      2 tokens/second

Time    Event              Tokens (before)    Result
 0s     4 requests         10                 6    (10 - 4)
 3s     6 requests         10 (refilled +6)   4    (10 - 6)
 4s     5 requests         6  (refilled +2)   1    (6 - 5)
 5s     3 requests         3  (refilled +2)   DENY 1 request (bucket has 3, need 3 for the batch
                                                   but second req after first is 2 then 1)
```

Allows bursts up to bucket capacity. Smooths sustained throughput. Ideal for APIs where short
bursts are expected (mobile clients waking up and syncing) but sustained throughput must be capped.

```python
# Token bucket implemented as an atomic Lua script in Redis
BUCKET_SCRIPT = """
local key        = KEYS[1]
local now        = tonumber(ARGV[1])
local rate       = tonumber(ARGV[2])   -- tokens per second
local capacity   = tonumber(ARGV[3])

local data = redis.call('HMGET', key, 'tokens', 'last_refill')
local tokens     = tonumber(data[1]) or capacity
local last_refill = tonumber(data[2]) or now

-- Refill tokens based on time elapsed
local elapsed = math.max(0, now - last_refill)
tokens = math.min(capacity, tokens + elapsed * rate)

if tokens >= 1 then
    tokens = tokens - 1
    redis.call('HMSET', key, 'tokens', tokens, 'last_refill', now)
    redis.call('EXPIRE', key, math.ceil(capacity / rate) + 1)
    return 1   -- allowed
else
    redis.call('HMSET', key, 'tokens', tokens, 'last_refill', now)
    return 0   -- denied
end
"""

script = r.register_script(BUCKET_SCRIPT)

def is_allowed_token_bucket(user_id: str, tokens_per_sec: float,
                             capacity: int) -> bool:
    key = f"rate:bucket:{user_id}"
    result = script(keys=[key], args=[time.time(), tokens_per_sec, capacity])
    return result == 1
```

The Lua script is mandatory: the read-modify-write on bucket state must be atomic, or two
concurrent requests on different Redis clients can both see a non-empty bucket and both proceed,
allowing double admission.

### Algorithm Comparison

```
                     Fixed      Sliding    Sliding     Token
                     Window     Log        Counter     Bucket
-------------------------------------------------------------------
Memory per key       O(1)       O(limit)   O(1)        O(1)
Boundary burst       Yes        No         ~No         No
Allows bursts        Brief      No         No          Yes
Accuracy             ~Low       High       ~High       High
Redis ops/request    2          4          4           1 (Lua)
Implementation       Trivial    Moderate   Moderate    Complex
Good default?        No         No         Yes         For burst APIs
```

## 5. Data Model

Rate limiters keep live state entirely in Redis. Only configuration lives in PostgreSQL.

```
Redis key schema:
  Fixed / Sliding counter:   rate:{user_id}:{endpoint}:{window_epoch}
  Sliding window log:        rate:log:{user_id}:{endpoint}           (sorted set)
  Token bucket:              rate:bucket:{user_id}:{endpoint}        (hash)
  Tier config cache:         rate:tier:{user_id}                     (string, 5min TTL)

Example keys:
  "rate:u123:search:28356"          →  counter = 42
  "rate:log:u123:login"             →  sorted set of float timestamps
  "rate:bucket:u123:upload"         →  hash {tokens: "7.3", last_refill: "1701388800.12"}
```

```sql
-- Rate limit tier configuration (loaded on startup, cached in Redis)
CREATE TABLE rate_limit_rules (
    id              SERIAL       PRIMARY KEY,
    rule_name       TEXT         UNIQUE NOT NULL,
    tier            TEXT         NOT NULL DEFAULT 'free',
    endpoint_prefix TEXT,                             -- NULL = all endpoints
    limit_count     INTEGER      NOT NULL,
    window_secs     INTEGER      NOT NULL,
    algorithm       TEXT         NOT NULL DEFAULT 'sliding_counter',
    burst_capacity  INTEGER,                          -- for token bucket only
    is_active       BOOLEAN      NOT NULL DEFAULT TRUE,
    updated_at      TIMESTAMPTZ  NOT NULL DEFAULT NOW()
);

-- Example rows
INSERT INTO rate_limit_rules (rule_name, tier, limit_count, window_secs) VALUES
  ('free_global',       'free',       60,    60),
  ('pro_global',        'pro',        600,   60),
  ('free_login',        'free',       5,     300),    -- 5 login attempts per 5 min
  ('free_search',       'free',       30,    60);

-- Violations log for abuse detection (written async)
CREATE TABLE rate_limit_violations (
    id              BIGSERIAL    PRIMARY KEY,
    client_key      TEXT         NOT NULL,   -- user_id or IP
    rule_name       TEXT         NOT NULL,
    request_path    TEXT,
    violated_at     TIMESTAMPTZ  NOT NULL DEFAULT NOW()
);
CREATE INDEX idx_violations_client ON rate_limit_violations(client_key, violated_at DESC);
```

## 6. Response Headers

Every rate-limited endpoint should return these headers on every response, not just on 429s.
Clients that respect these headers will back off automatically and write better retry logic.

```
HTTP/1.1 200 OK
X-RateLimit-Limit:      100
X-RateLimit-Remaining:  57
X-RateLimit-Reset:      1701388860    (Unix timestamp: when the current window resets)

HTTP/1.1 429 Too Many Requests
X-RateLimit-Limit:      100
X-RateLimit-Remaining:  0
X-RateLimit-Reset:      1701388860
Retry-After:            23            (seconds until client may retry)
Content-Type:           application/json

{
  "error": "rate_limit_exceeded",
  "limit": 100,
  "window": "60s",
  "retry_after": 23
}
```

## 7. Trade-offs Summary

| Concern                    | Solution                              | Trade-off                                          |
|----------------------------|---------------------------------------|----------------------------------------------------|
| Distributed consistency    | Redis INCR (atomic)                   | Redis becomes a latency dependency (~0.5ms)        |
| Redis unavailability       | Local in-memory fallback (1s TTL)     | Small over-admission during Redis downtime         |
| Burst traffic              | Token bucket algorithm                | More complex; requires Lua for atomicity           |
| Per-endpoint limits        | Endpoint segment in Redis key         | Key cardinality grows; monitor for explosion       |
| Multi-tier limits          | Config in PostgreSQL, cached in Redis | Config propagation lag up to cache TTL (5 minutes) |
| Strict 0-overage guarantee | Lua script for atomic check-and-incr  | Slightly higher Redis CPU per request              |

## 8. Senior Engineer Perspective

**The failure mode matters more than the algorithm.** When Redis goes down, your rate limiter
stops working. Fail open (allow all traffic) means an infrastructure failure removes your only
protection against API abuse — a Redis outage coinciding with a DDoS becomes catastrophic. Fail
closed (block all traffic) means your rate limiter causes an outage worse than the original problem.
The right choice depends on the threat model: for user-facing product APIs, fail open with alerting.
For authentication endpoints that protect against credential stuffing, fail closed.

**Local in-process caching eliminates most Redis round-trips.** Cache the user's rate limit tier
from PostgreSQL in application memory with a 60-second TTL. Cache the current window's counter in
memory for 1 second. In a 1-second window, a user makes at most a few requests — you can absorb
a 1-second overage to avoid the Redis round-trip. At 100,000 req/sec, this eliminates 99% of Redis
traffic.

**Distributed rate limiting has a fundamental race condition.** The check-then-act sequence (is
counter < limit? if yes, increment) is not atomic if done as two Redis commands. Between the GET
and the INCR, another server can increment the same key. Redis INCR is atomic by itself, so the
increment is safe — the danger is in making a decision before the increment. For limits like
1 req/10 seconds (authentication), use Lua to make check-and-increment atomic. For limits like
100 req/minute, the 1-2% overage from concurrent increments is acceptable without Lua.

**Rate limiting is a product decision wrapped in infrastructure.** What the limits are, whether
to warn users before blocking, how tiered limits should work, whether to show the limit in the
UI — these are product decisions. Engineers often implement the mechanics correctly but apply
wrong limits because they never asked the product team.
