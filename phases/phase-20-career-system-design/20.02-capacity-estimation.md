## 1. Why Capacity Estimation? (Theory)

Capacity estimation transforms vague requirements into concrete numbers that drive architectural decisions. "10 million users" means very different things:

- 10M users × 1 request/day = 116 RPS → a single server handles this
- 10M users × 100 requests/day = 11,600 RPS → you need caching and horizontal scaling
- 10M users × 10,000 requests/day = 1.16M RPS → you need serious distributed infrastructure

The numbers determine the design.

## 2. Essential Numbers to Memorize

```
TIME:
  1 day   = 86,400 seconds ≈ 10^5 seconds (close enough for estimation)
  1 month = 30 days = 2.6M seconds
  1 year  = 365 days = 31.5M seconds ≈ 3×10^7

STORAGE SIZES:
  1 KB = 10^3 bytes
  1 MB = 10^6 bytes
  1 GB = 10^9 bytes
  1 TB = 10^12 bytes
  1 PB = 10^15 bytes
  
  Short text (tweet):  280 bytes → call it 300 bytes
  Long text (article): 10-100 KB
  Image:               100 KB - 5 MB
  Video (1 min, 720p): 50-100 MB

TYPICAL REQUEST SIZES:
  JSON API response: 1 KB typical, 10 KB large
  Database row:      100 bytes - 1 KB
  HTTP header overhead: ~500 bytes

THROUGHPUT BENCHMARKS (order of magnitude):
  Simple in-memory cache read: 1 million ops/second
  Redis GET/SET: 100K-500K ops/second (single thread)
  PostgreSQL simple query (with index): 10K-50K queries/second
  PostgreSQL complex query: 1K-10K/second
  
LATENCY (approximate):
  L1 cache reference: 1 ns
  Main memory access: 100 ns
  SSD random read: 100 µs
  Network roundtrip (same datacenter): 1 ms
  Network roundtrip (cross-region): 100 ms
```

## 3. The Estimation Process

```
STEP 1: Daily Active Users → Requests Per Second
  
  Monthly active users (MAU): 100M
  Daily active users (DAU): typically 10-20% of MAU → 10-20M DAU
  Use: 10M DAU (conservative)
  
  Average requests per user per day: depends on app
    Social media: 50-100 requests/day (scrolling, likes, posts)
    E-commerce: 5-10 requests/day
    Chat: 100-200 requests/day
  
  FORMULA:
  RPS = (DAU × requests_per_user) / 86,400 seconds
  
  EXAMPLES:
  Twitter: 300M DAU × 100 req/day / 86,400 = 347,000 RPS
  WhatsApp: 2B DAU × 50 req/day / 86,400 = 1.16M RPS
  Small SaaS: 10K DAU × 20 req/day / 86,400 = 2.3 RPS (one server!)

STEP 2: Peak vs. Average
  Traffic is NOT uniform:
  Average RPS × peak factor = peak RPS
  
  Typical peak factors:
  E-commerce: 3-5x average (flash sales, business hours)
  Social media: 2-3x average (morning/evening usage)
  Always design for peak, not average.

STEP 3: Read/Write Ratio
  Most systems are read-heavy:
  Social media: 95% reads, 5% writes (10:1 to 100:1)
  E-commerce: 90% reads, 10% writes
  This guides caching strategy and database read replica decisions.

STEP 4: Storage Per Day / Year
  Writes per second × record size × seconds per day = daily storage
  
  Twitter example:
    500M tweets/day × 300 bytes = 150 GB/day
    × 365 days = 55 TB/year
    × 5 years = 275 TB total (just tweet text)
    Media would be 100-1000x more.
```

## 4. Worked Examples

### Twitter Feed

```
GIVEN: Twitter-like social media
REQUIREMENTS: 300M MAU, 150M DAU, 500M tweets/day, 500M timeline reads/day

TRAFFIC:
  Tweet write: 500M / 86,400 ≈ 6,000 RPS
  Timeline read: 500M / 86,400 ≈ 6,000 RPS

STORAGE:
  Text: 500M tweets/day × 280 bytes = 140 GB/day
  Images: 50% of tweets have images, avg 500KB
          = 250M × 500KB = 125 TB/day (!!)
  Videos: 10% have videos, avg 10MB = 500 TB/day (!!)
  → Content delivery and storage is the primary challenge

KEY INSIGHTS:
  - 6,000 read RPS is manageable with Redis cache
  - Media storage at petabyte scale needs CDN + object storage (S3)
  - Fan-out problem: 1 tweet → sent to millions of followers' timelines
    This is the famous Twitter "celebrity problem"
```

### Uber (ride-sharing location tracking)

```
GIVEN: 1M drivers, 10M riders
REQUIREMENTS: Real-time location tracking for drivers

LOCATION UPDATES:
  1M active drivers × 1 update every 5 seconds
  = 200,000 location writes/second
  
KEY INSIGHTS:
  200K writes/second is too high for PostgreSQL (max ~50K).
  Need: Redis (in-memory, very fast writes) or specialized
        geospatial databases.
  
  Driver locations: 1M × 32 bytes (lat/lng/timestamp/driver_id) = 32 MB
  All driver locations fit in RAM → in-memory is appropriate!
```

### URL Shortener

```
GIVEN: Bit.ly-like URL shortener
REQUIREMENTS: 500M URLs shortened (total), 100:1 read/write ratio

WRITES: 500M URLs / (5 years × 365 days × 86,400 seconds) ≈ 3 writes/second
READS: 300 reads/second (100:1 ratio)

STORAGE:
  500M records × 500 bytes = 250 GB → fits in a single PostgreSQL instance!
  
KEY INSIGHTS:
  This is a SMALL system. Reads are manageable without sharding.
  Add Redis cache for popular URLs.
  Single PostgreSQL primary + cache is the right architecture.
  (Interviewer: "What if we scaled to YouTube's size?" → then consider sharding)
```

NUMBERS TO KNOW BY HEART:

  A well-optimized server: 10,000-100,000 RPS

## 6. Senior Engineer Perspective

**Round generously and explain your assumptions:**

In interviews, exact numbers don't matter — order of magnitude does. "I'll assume 100 million users, 10% daily active, 10 requests per day each. That's 100M × 0.1 × 10 / 86,400 = roughly 1,200 requests per second." State your assumptions, show your math, arrive at a number. The interviewer can correct your assumptions if they're off.

**Estimation reveals the design:**

The most valuable output of capacity estimation is not the numbers themselves — it's the design implications. "At 1,200 RPS, a single PostgreSQL instance can handle reads directly. At 120,000 RPS, we need a caching layer." The estimation tells you which problems you actually have to solve.
