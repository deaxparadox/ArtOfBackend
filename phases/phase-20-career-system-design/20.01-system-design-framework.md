## 1. What Is a System Design Interview? (Theory)

A system design interview asks you to design a large-scale system — a URL shortener, a Twitter feed, a payment system — in 45-60 minutes. There's no single right answer. The interviewer is evaluating your ability to:

- Ask the right clarifying questions
- Think about scale and constraints
- Make and justify architectural trade-offs
- Communicate your thinking clearly
- Know when to use which technologies

**The interviewer's perspective:**

```
WHAT INTERVIEWERS WANT TO SEE:
  ✓ You drive the conversation (don't wait for prompts)
  ✓ You identify and resolve ambiguity before diving in
  ✓ You know the capacity orders of magnitude
  ✓ You can draw a coherent high-level architecture
  ✓ You understand data modeling for your chosen store
  ✓ You identify bottlenecks before they ask
  ✓ You can discuss trade-offs without being defensive

WHAT INTERVIEWERS DON'T WANT:
  ✗ Silence for 5 minutes while you think
  ✗ Jumping to implementation details before requirements
  ✗ Assuming requirements that weren't stated
  ✗ Claiming you'll use a technology without knowing why
  ✗ Refusing to make decisions ("it depends" with no follow-through)
```

## 2. The Framework: 6 Steps

### Step 1: Clarify Requirements (5 minutes)

Never start designing before you understand what you're building.

```
FUNCTIONAL REQUIREMENTS: What must the system do?
  "So to confirm, the core features are:
   1. Shorten a URL (POST /shorten)
   2. Redirect short URL to original (GET /{code})
   3. That's it — no analytics, no user accounts needed for now?"

NON-FUNCTIONAL REQUIREMENTS: How must it do it?
  Scale:        "How many users? How many URL creations per day? Redirects?"
  Availability: "Can we accept brief downtime or must it be 99.99%?"
  Latency:      "What's the acceptable redirect latency? Sub-100ms?"
  Consistency:  "If a URL is shortened, does every redirect immediately work?"
  Durability:   "Are shortened URLs permanent? What's the retention policy?"
  
WRITE THESE DOWN during the interview.
```

### Step 2: Capacity Estimation (5 minutes)

```
EXAMPLE: URL shortener capacity

Traffic:
  Reads:  100M redirects/day = 100M / 86,400 ≈ 1,200 RPS
  Writes: 1M new URLs/day = 1M / 86,400 ≈ 12 RPS
  Read/Write ratio: 100:1 (read-heavy)

Storage:
  1M new URLs/day × 365 days × 5 years = 1.8B records
  Each record: 100 bytes (short code + original URL + metadata)
  Storage: 1.8B × 100 bytes = 180 GB (manageable)

Bandwidth:
  Reads: 1,200 RPS × 100 bytes = 120 KB/s (trivial)

INSIGHT: This is read-heavy with small data per record.
         Great for caching. Single database node is sufficient for writes.
```

### Step 3: High-Level Design (10 minutes)

```
Draw the components:

Client → Load Balancer → App Servers → Database
                     ↓
                  Cache (Redis)

For URL shortener:
  POST /shorten → App Server → Generate code → Store in DB → Return short URL
  GET /{code}   → App Server → Check Cache → If miss: Query DB → Redirect
```

### Step 4: Data Model (5 minutes)

```sql
-- URL shortener schema

CREATE TABLE short_urls (
    code        CHAR(7) PRIMARY KEY,          -- the short code: "abc1234"
    original_url TEXT NOT NULL,               -- the full original URL
    user_id      BIGINT,                       -- nullable (anonymous allowed)
    created_at   TIMESTAMPTZ DEFAULT NOW(),
    expires_at   TIMESTAMPTZ,                 -- optional expiry
    click_count  BIGINT DEFAULT 0
);

CREATE INDEX idx_su_user ON short_urls(user_id);
CREATE INDEX idx_su_expires ON short_urls(expires_at) WHERE expires_at IS NOT NULL;
```

### Step 5: Detailed Design and Deep Dives (20 minutes)

Focus on the hard parts. For URL shortener:

```
CODE GENERATION: How do you generate 7-char unique codes?
  Option A: Random → check if exists → retry if collision
    Pro: simple
    Con: rare collision risk at scale
  
  Option B: Base62(auto-increment ID)
    Pro: guaranteed unique, deterministic
    Con: sequential (predictable codes)
  
  Option C: Hash (MD5 of URL) → take first 7 chars
    Pro: same URL always same code (deduplication)
    Con: hash collisions require collision handling

CACHING STRATEGY:
  Cache redirects in Redis with TTL.
  Cache the most popular URLs (80/20 rule: 20% of URLs = 80% of traffic).
  LRU eviction policy.
  What's the hit rate? If most URLs are accessed rarely, cache is less effective.

RATE LIMITING:
  Limit URL creation to prevent abuse.
  100 URL creates/hour per IP.
```

### Step 6: Identify Bottlenecks and Scale (5 minutes)

```
"The bottleneck at scale is:
  - Database reads: solved by Redis cache (cache hit rate ~80-90%)
  - Database writes: 12 RPS is trivial for any database
  
If we need to scale to 10x:
  - Add more app server instances (stateless, easy to scale)
  - Add Redis replicas for cache
  - Database read replicas for cache misses
  
If we need 100x (1.2M redirects/second):
  - Database sharding by code prefix
  - Global CDN for cache (edge caching)
```

## 3. Communication Tips

```
TALKING WHILE YOU DRAW:
  "I'm drawing the high-level components first. 
   The client hits a load balancer, which routes to one of several 
   app server instances. They all talk to a shared Redis cache and 
   a primary database. Let me explain each choice..."

MAKING TRADE-OFFS EXPLICIT:
  "I could use a NoSQL document store here for flexibility, 
   but since our access pattern is simple key-value lookup by short code,
   a relational database with an index works just as well and is simpler 
   operationally. I'll go with PostgreSQL."

ASKING FOR DIRECTION:
  "I could go deeper on the code generation algorithm or on caching.
   Which area is more interesting to you?"
  
RECOVERING FROM MISTAKES:
  "Actually, thinking about this more — storing click counts in the 
   main table will cause write contention at high traffic. 
   Better to use a separate analytics table or a message queue for counts."
```

## 4. Common Mistakes

```
MISTAKE 1: PREMATURE OPTIMIZATION
  "Before knowing scale, already designing sharded databases."
  Ask first, design accordingly.

MISTAKE 2: FORGETTING THE SIMPLE SOLUTION
  "This could be a single PostgreSQL database with an index."
  Complex distributed systems have high operational cost.
  Start simple, add complexity when justified by requirements.

MISTAKE 3: NOT JUSTIFYING CHOICES
  "I'll use Kafka." (why?)
  Every choice should have a reason:
  "I'll use Kafka because we need durable message storage and
   multiple consumer groups reading from the same event stream."

MISTAKE 4: DESIGNING IN SILENCE
  Think out loud. Even saying "I'm considering X vs Y" shows thinking.

MISTAKE 5: IGNORING FAILURE MODES
  "What happens if the database goes down during a write?"
  Always think about failure scenarios for critical paths.
```

## 5. Senior Engineer Perspective

**The design interview is a conversation, not a presentation:**

The best system design interviews feel like a senior engineer conversation, not an oral exam. You're working through a problem together. Ask the interviewer for feedback: "Does this approach make sense for your use case?" Adjust based on their response.

**Trade-offs are the substance of the interview:**

There is no "correct" answer to system design questions. The question is which trade-offs you choose and whether you understand them. Saying "I chose a relational database because our queries are complex and we need ACID guarantees — if we needed to scale writes to 100K/s I'd consider switching to DynamoDB" demonstrates the right thinking.
