# 20.09 — Design a Search Autocomplete System

## Why This Problem Is Hard

Autocomplete looks trivial: "user types a prefix, return matching suggestions." The difficulty is the operational envelope:

- **Latency**: The result must appear before the user finishes typing the next character. Target is under 100ms end-to-end, which means the data service must respond in under 20ms.
- **Scale**: Google Search handles ~8.5 billion queries per day. Each keystroke fires a request. A 5-character query fires 5 requests. You are now at 40 billion requests per day for autocomplete alone.
- **Freshness**: Trending queries ("earthquake in city X") must appear in suggestions within minutes, not weeks.
- **The trie is hard to distribute**: The trie is a tree structure. Trees do not shard naturally by key hash. A naive approach loads the entire trie on one machine; at Google's scale that does not fit in RAM.
- **Personalization**: User A searching for "py" should see "python" before "python monty" based on their history, while User B (a biologist) should see "pyrimidine."

## Data Model

### Trie Node Structure

```python
class TrieNode:
    children: dict[str, TrieNode]  # char -> child node
    top_k: list[tuple[str, int]]   # [(query, frequency), ...] sorted descending
    # top_k is stored at EVERY node, not just leaves.
    # This trades memory for O(1) lookup at any prefix depth.

class Trie:
    root: TrieNode

    def insert(self, query: str, frequency: int):
        node = self.root
        for char in query:
            if char not in node.children:
                node.children[char] = TrieNode()
            node = node.children[char]
            # Update top_k at each node along the path
            node.update_top_k(query, frequency)

    def search(self, prefix: str) -> list[str]:
        node = self.root
        for char in prefix:
            if char not in node.children:
                return []
            node = node.children[char]
        return [q for q, _ in node.top_k]  # O(1): already computed
```

The critical insight: storing `top_k` at every node is expensive in memory but eliminates traversal cost at query time. Without it, you would need to traverse the entire subtree under the prefix to find the top results — O(n) per query.

### Serialized Storage Format

The trie is serialized periodically to the Trie DB for durability:

```sql
CREATE TABLE trie_prefixes (
    prefix      VARCHAR(20) PRIMARY KEY,
    suggestions JSONB NOT NULL,     -- [{"query": "search", "freq": 9823}, ...]
    updated_at  TIMESTAMP NOT NULL
);

-- Example row:
-- prefix: "searc"
-- suggestions: [
--   {"query": "search engine", "freq": 9823456},
--   {"query": "search bar",    "freq": 4521123},
--   ...5 total
-- ]
```

This is a denormalized, read-optimized store. The trie structure lives in the cache servers; the DB holds snapshots for recovery.

## Deep Dive: The One Hard Problem — Sub-100ms at Scale

### Why the Trie Cannot Live on One Machine

A trie for the English web's query vocabulary:
- English vocabulary: ~170,000 words. Common queries: ~50 million distinct strings.
- Each trie node: ~1 KB including top_k list and children pointers.
- Full trie: ~50 million nodes * 1 KB = 50 GB.

50 GB fits in RAM on a single large machine. But at 870,000 req/sec, a single machine cannot handle the load. You need horizontal scaling.

### Sharding the Trie by Prefix

Trees do not shard by key hash (hashing "search" and "searched" sends them to different nodes, so you cannot answer "all queries starting with 'searc'" from one node). Instead, shard by prefix range:

```
Shard 1:  prefixes a -- f   (high frequency: "apple", "amazon", "facebook")
Shard 2:  prefixes g -- m   (high frequency: "google", "instagram")
Shard 3:  prefixes n -- z   (high frequency: "news", "youtube")
```

The client library knows the prefix-to-shard mapping. A request for prefix "sea" routes to Shard 3.

**Imbalance problem:** "a" queries vastly outnumber "z" queries. Solution: split at character *frequency*, not alphabet. Measure actual query distribution and cut shards at equal query volume boundaries. Reshard quarterly.

### CDN Caching of Popular Prefixes

Analysis of query logs shows that the top 1000 prefixes account for ~90% of all autocomplete requests. These are short prefixes like "a", "an", "the", "ho", "wh", etc.

Cache these at the CDN edge:

```
Cache key:   "autocomplete:en:{prefix}"
Cache TTL:   60 seconds (balance freshness vs. hit rate)
Hit rate:    ~90% for global traffic

At 870K req/sec and 90% cache hit rate:
  CDN handles:  783,000 req/sec
  Origin sees:   87,000 req/sec
```

This is the single most impactful optimization. The autocomplete service only needs to handle 10% of traffic.

### Query Aggregation Pipeline (Freshness)

The trie is not updated in real-time per query — that would require locking trie nodes on every write. Instead:

```
1. User submits query "ukraine war"
2. Query is logged to Kafka topic "query-log" (fire-and-forget, async)
3. Spark Streaming job reads from Kafka, aggregates query counts
   in a 1-hour tumbling window
4. At the end of each window, a Trie Builder job:
   a. Reads top-k queries per prefix from the aggregation result
   b. Merges with the previous trie (to avoid losing long-tail history)
   c. Applies decay to old queries (multiply frequency by 0.9 per day)
   d. Rebuilds the trie snapshot
   e. Publishes the new snapshot to the Trie Cache nodes via a
      "trie_update" Kafka topic

Latency from query to appearing in suggestions: ~1 hour
For true trending (minutes), use a separate "trending queries" service
that bypasses the trie and injects hot queries directly from Kafka.
```

## Alternative: Elasticsearch with Edge N-Gram Tokenizer

For teams without the engineering capacity to build a trie service, Elasticsearch provides a near-equivalent solution:

```json
PUT /queries
{
  "settings": {
    "analysis": {
      "analyzer": {
        "autocomplete": {
          "type": "custom",
          "tokenizer": "standard",
          "filter": ["lowercase", "autocomplete_filter"]
        }
      },
      "filter": {
        "autocomplete_filter": {
          "type": "edge_ngram",
          "min_gram": 1,
          "max_gram": 20
        }
      }
    }
  },
  "mappings": {
    "properties": {
      "query":     { "type": "text", "analyzer": "autocomplete" },
      "frequency": { "type": "integer" }
    }
  }
}
```

Edge n-grams tokenize "search" into ["s", "se", "sea", "sear", "searc", "search"] at index time. A match query on the `autocomplete` analyzer hits all documents whose query starts with the typed prefix. Sort by frequency descending.

**Trade-offs vs. trie:**

| | Trie | Elasticsearch |
|---|---|---|
| Latency | 2-5ms (in-memory) | 10-50ms |
| Operational complexity | High (custom service) | Medium (managed or self-hosted) |
| Fuzzy matching | Needs extra work | Built-in (fuzziness param) |
| Scaling | Manual shard management | Automatic shard routing |

For most products, Elasticsearch is the right answer. Build the custom trie only at Google/Baidu scale.

## Personalization Layer

The global top-k trie gives the same suggestions to everyone. Personalization blends:

```
final_suggestions = blend(global_top_k, user_history_top_k, weight=0.3)
```

```python
def get_suggestions(prefix: str, user_id: str) -> list[str]:
    global_results  = trie.search(prefix)           # top 5 globally
    user_results    = user_history.search(prefix, user_id)  # top 5 from user's past queries
    return rerank(global_results, user_results, user_weight=0.3)[:5]
```

User history is stored in a per-user inverted index (Redis sorted set works well: key = `user:{id}:prefix:{prefix}`, score = frequency):

```
ZADD user:42:prefix:py 10 "python tutorial"
ZADD user:42:prefix:py  3 "pyrimidine structure"
ZREVRANGE user:42:prefix:py 0 4  -> ["python tutorial", "pyrimidine structure"]
```

## Trade-offs Table

| Decision | Option A | Option B | Notes |
|---|---|---|---|
| Data structure | Trie with top_k at each node | Elasticsearch edge n-gram | Trie is faster; ES is simpler to operate |
| Update frequency | Real-time per query | Batch (hourly) | Batch wins for throughput; real-time adds write contention |
| Sharding | Prefix range | Hash of prefix | Range sharding required (hash breaks prefix locality) |
| Caching | CDN for top-1000 prefixes | Application-level cache | CDN first; dramatically reduces origin load |
| Personalization | Client-side blend | Server-side blend | Server-side (client does not hold user history) |

## Senior Engineer Perspective

**What juniors get wrong:**

1. Proposing a SQL LIKE query: `SELECT query FROM queries WHERE query LIKE 'sea%'`. This is a full table scan on billions of rows. Even with a B-tree index, it scans the prefix range but cannot return top-k by frequency efficiently.
2. Sharding the trie by hash. Hashing destroys prefix locality — the entire point of the trie.
3. Not accounting for CDN caching. The CDN is the actual scaling mechanism. The trie handles the long tail.

**The debounce optimization on the client:**
The client should not fire a request on every keystroke. A 50ms debounce means:
- User types "s", "e", "a", "r", "c", "h" rapidly (faster than 50ms per key)
- Only one request fires: for "search"
- This reduces request volume by 4-5x at no cost to user experience

**Why decay matters:**
Without frequency decay, "gangnam style" from 2012 would still appear in music-related autocomplete forever. Apply exponential decay to historical queries: multiply frequency by 0.9 for each day elapsed. Queries that are not searched fade out naturally.
