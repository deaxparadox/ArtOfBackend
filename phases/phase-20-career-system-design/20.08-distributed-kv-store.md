# 20.08 — Design a Distributed Key-Value Store (Redis / DynamoDB)

## Why This Problem Is Hard

A single-machine hash map is trivial. A distributed one forces you to confront the fundamental limits of distributed systems:

- **Sharding**: How do you split the keyspace across N nodes? How do you rebalance when a node is added or removed without rehashing everything?
- **Replication**: How many copies of each key do you keep? How do you keep them in sync? What happens when replicas disagree?
- **Failure detection**: How does the cluster discover that a node has died without a centralized coordinator that itself can fail?
- **CAP theorem**: When a network partition occurs, you must choose between consistency and availability. There is no middle ground in the worst case.
- **Conflict resolution**: When two nodes accept a write to the same key during a partition, which value wins when the partition heals?

```
        |            |            |
        +------------+------------+
                     |
           +---------+---------+
           |  Client Library   |
           |  (consistent hash,|
           |   retry, routing) |
           +---------+---------+
                     |
    +----------------+----------------+
    |                |                |
+---+----+      +----+---+      +-----+--+
| Node A |      | Node B |      | Node C |
|        |<---->|        |<---->|        |
| data   |      | data   |      | data   |

+--------+      +--------+      +--------+
    |                |                |
    +----------------+----------------+
             Gossip Protocol
             (failure detection,

```

The client library holds the consistent hash ring in memory. It computes which node owns a key locally — there is no central router. This removes the coordinator as a bottleneck and single point of failure.

## Data Model

Each key maps to a structured record internally:

```
Key (string, max 1KB)
  |
  v
{
  value:          bytes (up to 1MB),
  version_vector: { node_id: logical_clock, ... },
  ttl:            Unix timestamp or null,
  metadata: {
    created_at:   Unix timestamp,
    updated_at:   Unix timestamp,
    checksum:     CRC32
  }
}
```

The version vector (also called a vector clock) tracks causality. Each node increments its own entry when it accepts a write. On merge, a vector clock that is strictly greater than another dominates; if neither dominates, it is a conflict.

## Deep Dive: The Three Hard Problems

### 1. Consistent Hashing

**Naive modulo hashing** (`node = hash(key) % N`) breaks when N changes. Adding one node rehashes almost every key.

**Consistent hashing** maps both keys and nodes onto a circular ring of hash values (0 to 2^64 - 1). A key is owned by the first node whose position on the ring is >= the key's hash (clockwise lookup).

```
                    0
                    |
         +----------+----------+
         |                     |
  Node A (pos 90)          Node B (pos 210)
         |                     |
         +----------+----------+
                    |
              Node C (pos 330)

Key "foo" hashes to position 150 -> owned by Node B (next clockwise node)
Key "bar" hashes to position 320 -> owned by Node C
Key "baz" hashes to position 350 -> owned by Node A (wraps around)
```

**Virtual nodes (vnodes):** A single physical node is represented by V positions on the ring (e.g., V=150). This achieves:

- Uniform key distribution even when nodes have different capacities.
- When a node is removed, its keys scatter to multiple nodes rather than piling onto one neighbor.
- Adding a node steals a small fraction of keys from many nodes, not all keys from one.

```
Physical node mapping with vnodes (V=3 shown, V=150 in practice):

  Node A: positions [45, 190, 310]
  Node B: positions [80, 230, 350]
  Node C: positions [120, 270, 30]

Adding Node D: it claims some of A's slots, some of B's, some of C's.
Only ~k/N keys are remapped (k = total keys, N = number of nodes).
```

### 2. Replication and Quorum

Each key is replicated to the next N nodes clockwise on the ring (N = replication factor, typically 3).

**Quorum reads/writes** give tunable consistency:

```
N = 3  (total replicas)
W = 2  (writes that must succeed before acknowledging the client)
R = 2  (replicas that must respond before returning a read)

If W + R > N  (2 + 2 > 3): strong consistency guaranteed.
  A read must overlap with the latest write by at least one replica.

If W = 1, R = 1: maximum availability, eventual consistency.
If W = 3, R = 1: every write goes to all replicas (slow writes, fast reads).
```

**Sloppy quorum + hinted handoff:**
During a network partition, the required W replicas may be unreachable. With a strict quorum, the write fails. With a sloppy quorum, the write goes to W *any* available nodes, including nodes that are not the canonical owners. The node that accepted the write on behalf of the unavailable node stores a "hint" — metadata indicating which node should eventually receive this write. When the unavailable node recovers, hinted data is forwarded to it.

This trades strong consistency for availability during partitions — the DynamoDB/Cassandra default.

### 3. Gossip Protocol for Failure Detection

Centralized failure detection (e.g., a single heartbeat monitor) is a single point of failure. Gossip is a decentralized alternative:

```
Round 1: Node A randomly selects Node B and Node C. Shares its membership view.
Round 2: Node B shares with Node D and Node E. Node C shares with Node A and Node F.
...

After O(log N) rounds, all nodes have converged on the same membership view.
```

Each node maintains a heartbeat counter. If a node's counter has not incremented for T seconds (e.g., 3 rounds), it is marked suspicious. After another timeout, it is marked dead and its vnodes are transferred to neighbors.

**Phi accrual failure detector** (used by Cassandra): Instead of a binary alive/dead judgment, it outputs a suspicion level phi (0 to infinity). The application sets a threshold (e.g., phi > 8 = suspected dead). This adapts to variable network conditions.

## Conflict Resolution

When two writes to the same key happen concurrently on different replicas:

**Last-Write-Wins (LWW):**
```
Write 1: key="x", value="a", timestamp=1718000000.001
Write 2: key="x", value="b", timestamp=1718000000.002

Winner: Write 2 (higher timestamp)
Problem: clocks drift. Timestamp 2 could have happened before timestamp 1 physically.
```
LWW is simple and used by Redis and Cassandra (optionally). It can silently discard writes.

**Vector clocks:**
```
Initial state:  x = "a", vc = {A:1}
Node A updates: x = "b", vc = {A:2}   (dominates initial)
Node B updates: x = "c", vc = {B:1}   (concurrent with A:2 -- conflict!)

Neither vector clock dominates the other.
Resolution: return both values to the client ("siblings"), let the application merge.
```
DynamoDB originally used vector clocks but moved to LWW for simplicity. Riak still uses vector clocks with client-side merge.

## Comparison: Redis vs. DynamoDB vs. Cassandra

| Property | Redis | DynamoDB | Cassandra |
|---|---|---|---|
| Primary use case | Cache, session, pub/sub | OLTP, serverless | Wide-column, time-series |
| Consistency model | Strong (single-node), eventual (cluster) | Eventual (default), strong (opt-in) | Tunable (ONE to ALL) |
| Sharding | Hash slots (16384) | Automatic partition key | Consistent hash + vnodes |
| Replication | AOF + RDB + replica set | Managed (3 AZs) | Configurable RF |
| Failure detection | Sentinel / Cluster gossip | Managed | Gossip (phi accrual) |
| Conflict resolution | LWW (Redis does not expose conflicts) | LWW | LWW or manual |
| CAP placement | CP (cluster mode with quorum) | AP (default) | AP (default) |
| Query model | Key only | Key + sort key | Partition key + clustering key |
| Typical latency | < 1ms | 1-10ms | 2-10ms |

## CAP Theorem Placement

```
       Consistency
            |
            |   CP systems: HBase, Zookeeper, etcd
            |   (refuse writes during partition)
            |
            +---------------------------
            |   CA systems: single-node RDBMS
            |   (impossible in distributed setting)
            |
            +---------------------------
            |   AP systems: Cassandra, DynamoDB, CouchDB
                (accept writes, may return stale data)
            Availability
```

No distributed system can be both fully consistent and fully available during a network partition. The real question is not CP vs. AP but *how much staleness is acceptable* and *how long until convergence*.

## Trade-offs Table

| Decision | Option A | Option B | Notes |
|---|---|---|---|
| Sharding | Modulo hashing | Consistent hashing | Consistent hashing wins for elasticity |
| Consistency | Strong quorum (W+R>N) | Eventual (W=1, R=1) | Choose based on use case |
| Conflict resolution | LWW | Vector clocks | LWW is simpler; vector clocks are safer |
| Failure detection | Centralized heartbeat | Gossip | Gossip scales better, no SPOF |
| Replication | Synchronous | Asynchronous (hinted handoff) | Async = higher availability, staleness risk |

## Senior Engineer Perspective

**What juniors get wrong:**

1. Thinking consistent hashing is about "consistent" as in "strong consistency." It is about the ring being *consistently mapped* — it is a partitioning scheme, not a consistency guarantee.
2. Setting W=1 for writes because it is faster, then being surprised by data loss when a node dies before replication completes.
3. Not understanding that sloppy quorum means "eventual" in a concrete, measurable way: reads can return stale data for up to the replication lag window (often milliseconds, sometimes seconds under load).

**Why DynamoDB is not just a key-value store:**
DynamoDB supports a sort key, making each partition a sorted list of items. This supports range queries like `get all orders for user_123 between Jan 1 and Jan 31`. The partition key controls routing; the sort key controls ordering within a partition. Understanding this distinction separates junior from senior DynamoDB usage.

**The write amplification problem:**
With N=3 and W=2, every write generates 3 physical writes (one per replica). At 50,000 writes/sec with 10 KB average value, that is 1.5 GB/s of write traffic across the cluster. This is why nodes need high-bandwidth NICs and SSDs, not spinning disks.
