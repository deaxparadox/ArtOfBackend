## What Makes This System Hard

A news feed has an extreme read-to-write ratio (roughly 100:1), which sounds like it favors
pre-computation. Pre-compute the feed for every user on every write, and reads become instant.
The problem is called **the celebrity fan-out**.

When a user with 10 million followers posts, "pre-compute the feed for everyone" means writing to
10 million feed caches simultaneously. At 100,000 writes per second, that takes 100 seconds — far
too slow. But if you skip pre-computation for celebrity posts, your read path must query the
database at feed-load time, which explodes at 100,000 feed loads per second.

There is no solution that is both simple and correct at scale. The answer is a **hybrid fan-out**:
push model for regular users, pull model for celebrities. The complexity is in the merge step at
read time. Understanding this trade-off in depth — and being able to explain it with numbers — is
what separates a senior from a junior answer in a system design interview.

## 1. Requirements Clarification

### Functional Requirements
- Users can create posts (text, images, video)
- Users can follow other users
- Home feed: the user sees recent posts from accounts they follow
- Infinite scroll with cursor-based pagination
- Chronological ordering (discuss ranking/algorithmic as an extension)
- Like and comment (mention, but defer deep design)

### Non-Functional Requirements
- **Feed load latency**: under 200ms at p99
- **Write throughput**: 10 million posts per day
- **Read throughput**: 1 billion feed loads per day (100:1 read-heavy)
- **Consistency**: eventual — a new post can take 2-5 seconds to appear in all feeds
- **Scale**: 1 billion users, follow counts ranging from 0 to 50 million

### Clarifying Questions to Ask in an Interview
- Chronological or algorithmic feed?
- What is the follower count threshold for "celebrity"?
- How many posts should a feed page contain?
- Do we need read-your-own-writes consistency?
- Are deleted posts removed from existing feeds immediately or lazily?

### Out of Scope (for first pass)
- ML ranking and personalization
- Ad injection
- Stories and ephemeral content

## 2. Capacity Estimation

```
Users:           1 billion total, 300M daily active (DAU)
Posts per day:   10 million  →  ~115 posts/sec average, 1,000/sec peak
Feed loads/day:  1 billion   →  ~11,600/sec average, 100,000/sec peak

Per feed page:
  20 post IDs returned
  Post metadata fetched from cache: ~500 bytes/post
  Per feed response: ~10 KB

Follow graph:
  1B users × avg 200 follows = 200B directed edges
  Storage at 16 bytes/edge (follower_id + followee_id): 3.2 TB
  → Cassandra or sharded PostgreSQL

Fan-out analysis:
  Average user has 200 followers:
    Post → 200 Redis ZADD operations → completes in < 100ms
  
  Celebrity has 10M followers:
    Post → 10M Redis ZADD operations
    At 500K ops/sec: 20 seconds (unacceptable)
    → Cannot use fan-out-on-write for celebrities

Feed cache (Redis):
  300M DAU × feed of 200 post IDs × 8 bytes/ID = 480 GB
  Manageable in a Redis cluster; use 7-day TTL
```

                                    │  (Snowflake ID) │
                                    └───────┬────────┘
                                            │
                                    ┌───────▼────────┐
                                    │  Kafka topic:   │

## 4. Data Model

### PostgreSQL

```sql
-- Posts (Snowflake IDs for global uniqueness + time-ordering)
CREATE TABLE posts (
    post_id       BIGINT       PRIMARY KEY,   -- Snowflake ID
    user_id       BIGINT       NOT NULL,
    content       TEXT,
    media_urls    TEXT[],
    like_count    BIGINT       NOT NULL DEFAULT 0,
    comment_count BIGINT       NOT NULL DEFAULT 0,
    is_deleted    BOOLEAN      NOT NULL DEFAULT FALSE,
    created_at    TIMESTAMPTZ  NOT NULL DEFAULT NOW()
);
CREATE INDEX idx_posts_user_id ON posts(user_id, created_at DESC)
    WHERE is_deleted = FALSE;

-- Follow graph
CREATE TABLE follows (
    follower_id  BIGINT       NOT NULL,
    followee_id  BIGINT       NOT NULL,
    created_at   TIMESTAMPTZ  NOT NULL DEFAULT NOW(),
    PRIMARY KEY (follower_id, followee_id)
);
-- "Who follows X?" needed for fan-out
CREATE INDEX idx_follows_followee ON follows(followee_id, follower_id);
-- "Who does X follow?" needed for read-path merge
CREATE INDEX idx_follows_follower ON follows(follower_id, followee_id);

-- Users (follower_count maintained asynchronously)
CREATE TABLE users (
    user_id        BIGINT       PRIMARY KEY,
    username       TEXT         NOT NULL UNIQUE,
    follower_count BIGINT       NOT NULL DEFAULT 0,
    following_count BIGINT      NOT NULL DEFAULT 0,
    is_celebrity   BOOLEAN      NOT NULL DEFAULT FALSE,  -- updated by background job
    created_at     TIMESTAMPTZ  NOT NULL DEFAULT NOW()
);
```

### Redis

```
Feed cache (pre-built per user):
  Key:   feed:{user_id}
  Type:  Sorted Set
  Score: Unix timestamp (float)
  Value: post_id (string)
  TTL:   7 days (expire cold feeds)

  Examples:
    ZADD feed:123 1701388800.0 "456"
    ZADD feed:123 1701388750.0 "789"
    ZREVRANGE feed:123 0 19         -- top 20 most recent post IDs
    ZREVRANGEBYSCORE feed:123 +inf 1701302400 LIMIT 0 20   -- cursor-based pagination

Celebrity post index (pull-based):
  Key:   posts:{celebrity_user_id}
  Type:  Sorted Set
  Score: Unix timestamp
  Value: post_id
  TTL:   30 days

  ZADD posts:celeb456 1701388800.0 "post:999"
  ZREVRANGE posts:celeb456 0 9     -- last 10 posts from this celebrity

Post content cache:
  Key:   post:{post_id}
  Type:  Hash or JSON string
  TTL:   24 hours
  Value: {user_id, content, media_urls, like_count, created_at}

Celebrity follows list (per user):
  Key:   celeb_follows:{user_id}
  Type:  Set of celebrity user IDs
  TTL:   5 minutes
```

## 5. Deep Dive: Fan-Out Strategies

### Push Model (Fan-Out on Write)

When a user posts, immediately write the post ID into every follower's feed cache.

```python
def fan_out_on_write(post_id: int, author_id: int, timestamp: float) -> None:
    """Fan out to all followers' feed caches."""
    # Fetch followers in batches to avoid loading all IDs at once
    cursor = 0
    batch_size = 1000

    while True:
        followers = db.fetch(
            """
            SELECT follower_id FROM follows
            WHERE followee_id = %s
            ORDER BY follower_id
            LIMIT %s OFFSET %s
            """,
            (author_id, batch_size, cursor)
        )
        if not followers:
            break

        # Pipeline all Redis writes for this batch
        pipe = redis.pipeline()
        for follower in followers:
            feed_key = f"feed:{follower['follower_id']}"
            pipe.zadd(feed_key, {str(post_id): timestamp})
            # Trim feed to last 200 posts to control memory
            pipe.zremrangebyrank(feed_key, 0, -201)
        pipe.execute()

        cursor += batch_size
```

Pros: feed reads are O(1) — just ZREVRANGE from Redis.
Cons: a user with 10M followers triggers 10M Redis writes. At 100K Redis ops/sec, this takes
100 seconds. During a viral post surge, fan-out workers would create a massive write storm.

### Pull Model (Fan-Out on Read)

Do not write to followers' feeds at post time. At feed load time, query each followed user's
recent posts and merge them.

```python
def build_feed_on_read(user_id: int, limit: int = 20) -> list[int]:
    """Compute feed at read time by merging all followees' posts."""
    followees = db.fetch(
        "SELECT followee_id FROM follows WHERE follower_id = %s LIMIT 500",
        (user_id,)
    )

    all_post_ids = []
    # This is N separate cache lookups — expensive at scale
    for followee in followees:
        recent_posts = redis.zrevrange(
            f"posts:{followee['followee_id']}", 0, 10
        )
        all_post_ids.extend(recent_posts)

    # Sort by timestamp (embedded in Snowflake ID or fetched from cache)
    all_post_ids.sort(reverse=True)
    return all_post_ids[:limit]
```

Pros: zero write fan-out cost. A celebrity posting has zero immediate downstream cost.
Cons: a user following 500 accounts makes 500 cache lookups per feed load. At 100K feed loads/sec,
this becomes 50M cache ops/sec — far more expensive than the fan-out it avoided.

### Hybrid Model (Industry Standard)

Use fan-out on write for regular users. Skip it for celebrities and inject their posts at read time.

```python
CELEBRITY_THRESHOLD = 1_000_000  # followers

def on_post_created(post_id: int, author_id: int, timestamp: float) -> None:
    author = get_user(author_id)

    if author.follower_count < CELEBRITY_THRESHOLD:
        # Regular user: fan out to all followers' pre-built feeds
        fan_out_on_write(post_id, author_id, timestamp)
    else:
        # Celebrity: only write to their own post index
        # Followers will pull this at read time
        redis.zadd(f"posts:{author_id}", {str(post_id): timestamp})

def get_feed(user_id: int, cursor: float = None, limit: int = 20) -> list[int]:
    # Step 1: Get pre-built feed (from regular users followed)
    if cursor:
        feed_ids = redis.zrevrangebyscore(f"feed:{user_id}", cursor, '-inf',
                                          start=0, num=limit * 2)
    else:
        feed_ids = redis.zrevrange(f"feed:{user_id}", 0, limit * 2 - 1)

    # Step 2: Get celebrity post IDs at read time
    celebrity_ids = get_celebrity_follows(user_id)  # cached in Redis, 5min TTL
    for celeb_id in celebrity_ids:
        celeb_posts = redis.zrevrange(f"posts:{celeb_id}", 0, 9)
        feed_ids.extend(celeb_posts)

    # Step 3: Inject user's own recent posts (read-your-own-writes)
    own_posts = redis.zrevrange(f"posts:{user_id}", 0, 4)
    feed_ids.extend(own_posts)

    # Step 4: Sort, deduplicate, and paginate
    # Post IDs are Snowflake IDs — sort descending for newest-first
    unique_ids = list(dict.fromkeys(feed_ids))  # preserves order, deduplicates
    unique_ids.sort(reverse=True)
    return unique_ids[:limit]
```

### Celebrity Threshold Management

The threshold is a business parameter, not a fixed technical constant.

```
Becoming a celebrity:
  - Background job runs hourly, finds users who crossed threshold
  - Migrate: stop writing to followers' feeds on new posts
  - No need to retroactively remove past fan-out writes (stale data self-expires via TTL)

Leaving celebrity status (follower loss):
  - Use hysteresis: become celebrity at 1M, revert to regular at 800K
  - Prevents flip-flop when a user hovers near the threshold

Edge cases:
  - User with 5M followers posts 100 times in a day (content creator)
    → All 100 posts go into the celebrity index, no fan-out storm
  
  - User with 500K followers (below threshold) goes viral
    → Their posts are fan-out written, but 500K × 1 write = 500K ops → manageable
    → Background job upgrades them to celebrity after hourly check
    
  - Celebrity is followed by another celebrity
    → Both appear in the celebrity pull set of mutual followers
    → Merged at read time alongside pre-built feed
```

## 6. Scale and Trade-offs

| Concern                    | Solution                                   | Trade-off                                              |
|----------------------------|--------------------------------------------|--------------------------------------------------------|
| Celebrity fan-out          | Hybrid: push for regular, pull for celeb   | Complexity in read-path merge step                     |
| Feed latency               | Pre-built Redis sorted sets                | Memory cost; stale TTL for inactive users              |
| Post deletion              | Lazy filter at read time                   | Deleted post IDs linger in feed keys until evicted     |
| New follow                 | Backfill last N posts from new followee    | One-time write cost; inconsistent with past history    |
| Cold feed (cache miss)     | Rebuild on demand from follow graph        | 500ms+ first load; background pre-warm for active DAU |
| Read-your-own-writes       | Inject author's own posts at read time     | Small extra merge cost on every feed load              |
| Algorithmic ranking        | Replace timestamp score with ML score      | Requires online feature store; much more infrastructure|

### How Real Systems Handle It

Twitter uses a hybrid fan-out with roughly a 300K follower threshold. The pre-computed home
timeline lives in Redis. Obama, Katy Perry, and Justin Bieber are pull-based. A separate timeline
mixer service merges the pre-built feed with celebrity injections. This service is one of the
most expensive components Twitter operates.

Instagram uses push-only fan-out (no pull for celebrities) combined with an aggressive CDN for
media and an ML-ranked feed where the sorted set score is a ranking model output, not a timestamp.

Facebook uses its own TAO graph database to store the social graph and computes feeds using
EdgeRank (now a deep learning model). The graph traversal at read time is feasible because TAO is
purpose-built for high-throughput social graph reads, not a general-purpose database.

TikTok is architecturally different: the feed is primarily a recommendation engine, not a social
graph traversal. The "Following" tab is a small subset of the feed. Most users see content from
accounts they do not follow. This is a fundamentally different system (recommendation >> social
graph fan-out).

## 7. Senior Engineer Perspective

**The celebrity threshold is where the complexity lives, not in the fan-out itself.** At exactly
999,999 followers, a user still gets fan-out-on-write. When they post and their 999,999 followers
request the feed simultaneously, you push 1M writes to Redis in seconds — which is expensive but
survivable. The day they gain follower 1,000,001, they flip to the pull model and their posts
are no longer pre-propagated. A follower who used to see their posts at feed-load-cache-hit time
now triggers a celebrity index lookup. Make sure your monitoring tracks celebrity index hit rates.

**Feed TTL is a product and user experience decision.** If you expire feeds after 7 days, a user
who opens the app after 8 days gets a cold feed that must be rebuilt synchronously, adding 500ms
or more to their first load. The alternative — never expire feeds — means unbounded Redis memory
growth for every user who has ever existed. The practical answer is: expire cold feeds, pre-warm
feeds for users who are likely to return soon (users who were active in the last 30 days), and
accept the cold-start penalty for truly inactive accounts.

**Post deletion is harder than it looks.** Soft-deleting a post by setting `is_deleted = TRUE`
does not remove the post ID from any feed sorted set. Every feed read must check the deletion
status of every post ID it retrieves. At 20 posts per page and 100K feed loads/sec, you are making
2M deletion-status checks per second. Cache deletion status aggressively or run an async cleanup
job that removes deleted post IDs from feed sorted sets within minutes of deletion.

**The social graph is a first-class data structure at scale.** It is not a SQL JOIN. The
`follows` table with 200 billion rows does not respond to `SELECT follower_id FROM follows WHERE
followee_id = X` in time for a synchronous request. Store the follow graph in a purpose-built
adjacency list store: Cassandra wide rows, Redis sets, or a dedicated graph database. The fan-out
worker must read from this store, not from transactional PostgreSQL.

**Read-your-own-writes is a correctness requirement, not an optimization.** After a user posts
and scrolls to their home feed, they expect to see their own post immediately. A pre-built feed
cache that was last populated before the post was created will not contain it. Always inject the
author's own last 5-10 posts at feed load time, bypassing the pre-built cache entirely. This is
a small, bounded amount of work and it prevents a confusing user experience that would otherwise
generate support tickets.
