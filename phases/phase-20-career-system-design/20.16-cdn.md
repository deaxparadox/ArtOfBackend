## The Key Architectural Insight

A CDN is fundamentally a **globally distributed cache**. The core tension is between three forces that cannot be simultaneously maximized:

- **Freshness** — users want the latest content
- **Performance** — users want the nearest copy
- **Cost** — you want to minimize origin server load

Every CDN design decision is a trade-off between these three. Cache-Control headers, TTLs, purge APIs, and origin shields are all tools for navigating this triangle.

The second hard problem is **routing**: how does a user in Mumbai automatically get served from a Mumbai PoP (Point of Presence) and not a New York one? The answer is anycast routing and DNS-based steering — and getting this wrong adds hundreds of milliseconds of latency.

---

## Section 1: Requirements Clarification

Before drawing any architecture, establish scope:

| Question | Why it matters |
|---|---|
| What content types? (static assets, video, API responses, live streams) | Drives cache strategy — API responses need short TTLs, video needs byte-range support |
| Read/write ratio? | CDN is read-optimized — if writes dominate, you need a different design |
| How stale can content be? | Determines TTL strategy |
| Do we need HTTPS at the edge? | Edge needs TLS certificates — adds operational complexity |
| Any geography restrictions? (GDPR, geo-blocking) | Affects which PoPs serve which users |
| What is the expected traffic? (req/sec, GB/day) | Drives PoP count and capacity planning |

**Scope for this design:** General-purpose CDN serving static assets and cacheable API responses, global coverage, ~1 million requests/second at peak, sub-50ms latency to 95% of users.

## Section 2: Capacity Estimation

| Parameter | Estimate |
|---|---|
| Peak requests/second | 1,000,000 req/sec |
| Average response size | 100 KB (mix of HTML, images, JS, CSS) |
| Peak throughput | 1M * 100KB = **100 GB/sec** |
| Cache hit ratio target | 90% |
| Origin requests/sec | 1M * 10% = **100,000 req/sec** |
| Storage per PoP (hot cache) | 10 TB SSD |
| Number of PoPs | 100 globally |
| Total edge storage | 100 * 10 TB = **1 PB** |

At 90% cache hit ratio, origin traffic drops 10x. This is why CDN ROI is enormous — you serve 900,000 req/sec from cache and only 100,000 from your expensive origin servers.

**Bandwidth math:**
- 100 GB/sec total edge throughput
- 100 PoPs → 1 GB/sec average per PoP
- Peak PoPs (US East, EU West) carry 5-10x average: ~5-10 GB/sec each

## Section 3: High-Level Architecture

```
                    +------------------------------------------+
                    |           DNS / Anycast Routing          |
                    |   User DNS query --> nearest PoP IP      |
                    +------------------+-----------------------+
                                       |
               +-----------------------+-----------------------+
               |                       |                       |
               v                       v                       v
    +---------------------+  +-------------------+  +--------------------+
    |    PoP: US-East     |  |   PoP: EU-West    |  |   PoP: AP-South   |
    |                     |  |                   |  |                    |
    |  [Edge Server x4]   |  |  [Edge Server x4] |  |  [Edge Server x4] |
    |  [Edge Cache 10TB]  |  |  [Edge Cache 10TB]|  |  [Edge Cache 10TB]|
    +----------+----------+  +---------+---------+  +---------+----------+
               |  cache miss           | cache miss           | cache miss
               +---------------+-------+                      |
                               |                              |
                               v                              |
                    +---------------------+                   |
                    |   Regional Cache    | <-----------------+
                    |   (Origin Shield)   |
                    |   100TB NVMe SSD    |
                    +----------+----------+
                               |  cache miss
                               v
                    +---------------------+
                    |   Origin Server     |
                    |  (customer infra)   |
                    +---------------------+
```

**Cache hierarchy (3 layers):**
1. **Edge cache** — closest to users, 10TB per PoP, highest hit rate for popular content
2. **Regional cache / Origin shield** — mid-tier, aggregates misses from many edge PoPs, catches the long tail
3. **Origin** — the customer's actual servers, only hit for true misses

## Section 4: Deep Dive — The Hard Problems

### 4.1 Anycast Routing

**How does a user in Mumbai automatically reach the Mumbai PoP?**

Two mechanisms work together:

**DNS-based steering:**
```
User resolves: assets.example.com
    |
    v
CDN's authoritative DNS server
    |
    v  (sees user's resolver IP -- approximates geography)
    |
    v
Returns the IP of the nearest PoP
```

Each PoP has its own anycast IP block. The DNS response TTL is set low (30-60 seconds) so that when a PoP goes down, traffic re-routes quickly.

**BGP Anycast:**
Multiple PoPs announce the same IP prefix via BGP. Internet routers naturally route packets to the "closest" announcement (fewest BGP hops). This works at the IP routing layer without DNS involvement.

```
Same IP block: 192.0.2.0/24 announced by:
  - US-East PoP  via AS-1234
  - EU-West PoP  via AS-5678
  - AP-South PoP via AS-9012

User in London --> BGP selects EU-West path --> user hits EU-West PoP
User in Tokyo  --> BGP selects AP-South path --> user hits AP-South PoP
```

The downside: BGP anycast is coarse — it operates at the AS level, not city level. Large CDNs (Cloudflare, Akamai) combine anycast with application-layer health probes to fine-tune routing.

### 4.2 Cache Hierarchy and Origin Shield

Without an origin shield, every edge PoP independently sends misses to the origin:

```
100 PoPs x 10% miss rate x 1,000,000 req/sec = 100,000 req/sec to origin
Thundering herd scenario: 100 PoPs all miss on the same cold-start content
```

With origin shield, one regional cache acts as the "last stop" before origin:

```
Edge miss rate: 10%     --> 100,000 req/sec to regional cache
Regional hit rate: 80%  --> only 20,000 req/sec reach origin
Total origin reduction:  98% vs. no CDN
```

The regional cache is larger and slower (but still fast) — typically NVMe SSDs with 100TB capacity. It is the "long tail" catcher.

### 4.3 Cache Key Design

The cache key determines what constitutes a unique cacheable object. Getting this wrong causes either:
- **Over-caching:** serving stale or wrong content (e.g., caching a logged-in user's profile page)
- **Under-caching:** cache misses on content that is actually identical

Standard cache key: `scheme + host + path + query_string`

**Vary header:** If the origin responds with `Vary: Accept-Encoding`, the CDN must cache separate copies for gzip and non-gzip clients. Misusing `Vary: Cookie` breaks caching entirely — every user gets a unique cache entry.

```
Good cache key:  https://cdn.example.com/assets/logo.png
                 --> same for all users --> high hit rate

Bad cache key:   https://app.example.com/profile?user_id=123
                 --> unique per user --> 0% hit rate, wastes cache space
```

**Cache key normalization:** Sort query parameters. `?a=1&b=2` and `?b=2&a=1` are the same resource but would be different cache keys without normalization.

### 4.4 Cache Invalidation

TTL-based expiry is simple but creates a "stale window." Event-driven purge eliminates staleness but requires infrastructure.

**TTL-based via Cache-Control headers:**
```http
Cache-Control: max-age=86400          # 24 hours
Cache-Control: s-maxage=3600          # 1 hour for CDN, ignored by browsers
Cache-Control: no-store               # never cache (login pages, checkout)
Cache-Control: immutable              # content-hashed assets, cache forever
Cache-Control: max-age=300, stale-while-revalidate=60
```

The `stale-while-revalidate` directive is powerful: serve the stale copy immediately (zero latency) while triggering a background refresh. Users almost never see a cache-miss delay.

**Event-driven purge — two strategies:**

Strategy 1: **URL versioning** (preferred for static assets)
```
Old deployment: /assets/app.js
New deployment: /assets/app.abc123.js   <-- hash of file content in filename
```
Old URL stays cached forever (correct — old content is immutable). New URL is always a fresh miss. No purge API call needed. This is the gold standard for JS/CSS/image assets.

Strategy 2: **Purge API** (for dynamic content)
```bash
# Cloudflare purge API example
curl -X POST "https://api.cloudflare.com/client/v4/zones/{zone_id}/purge_cache" \
  -H "Authorization: Bearer {api_token}" \
  -d '{"files":["https://example.com/article/breaking-news-123"]}'
```
Purge must propagate to all 100 PoPs — takes 1-5 seconds. During propagation, some users may still get stale content. This is acceptable for most use cases but not for financial data.

**Surrogate keys / Cache tags** (Fastly, Varnish):
```
# Origin response header:
Surrogate-Key: product-456 category-electronics author-789

# Later, when product 456 changes:
POST /purge-tag  {"tag": "product-456"}
# Invalidates every cached response tagged with product-456 in one operation
```
This is the most powerful cache invalidation model — you can group related objects and invalidate them atomically without knowing every URL.

### 4.5 Consistent Hashing Within a PoP

Within a PoP, you have multiple edge servers (e.g., 20 servers). Which server caches which object?

**Random assignment:** Any server can serve any object, but the same object may be cached on 5 different servers — wasted memory, lower effective hit rate.

**Consistent hashing:** Object always maps to the same server(s) based on `hash(cache_key)`. Adding or removing a server only re-maps a fraction of objects.

```
Hash ring:
         [Server A]----------[Server B]----------[Server C]
              |                   |                   |
           hash 0-999         hash 1000-1999      hash 2000-2999

Request for "logo.png"  --> hash = 1500 --> Server B
Request for "video.mp4" --> hash = 800  --> Server A
Request for "style.css" --> hash = 2200 --> Server C
```

With virtual nodes (each physical server occupies multiple ring positions), load distributes evenly even with heterogeneous server capacities.

## Section 5: Data Model

CDN edge servers do not use a traditional database — they use in-memory LRU and on-disk caches. But the control plane needs state:

**Cache metadata (per-PoP local state):**
```sql
-- Stored in a local key-value store (RocksDB or Redis) per edge node
cache_entries
---------------------------------------------------------------------------
cache_key       VARCHAR(2048)  PRIMARY KEY   -- normalized URL
etag            VARCHAR(128)                 -- for If-None-Match conditional GETs
last_modified   TIMESTAMP
expires_at      TIMESTAMP                    -- TTL deadline
content_type    VARCHAR(128)
size_bytes      BIGINT
hit_count       BIGINT
last_accessed   TIMESTAMP
storage_path    VARCHAR(512)                 -- path on local SSD
```

**PoP registry (global control plane):**
```sql
pops
---------------------------------------------------------------------
pop_id          VARCHAR(32)   PRIMARY KEY    -- "us-east-1a"
region          VARCHAR(64)                  -- "North America"
anycast_prefix  VARCHAR(18)                  -- "192.0.2.0/24"
capacity_gbps   INT
health_status   VARCHAR(16)                  -- 'healthy','degraded','down'
last_heartbeat  TIMESTAMP
```

**Purge jobs (distributed purge tracking):**
```sql
purge_requests
---------------------------------------------------------------------
purge_id        UUID          PRIMARY KEY
zone_id         VARCHAR(64)
pattern         TEXT                         -- URL or surrogate tag
requested_at    TIMESTAMP
status          VARCHAR(16)                  -- 'pending','propagating','complete'
completed_pops  JSONB                        -- {"us-east": true, "eu-west": false}
```

## Section 6: Trade-offs and Failure Modes

| Scenario | Impact | Mitigation |
|---|---|---|
| PoP goes down | Users in that region hit next-nearest PoP | Short DNS TTL (30s) + BGP withdrawal + health checks every 5s |
| Cache stampede (TTL expires, 10k concurrent requests hit origin) | Origin overloaded | Cache locking: only one request fetches from origin, others wait for the result |
| Thundering herd on cold-start | All PoPs miss on new content simultaneously | Origin shield absorbs all misses; edge PoPs coalesce behind shield request |
| Stale cache after code deploy | Users see old JS/CSS, JavaScript errors | Content-hashed filenames with `Cache-Control: immutable` |
| Wrong Vary header set by origin | `Vary: Cookie` bypasses cache completely | CDN should strip or ignore `Vary: Cookie` for paths known to be cacheable |
| Origin is slow or down | Edge cannot refresh expired entries | `stale-if-error` header: serve stale content for up to N seconds if origin errors |

**stale-if-error pattern:**
```http
Cache-Control: max-age=300, stale-if-error=86400
```
If the origin returns a 5xx or is unreachable, serve cached content for up to 24 hours. Your site stays up during an origin outage. This is why adding a CDN in front of your origin also improves availability — not just performance.

## Section 7: Senior Engineer Perspective

In an interview, most candidates describe a basic two-layer CDN (edge → origin) and talk about TTLs. What separates strong candidates:

**1. Origin shield is non-obvious.** The thundering herd problem across 100 PoPs is real and expensive. Without a shield, a cold-start or TTL expiry at 100 PoPs simultaneously hits your origin with 100x the traffic. The regional shield layer is the standard mitigation — but you have to know to propose it.

**2. Cache invalidation is the core product.** Netflix, Amazon, and news sites all care deeply about "we just deployed code — how fast can we invalidate the entire CDN?" Surrogate keys let you purge by tag atomically — this is a major operational lever and a differentiator between CDN providers.

**3. The security surface.** CDN sits in front of your origin, making it also your DDoS mitigation layer, WAF (Web Application Firewall), and TLS termination point. A misconfigured CDN can serve data to the wrong geographic region (GDPR violations) or expose origin IPs (bypassing DDoS protection). Security is not an afterthought in CDN design.

**4. Build vs. buy is almost always buy.** Operating a CDN requires BGP peering agreements, physical PoP presence in data centers worldwide, 24/7 network operations. No startup builds this. The design interview tests whether you understand how a CDN works — not whether you would implement one from scratch.

> **The interview signal:** When asked "what happens when the CDN cache misses?", weak candidates say "it goes to origin." Strong candidates walk through the full hierarchy (edge → regional shield → origin), discuss stampede protection, and then ask "what is the origin's capacity? Because that determines whether we need a shield at all."
