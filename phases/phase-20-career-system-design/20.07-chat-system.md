# 20.07 — Design a Chat System (WhatsApp / Slack)

## Why This Problem Is Hard

Chat sounds simple: send a message, receive a message. The difficulty is in the operational constraints that emerge at scale:

- **Real-time delivery**: Users expect messages to appear in under 100ms. HTTP request-response is the wrong model; you need persistent connections.
- **Ordering guarantees**: Two users sending messages at the same millisecond creates ambiguity. Clocks drift. You cannot use `created_at` as the canonical order.
- **Online/offline presence**: Tracking whether 2 billion users are online is a distributed state management problem. Naive approaches kill your database.
- **Reliability vs. latency**: If the recipient is offline, you cannot drop the message. You need durable queues without adding latency for online users.
- **Fan-out for group chats**: A Slack channel with 5000 members means one message must reach 5000 WebSocket connections — potentially on 50 different servers.

## Capacity Estimation

```
DAU:                     500 million users
Messages per day:        100 billion  (~200 messages/user/day)
Messages per second:     100B / 86400 ~= 1.15 million msg/sec (peak ~3x = 3.5M/s)
Average message size:    1 KB (text)
Storage per day:         100B * 1 KB = 100 TB/day
Storage per year:        ~36 PB/year  (requires tiered/cold storage)

WebSocket connections:   ~50 million concurrent (10% of DAU online at any time)
Servers (at 50K conn/server): 50M / 50K = 1000 WebSocket servers

Presence store writes:   heartbeat every 30s per online user
                         50M / 30 ~= 1.67 million writes/sec to Redis
```

## High-Level Architecture

```
                              +------------------+
                              |   DNS / Load     |
                              |   Balancer       |
                              +--------+---------+
                                       |
              +------------------------+------------------------+
              |                        |                        |
    +---------+--------+   +-----------+---------+   +---------+--------+
    | WebSocket Server |   | WebSocket Server    |   | WebSocket Server |
    |   (WS-1)         |   |   (WS-2)            |   |   (WS-N)         |
    +---------+--------+   +-----------+---------+   +---------+--------+
              |                        |                        |
              +------------------------+------------------------+
                                       |
                           +-----------+-----------+
                           |   Message Queue       |
                           |   (Kafka / RabbitMQ)  |
                           +-----------+-----------+
                                       |
                           +-----------+-----------+
                           |   Message Service     |
                           |   (persistence +      |
                           |    fan-out logic)     |
                           +-----------+-----------+
                                       |
              +------------------------+------------------------+
              |                                                 |
  +-----------+----------+                         +-----------+----------+
  |   Cassandra Cluster  |                         |   Redis Cluster      |
  |   (message history)  |                         |   (presence, session |
  +----------------------+                         |    routing table)    |
                                                   +----------------------+
```

**Key flows:**

1. Client A connects to WS-1 via WebSocket. WS-1 registers `user_A → WS-1` in Redis.
2. Client A sends a message. WS-1 publishes it to Kafka.
3. Message Service consumes from Kafka, writes to Cassandra, then looks up recipient routing in Redis.
4. If recipient is on WS-2, Message Service pushes the message to WS-2 via an internal pub/sub channel. WS-2 delivers it to Client B.
5. If recipient is offline, the message stays in Cassandra. When Client B reconnects, it fetches unread messages.

## Data Model

```sql
-- Conversations (1-to-1 or group)
CREATE TABLE conversations (
    id          BIGSERIAL PRIMARY KEY,
    type        VARCHAR(10) NOT NULL CHECK (type IN ('direct', 'group')),
    name        VARCHAR(255),           -- null for direct chats
    created_at  TIMESTAMP NOT NULL DEFAULT now()
);

-- Membership
CREATE TABLE conversation_members (
    conversation_id BIGINT NOT NULL REFERENCES conversations(id),
    user_id         BIGINT NOT NULL,
    joined_at       TIMESTAMP NOT NULL DEFAULT now(),
    last_read_seq   BIGINT NOT NULL DEFAULT 0,  -- last sequence number read
    PRIMARY KEY (conversation_id, user_id)
);

-- Messages
CREATE TABLE messages (
    id              BIGSERIAL PRIMARY KEY,       -- Snowflake ID in practice
    conversation_id BIGINT NOT NULL,
    sender_id       BIGINT NOT NULL,
    content         TEXT,
    media_url       TEXT,
    seq             BIGINT NOT NULL,             -- per-conversation sequence number
    created_at      TIMESTAMP NOT NULL DEFAULT now(),
    status          VARCHAR(20) DEFAULT 'sent'   -- sent, delivered, read
);

CREATE INDEX idx_messages_conversation_seq ON messages(conversation_id, seq DESC);

-- Delivery/read receipts (separate from messages to avoid update contention)
CREATE TABLE message_receipts (
    message_id  BIGINT NOT NULL,
    user_id     BIGINT NOT NULL,
    event       VARCHAR(20) NOT NULL CHECK (event IN ('delivered', 'read')),
    occurred_at TIMESTAMP NOT NULL DEFAULT now(),
    PRIMARY KEY (message_id, user_id, event)
);
```

**Note on storage:** At Cassandra scale, the schema above is modeled as a wide row per conversation:

```
Partition key: conversation_id
Clustering key: seq DESC
```

This allows efficient fetching of the latest N messages with a single partition scan.

## Deep Dive: The Three Hard Problems

### 1. Real-Time Delivery: WebSocket vs. Long Polling

**WebSocket** is the right answer for chat. It provides a full-duplex, persistent TCP connection. The server can push messages to the client at any time.

```
Client                          Server
  |                               |
  |  --- HTTP Upgrade Request --> |
  |  <-- 101 Switching Protocols--|
  |                               |
  |  <-- push: new message -------|  (any time, no request needed)
  |  --- ack ----------------->   |
```

**Long polling** is the fallback for environments where WebSocket is blocked (corporate proxies). The client sends a request and the server holds it open until a message arrives or a timeout (typically 30s) is reached. It works but is wasteful: each "poll" consumes a connection.

**Connection draining on server restart:**
When a WebSocket server restarts, it must not abruptly drop 50,000 connections. The process:
1. Server receives SIGTERM.
2. It stops accepting new connections.
3. It sends a `connection_migrating` frame to all connected clients.
4. Clients reconnect to the load balancer (which routes to a healthy server).
5. Server waits for drain timeout (e.g., 30s), then exits.

### 2. Message Ordering: Why Timestamps Fail

```
User A sends at:   12:00:00.001 (server clock on WS-1)
User B sends at:   12:00:00.001 (server clock on WS-2, drifted by 2ms)
```

Both messages arrive at Message Service with identical timestamps. Which comes first? No answer.

**Solution: per-conversation sequence numbers**

A Redis counter per conversation atomically increments on each new message:

```python
seq = redis.incr(f"conv:{conversation_id}:seq")
message.seq = seq
```

Messages are ordered by `seq`, not `created_at`. The `seq` is monotonically increasing and authoritative.

**Alternative: Snowflake IDs**
Twitter's Snowflake generates 64-bit IDs that encode a timestamp (41 bits), machine ID (10 bits), and sequence (12 bits). These are globally sortable by creation time at millisecond granularity. For chat, per-conversation sequence numbers are better because they are dense (1, 2, 3...) and allow clients to detect gaps.

### 3. Online/Offline Presence

**Naive approach:** Store `is_online` in the users table. Every login/logout updates a row. At 50M concurrent users sending a heartbeat every 30s, this is 1.67M writes/sec to your database — not viable.

**Redis approach:**

```
Redis Hash: presence
  user_id_1 → { status: "online", last_seen: 1718000000, connection_id: "ws-1:conn-4921" }
  user_id_2 → { status: "away",   last_seen: 1717999830, connection_id: "ws-2:conn-0021" }
```

Each WebSocket server writes a heartbeat for its connected users every 30 seconds. A TTL of 65 seconds is set on each key. If the server crashes without graceful shutdown, the key expires and the user appears offline automatically.

**Presence fanout:**
When User A's status changes, every user who has A in their contact list should be notified. With 500 contacts each, this is potentially 500 writes per status change. To avoid a thundering herd:

- Use a pub/sub channel per user for presence updates.
- Clients subscribe only to presence of their current chat partner, not all contacts.
- Batch presence updates (do not send a new update if one was sent in the last 5s).

## WhatsApp Architecture Note

WhatsApp runs on a single persistent connection per phone (an XMPP-based protocol over TCP). Key design choices:

- Messages are stored on the server only until delivered. Once the recipient acknowledges receipt, the server deletes the message.
- For offline delivery, the server queues the message and delivers it when the user reconnects. The server does not hold messages longer than 30 days.
- End-to-end encryption means the server never has access to plaintext. It stores only ciphertext.
- Delivery receipt (double tick) fires when the server receives the message. Read receipt (blue tick) fires when the recipient opens the conversation.

## Trade-offs Table

| Decision | Option A | Option B | Winner |
|---|---|---|---|
| Transport | WebSocket | Long polling | WebSocket (lower latency, less overhead) |
| Message ordering | Wall clock timestamps | Per-conversation seq numbers | Seq numbers (monotonic, gap-detectable) |
| Message storage | PostgreSQL | Cassandra | Cassandra (write-heavy, time-series access pattern) |
| Presence storage | Database rows | Redis hash with TTL | Redis (high write throughput, auto-expire) |
| Offline delivery | Push notification only | Server-side queue + push | Server queue (more reliable) |
| Group fan-out | Client pulls | Server pushes via pub/sub | Server push (lower latency) |

## Senior Engineer Perspective

**What juniors get wrong:**

1. Using `ORDER BY created_at` for message ordering. This breaks under any clock skew.
2. Storing presence in the main database. The write volume is an order of magnitude too high.
3. Not planning for connection draining. Server restarts that drop 50K connections cause a thundering herd reconnect storm.

**What makes WhatsApp impressive:**
At its peak, WhatsApp handled 1 million connections per server using Erlang. Erlang's lightweight processes (not OS threads) and the BEAM VM's scheduler made this possible. Most systems use Go or Node.js today for similar reasons — event-loop or goroutine-based concurrency handles far more connections than thread-per-connection models.

**Scaling group chat:**
Group chat with 500 members is qualitatively different from 1-to-1 chat. Delivering to 500 members means 500 WebSocket lookups, 500 pub/sub publishes, and 500 potential push notifications for offline members. At Slack's scale, large channels use a different delivery path: a dedicated fan-out service with its own Kafka topic per large channel, written specifically for high-fan-out scenarios.

**The hidden complexity: message deduplication**
With at-least-once delivery, the same message can be delivered twice (e.g., network retry before the ACK arrives). Every client must deduplicate on message ID before rendering. The server must also deduplicate before writing to Cassandra — use the message ID as the idempotency key.
