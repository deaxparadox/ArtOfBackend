## 1. What Is It? (Theory)

Go has the best built-in profiling toolchain of any mainstream language. The `pprof` profiler is part of the standard library, the output formats are standardized, and the visualization is interactive and powerful. This is not an accident — the Go team designed `pprof` as a first-class tool because performance is a core Go value.

**What Go profiling measures:**

```
CPU PROFILE:    Where is goroutine CPU time being spent?
HEAP PROFILE:   Which code is allocating the most memory?
GOROUTINE:      How many goroutines exist and where are they blocked?
BLOCK PROFILE:  Which goroutines are blocked waiting for channels/mutexes?
MUTEX PROFILE:  Which mutexes have high contention?
TRACE:          Full event timeline (goroutines, GC, syscalls)
```

The power of Go's profiler is that you can answer all of these questions about a **production server** with zero restarts and minimal overhead.

## 2. How It Actually Works (Deep Mechanics)

### Enabling pprof in Your Server

Exposing pprof over HTTP is the standard production pattern. Add this to your main package:

```go
package main

import (
    "net/http"
    _ "net/http/pprof"  // blank import: registers pprof handlers on DefaultServeMux
    "log"
)

func main() {
    // Start pprof server on a separate port (never expose this publicly!)
    go func() {
        log.Println(http.ListenAndServe("localhost:6060", nil))
    }()

    // Your actual application server
    startApplicationServer()
}
```

This gives you these HTTP endpoints:
```
http://localhost:6060/debug/pprof/          → index of all profiles
http://localhost:6060/debug/pprof/profile   → 30-second CPU profile
http://localhost:6060/debug/pprof/heap      → current heap snapshot
http://localhost:6060/debug/pprof/goroutine → all goroutine stacks
http://localhost:6060/debug/pprof/block     → blocking profile
http://localhost:6060/debug/pprof/mutex     → mutex contention
http://localhost:6060/debug/pprof/trace     → execution trace
```

### Collecting and Analyzing CPU Profile

```bash
# Collect a 30-second CPU profile from a running server
go tool pprof http://localhost:6060/debug/pprof/profile?seconds=30

# While pprof is sampling, run your load test (in another terminal):
# k6 run --vus 50 --duration 30s script.js

# After 30 seconds, pprof drops you into an interactive session:
# (pprof) 

# Most useful pprof commands:
(pprof) top20        # top 20 functions by CPU time
(pprof) top20 -cum   # top 20 by CUMULATIVE time (including callees)
(pprof) web          # open interactive flamegraph in browser (requires graphviz)
(pprof) list myFunc  # show annotated source code for a function
(pprof) peek myFunc  # show callers and callees of a function

# Example top output:
#      flat  flat%   sum%        cum   cum%
#   1.23s   41.0%  41.0%     1.23s   41.0%  runtime.mallocgc
#   0.89s   29.7%  70.7%     0.89s   29.7%  json.Marshal
#   0.31s   10.3%  81.0%     0.31s   10.3%  net/http.(*response).Write
#   0.21s    7.0%  88.0%     2.44s   81.3%  main.handleRequest

# Reading:
# mallocgc = 41% of CPU time in memory allocation → too many allocations
# json.Marshal = 30% of CPU time → serialization is a bottleneck
```

### Generating an Interactive Flamegraph

```bash
# The most useful visualization: interactive HTML flamegraph
go tool pprof -http=:8080 http://localhost:6060/debug/pprof/profile?seconds=30

# Opens http://localhost:8080 in browser with:
# - Flamegraph view (default)
# - Top view
# - Source view
# - Peek view

# Save profile for later analysis
curl -o cpu.prof "http://localhost:6060/debug/pprof/profile?seconds=30"
go tool pprof -http=:8080 cpu.prof
```

### Heap Profile: Finding Memory Hogs

```bash
# Current heap allocation profile
go tool pprof http://localhost:6060/debug/pprof/heap

(pprof) top20         # top allocating functions
(pprof) top20 -inuse  # currently live allocations (not freed yet)
(pprof) top20 -alloc  # total ever allocated (including freed)
(pprof) web           # flamegraph of allocations
```

```go
// COMMON GO ALLOCATION PATTERN TO WATCH:
// Avoid allocating in hot loops

// BAD: allocates a new map on every request
func handleRequest(w http.ResponseWriter, r *http.Request) {
    data := map[string]interface{}{   // new map each time
        "user_id": getUserID(r),
        "timestamp": time.Now(),
    }
    json.NewEncoder(w).Encode(data)
}

// BETTER: use a sync.Pool to reuse objects
var responsePool = sync.Pool{
    New: func() interface{} {
        return make(map[string]interface{})
    },
}

func handleRequest(w http.ResponseWriter, r *http.Request) {
    data := responsePool.Get().(map[string]interface{})
    defer func() {
        // Clear and return to pool
        for k := range data { delete(data, k) }
        responsePool.Put(data)
    }()

    data["user_id"] = getUserID(r)
    data["timestamp"] = time.Now()
    json.NewEncoder(w).Encode(data)
}
```

### Writing Benchmarks

Go's built-in benchmark tool is the correct way to measure the performance of a specific function in isolation — before and after optimization:

```go
// user_service_test.go
package main

import "testing"

// Benchmark prefix + testing.B parameter = benchmark function
func BenchmarkGetUserJSON(b *testing.B) {
    user := &User{ID: 1, Name: "Alice", Email: "alice@example.com"}
    
    b.ResetTimer()  // don't include setup time in measurement
    
    for i := 0; i < b.N; i++ {
        // b.N is auto-adjusted so the benchmark runs for 1-2 seconds
        _, err := json.Marshal(user)
        if err != nil {
            b.Fatal(err)
        }
    }
}

func BenchmarkGetUserJSONParallel(b *testing.B) {
    user := &User{ID: 1, Name: "Alice", Email: "alice@example.com"}
    
    b.RunParallel(func(pb *testing.PB) {
        for pb.Next() {
            json.Marshal(user)
        }
    })
}
```

```bash
# Run benchmarks
go test -bench=. -benchmem ./...

# Output:
# BenchmarkGetUserJSON-8         2000000    723 ns/op   256 B/op   3 allocs/op
#                                ^count   ^time/op   ^memory/op ^allocations/op
# 
# -8 = 8 CPU cores
# 2M iterations in the benchmark
# 723 nanoseconds per operation
# 256 bytes allocated per operation
# 3 heap allocations per operation

# Compare two versions with benchstat:
go test -bench=. -count=5 ./... > old.txt
# (make your change)
go test -bench=. -count=5 ./... > new.txt
benchstat old.txt new.txt
# name              old time/op    new time/op    delta
# GetUserJSON-8       723ns ± 2%     421ns ± 1%   -41.76%  (p=0.008 n=5+5)
```

### The Execution Trace: Full System Visibility

The trace tool gives you a timeline of everything Go is doing — goroutines, GC, syscalls, scheduler events:

```bash
# Collect a 5-second trace
curl -o trace.out "http://localhost:6060/debug/pprof/trace?seconds=5"

# View in the browser
go tool trace trace.out
# Opens http://localhost:XXXX with:
# - Goroutine analysis (how many goroutines, where they're blocked)
# - Network blocking profile
# - Synchronization blocking profile
# - System calls
# - GC pauses (shown as vertical bars)
```

GC pauses are visible as red vertical bars interrupting all goroutines. If you see frequent long GC pauses, your program is allocating too much (the GC is running constantly to keep up).

## 3. Real-World Context

### CockroachDB's Performance Engineering

CockroachDB (distributed SQL, written in Go) has an entire engineering practice around pprof. When a customer reports slow queries, their engineers request a 30-second pprof heap + CPU profile from the customer's cluster. Because pprof is built-in and safe for production, this is zero-risk and instant.

Their engineering blog has posts where they trace performance improvements through pprof flamegraphs — from 40% CPU time in memory allocations down to 8% after optimizing hot paths with `sync.Pool` and pre-allocated buffers.

## 4. Senior Engineer Perspective

**What makes Go profiling uniquely good:**

1. **It's built in** — no external agent, no JVM JMX, no pip install
2. **Safe for production** — sampling overhead is under 1%
3. **Multiple profile types in one tool** — CPU, heap, goroutine, mutex, block
4. **Instant** — hit an HTTP endpoint, get a profile back in 30 seconds

**The benchmark workflow:**

Every performance optimization in Go should be driven by benchmarks. The process:
1. Write a benchmark for the function you want to optimize
2. Run it, record the baseline numbers
3. Make your change
4. Run it again, compare with `benchstat`
5. Only ship if `benchstat` shows a statistically significant improvement

This gives you reproducible, objective evidence that your optimization worked — not "it feels faster in testing."

**The goroutine leak pattern:**

A goroutine "leak" is a goroutine that was spawned but never exits. Over time, leaked goroutines accumulate and consume memory. The goroutine profile (`/debug/pprof/goroutine`) shows you all running goroutines and their stack traces. If the count grows over hours, you have a leak. The stack trace tells you where the goroutine was created.
