## 1. What Is It? (Theory)

A database connection is not free. Opening one requires:
- A TCP handshake (~1ms same datacenter)
- TLS negotiation if SSL is enabled (~5ms additional)
- PostgreSQL authentication protocol exchange (~2ms)
- PostgreSQL spawning a backend process (~10ms on a busy server)

Total: **15–20ms per new connection**. For a service handling 1,000 requests/second, creating a new connection per request would cost 15–20 seconds of pure connection overhead every second — clearly impossible.

Connection pooling solves this by maintaining a set of persistent, pre-authenticated connections and reusing them across requests. The TCP handshake happens once at pool startup, not once per request.

**The real-world analogy:**

Imagine a bank with 100 customers walking in per hour. Each transaction takes 5 minutes. Without pooling: each customer has their own dedicated teller who sits idle between transactions (100 tellers needed). With pooling: 10 tellers serve all 100 customers by picking up the next customer the moment the current one finishes (10 tellers, no idle waste).

The connection pool is the teller management system.

## 2. How It Actually Works (Deep Mechanics)

### Three Layers of Connection Pooling

Connection pooling can happen at three different places in your stack, each with different properties:

```
APPLICATION POOL (asyncpg, SQLAlchemy, HikariCP):
  Lives inside your application process.
  Each application instance has its own pool.
  
  Scope: one application server → database
  
  State: connections are NOT shared across app instances.
  Problem: 10 app servers × 20 pool connections = 200 connections to DB.
           With 100 app servers: 2,000 connections → PostgreSQL overwhelmed.

PROCESS-LEVEL POOL (PgBouncer):
  An external proxy that pools connections between your apps and the database.
  All app servers → PgBouncer → database.
  
  Scope: all application servers → database (centralized)
  
  Benefit: 1,000 client connections → 20 actual DB connections.
           The DB only sees 20 connections, regardless of how many app servers.

CONNECTION MULTIPLEXING (pgpool-II, Odyssey):
  More complex proxy that can also load-balance and replicate.
  Usually overkill — PgBouncer covers 99% of needs.
```

### PgBouncer in Depth

PgBouncer is a lightweight connection pooler for PostgreSQL. It runs as a separate process alongside your database or on a proxy server, accepting connections from your applications and multiplexing them onto fewer actual PostgreSQL connections.

```ini
# /etc/pgbouncer/pgbouncer.ini

[databases]
# Alias: myapp → actual connection details
myapp = host=localhost port=5432 dbname=myapp

[pgbouncer]
listen_port = 6432
listen_addr = *

# Pool mode is the most important decision:
pool_mode = transaction
# session:     One DB connection per client session (stateful; supports all PG features)
# transaction: One DB connection per transaction (most efficient; can't use SET, advisory locks)
# statement:   One DB connection per statement (almost never use)

max_client_conn = 10000    # max connections from your applications
default_pool_size = 20     # how many real PostgreSQL connections per database/user
min_pool_size = 5           # keep this many connections open even when idle
reserve_pool_size = 5       # emergency reserve for when pool is exhausted
server_idle_timeout = 600   # close idle server connections after 10 minutes

auth_type = md5
auth_file = /etc/pgbouncer/userlist.txt
```

**Pool mode trade-offs:**

```
TRANSACTION MODE (default recommendation):
  Client connects → starts a transaction → PgBouncer assigns a DB connection
  Transaction commits/rollbacks → DB connection returned to pool
  
  Benefit: 10,000 client connections → 20 actual DB connections
  Limitation: Cannot use these PG features between transactions:
    - SET, SET LOCAL (session settings)
    - Advisory locks (pg_advisory_lock)
    - LISTEN/NOTIFY (async notifications)
    - Prepared statements with certain configs
    - Temporary tables (use WITH / CTEs instead)
  
SESSION MODE (for compatibility):
  Client connects → gets a dedicated DB connection for entire session
  Client disconnects → connection returned to pool
  
  Benefit: Full PG feature compatibility
  Limitation: Less efficient — one DB connection per concurrent client
  Use when: you need session-level features PgBouncer can't proxy

CHOOSING:
  Start with TRANSACTION mode.
  Only fall back to SESSION mode if you hit a specific incompatibility.
```

### Application Pool Sizing with Little's Law

```python
import asyncpg

# Pool size formula (from topic 9.3):
# min_pool_size = target_qps × avg_query_latency_seconds
# with 1.5–2x headroom

# Example: FastAPI service with PostgreSQL
pool = await asyncpg.create_pool(
    dsn="postgresql://user:pass@pgbouncer:6432/myapp",
    
    # Sizing: 300 queries/sec expected, 30ms average query time
    # min = 300 × 0.030 = 9 connections
    # With 1.5x headroom: 14
    min_size=5,    # keep this many connections open always
    max_size=20,   # absolute maximum (add headroom + burst capacity)
    
    # Timeout behavior:
    command_timeout=30.0,     # kill queries running longer than 30s
    timeout=10.0,             # if pool is exhausted, wait max 10s for a connection
                              # instead of waiting forever (which would queue up threads)
    
    # Connection health:
    max_inactive_connection_lifetime=300.0,  # recycle idle connections every 5 min
)
```

### Detecting Pool Exhaustion

Pool exhaustion is one of the most common "why is the service down" incidents. It's subtle because the service isn't actually down — it's just queuing forever trying to get a database connection.

```
SYMPTOMS OF POOL EXHAUSTION:
  - Request latency spikes to 10s, 30s, then timeouts
  - Error: "TimeoutError: await acquire() timeout"
  - Error: "too many connections" from PostgreSQL
  - CPU on application servers: near zero (waiting, not computing)
  - CPU on database: normal or low (not overwhelmed — it just has no work)
  - Active connections in pg_stat_activity: at exactly max_connections

WHAT'S HAPPENING:
  Application has 20-connection pool.
  Incoming traffic sends 25 concurrent DB-using requests.
  20 get connections immediately. 5 queue, waiting.
  If those 20 are slow (maybe they're waiting for each other due to locks):
  The 5 waiting requests hit the timeout.
  User sees errors.

ROOT CAUSES:
  1. Pool too small for the traffic (size it with Little's Law)
  2. Queries are slower than expected (causing longer hold times → lower throughput)
  3. Traffic spike beyond pool capacity (need backpressure, see topic 2.12)
  4. Slow queries holding connections (lock contention, missing indexes)
  5. Application bug: connection not returned to pool (forgotten await/with block)
```

```sql
-- Check current PostgreSQL connection count
SELECT
    count(*) AS total_connections,
    count(*) FILTER (WHERE state = 'active') AS active,
    count(*) FILTER (WHERE state = 'idle') AS idle,
    count(*) FILTER (WHERE state = 'idle in transaction') AS idle_in_transaction,
    count(*) FILTER (WHERE wait_event IS NOT NULL) AS waiting
FROM pg_stat_activity
WHERE datname = 'myapp';

-- "idle in transaction" is the dangerous one:
-- A transaction was opened but not yet committed or rolled back.
-- It holds locks. It blocks VACUUM.
-- Usually caused by: bug in application code, exception not triggering rollback,
--                    long computation between BEGIN and COMMIT.

-- Find idle-in-transaction connections (sorted by duration):
SELECT
    pid,
    usename,
    application_name,
    now() - xact_start AS transaction_age,
    query
FROM pg_stat_activity
WHERE state = 'idle in transaction'
ORDER BY transaction_age DESC;

-- Kill connections idle in transaction for > 5 minutes (nuclear option):
SELECT pg_terminate_backend(pid)
FROM pg_stat_activity
WHERE state = 'idle in transaction'
  AND now() - xact_start > INTERVAL '5 minutes';
```

### The idle_in_transaction Killer

This deserves its own section because it's such a common production incident:

```
SCENARIO:
  1. Application opens a DB connection
  2. BEGIN transaction
  3. Queries some data
  4. Makes an external API call (Stripe, SendGrid, etc.)
  5. External API takes 45 seconds (timeout or slowness)
  6. During those 45 seconds: the DB connection is held, the transaction is open
  7. Any rows touched in step 3 are locked
  8. Any other queries needing those rows must wait
  9. If this happens for 10 concurrent requests: 10 connections locked, cascading waits

THE FIX:
  - Never hold a database transaction open while making external calls
  - Pattern: do all external work first, then open transaction and write atomically
  - Set idle_in_transaction_session_timeout:
  
  ALTER SYSTEM SET idle_in_transaction_session_timeout = '60s';
  SELECT pg_reload_conf();
  -- Connections idle in transaction > 60s are automatically terminated
```

## 3. Real-World Context

### Instagram's PgBouncer Configuration

Instagram (2016 Scaling Instagram post) documented running PgBouncer with transaction mode to handle millions of connections from their Django/Python application servers, funneled down to a much smaller number of actual PostgreSQL connections. Their configuration insight: pool_size should be set based on how many queries the database can handle concurrently at the target latency — not based on "how many connections feel comfortable."

### The Heroku Postgres Connection Limit

Heroku's small PostgreSQL plans have a limit of 25 connections. With multiple dyno workers, each needing a connection pool, teams quickly hit this limit. Heroku's solution: add PgBouncer between your app and the database. Their managed "Heroku Data for PostgreSQL" includes a PgBouncer sidecar option for exactly this reason.

This is not a Heroku-specific problem — it's universal. AWS RDS `db.t3.micro` allows 66 connections max. With 5 app servers × 20-connection pools = 100 connections. You're already over the limit. PgBouncer reduces the actual DB connections to 10-20 regardless of app server count.

## 4. Code Examples

### PgBouncer Health Check

```python
import asyncpg

async def check_pgbouncer_stats(pgbouncer_host: str, port: int = 6432) -> dict:
    """
    Connect to PgBouncer's admin database and fetch pool statistics.
    This tells you if your pools are healthy.
    """
    conn = await asyncpg.connect(
        host=pgbouncer_host,
        port=port,
        database="pgbouncer",  # special PgBouncer admin database
        user="pgbouncer_admin",
    )
    
    stats = await conn.fetch("SHOW POOLS;")
    await conn.close()
    
    result = []
    for pool in stats:
        utilization = (pool["cl_active"] + pool["cl_waiting"]) / max(pool["sv_pool_size"], 1)
        result.append({
            "database": pool["database"],
            "user": pool["user"],
            "clients_active": pool["cl_active"],   # clients currently using a server connection
            "clients_waiting": pool["cl_waiting"],  # clients waiting for a connection
            "servers_active": pool["sv_active"],    # server connections in use
            "servers_idle": pool["sv_idle"],        # server connections available
            "pool_size": pool["sv_pool_size"],
            "utilization_pct": round(utilization * 100, 1),
            "status": "⚠️ HIGH" if pool["cl_waiting"] > 5 else "✅ OK",
        })
    
    return result

# Use in a health check or monitoring dashboard
# Alert if cl_waiting > 0 for more than 30 seconds → pool is exhausted
```

## 5. Senior Engineer Perspective

**The one configuration that saves on-call engineers:**

Set `idle_in_transaction_session_timeout = '60s'` on your PostgreSQL server. This automatically kills connections that open a transaction and then hold it idle for over a minute. Without this, a single application bug (or external API timeout) can lock rows and block other queries indefinitely. This timeout is the backstop that prevents a small bug from cascading into a full outage.

**The pool sizing mistake everyone makes:**

Setting max_size too high because "more connections = more capacity." Above a certain point, more connections hurt because PostgreSQL spends more time on context switching and lock management between them. The sweet spot for PostgreSQL is usually 2–4 connections per CPU core on the database server. A 4-core database: 8–16 connections handles more throughput than 200 connections at 80% utilization of those 200. PgBouncer lets your application have 10,000 client connections while PostgreSQL only sees 16.
