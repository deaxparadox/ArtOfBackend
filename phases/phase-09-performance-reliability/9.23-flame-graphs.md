## 1. Flame Graphs: CPU Profiling in Production (9.23)

### What Is a Flame Graph?

A flame graph is a visualization of CPU time distribution across a call stack. It was invented by Brendan Gregg (Netflix) to make CPU profiler output understandable at a glance.

Every profiler produces call stack data: "at this moment, the CPU was executing function F, called by function E, called by function D..." A flame graph visualizes thousands of such samples as stacked rectangles.

**Reading a flame graph:**

```
FLAME GRAPH ANATOMY:

  ┌──────────────────────────────────────────────────────────────────────────────┐
  │                            main()                                            │
  ├─────────────────────────────────────┬────────────────────────────────────────┤
  │       handle_request() [60%]        │    background_tasks() [40%]            │
  ├───────────────────┬─────────────────┤                                        │
  │  db_query() [45%] │ auth() [15%]    │                                        │
  ├──────────────────────────────────── │────────────────────────────────────────┤
  │     psycopg2      │  jwt.decode()   │          celery.execute()              │
  └──────────────────────────────────────────────────────────────────────────────┘
         ↑                                          ↑
    Wide = more CPU time              These are your targets

RULES:
  - X-axis = alphabetical (within a stack level) NOT time order
  - Y-axis = call depth (bottom = entry point, top = leaf function)
  - Width = proportion of total CPU time
  - Look for: wide blocks near the top (hot leaf functions)
  - Look for: a wide block with nothing above it (the actual work)
  - Ignore: narrow blocks (they're fast, don't matter)
```

### Continuous Profiling: Always-On Production Profiling

**Traditional profiling:** You suspect a problem → run a profiler for 30 seconds → analyze. Reactive.

**Continuous profiling:** A low-overhead sampling profiler runs all the time, sending data to a central store. When an incident occurs, you already have profile data from before, during, and after. Proactive.

```bash
# Pyroscope: open-source continuous profiling server
# Install the server
docker run -p 4040:4040 grafana/pyroscope

# Python integration (auto-instruments FastAPI/Django)
pip install pyroscope-io

# In your application startup:
import pyroscope

pyroscope.configure(
    application_name = "my-api",
    server_address   = "http://pyroscope:4040",
    sample_rate      = 100,         # samples per second (low overhead)
    detect_subprocesses = True,
    oncpu_profiling  = True,        # CPU time profiling
    memory_profiling = True,        # heap profiling
    gil_only         = False,       # profile all threads, not just GIL-holder
    log_level        = "error",
)

# That's it. Pyroscope now continuously profiles in the background.
# Access the UI at http://localhost:4040 for flame graphs at any time range.
```

```go
// Go: pprof continuous profiling with Pyroscope
import "github.com/grafana/pyroscope-go"

func main() {
    pyroscope.Start(pyroscope.Config{
        ApplicationName: "my-api",
        ServerAddress:   "http://pyroscope:4040",
        ProfileTypes: []pyroscope.ProfileType{
            pyroscope.ProfileCPU,
            pyroscope.ProfileInuseObjects,
            pyroscope.ProfileAllocObjects,
            pyroscope.ProfileInuseSpace,
            pyroscope.ProfileAllocSpace,
        },
    })
    
    // Start your server normally
    http.ListenAndServe(":8080", handler)
}
```

**The power of continuous profiling for incidents:**

```
WITHOUT CONTINUOUS PROFILING:
  Incident occurs at 2pm.
  Engineers start profiling at 2:05pm.
  Profiling shows "db_query() taking 40% of CPU."
  But was this also happening yesterday? When did it start?
  Unknown — no historical data.

WITH CONTINUOUS PROFILING:
  Incident occurs at 2pm.
  Engineers open Pyroscope, look at the 1:55pm → 2:10pm window.
  Flame graph at 1:55: db_query() = 15% CPU (normal)
  Flame graph at 2:01: db_query() = 65% CPU (incident starts here)
  Engineers check what changed at 2:01: a deploy went out at 2:00.
  Root cause: new code introduced an unindexed query.
  Resolution time: minutes, not hours.
```

## 2. Goroutine and Thread Leaks (9.24)

### What Is a Goroutine Leak?

A goroutine (or thread) leak is when goroutines/threads are created but never exit. Over time, the leaked goroutines accumulate, consuming memory and potentially causing the program to slow down or crash.

In Go: each goroutine uses ~2–8KB of stack memory. One million leaked goroutines = 2–8GB of RAM just for stacks.

**The real-world analogy:**

Hiring contractors to complete tasks, but they never leave after finishing. They still occupy a desk, use office resources, and consume the coffee. Eventually, you run out of desks and coffee — even though most contractors are idle.

---

### Common Goroutine Leak Patterns

**Pattern 1: Channel leak — goroutine blocked on empty channel**

```go
// THE LEAK:
func processRequests(requests []Request) {
    results := make(chan Result)  // unbuffered channel
    
    for _, req := range requests {
        go func(r Request) {
            result := process(r)
            results <- result  // blocks if nobody reads!
        }(req)
    }
    
    // What if we return early due to an error?
    // → All goroutines are blocked on results <- result
    // → They'll never exit
    // → Memory leak

    for i := 0; i < len(requests); i++ {
        fmt.Println(<-results)
    }
}

// THE FIX: use context cancellation and buffered channels
func processRequests(ctx context.Context, requests []Request) {
    results := make(chan Result, len(requests))  // buffered: never blocks
    
    for _, req := range requests {
        go func(r Request) {
            select {
            case <-ctx.Done():
                return  // context cancelled: goroutine exits
            case results <- process(r):
            }
        }(req)
    }
    
    for i := 0; i < len(requests); i++ {
        select {
        case <-ctx.Done():
            return
        case result := <-results:
            fmt.Println(result)
        }
    }
}
```

**Pattern 2: HTTP server goroutine that never reads the response body**

```go
// THE LEAK in Go:
resp, err := http.Get("https://api.example.com/data")
if err != nil { return err }
// Forgot to read and close the body!
// The underlying TCP connection is held open by the goroutine serving the response body.

// THE FIX: ALWAYS close the response body
resp, err := http.Get("https://api.example.com/data")
if err != nil { return err }
defer resp.Body.Close()              // ← critical
body, err := io.ReadAll(resp.Body)  // also critical: read it all before closing
```

### Detecting Goroutine Leaks

**Using pprof goroutine profile:**

```bash
# See all goroutines and their stacks
curl http://localhost:6060/debug/pprof/goroutine?debug=2 | head -100

# If the goroutine count is growing:
# Take a snapshot every 5 minutes for 30 minutes and compare counts
for i in {1..6}; do
    count=$(curl -s http://localhost:6060/debug/pprof/goroutine | grep "^goroutine " | wc -l)
    echo "$(date): $count goroutines"
    sleep 300
done
# Stable count: no leak
# Growing count: goroutine leak
```

**goleak: automated goroutine leak detection in tests**

```go
// In tests, automatically fail if goroutines are leaked
package main_test

import (
    "testing"
    "go.uber.org/goleak"
)

func TestMyFunction(t *testing.T) {
    defer goleak.VerifyNone(t)  // ← fails test if goroutines leak
    
    // Your test code
    result := myFunction()
    
    // goleak checks at the end of the test: are there more goroutines now?
    // If yes: goroutine leak detected → test fails with leak location
}

// Or use in TestMain for all tests:
func TestMain(m *testing.M) {
    goleak.VerifyTestMain(m)
}
```

**Python thread leak detection:**

```python
import threading
import time

def check_thread_count():
    """Monitor active threads over time. Stable = no leak."""
    baseline = threading.active_count()
    
    while True:
        current = threading.active_count()
        if current > baseline + 50:
            # More than 50 extra threads appeared
            import logging
            logging.warning(
                "thread_leak_suspected",
                baseline=baseline,
                current=current,
                delta=current - baseline,
            )
        time.sleep(60)

# Periodic thread reporting (add to your monitoring)
from prometheus_client import Gauge
thread_count_gauge = Gauge("python_threads_total", "Active Python threads")

async def report_thread_count():
    while True:
        thread_count_gauge.set(threading.active_count())
        await asyncio.sleep(15)
```

## 3. The Diagnostic Decision Tree

```
SERVICE IS GETTING SLOW OR CRASHING — WHERE TO LOOK:

Memory growing indefinitely?
  → Likely memory leak
  → Check: tracemalloc comparison, heap profile
  → Look for: growing global caches, event listeners, closures

OOM killed repeatedly?
  → Memory leak (see above) or allocation rate too high
  → Check: max RSS at time of kill, heap snapshot
  → Look for: large allocations per request

Goroutine count growing indefinitely?
  → Goroutine leak
  → Check: pprof goroutine profile, goleak in tests
  → Look for: blocked channels, unreleased HTTP connections

CPU at 100% but throughput low?
  → CPU-bound bottleneck
  → Check: CPU flamegraph (py-spy, pprof)
  → Look for: wide blocks at top of flamegraph

Requests queueing, low CPU?
  → I/O bottleneck or pool exhaustion
  → Check: DB connection count, pool utilization, slow query log
  → Look for: connections stuck in "idle in transaction", high wait events

Service slow only at high traffic?
  → Concurrency problem — pool exhaustion, lock contention
  → Check: Little's Law calculation, DB wait events, p99 vs p50
  → Look for: queue depth growing under load
```
