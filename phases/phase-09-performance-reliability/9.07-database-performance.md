## 1. What Is It? (Theory)

For the vast majority of web applications, **the database is the bottleneck**. Not the Python interpreter. Not the Go garbage collector. Not JSON serialization. The database.

Why? Because your application server processes requests in memory at nanosecond-to-microsecond speeds. Your database involves disk I/O, lock management, query planning, and network round-trips — all operating at millisecond speeds.

The implication is direct: **database performance is application performance**. A team that knows how to optimize database queries will outperform a team that knows how to optimize application code, because they're targeting the real bottleneck.

**The real-world analogy:**

Think of a restaurant. The kitchen (your application code) can plate a dish in seconds. But if the delivery driver bringing ingredients from the warehouse takes 45 minutes per trip (your database query), the restaurant is slow regardless of how fast the kitchen works.

The kitchen efficiency is irrelevant if the ingredient delivery is the constraint.

## 2. How It Actually Works (Deep Mechanics)

### Where Database Time Goes

When a SQL query executes, time is divided across these phases:

```
QUERY LIFECYCLE:
  
  1. PARSING (1–5ms for complex queries, often <1ms)
     SQL text → Abstract Syntax Tree
     Syntax errors caught here.

  2. PLANNING (1–50ms for complex queries, often <5ms)
     Query planner evaluates multiple execution strategies.
     Chooses the estimated cheapest plan.
     Uses statistics about table sizes, index selectivity.
     Bad statistics → wrong plan → slow query.

  3. EXECUTION (varies enormously)
     Actual work: reading pages, applying predicates, sorting, joining.
     This is almost always where the time is.
     Sub-phases:
       a. Index lookup (O(log n)) → fast
       b. Sequential scan (O(n)) → can be fast for small tables
       c. Hash join / Merge join
       d. Sort (O(n log n)) if ORDER BY without index
       e. Aggregation

  4. DATA TRANSFER (proportional to result size)
     Returning rows to client.
     A query returning 100,000 rows is slow even if execution was fast.
     LIMIT early; never select * in production.
```

### pg_stat_statements: Your Database Performance Dashboard

`pg_stat_statements` is a PostgreSQL extension that tracks statistics for every query executed. It's the equivalent of having a profiler permanently running on your database.

```sql
-- Enable (add to postgresql.conf, then restart):
-- shared_preload_libraries = 'pg_stat_statements'
-- pg_stat_statements.track = all

-- View the extension
CREATE EXTENSION IF NOT EXISTS pg_stat_statements;

-- Find your slowest queries (by mean execution time)
SELECT
    round(mean_exec_time::numeric, 2) AS mean_ms,
    round(total_exec_time::numeric, 2) AS total_ms,
    calls,
    round(stddev_exec_time::numeric, 2) AS stddev_ms,   -- high stddev = inconsistent
    rows / calls AS avg_rows,
    left(query, 120) AS query_preview
FROM pg_stat_statements
WHERE calls > 100          -- only queries run at least 100 times
ORDER BY mean_exec_time DESC
LIMIT 20;

-- Find queries with the most total impact (mean × calls = total CPU time)
SELECT
    round((total_exec_time / 1000)::numeric, 2) AS total_seconds,
    calls,
    round(mean_exec_time::numeric, 2) AS mean_ms,
    left(query, 120) AS query_preview
FROM pg_stat_statements
ORDER BY total_exec_time DESC
LIMIT 20;

-- Find queries doing the most I/O (shared_blks_read = pages from disk)
SELECT
    shared_blks_read,
    shared_blks_hit,
    round(shared_blks_hit::numeric / NULLIF(shared_blks_hit + shared_blks_read, 0) * 100, 1) AS cache_hit_rate,
    round(mean_exec_time::numeric, 2) AS mean_ms,
    calls,
    left(query, 120) AS query_preview
FROM pg_stat_statements
WHERE shared_blks_read > 1000
ORDER BY shared_blks_read DESC
LIMIT 20;

-- Reset statistics (do this after a major change to get clean data)
SELECT pg_stat_statements_reset();
```

**Key metrics to watch:**

```
mean_exec_time:  Average milliseconds per execution.
                 Red flag: > 100ms for OLTP queries.
                 
total_exec_time: This query's total contribution to DB load.
                 A fast query called 1M times/day matters more than a slow query called once.
                 
stddev_exec_time: Standard deviation of execution time.
                  High stddev = sometimes fast, sometimes slow.
                  Points to: lock contention, vacuum conflicts, parameter sniffing.

rows / calls:    Average rows returned.
                 > 10,000 average rows → almost certainly should be paginated.
                 
shared_blks_read: Pages read from disk (not cache).
                  High disk reads = working set doesn't fit in shared_buffers.
                  Fix: increase shared_buffers, or add better indexes.
```

### The Slow Query Log

PostgreSQL can log queries that exceed a time threshold automatically:

```ini
# postgresql.conf
log_min_duration_statement = 1000   # log queries taking > 1 second
                                     # Set to 0 to log ALL queries (dev only)
                                     # Set to -1 to disable
log_statement = 'none'               # don't log every statement (too noisy)
log_duration = off

# For development: log everything (warning: very verbose under load)
log_min_duration_statement = 0
```

```bash
# View slow queries in PostgreSQL logs
tail -f /var/log/postgresql/postgresql-*.log | grep "duration:"

# Example output:
# 2024-06-15 14:30:45 UTC [12345] LOG:  duration: 2345.123 ms
#   statement: SELECT * FROM orders WHERE user_id = 123 ORDER BY created_at DESC
```

The slow query log is your first alert that something is wrong. When p99 latency spikes in production, check the slow query log first.

### EXPLAIN ANALYZE: The Query Surgeon's Tool

See the detailed treatment in [5.11 — EXPLAIN ANALYZE](../phase-05-databases/5.11-explain-analyze.md). The essential pattern:

```sql
-- Always use EXPLAIN ANALYZE (not just EXPLAIN) — ANALYZE actually runs the query
-- Always use BUFFERS — shows disk vs. cache reads
-- Wrap in BEGIN/ROLLBACK to avoid side effects for DML

BEGIN;
EXPLAIN (ANALYZE, BUFFERS, FORMAT TEXT)
  SELECT u.name, COUNT(o.id) AS order_count
  FROM users u
  JOIN orders o ON u.id = o.user_id
  WHERE u.created_at > NOW() - INTERVAL '30 days'
  GROUP BY u.id, u.name
  ORDER BY order_count DESC
  LIMIT 20;
ROLLBACK;

-- READ THE OUTPUT: look for these red flags:
-- "Seq Scan on [large table]" → missing index
-- "(cost=... rows=1) (actual rows=50000)" → bad statistics, run ANALYZE
-- "Sort Method: external merge Disk" → sort spilling to disk, increase work_mem
-- "(loops=N)" with large N in a nested loop → N+1 problem at DB level
-- "Buffers: shared hit=100 read=50000" → mostly disk reads, not cache
```

### Index Usage Statistics

```sql
-- Which indexes are actually being used?
SELECT
    schemaname,
    tablename,
    indexname,
    idx_scan AS times_used,
    pg_size_pretty(pg_relation_size(indexrelid)) AS index_size
FROM pg_stat_user_indexes
ORDER BY idx_scan ASC;  -- low idx_scan = rarely used

-- Unused indexes (cost writes, never help reads → drop them)
SELECT
    indexname,
    tablename,
    pg_size_pretty(pg_relation_size(indexrelid)) AS wasted_size
FROM pg_stat_user_indexes
WHERE idx_scan = 0
  AND indexrelid NOT IN (SELECT conindid FROM pg_constraint);
-- idx_scan=0 AND not a constraint index = completely unused

-- Cache hit rate (should be > 99% in production)
SELECT
    sum(heap_blks_hit) AS cache_hits,
    sum(heap_blks_read) AS disk_reads,
    round(
        sum(heap_blks_hit)::numeric /
        NULLIF(sum(heap_blks_hit) + sum(heap_blks_read), 0) * 100,
    2) AS cache_hit_rate
FROM pg_statio_user_tables;
-- < 99%: working set doesn't fit in shared_buffers
-- Fix: increase shared_buffers (set to 25% of available RAM)
```

## 3. Real-World Context

### The GitHub Database Incident (2021)

GitHub has written publicly about database incidents where a single unindexed query caused cascading slowdowns. One case: a rare but valid code path generated a query without a WHERE clause on an indexed column, causing a sequential scan of a multi-billion row table. This one query saturated I/O and slowed the entire replica cluster.

Their lesson: every query that runs in production must have been tested with EXPLAIN on production data volumes. Development data is too small to reveal full-table scan problems.

### The Shopify Black Friday Database Tuning

Every year Shopify conducts a "flash sale" drill before Black Friday to find database bottlenecks. Their process is straightforward:
1. Enable pg_stat_statements
2. Run the load test
3. Sort by total_exec_time DESC
4. Run EXPLAIN ANALYZE on the top 10 queries
5. Add missing indexes, fix query patterns
6. Repeat until all top queries are sub-100ms

This process is responsible for Black Friday ($5.6B in 2023) running without database incidents — not heroic engineering, just systematic profiling and optimization.

## 4. Code Examples

### Automating Query Performance Monitoring

```python
import asyncpg
import structlog

log = structlog.get_logger()

class SlowQueryDetector:
    """
    Middleware for asyncpg that logs slow queries with their execution plan.
    Use in staging/development to catch slow queries before they hit production.
    """
    
    SLOW_THRESHOLD_MS = 100
    
    def __init__(self, conn: asyncpg.Connection):
        self.conn = conn
    
    async def execute_with_timing(self, query: str, *args):
        import time
        start = time.perf_counter()
        
        try:
            result = await self.conn.fetch(query, *args)
            elapsed_ms = (time.perf_counter() - start) * 1000
            
            if elapsed_ms > self.SLOW_THRESHOLD_MS:
                log.warning(
                    "slow_query_detected",
                    elapsed_ms=round(elapsed_ms, 2),
                    query=query[:200],
                    threshold_ms=self.SLOW_THRESHOLD_MS,
                )
                # In development: automatically run EXPLAIN ANALYZE
                if is_development():
                    await self._explain(query, *args)
            
            return result
        except Exception as e:
            log.error("query_failed", query=query[:200], error=str(e))
            raise

    async def _explain(self, query: str, *args):
        """Run EXPLAIN ANALYZE and log the plan for debugging."""
        plan = await self.conn.fetch(
            f"EXPLAIN (ANALYZE, BUFFERS) {query}", *args
        )
        log.info("query_plan", plan="\n".join(row[0] for row in plan))
```

### Weekly Database Health Report

```python
async def database_health_report(pool: asyncpg.Pool) -> dict:
    """Generate a weekly performance report from pg_stat_statements."""
    async with pool.acquire() as conn:
        # Slowest queries by mean time
        slow_queries = await conn.fetch("""
            SELECT
                round(mean_exec_time::numeric, 1) AS mean_ms,
                calls,
                round(total_exec_time::numeric / 1000, 1) AS total_seconds,
                left(query, 200) AS query
            FROM pg_stat_statements
            WHERE calls > 50
            ORDER BY mean_exec_time DESC
            LIMIT 10
        """)
        
        # Highest impact queries (mean × calls)
        high_impact = await conn.fetch("""
            SELECT
                round(total_exec_time::numeric / 1000, 1) AS total_seconds,
                calls,
                round(mean_exec_time::numeric, 1) AS mean_ms,
                left(query, 200) AS query
            FROM pg_stat_statements
            ORDER BY total_exec_time DESC
            LIMIT 10
        """)
        
        # Unused indexes
        unused_indexes = await conn.fetch("""
            SELECT indexname, tablename,
                   pg_size_pretty(pg_relation_size(indexrelid)) AS size
            FROM pg_stat_user_indexes
            WHERE idx_scan = 0
              AND indexrelid NOT IN (SELECT conindid FROM pg_constraint)
            ORDER BY pg_relation_size(indexrelid) DESC
        """)
        
        return {
            "slowest_queries": [dict(r) for r in slow_queries],
            "highest_impact_queries": [dict(r) for r in high_impact],
            "unused_indexes": [dict(r) for r in unused_indexes],
        }
```

## 5. Senior Engineer Perspective

**What a junior gets wrong:**

Optimizing "the SQL query" by rewriting it more cleverly, when the real fix is adding an index. A query doing a sequential scan of 10 million rows cannot be "optimized in SQL" — it needs an index. No amount of query restructuring helps when the execution plan requires reading every row.

The second trap: looking only at slow queries. A query taking 200ms run once is a minor concern. A query taking 5ms run 100,000 times per minute is consuming 8 minutes of CPU per minute — it's the real bottleneck. Sort pg_stat_statements by **total_exec_time**, not just mean_exec_time.

**The database performance triage checklist:**

```
□ Is pg_stat_statements enabled?
□ What are the top 5 queries by total_exec_time?
□ EXPLAIN ANALYZE each of those top 5 queries
□ Are there sequential scans on large tables? → Add indexes
□ Are estimates way off from actuals? → Run ANALYZE tablename
□ Is cache hit rate > 99%? → If not, increase shared_buffers
□ Are there unused indexes consuming write overhead? → Drop them
□ Is average rows per query reasonable (< 10,000)? → Add pagination
□ Are any queries returning > 100ms consistently? → They need EXPLAIN treatment
```
