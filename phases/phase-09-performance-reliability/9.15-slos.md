## 1. The Three Terms Defined (9.15)

These three acronyms are often confused because they sound similar. They are distinct concepts with a specific relationship to each other.

### SLI — Service Level Indicator

An SLI is a **metric** — a specific, quantifiable measurement of some aspect of your service's behavior.

```
EXAMPLES OF SLIs:
  Request success rate = successful_requests / total_requests
  Request latency p99 = 99th percentile of response time distribution
  Availability = uptime_seconds / total_seconds
  Data freshness = time since last successful data sync
  Queue processing rate = messages processed / messages received
  
AN SLI IS JUST A NUMBER.
  It has no meaning until you attach a threshold to it.
  "Our p99 latency is 230ms" — is that good or bad?
  Depends on your target (SLO).
```

### SLO — Service Level Objective

An SLO is an **internal target** — the threshold you're trying to maintain for an SLI.

```
EXAMPLES OF SLOs:
  p99 request latency < 200ms
  Request success rate > 99.9%
  99.95% availability measured monthly
  Data freshness: sync completes within 30 minutes of source change
  
AN SLO IS YOUR INTERNAL COMMITMENT TO QUALITY.
  It's not a contract with customers (that's an SLA).
  It's the engineering team's standard.
  If you violate your SLO: internal alert, engineering prioritizes fixing it.
  
HOW STRICT TO SET IT:
  Current performance: p99 = 150ms, success rate = 99.95%
  Set SLO tighter than current: p99 < 200ms (buffer for degradation)
  Don't set it so tight it's constantly alerting.
  Don't set it so loose it's meaningless.
  Review and tighten quarterly as the system matures.
```

### SLA — Service Level Agreement

An SLA is a **contractual promise** — the commitment made to external customers or in a business contract, typically with financial penalties for violations.

```
EXAMPLES OF SLAs:
  "99.9% availability per month, or credits applied"
  "p99 API response time < 500ms for paid tier"
  "Support response within 4 business hours for enterprise customers"
  
AN SLA IS A BUSINESS COMMITMENT, NOT AN ENGINEERING TARGET.

RELATIONSHIP TO SLO:
  Your SLA must be LESS strict than your SLO (internal buffer).
  
  Why? If your SLO is 99.9% availability, you'll occasionally dip to 99.85%.
  If your SLA also promises 99.9%: you're paying penalties too often.
  
  Typical gap: SLO = 99.95% → SLA = 99.9%
  The gap gives you breathing room between "we're working on it" and "we owe credits."

FINANCIAL STAKES:
  AWS's EC2 SLA is 99.99% monthly uptime, or up to 30% service credits.
  Stripe's SLA for payment processing is 99.99%.
  Missing your SLA has real financial consequences.
  This is why SLAs are set conservatively relative to actual performance.
```

## 2. Error Budgets: How Reliability Trades Against Velocity (9.16)

An error budget is the amount of unreliability you're *allowed* before your SLO is violated. It's the inverse of your SLO: if you promise 99.9% availability, you have 0.1% "budget" for failures.

**The transformation this creates:**

Instead of "reliability is a goal we pursue endlessly," error budgets make reliability a **resource** — quantifiable, finite, and tradeable.

```
ERROR BUDGET CALCULATION:

SLO: 99.9% availability
Measurement period: 30 days

Total minutes in 30 days: 30 × 24 × 60 = 43,200 minutes
Allowed downtime (0.1% of 43,200): 43.2 minutes/month

If you've used:
  10 minutes → 33.2 minutes remaining → deploy fearlessly
  40 minutes → 3.2 minutes remaining → careful with risky changes
  50 minutes → -6.8 minutes → SLO violated → STOP new features, fix reliability
```

**How teams use error budgets:**

```
ERROR BUDGET IS AMPLE (< 50% burned):
  → Deploy frequently
  → Take risks with new features
  → Run chaos experiments
  → Do aggressive optimizations that might cause brief issues

ERROR BUDGET IS RUNNING LOW (50-90% burned):
  → Slow down deployments
  → Only deploy well-tested changes
  → Prioritize fixing reliability issues over new features
  → Investigate what burned the budget

ERROR BUDGET IS EXHAUSTED (> 100% burned):
  → Feature freeze (except reliability fixes)
  → On-call engineers focus exclusively on reliability
  → Post-mortem to understand what happened
  → Management is informed
  
ERROR BUDGET IS THE MECHANISM THAT ALIGNS ENGINEERING AND BUSINESS:
  "Why can't we ship this feature?" → "We have no error budget left this month."
  This is a quantitative answer. It removes the argument.
  Engineers can point to data, not opinion.
```

## 3. Measuring SLOs in Practice

### Setting Up SLO Dashboards with Prometheus

```python
from prometheus_client import Counter, Histogram

# Track what constitutes a "good" request for your SLO
request_total = Counter(
    "http_requests_total",
    "Total HTTP requests",
    ["method", "endpoint", "status_code"]
)

request_latency = Histogram(
    "http_request_duration_seconds",
    "Request latency",
    ["method", "endpoint"],
    buckets=[0.05, 0.1, 0.2, 0.5, 1.0, 2.5, 5.0, float("inf")]
)
```

```
PROMQL QUERIES FOR SLO DASHBOARDS:

# Availability SLO: what fraction of requests are "good" (2xx, 3xx)?
# SLO: 99.9% of requests should be successful (not 5xx)

# Good request rate
rate(http_requests_total{status_code!~"5.."}[5m])

# Total request rate  
rate(http_requests_total[5m])

# Availability (% good):
rate(http_requests_total{status_code!~"5.."}[5m]) 
  / rate(http_requests_total[5m])

# Error budget burn rate (are we burning budget faster than expected?):
1 - (
  rate(http_requests_total{status_code!~"5.."}[1h])
  / rate(http_requests_total[1h])
)
# If this number is > 0.001 (i.e., > 0.1% error rate) → burning budget

# Latency SLO: is p99 < 200ms?
histogram_quantile(0.99, rate(http_request_duration_seconds_bucket[5m])) < 0.2

# Multi-window error burn rate alert:
# Burn rate > 14.4 in last 1h AND burn rate > 6 in last 6h → page immediately
# This means: you're on track to exhaust your monthly budget in < 1 day
```

### The Multi-Window, Multi-Burn-Rate Alert

This is the alerting pattern recommended by Google's SRE book, designed to catch real SLO problems without excessive false alarms:

```yaml
# Alert rules for 99.9% availability SLO (30-day window)

groups:
- name: slo-alerts
  rules:
  # FAST BURN: burning 14.4x budget in 1h (budget gone in 2 days)
  - alert: SLOFastBurnRate
    expr: |
      (
        1 - (
          rate(http_requests_total{status_code!~"5.."}[1h])
          / rate(http_requests_total[1h])
        )
      ) > (14.4 * 0.001)
    for: 2m
    labels:
      severity: critical
    annotations:
      summary: "SLO error budget burning fast — check immediately"
      
  # SLOW BURN: burning 3x budget in 6h (budget gone in 10 days)
  - alert: SLOSlowBurnRate
    expr: |
      (
        1 - (
          rate(http_requests_total{status_code!~"5.."}[6h])
          / rate(http_requests_total[6h])
        )
      ) > (3 * 0.001)
    for: 15m
    labels:
      severity: warning
    annotations:
      summary: "SLO error budget burning slowly — investigate"
```

## 4. Real-World Context

### Google's SRE Book and Error Budgets

Google popularized error budgets in their "Site Reliability Engineering" book (2016). The key story: Google's SRE (Site Reliability Engineering) team used error budgets to resolve a cultural conflict between product teams (who wanted to ship fast) and reliability teams (who wanted to slow down).

The solution: *"If you have error budget remaining, deploy at whatever pace you want. If the budget runs out, the SRE team has authority to freeze deployments until reliability is restored."*

This made reliability discussions data-driven instead of opinion-driven. The product team couldn't argue with "you've used 95% of your budget" the same way they could argue with "I think this is too risky."

### Stripe's SLO Culture

Stripe defines SLOs for every API endpoint — not just aggregate availability. Their payment processing API has tighter SLOs than their reporting API, because a slow payment causes immediate user-visible harm, while a slow report is merely inconvenient.

This granularity is critical: a single "availability" SLO for your whole service hides which specific features are failing. You want to know: "Is it checkout that's slow, or is it only the order history page?"

## 5. Senior Engineer Perspective

**What a junior gets wrong:**

Setting an SLO based on what sounds good ("99.99%") without measuring what you currently achieve. If your service currently achieves 99.5% availability and you set an SLO of 99.99%, you'll be in constant violation, generating constant alerts, burning team morale. Set the SLO based on current measured performance, then improve from there.

**What separates a good SLO from a great one:**

A good SLO measures availability or latency. A great SLO measures what users actually care about — not "is the server up?" but "can users successfully complete a checkout?" A user-facing SLO based on checkout success rate is more valuable than a technical SLO on p99 HTTP response time.

**The error budget mindset:**

Every engineer should know the current error budget status the same way a pilot knows fuel levels. "We're at 40% budget burned this month, 20 days left" is the kind of visibility that prevents surprises. Teams that check this weekly make better deployment decisions than teams that discover they violated their SLO at the end of the month.
