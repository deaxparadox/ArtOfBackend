## 1. The Deployment Problem (Why These Patterns Exist)

The simplest deployment strategy is "stop the old version, start the new version." This guarantees downtime. At any moment between stop and start, users see errors.

The second simplest strategy is "start the new version on the same server, then switch traffic." This has zero downtime but if the new version is buggy, all users are affected immediately.

Blue/green and canary deployments are the production-grade answers to: **"How do we ship changes with zero downtime and minimal blast radius?"**

**The real-world analogy:**

**Blue/green** is like a pilot transferring flight control from the co-pilot to the pilot. The co-pilot (old version) is flying. The pilot (new version) takes the controls. If the pilot has a problem, the co-pilot takes back control immediately. At no point is nobody flying.

**Canary** is like a controlled drug trial — give the treatment to 1% of patients, watch for side effects, expand to 10%, 50%, 100% only if the first group is doing well. The 99% of patients (users) are protected while you verify.

## 2. Blue/Green Deployment (9.19)

### How It Works

```
STATE: Blue (v1.0) is running and serving 100% of traffic.

   Load Balancer
         │
         ▼
   [Blue: v1.0]    ← all traffic
   [Green: idle]   ← deployed but receiving no traffic

DEPLOYMENT STEP 1:
   Deploy v1.1 to Green environment.
   Green is now running v1.1 but receiving NO traffic.
   Run smoke tests against Green directly.

DEPLOYMENT STEP 2: Traffic switch
   Load Balancer
         │
         ├────► [Blue: v1.0]   ← 0% traffic now
         └────► [Green: v1.1]  ← 100% traffic now

   Traffic switched in seconds. Users experience no downtime.

ROLLBACK: if v1.1 has a critical bug
   Load Balancer
         │
         └────► [Blue: v1.0]  ← 100% traffic restored (Blue still running!)
   
   Rollback takes seconds. Blue was kept running just for this.
```

**Key requirement:** You need double the infrastructure — two identical environments. The "idle" environment exists only as a rollback option. This doubles infrastructure cost during the transition window.

### Blue/Green in Kubernetes

```yaml
# v1.0 deployment (Blue)
apiVersion: apps/v1
kind: Deployment
metadata:
  name: api-blue
spec:
  replicas: 5
  selector:
    matchLabels:
      app: api
      version: v1  # ← version label
  template:
    metadata:
      labels:
        app: api
        version: v1
    spec:
      containers:
      - name: api
        image: registry.example.com/api:v1.0

---
# Service: initially points to blue
apiVersion: v1
kind: Service
metadata:
  name: api-svc
spec:
  selector:
    app: api
    version: v1  # ← currently routing to v1 (Blue)
  ports:
  - port: 80
    targetPort: 8000
```

```bash
# DEPLOYMENT STEPS:

# Step 1: Deploy v2 as Green (no traffic yet — different selector)
kubectl apply -f api-green-deployment.yaml  # version: v2

# Step 2: Verify Green is healthy
kubectl get pods -l version=v2
kubectl exec -it $(kubectl get pod -l version=v2 -o jsonpath='{.items[0].metadata.name}') \
    -- curl http://localhost:8000/health

# Step 3: Switch traffic to Green (edit Service selector)
kubectl patch service api-svc \
    -p '{"spec": {"selector": {"version": "v2"}}}'
# This is instant — Service now routes to v2 pods

# Step 4: Verify metrics (watch for 2-5 minutes)
# Check: error rate, latency — any regression?

# Step 5a: Success → Delete Blue (save resources)
kubectl delete deployment api-blue

# Step 5b: Rollback → Switch back to Blue
kubectl patch service api-svc \
    -p '{"spec": {"selector": {"version": "v1"}}}'
# Instant rollback
```

## 3. Canary Deployment (9.20)

### How It Works

A canary deployment routes a small percentage of traffic to the new version while the majority remains on the old version. The name comes from "canary in a coal mine" — miners sent canaries in first to detect dangerous gases before humans entered.

```
CANARY PROGRESSION:

Week 0:  100% v1.0 (baseline)

Day 1:   1% v2.0 + 99% v1.0  ← watch for errors/regression
         If OK after 2 hours: continue

Day 2:   10% v2.0 + 90% v1.0 ← real traffic sample, bigger signals
         If OK: continue

Day 3:   25% → 50% → 100%    ← accelerate if metrics are clean

ROLLBACK AT ANY STEP:
         Reset to 100% v1.0
         Only 1-10% of users were affected by the bad version
```

### Why Canary > Blue/Green for Risk Management

Blue/green is all-or-nothing. If you switch to Green and it has a subtle bug that only affects 0.1% of requests, that 0.1% is 100% of your traffic. By the time you notice and rollback, thousands of users have been affected.

Canary is incremental. The 1% canary catches the bug while 99% of users are unaffected. Rollback affects only 1% of traffic, then returns to 100% v1.0.

### Canary in Kubernetes with Service Mesh (Istio)

```yaml
# VirtualService: controls traffic split
apiVersion: networking.istio.io/v1alpha3
kind: VirtualService
metadata:
  name: api-vs
spec:
  hosts:
  - api-svc
  http:
  - route:
    - destination:
        host: api-svc
        subset: v1   # stable
      weight: 90     # 90% of traffic
    - destination:
        host: api-svc
        subset: v2   # canary
      weight: 10     # 10% of traffic

---
# DestinationRule: defines subsets
apiVersion: networking.istio.io/v1alpha3
kind: DestinationRule
metadata:
  name: api-dr
spec:
  host: api-svc
  subsets:
  - name: v1
    labels:
      version: v1
  - name: v2
    labels:
      version: v2
```

```bash
# Gradually increase canary percentage
# After validating at 10% for 1 hour:
kubectl patch virtualservice api-vs --type merge \
    -p '{"spec":{"http":[{"route":[{"destination":{"host":"api-svc","subset":"v1"},"weight":50},{"destination":{"host":"api-svc","subset":"v2"},"weight":50}]}]}}'

# At 100%: remove the split entirely
kubectl patch virtualservice api-vs --type merge \
    -p '{"spec":{"http":[{"route":[{"destination":{"host":"api-svc","subset":"v2"},"weight":100}]}]}}'
```

### What to Observe During a Canary

```
METRICS TO WATCH:

ERROR RATE:
  v1 baseline error rate: 0.05%
  v2 canary error rate should not exceed: 0.1% (2x tolerance)
  
  If v2 error rate > 3× baseline: automatic rollback

LATENCY:
  v1 p99: 150ms
  v2 p99 should not exceed: 200ms (33% increase threshold)
  
  Higher latency may be acceptable if it's a known trade-off
  (e.g., new feature adds 10ms — acceptable)
  Unexpected latency regression → investigate or rollback

BUSINESS METRICS:
  For e-commerce: conversion rate, checkout completion rate
  For SaaS: feature usage, session length
  
  A deployment that looks fine technically but hurts conversion rate
  is still a problem. Canary lets you catch this before full rollout.

COMPARISON APPROACH:
  Always compare v2 metrics to v1 metrics IN THE SAME TIME WINDOW.
  Don't compare v2 at 3pm on Tuesday to v1 at 3am on Sunday.
  Simultaneously running both versions gives you a clean A/B comparison.
```

## 4. Choosing Between Blue/Green and Canary

```
USE BLUE/GREEN WHEN:
  ✓ You need the ability to instant-rollback
  ✓ Database migrations are involved (run migration, then switch traffic)
  ✓ Changes are all-or-nothing (e.g., external API contract change)
  ✓ You have budget for double infrastructure temporarily

USE CANARY WHEN:
  ✓ You want gradual rollout with real production traffic validation
  ✓ You want to limit blast radius of potential bugs
  ✓ Feature is independent and can work at partial traffic
  ✓ You have good metrics to compare old and new versions
  ✗ NOT when: database schema changes that aren't backward-compatible
  ✗ NOT when: breaking API changes (some users get v2, some get v1 → inconsistency)

USE BOTH:
  Many teams use blue/green to have a clean rollback environment
  AND canary within that to gradually shift traffic.
  Blue/green defines the "what" (two environments).
  Canary defines the "how fast" (traffic split percentage).
```

## 5. Senior Engineer Perspective

**What a junior gets wrong:**

Treating deployments as "press button, done." Deployment is an operation with risk proportional to the change size. A one-line CSS fix has different risk than a database migration. Having a deployment strategy means having a plan for: how traffic shifts, how rollback works, what metrics you watch, and when you declare success.

**The database migration problem with blue/green:**

The trickiest part of blue/green is database schema changes. If Blue uses schema v1 and Green requires schema v2, you can't just switch. You need to make the migration **backward-compatible**:

1. Deploy schema v2 that works with both v1 and v2 application code
2. Switch traffic from Blue to Green
3. After all Green traffic is confirmed good: clean up backward-compat shims

This is the expand-contract pattern from topic 5.16. Blue/green and expand-contract migrations are designed to work together.

**The business case for canary:**

On a high-traffic service, even a 0.1% bug rate is meaningful. At 1 million requests/day: 1,000 affected requests per day. A canary that catches this at 1% deployment means 10 affected requests before you notice and rollback. The math is compelling — small canary percentage, catch bugs early, limit user impact.
