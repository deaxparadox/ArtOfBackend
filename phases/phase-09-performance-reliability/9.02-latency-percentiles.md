## 1. What Is It? (Theory)

A percentile is a value below which a given percentage of observations fall. p99 latency of 200ms means: 99% of requests complete in under 200ms, and 1% take longer.

**Why this is the most important metric in backend engineering:**

Every time a user interacts with your service, they experience one of your latency distribution's values — not the average. The user who hits the p99 case doesn't care that 99 other users were fast. They experienced 200ms (or 5 seconds, or a timeout). Percentiles tell you what *individual users experience*. Averages do not.

**The real-world analogy:**

Imagine a hospital emergency room tracking "average wait time: 45 minutes." Sounds decent. But the distribution might be:
- 80% of patients seen in 15 minutes
- 15% wait 1 hour
- 5% wait 4+ hours (the critical cases that need escalation)

The average 45 minutes hides the 5% who are waiting 4 hours in pain. For them, the average is meaningless. The p95 of "1 hour" and p99 of "4 hours" tells the real story — and tells hospital management exactly where to focus.

Your backend is the same. The average is fine. The p99 is where the pain lives.

## 2. How It Actually Works (Deep Mechanics)

### The Latency Distribution

Latency is not normally distributed. It has a long tail — most requests are fast, but a small percentage are disproportionately slow. This is called a **long-tail distribution** or **heavy-tail distribution**.

```
TYPICAL WEB API LATENCY DISTRIBUTION:

Number of
requests
  │
  │  ██████
  │  ██████ █
  │  ██████ ██
  │  ██████ ██
  │  ██████ ████
  │  ██████ ███████
  │  ██████ ████████ █████
  │  ██████ ████████ ████████ ███
  └────────────────────────────────────────────►
  5ms  15ms  25ms  35ms  45ms  100ms  300ms  1000ms

  ↑           ↑                    ↑          ↑
  p50        p75                  p95        p99

Most requests cluster here.         Tail extends far to the right.
```

The tail exists because of:
- **Lock contention:** occasionally a request waits for a DB lock
- **GC pauses:** a garbage collection pause freezes all requests mid-flight
- **Connection pool exhaustion:** request waits for a DB connection to free up
- **Cold starts:** first request after a service restart is slow
- **Cache misses:** rare requests that bypass the cache
- **External API jitter:** third-party services occasionally spike

### What Each Percentile Tells You

```
p50 (median):
  "The experience of a typical user."
  Half your users see something better than this, half see something worse.
  A good p50 + a bad p99 = half your system is working well.

p75:
  "The experience of 3 in 4 users."
  Useful for understanding the upper-middle of your distribution.
  Less commonly cited, but good as an intermediate checkpoint.

p95:
  "1 in 20 users experiences this or worse."
  If you have 10,000 requests/minute: 500 users per minute see this.
  Usually the SLO target for non-critical services.
  A bad p95 is visible and will generate support tickets.

p99:
  "1 in 100 users experiences this or worse."
  If you have 10,000 requests/minute: 100 users per minute see this.
  The standard SLO target for user-facing, critical services.
  A bad p99 is felt but not loudly complained about — users just leave.

p999 (99.9th percentile):
  "1 in 1000 users."
  At 1M requests/day: 1000 users hit this every day.
  Matters for high-traffic services. Usually caused by the same root issue
  as p99 but more extreme.

Maximum:
  The single worst request you've seen.
  Useful for identifying outliers but not for SLOs — one bad request per
  hour at 10,000 req/s doesn't define your reliability.
```

### Tail Latency Amplification: The Hidden Danger of Microservices

This is one of the most underappreciated facts in distributed systems:

```
SCENARIO: a page load that makes 10 parallel backend service calls
  Each service has p99 latency of 10ms.

  What is the p99 latency of the page load?
  
  WRONG INTUITION: 10ms (they run in parallel)
  
  CORRECT MATH:
  P(at least one call takes > 10ms) = 1 - P(all calls take ≤ 10ms)
                                     = 1 - (0.99)^10
                                     = 1 - 0.904
                                     = 9.6%
  
  So ~10% of page loads will have at least one slow service call.
  That means p90 of the page load is already degraded — not just p99.

GENERALIZED:
  P(at least one slow) = 1 - (1 - slow_probability)^n_calls
  
  10 calls, each p99=10ms:  9.6% of page loads are slow (p90 level!)
  50 calls, each p99=10ms: 39.5% of page loads are slow (p60 level!)
  
THE LESSON:
  In a microservices architecture, tail latencies compound.
  A service with "good" p99 of 10ms, called 50 times per page, 
  causes 40% of page loads to experience a slow call.
  
  This is why Google's research on "hedged requests" matters so much:
  at scale, you cannot make each service fast enough individually —
  you need to handle the tail at the architecture level.
```

**Google's hedged request solution:**

```
HEDGED REQUESTS:
  Send the same request to TWO backend instances.
  Use whichever responds first. Cancel the other.
  
  Cost: ~2x backend traffic for the tailed requests
  Benefit: p99 becomes approximately p50 of two independent requests
  
  P(both slow) = P(slow)² = 0.01² = 0.0001
  Your new p99 is the old p50 of the underlying service.
  
  Used by: Google, Amazon for latency-sensitive services.
  When to use: when tail latency is causing real user impact and
               you can afford the extra traffic cost.
```

### Percentile Math: How They're Calculated

```python
def compute_percentile(latencies: list[float], p: float) -> float:
    """
    Compute the p-th percentile of a list of values.
    p is between 0 and 100.
    """
    sorted_values = sorted(latencies)
    n = len(sorted_values)
    index = (p / 100) * (n - 1)  # continuous index
    lower = int(index)
    upper = lower + 1
    if upper >= n:
        return sorted_values[-1]
    # Linear interpolation between adjacent values
    fraction = index - lower
    return sorted_values[lower] + fraction * (sorted_values[upper] - sorted_values[lower])

# Example
latencies = [5, 8, 10, 10, 12, 15, 18, 20, 45, 200]  # ms
print(f"p50: {compute_percentile(latencies, 50):.1f}ms")  # p50: 13.5ms
print(f"p90: {compute_percentile(latencies, 90):.1f}ms")  # p90: 56.5ms
print(f"p99: {compute_percentile(latencies, 99):.1f}ms")  # p99: 182.5ms
print(f"mean: {sum(latencies)/len(latencies):.1f}ms")     # mean: 34.3ms

# Notice: mean (34ms) makes it look "medium fast"
# p99 (182ms) reveals there are real tail latency problems
```

### Histograms: The Right Data Structure for Latency

In production monitoring (Prometheus), you don't store every request latency. You store histogram buckets — counts of requests that fell within latency ranges:

```python
# Prometheus histogram definition
from prometheus_client import Histogram

request_duration = Histogram(
    name="http_request_duration_seconds",
    documentation="HTTP request duration in seconds",
    labelnames=["method", "endpoint"],
    # Define buckets that cover your expected latency range
    buckets=[
        0.005,   # 5ms
        0.010,   # 10ms
        0.025,   # 25ms
        0.050,   # 50ms
        0.100,   # 100ms
        0.250,   # 250ms
        0.500,   # 500ms
        1.000,   # 1s
        2.500,   # 2.5s
        5.000,   # 5s
        float("inf"),  # catch-all
    ],
)

# Observation: a request took 47ms
request_duration.labels(method="GET", endpoint="/products").observe(0.047)
# This increments the ≤50ms bucket and all higher buckets

# PromQL to compute p99 from histogram data:
# histogram_quantile(0.99, rate(http_request_duration_seconds_bucket[5m]))
```

**Why histograms, not averages, in monitoring:**
- Histograms can compute any percentile at query time
- They are much cheaper to store than individual observations
- They aggregate correctly across multiple servers (averages don't aggregate)

## 3. Real-World Context

### How Stripe Defines SLOs

Stripe publishes (in engineering blog posts) that their API SLOs are defined in terms of percentiles. A p99 SLO of 100ms for payment confirmation means: if 1 in 100 payment confirmations is taking more than 100ms, something is wrong and must be fixed before deploying more changes.

They track p50, p95, and p99 simultaneously. A deployment that improves p50 but worsens p99 is *not* an improvement — it has made the experience of 1% of users worse, and that still fails the bar.

### Discord's 2023 Investigation

Discord engineers wrote about investigating message latency degradation. Their p50 was stable at ~3ms. Their p99 had drifted from ~5ms to ~40ms over several months as data grew. By watching percentiles, they noticed this trend months before it would have become a user complaint. This gave them time to investigate (Cassandra's compaction behavior at scale) and migrate (to ScyllaDB) without a crisis.

If they had only watched averages, the drift would have been invisible until it became catastrophic.

## 4. Code Examples

### Setting Up Latency Monitoring in FastAPI

```python
import time
from prometheus_client import Histogram, make_asgi_app
from starlette.middleware.base import BaseHTTPMiddleware
from fastapi import FastAPI

# Define latency histogram with meaningful buckets
REQUEST_LATENCY = Histogram(
    "http_request_duration_seconds",
    "HTTP request latency distribution",
    ["method", "endpoint", "status_code"],
    buckets=[0.005, 0.01, 0.025, 0.05, 0.1, 0.25, 0.5, 1.0, 2.5, 5.0, float("inf")],
)

class LatencyMiddleware(BaseHTTPMiddleware):
    async def dispatch(self, request, call_next):
        start = time.perf_counter()
        response = await call_next(request)
        elapsed = time.perf_counter() - start

        REQUEST_LATENCY.labels(
            method=request.method,
            endpoint=request.url.path,
            status_code=response.status_code,
        ).observe(elapsed)

        # Also expose the latency in response headers for debugging
        response.headers["X-Response-Time"] = f"{elapsed * 1000:.2f}ms"
        return response

app = FastAPI()
app.add_middleware(LatencyMiddleware)

# Expose metrics endpoint for Prometheus to scrape
metrics_app = make_asgi_app()
app.mount("/metrics", metrics_app)

# PromQL queries for your Grafana dashboard:
# 
# p99 latency (5-minute window):
# histogram_quantile(0.99,
#   rate(http_request_duration_seconds_bucket[5m])
# )
#
# p95 latency by endpoint:
# histogram_quantile(0.95,
#   sum by (endpoint, le) (
#     rate(http_request_duration_seconds_bucket[5m])
#   )
# )
#
# Alert: p99 over 500ms for 2 minutes
# - alert: HighP99Latency
#   expr: histogram_quantile(0.99, rate(http_request_duration_seconds_bucket[5m])) > 0.5
#   for: 2m
```

### Quick Percentile Analysis: Before You Add Monitoring

```python
import statistics
from collections import Counter

def latency_report(latencies_ms: list[float]) -> str:
    """
    Quick percentile analysis for ad-hoc debugging.
    Use this when you have a list of raw latency measurements.
    """
    if not latencies_ms:
        return "No data"

    sorted_ms = sorted(latencies_ms)
    n = len(sorted_ms)

    def pct(p):
        idx = int(n * p / 100)
        return sorted_ms[min(idx, n - 1)]

    lines = [
        f"Sample size:  {n} requests",
        f"Min:          {min(sorted_ms):.1f}ms",
        f"p50 (median): {pct(50):.1f}ms",
        f"p75:          {pct(75):.1f}ms",
        f"p95:          {pct(95):.1f}ms",
        f"p99:          {pct(99):.1f}ms",
        f"Max:          {max(sorted_ms):.1f}ms",
        f"Mean:         {statistics.mean(sorted_ms):.1f}ms  (compare to p50)",
        "",
        "Distribution:",
    ]

    # ASCII histogram (10 buckets)
    min_val, max_val = min(sorted_ms), max(sorted_ms)
    bucket_size = (max_val - min_val) / 10 if max_val > min_val else 1
    buckets = [0] * 10
    for v in sorted_ms:
        idx = min(int((v - min_val) / bucket_size), 9)
        buckets[idx] += 1

    for i, count in enumerate(buckets):
        lo = min_val + i * bucket_size
        hi = lo + bucket_size
        bar = "█" * (count * 40 // n)
        lines.append(f"  {lo:6.0f}–{hi:6.0f}ms  {bar} ({count})")

    return "\n".join(lines)

# Example usage: paste latency values from logs or monitoring
import random
sample = [random.gauss(20, 5) for _ in range(950)]  # most requests ~20ms
sample += [random.gauss(150, 30) for _ in range(50)]  # tail: 5% slow
print(latency_report(sample))
```

## 6. Trade-offs

**More granular percentiles = more storage cost:**
Storing p50/p95/p99/p999 requires more histogram buckets and more storage than just a mean. Modern systems (Prometheus, Datadog) make this cheap enough that there's no good reason to use only averages.

**Percentiles don't aggregate the same way averages do:**
You cannot average percentiles across servers. The p99 of (Server A + Server B) is NOT (p99_A + p99_B) / 2. You must aggregate the underlying histogram data and compute percentiles from the combined distribution. Prometheus's `histogram_quantile` function does this correctly automatically.

## 7. Senior Engineer Perspective

**What a junior gets wrong:**

The most common junior mistake with metrics is setting up dashboards that show the average, and then being surprised when users complain about slowness even when "the average looks fine." Average latency will always look better than it is because fast requests outnumber slow ones.

**The second trap:** watching p50 in isolation. "p50 is 10ms, we're great." p99 is 2000ms and users are timing out. p50 being good literally tells you the median experience is good — nothing about the tail.

**What a senior monitors:**

In production, a senior engineer will have three dashboards:
1. **Throughput** (requests/second, segmented by status code) — is the system handling load?
2. **Latency distribution** (p50, p95, p99 on the same chart) — how are individual requests doing?
3. **Error rate** (5xx / total) — is the system producing correct results?

These three, together, tell you everything about whether your system is healthy.

**The production alert philosophy:**

Alert on p99. Do not alert on mean. A mean alert will fire when it's too late (the p99 has been bad for an hour). A p99 alert fires when 1 in 100 users starts being hurt — early enough to intervene.
