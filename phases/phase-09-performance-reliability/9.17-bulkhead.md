## 1. The Bulkhead Pattern (9.17)

### What Is It?

The bulkhead pattern is named after ship compartments. A ship's hull is divided into watertight compartments (bulkheads). If the ship hits an iceberg and one compartment floods, the others remain sealed — the ship may take damage but doesn't sink.

In software: the bulkhead pattern **isolates failures by partitioning resources**, so that a failure in one area cannot consume all resources and bring down everything else.

**The problem it solves:**

```
WITHOUT BULKHEADS:
  Your service calls three dependencies: Database, Payment API, Recommendations API.
  All three share the same thread pool (50 threads).
  
  Payment API starts timing out (responding in 30 seconds instead of 200ms).
  
  Thread 1:  [handling request] → [calling Payment API] → [waiting 30 seconds...]
  Thread 2:  [handling request] → [calling Payment API] → [waiting 30 seconds...]
  Thread 3:  [handling request] → [calling Payment API] → [waiting 30 seconds...]
  ...
  Thread 50: [handling request] → [calling Payment API] → [waiting 30 seconds...]
  
  All 50 threads occupied. No threads for Database calls. No threads for Recommendation calls.
  Service is effectively DOWN even though only one dependency is slow.
  
  This is thread starvation caused by cascading failure.

WITH BULKHEADS:
  Separate thread pool per dependency (or per criticality tier):
  Database pool: 30 threads
  Payment API pool: 10 threads
  Recommendations pool: 10 threads
  
  Payment API starts timing out and fills its 10 threads.
  Database pool: 30 threads still available → service still works for non-payment operations
  Users can browse, add to cart, view order history — everything except payment.
  Graceful degradation instead of total failure.
```

### Implementing Bulkheads

```python
import asyncio
from concurrent.futures import ThreadPoolExecutor

# Separate semaphores as "soft" bulkheads for async contexts
class BulkheadSemaphore:
    """Limit concurrent calls to a dependency."""
    
    def __init__(self, name: str, max_concurrent: int):
        self.name = name
        self.semaphore = asyncio.Semaphore(max_concurrent)
        self._active = 0
    
    async def __aenter__(self):
        acquired = await asyncio.wait_for(
            self.semaphore.acquire(),
            timeout=5.0  # don't wait more than 5s for a slot
        )
        self._active += 1
        return self
    
    async def __aexit__(self, *args):
        self._active -= 1
        self.semaphore.release()

# Create a bulkhead per dependency
payment_bulkhead     = BulkheadSemaphore("payment", max_concurrent=10)
recommendations_bulkhead = BulkheadSemaphore("recommendations", max_concurrent=10)
database_bulkhead    = BulkheadSemaphore("database", max_concurrent=30)

@app.post("/checkout")
async def checkout(order: OrderRequest):
    # Each call is bounded by its own bulkhead
    async with database_bulkhead:
        inventory = await db.check_stock(order.product_id)
    
    async with payment_bulkhead:
        payment = await payment_service.charge(order.amount)
    
    async with database_bulkhead:
        confirmed = await db.confirm_order(order)
    
    # Recommendations are non-critical — skip if bulkhead is full
    try:
        async with asyncio.timeout(0):  # zero timeout = don't wait
            async with recommendations_bulkhead:
                suggestions = await recommendations.get(order.user_id)
    except (asyncio.TimeoutError, Exception):
        suggestions = []  # graceful degradation
    
    return {"order": confirmed, "suggestions": suggestions}
```

## 2. The Reliability Triad (9.18)

The three most important reliability mechanisms — timeout, retry, and fallback — form a "triad" that, together, handle almost every category of transient failure. Understanding how they work together is essential.

### Timeout: Bounding the Wait

A timeout says: "If this operation doesn't complete within N seconds, stop waiting and assume failure."

**Why timeouts are non-negotiable:**

Without timeouts, a single slow dependency can exhaust all threads/connections (see bulkhead problem above). Timeouts are the safety valve that prevents slow dependencies from becoming a complete outage.

```
WITHOUT TIMEOUT:
  External API hangs indefinitely.
  Your thread waits indefinitely.
  More requests arrive, more threads wait indefinitely.
  All threads exhausted.
  Service down.

WITH TIMEOUT (5 seconds):
  External API hangs.
  Your thread waits 5 seconds.
  Timeout fires: TimeoutError raised.
  Thread freed.
  Other requests continue being served.
  One user sees an error. Not all users.
```

### Retry: Handling Transient Failures

A retry says: "If this operation failed, try again — it might succeed next time."

The three components of a correct retry policy:

```
1. WHAT TO RETRY:
   ✓ Timeout → service may have recovered
   ✓ 503 Service Unavailable → server overloaded, might succeed
   ✓ Network error → transient
   ✗ 400 Bad Request → your request is wrong, retrying won't fix it
   ✗ 401/403 → auth issue, retrying won't fix it
   ✗ 422 Validation Error → bad data, retrying won't fix it

2. HOW MANY TIMES:
   Usually 2-3 retries for transient failures.
   After max retries: fail with error or fall back.

3. HOW LONG TO WAIT BETWEEN RETRIES:
   Exponential backoff + jitter (see topic 2.14).
   Don't retry instantly — give the service time to recover.
   Add randomness (jitter) so all retrying clients don't hammer at the same time.
```

### Fallback: Degrading Gracefully

A fallback says: "If this operation fails even after retries, use a lower-quality alternative."

```
FALLBACK EXAMPLES:

Product recommendations fail:
  Primary: personalized recommendations from ML service
  Fallback: popular products from cache
  Fallback-2: empty list (no recommendations section shown)
  
User profile slow:
  Primary: fresh profile from database
  Fallback: cached profile from Redis (may be slightly stale)
  Fallback-2: minimal profile from JWT claims (always available)
  
Search service down:
  Primary: Elasticsearch full-text search
  Fallback: PostgreSQL simple LIKE query (slower but functional)
  
Pricing service unavailable:
  Primary: dynamic pricing from pricing service
  Fallback: last known price from cache
  Final fallback: show "Price unavailable — call for pricing"
  (but NOT: show $0 or allow purchase at wrong price)
```

## 3. Putting It Together: The Complete Reliability Stack

These three patterns compose into a complete defense layer:

```python
from tenacity import retry, stop_after_attempt, wait_exponential_jitter
from circuitbreaker import circuit
import asyncio

# The complete reliability stack for a single external call:
@circuit(failure_threshold=5, recovery_timeout=30)  # circuit breaker
@retry(
    stop=stop_after_attempt(3),
    wait=wait_exponential_jitter(initial=1, max=10, jitter=2),
    reraise=True,
)
async def call_recommendations_service(user_id: int) -> list[dict]:
    async with asyncio.timeout(2.0):  # timeout: 2 seconds max
        async with recommendations_bulkhead:  # bulkhead: max 10 concurrent
            async with httpx.AsyncClient() as client:
                resp = await client.get(
                    f"http://recommendations-service/users/{user_id}",
                    timeout=2.0,  # also set at httpx level for defense-in-depth
                )
                resp.raise_for_status()
                return resp.json()["recommendations"]

async def get_page_data(user_id: int) -> dict:
    # Try to get recommendations with full reliability stack
    try:
        recommendations = await call_recommendations_service(user_id)
    except Exception:
        # Fallback: return empty list (degrade gracefully, don't fail the page)
        recommendations = []
    
    # Critical path: no fallback, this must work
    user = await get_user_from_db(user_id)
    
    return {
        "user": user,
        "recommendations": recommendations,  # may be empty if service is down
    }
```

**The decision matrix for each dependency:**

```
For each external dependency, decide:
  1. Timeout: how long is "too long" for this operation?
     → User-facing: 200ms–2s
     → Background job: 30s–5min
  
  2. Retry: is this operation safe to retry?
     → Idempotent operations (GET, PUT): yes
     → Non-idempotent without idempotency key (POST payment): NO
     → With idempotency key: yes
  
  3. Fallback: what do we do when all retries fail?
     → Is this critical path? → Return error to user
     → Is this optional/enhancement? → Return default/empty
     → Is there cached data? → Return stale cache
```

## 4. Senior Engineer Perspective

**The question every reliability review should ask:**

For every external call in your code: "What happens if this hangs for 5 minutes?" If the answer is "the entire service stops responding," that call needs a timeout and a bulkhead. If the answer is "a single user request fails," that might be acceptable. If the answer is "the feature shows a slightly degraded state," that's graceful degradation done right.

**The reliability triad is not optional for production code:**

A service that calls external dependencies without timeouts, without retries, and without fallbacks is a service waiting for its first outage. The question isn't "will these dependencies fail?" — they will. The question is "have you designed your system to survive those failures?"

**The order of operations:**

Design fallbacks first, then retries, then timeouts. Start by asking "what does graceful degradation look like here?" If there's a reasonable fallback, build it. Then add retries for idempotent calls. Then add timeouts to bound the worst case. This order ensures you're thinking about failure modes from the user experience perspective outward, not from the code perspective inward.
