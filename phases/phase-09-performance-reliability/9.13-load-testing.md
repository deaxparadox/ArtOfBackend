## 1. What Is It? (Theory)

Load testing is the practice of deliberately subjecting a system to controlled levels of traffic to understand how it performs under stress. It answers questions that you cannot answer from code review alone:

- What is our actual throughput ceiling?
- At what traffic level does latency degrade?
- Which component breaks first under load?
- Does the system recover after a traffic spike?

**The principle behind load testing:**

Development testing tells you if the code is correct. Load testing tells you if the system is reliable. These are different questions. Code can be perfectly correct and still collapse under real production traffic.

**The real-world analogy:**

A bridge engineer doesn't calculate whether a bridge can hold 10 cars by reading the blueprint. They test it — apply actual loads, look for deflection, stress-test the joints. The blueprint tells you what it *should* do. The load test tells you what it *actually* does.

Your backend is the same. A load test is the stress test that reveals what the system actually does, not what you hope it does.

## 2. Types of Load Tests

```
SMOKE TEST: "Does it work at all?"
  - 1-5 virtual users, a few minutes
  - Verifies the system functions before real testing begins
  - Run this first, before any heavier test

LOAD TEST: "How does it perform at expected traffic?"
  - Ramp to target load, hold for 5-15 minutes, ramp down
  - Measures: latency percentiles, error rate, throughput
  - Goal: verify you meet your SLOs at expected load

STRESS TEST: "Where does it break?"
  - Keep increasing load until something fails
  - Find the breaking point: what fails? (DB connections? CPU? Memory?)
  - Tells you your headroom before a real outage

SPIKE TEST: "Can it handle sudden bursts?"
  - Normal load → sudden 5-10x spike → back to normal
  - Tests auto-scaling response time, connection pool behavior
  - Simulates: product going viral, marketing email blast

SOAK TEST: "Does it degrade over time?"
  - Moderate load sustained for hours or days
  - Finds: memory leaks, connection leaks, gradual performance degradation
  - Run before any production launch or major feature release
  
ENDURANCE TEST: "Can it run for a long time?"
  - Similar to soak test but focuses on stability, not performance
  - Runs for 24-72 hours
```

## 3. How It Actually Works (Deep Mechanics)

### k6: The Developer-Friendly Load Testing Tool

k6 (Grafana Labs) is written in Go with a JavaScript API. Tests are scripts — reusable, versionable, and easy to integrate into CI.

```javascript
// k6 test: simulate users browsing and checking out
import http from 'k6/http';
import { check, sleep } from 'k6';
import { Rate, Trend } from 'k6/metrics';

// Custom metrics (tracked separately in output)
const errorRate = new Rate('error_rate');
const checkoutDuration = new Trend('checkout_duration');

// Test configuration
export let options = {
    stages: [
        { duration: '2m',  target: 0   },  // start at 0 (no ramp from existing test)
        { duration: '3m',  target: 50  },  // ramp up to 50 virtual users over 3 min
        { duration: '5m',  target: 50  },  // hold at 50 for 5 min (steady state)
        { duration: '3m',  target: 200 },  // spike to 200 (stress test)
        { duration: '5m',  target: 200 },  // hold at 200
        { duration: '2m',  target: 0   },  // ramp down
    ],
    thresholds: {
        // Test FAILS if these thresholds are breached
        'http_req_duration{scenario:default}': ['p95<200'],  // 95% of requests < 200ms
        'http_req_failed': ['rate<0.01'],                    // error rate < 1%
        'error_rate': ['rate<0.01'],
        'checkout_duration': ['p99<500'],                    // checkout < 500ms p99
    },
};

const BASE_URL = 'https://api.example.com';

// Setup: runs once before the test (not counted in metrics)
export function setup() {
    // Create test users, seed data if needed
    return { products: [1, 2, 3, 4, 5] };
}

// Main VU (virtual user) function: runs repeatedly per virtual user
export default function(data) {
    // Authenticate
    const loginRes = http.post(`${BASE_URL}/auth/login`, JSON.stringify({
        email: `user_${__VU}@test.com`,  // __VU = virtual user number (unique per user)
        password: 'testpassword',
    }), { headers: { 'Content-Type': 'application/json' } });
    
    check(loginRes, {
        'login successful': (r) => r.status === 200,
        'has token': (r) => r.json('access_token') !== '',
    });
    
    if (loginRes.status !== 200) {
        errorRate.add(1);
        return;  // skip rest of scenario if login failed
    }
    
    const token = loginRes.json('access_token');
    const headers = {
        'Content-Type': 'application/json',
        'Authorization': `Bearer ${token}`,
    };
    
    sleep(1);  // think time (realistic: users don't instantly click)
    
    // Browse products
    const productRes = http.get(`${BASE_URL}/api/products?page=1&limit=20`, { headers });
    check(productRes, { 'products loaded': (r) => r.status === 200 });
    
    sleep(2);  // browsing time
    
    // Checkout
    const productId = data.products[Math.floor(Math.random() * data.products.length)];
    
    const checkoutStart = Date.now();
    const checkoutRes = http.post(`${BASE_URL}/api/orders`, JSON.stringify({
        product_id: productId,
        quantity: 1,
    }), { headers });
    
    checkoutDuration.add(Date.now() - checkoutStart);
    
    check(checkoutRes, {
        'order created': (r) => r.status === 201,
        'has order id': (r) => r.json('id') !== '',
    });
    
    if (checkoutRes.status !== 201) {
        errorRate.add(1);
    }
    
    sleep(1);
}
```

```bash
# Run the test
k6 run script.js

# Run against staging with environment variable
k6 run -e BASE_URL=https://staging.example.com script.js

# Run with detailed output and upload to Grafana Cloud k6
k6 run --out cloud script.js

# Headless for CI (fails with non-zero exit if thresholds are breached)
k6 run --quiet script.js && echo "LOAD TEST PASSED" || echo "LOAD TEST FAILED"
```

### Locust: Python-Based, Realistic User Simulation

Locust (Python) is better when your test scenario requires complex logic or uses Python libraries:

```python
# locustfile.py
from locust import HttpUser, task, between, events
import json
import random

class ShopUser(HttpUser):
    # Users wait 1-3 seconds between tasks (think time)
    wait_time = between(1, 3)
    
    # Base URL (set via --host flag)
    # host = "https://api.example.com"
    
    def on_start(self):
        """Called when a user starts. Authenticate."""
        response = self.client.post("/auth/login", json={
            "email": f"user_{self.user_id}@test.com",
            "password": "testpassword",
        })
        if response.status_code == 200:
            self.token = response.json()["access_token"]
            self.client.headers.update({"Authorization": f"Bearer {self.token}"})
        else:
            self.token = None
    
    @task(5)  # weight 5: this task runs 5x more often than weight-1 tasks
    def browse_products(self):
        """Browse product catalog — most common action."""
        with self.client.get(
            "/api/products",
            params={"page": random.randint(1, 5), "limit": 20},
            name="/api/products",  # group similar requests together in stats
            catch_response=True,  # handle response ourselves
        ) as response:
            if response.status_code == 200:
                response.success()
            else:
                response.failure(f"Got status {response.status_code}")
    
    @task(2)
    def view_product_detail(self):
        product_id = random.randint(1, 100)
        self.client.get(f"/api/products/{product_id}", name="/api/products/[id]")
    
    @task(1)
    def place_order(self):
        if not self.token:
            return
        
        response = self.client.post(
            "/api/orders",
            json={"product_id": random.randint(1, 20), "quantity": 1},
            name="/api/orders",
        )
        
        if response.status_code not in (200, 201):
            # Custom failure tracking
            events.request.fire(
                request_type="POST",
                name="/api/orders",
                response_time=response.elapsed.total_seconds() * 1000,
                response_length=len(response.content),
                exception=Exception(f"Order failed: {response.status_code}"),
                context={},
            )
```

```bash
# Run headless with 100 users, spawning 10 per second, for 5 minutes
locust -f locustfile.py \
    --headless \
    --users 100 \
    --spawn-rate 10 \
    --run-time 5m \
    --host https://api.example.com

# Run with web UI for interactive control (localhost:8089)
locust -f locustfile.py --host https://api.example.com
# Navigate to http://localhost:8089
```

## 4. What to Monitor During a Load Test

```
REAL-TIME METRICS TO WATCH:

APPLICATION:
  □ Request rate (throughput)
  □ p50, p95, p99 latency
  □ Error rate (4xx, 5xx)
  □ Active connections

INFRASTRUCTURE:
  □ CPU utilization (all servers)
  □ Memory utilization
  □ Network I/O (bytes in/out)

DATABASE:
  □ Active connections (approaching max_connections?)
  □ Queries per second
  □ Average query latency
  □ Lock wait time (spike = contention)

CACHE:
  □ Hit rate (dropping = cache being evicted)
  □ Memory usage

WHEN TO STOP:
  → Error rate > 5% (system is degraded)
  → p99 > 10x normal (extreme tail latency)
  → CPU > 90% sustained (no headroom)
  → Any crash or restart loop
  → Database connections exhausted
```

## 5. Designing Realistic Tests

The biggest mistake in load testing is testing unrealistic scenarios:

```
UNREALISTIC:
  - All virtual users doing the same operation (e.g., all hitting GET /products)
  - No think time (users sending requests as fast as possible)
  - Test data that's nothing like production (empty tables, no relationships)
  - Load level unrelated to actual traffic patterns

REALISTIC:
  - Mix of operations matching actual user behavior (use analytics to determine ratios)
    e.g., for an e-commerce: 60% browse, 25% search, 10% view cart, 5% checkout
  - Think time: 1-5 seconds between actions (humans aren't robots)
  - Warm-up period before measuring (to fill caches and establish connections)
  - Test data: realistic sizes, realistic foreign key relationships
  - Load profile matching your traffic pattern (ramp up, steady, spike)
```

## 6. Senior Engineer Perspective

**What a junior gets wrong:**

Testing a single endpoint in isolation. "GET /products handles 10,000 req/s!" — but when you test the full user flow (login, browse, cart, checkout), the system falls over at 200 concurrent users because the checkout flow has a blocking database transaction that wasn't visible in the isolated test.

**The pre-launch checklist:**

```
□ Smoke test passes
□ Load test at 2x expected peak traffic passes (thresholds met)
□ Spike test: sudden 5x burst → system degrades gracefully, recovers
□ Soak test: 24h at 1x load → no memory growth, no degradation
□ Load test with downstream dependencies failing → graceful degradation
□ Load test results documented and compared to previous release
```

**Load testing in CI:**

Run the smoke test and a light load test on every PR. Run the full load test suite weekly or before major releases. If a load test fails, the PR doesn't merge. This catches performance regressions before they reach production, which is when they're cheap to fix.
