## Performance Fundamentals (9.1): Measure First, Optimize Second

```
THE PERFORMANCE ENGINEER'S RULES:

1. MEASURE BEFORE OPTIMIZING
   Without measurement, you optimize the wrong thing 80% of the time.
   The database query you rewrote for 3 days: 2ms.
   The forgotten external API call: 800ms.

2. USE PERCENTILES, NOT AVERAGES
   p50, p95, p99, p999 tell the truth. Averages hide it.
   10,000 requests at 5ms + 1 request at 5000ms = average 5.5ms (misleading).
   p99 = 5000ms (the truth: 1 in 100 users waits 5 seconds).

3. PROFILE BEFORE GUESSING
   Run a profiler. Look at the flame graph. The hottest function is your target.
   "Intuition-driven optimization" is expensive and often wrong.

4. OPTIMIZE BOTTLENECKS, NOT FAST CODE
   Making a 1ms function 2x faster saves 0.5ms.
   Making a 50ms database query 2x faster saves 25ms.
   Always optimize the bottleneck.

5. VERIFY IMPROVEMENT
   Measure before AND after. If p99 didn't improve: your optimization didn't work.
```

## Latency Percentiles (9.2)

```python
import time
import statistics

def measure_latency(operation_fn, n=1000):
    """Measure operation latency with percentile breakdown."""
    latencies = []
    for _ in range(n):
        start = time.perf_counter()
        operation_fn()
        latencies.append((time.perf_counter() - start) * 1000)  # ms

    latencies.sort()
    def p(pct): return latencies[int(len(latencies) * pct / 100)]

    return {
        "p50":  p(50),    # median: half of requests are faster than this
        "p75":  p(75),
        "p95":  p(95),    # 1 in 20 users experiences this
        "p99":  p(99),    # 1 in 100 users experiences this — your SLO target
        "p999": p(99.9),  # 1 in 1000 — tail latency
        "max":  max(latencies),  # worst case
        "mean": statistics.mean(latencies),  # often misleading
    }
```

## Little's Law (9.3) — See topic 0.5

Applied to sizing:

```python
# Pool sizing formula
def minimum_pool_size(target_rps: int, avg_latency_s: float) -> int:
    """Little's Law: Concurrency = Throughput × Latency"""
    return int(target_rps * avg_latency_s) + 1

# Example: 500 req/s API with 40ms avg DB query time
# pool_size = 500 × 0.040 = 20 connections minimum
# Use 1.5x for headroom: 30 connections
```

```sql
       rows / calls AS avg_rows
```

## SLOs, SLAs, SLIs (9.15)

```
SLI (Service Level Indicator): a metric you measure
  Examples: request latency p99, error rate, availability %

SLO (Service Level Objective): your internal reliability target
  Examples: p99 latency < 200ms, error rate < 0.1%, availability > 99.9%
  
SLA (Service Level Agreement): contractual commitment to customers
  Examples: 99.9% uptime, or we pay you back
  SLA should be less strict than your SLO (internal headroom)

ERROR BUDGET: How much unreliability you're allowed
  99.9% SLO for a 30-day month:
  Total minutes: 43,200
  Allowed downtime: 43,200 × 0.001 = 43.2 minutes/month
  
  If you burn this in week 1: no more risky deploys this month.
  If you have budget left: can take risks (experiments, faster deploys).

HOW TO SET SLOs:
  1. Measure current performance (baseline)
  2. Find what users actually care about (not everything matters equally)
  3. Set SLO slightly better than current (avoid alert fatigue)
  4. Review and tighten quarterly
```

## Horizontal vs. Vertical Scaling (9.12)

```
VERTICAL SCALING (scale up): bigger machine
  + Simple: no code changes
  - Expensive: premium pricing for large instances
  - Limited: can only get so big (server at 128 cores)
  - Single point of failure

HORIZONTAL SCALING (scale out): more machines
  + Resilient: one dying doesn't kill the service
  + Cheap: commodity hardware
  + Unlimited: add as many as needed
  - Requires stateless application (session in Redis, not in memory)
  - Adds operational complexity (load balancer, service discovery)

WHEN TO VERTICAL:
  Database servers (harder to shard)
  Cache servers (more RAM = more cached)
  Quick wins when horizontal requires major refactor

WHEN TO HORIZONTAL:
  Application servers (stateless)
  Web servers (any server serves any request)
  Queue workers (add workers for more throughput)
```

```python
# Netflix Chaos Monkey concept: randomly kill instances in production
# Goal: prove the system survives failures before users experience them

# Simple chaos injection in tests
import random
import asyncio

class ChaosMiddleware:
    """Inject artificial failures for resilience testing."""

    def __init__(self, app, failure_rate=0.01):
        self.app = app
        self.failure_rate = failure_rate

    async def __call__(self, scope, receive, send):
        if random.random() < self.failure_rate:
            # Simulate random failure
            choice = random.choice(["latency", "error", "timeout"])
            if choice == "latency":
                await asyncio.sleep(random.uniform(1, 5))
            elif choice == "error":
                from starlette.responses import JSONResponse
                await JSONResponse({"error": "chaos"}, status_code=500)(scope, receive, send)
                return
            elif choice == "timeout":
                await asyncio.sleep(30)  # force client timeout
        await self.app(scope, receive, send)
```

## Blue/Green and Canary Deployments (9.19-9.20)

```
BLUE/GREEN DEPLOYMENT:
  Blue: current production (100% traffic)
  Green: new version (0% traffic)
  
  Deploy → test green → switch load balancer → green gets 100%
  Rollback: switch LB back to blue (instant, no deployment needed)
  
  Pro: instant rollback, zero downtime
  Con: double infrastructure cost during transition

CANARY DEPLOYMENT:
  Deploy new version to 1% of servers.
  Monitor: error rate, latency, business metrics.
  If OK: increase to 5%, 10%, 25%, 50%, 100%.
  If bad: rollback (most traffic never touched the bad version).
  
  Pro: real production traffic tests, small blast radius
  Con: slower rollout, requires good monitoring

NGINX WEIGHTED ROUTING (canary):
upstream backend {
    server prod-v1.internal weight=95;  # old version
    server prod-v2.internal weight=5;   # new version (5%)
}
```
