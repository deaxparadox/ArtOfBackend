## 1. Database Connection Exhaustion: The Silent Killer (9.21)

### What Is It?

Database connection exhaustion is when all available database connections are in use and new requests must wait indefinitely for a connection to become free. The service appears operational — it's not crashed, it's not throwing errors — but requests pile up in a queue, latency climbs toward the timeout threshold, and eventually the service becomes unresponsive.

It's called "the silent killer" because the symptom (slow responses or timeouts) looks like many other problems. The root cause (connection starvation) is invisible without looking at the right metrics.

**The real-world analogy:**

A busy restaurant with only 10 tables. When all 10 tables are occupied, new customers must wait. If the party at table 3 is having a very slow dinner (a long transaction), other customers queue up. If all 10 parties are doing slow dinners simultaneously, the waiting list grows indefinitely. The restaurant isn't broken — it's just full. But from the outside, it looks like the service is down.

---

### How It Happens

```
SCENARIO: A service with a 20-connection pool and increasing load

t=0:    Traffic: 100 req/s. Average query: 10ms.
        Pool usage: 100 × 0.010 = 1 connection (Little's Law). Fine.

t=1min: External service starts responding slowly.
        One query now takes 5 seconds (external service called mid-transaction).
        Traffic: 100 req/s.
        Pool usage: 100 × 5.0 = 500 connections needed.
        Available: 20.
        480 requests queuing for connections.
        Latency for those requests: queue wait + 5s = much higher.

t=2min: Queue grows. Requests start timing out.
        Error logs: "TimeoutError: pool exhausted"
        More users retry their failed requests → traffic actually INCREASES.
        Positive feedback loop: more load → more queuing → more timeouts → more retries.

t=3min: Service effectively unusable.
        p99 latency: > 60 seconds (default timeout for many clients).
        Error rate: 40%.
        Root cause: one external API started timing out inside a transaction.
```

### Detection

```sql
-- PostgreSQL: count connections by state
SELECT
    state,
    wait_event_type,
    wait_event,
    COUNT(*) AS count
FROM pg_stat_activity
WHERE datname = 'myapp'
GROUP BY state, wait_event_type, wait_event
ORDER BY count DESC;

-- Result showing the problem:
-- state               | wait_event_type | wait_event | count
-- --------------------+-----------------+------------+------
-- active              | Client          | ClientRead |   18  ← 18 connections running
-- idle in transaction | Client          | ClientRead |    2  ← 2 stuck in open transactions
-- idle                |                 |            |    0  ← no idle connections available!

-- Find the stuck transactions
SELECT
    pid,
    usename,
    application_name,
    now() - xact_start AS transaction_age,
    now() - state_change AS state_age,
    state,
    query
FROM pg_stat_activity
WHERE state = 'idle in transaction'
   OR (state = 'active' AND now() - query_start > INTERVAL '10 seconds')
ORDER BY transaction_age DESC NULLS LAST;
```

```python
# Monitoring: track pool utilization
from prometheus_client import Gauge
import asyncpg

pool_size_gauge = Gauge("db_pool_size", "Total connections in pool")
pool_active_gauge = Gauge("db_pool_active", "Active (in-use) connections")
pool_idle_gauge = Gauge("db_pool_idle", "Idle connections")

async def update_pool_metrics(pool: asyncpg.Pool):
    pool_size_gauge.set(pool.get_size())
    pool_active_gauge.set(pool.get_size() - pool.get_idle_size())
    pool_idle_gauge.set(pool.get_idle_size())

# Alert when: active > 80% of max_size
# page when: active == max_size AND queries are queueing
```

### Prevention

The most effective preventions, in order of importance:

```
1. NEVER HOLD TRANSACTIONS OPEN DURING EXTERNAL CALLS
   
   BAD:
   BEGIN;
     result = db.query("SELECT * FROM orders WHERE id=$1", order_id)
     payment = stripe.charge(result.amount)  ← 800ms, holds connection!
     db.execute("UPDATE orders SET paid=true WHERE id=$1", order_id)
   COMMIT;
   
   GOOD:
   payment = stripe.charge(amount)  ← before transaction
   BEGIN;
     db.execute("UPDATE orders SET paid=true WHERE id=$1", order_id)
   COMMIT;

2. SET idle_in_transaction_session_timeout
   ALTER SYSTEM SET idle_in_transaction_session_timeout = '30s';
   SELECT pg_reload_conf();
   Kills transactions that open but don't commit/rollback within 30 seconds.
   
3. SET statement_timeout
   ALTER SYSTEM SET statement_timeout = '30s';
   Kills individual queries running longer than 30 seconds.
   
4. SET pool timeout (asyncpg example)
   pool = await asyncpg.create_pool(..., timeout=5.0)
   # Don't wait more than 5 seconds for a connection — fail fast
   
5. MONITOR and ALERT on pool utilization > 80%
```

## 2. Memory Leaks: Detection and Common Patterns (9.22)

### What Is a Memory Leak in a Managed Language?

In Python, Go, and other garbage-collected languages, you cannot accidentally "forget to free" memory the way you can in C. But you can **accidentally hold references** to objects longer than needed, preventing the garbage collector from freeing them.

The result is the same: memory that grows without bound until the process is OOM-killed by the OS.

**The real-world analogy:**

Your desk is a finite space. When you're done with a document, you file it away (garbage collected). A "memory leak" is when you absentmindedly keep putting documents on your desk and never file anything away. The desk fills up. You can't find anything. Eventually you need a bigger desk... then a bigger one... until you're fired (OOM killed).

---

### Common Memory Leak Patterns

**Pattern 1: Global caches without eviction**

```python
# THE LEAK:
class UserCache:
    cache = {}  # class variable = global, lives forever
    
    @classmethod
    def get(cls, user_id: int):
        if user_id not in cls.cache:
            cls.cache[user_id] = db.get_user(user_id)
        return cls.cache[user_id]
    
    # Problem: cache grows indefinitely — every unique user_id adds an entry
    # With 1M users over a week: cache has 1M entries, consuming GBs of RAM
    # Solution: use LRU cache with a size limit, or Redis

# THE FIX: use functools.lru_cache with maxsize
from functools import lru_cache

@lru_cache(maxsize=10_000)  # keeps only 10K most recently used
def get_user(user_id: int) -> dict:
    return db.get_user(user_id)

# Or: use cachetools with TTL
from cachetools import TTLCache
user_cache = TTLCache(maxsize=10_000, ttl=300)  # 5-minute TTL, 10K max entries
```

**Pattern 2: Event listeners never removed**

```python
# THE LEAK:
import signal
import atexit

shutdown_handlers = []  # list grows on each request!

@app.middleware("http")
async def register_handler(request, call_next):
    def cleanup():
        print("cleaning up for request", request.url)
    
    shutdown_handlers.append(cleanup)  # added but never removed!
    # With 1M requests/day: list has 1M entries after one day
    
    return await call_next(request)

# THE FIX: don't use mutable globals for per-request state
# Use request.state instead:
@app.middleware("http")
async def register_handler(request, call_next):
    def cleanup():
        print("cleaning up for request", request.url)
    
    request.state.cleanup = cleanup  # stored on the request, freed when request ends
    return await call_next(request)
```

**Pattern 3: Closures capturing large objects**

```python
# THE LEAK:
def create_route_handler(large_config_data):  # 100MB of configuration
    def handler(request):
        # handler closure captures large_config_data!
        process(large_config_data)
    return handler

handlers = []
for config in configs:
    handlers.append(create_route_handler(config))
# Each handler holds a reference to its config
# All configs are kept alive by the handlers list

# THE FIX: extract only what you need from large objects
def create_route_handler(config_id, relevant_config):  # only pass what's needed
    def handler(request):
        process(relevant_config)  # small extracted data, not the full config
    return handler
```

### Detecting Memory Leaks

```python
# DETECTION APPROACH 1: Monitor RSS over time
import psutil
import os

def get_memory_mb() -> float:
    process = psutil.Process(os.getpid())
    return process.memory_info().rss / 1024 / 1024  # MB

# If this number grows steadily over hours → memory leak
# Normal: grows during startup, stabilizes at a plateau
# Leak: keeps growing without plateau

# Log memory periodically
import asyncio

async def monitor_memory():
    while True:
        mem_mb = get_memory_mb()
        log.info("memory_usage", rss_mb=round(mem_mb, 1))
        # Alert if > threshold
        if mem_mb > 1500:  # alert at 1.5GB
            log.warning("high_memory_usage", rss_mb=round(mem_mb, 1))
        await asyncio.sleep(60)
```

```python
# DETECTION APPROACH 2: tracemalloc comparison
import tracemalloc

tracemalloc.start()
snapshot1 = tracemalloc.take_snapshot()

# ... run N requests ...
await asyncio.sleep(60)

snapshot2 = tracemalloc.take_snapshot()
top_stats = snapshot2.compare_to(snapshot1, "lineno")

print("Top 10 GROWING allocations:")
for stat in top_stats[:10]:
    print(stat)

# Output shows which lines of code allocated the most NEW memory
# between snapshot1 and snapshot2 → points to the leak source
```

```bash
# DETECTION APPROACH 3: watch RSS in production
# Add to your Prometheus metrics:
# process_resident_memory_bytes (built-in for most Python/Go/Node exporters)

# PromQL: memory growth rate over 24 hours
# deriv(process_resident_memory_bytes[24h]) > 10000000  (growing > 10MB/hr)
```

### The "Gradual Degradation" Signature

```
MEMORY LEAK TIMELINE:

Hour 0:   Process starts. Memory: 200MB.
Hour 2:   Memory: 250MB. (Normal: startup cache warming)
Hour 6:   Memory: 310MB. (Starting to concern)
Hour 12:  Memory: 420MB. (Definitely not normal)
Hour 18:  Memory: 580MB. (Alerting threshold)
Hour 24:  Memory: 780MB. (Near out-of-memory)
Hour 27:  OOM killed. Process restarts. Memory: 200MB. Repeat.

COMMON MISDIAGNOSIS:
  "Our service crashes every day around 3am."
  → Actually: OOM kill after 24 hours of memory growth.
  → Fix: find and fix the leak. Short-term: increase memory limit.
```

## 3. Senior Engineer Perspective

**Connection exhaustion prevention philosophy:**

The best prevention for connection exhaustion is writing code that respects the pool. Every database connection held is a scarce resource. Transactions should be short. External calls should happen outside transactions. Connection timeouts should be set everywhere. This discipline — treating connections like money, not like water — prevents the majority of pool exhaustion incidents.

**Memory leak prevention philosophy:**

The key question for any in-memory data structure: "What is the maximum size this can grow to, and what removes entries?" If you can't answer both parts, you have a potential leak. Every cache needs a size limit and an eviction policy. Every list needs something removing old entries. Every event listener needs to be eventually removed.

**The production soak test:**

A soak test (run your service at typical load for 24–72 hours while watching memory) is the most reliable way to detect memory leaks before they hit production. If memory is stable after 72 hours of realistic traffic, you probably don't have a leak.
