## 1. What Is It? (Theory)

The N+1 query problem is when fetching N records from a database requires N+1 database round trips instead of 1 or 2. It is the most common performance problem in applications that use an ORM, and it often lurks invisibly until the system is under real load.

The name: "1" query fetches the list. "N" queries fetch related data for each item in the list. Total: N+1 database round trips instead of 1 JOIN.

**The real-world analogy:**

You're managing an office and need to collect expense reports from 100 employees. The N+1 approach: you walk to each employee's desk one at a time, ask for their report, walk back, then walk to the next. 100 trips.

The correct approach: send a single email to all 100 employees asking them to submit reports, then collect them all at once. 1 trip.

The N+1 problem is your code making 100 individual trips when one would do.

## 2. How It Actually Works (Deep Mechanics)

### The Classic N+1 Pattern

```python
# DJANGO ORM — the N+1 trap
def get_user_order_summaries():
    users = User.objects.filter(is_active=True)  # Query 1: SELECT * FROM users
    
    result = []
    for user in users:
        result.append({
            "name": user.name,
            "email": user.email,
            # This line accesses user.orders — triggers a new query FOR EACH USER
            "order_count": user.orders.count(),  # Query 2, 3, 4... N+1
        })
    
    return result

# With 500 users: 1 query + 500 queries = 501 database round trips
# At 5ms per round trip: 501 × 5ms = 2,505ms = 2.5 SECONDS
# Just for what looks like simple "get users with order count"
```

Why doesn't Django warn you? Because each `user.orders.count()` call is valid Python — accessing a model attribute. The ORM lazily loads related objects. The problem is invisible in the code, but catastrophic in the database.

### The Detection Patterns

```
SYMPTOMS OF N+1:
  1. Endpoint is fast with 10 records, slow with 100
  2. Query count in logs grows linearly with list size
  3. All the slow queries are identical except for a different ID parameter
  4. Database CPU spikes during list endpoint calls

WHAT IT LOOKS LIKE IN LOGS:
  SELECT * FROM users WHERE is_active=true;   ← Query 1
  SELECT COUNT(*) FROM orders WHERE user_id=1; ← Query 2
  SELECT COUNT(*) FROM orders WHERE user_id=2; ← Query 3
  SELECT COUNT(*) FROM orders WHERE user_id=3; ← Query 4
  ... (repeated 497 more times)
  
  Notice: same query, different user_id value. That's the fingerprint.
```

### The Three Solutions

**Solution 1: JOIN (for SQL — the purest fix)**

```sql
-- Instead of 501 queries:
SELECT
    u.id,
    u.name,
    u.email,
    COUNT(o.id) AS order_count
FROM users u
LEFT JOIN orders o ON u.id = o.user_id
WHERE u.is_active = true
GROUP BY u.id, u.name, u.email;

-- 1 query. 1 round trip. Done.
-- Works regardless of how many users there are.
```

**Solution 2: Eager loading (ORM-level fix)**

```python
# DJANGO: select_related (for ForeignKey / OneToOne — uses SQL JOIN)
orders = Order.objects.select_related('user').filter(status='paid')
# Now order.user.name doesn't trigger a new query — data was fetched with JOIN

# DJANGO: prefetch_related (for ManyToMany / reverse FK — uses separate query)
users = User.objects.prefetch_related('orders').filter(is_active=True)
# 2 queries total:
# Query 1: SELECT * FROM users WHERE is_active=true
# Query 2: SELECT * FROM orders WHERE user_id IN (1, 2, 3, 4, ...500)
# Then Django matches them in Python — no more per-user queries

# DJANGO: combining both
users = User.objects.prefetch_related(
    Prefetch('orders', queryset=Order.objects.filter(status='paid').only('id', 'total_cents'))
).select_related('profile').filter(is_active=True)

# SQLALCHEMY: joinedload vs. selectinload
from sqlalchemy.orm import joinedload, selectinload

# joinedload: uses SQL JOIN (good for single related objects)
users = (session.execute(
    select(User)
    .options(joinedload(User.profile))  # JOIN
    .where(User.is_active == True)
)).scalars().all()

# selectinload: uses separate SELECT IN query (good for collections)
users = (session.execute(
    select(User)
    .options(selectinload(User.orders))  # SELECT WHERE id IN (...)
    .where(User.is_active == True)
)).scalars().all()
```

**Solution 3: DataLoader pattern (for GraphQL and service layers)**

The DataLoader pattern batches individual lookups into bulk queries. It's particularly important for GraphQL where resolvers are called per-field:

```python
from strawberry.dataloader import DataLoader
from collections import defaultdict

# The batch function: receives a LIST of IDs, returns results in same order
async def batch_load_orders(user_ids: list[int]) -> list[list[dict]]:
    """
    Receives: [1, 2, 3, 4, 5, ...]   (all user IDs needed by current request)
    Returns:  [[orders for user 1], [orders for user 2], ...]
    """
    # ONE query for all user IDs
    rows = await db.fetch(
        "SELECT id, user_id, total_cents, status FROM orders WHERE user_id = ANY($1::int[])",
        user_ids
    )
    
    # Group by user_id
    orders_by_user = defaultdict(list)
    for row in rows:
        orders_by_user[row["user_id"]].append(dict(row))
    
    # Return in same order as input (DataLoader requirement)
    return [orders_by_user.get(uid, []) for uid in user_ids]


# In your GraphQL resolver or service layer:
class UserType:
    @strawberry.field
    async def orders(self, info: strawberry.types.Info) -> list[OrderType]:
        # Instead of making a DB query here (which would be N+1),
        # we defer to the DataLoader which batches all pending requests
        return await info.context["orders_loader"].load(self.id)
        #                                               ^^^^^^^^^^^
        #                                    Each call adds to a batch.
        #                                    When event loop yields, batch is executed.

# In your request context setup:
@app.middleware("http")
async def setup_context(request, call_next):
    request.state.orders_loader = DataLoader(load_fn=batch_load_orders)
    return await call_next(request)
```

**How DataLoader batching works:**

```
REQUEST ARRIVES:
  GraphQL resolves User 1 → orders_loader.load(1)  → queued
  GraphQL resolves User 2 → orders_loader.load(2)  → queued
  GraphQL resolves User 3 → orders_loader.load(3)  → queued
  ...
  GraphQL resolves User N → orders_loader.load(N)  → queued
  
EVENT LOOP YIELDS (all resolvers have returned their coroutines):
  DataLoader fires: batch_load_orders([1, 2, 3, ..., N])
  ONE query: SELECT * FROM orders WHERE user_id = ANY([1..N])
  Results distributed back to each waiting resolver

WITHOUT DATALOADER:
  N separate queries, one per user, fired in parallel or serial
  
WITH DATALOADER:
  1 query with all user IDs in an IN clause
  Results: same, timing: dramatically better
```

## 3. Real-World Context

### How Facebook's TAO System Was Designed Around This Problem

Facebook's TAO (The Associations and Objects) system was designed specifically because N+1 patterns at Facebook's scale are catastrophic. TAO is a distributed cache that sits in front of MySQL and handles "give me the friends of user X, and for each friend give me their recent posts" — exactly the kind of nested relationship query that causes N+1 in a naive implementation.

The lesson: at scale, preventing N+1 is an architectural priority, not just a code quality concern.

### The Hidden GraphQL N+1

GraphQL makes N+1 particularly insidious because the schema allows clients to request any nested data. A naive implementation that doesn't use DataLoader will trigger an N+1 for every nested field the client requests.

Teams building GraphQL APIs without DataLoader typically discover this problem when load testing: their API handles 100 requests/second fine for shallow queries, but collapses to 5 requests/second when clients start querying nested data three levels deep. The query count goes from 1 to 1 × 50 × 20 × 10 = 10,000 per user request.

## 4. Detecting N+1 in Development

```python
# Django: install django-debug-toolbar (shows query count per request)
# Or: use a query counter in tests

from django.test.utils import CaptureQueriesContext
from django.db import connection

def test_user_list_no_n_plus_one(self):
    # Create 50 test users
    User.objects.bulk_create([User(name=f"User {i}") for i in range(50)])
    
    with CaptureQueriesContext(connection) as ctx:
        response = self.client.get("/api/users/")
    
    # Assert the number of queries is constant regardless of user count
    self.assertLessEqual(len(ctx.captured_queries), 3,
        f"Expected ≤3 queries, got {len(ctx.captured_queries)}. N+1 detected!")

# SQLAlchemy: use event listeners
from sqlalchemy import event

query_count = 0

@event.listens_for(engine, "before_cursor_execute")
def count_queries(conn, cursor, statement, parameters, context, executemany):
    global query_count
    query_count += 1

# In tests: reset query_count before the operation, check it after
```

## 5. Senior Engineer Perspective

**What a junior gets wrong:**

Adding `select_related` and `prefetch_related` everywhere "just in case." These have a cost: they always join and always fetch related data, even when you don't need it. If your endpoint only needs the user's name, prefetching their 500 orders is wasteful.

The correct approach: understand what data you actually need for each endpoint, then add exactly the eager loading required for that endpoint. Not more.

**The N+1 rule for code review:**

Any time you see a loop that accesses a model's related attribute inside the loop body, ask: "Is this triggering a database query per iteration?" If yes, it's an N+1.

```python
# Red flag in code review:
for user in users:
    send_email(user.email, context={"name": user.profile.display_name})
    #                                        ^^^^^^^^^^^^^^^^^^^^^^^^
    #                               Accessing related object inside loop
    #                               If 'profile' isn't prefetched: N+1
```

**The metric that reveals N+1 in production:**

In pg_stat_statements, look for queries where the call count is much higher than your request rate. If you serve 1000 requests/second and one query is called 50,000 times/second, you're making 50 DB calls per request. That's N+1 territory.
