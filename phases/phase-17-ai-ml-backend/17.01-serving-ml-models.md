## 1. What Is It? (Theory)

ML model serving is the engineering discipline of taking a trained model and making it available to production systems via an API. This is where machine learning and backend engineering intersect most directly.

A data scientist trains a model that achieves 94% accuracy. The backend engineer's job: make that model available, reliably, at scale, within acceptable latency. These engineering challenges are often harder than the model training itself.

**The gap between a trained model and production serving:**

```
DATA SCIENTIST'S WORLD:
  Jupyter notebook on a GPU workstation.
  "Run predict(input)" → get result.
  Batch mode: process 10,000 examples offline, wait for results.
  No concurrent requests. No SLAs. No uptime requirements.
  
PRODUCTION SERVING:
  HTTP API handling 500 requests/second.
  p99 latency: must be < 200ms.
  Must handle concurrent requests without race conditions.
  Must serve from CPU servers (GPU is expensive, not always available).
  Must handle model loading time (models are large: 1GB-100GB+).
  Must handle model version updates without downtime.
  Must monitor for accuracy drift and input distribution shift.
```

## 2. How It Actually Works (Deep Mechanics)

### The Model Serving Stack

```
CLIENT REQUEST
      ↓
LOAD BALANCER (routes requests, handles SSL)
      ↓
SERVING LAYER (your FastAPI/Flask/TorchServe service)
  - Input validation
  - Feature engineering (transform raw input to model features)
  - Model inference
  - Output post-processing
      ↓
MODEL (loaded in memory)
  - TensorFlow model / PyTorch model / scikit-learn model
  - Runs on: CPU or GPU
      ↓
RESPONSE (prediction result)
```

### The Inference Latency Budget

```
LATENCY BREAKDOWN FOR A TYPICAL ML API:
  
  Network (client → server):      5ms
  Input deserialization:          2ms
  Feature engineering:           15ms  ← often the hidden bottleneck
  Model inference:               50ms  ← what everyone focuses on
  Output processing:              3ms
  Serialization → client:         5ms
  
  Total: ~80ms
  
  INSIGHT: Feature engineering (15ms) is nearly 1/3 of the non-network time.
  Optimizing the model from 50ms to 40ms saves 10ms.
  Optimizing feature engineering from 15ms to 5ms also saves 10ms.
  Both matter. Profile before optimizing.
```

### Request Batching: The Core Inference Optimization

Batching is the single most impactful optimization in ML serving. GPUs are designed to process many inputs simultaneously — sending one input at a time is catastrophically inefficient:

```
WITHOUT BATCHING:
  100 requests arrive (each with one sample)
  GPU processes 1 sample, returns, processes 1 sample, returns... × 100
  
  GPU utilization: ~5% (waiting for data transfer between each single inference)
  Latency: 50ms per request (serial)
  Throughput: 20 requests/second

WITH DYNAMIC BATCHING:
  100 requests arrive
  Batching layer: "I'll collect requests for 5ms, then process them all at once"
  GPU processes 64 samples in one forward pass
  
  GPU utilization: ~90%
  Latency: 55ms (50ms + 5ms batch wait)
  Throughput: ~500 requests/second (25× improvement!)
  
  Trade-off: 5ms batch wait window adds to latency.
  Worth it when: throughput matters more than last-millisecond latency.

BATCH SIZE SELECTION:
  Too small (batch=1): poor GPU utilization, low throughput
  Too large (batch=256): high latency (wait for 256 requests to accumulate)
  Just right (batch=32-64): good utilization, reasonable wait time
  
  Optimal batch size is hardware and model dependent.
  Benchmark: start at 32, double until latency degrades unacceptably.
```

### A Production Model Serving Example

```python
import asyncio
import time
from fastapi import FastAPI
from pydantic import BaseModel
import torch
import numpy as np
from collections import deque
from typing import Any

app = FastAPI()

# Load model once at startup (not per-request!)
# weights_only=True: only load tensor data, not arbitrary Python objects.
# Prevents arbitrary code execution from malicious .pt files.
model = torch.load("model.pt", weights_only=True)
model.eval()

# Dynamic batching infrastructure
request_queue: asyncio.Queue = asyncio.Queue()
MAX_BATCH_SIZE = 32
MAX_BATCH_WAIT_MS = 5.0  # collect for 5ms, then process

class PredictionRequest(BaseModel):
    features: list[float]

class BatchProcessor:
    """Processes requests in batches for efficiency."""
    
    def __init__(self):
        self.queue: asyncio.Queue = asyncio.Queue()
        self.running = True
    
    async def start(self):
        """Background loop: collect and process batches."""
        while self.running:
            batch = []
            futures = []
            deadline = time.time() + MAX_BATCH_WAIT_MS / 1000
            
            # Collect requests up to batch size or deadline
            while len(batch) < MAX_BATCH_SIZE and time.time() < deadline:
                try:
                    wait_time = max(0, deadline - time.time())
                    future, features = await asyncio.wait_for(
                        self.queue.get(), timeout=wait_time
                    )
                    batch.append(features)
                    futures.append(future)
                except asyncio.TimeoutError:
                    break
            
            if batch:
                await self._process_batch(batch, futures)
    
    async def _process_batch(self, batch: list, futures: list):
        """Run model inference on a batch."""
        try:
            # Convert to tensor
            tensor = torch.tensor(batch, dtype=torch.float32)
            
            # GPU inference (if available)
            with torch.no_grad():
                predictions = model(tensor)
            
            results = predictions.numpy().tolist()
            
            # Return results to waiting requests
            for future, result in zip(futures, results):
                future.set_result(result)
        except Exception as e:
            for future in futures:
                future.set_exception(e)
    
    async def predict(self, features: list[float]) -> Any:
        """Add request to batch queue, wait for result."""
        future = asyncio.get_event_loop().create_future()
        await self.queue.put((future, features))
        return await future

batch_processor = BatchProcessor()

@app.on_event("startup")
async def startup():
    asyncio.create_task(batch_processor.start())

@app.post("/predict")
async def predict(request: PredictionRequest):
    result = await batch_processor.predict(request.features)
    return {"prediction": result}
```

## 3. Model Loading and Warm-Up

```python
import torch
import time
from contextlib import asynccontextmanager

# GLOBAL model variables
model = None
model_device = None

@asynccontextmanager
async def lifespan(app):
    """Load model at startup, clean up at shutdown."""
    global model, model_device
    
    # Determine device
    model_device = "cuda" if torch.cuda.is_available() else "cpu"
    
    print(f"Loading model on {model_device}...")
    start = time.time()
    
    # Load model
    # weights_only=True prevents arbitrary code execution from pickle.
    model = torch.load("model.pt", map_location=model_device, weights_only=True)
    model.eval()
    
    # WARM-UP: run a few dummy inferences to warm the JIT cache
    # Without warm-up: first real requests are slow (10-100x slower)
    dummy_input = torch.zeros(1, 10).to(model_device)
    for _ in range(10):
        with torch.no_grad():
            model(dummy_input)
    
    print(f"Model loaded and warmed up in {time.time() - start:.2f}s")
    
    yield  # app runs here
    
    # Cleanup
    del model
    if model_device == "cuda":
        torch.cuda.empty_cache()

app = FastAPI(lifespan=lifespan)

# NEVER load the model inside a route handler:
# @app.post("/predict")
# async def predict(...):
#     model = torch.load("model.pt")  ← WRONG! 10-second startup per request!
```

## 4. Feature Engineering at Serving Time

```python
# One of the most common sources of serving bugs:
# Training uses different feature computation than serving.
# This is called "training-serving skew."

# PROBLEMATIC PATTERN:
# Training code:
train_df["age_normalized"] = (train_df["age"] - train_mean) / train_std
model.fit(train_df[features])

# Serving code (WRONG):
serving_age_normalized = (user_age - 35)  # magic number, different from training!

# CORRECT PATTERN: save and load the preprocessing pipeline
from sklearn.pipeline import Pipeline
from sklearn.preprocessing import StandardScaler
import joblib

# Training: save the pipeline
pipeline = Pipeline([
    ("scaler", StandardScaler()),
    ("model", MyModel())
])
pipeline.fit(X_train, y_train)
joblib.dump(pipeline, "model_pipeline.pkl")  # save ENTIRE pipeline
# SECURITY NOTE: joblib uses pickle internally — only load from trusted sources
# (your own artifact store, not user-provided files).
# For untrusted data prefer JSON/msgspec for config; use ONNX for model weights.

# Serving: load and use the same pipeline
# Safe here: loading from your own controlled artifact store
pipeline = joblib.load("model_pipeline.pkl")

@app.post("/predict")
async def predict(features: dict):
    X = np.array([[features["age"], features["income"], ...]])
    prediction = pipeline.predict(X)  # uses EXACT same preprocessing
    return {"prediction": prediction[0]}
```

## 5. Real-World Context

### How Uber Serves ML Models

Uber uses Michelangelo — their internal ML platform. One insight from their engineering blog: the biggest challenge in ML serving isn't the inference itself. It's **feature computation latency**. Features like "trips by this driver in last 10 minutes" or "average rating of nearby drivers" require real-time database queries, joins, and aggregations.

Their solution: a **Feature Store** (see topic 17.12) that pre-computes and caches features, making them available at sub-millisecond latency at serving time.

### The Model Size Problem

Modern ML models are large:
- ResNet-50 image classifier: 90MB
- BERT text classifier: 400MB
- GPT-3: 175B parameters (~700GB)
- LLaMA-3 70B: ~140GB in float16

Loading a 400MB model takes ~5 seconds. On a pod restart, this causes significant cold start time. Solutions:
- Pre-load to RAM at container startup (warm containers)
- Use model quantization to reduce size (see 17.15)
- Model caching layers

## 6. Senior Engineer Perspective

**The training-serving skew is the #1 ML production bug:**

The gap between how features are computed during training and how they're computed at serving time causes silent errors — the model returns results, but they're wrong, and you won't know unless you monitor model accuracy in production. Always save and load the complete preprocessing pipeline, not just the model weights.

**GPU vs. CPU serving:**

GPUs are 10-100x faster for inference, but they're expensive ($0.50-$3/hour/GPU on cloud) and add operational complexity. For many models, optimized CPU inference (using ONNX Runtime, see 17.2) is fast enough at lower cost. Benchmark before committing to GPU.

The decision tree:
- Model inference < 50ms on CPU: serve on CPU
- Model inference 50-200ms on CPU: optimize first (quantization, ONNX), then consider GPU
- Model inference > 200ms on CPU: GPU probably necessary
- LLM inference: GPU is almost always required (the models are too large for CPU)
