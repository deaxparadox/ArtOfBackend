## 1. RAG: Retrieval Augmented Generation (17.7)

### What Is It?

RAG (Lewis et al., Facebook, 2020) is the pattern of grounding LLM responses in retrieved documents from your knowledge base. Instead of relying solely on what the model learned during training (which may be outdated or wrong), you first retrieve relevant documents and include them in the prompt context.

**Why RAG exists:**

```
THE HALLUCINATION PROBLEM:
  "What is our refund policy for enterprise customers?"
  
  LLM without RAG:
  "Enterprise customers typically have a 30-day return window with..."
  PROBLEM: This is made up. The model has no knowledge of YOUR refund policy.
           It's generating plausible-sounding text, not your actual policy.
  
  LLM with RAG:
  1. Embed the query: "refund policy enterprise customers"
  2. Retrieve: top-5 most relevant docs from your knowledge base
      [Enterprise Agreement §4.2: "Full refund within 90 days..."]
  3. Include docs in prompt: "Based on this policy document:..."
  4. LLM responds based on YOUR actual policy documents
  
  RESULT: The LLM is grounded in real information. Much less hallucination.

WHEN RAG IS THE RIGHT APPROACH:
  ✓ Q&A over company documents (policies, documentation, FAQs)
  ✓ Customer support using your knowledge base
  ✓ Code search and explanation across your codebase
  ✓ Research assistance over a corpus of papers
  ✓ Any case where the LLM needs information it wasn't trained on
```

### The RAG Pipeline

```python
from anthropic import Anthropic
from openai import OpenAI
import asyncpg
import json

anthropic_client = Anthropic()
openai_client = OpenAI()  # for embeddings

async def rag_query(
    question: str,
    db_pool: asyncpg.Pool,
    top_k: int = 5,
) -> str:
    """
    Full RAG pipeline:
    1. Embed the question
    2. Retrieve relevant documents from vector DB
    3. Build prompt with retrieved context
    4. Generate answer with LLM
    """
    
    # STEP 1: EMBED THE QUESTION
    embedding_response = openai_client.embeddings.create(
        model="text-embedding-3-small",
        input=question,
    )
    query_embedding = embedding_response.data[0].embedding
    
    # STEP 2: RETRIEVE RELEVANT DOCUMENTS (using pgvector)
    async with db_pool.acquire() as conn:
        rows = await conn.fetch("""
            SELECT
                id,
                title,
                content,
                source_url,
                1 - (embedding <=> $1::vector) AS similarity
            FROM knowledge_base
            WHERE 1 - (embedding <=> $1::vector) > 0.7  -- minimum relevance threshold
            ORDER BY embedding <=> $1::vector
            LIMIT $2
        """, query_embedding, top_k)
    
    if not rows:
        return "I don't have enough information to answer that question."
    
    # STEP 3: BUILD CONTEXT FROM RETRIEVED DOCUMENTS
    context_parts = []
    for row in rows:
        context_parts.append(
            f"**Source: {row['title']}** (similarity: {row['similarity']:.2f})\n"
            f"{row['content']}\n"
            f"URL: {row['source_url']}"
        )
    context = "\n\n---\n\n".join(context_parts)
    
    # STEP 4: GENERATE ANSWER WITH LLM
    response = anthropic_client.messages.create(
        model="claude-sonnet-4-6",
        max_tokens=1024,
        system="""You are a helpful assistant that answers questions based ONLY on 
the provided context. If the answer isn't in the context, say so clearly.
Do not make up information or draw from outside knowledge.
Always cite the source of your information.""",
        messages=[
            {
                "role": "user",
                "content": f"""Please answer the following question based on the provided context:

QUESTION: {question}

CONTEXT:
{context}

Answer based only on the context above. If the answer isn't there, say "I don't 
have that information in my knowledge base." Always cite which source you used."""
            }
        ],
    )
    
    return response.content[0].text
```

## 2. Vector Search Pipeline (17.8)

The vector search pipeline is the infrastructure that powers the retrieval step of RAG. It has three stages: **indexing** (at data ingestion time) and **querying** (at request time).

### Indexing: Building the Knowledge Base

```python
import asyncpg
from openai import OpenAI
import tiktoken
from typing import Generator

client = OpenAI()

def chunk_document(
    text: str,
    chunk_size: int = 512,
    chunk_overlap: int = 50,
    model: str = "text-embedding-3-small",
) -> Generator[str, None, None]:
    """
    Split a document into overlapping chunks.
    Overlapping ensures context isn't lost at chunk boundaries.
    
    CHUNKING STRATEGIES:
    - Fixed size (this): simple, predictable
    - Sentence-aware: split at sentence boundaries (better semantics)
    - Paragraph-aware: split at paragraph boundaries (preserves structure)
    - Recursive: try paragraph → sentence → word until size fits
    """
    encoding = tiktoken.encoding_for_model(model)
    tokens = encoding.encode(text)
    
    start = 0
    while start < len(tokens):
        end = min(start + chunk_size, len(tokens))
        chunk_tokens = tokens[start:end]
        chunk_text = encoding.decode(chunk_tokens)
        yield chunk_text
        start += chunk_size - chunk_overlap  # overlap for context continuity

async def index_document(
    pool: asyncpg.Pool,
    title: str,
    content: str,
    source_url: str,
    metadata: dict = None,
):
    """Index a document: chunk it, embed each chunk, store in pgvector."""
    
    chunks = list(chunk_document(content))
    
    if not chunks:
        return
    
    # Embed all chunks in a single API call (batch embedding is more efficient)
    embedding_response = client.embeddings.create(
        model="text-embedding-3-small",
        input=chunks,
    )
    
    async with pool.acquire() as conn:
        for i, (chunk, embedding_data) in enumerate(
            zip(chunks, embedding_response.data)
        ):
            await conn.execute("""
                INSERT INTO knowledge_base 
                (title, content, embedding, source_url, chunk_index, metadata)
                VALUES ($1, $2, $3, $4, $5, $6)
                ON CONFLICT (source_url, chunk_index) DO UPDATE SET
                    content = EXCLUDED.content,
                    embedding = EXCLUDED.embedding,
                    updated_at = NOW()
            """,
                title,
                chunk,
                embedding_data.embedding,
                source_url,
                i,
                json.dumps(metadata or {}),
            )
    
    print(f"Indexed '{title}': {len(chunks)} chunks")
```

### Advanced RAG Techniques

```python
# TECHNIQUE 1: HYBRID SEARCH (keyword + semantic)
# Combines BM25 text search with vector similarity for better recall

async def hybrid_search(question: str, pool: asyncpg.Pool, top_k: int = 5) -> list:
    query_embedding = get_embedding(question)
    
    async with pool.acquire() as conn:
        rows = await conn.fetch("""
            WITH semantic_search AS (
                SELECT id, content, title,
                       1 - (embedding <=> $1::vector) AS semantic_score
                FROM knowledge_base
                ORDER BY embedding <=> $1::vector
                LIMIT 20  -- over-retrieve, then rerank
            ),
            keyword_search AS (
                SELECT id, content, title,
                       ts_rank(search_vector, to_tsquery($2)) AS keyword_score
                FROM knowledge_base
                WHERE search_vector @@ to_tsquery($2)
                LIMIT 20
            )
            -- Reciprocal Rank Fusion to merge both result sets
            SELECT
                COALESCE(s.id, k.id) AS id,
                COALESCE(s.content, k.content) AS content,
                COALESCE(s.title, k.title) AS title,
                COALESCE(s.semantic_score, 0) * 0.7 + 
                COALESCE(k.keyword_score, 0) * 0.3 AS combined_score
            FROM semantic_search s
            FULL OUTER JOIN keyword_search k ON s.id = k.id
            ORDER BY combined_score DESC
            LIMIT $3
        """, query_embedding, question.replace(" ", " & "), top_k)
    return rows

# TECHNIQUE 2: RE-RANKING
# Retrieve more candidates, then use a cross-encoder to re-rank by relevance
from sentence_transformers import CrossEncoder

reranker = CrossEncoder("cross-encoder/ms-marco-MiniLM-L-6-v2")

async def retrieve_and_rerank(question: str, pool: asyncpg.Pool) -> list:
    # Retrieve more candidates than we need
    candidates = await retrieve_top_k(question, pool, top_k=20)
    
    # Re-rank using cross-encoder (more accurate than embedding similarity)
    pairs = [(question, doc["content"]) for doc in candidates]
    scores = reranker.predict(pairs)
    
    # Sort by cross-encoder score
    ranked = sorted(
        zip(candidates, scores),
        key=lambda x: x[1],
        reverse=True,
    )
    
    return [doc for doc, score in ranked[:5]]  # return top 5 after re-ranking
```

## 3. The RAG Database Schema

```sql
-- Complete pgvector schema for a RAG knowledge base

CREATE EXTENSION IF NOT EXISTS vector;

CREATE TABLE knowledge_base (
    id              BIGSERIAL PRIMARY KEY,
    title           TEXT NOT NULL,
    content         TEXT NOT NULL,       -- the chunk text
    embedding       vector(1536),         -- text-embedding-3-small dimensions
    source_url      TEXT NOT NULL,
    chunk_index     INTEGER NOT NULL DEFAULT 0,
    metadata        JSONB DEFAULT '{}',
    
    -- For hybrid search
    search_vector   tsvector GENERATED ALWAYS AS (
                        to_tsvector('english', coalesce(title, '') || ' ' || content)
                    ) STORED,
    
    -- Tracking
    created_at      TIMESTAMPTZ DEFAULT NOW(),
    updated_at      TIMESTAMPTZ DEFAULT NOW(),
    
    UNIQUE (source_url, chunk_index)
);

-- HNSW index for fast approximate nearest-neighbor search
CREATE INDEX idx_kb_embedding_hnsw ON knowledge_base 
    USING hnsw (embedding vector_cosine_ops)
    WITH (m = 16, ef_construction = 64);

-- GIN index for full-text search
CREATE INDEX idx_kb_search ON knowledge_base USING GIN (search_vector);

-- Index for filtering by source
CREATE INDEX idx_kb_source ON knowledge_base (source_url);
```

## 4. Senior Engineer Perspective

**The chunking strategy matters more than most realize:**

The quality of RAG answers depends heavily on how you chunk documents. A paragraph-level chunking strategy for documentation is very different from a sentence-level strategy for legal text. Experiment with different chunk sizes (256, 512, 1024 tokens) and measure answer quality.

Overlapping chunks (50-100 token overlap) prevent information loss at chunk boundaries — a critical fact can span the end of one chunk and the beginning of the next. Without overlap, that fact might not appear in either chunk's embedding space.

**Embedding model selection:**

The embedding model determines how "meaning" is captured. `text-embedding-3-small` is a good default. For domain-specific use (medical, legal, code), consider domain-specific embedding models. Always use the same model for indexing and querying — mixing models produces meaningless similarity scores.

**RAG evaluation:**

How do you know if your RAG system is working? Three metrics:
1. **Retrieval accuracy**: did the right documents get retrieved for each question?
2. **Answer faithfulness**: does the answer stay within the retrieved context?
3. **Answer correctness**: is the answer actually correct?

Tools: RAGAs, Ragas (Python library), human evaluation on a golden dataset.
