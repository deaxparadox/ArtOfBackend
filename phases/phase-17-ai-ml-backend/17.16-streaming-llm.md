## 1. Why Streaming Matters for LLMs (Theory)

LLMs generate text one token at a time. Generating a 500-word response takes 5-30 seconds. Without streaming, your user stares at a blank screen for 30 seconds and then sees the entire response at once — terrible UX.

With streaming, tokens appear as they're generated — like watching someone type in real time. The first token arrives in 200ms, and the user reads while the model continues generating.

```
WITHOUT STREAMING:
  User asks question
  [    5 seconds of nothing    ]
  Full response appears all at once

WITH STREAMING:
  User asks question
  Response appears... token... by... token...
  User starts reading while model is still generating
  
TIME TO FIRST TOKEN: 200ms (same either way)
PERCEIVED LATENCY: seconds vs milliseconds
```

## 2. How LLM Streaming Works

### The Server-Sent Events (SSE) Protocol

SSE is an HTTP protocol where the server pushes data to the client over a long-lived HTTP connection. It's simpler than WebSockets for unidirectional streams (server → client).

```
HTTP RESPONSE FOR SSE:
  Headers:
    Content-Type: text/event-stream
    Cache-Control: no-cache
    Connection: keep-alive
  
  Body (data flows continuously):
    data: {"type":"content_block_delta","delta":{"type":"text_delta","text":"Hello"}}
    
    data: {"type":"content_block_delta","delta":{"type":"text_delta","text":", world"}}
    
    data: {"type":"message_stop"}

Each event is a line starting with "data: ".
Empty line = end of one event.
```

### Consuming LLM Streaming APIs

```python
# PYTHON: receive streaming response from Anthropic

import anthropic

client = anthropic.Anthropic()

# Streaming with context manager (handles connection cleanup)
with client.messages.stream(
    model="claude-sonnet-4-6",
    max_tokens=1024,
    messages=[{"role": "user", "content": "Write a poem about the ocean"}],
) as stream:
    for text_chunk in stream.text_stream:
        print(text_chunk, end="", flush=True)
    
    # After stream completes, you can access the full message
    final_message = stream.get_final_message()
    print(f"\n\nTotal tokens: {final_message.usage.input_tokens + final_message.usage.output_tokens}")
```

## 3. Building a Streaming Backend Endpoint

Here's how to build a FastAPI endpoint that streams LLM output to clients via SSE:

```python
from fastapi import FastAPI
from fastapi.responses import StreamingResponse
import anthropic
import json
import asyncio

app = FastAPI()
client = anthropic.AsyncAnthropic()

async def llm_token_generator(user_message: str):
    """
    Async generator that yields SSE-formatted events.
    FastAPI's StreamingResponse reads from this generator.
    """
    
    # Send a "start" event so the client knows we've begun
    yield f"data: {json.dumps({'type': 'start'})}\n\n"
    
    try:
        async with client.messages.stream(
            model="claude-sonnet-4-6",
            max_tokens=1024,
            messages=[{"role": "user", "content": user_message}],
        ) as stream:
            # Yield each text chunk as an SSE event
            async for text_chunk in stream.text_stream:
                payload = json.dumps({"type": "chunk", "text": text_chunk})
                yield f"data: {payload}\n\n"
            
            # Get final message metadata
            final = await stream.get_final_message()
            done_payload = json.dumps({
                "type": "done",
                "input_tokens": final.usage.input_tokens,
                "output_tokens": final.usage.output_tokens,
            })
            yield f"data: {done_payload}\n\n"
    
    except Exception as e:
        error_payload = json.dumps({"type": "error", "message": str(e)})
        yield f"data: {error_payload}\n\n"


@app.post("/chat")
async def chat(request: dict):
    user_message = request.get("message", "")
    
    if not user_message:
        return {"error": "message is required"}
    
    return StreamingResponse(
        llm_token_generator(user_message),
        media_type="text/event-stream",
        headers={
            "Cache-Control": "no-cache",
            "X-Accel-Buffering": "no",  # disable nginx buffering
        },
    )
```

## 4. Consuming the Stream on the Frontend

```javascript
// JAVASCRIPT: consume SSE stream from your FastAPI backend

async function chat(message) {
    const response = await fetch('/chat', {
        method: 'POST',
        headers: { 'Content-Type': 'application/json' },
        body: JSON.stringify({ message }),
    });
    
    const reader = response.body.getReader();
    const decoder = new TextDecoder();
    let buffer = '';
    
    const outputElement = document.getElementById('output');
    outputElement.textContent = '';
    
    while (true) {
        const { done, value } = await reader.read();
        
        if (done) break;
        
        buffer += decoder.decode(value, { stream: true });
        const lines = buffer.split('\n');
        buffer = lines.pop();  // keep incomplete last line in buffer
        
        for (const line of lines) {
            if (!line.startsWith('data: ')) continue;
            
            const data = JSON.parse(line.slice(6));  // remove "data: " prefix
            
            if (data.type === 'chunk') {
                outputElement.textContent += data.text;
            } else if (data.type === 'done') {
                console.log(`Tokens: ${data.input_tokens} in, ${data.output_tokens} out`);
            } else if (data.type === 'error') {
                outputElement.textContent = `Error: ${data.message}`;
            }
        }
    }
}
```

## 5. Infrastructure Considerations for Streaming

```
NGINX BUFFERING:
  Nginx buffers responses by default.
  This breaks SSE — users won't see tokens until the buffer fills.
  
  FIX: Add to your nginx config:
  
  location /chat {
      proxy_pass http://backend;
      proxy_buffering off;          # disable response buffering
      proxy_cache off;
      proxy_set_header X-Accel-Buffering "no";
  }

LOAD BALANCERS:
  AWS ALB, GCP Load Balancer: support long-lived HTTP connections.
  Set idle timeout to 300+ seconds (LLM responses can take minutes).
  Default timeout is often 60 seconds — too short for long generations.

LONG-RUNNING CONNECTION MANAGEMENT:
  Streaming holds an HTTP connection open for the duration of generation.
  At 1000 concurrent users, you have 1000 open connections to the LLM.
  Your LLM API rate limits may be hit.
  Consider: request queuing, connection pooling, stream proxying.

RECONNECT HANDLING:
  SSE clients auto-reconnect by default.
  If the connection drops mid-stream, the client reconnects.
  Problem: the LLM starts over from the beginning.
  
  SOLUTION: generate a stream_id, cache partial results,
            resume from the last received event on reconnect.
```

## 6. Senior Engineer Perspective

**Streaming changes your architecture:**

Non-streaming is request-response: one request in, one response out. Streaming requires a persistent connection per request. This changes:
- How you handle timeouts (much longer)
- How you monitor errors (the response started as 200 OK, then failed mid-stream)
- How you bill users (by input/output tokens after stream ends, not on request start)
- How you handle load balancing and proxy configuration

**The error problem in streaming:**

When an error occurs mid-stream, you've already sent HTTP 200 and some tokens. You can't change the status code to 500. The client already started showing text. The solution: define an `error` event type in your SSE protocol, and send it when an error occurs. The client must handle it gracefully: show what was received, then display an error message.

**Streaming is the expected UX for LLMs:**

If you're building any user-facing LLM feature, streaming is not optional — it's the expected user experience. Users have been trained by ChatGPT and Claude.ai to expect tokens appearing immediately. A non-streaming LLM response with 5 seconds of loading screen will feel broken by comparison, even if the final answer is identical.
