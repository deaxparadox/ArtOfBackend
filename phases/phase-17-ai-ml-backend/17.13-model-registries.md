## 1. What Is a Model Registry? (Theory)

A model registry is a versioned catalog of trained ML models — tracking what was trained, with what hyperparameters, on what data, achieving what metrics. It serves as the handoff point between the data scientist who trains a model and the engineer who deploys it.

**The problem without a registry:**

```
WITHOUT A MODEL REGISTRY:
  
  Week 1: Alice trains model_v3.pkl, emails it to Bob
  Week 2: Bob deploys it. "Where did this come from?" Unknown.
  Week 3: Model performance degrades. "What changed?" Unknown.
  Week 4: Alice trains model_v4_final_FINAL.pkl (overwriting v3)
           Bob is still running model_v3.pkl from his local copy
           Team doesn't know which version is in production
           Can't roll back — the weights are gone
  
  RESULT: No reproducibility. No audit trail. No safe rollback.

WITH A MODEL REGISTRY:
  
  Alice trains a model → mlflow.log_model() → registry version 1.3.7
  Bob deploys version 1.3.7 → audit log records this
  Performance degrades → rollback to 1.3.6 in 1 command
  Six months later → reproduce ANY version's exact training run
```

## 2. MLflow: The Standard for Experiment Tracking

MLflow is the most widely used open-source ML lifecycle platform. It handles experiment tracking, model versioning, and deployment.

```python
import mlflow
import mlflow.sklearn
from sklearn.ensemble import RandomForestClassifier
from sklearn.metrics import accuracy_score, f1_score
import numpy as np

# Point MLflow at a tracking server (or use ./mlruns locally)
mlflow.set_tracking_uri("http://mlflow-server:5000")
mlflow.set_experiment("churn-prediction")

# TRAINING: log everything about this training run
with mlflow.start_run(run_name="rf-hypertuning-v2"):
    
    # Log hyperparameters
    params = {
        "n_estimators": 200,
        "max_depth": 10,
        "min_samples_split": 5,
        "random_state": 42,
    }
    mlflow.log_params(params)
    
    # Also log the data version
    mlflow.log_param("training_data_version", "v2026-06-15")
    mlflow.log_param("feature_set", "user_activity_v3")
    
    # Train
    model = RandomForestClassifier(**params)
    model.fit(X_train, y_train)
    
    # Log metrics
    y_pred = model.predict(X_test)
    mlflow.log_metric("accuracy", accuracy_score(y_test, y_pred))
    mlflow.log_metric("f1_score", f1_score(y_test, y_pred))
    mlflow.log_metric("training_samples", len(X_train))
    
    # Log the model artifact
    mlflow.sklearn.log_model(
        model,
        artifact_path="model",
        registered_model_name="churn-predictor",   # register in the model registry
    )
    
    # Log any figures (confusion matrix, ROC curve)
    import matplotlib.pyplot as plt
    fig, ax = plt.subplots()
    # ... create confusion matrix plot
    mlflow.log_figure(fig, "confusion_matrix.png")
    
    # Tag the run
    mlflow.set_tag("run_type", "hyperparameter_search")
    mlflow.set_tag("trained_by", "alice@company.com")
```

```python
# DEPLOYMENT: load a specific model version from the registry

import mlflow.sklearn

# Load the model registered as "churn-predictor" at version 5
model = mlflow.sklearn.load_model("models:/churn-predictor/5")

# Or load by stage: "Staging", "Production", "Archived"
prod_model = mlflow.sklearn.load_model("models:/churn-predictor/Production")

# Promote a version to Production (after testing in Staging)
client = mlflow.MlflowClient()
client.transition_model_version_stage(
    name="churn-predictor",
    version=5,
    stage="Production",
)

# Rollback: promote previous version back to Production
client.transition_model_version_stage(
    name="churn-predictor",
    version=4,
    stage="Production",
)
```

## 3. DVC: Data and Model Versioning with Git

DVC (Data Version Control) extends Git to track large ML artifacts — datasets and model weights — using Git-like commands but storing the actual files in S3/GCS/Azure:

```bash
# DVC WORKFLOW: track your dataset and model like code

# Initialize DVC in your git repo
dvc init
git commit -m "Initialize DVC"

# Track your training dataset (stored in S3, referenced in Git)
dvc add data/training_set.parquet
git add data/training_set.parquet.dvc .gitignore
git commit -m "Add training dataset v1"

# Push data to remote storage
dvc remote add -d myremote s3://my-ml-bucket/dvc-cache
dvc push

# Train model and track it
python train.py  # produces models/churn_model.pkl
dvc add models/churn_model.pkl
git add models/churn_model.pkl.dvc
git commit -m "Train churn model with RF, accuracy=0.94"
dvc push

# REPRODUCING: replay the entire pipeline deterministically
dvc repro   # re-runs only stages with changed inputs

# ROLLBACK: reproduce a previous model from Git history
git checkout HEAD~5  # go to 5 commits ago
dvc checkout        # pull the corresponding data/models from S3
python predict.py   # runs with the exact old model
```

```yaml
# dvc.yaml: define the pipeline as stages (like a Makefile for ML)
stages:
  prepare:
    cmd: python src/prepare.py
    deps:
      - src/prepare.py
      - data/raw.csv
    outs:
      - data/prepared.parquet

  train:
    cmd: python src/train.py --config params.yaml
    deps:
      - src/train.py
      - data/prepared.parquet
    params:
      - n_estimators
      - max_depth
    outs:
      - models/model.pkl
    metrics:
      - metrics/accuracy.json:
          cache: false    # metrics tracked in Git, not S3
```

## 4. Weights & Biases (W&B): Rich Experiment Tracking

W&B provides a more visual, collaborative experiment tracking experience:

```python
import wandb
import torch
import torch.nn as nn

# Initialize a run
run = wandb.init(
    project="image-classifier",
    config={
        "learning_rate": 0.001,
        "epochs": 50,
        "batch_size": 32,
        "architecture": "ResNet18",
    },
)

# Log metrics at every step
for epoch in range(config.epochs):
    train_loss, train_acc = train_epoch(model, train_loader)
    val_loss, val_acc = evaluate(model, val_loader)
    
    wandb.log({
        "epoch": epoch,
        "train_loss": train_loss,
        "train_accuracy": train_acc,
        "val_loss": val_loss,
        "val_accuracy": val_acc,
        "learning_rate": optimizer.param_groups[0]["lr"],
    })
    
    # Log model checkpoint as artifact
    if val_acc > best_acc:
        torch.save(model.state_dict(), "best_model.pth")
        artifact = wandb.Artifact("image-classifier", type="model")
        artifact.add_file("best_model.pth")
        run.log_artifact(artifact)

run.finish()

# W&B FEATURES THAT MAKE IT WORTH USING:
# - Side-by-side run comparison (compare 50 experiments in a table)
# - Hyperparameter parallel coordinates plot (see which params matter)
# - Automatic system metrics (GPU utilization, memory)
# - Alerts when runs fail or metrics degrade
# - Team collaboration: share experiment results via link
```

## 5. Choosing Your Tooling

```
TOOL COMPARISON:

MLflow:
  ✓ Self-hosted: free, full control
  ✓ Best for: teams that want one tool for tracking + registry + serving
  ✓ Integrates with: almost everything (sklearn, PyTorch, TF, Spark)
  ✗ UI is functional but not beautiful
  ✗ Requires running a tracking server

DVC:
  ✓ Git-native: works with your existing Git workflow
  ✓ Best for: tracking large datasets and model artifacts
  ✓ Data pipeline reproducibility is its strongest feature
  ✗ Experiment tracking is limited (just Git commits + params)
  ✗ No hosted option (it's a CLI tool, not a server)

W&B:
  ✓ Best UI and visualization in the category
  ✓ Excellent for deep learning (PyTorch, TF) where monitoring training curves matters
  ✓ Hosted SaaS (no server to manage)
  ✗ Cost at scale (pricing per user/usage)
  ✗ Data leaves your infrastructure (compliance concern for some teams)

COMMON COMBINATION:
  DVC for data versioning + MLflow for model registry = covers all bases
  W&B for teams that do a lot of deep learning experimentation
```

## 6. Senior Engineer Perspective

**A model registry is primarily about operations, not science:**

Data scientists think of experiment tracking as a research tool ("which hyperparameters work best?"). The backend engineer cares about the operational aspects: "which model is running in production right now?", "can I roll it back if it breaks?", "can I reproduce this model in 6 months for a compliance audit?"

A basic model registry needs just three things:
1. A unique version number for every deployed model
2. The ability to load any past version by its number
3. A record of which version is currently in production

MLflow provides all of this. Set it up from day one of any ML project that goes to production.

**Treat model weights like database migrations:**

Every schema change in your database has a migration file. Every trained model that goes to production should have a registry entry. You wouldn't deploy a database change without tracking it — don't deploy a model without tracking it either.
