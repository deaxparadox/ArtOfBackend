## 1. Why GPUs? (Theory)

Modern ML models — particularly neural networks — consist of enormous amounts of matrix multiplication. GPUs were originally designed for rendering 3D graphics, which also requires massive matrix operations. They excel at running thousands of small operations in parallel.

**CPU vs. GPU for matrix multiplication:**

```
MATRIX MULTIPLICATION: A × B where A and B are 4096×4096 matrices

CPU (Intel Core i9, 24 cores):
  - 24 cores × 8 operations/cycle × 3.6GHz = 691 GFLOPS (FP32)
  - Sequential: processes matrix row by row
  - Each operation can do complex branching logic
  - Memory: 64GB DDR5, shared with OS, applications

GPU (NVIDIA A100 80GB):
  - 6,912 CUDA cores × 2 operations/cycle × 1.41GHz = 19,500 GFLOPS (FP32)
  - 312,000 GFLOPS for TF32 (tensor core operations)
  - PARALLEL: processes all elements simultaneously
  - Memory: 80GB HBM2e, dedicated to GPU, 2TB/s bandwidth
  
SPEEDUP FOR MATRIX MULTIPLY: 400-2000x faster on GPU than CPU
This is why training a neural network that takes 1 hour on A100
would take 400+ hours on a CPU cluster.
```

**What CUDA is:**

CUDA (Compute Unified Device Architecture) is NVIDIA's programming platform for running code on GPUs. It lets you write C/C++ (or Python via libraries) that executes on the GPU's thousands of cores.

As a backend engineer, you rarely write raw CUDA. PyTorch and TensorFlow handle CUDA for you. But understanding CUDA concepts helps you debug GPU memory errors and performance issues.

## 2. GPU Memory: The Most Common Bottleneck

GPU memory (VRAM) is the #1 constraint in ML serving. Models must fit in VRAM to run on GPU.

```
MEMORY REQUIREMENTS FOR COMMON MODELS:

LLMs (inference):
  LLaMA 3.1 8B   (FP16): ~16 GB  → fits on 1x RTX 4090 (24GB)
  LLaMA 3.1 70B  (FP16): ~140 GB → requires 2x A100 (80GB each)
  LLaMA 3.1 405B (FP16): ~810 GB → requires 10+ A100s
  
  With 4-bit quantization:
  LLaMA 3.1 8B   (Q4):   ~4 GB   → runs on laptop GPU
  LLaMA 3.1 70B  (Q4):   ~35 GB  → fits on 1x A100 (80GB)

Vision Models:
  CLIP ViT-L/14:          ~600 MB
  Stable Diffusion XL:    ~6-8 GB
  
  
GPU HARDWARE LINEUP (2024):
  RTX 4090: 24GB VRAM, ~$1,600, consumer-grade
  A100 40GB: 40GB HBM2, $10,000+, datacenter
  A100 80GB: 80GB HBM2e, $15,000+, datacenter
  H100 80GB: 80GB HBM3, fastest currently, $30,000+
  
CLOUD PRICING (approximate per hour, on-demand):
  1x A100 (AWS p4d): ~$3.20/hr
  8x A100 (AWS p4d): ~$32/hr
  1x H100 (various): ~$5-8/hr
```

```python
# CHECKING AND MANAGING GPU MEMORY IN PYTORCH

import torch

# Check GPU availability and memory
print(f"CUDA available: {torch.cuda.is_available()}")
print(f"GPU count: {torch.cuda.device_count()}")
print(f"GPU name: {torch.cuda.get_device_name(0)}")

# Memory stats
allocated = torch.cuda.memory_allocated(0) / 1024**3    # GB actually used
reserved  = torch.cuda.memory_reserved(0) / 1024**3     # GB reserved by PyTorch's cache
total     = torch.cuda.get_device_properties(0).total_memory / 1024**3

print(f"Allocated: {allocated:.2f} GB")
print(f"Reserved:  {reserved:.2f} GB")
print(f"Total:     {total:.2f} GB")

# OOM (Out of Memory) errors: the most common GPU error
# "CUDA out of memory. Tried to allocate 1.50 GiB..."
# Solutions:
#   1. Reduce batch size
#   2. Use gradient checkpointing during training
#   3. Use mixed precision (FP16 instead of FP32 — cuts memory in half)
#   4. Use quantization (INT8/INT4)
#   5. Use a GPU with more VRAM

# Mixed precision inference (FP16): cuts memory ~50%, minimal accuracy loss
with torch.amp.autocast('cuda'):
    output = model(input_tensor)

# Move tensors to the right device
device = torch.device("cuda" if torch.cuda.is_available() else "cpu")
model = model.to(device)
input_tensor = input_tensor.to(device)
```

## 3. Multi-GPU: Fitting Models That Don't Fit on One GPU

When a model is larger than a single GPU's VRAM, you need multi-GPU strategies:

```
MULTI-GPU PARALLELISM STRATEGIES:

DATA PARALLELISM:
  - SAME model copied to each GPU
  - Different data batches on each GPU
  - Gradients averaged across GPUs after each batch
  - Use when: model fits in 1 GPU, want to train faster with more data
  
  Example: 4x A100 with same ResNet model
           Each GPU processes batch size=32
           Effective batch size = 4 × 32 = 128
           4x faster training

TENSOR PARALLELISM:
  - Model LAYERS split across GPUs
  - Layer outputs are shared between GPUs (all-reduce)
  - Use when: model doesn't fit in 1 GPU, layers too large
  
  Example: LLaMA 70B across 2x A100 80GB
           GPU 0: attention heads 0-31
           GPU 1: attention heads 32-63
           Both GPUs communicate on every layer forward pass

PIPELINE PARALLELISM:
  - Different LAYERS on different GPUs
  - Data flows through GPUs like a pipeline
  - Use when: many layers, high latency OK, want throughput
  
  Example: Transformer with 32 layers on 4 GPUs
           GPU 0: layers 1-8
           GPU 1: layers 9-16
           GPU 2: layers 17-24
           GPU 3: layers 25-32
```

```python
# DATA PARALLELISM (simplest, most common for training)
import torch
import torch.nn as nn

model = MyModel()

# Wrap with DataParallel: automatically splits batches across GPUs
if torch.cuda.device_count() > 1:
    model = nn.DataParallel(model)   # uses all available GPUs

model = model.cuda()

# DistributedDataParallel (more efficient for multi-node):
# Use torch.distributed.launch or torchrun

# TENSOR PARALLELISM (for LLM inference: vLLM handles this)
# vLLM --tensor-parallel-size 2  → splits model across 2 GPUs automatically
```

## 4. The GPU Infrastructure Stack

```
LAYERS OF GPU INFRASTRUCTURE:

1. PHYSICAL HARDWARE
   - GPU cards (NVIDIA A100, H100, RTX 4090)
   - Interconnect: NVLink (GPU-to-GPU), PCIe (GPU-to-CPU)
   - NVLink bandwidth: 600 GB/s vs PCIe: 64 GB/s
   - NVLink is critical for tensor parallelism

2. CUDA (NVIDIA's low-level GPU programming layer)
   - You almost never touch this directly
   - PyTorch, TensorFlow, JAX all sit on top of CUDA

3. CUDA LIBRARIES
   - cuDNN: GPU-accelerated deep learning operations
   - cuBLAS: GPU-accelerated linear algebra
   - NCCL: GPU communication for multi-GPU

4. ML FRAMEWORKS
   - PyTorch, TensorFlow, JAX
   - These call cuDNN/cuBLAS under the hood

5. ML SERVING
   - vLLM, TorchServe, Triton
   - These add batching, HTTP API, and production features

6. ORCHESTRATION
   - Kubernetes with GPU node pools
   - GPU operators (NVIDIA Device Plugin for k8s)

KUBERNETES + GPU:
  # GPU node label
  kubectl label nodes gpu-node01 accelerator=nvidia-a100
  
  # Pod spec requesting 1 GPU:
  resources:
    limits:
      nvidia.com/gpu: 1   # request 1 GPU
    requests:
      nvidia.com/gpu: 1
```

## 5. Monitoring GPU Utilization

```bash
# Watch GPU utilization in real time
nvidia-smi dmon -s pucvmet -d 1

# Fields explained:
# pwr:  power draw (Watts) — if low, GPU is idle
# gput: GPU utilization % — should be high (>90%) for good throughput
# memt: memory utilization % 
# sm:   streaming multiprocessors utilization
# temp: temperature (°C) — usually 70-85°C under load

# KEY INSIGHT:
# GPU utilization 20%: your serving is CPU-bound or batch size too small
# GPU utilization 95%: optimal — GPU is doing real work
# GPU memory 95%: at limit — reduce batch size or model precision

# Monitoring in Python:
import pynvml

pynvml.nvmlInit()
handle = pynvml.nvmlDeviceGetHandleByIndex(0)
info = pynvml.nvmlDeviceGetMemoryInfo(handle)
util = pynvml.nvmlDeviceGetUtilizationRates(handle)

print(f"GPU memory used: {info.used / 1024**3:.2f} GB / {info.total / 1024**3:.2f} GB")
print(f"GPU utilization: {util.gpu}%")
print(f"Memory bandwidth utilization: {util.memory}%")
```

## 6. Senior Engineer Perspective

**As a backend engineer, your GPU job is mostly economics:**

You rarely write CUDA code. Your actual GPU work is: choosing the right GPU tier for your workload, ensuring high GPU utilization (paying for idle GPU is expensive), monitoring GPU memory to avoid OOM crashes, and making the buy-vs-rent decision (on-prem GPU vs. cloud GPU).

**GPU utilization is your efficiency metric:**

If your model inference is running at 20% GPU utilization, you're paying for 100% of the GPU but using 20% of its compute. The fix is usually: increase batch size, use dynamic batching, or use a smaller GPU instance. An ML serving system running at 90% GPU utilization is 4x more cost-efficient than one at 20%.

**The GPU memory wall is real:**

Every architectural decision in LLM serving — quantization, tensor parallelism, KV cache management, dynamic batching — exists because of the GPU memory constraint. Understanding why a 70B parameter model needs 140GB of VRAM (70B × 2 bytes/float16 = 140GB) gives you the mental model for why techniques like quantization (4-bit = 17.5GB) and PagedAttention matter so much.
