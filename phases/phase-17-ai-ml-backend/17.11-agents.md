## 1. What Is an AI Agent? (Theory)

An AI agent is a system where an LLM doesn't just respond to a single prompt — it takes a series of actions autonomously to complete a goal. The LLM decides which tools to call, in what order, based on intermediate results.

**The difference between a chatbot and an agent:**

```
CHATBOT (stateless, single-turn):
  User: "What's the weather in Paris?"
  LLM: [reads training data] → "I can't check the weather directly..."
  
  The LLM is purely a text predictor. It knows no tools.

AGENT (stateful, multi-step):
  User: "Book me the cheapest flight to Paris for next Monday"
  
  LLM LOOP:
  Step 1: LLM decides to call search_flights(origin="NYC", destination="CDG", date="2026-06-22")
          → Tool returns: [{"flight": "AA123", "price": 450}, {"flight": "LH456", "price": 380}]
  
  Step 2: LLM decides to call check_availability(flight_id="LH456")
          → Tool returns: {"available": true, "seats_remaining": 3}
  
  Step 3: LLM decides to call book_flight(flight_id="LH456", passenger=user_profile)
          → Tool returns: {"confirmation": "LH-XK89Z2", "total": "$380"}
  
  Step 4: LLM returns to user: "Booked! LH456 on June 22, $380. Confirmation: LH-XK89Z2"
  
  The LLM made DECISIONS — which tools to call, what data to pass, when it's done.
```

## 2. Tool Calling: The Foundation of Agents

Tool calling (also called "function calling") is the mechanism that lets LLMs interact with external systems. The LLM outputs structured JSON specifying which tool to call and with what parameters — your code executes it.

```python
import anthropic
import json

client = anthropic.Anthropic()

# STEP 1: DEFINE TOOLS
# You tell the LLM what tools exist. The LLM chooses when to call them.
tools = [
    {
        "name": "get_weather",
        "description": "Get the current weather for a city. Returns temperature in Celsius and conditions.",
        "input_schema": {
            "type": "object",
            "properties": {
                "city": {
                    "type": "string",
                    "description": "The city name, e.g. 'Paris' or 'New York'",
                },
                "country_code": {
                    "type": "string",
                    "description": "ISO country code, e.g. 'FR' or 'US'",
                },
            },
            "required": ["city"],
        },
    },
    {
        "name": "send_email",
        "description": "Send an email to a user",
        "input_schema": {
            "type": "object",
            "properties": {
                "to": {"type": "string"},
                "subject": {"type": "string"},
                "body": {"type": "string"},
            },
            "required": ["to", "subject", "body"],
        },
    },
]

# STEP 2: IMPLEMENT THE ACTUAL TOOL FUNCTIONS
def get_weather(city: str, country_code: str = "") -> dict:
    # In production, call a real weather API
    return {"temperature": 18, "conditions": "partly cloudy", "city": city}

def send_email(to: str, subject: str, body: str) -> dict:
    # In production, call email service
    print(f"EMAIL SENT to {to}: {subject}")
    return {"status": "sent", "message_id": "msg-12345"}

# Map tool names to functions
TOOLS = {
    "get_weather": get_weather,
    "send_email": send_email,
}

# STEP 3: THE AGENT LOOP
def run_agent(user_message: str, max_iterations: int = 10) -> str:
    """
    The core agent loop:
    1. Send messages to LLM
    2. If LLM requests a tool call → execute it, add result to messages
    3. Repeat until LLM gives a final response (stop_reason = "end_turn")
    """
    messages = [{"role": "user", "content": user_message}]
    
    for iteration in range(max_iterations):
        response = client.messages.create(
            model="claude-sonnet-4-6",
            max_tokens=1024,
            tools=tools,
            messages=messages,
        )
        
        # LLM is done — return its final text response
        if response.stop_reason == "end_turn":
            return response.content[0].text
        
        # LLM wants to call tools
        if response.stop_reason == "tool_use":
            # Add the LLM's response to the conversation
            messages.append({"role": "assistant", "content": response.content})
            
            # Execute each requested tool call
            tool_results = []
            for block in response.content:
                if block.type == "tool_use":
                    tool_name = block.name
                    tool_input = block.input
                    
                    print(f"[AGENT] Calling {tool_name}({tool_input})")
                    
                    # Execute the tool
                    if tool_name in TOOLS:
                        result = TOOLS[tool_name](**tool_input)
                    else:
                        result = {"error": f"Unknown tool: {tool_name}"}
                    
                    tool_results.append({
                        "type": "tool_result",
                        "tool_use_id": block.id,
                        "content": json.dumps(result),
                    })
            
            # Add tool results to messages and continue the loop
            messages.append({"role": "user", "content": tool_results})
    
    return "Maximum iterations reached without completing task."

# USAGE:
result = run_agent("What's the weather in Paris? If it's raining, send an email to rain@example.com")
print(result)

# LLM OUTPUT (example):
# [AGENT] Calling get_weather({'city': 'Paris', 'country_code': 'FR'})
# Response: "The weather in Paris is partly cloudy at 18°C — no rain. 
#            No email needed!"
```

## 3. Agent Memory: Short-Term and Long-Term

Agents need memory to be useful across sessions. There are several types:

```
TYPES OF AGENT MEMORY:

1. IN-CONTEXT MEMORY (short-term, this conversation)
   - The messages array itself IS the memory
   - Limited by context window (128K tokens for Claude)
   - Forgotten when the session ends
   
2. SUMMARIZED MEMORY (compressed short-term)
   - When context grows large, summarize older messages
   - Keep the summary in the system prompt for the next session
   
3. EPISODIC MEMORY (long-term, retrieved)
   - Store past interactions in a vector database
   - Retrieve relevant past interactions at the start of each session
   - "Remember when you asked me about X last Tuesday?"
   
4. SEMANTIC MEMORY (long-term, factual)
   - User preferences, facts about the user, extracted knowledge
   - Stored in a structured DB and injected into system prompt
   - "You prefer Python over JavaScript. Your name is Alice."
   
5. PROCEDURAL MEMORY (behavior)
   - System prompt itself — how the agent should behave
   - Updated rarely (it's like long-term training)
```

```python
import asyncpg
from anthropic import Anthropic

client = Anthropic()

class AgentWithMemory:
    """
    Agent that persists conversation history and user facts between sessions.
    """
    
    def __init__(self, user_id: str, pool: asyncpg.Pool):
        self.user_id = user_id
        self.pool = pool
    
    async def get_user_facts(self) -> str:
        """Load stored facts about this user."""
        async with self.pool.acquire() as conn:
            row = await conn.fetchrow(
                "SELECT facts FROM user_memory WHERE user_id = $1",
                self.user_id,
            )
        if not row:
            return ""
        return f"\nUser facts:\n{row['facts']}"
    
    async def get_recent_summary(self) -> str:
        """Load summary of recent conversations."""
        async with self.pool.acquire() as conn:
            row = await conn.fetchrow(
                """SELECT summary FROM conversation_summaries 
                   WHERE user_id = $1 ORDER BY created_at DESC LIMIT 1""",
                self.user_id,
            )
        if not row:
            return ""
        return f"\nRecent conversation summary:\n{row['summary']}"
    
    async def save_facts(self, facts: str):
        async with self.pool.acquire() as conn:
            await conn.execute("""
                INSERT INTO user_memory (user_id, facts, updated_at)
                VALUES ($1, $2, NOW())
                ON CONFLICT (user_id) DO UPDATE SET facts = $2, updated_at = NOW()
            """, self.user_id, facts)
    
    async def chat(self, user_message: str, session_messages: list) -> str:
        user_facts = await self.get_user_facts()
        recent_summary = await self.get_recent_summary()
        
        system = f"""You are a helpful personal assistant.{user_facts}{recent_summary}

When you learn facts about the user (name, preferences, etc.), extract and remember them.
When responding, be concise and helpful."""
        
        session_messages.append({"role": "user", "content": user_message})
        
        response = client.messages.create(
            model="claude-sonnet-4-6",
            max_tokens=1024,
            system=system,
            messages=session_messages,
        )
        
        assistant_reply = response.content[0].text
        session_messages.append({"role": "assistant", "content": assistant_reply})
        
        return assistant_reply
```

## 4. Multi-Agent Systems

For complex tasks, multiple specialized agents can be orchestrated:

```python
# PATTERN: Orchestrator + Worker Agents
# The orchestrator decides which worker to delegate subtasks to.

class OrchestratorAgent:
    """Breaks a complex task into subtasks and delegates to specialist agents."""
    
    def __init__(self):
        self.client = Anthropic()
        self.workers = {
            "researcher": ResearchAgent(),
            "coder": CodingAgent(),
            "reviewer": ReviewAgent(),
        }
    
    async def execute(self, task: str) -> str:
        # Orchestrator decides on a plan
        plan_response = self.client.messages.create(
            model="claude-sonnet-4-6",
            max_tokens=1024,
            system="""You are an orchestrator. Break the task into subtasks.
Specify which agent handles each subtask: researcher, coder, or reviewer.
Output JSON: {"steps": [{"agent": "researcher", "task": "..."}]}""",
            messages=[{"role": "user", "content": task}],
        )
        
        plan = json.loads(plan_response.content[0].text)
        
        # Execute each step
        context = {}
        for step in plan["steps"]:
            agent_name = step["agent"]
            subtask = step["task"]
            worker = self.workers[agent_name]
            result = await worker.execute(subtask, context)
            context[agent_name] = result
        
        # Synthesize results
        return await self.synthesize(task, context)
```

## 5. Production Considerations

```
AGENT RELIABILITY CHALLENGES:

INFINITE LOOPS:
  Always set max_iterations.
  Some agents get stuck calling the same tool repeatedly.
  
TOOL CALL VALIDATION:
  LLMs sometimes call tools with invalid parameters.
  Validate inputs before executing.
  Return clear error messages so the LLM can self-correct.

COST CONTROL:
  Agents consume many more tokens than single calls.
  Each loop iteration = another LLM call.
  Set budgets: if cost > $X, stop the agent.
  
TIMEOUTS:
  A 10-step agent with 30-second LLM calls = 5 minutes total.
  Use async, set overall timeouts, provide progress updates.

OBSERVABILITY:
  Log every tool call and its result.
  Trace the full agent run for debugging.
  LangSmith, Langfuse, or custom structured logging work here.

HUMAN-IN-THE-LOOP:
  For irreversible actions (send email, make purchase, delete data),
  require human confirmation before executing.
  The agent presents a plan; human approves; agent executes.
```

## 6. Senior Engineer Perspective

**Agents are powerful but dangerous:**

An agent with access to a database, email system, and file system can cause significant damage if it misunderstands a task. The key design principle: **give agents the minimum permissions needed**, and **require confirmation for irreversible actions**.

Think of an agent like a junior employee. You wouldn't give a new hire root database access and tell them to "clean up the data." You'd give them read-only access first, review their proposed changes, then approve.

**When are agents actually worth the complexity?**

Agents make sense when:
1. The task requires multiple steps where each depends on the previous result
2. The sequence of steps isn't known in advance (dynamic decision making)
3. The task is too complex to encode in a single prompt

For anything that can be expressed as a linear pipeline with known steps, a deterministic workflow (not an agent) is better: predictable, cheaper, easier to debug.
