## 1. What Is It? (Theory)

LLM (Large Language Model) APIs provide access to foundation models — GPT-4, Claude, Gemini, Llama — through HTTP endpoints. Instead of training and hosting your own model (which requires millions of dollars and months of engineering), you call an API and pay per token.

For most applications: using LLM APIs is the right starting point. You focus on your product; the API provider handles model hosting, updates, and hardware.

**The backend engineer's job with LLM APIs:**

```
DATA SCIENTIST: "Use GPT-4 to classify customer support tickets"
BACKEND ENGINEER: Build the infrastructure around it:
  - API key management and rotation
  - Rate limiting and retry logic
  - Cost tracking (each call costs money per token)
  - Fallback to cheaper models for simple cases
  - Caching for repeated queries
  - Timeout handling (LLM calls can take 30-60 seconds)
  - Streaming responses to users (not waiting 30 seconds)
  - Prompt versioning
  - Monitoring (latency, cost, error rates)
```

## 2. How It Actually Works (Deep Mechanics)

### The Token Economy

```
TOKENS: how LLMs measure text (not characters, not words — sub-word units)
  
  Roughly: 1 token ≈ 4 characters ≈ 0.75 words (English)
  "Hello, world!" ≈ 4 tokens
  "The quick brown fox" ≈ 4 tokens
  
  PRICING (approximate, 2024):
  GPT-4o:         $5/million input tokens,  $15/million output tokens
  GPT-4o-mini:    $0.15/million input,       $0.60/million output
  Claude Sonnet:  $3/million input,          $15/million output
  Claude Haiku:   $0.25/million input,       $1.25/million output
  Gemini 1.5 Pro: $3.50/million input,       $10.50/million output
  
  COST EXAMPLE: customer support classification
  Input: "I'm having trouble logging in to my account. It says my password..."
  ~150 tokens input × $5/million = $0.00075 per call
  
  1 million calls/day × $0.00075 = $750/day → $22,500/month (use cheaper models!)
```

```python
client = Anthropic()

response = client.messages.create(
```

### Production-Grade LLM Client

```python
import anthropic
import asyncio
import time
from tenacity import retry, stop_after_attempt, wait_exponential_jitter
from functools import lru_cache
import hashlib
import json

class ProductionLLMClient:
    """
    Production-ready LLM client with:
    - Retry with exponential backoff
    - Timeout handling
    - Response caching for identical prompts
    - Cost tracking
    - Rate limiting awareness
    """
    
    def __init__(self):
        self.client = anthropic.AsyncAnthropic()
        self.total_input_tokens = 0
        self.total_output_tokens = 0
        self._cache: dict[str, str] = {}
    
    @retry(
        stop=stop_after_attempt(3),
        wait=wait_exponential_jitter(initial=1, max=30),
        reraise=True,
    )
    async def complete(
        self,
        system: str,
        messages: list[dict],
        model: str = "claude-haiku-4-5-20251001",  # default: cheapest model
        max_tokens: int = 1024,
        timeout: float = 30.0,
        use_cache: bool = True,
    ) -> str:
        # Cache key: hash of all inputs
        if use_cache:
            cache_key = hashlib.md5(
                json.dumps({"system": system, "messages": messages, "model": model}).encode()
            ).hexdigest()
            if cache_key in self._cache:
                return self._cache[cache_key]
        
        try:
            response = await asyncio.wait_for(
                self.client.messages.create(
                    model=model,
                    max_tokens=max_tokens,
                    system=system,
                    messages=messages,
                ),
                timeout=timeout,
            )
        except asyncio.TimeoutError:
            raise TimeoutError(f"LLM call timed out after {timeout}s")
        
        # Track costs
        self.total_input_tokens += response.usage.input_tokens
        self.total_output_tokens += response.usage.output_tokens
        
        result = response.content[0].text
        
        if use_cache:
            self._cache[cache_key] = result
        
        return result
    
    @property
    def estimated_cost_usd(self) -> float:
        """Rough cost estimate (Haiku pricing)."""
        return (
            self.total_input_tokens * 0.25 / 1_000_000 +
            self.total_output_tokens * 1.25 / 1_000_000
        )

# USAGE:
llm = ProductionLLMClient()

async def classify_ticket(ticket_text: str) -> str:
    return await llm.complete(
        system="Classify this support ticket into one of: billing, technical, account, other. Reply with ONLY the category.",
        messages=[{"role": "user", "content": ticket_text}],
        model="claude-haiku-4-5-20251001",  # cheap for classification
        max_tokens=10,         # classification needs very few tokens
        use_cache=True,        # cache identical tickets
    )
```

## 3. Model Selection Strategy

```
COST LADDER (cheapest → most capable):

TIER 1 (sub-cent per call, fast):
  Claude Haiku, GPT-4o-mini, Gemini Flash
  Use for: classification, extraction, simple Q&A, summarization of short text
  Latency: 0.5-2 seconds
  
TIER 2 (cents per call, moderate):
  Claude Sonnet, GPT-4o, Gemini 1.5 Pro
  Use for: complex reasoning, long-form generation, nuanced tasks
  Latency: 2-10 seconds

TIER 3 (dollars per call, most capable):
  Claude Opus, GPT-4 (legacy)
  Use for: tasks where quality matters more than cost
  Latency: 10-60 seconds

ROUTING STRATEGY:
  Start with Tier 1. If quality is insufficient, escalate to Tier 2.
  This gives 90% cost savings while maintaining quality for complex cases.
  
  async def smart_complete(prompt: str) -> str:
      # Try cheap model first
      result = await llm.complete(prompt, model="claude-haiku-4-5-20251001")
      
      # Check if result meets quality bar (e.g., is it too short?)
      if len(result) < 20 or "I cannot" in result:
          # Escalate to better model
          result = await llm.complete(prompt, model="claude-sonnet-4-6")
      
      return result
```

### Rate Limiting and Quotas

All LLM providers enforce rate limits:

```

  Tier 1: 40,000 tokens/minute, 1,000 requests/minute
```

## 5. Senior Engineer Perspective

**The LLM API is a dependency with unusual failure modes:**

Unlike a typical REST API that responds in milliseconds, LLM APIs can:
- Take 30-60 seconds (timeouts must be much longer than usual)
- Return partial results mid-generation (streaming is often better)
- Fail with rate limits at any time (retry logic is essential)
- Degrade quality without error codes (you need evaluation, not just error monitoring)
- Change behavior with model updates (Claude Sonnet 3 → 3.5 may produce different outputs)

**Cost control is a first-class concern:**

At scale, LLM costs can be significant. Always:
1. Log token counts per call type
2. Set token limits (`max_tokens`) to prevent runaway responses
3. Use the cheapest model that meets quality requirements
4. Cache responses for repeated identical prompts
5. Alert when daily costs exceed thresholds
