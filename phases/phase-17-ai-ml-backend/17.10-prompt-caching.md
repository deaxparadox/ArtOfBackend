## 1. What Is It? (Theory)

Prompt caching is a technique offered by LLM API providers (Anthropic, OpenAI) where repeated portions of prompts are cached server-side, significantly reducing both cost and latency for subsequent calls that share the same prefix.

Without optimization, every LLM call pays the full input token cost — even if 90% of the prompt is identical across calls. Prompt caching eliminates this waste.

---

## 2. How It Actually Works

### Anthropic Prompt Caching

```python
from anthropic import Anthropic

client = Anthropic()

# SCENARIO: You're building a coding assistant that always has a large
# system prompt + your entire codebase as context.
# Each user query adds ~100 tokens, but the context is 50,000 tokens.
# Without caching: pay 50,100 tokens per query.
# With caching: pay 50,000 tokens once, then ~100 tokens per query.

LARGE_CODEBASE_CONTEXT = """
# MyApp Codebase

## models/user.py
class User(Base):
    ...
    
## models/order.py
class Order(Base):
    ...

## services/payment_service.py
...
[50,000 tokens of code context]
"""

# FIRST CALL: writes the cache (costs slightly more, ~25% cache write premium)
response = client.messages.create(
    model="claude-sonnet-4-6",
    max_tokens=1024,
    system=[
        {
            "type": "text",
            "text": "You are an expert software engineer helping with this codebase.",
        },
        {
            "type": "text",
            "text": LARGE_CODEBASE_CONTEXT,
            "cache_control": {"type": "ephemeral"},  # ← mark for caching
        },
    ],
    messages=[
        {"role": "user", "content": "How does the payment service work?"}
    ],
)

# SECOND CALL: cache hit!
# The 50,000 token context is served from cache.
# Cost: cache_read_tokens × 10% of normal price + 100 new tokens × 100%
response2 = client.messages.create(
    model="claude-sonnet-4-6",
    max_tokens=1024,
    system=[
        {
            "type": "text",
            "text": "You are an expert software engineer helping with this codebase.",
        },
        {
            "type": "text",
            "text": LARGE_CODEBASE_CONTEXT,          # exact same content
            "cache_control": {"type": "ephemeral"},  # same cache marker
        },
    ],
    messages=[
        {"role": "user", "content": "What models does the ORM use?"}  # different question
    ],
)

# Check cache usage in the response
print(response2.usage.cache_read_input_tokens)   # tokens served from cache
print(response2.usage.cache_creation_input_tokens)  # tokens written to cache
print(response2.usage.input_tokens)              # new tokens this call

# COST SAVINGS:
# Without caching: 50,100 tokens × $3/million = $0.150 per call
# With caching:    50,000 cached × $0.30/million + 100 new × $3/million = $0.0153 per call
# SAVINGS: 90% cost reduction on the cached portion!
```

### Cache Duration and Invalidation

```
ANTHROPIC CACHE LIFETIME:
  "ephemeral" cache: valid for 5 minutes
  Cache is invalidated after 5 minutes of no access.
  Minimum cacheable content: 1024 tokens
  
CACHE MISS SCENARIOS:
  - Content changed (even one character difference)
  - Cache expired (5 minutes of no access)
  - Different model used
  - Content below minimum threshold
  
CACHE WRITE COST:
  Cache creation: 25% extra on those tokens (one-time)
  Cache reads: 10% of normal input token price
  
NET RESULT:
  Break-even if a cached prompt is accessed 1.4× (pays off after 1-2 reuses)
  Very profitable if the same context is used 100× per day
```

## 3. Token Optimization Strategies

Beyond caching, there are several ways to reduce token usage and thus API costs:

### Strategy 1: Choose the Right Model

```python
ROUTING STRATEGY:
  Classification tasks (is this spam?) → Haiku (cheap, fast)
  Summarization of short text → Haiku
  Complex reasoning, code generation → Sonnet
  Most creative or difficult tasks → Sonnet (Opus often overkill)
  
  Cost difference: Haiku is ~12x cheaper than Sonnet per token.
  Routing 80% of calls to Haiku = major cost reduction.
```

### Strategy 2: Minimize Unnecessary Tokens in Prompts

```python
# VERBOSE PROMPT (wasteful):
system = """You are a highly knowledgeable AI assistant with expertise in 
customer support. Your role is to help customers by providing accurate, 
helpful, and friendly responses. You should always be polite and professional.
When answering questions, please make sure to:
1. Be clear and concise
2. Provide accurate information
3. Be helpful and friendly
4. Only answer based on what you know
Please answer the following customer question:"""

# CONCISE PROMPT (saves 60 tokens per call):
system = """You are a customer support agent. Answer clearly and accurately.
Use only the provided context."""

# At 1M calls/day: 60 fewer tokens × $3/million = $180/day savings
```

### Strategy 3: Structured Output Instead of Free Text

```python
# Instead of asking the LLM to format a response in free text
# (which requires parsing and may include explanation text):

# UNSTRUCTURED (uses more tokens, harder to parse):
prompt = "Analyze this customer review and tell me the sentiment, key topics, and rating 1-5"
# LLM response: "The sentiment is positive. The customer mentions the price, delivery time..."

# STRUCTURED (fewer output tokens, machine-parseable):
prompt = """Analyze this review. Respond ONLY with valid JSON matching this exact schema:
{"sentiment": "positive|negative|neutral", "rating": 1-5, "topics": ["list", "of", "topics"]}

Review: {review_text}"""
# LLM response: {"sentiment": "positive", "rating": 4, "topics": ["price", "delivery"]}
```

### Strategy 4: Response Length Control

```python
# max_tokens: hard limit on output length
# For classification: max_tokens=10 (literally just "positive" or "negative")
# For summarization: max_tokens=150 (one paragraph)
# For code generation: max_tokens=2048 (allow full function)

response = client.messages.create(
    model="claude-haiku-4-5-20251001",
    max_tokens=5,   # "positive" or "negative" needs only 1-2 tokens
    system="Classify as positive or negative. Reply ONLY with the single word.",
    messages=[{"role": "user", "content": review_text}],
)
```

### Strategy 5: Batch Similar Requests

```python
# Instead of 100 individual classification calls:
# BAD (100 API calls):
for review in reviews:
    classify(review)

# GOOD (1 API call per batch):
BATCH_SIZE = 20
for batch in chunks(reviews, BATCH_SIZE):
    prompt = f"""Classify each of these {len(batch)} reviews as positive/negative.
Reply with a JSON array of results in the same order.

Reviews:
{json.dumps([r.text for r in batch], indent=2)}"""
    
    response = client.messages.create(...)
    classifications = json.loads(response.content[0].text)

```

## 4. Senior Engineer Perspective

**Prompt caching pays for itself almost immediately:**

If you have any system prompt longer than ~1,000 tokens that's reused across many calls (documentation context, persona definition, tool descriptions), prompt caching is almost always worth enabling. The math is straightforward: any prompt accessed more than twice in 5 minutes saves money.

**The token budget mindset:**

Treat input tokens like database reads — each has a cost. Before building an LLM feature, estimate the token budget:
- Average system prompt: N tokens
- Average context: N tokens
- Average user input: N tokens
- Average response: N tokens
- Total per call: N tokens × price × expected daily calls = daily cost

Build this estimate before committing to an architecture. An expensive prompt design might be fine at 1,000 calls/day but catastrophic at 1,000,000 calls/day.
