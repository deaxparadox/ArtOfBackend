## 1. What Is It? (Theory)

When you need to run LLMs on your own infrastructure — for cost, privacy, control, or latency — you need specialized inference infrastructure. LLMs have unique serving challenges that standard ML serving frameworks don't handle:

```
LLM UNIQUE CHALLENGES:

KV CACHE:
  During generation, each token attends to all previous tokens.
  The intermediate attention computations (KV cache) must be stored.
  For a 1000-token sequence: KV cache can be gigabytes.
  
  Naive serving: allocate max KV cache upfront for every request.
  Problem: 90% of allocated memory goes unused (most requests shorter than max).
  
  vLLM's solution: PagedAttention — allocate KV cache in pages (like virtual memory).
  Result: 3-4x more requests served per GPU simultaneously.

CONTINUOUS BATCHING:
  Traditional batching: wait for a full batch, process all at once.
  Problem: long requests block short requests (head-of-line blocking).
  
  vLLM's continuous batching: requests join/leave the batch dynamically.
  Short requests complete and new ones join while long requests are still generating.
  Result: much better GPU utilization, lower latency for short requests.

MEMORY REQUIREMENTS:
  LLaMA 3.1 70B: ~140GB in float16
  LLaMA 3.1 8B:  ~16GB in float16
  Typical GPU: 24GB (RTX 4090) to 80GB (A100/H100)
  
  Most consumer GPUs can't fit the popular models.
  Solutions: quantization (reduce precision), multi-GPU, model offloading.
```

## 2. vLLM: High-Throughput LLM Serving

vLLM (UC Berkeley, 2023) is the dominant open-source framework for high-throughput LLM serving. Its key innovation — PagedAttention — dramatically improves GPU utilization.

```bash
# Install
pip install vllm

# Start OpenAI-compatible server
python -m vllm.entrypoints.openai.api_server \
    --model meta-llama/Llama-3.1-8B-Instruct \
    --port 8000 \
    --tensor-parallel-size 1 \   # number of GPUs for tensor parallelism
    --gpu-memory-utilization 0.90 \
    --max-model-len 8192          # maximum context length

# Use with OpenAI-compatible client (drop-in replacement):
from openai import OpenAI

client = OpenAI(
    base_url="http://localhost:8000/v1",
    api_key="ignored",  # vLLM doesn't need a real key by default
)

response = client.chat.completions.create(
    model="meta-llama/Llama-3.1-8B-Instruct",
    messages=[
        {"role": "user", "content": "Explain quantum computing in simple terms."}
    ],
    max_tokens=500,
    temperature=0.7,
)
print(response.choices[0].message.content)
```

```
vLLM PERFORMANCE (A100 GPU, LLaMA 3.1 8B):
  Throughput: ~3,000 tokens/second
  Concurrent requests: 100+ with good latency
  
vLLM PERFORMANCE vs. HuggingFace transformers:
  transformers (naive): 1 request at a time, 200 tokens/second
  vLLM: 10-30x higher throughput via continuous batching + PagedAttention
```

## 3. Ollama: Development and Local Use

Ollama (2023) is designed for running LLMs locally with minimal setup. One command to pull and run a model:

```bash
# Install: curl -fsSL https://ollama.com/install.sh | sh

# Pull and run a model (downloads weights automatically)
ollama run llama3.2

# Interactive chat:
>>> Hello! Can you help me debug this Python code?

# API (OpenAI-compatible):
ollama serve &  # starts API server on port 11434

curl http://localhost:11434/api/chat \
    -d '{"model": "llama3.2", "messages": [{"role": "user", "content": "Hello"}]}'

# Python:
from openai import OpenAI
client = OpenAI(base_url="http://localhost:11434/v1", api_key="ignored")
response = client.chat.completions.create(
    model="llama3.2",
    messages=[{"role": "user", "content": "Hello"}],
)
```

```
WHEN TO USE OLLAMA:
  ✓ Local development (no cloud API keys needed)
  ✓ Testing prompts and pipelines offline
  ✓ Privacy-sensitive development (data never leaves your machine)
  ✓ Edge deployment on reasonably-spec'd machines

OLLAMA LIMITATIONS:
  ✗ Not optimized for high concurrent throughput (not for production serving)
  ✗ Limited to models that fit in available RAM/VRAM
  ✗ No multi-GPU support in basic configuration
```

## 4. llama.cpp: CPU Inference for LLMs

llama.cpp (Georgi Gerganov, 2023) makes LLM inference possible on CPUs by using quantization to reduce models to 4-bit or 8-bit precision:

```bash
# Compile llama.cpp
git clone https://github.com/ggerganov/llama.cpp
cd llama.cpp && make

# Download a GGUF model (quantized format)
wget https://huggingface.co/bartowski/Llama-3.2-1B-GGUF/resolve/main/Llama-3.2-1B-Q4_K_M.gguf

# Run inference on CPU
./llama-cli -m Llama-3.2-1B-Q4_K_M.gguf -p "The capital of France is" -n 100

# Start HTTP server
./llama-server -m model.gguf --port 8080 --ctx-size 4096

# CPU performance (example: LLaMA 3.2 1B Q4 on M3 MacBook Pro):
# ~60 tokens/second — fast enough for interactive use
```

```
QUANTIZATION LEVELS:
  Q2_K: 2-bit → smallest file, noticeable quality loss
  Q4_K_M: 4-bit → good balance of size and quality (recommended)
  Q8_0: 8-bit → near-full quality, 2x size vs Q4
  F16:  16-bit → full quality, largest size (needs GPU typically)
  
WHEN llama.cpp IS APPROPRIATE:
  ✓ No GPU available (CPU-only machines)
  ✓ Small models on developer laptops (1B-7B parameters)
  ✓ Embedded/edge devices with limited hardware
  ✓ Testing quantization effects before production deployment
  
FOR PRODUCTION: use vLLM on GPU. llama.cpp CPU is 10-100x slower.
```

## 5. The Self-Hosted vs. Managed API Decision

```
FACTORS FAVORING MANAGED API (Anthropic, OpenAI, Gemini):
  ✓ No GPU infrastructure to manage
  ✓ Latest models available immediately
  ✓ Scales automatically
  ✓ No upfront cost (pay per token)
  ✓ Lower engineering overhead
  ✓ Best for: prototyping, variable workloads

FACTORS FAVORING SELF-HOSTED (vLLM + open models):
  ✓ Lower cost at high volume (>1M tokens/day)
  ✓ Data privacy (sensitive data never leaves your infrastructure)
  ✓ Compliance requirements (HIPAA, GDPR data residency)
  ✓ Customization (fine-tune on your data)
  ✓ No API rate limits
  ✓ Offline capability
  
COST CROSSOVER POINT (rough estimate):
  API: $0.25-$5 per million tokens
  Self-hosted (A100 80GB on AWS): ~$3/hour ÷ 3000 tokens/second
               = $0.28 per million tokens (at high utilization)
  
  At 1M+ tokens/day: self-hosting often cheaper.
  Below 1M tokens/day: managed API usually cheaper (no idle GPU cost).
```
