## 1. What Are They? (Theory)

BentoML and Ray Serve are Python-native ML serving frameworks. They sit between "build your own Flask API" and specialized production servers like Triton, filling the gap for teams that want production-quality serving without the operational complexity of Triton.

**The serving framework landscape:**

```
LOW OVERHEAD → HIGH PERFORMANCE:

Flask/FastAPI (DIY):
  You write everything: batching, GPU allocation, health checks, scaling.
  Fine for simple models. Brittle at scale.

BentoML:
  Python-native, packages model + server as a "bento" (container).
  Handles batching, adaptive scaling, model packaging.
  Designed for: single-model serving at production scale.
  
Ray Serve:
  Built on Ray (distributed compute). Handles multi-model pipelines.
  "Compose multiple models into one service."
  Designed for: complex inference pipelines, multi-model systems.

TorchServe / TF Serving / Triton (from 17.2-17.4):
  High-performance, framework-specific (or Triton's multi-framework).
  Designed for: maximum GPU utilization in large-scale deployments.
  Requires more operational knowledge.
```

## 2. BentoML: Package and Serve Any ML Model

BentoML's core idea: a "Bento" is a self-contained unit of a model + its serving code + its dependencies + its configuration. You build a Bento; BentoML handles the rest.

```python
# STEP 1: SAVE A MODEL TO BENTOML'S LOCAL STORE
import bentoml
from sklearn.ensemble import RandomForestClassifier

# Train your model (or load from MLflow/file)
model = RandomForestClassifier(n_estimators=100)
model.fit(X_train, y_train)

# Save to BentoML model store
# BentoML stores model with metadata and gives it a version tag
saved_model = bentoml.sklearn.save_model(
    "churn_predictor",           # model name in BentoML store
    model,
    signatures={
        "predict": {"batchable": True, "batch_dim": 0},  # allow batching
    },
    metadata={"accuracy": 0.94, "f1": 0.91},
)
print(f"Saved: {saved_model.tag}")  # e.g., "churn_predictor:abc123"
```

```python
# STEP 2: DEFINE A BENTOML SERVICE
# This is your API definition. BentoML generates the HTTP server.

import bentoml
import numpy as np
from bentoml.io import NumpyNdarray, JSON

# Load the latest version of your model
runner = bentoml.sklearn.get("churn_predictor:latest").to_runner()

# Define the service
svc = bentoml.Service("churn_prediction_service", runners=[runner])

# Define an API endpoint
@svc.api(
    input=JSON(),                         # accepts JSON input
    output=JSON(),                        # returns JSON output
    route="/predict",
)
async def predict(features: dict) -> dict:
    """Predict churn probability for a user."""
    
    # Convert dict to numpy array (your model's expected input)
    input_array = np.array([[
        features["total_purchases_30d"],
        features["avg_order_value_30d"],
        features["days_since_last_purchase"],
    ]])
    
    # Run inference (async, BentoML handles batching multiple requests)
    prediction = await runner.predict.async_run(input_array)
    
    return {
        "user_id": features.get("user_id"),
        "churn_probability": float(prediction[0]),
        "churn_prediction": bool(prediction[0] > 0.5),
    }
```

```bash
# STEP 3: RUN AND DEPLOY

# Run locally
bentoml serve service:svc --reload

# Build a bento (packages model + code + dependencies into one artifact)
bentoml build

# Containerize (generates a Dockerfile automatically)
bentoml containerize churn_prediction_service:latest

# The resulting Docker image has everything:
# - Python environment
# - Model weights
# - Serving code
# - BentoML server
docker run -p 3000:3000 churn_prediction_service:latest
```

## 3. Ray Serve: Multi-Model Inference Pipelines

Ray Serve is built on top of Ray (a distributed computing framework). It shines when you need to compose multiple models into a single serving pipeline.

```python
import ray
from ray import serve
from starlette.requests import Request
import torch

ray.init()
serve.start()

# SCENARIO: a request goes through 3 models in sequence:
# 1. Language detector (decides language of input)
# 2. Appropriate language model for that language
# 3. Post-processor to format output

@serve.deployment(
    num_replicas=2,
    ray_actor_options={"num_cpus": 1.0},
)
class LanguageDetector:
    def __init__(self):
        from langdetect import detect
        self.detect = detect
    
    def detect_language(self, text: str) -> str:
        return self.detect(text)

@serve.deployment(
    num_replicas=1,
    ray_actor_options={"num_gpus": 0.5},  # share a GPU between replicas
)
class EnglishProcessor:
    def __init__(self):
        # Load English-specific model
        self.model = load_english_model()
    
    def process(self, text: str) -> str:
        return self.model.generate(text)

@serve.deployment(
    num_replicas=1,
    ray_actor_options={"num_gpus": 0.5},
)
class SpanishProcessor:
    def __init__(self):
        self.model = load_spanish_model()
    
    def process(self, text: str) -> str:
        return self.model.generate(text)

@serve.deployment
class Pipeline:
    """Orchestrates the multi-model pipeline."""
    
    def __init__(self, detector, english, spanish):
        self.detector = detector
        self.english = english
        self.spanish = spanish
    
    async def __call__(self, request: Request) -> dict:
        body = await request.json()
        text = body["text"]
        
        # Step 1: detect language
        language = await self.detector.detect_language.remote(text)
        
        # Step 2: route to appropriate processor
        if language == "en":
            result = await self.english.process.remote(text)
        elif language == "es":
            result = await self.spanish.process.remote(text)
        else:
            result = f"Unsupported language: {language}"
        
        return {"result": result, "language": language}

# DEPLOY THE PIPELINE
detector = LanguageDetector.bind()
english = EnglishProcessor.bind()
spanish = SpanishProcessor.bind()
pipeline = Pipeline.bind(detector, english, spanish)

serve.run(pipeline, route_prefix="/process")
```

```
RAY SERVE KEY FEATURES:

FRACTIONAL GPU ALLOCATION:
  ray_actor_options={"num_gpus": 0.5}
  Two replicas share one GPU — doubles throughput for GPU-idle workloads
  (Not available in most other frameworks)
  
INDEPENDENT SCALING:
  Each deployment scales independently.
  LanguageDetector is CPU-bound → scale with more CPUs.
  Processors are GPU-bound → scale with more GPU instances.
  
DYNAMIC REQUEST ROUTING:
  Route requests to different models based on content.
  Build A/B testing, language routing, complexity routing into the pipeline.

AUTOSCALING:
  num_replicas="auto" + configure min/max replicas + target concurrency
  Scales up and down based on queue depth.
```

## 4. BentoML vs. Ray Serve vs. Build Your Own

```
USE BENTOML WHEN:
  ✓ Serving a single model (or a few independent models)
  ✓ You want great out-of-the-box adaptive batching
  ✓ Easy containerization and deployment is a priority
  ✓ Team knows Python but not distributed systems
  ✓ Want a portable "bento" artifact deployable anywhere

USE RAY SERVE WHEN:
  ✓ Pipeline: multiple models compose a single response
  ✓ Need fractional GPU allocation (memory-efficient)
  ✓ Already using Ray for distributed training
  ✓ Need independent scaling of pipeline components
  ✓ Complex routing logic (A/B tests, language routing)

BUILD YOUR OWN (FastAPI) WHEN:
  ✓ Very simple model (one model, one endpoint, no batching)
  ✓ Team wants maximum control
  ✓ Existing FastAPI patterns on the team
  ✗ You'll manually implement everything BentoML gives for free
```

## 5. Senior Engineer Perspective

**BentoML solves the packaging problem:**

The hardest part of ML deployment isn't the model itself — it's packaging the model, its Python dependencies, its preprocessing code, and the serving logic into a single deployable artifact. BentoML's "bento" concept directly addresses this. The output is a Docker image that contains everything; no more "it works on my machine" failures because a team member has a different sklearn version.

**Ray Serve is distributed computing, not just serving:**

Ray Serve is powerful, but it brings Ray's operational complexity. You need to run a Ray cluster, understand Ray's resource model, and debug Ray-specific failures. For simple serving, this is overkill. For multi-model pipelines at scale — particularly when you need fractional GPU allocation or independent autoscaling of pipeline stages — it's the right tool.
