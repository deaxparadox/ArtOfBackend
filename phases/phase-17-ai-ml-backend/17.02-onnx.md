## 1. ONNX: Open Neural Network Exchange Format (17.2)

### What Is It?

ONNX (Open Neural Network Exchange, Microsoft + Facebook, 2017) is an open format for representing ML models. It allows models trained in one framework (PyTorch, TensorFlow, scikit-learn) to be exported and run in a different runtime — most importantly, ONNX Runtime (ORT), which is highly optimized for inference across CPU, GPU, and specialized accelerators.

**The problem ONNX solves:**

```
WITHOUT ONNX:
  Train in PyTorch → serve with PyTorch runtime
  Train in TensorFlow → serve with TensorFlow runtime
  
  Different frameworks have different performance characteristics.
  PyTorch is great for training but not always optimal for serving.
  Your serving infrastructure must carry the entire training framework.
  
  PyTorch model serving binary: ~2GB
  
WITH ONNX:
  Train in PyTorch → export to ONNX format → serve with ONNX Runtime
  
  ONNX Runtime is optimized for inference:
  - Operator fusion (combine adjacent operations)
  - Quantization (reduce precision from float32 to int8 → 4x smaller, faster)
  - Platform-specific acceleration (AVX-512 on Intel, NEON on ARM)
  - No training overhead
  
  ONNX Runtime serving binary: ~100MB
  Inference speed: typically 2-5x faster than PyTorch for the same model
```

### Exporting and Using ONNX

```python
import torch
import onnx
import onnxruntime as ort
import numpy as np

# STEP 1: Export PyTorch model to ONNX
model = MyModel()
# weights_only=True prevents arbitrary code execution from pickle
model.load_state_dict(torch.load("model.pt", weights_only=True))
model.eval()

# Define dummy input with the correct shape
dummy_input = torch.zeros(1, 10)  # batch_size=1, features=10

torch.onnx.export(
    model,
    dummy_input,
    "model.onnx",
    export_params=True,
    opset_version=17,         # ONNX opset version
    input_names=["features"],
    output_names=["predictions"],
    dynamic_axes={
        "features": {0: "batch_size"},     # batch dimension is dynamic
        "predictions": {0: "batch_size"},
    },
)

# STEP 2: Validate the exported model
onnx_model = onnx.load("model.onnx")
onnx.checker.check_model(onnx_model)
print("Model validated successfully")

# STEP 3: Run inference with ONNX Runtime
session = ort.InferenceSession(
    "model.onnx",
    providers=[
        "CUDAExecutionProvider",   # GPU (if available)
        "CPUExecutionProvider",    # CPU fallback
    ]
)

# Get input name from the model
input_name = session.get_inputs()[0].name  # "features"

# Run inference (much faster than PyTorch for production)
input_data = np.array([[1.0, 2.0, 3.0, ...]], dtype=np.float32)
predictions = session.run(None, {input_name: input_data})
print(predictions[0])  # model output

# PERFORMANCE COMPARISON (typical):
# PyTorch: 50ms per batch
# ONNX Runtime (CPU): 20ms per batch (2.5x faster)
# ONNX Runtime (GPU): 8ms per batch (6x faster)
# ONNX Runtime (quantized): 6ms per batch (8x faster, slight accuracy loss)
```

### ONNX Model Quantization

```python
from onnxruntime.quantization import quantize_dynamic, QuantType

# INT8 quantization: reduces model size 4x, speeds up CPU inference 2-4x
# Small accuracy loss (typically < 1% for classification tasks)
quantize_dynamic(
    model_input="model.onnx",
    model_output="model_quantized.onnx",
    weight_type=QuantType.QInt8,
)
```

## 2. TensorFlow Serving and TorchServe (17.3)

### TensorFlow Serving

TF Serving is a production ML serving system from Google, optimized for TensorFlow models:

```bash
# TensorFlow Serving via Docker
docker run -t --rm -p 8501:8501 \
    -v "/models/my_model:/models/my_model" \
    -e MODEL_NAME=my_model \
    tensorflow/serving

# REST API automatically available:
curl http://localhost:8501/v1/models/my_model:predict \
    -H "Content-Type: application/json" \
    -d '{"instances": [[1.0, 2.0, 3.0, 4.0]]}'

# gRPC API also available (lower latency for internal service calls):
# Port 8500
```

### TorchServe

TorchServe (AWS + Facebook) is PyTorch's official serving solution:

```bash
# Package model
torch-model-archiver \
    --model-name my_model \
    --version 1.0 \
    --model-file model.py \
    --serialized-file model.pt \
    --handler custom_handler.py \
    --export-path model_store/

# Start server
torchserve --start \
    --model-store model_store \
    --models my_model=my_model.mar \
    --ts-config config.properties

# Inference API:
curl http://localhost:8080/predictions/my_model \
    -H "Content-Type: application/json" \
    -d '{"data": [1.0, 2.0, 3.0]}'
```

## 3. Triton Inference Server (17.4): GPU Inference at Scale

NVIDIA Triton Inference Server is the production standard for high-throughput GPU-accelerated ML inference. It supports multiple model formats (ONNX, TensorFlow, PyTorch, TensorRT) and provides dynamic batching out of the box.

```
TRITON ARCHITECTURE:

  Client requests → Triton Server
                    ├── Model Repository (S3/local filesystem)
                    │   ├── model_1/ (ONNX format)
                    │   ├── model_2/ (TensorRT format — fastest on NVIDIA GPU)
                    │   └── ensemble/ (multi-model pipeline)
                    │
                    ├── Dynamic Batching Engine
                    │   Collects requests, forms optimal batches
                    │
                    ├── Model Instances (GPU or CPU)
                    │   Multiple concurrent model instances
                    │
                    └── HTTP/REST + gRPC endpoints

KEY TRITON FEATURES:
  ✓ Dynamic batching: accumulates requests within time window
  ✓ Concurrent model execution: multiple model instances in parallel
  ✓ Model ensembles: chain models in a pipeline (preprocessing → model → postprocessing)
  ✓ GPU memory management: efficient allocation across model instances
  ✓ Metrics endpoint: Prometheus-compatible, latency/throughput/GPU utilization
  ✓ TensorRT optimization: auto-converts models to fastest GPU format
```

```python
# Calling Triton from Python (tritonclient)
import tritonclient.http as httpclient
import numpy as np

client = httpclient.InferenceServerClient("localhost:8000")

# Check server health
assert client.is_server_ready()
assert client.is_model_ready("my_model")

# Prepare input
inputs = [
    httpclient.InferInput("features", [1, 10], "FP32")
]
inputs[0].set_data_from_numpy(np.array([[1.0, 2.0, 3.0, ...]], dtype=np.float32))

# Run inference
outputs = [httpclient.InferRequestedOutput("predictions")]
response = client.infer("my_model", inputs, outputs=outputs)
predictions = response.as_numpy("predictions")
```

## 4. When to Use Which Serving Solution

```
SIMPLE MODELS (scikit-learn, small PyTorch/TF):
  FastAPI + ONNX Runtime: simple, fast, no extra infrastructure
  Latency: sufficient for most use cases
  Operations: just a Python service

MEDIUM SCALE (PyTorch/TF, need batching, moderate GPU usage):
  TorchServe or TensorFlow Serving
  Managed batching, model versioning, REST/gRPC APIs
  Good for: single-model deployments at hundreds of RPS

HIGH SCALE (multiple models, GPU clusters, high throughput):
  NVIDIA Triton Inference Server
  Best-in-class GPU utilization, dynamic batching, model ensembles
  Used by: Netflix, Spotify, Google for production ML serving

LLM INFERENCE (see topic 17.6):
  vLLM, TGI (Text Generation Inference), Triton with TensorRT-LLM
  LLMs have unique requirements (KV cache, speculative decoding)
  Standard serving frameworks don't handle LLMs well
```
