## 1. What Is It? (Theory)

Every piece of data your program touches must live somewhere. Where it lives determines how fast you can access it — by orders of magnitude.

The memory hierarchy is the pyramid of storage systems in a computer, ordered from fastest+smallest+most expensive at the top, to slowest+largest+cheapest at the bottom.

**The real-world analogy:**

Think of a chef in a kitchen:
- **CPU registers** = what the chef holds in their hands right now (instant access, 1–2 items)
- **L1 cache** = ingredients on the counter directly in front of them (grab in 1 second)
- **L2/L3 cache** = ingredients in the refrigerator next to them (grab in 5 seconds)
- **RAM** = the pantry down the hall (30 second walk)
- **SSD** = the warehouse across the street (5 minute trip)
- **HDD** = a warehouse in another city (30 minute drive)
- **Network** = ordering from a supplier in another country (days)

A chef who keeps running to the other-country supplier for each ingredient will never keep up with orders. A chef who preloads commonly used ingredients onto their counter is fast. This is exactly how CPUs work — and it's why caching exists at every layer of software.

## 2. How It Actually Works (Deep Mechanics)

### The Full Hierarchy

```
MEMORY HIERARCHY

Level        Size          Latency        Bandwidth       Cost/GB
──────────────────────────────────────────────────────────────────
Registers    ~1 KB         0.3 ns         —               (in CPU)
L1 Cache     32–64 KB      ~1 ns          ~1 TB/s         (in CPU)
L2 Cache     256 KB–1 MB   ~4 ns          ~400 GB/s       (in CPU)
L3 Cache     4–64 MB       ~10–40 ns      ~200 GB/s       (in CPU)
RAM (DRAM)   4 GB–1 TB     ~100 ns        ~50 GB/s        ~$3/GB
NVMe SSD     100 GB–4 TB   ~100 µs        ~7 GB/s         ~$0.10/GB
SATA SSD     100 GB–4 TB   ~200 µs        ~500 MB/s       ~$0.07/GB
HDD          1–20 TB       ~10 ms         ~150 MB/s       ~$0.02/GB
Network      ∞             ~0.5–150ms+    ~10 MB–10 GB/s  ~$0.00 (leased)
──────────────────────────────────────────────────────────────────

RELATIVE SPEED (using RAM as baseline = 1):
  L1 cache:  100x faster than RAM
  L2 cache:  25x faster than RAM
  L3 cache:  3–10x faster than RAM
  RAM:       1x (baseline)
  NVMe SSD:  1000x SLOWER than RAM
  SATA SSD:  2000x SLOWER than RAM
  HDD:       100,000x SLOWER than RAM
  Network:   500,000x–1,500,000x SLOWER than RAM
```

**The number that should shock you:** Going from RAM to disk is 1,000x slower. Going from RAM to a cross-region network call is 1,500,000x slower. Every unnecessary disk read or network call in a hot code path is catastrophic.

### CPU Caches (L1/L2/L3)

The CPU cache is hardware-managed. Your code doesn't explicitly control it — the CPU automatically copies recently used data from RAM into cache. But you can write code that works *with* the cache or *against* it.

**Cache lines:** The cache operates in 64-byte chunks called cache lines. When you access one byte of RAM, the CPU fetches the surrounding 64 bytes. This is why sequential memory access is fast (next bytes are already in cache) and random access is slow (each access triggers a new cache line fetch).

```
SEQUENTIAL vs RANDOM ACCESS:

Array traversal (sequential — cache-friendly):
  arr[0], arr[1], arr[2], arr[3]...
  → CPU fetches cache line containing arr[0..15]
  → arr[1], arr[2]... already in L1 cache
  → Very fast (L1 hits)

HashMap traversal (random — cache-unfriendly):
  hashmap["key1"] → RAM address 0x4A23F0
  hashmap["key2"] → RAM address 0x9B12C8
  hashmap["key3"] → RAM address 0x1F4400
  → Each access hits a different cache line
  → Each one may be a RAM fetch (100ns each)
  → Very slow under load
```

**The practical consequence in Python:**

```python
import time

# Cache-friendly: sequential array access
arr = list(range(10_000_000))
start = time.perf_counter()
total = sum(arr)                    # Sequential, cache-friendly
print(f"Array sum: {time.perf_counter() - start:.3f}s")
# ~0.150s

# Cache-unfriendly: dictionary access with random keys
import random
d = {i: i for i in range(10_000_000)}
keys = list(d.keys())
random.shuffle(keys)
start = time.perf_counter()
total = sum(d[k] for k in keys)    # Random access, cache-unfriendly
print(f"Dict sum: {time.perf_counter() - start:.3f}s")
# ~0.900s — 6x slower, same amount of data
```

### RAM — The Working Memory

RAM (Random Access Memory) is where your running process lives: code, variables, data structures, stack frames, heap allocations.

Key properties:
- **Volatile:** RAM is erased when power is lost. Everything in RAM is temporary.
- **Byte-addressable:** You can read/write any individual byte.
- **Shared:** Multiple processes share RAM managed by the OS.

**What your Python/Node/Go process keeps in RAM:**
- The executable code
- All your variables, objects, lists, dicts
- The call stack (function calls, local variables)
- The heap (dynamic allocations — objects you `new` or `malloc`)
- File descriptor table, socket buffers
- OS kernel data structures

**Memory pressure:** When RAM fills up, the OS starts **swapping** — moving inactive memory pages to disk. This is catastrophic. An application that was accessing data at 50 GB/s (RAM) suddenly accesses it at 0.5 GB/s (SSD). The system appears frozen. Never let a backend service use more RAM than the machine has.

---

### Disk — The Persistent Store

Disks persist data across reboots. Your database files, log files, application code, and container images all live on disk.

**NVMe SSD vs SATA SSD vs HDD:**

```
NVMe SSD:  Connected directly to CPU via PCIe. Fastest.
           Read: ~3.5 GB/s seq, ~700K IOPS random
           Latency: ~100 µs
           Use: databases, high-performance applications

SATA SSD:  Connected via SATA interface (bottleneck: ~600 MB/s).
           Read: ~550 MB/s seq, ~100K IOPS random
           Latency: ~200 µs
           Use: general purpose, cheaper than NVMe

HDD:       Physical spinning disk. Mechanical seek time dominates.
           Read: ~150 MB/s sequential, ~100 IOPS random (!!!)
           Latency: ~10 ms per random read
           Use: cold storage, backups, large sequential data

KEY: Random I/O on HDD is devastating: 100 IOPS.
     A database doing random reads on HDD maxes at 100 queries/sec.
     Same database on NVMe SSD: 700,000 IOPS → 7000x more throughput.
     This is why "upgrade to SSD" is often the fastest win in old systems.
```

**Sequential vs random I/O:**

```
SEQUENTIAL READ: Reading a 100MB log file from start to finish.
  HDD: 100MB / 150 MB/s = 0.67 seconds
  SSD: 100MB / 3500 MB/s = 0.029 seconds

RANDOM READ: Looking up 1000 different rows in a database.
  HDD: 1000 seeks × 10ms = 10 seconds (!!)
  SSD: 1000 seeks × 0.1ms = 0.1 seconds

Databases are RANDOM access workloads.
HDDs are catastrophic for production databases.
```

---

### Network — The Slowest Layer

Network I/O is the bottom of the hierarchy — by far the slowest, and the most variable.

The key insight from [0.5 — Latency](./0.05-latency-bandwidth-throughput.md): a RAM read takes 100ns. A cross-region network round trip takes 100ms. That is a million times slower. Every unnecessary network call in a hot path is equivalent to making your code a million times slower for that operation.

**This is why every caching pattern exists.** Redis is just RAM on a machine nearby. An in-process cache is your application's own RAM. A CDN is RAM (or fast SSD) at an edge node near the user. All caches are "moving data up the memory hierarchy, closer to the CPU."

---

### How This Maps to Backend Architecture

Every backend caching layer maps directly to the memory hierarchy:

```
BACKEND CACHING LAYERS mapped to MEMORY HIERARCHY

In-process cache (e.g., Python dict, LRU cache)
  → Equivalent to: L3 cache / RAM
  → Latency: ~0 ms (same process memory)
  → Size: limited by server RAM
  → Drawback: not shared across servers

Redis / Memcached (remote in-memory cache)
  → Equivalent to: RAM on another machine
  → Latency: ~0.1–1 ms (LAN)
  → Size: limited by Redis server RAM
  → Benefit: shared across all app servers

Database (PostgreSQL on SSD)
  → Equivalent to: SSD
  → Latency: 1–50 ms (query execution + disk)
  → Size: limited by disk
  → Benefit: persistent, consistent

CDN (Cloudflare, CloudFront)
  → Equivalent to: RAM at a nearby PoP
  → Latency: ~1–5 ms (nearest edge)
  → Size: large (distributed across PoPs)
  → Benefit: physically close to user

Origin server (your app + DB, far away)
  → Equivalent to: network (the bottom)
  → Latency: 50–300 ms
  → Size: unlimited
  → Drawback: expensive, slow for distant users
```

**The caching strategy is always the same:** push data up the hierarchy, closer to the CPU, earlier in time.

## 3. Real-World Context

### Why PostgreSQL has a "shared_buffers" setting

PostgreSQL maintains its own in-memory cache of database pages called `shared_buffers`. By default, it's 128MB — embarrassingly small. For a production database, this should be 25% of total RAM.

Why? Because a database page read from `shared_buffers` (RAM) takes ~100ns. The same page read from the SSD takes ~100µs. That's 1000x slower. If your hot dataset fits in `shared_buffers`, your database runs at RAM speed. If it doesn't, every cache miss hits the disk.

This is why DBA's first question for a slow Postgres instance is: "What's your cache hit ratio?" A ratio below 99% means your data doesn't fit in RAM and you're hitting disk constantly.

```sql
-- Check PostgreSQL cache hit ratio
SELECT
  sum(heap_blks_hit) AS hits,
  sum(heap_blks_read) AS reads,
  round(sum(heap_blks_hit) /
    NULLIF(sum(heap_blks_hit) + sum(heap_blks_read), 0) * 100, 2) AS cache_hit_ratio
FROM pg_statio_user_tables;

-- Good: > 99%
-- Bad: < 95% → your working set doesn't fit in RAM → add RAM or tune queries
```

### Why Redis is fast

Redis is not magic. It is fast for one reason: **all data lives in RAM**. A Redis GET costs ~0.1ms. A PostgreSQL SELECT (hitting disk) costs 1–50ms. Redis is 10–500x faster because it's higher on the memory hierarchy.

The tradeoff: Redis stores all data in RAM (limited, expensive). PostgreSQL stores data on disk (large, cheap). The correct architecture: hot data in Redis, all data in PostgreSQL. Redis is just the "shelf" that holds today's ingredients.

### The "out of memory" incident

Common production disaster pattern:

1. Team adds a feature that loads a large dataset into memory
2. Works fine on the 8GB dev machine (dataset: 2GB)
3. Production machine has 4GB RAM (used 3.5GB already)
4. New feature runs → uses 2GB → total 5.5GB
5. OS starts swapping to disk
6. Everything using that memory is now disk-speed
7. Request latency goes from 20ms to 8000ms
8. Timeouts cascade
9. 3am phone call

The diagnostic signal: swap usage is non-zero on a production server. This should be treated as a P1 alert.

## 4. Code Examples

### Profiling memory usage in Python

```python
import tracemalloc
import linecache

def take_snapshot():
    """Take a memory snapshot and show top allocators."""
    snapshot = tracemalloc.take_snapshot()
    top_stats = snapshot.statistics("lineno")
    print("\nTop 10 memory allocations:")
    for stat in top_stats[:10]:
        frame = stat.traceback[0]
        filename = frame.filename
        lineno = frame.lineno
        line = linecache.getline(filename, lineno).strip()
        print(f"  {stat.size / 1024:.1f} KB  {filename}:{lineno}: {line}")

tracemalloc.start()

# Your code here — measure what it allocates
data = [{"id": i, "name": f"user_{i}", "email": f"user{i}@example.com"}
        for i in range(100_000)]

take_snapshot()
tracemalloc.stop()
```

### Implementing an in-process LRU cache

```python
from functools import lru_cache
import time

# Simple: Python's built-in LRU cache
# Stores results in RAM — the fastest possible cache
@lru_cache(maxsize=1000)  # keeps 1000 most recently used results
def get_user_permissions(user_id: int) -> list[str]:
    """
    Cache user permissions in-process RAM.
    First call: hits database (~5ms)
    Subsequent calls: hits RAM (~0 ms)
    """
    # Simulate DB call
    time.sleep(0.005)
    return ["read", "write"]  # normally from DB

# Manual LRU with TTL (for time-sensitive cached data)
from collections import OrderedDict
import threading

class TTLCache:
    """In-process cache with time-based expiry."""

    def __init__(self, maxsize: int = 1000, ttl_seconds: float = 60.0):
        self.maxsize = maxsize
        self.ttl = ttl_seconds
        self._cache: OrderedDict = OrderedDict()
        self._lock = threading.Lock()

    def get(self, key: str):
        with self._lock:
            if key not in self._cache:
                return None
            value, expiry = self._cache[key]
            if time.time() > expiry:
                del self._cache[key]
                return None
            self._cache.move_to_end(key)  # mark as recently used
            return value

    def set(self, key: str, value):
        with self._lock:
            if key in self._cache:
                self._cache.move_to_end(key)
            self._cache[key] = (value, time.time() + self.ttl)
            if len(self._cache) > self.maxsize:
                self._cache.popitem(last=False)  # evict oldest
```

### Checking memory pressure on a Linux server

```bash
# Is the server swapping? (should be 0 on a healthy backend server)
free -h
# Mem:   15.5G   12.1G   2.1G   512M   1.3G   3.4G
# Swap:     2G      0B     2G   ← 0 used is perfect; any swap usage is a problem

# What process is eating RAM?
ps aux --sort=-%mem | head -20

# Memory usage over time
vmstat 1 10
# si/so columns = swap in/out per second
# Non-zero = you're in trouble
```

**Language-agnostic pattern — where to cache at each layer:**

```
DATA ACCESS LATENCY + CACHING STRATEGY

Hot data (read >100/sec per server):
  → In-process cache (Python dict, LRU cache)
  → Latency: ~0ms, No network round trip
  → Risk: not shared, stale across restarts

Warm data (read 1–100/sec across all servers):
  → Redis / Memcached
  → Latency: ~0.5ms
  → Risk: cache invalidation, Redis memory limits

Cold data (read <1/sec):
  → Database (with indexes)
  → Latency: 1–50ms
  → This is where truth lives

Archive data (read rarely):
  → Object storage (S3, GCS)
  → Latency: 50–200ms
  → This is where history lives
```

## 5. Trade-offs

| Storage Level | Speed | Cost | Persistence | Shared Across Servers |
|---|---|---|---|---|
| In-process RAM | ~0ms | Free (server RAM) | No (dies on restart) | No |
| Redis | ~0.5ms | Medium ($) | Configurable | Yes |
| PostgreSQL (indexed) | ~5ms | Low ($) | Yes | Yes |
| PostgreSQL (unindexed) | ~100ms+ | Low ($) | Yes | Yes |
| SSD disk read | ~0.1ms | Low ($) | Yes | No |
| S3 / GCS | ~50–200ms | Very low (¢/GB) | Yes | Yes |

## 6. Senior Engineer Perspective

**What a junior gets wrong:**

The classic mistake: adding Redis caching to a query that runs fine at current load, without understanding why the cache will help or what "freshness" the cached data needs. Sometimes the database is fast enough. Sometimes a smarter index removes the need for caching. Redis is not a cure-all — it's a tool for moving hot data up the memory hierarchy.

The other mistake: ignoring memory entirely in Python/JavaScript (garbage-collected languages). "The runtime handles it" is true but dangerous. GC pauses are latency spikes. Memory leaks cause OOM kills. Keeping large objects in memory longer than needed fills RAM and triggers swapping.

**What a senior actually thinks:**

1. *"Where does this data live? How often is it read? How large is it?"* — These three questions determine the caching strategy.

2. *"What is the memory footprint of this request?"* — Large API responses, loading entire database tables into memory, accumulating results in lists before streaming — all common ways to blow up RAM.

3. *"What is the cache hit ratio and what happens on a miss?"* — A cache with a 50% hit ratio that serves 1000 RPS still sends 500 RPS to the database. If the database can only handle 200 RPS, you have a problem.

**The one mental shift:**

Stop thinking of data as "in the database." Think of data as living in a hierarchy: it might be in your RAM right now, or in Redis 1ms away, or in the DB 10ms away, or on an archive disk 200ms away. Your job is to ensure that hot data is always as high in the hierarchy as it can be, without violating correctness constraints.

Every cache is just answering: *"How do I avoid the trip to the warehouse for ingredients I use every day?"*
