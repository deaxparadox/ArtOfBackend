## 1. What Is It? (Theory)

Computers store numbers. They don't store letters. "A" is stored as the number 65. "B" is 66. "中" is 20013. An encoding is the agreement between the writer and the reader about which number maps to which character.

If the writer and reader use different encoding agreements, the result is **mojibake** — garbled text that looks like `Ã©lÃ¨ve` instead of `élève`. This has destroyed countless production systems, corrupted databases, and caused security vulnerabilities.

**The history in 3 sentences:** ASCII (1963) encoded 128 English characters in 7 bits. Then the world realized non-English speakers existed, and dozens of incompatible encodings appeared (Latin-1, Shift-JIS, GB2312, KOI8-R). Unicode (1991) unified every character in every language into one standard, and UTF-8 became the dominant encoding for transmitting Unicode across the internet.

**The rule you need right now:**

> **Always use UTF-8. Everywhere. No exceptions. If you must choose an encoding, choose UTF-8.**

## 2. How It Actually Works (Deep Mechanics)

### ASCII (American Standard Code for Information Interchange)

7-bit encoding. 128 characters total (0–127). Covers:
- 0–31: Control characters (newline=10, carriage return=13, tab=9, null=0)
- 32–126: Printable characters (space, A–Z, a–z, 0–9, punctuation)
- 127: Delete

```
ASCII table (selected):
  Decimal  Hex   Character
  ───────────────────────
  65       0x41  A
  66       0x42  B
  97       0x61  a
  48       0x30  0
  49       0x31  1
  32       0x20  (space)
  10       0x0A  \n (newline)
  13       0x0D  \r (carriage return)
  0        0x00  \0 (null)

"Hello" in ASCII bytes: 72 101 108 108 111
In hex: 48 65 6C 6C 6F
```

ASCII is still relevant because HTTP headers, JSON keys, URIs, and most protocol control characters are ASCII. When something breaks with non-ASCII content, ASCII is the baseline you debug from.

### The Pre-Unicode Chaos

ASCII only had 128 characters. The rest of the world's characters (accented European letters, Cyrillic, Greek, Arabic, Chinese, Japanese, Korean...) needed more. Different countries created different encodings, all incompatible:

### Unicode — The Universal Standard

Unicode is a character set (not an encoding) that assigns a unique **code point** to every character in every writing system. As of 2024, Unicode 15.1 defines 149,813 characters.

**Code points** are written as `U+XXXX` where XXXX is a hex number:

```
U+0041  →  A  (Latin capital A)
U+00E9  →  é  (Latin e with acute accent)
U+4E2D  →  中 (Chinese character for "middle/China")
U+1F600 →  😀 (Grinning face emoji)
U+0000  →  NULL (control character)
U+FFFD  →  ? (replacement character — the box-with-? you see for unknown chars)
```

Unicode is organized into **planes**:
- Plane 0 (U+0000–U+FFFF): Basic Multilingual Plane (BMP) — most common characters
- Plane 1 (U+10000–U+1FFFF): Supplementary Multilingual Plane — historic scripts, musical notation, many emoji
- Planes 2–16: More scripts, CJK extensions, private use areas

**Unicode ≠ UTF-8.** Unicode is the standard (what code points mean). UTF-8, UTF-16, UTF-32 are encodings (how to store those code points as bytes).

---

### UTF-8 — The Dominant Encoding

UTF-8 encodes Unicode code points as 1–4 bytes. It is variable-length.

```
CODE POINT RANGE     BYTES    BYTE PATTERN
──────────────────────────────────────────────────────
U+0000   – U+007F    1 byte   0xxxxxxx
U+0080   – U+07FF    2 bytes  110xxxxx 10xxxxxx
U+0800   – U+FFFF    3 bytes  1110xxxx 10xxxxxx 10xxxxxx
U+10000  – U+10FFFF  4 bytes  11110xxx 10xxxxxx 10xxxxxx 10xxxxxx

EXAMPLES:
  'A' (U+0041 = 65):
    Binary: 01000001
    UTF-8:  01000001  (1 byte — fits in ASCII range)
    Hex: 0x41

  'é' (U+00E9 = 233):
    Binary: 11101001
    UTF-8:  11000011 10101001  (2 bytes)
    Hex: 0xC3 0xA9

  '中' (U+4E2D = 20013):
    Binary: 100111000101101
    UTF-8:  11100100 10111000 10101101  (3 bytes)
    Hex: 0xE4 0xB8 0xAD

  '😀' (U+1F600 = 128512):
    UTF-8:  11110000 10011111 10011000 10000000  (4 bytes)
    Hex: 0xF0 0x9F 0x98 0x80
```

**Why UTF-8 won:**
1. **ASCII compatible:** The first 128 UTF-8 code points are identical to ASCII. All ASCII files are valid UTF-8. Backwards compatible.
2. **Self-synchronizing:** You can always find the start of a character by looking for bytes starting with `0` or `11`. You never have to read from the beginning.
3. **No null bytes in ASCII range:** Old C code that uses null-terminated strings works for ASCII content.
4. **Space-efficient for Western text:** English text uses 1 byte per character (same as ASCII).

---

### UTF-16 and UTF-32

**UTF-16:** Encodes most common characters as 2 bytes, rare characters (above U+FFFF) as 4 bytes. Used internally by Windows, Java, JavaScript (but not over the wire). The source of "surrogate pairs" — the representation of 4-byte characters as two 2-byte sequences.

**UTF-32:** Fixed 4 bytes per character. Simple (O(1) character indexing), wastes space (4x ASCII). Rarely used in practice except for internal processing.

**BOM (Byte Order Mark):** UTF-16 has an endianness problem — should the 2-byte value be stored big-endian or little-endian? The BOM (`U+FEFF`) at the start of a file tells the reader which endian to use. UTF-8 technically has a BOM too but it's redundant (UTF-8 has no endianness) and causes bugs — **never write a UTF-8 BOM.**

---

### The BOM Disaster (a real production bug)

A file written by Excel with UTF-8 BOM (`0xEF 0xBB 0xBF` at the start):

```
0xEF 0xBB 0xBF 0x69 0x64 0x2C 0x6E 0x61 0x6D 0x65
                i   d   ,   n   a   m   e
```

When you read this file and strip/split on commas, the first "column" is `﻿id` (BOM + "id"), not `id`. Your CSV parser silently reads the header wrong. The `id` column is not found. Queries fail. Data silently misrouted.

Fix: always open files with `encoding="utf-8-sig"` in Python to automatically strip the BOM.

---

### URL Encoding (Percent Encoding)

URIs can only contain a subset of ASCII characters (letters, digits, `-._~`). Everything else must be percent-encoded: `%XX` where XX is the hex value of the byte.

```
URL encoding examples:
  space     → %20  (or + in query strings)
  /         → %2F
  ?         → %3F
  #         → %23
  &         → %26
  é (UTF-8: 0xC3 0xA9) → %C3%A9
  中 (UTF-8: 0xE4 0xB8 0xAD) → %E4%B8%AD

Example URL with non-ASCII:
  https://example.com/search?q=café
  → https://example.com/search?q=caf%C3%A9
```

## 3. Real-World Context

### The Mojibake problem in databases

Classic production bug: you have a MySQL database with `latin1` charset (the MySQL default before 5.5). You insert UTF-8 data containing `é` (bytes: `0xC3 0xA9`). MySQL stores those 2 bytes as 2 Latin-1 characters: `Ã©`. When you read back, you get `Ã©` instead of `é`. Your database is silently corrupting non-ASCII content.

Fix: use `utf8mb4` in MySQL (not `utf8` — MySQL's `utf8` only supports 3-byte characters, which excludes emoji). In PostgreSQL, use `UTF8` (the default and correct choice).

```sql
-- MySQL: check what charset your tables use
SHOW CREATE TABLE users\G

-- MySQL: create table with correct charset
CREATE TABLE users (
    name VARCHAR(255)
) CHARACTER SET utf8mb4 COLLATE utf8mb4_unicode_ci;

-- PostgreSQL: always UTF8 by default
\l   -- shows encoding for each database
```

### The Twitter emoji incident

Early Twitter stored tweets in a MySQL database with `utf8` charset (3-byte max). When users posted emoji (4-byte UTF-8), MySQL silently truncated the tweet at the emoji. Users lost data. The fix was migrating to `utf8mb4`.

### Length vs. byte length

In UTF-8, `len("café") = 4` (characters) but `len("café".encode("utf-8")) = 5` (bytes, because `é` is 2 bytes). This matters when:
- Database column has `VARCHAR(100)` — is that 100 characters or 100 bytes?
- Truncating strings — truncating at byte boundary can split a multi-byte character
- Hashing — always hash bytes, not characters

```python
text = "café"
print(len(text))                    # 4 (Unicode characters)
print(len(text.encode("utf-8")))    # 5 (UTF-8 bytes)
print(len(text.encode("utf-16")))   # 10 (UTF-16 bytes, includes 2-byte BOM)

# Safe string truncation (never cut a multi-byte character)
def truncate_utf8_bytes(text: str, max_bytes: int) -> str:
    """Truncate string to max_bytes without splitting multi-byte characters."""
    encoded = text.encode("utf-8")[:max_bytes]
    # Decode with 'ignore' to drop any partial multi-byte character at the end
    return encoded.decode("utf-8", errors="ignore")
```

## 4. Code Examples

### Encoding and decoding correctly in Python

```python
# Python 3: str is Unicode, bytes is raw bytes. Never confuse them.

# Text → bytes (encode)
text: str = "Hello, 世界! 😀"
utf8_bytes: bytes = text.encode("utf-8")
print(utf8_bytes)
# b'Hello, \xe4\xb8\x96\xe7\x95\x8c! \xf0\x9f\x98\x80'

# Bytes → text (decode)
decoded: str = utf8_bytes.decode("utf-8")
print(decoded)  # Hello, 世界! 😀

# WRONG: trying to decode UTF-8 bytes as Latin-1
try:
    wrong = utf8_bytes.decode("latin-1")
    print(wrong)  # Hello, ä¸–ç•Œ! ð — mojibake
except UnicodeDecodeError:
    pass  # Won't error for latin-1 (every byte is valid), just wrong output

# Handle encoding errors gracefully
bad_bytes = b"Hello \xff World"  # 0xFF is not valid UTF-8
safe = bad_bytes.decode("utf-8", errors="replace")   # Hello  World
strict = bad_bytes.decode("utf-8", errors="ignore")  # Hello  World (drops bad bytes)

# Always specify encoding when opening files
with open("data.txt", "r", encoding="utf-8") as f:
    content = f.read()

# Strip UTF-8 BOM (from Excel-generated CSVs)
with open("data.csv", "r", encoding="utf-8-sig") as f:
    content = f.read()
```

### Detecting and normalizing encoding

```python
import chardet  # pip install chardet

def detect_and_decode(raw_bytes: bytes) -> str:
    """Detect encoding and decode safely."""
    result = chardet.detect(raw_bytes)
    encoding = result["encoding"] or "utf-8"
    confidence = result["confidence"]

    if confidence < 0.8:
        # Low confidence — fall back to UTF-8 with error replacement
        return raw_bytes.decode("utf-8", errors="replace")

    return raw_bytes.decode(encoding, errors="replace")

# Unicode normalization (important for comparisons)
import unicodedata

# 'é' can be represented two ways:
e_precomposed = "é"         # Single code point: é
e_decomposed  = "é"       # Two code points: e + combining acute accent

print(e_precomposed == e_decomposed)    # False (different code points!)
print(len(e_precomposed))               # 1
print(len(e_decomposed))               # 2

# Normalize to NFC (canonical composition) before comparing
nfc_a = unicodedata.normalize("NFC", e_precomposed)
nfc_b = unicodedata.normalize("NFC", e_decomposed)
print(nfc_a == nfc_b)  # True (both become the single precomposed form)

# Always normalize user input before storing or comparing
def normalize_text(text: str) -> str:
    return unicodedata.normalize("NFC", text)
```

### URL encoding

```python
from urllib.parse import quote, unquote, urlencode

# Encode a string for use in a URL
query = "café au lait"
encoded = quote(query)
print(encoded)  # caf%C3%A9%20au%20lait

# Encode query parameters
params = {"q": "café", "page": 2, "lang": "zh-CN"}
query_string = urlencode(params)
print(query_string)  # q=caf%C3%A9&page=2&lang=zh-CN

# Decode
print(unquote("caf%C3%A9%20au%20lait"))  # café au lait
```

## 5. Common Encoding Rules

```
THE ENCODING RULES EVERY BACKEND ENGINEER MUST FOLLOW:

1. ALWAYS use UTF-8 for everything
   - Database charset: utf8mb4 (MySQL) or UTF8 (PostgreSQL)
   - File I/O: always specify encoding="utf-8"
   - HTTP responses: always include Content-Type: text/html; charset=utf-8
   - JSON: always UTF-8 (the spec requires it)

2. NEVER use float for money
   - Store in integer (cents) or use Decimal type

3. ALWAYS normalize Unicode before comparing or storing
   - unicodedata.normalize("NFC", text)

4. ALWAYS use len(text.encode("utf-8")) for byte-length, len(text) for char-length
   - They differ for any non-ASCII content

5. ALWAYS strip BOM when reading files from Windows/Excel
   - encoding="utf-8-sig" in Python

6. Database connections need charset specified
   - SQLAlchemy: "mysql+pymysql://user:pass@host/db?charset=utf8mb4"
```

## 6. Senior Engineer Perspective

**What a junior gets wrong:**

The single most common encoding mistake: treating `str` and `bytes` as interchangeable. In Python 3, they are completely separate types. You cannot concatenate them, compare them, or use one where the other is expected. The conversion happens at `.encode()` and `.decode()`, and you must specify the encoding explicitly.

The second mistake: using MySQL's `utf8` charset (3 bytes max) instead of `utf8mb4` (4 bytes). Emoji are 4-byte UTF-8 characters. MySQL's `utf8` silently discards them or errors. This has caused data loss at every company that didn't know this.

**What a senior actually thinks:**

"Where is the encoding boundary in this request?"

Every backend request crosses encoding boundaries:
- Browser sends bytes over HTTP
- Your framework decodes them to strings (using Content-Type header)
- Your code processes strings
- You encode back to bytes for the database
- Database stores bytes and returns bytes (or strings depending on driver)
- You encode bytes to send in the HTTP response

At each crossing, the encoding must be specified and consistent. Gaps cause mojibake. The senior engineer traces data flow and verifies the encoding at every boundary.

**The production invariant:**

Every string that enters your system from outside must have its encoding declared. If the encoding is not declared, you are guessing — and you will guess wrong on some inputs. Validate and normalize at the boundary; trust internally.
