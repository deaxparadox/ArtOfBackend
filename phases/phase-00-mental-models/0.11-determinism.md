## 1. What Is It? (Theory)

A **deterministic** system always produces the same output given the same input. No matter when you run it, how many times, or on which machine — same input = same output.

A **non-deterministic** system may produce different outputs even with identical inputs. The output depends on factors beyond the input itself: the current time, random numbers, race conditions, network state, other concurrent processes.

**Why this matters for backend engineers:**

- Deterministic code is testable, reproducible, debuggable, and safe to retry
- Non-deterministic code is hard to test, hard to reproduce bugs in, and dangerous to retry
- Most backend bugs occur at the boundary between deterministic logic and non-deterministic external state (time, randomness, concurrent database writes, network)

## 2. How It Actually Works

### Sources of Non-Determinism

**1. Time (`now()`)**

```python
# Non-deterministic: result changes depending on WHEN you call it
def is_promo_active(promo):
    return promo.start_date <= datetime.now() <= promo.end_date

# Problem: You can't test this reliably.
# At 11:59pm the test passes. At 12:01am it fails.
# In production on a different server, "now" might differ by milliseconds.

# Deterministic: inject the current time
def is_promo_active(promo, now: datetime):
    return promo.start_date <= now <= promo.end_date

# Now testable: pass any time you want
assert is_promo_active(promo, datetime(2024, 6, 15)) == True
assert is_promo_active(promo, datetime(2024, 12, 31)) == False
```

**2. Random numbers**

```python
import random

# Non-deterministic
def assign_to_ab_group(user_id: int) -> str:
    return "A" if random.random() < 0.5 else "B"

# Problem: same user gets different group on each call
# Can't reproduce specific test scenarios
# Can't guarantee 50/50 split

# Deterministic: use user_id as seed (same user always same group)
def assign_to_ab_group(user_id: int) -> str:
    rng = random.Random(user_id)  # seed from user_id
    return "A" if rng.random() < 0.5 else "B"

# Or: use hash (no random at all)
import hashlib
def assign_to_ab_group(user_id: int) -> str:
    h = int(hashlib.md5(str(user_id).encode()).hexdigest(), 16)
    return "A" if h % 2 == 0 else "B"
# Same user_id → same group → always deterministic
```

**3. Ordering in databases**

```sql
-- Non-deterministic: ORDER BY not specified
-- Returns rows in arbitrary order (heap scan, index scan, parallelism)
SELECT * FROM users LIMIT 10;
-- Different result each query, different result on different servers

-- Deterministic: always specify ORDER BY
SELECT * FROM users ORDER BY created_at DESC, id DESC LIMIT 10;
-- Same result every time for the same data
```

**4. Race conditions**

```python
# Non-deterministic: two concurrent requests can both "win"
def increment_counter(user_id: int):
    user = db.get(user_id)          # Both threads read count=5
    user.request_count += 1         # Both threads compute count=6
    db.save(user)                   # Both threads write count=6
    # Result: count=6 instead of 7 — one increment lost

# Deterministic: atomic update
def increment_counter(user_id: int):
    db.execute(
        "UPDATE users SET request_count = request_count + 1 WHERE id = %s",
        (user_id,)
    )
    # Database executes this atomically — result is always correct
```

**5. Network order in distributed systems**

```
Server A sends message at t=0: "Set balance to $100"
Server B sends message at t=1: "Set balance to $200"

Both arrive at the database. Which arrives first?
In a distributed system: UNDEFINED. It depends on network conditions.

If A arrives first: final balance = $200 ✓
If B arrives first: final balance = $100 ✗ (wrong!)

Solution: use vector clocks, CRDT, or single-writer patterns
to impose deterministic ordering.
```

### Determinism and Testability

The core reason to write deterministic code: **it's testable**. You can write a test that always passes or always fails — not one that flakes depending on when it runs.

```python
# FLAKY TEST — non-deterministic
def test_promo_is_active():
    promo = Promo(start="2024-01-01", end="2025-12-31")
    assert is_promo_active(promo)  # Passes in 2024. Fails in 2026. ← FLAKY

# RELIABLE TEST — deterministic (inject time)
def test_promo_is_active():
    promo = Promo(start="2024-01-01", end="2025-12-31")
    assert is_promo_active(promo, now=datetime(2024, 6, 15))  # Always passes
    assert not is_promo_active(promo, now=datetime(2026, 1, 1))  # Always fails
```

### Making Code Deterministic — Practical Techniques

| Non-determinism source | Technique | Example |
|---|---|---|
| `datetime.now()` | Inject time as parameter | `def f(now: datetime)` |
| `random()` | Inject RNG or use deterministic hash | `Random(seed)` |
| External API calls | Dependency injection / mocking | `def f(api_client)` |
| Database state | Seed test DB / use transactions | Rollback after each test |
| Concurrent writes | Atomic DB operations / locks | `UPDATE ... SET x = x + 1` |
| Unordered queries | Always include `ORDER BY` | `ORDER BY id ASC` |
| File system | Inject file path / mock FS | `def f(file_reader)` |
| `os.environ` | Inject config / test overrides | `Settings(env=test_env)` |

---

### Non-Determinism That Cannot Be Eliminated

Some non-determinism is essential and unavoidable:
- **Security:** Random number generation for cryptographic keys must be non-deterministic. A deterministic key is predictable — therefore insecure.
- **Performance:** Query planner choices, OS scheduling, GC timing.
- **Physics:** Network delays, hardware failures.

The goal is not to eliminate all non-determinism — it's to **isolate non-determinism at the boundaries** and make your business logic deterministic. Push randomness and time to the edges. Keep the core deterministic.

```
Good architecture:
  [Random UUID generator] → inject → [Deterministic business logic]
  [Clock (time.now())]    → inject → [Deterministic business logic]
  [External API client]   → inject → [Deterministic business logic]

The business logic never directly calls random() or now().
It receives those values from outside.
This makes business logic fully testable and reproducible.
```

## 3. Real-World Context

### Reproducible builds

Determinism in build systems: Docker images must be deterministic — the same Dockerfile should produce the same image on any machine and at any time. Non-determinism in builds (e.g., `pip install requests` without pinning versions) means your image built today may differ from the one built last week, causing mysterious production-only bugs.

Solution: always pin dependencies (`requirements.txt`, `package-lock.json`, `go.sum`).

### Kafka message ordering

Kafka guarantees ordering **within a partition** but not across partitions. If you produce messages to multiple partitions, consumers may see them in a different order than they were produced. Systems that depend on message ordering must either:
1. Use a single partition (no parallelism — hurts throughput)
2. Use a partition key (all messages for a user_id go to the same partition)
3. Use sequence numbers and reorder at the consumer

## 4. Senior Engineer Perspective

**What a junior gets wrong:**

Writing tests against live time (`datetime.now()`) or random state, then wondering why tests pass on their laptop but fail in CI at 2am. Flaky tests are almost always caused by hidden non-determinism — time, randomness, test ordering (test A modifies state that test B depends on), or network calls.

**What a senior actually does:**

Uses dependency injection for every source of non-determinism. Not because it's "good design" in the abstract, but because it makes code testable and bugs reproducible. When something fails in production, the ability to reproduce it locally — by injecting the exact timestamp, the exact random seed, the exact mocked external API response — is the difference between a 10-minute fix and a 3-day investigation.

**The rule:**

> Deterministic logic + injected non-determinism at the boundary = testable, reproducible, debuggable systems.
