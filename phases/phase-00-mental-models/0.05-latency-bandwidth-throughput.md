## 1. What Is It? (Theory)

Three numbers govern every performance decision in backend engineering. Confuse them and your optimization efforts land in the wrong place. Understand them and you can predict system behavior before writing a line of code.

**Latency** — How long does a single operation take?
**Bandwidth** — How much data can flow per unit of time (theoretical maximum)?
**Throughput** — How much data actually flows per unit of time (observed reality)?

The confusion: most engineers use "bandwidth" and "throughput" interchangeably. They are not the same.

**The real-world analogy — a highway:**

- **Bandwidth** = number of lanes on the highway (capacity)
- **Throughput** = actual number of cars passing per hour (utilization)
- **Latency** = time for one car to drive from start to finish

A 6-lane highway (high bandwidth) during a traffic jam has terrible throughput. A 2-lane road with no traffic has throughput close to its bandwidth. A sports car on an empty highway has low latency. A truck full of hard drives has high throughput but high latency.

## 2. How It Actually Works (Deep Mechanics)

### Latency

Latency is the **time elapsed from initiating an operation to receiving its result**.

```
Latency = Time(first_byte_received) - Time(request_sent)
```

**Components of latency in a typical HTTP request:**

```
Total Latency = Propagation + Transmission + Processing + Queuing

PROPAGATION  Time for signal to travel the physical distance.
             Governed by physics (speed of light). Cannot be optimized away.
             NY → London: ~35ms one-way minimum.

TRANSMISSION Time to push all the bits onto the wire.
             Governed by bandwidth. Larger payload = more transmission time.
             1MB at 1Gbps = 8ms. 1MB at 100Mbps = 80ms.

PROCESSING   Time for router/server/database to do work.
             Governed by CPU, disk, memory. Your code lives here.
             Varies: 1ms (cached result) to 5s (complex query).

QUEUING      Time spent waiting because something upstream is busy.
             The most insidious. Can be zero or catastrophically high.
             Appears at: network buffers, request queues, database connection pool.
```

**Latency is not an average — it's a distribution:**

This is the most important and most ignored fact about latency. Averages lie.

```
Example: 1000 requests with these response times (ms):
  200, 200, 201, 199, 200, 202, 198, 200, 200... (990 requests)
  800, 900, 1200, 5000, 8000, 10000, 15000... (10 requests)

Average:  ~345ms  ← looks OK
Median:   200ms   ← also looks OK
p95:      900ms   ← 1 in 20 users sees this
p99:      5000ms  ← 1 in 100 users waits 5 seconds
p999:     15000ms ← 1 in 1000 users waits 15 seconds
```

That "1 in 100" doesn't sound bad until you realize: if your app makes 5 API calls per page load, each with independent p99=5s, then (1 - 0.99^5) = 4.9% of page loads experience at least one 5-second timeout. Nearly 1 in 20 users has a broken experience.

**This is why you must always monitor p95 and p99, never just averages.**

### Bandwidth

Bandwidth is the **maximum rate** at which data can be transferred across a network link.

```
Common bandwidths:
  Home WiFi (802.11n):      ~150 Mbps
  Home WiFi (802.11ax):     ~1 Gbps
  Home fiber (consumer):    100 Mbps – 1 Gbps
  Data center internal:     10 Gbps – 100 Gbps
  Backbone fiber:           100 Gbps – 400 Gbps
  Undersea cable (total):   Tbps across hundreds of fiber pairs
```

**Bandwidth ≠ throughput** because:
- Protocol overhead steals bandwidth (TCP headers, TLS overhead)
- Congestion causes backoff (TCP reduces send rate when packets drop)
- Encryption adds CPU overhead, slowing effective throughput
- Packet loss causes retransmission, wasting bandwidth

A "1 Gbps" link realistically delivers 600–800 Mbps of application data.

---

### Throughput

Throughput is the **actual observed rate** of successful data transfer or operations completed.

Two kinds you care about:
- **Data throughput:** megabytes per second transferred
- **Request throughput:** requests per second (RPS/QPS) handled

```
Throughput (requests/sec) = Concurrency / Latency

This is Little's Law (from queueing theory):
  L = λW

  L = average number of items in system (concurrency)
  λ = average arrival rate (throughput)
  W = average time in system (latency)

Rearranged:
  λ = L / W
  Throughput = Concurrency / Latency
```

**Little's Law example:**

Your API has:
- Average latency: 100ms (0.1 seconds)
- 10 concurrent workers

Maximum throughput: 10 / 0.1 = **100 requests/second**

To double throughput to 200 RPS, you can either:
1. Double concurrency (add more workers/threads) — from 10 to 20
2. Halve latency (optimize the slow path) — from 100ms to 50ms

Both achieve the same result. Knowing which is cheaper guides your optimization.

### The Latency-Bandwidth Trade-off

High bandwidth does NOT mean low latency. They are independent dimensions.

```
           HIGH BANDWIDTH
                │
                │
LOW LATENCY ────┼──── HIGH LATENCY
                │
                │
           LOW BANDWIDTH

Examples:
  Low latency, high bandwidth:  Data center fiber (1ms RTT, 100Gbps)
  Low latency, low bandwidth:   Ethernet on a local LAN (0.1ms, 100Mbps)
  High latency, high bandwidth: Geostationary satellite (600ms, 100Mbps)
  High latency, low bandwidth:  3G mobile in a rural area (150ms, 1Mbps)
```

**The satellite problem (why Starlink exists):**

Traditional geostationary satellites orbit at 35,786 km altitude. At the speed of light, that's ~240ms one-way. Round trip: 480ms minimum. No amount of bandwidth improvements can fix this — it's physics. Elon Musk's Starlink uses low-Earth orbit satellites at ~550 km altitude, giving ~3ms one-way latency. Same bandwidth, 80x less latency.

---

### Bandwidth Delay Product

When you have high bandwidth AND high latency, data "in flight" creates a hidden buffer:

```
Bandwidth Delay Product (BDP) = Bandwidth × Round Trip Time

Example: Transcontinental fiber
  Bandwidth: 1 Gbps = 125 MB/s
  RTT: 100ms = 0.1s

  BDP = 125 MB/s × 0.1s = 12.5 MB

This means: to fully utilize a 1 Gbps transcontinental link,
you need 12.5 MB of data "in flight" at all times.

TCP's send window must be large enough to hold BDP worth of data.
Default TCP window size: 65 KB → massively underutilizes high-BDP links.
TCP window scaling (RFC 1323) fixes this, but must be enabled.
```

This is why large file transfers across the globe are often slower than expected — TCP's default window is too small for high-BDP paths.

### Numbers Every Backend Engineer Must Know

Memorize this table. It will save you from bad decisions.

```
LATENCY REFERENCE TABLE (approximate, 2024)

Operation                           Latency
────────────────────────────────────────────────────────
L1 cache read                       ~0.5 ns
Branch misprediction                ~5 ns
L2 cache read                       ~7 ns
Mutex lock/unlock                   ~25 ns
L3 cache read                       ~40 ns
Main memory (RAM) read              ~100 ns       ← 200x slower than L1
Compress 1KB (Snappy)               ~3 µs
Read 4KB from SSD (random)          ~100 µs
Read 1MB sequentially from SSD      ~1 ms         ← 1,000x slower than RAM
Round trip in same datacenter       ~0.5 ms
Database query (indexed, local)     ~1–5 ms
Round trip cross-region (US→EU)     ~80 ms
Read 1MB sequentially from HDD      ~20 ms
Send packet US → Europe → US        ~150 ms       ← 150,000x slower than RAM
TLS handshake (TLS 1.3)             ~1 RTT = 80ms (cross-region)
DNS resolution (uncached)           ~50–100 ms
────────────────────────────────────────────────────────

KEY INSIGHT: Going from RAM to network is 150,000x slower.
Every unnecessary network call is catastrophically expensive.
```

## 3. Real-World Context

### The Tail Latency Tax at Scale

Amazon famously studied how latency affects revenue: every 100ms of added latency costs them 1% in sales. At Amazon's scale, 100ms = tens of millions of dollars per year.

But there's a deeper problem at scale called **tail latency amplification**:

```
Scenario: Your page load makes 100 backend calls (microservices architecture)
  Each service has p99 latency of 10ms
  Probability that AT LEAST ONE call takes >10ms:
  
  P(any slow) = 1 - P(all fast) = 1 - (0.99)^100 = 63%
  
  Your page has a 63% chance of being slow,
  even though each service is only slow 1% of the time.

Google's solution: "Hedged requests" — send the same request to 2 servers,
use whichever responds first, cancel the other. Costs ~2x backend traffic,
but eliminates tail latency completely.
```

### How Cloudflare uses the bandwidth/latency distinction

Cloudflare operates 300+ Points of Presence (PoPs) globally. When you make a request to a Cloudflare-protected site:

- Without Cloudflare: Your request travels from your city to the origin server (maybe 150ms)
- With Cloudflare: Your request hits the nearest PoP (~5ms), which has the cached response

Cloudflare didn't increase the bandwidth of the connection. They **reduced the physical distance** (propagation latency). The bandwidth is the same. The latency is 30x better. This is the core insight of edge computing.

### The database connection pool queue problem

Classic production failure: your API handles 1000 requests/second fine. At 1200 requests/second it starts seeing 500ms latency spikes. At 1500 it falls over.

```
DIAGNOSIS USING LITTLE'S LAW:

Your database connection pool: 10 connections
Average DB query time: 10ms

Maximum DB throughput: 10 / 0.010 = 1000 queries/second

At 1200 requests/second: queries queue up. Queue length grows.
Queuing adds latency. Latency increases. Queue grows faster.
Positive feedback loop → cascade failure.

FIX: Increase pool size to 20 → 2000 queries/sec capacity.
OR: Add caching → reduce DB queries per request.
OR: Optimize query → reduce latency from 10ms to 5ms → doubles capacity.
```

## 4. Code Examples

### Measuring latency correctly — percentiles, not averages

```python
import time
import statistics
import httpx

async def measure_latency(url: str, n: int = 100) -> dict:
    """
    Make N requests and report latency distribution.
    Never use average. Always use percentiles.
    """
    latencies = []

    async with httpx.AsyncClient() as client:
        for _ in range(n):
            start = time.perf_counter()         # high-resolution timer
            await client.get(url)
            elapsed_ms = (time.perf_counter() - start) * 1000
            latencies.append(elapsed_ms)

    latencies.sort()

    def percentile(data: list, p: float) -> float:
        idx = int(len(data) * p / 100)
        return round(data[min(idx, len(data) - 1)], 2)

    return {
        "n":         n,
        "min_ms":    round(min(latencies), 2),
        "p50_ms":    percentile(latencies, 50),   # median
        "p75_ms":    percentile(latencies, 75),
        "p95_ms":    percentile(latencies, 95),   # 1 in 20 users
        "p99_ms":    percentile(latencies, 99),   # 1 in 100 users
        "max_ms":    round(max(latencies), 2),
        "mean_ms":   round(statistics.mean(latencies), 2),  # lies
    }

# Run it
import asyncio
result = asyncio.run(measure_latency("https://httpbin.org/get", n=50))
print(result)
# {'n': 50, 'min_ms': 182.4, 'p50_ms': 210.1, 'p95_ms': 380.2,
#  'p99_ms': 520.7, 'max_ms': 543.1, 'mean_ms': 223.8}

# The mean (223ms) looks fine.
# The p99 (520ms) is what 1 in 100 users experiences.
# Focus on p99, not mean.
```

### Applying Little's Law to size a thread pool

```python
def calculate_pool_size(
    target_rps: int,
    avg_latency_ms: float,
    safety_factor: float = 1.5
) -> dict:
    """
    Use Little's Law to calculate required concurrency.

    L = λ × W
    concurrency = throughput × latency

    Args:
        target_rps: requests per second you need to handle
        avg_latency_ms: average time each request takes
        safety_factor: headroom above minimum (1.5 = 50% buffer)
    """
    avg_latency_s = avg_latency_ms / 1000
    min_concurrency = target_rps * avg_latency_s
    recommended = int(min_concurrency * safety_factor) + 1

    return {
        "target_rps": target_rps,
        "avg_latency_ms": avg_latency_ms,
        "minimum_threads": int(min_concurrency) + 1,
        "recommended_threads": recommended,
        "max_rps_with_recommended": round(recommended / avg_latency_s),
    }

# Example: API that does 50ms DB queries, needs to handle 200 RPS
result = calculate_pool_size(target_rps=200, avg_latency_ms=50)
print(result)
# {'target_rps': 200, 'avg_latency_ms': 50,
#  'minimum_threads': 11, 'recommended_threads': 16,
#  'max_rps_with_recommended': 320}
# Need at least 11 threads. Use 16 for headroom.
```

### Measuring bandwidth utilization

```python
import time

def measure_throughput(data_size_bytes: int, operation_fn) -> dict:
    """
    Measure actual throughput of any operation.
    Compare to theoretical maximum to find efficiency.
    """
    start = time.perf_counter()
    operation_fn()
    elapsed = time.perf_counter() - start

    throughput_mbps = (data_size_bytes / elapsed) / (1024 * 1024)
    throughput_gbps = throughput_mbps / 1024

    return {
        "data_mb":        round(data_size_bytes / 1024 / 1024, 2),
        "elapsed_s":      round(elapsed, 3),
        "throughput_mbps": round(throughput_mbps, 1),
        "throughput_gbps": round(throughput_gbps, 3),
    }

# Measure reading 100MB from disk
import os
def read_file():
    with open("/tmp/testfile", "rb") as f:
        f.read()

# First create a test file
with open("/tmp/testfile", "wb") as f:
    f.write(os.urandom(100 * 1024 * 1024))  # 100MB random data

result = measure_throughput(100 * 1024 * 1024, read_file)
print(result)
# {'data_mb': 100.0, 'elapsed_s': 0.082, 'throughput_mbps': 1219.5, 'throughput_gbps': 1.191}
```

**Language-agnostic pattern — the three questions for every performance problem:**

```
1. WHERE is the time going?
   → Profile. Find which operation consumes the latency budget.
   → Latency breakdown: DNS? TCP? TLS? Server processing? DB query? Serialization?

2. WHAT is the bottleneck?
   → Apply Little's Law: is concurrency too low, or latency too high?
   → Is it CPU-bound, I/O-bound, or network-bound?

3. HOW MUCH headroom is there?
   → Current throughput vs. maximum theoretical (Little's Law)
   → At what load does queuing begin? (That's your real capacity limit)
```

## 6. Trade-offs

| Optimization | Latency gain | Cost |
|---|---|---|
| CDN (edge caching) | Removes propagation latency | Cache invalidation complexity |
| Connection keep-alive | Removes TCP handshake (~10ms) | Server manages persistent connections |
| HTTP/2 multiplexing | Removes head-of-line blocking | More complex protocol |
| Async I/O | Removes thread blocking (adds concurrency) | More complex code |
| Caching (Redis) | DB query → RAM read (50ms → 0.1ms) | Cache invalidation, stale data |
| Read replicas | Spreads read load (more throughput) | Replication lag (stale reads) |
| Compression (gzip) | Reduces transmission latency | Adds CPU latency |

## 7. Senior Engineer Perspective

**What a junior gets wrong:**

Juniors fix latency by "making the code faster" — rewriting loops, switching libraries. This is almost never the bottleneck. The real latency budget is almost always dominated by I/O: database queries, external HTTP calls, disk reads.

The second mistake: optimizing average latency while ignoring p99. The average is easy to make look good. The p99 is what your worst users experience, and on a large enough user base, "worst users" is still millions of people.

**What a senior actually thinks:**

1. *"What is my latency budget for this operation?"* — Define the SLO first (e.g., p99 < 200ms). Then measure where you actually are.

2. *"Where is the time going?"* — Break down the request into phases. DNS, TCP, TLS, queue wait, DB query, computation, serialization. Each phase has a budget.

3. *"Am I CPU-bound or I/O-bound?"* — If the CPU is at 5% and the query takes 100ms, adding more CPU does nothing. The bottleneck is the DB.

4. *"What does Little's Law say my headroom is?"* — Before adding more servers, check if the current servers are even close to capacity. Adding servers to a system that's I/O-bound on a shared database just increases DB contention.

**The interview angle:**

"Our API is slow — how would you diagnose it?" is one of the most common senior engineering questions. The answer that gets the job:

1. Measure first — get percentile breakdown, not averages
2. Break down the latency — identify which component consumes the budget
3. Apply Little's Law — is it a concurrency problem or a latency problem?
4. Find the bottleneck — CPU, DB, network, external API?
5. Optimize the bottleneck, measure again, repeat

The wrong answer: "I'd optimize the database queries." Maybe. But you don't know until you measure.
