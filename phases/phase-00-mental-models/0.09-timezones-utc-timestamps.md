## 1. What Is It? (Theory)

Time is one of the most dangerous concepts in backend engineering. It seems trivial — just a number, right? — until a bug caused by daylight saving time takes down a production system at 2am on the first Sunday of November.

**The rule before everything else:**

> **Store all timestamps in UTC. Always. Convert to local time only when displaying to humans. Never store local time in your database.**

Why this single rule matters: UTC has no daylight saving time. UTC has no offsets. UTC is unambiguous. Local time is ambiguous — "2am on November 3, 2024" in New York does not exist (clocks fall back, that hour happens twice), and storing it creates data that cannot be correctly interpreted.

## 2. How It Actually Works (Deep Mechanics)

### UTC (Coordinated Universal Time)

UTC is the world's time standard. It is:
- Based on International Atomic Time (TAI) plus leap seconds
- The basis for all time zones (every zone is UTC+offset or UTC-offset)
- Never affected by Daylight Saving Time
- The correct format for storing, transmitting, and comparing timestamps

```
UTC offsets for selected zones:
  UTC+0:00  London (winter), Dublin, Reykjavik
  UTC+1:00  Paris, Berlin, Rome (winter)
  UTC+2:00  Helsinki, Kyiv, Johannesburg
  UTC+3:00  Moscow, Riyadh
  UTC+5:30  Mumbai, Kolkata (note: non-integer offset!)
  UTC+8:00  Beijing, Singapore, Perth
  UTC+9:00  Tokyo, Seoul
  UTC-5:00  New York (winter), EST
  UTC-8:00  Los Angeles (winter), PST

Important: offsets change with DST.
  New York summer: UTC-4 (EDT)
  New York winter: UTC-5 (EST)
```

---

### Unix Timestamps

A Unix timestamp (also called Unix time, POSIX time, epoch time) is the number of **seconds** (or milliseconds) elapsed since **January 1, 1970, 00:00:00 UTC**, not counting leap seconds.

```
January 1, 1970 00:00:00 UTC = Unix timestamp 0

Examples:
  2024-01-01 00:00:00 UTC = 1704067200
  2024-06-15 12:30:00 UTC = 1718450200
  Current time: (run `date +%s` in terminal)

Unix timestamps:
  → Are always UTC (no timezone ambiguity)
  → Are always integers (no formatting ambiguity)
  → Are easy to compare, sort, subtract
  → Are the correct way to store time in databases
```

**The 2038 problem:**

32-bit Unix timestamps overflow on **January 19, 2038, at 03:14:07 UTC** (when the signed 32-bit counter reaches 2,147,483,647). This is the Y2K of our time. Embedded systems, old Linux kernels, and legacy databases storing timestamps as 32-bit integers will fail. Always use 64-bit timestamps (milliseconds or seconds).

**Millisecond vs. second timestamps:**

```python
import time

# Second precision (enough for most events)
ts_seconds = int(time.time())       # e.g., 1718450200

# Millisecond precision (needed for high-frequency events, logs)
ts_ms = int(time.time() * 1000)     # e.g., 1718450200123

# Nanosecond precision (needed for profiling, databases)
import time
ts_ns = time.time_ns()              # e.g., 1718450200123456789
```

### Daylight Saving Time (DST) — The Source of Production Bugs

DST is the practice of advancing clocks by 1 hour in spring ("spring forward") and falling back 1 hour in autumn ("fall back"). Not all countries observe it. Of those that do, they don't change on the same date.

**The two DST bugs:**

**Spring forward (clocks jump 1 hour):**
```
March 10, 2024, 2:00am EST becomes 3:00am EDT
The hour from 2:00am to 2:59am DOES NOT EXIST.

If you schedule a job to run at 2:30am: IT NEVER RUNS.
If a user sets an alarm for 2:30am: they miss it.
```

**Fall back (clocks repeat 1 hour):**
```
November 3, 2024, 2:00am EDT becomes 1:00am EST
The hour from 1:00am to 1:59am HAPPENS TWICE.

"1:30am on Nov 3" is ambiguous — which 1:30am?
If you store this in local time, you cannot tell which one.
If your system processes events, it may process the 1-hour block twice.
```

**Real incident: Airline booking bug**

An airline stored booking timestamps in local US time. On fall-back Sunday, all bookings created during the repeated hour appeared to be out of order. Their fraud detection system (which relied on chronological ordering) flagged all those bookings as suspicious and declined them. Thousands of legitimate bookings were rejected.

The fix: store in UTC. The problem disappears entirely.

### Time Zone Database (tzdata)

Time zones are not just offsets. They are historical records of what offset a location used, when DST transitions happened, and when policies changed. The **IANA Time Zone Database** (tzdata) is the authoritative source.

```
Time zone names follow the pattern: Region/City
  America/New_York    (not "EST" or "US Eastern")
  Europe/London       (not "GMT" or "UK")
  Asia/Kolkata        (not "IST" — "IST" is ambiguous: India, Israel, Ireland)
  Pacific/Auckland
  UTC

NEVER use abbreviations like EST, PST, CET — they are ambiguous.
  IST = Indian Standard Time (UTC+5:30)
  IST = Irish Standard Time (UTC+1:00)
  IST = Israel Standard Time (UTC+2:00)
  All three are called "IST". Use IANA names instead.
```

---

### ISO 8601 — The Standard Date Format

When you must represent time as a string (in API responses, logs, JSON), use ISO 8601:

```
ISO 8601 formats:
  Date only:          2024-06-15
  Time only:          14:30:00
  Combined (UTC):     2024-06-15T14:30:00Z          (Z = UTC)
  With offset:        2024-06-15T14:30:00+05:30      (Mumbai)
  With milliseconds:  2024-06-15T14:30:00.123Z
  With microseconds:  2024-06-15T14:30:00.123456Z

NEVER use:
  06/15/2024     ← ambiguous (US format)
  15/06/2024     ← ambiguous (European format)
  15-Jun-2024    ← not machine-parseable
  2024-6-5       ← not zero-padded (breaks sorting)
  Jun 15, 2024 2:30pm EST  ← ambiguous timezone abbreviation
```

ISO 8601 strings sort correctly as strings (because they go from most to least significant: year → month → day → hour → minute → second). `"2024-01-01" < "2024-06-15"` is true as a string comparison. This is intentional.

### Storing Time in Databases

```
PostgreSQL:
  TIMESTAMP           — stores date and time, NO timezone info
                        (naive timestamp — do not use for UTC timestamps)
  TIMESTAMPTZ         — stores date and time WITH timezone offset
                        (converts to UTC on storage, converts back on retrieval)
  → Always use TIMESTAMPTZ. Never use TIMESTAMP for real-world times.

MySQL:
  DATETIME            — stores local time, no timezone
  TIMESTAMP           — stores UTC (converts from session timezone on write)
  → Use DATETIME with application-level UTC enforcement, OR
  → Use TIMESTAMP (but beware the range limit: 1970–2038)
  → For > 2038: store as BIGINT (Unix milliseconds)

Best practice for any database:
  → Store as BIGINT (Unix milliseconds since epoch)
  → Unambiguous, timezone-free, no database driver weirdness
  → Easy to compare, sort, and do arithmetic
  → Works identically across all databases
```

## 3. Real-World Context

### The Cloudflare 1-second outage (2017)

Cloudflare's software had a bug related to a leap second. On December 31, 2016, at 23:59:60 UTC, the world added a leap second (UTC occasionally adds one to stay in sync with Earth's rotation). Cloudflare's Go code included `assert time.Now() >= lastTime` — but during the leap second, `time.Now()` briefly went backward (the OS clock adjusted), making the assertion fail. Several Cloudflare services crashed for about 2 hours.

The lesson: time can go backward. Always handle this. Use monotonic clocks for measuring intervals, not wall clocks.

### The Boeing 737 MAX MCAS clock issue

Embedded systems in aircraft often use 32-bit unsigned integers for timestamps (milliseconds since boot). These wrap around every ~49 days. If an aircraft flies for >49 days without being power-cycled... the timestamp counter wraps to zero. This has caused silent issues in flight management computers where "49 days since boot" became "0 days since boot."

The lesson: always use 64-bit timestamps. The cost is 4 extra bytes. The risk of not doing so is catastrophic.

### Every company's UTC migration story

Almost every company that started storing local time eventually migrated to UTC. The pattern:
1. "We only have users in one timezone, it's fine."
2. Company expands internationally.
3. Data from different timezones is now in one table with incomparable timestamps.
4. Painful, expensive migration to UTC.

The migration is hard: you have to determine what timezone every old record was stored in, convert it, handle DST edge cases, update reporting. Doing it right from day one costs nothing.

## 4. Code Examples

### The correct way to work with time in Python

```python
from datetime import datetime, timezone, timedelta
import time

# WRONG: naive datetime (no timezone info — is this UTC? Local? Unknown?)
naive = datetime(2024, 6, 15, 14, 30, 0)
print(naive.tzinfo)  # None — dangerous

# RIGHT: always use timezone-aware datetimes
utc_now = datetime.now(timezone.utc)
print(utc_now)  # 2024-06-15 14:30:00.123456+00:00

# Unix timestamp → UTC datetime
ts = 1718450200
dt = datetime.fromtimestamp(ts, tz=timezone.utc)
print(dt)  # 2024-06-15 12:36:40+00:00

# UTC datetime → Unix timestamp
ts_back = dt.timestamp()
print(int(ts_back))  # 1718450200

# Store as ISO 8601 string (for JSON APIs)
iso_string = utc_now.isoformat()
print(iso_string)  # 2024-06-15T14:30:00.123456+00:00

# Parse ISO 8601 string
from datetime import datetime
parsed = datetime.fromisoformat("2024-06-15T14:30:00+00:00")
print(parsed.tzinfo)  # UTC+00:00

# Convert UTC to local time for display
import zoneinfo  # Python 3.9+
kolkata = zoneinfo.ZoneInfo("Asia/Kolkata")
kolkata_time = utc_now.astimezone(kolkata)
print(kolkata_time)  # 2024-06-15 20:00:00.123456+05:30

# NEVER do this — converting without specifying timezone
# datetime.now()  ← returns local machine time, no tz info
# datetime.utcnow()  ← returns UTC but as NAIVE datetime (no tzinfo!)
```

### Using arrow (a better datetime library)

```python
import arrow  # pip install arrow

# Current time in UTC
now = arrow.utcnow()
print(now)  # 2024-06-15T14:30:00.123456+00:00

# Convert to local timezone for display
in_kolkata = now.to("Asia/Kolkata")
print(in_kolkata)  # 2024-06-15T20:00:00.123456+05:30

# Humanize (for display)
past = arrow.utcnow().shift(hours=-3)
print(past.humanize())  # "3 hours ago"

# Parse multiple formats
t1 = arrow.get("2024-06-15T14:30:00Z")
t2 = arrow.get("2024-06-15 14:30:00", "YYYY-MM-DD HH:mm:ss")
t2_utc = t2.replace(tzinfo="UTC")  # MUST specify timezone when parsing naive strings

# Time arithmetic
tomorrow = arrow.utcnow().shift(days=1)
in_30_min = arrow.utcnow().shift(minutes=30)

# Unix timestamp
ts = arrow.utcnow().timestamp()  # seconds
ts_ms = int(arrow.utcnow().timestamp() * 1000)  # milliseconds
```

### Database — storing and retrieving with PostgreSQL (SQLAlchemy)

```python
from sqlalchemy import Column, BigInteger, func
from sqlalchemy.dialects.postgresql import TIMESTAMP
from datetime import datetime, timezone

class Event(Base):
    __tablename__ = "events"

    id = Column(BigInteger, primary_key=True)

    # Option 1: PostgreSQL TIMESTAMPTZ (best for Postgres-specific apps)
    created_at = Column(
        TIMESTAMP(timezone=True),
        nullable=False,
        server_default=func.now()   # DB sets current UTC time on insert
    )

    # Option 2: BIGINT (milliseconds since epoch) — portable, unambiguous
    created_at_ms = Column(BigInteger, nullable=False)

# Insert with UTC time
new_event = Event(
    created_at_ms=int(datetime.now(timezone.utc).timestamp() * 1000)
)

# Query events in last 24 hours
from datetime import timedelta
cutoff_ms = int((datetime.now(timezone.utc) - timedelta(days=1)).timestamp() * 1000)
recent = db.query(Event).filter(Event.created_at_ms >= cutoff_ms).all()
```

**Language-agnostic rules:**

```
TIME HANDLING RULES

STORAGE:
  → Store in UTC always
  → Use TIMESTAMPTZ in PostgreSQL, BIGINT(ms) for portability
  → Never store local time in the database

TRANSMISSION (APIs):
  → Use ISO 8601 with explicit UTC offset: "2024-06-15T14:30:00Z"
  → Include timezone info in every timestamp in your API
  → Never return bare date strings ("2024-06-15") for timestamps — add time too

DISPLAY:
  → Convert UTC → user's local timezone only at the display layer
  → Store user's timezone preference (IANA name) in their profile
  → Never store the result of a local-time conversion

COMPARISON:
  → Always compare timestamps in UTC
  → For "within 24 hours": compare Unix ms values, not date strings

SCHEDULING:
  → Store "run at 9am New York time" as the IANA timezone + wall clock time
  → Compute next UTC execution time at schedule time
  → Recompute after DST transitions (the UTC time changes even if wall time doesn't)
```

## 6. Senior Engineer Perspective

**What a junior gets wrong:**

Three things, in order of frequency:

1. **Storing local time in the database** — "Our users are all in one timezone." They won't be forever. And even if they are: DST still breaks things. Always UTC.

2. **Using `datetime.utcnow()`** — This returns a naive datetime (no `tzinfo`). It looks like UTC but Python doesn't know that. When you pass it to SQLAlchemy or serialize it to JSON, timezone information is lost. Use `datetime.now(timezone.utc)` instead.

3. **Parsing user input without specifying timezone** — `datetime.strptime("2024-06-15 14:30", "%Y-%m-%d %H:%M")` produces a naive datetime. If the user said "14:30 in Mumbai" and you don't record that, you've lost information forever.

**What a senior actually thinks:**

"What is the user's intent, and what timezone context do I need to preserve it?"

When a user books a flight from Mumbai to New York at "6am", they mean 6am Mumbai time on departure day. You store that as: `{"wall_time": "06:00", "timezone": "Asia/Kolkata", "date": "2024-12-15"}`. You separately compute the UTC timestamp for sorting and comparison. The wall time is what the user intended. The UTC timestamp is what the system uses.

These are two different things. Confusing them destroys data.

**The most expensive timezone bug:**

In 2012, an automated trading system at Knight Capital failed due to a software deployment bug, but many related systems have failed due to timezone mismatches in trading order timestamps. A timestamp off by 1 hour means an order that should be "during market hours" appears to arrive "after market close." In HFT, this can mean the difference between a valid trade and a regulatory violation.

In domains with financial, medical, or legal significance — always, always UTC with explicit timezone when displaying.
