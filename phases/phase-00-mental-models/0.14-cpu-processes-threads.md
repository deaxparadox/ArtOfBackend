## 1. What Is It? (Theory)

Your backend code is text. The CPU executes electrical signals. Between those two facts is a rich system of processes, threads, schedulers, and system calls that determines how fast your code runs, how many requests it can handle simultaneously, and what happens when it blocks.

Understanding this is what separates engineers who can reason about performance from those who just guess.

**The real-world analogy:**

A CPU core is like a single chef. A multi-core CPU is like a kitchen with multiple chefs. **Processes** are like separate kitchens — completely isolated, their own tools, ingredients, ventilation. **Threads** are like multiple chefs working in the same kitchen — sharing the same tools and workspace, able to hand ingredients to each other, but needing to coordinate so they don't grab the same knife simultaneously.

The OS **scheduler** is the restaurant manager, deciding which chef works on which dish at which moment, constantly switching them between tasks to keep everyone productive.

## 2. How It Actually Works

### From Source Code to CPU Instructions

```
YOUR CODE (Python):
  def add(a, b): return a + b

  ▼ Python interpreter (CPython)

BYTECODE:
  LOAD_FAST  a
  LOAD_FAST  b
  BINARY_ADD
  RETURN_VALUE

  ▼ CPython JIT / C runtime

MACHINE CODE (x86-64 assembly):
  mov  rax, [rbp-8]   ; load a
  add  rax, [rbp-16]  ; add b
  ret                 ; return

  ▼ CPU

ELECTRICAL SIGNALS:
  Billions of transistors switching on/off per second
```

**Key point:** Python is slow not because `a + b` is slow, but because each bytecode operation passes through layers of interpretation overhead. Go and Rust compile directly to machine code — fewer layers, faster execution.

---

### Processes

A **process** is a running instance of a program. Each process has:
- Its own isolated virtual address space (memory)
- Its own file descriptors (open files, sockets)
- Its own environment variables
- At least one thread (the main thread)

```
PROCESS MEMORY LAYOUT:
┌────────────────────┐ High address
│       Stack        │ Function calls, local variables (grows downward)
│         ↓          │
│                    │
│         ↑          │
│        Heap        │ Dynamically allocated objects (malloc, new)
├────────────────────┤
│  Initialized data  │ Global/static variables with initial values
├────────────────────┤
│ Uninitialized data │ Global/static variables (zero-initialized)
├────────────────────┤
│       Code         │ Your compiled program instructions (read-only)
└────────────────────┘ Low address
```

**Process isolation is both a feature and a cost:**
- Feature: Process A cannot corrupt Process B's memory (a crash in one doesn't crash the other)
- Cost: Communication between processes is expensive (requires IPC: pipes, sockets, shared memory, message passing)

---

### Threads

A **thread** is the unit of execution within a process. A process can have many threads. Threads within the same process share:
- Heap memory
- Global variables
- File descriptors
- Code

Each thread has its own:
- Stack (separate local variables for each function call chain)
- Program counter (which instruction it's currently executing)
- CPU registers

```
MULTI-THREADED PROCESS:

Process (shared):
  Heap: [user_cache = {...}, db_connection_pool = [...]]
  Code: [all your functions]
  File Descriptors: [socket to DB, socket to Redis, log file]

Thread 1 (its own stack):       Thread 2 (its own stack):
  handling request A               handling request B
  local: user = User(id=1)         local: user = User(id=2)
  local: response = {}             local: response = {}
```

**The cost of creating a thread:**
- ~1–8 MB of stack memory per thread
- OS kernel data structures
- Context switch overhead

A server with 1000 threads uses 1–8 GB of RAM just for thread stacks, before any application data.

---

### Context Switching

A single CPU core can only execute one thread at a time. The OS gives each thread a **time slice** (~1–10ms) then switches to another thread. This is **context switching**.

```
CONTEXT SWITCH:

Thread A running...
  CPU saves A's registers, program counter, stack pointer
  CPU loads B's registers, program counter, stack pointer
Thread B running...
  (time slice expires)
  CPU saves B's state
  CPU loads A's state
Thread A continues from where it left off...

Cost of one context switch: ~1–10 µs
With 1000 threads on a 4-core server:
  250 threads per core
  Each thread gets a time slice every 250 × 1-10ms = 0.25–2.5 seconds
  Thousands of context switches per second
  Just the overhead: 1000 switches × 10µs = 10ms/second per core on overhead
```

This is why **massive thread counts degrade performance** — the server spends more time switching between threads than actually running user code.

### System Calls

Your user-space code cannot directly access hardware. To read from disk, open a socket, allocate memory, or interact with the OS, your code makes a **system call** — a request to the OS kernel.

```
SYSTEM CALL FLOW:

User space code:
  open("file.txt")   ← your Python/Go/Node code

  ▼ system call interface

Kernel space:
  sys_open()         ← kernel handles hardware access
  VFS layer
  Filesystem driver
  Disk driver
  Hardware: SSD/HDD

  ▼ return value

User space:
  file descriptor = 3   ← you get a handle back
```

**Cost of a system call:** ~100ns–10µs depending on what it does. Thousands of system calls per request (reading files, writing to sockets, memory allocation) add up. This is why in-process operations are always faster than operations that cross the kernel boundary.

---

### Python's GIL (Global Interpreter Lock)

This is critical for Python backend engineers. Python's reference implementation (CPython) has a **Global Interpreter Lock** — a mutex that allows only one thread to execute Python bytecode at a time.

```
GIL BEHAVIOR:

Thread A (running Python):   ═══════════════╗
                                           GIL wait
Thread B (waiting for GIL): ───────────────╚═══════════
                                           GIL acquired

Only one thread runs Python code at a time,
even on a multi-core CPU.

EXCEPTION: I/O operations release the GIL.
  Thread A: waiting for database response → releases GIL
  Thread B: can now run Python code        → acquires GIL
  Thread A: DB response arrives → waits to reacquire GIL
```

**Implications for Python backends:**

| Workload | GIL impact | Solution |
|---|---|---|
| I/O-bound (DB, HTTP, disk) | Low — GIL released during I/O | Multi-threading works fine |
| CPU-bound (image processing, ML, crypto) | High — GIL blocks other threads | Use `multiprocessing` or move to Go/Rust |

**This is why Python web servers use multiple worker processes (Gunicorn) or async I/O (asyncio), not multiple threads.** Multiple processes = no GIL (each process has its own). Async = no threads needed for I/O concurrency.

### The Server Concurrency Model

Different languages/frameworks handle concurrent requests differently:

```
MODEL 1: Thread per request (Java Spring, old Rails)
  New request → new thread (or take from pool) → handle → return to pool
  
  Pro: Simple, blocking code style
  Con: Limited by thread pool size, context switch overhead
  
  Capacity: ~500–2000 concurrent requests per server (before performance degrades)

MODEL 2: Async event loop (Node.js, Python asyncio, Go)
  Single thread → event loop → handles I/O callbacks non-blocking
  
  Pro: Handles thousands of concurrent connections with one thread
  Con: CPU-bound work blocks the entire loop, more complex code
  
  Capacity: ~10,000–100,000 concurrent requests per server

MODEL 3: Goroutines (Go)
  M goroutines mapped to N OS threads (M:N scheduling)
  Goroutines: ~2–8 KB stack (vs ~1–8 MB for OS thread)
  
  Pro: Both parallel and concurrent, cheap to create
  Con: Go-specific
  
  Capacity: Millions of concurrent goroutines possible

MODEL 4: Multi-process (Python Gunicorn, PHP-FPM)
  N worker processes, each handles requests independently
  
  Pro: True CPU parallelism (no GIL), process isolation
  Con: More memory (each process has its own heap), IPC for shared state
  
  Capacity: N workers × M requests each
```

## 3. Real-World Context

### Why Nginx handles 10,000 connections and Apache handles ~150

**Apache (old model):** One thread (or process) per connection. At 150 concurrent connections, you have 150 threads. At 1000, the context switching overhead crushes performance. Memory: 150 × 8MB stack = 1.2GB just for stacks.

**Nginx:** Single master process, a few worker processes, each running an async event loop. One worker handles thousands of connections non-blocking. At 10,000 concurrent connections: minimal memory, no context switching.

This difference is what C10K problem research (early 2000s) solved — how to handle 10,000 concurrent connections. The answer: async I/O, not thread-per-connection.

### The Python multiprocessing vs. threading decision

```python
# CPU-bound work: use multiprocessing (bypass GIL)
import multiprocessing

def cpu_heavy_task(data):
    return sum(x**2 for x in data)  # CPU-bound

with multiprocessing.Pool(processes=4) as pool:
    results = pool.map(cpu_heavy_task, chunks_of_data)  # True parallelism

# I/O-bound work: threading or asyncio is fine (GIL released during I/O)
import concurrent.futures

def fetch_user(user_id):
    return requests.get(f"https://api.example.com/users/{user_id}")

with concurrent.futures.ThreadPoolExecutor(max_workers=20) as executor:
    futures = [executor.submit(fetch_user, uid) for uid in user_ids]
    results = [f.result() for f in futures]  # Fine for I/O-bound
```

## 4. Code Examples

### Measuring context switch overhead

```python
import time
import threading

def count_context_switches(n_threads: int, work_per_thread: int):
    """Measure impact of thread count on throughput."""
    results = []

    def worker():
        total = 0
        for i in range(work_per_thread):
            total += i   # CPU work
        results.append(total)

    start = time.perf_counter()
    threads = [threading.Thread(target=worker) for _ in range(n_threads)]
    for t in threads: t.start()
    for t in threads: t.join()
    elapsed = time.perf_counter() - start

    print(f"Threads: {n_threads:4d}, Time: {elapsed:.3f}s, "
          f"Throughput: {n_threads * work_per_thread / elapsed:.0f} ops/s")

# Python GIL means threads don't run in parallel for CPU work
count_context_switches(1,   10_000_000)   # baseline
count_context_switches(4,   10_000_000)   # NOT 4x faster due to GIL
count_context_switches(16,  10_000_000)   # may be slower (context switch overhead)
```

## 5. Senior Engineer Perspective

**What a junior gets wrong:**

Assuming that adding more threads always means more throughput. It doesn't. For I/O-bound workloads, more threads help until the context switch overhead dominates. For CPU-bound workloads in Python, more threads help zero because of the GIL.

Also: not understanding why Python web apps use `multiprocessing` workers (Gunicorn's `--workers` flag). Each Gunicorn worker is a separate process. 4 workers = 4 CPU cores fully utilized. Without this, a single-process Python server uses only one core regardless of CPU count.

**The one thing that changes how you configure servers:**

Know whether your application is **CPU-bound** or **I/O-bound** before choosing your concurrency model.

- I/O-bound (DB queries, HTTP calls, file reads): async I/O or thread pool — one thread handles thousands of connections
- CPU-bound (image resizing, ML inference, crypto): multiprocessing — one process per CPU core

Getting this wrong means leaving 90% of your server's capacity unused.
