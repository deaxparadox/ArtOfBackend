## 1. What Is It? (Theory)

CAP theorem, proved by Eric Brewer in 2000 and formalized by Gilbert and Lynch in 2002, states:

> **A distributed data store can guarantee at most 2 of these 3 properties simultaneously:**
> - **C**onsistency
> - **A**vailability
> - **P**artition Tolerance

This is one of the most cited theorems in distributed systems — and one of the most misunderstood.

**Definitions:**

**Consistency (C):** Every read returns the most recent write (or an error). All nodes see the same data at the same time. No stale reads.

**Availability (A):** Every request receives a response (not an error). The system is always operational. It may return stale data, but it responds.

**Partition Tolerance (P):** The system continues operating even when network partitions occur — when some nodes cannot communicate with others.

**The key insight: P is not optional.**

Network partitions *will* happen in any real distributed system. Routers fail. Cables cut. Cloud zones lose connectivity. If your system spans more than one machine, you must tolerate partitions. Therefore:

> **In practice, you are always choosing between C and A when a partition occurs.**

CA (not partition tolerant) only makes sense for single-node databases. Once you have replication or multiple nodes, you must handle partitions — so the real choice is always CP vs AP.

## 2. How It Actually Works

### What Happens During a Partition

```
NORMAL OPERATION (no partition):
  Client → [Node A] ←──synced──→ [Node B]
  Write to A → A syncs to B → both have latest data
  Read from B → returns latest ✓

DURING A PARTITION (A and B can't communicate):
  Client → [Node A]    ✗    [Node B]
  A and B are disconnected.

  A receives a write. Cannot sync to B.
  NOW YOU MUST CHOOSE:

  CHOICE 1 — CP (Consistency over Availability):
    A and B refuse requests until partition heals.
    "I will not answer. I might give you stale or inconsistent data."
    → Consistent but not available.

  CHOICE 2 — AP (Availability over Consistency):
    A and B both accept reads and writes independently.
    When partition heals, they reconcile (may have conflicting writes).
    "I will answer, but I might be stale."
    → Available but not (strongly) consistent.
```

---

### Real Databases Mapped to CAP

| Database | CAP Choice | Why |
|---|---|---|
| PostgreSQL (single node) | CA | No partition — single machine. Not truly distributed. |
| PostgreSQL (with replication) | CP | Synchronous replication: wait for replica ack or fail. Async replication: AP-ish. |
| MongoDB (majority writes) | CP | Majority write concern = must confirm with quorum. |
| CouchDB | AP | Always available, eventually consistent. |
| Cassandra | AP | Highly available, tunable consistency (default: eventual). |
| DynamoDB | AP (default) | Eventually consistent reads by default; strongly consistent at extra cost. |
| Redis (single node) | CA | Single node. |
| Redis Cluster | AP | Available with eventual consistency during partitions. |
| HBase | CP | Uses ZooKeeper for coordination — halts on partition. |
| Etcd / ZooKeeper | CP | Consensus-based, refuses writes without quorum. |
| Spanner (Google) | CP | Globally consistent using TrueTime API. |

### PACELC — The Extended Model

CAP only describes behavior during a partition. Eric Brewer later acknowledged the model has a gap: **what about when there's no partition?** Even without a partition, there's a latency vs. consistency trade-off.

**PACELC:** (Partition → else Latency vs. Consistency)

```
If Partition → choose between Availability and Consistency (same as CAP)
Else (normal operation) → choose between Latency and Consistency

PACELC ratings:
  DynamoDB:   PA/EL  (partition: available; normal: low latency, eventual consistency)
  Cassandra:  PA/EL  (same — tunable but defaults to speed)
  Spanner:    PC/EC  (partition: consistent; normal: consistent but higher latency)
  CRDB:       PC/EC  (CockroachDB — globally consistent)
  MongoDB:    PC/EC  (with majority write concern)
```

---

### Consistency Models (A Spectrum)

CAP's "C" is linearizability (strongest). But consistency is a spectrum:

```
STRONG ←──────────────────────────────────────────────────► WEAK

LINEARIZABILITY       SEQUENTIAL       CAUSAL       EVENTUAL
(Strongest)           CONSISTENCY    CONSISTENCY   CONSISTENCY
                                                    (Weakest)

What it means:
  Linearizability:  Every read returns the latest write. Operations appear
                    instantaneous and globally ordered.
                    Cost: high latency (must coordinate globally)
                    Example: Spanner, etcd

  Sequential:       All operations appear to occur in some global order.
                    Each process sees operations in the order it executed them.
                    Example: single-leader replication with sync followers

  Causal:           Causally related operations are seen in order.
                    "You posted a reply → others see the post before the reply"
                    Example: Facebook Timeline

  Read-your-writes: After a write, your own reads see that write.
                    Others may not see it yet.
                    Example: Sticky sessions on a read replica

  Eventual:         If no new updates, all replicas eventually converge.
                    Reads may return stale data.
                    Example: Cassandra, DynamoDB default
```

## 3. Real-World Context

### How Cassandra chose AP

Cassandra was built by Facebook to power their Inbox search — a feature requiring massive write throughput and always-on availability over strict consistency. When you lose a message in your inbox, it's annoying. When the inbox service is down, everyone notices.

Cassandra uses **eventual consistency**: writes go to any available node, replicas sync asynchronously. If two nodes have conflicting values (due to a partition), the one with the highest timestamp wins (last-write-wins). This can lose writes, but the service stays available.

Tunable consistency: Cassandra lets you choose per-query: `QUORUM` (majority must respond), `ONE` (any one replica), `ALL` (all replicas). Higher consistency = higher latency = lower availability.

### How Google Spanner chose CP

Spanner is globally distributed across multiple continents but provides linearizable consistency. How? By using GPS clocks and atomic clocks in data centers to implement **TrueTime** — a global clock with bounded uncertainty. Spanner can determine the order of events globally by waiting for the uncertainty window to pass.

Cost: ~5–10ms extra latency per transaction (waiting for clock uncertainty). For Google Ads, the financial integrity is worth it.

### The bank account example

For a bank:

```
CP is the only acceptable choice.
  If my account says $1000 and I withdraw $900, then
  simultaneously my partner withdraws $900 from another ATM:
  
  AP system: both ATMs see $1000, both approve. Bank loses $800.
  CP system: one ATM sees the other's withdrawal first.
             Second withdrawal fails.

No bank can tolerate AP for balance operations.
```

For a social media "like" count:

```
AP is perfectly acceptable.
  You click Like. Your friend sees 1,234 likes.
  You see 1,235 likes.
  
  Neither of you is hurt. Eventual consistency is fine.
  AP gives you higher availability and lower latency.
```

The CAP choice depends entirely on what "correctness" means for your data.

## 4. Code Examples

### Tuning consistency in Cassandra

```python
from cassandra.cluster import Cluster
from cassandra.policies import ConsistencyLevel

cluster = Cluster(["node1", "node2", "node3"])
session = cluster.connect("mydb")

# HIGH CONSISTENCY: must hear from majority of replicas
# Slower, but you won't read stale data
session.default_consistency_level = ConsistencyLevel.QUORUM

result = session.execute("SELECT balance FROM accounts WHERE id = %s", [user_id])

# HIGH AVAILABILITY: respond from any single replica
# Faster, but may be stale
session.default_consistency_level = ConsistencyLevel.ONE
```

### DynamoDB: eventual vs. strongly consistent reads

```python
import boto3

dynamodb = boto3.resource("dynamodb")
table = dynamodb.Table("BankAccounts")

# Eventually consistent (default): cheaper, may be stale
response = table.get_item(Key={"user_id": "123"})

# Strongly consistent: guaranteed latest, costs 2x read units
response = table.get_item(
    Key={"user_id": "123"},
    ConsistentRead=True  # CP behavior
)

# For bank balances: always use ConsistentRead=True
# For user profile reads: eventually consistent is fine and cheaper
```

## 5. Trade-offs Summary

| Choice | Consistency | Availability | Latency | Use When |
|---|---|---|---|---|
| CP | Strong | Reduced during partition | Higher (coordination) | Money, inventory, auth tokens |
| AP | Eventual | Always available | Lower (no coordination) | Social feeds, view counts, recommendations |
| CA | Strong | High (no partitions) | Low (single node) | Development, small datasets, single-region |

## 6. Senior Engineer Perspective

**What a junior gets wrong:**

Treating CAP as "choose 2 of 3" at system design time and never thinking about it again. CAP applies **per-operation**, not per-system. Your database might be CP for write operations but AP for reads. DynamoDB is AP by default but CP when you set `ConsistentRead=True`. Good engineers tune consistency per use case.

Also: confusing availability (CAP) with uptime. CAP availability means "every request gets a response" — not "the service is up 99.99% of the time." A CP system can have 99.999% uptime and still be "not available" in the CAP sense during the rare seconds of a partition.

**What a senior actually asks:**

For every piece of data: "What is the cost of reading stale data vs. the cost of being unavailable?"

- **Financial data:** stale = catastrophic, unavailable = annoying → CP
- **Session tokens:** stale = security hole, unavailable = logged-out users → CP
- **User avatars:** stale = shows old photo, unavailable = broken profile → AP
- **Social feed:** stale = slightly old posts, unavailable = blank screen → AP

This cost analysis is the actual work of system design. CAP gives you the vocabulary.
