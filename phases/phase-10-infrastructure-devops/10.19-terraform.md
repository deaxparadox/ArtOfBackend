## 1. Terraform (10.19): Declarative Infrastructure

Terraform (HashiCorp, 2014) is the most widely used IaC tool. You describe **what** infrastructure you want; Terraform figures out **how** to create or update it to match your description.

### The Core Concepts

```
PROVIDERS:   Plugins that know how to talk to a specific cloud/service
             (AWS, GCP, Azure, Cloudflare, GitHub, Datadog...)

RESOURCES:   Individual infrastructure components
             (aws_instance, aws_rds_cluster, aws_s3_bucket...)

STATE:       Terraform's record of what currently exists
             Stored in terraform.tfstate (local or remote)
             Used to calculate what needs to change

PLAN:        A preview of changes before applying them
             "Will create 3 resources, modify 1, destroy 0"

APPLY:       Actually make the changes
```

### A Real Terraform Configuration

```hcl
# main.tf

terraform {
  required_version = ">= 1.6.0"
  
  required_providers {
    aws = {
      source  = "hashicorp/aws"
      version = "~> 5.0"  # allow 5.x updates but not 6.x
    }
  }
  
  # Store state in S3 (required for teams — never use local state)
  backend "s3" {
    bucket         = "mycompany-terraform-state"
    key            = "production/api.tfstate"
    region         = "us-east-1"
    dynamodb_table = "terraform-state-lock"  # prevents concurrent applies
    encrypt        = true
  }
}

provider "aws" {
  region = "us-east-1"
}

# ─── VARIABLES ─────────────────────────────────────────────────
variable "environment" {
  description = "Environment name (production, staging)"
  type        = string
}

variable "db_password" {
  description = "PostgreSQL password"
  type        = string
  sensitive   = true  # won't appear in plan output
}

# ─── DATA SOURCES (read existing resources) ────────────────────
data "aws_vpc" "main" {
  id = "vpc-0abc123"  # reference existing VPC (not managed by this config)
}

data "aws_subnets" "private" {
  filter {
    name   = "vpc-id"
    values = [data.aws_vpc.main.id]
  }
  tags = {
    Tier = "private"
  }
}

# ─── SECURITY GROUP ─────────────────────────────────────────────
resource "aws_security_group" "rds" {
  name        = "${var.environment}-rds-sg"
  description = "PostgreSQL security group"
  vpc_id      = data.aws_vpc.main.id

  ingress {
    from_port       = 5432
    to_port         = 5432
    protocol        = "tcp"
    security_groups = [aws_security_group.app.id]  # reference another resource
  }

  tags = {
    Environment = var.environment
    ManagedBy   = "terraform"
  }
}

# ─── RDS POSTGRESQL ─────────────────────────────────────────────
resource "aws_db_instance" "main" {
  identifier     = "${var.environment}-postgres"
  engine         = "postgres"
  engine_version = "16.3"
  instance_class = "db.t3.medium"
  
  allocated_storage     = 100
  max_allocated_storage = 500  # auto-scaling storage up to 500GB
  storage_encrypted     = true
  
  db_name  = "myapp"
  username = "dbadmin"
  password = var.db_password  # from variables (stored in AWS Secrets Manager or TF Cloud)
  
  vpc_security_group_ids = [aws_security_group.rds.id]
  db_subnet_group_name   = aws_db_subnet_group.main.name
  
  backup_retention_period = 7
  backup_window          = "03:00-04:00"    # 3-4am UTC
  maintenance_window     = "Mon:04:00-Mon:05:00"
  
  deletion_protection = true  # CRITICAL: prevents accidental terraform destroy
  
  # Free snapshot before destroy (extra protection)
  final_snapshot_identifier = "${var.environment}-final-snapshot"
  skip_final_snapshot       = false
  
  tags = {
    Environment = var.environment
    ManagedBy   = "terraform"
  }
}

# ─── OUTPUTS (expose values for use by other configs) ───────────
output "rds_endpoint" {
  value       = aws_db_instance.main.endpoint
  description = "PostgreSQL connection endpoint"
}

output "rds_port" {
  value = aws_db_instance.main.port
}
```

### Terraform Workflow

```bash
# Initialize: download providers and configure backend
terraform init

# Preview changes without applying them
terraform plan

# Apply the changes (prompts for confirmation)
terraform apply

# Apply without prompts (for CI/CD)
terraform apply -auto-approve

# Plan and save to a file (review, then apply exactly that plan)
terraform plan -out=tfplan
terraform apply tfplan

# Destroy all resources (use with extreme caution!)
terraform destroy

# Target a specific resource (useful for debugging)
terraform plan -target=aws_db_instance.main
terraform apply -target=aws_db_instance.main

# Import existing infrastructure into Terraform state
terraform import aws_s3_bucket.existing-bucket my-existing-bucket

# Show the current state
terraform show
terraform state list  # list all resources in state
```

### Terraform State: The Most Important Concept

```
STATE FILE (terraform.tfstate):
  Terraform's memory of what it created.
  Maps your config to real cloud resource IDs.
  
  Without state: Terraform doesn't know what already exists.
  It would try to create everything again, causing duplicates or errors.
  
  With state: Terraform computes the diff:
    "You want: vpc, subnet, RDS"
    "I already created: vpc, subnet, RDS (IDs: xyz...)"
    "Nothing to do."

CRITICAL RULES:
  1. NEVER store state in git (contains sensitive values, causes conflicts)
  2. ALWAYS use remote state for teams (S3 + DynamoDB locking)
  3. NEVER manually edit the state file
  4. ALWAYS use state locking (prevents two people applying simultaneously)
```

## 2. Pulumi (10.20): IaC with Real Programming Languages

Pulumi uses actual programming languages (TypeScript, Python, Go) instead of HCL:

```typescript
// Pulumi TypeScript: same infrastructure as Terraform above
import * as aws from "@pulumi/aws";
import * as pulumi from "@pulumi/pulumi";

const config = new pulumi.Config();
const environment = config.require("environment");
const dbPassword = config.requireSecret("dbPassword");  // encrypted

const rdsSecurityGroup = new aws.ec2.SecurityGroup("rds-sg", {
    description: "PostgreSQL security group",
    vpcId: existingVpcId,
    ingress: [{
        fromPort: 5432,
        toPort: 5432,
        protocol: "tcp",
        securityGroups: [appSecurityGroup.id],
    }],
    tags: { Environment: environment, ManagedBy: "pulumi" },
});

const db = new aws.rds.Instance("main", {
    engine: "postgres",
    engineVersion: "16.3",
    instanceClass: aws.rds.InstanceType.T3_Medium,
    dbName: "myapp",
    username: "dbadmin",
    password: dbPassword,
    allocatedStorage: 100,
    deletionProtection: true,
    vpcSecurityGroupIds: [rdsSecurityGroup.id],
    tags: { Environment: environment },
});

// TypeScript gives you: loops, conditionals, functions, type safety
// Much more powerful than HCL for complex configurations

export const endpoint = db.endpoint;
```

**Terraform vs. Pulumi:**

```
TERRAFORM:
  ✓ Larger community, more documentation, more Terraform-specific tooling
  ✓ HCL is easy to read for non-programmers
  ✓ Industry standard — every ops team knows it
  ✗ Limited programming constructs (loops are awkward, functions are limited)
  ✗ HCL can become hard to maintain for complex infra

PULUMI:
  ✓ Full programming language (loops, functions, classes, type safety)
  ✓ Better for complex, dynamic infrastructure
  ✓ Share code with your application (same language)
  ✗ Smaller community
  ✗ State management is Pulumi SaaS (or manual setup)

RECOMMENDATION:
  Teams without strong programming preference → Terraform
  Teams with TypeScript/Python expertise → Pulumi is excellent
```

## 3. Ansible (10.21): Configuration Management

Ansible manages the configuration of existing servers — installing software, managing files, applying settings. It's complementary to Terraform: Terraform creates the infrastructure, Ansible configures it.

```yaml
# playbook.yml: configure a fresh Ubuntu server for our application
---
- name: Configure application servers
  hosts: app_servers
  become: true         # sudo
  vars:
    app_user: appuser
    app_dir: /opt/myapp
    python_version: "3.12"

  tasks:
  - name: Install system dependencies
    apt:
      name:
        - python{{ python_version }}
        - python{{ python_version }}-pip
        - nginx
        - supervisor
      state: present
      update_cache: yes

  - name: Create application user
    user:
      name: "{{ app_user }}"
      system: true
      shell: /bin/false
      home: "{{ app_dir }}"
      create_home: true

  - name: Copy application files
    copy:
      src: ../dist/
      dest: "{{ app_dir }}/current/"
      owner: "{{ app_user }}"
      group: "{{ app_user }}"

  - name: Configure Nginx
    template:
      src: templates/nginx.conf.j2
      dest: /etc/nginx/sites-available/myapp
    notify: Restart nginx      # trigger handler when this task changes

  - name: Enable Nginx site
    file:
      src: /etc/nginx/sites-available/myapp
      dest: /etc/nginx/sites-enabled/myapp
      state: link

  handlers:
  - name: Restart nginx
    service:
      name: nginx
      state: restarted
```

```bash
# Run against all hosts in inventory
ansible-playbook -i inventory/production playbook.yml

# Run against specific hosts only
ansible-playbook -i inventory/production playbook.yml --limit web_servers

# Dry-run (check mode)
ansible-playbook -i inventory/production playbook.yml --check

# Show what would change
ansible-playbook -i inventory/production playbook.yml --diff
```

**When Ansible vs. Docker:**

Ansible is excellent for configuring base servers before containers, for server maintenance (rotating certificates, updating system packages), and for environments where Docker isn't used. For containerized applications on Kubernetes, Docker handles most of what Ansible would traditionally do.

## 4. Senior Engineer Perspective

**The golden rule of IaC:**

If you can't reproduce your entire infrastructure from your IaC code and your secrets manager, you have a problem. Run `terraform destroy` on staging and `terraform apply` it back — it should work. If it doesn't, you have undocumented manual changes. Fix them.

**State is precious:**

A corrupted or lost Terraform state file is a production incident. Remote state in S3 with versioning enabled means you can restore previous state versions. State locking via DynamoDB prevents two engineers from applying simultaneously (which causes state corruption). Set both up before your first `terraform apply` in a shared environment.

**The module pattern:**

As infrastructure grows, Terraform configurations become repetitive. Use modules to encapsulate reusable patterns:

```hcl
# Reusable module: modules/rds/main.tf
module "production_db" {
  source      = "./modules/rds"
  environment = "production"
  db_password = var.db_password
  instance_class = "db.r6g.large"
}

module "staging_db" {
  source      = "./modules/rds"
  environment = "staging"
  db_password = var.staging_db_password
  instance_class = "db.t3.medium"
}
# Same logic, different parameters
```
