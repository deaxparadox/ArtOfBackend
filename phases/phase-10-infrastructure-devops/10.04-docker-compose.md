## 1. What Is It? (Theory)

Docker Compose is a tool for defining and running multi-container applications. Instead of running multiple `docker run` commands with long argument lists, you define all your services, networks, and volumes in a single `docker-compose.yml` file and start everything with `docker compose up`.

**The problem it solves:**

A real application isn't just one container. It's a web server, a database, a cache, a message queue, maybe a background worker. Before Docker Compose, spinning up a full development environment meant:

```bash
# Without Compose: painful
docker run -d --name postgres -e POSTGRES_DB=myapp -e POSTGRES_USER=dev \
    -e POSTGRES_PASSWORD=devpass -p 5432:5432 postgres:16

docker run -d --name redis -p 6379:6379 redis:7-alpine

docker run -d --name api --link postgres --link redis \
    -e DATABASE_URL=postgresql://dev:devpass@postgres:5432/myapp \
    -e REDIS_URL=redis://redis:6379 \
    -p 8000:8000 myapp:latest

docker run -d --name worker --link postgres --link redis \
    -e DATABASE_URL=postgresql://dev:devpass@postgres:5432/myapp \
    -e REDIS_URL=redis://redis:6379 \
    myapp:latest worker

# Now stop everything... and hope you remember all the container names
docker stop postgres redis api worker
docker rm postgres redis api worker
```

With Compose, you write that once, then it's two commands for life: `docker compose up` and `docker compose down`.

**The real-world analogy:**

Docker Compose is like a recipe card for your kitchen. Instead of memorizing every ingredient, measurement, and step for a complex meal, you write it once. Next time: follow the recipe. Share the recipe with your team. Everyone makes the exact same meal.

## 2. How It Actually Works (Deep Mechanics)

### The docker-compose.yml File

```yaml
# docker-compose.yml — complete development environment for a FastAPI + PostgreSQL + Redis app

version: "3.9"  # optional in modern Compose, but good for documentation

services:
  # ─── API SERVICE ──────────────────────────────────────────────
  api:
    build:
      context: .           # build from Dockerfile in current directory
      dockerfile: Dockerfile
      target: runtime      # use the 'runtime' stage (multi-stage build)
    image: myapp:dev       # tag the built image
    
    ports:
      - "8000:8000"        # host_port:container_port
    
    environment:
      - DATABASE_URL=postgresql://dev:devpass@postgres:5432/myapp
      - REDIS_URL=redis://redis:6379
      - LOG_LEVEL=debug
      # Never put production secrets here! Use env_file or external secrets.
    
    volumes:
      - .:/app             # mount current directory into container (live reload!)
      # Changes to your source code are immediately visible inside the container
      # because you're mounting the actual directory, not a copy
    
    depends_on:
      postgres:
        condition: service_healthy  # wait until postgres health check passes
      redis:
        condition: service_healthy
    
    command: uvicorn main:app --host 0.0.0.0 --port 8000 --reload
    # --reload: auto-restart when code changes (dev only)
    
    restart: unless-stopped  # restart on crash (not on manual stop)
    
    networks:
      - backend

  # ─── BACKGROUND WORKER ────────────────────────────────────────
  worker:
    build:
      context: .
      target: runtime
    
    environment:
      - DATABASE_URL=postgresql://dev:devpass@postgres:5432/myapp
      - REDIS_URL=redis://redis:6379
    
    volumes:
      - .:/app
    
    depends_on:
      - api      # wait for API to be up (not health check, just started)
    
    command: celery -A app.worker worker --loglevel=info
    
    networks:
      - backend

  # ─── POSTGRESQL DATABASE ───────────────────────────────────────
  postgres:
    image: postgres:16-alpine   # use specific version, not latest
    
    environment:
      POSTGRES_DB: myapp
      POSTGRES_USER: dev
      POSTGRES_PASSWORD: devpass
    
    ports:
      - "5432:5432"       # expose to host so you can connect with pgAdmin/psql
    
    volumes:
      - postgres_data:/var/lib/postgresql/data   # named volume: data persists
      - ./init.sql:/docker-entrypoint-initdb.d/init.sql  # run on first start
    
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U dev -d myapp"]
      interval: 5s
      timeout: 5s
      retries: 5
      start_period: 10s
    
    networks:
      - backend

  # ─── REDIS CACHE ──────────────────────────────────────────────
  redis:
    image: redis:7-alpine
    
    ports:
      - "6379:6379"
    
    command: redis-server --appendonly yes  # enable AOF persistence
    
    volumes:
      - redis_data:/data
    
    healthcheck:
      test: ["CMD", "redis-cli", "ping"]
      interval: 5s
      retries: 3
    
    networks:
      - backend

# ─── VOLUMES ────────────────────────────────────────────────────
# Named volumes: managed by Docker, persist across container restarts
# (unlike bind mounts, Docker controls where these live on the host)
volumes:
  postgres_data:
  redis_data:

# ─── NETWORKS ───────────────────────────────────────────────────
# All services in the same network can reach each other by service name
# (postgres is reachable at hostname "postgres", redis at "redis")
networks:
  backend:
    driver: bridge
```

### The Essential Commands

```bash
# Start all services (builds images if needed)
docker compose up

# Start detached (background)
docker compose up -d

# Build or rebuild images before starting
docker compose up --build

# Start only specific services
docker compose up api postgres

# Stop everything (preserves volumes)
docker compose down

# Stop everything AND delete volumes (clean slate)
docker compose down -v

# View logs
docker compose logs              # all services
docker compose logs api          # specific service
docker compose logs -f api       # follow (stream) logs

# Execute a command in a running service
docker compose exec api bash            # open a shell
docker compose exec api pytest tests/   # run tests
docker compose exec postgres psql -U dev -d myapp  # database shell

# Check status
docker compose ps

# Scale a service (run multiple instances)
docker compose up --scale worker=3

# Restart a specific service
docker compose restart api
```

## 3. Development Patterns

### Live Code Reload with Volume Mounts

The most powerful development pattern: mount your source code into the container so changes are reflected immediately without rebuilding the image.

```yaml
services:
  api:
    volumes:
      - .:/app                        # mount entire project
      - /app/node_modules             # EXCEPTION: don't mount node_modules from host
                                      # (Mac/Windows host fs is slow for node_modules)
    command: uvicorn main:app --reload  # restart on file changes
```

This gives you the isolation of Docker with the developer experience of running locally.

### Override Files for Different Environments

```yaml
# docker-compose.override.yml (automatically applied when docker-compose.yml exists)
# Use for development-specific settings

services:
  api:
    environment:
      - DEBUG=true
      - LOG_LEVEL=debug
    volumes:
      - .:/app  # dev: live code mount
    # In production: no volume mount, code is baked into the image
```

```yaml
# docker-compose.prod.yml (explicitly applied for production smoke testing)
# docker compose -f docker-compose.yml -f docker-compose.prod.yml up

services:
  api:
    environment:
      - DEBUG=false
      - LOG_LEVEL=info
    deploy:
      resources:
        limits:
          memory: 512m
```

### Testing Services in Isolation

```yaml
# docker-compose.test.yml — for CI/CD testing
services:
  api:
    build: .
    environment:
      - DATABASE_URL=postgresql://test:test@postgres-test:5432/testdb
    depends_on:
      postgres-test:
        condition: service_healthy
    command: pytest tests/ -v

  postgres-test:
    image: postgres:16-alpine
    environment:
      POSTGRES_DB: testdb
      POSTGRES_USER: test
      POSTGRES_PASSWORD: test
    tmpfs:
      - /var/lib/postgresql/data  # use RAM disk for tests (fastest, auto-cleaned)
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U test -d testdb"]
      interval: 3s
      retries: 5
```

```bash
# Run tests in CI
docker compose -f docker-compose.test.yml run --rm api
```

## 4. Real-World Context

### The "It Works on My Machine" Killer

The most common use of Docker Compose at companies is onboarding. New engineer joins → checks out the repo → `docker compose up` → the entire development environment starts. No "install PostgreSQL 14 (not 15!) and set these env vars and..." instructions. No hours configuring local services.

Shopify famously had a "dev" tool built on top of Docker Compose that let any engineer spin up any microservice's dependencies instantly. This eliminated an entire class of "works on my machine" issues.

### Compose in CI/CD

GitHub Actions example:

```yaml
# .github/workflows/test.yml
jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      
      - name: Run tests
        run: |
          docker compose -f docker-compose.test.yml run --rm api
        # This starts postgres, runs all tests, then tears everything down
        # Completely self-contained: no manual postgres setup needed in CI
```

## 5. Senior Engineer Perspective

**What a junior gets wrong:**

Using Docker Compose in production directly. Docker Compose is a development and testing tool. For production, use Kubernetes, ECS, or a managed container service. Compose doesn't handle:
- Rolling updates (zero-downtime deploys)
- Self-healing (restart policies are basic)
- Service discovery at scale
- Distributed deployments across multiple hosts
- Resource scheduling

**The volume mount for development:**

Mounting your source code (`- .:/app`) is one of the most useful development tricks. But it comes with a cost: on macOS and Windows, the host filesystem is significantly slower than Linux filesystem. `node_modules` (thousands of small files) is particularly painful. The fix: mount `node_modules` as a named Docker volume (`/app/node_modules`), not from the host.

**The `depends_on` vs. actual readiness:**

`depends_on: postgres` only waits for the postgres container to START — not for PostgreSQL to be READY to accept connections. PostgreSQL takes ~1-3 seconds to initialize. Without `condition: service_healthy`, your app may crash on startup trying to connect to a postgres that hasn't finished initializing. Always use `condition: service_healthy` paired with a proper `healthcheck`.
