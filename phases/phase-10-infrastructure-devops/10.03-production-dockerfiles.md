## 1. What Is It? (Theory)

Writing a Dockerfile that "works" is trivial. Writing a Dockerfile that is **production-grade** — small, fast to build, secure, reproducible, and observable — requires deliberate choices at every line.

This topic is a collection of the concrete patterns and decisions that separate a Dockerfile that ships code from a Dockerfile that ships code reliably, safely, and efficiently.

**The four qualities of a production Dockerfile:**

1. **Small image:** fewer bytes = faster pulls, less storage cost, smaller attack surface
2. **Fast rebuild:** layer cache hit rate determines CI speed — wrong layer order means rebuilding everything on every commit
3. **Secure:** non-root user, no secrets in image, no unnecessary tools
4. **Reproducible:** same build produces the same image regardless of when it runs

## 2. Language-Specific Production Patterns

### Python (FastAPI / Django)

```dockerfile
# ─── BUILDER STAGE ───────────────────────────────────────────────
FROM python:3.12-slim AS builder

# Install only what's needed to build Python C extensions
# Kept in builder stage, not in final image
RUN apt-get update && apt-get install -y --no-install-recommends \
    gcc \
    libpq-dev \
  && rm -rf /var/lib/apt/lists/*

WORKDIR /build

# Install Python dependencies
# Copy requirements first to maximize layer cache reuse
COPY requirements.txt .
RUN pip install --no-cache-dir --user -r requirements.txt
# --user: installs to /root/.local (easy to copy to other stage)
# --no-cache-dir: don't cache pip downloads (smaller layer)


# ─── RUNTIME STAGE ───────────────────────────────────────────────
FROM python:3.12-slim AS runtime

# Install only runtime system libs (not build tools)
RUN apt-get update && apt-get install -y --no-install-recommends \
    libpq5 \    # PostgreSQL client library (runtime only, not development headers)
    curl \      # for health check only
  && rm -rf /var/lib/apt/lists/* \
  && apt-get clean

# Create non-root user with explicit UID/GID (reproducible, predictable)
RUN groupadd --gid 1001 appgroup && \
    useradd --uid 1001 --gid 1001 --no-create-home --shell /bin/false appuser

# Copy Python packages from builder
COPY --from=builder --chown=appuser:appgroup /root/.local /home/appuser/.local

WORKDIR /app

# Copy application code (changes most frequently → at the end)
COPY --chown=appuser:appgroup . .

# Switch to non-root user AFTER all setup (chown during COPY is efficient)
USER appuser

ENV PATH=/home/appuser/.local/bin:$PATH
ENV PYTHONDONTWRITEBYTECODE=1
ENV PYTHONUNBUFFERED=1
ENV PYTHONFAULTHANDLER=1   # print Python traceback on segfault

EXPOSE 8000

HEALTHCHECK --interval=30s --timeout=5s --start-period=10s --retries=3 \
  CMD curl -f http://localhost:8000/health || exit 1

# Use exec form (not shell form) for CMD:
# exec form ["executable", "arg"]: process is PID 1, receives signals correctly
# shell form "executable arg": spawns /bin/sh which doesn't forward signals
CMD ["uvicorn", "main:app", \
     "--host", "0.0.0.0", \
     "--port", "8000", \
     "--workers", "1", \
     "--no-access-log"]
```

### Node.js (Express / NestJS)

```dockerfile
FROM node:20-alpine AS builder

WORKDIR /build

# Copy package files first (cache npm install as long as package*.json doesn't change)
COPY package.json package-lock.json ./

# ci = clean install (uses package-lock.json exactly, faster and more reproducible than npm install)
# --production=false: we need devDependencies to build TypeScript
RUN npm ci

COPY . .

# Compile TypeScript
RUN npm run build

# Prune devDependencies for production (removes TypeScript, test tools, etc.)
RUN npm prune --production


FROM node:20-alpine AS runtime

RUN addgroup -g 1001 appgroup && adduser -u 1001 -G appgroup -s /bin/sh -D appuser

WORKDIR /app

# Copy built files and production node_modules
COPY --from=builder --chown=appuser:appgroup /build/dist ./dist
COPY --from=builder --chown=appuser:appgroup /build/node_modules ./node_modules
COPY --from=builder --chown=appuser:appgroup /build/package.json .

USER appuser

EXPOSE 3000

CMD ["node", "dist/main.js"]
```

### Go (Gin / Echo / chi)

```dockerfile
FROM golang:1.22-alpine AS builder

# Install CA certificates (needed for HTTPS calls from the binary)
RUN apk add --no-cache ca-certificates

WORKDIR /build

# Copy go.mod and go.sum first (dependency layer cache)
COPY go.mod go.sum ./
RUN go mod download

# Copy and build (CGO_ENABLED=0: static binary, no C runtime dependency)
COPY . .
RUN CGO_ENABLED=0 GOOS=linux go build \
    -ldflags="-s -w" \    # -s: strip debug info, -w: strip DWARF tables → smaller binary
    -trimpath \           # remove absolute path info from binary (reproducible, security)
    -o /app/server ./cmd/server


# Final stage: FROM scratch — literally nothing
# This is the smallest possible base: zero OS, zero shell, zero attack surface
FROM scratch AS runtime

# Copy only what's needed
COPY --from=builder /etc/ssl/certs/ca-certificates.crt /etc/ssl/certs/  # for HTTPS
COPY --from=builder /app/server /server

EXPOSE 8080

# UID 1001: run as non-root (scratch has no useradd, but we can use numeric UID)
USER 1001:1001

ENTRYPOINT ["/server"]
```

**The Go scratch image:** Go compiles to a static binary that has no external dependencies. A `FROM scratch` image contains only your binary and nothing else. Total image size: ~10-15MB. Versus Python: 150-200MB minimum. This is one reason Go is the preferred language for building infrastructure tools.

## 3. Layer Ordering: The Cache Optimization

The order of Dockerfile instructions is a performance engineering decision. Wrong order = rebuilding everything on every commit. Correct order = millisecond rebuilds for code changes.

```
RULE: Put instructions that change RARELY near the top.
      Put instructions that change OFTEN near the bottom.

EXAMPLE: Python application

FREQUENTLY CHANGES:
  Your application code (src/, app/)
  Config files

RARELY CHANGES:
  System packages (apt-get install)
  Python runtime dependencies (pip install)
  Base image version

CORRECT ORDER:
  FROM python:3.12-slim          ← never changes during development
  RUN apt-get install ...        ← changes rarely (adding new system lib)
  COPY requirements.txt .        ← changes occasionally (adding new package)
  RUN pip install ...            ← rebuilds only when requirements.txt changes
  COPY . .                       ← changes on every commit
  CMD [...]

WRONG ORDER:
  FROM python:3.12-slim
  COPY . .                       ← changes on every commit → INVALIDATES cache for everything below
  RUN pip install -r requirements.txt  ← rebuilds every time! pip install is slow (1-5 min)
  CMD [...]
  
  With wrong order: every git commit triggers a full pip install.
  With correct order: pip install cache hits 99% of commits (only misses on requirements.txt changes).
```

## 4. BuildKit: Docker's Modern Build Engine

BuildKit (default since Docker 23.x) provides:
- **Parallel stage builds:** multiple FROM stages build in parallel
- **Persistent build cache:** cache across CI builds (not just local)
- **Secret mounting:** access secrets without baking them into layers

```dockerfile
# SECRET MOUNTING: access a secret during build without it appearing in image
# (e.g., private PyPI index, SSH key to clone a private repo)

# syntax=docker/dockerfile:1
FROM python:3.12-slim

# Mount a secret at build time — it's available only during this RUN command
# and is NOT stored in the image layer
RUN --mount=type=secret,id=pip_token \
    pip install \
    --extra-index-url https://$(cat /run/secrets/pip_token)@private.pypi.example.com \
    my-private-package

# Build with: docker build --secret id=pip_token,env=PIP_TOKEN .
```

```bash
# Mount SSH key to access private git repos during build
docker build --ssh default=$SSH_AUTH_SOCK .

# In Dockerfile:
# RUN --mount=type=ssh git clone git@github.com:myorg/private-repo.git
```

## 5. The Production Dockerfile Checklist

```
BEFORE SHIPPING:

IMAGE SIZE:
□ Multi-stage build used (if any compilation or build tools needed)
□ .dockerignore created and comprehensive
□ apt-get cleanup in same RUN command as install
□ --no-cache-dir for pip
□ docker history myimage:tag reviewed (no unexpectedly large layers)

SECURITY:
□ Non-root user created and USER instruction present
□ No secrets, tokens, or credentials in any layer (check with docker history)
□ Base image pinned to a specific version tag (not :latest)
□ No unnecessary tools (vim, curl, wget) in production image
□ HEALTHCHECK defined

REPRODUCIBILITY:
□ Base image uses digest or exact version tag
□ pip install uses pinned requirements.txt (not requirements without versions)
□ npm ci used (not npm install)
□ go mod download uses go.sum (not just go.mod)

SIGNALS:
□ CMD uses exec form ["cmd", "arg"] not shell form "cmd arg"
□ Application handles SIGTERM gracefully (see topic 2.10)
□ Container starts within HEALTHCHECK start-period

LAYER CACHE:
□ Dependencies copied and installed before application code
□ COPY . . is near the bottom of the Dockerfile
```

## 6. Senior Engineer Perspective

**The build time question no one asks:**

How long does your CI pipeline take? If the answer includes "5 minutes for the Docker build step," your layer ordering is probably wrong. Correct layer ordering means a typical commit causes only the last 1-2 layers to rebuild (COPY . . and CMD) — which should take seconds, not minutes.

**The image scanning requirement:**

Every production image should be scanned for known CVEs before deployment. This is not optional. The tools are free (Trivy, Grype) and trivially integrated into CI:

```bash
# In CI, after build:
trivy image --exit-code 1 --severity HIGH,CRITICAL myapp:$CI_COMMIT_SHA
# Fails the pipeline if HIGH or CRITICAL vulnerabilities are found
# This catches: OpenSSL vulnerabilities in the base image, compromised pip packages, etc.
```

**The distroless alternative:**

For maximum security, consider Google's distroless images (`gcr.io/distroless/python3`) — they contain only the language runtime and its dependencies, with no shell, no package manager, no coreutils. An attacker who gains code execution in a distroless container has nowhere to run.
