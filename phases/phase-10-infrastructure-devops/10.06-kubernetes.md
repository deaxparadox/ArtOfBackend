## 1. What Is Kubernetes? (10.6)

Kubernetes (K8s, from Google, 2014) is a container orchestration system — it manages where and how containers run across a cluster of machines.

Docker answers "how do I run this container on this machine?" Kubernetes answers "how do I run this container across a fleet of machines, keep it healthy, scale it, and update it without downtime?"

**What Kubernetes actually does:**

```
WITHOUT KUBERNETES:
  You have 5 servers. You manually:
    - Choose which server gets which container
    - Monitor: is the container still running on server 3?
    - Replace: if it crashed, restart it somewhere
    - Scale: add 3 more containers for the Black Friday spike
    - Update: roll out a new version without taking down all instances at once
    - Balance: your new code is faster, maybe server 3 can take more load now
    
  This is a full-time job. It doesn't scale to 500 containers.

WITH KUBERNETES:
  You declare: "I want 5 replicas of this container, with these resources, using this image"
  Kubernetes does:
    → Decides which node has capacity (scheduler)
    → Starts the containers (kubelet)
    → Monitors health (health checks) and restarts on failure
    → Rebalances if a node goes down
    → Scales up/down based on CPU/memory metrics (HPA)
    → Rolls out new versions gradually, rollback on failure
    
  You describe the desired state. Kubernetes makes reality match it.
```

**The real-world analogy:**

Kubernetes is like a city government for containers. The city says "we need hospitals running in every district." The government figures out which buildings to use, replaces hospitals that burn down, builds more if the population grows, and manages the roads (networking) that connect them. You don't manage individual buildings — you manage policies.

## 2. Kubernetes Architecture (10.7)

```
KUBERNETES CLUSTER:

CONTROL PLANE (the "brain" — usually 3 nodes for HA):
  ┌─────────────────────────────────────────────────┐
  │  API Server (kube-apiserver)                    │
  │    The single entry point for all K8s operations │
  │    All kubectl commands go here                  │
  │                                                 │
  │  etcd                                           │
  │    Distributed key-value store                  │
  │    Stores the entire cluster state              │
  │    "Source of truth" — if etcd is lost, the     │
  │    cluster state is lost                        │
  │                                                 │
  │  Scheduler (kube-scheduler)                     │
  │    Decides which Node a new Pod runs on          │
  │    Considers: resource requests, node capacity,  │
  │    affinity rules, taints/tolerations            │
  │                                                 │
  │  Controller Manager (kube-controller-manager)   │
  │    Runs "controllers" — control loops that watch │
  │    for state drift and correct it:              │
  │    ReplicaSet controller: "I want 5 pods,       │
  │    there are only 3, create 2 more"             │
  └─────────────────────────────────────────────────┘
          │ API calls
          ▼
WORKER NODES (where your containers actually run):
  ┌───────────────────────────────────┐
  │  kubelet                          │
  │    Agent on each node             │
  │    Reads pod spec from API server │
  │    Starts/stops containers        │
  │    Reports node and pod status    │
  │                                   │
  │  kube-proxy                       │
  │    Manages network rules (iptables/ipvs)
  │    Implements Service load balancing
  │                                   │
  │  Container Runtime (containerd)   │
  │    Actually runs containers       │
  │    (Docker Engine uses containerd)│
  │                                   │
  │  Pods (1 or more containers)      │
  │    Your actual application        │
  └───────────────────────────────────┘
```

## 3. Kubernetes Objects (10.8)

**Everything in Kubernetes is an object** — a desired state declaration stored in etcd. The control plane works to make reality match these declarations.

### Pod: The Smallest Deployable Unit

```yaml
# A Pod runs one or more containers in the same network namespace
# (containers in the same pod share localhost and volumes)
apiVersion: v1
kind: Pod
metadata:
  name: api-pod
  labels:
    app: api            # labels are key-value pairs used for selection
    version: v1.2.3
spec:
  containers:
  - name: api
    image: registry.example.com/api:v1.2.3
    ports:
    - containerPort: 8000
    
    resources:
      requests:               # minimum resources (used for scheduling)
        cpu: "100m"           # 0.1 CPU core
        memory: "128Mi"
      limits:                 # maximum resources (enforced by cgroups)
        cpu: "500m"           # 0.5 CPU core
        memory: "512Mi"
    
    env:
    - name: DATABASE_URL
      valueFrom:
        secretKeyRef:
          name: app-secrets
          key: database-url
    
    readinessProbe:
      httpGet:
        path: /ready
        port: 8000
      initialDelaySeconds: 5
      periodSeconds: 5
    
    livenessProbe:
      httpGet:
        path: /live
        port: 8000
      initialDelaySeconds: 10
      periodSeconds: 10
```

**Note:** You almost never create Pods directly. You create Deployments that manage Pods automatically.

### Deployment: Managing Multiple Pod Replicas

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: api
  namespace: production
spec:
  replicas: 5           # run 5 instances of this Pod

  selector:
    matchLabels:
      app: api          # this deployment manages pods with label app=api

  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxSurge: 1          # allow 1 extra pod during update (6 pods temporarily)
      maxUnavailable: 0    # never reduce below 5 pods (zero downtime)

  template:               # the pod template (everything after here is like a Pod spec)
    metadata:
      labels:
        app: api
    spec:
      terminationGracePeriodSeconds: 60
      containers:
      - name: api
        image: registry.example.com/api:v1.2.3
        ...
```

### Service: Stable Network Access to Pods

Pods are ephemeral — they get new IPs when restarted. A Service provides a stable IP and DNS name that load-balances across matching pods.

```yaml
apiVersion: v1
kind: Service
metadata:
  name: api-svc
spec:
  selector:
    app: api          # routes to pods with this label
  ports:
  - port: 80          # service port
    targetPort: 8000  # container port
  type: ClusterIP     # internal only (default)
  # type: LoadBalancer  → creates cloud LB with external IP
  # type: NodePort    → exposes on each node's IP
```

```
SERVICE TYPES:
  ClusterIP:      Only accessible within the cluster
                  Use for: internal service-to-service communication
  
  NodePort:       Accessible via <NodeIP>:<NodePort> from outside
                  Use for: development, testing (not production)
  
  LoadBalancer:   Cloud provider creates a load balancer with external IP
                  Use for: production traffic entry point
  
  ExternalName:   DNS alias for an external service
                  Use for: abstracting external services (database.example.com)
```

### ConfigMap and Secret: Configuration Management

```yaml
# ConfigMap: non-sensitive configuration
apiVersion: v1
kind: ConfigMap
metadata:
  name: api-config
data:
  LOG_LEVEL: "info"
  REDIS_URL: "redis://redis-svc:6379"
  MAX_WORKERS: "4"

---
# Secret: sensitive data (base64 encoded, not encrypted by default!)
# For real encryption: use Sealed Secrets, AWS Secrets Manager, or Vault
apiVersion: v1
kind: Secret
metadata:
  name: app-secrets
type: Opaque
data:
  database-url: cG9zdGdyZXNxbDovL3VzZXI6cGFzc0Bob3N0L2Ri  # base64 encoded
  jwt-secret: bXlzdXBlcnNlY3JldGtleQ==
```

```yaml
# Using ConfigMap and Secret in a Deployment
spec:
  containers:
  - name: api
    envFrom:
    - configMapRef:
        name: api-config     # inject all ConfigMap keys as env vars
    env:
    - name: DATABASE_URL
      valueFrom:
        secretKeyRef:
          name: app-secrets
          key: database-url  # inject specific secret key
```

## 4. Kubernetes Networking (10.9)

```
KUBERNETES NETWORKING MODEL:
  Every Pod gets a unique IP address.
  Pods can communicate with any other Pod directly (no NAT within the cluster).
  Services provide stable IPs and DNS for groups of pods.

DNS RESOLUTION IN KUBERNETES:
  Service name:   api-svc
  Within same namespace: api-svc → resolves to Service ClusterIP
  Cross-namespace:       api-svc.production.svc.cluster.local
  
  Example: your payment service connecting to the database:
  DATABASE_URL = postgresql://postgres-svc.production.svc.cluster.local:5432/mydb
```

### Ingress: HTTP Routing at the Cluster Edge

```yaml
# Ingress: HTTP(S) routing from outside the cluster to internal services
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: api-ingress
  annotations:
    nginx.ingress.kubernetes.io/rewrite-target: /
    cert-manager.io/cluster-issuer: letsencrypt-prod  # auto-provision TLS cert
spec:
  tls:
  - hosts:
    - api.example.com
    secretName: api-tls-cert  # cert-manager writes the cert here
  
  rules:
  - host: api.example.com
    http:
      paths:
      - path: /api/v1
        pathType: Prefix
        backend:
          service:
            name: api-svc
            port:
              number: 80
      - path: /admin
        pathType: Prefix
        backend:
          service:
            name: admin-svc
            port:
              number: 80
```

### NetworkPolicy: Firewall Rules for Pods

```yaml
# Restrict which pods can talk to each other (zero-trust networking)
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: api-network-policy
spec:
  podSelector:
    matchLabels:
      app: api          # this policy applies to pods with app=api label

  policyTypes:
  - Ingress
  - Egress

  ingress:
  - from:
    - podSelector:
        matchLabels:
          app: nginx    # only nginx ingress controller can reach api pods
    ports:
    - port: 8000

  egress:
  - to:
    - podSelector:
        matchLabels:
          app: postgres  # api pods can only reach postgres pods
    ports:
    - port: 5432
  - to:
    - podSelector:
        matchLabels:
          app: redis
    ports:
    - port: 6379
  # api pods cannot reach the internet or other internal services by default
```

## 5. Helm: The Kubernetes Package Manager (10.10)

Helm lets you package Kubernetes YAML files into reusable "charts" with configurable values. Instead of maintaining separate YAML for dev/staging/prod with small differences, you maintain one chart with `values.yaml` files per environment.

```yaml
# values.yaml (defaults)
replicaCount: 3
image:
  repository: registry.example.com/api
  tag: latest  # override in CI with specific SHA
  pullPolicy: IfNotPresent

service:
  port: 80

ingress:
  enabled: true
  host: api.example.com

resources:
  requests:
    cpu: 100m
    memory: 128Mi
  limits:
    cpu: 500m
    memory: 512Mi

env:
  LOG_LEVEL: info
  MAX_WORKERS: "4"
```

```yaml
# values-production.yaml (production overrides)
replicaCount: 10
resources:
  requests:
    cpu: 500m
    memory: 512Mi
  limits:
    cpu: 2000m
    memory: 2Gi
env:
  LOG_LEVEL: warn
```

```bash
# Install or upgrade a release
helm upgrade --install myapp ./helm/myapp \
    --namespace production \
    --values ./helm/myapp/values-production.yaml \
    --set image.tag=$CI_COMMIT_SHA \   # override image tag from CI
    --wait \                            # wait for pods to be ready
    --timeout 5m                        # fail if not ready in 5 minutes

# Rollback to previous release
helm rollback myapp 1

# List releases
helm list --namespace production

# See rendered YAML (dry-run, for debugging)
helm template myapp ./helm/myapp --values values-production.yaml
```

## 6. Real-World Context

### Why Kubernetes Became the Standard

Before Kubernetes, companies built their own container orchestration systems. Google had Borg (internal, not open-source). Twitter had Mesos. Amazon had ECS. Kubernetes unified the industry around a single open standard, allowing tooling, skills, and best practices to be shared.

Today, a Kubernetes engineer at a startup can switch to a Fortune 500 company and deploy on Day 1 — the concepts are identical even if the scale differs.

### The Learning Curve Is Real

Kubernetes is complex. It was designed for Google-scale operations and carries that complexity with it. A new team deploying to Kubernetes for the first time typically spends 2-4 weeks just understanding the abstractions before becoming productive.

The alternative for small teams: use managed container services (AWS App Runner, Google Cloud Run, Fly.io) which handle orchestration for you. Move to Kubernetes when you need the control it provides — multi-region, custom networking, stateful workloads.

## 7. Senior Engineer Perspective

**The one thing that makes Kubernetes work in production:**

Understanding the reconciliation loop. Kubernetes doesn't "deploy" — it constantly reconciles actual state with desired state. If you want 5 replicas and there are 3: K8s creates 2 more. If there are 7 (a bug): K8s deletes 2. This continues forever. You manage the desired state (YAML in Git); Kubernetes manages the actual state.

This means: **GitOps is the natural operating model for Kubernetes.** Store your K8s manifests in Git. Merge changes to deploy. Rollback by reverting a commit. ArgoCD or Flux watch the git repo and apply changes automatically — this is the production standard.

**Resource requests are the most underappreciated setting:**

If you don't set `requests.cpu` and `requests.memory`, the Kubernetes scheduler has no information to make good placement decisions. Pods get scheduled onto already-full nodes, causing OOM kills and CPU throttling. Always set both requests and limits. The requests control scheduling; the limits control runtime behavior.
