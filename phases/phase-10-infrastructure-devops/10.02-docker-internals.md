## 1. What Is It? (Theory)

Docker containers are not virtual machines. A VM has a complete hypervisor, a guest OS, and emulated hardware — heavy and slow. A Docker container is a **process on the host OS** that thinks it's isolated. The isolation is real, but it's achieved through Linux kernel features, not through a separate OS instance.

Understanding this distinction has practical consequences:
- Why containers start in milliseconds (no OS boot)
- Why a container bug can crash the host kernel (they share a kernel)
- Why resource limits work the way they do
- Why Docker doesn't actually work on Windows/macOS without a Linux VM underneath

**The real-world analogy:**

A container is like a strict private office in an open-plan workspace. You have your own desk, your own locked cabinets, your own computer — you can't see your neighbors' work. But you're still breathing the same air, using the same building's electricity, and if the building's HVAC system breaks, everyone is affected.

A virtual machine is like renting a separate building. Complete isolation. But much more expensive and slower to set up.

## 2. How It Actually Works: The Three Linux Primitives

Docker is built on three Linux kernel features that have existed for years: **namespaces**, **cgroups**, and **union filesystems**. Docker is essentially a user-friendly wrapper around these three concepts.

---

### Linux Namespaces: The Illusion of Isolation

A namespace wraps a system resource so that a process sees only a private copy of it, while other processes see a different copy. Linux has seven namespace types:

```
PID NAMESPACE:
  Container's init process has PID 1 (inside the namespace).
  On the host, it might be PID 5432.
  Container can't see host processes (or other containers' processes).
  
  $ docker exec container ps aux
  PID  COMMAND
  1    uvicorn main:app   ← looks like it's the only process
  
  $ ps aux | grep uvicorn (on host)
  5432  uvicorn main:app  ← actually PID 5432

NETWORK NAMESPACE:
  Container has its own virtual network interface, IP address, routing table.
  Container 1: eth0 = 172.17.0.2
  Container 2: eth0 = 172.17.0.3
  Host: docker0 bridge connects them
  
  From inside container: can't see host network interfaces
  From host: can see all container interfaces

MOUNT NAMESPACE:
  Container has its own view of the filesystem.
  Container sees /usr/lib/python3.12, /app, /tmp — its own isolated tree.
  Changes to the filesystem inside the container don't affect the host.
  
  Exception: volumes explicitly mounted with -v

USER NAMESPACE:
  Container user 0 (root) maps to an unprivileged host user.
  "Root" inside the container is not root on the host.
  Essential for rootless Docker.

IPC NAMESPACE:
  Container has its own shared memory, message queues, semaphores.
  Inter-process communication is isolated between containers.

UTS NAMESPACE:
  Container has its own hostname.
  $ hostname inside container = container name
  $ hostname on host = real hostname

CGROUP NAMESPACE:
  Container sees only its own cgroup hierarchy.
  Can't see other containers' resource usage.
```

---

### cgroups (Control Groups): Resource Limits

While namespaces control what a process can *see*, cgroups control what a process can *use*. They provide resource limiting, prioritization, and accounting.

```
MEMORY CGROUP:
  --memory=512m → container can use max 512MB RAM
  --memory-swap=512m → same limit for swap (no additional swap)
  
  If container exceeds limit: OOM killer fires → process killed.
  This is the mechanism behind Kubernetes resource limits.

CPU CGROUP:
  --cpus=0.5 → container can use at most 50% of one CPU core
  --cpu-shares=1024 → relative weight for CPU time when system is busy
  
  Kubernetes: requests.cpu=100m (0.1 core) limits.cpu=500m (0.5 core)

BLOCK I/O CGROUP:
  --device-read-bps=/dev/sda:10mb → limit disk read to 10MB/s
  
  Prevents one noisy container from saturating the disk for all others.

NETWORK BANDWIDTH:
  Docker itself doesn't limit network bandwidth.
  Use Kubernetes NetworkPolicy + traffic shaping for this.
```

```bash
# See cgroup limits for a running container
docker inspect mycontainer | jq '.[0].HostConfig.Memory'
# 536870912 (512MB in bytes)

# See actual resource usage
docker stats mycontainer
# CONTAINER  CPU %  MEM USAGE / LIMIT   MEM %   NET I/O
# mycontainer  2.3%  156MiB / 512MiB    30.5%   1.2MB / 3.4MB

# On the host, see cgroup hierarchy
cat /sys/fs/cgroup/memory/docker/<container_id>/memory.limit_in_bytes
# 536870912
```

### Union Filesystems (OverlayFS): The Layer Cake

Docker's image layers are implemented using OverlayFS (or other union filesystem implementations like AUFS, devicemapper). The concept: multiple directory layers "stacked" together, with higher layers overriding lower ones.

```
HOW OVERLAYFS WORKS:

Lower layers (read-only image layers):
  Layer 1 (python:3.12-slim base): /usr/lib/python3.12, /bin, /etc, ...
  Layer 2 (WORKDIR /app):          /app/ (empty)
  Layer 3 (pip install):           /app/.local/lib/python3.12/...
  Layer 4 (COPY . .):              /app/main.py, /app/requirements.txt

Upper layer (writable container layer):
  Every change a running container makes goes here.
  Write to /app/data.txt → goes to upper layer, not lower layers.
  Lower layers remain unchanged.

What the container sees (merged view):
  /usr/lib/python3.12   ← from layer 1
  /app/main.py          ← from layer 4
  /app/data.txt         ← from upper layer (runtime write)

COPY-ON-WRITE:
  Container reads /etc/hosts → reads from lower layer (fast)
  Container writes /etc/hosts → OverlayFS copies file to upper layer, then writes
  Now both copies exist: original in lower (for other containers to read), 
  modified in upper (for this container's writes)

BENEFIT:
  1,000 containers from the same image:
  - Lower layers: shared once on disk (e.g., 500MB python:3.12-slim = one copy)
  - Upper layers: one per container, but only containing actual changes
  - Total disk usage: ~500MB (shared) + N × (container-specific changes)
  - NOT: 1,000 × 500MB = 500GB
```

### Container Networking: How Containers Talk

```
DEFAULT BRIDGE NETWORK (docker0):

Host machine:
  eth0: 192.168.1.100 (connects to your router/internet)
  docker0: 172.17.0.1 (virtual bridge, created by Docker)

Container 1:
  eth0: 172.17.0.2 (virtual interface on bridge)
  Traffic to internet: → docker0 → host eth0 → internet (via iptables NAT)

Container 2:
  eth0: 172.17.0.3

Container-to-container:
  Container 1 pings 172.17.0.3 → goes through docker0 bridge → Container 2
  Can also use container names on user-defined networks: ping container_2

Port publishing (-p 8000:8000):
  host:8000 → iptables DNAT rule → container:8000
  External traffic to host port 8000 is forwarded to the container
```

## 3. Why This Matters for Debugging

Understanding these internals gives you concrete debugging tools:

```bash
# See what namespaces a container uses
ls /proc/$(docker inspect --format '{{.State.Pid}}' mycontainer)/ns/
# lrwxrwxrwx  mnt -> mnt:[4026532485]
# lrwxrwxrwx  pid -> pid:[4026532487]
# lrwxrwxrwx  net -> net:[4026532490]

# Inspect container networking from the host
ip netns exec container_netns ip addr show
# Or: nsenter --target <pid> --net ip addr

# See cgroup resource usage
cat /sys/fs/cgroup/memory/docker/<container_id>/memory.usage_in_bytes

# Debug a crashed container (explore its filesystem after crash)
docker run -it --rm --volumes-from crashed_container alpine sh
# Now explore /app/, /var/log/ etc. — the crashed container's filesystem

# See OverlayFS layers for an image
docker inspect myapp:latest | jq '.[0].GraphDriver'
# Shows the lower directory (image layers) and upper directory (container writes)
```

## 4. Security Implications

```
CONTAINER SECURITY MODEL:

ISOLATION PROVIDED:
  ✓ Process isolation (PID namespace — can't kill host processes)
  ✓ Filesystem isolation (mount namespace — can't access host files by default)
  ✓ Network isolation (net namespace — own IP, can't sniff other containers)
  ✓ Resource isolation (cgroups — can't exhaust host resources if limits set)

NOT PROVIDED:
  ✗ Kernel isolation — container shares the host's kernel
    Kernel vulnerability → can potentially affect all containers
  ✗ True privilege isolation if running as root
    Root inside container + kernel vulnerability = potential host compromise
  ✗ GPU isolation (by default)
  ✗ Time isolation — clock is shared

BEST PRACTICES THAT FOLLOW FROM INTERNALS:
  1. Run as non-root (USER appuser in Dockerfile)
     → Even if container is compromised, attacker has limited host privileges
  2. Use --read-only flag for immutable containers
     → Container filesystem is read-only, any writes fail (prevents tampering)
  3. Use --no-new-privileges
     → Container processes can't gain new privileges via setuid/setgid
  4. Set resource limits (--memory, --cpus)
     → Prevents one container from consuming all host resources
  5. Use seccomp profiles
     → Restrict which syscalls the container can make
     → Docker's default seccomp profile already blocks ~44 dangerous syscalls
```

## 5. Senior Engineer Perspective

**What changes once you understand internals:**

You stop thinking of containers as "mini VMs" and start thinking of them as "processes with strong opinions about their environment." This matters for:

- **Security:** a root container + kernel CVE = serious breach. Non-root containers + seccomp profiles = attack surface is dramatically reduced.
- **Resource planning:** each container's CPU/memory isn't independent — they all share the same kernel, same hardware, same network. Cgroup limits are the only thing preventing one container from starving others.
- **Debugging:** when a container "acts weird," you can use nsenter, cat /proc/*/ns, inspect cgroups — the host's Linux tools still work on container processes. The container is just a process.

**The Docker-on-Mac and Docker-on-Windows reality:**

Docker requires a Linux kernel. On macOS and Windows, Docker Desktop runs a lightweight Linux VM (using HyperKit on Mac, WSL2 on Windows). Your containers run inside that VM. This is why Docker on macOS is slower than on Linux — there's a VM layer. In production (which runs on Linux), there's no VM, just namespaces and cgroups.
