## 1. CI/CD Fundamentals (10.12)

### What Is CI/CD?

**Continuous Integration (CI):** Every code change is automatically built, tested, and validated. If tests fail, the developer is notified immediately. The goal: catch problems when they're cheapest to fix (within minutes of introduction, not weeks later).

**Continuous Delivery (CD):** Every change that passes CI is automatically deployable to production. The deployment itself may be manual (click a button) or automatic (Continuous Deployment).

**Continuous Deployment:** The deployment to production is also automated. Every commit that passes CI is deployed to production automatically.

```
CI/CD PIPELINE:

Developer pushes code
        │
        ▼
[LINT]         Check code style, formatting (ruff, eslint, gofmt)
        │      Fast: 10-30 seconds
        ▼
[TEST]         Unit tests, integration tests
        │      Medium: 1-5 minutes
        ▼
[BUILD]        Build Docker image, compile binaries
        │      Medium: 1-5 minutes
        ▼
[SCAN]         Vulnerability scan (Trivy), SAST (Semgrep, CodeQL)
        │      Medium: 1-3 minutes
        ▼
[PUSH]         Push image to container registry
        │      Fast: 30 seconds
        ▼
[DEPLOY-STAGING]  Deploy to staging environment
        │          Run smoke tests
        ▼
[DEPLOY-PROD]    (Manual approval) or (Automatic for continuous deployment)
        │
        ▼
Production running new version

TOTAL TIME (good CI/CD): 5-15 minutes from push to production
TOTAL TIME (bad CI/CD): 2-6 hours of manual steps
```

**The real-world analogy:**

CI/CD is the assembly line of software. Without it, each feature goes through a custom, manual process — like hand-crafting cars. With it, every commit flows through the same automated process — like a modern factory. Consistency, speed, and reliability are built into the process, not the individual.

## 2. GitHub Actions: The Modern Standard (10.13)

GitHub Actions is an event-driven CI/CD system built into GitHub. Workflows are YAML files in `.github/workflows/`.

```yaml
# .github/workflows/ci.yml
name: CI/CD Pipeline

on:
  push:
    branches: [main, develop]    # trigger on push to these branches
  pull_request:
    branches: [main]             # trigger on PRs targeting main

# Prevent multiple concurrent runs for the same branch
concurrency:
  group: ${{ github.workflow }}-${{ github.ref }}
  cancel-in-progress: true  # cancel old run if new push arrives

jobs:
  # ─── JOB 1: LINT AND FORMAT ──────────────────────────────────
  lint:
    runs-on: ubuntu-latest  # GitHub-hosted runner
    
    steps:
    - uses: actions/checkout@v4
    
    - uses: actions/setup-python@v5
      with:
        python-version: "3.12"
        cache: "pip"          # cache pip packages between runs
    
    - run: pip install ruff mypy
    
    - name: Lint with ruff
      run: ruff check . --output-format github
      # --output-format github: shows inline annotations in PR

    - name: Type check with mypy
      run: mypy app/ --ignore-missing-imports

  # ─── JOB 2: TESTS ─────────────────────────────────────────────
  test:
    runs-on: ubuntu-latest
    needs: lint  # only run after lint passes
    
    services:
      # Spin up a PostgreSQL service container alongside this job
      postgres:
        image: postgres:16-alpine
        env:
          POSTGRES_DB: testdb
          POSTGRES_USER: testuser
          POSTGRES_PASSWORD: testpass
        ports:
          - 5432:5432
        options: >-
          --health-cmd pg_isready
          --health-interval 5s
          --health-timeout 3s
          --health-retries 5
    
    steps:
    - uses: actions/checkout@v4
    
    - uses: actions/setup-python@v5
      with:
        python-version: "3.12"
        cache: "pip"
    
    - run: pip install -r requirements.txt -r requirements-dev.txt
    
    - name: Run tests
      env:
        DATABASE_URL: postgresql://testuser:testpass@localhost:5432/testdb
        SECRET_KEY: test-secret-key-not-for-production
      run: |
        pytest tests/ \
          --cov=app \
          --cov-report=xml \
          --cov-report=term-missing \
          --junitxml=test-results.xml \
          -v
    
    - name: Upload coverage
      uses: codecov/codecov-action@v4
      with:
        file: coverage.xml
    
    - name: Upload test results
      if: always()  # upload even if tests fail
      uses: actions/upload-artifact@v4
      with:
        name: test-results
        path: test-results.xml

  # ─── JOB 3: BUILD AND PUSH ─────────────────────────────────────
  build:
    runs-on: ubuntu-latest
    needs: test
    permissions:
      contents: read
      packages: write  # needed for GHCR push
    
    outputs:
      image-tag: ${{ steps.meta.outputs.tags }}
      image-digest: ${{ steps.push.outputs.digest }}
    
    steps:
    - uses: actions/checkout@v4
    
    # Set up Docker Buildx (enables BuildKit features)
    - uses: docker/setup-buildx-action@v3
    
    # Login to GitHub Container Registry
    - uses: docker/login-action@v3
      with:
        registry: ghcr.io
        username: ${{ github.actor }}
        password: ${{ secrets.GITHUB_TOKEN }}
    
    # Generate image tags and labels
    - uses: docker/metadata-action@v5
      id: meta
      with:
        images: ghcr.io/${{ github.repository }}
        tags: |
          type=sha,prefix=sha-,format=short   # sha-abc1234
          type=raw,value=latest,enable={{is_default_branch}}  # latest on main only
    
    # Build and push (uses BuildKit layer caching)
    - uses: docker/build-push-action@v5
      id: push
      with:
        context: .
        push: true
        tags: ${{ steps.meta.outputs.tags }}
        labels: ${{ steps.meta.outputs.labels }}
        cache-from: type=gha       # use GitHub Actions cache for layers
        cache-to: type=gha,mode=max
    
    # Scan for vulnerabilities
    - uses: aquasecurity/trivy-action@master
      with:
        image-ref: ghcr.io/${{ github.repository }}:sha-${{ github.sha }}
        format: sarif
        output: trivy-results.sarif
        severity: CRITICAL,HIGH
        exit-code: 1              # fail pipeline on CRITICAL/HIGH vulns

  # ─── JOB 4: DEPLOY TO STAGING ─────────────────────────────────
  deploy-staging:
    runs-on: ubuntu-latest
    needs: build
    if: github.ref == 'refs/heads/main'  # only on main branch
    environment: staging  # requires environment secrets configured in GitHub
    
    steps:
    - uses: actions/checkout@v4
    
    - name: Update Kubernetes deployment
      run: |
        kubectl set image deployment/api \
          api=ghcr.io/${{ github.repository }}:sha-${{ github.sha }} \
          --namespace staging
        kubectl rollout status deployment/api --namespace staging --timeout=5m
      env:
        KUBECONFIG_DATA: ${{ secrets.STAGING_KUBECONFIG }}
    
    - name: Run smoke tests
      run: |
        sleep 10  # wait for pods to be fully ready
        curl -f https://staging.api.example.com/health || exit 1

  # ─── JOB 5: DEPLOY TO PRODUCTION ──────────────────────────────
  deploy-production:
    runs-on: ubuntu-latest
    needs: deploy-staging
    environment:
      name: production
      url: https://api.example.com  # shown in GitHub deployment dashboard
    # environment "production" requires manual approval in GitHub settings
    
    steps:
    - name: Deploy to production
      run: |
        kubectl set image deployment/api \
          api=ghcr.io/${{ github.repository }}:sha-${{ github.sha }} \
          --namespace production
        kubectl rollout status deployment/api --namespace production --timeout=10m
```

## 3. Deployment Strategies (10.17)

```
RECREATE:
  Stop old version. Start new version.
  Downtime: YES (gap between stop and start)
  Use: development, when downtime is acceptable
  
ROLLING UPDATE (default in Kubernetes):
  Replace pods one at a time (or in batches).
  Old and new versions run simultaneously during update.
  Downtime: NO (always at least N-1 pods running)
  Risk: if schema change is not backward-compatible, old pods break
  
BLUE/GREEN:
  Run new version alongside old. Switch traffic atomically.
  Downtime: NO
  Instant rollback: YES (just switch traffic back)
  Cost: 2× infrastructure during transition
  
CANARY:
  Route small % of traffic to new version.
  Monitor. Gradually increase %.
  Downtime: NO
  Blast radius: small (only canary users affected)
  Best for: large services where even 0.1% error matters
  
A/B TESTING:
  Route specific user segments to specific versions.
  Based on: header, cookie, user ID, geography.
  Goal: measure product metrics (conversion, engagement), not just error rate
  Tools: feature flags, Istio traffic management
```

## 4. Infrastructure as Code (10.18): Never Click in a Console

**The principle:** Every infrastructure resource (server, database, network) must be defined in code, stored in version control, and applied through an automated process. Manual console clicks produce infrastructure that:
- Can't be reproduced (what did you click exactly?)
- Can't be reviewed (no pull request for "I clicked this button")
- Can't be rolled back (no history of what changed)
- Creates drift (staging != production because someone clicked differently)

```
INFRASTRUCTURE AS CODE TOOLS:
  Terraform   → declarative, cloud-agnostic, most popular
  Pulumi      → imperative (real programming languages), newer
  Ansible     → procedural, agentless, good for config management
  CloudFormation → AWS-native, verbose YAML/JSON
  CDK         → AWS CDK: define CloudFormation with TypeScript/Python
  Crossplane  → Kubernetes-native IaC
```

## 5. Senior Engineer Perspective

**The one CI/CD metric that matters:**

Deployment frequency. Teams that deploy more frequently have fewer incidents because each deployment is smaller. A weekly deploy is a week's worth of changes — hard to isolate the problem. A deploy every hour is one commit's worth — trivial to identify and rollback.

**The manual approval gate:**

Many teams add a manual approval requirement before production deploys. This is reasonable for high-stakes changes but becomes a bottleneck for routine deploys. The decision: is the risk of fully automated deployment lower than the risk of deploys being delayed because nobody is available to approve? For most mature teams with good test coverage: automate it. The tests are your approval gate.

**GitOps: the evolved CD pattern:**

Store your K8s manifests (or Helm values) in Git. Use ArgoCD or Flux to watch the Git repository and automatically apply changes to the cluster when the repository changes. CI builds and pushes the image, then updates the image tag in the manifests repo. CD watches the manifests repo and deploys. Rollback = revert a commit in the manifests repo.

```
CI (code repo):    push code → test → build → push image → update manifest
CD (config repo):  ArgoCD watches → detect change → apply to cluster
```
