## 1. What Is It? (Theory)

Docker is a platform for building, shipping, and running applications inside **containers** — lightweight, isolated environments that package your application with everything it needs to run: the code, the runtime, the libraries, the configuration.

Before Docker, deploying an application meant wrestling with environment differences: "It works on my machine." Your laptop has Python 3.11 and Ubuntu 22.04 with a specific libpq version. The staging server has Python 3.9 and CentOS 7. Production has Python 3.10 and something nobody can quite remember. Every deployment was a potential surprise.

Docker eliminates this by shipping the environment alongside the code. The container runs identically on your laptop, in CI, in staging, and in production. Same binary. Same dependencies. No more "works on my machine."

**The real-world analogy:**

Before shipping containers (the physical kind), international trade was chaotic. Cargo was loaded piece-by-piece onto ships, in different sizes and shapes. Each port needed different equipment. Loading and unloading was slow, labor-intensive, and expensive.

The shipping container standardized everything: a standard-sized metal box that fits on any ship, truck, or rail car, handled by standardized equipment at every port. The same container goes from a factory in China to a warehouse in Ohio without anyone caring what's inside — the interface is always the same.

Docker containers do the same for software. Any Docker container runs on any Docker-enabled machine. The interface is standardized. The contents are yours.

## 2. How It Actually Works (Deep Mechanics)

### The Three Core Concepts

**Image:** A read-only template that defines what the container will contain. Like a class definition in OOP — the blueprint, not the running instance. An image contains the filesystem, runtime, dependencies, and startup command. Images are built once and shared.

**Container:** A running instance of an image. Like an object instantiated from a class. Containers have their own isolated process space, network, and filesystem (writable layer on top of the read-only image). You can have many containers from the same image.

**Layer:** Images are built from layers — each Dockerfile instruction creates one layer. Layers are cached and reused across builds and across images that share base layers.

```
IMAGE LAYERS (example: a Python FastAPI app):

Layer 5: COPY . /app                           ← your application code
Layer 4: RUN pip install -r requirements.txt   ← your Python dependencies
Layer 3: COPY requirements.txt /app            ← requirements file
Layer 2: WORKDIR /app                          ← set working directory
Layer 1: FROM python:3.12-slim                 ← Python 3.12 + Debian slim OS

These layers stack to form the complete filesystem.
If you change requirements.txt: only layers 3-5 rebuild.
Layer 1 and 2 are pulled from cache instantly.
```

**Why layers matter for build speed:**

The Docker build cache is keyed by the layer hash. If the instruction and all its inputs haven't changed, Docker uses the cached layer. Layers invalidate the cache for all layers beneath them — so putting frequently-changing instructions (COPY your code) at the bottom and rarely-changing ones (apt-get install, pip install) near the top minimizes rebuild time.

---

### The Dockerfile

A Dockerfile is a text file containing instructions for building an image. Every instruction becomes a layer.

```dockerfile
# SYNTAX:
FROM base_image        ← start from an existing image
RUN command            ← execute a shell command (creates a new layer)
COPY src dst           ← copy files into the image
ENV KEY=VALUE          ← set environment variable
WORKDIR /path          ← set working directory for subsequent instructions
EXPOSE port            ← document which port the container listens on (doesn't actually open it)
USER username          ← switch to a non-root user
CMD ["executable"]     ← default command when container starts (can be overridden)
ENTRYPOINT ["exec"]    ← command that always runs (CMD becomes its arguments)
```

**A production-grade Dockerfile:**

```dockerfile
# STAGE 1: Install dependencies (separate from final image)
FROM python:3.12-slim AS builder

# Install system build dependencies (only in builder, not final image)
RUN apt-get update && apt-get install -y \
    build-essential \
    libpq-dev \
  && rm -rf /var/lib/apt/lists/*  # ← critical: clean apt cache in same RUN layer

WORKDIR /app

# Copy requirements first (cache this layer as long as requirements.txt doesn't change)
COPY requirements.txt .
RUN pip install --no-cache-dir --user -r requirements.txt


# STAGE 2: Final runtime image (much smaller — no build tools)
FROM python:3.12-slim AS runtime

# Create a non-root user (security: containers shouldn't run as root)
RUN useradd --create-home --no-log-init appuser

# Copy installed packages from builder stage
COPY --from=builder /root/.local /home/appuser/.local

# Set up the working directory
WORKDIR /app

# Copy application code (changes frequently → put last for cache efficiency)
COPY --chown=appuser:appuser . .

# Switch to non-root user
USER appuser

# Environment settings
ENV PATH=/home/appuser/.local/bin:$PATH
ENV PYTHONDONTWRITEBYTECODE=1  # don't write .pyc files
ENV PYTHONUNBUFFERED=1          # unbuffered output (logs appear immediately)

# Document the port (informational only)
EXPOSE 8000

# Health check (used by Docker and Kubernetes)
HEALTHCHECK --interval=30s --timeout=10s --start-period=5s --retries=3 \
  CMD curl -f http://localhost:8000/health || exit 1

# The command that runs when the container starts
CMD ["uvicorn", "main:app", "--host", "0.0.0.0", "--port", "8000", \
     "--workers", "1", "--no-access-log"]
```

### .dockerignore: What NOT to Include

Every file in the build context is sent to the Docker daemon. Large contexts slow builds. More importantly, accidentally including secrets is catastrophic.

```
# .dockerignore
.git/               ← git history (can be large, and exposes secrets in history)
.env                ← secrets! never include
.env.local
__pycache__/
*.pyc
*.pyo
.pytest_cache/
.coverage
htmlcov/
dist/
build/
node_modules/
*.log
*.md                ← documentation (not needed at runtime)
tests/              ← tests aren't needed in the runtime image
docs/
.vscode/
.idea/
```

### Multi-Stage Builds: The Game-Changer

Multi-stage builds let you use one image to build your application and a different (smaller) image to run it. This is the single most impactful Dockerfile optimization.

```
WITHOUT MULTI-STAGE:
  FROM python:3.12                   ← 920MB base image
  RUN apt-get install build-essential ← adds 200MB
  RUN pip install ...                 ← adds 300MB
  COPY . .
  ← Final image: 1.4GB

WITH MULTI-STAGE:
  FROM python:3.12 AS builder         ← 920MB (only used for building)
  RUN apt-get install build-essential
  RUN pip install --user ...
  
  FROM python:3.12-slim AS runtime    ← 130MB base image
  COPY --from=builder /root/.local /root/.local  ← only compiled packages
  COPY . .
  ← Final image: 180MB
  
  8x smaller image = faster pulls, less storage, smaller attack surface
```

## 3. Real-World Context

### Why Uber Migrated to Docker (and What They Learned)

Uber's engineering blog describes their Docker migration: before containers, deploying a new service required coordinating with the infrastructure team to provision VMs, install dependencies, and configure the environment — a process taking days to weeks. With Docker, a new service deploy takes minutes. The service team writes a Dockerfile, pushes to the container registry, and the orchestration platform handles the rest.

Their key insight: **Docker's main value isn't performance — it's standardization**. The interface contract (a container image) is the same for every service. Operations teams don't need to know if a service uses Python or Go or Ruby. It's a container.

### The Security Lesson: Always Run as Non-Root

In 2020, a container breakout vulnerability (CVE-2020-14386) allowed attackers to escape the container namespace if the container was running as root. Containers running as non-root were unaffected — the process didn't have the privileges needed to exploit the vulnerability.

This is why every production Dockerfile should include `USER appuser` — it's defense in depth. Even if an attacker compromises your container, they're limited to the low-privilege user account, not root.

## 4. Code Examples

### Building and Running

```bash
# Build an image from the Dockerfile in the current directory
docker build -t myapp:latest .
docker build -t myapp:v1.2.3 .              # specific version tag
docker build -t myapp:latest --no-cache .   # ignore build cache

# Run a container
docker run myapp:latest
docker run -d myapp:latest                  # detached (background)
docker run -p 8000:8000 myapp:latest       # map host port to container port
docker run -e DATABASE_URL=postgres://... myapp:latest  # pass env var
docker run -v /host/path:/container/path myapp:latest   # mount volume

# Inspect a running container
docker ps                                   # list running containers
docker logs container_id                    # view stdout/stderr
docker exec -it container_id /bin/sh       # open a shell inside
docker inspect container_id                # full container metadata

# Image management
docker images                               # list local images
docker image rm myapp:latest                # delete an image
docker pull python:3.12-slim               # pull from registry
docker push registry.example.com/myapp:v1 # push to registry

# Cleanup (important in CI and on dev machines)
docker system prune -f                     # remove unused containers/images
docker system prune -af                    # remove ALL unused, including unreferenced images
```

### Checking Your Image Size

```bash
# See all layers and their sizes
docker history myapp:latest

# Output:
# IMAGE         CREATED       CREATED BY                                SIZE
# a3b4c5d6e7f8  2 hours ago   CMD ["uvicorn" "main:app"...]              0B
# f8e7d6c5b4a3  2 hours ago   COPY . .                                   42kB
# e7f8a3b4c5d6  2 hours ago   COPY --from=builder ...                    180MB
# ...
# 1234567890ab  2 weeks ago   /bin/sh -c #(nop) ADD file:...             130MB (base layer)

# Compare image sizes
docker images | grep myapp
# myapp    v2-multistage    180MB  ← with multi-stage
# myapp    v1-single        1.4GB  ← without multi-stage
```

## 6. Common Mistakes

```
MISTAKE 1: Running as root
  Fix: add RUN useradd appuser && USER appuser

MISTAKE 2: Not using .dockerignore
  Fix: create .dockerignore before any docker build

MISTAKE 3: Apt-get without cleanup in same layer
  Fix: ... && rm -rf /var/lib/apt/lists/*  (in the same RUN command)
  
  Why: each RUN is a layer. If you clean in a separate RUN:
  Layer N:    apt-get install → +500MB
  Layer N+1:  rm -rf → -500MB (from layer N+1's perspective)
  But layer N still exists and contributes to total image size.
  Cleanup MUST be in the same RUN command as the install.

MISTAKE 4: Copying too much (no .dockerignore)
  Fix: create a comprehensive .dockerignore

MISTAKE 5: One Dockerfile layer per RUN (no combining)
  Fix: combine related apt-get/npm commands:
  RUN apt-get update && apt-get install -y git curl vim && rm -rf /var/lib/apt/lists/*
  Not: three separate RUN commands for each

MISTAKE 6: Not pinning base image versions
  FROM python:latest  ← DON'T: "latest" tag changes
  FROM python:3.12.3-slim  ← DO: pin the exact version for reproducible builds

MISTAKE 7: Storing secrets in the image
  ENV SECRET_KEY=mysecret  ← DON'T: visible in docker inspect, docker history
  ← DO: pass secrets at runtime via -e or a secrets manager
```

## 7. Senior Engineer Perspective

**What a junior gets wrong:**

Building giant images because "it just works." A 2GB image works fine on your laptop. In production with 50 pods auto-scaling, each pod pull is 2GB, taking 3-5 minutes on an AWS ECS cluster. During a traffic spike when you need 10 new instances FAST, you're waiting 5 minutes per instance pull. A well-built 150MB image pulls in 20 seconds.

The second mistake: including development tools (`vim`, `curl`, `strace`) in production images. These are legitimate debug tools, but they're also an attacker's toolkit if the container is compromised. Production images should contain only what's needed to run the application. Nothing more.

**The philosophy of container images:**

A container image is not your server. It's not a VM. It's a **single-purpose, immutable artifact** that does one job. The 12-factor principle of stateless, single-responsibility applies here: one container, one process, one concern. If you find yourself putting a web server and a cron job and a background worker in the same container — split them into separate containers that share the same image layers.

**The test for a good production Dockerfile:**

```bash
# 1. Does it build successfully on a clean machine? (no hidden dependencies)
docker build --no-cache -t myapp:test .

# 2. Is the image small? (under 300MB for most Python/Node apps)
docker images myapp:test

# 3. Does it run as non-root?
docker inspect myapp:test | grep User

# 4. Does it pass Hadolint?
hadolint Dockerfile

# 5. Are there known vulnerabilities?
trivy image myapp:test
```
