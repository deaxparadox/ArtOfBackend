## 1. What Is It? (Theory)

A container registry is a storage and distribution system for Docker images. It's the "npm registry" or "PyPI" of the container world — but instead of code packages, it stores container images.

The workflow is always: **build** the image locally or in CI → **push** to the registry → **pull** from the registry on the server where it runs. The registry is the handoff point between where code is built and where it runs.

**Why a dedicated registry matters:**

You could theoretically copy image files from one machine to another manually (docker save / docker load). But registries provide:
- Content-addressable storage (same image layer stored once regardless of how many images use it)
- Authentication and access control
- Image scanning for vulnerabilities
- Geographic distribution (pull from the nearest datacenter)
- Immutability (image digest is a cryptographic hash — the image cannot be changed)
- Lifecycle management (retention policies, cleanup of old images)

## 2. The Major Registries

### Docker Hub (hub.docker.com)

The original and largest container registry. All official base images (`python:3.12`, `postgres:16`, `redis:7`) live here.

```bash
# Public image (anyone can pull)
docker pull python:3.12-slim

# Your own image (free for public, paid for private)
docker push nitishkushwaha/myapp:v1.0

# Pull your own image
docker pull nitishkushwaha/myapp:v1.0
```

**Limitations:** Docker Hub has rate limits for unauthenticated pulls (100 pulls per 6 hours from a single IP). In CI, this causes random build failures. Always authenticate or use a paid plan.

**Docker Hub in production:** Generally only for base images pulled via Dockerfile. Your own application images should be in a cloud provider's registry for better performance and access control.

---

### AWS ECR (Elastic Container Registry)

The registry for AWS-native workloads. Deeply integrated with ECS, EKS, Lambda, and CodeBuild.

```bash
# Authenticate (valid for 12 hours)
aws ecr get-login-password --region us-east-1 | \
    docker login --username AWS --password-stdin \
    123456789.dkr.ecr.us-east-1.amazonaws.com

# Create a repository (do this once)
aws ecr create-repository --repository-name myapp --region us-east-1

# Tag your local image with the ECR URL
docker tag myapp:latest 123456789.dkr.ecr.us-east-1.amazonaws.com/myapp:latest
docker tag myapp:latest 123456789.dkr.ecr.us-east-1.amazonaws.com/myapp:v1.2.3

# Push
docker push 123456789.dkr.ecr.us-east-1.amazonaws.com/myapp:latest
docker push 123456789.dkr.ecr.us-east-1.amazonaws.com/myapp:v1.2.3

# In your ECS task definition or Kubernetes deployment:
# image: 123456789.dkr.ecr.us-east-1.amazonaws.com/myapp:v1.2.3
```

**ECR features:**
- **Lifecycle policies:** automatically delete images older than N days or keep only the N most recent
- **Image scanning:** integrates with Amazon Inspector for vulnerability scanning
- **Private:** no public access unless explicitly enabled
- **Cross-account access:** share images across AWS accounts via IAM policies
- **Pricing:** $0.10/GB/month + standard data transfer

### GitHub Container Registry (GHCR)

Integrated with GitHub Actions, free for public repositories.

```bash
# Authenticate with a Personal Access Token (or GITHUB_TOKEN in Actions)
echo $GITHUB_TOKEN | docker login ghcr.io -u USERNAME --password-stdin

# Push
docker tag myapp:latest ghcr.io/nitishkushwaha/myapp:latest
docker push ghcr.io/nitishkushwaha/myapp:latest
```

```yaml
# GitHub Actions: build and push automatically
name: Build and Push

on:
  push:
    branches: [main]

jobs:
  build:
    runs-on: ubuntu-latest
    permissions:
      contents: read
      packages: write  # needed to push to GHCR

    steps:
    - uses: actions/checkout@v4

    - name: Log in to GHCR
      uses: docker/login-action@v3
      with:
        registry: ghcr.io
        username: ${{ github.actor }}
        password: ${{ secrets.GITHUB_TOKEN }}  # GITHUB_TOKEN is auto-provided

    - name: Build and push
      uses: docker/build-push-action@v5
      with:
        context: .
        push: true
        tags: |
          ghcr.io/${{ github.repository }}:latest
          ghcr.io/${{ github.repository }}:${{ github.sha }}
        cache-from: type=gha        # use GitHub Actions cache
        cache-to: type=gha,mode=max # populate the cache
```

**GHCR is excellent when:** your code is on GitHub, you use GitHub Actions for CI, and you want zero-configuration integration.

### Google Container Registry / Artifact Registry

```bash
# Authenticate
gcloud auth configure-docker us-central1-docker.pkg.dev

# Push to Artifact Registry (GCR's successor)
docker tag myapp:latest us-central1-docker.pkg.dev/my-project/myrepo/myapp:v1.0
docker push us-central1-docker.pkg.dev/my-project/myrepo/myapp:v1.0
```

## 3. Image Tagging Strategy

Tags are mutable references (`:latest` can point to a different image tomorrow). Digests are immutable (cryptographic hash of the image content).

```
TAGGING STRATEGIES:

:latest                     ← NEVER rely on this in production
                              (changes on every push, breaks reproducibility)

:v1.2.3                     ← semantic version (good for libraries/public releases)
:1.2.3                      ← same but without 'v' prefix

:git-sha (short)            ← e.g., :a3f4d2b (first 7 chars of commit SHA)
:git-full-sha               ← full 40-char SHA (most reproducible)
:20240615-142300            ← timestamp (sortable but doesn't tell you what changed)

:main-20240615-a3f4d2b      ← branch + date + sha (comprehensive)

PRODUCTION RECOMMENDATION:
  Tag with the git SHA for every push:
    myapp:a3f4d2b           ← deploy this to staging
  Tag with the version for releases:
    myapp:v1.2.3            ← deploy this to production

  This way:
  - You can always reproduce "what was deployed on June 15" by finding the SHA
  - You can roll back to any previous SHA
  - :latest is just a convenience tag for local dev — never use in K8s manifests
```

**Using digest for maximum reproducibility:**

```bash
# Get the digest (sha256) of an image
docker inspect --format='{{index .RepoDigests 0}}' python:3.12-slim
# python@sha256:abc123...

# In Dockerfile: use digest instead of tag (immune to tag mutation)
FROM python@sha256:abc123...
# Nobody can change what this resolves to — the hash IS the content
```

## 4. Lifecycle Management and Cleanup

Images accumulate quickly in production. A team pushing several times per day generates thousands of images per year. Uncleaned registries become expensive.

```bash
# ECR: lifecycle policy (via AWS CLI or Terraform)
aws ecr put-lifecycle-policy \
    --repository-name myapp \
    --lifecycle-policy-text '{
      "rules": [
        {
          "rulePriority": 1,
          "description": "Keep only last 10 tagged images",
          "selection": {
            "tagStatus": "tagged",
            "tagPrefixList": ["v"],
            "countType": "imageCountMoreThan",
            "countNumber": 10
          },
          "action": {"type": "expire"}
        },
        {
          "rulePriority": 2,
          "description": "Delete untagged images after 1 day",
          "selection": {
            "tagStatus": "untagged",
            "countType": "sinceImagePushed",
            "countUnit": "days",
            "countNumber": 1
          },
          "action": {"type": "expire"}
        }
      ]
    }'
```

## 5. Senior Engineer Perspective

**The image immutability principle:**

Once you push `myapp:a3f4d2b` and deploy it to production, never push a different image with the same tag. If you need to update, build a new image and give it a new tag. This ensures:
- You can always reproduce what was running in production
- Rollbacks are just changing the tag in your deployment manifest
- CI/CD pipelines are deterministic — the same tag always means the same code

**The one mistake that causes mysterious production bugs:**

Using `:latest` in Kubernetes manifests. If you have `image: myapp:latest` in your deployment and someone pushes a new `:latest` tag, Kubernetes will pull the new image on the next pod restart — even if you didn't do a deployment. Your pods quietly run different code. This is how "I didn't deploy anything but the behavior changed" incidents happen.

**Always use a specific, immutable tag in production deployments.** `:latest` is for your local `docker run` commands, nothing else.
