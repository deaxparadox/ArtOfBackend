## GraphQL Fundamentals (6.7)

GraphQL (Facebook, 2015) is a query language for APIs and a runtime for executing those queries. Clients specify exactly what data they need; the server returns exactly that — no more, no less.

**The problem GraphQL solves:**

```
REST PROBLEM 1: Over-fetching
  GET /users/123
  Returns 50 fields — but the client only needs name and email.
  → Wastes bandwidth, especially on mobile.

REST PROBLEM 2: Under-fetching (N+1 HTTP calls)
  GET /users/123          → get user
  GET /users/123/orders   → get orders
  GET /orders/456/items   → get order items
  3 round trips to render one page.

GRAPHQL SOLUTION:
  One request, client declares what it needs:
  {
    user(id: 123) {
      name
      email
      orders(status: "paid") {
        id
        totalCents
        items {
          productName
          quantity
        }
      }
    }
  }
  → One request, exactly the needed fields, any depth.
```

## GraphQL Operations

### Queries (Read)

```graphql
# Query with arguments
query GetUser($userId: ID!) {
    user(id: $userId) {
        id
        name
        email
        orders(status: PAID, limit: 5) {
            id
            totalCents
            createdAt
        }
    }
}

# Multiple queries in one request
query Dashboard($userId: ID!) {
    user(id: $userId) { name, email }
    notifications(userId: $userId, unread: true) { id, message }
    recentOrders(userId: $userId, limit: 3) { id, status }
}
```

### Mutations (Write)

```graphql
mutation CreateOrder($input: CreateOrderInput!) {
    createOrder(input: $input) {
        id
        status
        totalCents
        createdAt
    }
}

# Variables:
# { "input": { "userId": 123, "productId": 456, "quantity": 2 } }
```

### Subscriptions (Real-time)

```graphql
subscription OrderUpdates($orderId: ID!) {
    orderStatusChanged(orderId: $orderId) {
        id
        status
        updatedAt
    }
}
# Server pushes updates over WebSocket when order status changes
```

## GraphQL Schema Design (6.8)

```graphql
# schema.graphql — the contract between client and server

type User {
    id: ID!              # ! = non-null
    email: String!
    name: String!
    orders(
        status: OrderStatus
        limit: Int = 10
    ): [Order!]!          # non-null list of non-null Orders
    createdAt: DateTime!
}

type Order {
    id: ID!
    user: User!
    items: [OrderItem!]!
    status: OrderStatus!
    totalCents: Int!
    createdAt: DateTime!
}

type OrderItem {
    id: ID!
    product: Product!
    quantity: Int!
    unitPriceCents: Int!
}

enum OrderStatus {
    PENDING
    PAID
    SHIPPED
    DELIVERED
    CANCELLED
}

# Input types for mutations
input CreateOrderInput {
    userId: ID!
    items: [OrderItemInput!]!
}

input OrderItemInput {
    productId: ID!
    quantity: Int!
}

# Root types
type Query {
    user(id: ID!): User
    users(limit: Int = 20, offset: Int = 0): [User!]!
    order(id: ID!): Order
}

type Mutation {
    createOrder(input: CreateOrderInput!): Order!
    cancelOrder(orderId: ID!): Order!
}

type Subscription {
    orderStatusChanged(orderId: ID!): Order!
}
```

## Resolvers (How GraphQL Fetches Data)

```python
# Strawberry (Python GraphQL)
import strawberry
from typing import Optional

@strawberry.type
class User:
    id: int
    email: str
    name: str

    @strawberry.field
    async def orders(self, info: strawberry.types.Info, status: Optional[str] = None) -> list["Order"]:
        # Resolver for the orders field
        return await info.context["db"].get_orders(user_id=self.id, status=status)

@strawberry.type
class Query:
    @strawberry.field
    async def user(self, info: strawberry.types.Info, id: int) -> Optional[User]:
        return await info.context["db"].get_user(id)

schema = strawberry.Schema(query=Query)
```

## The N+1 Problem in GraphQL + DataLoader (6.9)

GraphQL's resolver-per-field design creates N+1 database queries:

```
Query: { users { orders { items } } }

Without DataLoader:
  1. Fetch all users: SELECT * FROM users               (1 query)
  2. For each user (100 users):
     SELECT * FROM orders WHERE user_id = ?            (100 queries)
  3. For each order (500 orders):
     SELECT * FROM order_items WHERE order_id = ?      (500 queries)
  Total: 601 queries for one GraphQL request!

DataLoader: batch + cache resolver calls
  1. Fetch all users: SELECT * FROM users               (1 query)
  2. Collect all user_ids [1,2,3...100], make ONE query:
     SELECT * FROM orders WHERE user_id = ANY([1..100]) (1 query)
  3. Collect all order_ids, make ONE query:
     SELECT * FROM order_items WHERE order_id = ANY([...]) (1 query)
  Total: 3 queries! ← same data, 200x fewer queries
```

```python
from strawberry.dataloader import DataLoader

async def batch_load_orders(user_ids: list[int]) -> list[list[Order]]:
    """Load orders for many users in one query."""
    all_orders = await db.fetch(
        "SELECT * FROM orders WHERE user_id = ANY($1::int[])",
        user_ids
    )
    # Group orders by user_id
    orders_by_user = {uid: [] for uid in user_ids}
    for order in all_orders:
        orders_by_user[order["user_id"]].append(order)
    return [orders_by_user[uid] for uid in user_ids]

# Use in resolver
@strawberry.type
class User:
    id: int

    @strawberry.field
    async def orders(self, info: strawberry.types.Info) -> list[Order]:
        return await info.context["orders_loader"].load(self.id)

# Set up loader per request (not shared across requests)
orders_loader = DataLoader(load_fn=batch_load_orders)
```

## GraphQL vs. REST (6.10)

```
USE GRAPHQL WHEN:
  ✓ Mobile clients with bandwidth constraints (request exactly what's needed)
  ✓ Many different clients needing different data shapes (web, iOS, Android)
  ✓ Rapidly evolving APIs (add fields freely; old clients still work)
  ✓ Complex nested data where multiple REST calls would be needed
  ✓ Real-time features needed (subscriptions)
  ✓ You want to expose a public API that third parties query flexibly

USE REST WHEN:
  ✓ Simple CRUD operations
  ✓ File uploads and downloads
  ✓ Strong HTTP caching requirements (GET /products is cacheable; POST /graphql isn't)
  ✓ Webhooks and event-driven integrations
  ✓ Your team is more familiar with REST
  ✓ Simpler server-side implementation needed
  ✓ The API consumers are well-known and their needs are fixed

THE HONEST COMPARISON:
  GraphQL solves real problems (over-fetching, multiple round trips).
  It also introduces real complexity (DataLoader, auth per field, caching harder).
  
  Not every API needs GraphQL. Internal microservices rarely do.
  Public-facing APIs serving many diverse clients: GraphQL shines.
```

## Apollo Server & Federation (6.11)

```javascript
// Apollo Server (Node.js GraphQL server)
import { ApolloServer } from '@apollo/server';
import { expressMiddleware } from '@apollo/server/express4';

const server = new ApolloServer({
    typeDefs,
    resolvers,
    plugins: [ApolloServerPluginLandingPageGraphQLPlayground()],
});

await server.start();
app.use('/graphql', expressMiddleware(server, {
    context: async ({ req }) => ({
        user: await authenticateRequest(req),
        db: databasePool,
        ordersLoader: new DataLoader(batchLoadOrders),
    }),
}));

// Apollo Federation: combine multiple GraphQL services into one schema
// User service owns User type
// Order service owns Order type and EXTENDS User with orders field
// Apollo Gateway combines them into one schema
// Client queries as if it's one service, requests routed to correct subgraph
```

## GraphQL Security: Depth Limiting & Complexity (6.13)

```python
# Malicious query: exponential data fetching
query {
    users {
        friends {
            friends {
                friends {
                    friends { id }
                }
            }
        }
    }
}
# 4 levels deep: if 100 users each have 100 friends: 100^4 = 100M rows!

# Protection 1: Depth limiting
import strawberry
from strawberry.extensions import QueryDepthLimiter

schema = strawberry.Schema(
    query=Query,
    extensions=[QueryDepthLimiter(max_depth=5)]
)

# Protection 2: Query complexity limit
from strawberry.extensions import MaxTokensLimiter

schema = strawberry.Schema(
    query=Query,
    extensions=[MaxTokensLimiter(max_token_count=1000)]
)

# Production GraphQL security checklist:
# □ Depth limiting (max 5-7 levels)
# □ Complexity limit (cost per field, reject expensive queries)
# □ Timeout per query (5-30 seconds)
# □ Persisted queries (only allow pre-approved queries in production)
# □ Authentication enforced in context, not in resolvers
# □ Authorization per field (not just per operation)
# □ Rate limiting (queries per user per minute)
# □ No introspection in production (hides schema from attackers)
```
