## REST Fundamentals (6.1)

REST (Representational State Transfer) is an architectural style for distributed hypermedia systems, defined by Roy Fielding in his 2000 PhD dissertation. The key word: *architectural style*, not a protocol or standard. REST is a set of constraints.

**The 6 REST constraints (what people miss):**

```
1. CLIENT-SERVER
   Separation of concerns: client handles UI, server handles data.
   Improves portability (multiple clients for one server).

2. STATELESS
   Each request contains ALL information needed to process it.
   No session state stored on server between requests.
   → Auth token in each request (JWT), not server-side sessions.

3. CACHEABLE
   Responses must declare whether they're cacheable.
   → Cache-Control headers on every response.

4. UNIFORM INTERFACE (the key constraint)
   Four sub-constraints:
   a. Resource identification: resources identified by URIs
   b. Manipulation through representations: JSON/XML returned, not direct DB access
   c. Self-descriptive messages: each message includes enough info to process it
   d. HATEOAS (hypermedia): responses include links to next possible actions

5. LAYERED SYSTEM
   Client doesn't know if it's talking to origin server, LB, or CDN.
   Each layer only knows the layer it interfaces with.

6. CODE ON DEMAND (optional)
   Server can send executable code to client (JavaScript).
```

**What most "REST" APIs actually implement:** Constraints 1, 2, 3, and part of 4a. Very few implement HATEOAS (4d). Almost nothing implements it fully. "REST" in practice means "HTTP API with JSON" — pragmatic, not academic.

## RESTful Resource Design (6.2)

```
REST RESOURCE DESIGN RULES:

NOUNS, NOT VERBS:
  ✗ /getUser             ← verb-based (RPC style)
  ✗ /createOrder
  ✓ /users               ← noun-based (resource)
  ✓ /orders

COLLECTIONS AND ITEMS:
  /users                 → collection of all users
  /users/123             → single user (item)
  /users/123/orders      → orders belonging to user 123
  /users/123/orders/456  → specific order of user 123

PLURAL NOUNS:
  ✓ /users (not /user)
  ✓ /products (not /product)
  ✓ /orders (not /order)

HIERARCHY FOR RELATIONSHIPS:
  /users/123/orders      → orders owned by user (user is parent)
  /orders/456/items      → items in a specific order

AVOID DEEP NESTING (max 2-3 levels):
  ✗ /users/123/orders/456/items/789/reviews
  ✓ /order-items/789/reviews  (flatten where relationships aren't the focus)
  ✓ GET /reviews?order_item_id=789

ACTIONS THAT DON'T FIT CRUD:
  POST /orders/456/cancel        → cancel action (verb as sub-resource)
  POST /payments/456/refund
  POST /users/123/verify-email
  
  These are acceptable when the action doesn't map to a resource.
```

## HTTP Methods Semantics (6.3)

```
METHOD    OPERATION   IDEMPOTENT   SAFE   BODY   RESPONSE
GET       Read        Yes          Yes    No     Resource data
POST      Create      No*          No     Yes    Created resource
PUT       Replace     Yes          No     Yes    Updated resource
PATCH     Partial     No*          No     Yes    Updated resource
DELETE    Delete      Yes          No     No     204 No Content

* POST and PATCH CAN be made idempotent with idempotency keys

CORRECT USAGE:

GET /users/123
  → Return user 123
  → 200 OK with user data
  → 404 Not Found if user doesn't exist

POST /users
  Body: { "email": "alice@example.com", "name": "Alice" }
  → Create new user
  → 201 Created with user data and Location: /users/456 header

PUT /users/123
  Body: { "email": "alice@example.com", "name": "Alice Smith" }
  → Replace entire user 123 (all fields required)
  → 200 OK with updated user

PATCH /users/123
  Body: { "name": "Alice Smith" }  ← partial update (only what's changing)
  → Update only the name
  → 200 OK with updated user

DELETE /users/123
  → Delete user 123
  → 204 No Content (no body)
  → 404 Not Found if user doesn't exist

HEAD /users/123
  → Same as GET but no body (check if resource exists, get headers)
  → 200 OK or 404 Not Found

OPTIONS /users
  → What methods are allowed on this endpoint?
  → 200 OK with Allow: GET, POST, DELETE header
```

## REST API Design Best Practices (6.4)

### Filtering, Sorting, Pagination

```
FILTERING:
  GET /orders?status=paid
  GET /orders?status=paid&user_id=123
  GET /orders?min_total=1000&max_total=5000
  GET /users?created_after=2024-01-01&role=admin

SORTING:
  GET /orders?sort=created_at          ascending
  GET /orders?sort=-created_at         descending (minus prefix)
  GET /orders?sort=-total_cents,created_at  multiple fields

PAGINATION (see topic 2.18 for performance):
  Offset: GET /orders?page=2&limit=20
  Cursor: GET /orders?cursor=eyJ...&limit=20
  
SEARCH:
  GET /products?q=wireless+headphones
  GET /users?search=alice

FIELD SELECTION (reduce response size):
  GET /users?fields=id,email,name
  GET /orders?include=user,items
```

### Response Envelope vs. Direct

```json
// DIRECT (preferred for REST):
// GET /users/123 → 200 OK
{ "id": 123, "email": "alice@example.com", "name": "Alice" }

// ENVELOPE (common but verbose):
// GET /users/123 → 200 OK
{
  "data": { "id": 123, "email": "alice@example.com" },
  "meta": { "timestamp": "2024-06-15T14:30:00Z" }
}

// COLLECTION:
// GET /users → 200 OK
{
  "items": [...],
  "total": 1532,
  "page": 1,
  "limit": 20
}
// OR with cursor:
{
  "items": [...],
  "next_cursor": "eyJ...",
  "has_more": true
}
```

### Error Responses (Consistent Format Is Critical)

```json
// 4xx errors: always explain what's wrong
// 422 Unprocessable Entity
{
  "error": "validation_failed",
  "message": "Request body validation failed",
  "details": [
    { "field": "email", "message": "must be a valid email address" },
    { "field": "age", "message": "must be at least 13" }
  ],
  "request_id": "abc-123"
}

// 404 Not Found
{
  "error": "not_found",
  "message": "User with id 123 not found",
  "request_id": "def-456"
}

// 500 Internal Server Error (NEVER leak internals)
{
  "error": "internal_error",
  "message": "An unexpected error occurred",
  "request_id": "ghi-789"  ← user can provide this to support
}
```

## HATEOAS (6.5)

HATEOAS (Hypermedia As The Engine Of Application State) is the REST constraint most APIs ignore. In pure REST, responses include links to valid next actions.

```json
// HATEOAS-compliant order response:
{
  "id": 456,
  "status": "pending",
  "total_cents": 5000,
  "_links": {
    "self": { "href": "/orders/456" },
    "cancel": { "href": "/orders/456/cancel", "method": "POST" },
    "pay": { "href": "/orders/456/payment", "method": "POST" },
    "user": { "href": "/users/123" }
  }
}
// Client follows links instead of hardcoding URLs
// API can change URLs; clients just follow the links
```

**Reality:** HATEOAS is rarely implemented in practice. REST without HATEOAS is more accurately "HTTP + JSON API" — which is what 99% of APIs are. HATEOAS adds overhead for minimal practical benefit unless your API consumers are expected to be generic hypermedia clients.

## OpenAPI / Swagger (6.6)

OpenAPI is a standard for describing REST APIs in a machine-readable format (YAML/JSON). It enables automatic client generation, documentation, and validation.

```yaml
# openapi.yaml
openapi: 3.1.0
info:
  title: My API
  version: 1.0.0

paths:
  /users/{userId}:
    get:
      summary: Get a user
      parameters:
        - name: userId
          in: path
          required: true
          schema: { type: integer }
      responses:
        '200':
          description: User found
          content:
            application/json:
              schema:
                $ref: '#/components/schemas/User'
        '404':
          $ref: '#/components/responses/NotFound'

components:
  schemas:
    User:
      type: object
      required: [id, email, name]
      properties:
        id: { type: integer }
        email: { type: string, format: email }
        name: { type: string }
```

**FastAPI auto-generates OpenAPI:**

```python
from fastapi import FastAPI
app = FastAPI()

# FastAPI generates /openapi.json and /docs automatically
# Every route, parameter, and response model creates the schema
@app.get("/users/{user_id}", response_model=UserResponse, tags=["users"])
async def get_user(user_id: int):
    """Get a user by ID."""  # This becomes the summary/description
    ...
```

**OpenAPI tools:**
- Swagger UI: interactive documentation (browse and test API)
- Redoc: beautiful documentation site
- openapi-generator: generate client SDKs in 40+ languages
- Spectral: OpenAPI linting and style guide enforcement

## Senior Engineer Perspective on REST

**What a junior gets wrong:**

Calling everything "REST" that uses HTTP. True REST is a set of constraints, not just "HTTP API." More importantly: blindly following REST for operations that don't fit CRUD. An endpoint like `POST /orders/456/cancel` isn't "un-RESTful" — it's a pragmatic design decision. Use REST principles as guidelines, not laws.

**What a senior designs for:**

Backwards compatibility. Every field you add is a commitment. Every field you remove is a breaking change. Design APIs assuming clients will be out in the wild for years without updating. Make all new fields optional. Never remove fields without versioning (see topic 2.19).

**The pragmatic REST checklist:**

```
□ Nouns in URLs (not verbs)
□ Plural resource names
□ Consistent HTTP methods for CRUD
□ Correct HTTP status codes (201 for create, 204 for delete)
□ Consistent error format with helpful messages
□ Pagination on all list endpoints
□ OpenAPI spec generated or maintained
□ API versioning strategy defined (/api/v1/)
□ Request IDs on all responses
□ Response schema documented (avoid surprises)
```
