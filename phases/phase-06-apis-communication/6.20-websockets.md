## WebSockets (6.20)

WebSockets provide a **full-duplex, persistent connection** between client and server. Unlike HTTP (request/response), WebSocket connections stay open — both sides can send messages at any time.

```
HTTP (request/response):
  Client ──── GET /messages ────► Server
  Client ◄──── 200 + data ─────── Server
  (each "check for new messages" = a new request)

WebSocket (persistent, bidirectional):
  Client ──── upgrade to WebSocket ────► Server (HTTP upgrade, 1 time)
  Client ◄──── message ─────────────── Server (server pushes!)
  Client ──── message ──────────────►  Server (client sends!)
  Client ◄──── message ─────────────── Server
  (connection stays open until explicitly closed)
```

### WebSocket Handshake

```http
Client sends HTTP upgrade request:
  GET /ws HTTP/1.1
  Host: api.example.com
  Upgrade: websocket
  Connection: Upgrade
  Sec-WebSocket-Key: dGhlIHNhbXBsZSBub25jZQ==
  Sec-WebSocket-Version: 13

Server responds:
  HTTP/1.1 101 Switching Protocols
  Upgrade: websocket
  Connection: Upgrade
  Sec-WebSocket-Accept: s3pPLMBiTxaQ9kYGzzhZRbK+xOo=

Connection is now a WebSocket — no more HTTP.
Data travels as "frames" (binary or text).
```

### WebSocket Server (FastAPI)

```python
from fastapi import FastAPI, WebSocket, WebSocketDisconnect
from typing import dict

app = FastAPI()

class ConnectionManager:
    def __init__(self):
        self.active_connections: dict[str, WebSocket] = {}

    async def connect(self, user_id: str, websocket: WebSocket):
        await websocket.accept()
        self.active_connections[user_id] = websocket

    def disconnect(self, user_id: str):
        self.active_connections.pop(user_id, None)

    async def send_to_user(self, user_id: str, message: dict):
        if ws := self.active_connections.get(user_id):
            await ws.send_json(message)

    async def broadcast(self, message: dict):
        for ws in self.active_connections.values():
            await ws.send_json(message)

manager = ConnectionManager()

@app.websocket("/ws/{user_id}")
async def websocket_endpoint(websocket: WebSocket, user_id: str):
    await manager.connect(user_id, websocket)
    try:
        while True:
            data = await websocket.receive_json()
            # Echo back or route to appropriate handler
            await manager.send_to_user(user_id, {"echo": data})
    except WebSocketDisconnect:
        manager.disconnect(user_id)

# Notify a user from a background task / other endpoint
@app.post("/notify/{user_id}")
async def notify_user(user_id: str, message: dict):
    await manager.send_to_user(user_id, message)
    return {"sent": True}
```

### Production WebSocket Concerns

```
SCALING WEBSOCKETS:
  WebSockets are stateful (user connected to specific server).
  Load balancer routes user to Server A.
  Server B can't send messages to users on Server A.
  
  SOLUTION: PubSub via Redis
  Server A: user connects → subscribe to Redis channel "user:123"
  Server B: wants to send message → publish to Redis channel "user:123"
  Server A: receives from Redis → sends over WebSocket to user
  
  All servers share Redis → any server can reach any user.

RECONNECTION:
  WebSockets break (mobile network change, server restart).
  Clients must implement automatic reconnection with exponential backoff.
  Server must handle duplicate connections (new connection for same user).

HEARTBEAT / PING-PONG:
  WebSocket protocol has built-in ping/pong frames.
  Server sends ping every 30s → client responds with pong.
  If no pong: connection is dead → close and clean up.
  Prevents "ghost connections" from clients that closed without proper disconnect.
```

## Server-Sent Events (SSE) (6.21)

SSE provides **one-directional server-to-client streaming** over HTTP. Simpler than WebSockets when the client doesn't need to send data.

```
SSE: Server pushes to client over HTTP (one direction)
  Client ──── GET /events ────►  Server
  Client ◄──── event stream ───── Server (infinite response)
  (client can't send back over the same connection)
  
  Good for: live feed, notifications, progress updates, AI streaming responses
```

```python
# FastAPI SSE endpoint
from fastapi import FastAPI
from fastapi.responses import StreamingResponse
import asyncio, json

app = FastAPI()

@app.get("/events/orders/{user_id}")
async def order_events(user_id: int):
    async def event_stream():
        # Subscribe to Redis pubsub or database changes
        async for event in subscribe_to_user_events(user_id):
            # SSE format: "data: <json>\n\n"
            yield f"data: {json.dumps(event)}\n\n"
            # Optional: send event type
            # yield f"event: order_update\ndata: {json.dumps(event)}\n\n"
            # Optional: send event ID (allows reconnection from last received)
            # yield f"id: {event['id']}\ndata: {json.dumps(event)}\n\n"

    return StreamingResponse(
        event_stream(),
        media_type="text/event-stream",
        headers={
            "Cache-Control": "no-cache",
            "X-Accel-Buffering": "no",  # prevent Nginx buffering
        }
    )
```

```javascript
// SSE client (browser)
const eventSource = new EventSource('/events/orders/123');

eventSource.onmessage = (event) => {
    const data = JSON.parse(event.data);
    console.log('Received:', data);
};

eventSource.addEventListener('order_update', (event) => {
    // Typed event handler
});

eventSource.onerror = () => {
    // Auto-reconnects after 3s by default
};
```

**SSE advantages over WebSockets:**
- Works over HTTP/1.1 (no upgrade needed)
- Automatic reconnection built into browser
- Works through HTTP proxies that don't support WebSockets
- Simple: one-directional push

## Long Polling vs. SSE vs. WebSockets (6.22)

```
POLLING (client pulls every N seconds):
  Client: GET /notifications?since=last_id   (every 5 seconds)
  Pros: simple, works everywhere
  Cons: unnecessary requests when no new data, delay up to polling interval

LONG POLLING (client holds connection open):
  Client sends request. Server holds it open until data arrives (or 30s timeout).
  Then client immediately sends another request.
  Pros: near-real-time, works through proxies
  Cons: 1 connection per client, complex server side

SSE (server pushes over HTTP):
  Client opens one persistent HTTP connection.
  Server pushes events as they happen.
  Pros: simple, built-in reconnection, HTTP native
  Cons: one-directional, some proxies buffer responses

WEBSOCKET (full duplex):
  Persistent bidirectional connection.
  Pros: lowest latency, bidirectional, efficient
  Cons: more complex, stateful scaling needed, proxy issues

DECISION GUIDE:
  Client only needs to receive data: SSE
  Client needs to send AND receive data: WebSocket
  Can tolerate 1-10s delay, simplicity needed: Long polling
  Can tolerate delay, no infra changes: Polling
```

## Socket.IO (6.23)

Socket.IO is a library that wraps WebSockets with fallbacks, rooms, namespaces, and automatic reconnection:

```javascript
// Socket.IO server (Node.js)
import { Server } from 'socket.io';
const io = new Server(httpServer);

io.on('connection', (socket) => {
    const userId = socket.handshake.auth.userId;
    socket.join(`user:${userId}`);  // rooms: group connections

    socket.on('chat_message', (data) => {
        // Broadcast to room
        io.to(`room:${data.roomId}`).emit('chat_message', {
            from: userId,
            message: data.message,
        });
    });

    socket.on('disconnect', () => {
        console.log('User disconnected:', userId);
    });
});

// Send from outside (e.g., from an API endpoint)
io.to(`user:${userId}`).emit('notification', { message: 'You have a new order' });
```

**Socket.IO vs. raw WebSocket:**
- Socket.IO: rooms, namespaces, automatic reconnect, fallbacks, acknowledgements
- Raw WebSocket: simpler, lighter, more control

Use Socket.IO for chat, multiplayer games, collaborative tools. Use raw WebSocket for simpler server-push scenarios.

## WebRTC (6.24)

WebRTC (Web Real-Time Communication) enables direct peer-to-peer media communication between browsers. Backend role: signaling server.

```
WEBRTC ARCHITECTURE:
  Browser A  ←── signaling (offer/answer) ──→  Signaling Server (WebSocket)
  Browser A  ←── direct media stream ─────────→  Browser B
  (video/audio goes peer-to-peer, NOT through your server)

  STUN server: helps peers discover their public IP (NAT traversal)
  TURN server: relay server when direct P2P is impossible (NAT/firewall)

YOUR BACKEND ROLE:
  - Signaling server: relay SDP offers/answers via WebSocket
  - TURN server: relay media when direct connection fails
  - Session management: who is in which call

USE CASES:
  Video calls (Zoom-like), voice calls, screen sharing
  P2P file transfer, real-time collaboration

WHEN NOT WEBRTC:
  If you need server-side recording/processing: use a media server (Janus, Mediasoup)
  Simple notifications/chat: WebSocket is simpler
```
