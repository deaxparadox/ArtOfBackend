## MQTT (6.25): IoT Messaging Protocol

MQTT (Message Queuing Telemetry Transport) is a lightweight publish/subscribe protocol designed for devices with limited bandwidth and battery (IoT, sensors, embedded systems).

```
MQTT CONCEPTS:
  Broker:      Central message router (Mosquitto, HiveMQ, AWS IoT Core)
  Publisher:   Sends messages to a topic
  Subscriber:  Receives messages from topics
  Topic:       Hierarchical identifier: "sensors/building1/floor3/temperature"

QoS LEVELS:
  0: At most once  — fire and forget, possible loss
  1: At least once — guaranteed delivery, possible duplicates  
  2: Exactly once  — guaranteed exactly once, most expensive

TOPIC WILDCARDS:
  sensors/+/temperature   → + = one level wildcard (any building)
  sensors/#               → # = multi-level wildcard (all subtopics)
```

```python
import paho.mqtt.client as mqtt

# MQTT subscriber (IoT data collection)
def on_message(client, userdata, message):
    topic = message.topic
    payload = message.payload.decode()
    print(f"Topic: {topic}, Data: {payload}")
    # Store in time-series database

client = mqtt.Client()
client.on_message = on_message
client.connect("mqtt.example.com", 1883)
client.subscribe("sensors/#")  # all sensor data
client.loop_forever()

# MQTT publisher (a sensor device or simulator)
client.publish("sensors/building1/floor3/temperature", "23.5", qos=1)
```

**When MQTT:** IoT devices, sensor networks, home automation (Home Assistant uses MQTT). **Not for:** regular web APIs (use REST/WebSocket).

## AMQP (6.26): Advanced Message Queuing Protocol

AMQP is the wire protocol behind RabbitMQ (see Phase 12 for RabbitMQ in depth). Understanding AMQP explains RabbitMQ's routing model.

```
AMQP CONCEPTS:
  Exchange:  Receives messages and routes to queues
  Queue:     Stores messages until consumed
  Binding:   Rule linking exchange to queue (based on routing key)
  
  EXCHANGE TYPES:
  Direct:  Route to queue with exact matching routing key
  Topic:   Route by pattern match (*.error, logs.#)
  Fanout:  Broadcast to all bound queues
  Headers: Route by message header attributes

AMQP vs MQTT:
  AMQP: enterprise messaging (RabbitMQ, ActiveMQ), complex routing
  MQTT: IoT, lightweight, topic hierarchies, simple pub/sub
```

## tRPC (6.27): End-to-End TypeScript Type Safety

tRPC is a TypeScript-first RPC framework — no schema definition file, no code generation. The API contract IS the TypeScript types.

```typescript
// Server (Next.js or standalone Node.js)
import { initTRPC } from '@trpc/server';
import { z } from 'zod';

const t = initTRPC.create();
const router = t.router;
const publicProcedure = t.procedure;

export const appRouter = router({
    // Typed: input validated by Zod, output inferred from return type
    getUser: publicProcedure
        .input(z.object({ id: z.number() }))
        .query(async ({ input }) => {
            const user = await db.getUser(input.id);
            return user;  // TypeScript infers the return type
        }),

    createUser: publicProcedure
        .input(z.object({
            email: z.string().email(),
            name: z.string().min(1),
        }))
        .mutation(async ({ input }) => {
            return await db.createUser(input);
        }),
});

export type AppRouter = typeof appRouter;  // exported for client
```

```typescript
// Client (auto-complete, type-safe, no manual type definitions)
import { createTRPCProxyClient, httpBatchLink } from '@trpc/client';
import type { AppRouter } from './server';

const client = createTRPCProxyClient<AppRouter>({
    links: [httpBatchLink({ url: '/api/trpc' })],
});

// TypeScript KNOWS the return type is User — no manual typing
const user = await client.getUser.query({ id: 123 });
// user.email, user.name — fully typed, autocomplete works

// Type error caught at compile time:
await client.createUser.mutate({ email: 'not-an-email' });
// TypeScript error: "email" must be a valid email
```

**tRPC use case:** Full-stack TypeScript apps where frontend and backend share the same TypeScript types. Especially powerful with Next.js (same codebase). **Not for:** External APIs consumed by third parties (gRPC/REST has broader language support).

## JSON-RPC (6.28)

JSON-RPC is a lightweight remote procedure call protocol using JSON. Very simple — specify method name and parameters:

```json
// Request
{"jsonrpc": "2.0", "method": "getUser", "params": {"id": 123}, "id": 1}

// Response
{"jsonrpc": "2.0", "result": {"id": 123, "name": "Alice"}, "id": 1}

// Error
{"jsonrpc": "2.0", "error": {"code": -32601, "message": "Method not found"}, "id": 1}

// Batch: multiple calls in one request
[
  {"jsonrpc": "2.0", "method": "getUser", "params": {"id": 1}, "id": 1},
  {"jsonrpc": "2.0", "method": "getUser", "params": {"id": 2}, "id": 2}
]
```

**Used by:** Ethereum blockchain APIs (web3.py, ethers.js), Solana, Bitcoin Core, some internal enterprise services. Not common for new REST-style APIs.

## Thrift (6.29) and MessagePack (6.30)

**Thrift (Apache Thrift, Facebook 2007):**

An alternative to Protocol Buffers. Similar concept: define schema in .thrift file, generate code in multiple languages. Less popular than gRPC/protobuf today but found in older systems (Cassandra, HBase use Thrift interfaces).

```thrift
// user.thrift
struct User {
    1: i64 id,
    2: string email,
    3: string name,
}

service UserService {
    User getUser(1: i64 user_id),
    list<User> listUsers(1: i32 limit, 2: i32 offset),
}
```

**MessagePack:**

Binary serialization format like protobuf but without a schema (schema-less). JSON compatible (same data model) but 2-4x smaller and faster.

```python
import msgpack

# Encode (like json.dumps but binary)
data = {"user_id": 123, "email": "alice@example.com", "score": 9.5}
packed = msgpack.packb(data, use_bin_type=True)
print(len(packed))    # 42 bytes vs 57 bytes JSON

# Decode
unpacked = msgpack.unpackb(packed, raw=False)

# Use case: Redis values, internal service communication, anywhere JSON is too slow/large
# Without a schema: easier than protobuf but less type safety
```

**MessagePack vs. Protobuf:**
- MessagePack: schema-free (flexible, no .proto files), slightly slower
- Protobuf: schema-required (safer, better tooling), faster
