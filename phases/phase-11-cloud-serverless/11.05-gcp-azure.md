## 1. Google Cloud Platform (11.21-11.27)

GCP (Google Cloud Platform) is the cloud that Google built to run its own services — Search, YouTube, Gmail, Google Maps. The underlying infrastructure is genuinely impressive: Google's global fiber network (Andromeda SDN), the Borg scheduler that inspired Kubernetes, Spanner (globally consistent database), and Bigtable (the original NoSQL that inspired HBase and Cassandra).

**GCP's key differentiators:**
- Data analytics (BigQuery is unmatched for serverless SQL analytics)
- Machine learning (TPUs, Vertex AI, the original TensorFlow)
- Kubernetes expertise (K8s was born at Google — GKE is the most mature managed K8s)
- Networking (Google's global backbone is exceptional)

### Cloud Run: The Best Serverless Container Platform (11.21)

Cloud Run is the standout GCP service that many consider the best serverless container platform available. It runs any container with an HTTP server, scales to zero, and charges only for request-handling time.

```
CLOUD RUN vs LAMBDA:

LAMBDA:
  - Run any function (arbitrary code, not just HTTP)
  - 15-minute max execution time
  - 10GB max ephemeral storage
  - Complex event integrations (S3, DynamoDB streams, etc.)
  - Vendor-specific: your code must fit Lambda's event model

CLOUD RUN:
  - Run any container with an HTTP server
  - No vendor lock-in: your container runs anywhere
  - 60-minute max request timeout
  - Same code runs locally (docker run) and in production
  - Simpler mental model: if it works as a container, it works on Cloud Run

CLOUD RUN SWEET SPOT:
  - Containerized APIs that need to scale to zero (pay-per-request billing)
  - Batch processing triggered by HTTP
  - Simple microservices that don't need complex AWS Lambda event sources
```

```yaml
# Deploy to Cloud Run via gcloud CLI
gcloud run deploy myapi \
  --image gcr.io/myproject/myapi:v1.2.3 \
  --region us-central1 \
  --platform managed \
  --allow-unauthenticated \
  --port 8000 \
  --memory 512Mi \
  --cpu 1 \
  --min-instances 0 \          # scale to zero
  --max-instances 100 \        # scale up to 100
  --concurrency 80 \           # 80 requests per container instance
  --set-env-vars "DATABASE_URL=$(cat .db_url)" \
  --set-secrets "JWT_SECRET=jwt-secret:latest"
```

**Cloud Run pricing:** ~$0.00002400 per vCPU-second + $0.00000250 per GB-second. At zero traffic: $0. This makes it genuinely cost-effective for APIs with variable traffic.

### BigQuery: Serverless Analytics (11.25)

BigQuery is GCP's serverless data warehouse. It processes petabytes of data using SQL with no infrastructure to manage.

```sql
-- BigQuery query: process 10 years of event data in seconds
-- No indexes to create, no partitions to tune manually

SELECT
    DATE_TRUNC(event_timestamp, MONTH) AS month,
    country,
    event_type,
    COUNT(*) AS events,
    COUNT(DISTINCT user_id) AS unique_users
FROM `myproject.analytics.events`
WHERE event_timestamp >= '2020-01-01'
  AND event_type IN ('purchase', 'signup', 'view_product')
GROUP BY 1, 2, 3
ORDER BY 1 DESC, 4 DESC

-- BigQuery executes this against 100TB of data in ~10 seconds
-- Cost: $5 per TB scanned (on-demand) — this query might cost $0.50
-- No server to provision, no cluster to manage
```

**BigQuery use cases:**
- Analytics on large datasets (event logs, user behavior, financial data)
- Ad-hoc SQL queries on petabyte-scale data
- Exporting aggregated data for business intelligence (Looker, Data Studio)
- Machine learning on large datasets (BigQuery ML)

### GKE: The Most Mature Managed Kubernetes (11.22)

GKE was built by the team that created Kubernetes. It remains the reference implementation for managed Kubernetes. Features like Autopilot (fully managed node management), integration with Cloud Armor (WAF), and native GCP workload identity make GKE the benchmark other managed K8s providers follow.

## 2. Microsoft Azure (11.28-11.31)

Azure (Microsoft, 2010) is the dominant cloud for enterprise organizations using Microsoft technologies. Its strongest differentiators: Active Directory integration (Azure AD), hybrid cloud scenarios (Azure Arc), and enterprise compliance certifications.

### Azure Functions (11.28): Serverless in the Microsoft Ecosystem

```csharp
// Azure Functions: serverless functions for .NET, Node.js, Python, Java
using Microsoft.Azure.Functions.Worker;
using Microsoft.Extensions.Logging;

public class OrderProcessor
{
    private readonly ILogger<OrderProcessor> _logger;

    [Function("ProcessOrder")]
    public async Task Run(
        [ServiceBusTrigger("orders-queue", Connection = "ServiceBusConnection")]
        string message,   // automatically deserializes from Azure Service Bus
        FunctionContext context)
    {
        _logger.LogInformation("Processing: {Message}", message);
        var order = JsonSerializer.Deserialize<Order>(message);
        await processOrderAsync(order);
    }
    
    // HTTP trigger
    [Function("GetOrder")]
    public IActionResult GetOrder(
        [HttpTrigger(AuthorizationLevel.Function, "get", Route = "orders/{id}")] 
        HttpRequest req,
        string id)
    {
        var order = orderRepository.GetById(id);
        return order is null ? new NotFoundResult() : new OkObjectResult(order);
    }
}
```

### Azure Cosmos DB (11.29): Globally Distributed NoSQL

CosmosDB is Azure's globally distributed multi-model database. It guarantees single-digit millisecond reads and writes globally (by keeping data close to users in chosen regions) with a 99.999% SLA.

```
COSMOS DB KEY CONCEPTS:
  API choices: SQL (document), MongoDB, Cassandra, Gremlin (graph), Table
  Partitioning: data distributed by partition key
  Consistency levels: 5 levels (Strong, Bounded Staleness, Session, Consistent Prefix, Eventual)
  
  RU (Request Units): abstract billing unit
  Reading 1KB document ≈ 1 RU
  Writing 1KB document ≈ 5 RUs
  You provision RUs/second or use autoscale (scales 10×-100× automatically)
  
  GLOBAL DISTRIBUTION:
  Enable multi-region with one click.
  Each write → replicated to all regions asynchronously.
  Client SDK routes reads to nearest region.
  
  USE COSMOS DB WHEN:
  ✓ Global user base needing local reads/writes
  ✓ Already deep in Azure ecosystem
  ✓ Need Cassandra/MongoDB compatibility on Azure
  ✗ CosmosDB is expensive for high-volume operations
  ✗ Don't use for: relational data (use Azure SQL Database instead)
```

## 3. Choosing a Cloud Provider

```
AWS:
  ✓ Largest service catalog (200+ services)
  ✓ Largest market share (~32%) → most engineers know it
  ✓ Best startup credits and ecosystem
  ✓ Deepest global reach (33 regions)
  ✗ Most complex (too many services, steep learning curve)
  Best for: startups, new projects, maximum flexibility

GCP:
  ✓ Best for data analytics (BigQuery is unmatched)
  ✓ Best Kubernetes expertise (GKE, Autopilot)
  ✓ Cloud Run is genuinely excellent for containers
  ✓ Best ML/AI services (Vertex AI, TPUs)
  ✗ Smaller ecosystem than AWS
  ✗ Enterprise sales less mature than AWS/Azure
  Best for: data-heavy workloads, ML, Kubernetes-first shops

AZURE:
  ✓ Best for Microsoft-stack enterprises (Active Directory, Office 365)
  ✓ Hybrid cloud (Azure Arc, ExpressRoute)
  ✓ Best enterprise compliance certifications
  ✓ Strong for .NET applications
  ✗ Complex pricing
  ✗ Console UX is often criticized
  Best for: enterprises with Microsoft agreements, hybrid scenarios

MULTI-CLOUD:
  In theory: avoid vendor lock-in, use best-of-breed services
  In practice: doubles operational complexity, duplicates training costs
  When it makes sense: regulatory requirements, M&A, specific service gaps
  
  Most teams: pick one cloud, go deep, become excellent at it.
  "Multi-cloud by accident" (one team chose AWS, another chose GCP) is the
  most common scenario — and the most painful.
```

## 4. Senior Engineer Perspective

**The AWS vs. GCP vs. Azure honest assessment:**

For most new projects: AWS is the safe choice — largest ecosystem, most engineers know it, most consulting help available. GCP wins for data engineering and ML workloads. Azure wins for enterprises with Microsoft agreements.

The key insight: **cloud provider choice is mostly a team expertise and cost decision**, not a technical correctness decision. A great team on GCP will outperform a mediocre team on AWS. Choose the cloud your team knows best or can learn fastest.

**Cloud Run deserves more attention than it gets:**

The combination of "any container with an HTTP server" + "scale to zero" + "no cold starts for warm instances" is genuinely compelling. Teams spending significant money on ECS Fargate for services with variable traffic should evaluate Cloud Run. The operational simplicity (no task definitions, no services, no load balancer config) often reduces deployment complexity dramatically.
