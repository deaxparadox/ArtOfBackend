## 1. Serverless Fundamentals (11.32)

"Serverless" is a deployment model, not an absence of servers. There are always servers — you just don't provision, manage, or think about them. You write code; the platform handles execution, scaling, and operations.

**The core tradeoff:**

```
TRADITIONAL SERVERS / CONTAINERS:
  You control: OS, runtime, scaling, availability
  You pay for: always-on capacity (even at zero traffic)
  You manage: patches, upgrades, cluster configuration
  Latency: consistently low (no cold start)
  
SERVERLESS:
  Platform controls: OS, runtime, scaling
  You pay for: actual execution time (near-zero cost at zero traffic)
  Platform manages: everything operational
  Latency: variable (cold start on idle functions/containers)

WHEN SERVERLESS WINS:
  ✓ Variable or unpredictable traffic (pay only for what you use)
  ✓ Event-driven workflows (image processing, document parsing, notifications)
  ✓ API backends for applications with low-to-medium sustained traffic
  ✓ Scheduled tasks (crons, batch jobs)
  ✓ Webhooks and integrations
  ✓ Teams that want to minimize operational burden

WHEN SERVERLESS DOESN'T WIN:
  ✗ High, sustained traffic (containers are cheaper at scale)
  ✗ Latency-sensitive real-time applications (cold starts are unacceptable)
  ✗ Long-running processes (Lambda max: 15 minutes)
  ✗ Stateful services (serverless is stateless by design)
  ✗ Complex local development needs
```

## 2. The Cold Start Problem (11.33)

Cold starts are the central challenge of serverless architectures. Understanding them helps you architect around them.

**What causes a cold start:**

```
LAMBDA COLD START BREAKDOWN:
  
  1. Firecracker microVM initialization (~125ms)
     AWS spins up a minimalist VM for your function
  
  2. Language runtime initialization
     Python: ~150ms (import Python interpreter + stdlib)
     Node.js: ~100ms
     Go: ~50ms (statically compiled)
     Java JVM: ~500ms-5s (JVM startup, class loading, JIT warmup)
  
  3. Your package download and initialization
     Small packages (< 5MB): ~50ms
     Large packages (50MB+): ~500ms+
     Model loading in ML functions: can be minutes
  
  4. Your module-level code execution
     Database connection establishment
     Configuration loading
     Any code outside your handler function
  
WARM INVOCATIONS:
  Same container reused → only handler code runs
  Latency: just your handler execution time
  
THE SPIKE PROBLEM:
  Traffic pattern: 0 → 1000 req/s suddenly (product goes viral)
  Lambda must create 1000 containers simultaneously
  All 1000 are cold starts
  Your p99 latency spikes to cold start time × processing time
  Users experience this as the service being "slow"
```

**Cold start mitigation strategies:**

```python
# STRATEGY 1: Minimize your package size
# Lambda loads your deployment package on cold start.
# Every MB adds ~5-10ms to cold start.

# .lambda_ignore (similar to .dockerignore)
# Exclude: tests, documentation, dev dependencies, .pyc caches
# Use Lambda layers for large shared dependencies

# STRATEGY 2: Initialize expensive resources globally (not in handler)
# GOOD: initialized on cold start, reused on warm invocations
dynamodb = boto3.resource('dynamodb')
table = dynamodb.Table(os.environ['TABLE_NAME'])
s3 = boto3.client('s3')

# Lazy connection: establish on first invocation, reuse after
_db_connection = None
def get_db():
    global _db_connection
    if _db_connection is None or not _db_connection.is_connected():
        _db_connection = create_connection()
    return _db_connection

def lambda_handler(event, context):
    conn = get_db()  # reuses existing connection
    # process event...

# STRATEGY 3: Provisioned Concurrency (guaranteed warm instances)
# AWS keeps N instances permanently initialized
# Cost: you pay per hour for provisioned concurrency
# Typical: provision during business hours, let scale to zero at night
# Configure in Terraform or AWS Console per function

# STRATEGY 4: Use runtimes with fast cold starts
# Go and Rust: <100ms cold starts
# Python: ~200-400ms (acceptable for most APIs)
# Java JVM: 3-10s (unacceptable for user-facing APIs → use SnapStart or GraalVM)

# STRATEGY 5: Keep functions warm (lightweight, works for low traffic)
# Schedule EventBridge rule to invoke function every 5 minutes
# Prevents the function from going completely cold
# Caveat: doesn't help during traffic spikes where you need many instances
```

## 3. Cloudflare Workers: Edge Computing (11.35-11.36)

Cloudflare Workers are JavaScript (or WASM) functions that run at Cloudflare's edge network — 300+ cities globally. They execute in V8 isolates (not containers), giving them:
- Near-zero cold starts (<1ms)
- Execution within milliseconds of your users
- 50ms CPU time limit (designed for fast edge logic, not heavy processing)

**The mental model for Cloudflare Workers:**

```
TRADITIONAL BACKEND:
  User in Tokyo → 150ms → your server in Virginia → response

CLOUDFLARE WORKERS:
  User in Tokyo → 5ms → Tokyo edge node (running your worker code) → response
  
  Workers run IN the CDN — not next to your origin server.
  The user never needs to reach Virginia for cacheable or logic-only operations.
```

```javascript
// Cloudflare Worker: runs at edge, not in your datacenter
// Every request to your domain goes through this code at the nearest PoP

export default {
    async fetch(request, env, ctx) {
        const url = new URL(request.url);

        // 1. Auth at the edge — validate JWT without hitting origin
        if (url.pathname.startsWith('/api/')) {
            const token = request.headers.get('Authorization')?.slice(7);
            
            if (!token) {
                return new Response('Unauthorized', { status: 401 });
            }
            
            // Verify JWT using Web Crypto API (available in Workers)
            const payload = await verifyJWT(token, env.JWT_SECRET);
            if (!payload) {
                return new Response('Invalid token', { status: 401 });
            }
            
            // Add verified user info to request for origin
            const newHeaders = new Headers(request.headers);
            newHeaders.set('X-User-Id', payload.sub);
            newHeaders.set('X-User-Role', payload.role);
            
            const modifiedRequest = new Request(request, { headers: newHeaders });
            return fetch(modifiedRequest);  // forward to origin
        }
        
        // 2. A/B testing at edge — no origin round trip needed
        if (url.pathname === '/home') {
            const userId = request.headers.get('X-User-Id') || 'anonymous';
            const variant = userId.charCodeAt(0) % 2 === 0 ? 'A' : 'B';
            
            url.searchParams.set('variant', variant);
            return fetch(new Request(url.toString(), request));
        }
        
        // 3. Geo-based routing
        const country = request.cf?.country;
        if (country === 'DE' || country === 'FR') {
            // Route European users to EU servers
            const euOrigin = 'https://eu.api.example.com';
            return fetch(new Request(euOrigin + url.pathname, request));
        }
        
        return fetch(request);  // default: forward to origin
    }
}
```

### Cloudflare's Edge Data Primitives (11.36)

```javascript
// KV: globally distributed key-value store
// Eventually consistent (async replication to all PoPs)
// Good for: user preferences, feature flags, rate limit counters

const value = await env.MY_KV.get('feature:dark-mode');
await env.MY_KV.put('session:abc123', JSON.stringify(userData), {
    expirationTtl: 3600  // 1 hour TTL
});

// R2: S3-compatible object storage with zero egress fees
// Exactly like S3 API, but no $0.09/GB egress charges
const object = await env.MY_BUCKET.get('images/user-123/avatar.jpg');
await env.MY_BUCKET.put('uploads/file.pdf', fileData, {
    httpMetadata: { contentType: 'application/pdf' }
});

// D1: SQLite databases at the edge
// ACID-compliant, SQL queries, runs in Workers
const result = await env.DB.prepare(
    "SELECT * FROM users WHERE email = ?"
).bind(email).first();

// Durable Objects: stateful singletons at the edge
// Use for: collaborative editing, real-time game state, chat rooms
// Each Durable Object is a single-threaded actor with its own storage
export class ChatRoom {
    constructor(state, env) {
        this.state = state;
        this.connections = new Set();
    }
    
    async fetch(request) {
        const webSocketPair = new WebSocketPair();
        const [client, server] = Object.values(webSocketPair);
        
        this.state.acceptWebSocket(server);
        this.connections.add(server);
        
        return new Response(null, { status: 101, webSocket: client });
    }
    
    async webSocketMessage(ws, message) {
        // Broadcast to all connections in this room
        for (const conn of this.connections) {
            if (conn.readyState === 1) {
                conn.send(message);
            }
        }
    }
}
```

## 4. Fly.io: Deploy Containers Globally (11.37)

Fly.io runs Docker containers globally on their own network, close to users. Unlike Lambda or Cloud Run, you bring a container — any container. Unlike traditional hosting, Fly automatically places your containers near users.

```bash
# Deploy any Docker container globally with Fly
fly launch --image myapp:latest

# fly.toml configuration
app = "my-api"

[build]
  image = "registry.example.com/myapp:v1.2.3"

[env]
  LOG_LEVEL = "info"

[[services]]
  internal_port = 8000
  protocol = "tcp"

  [[services.ports]]
    port = 443
    handlers = ["tls", "http"]

  [services.concurrency]
    type = "requests"
    hard_limit = 250
    soft_limit = 200

  [[services.tcp_checks]]
    interval = "10s"
    timeout = "2s"
    grace_period = "5s"

[[regions]]
  # Deploy to these regions
  primary = "iad"        # Northern Virginia
  backup = ["lhr", "sin"]  # London, Singapore
```

**Fly.io sweet spot:** Applications that need to run close to users globally but want the simplicity of "give me a container, I handle the rest." Between Lambda's AWS lock-in and Kubernetes' complexity.

## 5. Railway and Render: Simplest Deployment Platforms (11.38)

For teams who want to deploy without dealing with AWS, Kubernetes, or any infrastructure configuration:

```
RAILWAY:
  Connect GitHub repo → Railway deploys automatically on every push
  Supports: Node.js, Python, Go, Ruby, Docker
  Databases: managed PostgreSQL, MySQL, Redis, MongoDB
  Cost: ~$5/month for small services
  Best for: startups, side projects, MVP validation
  
RENDER:
  Similar to Railway but with free tier
  Web services, background workers, cron jobs, static sites
  Managed databases (PostgreSQL)
  Cost: free tier → $7/month for always-on services
  Best for: indie developers, small teams

BOTH:
  ✓ Zero DevOps needed (no Terraform, no Docker, no K8s)
  ✓ Automatic TLS, custom domains
  ✓ Preview environments for pull requests
  ✗ Limited control compared to AWS/GCP
  ✗ More expensive per compute unit at scale
  ✗ Not suitable for regulated industries (limited compliance certifications)
```

## 6. The Trade-off Matrix (11.39)

```
DEPLOYMENT OPTION COMPARISON:

                Complexity   Cost at low   Cost at high   Cold Start   Control
                            traffic       traffic
EC2             Low-Medium   High         Low            None         High
ECS Fargate     Medium       Medium       Medium         None         Medium
Kubernetes      High         Medium       Low            None         Very High
Lambda          Low          Very low     Medium-High    Yes          Low
Cloud Run       Low          Very low     Medium         Minimal      Medium
Cloudflare Wkr  Medium       Very low     Very low       None         Medium
Railway/Render  Very Low     Low-Medium   Expensive      None         Low

DECISION GUIDE:
  Traffic: Very spiky (0 → burst) → Lambda / Cloud Run (pay for what you use)
  Traffic: Steady medium → ECS Fargate (predictable, no cold starts)
  Traffic: Very high (>10k req/s sustained) → ECS + Reserved, or Kubernetes
  Team: Small, no DevOps → Railway / Render / Cloud Run
  Latency: Sub-10ms needed → Cloudflare Workers (edge, no cold starts)
  Control: Full OS control needed → EC2
```

## 7. Cost Modeling: Avoiding Bill Shock (11.40)

The cloud's pay-per-use model can produce surprises. Always model costs before committing to an architecture.

```python
# Monthly cost estimation for a typical API

def estimate_monthly_cost(
    requests_per_day: int,
    avg_duration_ms: float,
    avg_response_kb: float,
    memory_mb: int = 512,
) -> dict:
    """
    Estimate AWS Lambda + API Gateway monthly cost.
    """
    requests_per_month = requests_per_day * 30
    
    # Lambda pricing (us-east-1, 2024)
    lambda_request_cost = requests_per_month * 0.0000002  # $0.20 per 1M requests
    gb_seconds = (memory_mb / 1024) * (avg_duration_ms / 1000) * requests_per_month
    lambda_duration_cost = gb_seconds * 0.0000166667  # $0.0000166667 per GB-second
    
    # API Gateway (HTTP API) pricing
    api_gateway_cost = requests_per_month * 0.000001  # $1.00 per 1M requests
    
    # Data transfer
    transfer_gb = (avg_response_kb / 1024) * requests_per_month / 1024
    data_transfer_cost = max(0, transfer_gb - 1) * 0.09  # $0.09/GB after first 1GB
    
    total = lambda_request_cost + lambda_duration_cost + api_gateway_cost + data_transfer_cost
    
    return {
        "requests_per_month": f"{requests_per_month:,}",
        "lambda_requests": f"${lambda_request_cost:.2f}",
        "lambda_duration": f"${lambda_duration_cost:.2f}",
        "api_gateway": f"${api_gateway_cost:.2f}",
        "data_transfer": f"${data_transfer_cost:.2f}",
        "total_monthly": f"${total:.2f}",
    }

# Example: 100,000 requests/day, 200ms avg, 5KB avg response
print(estimate_monthly_cost(100_000, 200, 5))
# Lambda requests: $0.60
# Lambda duration (512MB, 200ms): $10.00
# API Gateway: $3.00
# Data transfer: ~$0
# Total: ~$13.60/month

# Comparison: t3.micro EC2 + ALB running 24/7
# EC2: $8.47/month
# ALB: ~$18/month
# Total: ~$26.47/month
# Lambda is cheaper for this workload!

# But at 10M requests/day:
# Lambda: ~$1,360/month
# EC2 (3× t3.medium + ALB): ~$120/month
# Lambda is 11× more expensive at high scale
```

**Key cost optimization tips:**

```
FREE TIER (12 months new AWS accounts + always free):
  Lambda: 1M requests/month + 400,000 GB-seconds
  S3: 5GB storage + 20,000 GET requests + 2,000 PUT requests
  DynamoDB: 25GB storage + 25 RCU + 25 WCU
  CloudFront: 1TB data transfer + 10M HTTP requests

COST TRAPS TO AVOID:
  1. NAT Gateway data processing: $0.045/GB. Use S3/DynamoDB VPC endpoints.
  2. Data transfer between AZs: $0.01/GB. Keep services in same AZ where possible.
  3. CloudWatch Logs storage: logs accumulate. Set retention policies.
  4. Unused Elastic IPs: $0.005/hour if not attached. Release them.
  5. RDS Multi-AZ: double the cost. Necessary for production, overkill for dev.
  6. Lambda concurrency limits: if you don't set reserved concurrency, one runaway
     function can consume all account concurrency, starving other functions.

COST MONITORING:
  □ AWS Cost Explorer: see spending by service, by tag, over time
  □ AWS Budgets: alert when spending exceeds threshold
  □ Cost Anomaly Detection: ML-based unusual spending alerts
  □ Resource tagging: tag everything by team/service/environment
     (enables cost attribution: "team X is spending $Y/month")
```
