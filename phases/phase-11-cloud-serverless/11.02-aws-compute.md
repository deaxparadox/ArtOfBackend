## 1. EC2: Virtual Machines (11.2)

EC2 (Elastic Compute Cloud) is AWS's virtual machine service. You get a virtual server with a chosen OS, CPU, memory, and disk — and you pay by the hour or second.

**When to use EC2:**
- Long-running services that aren't containerized
- Stateful applications that need specific OS configurations
- When you need full control over the underlying system
- Databases (though RDS is usually better)

### Instance Families and Types

```
INSTANCE NAMING: [family][generation].[size]
  c7g.2xlarge
  │││ └──────── size (nano, micro, small, medium, large, xlarge, 2xlarge, 4xlarge...)
  │└┘────────── processor generation (higher = newer, faster, cheaper per unit)
  └──────────── family (determines optimized characteristic)

FAMILIES:
  t4g, t3:    General purpose, burstable (t = "turbo" burst)
              Best for: dev/staging, variable load, small production services
              Cost: cheapest option (~$0.01-0.05/hr)
              
  m7g, m6i:  General purpose, balanced compute/memory (m = "medium")
              Best for: most production web servers, caches
              
  c7g, c6i:  Compute optimized (c = "compute")
              Best for: CPU-heavy work (media encoding, HPC, gaming)
              
  r7g, r6i:  Memory optimized (r = "ram")
              Best for: databases, caches, in-memory analytics (Redis, large Postgres)
              
  p4, p3:    GPU instances (p = "parallel")
              Best for: ML training, inference
              
  i4i, i3:   Storage optimized (i = "I/O")
              Best for: NoSQL databases, data warehousing, distributed filesystems

PROCESSORS:
  [no suffix]: Intel
  a or g:      AWS Graviton (ARM-based)
  
  Graviton (ARM) ADVANTAGE:
  20% cheaper per unit + typically 20-40% better performance-per-dollar
  vs equivalent Intel instance. Most workloads run on ARM unchanged.
  Always evaluate Graviton first for new workloads.
```

### Pricing Models

```
ON-DEMAND:
  Pay per second (Linux) or per hour (Windows).
  No commitment. Highest price.
  Use for: dev, unpredictable workloads, short-term needs.
  Price example: m5.large = $0.096/hr (~$70/month)

RESERVED INSTANCES (1 or 3 year commitment):
  30-70% discount vs on-demand.
  Best for: stable, predictable production workloads.
  Price example: m5.large, 1yr, no upfront = $0.062/hr (~$45/month, 35% savings)

SPOT INSTANCES:
  70-90% discount. AWS can reclaim with 2-minute notice.
  Best for: stateless batch jobs, fault-tolerant workers, dev environments.
  Mechanism: you bid on unused capacity. If spot price > your bid → terminated.
  Price example: m5.large spot = ~$0.029/hr (~$21/month, 70% savings)

SAVINGS PLANS (modern alternative to Reserved):
  Commit to a specific $/hour spend across any instance type.
  More flexible than Reserved Instances (change instance type freely).
  1-year or 3-year commitment.
```

### Key EC2 Concepts

```
AMI (Amazon Machine Image):
  A snapshot of an OS + software configuration.
  Like a Docker image but for VMs.
  AWS provides: Amazon Linux 2023, Ubuntu 22.04, Windows Server...
  You can create custom AMIs with pre-installed software.

EBS (Elastic Block Store):
  Persistent block storage attached to EC2 (like a hard drive).
  Survives instance stops/starts.
  Types: gp3 (general SSD), io2 (high IOPS SSD), st1 (throughput HDD)
  gp3 is the default and right choice for most workloads.

Instance Store:
  Physically attached storage — much faster than EBS.
  NOT persistent: data is lost when instance stops or terminates.
  Use for: temporary data, buffers, caches.

Security Groups:
  Stateful firewall for EC2 instances. (See topic 1.20)
```

## 2. Lambda: Serverless Functions (11.8)

Lambda is AWS's serverless compute service. You provide a function — a piece of code — and AWS runs it in response to events. You pay only for the compute time your function uses, in 1-millisecond increments.

**The paradigm shift Lambda represents:**

Traditional computing: you provision a server, it runs 24/7, you pay whether it has work or not. Lambda: you upload code, AWS runs it only when triggered, you pay only for the milliseconds it executes. A Lambda function processing 1 million requests at 100ms each costs approximately $0.20 per month. The equivalent EC2 instance running 24/7 costs $15-50/month.

### How Lambda Actually Works

```
LAMBDA EXECUTION MODEL:

COLD START (first invocation or after idle period):
  1. AWS allocates a micro-VM (Firecracker VM, ~125ms initialization)
  2. Lambda service downloads your deployment package (~50-500ms)
  3. Container starts, runtime initialized (~50ms for Python/Node, 500ms-5s for Java)
  4. Your function handler is called
  Total cold start: 100ms (Go/Rust) to 5-10s (Java JVM)

WARM START (subsequent invocations within ~5-15 minutes):
  1. The same container from the previous invocation is reused
  2. Your handler is called immediately
  Total: function execution time only (your code, no initialization overhead)

CONTAINER REUSE IMPLICATIONS:
  Global variables and connections initialized outside the handler
  PERSIST between warm invocations:
  
  # This code runs only on cold start:
  import psycopg2
  connection = psycopg2.connect(os.environ['DATABASE_URL'])  # initialized once
  
  # This runs on every invocation:
  def handler(event, context):
      cursor = connection.cursor()  # reuses the persistent connection
      cursor.execute("SELECT ...")
      return result
  
  Initialize expensive resources (DB connections, loaded ML models) globally.
  They'll be reused across warm invocations.
```

### Lambda Configuration

```python
# lambda_function.py

import json
import os
import boto3

# Initialized once on cold start (shared across warm invocations)
s3_client = boto3.client('s3')
dynamodb = boto3.resource('dynamodb')
table = dynamodb.Table(os.environ['TABLE_NAME'])

def lambda_handler(event, context):
    """
    event:    The trigger event (API Gateway request, S3 event, SQS message, etc.)
    context:  Lambda runtime info (remaining time, function name, memory limit, etc.)
    
    Returns: For API Gateway: {"statusCode": 200, "body": "..."}
             For SQS: None or {"batchItemFailures": [...]}
             For async: anything (or nothing)
    """
    
    # Remaining execution time
    if context.get_remaining_time_in_millis() < 1000:
        # Less than 1 second left — fail gracefully rather than timing out
        return {"statusCode": 503, "body": "Function near timeout"}
    
    # Handle API Gateway proxy event
    if event.get('httpMethod'):
        path = event['path']
        method = event['httpMethod']
        body = json.loads(event.get('body') or '{}')
        
        # Your business logic here
        result = process_request(path, method, body)
        
        return {
            "statusCode": 200,
            "headers": {
                "Content-Type": "application/json",
                "Access-Control-Allow-Origin": "*",
            },
            "body": json.dumps(result),
        }
    
    # Handle SQS event
    if 'Records' in event and event['Records'][0].get('eventSource') == 'aws:sqs':
        failed_items = []
        for record in event['Records']:
            try:
                message = json.loads(record['body'])
                process_message(message)
            except Exception as e:
                # Report individual failures (partial batch success)
                failed_items.append({"itemIdentifier": record['messageId']})
        
        return {"batchItemFailures": failed_items}
```

### Lambda Cold Start: The Root Problem

```
COLD START TIMES BY RUNTIME (approximate):
  Go (Lambda's provided.al2):  ~50-100ms
  Python 3.12:                 ~200-400ms  ← acceptable for most use cases
  Node.js 20:                  ~100-300ms
  Ruby:                        ~200-400ms
  Java 21 (JVM):               3-10 seconds  ← often unacceptable
  Java 21 (GraalVM native):    ~100ms        ← acceptable

COLD START MITIGATION:
  1. Provisioned Concurrency: pre-warm N Lambda instances
     AWS keeps those N instances always-warm. No cold starts.
     Cost: you pay for the provisioned capacity 24/7.
     Use for: critical endpoints where cold starts are unacceptable.
  
  2. Lighter runtimes: Python/Node/Go over Java
  
  3. Minimize package size: Lambda loads your code on cold start.
     A 10MB package cold starts faster than a 100MB package.
     Strip unnecessary dependencies.
  
  4. SnapStart (Java): saves a post-initialization snapshot.
     Resuming from snapshot: ~1-2s vs. 3-10s for full JVM init.
  
  5. Scheduled warmup: send a dummy event every 5 minutes.
     This keeps instances warm without paying for Provisioned Concurrency.
     Downside: not reliable under real burst traffic.
```

## 3. API Gateway (11.9): HTTP in Front of Lambda

API Gateway sits in front of Lambda and translates HTTP requests into Lambda invocations.

```
API GATEWAY → LAMBDA FLOW:

User → HTTPS → API Gateway → Lambda invocation → Your function → Response

API Gateway handles:
  - TLS termination
  - Request routing (/users → userLambda, /orders → orderLambda)
  - Authentication (API keys, Cognito JWT, custom authorizers)
  - Rate limiting (per API, per IP, per user)
  - Request/response transformation
  - Caching

API GATEWAY TYPES:
  HTTP API (newer, cheaper, faster):
    - Lower latency (~10ms vs ~30ms for REST API)
    - 70% cheaper than REST API
    - Supports: JWT auth, Lambda, HTTP backends
    - Missing: full request transformation, some advanced features
    Best for: most new Lambda-based APIs
  
  REST API (classic, more features):
    - Full request/response transformation
    - API keys, usage plans
    - Mock integrations
    - Better for: legacy use, advanced request mapping
  
  WebSocket API:
    - Persistent WebSocket connections → Lambda handlers
    - For: real-time chat, notifications, streaming
```

```python
# Lambda function designed for API Gateway HTTP API
def lambda_handler(event, context):
    # HTTP API event structure
    method = event['requestContext']['http']['method']
    path = event['requestContext']['http']['path']
    user_id = event['requestContext']['authorizer']['jwt']['claims']['sub']
    query_params = event.get('queryStringParameters') or {}
    body = json.loads(event.get('body') or '{}')
    
    # Route based on method + path
    if method == 'GET' and path.startswith('/users/'):
        user_id_param = path.split('/')[-1]
        user = get_user(user_id_param)
        if not user:
            return {
                "statusCode": 404,
                "body": json.dumps({"error": "User not found"}),
            }
        return {
            "statusCode": 200,
            "body": json.dumps(user),
        }
    
    return {"statusCode": 404, "body": json.dumps({"error": "Not found"})}
```

## 4. ECS and EKS: Running Containers on AWS (11.12)

### ECS (Elastic Container Service)

ECS is AWS's native container orchestrator. Unlike Kubernetes (EKS), ECS is simpler and more AWS-integrated but less flexible.

```
ECS LAUNCH TYPES:

FARGATE (serverless containers — recommended):
  → You define the CPU and memory for your task.
  → AWS provisions the underlying EC2 instances (you don't see them).
  → Pay per second of compute.
  → No cluster management: AWS handles OS updates, scaling, placement.
  → Best for: most containerized applications.
  
EC2 LAUNCH TYPE:
  → You manage a pool of EC2 instances.
  → ECS schedules containers onto those instances.
  → More control, cheaper at scale, more operational overhead.
  → Best for: workloads needing specific instance types (GPU), or very large scale.

ECS KEY CONCEPTS:
  Cluster:         A logical grouping of tasks and services
  Task Definition: Blueprint for a container group (like a Docker Compose service)
  Task:            A running instance of a task definition
  Service:         Maintains N running tasks, integrates with load balancers
```

```hcl
# Terraform: ECS Fargate service
resource "aws_ecs_task_definition" "api" {
  family                   = "api"
  requires_compatibilities = ["FARGATE"]
  network_mode             = "awsvpc"
  cpu                      = 512    # 0.5 vCPU
  memory                   = 1024   # 1 GB RAM
  
  task_role_arn      = aws_iam_role.api_task_role.arn
  execution_role_arn = aws_iam_role.ecs_execution_role.arn
  
  container_definitions = jsonencode([{
    name  = "api"
    image = "${aws_ecr_repository.api.repository_url}:latest"
    
    portMappings = [{
      containerPort = 8000
      protocol      = "tcp"
    }]
    
    environment = [
      { name = "LOG_LEVEL", value = "info" }
    ]
    
    secrets = [
      { name = "DATABASE_URL", valueFrom = aws_secretsmanager_secret.db_url.arn }
    ]
    
    logConfiguration = {
      logDriver = "awslogs"
      options = {
        "awslogs-group"         = aws_cloudwatch_log_group.api.name
        "awslogs-region"        = "us-east-1"
        "awslogs-stream-prefix" = "api"
      }
    }
    
    healthCheck = {
      command     = ["CMD-SHELL", "curl -f http://localhost:8000/health || exit 1"]
      interval    = 30
      timeout     = 5
      retries     = 3
      startPeriod = 10
    }
  }])
}

resource "aws_ecs_service" "api" {
  name            = "api"
  cluster         = aws_ecs_cluster.main.id
  task_definition = aws_ecs_task_definition.api.arn
  desired_count   = 3
  launch_type     = "FARGATE"
  
  network_configuration {
    subnets          = aws_subnet.private[*].id
    security_groups  = [aws_security_group.api.id]
    assign_public_ip = false  # private subnet, NAT gateway for outbound
  }
  
  load_balancer {
    target_group_arn = aws_lb_target_group.api.arn
    container_name   = "api"
    container_port   = 8000
  }
  
  deployment_circuit_breaker {
    enable   = true   # auto rollback if deployment fails health checks
    rollback = true
  }
}
```

### EKS (Elastic Kubernetes Service) (11.12)

EKS is managed Kubernetes on AWS — AWS manages the control plane (API server, etcd, scheduler), you manage the worker nodes (or use Fargate for managed nodes too).

```
ECS vs EKS:

ECS:
  ✓ Simpler: less configuration, less concepts to learn
  ✓ AWS-native: deep integration with IAM, ALB, CloudWatch out-of-box
  ✓ Fargate is excellent: no node management
  ✗ AWS-specific: skills and tooling don't transfer to other clouds
  ✗ Less powerful: no custom operators, limited plugin ecosystem

EKS:
  ✓ Standard: Kubernetes knowledge transfers everywhere
  ✓ Ecosystem: Helm charts, operators, GitOps tools all work
  ✓ More control: custom networking, scheduling, operators
  ✗ More complex: more to learn, more to configure
  ✗ More expensive: control plane costs $0.10/hr minimum
  
RECOMMENDATION:
  New project, team new to containers → ECS Fargate (simpler)
  Team already knows Kubernetes → EKS
  Multi-cloud strategy → EKS (same YAML works everywhere)
  Small company, wants to move fast → ECS Fargate
```

## 5. CloudFront: AWS CDN (11.13)

CloudFront is AWS's Content Delivery Network — a global network of edge locations that cache and serve content close to users.

```
HOW CLOUDFRONT WORKS:

User in Tokyo requests https://cdn.example.com/image.jpg

Without CloudFront:
  Tokyo user → us-east-1 origin → 150ms RTT
  
With CloudFront:
  Tokyo user → Tokyo edge location (~5ms RTT)
  If cached: served immediately (~5ms total)
  If not cached: Tokyo edge → us-east-1 origin → cached at Tokyo edge

CLOUDFRONT USE CASES:
  1. Static assets: images, CSS, JavaScript → cache at edge, serve globally fast
  2. Dynamic API responses: short-lived cache for public API responses
  3. Video streaming: adaptive bitrate streaming from S3
  4. Security: WAF, DDoS protection, TLS termination at the edge

CLOUDFRONT BEHAVIORS:
  Path pattern: /api/*         → forward to ALB origin, no caching
  Path pattern: /static/*      → forward to S3 origin, cache 1 year
  Path pattern: /images/*      → forward to S3, cache 30 days
  
  Each behavior can have different:
  - Cache settings (TTL, cache key)
  - Origin (S3, ALB, custom HTTP)
  - Security settings (WAF rules, geo restrictions)
```

## 6. Senior Engineer Perspective

**The EC2 → Container → Lambda evolution:**

Most teams follow this path: start with EC2 (familiar), move to containers on ECS (reproducible environments, easier scaling), eventually add Lambda for event-driven workloads (pay-per-use, no idle cost). Each step trades operational simplicity for cost efficiency and scalability.

**The Lambda cost math that surprises people:**

Lambda is not always cheapest. At sustained high throughput (1,000+ req/second, 24/7), a small Fargate service can be cheaper. Lambda shines when traffic is spiky or low — you're not paying for idle capacity. Calculate before assuming Lambda saves money.

**The cold start calculation:**

Before using Lambda for a user-facing endpoint, calculate the worst case: p99 latency = max cold start time + max execution time. For a Python Lambda that takes 400ms cold start + 200ms execution = 600ms p99. That might be acceptable for a checkout API but unacceptable for an autocomplete suggestion API. Use Provisioned Concurrency for anything where cold starts exceed your latency budget.
