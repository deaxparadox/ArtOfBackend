## 1. S3: Object Storage, the Backbone of the Internet (11.3)

S3 (Simple Storage Service) is the most used service in all of AWS. It stores objects — any file, any size, up to 5TB per object — with eleven nines of durability (99.999999999%). It is the foundation of countless architectures: user-uploaded content, backups, data lakes, static website hosting, Lambda deployment packages.

**Understanding S3:**

```
S3 DATA MODEL:
  Bucket:    A globally unique namespace (like a drive letter, but unique worldwide)
  Object:    A file stored at a key (path-like string)
  Key:       The "filename" — can contain "/" to simulate directories
  
  bucket: my-app-uploads
  key: users/123/avatars/profile.jpg
  → Accessible at: https://my-app-uploads.s3.amazonaws.com/users/123/avatars/profile.jpg
  
  S3 has NO actual directories — just keys with "/" in them.
  "Folders" in the console are just keys sharing a prefix.

DURABILITY vs AVAILABILITY:
  Durability: 99.999999999% (11 nines)
    → If you store 10 billion objects, you might lose 1 per 10,000 years
    → Objects are stored redundantly across multiple AZs
  
  Availability: 99.99% (S3 Standard) → 52 minutes downtime/year
    → For archive storage (S3 Glacier): availability is lower (3-5 hours for retrieval)
```

### S3 Storage Classes

```
S3 STANDARD:        Most accessed data. $0.023/GB/month.
S3 INTELLIGENT-TIERING: Auto-moves objects to cheaper tier if not accessed.
                        Best for: unpredictable access patterns.
S3 STANDARD-IA:     Infrequent access (30-day minimum). $0.0125/GB/month.
                    30% cheaper, $0.01/GB retrieval charge.
S3 ONE ZONE-IA:     Like Standard-IA but one AZ only. Not HA. Even cheaper.
S3 GLACIER INSTANT: Long-term archive, millisecond retrieval. $0.004/GB/month.
S3 GLACIER FLEXIBLE: Minutes-hours retrieval. $0.0036/GB/month.
S3 GLACIER DEEP ARCHIVE: 12-hour retrieval. $0.00099/GB/month (~$1/TB/month!)
                          Best for: regulatory backups, audit logs kept 7 years.

TYPICAL LIFECYCLE POLICY:
  0-30 days: Standard (hot data)
  30-90 days: Standard-IA (cooling down)
  90+ days: Glacier (archive)
  7 years: Delete (or keep forever for compliance)
```

### S3 Patterns

```python
import boto3
from botocore.config import Config

s3 = boto3.client('s3', config=Config(signature_version='s3v4'))

# PRESIGNED URL: let clients upload directly to S3 (bypass your server)
# Your server never handles the file bytes — just issues a signed URL
def get_upload_url(bucket: str, key: str, content_type: str) -> dict:
    """
    Client uploads directly to S3 using this URL.
    Your server doesn't receive the file — saves bandwidth and compute.
    """
    presigned_url = s3.generate_presigned_url(
        'put_object',
        Params={
            'Bucket': bucket,
            'Key': key,
            'ContentType': content_type,
        },
        ExpiresIn=3600,  # URL valid for 1 hour
    )
    return {
        "upload_url": presigned_url,
        "public_url": f"https://{bucket}.s3.amazonaws.com/{key}",
    }

# STREAMING DOWNLOAD (don't load entire file into memory)
def process_large_s3_file(bucket: str, key: str):
    response = s3.get_object(Bucket=bucket, Key=key)
    stream = response['Body']
    
    # Process in chunks — memory stays constant regardless of file size
    for chunk in stream.iter_chunks(chunk_size=1024 * 1024):  # 1MB chunks
        process_chunk(chunk)

# MULTIPART UPLOAD (for large files > 100MB)
# S3 has a 5GB limit per PUT. For larger files: multipart upload.
# boto3's TransferConfig handles this automatically:
from boto3.s3.transfer import TransferConfig

config = TransferConfig(
    multipart_threshold=8 * 1024 * 1024,    # 8MB: use multipart for larger files
    max_concurrency=10,                       # parallel upload threads
    multipart_chunksize=8 * 1024 * 1024,     # 8MB per part
)

s3.upload_file(
    'large_video.mp4',
    'my-bucket',
    'videos/large_video.mp4',
    Config=config,
    ExtraArgs={'ContentType': 'video/mp4'},
)
```

## 2. RDS: Managed Relational Databases (11.4)

RDS (Relational Database Service) is a managed service that runs PostgreSQL, MySQL, MariaDB, Oracle, or SQL Server. AWS handles backups, OS patches, minor version upgrades, and failover.

```
RDS vs SELF-MANAGED POSTGRES:

SELF-MANAGED:
  ✓ Full control (tune anything, install any extension)
  ✓ Cheaper at large scale (no RDS overhead)
  ✗ You manage: OS updates, PostgreSQL upgrades, backups, HA, monitoring

RDS:
  ✓ Automated backups (point-in-time recovery up to 35 days)
  ✓ Multi-AZ standby with automatic failover
  ✓ Minor version auto-upgrades
  ✓ Performance Insights (visual query analysis)
  ✓ Enhanced monitoring
  ✗ More expensive (~20-30% overhead vs EC2 self-managed)
  ✗ Some restrictions (can't install arbitrary OS packages)
  ✗ Some PostgreSQL extensions not available

FOR MOST TEAMS: RDS is the right choice. The operational savings outweigh the cost.
```

**Key RDS settings for production:**

```hcl
resource "aws_db_instance" "main" {
  # ... other settings ...
  
  multi_az = true   # standby in another AZ, auto-failover in 60-120s
  
  backup_retention_period = 7          # keep 7 days of automated backups
  backup_window = "03:00-04:00"        # backup during low-traffic hours (UTC)
  maintenance_window = "Mon:04:00-Mon:05:00"  # OS/engine updates
  
  deletion_protection = true           # prevent accidental terraform destroy
  
  # Performance Insights: visualize queries over time
  performance_insights_enabled = true
  performance_insights_retention_period = 7  # days
  
  # Enable enhanced monitoring (CloudWatch metrics every 1-60 seconds)
  monitoring_interval = 60  # seconds; 0 = disabled
  monitoring_role_arn = aws_iam_role.rds_monitoring.arn
  
  # Storage auto-scaling: avoid running out of disk
  allocated_storage     = 100
  max_allocated_storage = 1000  # auto-scale up to 1TB
  
  # Encryption at rest
  storage_encrypted = true
  kms_key_id        = aws_kms_key.rds.arn
}
```

### Aurora: AWS's High-Performance Database (11.5)

Aurora is AWS's proprietary MySQL/PostgreSQL-compatible database. It was built from scratch for the cloud and offers 5× the throughput of standard MySQL and 3× the throughput of standard PostgreSQL.

```
HOW AURORA DIFFERS FROM RDS POSTGRES:

STORAGE:
  RDS Postgres: Standard EBS volumes, replication over network to standby
  Aurora: 6-way distributed storage (2 copies per AZ × 3 AZs)
          Storage auto-scales 10GB → 128TB
          No need to provision storage upfront
  
REPLICATION:
  RDS: Physical streaming replication (one standby, automatic failover)
  Aurora: Up to 15 read replicas with < 100ms replica lag
          Replicas share the same storage layer (no data copying!)
          Replicas promoted to primary in < 30 seconds
  
AURORA SERVERLESS v2:
  Scales compute capacity automatically, per ACU (Aurora Capacity Unit)
  Min: 0.5 ACUs → Max: 128 ACUs
  Scales in seconds (not minutes like traditional RDS)
  Best for: variable workloads, development, multi-tenant SaaS
  
WHEN AURORA OVER RDS POSTGRES:
  ✓ Need more than 1 read replica
  ✓ Need faster failover (< 30s vs. 60-120s for RDS)
  ✓ Variable workload (Aurora Serverless)
  ✓ Need up to 128TB storage
  ✗ Aurora is ~20% more expensive than RDS at same instance size
  ✗ More AWS-specific (less standard Postgres behavior)
```

## 3. ElastiCache: Managed Redis/Memcached (11.7)

ElastiCache is managed Redis (or Memcached). AWS handles replication, failover, patching, and monitoring.

```python
import redis

# ElastiCache Redis connection (works the same as self-hosted Redis)
cache = redis.Redis(
    host=os.environ['ELASTICACHE_ENDPOINT'],  # primary endpoint
    port=6379,
    ssl=True,         # ElastiCache supports TLS in transit
    decode_responses=True,
)

# For Redis Cluster mode (multiple shards):
from redis.cluster import RedisCluster

cluster = RedisCluster(
    startup_nodes=[{"host": os.environ['CLUSTER_ENDPOINT'], "port": 6379}],
    decode_responses=True,
    ssl=True,
)
```

**ElastiCache cluster modes:**

```
CLUSTER MODE DISABLED (Replication Group):
  One primary node, up to 5 read replicas.
  All data on one shard.
  Automatic failover: replica promoted in ~1 minute.
  Max data: limited by one node's memory.
  Best for: most applications. Simple.

CLUSTER MODE ENABLED (Redis Cluster):
  Data sharded across 1-500 shards.
  Each shard: one primary + up to 5 replicas.
  Max data: up to 500 × single node capacity.
  Horizontal scaling for very large datasets.
  Complexity: Multi-key operations must target same slot (hash tag {}).
  Best for: very large caches (> 100GB), high throughput requirements.
```

## 4. SQS: Managed Queue Service (11.10)

```python
import boto3, json

sqs = boto3.client('sqs', region_name='us-east-1')
QUEUE_URL = os.environ['SQS_QUEUE_URL']

# PRODUCER: send messages
sqs.send_message(
    QueueUrl=QUEUE_URL,
    MessageBody=json.dumps({
        "order_id": 123,
        "action": "process_payment",
    }),
    MessageGroupId="payments",      # FIFO queue: ordering within group
    MessageDeduplicationId=str(123),  # FIFO: prevent duplicate processing
)

# CONSUMER: receive and process messages
response = sqs.receive_message(
    QueueUrl=QUEUE_URL,
    MaxNumberOfMessages=10,
    WaitTimeSeconds=20,     # long polling: block up to 20s waiting for messages
    VisibilityTimeout=300,  # message invisible to others for 5 minutes while processing
)

for message in response.get('Messages', []):
    try:
        body = json.loads(message['Body'])
        process_order(body['order_id'])
        
        # DELETE after successful processing
        sqs.delete_message(
            QueueUrl=QUEUE_URL,
            ReceiptHandle=message['ReceiptHandle'],
        )
    except Exception as e:
        # Don't delete — message returns to queue after VisibilityTimeout
        logger.error("Failed to process message", error=str(e))
```

**SQS key concepts:**

```
STANDARD QUEUE:
  - At-least-once delivery (occasional duplicates)
  - Best-effort ordering (not guaranteed)
  - Up to 120,000 TPS
  - Use for: most workloads, idempotent consumers

FIFO QUEUE:
  - Exactly-once processing (deduplication)
  - First-in, first-out ordering within a message group
  - Up to 3,000 TPS (or 300 without batching)
  - Use for: order-sensitive processing, financial transactions

VISIBILITY TIMEOUT:
  When a consumer receives a message, it becomes invisible to others.
  If the consumer crashes before deleting it:
  After VisibilityTimeout seconds → message returns to queue.
  Another consumer can then process it.
  Set to: expected max processing time × 3 (safety margin)

DEAD LETTER QUEUE (DLQ):
  After N failed attempts (maxReceiveCount), message goes to DLQ.
  DLQ stores unprocessable messages for manual inspection.
  Configure DLQ redrive policy on every production queue.
```

## 5. CloudWatch: Observability (11.17)

CloudWatch is AWS's integrated observability service — metrics, logs, alarms, and dashboards.

```python
import boto3, time

cloudwatch = boto3.client('cloudwatch')

# CUSTOM METRICS: send application metrics to CloudWatch
def record_metric(name: str, value: float, unit: str = 'Count', dimensions: dict = None):
    cloudwatch.put_metric_data(
        Namespace='MyApp',  # your custom namespace
        MetricData=[{
            'MetricName': name,
            'Value': value,
            'Unit': unit,
            'Timestamp': time.time(),
            'Dimensions': [
                {'Name': k, 'Value': v}
                for k, v in (dimensions or {}).items()
            ],
        }]
    )

# Use:
record_metric('OrdersProcessed', 1, dimensions={'Environment': 'production'})
record_metric('PaymentLatency', 234, 'Milliseconds', {'PaymentProvider': 'stripe'})
```

## 6. AWS Secrets Manager & SSM Parameter Store (11.16)

```python
import boto3, json

secrets_client = boto3.client('secretsmanager', region_name='us-east-1')
ssm_client = boto3.client('ssm', region_name='us-east-1')

# Secrets Manager: for sensitive credentials with rotation
def get_database_credentials() -> dict:
    response = secrets_client.get_secret_value(
        SecretId='prod/myapp/database'
    )
    return json.loads(response['SecretString'])

# Usage at startup:
db_creds = get_database_credentials()
DATABASE_URL = f"postgresql://{db_creds['username']}:{db_creds['password']}@{db_creds['host']}/{db_creds['dbname']}"

# SSM Parameter Store: for non-secret configuration (or secrets without rotation)
def get_parameter(name: str) -> str:
    response = ssm_client.get_parameter(
        Name=name,
        WithDecryption=True  # decrypt SecureString parameters
    )
    return response['Parameter']['Value']

# Usage:
redis_url = get_parameter('/production/myapp/redis-url')
feature_flags = json.loads(get_parameter('/production/myapp/feature-flags'))
```

**Secrets Manager vs. SSM Parameter Store:**

```
SECRETS MANAGER:
  ✓ Automatic rotation (Lambda-based, AWS-managed for RDS/Redshift/DocumentDB)
  ✓ Cross-account access
  ✓ Richer API (versioning, metadata)
  ✗ Cost: $0.40/secret/month + $0.05 per 10,000 API calls
  Use for: database passwords that rotate, API keys, anything you rotate

SSM PARAMETER STORE:
  ✓ Free tier (Standard parameters, non-encrypted)
  ✓ SecureString: KMS-encrypted, $0.05 per advanced parameter
  ✗ No automatic rotation
  ✗ 10KB max value size
  Use for: configuration values, non-rotating API keys, feature flags
```

## 7. Senior Engineer Perspective

**The S3 pre-signed URL pattern is underused:**

Teams that handle file uploads through their application servers are burning compute and bandwidth unnecessarily. The correct pattern: client requests a pre-signed URL from your API → client uploads directly to S3 → client notifies your API that upload is complete → your API processes the file from S3. Your server only sees a notification, not the file bytes. This scales to any file size with zero server memory pressure.

**RDS Multi-AZ is not optional for production:**

A single-AZ RDS database is a single point of failure. The standby costs approximately 2× the primary instance. The math: if an outage costs your company $10,000/hour and happens once per 6 months, you need the outage to last only 10 minutes to justify the cost difference. For any real business: Multi-AZ pays for itself.

**The SQS message visibility timeout gotcha:**

If you set `VisibilityTimeout=30s` but your processing takes 45 seconds, the message becomes visible again after 30 seconds. SQS delivers it to another consumer. You now have two consumers processing the same message simultaneously. Set visibility timeout to at least 3× your expected processing time. Call `ChangeMessageVisibility` for long-running processing to extend the timeout.
