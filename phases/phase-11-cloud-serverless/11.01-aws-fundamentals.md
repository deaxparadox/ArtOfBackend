## 1. The Mental Model for Cloud (11.1)

Before understanding any specific AWS service, you need the right mental model for what "the cloud" actually is.

**The cloud is not magic. It is other people's computers.**

More precisely: AWS is a global network of data centers that rents computing resources — servers, storage, networking, and managed services — by the minute or hour. You use what you need, pay for what you use, and return it when done.

**The real-world analogy:**

Owning physical servers is like buying a car. You pay the full upfront cost, you're responsible for maintenance, and you're stuck with the same car whether you drive 1 mile or 10,000 miles per month. The cloud is like Uber — you pay only for the trips you take, the car scales to your needs (a minivan for 6 people, a sedan for 1), and someone else handles the maintenance.

## 2. AWS Global Infrastructure

### Regions

A **Region** is a physical geographic location where AWS has data centers. Each region is completely independent — it has its own power, cooling, network infrastructure, and is isolated from failures in other regions.

```
SELECTED AWS REGIONS (2024):
  us-east-1       Northern Virginia (the original, largest, cheapest)
  us-west-2       Oregon
  eu-west-1       Ireland
  eu-central-1    Frankfurt
  ap-southeast-1  Singapore
  ap-northeast-1  Tokyo
  ap-south-1      Mumbai
  sa-east-1       São Paulo
  Total: 33 regions, growing

CHOOSING A REGION:
  1. Latency: choose closest to your users
  2. Compliance: some data must stay in specific geography (GDPR, HIPAA)
  3. Price: us-east-1 is typically 10-20% cheaper than other regions
  4. Service availability: new AWS services launch in us-east-1 first,
     sometimes months before other regions

IMPORTANT: Services in different regions are isolated.
  An EC2 instance in us-east-1 CANNOT directly talk to one in eu-west-1
  without going over the public internet (or AWS PrivateLink).
  Data stored in eu-west-1 stays in eu-west-1 (GDPR compliance).
```

### Availability Zones

A **Region** contains multiple **Availability Zones (AZs)** — physically separate data centers within the same geographic region. They have independent power, cooling, and physical network paths, but are connected by ultra-low-latency private fiber.

```
REGION: us-east-1 (Northern Virginia)
  AZ: us-east-1a ← separate physical data center
  AZ: us-east-1b ← separate physical data center (miles away)
  AZ: us-east-1c ← separate physical data center
  AZ: us-east-1d
  AZ: us-east-1e
  AZ: us-east-1f

WHY AZs MATTER:
  If us-east-1a has a power failure (it has happened):
  → Services only in us-east-1a: DOWN
  → Services spread across us-east-1a and us-east-1b: HEALTHY
  
  Multi-AZ is the minimum for production availability.
  Single-AZ is the single most common cause of avoidable outages.
  
MULTI-AZ ARCHITECTURE:
  Load Balancer → spans all AZs
  App servers: 2+ instances, one per AZ
  RDS: primary in 1a, standby replica in 1b
  If 1a fails: LB stops routing there, RDS promotes 1b to primary
  Users see: brief slowness (60-120s for RDS failover), then normal

GLOBAL SERVICES (not region-specific):
  IAM, Route 53, CloudFront, WAF, Organizations
  These are inherently global — you configure once, they work everywhere.
```

## 3. IAM: Identity and Access Management (11.15)

IAM is arguably the most important AWS service. Every action in AWS is an API call. IAM controls who (or what) can make which API calls.

**If IAM is misconfigured, every other security measure is meaningless.**

### Core IAM Concepts

```
USERS: Human identities (your developers, admins)
  Create a user for each human who accesses AWS console/CLI.
  Users have permanent credentials (password + access keys).
  
GROUPS: Collections of users
  Apply permissions to a group, add users to the group.
  Easier than managing permissions per-user.
  
ROLES: Temporary identities (for services and cross-account access)
  An EC2 instance, Lambda function, or ECS task assumes a Role.
  Roles issue temporary credentials (valid 1-12 hours).
  No permanent access keys stored in code.
  
POLICIES: JSON documents defining what actions are allowed/denied
  Attached to users, groups, or roles.
  Follow the principle of least privilege.
```

### The Principle of Least Privilege

The most important IAM rule: **grant only the minimum permissions required to do the job.**

```json
// BAD: Admin access for an application that only reads S3
{
    "Effect": "Allow",
    "Action": "*",
    "Resource": "*"
}

// GOOD: Only the specific actions on the specific resources needed
{
    "Version": "2012-10-17",
    "Statement": [
        {
            "Effect": "Allow",
            "Action": [
                "s3:GetObject",
                "s3:ListBucket"
            ],
            "Resource": [
                "arn:aws:s3:::my-upload-bucket",
                "arn:aws:s3:::my-upload-bucket/*"
            ]
        }
    ]
}
```

### IAM Roles for EC2/ECS/Lambda (The Right Pattern)

```python
# WRONG: hardcoding credentials in application code
import boto3

s3 = boto3.client(
    's3',
    aws_access_key_id='AKIAIOSFODNN7EXAMPLE',     # in code = catastrophic
    aws_secret_access_key='wJalrXUtnFEMI/...',
)

# WRONG: storing credentials in environment variables in the image
# AWS_ACCESS_KEY_ID=AKIAIOSFODNN7EXAMPLE  ← visible in docker inspect, logs

# RIGHT: use IAM roles — boto3 automatically finds credentials from instance metadata
s3 = boto3.client('s3')  # automatically uses EC2 instance role or Lambda execution role

# The AWS SDK checks credentials in this order:
# 1. Environment variables (AWS_ACCESS_KEY_ID, etc.)
# 2. ~/.aws/credentials file
# 3. IAM instance profile (EC2/ECS/Lambda role) ← use this in production
# 4. Container credentials endpoint (ECS)
```

```yaml
# Terraform: attach a role to an ECS task
resource "aws_iam_role" "api_task_role" {
  name = "api-task-role"

  assume_role_policy = jsonencode({
    Version = "2012-10-17"
    Statement = [{
      Effect    = "Allow"
      Principal = { Service = "ecs-tasks.amazonaws.com" }
      Action    = "sts:AssumeRole"
    }]
  })
}

resource "aws_iam_role_policy" "api_s3_access" {
  name = "api-s3-access"
  role = aws_iam_role.api_task_role.id

  policy = jsonencode({
    Version = "2012-10-17"
    Statement = [{
      Effect   = "Allow"
      Action   = ["s3:GetObject", "s3:PutObject"]
      Resource = ["arn:aws:s3:::${var.upload_bucket}/*"]
    }]
  })
}

resource "aws_ecs_task_definition" "api" {
  task_role_arn      = aws_iam_role.api_task_role.arn  # role the application uses
  execution_role_arn = aws_iam_role.ecs_execution_role.arn  # role ECS uses to pull image + log
  ...
}
```

### IAM Best Practices

```
ACCOUNTS:
  □ Use AWS Organizations with multiple accounts
    (dev / staging / production in separate accounts — blast radius isolation)
  □ Root account: only for billing, lock it with MFA, never use for daily work
  □ Enable MFA for all human users

PERMISSIONS:
  □ Principle of least privilege: only what's needed, nothing more
  □ Deny by default: IAM starts with deny, you explicitly allow
  □ Prefer roles over users for applications (temporary credentials)
  □ Rotate access keys regularly (better: don't use them — use roles)
  □ Review permissions quarterly with AWS Access Analyzer

MONITORING:
  □ CloudTrail enabled in all regions (audit log of all API calls)
  □ Alert on: root account usage, failed authentication, privilege escalation
  □ AWS Config: detect policy changes
  □ GuardDuty: detect credential compromise
```

## 4. Real-World Context

### The Capital One Data Breach (2019)

A misconfigured AWS WAF allowed an attacker to perform SSRF (Server-Side Request Forgery) against an EC2 instance. The instance had an overly permissive IAM role that allowed listing all S3 buckets and reading their contents.

Result: 106 million customer records exposed.

The root cause: an IAM role that had `s3:ListAllMyBuckets` and `s3:GetObject` on `arn:aws:s3:::*` (all S3 buckets) instead of only the specific buckets the application needed.

With least-privilege IAM: the compromised role could have only accessed the specific bucket needed by the application. The attacker finds the SSRF vulnerability but can only read/write one bucket — a much smaller breach.

**Lesson: IAM misconfiguration is the #1 cause of AWS data breaches.**

### The AWS Account Strategy

Every production AWS workload should follow this structure:

```
AWS Organization
├── Management Account (billing only, no workloads)
├── Security Account (GuardDuty, Security Hub, audit logs)
├── Shared Services Account (ECR, Artifactory, shared tools)
├── Production Account (production workloads only)
├── Staging Account (staging workloads)
└── Development Account (developer sandboxes)

WHY MULTIPLE ACCOUNTS?
  - Production IAM mistakes can't affect staging accounts
  - A compromised development EC2 can't access production S3
  - Billing is clearly separated
  - Security policies enforced at the Organization level (SCPs)
```

## 5. Senior Engineer Perspective

**What a junior gets wrong:**

Creating IAM users for applications and storing their access keys in `.env` files or environment variables. This is the most common AWS security mistake. Instead: every application running in AWS should assume an IAM role. Roles issue temporary credentials automatically — no access keys in code, no rotation needed, credentials expire automatically.

**The IAM audit every senior runs on a new account:**

```bash
# Find users with access keys (should be minimal — humans should use SSO)
aws iam generate-credential-report
aws iam get-credential-report --output text | base64 -d | grep -E "true,true"
# ^ users with both console access AND access keys = high risk

# Find over-permissive policies
aws iam list-policies --scope Local | grep -i admin
# Any policy with "admin" or "full" in the name is suspicious

# Use AWS Access Analyzer to find unintended external access
aws accessanalyzer list-findings --analyzer-name myanalyzer
```

**The one IAM principle to internalize:**

"What is the worst possible thing someone could do if this credential was compromised?" The answer should be bounded and acceptable. If a stolen Lambda execution role credential would give an attacker read access to one S3 bucket: recoverable. If it gives them write access to the production RDS database: catastrophic. Design roles with the worst case in mind.
