## 1. What Is a VPC?

A VPC (Virtual Private Cloud) is your private, isolated network within AWS. Every resource you create in AWS lives inside a VPC. It's the networking foundation for your entire cloud infrastructure.

**The real-world analogy:**

The internet is like a public street — anyone can see and reach anything on it. A VPC is like a private gated community — the internal streets, houses, and services are visible only to members of the community. The gate (Internet Gateway) controls what the outside world can reach.

## 2. VPC Architecture: The Standard Production Design

```
PRODUCTION VPC ARCHITECTURE (us-east-1, 3 AZs):

VPC: 10.0.0.0/16 (65,536 IP addresses)

PUBLIC SUBNETS (internet-accessible):
  10.0.1.0/24  us-east-1a  ← Internet Gateway for inbound traffic
  10.0.2.0/24  us-east-1b  ← Load balancers, NAT gateways, bastion hosts
  10.0.3.0/24  us-east-1c

PRIVATE SUBNETS (no direct internet exposure):
  10.0.11.0/24 us-east-1a  ← Application servers (ECS, EKS, EC2)
  10.0.12.0/24 us-east-1b  ← They access internet via NAT Gateway
  10.0.13.0/24 us-east-1c

DATABASE SUBNETS (most restricted):
  10.0.21.0/24 us-east-1a  ← RDS, ElastiCache
  10.0.22.0/24 us-east-1b  ← No outbound internet access
  10.0.23.0/24 us-east-1c

TRAFFIC FLOWS:

Internet → IGW → ALB (public subnet) → App (private subnet) → RDS (database subnet)
                                             ↓ outbound
                                       NAT Gateway (public subnet)
                                             ↓
                                        Internet (APIs, package downloads)
```

### Key VPC Components

```
INTERNET GATEWAY (IGW):
  Enables communication between public subnets and the internet.
  One per VPC. Highly available by design.
  Attach to VPC, update route table: 0.0.0.0/0 → IGW.

NAT GATEWAY:
  Allows private subnet resources to initiate outbound internet connections.
  (private EC2 can download packages, call external APIs)
  Blocks unsolicited inbound connections from internet.
  Cost: ~$32/month per AZ + $0.045/GB processed.
  Deploy in each AZ (for HA) with separate route tables per AZ.

ROUTE TABLES:
  Every subnet has a route table defining where traffic goes.
  
  Public subnet route table:
    10.0.0.0/16 → local (within VPC)
    0.0.0.0/0   → igw-xxxxxx (internet)
  
  Private subnet route table:
    10.0.0.0/16 → local
    0.0.0.0/0   → nat-xxxxxx (via NAT Gateway)
  
  Database subnet route table:
    10.0.0.0/16 → local
    (no default route → no internet access at all)

SECURITY GROUPS:
  Stateful firewall for instances.
  "Allow inbound 5432 from app security group"
  Applied to: EC2, RDS, ECS tasks, Lambda (in VPC), etc.

NETWORK ACLs:
  Stateless firewall for subnets.
  Applied before security groups.
  Explicit deny rules (unlike security groups which are allow-only).
  Use for subnet-level blocking (e.g., block an IP range from the entire private tier).
```

## 3. VPC in Terraform

```hcl
# Complete production VPC with Terraform
module "vpc" {
  source  = "terraform-aws-modules/vpc/aws"
  version = "5.5.0"

  name = "production"
  cidr = "10.0.0.0/16"

  azs              = ["us-east-1a", "us-east-1b", "us-east-1c"]
  
  # Public subnets: ALB, NAT gateways
  public_subnets   = ["10.0.1.0/24", "10.0.2.0/24", "10.0.3.0/24"]
  
  # Private subnets: application servers
  private_subnets  = ["10.0.11.0/24", "10.0.12.0/24", "10.0.13.0/24"]
  
  # Database subnets: RDS, ElastiCache
  database_subnets = ["10.0.21.0/24", "10.0.22.0/24", "10.0.23.0/24"]

  enable_nat_gateway = true
  single_nat_gateway = false  # one NAT per AZ for HA (vs. one shared NAT)
                               # single_nat_gateway = true: cheaper ($32/mo vs $96/mo)
                               # but if public AZ fails, private subnets lose internet
  
  enable_dns_hostnames = true  # needed for RDS, EKS

  tags = {
    Environment = "production"
    ManagedBy   = "terraform"
  }
}

# Security group for application servers
resource "aws_security_group" "app" {
  name   = "app-sg"
  vpc_id = module.vpc.vpc_id

  # Allow traffic from load balancer only
  ingress {
    from_port       = 8000
    to_port         = 8000
    protocol        = "tcp"
    security_groups = [aws_security_group.alb.id]  # ONLY from the ALB
  }

  # Allow all outbound (to reach database, redis, external APIs)
  egress {
    from_port   = 0
    to_port     = 0
    protocol    = "-1"
    cidr_blocks = ["0.0.0.0/0"]
  }
}

# Security group for database (most restrictive)
resource "aws_security_group" "rds" {
  name   = "rds-sg"
  vpc_id = module.vpc.vpc_id

  # Allow only from application security group
  ingress {
    from_port       = 5432
    to_port         = 5432
    protocol        = "tcp"
    security_groups = [aws_security_group.app.id]  # ONLY app servers
  }
  # No egress rule = no outbound traffic (databases don't initiate connections)
}
```

## 4. VPC Endpoints: Access AWS Services Without Leaving the VPC

By default, when your EC2 instance in a private subnet calls S3, the traffic goes through the NAT Gateway — costing money ($0.045/GB) and adding latency.

VPC Endpoints keep the traffic within the AWS network:

```hcl
# S3 Gateway Endpoint: free, enables private subnet to reach S3 without NAT
resource "aws_vpc_endpoint" "s3" {
  vpc_id            = module.vpc.vpc_id
  service_name      = "com.amazonaws.us-east-1.s3"
  vpc_endpoint_type = "Gateway"
  
  route_table_ids = module.vpc.private_route_table_ids
}

# DynamoDB Gateway Endpoint: free
resource "aws_vpc_endpoint" "dynamodb" {
  vpc_id            = module.vpc.vpc_id
  service_name      = "com.amazonaws.us-east-1.dynamodb"
  vpc_endpoint_type = "Gateway"
  route_table_ids   = module.vpc.private_route_table_ids
}

# Secrets Manager Interface Endpoint: ~$7/month/AZ but keeps secrets traffic private
resource "aws_vpc_endpoint" "secrets_manager" {
  vpc_id              = module.vpc.vpc_id
  service_name        = "com.amazonaws.us-east-1.secretsmanager"
  vpc_endpoint_type   = "Interface"
  subnet_ids          = module.vpc.private_subnets
  security_group_ids  = [aws_security_group.vpc_endpoints.id]
  private_dns_enabled = true  # DNS resolves secretsmanager to private endpoint
}
```

## 5. Senior Engineer Perspective

**The NAT Gateway cost trap:**

Many teams don't realize how expensive NAT Gateway data processing can be. At $0.045/GB, a service processing 10TB/month of outbound traffic = $450/month just in NAT fees. VPC Endpoints for S3 and DynamoDB are free and eliminate this cost for those services. For ECS services pulling large Docker images: use ECR's VPC interface endpoint to avoid NAT charges on image pulls.

**The single NAT Gateway vs. multi-AZ tradeoff:**

If you use `single_nat_gateway = true` (one NAT in one public AZ), all your private subnets route through that one AZ. If the AZ with the NAT Gateway fails: your entire private tier loses internet access even though the applications themselves are in other AZs. For production systems, use one NAT Gateway per AZ (`single_nat_gateway = false`). Yes, it costs 3× as much, but it's correct HA.

**The private database subnet is non-negotiable:**

Every production database must be in a private subnet with no default route to the internet. A database accessible from the internet will be compromised. The attack timeline is measured in minutes, not hours. Security groups are defense in depth; private subnets are the primary defense.
