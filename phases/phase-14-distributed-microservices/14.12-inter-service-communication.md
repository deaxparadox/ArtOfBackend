## 1. What Is It? (Theory)

Once you have multiple services, they must communicate. Every inter-service call is a network call (topic 14.1 — fallacies), so you must choose your communication pattern carefully. The two fundamental modes are:

- **Synchronous**: caller waits for the response
- **Asynchronous**: caller sends a message and continues without waiting

This is not just a technical choice — it determines the coupling between services, their failure behavior, and the user experience.

---

## 2. Synchronous Communication (REST/gRPC)

```
WHEN TO USE:
  ✓ Caller needs the result to continue (create order → need order_id to continue)
  ✓ Real-time response required (user is waiting for the result)
  ✓ Simple request-reply semantics needed
  ✓ Result is used immediately in the same request handler

SYNCHRONOUS CALL CHAIN RISKS:
  
  User → OrderService → PaymentService → InventoryService → ShippingService
  
  Each call adds latency: 50ms + 30ms + 20ms + 40ms = 140ms minimum
  
  If InventoryService is down:
  → OrderService returns error
  → User sees error
  → No order placed
  
  If ShippingService is slow (500ms instead of 40ms):
  → Entire chain is 550ms
  → User waits
  
  RULES FOR SYNCHRONOUS CALLS:
  1. Keep chains short (max 2-3 hops from the user-facing service)
  2. Set timeouts on every call (see topic 2.15)
  3. Use circuit breakers (see topic 14.16)
  4. For non-critical dependencies: use fallbacks (see topic 9.17)

HTTP/REST: Universal, human-readable, great for public APIs and browser clients
gRPC: Binary, efficient, generated clients, great for internal service calls
  See topics 6.1-6.19 for full comparison.
```

## 3. Asynchronous Communication (Events/Messages)

```
WHEN TO USE:
  ✓ Caller doesn't need the result immediately
  ✓ Multiple services need to react to the same event
  ✓ Loose coupling is more important than immediate consistency
  ✓ Resilience to downstream service failures is required
  ✓ Workload leveling (absorb spikes, process at sustainable rate)

ASYNC FLOW:
  User → OrderService
  OrderService: create order record, publish "order.placed" event → returns 201 Created
  
  Independently (consumers react in their own time):
  PaymentService: processes payment (may take 2 seconds)
  InventoryService: reserves items
  NotificationService: sends confirmation email
  AnalyticsService: records the event
  
  User gets immediate response.
  Processing happens in the background.
  Failures in PaymentService don't fail the order creation.

THE COUPLING DIFFERENCE:
  
  SYNCHRONOUS:
  OrderService knows about PaymentService, InventoryService.
  Adding NotificationService: change OrderService.
  
  ASYNCHRONOUS:
  OrderService knows about nobody.
  Adding NotificationService: subscribe to "order.placed" event.
  OrderService unchanged.
```

## 4. Choosing Sync vs. Async Per Operation

```
OPERATION                   PATTERN        REASON
─────────────────────────────────────────────────────────────────────
"Does user exist?"          SYNC           Need answer before proceeding
"Get product details"       SYNC           User waiting for display
"Create order"              SYNC (creation) + ASYNC (side effects)
  - Create order record     SYNC           User needs order_id
  - Charge payment          SYNC           Must succeed to confirm order
  - Send email              ASYNC          User doesn't need to wait
  - Update analytics        ASYNC          Not user-critical
  - Sync to CRM             ASYNC          Not time-critical

"Process batch export"      ASYNC          Long-running, background
"Update recommendation"     ASYNC          Eventual consistency fine
"Sync inventory from ERP"   ASYNC          Bulk operation
"Send webhook to customer"  ASYNC + RETRY  External call, may fail

THE HYBRID PATTERN (most correct for complex operations):
  
  @app.post("/orders")
  async def create_order(data: OrderRequest, user = Depends(auth)):
      # SYNCHRONOUS: steps that determine the response
      order = await db.create_order(data)
      payment = await payment_service.charge(order.id, data.amount_cents)
      await inventory_service.reserve(order.id, data.product_id, data.quantity)
      
      # ASYNCHRONOUS: side effects that don't affect the response
      await event_bus.publish("order.confirmed", {"order_id": order.id})
      # Email, analytics, CRM sync all happen in background
      
      return {"order_id": order.id, "status": "confirmed"}
```

## 5. The Service Mesh for Internal Communication

For internal service-to-service calls, a **service mesh** (Istio, Linkerd) provides:
- mTLS between all services (mutual authentication)
- Circuit breaking at the infrastructure level
- Distributed tracing without application code changes
- Traffic management (canary deploys, traffic splitting)

See topics 14.18-14.19 for full service mesh coverage.

---

## 6. Senior Engineer Perspective

**The synchronous chain is the most common microservices design mistake:**

Chains of synchronous calls create implicit coupling: if any service in the chain is slow or down, the entire chain fails. The User → A → B → C → D pattern means all 4 services must be available for any user request to succeed.

Design rule: a user-facing request should call AT MOST one or two downstream services synchronously. Everything else should be async (event-driven).

If you find yourself needing a chain longer than 2-3 hops for a single user request: reconsider whether those operations truly need to be synchronous. Most "side effects" (notifications, analytics, CRM sync) are naturally async.
