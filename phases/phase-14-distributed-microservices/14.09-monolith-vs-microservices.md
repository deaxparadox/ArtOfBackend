## 1. What Is It? (Theory)

Every backend system makes a fundamental architectural choice: deploy as one application (monolith) or as many independent services (microservices). This decision shapes everything — your deployment model, operational complexity, team structure, and failure modes.

The industry narrative shifted strongly toward microservices in the 2013-2020 period, leading many teams to adopt microservices not because they solved a real problem but because it seemed "modern." This created a generation of over-engineered systems with distributed systems complexity but without the benefits that microservices were designed to provide.

The honest assessment: **microservices are the right choice for some problems. For many others, they are dramatically the wrong choice.**

## 2. How It Actually Works (Deep Mechanics)

### The Monolith

```
MONOLITH: one deployable unit, one codebase, one database.

WHAT "MONOLITH" DOESN'T MEAN:
  ✗ Big ball of mud (no structure, tangled code)
  ✗ Can't be well-structured (see topic 14.10: modular monolith)
  ✗ Can't scale (horizontal scaling still works — just deploy more instances)
  
WHAT MONOLITH ACTUALLY MEANS:
  One deployment artifact.
  All code in one repository (or a few related repos).
  Typically one database schema.
  Function calls between components (not HTTP/gRPC).

MONOLITH STRENGTHS:
  ✓ SIMPLE TRANSACTIONS: function calls are transactions.
    order_service.create_order()
    payment_service.charge()  ← same process, same transaction if needed
  
  ✓ SIMPLE DEBUGGING: all logs in one place.
    One grep. One log file. One stack trace.
    
  ✓ SIMPLE DEPLOYMENT: one artifact to build, test, deploy.
    No service coordination. No versioning across services.
    
  ✓ SIMPLE DEVELOPMENT: function-level code navigation.
    IDE works: go-to-definition, refactor, find-all-usages.
    
  ✓ LOW LATENCY: function calls (nanoseconds) vs service calls (milliseconds).
  
  ✓ NO DISTRIBUTED SYSTEM COMPLEXITY:
    No saga patterns, no idempotency keys, no circuit breakers.
    Just functions and transactions.

MONOLITH WEAKNESSES:
  ✗ ALL-OR-NOTHING DEPLOYMENT:
    Update the shipping component → redeploy everything.
    A bug in shipping delays shipping AND payments AND user management.
    
  ✗ SCALING COUPLES:
    The payment processor needs 10x more compute than user management.
    With a monolith: scale both together (wasteful).
    
  ✗ TECHNOLOGY COUPLING:
    All components must use the same language/runtime/framework.
    Want to use Python for ML and Go for high-throughput? Hard.
    
  ✗ TEAM SCALING CHALLENGES:
    100 engineers all touching the same codebase.
    Merges conflict. Deployments break each other's code.
    Code ownership is unclear.
```

### Microservices

```
MICROSERVICES: many independently deployable services,
               each with its own database.

MICROSERVICES STRENGTHS:
  ✓ INDEPENDENT DEPLOYMENT:
    Update the shipping service → deploy only the shipping service.
    Other services unaffected. Lower risk per deployment.
    
  ✓ INDEPENDENT SCALING:
    Payment processor is hot → scale it alone.
    Other services: no change needed.
    
  ✓ TECHNOLOGY FREEDOM:
    ML service: Python + PyTorch
    Order service: Go
    Legacy service: Java
    Each team chooses the right tool.
    
  ✓ TEAM INDEPENDENCE:
    10 teams × 10 services = each team owns their service.
    Deploy independently. No coordination required.
    Clearer ownership and accountability.
    
  ✓ FAILURE ISOLATION:
    Recommendation service crashes → order service still works.
    (Properly designed with circuit breakers, see topic 14.16)

MICROSERVICES WEAKNESSES:
  ✗ DISTRIBUTED SYSTEM COMPLEXITY:
    Every inter-service call adds: latency, failure modes, retries, circuit breakers.
    10 services × 5 dependencies each = 50 failure points to manage.
    
  ✗ DISTRIBUTED TRANSACTIONS (topic 14.7-14.8):
    Need atomicity across services? Saga patterns. Outbox. Complexity.
    
  ✗ OPERATIONAL OVERHEAD:
    10 services × health checks, deployment pipelines, monitors, alerts = 10x work.
    
  ✗ TESTING COMPLEXITY:
    Integration tests need all services running.
    Contract testing required to catch API compatibility breaks.
    
  ✗ DEBUGGING DIFFICULTY:
    "Why did this order fail?" — trace through 6 services, 6 log systems.
    Requires distributed tracing (OpenTelemetry) to be useful.
    
  ✗ VERSIONING OVERHEAD:
    Service A calls Service B v2.3.1.
    Service B deploys v2.4.0 with a breaking change.
    Service A now breaks. All callers must be updated.
```

## 3. The Decision Framework

```
CHOOSE MONOLITH WHEN:
  ✓ Team is < 10-15 engineers
  ✓ Domain is not well-understood yet (premature decomposition is painful)
  ✓ Single deployment unit simplifies operations significantly
  ✓ The system is small enough that one codebase works well
  ✓ Strong consistency requirements across many operations
  ✓ You're starting a new product (validate before you complicate)

CHOOSE MICROSERVICES WHEN:
  ✓ Team is > 20-30 engineers AND different parts have different team ownership
  ✓ Different components have dramatically different scaling requirements
  ✓ Different components need different technology stacks
  ✓ Independent deployment cadence is genuinely required
  ✓ You've already validated the domain (stability prevents premature splitting)
  ✓ You have strong observability, CI/CD, and operational infrastructure

THE LITMUS TEST:
  "Can a new engineer make a change and deploy it without affecting
   other teams or requiring cross-team coordination?"
  
  Monolith: NO (shared codebase, shared deployment)
  Microservices: YES (each service independently deployable)
  
  If the answer is "yes but we have processes to coordinate" —
  you don't have real microservices, you have a distributed monolith.
```

## 4. Real-World Context

### Amazon's Evolution

Amazon started as a monolith (Amazon.com, 1995). As the company grew and the codebase became unmanageable, they migrated to service-oriented architecture — the technical implementation of the famous "Two Pizza Team" principle (if a team needs more than two pizzas to feed everyone, it's too big).

This wasn't microservices-as-fashion — it was a response to concrete problems: the codebase was so large that changing the checkout process required coordinating with 20 teams. Each team wanted to deploy at their own pace. The monolith was creating real organizational friction.

### Monzo's Microservices from Day One

Monzo (UK digital bank, founded 2015) built microservices from their first day. With 1,500+ microservices (as of 2021), their architecture is extremely fine-grained. The benefit: each service can be independently scaled, deployed, and owned. The cost: massive operational complexity. They invested heavily in internal tooling to make this manageable.

Their advice: "We'd probably start with a monolith and migrate incrementally if we were starting again. The operational overhead of microservices was significant."

### The Shopify Modular Monolith

Shopify runs largely on a Rails monolith (as of 2024 — the core platform). They've invested heavily in making it a well-structured modular monolith with clear boundaries between components. For specific high-scale components (checkout, payments), they've extracted microservices. But the core remains a monolith with carefully managed module boundaries.

## 5. Senior Engineer Perspective

**The microservices hype cycle:**

From ~2015 to ~2022, microservices were presented as the obvious correct architecture for everything. This led to countless startups with 3 engineers building microservices, which meant they were spending 70% of their time on infrastructure and 30% on product features.

The pendulum has swung back. The Amazon Prime Video "monolith > microservices" post (2023) is the most notable example: moving from microservices to a monolith reduced cost by 90% for their specific use case.

**The migration order:**

The correct order is: monolith first → prove the domain → extract services where the pain is concrete.

"Where the pain is concrete" means:
- A specific component needs to be deployed 10x per day independently
- A specific component needs 100x more scaling than the rest
- A specific team is constantly blocked by another team's code
- A specific component needs a fundamentally different technology

These are signals to extract one service. Not "microservices are best practice, therefore let's decompose everything."
