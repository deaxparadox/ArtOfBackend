## 1. What Is It? (Theory)

In a microservices architecture, services need to find each other. But service instances are ephemeral — they're deployed, scaled up and down, and restarted constantly. You can't hardcode IP addresses.

Service discovery is the mechanism by which services find the network location (host + port) of other services dynamically.

**The real-world analogy:**

In a city, businesses have fixed addresses (physical buildings). In a microservices cluster, services have dynamic "addresses" (IPs that change every deployment). Service discovery is like a city's 411 service: "Where is the nearest pharmacy?" → discovers current location.

## 2. The Two Discovery Patterns

### Client-Side Discovery

```
CLIENT-SIDE DISCOVERY:
  
  Service Registry (Consul, Eureka, etcd):
    A database of "service name → list of healthy instance IPs"
    
  Client (ServiceA):
    1. Query registry: "Where are instances of ServiceB?"
    2. Registry returns: ["10.0.1.45:8080", "10.0.1.46:8080", "10.0.1.47:8080"]
    3. Client selects one (round-robin, random, least-connections)
    4. Client calls the selected instance directly
  
  PROS:
  ✓ Client has full control over load balancing strategy
  ✓ No extra network hop (direct call after discovery)
  ✓ Client can make sophisticated decisions (prefer same AZ, avoid slow instances)
  
  CONS:
  ✗ Every service must implement service discovery logic
  ✗ Every service must have the discovery library for its language
  ✗ More code per service
  
  USED BY: Netflix's Ribbon (Java), some Go service mesh implementations

SERVICE REGISTRATION:
  Services register on startup, deregister on shutdown.
  Health check: registry polls the service health endpoint.
  If unhealthy: removed from the registry.
  
  // Go: registering with Consul
  agent.ServiceRegister(&api.AgentServiceRegistration{
      ID:      "order-service-1",
      Name:    "order-service",
      Address: "10.0.1.45",
      Port:    8080,
      Check: &api.AgentServiceCheck{
          HTTP:     "http://10.0.1.45:8080/health",
          Interval: "10s",
          Timeout:  "3s",
      },
  })
```

### Server-Side Discovery (Most Common in Kubernetes)

```
SERVER-SIDE DISCOVERY:
  
  Load Balancer / Service Proxy sits in front:
    ServiceA → Load Balancer → ServiceB (one of several instances)
    ServiceA doesn't know how many ServiceB instances exist.
    ServiceA just calls a stable DNS name or VIP.
  
  KUBERNETES DNS-BASED DISCOVERY:
    Deploy ServiceB as a Kubernetes Service:
    service name = "payment-service" in namespace "production"
    
    DNS entry automatically created:
    payment-service.production.svc.cluster.local
    
    ServiceA calls: http://payment-service/pay
    Kubernetes DNS resolves → ClusterIP (virtual IP)
    kube-proxy routes to a healthy Pod
    
  PROS:
  ✓ No discovery code in services (just use DNS)
  ✓ Works with any language/framework
  ✓ Kubernetes handles everything automatically
  
  CONS:
  ✗ Extra hop through load balancer/kube-proxy
  ✗ Less client control over selection strategy
  
  USED BY: Kubernetes (default), AWS ELB, Nginx upstream
```

## 3. Consul: Service Discovery + Configuration + Health Checking

```
CONSUL IS THREE THINGS:
  1. Service Registry: register/discover services
  2. KV Store: store configuration
  3. Health Checking: monitor service health

# Register service in Consul (JSON config)
{
    "service": {
        "name": "payment-service",
        "id": "payment-service-1",
        "address": "10.0.1.45",
        "port": 8080,
        "tags": ["v2.3.1", "production"],
        "check": {
            "http": "http://10.0.1.45:8080/health",
            "interval": "10s",
            "deregister_critical_service_after": "30s"
        }
    }
}

# Discover services via Consul API
curl http://consul:8500/v1/health/service/payment-service?passing=true
# Returns: list of healthy instances with their IP:port

# Via Consul DNS (services can query by DNS):
dig @consul-dns-port payment-service.service.consul
# Returns A records for all healthy instances
```

## 4. Service Discovery in Kubernetes

In Kubernetes, service discovery is built-in via CoreDNS and Kubernetes Services:

```yaml
# Kubernetes Service: stable network endpoint for pods
apiVersion: v1
kind: Service
metadata:
  name: payment-svc
  namespace: production
spec:
  selector:
    app: payment         # routes to pods with this label
  ports:
  - port: 80
    targetPort: 8080
  type: ClusterIP        # stable virtual IP within the cluster

# DNS resolution within cluster:
# payment-svc                          (same namespace)
# payment-svc.production              (cross-namespace)
# payment-svc.production.svc.cluster.local  (fully qualified)

# Environment variables (automatically injected by Kubernetes):
# PAYMENT_SVC_SERVICE_HOST=10.96.123.45
# PAYMENT_SVC_SERVICE_PORT=80
```

## 5. Senior Engineer Perspective

**For Kubernetes: you don't need Consul:**

Kubernetes provides service discovery out-of-the-box via DNS. Unless you're running non-Kubernetes workloads that also need service discovery, adding Consul to a Kubernetes cluster is unnecessary complexity.

Use Consul when:
- You have services outside Kubernetes (bare metal, VMs, legacy)
- You need cross-datacenter service discovery
- You need the configuration store features of Consul

For pure Kubernetes: just use Kubernetes Services and DNS.
