## Why Distributed Systems Are Hard (14.1)

The 8 fallacies of distributed computing (see topic 0.12) summarize it: everything that's true about a single machine stops being true across a network. Add to this:

```
PARTIAL FAILURE: the hardest problem in distributed systems
  In a single machine: something either works or it crashes entirely.
  In a distributed system: a service can be alive but not processing requests,
  responding but returning wrong answers, or slow rather than down.
  "Did the request succeed?" becomes unanswerable across a network.

TIMING UNCERTAINTY:
  Messages arrive out of order (UDP).
  Clocks differ across machines (NTP doesn't solve this entirely).
  "Event A happened before Event B" can be impossible to determine.

CONSISTENCY vs AVAILABILITY:
  During a network partition: you must choose one (CAP theorem, topic 0.13).

THE TWO GENERALS PROBLEM:
  Provably impossible to achieve perfect consensus over an unreliable network.
  Practical systems use probabilistic guarantees instead of perfect guarantees.
```

## Consensus: Raft Algorithm (14.5)

```
RAFT: the consensus algorithm behind etcd (Kubernetes' brain) and CockroachDB

ROLES:
  Leader:   handles all client requests, replicates to followers
  Follower: accepts log entries from leader, votes in elections
  Candidate: requesting votes to become leader

LEADER ELECTION:
  If no heartbeat from leader for election timeout:
  → Follower becomes Candidate, increments term, requests votes
  → Majority of votes → becomes Leader
  → Leader sends heartbeats to maintain authority

LOG REPLICATION:
  Leader receives request → appends to its log → sends to followers
  Followers append to log → send acknowledgment
  Majority acknowledged → Leader commits → sends commit notification
  
  QUORUM: majority (N/2 + 1) must acknowledge before commit
  3-node cluster: 2 must agree
  5-node cluster: 3 must agree

  WHY QUORUM: if leader dies after commit but before notifying all followers,
  any node with the committed log can become leader.
  A node without the commit cannot win election (can't get quorum votes).
```

## Microservices Principles (14.9-14.21)

### Monolith vs. Microservices

```
MONOLITH:
  ✓ Simple to develop, test, deploy
  ✓ No network overhead between components
  ✓ ACID transactions across the whole system
  ✓ Easy to refactor (rename function → all callers updated)
  ✗ Hard to scale individual components
  ✗ Large team coordination overhead
  ✗ One slow component slows everything
  
MICROSERVICES:
  ✓ Independent deployment (deploy payment service without deploying users service)
  ✓ Independent scaling (scale only the hot services)
  ✓ Technology freedom (Python for ML, Go for APIs, Java for batch)
  ✓ Failure isolation (payment service down ≠ product catalog down)
  ✗ Network overhead (10µs function call → 1ms service call)
  ✗ No cross-service transactions (must use sagas)
  ✗ Distributed debugging (need tracing across services)
  ✗ Operational complexity multiplied by number of services

WHEN TO MOVE TO MICROSERVICES:
  Conway's Law: your architecture will mirror your org structure.
  If you have 5 teams that work independently → consider microservices.
  If you have 1 team → monolith is almost always better.
  
  Start monolith → extract services when:
  1. A specific component needs independent scaling
  2. Teams are stepping on each other constantly
  3. One component needs a different technology
  4. Deployment coupling is causing too much friction
```

### The Modular Monolith (14.10)

```python
# Modular monolith: strict module boundaries, shared codebase
# Gives independence without distributed systems complexity

# Module: orders
# orders/
#   api.py       (HTTP endpoints)
#   service.py   (business logic)
#   repository.py (data access)
#   events.py    (events published by this module)
#   __init__.py  (public interface — only these can be imported externally)

# orders/__init__.py
from .service import OrderService, CreateOrderRequest
from .events import OrderCreatedEvent
# Nothing else should be imported by other modules

# External access: only through public interface
from orders import OrderService  # OK
from orders.repository import OrderRepository  # VIOLATION — internal detail
```

### Service Decomposition

```
DECOMPOSITION STRATEGIES:

BY BUSINESS CAPABILITY:
  Orders service, Payments service, Inventory service, Users service
  Each service owns one business domain end-to-end
  
BY SUBDOMAIN (DDD):
  Identify bounded contexts: where concepts have different meanings
  "Product" in catalog (description, images) vs "Product" in inventory (stock levels)
  → Separate services with separate Product models

BY DATA OWNERSHIP:
  Each service owns its own database
  Cross-service queries become API calls (or async events)
  Service A NEVER reads Service B's database directly
```

## Distributed Transactions: Saga Pattern (14.8, 14.21-14.22)

```python
# Traditional ACID transaction: works within one service
# Cross-service: use Saga pattern

# SAGA: sequence of local transactions, each publishing events to trigger the next
# On failure: compensating transactions undo completed steps

# CHOREOGRAPHY (event-driven, decentralized):
# Each service listens for events and decides its next action

# Order Service publishes:
await event_bus.publish("order.created", {"order_id": 123, "amount": 5000})

# Inventory Service listens:
@event_handler("order.created")
async def reserve_inventory(event):
    success = await inventory.reserve(event["order_id"])
    if success:
        await event_bus.publish("inventory.reserved", {"order_id": event["order_id"]})
    else:
        await event_bus.publish("inventory.reservation_failed", {"order_id": event["order_id"]})

# Payment Service listens for inventory.reserved:
@event_handler("inventory.reserved")
async def charge_payment(event):
    success = await payment.charge(event["order_id"])
    if success:
        await event_bus.publish("payment.completed", {"order_id": event["order_id"]})
    else:
        # Compensate: tell inventory to release reservation
        await event_bus.publish("payment.failed", {"order_id": event["order_id"]})

@event_handler("payment.failed")
async def release_inventory_compensation(event):
    await inventory.release(event["order_id"])

# ORCHESTRATION (centralized saga orchestrator):
class OrderSaga:
    async def execute(self, order_id: int):
        # Step 1
        if not await inventory_service.reserve(order_id):
            await self.compensate([])
            return False

        # Step 2
        if not await payment_service.charge(order_id):
            await self.compensate(["inventory"])
            return False

        # Step 3
        await order_service.confirm(order_id)
        return True

    async def compensate(self, completed_steps: list):
        if "inventory" in completed_steps:
            await inventory_service.release(self.order_id)
```

## Conway's Law (14.20) and When NOT to Use Microservices (14.21)

```
CONWAY'S LAW:
  "Organizations which design systems are constrained to produce designs
   which are copies of the communication structures of these organizations."
  
  If you have 3 teams: you'll have 3 services (or a monolith with 3 modules).
  Trying to have 10 services with 2 teams: constant coordination overhead.
  
  INVERSE CONWAY MANEUVER:
  Design org structure to match desired system architecture.
  Want microservices? Create autonomous teams that can own a service end-to-end.

WHEN NOT TO USE MICROSERVICES:
  ✗ Startup / small team (< 10 engineers): operational overhead outweighs benefits
  ✗ When you don't understand the domain yet (premature decomposition)
  ✗ When you need cross-service transactions frequently (saga is complex)
  ✗ When latency is critical (service calls add overhead)
  ✗ When you don't have good observability (distributed debugging is hard)
  
  Amazon's famous internal memo: Jeff Bezos mandated APIs between teams in 2002.
  This enabled AWS. But Amazon was already a large organization.
  
  Don't cargo-cult microservices for a 5-person team.
  Start with a well-structured monolith. Extract services when the pain is real.
```

## Global ID Generation (14.26)

```python
# Snowflake ID: Twitter's distributed ID generation
# 64 bits: 41-bit timestamp + 10-bit machine ID + 12-bit sequence

import time
import threading

class SnowflakeIDGenerator:
    EPOCH = 1577836800000  # 2020-01-01 as offset (42 years of IDs)
    MACHINE_ID_BITS = 10
    SEQUENCE_BITS = 12

    def __init__(self, machine_id: int):
        self.machine_id = machine_id & ((1 << self.MACHINE_ID_BITS) - 1)
        self.sequence = 0
        self.last_timestamp = -1
        self._lock = threading.Lock()

    def generate(self) -> int:
        with self._lock:
            ts = int(time.time() * 1000) - self.EPOCH
            if ts == self.last_timestamp:
                self.sequence = (self.sequence + 1) & ((1 << self.SEQUENCE_BITS) - 1)
                if self.sequence == 0:  # sequence overflow: wait for next ms
                    while ts <= self.last_timestamp:
                        ts = int(time.time() * 1000) - self.EPOCH
            else:
                self.sequence = 0
            self.last_timestamp = ts
            return (ts << (self.MACHINE_ID_BITS + self.SEQUENCE_BITS)) | \
                   (self.machine_id << self.SEQUENCE_BITS) | self.sequence

# 4096 IDs/ms/machine, globally unique, sortable by time
# Used by Twitter, Discord (which generates 1M+ IDs per day for messages)
```
