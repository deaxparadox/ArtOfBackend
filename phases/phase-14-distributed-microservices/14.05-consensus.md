## 1. What Is It? (Theory)

Consensus algorithms solve the fundamental distributed systems problem: **how do multiple nodes agree on a single value, even if some nodes fail?**

This problem underlies almost everything in distributed systems: leader election, distributed transactions, replicated logs, and cluster coordination. If you can solve consensus, you can build reliable distributed systems on top of it.

**The consensus problem:**

```
GIVEN: N nodes, some may fail
GOAL:  All surviving nodes agree on the same value
CONSTRAINTS:
  - Safety: two nodes never commit different values
  - Liveness: the system eventually makes progress (commits something)
  - Fault tolerance: the system works as long as a majority (quorum) of nodes are healthy
```

**Why majority quorum?**

If you require a majority (N/2 + 1) to agree before committing, then any two majorities overlap by at least one node — ensuring that the node with the latest data is always included in any quorum:

```
5-node cluster:
  Quorum = 3 (majority)
  Quorum A: [1, 2, 3]
  Quorum B: [3, 4, 5]
  Overlap: [3] ← always at least one common node
  
  The overlap node ensures no two majorities can make inconsistent decisions.
  Even if 2 nodes fail: 3 remaining form a quorum and can continue.
```

## 2. Paxos: The Original Consensus Algorithm

Paxos (Leslie Lamport, 1989) was the first formally proven consensus algorithm. It's notoriously difficult to understand and implement correctly — Lamport wrote a "simplified" version in 2001 and it was still considered complex.

```
PAXOS PHASES:

Phase 1 (PREPARE):
  Proposer → Prepare(n) → Acceptors
  n = ballot number (unique, monotonically increasing)
  
  Acceptors respond:
  - Promise: "I won't accept any proposal numbered < n"
  - If I've already accepted a proposal: tell the proposer what it was

Phase 2 (ACCEPT):
  If proposer received promises from a majority:
  → Proposer: Accept(n, value) → Acceptors
    value = the proposer's value, OR the highest-numbered value already accepted
  
  Acceptors: accept if they haven't promised a higher-numbered proposal
  
  If majority accepts: value is CHOSEN (consensus reached)

Phase 3 (LEARN):
  Inform learners of the chosen value.

WHY PAXOS IS HARD:
  Multi-Paxos (for repeated decisions on a log): complex
  Leader changes: complex
  Disk failures: complex
  The gap between the paper and a correct implementation: enormous
  
  Google's Chubby lock service uses Paxos.
  Google's own engineers described implementing it as "significant challenge."
  A correct Paxos implementation is typically 10,000+ lines of code.
```

## 3. Raft: Understandable Consensus

Raft (Diego Ongaro and John Ousterhout, 2014) was explicitly designed to be understandable. The paper is titled "In Search of an Understandable Consensus Algorithm." It decomposes the consensus problem into three relatively independent subproblems: leader election, log replication, and safety.

```
RAFT ROLES:
  FOLLOWER:  Default state. Accepts log entries from leader. Votes in elections.
  CANDIDATE: Transitional state during election. Requests votes.
  LEADER:    One per cluster. Handles all client requests. Replicates log.

RAFT ALGORITHM:

LEADER ELECTION:
  Followers expect a heartbeat from the leader every N milliseconds.
  If no heartbeat within timeout: follower becomes CANDIDATE.
  
  Candidate increments its "term" (election epoch) and requests votes:
  "I'm requesting votes for term 7. My log is up to date."
  
  Other nodes vote YES if:
    - They haven't already voted in this term
    - The candidate's log is at least as up-to-date as theirs
  
  If candidate receives votes from a majority: becomes LEADER for term 7.
  Immediately sends heartbeats to prevent new elections.

LOG REPLICATION (once a leader is elected):
  1. Client sends a request to the leader.
  2. Leader appends the request to its log (as an "uncommitted" entry).
  3. Leader sends AppendEntries RPC to all followers.
  4. Followers append to their local logs, acknowledge.
  5. When a MAJORITY of followers acknowledge: leader COMMITS the entry.
  6. Leader responds success to client.
  7. Leader notifies followers the entry is committed (in next AppendEntries).

SAFETY PROPERTY:
  A log entry can only be committed if it's present on a majority of servers.
  Therefore: no two leaders can commit conflicting entries.
  (They'd need overlapping majorities, which would prevent conflicting commits.)
```

### Raft in Practice: etcd

etcd (used by Kubernetes) is built on Raft:

```bash
# etcd cluster: 3 or 5 nodes for HA (odd number for clear majority)
# 3 nodes: tolerates 1 failure (2/3 majority)
# 5 nodes: tolerates 2 failures (3/5 majority)
# 7 nodes: tolerates 3 failures (4/7 majority)
# Beyond 7: write performance degrades (too many nodes to get quorum from)

# Check etcd cluster health
ETCDCTL_API=3 etcdctl endpoint health \
    --endpoints=https://etcd-1:2379,https://etcd-2:2379,https://etcd-3:2379

# Check which node is the leader
ETCDCTL_API=3 etcdctl endpoint status \
    --endpoints=https://etcd-1:2379,https://etcd-2:2379,https://etcd-3:2379 \
    --write-out=table
# Shows: IS LEADER column for each node

# Write only succeeds if majority are available
# Read can be: serialized (strong, via leader) or linearized (via quorum)
ETCDCTL_API=3 etcdctl get /kubernetes/leader --consistency linearizable
```

## 4. Real-World Context

### How Kubernetes Relies on Raft

Kubernetes uses etcd as its sole source of truth for cluster state. All cluster objects (pods, services, deployments, secrets) are stored in etcd. When you run `kubectl apply -f deployment.yaml`:

1. `kubectl` sends the request to the Kubernetes API server
2. API server validates and writes to etcd
3. etcd replicates the write across its Raft cluster (typically 3 nodes)
4. Only after a majority of etcd nodes commit the write does the API server respond success
5. The Kubernetes scheduler and controllers read from etcd and act accordingly

If the Kubernetes etcd cluster loses quorum (more than half the nodes fail), **the entire Kubernetes cluster stops accepting new deployments** (CP behavior: consistent but not available during partition). This is why production Kubernetes should always have 3+ etcd nodes across different availability zones.

### CockroachDB and Multi-Region Raft

CockroachDB uses a separate Raft group for each partition (range) of data. This means thousands of Raft instances running simultaneously, each managing one slice of the keyspace. When you write a row, the Raft group for that row's range replicates the write across 3-5 nodes.

For global deployments: CockroachDB's "super region" feature uses Raft to maintain a majority of replicas within a region (for low-latency local writes) while also maintaining replicas globally (for disaster recovery). This is Raft-based geo-distribution in production.

## 5. Senior Engineer Perspective

**You almost certainly don't implement consensus:**

Understanding Raft conceptually is important for reasoning about distributed systems. Actually implementing a consensus algorithm from scratch in production is something almost no backend engineer should do. Use etcd, ZooKeeper, or Consul — systems where experts have spent years ensuring correctness.

When you encounter consensus in practice, it's usually in the form of: "Is etcd healthy? Does the Kubernetes cluster have quorum? Is ZooKeeper available for service discovery?" Understanding Raft helps you answer these questions and reason about failure modes.

**The quorum math matters for availability:**

3-node cluster: tolerate 1 failure (always have at least 2 for quorum). Down for maintenance? Take only 1 node down at a time.

5-node cluster: tolerate 2 failures. During a rolling upgrade: safe to take 2 nodes down simultaneously if the other 3 are healthy.

This math directly determines how you design your deployment and maintenance procedures for consensus-based infrastructure.
