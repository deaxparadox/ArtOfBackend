## 1. API Gateway Pattern (14.14)

### What Is It?

An API Gateway is a single entry point that all clients (web, mobile, external partners) use to access your microservices. It routes requests to the appropriate backend services, handles cross-cutting concerns, and shields clients from the internal service topology.

```
WITHOUT API GATEWAY (messy):
  Web App → directly calls Order Service
  Web App → directly calls Payment Service
  Web App → directly calls User Service
  Mobile App → directly calls different endpoints on same services
  External Partner → calls services with different auth mechanism
  
  Problems:
  - Web and mobile need different response formats
  - Auth logic duplicated in every service
  - Rate limiting duplicated in every service
  - Clients must know about all service URLs and ports
  - Service refactoring requires updating all clients

WITH API GATEWAY (clean):
  Web App → API Gateway → routes to → Order/Payment/User Services
  Mobile App → API Gateway → routes to → same services (possibly different response)
  External Partner → API Gateway → routes to → (with different auth)
  
  Gateway handles: auth, rate limiting, SSL termination, logging, routing
  Services: pure business logic only
```

### What the Gateway Handles

```python
# API Gateway responsibilities (example using Kong or AWS API Gateway concepts)

ROUTING:
  GET  /api/v1/users/*    → user-service
  POST /api/v1/orders     → order-service
  GET  /api/v1/products/* → catalog-service

AUTHENTICATION:
  Validate JWT token before forwarding to service.
  Inject user identity as a header: X-User-Id, X-User-Role
  Services trust this header (no need for each service to verify JWT)

RATE LIMITING:
  100 req/min per authenticated user
  10 req/min per IP for unauthenticated
  No rate limiting code in any service

SSL TERMINATION:
  Accept HTTPS from clients.
  Forward HTTP internally (simpler for internal services).

CORS:
  Handle preflight requests centrally.
  Services don't need CORS headers.

REQUEST LOGGING:
  Log every request with: method, path, status, latency, user_id
  Services focus on business logging, not access logging.

RESPONSE TRANSFORMATION:
  Aggregate responses from multiple services into one response.
  Strip internal fields before returning to client.

CIRCUIT BREAKING:
  If order-service is down: return 503 from gateway.
  Don't forward requests that will fail.
```

### API Gateway Products

```
SELF-HOSTED:
  Kong: Lua-based, plugin ecosystem, Postgres-backed config
  Nginx + custom modules: maximum performance, manual config
  Traefik: cloud-native, automatic config from K8s annotations
  
MANAGED (AWS):
  AWS API Gateway (HTTP API): ~$1/million requests, Lambda integration
  AWS API Gateway (REST API): more features, more expensive
  
MANAGED (GCP):
  Cloud Endpoints: Google's API management
  Apigee: enterprise API management (acquired by Google)
  
SERVICE MESH (alternative approach):
  Istio / Linkerd: sidecars handle routing, auth, observability per service
  See topics 14.18-14.19
```

## 2. BFF: Backend for Frontend Pattern (14.15)

### What Is It?

BFF (Backend for Frontend) is a variation of the API Gateway where **each frontend client has its own dedicated backend service** that aggregates and formats data specifically for that client's needs.

```
THE PROBLEM WITH ONE API GATEWAY FOR ALL CLIENTS:

  Mobile App:
    Needs: lightweight responses (small screen, limited bandwidth)
    Screen: product name, thumbnail, price, "Add to Cart" button
    API call: GET /products/123
    Needs: 3 fields
    
  Web App:
    Needs: rich responses (large screen, fast connection)
    Page: full product details, reviews, related products, inventory, recommendations
    API call: GET /products/123
    Needs: 20+ fields
    
  Compromise API:
    Returns everything → mobile gets too much (wasted bandwidth)
    OR returns minimal → web must make multiple calls (worse UX)
    
  Neither client is optimally served.

THE BFF SOLUTION:

  Mobile BFF (mobile-specific backend):
    GET /products/123 (from mobile app)
    → fetches from catalog-service, inventory-service
    → returns: {name, thumbnail_url, price, in_stock}
    → optimized for mobile: small, fast

  Web BFF (web-specific backend):
    GET /products/123 (from web app)
    → fetches from catalog-service, inventory-service, review-service, recommendation-service
    → aggregates + formats for web display
    → returns: full rich product page data

  Each BFF is owned by the frontend team (or a full-stack team).
  Decouples frontend evolution from backend services.
```

```python
# Python BFF example: aggregates data for web product page

@app.get("/products/{product_id}")
async def get_product_for_web(product_id: int, user_id: int | None = None):
    """
    Web BFF: aggregates catalog + inventory + reviews + recommendations.
    Optimized for the web product detail page.
    """
    # Fetch all data concurrently
    product, inventory, reviews, recommendations = await asyncio.gather(
        catalog_service.get_product(product_id),
        inventory_service.get_inventory(product_id),
        review_service.get_top_reviews(product_id, limit=5),
        recommendation_service.get_similar(product_id, user_id=user_id),
    )
    
    # Shape the response for the web product page
    return {
        "id": product.id,
        "name": product.name,
        "description": product.description,
        "price_cents": product.price_cents,
        "images": product.images,               # full image list
        "specifications": product.specifications,
        "stock": {
            "in_stock": inventory.quantity > 0,
            "quantity": inventory.quantity,
        },
        "reviews": {
            "average": reviews.average_rating,
            "count": reviews.total_count,
            "top_reviews": reviews.items,
        },
        "recommendations": [
            {"id": p.id, "name": p.name, "price": p.price_cents, "thumbnail": p.thumbnail}
            for p in recommendations
        ],
    }

# Mobile BFF: lighter response for the same product
@app.get("/products/{product_id}")  # running as mobile-bff service
async def get_product_for_mobile(product_id: int):
    product, inventory = await asyncio.gather(
        catalog_service.get_product(product_id),
        inventory_service.get_inventory(product_id),
    )
    # Minimal response for mobile
    return {
        "id": product.id,
        "name": product.name,
        "price_cents": product.price_cents,
        "thumbnail": product.thumbnail,
        "in_stock": inventory.quantity > 0,
    }
```

## 3. API Gateway vs. BFF vs. Service Mesh

```
API GATEWAY:
  One centralized entry point for all clients.
  Good for: common cross-cutting concerns (auth, rate limiting, routing).
  Not good for: client-specific aggregation (makes gateway complex).

BFF:
  Client-specific aggregation and formatting.
  Owned by frontend teams.
  Good for: optimizing responses per client.
  Not good for: shared concerns (each BFF would duplicate auth logic).

COMMON COMBINATION:
  API Gateway handles: auth, rate limiting, SSL, routing to BFFs
  BFFs handle: aggregation, formatting per client
  
  Internet → API Gateway (auth, rate limits) → Mobile BFF or Web BFF or Partner API
                                               → backend microservices

SERVICE MESH:
  Handles: mTLS between services, observability, circuit breaking
  Different layer than API Gateway (internal, not edge)
  Works alongside API Gateway, not instead of it.
```

## 4. Senior Engineer Perspective

**The BFF is often the missing pattern:**

Teams building for multiple clients (web, iOS, Android) often struggle with APIs that work for one client but are awkward for others. The BFF pattern solves this cleanly: each client gets an API shaped for its needs, owned by the team closest to that client.

**The API Gateway as a "God Service" antipattern:**

A single API Gateway that does routing + auth + rate limiting + aggregation + formatting + transformation becomes a complex service that every change must go through. Keep the gateway focused on cross-cutting concerns (auth, routing, rate limiting). Push aggregation and formatting into BFFs that are specific to each client. This distributes the work and avoids making the gateway a bottleneck.
