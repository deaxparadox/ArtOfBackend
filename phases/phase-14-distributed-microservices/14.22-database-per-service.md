## 1. Database-per-Service Pattern (14.22)

### What Is It?

The database-per-service pattern states that each microservice owns its own database — no two services share a database schema. This is arguably the most important constraint in microservices architecture.

```
THE FUNDAMENTAL RULE:
  Service A's database → accessible ONLY by Service A
  Service B's database → accessible ONLY by Service B
  
  If Service B needs data from Service A:
  → B calls A's API (synchronous)
  → OR A publishes an event that B consumes (asynchronous)
  → B NEVER directly queries A's database tables

WHY THIS RULE EXISTS:
  
  Without it: services are coupled at the database level.
  
  Order Service → reads payments table (directly)
  Payment Service → changes payments table schema
  Order Service → BREAKS (column renamed, type changed, table moved)
  
  The database becomes shared infrastructure → tight coupling → no independent deployment.
  
  This is the "shared database antipattern" (topic 14.23).

ENFORCING THE RULE:
  Option 1: Physical separation (each service has its own database server)
    Pros: complete isolation
    Cons: expensive, more infrastructure to manage
    
  Option 2: Schema separation (same server, different schemas/databases)
    Example: PostgreSQL: service_a_db, service_b_db (separate databases on same cluster)
    Pros: shared infrastructure, clear separation
    Cons: services could still bypass the rule (credentials control)
    
  Option 3: Schema prefix + strict access control
    Example: PostgreSQL schema per service, credentials restricted to own schema
    Pros: cheapest, good for smaller teams
    Cons: requires discipline and access control enforcement
```

## 2. Shared Database Antipattern (14.23)

```
THE SHARED DATABASE ANTIPATTERN:

  OrderService ──┐
  PaymentService ─┼── SAME DATABASE → tight coupling
  UserService ───┘

SIGNS YOU HAVE THIS PROBLEM:
  - Services query each other's tables directly (JOIN across service tables)
  - Schema changes require cross-team coordination
  - Services cannot be deployed independently (shared schema migration)
  - "We need to add a column to the orders table for payments logic"
    → orders and payments are in the wrong service boundary

THE MIGRATION PATH:
  1. Identify which tables each service "owns"
  2. Move tables to separate schemas/databases
  3. Replace cross-service queries with API calls or event subscriptions
  4. This is painful but necessary for true service independence

WHEN SHARED DATABASE IS ACCEPTABLE:
  During early development, before services have significant differences
  For small teams where operational simplicity > architectural purity
  When services are so tightly coupled they should be ONE service
```

## 3. Event-Driven Data Synchronization (14.24)

When services need data from each other but you can't share databases, event-driven sync is the solution:

```
PATTERN: SERVICE A publishes events → SERVICE B maintains its own copy

Example: Order Service needs user's email address (owned by User Service)

WRONG: Order Service calls User Service for every email needed
  → Tight coupling, Order Service breaks if User Service is down
  → N+1 problem: 1000 emails = 1000 calls to User Service

RIGHT: User Service publishes UserUpdated events → Order Service caches them

  User Service:
    When user profile changes → publish "user.updated" event with:
    {user_id: 123, email: "alice@example.com", name: "Alice"}
  
  Order Service:
    Subscribes to "user.updated" → maintains local user_emails table:
    | user_id | email                  | updated_at |
    | 123     | alice@example.com      | 2024-06-15 |
  
  Now: Order Service queries its own local copy, no dependency on User Service
  
  Trade-off:
  ✓ Order Service works even if User Service is down
  ✓ No cross-service queries
  ✗ Eventual consistency: email update may take seconds to propagate
  ✗ More storage (denormalized copy of user data)
  
  Acceptable for: most notification/communication use cases
  Not acceptable for: authorization checks (use real-time calls for security-critical data)
```

## 4. Distributed Caching Patterns (14.25)

```
DISTRIBUTED CACHE TOPOLOGY OPTIONS:

1. LOCAL CACHE (in-process):
   Each service instance has its own in-memory cache.
   + Fastest (no network hop)
   - Inconsistent: instance A has new value, instance B has old value
   - No sharing across instances
   Best for: static configuration, rarely-changing data

2. SHARED REDIS (most common):
   All service instances share one Redis cluster.
   + Consistent across all instances
   + Survives service restarts
   - Network hop (~0.5ms)
   - Redis is a shared dependency
   Best for: session data, frequently accessed DB results, rate limiting

3. CACHE-ASIDE BY SERVICE BOUNDARY:
   Each service has its own Redis cluster.
   Catalog Service → Catalog Redis cache
   Order Service → Order Redis cache
   
   + No cross-service cache pollution
   + Independent scaling per service
   - More Redis clusters to manage
   Best for: mature microservices with clear data ownership

4. READ-YOUR-WRITES ACROSS SERVICES:
   Problem: User updates profile → cache invalidated on UserService Redis
            But OrderService has cached old profile → stale
   
   Solution: Event-driven cache invalidation
   UserService: publish "user.updated" event
   OrderService: subscribes → invalidates its cached copy of that user
```

## 5. Global ID Generation (14.26)

In microservices, multiple services create IDs for entities. These IDs must be:
- **Globally unique** (no collisions across services and data centers)
- **Sortable by time** (for pagination, ordering, and debugging)
- **Not dependent on a central coordinator** (which would be a single point of failure)

```
SNOWFLAKE IDs (Twitter, 2010):

64-bit integer structured as:
  41 bits: millisecond timestamp (since Jan 1, 2010)
  10 bits: worker ID (datacenter 5 bits + machine 5 bits)
  12 bits: sequence number (within same millisecond)
  
  → 2^41 ms ≈ 69 years of IDs before overflow
  → 2^10 = 1024 workers
  → 2^12 = 4096 IDs per worker per millisecond
  → 4 billion IDs per second total capacity
  
  Sortable: IDs generated later are larger (timestamp is the high bits)
  Globally unique: worker ID ensures no two workers generate the same ID
  No coordinator: each worker generates independently

ULID (Universally Unique Lexicographically Sortable Identifier):

  01JKXM2Z7Q3R4S5T6U7V8W9X0A
  ←10 chars→ ← 16 chars        →
  Timestamp   Randomness
  (48 bits)   (80 bits)
  
  Human-readable (vs. binary Snowflake)
  URL-safe (no special characters)
  Sortable (lexicographic sort = chronological sort)

UUID v7 (2023 standard):

  0190a0f6-3d7b-7000-8f8e-e9e4c4e47a82
  ↑timestamp↑   ↑ver↑  ↑random         ↑
  
  Standard UUID format (works everywhere UUIDs work)
  Time-ordered (like ULID but in UUID format)
  Globally unique (128 bits total)
  
  PostgreSQL 17+: gen_uuidv7() built-in

CHOOSING:
  Simple distributed service needing sortable IDs: UUID v7 (most compatible)
  Need short, URL-safe IDs: ULID
  High-throughput, custom coordinator: Snowflake
  Simple but sufficient: UUID v4 (random, not sortable)
```

```python
# UUID v7 (using python-ulid or manual implementation)
import time
import os
import struct

def generate_uuid_v7() -> str:
    """Generate a UUID v7 (time-ordered UUID)."""
    timestamp_ms = int(time.time() * 1000)
    random_bytes = os.urandom(10)
    
    # Pack: 48-bit timestamp + version (7) + 74 random bits
    high = (timestamp_ms << 16) | 0x7000 | (random_bytes[0] << 4 & 0x0FFF)
    mid = ((random_bytes[0] & 0x3F) << 8) | random_bytes[1]
    low = struct.unpack(">Q", random_bytes[2:])[0] & 0x3FFFFFFFFFFFFFFF
    
    return f"{high:016x}-{mid:04x}-{low:016x}"

# Or use the uuid library (Python 3.13+):
# import uuid
# uuid.uuid7()  # coming in Python 3.13

# Or use the python-ulid package:
from ulid import ULID
id = ULID()  # generates a time-ordered, URL-safe ID
print(str(id))  # 01JM3NVW0T...
```
