## 1. Circuit Breakers in Microservices (14.16)

Circuit breakers in topic 2.13 covered the application-level pattern. In microservices, they're critical for preventing cascade failures.

```
MICROSERVICES CIRCUIT BREAKER SCENARIO:

Normal: OrderService → PaymentService (responding in 50ms)

PaymentService starts timing out (DB is overloaded):
  OrderService waits 5 seconds per request
  Thread pool fills with waiting threads
  OrderService becomes slow (waiting for PaymentService)
  Users complain about slow orders
  OrderService threads exhausted
  OrderService stops responding entirely
  
  ONE SLOW SERVICE TOOK DOWN A HEALTHY SERVICE.
  This is cascade failure.

WITH CIRCUIT BREAKER:
  PaymentService starts timing out.
  Circuit breaker opens after 5 failures.
  
  Now: OrderService calls PaymentService
  → Circuit breaker is OPEN
  → Returns error immediately (no waiting)
  → OrderService: "payment temporarily unavailable"
  → User: "payment processing delayed, we'll try again"
  → OrderService threads NOT exhausted
  → OrderService continues serving other requests
  
  After 30 seconds: circuit tries one request (HALF-OPEN)
  If PaymentService responds: circuit CLOSES. Normal operation.
  If still failing: stays OPEN another 30 seconds.

IMPLEMENTATION WITH resilience4j (Java):
  CircuitBreaker cb = CircuitBreaker.ofDefaults("paymentService");
  // Configure: failure threshold, wait duration, slow call threshold
  
  // Decorates the call:
  Supplier<Payment> supplier = CircuitBreaker.decorateSupplier(cb, 
      () -> paymentService.charge(orderId, amount));
  
  try {
      Payment result = supplier.get();
  } catch (CallNotPermittedException e) {
      // Circuit is OPEN - use fallback
      return queueForLaterPayment(orderId, amount);
  }

RESILIENCE4J CIRCUIT BREAKER CONFIG:
  failure-rate-threshold: 50        # open when 50% of calls fail
  slow-call-rate-threshold: 80      # open when 80% of calls are slow
  slow-call-duration-threshold: 2s  # "slow" = > 2 seconds
  wait-duration-in-open-state: 30s  # stay open for 30s before retry
  permitted-number-of-calls-in-half-open-state: 3  # test with 3 calls
```

## 2. Distributed Tracing (14.17)

When a user request touches 6 services before returning, traditional logging fails: the logs are spread across 6 different services' log files. Distributed tracing solves this by tracking the entire request as one trace across all services.

```
TRACE ANATOMY:

User request: "View order #123"

Trace ID: abc-123-def (same for ALL services)
  └── Span: api-gateway (10ms)
      └── Span: order-service (40ms)
          ├── Span: db.query "SELECT * FROM orders" (15ms)
          └── Span: payment-service/getStatus (20ms)
              └── Span: db.query "SELECT * FROM payments" (8ms)
          
Waterfall view (in Jaeger/Tempo):
  api-gateway     |████|
  order-service   |          ████████████████████|
  db.orders       |          |█████████|
  payment-svc     |                    |████████|
  db.payments     |                    |████|
  
Total: 50ms visible breakdown
"The 20ms in payment-service → most of it is the DB query (8ms) + network (12ms)"

KEY INSIGHT: you can instantly see WHERE the time is going across all services.
Without tracing: grep through 6 log files, manually correlate timestamps.
```

```python
# OpenTelemetry auto-instrumentation (adds tracing with minimal code)
from opentelemetry import trace
from opentelemetry.sdk.trace import TracerProvider
from opentelemetry.sdk.trace.export import BatchSpanProcessor
from opentelemetry.exporter.otlp.proto.grpc.trace_exporter import OTLPSpanExporter
from opentelemetry.instrumentation.fastapi import FastAPIInstrumentor
from opentelemetry.instrumentation.httpx import HTTPXClientInstrumentor
from opentelemetry.instrumentation.asyncpg import AsyncPGInstrumentor

# Setup (once at startup)
provider = TracerProvider()
provider.add_span_processor(
    BatchSpanProcessor(OTLPSpanExporter(endpoint="http://otel-collector:4317"))
)
trace.set_tracer_provider(provider)

# Auto-instrument HTTP framework and HTTP client and DB driver
FastAPIInstrumentor.instrument_app(app)
HTTPXClientInstrumentor().instrument()
AsyncPGInstrumentor().instrument()

# Now every request automatically:
# 1. Creates a span for each HTTP handler
# 2. Propagates W3C traceparent header to downstream services
# 3. Creates spans for DB queries
# 4. Sends all to your tracing backend (Jaeger, Tempo, Zipkin, Datadog)
```

## 3. Sidecar Pattern (14.18)

The sidecar pattern runs a helper process alongside each service instance, handling infrastructure concerns so the service doesn't have to.

```
SIDECAR CONCEPT:

Pod (in Kubernetes):
  ┌─────────────────────────────────────────┐
  │  Application Container (order-service)  │
  │  Port 8000                              │
  │  Code: business logic only              │
  └─────────────────────────────────────────┘
  ┌─────────────────────────────────────────┐
  │  Sidecar Container (Envoy proxy)        │
  │  Intercepts: all inbound/outbound traffic│
  │  Handles:                               │
  │    - mTLS (encrypt all traffic)        │
  │    - Circuit breaking                  │
  │    - Retries                           │
  │    - Metrics collection                │
  │    - Distributed tracing               │
  └─────────────────────────────────────────┘
  (Shared network namespace: sidecar sees all traffic)

SIDECAR BENEFITS:
  Service code: pure business logic
  Infrastructure concerns: handled by sidecar, language-agnostic
  
  If you have Go, Java, Python services: sidecar handles mTLS for all of them.
  No need for a TLS library in each language.
  No circuit breaker library in each language.
  One place to configure, one place to upgrade.
```

## 4. Service Mesh (14.19)

A service mesh is the combination of: sidecars on every service + a control plane to manage them.

```
SERVICE MESH ARCHITECTURE (Istio):

CONTROL PLANE (istiod):
  - Distributes configuration to all sidecars
  - Issues TLS certificates for mTLS
  - Manages traffic routing rules
  - Collects telemetry

DATA PLANE (Envoy sidecars on every pod):
  - Enforces mTLS
  - Implements circuit breaking
  - Collects metrics and traces
  - Routes traffic per control plane rules

WHAT THE MESH PROVIDES:
  ✓ Automatic mTLS between ALL services (zero-trust networking)
  ✓ Traffic management (canary deployments, A/B testing at service level)
  ✓ Observability (metrics, logs, traces for ALL service-to-service traffic)
  ✓ Resilience policies (circuit breaking, retries, timeouts)
  ✓ All without code changes to your services

WHEN TO USE A SERVICE MESH:
  ✓ You have many (20+) microservices and operational overhead is growing
  ✓ Zero-trust security is required (mTLS everywhere)
  ✓ You need traffic management without code changes
  ✓ You have polyglot services (hard to add resiliency libraries to each language)
  
WHEN NOT TO USE A SERVICE MESH:
  ✗ You have < 10 services (overhead outweighs benefits)
  ✗ Your team doesn't have Istio/service mesh expertise
  ✗ You can achieve the same goals with application-level libraries

THE COST:
  Istio adds complexity: control plane, sidecar injection, CRD-based config.
  Learning curve is steep.
  sidecar adds ~50-100MB RAM per pod, ~5-10ms latency per hop.
  
  Linkerd: simpler, lower overhead (Rust-based micro-proxy), less features.
  Consider Linkerd over Istio for teams new to service meshes.
```

## 5. Senior Engineer Perspective

**The observability imperative:**

Without distributed tracing, debugging a production issue in a 10-service system can take hours — manually correlating timestamps across log files. With OpenTelemetry (standardized, vendor-neutral), you get tracing, metrics, and logs from one SDK. Set it up from day one. It pays dividends every time there's an incident.

**The service mesh decision:**

Service meshes are powerful but complex. Many teams add Istio before they have the operational experience to run it effectively. The result: unexpected outages caused by Istio configuration mistakes, and engineers who aren't sure if a problem is in their service or in the mesh.

The recommendation: build experience with application-level circuit breakers and observability first. Add a service mesh when you have both the scale that justifies it and the operational expertise to run it. Consider Linkerd (simpler) before Istio (more powerful but harder).
