## 1. What Is It? (Theory)

Consistency models define the guarantees a distributed system makes about the order and visibility of operations. They form a spectrum from strongest (linearizability — operations appear instantaneous and globally ordered) to weakest (eventual consistency — all replicas will eventually agree, but no guarantee on when or in what order).

Understanding this spectrum lets you choose the right consistency level for each operation in your system — not treating all data as equally critical.

---

## 2. The Consistency Spectrum

```
STRONGEST ←────────────────────────────────────────────→ WEAKEST
LINEARIZABILITY   SEQUENTIAL   CAUSAL   MONOTONIC   EVENTUAL
```

### Linearizability (Strongest)

```
DEFINITION: All operations appear to execute atomically at a single point in time,
consistent with real time. After a write completes, all subsequent reads see it.

GUARANTEES:
  - A read always returns the latest write
  - No observer can see "time go backwards"
  - Global ordering of all operations

EXAMPLE:
  t=0: A writes x=1
  t=1: A reads x → must see 1
  t=1: B reads x → must see 1 (even though B didn't write it)
  
  If ANY reader can see x=0 after the write completes: NOT linearizable.

IMPLEMENTED BY:
  Zookeeper, etcd (via Raft consensus)
  Google Spanner (TrueTime)
  PostgreSQL + synchronous replication
  
COST:
  High latency (must coordinate across all replicas before responding)
  Lower availability during partitions
```

### Causal Consistency

```
DEFINITION: Operations that have a causal relationship are seen by all nodes
in the same order. Concurrent (causally unrelated) operations may be seen
in different orders by different nodes.

CAUSAL RELATIONSHIP:
  A happens before B if:
  - A and B are on the same node and A executes before B
  - A sends a message that B receives
  - Transitively through a chain of the above

EXAMPLE:
  Alice posts a comment.
  Bob reads the comment and replies to it.
  
  CAUSALITY: Alice's comment → Bob's reply
  
  CAUSAL CONSISTENCY GUARANTEES: everyone sees Alice's comment
  BEFORE they see Bob's reply (because Bob saw the comment before replying).
  
  But: Charlie's unrelated post and Alice's comment have no causal relationship.
  Different nodes may see them in different orders. That's fine.

EXAMPLE SYSTEM:
  Amazon's internal Dynamo-style systems
  Some configurations of Apache Cassandra
  
REAL-WORLD USE CASE:
  Social media feeds: you always see a reply after the post it replies to,
  even if the posts from other people appear in different orders.
```

### Read-Your-Writes Consistency

```
DEFINITION: After a client writes a value, subsequent reads by the SAME CLIENT
always see that write (or a later value). Other clients may still see stale data.

EXAMPLE:
  User updates their profile picture.
  User immediately navigates to their profile page.
  GUARANTEE: user sees their NEW profile picture (not the old one).
  
  Meanwhile: other users may still see the old picture for a few seconds.
  That's acceptable — read-your-writes is about the writer's experience.

IMPLEMENTATION:
  Route all requests from the same user session to the same replica.
  OR: write to primary, read from primary for a few seconds after writes.
  
IN PRACTICE:
  Most applications need this for good UX.
  "I just changed my password, why is my old password still working?" → violated read-your-writes
  
IMPLEMENTATION IN DynamoDB:
  Use ConsistentRead=True for reads immediately after writes.
  Or: maintain session consistency via the application layer.
```

### Eventual Consistency

```
DEFINITION: If no new writes occur, all replicas will eventually converge
to the same value. No guarantee on when or how many stale reads occur.

EXAMPLE:
  DNS: update an A record.
  Some nameservers see the change immediately.
  Others see it in 5 minutes. Others in 4 hours.
  Eventually: all nameservers have the new record.
  
  S3: update a file.
  Read immediately after update: might get old version.
  Read 1 second later: might still get old version.
  Read 5 seconds later: probably get new version.
  "Eventually": guaranteed convergence, but timing undefined.

IMPLEMENTATION STRATEGIES when replicas diverge:
  
  LAST-WRITE-WINS (LWW):
    Use timestamps. Latest write wins. Simple.
    Risk: clock skew means "latest" timestamp isn't actually latest.
    Common in: Cassandra, DynamoDB.
  
  MULTI-VERSION CONCURRENCY CONTROL (MVCC):
    Keep multiple versions. Client or system resolves conflicts.
    Used by: CouchDB (client-side merge), some Riak configurations.
  
  CONFLICT-FREE REPLICATED DATA TYPES (CRDTs):
    Data structures designed so any merge produces correct results.
    No conflicts possible because merge is mathematically defined.
    Examples: G-Counter (always grows), OR-Set (add-wins on merge).
    Used by: Riak, some Redis Cluster configs.
    
WHEN EVENTUAL CONSISTENCY IS APPROPRIATE:
  ✓ Shopping cart (losing a cart item during a 2-second partition is rare and survivable)
  ✓ View counters (exact count doesn't matter, approximate is fine)
  ✓ Recommendations (stale data for a few seconds is imperceptible)
  ✓ DNS records (designed for eventual propagation)
  
WHEN IT'S NOT APPROPRIATE:
  ✗ Account balances (can't show stale balance before a purchase)
  ✗ Inventory reservation (can't sell the last item twice)
  ✗ Authentication (password change should take effect immediately)
```

## 3. Choosing the Right Model

```
OPERATION                    CONSISTENCY NEEDED     REASON
────────────────────────────────────────────────────────────────────
Check account balance        Linearizable           Cannot be stale before debit
Debit an account             Linearizable           Atomic, no double-debit
Reserve inventory            Linearizable           Can't oversell
Place an order               Linearizable           Atomicity across tables
Read user profile            Read-your-writes       User sees their own changes
View product listing         Eventual               Stale catalog is fine
View comment thread          Causal                 Replies must appear after posts
View notification count      Eventual               Approximate count is fine
Update preferences           Read-your-writes       User sees their own changes
Analytics dashboard          Eventual               Stale by definition (batch)
Session authentication       Read-your-writes       Must see recent password change
```

## 4. Senior Engineer Perspective

**The consistency model design session:**

For every data entity in your system, have a design discussion: "What consistency model does this data need?" Write it down. The answer should inform your database choice, your replication configuration, and your application-level consistency mechanisms (idempotency, client-side logic).

Teams that skip this discussion usually end up with eventual consistency everywhere (the easiest to build) and discover too late that some operations needed stronger guarantees.

**The eventual consistency mental model:**

Think of eventually consistent data as "a best-effort view of the current state." It's good enough for most reads but unreliable for decisions. A good rule: if an operation has consequences (money movement, inventory deduction, scheduling a unique resource), it needs stronger consistency. If an operation is read-only with no downstream effects (showing a counter, displaying a feed), eventual consistency is usually fine.
