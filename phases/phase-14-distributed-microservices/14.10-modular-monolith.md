## 1. What Is It? (Theory)

A modular monolith is a single deployable application that is internally structured with clear module boundaries — strict separation between domains, enforced via code organization and dependency rules.

It gets the simplicity benefits of a monolith (one deployment, transactions, fast calls) while forcing the discipline benefits of microservices (clear ownership, boundaries, independent development).

The modular monolith is arguably the best starting architecture for most applications and the least discussed.

## 2. How It Actually Works

```
MODULE STRUCTURE (e.g., a Django application):

myapp/
├── orders/              ← the "Orders" module
│   ├── __init__.py      ← public interface: what other modules can import
│   ├── models.py        ← Order, OrderItem models
│   ├── services.py      ← OrderService (business logic)
│   ├── repository.py    ← OrderRepository (data access)
│   ├── api.py           ← HTTP endpoints (internal only, not exported)
│   └── events.py        ← Events this module can publish
│
├── payments/            ← the "Payments" module
│   ├── __init__.py      ← public interface
│   ├── models.py
│   ├── services.py
│   └── events.py
│
├── inventory/           ← the "Inventory" module
│   ├── __init__.py
│   └── ...
│
└── shared/              ← shared utilities (logging, authentication)
    └── ...

THE CRITICAL RULE:
  Modules ONLY import from each other via the public __init__.py interface.
  NEVER: from orders.models import Order  ← internal detail
  ALWAYS: from orders import OrderService  ← public interface

  This enforces:
  - Clear ownership: each module team owns their internal implementation
  - Easy extraction: when ready to split, the module is already isolated
  - Testability: each module can be tested independently
```

### Enforcing Module Boundaries

```python
# Python: use architectural linting tools

# .architectural-rules.yaml
modules:
  - name: orders
    public: [OrderService, OrderCreatedEvent, OrderStatus]
    private: [Order, OrderItem, OrderRepository]
  
  - name: payments
    public: [PaymentService, PaymentProcessedEvent]
    private: [Payment, PaymentRepository]
  
  # Dependency rules
  allowed_dependencies:
    payments: [orders]  # payments can depend on orders
    orders: [inventory]  # orders can depend on inventory
    # orders CANNOT depend on payments (no circular dependency)

# Tools:
# Python: import-linter (pip install import-linter)
# Java: ArchUnit
# .NET: NDepend
# Go: structcheck (enforces package boundaries)
```

## 3. When to Use It

```
MODULAR MONOLITH IS IDEAL WHEN:
  ✓ Team is growing (5-30 engineers) but not yet at microservices scale
  ✓ Domain is evolving (premature microservices lock you into wrong boundaries)
  ✓ You want to be ready to extract services when needed
  ✓ Strong consistency requirements within modules
  ✓ Operational simplicity is important (one deployment is much simpler)

MIGRATION PATH:
  1. Start as modular monolith with clear module boundaries
  2. Each module has a well-defined public interface
  3. When a specific module needs independent scaling/deployment:
     a. The module already has clear boundaries
     b. Move it to its own service
     c. Replace the function calls with HTTP/gRPC calls
     d. The interface contract is already defined
     e. Migration takes days, not months

  This is the "strangler fig" pattern applied at module granularity.
```

## 4. Real-World Context

Shopify's engineering blog describes their approach: they spent years adding strict boundaries to their Rails monolith. Components (their term for modules) cannot violate each other's boundaries. This allowed them to gradually extract specific high-load components as microservices while keeping the simpler components in the monolith.

Stack Overflow runs on a monolith serving 100M+ visitors per month. They've written extensively about how a well-optimized monolith outperforms many microservices architectures in their specific use case.

---

## 5. Senior Engineer Perspective

**The modular monolith as a stepping stone:**

The best microservices migrations I've seen follow this pattern:
1. Modular monolith first (enforce module boundaries from day one)
2. Identify the module with the strongest case for extraction (scaling, deployment, team ownership)
3. Extract one service
4. Operate it well. Learn the distributed systems overhead.
5. Extract the next module only when the same case arises

Teams that start with microservices from day one often get the boundaries wrong and spend years fighting with tight coupling between services — a distributed monolith with all the costs and none of the benefits.
