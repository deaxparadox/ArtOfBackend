## 1. What Is It? (Theory)

When you need operations to span multiple services or databases but 2PC is too fragile, you need alternative patterns that achieve eventual consistency without global locks. This topic compares the practical options for distributed coordination.

**The spectrum of approaches:**

```
MOST CONSISTENT ←────────────────────────────────→ MOST AVAILABLE

2PC          TCC          SAGA         OUTBOX + EVENTUAL
Global lock  Reserve-     Compensate   No global lock
             confirm-     on failure   Eventually consistent
             cancel
```

## 2. Saga (Deep Dive for Distributed Transactions)

A Saga is a sequence of local transactions where each step publishes an event that triggers the next step, and each step has a compensating transaction for rollback. We covered the mechanics in [12.21 — Saga pattern](../phase-12-messaging-events/12.21-saga.md). Here's how it addresses the distributed transaction problem:

```
THE SAGA GUARANTEE:
  "Either all local transactions complete successfully,
  OR compensating transactions undo the completed steps."
  
  This is NOT the same as ACID atomicity.
  
  ACID atomicity: the operation either happens or it doesn't.
                  Intermediate states are invisible to other transactions.
  
  Saga atomicity: all steps eventually complete or are compensated.
                  Intermediate states ARE visible (the order is PLACED
                  while payment is still PROCESSING).
  
  SAGA ISOLATION LEVELS:
  None, by default. You may observe:
    - Order placed, payment not yet processed
    - Order placed, payment failed, inventory not yet released
    
  For many business processes: this is acceptable.
  Users can see "order processing" as an intermediate state.
  
WHEN SAGA IS APPROPRIATE:
  ✓ Long-running business processes (order placement, booking, onboarding)
  ✓ Each step is a meaningful business action (not just a DB write)
  ✓ Intermediate states are visible to users (expected in the domain)
  ✓ Compensation actions exist for each step
  
WHEN SAGA IS NOT APPROPRIATE:
  ✗ Strict isolation required (concurrent conflicting sagas would collide)
  ✗ Very short operations (the overhead isn't worth it)
  ✗ No meaningful compensation exists (can't un-send an email, can only send a correction)
```

## 3. TCC: Try-Confirm-Cancel

TCC (Try-Confirm-Cancel) is a distributed transaction pattern that maintains availability while providing stronger consistency than sagas:

```
TCC PHASES:

TRY: Reserve the resources, check all preconditions.
  Does NOT make permanent state changes.
  Does reserve: inventory slots, payment holds, appointment times.
  
  If any TRY fails: cancel everything. No permanent changes.
  
CONFIRM: Make reservations permanent.
  Called ONLY if ALL tries succeeded.
  Business operation: payment captured, inventory deducted, appointment confirmed.
  
CANCEL: Release reservations.
  Called if ANY try failed, or after a timeout.
  Releases: inventory hold, payment authorization, appointment block.

EXAMPLE: Hotel + Flight + Car booking
  
  TRY phase:
    Hotel: reserve room (mark as "pending" for 10 minutes, not "booked")
    Flight: hold seat (mark as "held", not "sold")
    Car: hold rental (mark as "pending", not "reserved")
    
    If any of these fails → CANCEL all others.
  
  CONFIRM phase (all tries succeeded):
    Hotel: mark room as "booked"
    Flight: mark seat as "sold"
    Car: mark rental as "reserved"
    
  CANCEL phase (any try failed):
    Hotel: release "pending" room back to available
    Flight: release "held" seat back to available
    Car: release "pending" rental back to available

COMPARISON WITH SAGA:
  TCC: the intermediate "held" state is real but not final.
       Users don't see "partial success" in TCC (they see "pending").
  Saga: intermediate steps DO take effect (payment charged, then compensation).
  
  TCC: less likely to require compensation (resources held, not consumed)
  Saga: more likely to require compensation (steps fully execute)
  
REAL IMPLEMENTATIONS:
  Seata TCC (Alibaba, open-source)
  DTM (Go-based distributed transaction framework)
  Custom implementations at Airbnb, Booking.com
```

## 4. Best-Effort: Accept Inconsistency with Alerting

For some operations, the pragmatic answer is: try your best, handle failures with reconciliation.

```
BEST-EFFORT APPROACH:

STEP 1: Execute local transaction
STEP 2: Send events/notifications to other services
  If notifications fail: log, alert, retry asynchronously
STEP 3: Reconciliation job (nightly/hourly)
  Find inconsistencies between services
  Apply business rules to resolve them

EXAMPLE: Invoice generation after payment
  
  Payment service: payment recorded ✓
  Invoice service: create invoice
    → If this fails: log failure, alert, retry
    → Reconciliation: "payments without invoices older than 1 hour" → create retroactively
  
  TRADE-OFF:
  ✓ Simple to implement
  ✓ Each service is truly independent
  ✗ Brief inconsistency windows
  ✗ Requires reconciliation infrastructure
  
WHEN BEST-EFFORT IS APPROPRIATE:
  ✓ Financial operations like invoicing/reporting (not the actual payment)
  ✓ Analytics and audit logs (slightly delayed is acceptable)
  ✓ Email notifications (retry on failure is sufficient)
  ✓ Any operation where "eventually correct" is good enough
```

## 5. Choosing the Right Pattern

```
DECISION GUIDE:

Need ACID guarantees across services?
  → Reconsider your service boundaries. Should this be one service?
  → If you truly need it: 2PC with understood blocking risk.

Long-running business process with meaningful compensation?
  → SAGA (choreography for loose coupling, orchestration for visibility)

Need to coordinate multiple operations before any take effect?
  → TCC (resources held, not consumed, until all ready)

Operations with loose coupling and eventual consistency is OK?
  → Outbox pattern + events (most microservices use this)

Analytics, notifications, audit logs?
  → Best-effort with reconciliation

THE PRACTICAL REALITY:
  Most microservices use a combination:
  - SAGA for multi-step business processes (order placement)
  - Outbox pattern for reliable event publishing
  - TCC for booking/reservation systems
  - Best-effort for analytics and notifications
  
  2PC is rarely the right answer for microservices.
```

## 6. Senior Engineer Perspective

**The distributed transaction is usually a design smell:**

When you find yourself reaching for 2PC or complex saga orchestration across many services, ask: "Did I split this domain too aggressively?" Strong transactional requirements across services often indicate that those services belong together in a single service with a single database. The need for distributed transactions can be a signal that your service boundaries are in the wrong place.

The best distributed transaction is often no distributed transaction — achieved by keeping tightly coupled operations in the same service.
