## 1. What Is It? (Theory)

ClickHouse (Yandex, 2016; open-source) is a column-oriented OLAP database designed for real-time analytical queries on massive datasets. It is the fastest SQL OLAP database in common benchmarks — frequently processing billions of rows per second on a single server.

ClickHouse powers analytics at companies like Cloudflare (10 trillion events/day), Uber, Airbnb, and thousands of others who need fast analytics on large event streams.

**The ClickHouse positioning:**

```
BigQuery:   Serverless, zero-ops, excellent for ad-hoc analytics, GCP-specific
Snowflake:  Managed, separates compute/storage, broad cloud support, expensive
Redshift:   AWS-native, mature, moderate performance
ClickHouse: Self-hosted or managed, fastest for high-throughput event data,
            excellent for product/product analytics, requires more ops knowledge
```

## 2. How It Actually Works (Deep Mechanics)

### The MergeTree Engine: ClickHouse's Core

```sql
-- The MergeTree family is ClickHouse's primary table engine
CREATE TABLE events (
    -- Columns stored separately on disk, sorted by ORDER BY
    timestamp     DateTime,
    user_id       UInt64,
    event_type    LowCardinality(String),  -- LowCardinality: dictionary encoding
    page          String,
    country       LowCardinality(String),
    session_id    String,
    amount_cents  Nullable(Int32),
)
ENGINE = MergeTree()
PARTITION BY toYYYYMM(timestamp)  -- data partitioned by month
ORDER BY (timestamp, user_id)     -- data sorted by this within each partition
-- Sorting enables: range scans on timestamp, fast user lookups, compression

-- DATA STORAGE:
-- Each column is a separate file on disk.
-- timestamp.bin: [compressed timestamps]
-- user_id.bin:   [compressed user IDs]
-- event_type.bin:[dictionary-encoded event types]
-- etc.

-- PARTITION PRUNING:
-- Query for data in June 2024:
-- WHERE toYYYYMM(timestamp) = 202406
-- → Only reads the June partition, skips all others
-- → Massive I/O reduction for time-range queries
```

### Why ClickHouse Is Fast

```
1. VECTORIZED EXECUTION:
   SQL query → compiled to CPU vector instructions (SIMD)
   Process 512 bits (8 integers) at once instead of 1
   
   Traditional DB: SUM(amount) → iterate, add one by one
   ClickHouse: SUM(amount) → load 8 values, add all 8, repeat
   → 8x CPU efficiency for arithmetic operations

2. LATE MATERIALIZATION:
   Start with the most selective filter first.
   Only fetch other columns for rows that pass all filters.
   
   WHERE country = 'US' AND event_type = 'purchase' AND amount > 1000
   → First apply country filter (e.g., reduces to 20% of rows)
   → Then event_type filter (reduces to 5% of rows)
   → Then amount filter (reduces to 0.1% of rows)
   → Only read all columns for the 0.1% that passed all filters
   
3. COMPRESSION:
   Column data compresses much better than row data.
   LZ4 compression for speed, ZSTD for size.
   
   event_type column: purchase/view/cart/purchase/purchase/view...
   After dictionary encoding: 2/1/3/2/2/1... (small integers)
   After run-length encoding (sorted data): 2×3, 1×2, 3×1...
   Compression ratio: 5-20x vs raw data
   
4. PARALLEL PROCESSING:
   Each query uses all available CPU cores.
   Each core processes a different range of data.
   100 billion rows ÷ 64 cores = 1.5 billion rows per core
```

### Writing Analytical Queries

```sql
-- ClickHouse SQL is standard SQL with extensions

-- Q1: Daily active users for the last 30 days
SELECT
    toDate(timestamp)        AS date,
    uniq(user_id)            AS daily_active_users,  -- approximate distinct count
    count()                  AS events,
    countIf(event_type = 'purchase') AS purchases
FROM events
WHERE timestamp >= today() - 30
GROUP BY date
ORDER BY date

-- uniq() is ClickHouse's approximate COUNT(DISTINCT) - much faster than exact
-- countIf() = conditional count without subquery

-- Q2: Top 10 pages by revenue in the last week
SELECT
    page,
    sum(amount_cents) / 100.0  AS revenue,
    count()                    AS purchases,
    uniq(user_id)              AS buyers
FROM events
WHERE event_type = 'purchase'
  AND timestamp >= now() - INTERVAL 7 DAY
GROUP BY page
ORDER BY revenue DESC
LIMIT 10

-- Q3: Retention analysis - % of users who purchased again after first purchase
WITH first_purchases AS (
    SELECT user_id, min(timestamp) AS first_purchase_date
    FROM events
    WHERE event_type = 'purchase'
    GROUP BY user_id
)
SELECT
    toStartOfWeek(fp.first_purchase_date) AS cohort_week,
    count(DISTINCT fp.user_id) AS cohort_size,
    countDistinctIf(e.user_id,
        e.event_type = 'purchase' AND
        e.timestamp > fp.first_purchase_date + INTERVAL 7 DAY AND
        e.timestamp <= fp.first_purchase_date + INTERVAL 14 DAY
    ) AS week2_retained,
FROM first_purchases fp
LEFT JOIN events e ON fp.user_id = e.user_id
GROUP BY cohort_week
ORDER BY cohort_week

-- This query on 1 billion events completes in ~10 seconds in ClickHouse.
-- The same query on PostgreSQL with 1 billion rows: hours.
```

## 3. Real-World Context

### Cloudflare's ClickHouse Deployment

Cloudflare uses ClickHouse to process 10 trillion events per day — every HTTP request logged and made available for analytics in real-time. Their engineering blog details the architecture: events streamed via Kafka → ClickHouse, analysts query within seconds of data arrival.

The scale: ClickHouse handles this on a few hundred nodes. Equivalent processing in PostgreSQL would require thousands of nodes and still not achieve real-time latency.

### ClickHouse for Product Analytics

Plausible Analytics (privacy-focused Google Analytics alternative) uses ClickHouse for all its analytics. A query like "how many unique visitors did this website get from Germany last month, broken down by device type?" runs in ~50ms even for sites with millions of daily visits, because ClickHouse can efficiently skip irrelevant data using partition pruning.

## 5. Senior Engineer Perspective

**The ReplacingMergeTree and deduplication:**

ClickHouse's MergeTree doesn't support updates — it's append-only. For changing data (updating a user's attributes), use `ReplacingMergeTree` which keeps only the latest row per primary key. But deduplication happens lazily during background merges, so you may see duplicates in queries. Use `FINAL` keyword to force deduplication at query time (slower but accurate):

```sql
SELECT * FROM users FINAL WHERE id = 12345;
```

**ClickHouse is not a general-purpose database:**

Don't try to use ClickHouse as your application's OLTP database. No proper transactions, no point queries, updates are expensive. It's an analytics database. Use PostgreSQL for your application, stream events to ClickHouse for analytics. They serve completely different purposes.
