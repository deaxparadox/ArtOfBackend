## 1. What Is It? (Theory)

Understanding Elasticsearch's internals explains why it behaves the way it does: why search is near-real-time but not instantaneous, why shard count is a permanent decision, why merges cause latency spikes, and how to tune for your specific access patterns.

## 2. How It Actually Works (Deep Mechanics)

### Lucene: The Engine Under Elasticsearch

Each Elasticsearch shard is a Lucene index. Lucene is the actual search library — Elasticsearch wraps it in a distributed, HTTP-accessible service.

```
LUCENE INDEX STRUCTURE:

  Shard 0 (a single Lucene index):
  ├── Segment 1 (immutable)
  │   ├── Inverted index for these documents
  │   ├── Stored fields (original document values)
  │   ├── Doc values (column-store for sorting/aggregations)
  │   └── Term vectors (for more-like-this queries)
  ├── Segment 2 (immutable)
  ├── Segment 3 (immutable, currently being written)
  └── .liv file (marks deleted documents)

SEGMENTS:
  Each segment is a complete Lucene index for a subset of documents.
  Segments are IMMUTABLE — once written, they never change.
  Updates = mark old document deleted + create new document in new segment.
  Deletes = mark document in .liv file (not physically removed).
  
  Search reads ALL segments and merges results.
  More segments = slower search (must scan each).
  Background merge process combines small segments into larger ones.
```

### Near-Real-Time Search: The Refresh Cycle

```
WHY SEARCH IS NEAR-REAL-TIME (not instantaneous):

Document indexed → lives in write buffer (in-memory)
                ↓ (every 1 second by default: "refresh_interval")
           New segment created
                ↓
         Search can now find the document

REFRESH (default: every 1 second):
  Flushes in-memory buffer to a new segment.
  New segment is immediately searchable.
  Cost: ~10ms per refresh (negligible usually).
  
FLUSH (periodic, when buffer is full, or explicit):
  Writes segments to disk (fsync).
  Ensures durability (survives restart).
  More expensive than refresh.

IMPACT:
  Document indexed at t=0.000
  Refresh fires at t=1.000
  Document searchable at t=1.001
  
  1-second delay is "near-real-time" — fine for most use cases.
  Not real-time: logs, audit systems where milliseconds matter.
  
  To make specific index immediately searchable:
    POST /products/_refresh
  
  To disable refresh (bulk indexing performance):
    PUT /products/_settings
    {"index": {"refresh_interval": "-1"}}
  Then re-enable after bulk load.
```

### Segment Merges: Why Elasticsearch Slows Under Writes

```
MERGE PROBLEM:

  Start: [Seg1: 1000 docs][Seg2: 1000 docs][Seg3: 500 docs]
  After more indexing:
         [Seg1: 1000][Seg2: 1000][Seg3: 500][Seg4: 300][Seg5: 200]
  
  Elasticsearch background merges:
  [Seg3+Seg4+Seg5 merged] → [Seg6: 1000 docs] (new segment)
  Old segments deleted.
  
  DURING MERGE:
  Old and new segments both exist on disk temporarily.
  Disk usage can spike to 2x normal during large merges.
  CPU usage for merge competes with search requests.
  
  FORCE MERGE (for read-only indices like archived logs):
  POST /logs-2024-01/_forcemerge?max_num_segments=1
  Compresses all segments into one. After: fastest possible search.
  Do NOT force merge on active write indices (causes huge I/O spike).
```

### Shard Sizing: Getting It Right Before You Deploy

```
SHARD COUNT: the most important Elasticsearch decision (and hard to change)

TOO FEW SHARDS:
  Cannot parallelize across multiple nodes.
  One shard can only live on one node.
  If you have 1 shard: all search goes through 1 node, no matter how big the cluster.

TOO MANY SHARDS:
  Each shard has overhead (~1.5MB heap, file handles, etc.)
  10,000 shards on 3 nodes = 3,333 shards per node = OOM risk
  "Oversharding" is a very common production mistake

ELASTICSEARCH SIZING GUIDANCE (2024):
  Shard size: 10-40GB per shard (sweet spot for performance)
  Heap: 50% or less of each node's RAM allocated to ES heap (rest for OS page cache)
  
  Calculate:
    Expected data volume: 100GB
    Target shard size: 25GB
    Number of shards: 100GB / 25GB = 4 shards
  
  Start conservative: 1-3 shards for small indices.
  Scale out by increasing shard count via reindexing when needed.
  
CHANGING SHARD COUNT:
  You CANNOT change the number of shards in an existing index.
  To change: reindex to a new index with different shard count.
  Use aliases to make this transparent to clients:
  
    POST /_aliases
    {
      "actions": [
        {"remove": {"index": "products-v1", "alias": "products"}},
        {"add": {"index": "products-v2", "alias": "products"}}
      ]
    }
  
  Applications always read/write to "products" alias, not the versioned index.
  Reindexing doesn't require downtime.
```

## 3. Real-World Context

### The 30-Shard Legacy Problem

Many teams following outdated Elasticsearch documentation created indices with 5 shards (the old default) for small datasets. Over time they have 200 small indices × 5 shards = 1,000 shards on a cluster meant for a few hundred. The cluster becomes slow and memory-hungry from shard overhead. This is called "oversharding."

Elastic's guidance since 7.x: use 1 shard by default, scale up when individual shards exceed 40GB. Use ILM (Index Lifecycle Management) to roll over time-series indices.

### ILM: The Correct Way to Manage Time-Series Data

For log data (millions of documents per day), Index Lifecycle Management automatically rolls indices:

```json
// ILM Policy: hot → warm → cold → delete
{
    "policy": {
        "phases": {
            "hot": {
                "min_age": "0ms",
                "actions": {
                    "rollover": {"max_size": "40gb", "max_age": "1d"},
                    "set_priority": {"priority": 100}
                }
            },
            "warm": {
                "min_age": "7d",
                "actions": {
                    "shrink": {"number_of_shards": 1},
                    "forcemerge": {"max_num_segments": 1},
                    "set_priority": {"priority": 50}
                }
            },
            "cold": {
                "min_age": "30d",
                "actions": {"freeze": {}}  // reduce memory, searchable but slower
            },
            "delete": {
                "min_age": "90d",
                "actions": {"delete": {}}
            }
        }
    }
}
```

## 4. Senior Engineer Perspective

**The single most important operational habit:**

Monitor shard count and shard size continuously. `GET /_cat/shards?v` shows all shards, their sizes, and which node hosts them. A cluster where shards are consistently growing beyond 50GB, or where shard count is growing unbounded, is heading toward problems. Catch these early while reindexing is still affordable (before the data grows 10x more).

**Replicas are for availability, not just performance:**

A replica shard is a complete copy of a primary shard. It serves reads (distributing read load), and when a primary fails, a replica is promoted to primary. For production: always `number_of_replicas: 1` minimum. During bulk indexing, temporarily set to 0 (faster writes), then restore to 1 after.
