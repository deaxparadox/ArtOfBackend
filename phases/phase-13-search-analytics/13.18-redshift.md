## 1. What Is It? (Theory)

Amazon Redshift (Amazon, 2012) is AWS's managed cloud data warehouse. It's PostgreSQL-compatible (you can use PostgreSQL clients), column-oriented, and deeply integrated with the AWS ecosystem (S3, Glue, Kinesis, QuickSight).

Redshift was the first cloud data warehouse to make enterprise analytics accessible to startups, launching at prices far below traditional on-premise options like Oracle or Teradata.

## 2. How Redshift Works

### Cluster Architecture

```
REDSHIFT CLUSTER:
  
  Leader Node:
    Receives SQL queries
    Creates query execution plan
    Distributes work to compute nodes
    Merges results
    
  Compute Nodes (1-128):
    Store and process data
    Each node has: CPU, memory, local SSD/HDD
    Data distributed across nodes via distribution key
    
  Node types:
    ra3.xlplus: 4 vCPU, 32GB RAM, 64TB S3 managed storage
    ra3.4xlarge: 12 vCPU, 96GB RAM, 128TB S3 managed storage  
    ra3.16xlarge: 48 vCPU, 384GB RAM, 128TB S3 managed storage

REDSHIFT SERVERLESS (newer option):
  No cluster to provision.
  Pays per compute-unit-second (like BigQuery's per-query model).
  Best for: variable workloads, development, cost unpredictability is OK.
```

### Distribution and Sort Keys

```sql
-- Distribution style affects how data is distributed across nodes
-- Affects: query performance (for joins), data locality

CREATE TABLE orders (
    order_id     BIGINT NOT NULL,
    user_id      BIGINT NOT NULL,
    created_at   TIMESTAMP NOT NULL,
    total_cents  INTEGER NOT NULL,
)
DISTSTYLE KEY              -- distribute by the distkey column
DISTKEY (user_id)          -- orders for same user go to same node
                           -- → queries filtering by user_id avoid data movement

SORTKEY (created_at);      -- physical sort order on disk
                           -- → range queries on created_at skip blocks efficiently

-- Alternative distribution styles:
-- DISTSTYLE EVEN: round-robin (no join optimization, default)
-- DISTSTYLE ALL: copy full table to each node (small dimension tables)
-- DISTSTYLE AUTO: Redshift chooses automatically (good default)
```

### Loading Data (COPY Command)

```sql
-- The fast way to load data into Redshift: COPY from S3
COPY orders
FROM 's3://my-bucket/orders/'
IAM_ROLE 'arn:aws:iam::123456789:role/RedshiftCopyRole'
FORMAT AS PARQUET;  -- or CSV, JSON, Avro

-- COPY is parallelized across all compute nodes.
-- Each node loads from S3 simultaneously.
-- Can load billions of rows in minutes.
-- Never use single-row INSERT for bulk loading -- 1000x slower than COPY.
```

## 3. Redshift vs. Snowflake vs. BigQuery (AWS Context)

```
FOR AWS-NATIVE TEAMS:

REDSHIFT:
  ✓ Deep AWS integration (native S3, Glue, IAM, VPC)
  ✓ Familiar if team knows PostgreSQL (compatible wire protocol)
  ✓ Strong performance for large-scale analytics
  ✓ Predictable cost with reserved instances
  ✓ Redshift Spectrum: query S3 directly without loading
  ✗ Less innovative than Snowflake/BigQuery in recent years
  ✗ Cluster management overhead (right-sizing, vacuuming)

SNOWFLAKE ON AWS:
  ✓ Time travel, data sharing, zero-copy clone
  ✓ Multi-cluster for concurrency
  ✓ Better separation of storage and compute
  ✗ Premium pricing compared to Redshift

BIGQUERY ON AWS:
  ✗ GCP service — doesn't run natively on AWS
  (BigQuery Omni exists but is a GCP product)

DECISION FOR AWS TEAMS:
  Strong AWS integration needed: Redshift
  Enterprise features + multi-cloud: Snowflake on AWS
  Existing Redshift investment: stay on Redshift (migration is costly)
  Starting fresh, care about cost: evaluate Redshift Serverless
```

## 4. Senior Engineer Perspective

**Vacuum and Analyze: the operational maintenance:**

Redshift requires manual maintenance that BigQuery and Snowflake handle automatically:
- `VACUUM`: reclaims disk space from deleted rows, re-sorts data
- `ANALYZE`: updates statistics used by the query optimizer

Without regular vacuum and analyze, query performance degrades over time as statistics become stale and disk space fills with "soft-deleted" rows. Automate this:

```sql
-- Run nightly (via Redshift scheduled query):
VACUUM DELETE ONLY tablename;  -- faster: only reclaim deleted space
ANALYZE tablename;

-- Or enable automatic vacuum (enabled by default in newer Redshift):
-- ALTER TABLE orders SET (autovacuum_enabled = true);
```

**Redshift Spectrum: the data lake bridge:**

Redshift Spectrum lets you query data in S3 directly without loading it into Redshift. This is particularly powerful for keeping "cold" historical data in S3 (cheap storage) while querying it with Redshift's SQL engine:

```sql
-- External table pointing to S3 data
CREATE EXTERNAL TABLE spectrum.historical_orders (
    order_id bigint, user_id bigint, total_cents int, created_at timestamp
)
PARTITIONED BY (year int, month int)
STORED AS PARQUET
LOCATION 's3://my-data-lake/historical-orders/';

-- Query joins Redshift table with S3 data seamlessly
SELECT h.*, o.status
FROM spectrum.historical_orders h  -- reads from S3 via Spectrum
JOIN orders o ON h.order_id = o.id  -- from Redshift cluster storage
WHERE h.year = 2022;
```
