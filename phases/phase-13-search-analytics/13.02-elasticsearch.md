## 1. What Is It? (Theory)

Elasticsearch (2010, Shay Banon) is a distributed, RESTful search and analytics engine built on Apache Lucene. It exposes Lucene's powerful full-text search capabilities through a JSON over HTTP API, adds horizontal scaling across multiple nodes, and provides near-real-time indexing.

Elasticsearch is used for:
- Full-text search (e-commerce product search, documentation search)
- Log analysis (the "E" in ELK stack: Elasticsearch + Logstash + Kibana)
- Metrics analytics and dashboards
- Geospatial search
- Autocomplete and suggestion

It is the world's most popular enterprise search engine, deployed by Wikipedia, GitHub, Netflix, Uber, and thousands of others.

## 2. How It Actually Works (Deep Mechanics)

### The Distributed Architecture

```
ELASTICSEARCH CLUSTER (3 nodes):

  Node 1 (Master-eligible):     Node 2:              Node 3:
  ├── Index "products"          ├── Index "products"  ├── Index "products"
  │   ├── Shard 0 (Primary)     │   ├── Shard 1 (P)   │   ├── Shard 2 (P)
  │   └── Shard 2 (Replica)     │   └── Shard 0 (R)   │   └── Shard 1 (R)
  └── Index "logs"              └── Index "logs"       └── Index "logs"
      └── Shard 0 (Primary)         └── Shard 1 (P)       └── Shard 2 (P)

SHARD: a single Lucene instance — the actual search unit
PRIMARY SHARD: handles all writes for that shard
REPLICA SHARD: copy of a primary — serves reads, provides fault tolerance

WRITE PATH:
  Client → any node → node routes to correct primary shard
  → primary writes + replicates to replicas
  → returns success when all replicas acknowledged

READ PATH:
  Client → any node → node routes to primary OR replica for load balancing
  → shard executes search → returns results
  → coordinating node merges results from all shards
  → returns ranked results to client
```

### Indexing Documents

```python
# Index a single document
await es.index(
    index="products",
    id="prod-123",        # provide ID or let ES generate one
    document={
        "name": "Nike Air Max Running Shoes",
        "description": "Lightweight running shoes with cushioned sole",
        "price_cents": 12999,
        "category": "footwear",
        "brand": "Nike",
        "tags": ["running", "sports", "sneakers"],
        "in_stock": True,
        "created_at": "2024-06-15",
    }
)

# Bulk indexing (much more efficient for large datasets)
from elasticsearch.helpers import async_bulk

async def index_products(products: list[dict]):
    actions = [
        {
            "_index": "products",
            "_id": product["id"],
            "_source": product,
        }
        for product in products
    ]
    
    success, failed = await async_bulk(
        es,
        actions,
        chunk_size=500,      # 500 documents per bulk request
        max_retries=3,
    )
    return success, failed
```

## 3. Real-World Context

### How GitHub Uses Elasticsearch

GitHub uses Elasticsearch to power code search — indexing billions of files across millions of repositories. Their engineering blog describes challenges unique to code search: code is not natural language (symbols, operators, function names), queries are often exact substring matches, users want to search by language, repository, or file path.

Their solution involves custom analyzers that treat code symbols differently from prose text, and a combination of Elasticsearch full-text search with pattern-based filtering.

### The Relevance Tuning Reality

Elasticsearch's default BM25 scoring works well for general use cases. For specialized domains (legal, medical, e-commerce), relevance tuning requires:
- **Field boosting:** product name match matters more than description match
- **Function score:** boost by popularity, recency, or business rules
- **Learning to Rank:** use ML models trained on click data
- **Phrase matching boost:** "running shoes" as a phrase scores higher than the words separately

Real production search systems at scale typically have dedicated search relevance engineers who run A/B tests to measure which ranking changes improve click-through rates and conversions.

## 4. Senior Engineer Perspective

**The mapping immutability problem:**

Once you've defined a field mapping in Elasticsearch, you can't change it. If you define `price` as a `keyword` and later need it as `integer` (to do range queries), you must reindex — create a new index with the correct mapping, re-index all documents, switch the alias to point to the new index.

This is expensive at scale. Invest time upfront in designing your mapping carefully. Rules:
- Use `keyword` for values you'll filter, aggregate, or sort on (category, brand, status)
- Use `text` (with analyzer) for values you'll search with full-text (name, description)
- Use the `.keyword` subfield on text fields when you need both
- Never use `keyword` for numeric data you'll do range queries on

**The sync between your database and Elasticsearch:**

Elasticsearch is a derived store — its data comes from your primary database. Keeping them in sync is a persistent engineering challenge:
- **Real-time sync via Debezium**: tap PostgreSQL WAL → stream to Elasticsearch via Kafka Connect
- **Application-level dual-write**: update DB + update ES in the same request handler (risk: one can fail)
- **Periodic re-index**: cron job that re-syncs changes (has latency, expensive for large data)

The CDC approach (Debezium) is the gold standard: millisecond latency, reliable, no application code required.
