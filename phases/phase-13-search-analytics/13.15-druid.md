## 1. What Is It? (Theory)

Apache Druid (Metamarkets, 2011; Apache 2018) is a high-performance real-time analytics database designed for powering **user-facing analytics applications** — dashboards and exploratory analytics that users interact with directly, not just internal analyst tools.

The key distinction from ClickHouse or BigQuery: Druid is built for sub-second query response times even on large datasets, making it suitable for dashboards that users expect to respond instantly when they change filters.

---

## 2. Druid's Design Choices

```
DRUID vs. CLICKHOUSE:

CLICKHOUSE:
  For: internal analytics, data engineering, complex queries
  Response time: seconds to minutes for complex queries
  Concurrent users: dozens to hundreds
  
DRUID:
  For: user-facing dashboards, high-concurrency analytics
  Response time: sub-second to seconds always (or query fails fast)
  Concurrent users: thousands
  Trade-off: less flexible queries, more opinionated architecture

DRUID ARCHITECTURAL ADVANTAGES FOR REAL-TIME:
  1. Segment-based storage: data pre-aggregated and sorted
  2. Bitmap indexes on all dimensions: fast multi-filter queries
  3. Pre-computed aggregations: counts, sums stored as numbers, not raw rows
  4. In-memory caching: hot segments served from RAM
  5. Kafka-native ingestion: query data within seconds of ingestion
```

### Druid Schema: Dimensions and Metrics

```
DRUID DATA MODEL:
  Timestamp: always required (Druid is time-series first)
  Dimensions: categorical data for filtering/grouping (user_id, country, event_type)
  Metrics: numerical data for aggregation (count, sum, max, min)

EXAMPLE EVENT:
  {
    "timestamp": "2024-06-15T14:30:00Z",   ← required
    "user_id": "user-123",                  ← dimension
    "country": "US",                        ← dimension
    "event_type": "purchase",               ← dimension
    "product_id": "prod-456",               ← dimension
    "amount_cents": 5000,                   ← metric
    "session_id": "sess-789",               ← dimension
  }

DRUID PRE-AGGREGATES (roll-up):
  If multiple events have the same dimension values in the same minute,
  Druid can roll them up:
  
  Before rollup (10 events):
    timestamp=2024-06-15T14:30, country=US, event_type=view, amount=0
    timestamp=2024-06-15T14:30, country=US, event_type=view, amount=0
    timestamp=2024-06-15T14:30, country=US, event_type=view, amount=0
    ...
  
  After rollup (1 row):
    timestamp=2024-06-15T14:30, country=US, event_type=view, count=10, sum_amount=0
  
  QUERY: "How many views from US in the last hour?"
  WITHOUT rollup: scan 1M raw events
  WITH rollup: scan 60 pre-aggregated minute-rows
  10,000x less data to scan.
```

## 3. When Druid vs. ClickHouse

```
CHOOSE DRUID WHEN:
  ✓ User-facing dashboards that must respond in < 1s
  ✓ High-concurrency analytics (1000+ concurrent dashboard users)
  ✓ Real-time data ingestion from Kafka with sub-second query availability
  ✓ Time-series analytics as the primary use case
  ✓ Pre-aggregation makes sense for your data (reduces volume significantly)

CHOOSE CLICKHOUSE WHEN:
  ✓ Ad-hoc complex queries (Druid is limited in SQL expressiveness)
  ✓ Internal analysts (not user-facing, higher query latency OK)
  ✓ Full SQL support (joins, subqueries, window functions)
  ✓ Simpler operational model (Druid has many components: Coordinator, Broker, Historical, MiddleManager...)
  ✓ Cost (Druid's multi-component architecture requires more nodes)

CHOOSE BIGQUERY/SNOWFLAKE WHEN:
  ✓ Serverless (zero ops)
  ✓ Ad-hoc complex analytics
  ✓ Data warehouse use case (not just event analytics)
  ✓ Integration with data transformation tools (dbt)
```

## 4. Real-World Context

### Druid at Netflix

Netflix uses Apache Druid to power their A/B testing analytics platform. When a product team runs an experiment (e.g., "does this new UI increase engagement?"), they need to query experiment results interactively — slicing by country, device type, user segment — and get answers in under a second. Druid handles hundreds of concurrent analysts querying billions of experiment events.

### Druid at Twitter (historical)

Twitter used Druid to power their real-time analytics: "how many impressions did this tweet get in the last 5 minutes, broken down by country?" This required ingesting millions of events per second from Kafka and making them queryable almost immediately. Druid's Kafka-native ingestion and pre-aggregation made this feasible.
