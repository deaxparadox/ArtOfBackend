## 1. What Is It? (Theory)

When users type "running shoes for marathons" into a search box, they don't expect the database to find rows where the column value is exactly "running shoes for marathons." They expect to find products about running, shoes, marathons — even if no single product description contains those exact words in that order.

Full-text search is the set of techniques that makes this possible. It solves a fundamentally different problem than SQL WHERE clauses: instead of equality matching on exact values, it finds documents based on **relevance to a query**.

**The real-world analogy:**

A standard database index is like a phone book — you look up an exact name and find the entry. Full-text search is like a book's index at the back — you look up a concept ("photosynthesis") and find every page where the topic appears, often ranked by how prominently it's discussed.

The phone book requires an exact match. The book index works with concepts, synonyms, and context.

## 2. How It Actually Works (Deep Mechanics)

Full-text search works in two phases: **indexing** (processing documents when they're added) and **searching** (processing the query and finding matches).

### Phase 1: Indexing — Building the Inverted Index

```
DOCUMENT 1: "The quick brown fox jumps over the lazy dog"
DOCUMENT 2: "Quick brown foxes run over hills"
DOCUMENT 3: "The dog ran quickly through the hills"

ANALYSIS PIPELINE (applied to each document during indexing):

Step 1: TOKENIZATION — split text into tokens
  "The quick brown fox jumps over the lazy dog"
  → ["The", "quick", "brown", "fox", "jumps", "over", "the", "lazy", "dog"]

Step 2: LOWERCASE — normalize capitalization
  → ["the", "quick", "brown", "fox", "jumps", "over", "the", "lazy", "dog"]

Step 3: STOP WORD REMOVAL — remove common, low-value words
  Stop words: ["the", "a", "an", "is", "over", "in", "at", "to", "for"]
  → ["quick", "brown", "fox", "jumps", "lazy", "dog"]

Step 4: STEMMING / LEMMATIZATION — reduce to root form
  Stemming:      "jumps" → "jump", "running" → "run", "foxes" → "fox"
  Lemmatization: "ran" → "run", "better" → "good" (uses dictionary)
  → ["quick", "brown", "fox", "jump", "lazi", "dog"]

INVERTED INDEX:
  Token      → Document IDs (and positions)
  "quick"    → [doc1: pos=2, doc2: pos=1]
  "brown"    → [doc1: pos=3, doc2: pos=2]
  "fox"      → [doc1: pos=4, doc2: pos=3]  ← "foxes" also → "fox"
  "jump"     → [doc1: pos=5]
  "run"      → [doc2: pos=4, doc3: pos=2]  ← "run" and "ran" both → "run"
  "lazi"     → [doc1: pos=6]               ← "lazy" → "lazi"
  "dog"      → [doc1: pos=7, doc3: pos=4]
  "hill"     → [doc2: pos=6, doc3: pos=5]
  "quick"    → [doc1, doc2]
```

**The inverted index** is like the back-of-book index. It maps terms to the documents containing them. Looking up a term gives instant access to all matching documents — O(1) lookup regardless of total document count.

### Phase 2: Searching — Scoring and Ranking

When a user queries "quick fox," the search engine:

```
1. ANALYZE THE QUERY (same pipeline as indexing):
   "quick fox" → tokenize → lowercase → stem → ["quick", "fox"]

2. LOOK UP EACH TERM IN THE INVERTED INDEX:
   "quick" → [doc1, doc2]
   "fox"   → [doc1, doc2]

3. INTERSECT (for AND) or UNION (for OR):
   "quick AND fox" → [doc1, doc2] (both contain both terms)

4. SCORE / RANK each document:
   Which document is MORE RELEVANT to "quick fox"?
   
   BM25 SCORING (the modern standard, replaces TF-IDF):
   
   TF (Term Frequency): how often the term appears in the document
     doc1: "quick" appears 1/6 non-stop words
     doc2: "quick" appears 1/5 non-stop words
   
   IDF (Inverse Document Frequency): how rare the term is across all documents
     "quick" appears in 2/3 documents → low IDF (common = less important)
     "jumps" appears in 1/3 documents → high IDF (rare = more important)
   
   BM25 combines TF and IDF with saturation (penalizes overly long docs
   that just stuff keywords): score = IDF × TF × (k1+1) / (TF + k1×(1-b+b×dl/avgdl))
   
   Lower IDF + higher TF → higher score.
   Rarer terms in a document → more relevant result.

5. RETURN sorted by score (most relevant first)
```

### Analyzers: The Heart of Full-Text Search

The analysis pipeline (tokenizer + filters) is configurable. The right analyzer makes search dramatically better:

```
STANDARD ANALYZER (default):
  "Running Shoes for Marathons"
  → lowercases, splits on whitespace/punctuation
  → ["running", "shoes", "for", "marathons"]
  Problem: "running" ≠ "run", "ran", "runs"

ENGLISH ANALYZER:
  "Running Shoes for Marathons"
  → lowercase, stopword removal, English stemmer
  → ["run", "shoe", "marathon"]
  Now: "runners wear running shoes in marathons" ALL match "run shoe marathon"

CUSTOM ANALYZER FOR PRODUCT SEARCH:
  → lowercase
  → ASCII folding: "café" → "cafe", "naïve" → "naive"
  → synonym expansion: "sofa" → ["sofa", "couch", "settee"]
  → edge-ngram: "runn" → ["r", "ru", "run", "runn"] (autocomplete)
```

## 3. Real-World Context

### Why SQL LIKE Is Not Full-Text Search

```sql
-- SQL LIKE: pattern matching, not full-text search
SELECT * FROM products WHERE description LIKE '%running shoes%';

-- Problems:
-- 1. Cannot find "shoes for running" (word order matters)
-- 2. Cannot find "running shoe" (plural mismatch)
-- 3. No relevance ranking (all matches equal)
-- 4. Cannot handle typos ("runnng shoes" finds nothing)
-- 5. Performance: requires full table scan (no index optimization)
-- 6. Cannot understand "sneakers" as synonym for "shoes"

-- Full-text search handles ALL of these.
-- LIKE is for pattern matching (phone numbers, codes).
-- Full-text search is for natural language queries.
```

### How Google Built Its Search

Google's original PageRank paper (1998) built on these same inverted index principles but added one critical dimension: link-based authority (a page linked by many other pages is authoritative). Modern Google search combines:
- Inverted index for term matching (fast retrieval)
- BM25-style relevance scoring
- PageRank for authority
- Machine learning models (BERT, MUM) for semantic understanding
- Hundreds of other signals (freshness, user engagement, etc.)

The inverted index you build in Elasticsearch is structurally identical to what Google uses — Google's scale and sophistication differ, but the fundamental data structure is the same.

## 4. PostgreSQL Full-Text Search: When It's Enough

Before adding Elasticsearch, check whether PostgreSQL's built-in full-text search covers your needs:

```sql
-- Create a text search column
ALTER TABLE products ADD COLUMN search_vector tsvector;

-- Populate it
UPDATE products SET search_vector = to_tsvector('english', 
    name || ' ' || COALESCE(description, '') || ' ' || COALESCE(brand, '')
);

-- Create a GIN index (fast for containment queries)
CREATE INDEX idx_products_search ON products USING GIN (search_vector);

-- Auto-update via trigger
CREATE OR REPLACE FUNCTION update_product_search_vector()
RETURNS TRIGGER AS $$
BEGIN
    NEW.search_vector = to_tsvector('english',
        COALESCE(NEW.name, '') || ' ' ||
        COALESCE(NEW.description, '') || ' ' ||
        COALESCE(NEW.brand, '')
    );
    RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER product_search_update
    BEFORE INSERT OR UPDATE ON products
    FOR EACH ROW EXECUTE FUNCTION update_product_search_vector();

-- Query with ranking
SELECT
    name,
    ts_rank(search_vector, query) AS rank
FROM products,
     to_tsquery('english', 'running & shoes') query
WHERE search_vector @@ query
ORDER BY rank DESC
LIMIT 20;

-- With phrase search (must appear together)
WHERE search_vector @@ phraseto_tsquery('english', 'running shoes')

-- Highlighting matching terms
SELECT
    name,
    ts_headline('english', description, query, 'MaxWords=30,MinWords=15') AS excerpt
FROM products, to_tsquery('english', 'running') query
WHERE search_vector @@ query;
```

**PostgreSQL FTS is sufficient when:**
- Document count < 1 million
- No complex relevance tuning needed
- Already using PostgreSQL (no new infrastructure)
- Simple query patterns (no faceting, no autocomplete)

**Need Elasticsearch when:**
- Millions of documents
- Complex relevance tuning
- Faceted navigation (filter by category, price range, brand simultaneously)
- Autocomplete with typo tolerance
- Geospatial search
- Analytics on search patterns

## 5. Key Concepts Glossary

```
TERM:           A single searchable token after analysis.
TOKENIZATION:   Splitting text into terms.
ANALYZER:       Pipeline of tokenizer + filters applied to text.
STEMMING:       Reducing words to root form (running→run).
LEMMATIZATION:  Reducing words to dictionary form (are→be, ran→run).
STOP WORDS:     Common words filtered out (the, a, is).
INVERTED INDEX: Maps terms to document IDs (like a book's index).
TF:             Term Frequency — how often a term appears in a document.
IDF:            Inverse Document Frequency — how rare a term is across documents.
BM25:           Modern relevance scoring formula (successor to TF-IDF).
PRECISION:      What fraction of returned results are relevant.
RECALL:         What fraction of all relevant results were returned.
FUZZY SEARCH:   Finding terms that are "close" to the query (typo tolerance).
FACETS:         Aggregated filters (count by category, price range, etc.).
HIGHLIGHTING:   Showing the matching portion of the document in results.
```

## 6. Senior Engineer Perspective

**The precision/recall tradeoff:**

Every search system lives on a spectrum between precision ("I only return results I'm very confident about") and recall ("I return everything that might be relevant"). High precision = fewer but more accurate results. High recall = more results but noisier.

The right balance depends on the use case:
- Internal document search: high recall (users expect to find everything)
- E-commerce product search: balanced (frustrating to miss products, but also frustrating to see irrelevant ones)
- Legal/compliance search: very high recall (can't miss anything)

Analyzers (stemming, synonyms, fuzzy matching) generally improve recall at the cost of precision.

**Build search with relevance feedback:**

The best search systems learn from user behavior. Which results do users click? Which search terms return no results? Which queries lead to immediate reformulations? This feedback loop improves relevance over time. Track: zero-result queries (improve the index), low-click queries (improve ranking), reformulation patterns (improve suggestions). Elasticsearch makes all of this possible but you have to build the feedback mechanism yourself.
