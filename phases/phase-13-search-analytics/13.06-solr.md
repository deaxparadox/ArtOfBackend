## 1. What Is It? (Theory)

Apache Solr (CNET Networks, 2004; Apache Software Foundation 2006) is the original open-source enterprise search platform. It was the dominant search solution for a decade before Elasticsearch emerged. Solr is built on Lucene (the same foundation as Elasticsearch) and offers comparable full-text search capabilities.

Solr is now less commonly chosen for new projects, but remains in heavy production use at many enterprises. Understanding it is important because you'll encounter it in existing codebases.

## 2. Solr vs. Elasticsearch

```
ARCHITECTURE:
  Solr: XML/JSON configuration files, schema.xml defines fields
  Elasticsearch: API-driven configuration, mappings defined via JSON API
  
  Solr requires schema changes → XML file edit → restart (schema mode)
  OR: schemaless mode (schema auto-detected, less control)
  
  Elasticsearch: change mappings via API, no restart required.
  Winner: Elasticsearch (much more operational flexibility)

QUERY SYNTAX:
  Solr: Lucene query syntax + Solr-specific syntax
    q=description:"running shoes" AND in_stock:true AND price_cents:[5000 TO 20000]
  
  Elasticsearch: JSON Query DSL
    {"bool": {"must": [...], "filter": [...]}}
  
  Both are powerful. Solr's syntax is more compact for simple queries.
  Elasticsearch's JSON is more expressive for complex queries.

DISTRIBUTED SEARCH:
  SolrCloud: Solr's distributed mode (ZooKeeper-based)
  Elasticsearch: built-in distributed from day one
  
  Elasticsearch's distributed model is simpler to operate and reason about.
  SolrCloud works well but requires ZooKeeper knowledge.

COMMUNITY AND ECOSYSTEM:
  Solr: Apache project, mature, stable, conservative development
  Elasticsearch: faster feature development, larger community, more resources

WHEN YOU ENCOUNTER SOLR:
  Legacy enterprise applications (pre-2014 tech stacks)
  Organizations with existing Solr expertise
  Apache ecosystem shops (Hadoop, Spark, HDFS — Solr integrates naturally)
  When Elastic license concerns exist and OpenSearch isn't preferred
```

## 3. Solr Basics

```xml
<!-- schema.xml: define your fields (classic mode) -->
<schema name="products" version="1.6">
    <field name="id" type="string" indexed="true" stored="true" required="true"/>
    <field name="name" type="text_en" indexed="true" stored="true"/>
    <field name="description" type="text_en" indexed="true" stored="true"/>
    <field name="price_cents" type="pint" indexed="true" stored="true"/>
    <field name="category" type="string" indexed="true" stored="true"/>
    <field name="in_stock" type="boolean" indexed="true" stored="true"/>
    
    <uniqueKey>id</uniqueKey>
</schema>
```

```bash
# Solr query via HTTP
curl "http://localhost:8983/solr/products/select?\
q=description:running+AND+description:shoes\
&fq=in_stock:true&fq=price_cents:[5000+TO+20000]\
&sort=score+desc\
&rows=20&start=0\
&facet=true&facet.field=category&facet.field=brand\
&hl=true&hl.fl=description\
&fl=id,name,price_cents,score"
```

## 4. Senior Engineer Perspective

**The honest recommendation in 2024:**

For new projects, Elasticsearch or OpenSearch is almost always the better choice. They have better tooling, more documentation, better managed services, and more active development. Solr is excellent software and runs in production at massive scale (Netflix uses it for content search), but its developer experience and operational model are less polished.

If you're inheriting a Solr deployment: it likely runs fine, upgrade carefully, don't migrate just because Elasticsearch is newer. If you're starting fresh: Elasticsearch or OpenSearch.
