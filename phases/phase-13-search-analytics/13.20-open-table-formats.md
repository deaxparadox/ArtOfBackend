## 1. What Is It? (Theory)

Open table formats are metadata layers that sit on top of object storage (S3, GCS) and Parquet/ORC files, adding ACID transactions, schema evolution, time travel, and efficient query planning to raw file-based data lakes.

**The problem they solve:**

```
S3 + PARQUET WITHOUT A TABLE FORMAT:
  
  "Update the price of product 123 to $49.99"
  Parquet is immutable. You can't update a row.
  
  WORKAROUNDS (all bad):
  1. Delete the Parquet file, rewrite the entire table with the change.
     For 1TB table: 1TB of rewriting for 1 row change. Catastrophically slow.
  
  2. Write a new "change" file and merge at query time.
     Every query must merge thousands of change files. Gets slower over time.
  
  3. Prohibit updates entirely.
     Impossible for GDPR compliance (must delete user data).
  
WITH A TABLE FORMAT (Iceberg/Delta/Hudi):
  These handle it correctly:
  - Track which Parquet files contain which rows
  - Write only new/changed files
  - Metadata points queries to the correct set of files
  - Support DELETE, UPDATE, MERGE with reasonable performance
```

## 2. The Three Open Table Formats

### Apache Iceberg (Netflix, 2018; Apache 2019)

```
ICEBERG ARCHITECTURE:

  Table metadata (JSON in S3):
  ├── metadata.json         ← current table state + schema + snapshots
  ├── snap-001.avro         ← snapshot: which files constitute the table at this point
  ├── snap-002.avro         ← next snapshot (after an insert/update/delete)
  └── manifest files        ← lists of data files with statistics
  
  Data files (Parquet in S3):
  ├── data/2024/06/data-001.parquet
  ├── data/2024/06/data-002.parquet
  └── delete-files/delete-001.parquet  ← tracks which rows are deleted

ICEBERG TRANSACTION MODEL:
  Every write creates a new snapshot (MVCC similar to PostgreSQL).
  Readers see a consistent snapshot; writers create new snapshots.
  ACID transactions: atomic (all-or-nothing write), isolated reads.
  
KEY FEATURES:
  ✓ ACID transactions
  ✓ Time travel (query past snapshots)
  ✓ Schema evolution (add/rename/drop columns without full rewrite)
  ✓ Partition evolution (change partition scheme without migration)
  ✓ Hidden partitioning (query by date, Iceberg handles partitioning)
  ✓ Multi-engine: Spark, Trino, Flink, BigQuery, Snowflake, Athena
  ✓ Row-level deletes (for GDPR compliance)
```

```python
# Apache Iceberg with PyIceberg (Python)
from pyiceberg.catalog import load_catalog
from pyiceberg.schema import Schema
from pyiceberg.types import NestedField, StringType, LongType

catalog = load_catalog("default", **{
    "type": "rest",
    "uri": "http://iceberg-rest-catalog:8181",
    "s3.region": "us-east-1",
})

# Create a table
schema = Schema(
    NestedField(1, "order_id", LongType(), required=True),
    NestedField(2, "user_id", LongType(), required=True),
    NestedField(3, "amount_cents", LongType()),
)

table = catalog.create_table(
    "analytics.orders",
    schema=schema,
    partition_spec=...,
    properties={
        "write.target-file-size-bytes": "134217728",  # 128MB files
    },
)

# Read with Pandas (or Spark)
df = table.scan().to_pandas()

# Time travel: read as of a specific snapshot
old_df = table.scan(
    snapshot_id=123456789
).to_pandas()

# Schema evolution (no data rewrite!)
with table.update_schema() as update:
    update.add_column("discount_cents", LongType())
```

### Delta Lake (Databricks, 2019)

```
DELTA LAKE:
  Developed by Databricks (creators of Apache Spark).
  Transaction log: JSON files in the table directory (_delta_log/).
  
  _delta_log/
  ├── 00000000000000000000.json  ← transaction 0: initial table creation
  ├── 00000000000000000001.json  ← transaction 1: insert
  ├── 00000000000000000002.json  ← transaction 2: delete some rows
  └── 00000000000000000010.checkpoint.parquet  ← compact log every 10 txns

KEY FEATURES:
  ✓ ACID transactions
  ✓ Time travel (30 days by default)
  ✓ Schema enforcement + evolution
  ✓ MERGE (upsert) statement
  ✓ VACUUM to remove old files
  ✓ Deep Databricks integration
  ✓ Optimized for Spark workloads
  ✗ Historically less multi-engine (improving: Delta Standalone, Delta-RS)
```

### Apache Hudi (Uber, 2016; Apache 2019)

```
HUDI:
  Designed for "incremental" processing — efficient handling of new/changed records.
  
  TWO TABLE TYPES:
  Copy-on-Write (CoW): like Iceberg, rewrite files on update
    Fast reads, slower writes
  Merge-on-Read (MoR): write to delta logs, merge at read time
    Fast writes, slightly slower reads (merge at read)
  
KEY FEATURES:
  ✓ Incremental queries (what changed since timestamp T?)
  ✓ Kafka/DeltaStreamer: native streaming ingestion
  ✓ Efficient upserts at scale (deduplication at ingestion time)
  ✓ Best at: CDC ingestion from databases
  ✗ More complex to operate than Iceberg/Delta
```

## 3. Which Format to Choose

```
                Apache Iceberg         Delta Lake           Apache Hudi
Multi-engine:   Excellent              Good (improving)     Good
Time Travel:    Excellent              Good                 Limited
CDC/Upserts:    Good                   Good                 Excellent (built for it)
Streaming:      With Flink/Kafka       With Spark Streaming Native DeltaStreamer
AWS support:    Excellent (Athena,     Via open delta-rs    Via S3 + EMR
                Glue, EMR)
Databricks:     Works well             Native               Works well
Schema evol:    Excellent (partition   Good                 Good
                evolution unique)

CHOOSE ICEBERG WHEN:
  ✓ Multi-engine access is important (Spark + Trino + Flink + BigQuery)
  ✓ AWS-native workloads (best Athena/Glue integration)
  ✓ Partition evolution needed (change how data is partitioned over time)
  ✓ Starting a new lakehouse (best community momentum in 2024)

CHOOSE DELTA LAKE WHEN:
  ✓ Primarily using Databricks
  ✓ Heavy Spark workloads
  ✓ Team already familiar with Delta

CHOOSE HUDI WHEN:
  ✓ Streaming CDC ingestion is the primary use case
  ✓ Need efficient upserts at very high velocity
  ✓ Merge-on-Read pattern fits your workload
```

## 4. Real-World Context

### Netflix's Iceberg Adoption

Netflix developed Apache Iceberg internally to solve a critical problem: their petabyte-scale Parquet-based data lake had no efficient way to handle GDPR deletion requests. Deleting a user's data required rewriting entire Parquet files — astronomically expensive at their scale.

Iceberg's row-level deletes let Netflix track which rows to exclude in a small delete file, without rewriting the entire dataset. A user's data appears deleted to all queries immediately after the delete operation, even though the underlying Parquet files are unchanged until the next compaction.

### The Iceberg Momentum in 2024

As of 2024, Apache Iceberg is winning the open table format race. Snowflake added Iceberg tables. BigQuery added Iceberg support (Biglake). AWS Athena and Glue have excellent Iceberg support. Databricks maintains Delta Lake but added Iceberg compatibility. The industry is converging on Iceberg as the open standard — Databricks and Snowflake both need to support it to avoid being locked out of the broader data ecosystem.
