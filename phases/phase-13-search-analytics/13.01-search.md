## Full-Text Search Fundamentals (13.1)

```
HOW FULL-TEXT SEARCH WORKS:

1. TOKENIZATION: split text into tokens
   "The quick brown fox" → ["the", "quick", "brown", "fox"]

2. NORMALIZATION: standardize tokens
   Lowercasing: "Quick" → "quick"
   Stemming: "running" → "run", "runs" → "run"
   Stop words: remove "the", "a", "is"

3. INVERTED INDEX: map token → list of documents
   "quick" → [doc:1, doc:5, doc:23]
   "fox"   → [doc:1, doc:8]
   
   Query "quick fox":
   → docs with "quick" AND "fox" = intersection = [doc:1]
   → Sort by relevance score (TF-IDF, BM25)

4. RANKING (BM25 — the modern standard):
   Score = term frequency in document × inverse document frequency
   Documents with rare query terms rank higher
   Documents with many repeats of the term rank higher
   But diminishing returns on term frequency
```

## Elasticsearch (13.2-13.4)

```python
from elasticsearch import AsyncElasticsearch

es = AsyncElasticsearch(["http://elasticsearch:9200"])

# INDEX CREATION: define mappings (schema)
await es.indices.create(
    index="products",
    body={
        "mappings": {
            "properties": {
                "name":        {"type": "text", "analyzer": "english"},
                "description": {"type": "text", "analyzer": "english"},
                "category":    {"type": "keyword"},    # exact match, not analyzed
                "price_cents": {"type": "integer"},
                "tags":        {"type": "keyword"},
                "created_at":  {"type": "date"},
            }
        },
        "settings": {
            "number_of_shards": 3,    # distribute across 3 nodes
            "number_of_replicas": 1,  # 1 replica per shard (HA)
        }
    }
)

# INDEX A DOCUMENT
await es.index(index="products", id="123", document={
    "name": "Wireless Bluetooth Headphones",
    "description": "Premium noise-cancelling headphones with 30-hour battery",
    "category": "electronics",
    "price_cents": 9999,
    "tags": ["audio", "wireless", "premium"],
    "created_at": "2024-06-15",
})

# SEARCH QUERY
result = await es.search(
    index="products",
    body={
        "query": {
            "bool": {
                "must": [
                    {"multi_match": {
                        "query": "wireless headphones",
                        "fields": ["name^3", "description"],  # name is 3x more important
                        "type": "best_fields",
                    }}
                ],
                "filter": [
                    {"term": {"category": "electronics"}},
                    {"range": {"price_cents": {"lte": 15000}}},
                ],
                "should": [
                    {"term": {"tags": "premium"}},  # boost if premium
                ],
            }
        },
        "sort": [
            {"_score": "desc"},           # primary: relevance
            {"price_cents": "asc"},       # secondary: cheapest first
        ],
        "highlight": {
            "fields": {"name": {}, "description": {}},  # highlight matching terms
        },
        "from": 0,
        "size": 20,
        "aggs": {
            "by_category": {"terms": {"field": "category"}},
            "price_ranges": {
                "range": {
                    "field": "price_cents",
                    "ranges": [
                        {"to": 5000}, {"from": 5000, "to": 15000}, {"from": 15000}
                    ]
                }
            }
        }
    }
)

# Parse results
hits = result["hits"]["hits"]
for hit in hits:
    print(hit["_score"], hit["_source"]["name"])
    if "highlight" in hit:
        print(hit["highlight"]["name"])

# Aggregations (facets for filtering UI)
categories = result["aggregations"]["by_category"]["buckets"]
# [{"key": "electronics", "doc_count": 45}, {"key": "audio", "doc_count": 23}]
```

## Search Engine Comparison (13.7-13.9)

```
POSTGRESQL FTS:
  ✓ Already have PostgreSQL
  ✓ ACID guarantees (search stays in sync with data)
  ✗ Limited relevance tuning
  ✗ Slow for large text corpora
  Use when: < 1M documents, simple search needs

ELASTICSEARCH / OPENSEARCH:
  ✓ Best relevance scoring (BM25)
  ✓ Powerful aggregations (facets, analytics)
  ✓ Scalable to billions of documents
  ✗ Eventual consistency with your database (sync needed)
  ✗ Operational complexity
  Use when: complex search, analytics, large scale

TYPESENSE:
  ✓ Typo tolerance (works even if user misspells)
  ✓ Easy to set up
  ✓ Open source + managed cloud
  ✗ Smaller ecosystem
  Use when: product search, instant autocomplete, typo-tolerant search

MEILISEARCH:
  ✓ Instant search, developer-friendly
  ✓ Typo tolerance, faceting
  ✗ Less powerful than Elasticsearch for complex use cases
  Use when: developer documentation search, small-medium scale

ALGOLIA:
  ✓ Best developer experience, instant results
  ✓ Managed, no ops
  ✗ Expensive at scale
  Use when: you need search fast, budget allows
```

## Vector Databases (13.10-13.12)

```python
# Vector search: find semantically similar content (not keyword matching)
# Use case: "find products similar to 'cozy winter jacket'"
# → Even if product description doesn't contain those exact words

# Step 1: Embed text using an LLM embedding model
from openai import OpenAI
client = OpenAI()

def embed(text: str) -> list[float]:
    response = client.embeddings.create(
        model="text-embedding-3-small",
        input=text
    )
    return response.data[0].embedding  # 1536-dimensional vector

# Step 2: Store embedding in vector database
import psycopg2  # with pgvector extension

# CREATE EXTENSION vector;
# ALTER TABLE products ADD COLUMN embedding vector(1536);
# CREATE INDEX ON products USING ivfflat (embedding vector_cosine_ops);

# Upsert embedding
embedding = embed(product.name + " " + product.description)
await db.execute(
    "UPDATE products SET embedding = $1 WHERE id = $2",
    embedding, product.id
)

# Step 3: Search by semantic similarity
query_embedding = embed("cozy winter jacket")
similar_products = await db.fetch("""
    SELECT id, name, 1 - (embedding <=> $1::vector) AS similarity
    FROM products
    WHERE embedding IS NOT NULL
    ORDER BY embedding <=> $1::vector  -- cosine distance
    LIMIT 20
""", query_embedding)

# pgvector: PostgreSQL extension for vector search
# Pinecone: managed vector database, large scale
# Weaviate: open source, hybrid search (keyword + vector)
# Qdrant: open source, written in Rust, fast
```

## OLTP vs. OLAP (13.13)

```
OLTP (Online Transactional Processing):
  Purpose: day-to-day operations (insert/update/delete individual rows)
  Query pattern: simple queries on small amounts of data
  Example: INSERT ORDER, UPDATE USER, SELECT * WHERE id=123
  Database: PostgreSQL, MySQL (row-oriented storage)
  
OLAP (Online Analytical Processing):
  Purpose: business intelligence, analytics, aggregations
  Query pattern: aggregate large amounts of data
  Example: "Total revenue per region per month for last 3 years"
  Database: ClickHouse, BigQuery, Snowflake (columnar storage)

WHY COLUMNAR STORAGE IS FASTER FOR ANALYTICS:
  Row storage: entire row read even for 2 columns out of 50
  Column storage: only the 2 needed columns read
  
  Plus: columnar data compresses much better (same type = better compression)
  Result: 10-100x faster for analytical queries
```

## ClickHouse (13.14)

```sql
-- ClickHouse: columnar OLAP database, extremely fast for aggregations

CREATE TABLE events (
    timestamp  DateTime,
    user_id    UInt64,
    event_type String,
    page       String,
    country    String
) ENGINE = MergeTree()
ORDER BY (timestamp, user_id)    -- primary sort key (affects query speed)
PARTITION BY toYYYYMM(timestamp) -- partition by month (speeds up time-range queries)
TTL timestamp + INTERVAL 1 YEAR; -- auto-delete after 1 year

-- Query: daily active users for last 30 days
SELECT
    toDate(timestamp) AS date,
    uniqExact(user_id) AS dau    -- exact distinct count
FROM events
WHERE timestamp >= today() - 30
  AND event_type = 'page_view'
GROUP BY date
ORDER BY date;

-- ClickHouse can process this across BILLIONS of rows in seconds.
-- Same query in PostgreSQL on 1B rows: hours.
```
