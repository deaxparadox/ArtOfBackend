## 1. What Is It? (Theory)

Snowflake (2012, Benoit Dageville, Thierry Cruanes, Marcin Zukowski) is a cloud data warehouse that pioneered the separation of storage and compute in data warehousing. Before Snowflake, data warehouses coupled storage and compute on the same nodes — scaling required adding more nodes (both storage and compute) even if you only needed more of one.

Snowflake's innovation: store data in S3 (cheap, unlimited), run compute on separate virtual warehouses (pay only when running queries). This architectural decision made Snowflake one of the most successful enterprise software companies in history.

## 2. How Snowflake Works

### The Three-Layer Architecture

```
CLOUD SERVICES LAYER:
  Authentication, metadata, query optimization, transaction management
  Runs 24/7 (you don't control this)

COMPUTE LAYER (virtual warehouses):
  XS  → 1 "server" unit  → ~2 credits/hour
  S   → 2 server units   → ~4 credits/hour  
  M   → 4 server units   → ~8 credits/hour
  L   → 8 server units   → ~16 credits/hour
  XL  → 16 server units  → ~32 credits/hour
  ... up to 6XL
  
  AUTO-SUSPEND: automatically shuts down after N minutes of inactivity
  → You stop paying when nobody is querying
  
  MULTI-CLUSTER: for high concurrency, run multiple warehouses in parallel
  → Warehouse 1, 2, 3 all serve the same data, distribute load

STORAGE LAYER (S3/Azure Blob/GCS):
  All data stored in compressed columnar format (similar to Parquet)
  Snowflake manages the files, S3 charges ($23/TB/month)
  Data persists independently of compute
```

### The Killer Feature: Time Travel and Zero-Copy Cloning

```sql
-- TIME TRAVEL: query data as it existed at any past point (up to 90 days)
SELECT * FROM orders AT(timestamp => '2024-06-01 12:00:00'::timestamp);
SELECT * FROM orders BEFORE(statement => '<query_id>');  -- before this query ran

-- Accidentally dropped a table? Recover it:
UNDROP TABLE orders;  -- available for 7 days after drop (default)

-- ZERO-COPY CLONING: create a full copy without duplicating storage
CREATE DATABASE prod_clone CLONE production;  -- INSTANT, uses zero extra storage
-- Clone shares the underlying S3 files with the source until it diverges
-- Perfect for: testing migrations, development environments, point-in-time snapshots

-- DATA SHARING: share data across Snowflake accounts without copying
CREATE SHARE partner_share;
GRANT USAGE ON DATABASE analytics TO SHARE partner_share;
GRANT SELECT ON TABLE analytics.public.metrics TO SHARE partner_share;
-- Partner can query your data in real-time (no ETL, no copy, you control access)
```

## 3. Snowflake vs. BigQuery

```
                    Snowflake              BigQuery
Cloud:              Multi-cloud            GCP only
                    (AWS, Azure, GCP)      
Compute model:      Virtual warehouses     Serverless slots
                    (always-on when used)  (auto-scale, pay per query)
Pricing:            Credits/hour           Per TB scanned
Concurrency:        Multi-cluster          Automatic (up to slot limits)
Time Travel:        90 days               7 days (via table snapshots)
Zero-copy Clone:    Yes (instant)          Limited
Data Sharing:       Snowflake Marketplace  BigQuery Analytics Hub
SQL Dialect:        Standard SQL           Standard SQL (more GCP-native)
Performance:        Excellent              Excellent
Semi-structured:    VARIANT (JSON)         JSON/ARRAY natively
Best for:           Enterprise analytics,  GCP-native, serverless,
                    multi-cloud            pay-per-query analytics
Cost:               Higher for continuous  Higher for frequent large queries
                    heavy usage            Lower for sporadic queries
```

## 4. When to Choose Snowflake

```
CHOOSE SNOWFLAKE WHEN:
  ✓ Multi-cloud: running on AWS or Azure (BigQuery is GCP-only)
  ✓ Enterprise features: time travel, data sharing, governance
  ✓ Data marketplace: buying/selling data as a product
  ✓ Complex semi-structured data (JSON stored as VARIANT)
  ✓ Require dedicated compute capacity (predictable performance)
  ✓ Organization already has Snowflake contracts/expertise

CHOOSE BIGQUERY WHEN:
  ✓ GCP-native infrastructure
  ✓ Truly serverless (no idle costs)
  ✓ Ad-hoc, unpredictable query patterns (pay per query)
  ✓ Strong integration with Google ML services
  ✓ Lower cost for sporadic queries
```

## 5. Senior Engineer Perspective

**Snowflake's pricing surprise: credit consumption during idle time:**

A virtual warehouse configured with `AUTO_SUSPEND = 60` means it keeps running for 60 minutes after the last query. If you accidentally leave a large warehouse running overnight, you'll see a significant charge. Always set `AUTO_SUSPEND` to 60-120 seconds for development warehouses.

**The Snowflake Data Cloud:**

Snowflake's Data Marketplace allows organizations to share and monetize data without moving it. A company with weather data can sell access to their Snowflake dataset; buyers query it directly from their own Snowflake account. This "data as a product" model is why Snowflake is positioned as more than a database — it's infrastructure for a data economy.
