## 1. What Is It? (Theory)

Traditional full-text search matches keywords. A search for "happy" doesn't find documents about "joyful" or "elated" even though they mean the same thing. Vector search solves this by representing text (or images, audio, code) as points in high-dimensional space where **semantic similarity corresponds to geometric proximity**.

**The transformation:**

```
WORD/SENTENCE → EMBEDDING MODEL → VECTOR (1536 numbers)

"I love running shoes" → [0.234, -0.891, 0.123, ..., 0.456]  (1536 dimensions)
"athletic footwear"   → [0.231, -0.889, 0.121, ..., 0.459]  (very similar!)
"coffee mug"          → [-0.781, 0.234, -0.567, ..., 0.123] (very different)

CLOSENESS IN VECTOR SPACE = SEMANTIC SIMILARITY
  If two texts are semantically similar → their vectors are close together
  If two texts are semantically different → their vectors are far apart
```

**The real-world analogy:**

Think of a map. Cities that are geographically close to each other are on the map close together. Now imagine a "meaning map" where words and phrases are placed so that semantically similar concepts are physically nearby. "Running" and "jogging" are nearby. "Running" and "cooking" are far apart. Vector search finds the nearest neighbors in this semantic space.

## 2. How Embeddings Work

```
THE EMBEDDING PROCESS:

1. TRAIN (done by AI companies, you don't do this):
   A neural network learns to map text to vectors such that:
   similar meanings → similar vectors
   
   Training data: billions of text pairs with similarity labels
   Result: a model that can convert any text to a vector

2. EMBED (you do this):
   Pass text through the embedding model → get vector
   
   from openai import OpenAI
   client = OpenAI()
   
   response = client.embeddings.create(
       model="text-embedding-3-small",  # 1536-dimensional vectors
       input="Nike Air Max running shoes",
   )
   embedding = response.data[0].embedding  # list of 1536 floats

3. STORE:
   Store the vector alongside the original document
   Index for fast nearest-neighbor search

4. QUERY:
   Embed the query with the SAME model
   Find documents with vectors closest to the query vector

KEY REQUIREMENT:
  Query and documents must use the SAME embedding model.
  Different models produce incompatible vector spaces.
  If you switch models: must re-embed all documents.
```

### Cosine Similarity vs. Euclidean Distance

```
COSINE SIMILARITY:
  Measures the angle between two vectors.
  Values: -1 (opposite) to 1 (identical direction)
  Used by: most text embedding models
  
  doc_a = [0.234, -0.891, ...]
  doc_b = [0.231, -0.889, ...]
  
  cosine_similarity = dot_product(a, b) / (||a|| × ||b||)
  
  High cosine similarity (close to 1) → semantically similar
  
EUCLIDEAN DISTANCE (L2):
  Measures the straight-line distance between two points.
  Used by: some image embedding models
  
  Small distance → similar vectors

RECOMMENDATION:
  For text: use cosine similarity (or inner product for normalized vectors)
  For normalized vectors: inner product = cosine similarity (faster computation)
```

## 3. ANN Algorithms: Making Nearest-Neighbor Fast

A database with 10 million documents can't check the distance to every document for every query — that's 10M distance calculations per query. **Approximate Nearest Neighbor (ANN)** algorithms trade a small accuracy loss for massive speed gains.

```
EXACT NEAREST NEIGHBOR:
  Check every document: O(n × d) where d = dimensions
  10M docs × 1536 dims = 15 billion operations per query
  Too slow for production (would take seconds)

APPROXIMATE NEAREST NEIGHBOR (ANN):
  Build an index structure that lets you find "close enough" neighbors quickly
  
  HNSW (Hierarchical Navigable Small World) — most commonly used:
  
  Layer 3 (sparse): [doc 1] ─────────────────────── [doc 9] ─── [doc 100]
                                                                       │
  Layer 2 (medium): [doc 1] ── [doc 5] ───── [doc 9] ─── [doc 50] ─ [doc 100]
                                                                         │
  Layer 1 (dense):  [doc 1] [doc 2] [doc 3] [doc 4] [doc 5] ... [doc 100]
  
  Query: start at highest layer, find approximate nearest, descend to next layer,
         refine, repeat until lowest layer.
  
  Time: O(log n) instead of O(n)
  Trade-off: finds ~95-99% of true nearest neighbors (not exact)
  
  For search applications: 1% approximation error is invisible to users.

OTHER ANN ALGORITHMS:
  IVF (Inverted File Index): partition space into clusters, search most relevant
  Product Quantization: compress vectors to save memory
  Annoy: tree-based (Spotify's recommendation library)
  ScaNN: Google's SIMD-optimized for GPU acceleration
```

## 4. Hybrid Search: Combining Keyword + Semantic

Pure vector search misses exact matches. Pure keyword search misses semantic matches. Hybrid search combines both:

```python
# Hybrid search: keyword BM25 + vector similarity → merged results

async def hybrid_search(query: str, top_k: int = 20) -> list[dict]:
    # STEP 1: Embed the query
    embedding = await get_embedding(query)
    
    # STEP 2: Run both searches in parallel
    keyword_results, vector_results = await asyncio.gather(
        elasticsearch_bm25_search(query, top_k=100),
        vector_db_search(embedding, top_k=100),
    )
    
    # STEP 3: Reciprocal Rank Fusion (RRF) — merge ranked lists
    # RRF formula: score = sum(1 / (k + rank)) where k=60 is a constant
    scores = {}
    
    for rank, result in enumerate(keyword_results, start=1):
        doc_id = result["id"]
        scores[doc_id] = scores.get(doc_id, 0) + 1 / (60 + rank)
    
    for rank, result in enumerate(vector_results, start=1):
        doc_id = result["id"]
        scores[doc_id] = scores.get(doc_id, 0) + 1 / (60 + rank)
    
    # STEP 4: Sort by combined score
    sorted_ids = sorted(scores, key=scores.get, reverse=True)
    return [get_document(doc_id) for doc_id in sorted_ids[:top_k]]

# WHEN EACH SEARCH TYPE WINS:

# Query: "Nike Air Max 90"  → keyword search wins (exact product name)
# Query: "comfortable shoes for standing all day" → vector search wins (semantic)
# Query: "red wedding dress" → hybrid wins (color=keyword, occasion=semantic)
```

## 5. Real-World Context

### Why Vector Search Transformed Product Discovery

Traditional e-commerce search fails for "outfit inspiration" or "shoes that go with a blue dress" — queries that describe what you want functionally, not what the product is literally called. Embedding-based search understands these intent-driven queries.

Pinterest, Spotify, Netflix, and every major recommendation system now uses vector search at their core: similar pins, songs like this, shows you might enjoy. These are all nearest-neighbor queries in embedding space.

### The RAG Architecture

Vector search is the retrieval mechanism in RAG (Retrieval-Augmented Generation):

```
User question: "How do I return a product I bought last week?"

STEP 1: Embed the question → query vector
STEP 2: Find top-5 most similar policy/FAQ documents in vector DB
STEP 3: Include those documents as context in the LLM prompt
STEP 4: LLM generates an accurate, grounded answer

WITHOUT VECTOR SEARCH: LLM might hallucinate your return policy
WITH VECTOR SEARCH: LLM answers based on your actual policy documents
```

## 6. Senior Engineer Perspective

**The embedding model lock-in:**

Once you embed your 10 million documents with OpenAI's `text-embedding-3-small` model, your entire vector index depends on that model. If you want to switch to a better model (or reduce cost), you must re-embed everything. Plan for this: use an abstraction layer, budget for periodic re-embedding as models improve.

**Dimensions vs. performance:**

Larger embedding dimensions (3072 vs 1536 vs 512) generally improve quality but increase storage and computation cost. For most applications, `text-embedding-3-small` (1536 dims) or even smaller models (384 dims from sentence-transformers) provide sufficient quality at much lower cost. Always benchmark before choosing the largest available model.
