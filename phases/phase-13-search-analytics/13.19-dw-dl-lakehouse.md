## 1. What Is It? (Theory)

Three architectural patterns for storing and analyzing large-scale data have emerged over the last two decades. Understanding which to use (or how they coexist) is essential for designing a modern data platform.

---

## 2. The Three Architectures

### Data Warehouse (1990s–present)

```
WHAT IT IS:
  Structured, relational storage optimized for SQL analytics.
  Data is cleaned, transformed, and loaded (ETL) before storage.
  Schema is defined before data arrives (schema-on-write).
  
EXAMPLES: Snowflake, BigQuery, Redshift, ClickHouse

STRUCTURE:
  Raw data sources → ETL pipeline → transformed, clean data → Data Warehouse
  
  Data in the warehouse: structured, typed, validated, consistent.
  Queries: fast, predictable.
  
STRENGTHS:
  ✓ Fast queries on structured data
  ✓ ACID transactions
  ✓ Strong governance and data quality
  ✓ Familiar SQL interface for analysts

WEAKNESSES:
  ✗ Expensive storage (proprietary formats, can't use cheaper object storage)
  ✗ Schema rigidity: schema must be designed upfront
  ✗ Can't store unstructured data (images, PDFs, logs in their raw form)
  ✗ ETL pipelines are brittle — source schema changes break the pipeline
  ✗ Data duplication: raw data + warehouse copy
```

### Data Lake (2010s)

```
WHAT IT IS:
  Store everything in cheap object storage (S3, GCS) in any format.
  Schema applied when reading (schema-on-read).
  No transformation required before storage.
  
EXAMPLES: S3 + Spark/Presto, Azure Data Lake Storage, GCS

STRUCTURE:
  Raw data sources → dump to S3 as-is (CSV, JSON, Parquet, images, logs)
  
  When you need to analyze: read raw data, apply schema, compute result.
  
STRENGTHS:
  ✓ Cheap storage ($0.023/GB/month on S3 vs. $23/GB/month on Redshift)
  ✓ Store everything (unstructured, semi-structured, structured)
  ✓ Schema flexibility: add new data types without migration
  ✓ Enables ML training on raw data
  ✓ Durability: S3 has 11 nines durability

WEAKNESSES:
  ✗ Slow queries without optimization (scanning raw CSVs is slow)
  ✗ No ACID transactions (can't update rows, no rollback)
  ✗ "Data swamp": without governance, becomes untrustworthy
  ✗ Harder for analysts (not SQL-friendly out of the box)
  ✗ No data versioning (delete a file = data gone)
```

### Data Lakehouse (2020s–present)

```
WHAT IT IS:
  Combines data lake (cheap object storage) + data warehouse (ACID, SQL)
  in a single architecture.
  
  Data stored in open formats (Parquet/ORC) in object storage.
  Table format layer (Iceberg, Delta Lake, Hudi) adds:
    - ACID transactions
    - Schema evolution
    - Time travel
    - Row-level updates/deletes
  Multiple engines read the same data (Spark, Presto, BigQuery, Snowflake)
  
EXAMPLES: Delta Lake (Databricks), Apache Iceberg, Apache Hudi

STRUCTURE:
  Raw data → land in S3/GCS (as-is or minimally processed)
  Table format layer: manages files, tracks versions, enforces schema
  Query engines: Spark, Trino, BigQuery, Athena all read same data

STRENGTHS:
  ✓ Cheap storage (S3) + fast queries (columnar format + partitioning)
  ✓ ACID transactions (update/delete without full table rewrite)
  ✓ Time travel (query data as of any past timestamp)
  ✓ One copy of data (no duplication between lake and warehouse)
  ✓ Open standards (not vendor lock-in)
  ✓ Works for both BI analytics AND ML training
  
WEAKNESSES:
  ✗ More complex to operate than a managed warehouse
  ✗ Emerging ecosystem (still maturing)
  ✗ Requires compute engine management (Spark/Trino cluster)
```

## 3. Side-by-Side Comparison

```
                DATA WAREHOUSE        DATA LAKE           DATA LAKEHOUSE
─────────────────────────────────────────────────────────────────────────────
Storage:        Proprietary/managed   Object storage (S3)  Object storage (S3)
Cost:           High ($20-100/TB/mo)  Low ($0.02/GB/mo)    Low ($0.02/GB/mo)
Format:         Proprietary           Any (CSV, JSON, ORC)  Open (Parquet/ORC)
Schema:         Write-time (rigid)    Read-time (flexible)  Write-time (flexible)
ACID:           Yes                   No                   Yes (via table format)
SQL:            Excellent             Via Spark/Presto      Via Spark/Trino/etc.
ML Training:    Possible but harder   Native               Native
Time Travel:    Snowflake (90d)        No                   Yes (configurable)
Update/Delete:  Standard SQL          No (file-level only)  Yes (row-level)
Concurrency:    High                  Low                   Moderate
Governance:     Built-in              Manual                Catalog required
Maturity:       Decades               ~10 years             ~3-5 years

BEST FOR:
  Data Warehouse: BI reporting, SQL analytics, business metrics
  Data Lake: raw storage, ML feature stores, compliance archives, unstructured
  Lakehouse: unified analytics + ML + cost efficiency at scale
```

## 4. The Modern Architecture

Most large companies use all three in a layered approach:

```
MODERN DATA PLATFORM ARCHITECTURE:

BRONZE (Raw):
  All data lands here first, unchanged.
  Format: as ingested (JSON events, CSV exports, CDC binary)
  Storage: S3/GCS (cheapest)
  Purpose: source of truth, disaster recovery, replay
  Retention: 7 years (compliance)
  
SILVER (Cleaned):
  Bronze data cleaned, normalized, deduped.
  Format: Parquet + Iceberg/Delta (columnar, partitioned)
  Storage: S3/GCS
  Purpose: analytics, ML feature store
  Access: Spark, Trino, Athena
  
GOLD (Business-Ready):
  Silver data aggregated into business-level tables.
  Format: data warehouse (Snowflake, BigQuery, Redshift)
  OR: Parquet in a lakehouse with BI engine
  Purpose: dashboards, reports, self-serve analytics
  Access: Looker, Tableau, direct SQL

FLOW:
  Operational DBs → CDC → Bronze → Spark/dbt → Silver → dbt → Gold → BI tools
```

## 5. Senior Engineer Perspective

**The data swamp antipattern:**

Data lakes frequently become "data swamps" — data lands, nobody documents it, nobody knows what's there, nobody trusts the data quality. The lakehouse architecture with a metadata catalog (AWS Glue, Apache Atlas, DataHub) is how teams prevent this. Every dataset needs: an owner, a schema, data quality checks, and documentation. Without governance, cheap storage creates expensive confusion.

**The lakehouse is winning the architectural debate:**

As of 2024, the trend is clearly toward lakehouses. The cost difference between proprietary warehouse storage and S3 is too large to ignore at scale. The table formats (especially Apache Iceberg, covered in topic 13.20) have matured enough to be production-ready. Major cloud providers are adding lakehouse capabilities to their services (BigQuery external tables + Iceberg, Snowflake Iceberg tables). The question is no longer "warehouse vs. lake" but "which table format for the lakehouse?"
