## 1. Why Dedicated Vector Databases?

When pgvector hits its limits (scale, performance, or feature requirements), dedicated vector databases provide:
- Optimized storage layouts for vectors
- Advanced ANN algorithms with fine-tuned parameters
- Better multi-filter performance (pre-filtering then vector search)
- Horizontal scaling designed for vector workloads
- Purpose-built monitoring and management

## 2. The Main Players

### Pinecone: The Managed Standard

```python
from pinecone import Pinecone, ServerlessSpec

pc = Pinecone(api_key="YOUR_API_KEY")

# Create index
pc.create_index(
    name="products",
    dimension=1536,
    metric="cosine",
    spec=ServerlessSpec(cloud="aws", region="us-east-1")
)

index = pc.Index("products")

# Upsert vectors (id + vector + metadata)
index.upsert(vectors=[
    {
        "id": "prod-123",
        "values": [0.234, -0.891, ...],  # 1536-dimensional embedding
        "metadata": {
            "name": "Nike Air Max",
            "category": "footwear",
            "price_cents": 12999,
            "in_stock": True,
        }
    },
    ...
])

# Query with metadata filters
results = index.query(
    vector=[0.234, -0.891, ...],    # query embedding
    top_k=20,
    filter={
        "category": {"$eq": "footwear"},
        "price_cents": {"$lte": 20000},
        "in_stock": {"$eq": True},
    },
    include_metadata=True,
)

# PRICING (serverless, 2024):
# Storage: $0.08 per GB/month (1536-dim: 6.1KB × 1M docs = ~6.1GB = ~$0.49/month)
# Queries: ~$0.000002 per query (1M queries = $2)
# Very affordable for most use cases.
```

### Qdrant: Open-Source, Rust-Based, Excellent Performance

```python
from qdrant_client import QdrantClient, models

client = QdrantClient("localhost", port=6333)

# Create collection
client.create_collection(
    collection_name="products",
    vectors_config=models.VectorParams(
        size=1536,
        distance=models.Distance.COSINE,
    ),
)

# Add vectors
client.upsert(
    collection_name="products",
    points=[
        models.PointStruct(
            id=123,
            vector=[0.234, -0.891, ...],
            payload={
                "name": "Nike Air Max",
                "category": "footwear",
                "price_cents": 12999,
            }
        )
    ],
)

# Search with filters
results = client.search(
    collection_name="products",
    query_vector=[0.234, -0.891, ...],
    query_filter=models.Filter(
        must=[
            models.FieldCondition(
                key="category",
                match=models.MatchValue(value="footwear")
            ),
            models.FieldCondition(
                key="price_cents",
                range=models.Range(lte=20000)
            ),
        ]
    ),
    limit=20,
)
```

### Weaviate: Vector DB with GraphQL and Auto-Embedding

```python
import weaviate
from weaviate.classes.query import MetadataQuery

client = weaviate.connect_to_local()

# Weaviate can auto-embed if you configure a vectorizer
# (integrates with OpenAI, Cohere, HuggingFace, etc.)
collection = client.collections.get("Product")

# Add without pre-computing embedding (Weaviate calls the embedding API):
collection.data.insert({
    "name": "Nike Air Max Running Shoes",
    "description": "Lightweight shoes with cushioning",
    "price_cents": 12999,
    "category": "footwear",
})

# Search using natural language (Weaviate embeds the query):
results = collection.query.near_text(
    query="comfortable shoes for running",
    filters=weaviate.classes.query.Filter.by_property("in_stock").equal(True),
    limit=10,
    return_metadata=MetadataQuery(certainty=True),
)

# Weaviate's auto-embedding reduces code (no need to call embedding API yourself)
# Trade-off: vendor dependency on Weaviate's model integrations
```

## 4. The Filtering Challenge

The hardest problem in vector databases: how to efficiently combine vector search with metadata filtering.

```
NAIVE APPROACH (slow):
  1. Find top-10,000 approximate nearest neighbors
  2. Filter by metadata (category=footwear, price<$200)
  3. Return top-20 from filtered set
  
  Problem: if only 0.1% of documents match the filter,
  you need to retrieve 20,000 neighbors just to get 20 that pass the filter.

SMART APPROACHES:
  PRE-FILTERING (Qdrant's default):
    Build separate indexes on filterable fields.
    First apply metadata filter → get matching document IDs.
    Then search only within those IDs for vector similarity.
    
    Fast when filter is selective (returns few documents).
    Slower when filter returns many documents (still scan many vectors).
  
  POST-FILTERING:
    Find top-K nearest vectors.
    Then apply metadata filter.
    
    Fast when filter isn't very selective.
    Inefficient when filter is very selective.
  
  HYBRID (Qdrant and Pinecone use adaptive approaches):
    Choose pre or post filtering based on estimated filter selectivity.
    Most production vector DBs use this adaptive approach.
```

## 5. Senior Engineer Perspective

**The vector DB landscape is moving fast:**

In 2023-2024, every major database added vector support (PostgreSQL with pgvector, Redis with RedisSearch, MongoDB Atlas, Elasticsearch, etc.). Dedicated vector databases (Pinecone, Qdrant, Weaviate) are still faster and more feature-rich for pure vector workloads, but the gap is narrowing.

**My recommendation for 2024:**

1. Start with pgvector (zero new infrastructure, good enough for millions of vectors)
2. If you outgrow it: evaluate Qdrant (open-source, excellent performance, self-hosted option)
3. If you need zero-ops managed at large scale: Pinecone

Don't choose a vector database before you've validated that you actually need one. Many teams benefit from simple semantic search without the overhead of a dedicated vector database.
