## 1. What Is It? (Theory)

Elasticsearch's Query DSL (Domain-Specific Language) is a JSON-based language for constructing queries. It's more powerful than SQL for search — it supports full-text relevance, fuzzy matching, geographic queries, and complex aggregations in a single request.

Understanding the two fundamental query types is essential:

- **Query context:** "How well does this document match?" — computes a relevance score, affects ranking
- **Filter context:** "Does this document match yes/no?" — no scoring, faster, cacheable

The rule: use filters for exact conditions (in_stock=true, category="shoes"), use queries for relevance (keyword search in text fields). Filters are much faster because they're cached and don't compute scores.

## 2. Core Query Types

### match: Full-Text Search on Text Fields

```json
GET /products/_search
{
  "query": {
    "match": {
      "description": "running shoes"
      // Analyzed: ["run", "shoe"]
      // Finds documents containing both terms (AND by default in newer ES)
    }
  }
}

// Control AND vs OR:
{
  "query": {
    "match": {
      "description": {
        "query": "running shoes marathon",
        "operator": "and"   // all terms must match
        // vs "or" (any term)
      }
    }
  }
}

// Fuzzy matching (typo tolerance):
{
  "query": {
    "match": {
      "description": {
        "query": "runnng shoes",  // typo
        "fuzziness": "AUTO"      // auto-determines edit distance based on word length
      }
    }
  }
}
```

### term: Exact Match on Keyword Fields

```json
// term: no analysis, exact match
// Use for: keyword fields, numbers, booleans, IDs
{
  "query": {
    "term": {
      "category": "footwear"    // exact match, case-sensitive
    }
  }
}

// terms: match any of these values
{
  "query": {
    "terms": {
      "brand": ["Nike", "Adidas", "Puma"]
    }
  }
}

// range: numeric/date ranges
{
  "query": {
    "range": {
      "price_cents": {
        "gte": 5000,    // >= 50.00
        "lte": 20000    // <= 200.00
      }
    }
  }
}
```

### bool: Combining Multiple Queries

The `bool` query is the most important — it combines multiple queries with AND/OR/NOT logic:

```json
{
  "query": {
    "bool": {
      "must": [
        // MUST match → contributes to score (AND)
        {
          "match": {
            "description": "running shoes"
          }
        }
      ],
      "filter": [
        // MUST match → does NOT contribute to score (AND, cached)
        { "term": { "in_stock": true } },
        { "range": { "price_cents": { "lte": 20000 } } }
      ],
      "should": [
        // SHOULD match → boosts score but not required (OR)
        { "term": { "brand": "Nike" } },
        { "match": { "tags": "marathon" } }
      ],
      "must_not": [
        // MUST NOT match → excludes documents (NOT, cached)
        { "term": { "category": "sandals" } }
      ]
    }
  }
}
// Result: documents that:
// - contain "running shoes" in description (scored)
// - are in stock (filtered, not scored)
// - cost <= $200 (filtered, not scored)
// - preferably Nike OR mention marathon (optional boosts)
// - are NOT sandals (excluded)
```

### multi_match: Search Across Multiple Fields

```json
{
  "query": {
    "multi_match": {
      "query": "running shoes",
      "fields": ["name^3", "description", "brand^2"],
      // ^3: name matches worth 3x more than description
      "type": "best_fields"  
      // best_fields: score from best matching field
      // most_fields: combine scores from all matching fields
      // cross_fields: query terms can span fields ("Nike" in brand, "Air" in name)
      // phrase: search as phrase across fields
    }
  }
}
```

## 3. Aggregations: Analytics Within Search

Aggregations let you compute analytics alongside search results — the same request returns both search results AND facets/statistics:

```python
result = await es.search(
    index="products",
    body={
        "query": {
            "bool": {
                "must": [{"match": {"description": "running shoes"}}],
                "filter": [{"term": {"in_stock": True}}]
            }
        },
        "size": 20,     # return 20 search results
        
        # AGGREGATIONS: compute on all matching documents (not just page 20)
        "aggs": {
            # Faceted navigation: count by category
            "by_category": {
                "terms": {
                    "field": "category.keyword",
                    "size": 10
                }
            },
            
            # Price range buckets
            "price_ranges": {
                "range": {
                    "field": "price_cents",
                    "ranges": [
                        {"key": "Under $50", "to": 5000},
                        {"key": "$50-$100", "from": 5000, "to": 10000},
                        {"key": "$100-$200", "from": 10000, "to": 20000},
                        {"key": "Over $200", "from": 20000}
                    ]
                }
            },
            
            # Price statistics
            "price_stats": {
                "stats": {"field": "price_cents"}
                # Returns: min, max, avg, sum, count
            },
            
            # Nested agg: top brands per category
            "by_category_with_brands": {
                "terms": {"field": "category.keyword", "size": 5},
                "aggs": {
                    "top_brands": {
                        "terms": {"field": "brand.keyword", "size": 3}
                    }
                }
            },
            
            # Date histogram (for time-series)
            "orders_over_time": {
                "date_histogram": {
                    "field": "created_at",
                    "calendar_interval": "month"
                }
            }
        }
    }
)

# Parse results
hits = result["hits"]["hits"]                    # search results
categories = result["aggregations"]["by_category"]["buckets"]
# [{"key": "footwear", "doc_count": 234}, {"key": "sports", "doc_count": 145}]

price_ranges = result["aggregations"]["price_ranges"]["buckets"]
# [{"key": "Under $50", "doc_count": 45}, ...]

stats = result["aggregations"]["price_stats"]
# {"min": 999, "max": 59999, "avg": 12456.7, "count": 456}
```

## 4. Highlight and Suggest

```python
result = await es.search(
    index="products",
    body={
        "query": {"match": {"description": "running shoes"}},
        
        # HIGHLIGHT: show matching snippet with terms emphasized
        "highlight": {
            "fields": {
                "description": {
                    "fragment_size": 150,       # words per fragment
                    "number_of_fragments": 3,   # max fragments to return
                    "pre_tags": ["<em>"],        # HTML tag wrapping match
                    "post_tags": ["</em>"]
                }
            }
        },
        
        # SUGGEST: autocomplete / spell-check
        "_source": ["name", "price_cents"],  # only return these fields
    }
)

# Accessing highlights
for hit in result["hits"]["hits"]:
    product = hit["_source"]
    highlights = hit.get("highlight", {}).get("description", [])
    # ["... lightweight <em>running</em> <em>shoes</em> for daily use ..."]
    
# Autocomplete with completion suggester (requires special mapping):
await es.search(
    index="products",
    body={
        "suggest": {
            "product-suggest": {
                "prefix": "runn",
                "completion": {
                    "field": "name_suggest",    # requires completion field type
                    "size": 5,
                    "fuzzy": {"fuzziness": 1}
                }
            }
        }
    }
)
```

## 5. Senior Engineer Perspective

**Filter before you score:**

The most common Elasticsearch performance mistake is putting conditions in `must` that should be in `filter`. `must` computes relevance scores (CPU-intensive). `filter` just checks yes/no (cacheable, fast). Always ask: "Does this condition affect relevance, or is it just a hard requirement?" If the latter: use `filter`.

```json
// SLOW (must computes scores for all three conditions):
"must": [
    {"match": {"description": "running shoes"}},
    {"term": {"in_stock": true}},
    {"range": {"price_cents": {"lte": 20000}}}
]

// FAST (filter for exact conditions, must only for full-text):
"must": [{"match": {"description": "running shoes"}}],
"filter": [
    {"term": {"in_stock": true}},
    {"range": {"price_cents": {"lte": 20000}}}
]
```

**Prefer `keyword` aggregations over `text` aggregations:**

Aggregating on a `text` field requires loading analyzed terms — it works but is expensive and may give unexpected results (stemmed tokens instead of original values). Always map fields that will be aggregated as `keyword` (or use `.keyword` subfield).
