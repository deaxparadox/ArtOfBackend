## 1. What Is It? (Theory)

BigQuery (Google, 2010) is Google's fully managed, serverless data warehouse. You load data, run SQL queries, and pay only for the data scanned — no clusters to provision, no tuning required, no capacity planning.

The key word is **serverless**: there are no instances to manage. BigQuery automatically scales to thousands of compute slots for large queries, then scales down to zero when idle. You pay for what you use, not for what you provision.

---

## 2. How BigQuery Works

### The Dremel Architecture

BigQuery is built on Google's Dremel research paper (2010). The core insight: separate storage from compute, use massively parallel query execution on columnar storage.

```
QUERY EXECUTION:

  SQL query submitted
        ↓
  Query is compiled and broken into subtasks
        ↓
  Thousands of "slots" (compute workers) assigned
  Each slot processes a subset of the data in parallel
        ↓
  Each slot reads ONLY the columns referenced in the query
  from Google's Colossus distributed file system
        ↓
  Results merged and returned to the user
        ↓
  Slots released (you don't pay for them when idle)

IMPLICATION:
  A query on 1TB of data → distributed across 1000+ workers
  Each worker reads ~1GB
  Processing time: seconds, not hours
  
  Same query on a traditional data warehouse: might take 30 minutes
  (single node, sequential processing)
```

### Pricing Model

```
BIGQUERY PRICING (on-demand, 2024):
  Storage: $0.02 per GB/month ($20 per TB)
  Queries: $5 per TB scanned ($5 per 1TB read)
  
  First 1TB/month of queries: FREE
  First 10GB/month of storage: FREE

COST EXAMPLES:
  Small analytics team, 100GB data, 10TB queries/month:
    Storage: $2
    Queries: 10 × $5 = $50
    Total: ~$52/month
  
  Medium company, 1TB data, 100TB queries/month:
    Storage: $20
    Queries: 100 × $5 = $500
    Total: ~$520/month
  
  Large company, 10TB data, 1PB queries/month:
    Storage: $200
    Queries: 1000 × $5 = $5,000
    Total: ~$5,200/month
    
  FLAT-RATE PRICING (for high-usage):
    Enterprise: commit to N slots (compute), unlimited queries
    100 slots ≈ $2,000/month (unlimited queries within that compute capacity)

COST OPTIMIZATION:
  SELECT only needed columns (avoid SELECT *)
  Use partition filters (WHERE date >= '2024-01-01')
  Cluster tables on common filter columns
  Cache query results (same query within 24h: free re-run)
```

## 3. Key BigQuery Features

### Partitioning and Clustering

```sql
-- Partitioned table: BigQuery skips irrelevant partitions
CREATE TABLE `project.dataset.events`
PARTITION BY DATE(timestamp)    -- partition by date
CLUSTER BY user_id, event_type  -- cluster within partitions for co-location
AS SELECT * FROM source_table;

-- Query that benefits from partitioning + clustering:
SELECT user_id, count(*) AS events
FROM `project.dataset.events`
WHERE DATE(timestamp) = '2024-06-15'  -- partition pruning: reads only June 15 data
  AND event_type = 'purchase'         -- clustering benefits: co-located events
GROUP BY user_id

-- BigQuery estimates bytes scanned before executing:
-- "This query will process 2.1 GB when run."
-- Let you see the cost before committing to running it.
```

### Streaming Inserts vs. Batch Loads

```python
from google.cloud import bigquery

client = bigquery.Client()
table_ref = client.dataset("analytics").table("events")

# STREAMING INSERT (real-time, data available immediately):
errors = client.insert_rows_json(
    table_ref,
    [
        {
            "timestamp": "2024-06-15T14:30:00Z",
            "user_id": "user-123",
            "event_type": "purchase",
            "amount_cents": 5000,
        }
    ]
)
# Available for querying immediately after insert.
# Cost: $0.01 per 200MB of streamed data.

# BATCH LOAD (from GCS, CSV/JSON/Parquet, cheaper):
job_config = bigquery.LoadJobConfig(
    source_format=bigquery.SourceFormat.PARQUET,
    write_disposition=bigquery.WriteDisposition.WRITE_APPEND,
)
job = client.load_table_from_uri(
    "gs://my-bucket/events/*.parquet",
    table_ref,
    job_config=job_config,
)
job.result()  # wait for completion
# Free (no charge for batch loads from GCS)
# Available for querying after job completes (minutes to hours)
```

## 4. When to Use BigQuery

```
BIGQUERY IDEAL USE CASES:
  ✓ Ad-hoc analytics by data analysts (pay per query, no infrastructure)
  ✓ Reporting and dashboards (Looker, Data Studio, Tableau)
  ✓ Data warehouse for GCP-native workloads
  ✓ When you need SQL on data stored in GCS (external tables)
  ✓ ML training on large datasets (BigQuery ML)
  ✓ Low query frequency (pay-per-query is cost-effective at low usage)

WHEN BIGQUERY IS NOT IDEAL:
  ✗ Real-time analytics requiring sub-second latency (use ClickHouse/Druid)
  ✗ Very high query frequency (hundreds of TB/day: flat-rate pricing or ClickHouse)
  ✗ Heavy ETL transformations (use Spark or Dataflow instead)
  ✗ Non-GCP infrastructure (AWS → prefer Redshift or Athena)
  ✗ Row-level updates (BigQuery supports DML but expensive for OLTP patterns)
```

## 5. Senior Engineer Perspective

**The query cost surprise:**

`SELECT * FROM large_table` on a 10TB table = $50 in one query. Without `SELECT *` discipline and partition filters, BigQuery costs can explode. Always select only needed columns and always include date filters on partitioned tables. Run a cost estimate before executing expensive ad-hoc queries.

**BigQuery for RAG:**

BigQuery's vector search feature (2023) allows storing embeddings in BigQuery and running k-nearest-neighbor queries alongside SQL. For teams already using BigQuery as their analytics warehouse, this enables hybrid semantic + structured search without a separate vector database:

```sql
SELECT query.id, base.content, distance
FROM VECTOR_SEARCH(
    TABLE `project.dataset.embeddings`,
    'embedding_column',
    (SELECT id, embedding FROM `project.dataset.query_embeddings`),
    top_k => 10,
    distance_type => 'COSINE'
) AS result
JOIN `project.dataset.content` base ON result.base.id = base.id
```
