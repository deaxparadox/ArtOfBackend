## 1. What Is It? (Theory)

pgvector is a PostgreSQL extension that adds vector storage and similarity search to PostgreSQL. Instead of running a separate vector database, you store embeddings directly in your existing PostgreSQL database.

**Why this matters:**

For teams already running PostgreSQL, pgvector means zero additional infrastructure to deploy, zero new operational knowledge required, and the ability to join vector search with your regular data in a single SQL query. For many applications, pgvector is sufficient and dramatically simpler than deploying a dedicated vector database.

## 2. How It Actually Works

### Setup and Schema

```sql
-- Enable the extension (once per database)
CREATE EXTENSION IF NOT EXISTS vector;

-- Add an embedding column to your existing products table
ALTER TABLE products ADD COLUMN embedding vector(1536);
-- 1536 = dimensions of text-embedding-3-small

-- Create a vector index for fast approximate nearest-neighbor search
-- HNSW (best balance of build time and query performance):
CREATE INDEX ON products USING hnsw (embedding vector_cosine_ops);

-- IVFFlat (faster to build, slightly less accurate):
-- CREATE INDEX ON products USING ivfflat (embedding vector_cosine_ops)
-- WITH (lists = 100);  -- 100 = sqrt(n_rows) is a good starting point
```

### Generating and Storing Embeddings

```python
import asyncpg
from openai import AsyncOpenAI

openai = AsyncOpenAI()

async def embed_text(text: str) -> list[float]:
    response = await openai.embeddings.create(
        model="text-embedding-3-small",
        input=text,
    )
    return response.data[0].embedding

async def index_product(pool: asyncpg.Pool, product: dict):
    """Store product with its embedding."""
    # Combine relevant fields for rich embedding
    text_to_embed = f"{product['name']} {product['description']} {product['brand']}"
    embedding = await embed_text(text_to_embed)
    
    await pool.execute(
        "UPDATE products SET embedding = $1 WHERE id = $2",
        embedding,  # asyncpg handles list[float] → vector conversion
        product["id"],
    )

async def backfill_embeddings(pool: asyncpg.Pool):
    """Backfill embeddings for existing products."""
    products = await pool.fetch(
        "SELECT id, name, description, brand FROM products WHERE embedding IS NULL"
    )
    for product in products:
        await index_product(pool, product)
```

### Searching

```python
async def semantic_search(
    pool: asyncpg.Pool,
    query: str,
    limit: int = 20,
    category_filter: str | None = None,
) -> list[dict]:
    """Find semantically similar products."""
    
    # Embed the query
    query_embedding = await embed_text(query)
    
    # Build the query
    sql = """
        SELECT
            id,
            name,
            price_cents,
            category,
            1 - (embedding <=> $1::vector) AS similarity
        FROM products
        WHERE embedding IS NOT NULL
          AND ($2::text IS NULL OR category = $2)
        ORDER BY embedding <=> $1::vector  -- cosine distance (smaller = more similar)
        LIMIT $3
    """
    # <=> : cosine distance
    # <-> : Euclidean distance
    # <#> : negative inner product (for inner product similarity)
    
    rows = await pool.fetch(sql, query_embedding, category_filter, limit)
    
    return [
        {
            "id": row["id"],
            "name": row["name"],
            "price_cents": row["price_cents"],
            "similarity": round(float(row["similarity"]), 4),
        }
        for row in rows
    ]

# Example results:
# query: "comfortable shoes for standing all day"
# Results: [
#   {"name": "Birkenstock Boston Clogs", "similarity": 0.9234},
#   {"name": "Nike React Infinity Run", "similarity": 0.8912},
#   {"name": "New Balance 990v5", "similarity": 0.8756},
# ]
```

### Hybrid Search with pgvector

```sql
-- Combine vector similarity with full-text search in one query
SELECT
    p.id,
    p.name,
    p.price_cents,
    ts_rank(p.search_vector, plainto_tsquery($1)) AS text_score,
    1 - (p.embedding <=> $2::vector)             AS vector_score,
    -- Weighted combination
    0.3 * ts_rank(p.search_vector, plainto_tsquery($1)) +
    0.7 * (1 - (p.embedding <=> $2::vector))     AS hybrid_score
FROM products p
WHERE p.embedding IS NOT NULL
  AND (
    p.search_vector @@ plainto_tsquery($1)       -- text match
    OR (1 - (p.embedding <=> $2::vector)) > 0.7  -- OR semantically similar
  )
ORDER BY hybrid_score DESC
LIMIT 20;
-- Pass: $1 = query text, $2 = query embedding
```

## 3. Performance Considerations

```
HNSW INDEX PARAMETERS:
  m (default 16): connections per node. Higher = better recall, more memory.
    - Small datasets (<100k): m=8
    - General use: m=16
    - High recall needed: m=32
  
  ef_construction (default 64): search depth during build. Higher = slower build, better index.
    - Development: ef_construction=64
    - Production: ef_construction=128

QUERY PERFORMANCE:
  SET hnsw.ef_search = 40;  -- search depth at query time (default 40)
  -- Higher = better recall but slower. Tune per use case.

SCALING LIMITS:
  pgvector HNSW works well up to ~1-5M vectors.
  Beyond that: consider dedicated vector databases (Pinecone, Qdrant).
  
  Memory: each vector(1536) = 1536 × 4 bytes = 6.1KB
  1M vectors ≈ 6.1GB just for embeddings (plus index overhead ~2x)
  A 32GB PostgreSQL instance can handle ~2-3M vectors comfortably.
```

## 4. When pgvector vs. Dedicated Vector DB

```
USE pgvector WHEN:
  ✓ Already using PostgreSQL
  ✓ Dataset < 5M vectors
  ✓ Joins with relational data are important
  ✓ Operational simplicity (no new system to learn and manage)
  ✓ Neon, Supabase, or other managed PG already includes pgvector

USE DEDICATED VECTOR DB (Pinecone, Qdrant, Weaviate) WHEN:
  ✓ > 5M vectors with high query throughput required
  ✓ You need advanced filtering options pgvector lacks
  ✓ You need ANN algorithms with more control (IVF, PQ quantization)
  ✓ Multi-modal search (images + text + audio)
  ✓ The vector search is completely separate from your relational data
```

## 5. Senior Engineer Perspective

**The embedding freshness problem:**

Products change over time (price updates, description changes, new features). When the source data changes, the embedding becomes stale — it no longer accurately represents the document. Build a pipeline that re-embeds documents when they change significantly. For critical fields: use database triggers or Kafka CDC to detect changes and queue re-embedding jobs.

**Start with pgvector, migrate if needed:**

For most teams adding semantic search, pgvector is the right starting point. It's free, requires no new infrastructure, and handles millions of vectors well. If you outgrow it (more than a few million vectors, or sub-10ms latency requirements at scale), migrate to a dedicated vector database then. Don't prematurely optimize by deploying Pinecone on day one when pgvector would serve you fine for years.
