## 1. Why Distributed Tracing? (Theory)

In a monolith, a slow request is easy to debug: add timing logs, find the slow function. In a microservices architecture, a single user request might pass through 8 services. The user experiences 3 seconds of slowness. Which service caused it?

Without distributed tracing, you're correlating timestamps across 8 different log files. With distributed tracing, you see a single view of the entire request journey.

```
THE PROBLEM (8 services, one slow request):

Service A  B  C  D  E  F  G  H
Logs: [14:30:00.001] [14:30:00.050] [14:30:00.100] ... [14:30:03.000]

Without tracing: 
  Which service took the extra 2.9 seconds?
  Which service called which?
  Was it one slow service or accumulated latency?

With distributed tracing:
  ┌─────────────────────────────────────────────────────────────────┐
  │ Trace ID: abc-123-def                                           │
  │                                                                 │
  │ api-gateway    [==================================] 3000ms total │
  │   auth-svc     [=]                                 50ms         │
  │   order-svc    [================================] 2900ms ← slow!│
  │     postgres   [============================]     2800ms ← DB!  │
  │     SQL: SELECT * FROM orders WHERE user_id=?                   │
  │       (missing index on user_id!)                               │
  └─────────────────────────────────────────────────────────────────┘
  
Answer in seconds, not hours.
```

## 2. How Distributed Tracing Works

```
TRACING CONCEPTS:

TRACE:
  Represents a single end-to-end request through the system.
  Identified by a globally unique trace_id.
  Contains all spans for that request.

SPAN:
  One unit of work within a trace.
  Has: span_id, parent_span_id, trace_id, service name, operation name,
       start time, duration, tags (key-value), status (ok/error).
  
  Span hierarchy:
  trace_id: "abc-123"
  ├── span: api-gateway (parent_span_id: null)     50ms → 3000ms
  │   ├── span: auth-service (parent: api-gateway) 50ms → 100ms  
  │   └── span: order-service (parent: api-gateway) 100ms → 3000ms
  │       └── span: postgres (parent: order-service) 110ms → 2910ms

CONTEXT PROPAGATION:
  How does the trace_id and span_id travel between services?
  
  Service A calls Service B via HTTP.
  A adds trace context to headers:
    traceparent: 00-abc123def-parentspanid-01
  
  B reads the header, knows it's part of trace "abc123def".
  B creates a new span with the received parent span ID.
  B's spans appear as children of A's span in the trace.
```

## 3. OpenTelemetry: The Standard

OpenTelemetry (OTel) is the open standard for generating, collecting, and exporting telemetry data (traces, metrics, logs). It replaced vendor-specific SDKs (Datadog traces, Jaeger client, Zipkin client) with one unified API.

```python
# OpenTelemetry setup for Python + FastAPI

from opentelemetry import trace
from opentelemetry.sdk.trace import TracerProvider
from opentelemetry.sdk.trace.export import BatchSpanProcessor
from opentelemetry.exporter.otlp.proto.grpc.trace_exporter import OTLPSpanExporter
from opentelemetry.instrumentation.fastapi import FastAPIInstrumentor
from opentelemetry.instrumentation.sqlalchemy import SQLAlchemyInstrumentor
from opentelemetry.instrumentation.httpx import HTTPXClientInstrumentor

# STEP 1: Configure the tracer
provider = TracerProvider()
otlp_exporter = OTLPSpanExporter(
    endpoint="http://otel-collector:4317",  # OpenTelemetry Collector
)
provider.add_span_processor(BatchSpanProcessor(otlp_exporter))
trace.set_tracer_provider(provider)

# STEP 2: Auto-instrument common libraries
# These libraries automatically create spans for their operations
FastAPIInstrumentor.instrument_app(app)      # HTTP request spans
SQLAlchemyInstrumentor().instrument()         # SQL query spans
HTTPXClientInstrumentor().instrument()        # Outgoing HTTP call spans

# Now FastAPI creates spans automatically for every request.
# SQLAlchemy creates spans for every query.
# httpx creates spans for every outgoing call.
# All correlated via trace context propagation.

# STEP 3: Manual instrumentation for business operations
tracer = trace.get_tracer(__name__)

async def process_payment(order_id: str, amount: float):
    with tracer.start_as_current_span("process_payment") as span:
        span.set_attribute("order.id", order_id)
        span.set_attribute("payment.amount", amount)
        
        try:
            result = await charge_card(order_id, amount)
            span.set_attribute("payment.status", "success")
            return result
        except PaymentDeclinedException as e:
            span.set_status(trace.Status(trace.StatusCode.ERROR, str(e)))
            span.record_exception(e)
            raise
```

## 4. The OpenTelemetry Collector

```yaml
# otel-collector-config.yaml
# The collector receives telemetry, processes it, and exports to backends

receivers:
  otlp:
    protocols:
      grpc:
        endpoint: 0.0.0.0:4317
      http:
        endpoint: 0.0.0.0:4318

processors:
  batch:
    timeout: 1s
    send_batch_size: 1024
  
  # Add deployment metadata to all spans
  resource:
    attributes:
      - key: deployment.environment
        value: production
        action: upsert

exporters:
  # Send traces to Jaeger
  jaeger:
    endpoint: jaeger:14250
    tls:
      insecure: true
  
  # Also send to Datadog (same traces, two destinations)
  datadog:
    api:
      key: ${DD_API_KEY}

service:
  pipelines:
    traces:
      receivers: [otlp]
      processors: [batch, resource]
      exporters: [jaeger, datadog]
```

## 5. Context Propagation in Practice

```python
# How trace context crosses service boundaries

import httpx
from opentelemetry.propagate import inject, extract

# Service A calling Service B:
async def call_service_b(trace_data: dict) -> dict:
    headers = {}
    inject(headers)  # OTel adds traceparent header automatically
    # headers now: {"traceparent": "00-abc123-parentspanid-01"}
    
    async with httpx.AsyncClient() as client:
        response = await client.post(
            "http://service-b/api/process",
            json=trace_data,
            headers=headers,
        )
    return response.json()

# Service B extracting context:
# FastAPIInstrumentor does this automatically:
# It reads the traceparent header, extracts trace_id + parent_span_id,
# and creates a child span in the same trace.

# W3C TRACEPARENT FORMAT:
# traceparent: 00-{trace-id}-{parent-span-id}-{flags}
# traceparent: 00-4bf92f3577b34da6a3ce929d0e0e4736-00f067aa0ba902b7-01
#              ^^ version       ^^ 32 hex chars trace  ^^ 16 hex chars span
```

## 6. Senior Engineer Perspective

**OpenTelemetry is the best engineering bet you can make for observability:**

Instrument with OpenTelemetry once. The output can flow to any backend: Jaeger, Tempo, Datadog, New Relic, Dynatrace, Honeycomb. Switching observability backends no longer requires re-instrumenting your codebase. The 2023-2024 standardization on OTel is the most important shift in observability tooling in years.

**Sampling is not optional at scale:**

If you trace every request, at 10,000 requests/second you're sending 10,000 traces/second to your tracing backend. Storage and processing costs become prohibitive. Sampling strategy:

- **Head-based sampling**: decide at the start of a request whether to trace it (e.g., 1% of all requests). Simple but misses important slow/error requests.
- **Tail-based sampling**: buffer all spans, decide after the request completes (e.g., always keep error traces, sample normal ones at 0.1%). More expensive but keeps high-value traces.
