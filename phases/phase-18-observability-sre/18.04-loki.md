## 1. What Is Loki? (Theory)

Loki (Grafana Labs, 2018) is a log aggregation system designed to be cost-efficient and simple to operate. Its key design choice: **don't index log content, only index log metadata (labels)**.

**Loki vs. Elasticsearch — the key trade-off:**

```
ELASTICSEARCH:
  Indexes EVERY WORD in every log line.
  A log line "User 123 failed payment for $99.99" creates index entries
  for "User", "123", "failed", "payment", "$99.99"
  
  Result:
  - Very fast full-text search on any word or field
  - Very expensive: index storage is often 2-5x the raw log size
  - Complex to operate (JVM tuning, shard management)

LOKI:
  Indexes ONLY LABELS: {service="payment", level="error", region="us-east-1"}
  Does NOT index log content.
  
  Result:
  - Full-text search requires scanning log lines (slower on large datasets)
  - Very cheap: storage is close to compressed raw log size
  - Simple to operate: S3-compatible object storage as backend
  - Trade: slower arbitrary text search, faster and cheaper label-based queries
```

## 2. Loki Architecture

```
LOKI COMPONENTS:

PROMTAIL (agent):
  Runs on each server, tails log files.
  Adds labels (service name, environment, host).
  Pushes to Loki via HTTP.

LOKI (storage engine):
  Receives log streams.
  Compresses and stores chunks in S3 / GCS / local disk.
  Maintains a small index of labels only.

GRAFANA (visualization):
  Query Loki using LogQL language.
  Correlate logs with Prometheus metrics in the same dashboard.

FLOW:
  Application → Promtail → Loki → Grafana
```

```yaml
  - url: http://loki:3100/loki/api/v1/push

scrape_configs:
  # Scrape Docker container logs
  - job_name: containers
    docker_sd_configs:
      - host: unix:///var/run/docker.sock
        refresh_interval: 5s
    relabel_configs:
      - source_labels: [__meta_docker_container_name]
        target_label: container
      - source_labels: [__meta_docker_container_label_com_docker_compose_service]
        target_label: service

  # Scrape log files directly
  - job_name: system
    static_configs:
      - targets: [localhost]
        labels:
          job: varlogs
          __path__: /var/log/*.log
```

## 4. LogQL: Querying Loki

LogQL is Loki's query language. It has two parts: a log stream selector (labels) and an optional filter/expression.

```
LOGQL SYNTAX:
  {label_selector} | filter_expression

LABEL SELECTORS:
  {service="payment-service"}                    → all logs from payment-service
  {service="payment-service", level="error"}     → errors from payment-service
  {service=~"payment.*"}                         → regex: all payment* services
  {service!="health-checker"}                    → exclude health checker logs

LOG FILTERS (applied to log content):
  {service="payment-service"} |= "payment_failed"     → contains this text
  {service="payment-service"} != "health_check"       → does NOT contain
  {service="payment-service"} |~ "error.*timeout"     → regex match

PARSING JSON LOGS:
  {service="payment-service"} | json | level="error"
  (parse JSON → filter by level field)
  
  {service="payment-service"} | json | amount > 1000
  (parse JSON → filter by amount field)

METRIC QUERIES (convert logs to metrics):
  rate({service="payment-service", level="error"}[5m])
  → errors per second over 5-minute window

  count_over_time({service="payment-service"} |= "payment_failed"[1h])
  → total failed payments in last hour

PRACTICAL EXAMPLES:
  # All errors in last 15 minutes, payment service
  {service="payment-service"} | json | level = "error"
  
  # Find all logs for a specific user
  {service=~".*"} |= "user_id\":\"u-123\""
  
  # Find all logs for a trace
  {service=~".*"} |= "trace_id\":\"abc-def-123\""
```

## 5. Loki with Docker Compose (Local Setup)

```yaml
# docker-compose.yml: local Loki + Grafana stack

version: '3'
services:
  loki:
    image: grafana/loki:2.9.0
    ports:
      - "3100:3100"
    volumes:
      - loki-data:/loki
    command: -config.file=/etc/loki/local-config.yaml

  promtail:
    image: grafana/promtail:2.9.0
    volumes:
      - /var/log:/var/log
      - /var/run/docker.sock:/var/run/docker.sock
      - ./promtail.yml:/etc/promtail/config.yml
    command: -config.file=/etc/promtail/config.yml

  grafana:
    image: grafana/grafana:10.0.0
    ports:
      - "3000:3000"
    environment:
      - GF_AUTH_ANONYMOUS_ENABLED=true
      - GF_AUTH_ANONYMOUS_ORG_ROLE=Admin
    volumes:
      - grafana-data:/var/lib/grafana

volumes:
  loki-data:
  grafana-data:
```

## 6. Loki vs. Elasticsearch: When to Choose Each

```
CHOOSE LOKI WHEN:
  ✓ Budget-sensitive: Loki + S3 storage is 5-10x cheaper than Elasticsearch
  ✓ Already using Grafana + Prometheus (tight integration)
  ✓ Log queries are primarily label-based (service, level, environment)
  ✓ Small to medium log volumes (< 10 GB/day)
  ✓ Simple operational requirements (Loki is much easier to run)

CHOOSE ELASTICSEARCH WHEN:
  ✓ Need fast arbitrary full-text search on log content
  ✓ Complex log parsing and transformation (Logstash pipelines)
  ✓ Large enterprise with existing Kibana investment
  ✓ Need complex aggregations on log field values
  ✓ Full-text search is performance-critical

THE TREND (2024):
  Many teams that built on Elasticsearch are migrating to Loki + S3.
  The cost savings (especially at scale) are significant.
  Loki's full-text search has improved enough for most use cases.
  Grafana integration is excellent if you're already using Grafana.
```

## 7. Senior Engineer Perspective

**Loki's "don't index everything" design is a brilliant engineering trade-off:**

Elasticsearch's power comes from indexing every word — but that indexing costs significant CPU, RAM, and disk. Loki bet that most log queries start with "show me logs from service X with level=error" — which only requires label filtering. Full-text search within those results is then a scan of a small result set, not a billion-document index.

For most production debugging workflows, this trade is correct. You almost always narrow by label first. The scan for arbitrary text within a narrow label set is fast enough.

Where Elasticsearch wins: compliance/security teams doing broad searches across all services for specific strings (e.g., "find every log that mentions a specific email address across all services"). Loki would require scanning everything.
