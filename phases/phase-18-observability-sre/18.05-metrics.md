## 1. What Are Metrics? (Theory)

Metrics are numerical measurements of your system sampled over time. Unlike logs (discrete events) and traces (request journeys), metrics answer questions like:
- How many requests per second?
- What's the p99 latency right now?
- How full is the database connection pool?
- How much memory is being used?

Metrics are optimized for:
- **Aggregation**: "average error rate across all instances"
- **Trending**: "memory growing over the past week"
- **Alerting**: "alert if p99 > 500ms for 5 minutes"
- **Dashboards**: live visualization of system health

## 2. The Four Metric Types

### Type 1: Counter

A counter only increases. It counts events. It resets to zero when the process restarts.

```python
from prometheus_client import Counter

# A counter for HTTP requests
http_requests_total = Counter(
    'http_requests_total',
    'Total number of HTTP requests',
    ['method', 'path', 'status_code'],   # labels = dimensions
)

# Increment when a request is handled
http_requests_total.labels(
    method='GET',
    path='/api/users',
    status_code='200',
).inc()

# QUERYING (PromQL):
# rate(http_requests_total[5m])
# → requests per second over a 5-minute window
# (rate handles counter resets automatically)
```

```
COUNTERS IN PRACTICE:
  http_requests_total
  http_errors_total
  payments_processed_total
  bytes_sent_total
  user_signups_total
  
WHY COUNTERS, NOT GAUGES FOR EVENT COUNTS:
  A gauge that tracks "requests this minute" would need resetting.
  Who resets it? What if the process crashes mid-minute?
  A counter with rate() avoids all these problems.
```

### Type 2: Gauge

A gauge can go up or down. It represents a current value.

```python
from prometheus_client import Gauge

# Active connections (goes up and down)
active_connections = Gauge(
    'active_connections',
    'Number of currently active connections',
)

active_connections.inc()   # connection opened
active_connections.dec()   # connection closed
active_connections.set(42) # set to specific value

# Other examples:
memory_usage = Gauge('memory_usage_bytes', 'Current memory usage')
queue_depth = Gauge('queue_depth', 'Messages waiting in queue')
cache_hit_ratio = Gauge('cache_hit_ratio', 'Cache hit ratio (0-1)')

# QUERYING (PromQL):
# active_connections                    → current value
# max_over_time(active_connections[1h]) → max over last hour
```

### Type 3: Histogram

A histogram measures the distribution of a value. You pre-define buckets; events are counted in each bucket.

```python
from prometheus_client import Histogram

# Request duration histogram
request_duration = Histogram(
    'request_duration_seconds',
    'Request duration in seconds',
    buckets=[.005, .01, .025, .05, .1, .25, .5, 1, 2.5, 5, 10],
    # buckets: 5ms, 10ms, 25ms, 50ms, 100ms, 250ms, 500ms, 1s, 2.5s, 5s, 10s
)

# Measure a request (context manager):
with request_duration.time():
    result = handle_request()

# Or manually:
import time
start = time.time()
result = handle_request()
request_duration.observe(time.time() - start)

# QUERYING (PromQL):
# histogram_quantile(0.99, rate(request_duration_seconds_bucket[5m]))
# → p99 latency over last 5 minutes (THE most important metric)
```

```
HOW HISTOGRAMS WORK INTERNALLY:

Prometheus stores three time series per histogram:
  request_duration_seconds_bucket{le="0.005"} = 1234  → ≤5ms requests
  request_duration_seconds_bucket{le="0.01"}  = 3456  → ≤10ms requests
  request_duration_seconds_bucket{le="0.025"} = 7890  → ≤25ms requests
  ...
  request_duration_seconds_bucket{le="+Inf"}  = 12000 → total requests
  request_duration_seconds_count = 12000              → same as +Inf bucket
  request_duration_seconds_sum = 1234.5               → total duration

From these, Prometheus can estimate any percentile using interpolation.
Note: these are ESTIMATES — bucket boundaries affect precision.
```

### Type 4: Summary

A summary also measures distributions, but computes quantiles on the client side (not Prometheus server).

```python
from prometheus_client import Summary

request_duration = Summary(
    'request_duration_seconds',
    'Request duration',
)

with request_duration.time():
    handle_request()

# Stores: request_duration_seconds_count, request_duration_seconds_sum
# Pre-computed quantiles at collection time
```

```
HISTOGRAM vs. SUMMARY:

HISTOGRAM:
  ✓ Quantiles computed server-side (Prometheus does the math)
  ✓ Multiple instances can be aggregated across pods
  ✓ More flexible (compute any percentile after the fact)
  ✗ Less precise (limited by bucket boundaries)
  → USE FOR: request latency, response size
  → ALWAYS prefer histogram for latency monitoring

SUMMARY:
  ✗ Quantiles computed client-side (each instance computes its own)
  ✗ Cannot aggregate across multiple instances
  ✓ More precise (exact quantiles, not estimates)
  → USE FOR: situations where precision matters and you have one instance
```

## 3. Labels: The Power of Dimensions

Labels add dimensions to metrics. They let you slice the same metric by different properties.

```python
# BAD: no labels
requests_total = Counter('requests_total', 'Total requests')
requests_total.inc()  # can't tell which endpoint or status

# GOOD: labels add dimensions
requests = Counter(
    'http_requests_total',
    'HTTP requests',
    ['method', 'endpoint', 'status'],
)
requests.labels(method='GET', endpoint='/api/users', status='200').inc()
requests.labels(method='POST', endpoint='/api/orders', status='500').inc()

# QUERYING:
# sum(rate(http_requests_total{status=~"5.."}[5m])) by (endpoint)
# → error rate per endpoint
# 
# rate(http_requests_total{endpoint="/api/orders"}[5m])
# → request rate for orders endpoint specifically
```

```
CARDINALITY WARNING:
  High-cardinality labels WILL break Prometheus.
  
  BAD: labels={user_id="u-12345"}  → millions of unique time series
  BAD: labels={trace_id="abc123"}  → one time series per request
  
  Prometheus stores each unique label combination as a separate time series.
  1M users × 10 endpoints = 10M time series = out of memory.
  
  RULE: labels should have LOW cardinality (< 1000 unique values).
  Service name: fine (dozens of services)
  HTTP method: fine (GET, POST, PUT, DELETE)
  Status code: fine (200, 400, 404, 500)
  User ID: NEVER
  Request ID: NEVER
```

## 4. The Four Golden Signals

Google's SRE book defines the four metrics every service should have:

```
1. LATENCY: how long requests take
   Metric: histogram_quantile(0.99, rate(request_duration_seconds_bucket[5m]))
   Alert: p99 > 500ms for 5 minutes

2. TRAFFIC: how much demand the system is receiving
   Metric: rate(http_requests_total[5m])
   Use: capacity planning, normal baseline, anomaly detection

3. ERRORS: how often requests fail
   Metric: rate(http_errors_total[5m]) / rate(http_requests_total[5m])
   Alert: error rate > 1% for 5 minutes

4. SATURATION: how "full" the system is
   Metric: cpu_usage, memory_usage, connection_pool_usage
   Alert: CPU > 85% for 10 minutes
```

## 5. Senior Engineer Perspective

**Instrument for alerts, not for archives:**

The most common mistake is instrumenting everything without thinking about what you'll alert on or dashboard. Metrics have cost (scraping, storage, cardinality). Start with the Four Golden Signals — latency, traffic, errors, saturation — for every service. Everything else is secondary.

**Histograms over gauges for latency:**

If you have to pick one metric type for request latency, always pick histogram. The ability to compute p99 in Prometheus queries — and see how that p99 changes over time — is irreplaceable for SLO tracking. A gauge showing "average latency" is nearly useless: averages hide long-tail outliers, which are exactly the requests breaking user experience.
