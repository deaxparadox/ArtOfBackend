## 1. What Is Synthetic Monitoring? (Theory)

Synthetic monitoring is the practice of simulating user behavior against your production system to detect problems before real users do. A "synthetic check" is an automated script that runs from external locations, mimics a user action, and alerts if it fails or is slow.

**The gap synthetic monitoring fills:**

```
YOUR INTERNAL METRICS:
  Prometheus: "p99 latency = 150ms, error rate = 0.2%"
  Looks healthy. But...
  
  Is the login flow actually working right now?
  Can users in Germany access the site?
  Is the payment page rendering without JavaScript errors?
  Is your SSL certificate about to expire?
  
  Your internal metrics measure server-side health.
  Synthetic monitoring measures USER-VISIBLE health.
  
SCENARIO:
  Your server returns 200 OK with correct latency.
  But the page contains a broken JavaScript reference.
  Users see a blank screen.
  Your metrics look perfectly healthy.
  
  A synthetic check that actually clicks "Login" would catch this in 60 seconds.
  Without it: you find out when users complain.
```

## 2. Uptime Kuma: Simple, Self-Hosted Monitoring

Uptime Kuma is a lightweight, open-source monitoring tool. Install it once, point it at your services, and get beautiful status dashboards.

```yaml
# docker-compose.yml

version: '3'
services:
  uptime-kuma:
    image: louislam/uptime-kuma:1
    ports:
      - "3001:3001"
    volumes:
      - uptime-kuma-data:/app/data
    restart: unless-stopped

volumes:
  uptime-kuma-data:
```

```
UPTIME KUMA MONITOR TYPES:

HTTP(S): Check URL, verify status code and response
  - URL: https://api.yourcompany.com/health
  - Expected status: 200
  - Check every: 60 seconds
  - Keyword check: response body must contain "ok"

TCP PORT: Is the port open?
  - Host: database.internal, Port: 5432
  
PING: Is the host reachable?
  - Checks ICMP, detects network issues

DNS: Is the DNS record correct?
  - Verify A record returns expected IP

SSL CERTIFICATE: Is cert valid and not expiring soon?
  - Alerts when certificate expires within 30 days
  
DOCKER CONTAINER: Is the container running?
  - Monitors specific containers on the same Docker host

NOTIFICATION CHANNELS:
  Slack, Discord, Telegram, PagerDuty, email, webhook
  Configure: alert after 2 consecutive failures (reduces false positives)

PUBLIC STATUS PAGE:
  Uptime Kuma generates a public status page (like statuspage.io)
  Users can see historical uptime, current status
  You can display incidents and maintenance windows
```

## 3. Checkly: API and Browser Testing at Scale

Checkly is a commercial synthetic monitoring platform designed for more sophisticated checks — including actual browser automation.

```javascript
// Checkly: API check (runs from global locations every minute)

const { expect } = require('@playwright/test');

// This check runs against your production API from:
// us-east-1, eu-west-1, ap-southeast-1, etc.

// API check: test the login endpoint
const response = await fetch('https://api.yourcompany.com/auth/login', {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify({
        email: process.env.CHECKLY_TEST_EMAIL,
        password: process.env.CHECKLY_TEST_PASSWORD,
    }),
});

expect(response.status).toBe(200);
const body = await response.json();
expect(body.token).toBeTruthy();
expect(body.user.email).toBe(process.env.CHECKLY_TEST_EMAIL);

// This runs every 60 seconds from 5 global locations.
// If it fails 2 consecutive times from 2+ locations: alert fires.
```

```javascript
// Checkly: Browser check (Playwright test against production)

const { chromium } = require('playwright');

// This opens a real Chrome browser against your production site
// Tests the actual user-visible checkout flow

const browser = await chromium.launch();
const page = await browser.newPage();

// Simulate: user adds product to cart and checks out
await page.goto('https://yourcompany.com/products/widget-pro');
await page.click('[data-testid="add-to-cart"]');
await page.click('[data-testid="checkout-button"]');

// Verify cart page loaded correctly
await expect(page.locator('[data-testid="cart-total"]')).toBeVisible();
const total = await page.locator('[data-testid="cart-total"]').textContent();
expect(parseFloat(total.replace('$', ''))).toBeGreaterThan(0);

// Time the full flow
const startTime = Date.now();
await page.click('[data-testid="proceed-to-payment"]');
await page.waitForURL('**/checkout/payment');
const timeToPayment = Date.now() - startTime;

// Alert if checkout navigation takes > 3 seconds
expect(timeToPayment).toBeLessThan(3000);

await browser.close();
```

## 4. Multi-Region Synthetic Monitoring

```
WHY MULTI-REGION MATTERS:

Single-region check (e.g., only from us-east-1):
  - Your user in Tokyo can't access the site
  - Your check from us-east-1 succeeds
  - You don't know there's a problem
  
Multi-region checks:
  - Check from: us-east-1, eu-west-1, ap-southeast-1
  - Tokyo user's region covered
  - CDN misconfiguration in Asia detected immediately

BEST PRACTICE: Alert only if failing from 2+ regions
  One region failing = likely a regional CDN issue (may resolve quickly)
  Two regions failing = problem likely on your origin servers
  
  This dramatically reduces false positives from regional network hiccups.
```

## 5. What to Monitor with Synthetic Checks

```
CRITICAL USER JOURNEYS (always monitor):
  ✓ Can users log in?
  ✓ Can users sign up?
  ✓ Does the main product page load?
  ✓ Can users complete checkout/payment?
  ✓ Does the API health endpoint return healthy?
  
INFRASTRUCTURE BASICS:
  ✓ SSL certificate expiry (alert at 30 days remaining)
  ✓ DNS resolution (correct IP returned)
  ✓ Main public-facing domains up
  
SECONDARY CHECKS:
  ✓ Email delivery (sign up, wait for welcome email via IMAP check)
  ✓ Search functionality
  ✓ Key API endpoints

DON'T MONITOR:
  ✗ Every endpoint (too noisy, expensive)
  ✗ Endpoints that require complex auth state
  ✗ Very low-traffic features (false positives from natural variation)
```

## 6. Senior Engineer Perspective

**Synthetic monitoring is the closest thing to being a user:**

Every other monitoring signal is indirect: CPU usage, error rates, trace durations — these measure the machine, not the experience. A synthetic check that actually clicks "Add to Cart" and verifies the total updates correctly measures what the user sees. That's uniquely valuable.

The most common production incident that synthetic monitoring catches first: frontend deploy introduces a JavaScript error that crashes the page. Server-side metrics look fine (returning 200 OK with correct HTML), but the page is broken. No internal metric catches this. A Playwright browser check catches it within 60 seconds.

**Start simple, add complexity incrementally:**

Uptime Kuma on a $5 VPS is dramatically better than no synthetic monitoring. Start there. Once you've experienced the value — "we found out our staging login was broken before users did" — you'll naturally want more sophisticated checks like Checkly's browser tests. Don't let perfect be the enemy of good.
