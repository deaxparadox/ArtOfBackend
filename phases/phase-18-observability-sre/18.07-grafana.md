## 1. What Is Grafana? (Theory)

Grafana is a visualization platform that connects to any data source (Prometheus, Loki, Elasticsearch, PostgreSQL, InfluxDB, CloudWatch...) and lets you build dashboards and set up alerts.

Its power: it's source-agnostic. In a single dashboard, you can show:
- Prometheus metrics (request rate, latency)
- Loki logs (error log tail)
- PostgreSQL query results (business metrics)
- S3 cost data

Everything in one pane of glass, correlating time ranges automatically.

## 2. Data Sources and Panels

```
GRAFANA ARCHITECTURE:

DATA SOURCES (plugins that connect to backends):
  Prometheus   → metrics (most common)
  Loki         → logs
  Elasticsearch → logs + metrics
  PostgreSQL   → SQL query results
  CloudWatch   → AWS metrics
  InfluxDB     → time-series
  +100 more community plugins

PANELS (visualizations):
  Time series  → line chart over time (latency, throughput)
  Stat         → single number with color threshold (current error rate)
  Gauge        → dial chart (current CPU usage %)
  Bar chart    → comparison across categories
  Table        → raw data, sortable
  Logs         → streaming log tail from Loki
  Heatmap      → latency distribution over time
  Alert list   → currently firing alerts
```

## 4. Grafana Alerting

Grafana can alert based on any panel query — across any data source.

```yaml
# Grafana alert (defined in UI or as code with Grafana as code)
# This alert fires when p99 latency > 500ms for 5 minutes

Alert: "High p99 Latency"

Query (PromQL):
  histogram_quantile(
    0.99,
    sum by (le) (rate(request_duration_seconds_bucket[5m]))
  )

Condition:
  IS ABOVE 0.5   (500ms = 0.5 seconds)

For: 5m          (must be above threshold for 5 minutes before firing)

Labels:
  severity: critical
  team: backend

Notification policy:
  severity=critical → PagerDuty (pages on-call engineer)
  severity=warning  → Slack #alerts channel
```

```
GRAFANA ALERT STATES:
  Normal:    query below threshold
  Pending:   query above threshold, waiting for "For" duration
  Firing:    threshold exceeded for required duration → notification sent
  Resolved:  metric went back below threshold → resolution notification
```

## 5. Dashboard as Code (Grafana Provisioning)

```yaml
# grafana/provisioning/dashboards/dashboard.yml
# Grafana loads this on startup — no clicking in the UI needed

apiVersion: 1
providers:
  - name: 'default'
    folder: 'Services'
    type: file
    options:
      path: /var/lib/grafana/dashboards
```

```json
// grafana/dashboards/payment-service.json (abbreviated)
{
  "title": "Payment Service",
  "uid": "payment-service",
  "panels": [
    {
      "title": "Request Rate",
      "type": "timeseries",
      "targets": [
        {
          "expr": "rate(http_requests_total{service=\"payment-service\"}[5m])",
          "legendFormat": "{{ method }} {{ endpoint }}"
        }
      ],
      "fieldConfig": {
        "defaults": {
          "unit": "reqps",
          "thresholds": {
            "steps": [
              {"color": "green", "value": 0},
              {"color": "yellow", "value": 1000},
              {"color": "red", "value": 2000}
            ]
          }
        }
      }
    }
  ]
}
```

## 6. The Grafana Ecosystem

```
GRAFANA STACK (all maintained by Grafana Labs):

Grafana      → visualization, alerting
Prometheus   → metrics storage and query (not made by Grafana Labs, but tight integration)
Loki         → log aggregation (made by Grafana Labs)
Tempo        → distributed tracing backend (made by Grafana Labs)
Mimir        → scalable Prometheus (long-term storage)
Pyroscope    → continuous profiling

This stack is often called the "LGTM stack":
Loki + Grafana + Tempo + Mimir

Together they provide all three pillars of observability:
  Logs:    Loki
  Metrics: Prometheus/Mimir
  Traces:  Tempo
All visualized in Grafana with one unified query language (LogQL/PromQL/TraceQL).
```

## 7. Senior Engineer Perspective

**The dashboard should tell a story:**

A useful Grafana dashboard answers: "Is this service healthy right now?" in under 10 seconds. The first row should always be big stat panels — request rate, error rate, p99 latency. If you need to scroll to find out if there's a problem, the dashboard isn't serving its purpose.

**Template variables for reusability:**

Create dashboards with template variables (e.g., `$service`) so one dashboard template works for 20 services. Instead of building and maintaining 20 dashboards, you build one and switch the variable. This is also how you create "drill-down" dashboards: a top-level dashboard shows all services; clicking on a service variable switches the dashboard to focus on just that service.
