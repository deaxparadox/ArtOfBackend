## 1. What Is a Tracing Backend? (Theory)

After your application generates trace data (using OpenTelemetry or another SDK), that data needs to go somewhere for storage and querying. Tracing backends are the databases and query UIs for trace data.

Three main open-source options:
- **Jaeger** (Uber, 2017): the original open-source distributed tracer
- **Grafana Tempo** (Grafana Labs, 2020): cost-optimized, S3-backed
- **Zipkin** (Twitter, 2012): the oldest, now mostly legacy

---

## 2. Jaeger

Jaeger (Uber Engineering, 2017) was built to solve Uber's distributed tracing needs across hundreds of microservices.

```
JAEGER ARCHITECTURE:

Applications → Jaeger Agent → Jaeger Collector → Storage
                    ↓                                ↓
              (runs per host,          Elasticsearch / Cassandra
               buffers spans)
                                        ↓
                                   Jaeger Query
                                   (web UI, API)

JAEGER STORAGE BACKENDS:
  In-memory (development only): no persistence
  Cassandra: original production backend, good for high write throughput
  Elasticsearch: most common today, good query capabilities
  Badger: embedded key-value store for single-node setups
```

```yaml
# docker-compose.yml: minimal Jaeger setup (all-in-one for development)

version: '3'
services:
  jaeger:
    image: jaegertracing/all-in-one:1.55
    ports:
      - "16686:16686"   # Jaeger UI
      - "14250:14250"   # gRPC collector (for OTel exporter)
      - "4317:4317"     # OTLP gRPC (OpenTelemetry Protocol)
      - "4318:4318"     # OTLP HTTP
    environment:
      - COLLECTOR_OTLP_ENABLED=true
```

```
JAEGER UI FEATURES:
  - Search traces by: service, operation, tags, duration, time range
  - Trace timeline: horizontal bar chart showing span durations
  - Service dependencies: auto-generated graph of service calls
  - Compare traces: side-by-side comparison of two traces
  
JAEGER SEARCH EXAMPLES:
  Service: order-service
  Operation: POST /api/orders
  Tags: error=true
  Duration: >500ms
  Time range: last 1 hour
  → Find all slow or failed order creation requests
```

## 3. Grafana Tempo

Tempo (Grafana Labs, 2020) is designed for one thing: store and retrieve traces as cheaply as possible.

**Tempo's key innovation:** it only stores traces, nothing else. It uses S3-compatible object storage (instead of Elasticsearch/Cassandra), making it dramatically cheaper at scale.

```
TEMPO ARCHITECTURE:

Applications → OpenTelemetry Collector → Tempo
                                              ↓
                                     S3 / GCS / Azure Blob
                                              ↓
                                     Grafana (visualization)
                                     TraceQL queries

TEMPO ADVANTAGES:
  ✓ S3 storage: 1/10th the cost of Elasticsearch
  ✓ Simple to operate: no Elasticsearch cluster to manage
  ✓ Deep Grafana integration: correlate metrics, logs, traces in one UI
  ✓ Exemplars: Prometheus metrics can include trace IDs
    (click a spike in your error rate → jump directly to a trace from that time)

TEMPO LIMITATION:
  ✗ No built-in trace search without Tempo Search (adds an index)
    Without search enabled: query by trace ID only
    With Tempo Search: query by service, duration, tags (uses more storage)
```

```yaml
# tempo.yml — production Tempo configuration

server:
  http_listen_port: 3200

distributor:
  receivers:
    otlp:
      protocols:
        grpc:
          endpoint: 0.0.0.0:4317
        http:
          endpoint: 0.0.0.0:4318

ingester:
  trace_idle_period: 10s
  max_block_duration: 5m

storage:
  trace:
    backend: s3
    s3:
      bucket: my-tempo-traces
      endpoint: s3.us-east-1.amazonaws.com
    wal:
      path: /var/tempo/wal
    local:
      path: /var/tempo/blocks

query_frontend:
  search:
    duration_slo: 5s
    throughput_bytes_slo: 1.073741824e+09
```

## 4. Zipkin

Zipkin (Twitter, 2012) was one of the first distributed tracing systems. It's simpler and lighter than Jaeger or Tempo but lacks many modern features.

```
ZIPKIN STATUS (2024):
  Still maintained, still widely deployed (especially in Java ecosystems)
  Spring Boot auto-configures Zipkin out of the box
  Lacks: advanced query, service dependency graphs
  
  WHEN TO USE:
  ✓ Already running Zipkin and it's working fine
  ✓ Spring Boot applications where auto-configuration is valuable
  ✓ Simplicity is a priority (fewer components than Jaeger)
  
  WHEN TO CHOOSE SOMETHING ELSE:
  ✗ New project (choose Tempo for cost, Jaeger for features)
  ✗ Need advanced trace querying
  ✗ Large scale (Zipkin scales less well than alternatives)
```

## 5. Choosing Your Tracing Backend

```
COMPARISON:
                        JAEGER          TEMPO           ZIPKIN
────────────────────────────────────────────────────────────────
STORAGE                 ES/Cassandra    S3 (cheap!)     MySQL/ES
COST                    Medium          Low             Low
QUERY CAPABILITIES      Excellent       Good            Basic
GRAFANA INTEGRATION     Good            Excellent       Good
KUBERNETES NATIVE       Good            Excellent       OK
COMMUNITY               Large           Growing         Declining
COMPLEXITY              Medium          Low             Low

RECOMMENDATION:
  New project + already using Grafana: TEMPO
  New project + want best query/UI: JAEGER
  Spring Boot shops with existing setup: ZIPKIN (don't change what works)
  Enterprise with Datadog/NR: use their built-in tracing (not a separate tool)
```

## 6. Exemplars: Connecting Metrics to Traces

One of the most powerful features in modern observability is **exemplars** — a way to embed trace IDs directly in Prometheus metrics. This creates a direct link from a metrics anomaly to a specific trace.

```python
# Prometheus exemplars: attach a trace ID to a histogram observation

from prometheus_client import Histogram
from opentelemetry import trace

REQUEST_LATENCY = Histogram(
    'request_duration_seconds',
    'HTTP request duration',
    buckets=[.005, .01, .025, .05, .1, .25, .5, 1, 2.5, 5],
)

async def handle_request():
    with REQUEST_LATENCY.time():
        result = await process()
    
    # Get current trace ID and add as exemplar
    current_span = trace.get_current_span()
    span_context = current_span.get_span_context()
    if span_context.is_valid:
        trace_id = format(span_context.trace_id, '032x')
        REQUEST_LATENCY.observe(
            duration,
            exemplar={"trace_id": trace_id},
        )

# IN GRAFANA:
# You see a spike in p99 latency on your dashboard.
# On the histogram panel, you click the spike.
# Grafana shows exemplar data points — data points with trace IDs.
# You click one exemplar → Grafana opens Tempo → you see the exact trace.
# 
# From "something is slow" to "this specific request was slow, here's why"
# in 3 clicks. No manual searching.
```

## 7. Senior Engineer Perspective

**Tempo + Grafana is the best value for new projects:**

For teams already using Grafana + Prometheus, adding Tempo is almost free: it's another Grafana Labs product, the integration is seamless, and S3 storage makes the cost negligible. The exemplars feature creates the most valuable observability correlation: metrics → traces with one click.

Jaeger is better if you need the most powerful trace query capabilities or you're not already in the Grafana ecosystem. Its UI and search are more mature.
