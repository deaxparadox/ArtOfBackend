## 1. What Is Datadog? (Theory)

Datadog is the dominant commercial observability platform. Unlike the open-source Prometheus + Grafana + Loki stack, Datadog is a fully managed SaaS product: you install an agent, and everything flows to Datadog's cloud platform. No infrastructure to run, no dashboards to build from scratch.

**The Datadog pitch:**

```
OPEN-SOURCE STACK (Prometheus + Grafana + Loki + Jaeger):
  ✓ Free (you pay only for infrastructure to run it)
  ✓ Full control
  ✗ You operate it: upgrades, disk management, scaling
  ✗ Integration requires manual wiring
  ✗ Alerting and dashboards require manual setup
  ✗ Distributed tracing requires separate tool (Jaeger/Tempo)

DATADOG:
  ✓ Everything in one place: metrics + logs + traces + profiling
  ✓ Auto-discovery: detects running services, auto-instruments many
  ✓ Ready-made dashboards for 500+ technologies
  ✓ Powerful correlation: click a spike in metrics → see traces for that period
  ✓ AI-powered anomaly detection
  ✗ Expensive: $15-23/host/month for infra, more for APM and logs
  ✗ Data leaves your infrastructure (compliance consideration)
  ✗ Vendor lock-in
```

## 2. The Datadog Agent

The Datadog agent is a daemon that runs on each host and collects everything:

```bash
# Install Datadog agent (Linux)
DD_AGENT_MAJOR_VERSION=7 DD_API_KEY=<your_api_key> \
    bash -c "$(curl -L https://s3.amazonaws.com/dd-agent-testing/scripts/install_script.sh)"

# The agent auto-discovers:
# - Running processes (nginx, postgres, redis, etc.)
# - Kubernetes pods and services
# - Docker containers
# - JVM metrics (JMX)
# - System metrics (CPU, memory, disk, network)

# Enable specific integrations in /etc/datadog-agent/conf.d/
# Example: postgres.d/conf.yaml
init_config:

instances:
  - dbm: true
    host: localhost
    port: 5432
    username: datadog
    password: <password>
    dbname: postgres
```

## 3. APM (Application Performance Monitoring)

Datadog APM provides distributed tracing with auto-instrumentation — you add one library and it automatically instruments all HTTP calls, database queries, cache operations.

```python
# Python APM setup (one file, then auto-instrumentation kicks in)

# 1. Install
# pip install ddtrace

# 2. Start application with ddtrace-run
# ddtrace-run python app.py
# That's it — ddtrace patches all common libraries automatically:
# requests, httpx, psycopg2, redis, celery, fastapi, flask, django...

# OR: manual instrumentation for custom operations
from ddtrace import tracer

@tracer.wrap(service="payment-service", resource="process_payment")
def process_payment(order_id: str, amount: float):
    # This function appears as a span in Datadog APM
    with tracer.trace("payment.validate") as span:
        span.set_tag("order_id", order_id)
        span.set_tag("amount", amount)
        validate_payment(amount)
    
    charge_result = charge_card(order_id, amount)
    
    if not charge_result.success:
        tracer.current_span().set_error_info(
            error_type="PaymentDeclined",
            error_msg=charge_result.error,
        )
        raise PaymentDeclinedException(charge_result.error)
    
    return charge_result
```

## 4. Datadog Dashboards and Monitors

```
DATADOG MONITORS (alerting):

METRIC ALERT:
  "Alert when error rate > 1% for 5 minutes on service:payment-service"
  
  Query: sum:http.server.errors{service:payment-service}.as_rate() 
         / sum:http.server.requests{service:payment-service}.as_rate()
  
  Threshold: > 0.01 (1%)
  Evaluation window: 5 minutes
  Notify: @pagerduty-oncall-team
  
ANOMALY ALERT:
  "Alert when latency deviates from historical baseline"
  Uses machine learning to detect unusual patterns.
  Good for: traffic that has seasonal patterns (more traffic on weekdays)

APM ALERT:
  "Alert when p99 latency for GET /api/orders > 500ms"
  Directly on trace data — no need to instrument metrics separately.

LOG ALERT:
  "Alert when more than 100 'payment_failed' log events in 5 minutes"
  Triggered by log patterns, not metrics.

COMPOSITE ALERT:
  "Alert when BOTH error rate is high AND p99 latency is high"
  Reduces false positives from single-signal alerts.
```

## 5. Datadog vs. Open Source Stack: The Cost Calculation

```
COST COMPARISON (example: 20-service company, 50 hosts):

OPEN-SOURCE STACK:
  Infrastructure to run it: 3 EC2 instances (~$300/month)
  Engineering time to maintain: ~2 hours/week × $100/hr = $800/month
  Total: ~$1,100/month

DATADOG (APM + Logs + Infrastructure):
  Infrastructure monitoring: $15/host × 50 = $750/month
  APM: $31/host × 50 = $1,550/month
  Log management: $0.10/GB × 50GB/day × 30 days = $150/month
  Total: ~$2,450/month
  
BUT ALSO:
  Zero infrastructure maintenance
  Built-in dashboards for all services
  Customer support
  
THE DECISION:
  < $5K/month observability budget: open-source stack
  $5-20K/month: Datadog might be worth it
  > $20K/month or engineering-time-constrained: Datadog
  
  Most startups start with Datadog for speed,
  then migrate to open-source as they scale and the bill grows.
```

## 6. Senior Engineer Perspective

**Datadog is a productivity tool that becomes a budget problem:**

At a startup with 5 engineers, Datadog is often worth every dollar. You get production observability in a day instead of a week. The dashboards, APM, and log correlation work without configuration. You ship faster.

As you scale to 100 hosts and start seeing $10K+/month Datadog bills, the calculation changes. Many companies at that scale bring in a dedicated SRE to migrate to the open-source stack and save hundreds of thousands of dollars per year.

**The correlation feature is Datadog's killer advantage:**

The one thing that's genuinely hard to replicate in open-source is Datadog's seamless correlation between signals: click a spike in your error rate → automatically see the distributed traces from that exact time window → click into the trace → see the logs from that request. In Grafana + Prometheus + Loki + Jaeger, this correlation requires manual time-range coordination. Datadog does it in one click.
