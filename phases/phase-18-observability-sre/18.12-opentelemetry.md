## 1. What Is OpenTelemetry? (Theory)

OpenTelemetry (OTel) is the CNCF standard for generating, collecting, and exporting telemetry data — traces, metrics, and logs — from any application, in any language, to any backend.

Before OTel, every observability vendor had its own SDK:
- Want Datadog? Use Datadog's tracer SDK.
- Want Jaeger? Use Jaeger's client library.
- Want New Relic? Use New Relic's agent.

Switching vendors meant re-instrumenting your entire codebase. OTel solved this by creating a **vendor-neutral API**: instrument once with OTel, then route the data to any backend by changing configuration.

**OTel's role in the ecosystem:**

```
BEFORE OPENTELEMETRY:
  Code → [Datadog SDK] → Datadog
  To switch to Jaeger: rewrite instrumentation

WITH OPENTELEMETRY:
  Code → [OTel API/SDK] → [OTel Collector] → Datadog
                                            → Jaeger
                                            → Tempo
                                            → Honeycomb
  To switch backends: change one config line in the collector.
  Your application code never changes.
```

## 2. OpenTelemetry Architecture

```
OPENTELEMETRY COMPONENTS:

1. OTel API
   Language-level API for creating traces, metrics, logs.
   Stable contract: your application code depends on this.
   Noop by default: if no SDK is configured, nothing happens.

2. OTel SDK
   Implementation of the API.
   Configured per-application: sampling, exporters, processors.
   Auto-instrumentation: patches common libraries automatically.

3. OTel Collector (optional but recommended)
   Receives telemetry from applications.
   Processes: filters, adds attributes, samples.
   Exports: sends to one or more backends.
   Runs as a sidecar or standalone service.

4. Exporters
   Translates OTel format → backend-specific format.
   OTLP: native OTel protocol (most backends support it now)
   Jaeger exporter, Zipkin exporter, Prometheus exporter, etc.

FLOW:
  Application (OTel SDK)
      ↓ OTLP gRPC/HTTP
  OTel Collector
      ├── filter, batch, sample
      ├──→ Tempo (traces)
      ├──→ Prometheus/Mimir (metrics)
      └──→ Loki (logs)
```

## 3. Auto-Instrumentation: Zero Code Change

OTel provides auto-instrumentation that patches common libraries at startup. You get spans for HTTP requests, DB queries, cache calls — without changing a single line of application code.

```bash
# Python: instrument any application without code changes

pip install opentelemetry-distro opentelemetry-exporter-otlp

# Install auto-instrumentation for all detected libraries:
opentelemetry-bootstrap -a install
# Detects: fastapi, django, flask, sqlalchemy, redis, httpx, requests...
# Installs: opentelemetry-instrumentation-fastapi, etc.

# Run your application with auto-instrumentation:
opentelemetry-instrument \
    --traces_exporter otlp \
    --metrics_exporter otlp \
    --logs_exporter otlp \
    --exporter_otlp_endpoint http://otel-collector:4317 \
    --service_name payment-service \
    python app.py

# Result: every HTTP request, SQL query, and Redis call
# automatically creates a span. No code changes needed.
```

```java
// Java: even simpler — just add a JVM agent

// Download: opentelemetry-javaagent.jar

// Run:
java -javaagent:opentelemetry-javaagent.jar \
     -Dotel.service.name=order-service \
     -Dotel.traces.exporter=otlp \
     -Dotel.exporter.otlp.endpoint=http://otel-collector:4317 \
     -jar app.jar

// The Java agent instruments:
// Spring, Hibernate, JDBC, Kafka, RabbitMQ, gRPC, HTTP clients...
// No code changes. Just add the JVM flag.
```

## 4. The OTel Collector in Production

```yaml
# otel-collector-config.yaml — production configuration

receivers:
  otlp:
    protocols:
      grpc:
        endpoint: 0.0.0.0:4317
      http:
        endpoint: 0.0.0.0:4318
  
  # Also scrape Prometheus metrics from services
  prometheus:
    config:
      scrape_configs:
        - job_name: 'otel-collector'
          static_configs:
            - targets: ['localhost:8888']

processors:
  # Batch spans before exporting (performance)
  batch:
    timeout: 1s
    send_batch_size: 1024
  
  # Add environment info to all telemetry
  resource:
    attributes:
      - key: deployment.environment
        value: production
        action: upsert
  
  # Sample: keep 100% of error traces, 1% of success traces
  tail_sampling:
    decision_wait: 10s
    policies:
      - name: errors-policy
        type: status_code
        status_code:
          status_codes: [ERROR]
      - name: slow-traces-policy
        type: latency
        latency:
          threshold_ms: 1000
      - name: probabilistic-policy
        type: probabilistic
        probabilistic:
          sampling_percentage: 1

exporters:
  # Traces → Grafana Tempo
  otlp/tempo:
    endpoint: http://tempo:4317
    tls:
      insecure: true
  
  # Metrics → Prometheus
  prometheus:
    endpoint: 0.0.0.0:8889
  
  # Logs → Grafana Loki
  loki:
    endpoint: http://loki:3100/loki/api/v1/push

service:
  pipelines:
    traces:
      receivers: [otlp]
      processors: [batch, resource, tail_sampling]
      exporters: [otlp/tempo]
    
    metrics:
      receivers: [otlp, prometheus]
      processors: [batch, resource]
      exporters: [prometheus]
    
    logs:
      receivers: [otlp]
      processors: [batch, resource]
      exporters: [loki]
```

## 5. OTel Semantic Conventions

OpenTelemetry defines standard attribute names so telemetry from different libraries is consistent:

```
SEMANTIC CONVENTIONS (standard attribute names):

HTTP:
  http.method: "GET"
  http.url: "https://api.example.com/users"
  http.status_code: 200
  http.request_content_length: 1234

DATABASE:
  db.system: "postgresql"
  db.name: "production"
  db.operation: "SELECT"
  db.statement: "SELECT * FROM users WHERE id = ?"  (sanitized)

MESSAGING:
  messaging.system: "kafka"
  messaging.destination: "orders"
  messaging.operation: "publish"

NETWORK:
  net.peer.name: "db.internal"
  net.peer.port: 5432

Using standard attributes means:
  - Jaeger, Tempo, Datadog all understand your spans the same way
  - Pre-built dashboards work without customization
  - Service dependency graphs are accurate
```

## 6. Senior Engineer Perspective

**OTel is the right foundation for any new project:**

The observability tooling landscape changes quickly. Vendors that were dominant 5 years ago are now legacy. Building on OTel means you're not locked into any particular vendor's tooling — you're building on an open standard backed by Google, Microsoft, AWS, and 150+ other companies. The instrumentation you write today will work with whatever backend the community converges on in 5 years.

**The collector is optional but worthwhile:**

You can export directly from application to backend (skip the collector). This works and is simpler. But the collector gives you:
1. Tail-based sampling (need to see the full trace before deciding to keep it)
2. Fan-out (same data to multiple backends simultaneously)
3. Centralized configuration (change sampling rate without redeploying apps)
4. Resource reduction (batch/compress before sending, not per-app)

For production systems at any meaningful scale, the collector pays for itself.
