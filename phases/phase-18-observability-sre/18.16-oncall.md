## 1. What Is On-Call? (Theory)

On-call is the practice of having engineers available 24/7 to respond to production incidents. When an alert fires outside business hours, the on-call engineer gets paged and is responsible for investigating and resolving the issue.

On-call is unavoidable for any service that has users who depend on it around the clock. The question isn't whether to have on-call — it's how to make it sustainable and effective.

```
THE ON-CALL REALITY:

GOOD ON-CALL:
  - Alerts fire rarely (SLO-based alerting, well-tuned thresholds)
  - Clear runbooks for common incidents
  - Systems are well-instrumented (you can find the problem)
  - Incidents are resolved in < 30 minutes
  - Post-mortems improve the system
  - Engineers feel prepared and supported

BAD ON-CALL:
  - Pages every other night (alert fatigue, burnout)
  - No runbooks ("good luck figuring it out")
  - Poor observability (you can't find what broke)
  - Incidents last hours
  - Same incidents repeat (no post-mortems or action items)
  - Engineers dread being on-call, leave teams

THE GOAL: on-call should be boring.
  If your on-call is exciting, something is wrong with your system.
  Excitement means unplanned work, unclear processes, poor automation.
```

## 2. Incident Response Process

When a page fires, a structured response process reduces the time to resolution:

```
INCIDENT RESPONSE PHASES:

1. DETECTION (0-2 minutes)
   Alert fires. On-call engineer is paged.
   Acknowledge the alert (stops escalation to backup).
   
2. TRIAGE (2-10 minutes)
   What is broken? How severe is it?
   
   Severity levels:
   SEV1: Complete service outage, all users affected → page everyone
   SEV2: Major degradation, many users affected → page team
   SEV3: Minor degradation, small % of users → handle during business hours
   
3. COMMUNICATE (immediately for SEV1/2)
   Post in incident Slack channel: "I'm investigating [alert name]"
   This prevents duplicate investigation and keeps stakeholders informed.
   
4. INVESTIGATE (5-30 minutes)
   Follow the runbook for the alert.
   Check metrics (what changed?), logs (what errors?), traces (where?).
   Narrow down the root cause.
   
5. MITIGATE (as soon as cause is known)
   Restore service first, root cause later.
   Options: rollback deployment, increase replicas, restart service, disable feature flag
   
6. COMMUNICATE RESOLUTION
   "Service restored. Root cause: bad deployment. Working on post-mortem."
   
7. POST-MORTEM (within 24-48 hours)
   Document what happened, why, and what we're doing to prevent it.
```

## 3. Runbooks: Your First Line of Defense

A runbook is a documented procedure for handling a specific alert or incident type. Good runbooks reduce response time and the expertise required to resolve incidents.

```markdown
# Runbook: PaymentService-HighErrorRate

**Alert:** PaymentAPIHighBurnRate  
**Severity:** SEV1 (if burn rate > 14.4x), SEV2 (if 6-14x)

## What This Means
The payment service is returning 5xx errors at a rate that will exhaust 
our 30-day error budget within 2 days.

## Immediate Steps (< 5 minutes)

1. Check Grafana dashboard: "Payment Service Overview"
   Link: [dashboard-link]
   
   Look at:
   - Which endpoints are failing? (pie chart "Errors by Endpoint")
   - When did it start? (correlate with deployments in deploy log)
   - Error messages in Loki: `{service="payment-service"} |= "error"`

2. Check recent deployments:
   `kubectl rollout history deployment/payment-service`
   Did a deployment coincide with the alert time?

3. If deployment caused it → ROLLBACK IMMEDIATELY:
   `kubectl rollout undo deployment/payment-service`
   Wait 2 minutes, check if error rate returns to normal.

## If Rollback Doesn't Help

4. Check database connectivity:
   `kubectl exec -it payment-pod -- psql $DATABASE_URL -c "SELECT 1"`
   If database is unreachable → escalate to DBA

5. Check downstream dependencies:
   `kubectl logs -l app=payment-service --since=5m | grep "connection refused"`
   If Stripe API is unreachable → check status.stripe.com

6. Check resource exhaustion:
   `kubectl top pods -l app=payment-service`
   If CPU/memory at limits → scale up:
   `kubectl scale deployment/payment-service --replicas=5`

## Escalation
If not resolved in 30 minutes: escalate to [backup engineer]
```

## 4. Blameless Post-Mortems

A post-mortem is a document written after every significant incident. The word "blameless" is critical: the goal is to understand systemic failures, not to find someone to blame.

```markdown
# Post-Mortem: Payment Service Outage — 2026-06-15

## Summary
On June 15, 2026, from 14:28-14:58 UTC (30 minutes), 
the payment service returned 503 errors to 100% of requests.
Approximately 2,400 payment attempts failed.

## Timeline
14:28 — Deployment of payment-service v2.3.1 completed
14:29 — Error rate begins rising (Prometheus alert threshold: 1%)
14:31 — Burn rate alert fires (SEV1), on-call (Alice) paged
14:33 — Alice acknowledges, posts in #incidents
14:35 — Alice identifies the deployment as root cause from Grafana
14:38 — Alice initiates rollback
14:40 — Rollback completes, error rate returns to normal
14:42 — Alert resolves, service confirmed healthy
14:58 — Post-mortem opened, root cause confirmed

## Root Cause
The v2.3.1 deployment included a database migration that renamed 
the `user_id` column to `customer_id`. The migration ran successfully,
but the application code was not updated (still referencing `user_id`).
Every query to the database failed with "column 'user_id' does not exist".

## What Went Well
- Monitoring detected the issue within 1 minute of deployment
- On-call response was swift (2 minutes to acknowledge, 7 to identify cause)
- Rollback process worked correctly and took 2 minutes
- Total downtime: 30 minutes (within our SLO budget)

## What Went Poorly
- The migration and code change were in the same PR but deployed separately
- No canary deployment to detect the failure on 1% traffic first
- No automated rollback trigger when error rate exceeds threshold

## Action Items
| Action | Owner | Due Date |
|--------|-------|----------|
| Require migrations to be backward-compatible (support both old and new column name for 1 deployment) | Bob | 2026-06-22 |
| Implement canary deployments for payment-service | Alice | 2026-06-29 |
| Add pre-deployment integration test that runs actual payment flow | Carol | 2026-07-06 |
| Update runbook with column migration failure scenario | Alice | 2026-06-20 |
```

## 5. Sustainable On-Call Practices

```
MAKING ON-CALL SUSTAINABLE:

1. ROTATION SIZE: minimum 4-5 engineers per rotation
   Each engineer is on-call ~once per month
   Smaller rotations → burnout from frequent paging

2. HANDOFF: overlap between shifts
   Outgoing on-call: brief incoming on-call on ongoing issues
   Write a handoff note: active alerts, known issues, recent changes

3. COMPENSATION: off-call relief after heavy on-call weeks
   If you were paged 10 times and woke up at 3am twice: take Monday off

4. ALERT HYGIENE: review alerts monthly
   Alerts that fired but required no action: tune or delete them
   "If it fired and nothing was wrong, it's a false positive"
   
5. RUNBOOKS: write them before you go on-call
   New team member comes on-call: they have runbooks for every alert
   If you spend time figuring out what an alert means: write a runbook

6. POST-MORTEM ALL SEV1/2 incidents:
   No post-mortem = repeating incidents
   The post-mortem is more important than the fix
```

## 6. Senior Engineer Perspective

**On-call culture reveals engineering culture:**

If your on-call is miserable (constant pages, unclear processes, repeated incidents), that's a symptom of deeper problems: insufficient monitoring, missing runbooks, no learning from failures. Fixing on-call means fixing the engineering culture and the system quality.

The best signal for a healthy system: new engineers who go on-call for the first time have a quiet week. They don't get paged because the system is reliable, the runbooks exist, and previous on-calls reduced toil. That's the goal.

**The blameless post-mortem is a cultural practice, not a document:**

Writing a post-mortem is easy. Truly practicing "blameless" is hard. When an incident happens, the human instinct is to find someone responsible. Resist it. The SRE practice treats incidents as system failures: "the system allowed this to happen." Engineers are not robots — they make mistakes. The question is why the system didn't catch the mistake before it caused harm.

Teams that practice blameless post-mortems build safer systems faster, because engineers share mistakes openly instead of hiding them.
