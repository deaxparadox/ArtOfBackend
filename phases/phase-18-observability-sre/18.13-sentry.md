## 1. What Is Sentry? (Theory)

Sentry is an error tracking and performance monitoring platform. While Prometheus tells you your error rate is 2%, Sentry tells you **exactly which exceptions are being thrown**, how many users are affected, which line of code caused them, and the full stack trace.

**The gap Sentry fills:**

```
PROMETHEUS:
  "Error rate: 2% (up from 0.1%)"
  You know something is wrong. You don't know what.

SENTRY:
  "TypeError: Cannot read property 'id' of undefined
   payment_service.py, line 142, in process_payment
   Affected: 1,247 users
   First seen: 2 minutes ago
   Likely caused by: commit abc123 by Alice"
   
  Stack trace, user context, breadcrumbs (what the user did before),
  environment variables, all in one view.
```

## 2. Setting Up Sentry

```python
# Python / FastAPI

import sentry_sdk
from sentry_sdk.integrations.fastapi import FastApiIntegration
from sentry_sdk.integrations.sqlalchemy import SqlalchemyIntegration
from sentry_sdk.integrations.redis import RedisIntegration

sentry_sdk.init(
    dsn="https://your-key@sentry.io/your-project-id",
    
    environment="production",
    release="my-app@1.2.3",  # tag errors to a release/commit
    
    # Sample rate for performance monitoring (traces)
    traces_sample_rate=0.1,  # 10% of requests
    
    # Profiles (flame graphs in production)
    profiles_sample_rate=0.1,
    
    integrations=[
        FastApiIntegration(transaction_style="endpoint"),
        SqlalchemyIntegration(),
        RedisIntegration(),
    ],
    
    # Control what data is sent (GDPR)
    before_send=scrub_sensitive_data,
)

def scrub_sensitive_data(event, hint):
    """Remove sensitive data before sending to Sentry."""
    if "request" in event:
        # Don't send authorization headers
        event["request"].get("headers", {}).pop("Authorization", None)
        event["request"].get("headers", {}).pop("Cookie", None)
    return event
```

## 3. Sentry Features Deep Dive

```
ERROR GROUPING (issues):
  Sentry groups similar errors into "issues".
  100 users hitting the same ValueError → one issue, not 100 alerts.
  
  Groups by: exception type + file + line + message pattern.
  
  Each issue shows:
  - Affected users count
  - Events per hour graph
  - First/last occurrence
  - Assignee
  - Release where first seen
  - Git blame (who last touched that line)

RELEASE TRACKING:
  Tag your Sentry init with: release="git-sha-abc123"
  After a deploy, Sentry shows:
  - New issues introduced in this release
  - Resolved issues (error disappeared in new release)
  - Regression issues (was resolved, now broken again)
  
  This turns "something broke" into "the deploy at 14:30 introduced 3 new errors".

PERFORMANCE MONITORING:
  Sentry captures distributed traces for sampled requests.
  Shows: slow database queries, slow HTTP calls, N+1 query patterns.
  
  Transaction: one HTTP request end-to-end
  Spans: child operations (SQL queries, cache calls, external HTTP)
  
  Highlights: operations that are slow or occurring unexpectedly often

ALERTS:
  "Alert when new issue seen in production"
  "Alert when issue affects > 100 users"
  "Alert when error rate increases by 10%"
  Sends to: Slack, PagerDuty, email, webhook
```

## 4. Sentry in Different Languages

```javascript
// Node.js / TypeScript
import * as Sentry from "@sentry/node";

Sentry.init({
    dsn: "https://your-key@sentry.io/project-id",
    environment: process.env.NODE_ENV,
    release: process.env.GIT_SHA,
    tracesSampleRate: 0.1,
    integrations: [
        new Sentry.Integrations.Http({ tracing: true }),
        new Sentry.Integrations.Express({ app }),
    ],
});

// Express middleware (must be first)
app.use(Sentry.Handlers.requestHandler());
app.use(Sentry.Handlers.tracingHandler());

// ... your routes ...

// Error handler (must be after routes)
app.use(Sentry.Handlers.errorHandler());
```

```go
// Go
import "github.com/getsentry/sentry-go"

func main() {
    sentry.Init(sentry.ClientOptions{
        Dsn:              "https://key@sentry.io/project",
        Environment:      "production",
        Release:          os.Getenv("GIT_SHA"),
        TracesSampleRate: 0.1,
    })
    defer sentry.Flush(2 * time.Second)
    
    // Capture panic in a goroutine
    defer sentry.Recover()
    
    // Manual capture
    if err != nil {
        sentry.CaptureException(err)
    }
}
```

## 5. Sentry vs. Logs for Error Tracking

```
SENTRY vs. LOGS FOR ERRORS:

LOGS:
  ✓ Complete history, every occurrence
  ✓ Can search for anything
  ✗ No grouping: 1000 log lines vs. 1 Sentry issue
  ✗ No user impact information
  ✗ No automatic assignee/alert
  ✗ Context must be manually added

SENTRY:
  ✓ Automatic deduplication/grouping
  ✓ User impact (how many users affected)
  ✓ Release correlation (which deploy introduced it)
  ✓ Full stack trace always captured
  ✓ Breadcrumbs (what happened before the error)
  ✗ Sampled for performance monitoring (not all requests)
  ✗ Cost at high error volumes
  ✗ Sensitive data considerations

RECOMMENDATION: Use BOTH.
  Logs for: operational debugging, audit trails, metrics
  Sentry for: exception tracking, error prioritization, developer alerting
  They complement each other — don't replace logs with Sentry.
```

## 6. Senior Engineer Perspective

**Sentry changes how developers relate to production:**

Without Sentry, errors are invisible until a user reports them. With Sentry, every exception in production lands directly in Slack or email for the team responsible. This creates immediate feedback: you deploy, you watch Sentry for 15 minutes, you see if you introduced regressions.

This is culturally important. It shifts the team from "deploy and hope" to "deploy and watch." Engineers who see their errors in real-time write more defensive code, add better error handling, and feel more ownership of production quality.

**The release tracking feature is underused:**

Connecting Sentry releases to your CI/CD pipeline (pass the git SHA to `sentry.init(release=sha)`) and uploading source maps gives you something invaluable: exact attribution of new errors to specific commits and the developers who wrote them. This turns a production incident from "something is broken, investigate for an hour" to "commit abc123 by Alice introduced this error 5 minutes after deploy."
