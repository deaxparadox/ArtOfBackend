## 1. What Is eBPF? (Theory)

eBPF (extended Berkeley Packet Filter) is a Linux kernel technology that lets you run sandboxed programs inside the Linux kernel without modifying kernel source code or loading kernel modules. It's like a superpower: attach a small program to any kernel event and collect data or modify behavior — safely and efficiently.

For backend engineers, eBPF's most relevant use case is **observability without code changes**. You can instrument a running application — capture network traffic, system calls, function calls — without restarting it or modifying its code.

**The analogy:**

```
TRADITIONAL INSTRUMENTATION:
  You want to know every file a process opens.
  Option 1: Modify the application to log every open() call.
            (Requires code change, redeploy)
  Option 2: strace the process.
            (Works but 10-100x slower — strace intercepts every syscall)

eBPF:
  Attach a tiny eBPF program to the open() syscall in the kernel.
  The program runs in the kernel itself — microsecond overhead.
  Zero application code change. Zero restart.
  Watch any running process in real time.
```

## 2. How eBPF Works

```
eBPF EXECUTION MODEL:

1. Write eBPF program (C-like language)
   "When process X calls read(), record filename and duration"

2. Compile to eBPF bytecode

3. Load bytecode into the Linux kernel
   The kernel verifier checks safety:
   - No infinite loops
   - No memory access out of bounds
   - Must terminate
   
4. Attach to a kernel probe (kprobe, tracepoint, uprobe, network hook)
   "Run this when open() is called"
   "Run this when a TCP packet arrives"
   "Run this when function foo() in process bar is called"

5. eBPF program runs in kernel space (fast, low overhead)
   Writes data to eBPF maps (shared memory between kernel and user space)

6. User-space tool reads from eBPF maps
   Displays statistics, generates logs, aggregates metrics

OVERHEAD:
  Traditional tracing (strace): 100x slowdown
  eBPF: typically 1-5% overhead
  This makes eBPF usable in production, not just dev/debugging.
```

## 3. eBPF in Practice: Observability Tools

### bpftrace: eBPF One-Liners

```bash
# bpftrace: like awk for the kernel

# Trace all open() calls system-wide
bpftrace -e 'tracepoint:syscalls:sys_enter_open { printf("%s %s\n", comm, str(args->filename)); }'

# Trace TCP connections being established
bpftrace -e 'tracepoint:sock:inet_sock_set_state /args->newstate == 1/ { printf("%s connect: %d.%d.%d.%d:%d\n", comm, (args->daddr>>0)&0xFF, (args->daddr>>8)&0xFF, (args->daddr>>16)&0xFF, (args->daddr>>24)&0xFF, args->dport); }'

# Profile Python function calls (no code change required)
bpftrace -e 'uprobe:/usr/bin/python3:PyObject_Call { @[ustack] = count(); }'

# Measure read() system call latency per process
bpftrace -e 'tracepoint:syscalls:sys_enter_read { @start[pid] = nsecs; }
             tracepoint:syscalls:sys_exit_read /@start[pid]/ { 
               @latency[comm] = hist((nsecs - @start[pid]) / 1000);
               delete(@start[pid]);
             }'
```

### Pixie: No-Instrumentation Observability for Kubernetes

Pixie (acquired by New Relic) uses eBPF to automatically collect traces, metrics, and logs from Kubernetes pods without any code changes:

```bash
# Install Pixie in your Kubernetes cluster
curl -fsSL https://withpixie.ai/install.sh | bash
px deploy

# Immediately get:
# - HTTP request/response traces (including body)
# - SQL queries with full statements
# - DNS queries
# - Network topology (which services talk to which)
# - CPU flamegraphs
# - Memory usage
# All without changing any application code.

# Query with PxL (Pixie's SQL-like language):
px run px/http_data    # all HTTP traffic
px run px/db_query_latency  # database query latency
px run px/pod_memory_usage  # memory per pod
```

### Cilium: eBPF-Based Networking and Security

```
CILIUM:
  Kubernetes networking plugin that uses eBPF instead of iptables.
  
  Traditional k8s networking: iptables rules → slow (O(n) matching)
  Cilium: eBPF programs → fast (O(1) lookup in hash tables)
  
  FEATURES:
  Network Policy: control which pods can talk to which
  Load Balancing: replace kube-proxy with eBPF-based LB
  Observability: Hubble UI shows real-time service communication map
  
  The Hubble UI (Cilium's observability tool) shows:
  → Which service is talking to which
  → HTTP methods and URLs being called
  → TCP connections
  → DNS queries
  All collected at the kernel level, zero code changes.
```

## 4. BCC (BPF Compiler Collection): Higher-Level Tools

The BCC project provides Python/Lua wrappers for common eBPF patterns:

```python
#!/usr/bin/python3
# Trace slow disk I/O using BCC

from bcc import BPF
from time import sleep

# BPF program (C code run in the kernel)
bpf_text = """
#include <uapi/linux/ptrace.h>
#include <linux/blkdev.h>

BPF_HASH(start, struct request *);
BPF_HISTOGRAM(dist);

// When I/O request starts
void trace_start(struct pt_regs *ctx, struct request *req) {
    u64 ts = bpf_ktime_get_ns();
    start.update(&req, &ts);
}

// When I/O request completes
void trace_completion(struct pt_regs *ctx, struct request *req) {
    u64 *tsp = start.lookup(&req);
    if (tsp != 0) {
        u64 delta = bpf_ktime_get_ns() - *tsp;
        dist.increment(bpf_log2l(delta / 1000));  // microseconds
        start.delete(&req);
    }
}
"""

b = BPF(text=bpf_text)
b.attach_kprobe(event="blk_start_request", fn_name="trace_start")
b.attach_kprobe(event="blk_account_io_completion", fn_name="trace_completion")

print("Tracing disk I/O latency... Hit Ctrl+C to stop.")
sleep(10)
print("\nI/O latency distribution (microseconds):")
b["dist"].print_log2_hist("usecs")
```

## 5. When eBPF Is the Right Tool

```
eBPF SHINES WHEN:
  ✓ You can't modify application code (vendor software, legacy apps)
  ✓ You need production debugging without restarts
  ✓ You want zero-overhead continuous profiling
  ✓ Kubernetes networking/security policy enforcement (Cilium)
  ✓ Security: detect and alert on unusual syscall patterns (Falco)
  ✓ Performance profiling: identify hot paths without instrumentation
  
eBPF IS OVERKILL WHEN:
  ✗ You can add instrumentation to application code (OTel is easier)
  ✗ High-level metrics suffice (Prometheus handles most cases)
  ✗ Team lacks Linux kernel knowledge (steep learning curve)
  
ENTRY-LEVEL eBPF TOOLS (no kernel knowledge needed):
  Pixie/Groundcover: deploy to k8s, get automatic observability
  Cilium/Hubble: install as k8s CNI, get network observability
  
DEEP-LEVEL eBPF (requires kernel knowledge):
  bpftrace: one-liners for specific kernel investigations
  BCC: write custom eBPF programs in Python
```

## 6. Senior Engineer Perspective

**eBPF is the future of cloud-native observability:**

The ability to observe any system without code changes is transformative. In production, you often can't restart services, modify code, or deploy changes to investigate a problem. eBPF lets you attach observability to running services in real time.

Tools like Pixie and Cilium/Hubble are making eBPF accessible without requiring kernel expertise. If you're on Kubernetes, deploying Cilium and Hubble gives you a complete service communication map and network observability "for free" — no application changes needed.

**The security applications are equally important:**

Falco (CNCF project) uses eBPF to detect anomalous behavior in running containers — processes spawning unexpected child processes, files being modified in unexpected locations, unusual network connections. Traditional signature-based detection misses zero-days. Behavioral detection via eBPF catches unusual patterns regardless of the specific attack.
