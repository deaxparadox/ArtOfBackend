## 1. What Is the ELK Stack? (Theory)

ELK = **E**lasticsearch + **L**ogstash + **K**ibana. It's the most widely deployed log aggregation and visualization stack. Each component has a specific role:

```
THE ELK PIPELINE:

Application → [Logstash / Filebeat] → Elasticsearch → Kibana

FILEBEAT (data shipper):
  Lightweight agent running on each server.
  Tails log files or receives logs over TCP/UDP.
  Ships logs to Logstash or directly to Elasticsearch.

LOGSTASH (data processor, optional):
  Parses and transforms logs before indexing.
  Filter plugins: grok (regex-based parsing), mutate (rename/add fields)
  Can aggregate from multiple sources, enrich with geolocation data.
  Heavier resource usage than Filebeat alone.

ELASTICSEARCH (storage and search engine):
  Distributed full-text search engine (built on Apache Lucene).
  Each log line becomes a JSON document, indexed for fast search.
  Supports: full-text search, aggregations, date histograms, geolocation.

KIBANA (visualization):
  Web UI for Elasticsearch.
  Build dashboards, run ad-hoc queries, visualize time-series data.
  Alerting: trigger PagerDuty/Slack when error count spikes.
```

## 2. How Elasticsearch Stores Logs

```
ELASTICSEARCH DATA MODEL:

INDEX: like a database table
  logs-2026-06-15   (daily rotation is standard)
  logs-2026-06-16
  ...
  
DOCUMENT: like a database row (one log entry)
  {
    "@timestamp": "2026-06-15T14:30:00.123Z",
    "level": "error",
    "service": "payment-service",
    "user_id": "123",
    "event": "payment_failed",
    "amount": 99.99,
    "message": "Payment failed: Card declined"
  }

SHARD: a partition of an index
  An index is split into shards for horizontal scaling.
  Each shard is an independent Lucene index.
  More shards = more parallelism (more search nodes can help).

REPLICA: copy of a shard
  Each shard can have 1+ replicas on different nodes.
  Provides redundancy + read scaling.

INDEX LIFECYCLE MANAGEMENT (ILM):
  Hot tier:  recent logs (0-7 days), fast SSDs, high search load
  Warm tier: week-old logs, cheaper hardware
  Cold tier: month-old logs, object storage
  Delete:    logs older than N days (configure per compliance requirement)
```

## 3. Shipping Logs with Filebeat

```yaml
# filebeat.yml — collect logs and send to Elasticsearch

filebeat.inputs:
  - type: log
    enabled: true
    paths:
      - /var/log/app/*.log
    json.keys_under_root: true  # parse JSON log lines into fields
    json.overwrite_keys: true

  - type: docker
    containers.ids: '*'         # collect from all Docker containers

processors:
  - add_host_metadata: ~        # add hostname, IP to every log
  - add_docker_metadata: ~      # add container name, image

output.elasticsearch:
  hosts: ["http://elasticsearch:9200"]
  index: "logs-%{+yyyy.MM.dd}"
  
  # For production with authentication:
  # username: "elastic"
  # password: "${ELASTICSEARCH_PASSWORD}"
  # ssl.certificate_authorities: ["/etc/ca.crt"]
```

## 4. Kibana: Querying and Visualizing Logs

```
KIBANA DISCOVER (ad-hoc log search):

Query syntax (KQL - Kibana Query Language):
  level: "error"                           → all error logs
  level: "error" AND service: "payment*"  → errors in payment services
  amount > 100 AND level: "error"          → large failed transactions
  event: "payment_failed" AND NOT user_id: "test-*"  → real failures

Time range filtering:
  Last 15 minutes  → active incident investigation
  Last 24 hours    → trend analysis
  Custom range     → post-mortem analysis

COMMON QUERIES IN PRACTICE:
  "Show me all errors in the last hour for service=payment-service"
  level: "error" AND service: "payment-service"
  
  "Show me all requests for user 123"
  user_id: "123"
  
  "Show me all requests with trace_id matching this trace"
  trace_id: "abc-def-123"

KIBANA DASHBOARDS:
  Time-series graph: errors per minute over 24 hours
  Pie chart: error breakdown by service
  Data table: top 10 most common error messages
  Heat map: request volume by hour × day of week
```

## 5. The ELK Stack in Production

```
ELASTICSEARCH SIZING GUIDELINES:
  
  Log volume: 1 GB/day of logs
  Retention: 30 days
  Storage needed: ~30 GB raw + ~30 GB replicas = ~60 GB
  
  Start with: 3-node cluster (HA), each node 8GB RAM, 100GB SSD
  
  RAM rule: Elasticsearch heap = 50% of RAM, but max 31GB.
  A 16GB node → 8GB heap. Don't go above 31GB heap (JVM limitation).

COMMON PRODUCTION ISSUES:

  1. DISK FULL:
     ILM not configured → old indices never deleted.
     Fix: configure ILM to delete logs older than 30 days.
  
  2. SLOW QUERIES:
     Querying without time bounds → searches all shards.
     Fix: always include @timestamp range in queries.
  
  3. SHARD COUNT TOO HIGH:
     Daily index creation × years = thousands of shards.
     Fix: use ILM to merge old shards, delete aged indices.
  
  4. HEAP PRESSURE:
     Large aggregations (cardinality of user_id) exhaust heap.
     Fix: limit aggregation cardinality, increase JVM heap.

WHEN ELK IS OVERKILL:
  < 1 GB/day logs: CloudWatch, Datadog, or Papertrail is cheaper and simpler.
  No dedicated ops team: managed ELK (Elastic Cloud) or Grafana Loki.
  
WHEN ELK IS WORTH IT:
  > 10 GB/day logs (cloud managed gets very expensive)
  Compliance requires logs stored on-prem
  Complex log parsing requirements
  Large engineering team that needs powerful queries
```

## 6. Senior Engineer Perspective

**Index management is the hidden operational burden:**

Running Elasticsearch in production is not hard to start. It becomes difficult when you have months of data, misconfigured ILM, and suddenly your cluster is 90% full. The engineering tax of operating a large Elasticsearch cluster — capacity planning, shard management, version upgrades — is significant.

The honest advice: unless you have more than 10 GB/day of logs and a team with operational experience, start with a managed service. Grafana Loki (topic 18.4) is dramatically cheaper and easier to run than Elasticsearch, though it trades query flexibility for cost and simplicity.
