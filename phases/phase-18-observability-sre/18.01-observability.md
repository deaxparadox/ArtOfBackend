## The Three Pillars of Observability (18.1)

```
LOGS: what happened
  Text records of discrete events.
  "User 123 placed order 456 at 14:30:00"
  Good for: debugging specific incidents, audit trails

METRICS: how many / how fast / how much
  Numerical measurements sampled over time.
  "p99 request latency: 245ms at 14:30"
  "Error rate: 0.3% at 14:30"
  Good for: dashboards, SLO tracking, alerting

TRACES: where time was spent
  Distributed request paths with timing.
  "Request took 245ms: 5ms auth, 180ms DB query, 60ms serialization"
  Good for: finding bottlenecks, debugging cross-service issues

OBSERVABILITY vs MONITORING:
  Monitoring: checking known failure modes ("is the server up?")
  Observability: ability to ask new questions without deploying new code
  → Logs, metrics, traces that let you investigate any unknown failure
```

## Structured Logging (18.2)

```python
# structlog: the production-grade structured logging library for Python
import structlog
import logging

structlog.configure(
    processors=[
        structlog.contextvars.merge_contextvars,       # per-request context
        structlog.processors.add_log_level,
        structlog.processors.TimeStamper(fmt="iso", utc=True),
        structlog.processors.StackInfoRenderer(),
        structlog.dev.set_exc_info,
        structlog.processors.JSONRenderer(),           # production: JSON
        # structlog.dev.ConsoleRenderer(),             # development: pretty
    ],
    wrapper_class=structlog.make_filtering_bound_logger(logging.INFO),
)

log = structlog.get_logger()

# Bind request-level context (applies to all logs in this scope)
structlog.contextvars.bind_contextvars(
    request_id="abc-123",
    user_id=42,
    service="api",
    version="v1.2.3",
)

# All subsequent log calls include the bound context
log.info("order_placed", order_id=789, amount_cents=5000)
# {"event": "order_placed", "order_id": 789, "amount_cents": 5000,
#  "request_id": "abc-123", "user_id": 42, "timestamp": "2024-06-15T14:30:00Z"}

log.error("payment_failed", order_id=789, error="card_declined")
# Same bound context included automatically
```

## Prometheus + Grafana (18.6-18.7)

```python
# Prometheus: pull-based metrics collection
from prometheus_client import Counter, Histogram, Gauge, start_http_server
import time

# Define metrics
request_count = Counter(
    "http_requests_total",
    "Total HTTP requests",
    ["method", "endpoint", "status_code"]
)

request_duration = Histogram(
    "http_request_duration_seconds",
    "HTTP request duration",
    ["method", "endpoint"],
    buckets=[0.005, 0.01, 0.025, 0.05, 0.1, 0.25, 0.5, 1.0, 2.5, 5.0]
)

active_connections = Gauge(
    "active_connections",
    "Number of active WebSocket connections"
)

# Instrument FastAPI
from starlette.middleware.base import BaseHTTPMiddleware

class PrometheusMiddleware(BaseHTTPMiddleware):
    async def dispatch(self, request, call_next):
        start = time.perf_counter()
        response = await call_next(request)
        duration = time.perf_counter() - start

        request_count.labels(
            method=request.method,
            endpoint=request.url.path,
            status_code=response.status_code,
        ).inc()

        request_duration.labels(
            method=request.method,
            endpoint=request.url.path,
        ).observe(duration)

        return response

# Expose metrics endpoint for Prometheus to scrape
# GET /metrics → returns all metrics in Prometheus format
start_http_server(9090)  # or use prometheus_fastapi_instrumentator
```

## OpenTelemetry (18.12): The Standard

```python
# OpenTelemetry: one SDK for all observability signals (traces, metrics, logs)
from opentelemetry import trace
from opentelemetry.sdk.trace import TracerProvider
from opentelemetry.sdk.trace.export import BatchSpanProcessor
from opentelemetry.exporter.otlp.proto.grpc.trace_exporter import OTLPSpanExporter
from opentelemetry.instrumentation.fastapi import FastAPIInstrumentor
from opentelemetry.instrumentation.httpx import HTTPXClientInstrumentor
from opentelemetry.instrumentation.asyncpg import AsyncPGInstrumentor
from opentelemetry.instrumentation.redis import RedisInstrumentor

# Configure tracing
provider = TracerProvider()
provider.add_span_processor(
    BatchSpanProcessor(
        OTLPSpanExporter(endpoint="http://otel-collector:4317")
    )
)
trace.set_tracer_provider(provider)

# Auto-instrument everything
FastAPIInstrumentor.instrument_app(app)
HTTPXClientInstrumentor().instrument()
AsyncPGInstrumentor().instrument()
RedisInstrumentor().instrument()

# Now every request automatically:
# 1. Creates a trace with spans for: HTTP handler, DB queries, Redis calls, outgoing HTTP
# 2. Propagates W3C traceparent header to downstream services
# 3. Sends to OTLP collector → Jaeger/Tempo/Datadog

# Manual spans for important business operations
tracer = trace.get_tracer(__name__)

@app.post("/orders")
async def create_order(data: OrderRequest):
    with tracer.start_as_current_span("create_order") as span:
        span.set_attribute("user.id", current_user.id)
        span.set_attribute("order.amount_cents", data.total_cents)

        with tracer.start_as_current_span("check_inventory"):
            available = await inventory.check(data.product_id, data.quantity)

        with tracer.start_as_current_span("process_payment"):
            payment = await payment_gateway.charge(data.amount)
            span.set_attribute("payment.id", payment.id)

        return await db.create_order(data)
```

## SLOs, Error Budgets, On-Call (18.14-18.16)

```
DEFINING AN SLO:

1. Identify what users care about:
   - "Can they complete a purchase?" → order API availability
   - "Is it fast enough?" → order API p99 latency

2. Define the SLI (what to measure):
   SLI = (good requests) / (total requests)
   Good = status_code < 500 AND latency < 500ms

3. Set the SLO:
   99.9% availability → 43 minutes downtime/month
   95th percentile latency < 200ms

4. Track the error budget:
   Budget = (1 - 0.999) × 43,200 minutes = 43 minutes/month
   Burned so far: 12 minutes → 31 minutes remaining

BURN RATE ALERTING:
  If you burn 5% of monthly budget in 1 hour → page someone immediately
  If you burn 2% of monthly budget in 6 hours → page someone soon
  This is more actionable than "alert when error rate > X%"
  because it tells you "at this rate, you'll miss your SLO"

ON-CALL RUNBOOK TEMPLATE:
  Alert name: HighErrorRate
  What broke: API returning 5xx errors
  Likely causes:
    1. Database overloaded (check pg_stat_activity)
    2. Redis unreachable (check redis-cli ping)
    3. Deployment just happened (check kubectl rollout status)
  Investigation steps:
    1. kubectl logs -l app=api --tail=100 | grep ERROR
    2. Check Grafana dashboard: /d/api-latency
    3. Check PagerDuty for related alerts
  Escalation: if not resolved in 15 min → page @backend-team
```
