## 1. What Is a Runbook? (Theory)

A runbook is a document that tells an on-call engineer — who may be half-asleep at 3am — exactly how to handle a specific alert or incident. It's the difference between "stare at dashboards for 45 minutes hoping something makes sense" and "follow these 5 steps to resolve this in 10 minutes."

The name comes from traditional operations: physical books operators ran through during procedures.

```
WHY RUNBOOKS MATTER:

SCENARIO WITHOUT RUNBOOK:
  3:15am. Alert: "PaymentService-HighErrorRate"
  On-call (Alice) wakes up.
  
  "Okay, payment service has high errors... what do I check?"
  Alice spends 15 minutes exploring dashboards, not sure where to look.
  She runs some queries, finds the problem is a database connection issue.
  "How do I restart the connection pool without downtime?"
  Another 20 minutes searching Slack history for previous incidents.
  Total: 45 minutes, degraded service the entire time.

SCENARIO WITH RUNBOOK:
  3:15am. Alert fires. Notification includes runbook link.
  Alice opens the runbook on her phone.
  Step 1: Check [this Grafana dashboard] for DB connection pool usage.
  Step 2: If pool exhausted, run: kubectl rollout restart deployment/payment-service
  Step 3: Verify error rate returns to < 0.1% within 5 minutes.
  Total: 8 minutes. Service restored quickly.
```

## 2. Runbook Structure

A good runbook follows a consistent structure. Engineers should be able to execute it under stress:

```markdown
# Runbook: [Alert Name]

**Severity:** SEV1 / SEV2 / SEV3
**Owner:** [Team]
**Last updated:** [Date]

## What This Alert Means
One paragraph, plain English. What is broken and why this alert exists.

## Impact
- Who is affected: all users / X% of users / users in region Y
- What can't they do: can't check out / can't log in / degraded performance

## Quick Links
- Dashboard: [link]
- Logs: [link to pre-filtered Loki/Kibana query]
- Service: [link to service in k8s/consul]

## Diagnostic Steps

### Step 1: Confirm the scope (2 minutes)
1. Open the [Dashboard Link]
2. Check [metric name] — is it a single endpoint or all endpoints?
3. Check [log query] for error messages

### Step 2: Check recent changes (2 minutes)
```
kubectl rollout history deployment/payment-service
git log --oneline -10 deployment/payment-service
```
Did the problem start after a deployment?

### Step 3: Assess root cause

**If deployment caused it:**
→ Rollback: `kubectl rollout undo deployment/payment-service`
→ Verify error rate returns to normal within 5 minutes

**If database is the issue:**
→ Check connectivity: [command]
→ If connection pool exhausted: [restart command]
→ If database is down: escalate to [DBA on-call]

**If external API is down:**
→ Check [status page link]
→ Enable fallback mode: `kubectl set env deployment/payment-service STRIPE_FALLBACK=true`

## Escalation
If not resolved in 30 minutes: page [backup engineer]
If database involvement: page [DBA]

## Resolution
1. Verify error rate < 0.1% for 5+ minutes
2. Resolve alert in PagerDuty
3. Post in #incidents: "[brief description] — resolved"
4. If SEV1/2: create post-mortem within 24 hours

## Common Issues & Solutions
| Symptom | Likely Cause | Fix |
|---------|--------------|-----|
| Error rate > 10% | Bad deployment | Rollback |
| Connection errors | DB pool exhausted | Restart pods |
| 503 errors from load balancer | All pods unhealthy | Check pod logs |
```

## 3. When to Write Runbooks

```
WRITE A RUNBOOK:
  ✓ Before the first time an alert fires (proactive)
  ✓ Immediately after any incident (while context is fresh)
  ✓ When you notice an engineer struggling with an alert
  ✓ When an alert fires for the first time

RULES FOR GOOD RUNBOOKS:
  ✓ Written for 3am-brain: tired, stressed, unfamiliar
  ✓ Includes exact commands (don't make them remember syntax)
  ✓ Links to dashboards (don't make them search)
  ✓ Has explicit escalation paths (don't make them guess)
  ✓ Is kept up to date (stale runbooks are dangerous)
  ✓ Is tested by someone other than the author

ANTI-PATTERNS:
  ✗ "Check the logs" (which logs? where?)
  ✗ "Investigate further if needed" (needs more specifics)
  ✗ "Contact Alice" (Alice might be asleep or unavailable)
  ✗ Last updated 18 months ago (might be wrong now)
```

## 4. Runbook Automation: From Documentation to Code

The best runbook eventually becomes automated. If you find yourself running the same 5 commands every time an alert fires, those commands should run automatically.

```python
# Evolution of a runbook action:

# STAGE 1: Manual (documented in runbook)
# "Run: kubectl rollout restart deployment/payment-service"

# STAGE 2: Scripted
# scripts/restart_payment_service.sh:
#!/bin/bash
set -euo pipefail
DEPLOYMENT="payment-service"
kubectl rollout restart deployment/$DEPLOYMENT
echo "Waiting for rollout..."
kubectl rollout status deployment/$DEPLOYMENT --timeout=120s
echo "Verifying error rate..."
# Check Prometheus API
ERROR_RATE=$(curl -s 'http://prometheus:9090/api/v1/query?query=...' | jq '.data.result[0].value[1]')
if (( $(echo "$ERROR_RATE < 0.01" | bc -l) )); then
  echo "Error rate normal: $ERROR_RATE. Recovery successful."
else
  echo "Error rate still high: $ERROR_RATE. Escalating."
  exit 1
fi

# STAGE 3: Automated response (triggered by alert)
# PagerDuty webhook → Lambda → runs the script automatically
# Human is only involved if the automated response fails
# This eliminates a class of 3am pages entirely
```

## 5. Runbook as Code

Store runbooks with your code — not in a separate wiki that goes stale:

```
RECOMMENDED APPROACH:

Repository structure:
  services/
    payment-service/
      README.md
      runbooks/
        high-error-rate.md   ← specific runbooks
        db-connection.md
        deployment-rollback.md
  
  sre/
    runbooks/
      infrastructure/
        disk-full.md
        node-not-ready.md
      shared/
        rollback-deployment.md

BENEFITS:
  - Runbooks live next to the code they describe
  - Updated in the same PR as code changes that affect them
  - Code review catches stale runbooks
  - Git history shows when and why runbooks changed

INCLUDE RUNBOOK LINK IN ALERT:
  annotations:
    runbook_url: "https://github.com/org/repo/blob/main/services/payment-service/runbooks/high-error-rate.md"
  
  PagerDuty, Alertmanager, Grafana all support showing a runbook link
  directly in the alert notification.
```

## 6. Senior Engineer Perspective

**A runbook is an investment in sleep:**

Every hour you spend writing a good runbook saves multiple hours of future on-call investigation. The calculus is simple: a runbook takes 2 hours to write. Each incident it helps with saves 30 minutes. After 4 incidents (likely within 6 months), it's paid for itself. After 20 incidents, you've saved 10 hours of 3am misery.

**Write runbooks for the person who knows nothing about this service:**

The best test of a runbook: can someone who has never touched this service use it to resolve the incident? If the answer is no, the runbook is too dependent on implicit knowledge. Make it explicit. Spell out the exact commands. Link to the exact dashboard panels. Assume nothing.
