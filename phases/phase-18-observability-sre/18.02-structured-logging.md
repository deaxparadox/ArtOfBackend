## 1. Why Structured Logging? (Theory)

Traditional logging produces human-readable text. Structured logging produces machine-readable data (usually JSON). The difference sounds minor, but it's transformative at scale.

```
UNSTRUCTURED LOG (traditional):
  "2026-06-15 14:30:00 ERROR User 123 payment failed: Card declined, amount=99.99"
  
  Problems:
  - Different engineers format similarly named fields differently
  - Can't reliably extract "user_id" — it's buried in a sentence
  - Can't filter logs by amount > 50 (it's text, not a number)
  - Different date formats across services make correlation painful

STRUCTURED LOG (JSON):
  {
    "timestamp": "2026-06-15T14:30:00.123Z",
    "level": "error",
    "service": "payment-service",
    "user_id": "123",
    "event": "payment_failed",
    "amount": 99.99,
    "error": "Card declined",
    "trace_id": "abc-def-123"
  }
  
  Benefits:
  - Any field is immediately filterable/searchable
  - Types are preserved (amount is a number, not text)
  - Machine-parseable: Elasticsearch/Loki ingest it perfectly
  - Consistent across all services (agree on field names once)
```

## 2. Log Levels: When to Use Each

```
DEBUG:   Detailed diagnostic information
         Only in development or when debugging a production issue.
         Example: "Executing SQL: SELECT * FROM users WHERE id=123"
         Production: DISABLED (too much noise, expensive to store)
         
INFO:    Significant business events that completed normally
         Example: "User 123 placed order 456 for $99.99"
         Production: ENABLED
         
WARNING: Something unexpected happened but the request succeeded
         Operator should be aware but it's not urgent
         Example: "Payment retry #2 succeeded after initial failure"
         Production: ENABLED (alert if high volume)
         
ERROR:   A request failed. The system is degraded. Human action needed.
         Example: "Payment failed: database connection timed out"
         Production: ENABLED (alert immediately)
         
CRITICAL/FATAL: System is in danger. Immediate action needed.
         Example: "Cannot connect to primary database"
         Production: PAGE ON-CALL IMMEDIATELY
```

## 3. Implementing Structured Logging in Python

```python
import structlog
import logging
import sys

# Configure structlog for production (JSON output)
structlog.configure(
    processors=[
        structlog.contextvars.merge_contextvars,     # per-request context
        structlog.processors.add_log_level,
        structlog.processors.TimeStamper(fmt="iso", utc=True),
        structlog.processors.StackInfoRenderer(),
        structlog.dev.set_exc_info,
        structlog.processors.JSONRenderer(),          # JSON for production
    ],
    logger_factory=structlog.PrintLoggerFactory(file=sys.stdout),
)

logger = structlog.get_logger()

# BASIC USAGE:
logger.info("user_logged_in", user_id="123", ip_address="1.2.3.4")
# Output: {"level": "info", "timestamp": "2026-06-15T14:30:00Z", 
#           "event": "user_logged_in", "user_id": "123", "ip_address": "1.2.3.4"}

logger.error("payment_failed", user_id="123", amount=99.99, error="Card declined")
# Output: {"level": "error", ..., "event": "payment_failed", "amount": 99.99, ...}

# BIND CONTEXT: add fields to all subsequent logs in this scope
log = logger.bind(request_id="req-abc", user_id="123")
log.info("starting_order_processing")
log.info("inventory_checked", item_id="item-456")
log.info("payment_initiated", amount=99.99)
# All three logs will contain request_id and user_id automatically

# PER-REQUEST CONTEXT (FastAPI middleware):
import uuid
from contextvars import ContextVar

request_id_var: ContextVar[str] = ContextVar("request_id", default="")

async def log_middleware(request, call_next):
    request_id = str(uuid.uuid4())
    structlog.contextvars.clear_contextvars()
    structlog.contextvars.bind_contextvars(
        request_id=request_id,
        method=request.method,
        path=request.url.path,
    )
    
    response = await call_next(request)
    
    logger.info(
        "request_completed",
        status_code=response.status_code,
    )
    return response
```

## 4. What to Log (and What NOT to Log)

```
ALWAYS LOG:
  ✓ Business events: user signed up, order placed, payment succeeded/failed
  ✓ System events: service started, config loaded, background job started/completed
  ✓ Errors with context: what failed, who it affected, what the system state was
  ✓ External calls: API calls to third parties, database queries (at DEBUG level)
  ✓ Security events: login attempts (success/failure), permission denied, token refresh
  
NEVER LOG:
  ✗ Passwords (even hashed ones)
  ✗ Full credit card numbers (PCI-DSS compliance)
  ✗ API keys or tokens
  ✗ Social security numbers or passport numbers
  ✗ Full request/response bodies in production (may contain sensitive data)
  
  Scrubbing sensitive data:
  
  SAFE: log last 4 of card: "card_last4": "4242"
  SAFE: log token prefix: "token_prefix": "sk_live_abc..."
  SAFE: log email domain only: "email_domain": "gmail.com"
  UNSAFE: "email": "user@gmail.com"  (PII in most jurisdictions)
  UNSAFE: "password": "hunter2"      (never, ever)

LOG SIZE DISCIPLINE:
  Long log messages become expensive at scale.
  1 million requests/day × 1KB per log = 1GB of logs per day
  Keep log fields minimal and meaningful.
  Avoid logging full request bodies or large payloads.
```

## 5. Structured Logging in Node.js (Pino)

```javascript
const pino = require('pino');

// Production configuration: JSON output, fast
const logger = pino({
    level: process.env.LOG_LEVEL || 'info',
    formatters: {
        level: (label) => ({ level: label }),
    },
    timestamp: pino.stdTimeFunctions.isoTime,
});

// Usage
logger.info({ user_id: '123', event: 'user_login' }, 'User logged in');
// Output: {"level":"info","time":"2026-06-15T14:30:00.123Z","user_id":"123","event":"user_login","msg":"User logged in"}

logger.error({ err, user_id: '123', amount: 99.99 }, 'Payment failed');
// Pino serializes 'err' with stack trace automatically

// Child logger: bind context for a request
const reqLogger = logger.child({ request_id: 'req-abc', user_id: '123' });
reqLogger.info({ event: 'order_processing_started' }, 'Processing order');
// request_id and user_id appear in every log from reqLogger
```

## 6. Senior Engineer Perspective

**Correlation IDs are worth more than anything else:**

A single `trace_id` field that appears in every log line for a request — and gets passed through every downstream service call — is the highest-value addition to any logging system. Without it, debugging a failed request requires correlating timestamps across services. With it, one query retrieves the complete story of that request.

Implement correlation IDs first, before adding any other observability tooling. It's one day of work and it will save hundreds of debugging hours.

**Log events, not code paths:**

Bad: `logger.info("Entered processPayment function")`
Good: `logger.info("payment_started", user_id=..., amount=...)`

The first tells you what the code is doing. The second tells you what the business is doing. When you're debugging a production incident, you want to understand what happened from a business perspective. Code path logs are noise; business event logs are signal.
