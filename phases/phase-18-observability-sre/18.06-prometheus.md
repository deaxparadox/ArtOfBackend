## 1. What Is Prometheus? (Theory)

Prometheus (SoundCloud, 2012; CNCF graduated 2018) is a pull-based metrics system. Instead of services pushing metrics to a central server, Prometheus **scrapes** metrics from your services on a schedule (every 15 seconds by default).

**Pull vs. push — why Prometheus chose pull:**

```
PUSH (StatsD, Graphite, Datadog agent):
  Services actively send metrics to the central server.
  Problem: if the central server is down, metrics are lost.
  Problem: hard to tell if a service is down vs. just sending no metrics.

PULL (Prometheus):
  Prometheus visits each service's /metrics endpoint.
  If a service doesn't respond: Prometheus knows it's down.
  If Prometheus is down: service is unaffected, metrics accumulate.
  Easier to test: curl http://service/metrics and see what would be scraped.
  
  One practical downside: services must be reachable from Prometheus.
  For short-lived jobs (cron, batch): use Pushgateway as a middleman.
```

## 2. How Prometheus Works

```
PROMETHEUS ARCHITECTURE:

Service A          Service B          Service C
[/metrics]         [/metrics]         [/metrics]
    ↑                  ↑                  ↑
    └──────────────────┼──────────────────┘
                       │ HTTP scrape every 15s
                  ┌────▼─────┐
                  │Prometheus │
                  │ server   │
                  │ (TSDB)   │
                  └────┬─────┘
                       │
                  ┌────▼─────┐    ┌──────────────┐
                  │ Alert-   │───→│  PagerDuty   │
                  │ manager  │    │  Slack       │
                  └──────────┘    └──────────────┘
                       │
                  ┌────▼─────┐
                  │  Grafana  │ (visualization)
                  └──────────┘

SERVICE DISCOVERY:
  Static: list services in prometheus.yml
  Dynamic: Kubernetes service discovery, Consul, EC2, etc.
  Prometheus finds new pods/services automatically.
```

## 3. Instrumenting Your Application

```python
from prometheus_client import Counter, Histogram, Gauge, start_http_server
import time

# Define metrics
REQUEST_COUNT = Counter(
    'http_requests_total',
    'Total HTTP requests',
    ['method', 'endpoint', 'status_code'],
)
REQUEST_LATENCY = Histogram(
    'request_duration_seconds',
    'HTTP request latency',
    ['endpoint'],
    buckets=[.005, .01, .025, .05, .1, .25, .5, 1, 2.5, 5],
)
ACTIVE_REQUESTS = Gauge('active_requests', 'Currently active requests')

# FastAPI integration
from fastapi import FastAPI, Request
import time

app = FastAPI()

@app.middleware("http")
async def metrics_middleware(request: Request, call_next):
    ACTIVE_REQUESTS.inc()
    start = time.time()
    
    response = await call_next(request)
    
    duration = time.time() - start
    REQUEST_COUNT.labels(
        method=request.method,
        endpoint=request.url.path,
        status_code=str(response.status_code),
    ).inc()
    REQUEST_LATENCY.labels(endpoint=request.url.path).observe(duration)
    ACTIVE_REQUESTS.dec()
    
    return response

# Expose /metrics endpoint
from prometheus_client import generate_latest, CONTENT_TYPE_LATEST
from fastapi.responses import Response

@app.get("/metrics")
def metrics():
    return Response(generate_latest(), media_type=CONTENT_TYPE_LATEST)
```

## 4. PromQL: Prometheus Query Language

PromQL is the query language for Prometheus. It's powerful but has a learning curve.

```promql
# INSTANT VECTOR QUERIES (current value):
http_requests_total                          # all time series for this metric
http_requests_total{status_code="500"}       # filter by label
http_requests_total{endpoint=~"/api/.*"}     # regex label match

# RANGE VECTOR QUERIES (data over a time window):
http_requests_total[5m]                      # all values over last 5 minutes
                                             # (used as input to functions)

# RATE AND INCREASE:
rate(http_requests_total[5m])                # per-second rate over 5m
increase(http_requests_total[1h])            # total increase over last hour

# AGGREGATION:
sum(rate(http_requests_total[5m]))           # total request rate across all instances
sum by (endpoint) (rate(http_requests_total[5m]))   # request rate per endpoint
avg(active_connections)                       # average across all instances

# ERROR RATE (fraction of requests that are errors):
sum(rate(http_requests_total{status_code=~"5.."}[5m]))
  /
sum(rate(http_requests_total[5m]))

# P99 LATENCY:
histogram_quantile(
  0.99,
  sum by (le, endpoint) (rate(request_duration_seconds_bucket[5m]))
)

# CPU USAGE PERCENTAGE:
100 - (avg by (instance) (rate(node_cpu_seconds_total{mode="idle"}[5m])) * 100)

# MEMORY USAGE:
node_memory_MemAvailable_bytes / node_memory_MemTotal_bytes
```

## 5. Alerting Rules

```yaml
# prometheus/alerts.yml — alerting rules

groups:
  - name: application_alerts
    rules:
      # Alert when error rate > 1% for 5 minutes
      - alert: HighErrorRate
        expr: |
          sum(rate(http_requests_total{status_code=~"5.."}[5m]))
          /
          sum(rate(http_requests_total[5m]))
          > 0.01
        for: 5m   # must be true for 5 minutes before firing
        labels:
          severity: critical
        annotations:
          summary: "High error rate on {{ $labels.service }}"
          description: "Error rate is {{ $value | humanizePercentage }}"

      # Alert when p99 latency > 500ms for 5 minutes
      - alert: HighLatency
        expr: |
          histogram_quantile(
            0.99,
            sum by (le, endpoint) (rate(request_duration_seconds_bucket[5m]))
          ) > 0.5
        for: 5m
        labels:
          severity: warning
        annotations:
          summary: "High p99 latency: {{ $value | humanizeDuration }}"

      # Alert when a service goes down
      - alert: ServiceDown
        expr: up == 0
        for: 1m
        labels:
          severity: critical
        annotations:
          summary: "Service {{ $labels.job }} is down"

      # Alert when disk is getting full
      - alert: DiskSpaceLow
        expr: |
          (node_filesystem_avail_bytes{fstype!~"tmpfs|fuse.lxcfs"}
           / node_filesystem_size_bytes) < 0.15
        for: 10m
        labels:
          severity: warning
        annotations:
          summary: "Disk space low: {{ $value | humanizePercentage }} free"
```

## 6. Prometheus Configuration

```yaml
# prometheus.yml

global:
  scrape_interval: 15s        # how often to scrape
  evaluation_interval: 15s    # how often to evaluate alert rules

alerting:
  alertmanagers:
    - static_configs:
        - targets: ['alertmanager:9093']

rule_files:
  - "alerts.yml"

scrape_configs:
  # Scrape your own metrics (meta)
  - job_name: 'prometheus'
    static_configs:
      - targets: ['localhost:9090']

  # Scrape your services
  - job_name: 'api-gateway'
    static_configs:
      - targets: ['api-gateway:8080']
    metrics_path: /metrics

  # Kubernetes: auto-discover pods with specific annotation
  - job_name: 'kubernetes-pods'
    kubernetes_sd_configs:
      - role: pod
    relabel_configs:
      - source_labels: [__meta_kubernetes_pod_annotation_prometheus_io_scrape]
        action: keep
        regex: 'true'
```

## 7. Senior Engineer Perspective

**The 5-minute window in rate() is the most common mistake:**

New Prometheus users often use `rate(metric[1m])` for more responsive dashboards. But Prometheus needs at least 2 scrape intervals in the window for a reliable rate. With 15s scrape intervals and a 1-minute window, you only have 4 data points — one missed scrape causes a 25% gap. Use `[5m]` (20 data points) for stable rates. Use `[1m]` only if you accept noisier graphs.

**Cardinality is the Prometheus killer:**

The most common Prometheus failure mode is cardinality explosion: someone adds `user_id` or `request_id` as a label. Prometheus creates a new time series per unique label value. With 1 million users × 10 metrics × 5 labels = 50 million time series. Memory and disk usage spike. Prometheus crashes. Always review new label additions in code review.
