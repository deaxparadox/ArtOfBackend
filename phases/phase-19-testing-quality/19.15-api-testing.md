## 1. What Is API Testing? (Theory)

API testing means sending real HTTP requests to your API and verifying the responses. Unlike unit tests (which mock everything) or E2E tests (which use a browser), API tests exercise your HTTP layer directly.

**The role of API testing tools:**

```
EXPLORATORY TESTING:
  "I want to try this endpoint manually while building it"
  
  curl works but is stateless — you'd have to re-type everything.
  API clients let you save requests, organize them into collections,
  and run them repeatedly with one click.

AUTOMATED API TESTING:
  Write tests with assertions, run them in CI.
  "POST /users with valid data returns 201"
  "GET /users/99999 returns 404"
  "Authentication endpoints reject invalid tokens"

CONTRACT VERIFICATION:
  "The API response matches the documented schema"
  Works with OpenAPI/Swagger specs.
```

## 2. Postman

Postman is the most widely used API client. It's GUI-based, supports collections, environments, and scripted tests.

```javascript
// Postman test script (runs after receiving response)
// Written in JavaScript, runs in Postman's sandbox

// TEST STRUCTURE:
pm.test("Status code is 201", function () {
    pm.response.to.have.status(201);
});

pm.test("Response has user ID", function () {
    const json = pm.response.json();
    pm.expect(json.id).to.be.a('string');
    pm.expect(json.id).to.not.be.empty;
});

pm.test("Response has correct email", function () {
    const json = pm.response.json();
    pm.expect(json.email).to.equal(pm.variables.get("test_email"));
});

pm.test("Created_at is a valid date", function () {
    const json = pm.response.json();
    const date = new Date(json.created_at);
    pm.expect(date.toString()).to.not.equal('Invalid Date');
});

// CHAINING REQUESTS: save values from one response for the next
pm.test("Save user ID for next request", function () {
    const json = pm.response.json();
    pm.environment.set("user_id", json.id);
    // Later requests can use {{user_id}} in URL or body
});
```

```bash
# Run Postman collection in CI with Newman
npm install -g newman

newman run postman_collection.json \
    --environment staging.json \
    --reporters cli,junit \
    --reporter-junit-export test-results.xml
```

## 3. Bruno: The Git-Friendly Alternative

Bruno stores collections as plain text files — not JSON in a proprietary format. Collections live in your repository alongside code.

```javascript
// users/create-user.bru
meta {
  name: Create User
  type: http
  seq: 1
}

post {
  url: {{baseUrl}}/api/users
  body: json
  auth: none
}

headers {
  Content-Type: application/json
}

body:json {
  {
    "email": "{{testEmail}}",
    "name": "Test User"
  }
}

tests {
  test("should return 201", function() {
    expect(res.status).to.equal(201);
  });
  
  test("should return created user", function() {
    expect(res.body.email).to.equal(bru.getEnvVar("testEmail"));
    expect(res.body.id).to.be.a("string");
  });
  
  // Save for chained requests
  test("save user id", function() {
    bru.setEnvVar("userId", res.body.id);
  });
}
```

```bash
# environments/staging.json
{
    "name": "Staging",
    "variables": {
        "baseUrl": "https://staging.yourcompany.com",
        "testEmail": "test@example.com"
    }
}

# Run in CI
npx @usebruno/cli run --env Staging users/
```

**Bruno advantages:**
- Collections are plain text files (`.bru` files) — diff-able in git
- No account required
- Open source
- Can be committed alongside the API code

## 4. Hoppscotch

Hoppscotch is an open-source web-based API client. Notable for being entirely in-browser (no install), self-hostable, and having a clean modern UI.

```
HOPPSCOTCH FEATURES:
  Web-based: works in any browser, no install
  REST, GraphQL, WebSocket, Server-Sent Events support
  Collections and environments
  Real-time SSE/WebSocket testing
  Import from Postman, Swagger/OpenAPI
  CLI tool: hopp for CI integration
  Self-hostable: deploy on your own infrastructure
  
BEST FOR:
  Teams that want open-source + self-hosting
  Quick prototyping without installing anything
  Testing WebSockets and SSE (better than Postman)
  GraphQL testing with schema introspection
```

## 5. API Testing in CI/CD

```yaml
# GitHub Actions: run API tests against staging on deploy

name: API Tests
on:
  workflow_dispatch:     # manual trigger
  deployment:           # on deploy to staging

jobs:
  api-tests:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      
      - name: Wait for deployment
        run: |
          for i in {1..30}; do
            if curl -s ${{ vars.STAGING_URL }}/health | grep -q '"status":"ok"'; then
              echo "Service is up"
              break
            fi
            echo "Waiting... ($i/30)"
            sleep 10
          done
      
      - name: Run Bruno API tests
        uses: nickvdyck/newman-action@v1
        with:
          collection: api-tests/collection.json
          environment: api-tests/environments/staging.json
          fail-on-error: true
```

## 6. OpenAPI-Driven Testing

```python
# Validate that your API matches its OpenAPI spec
# pip install schemathesis

import schemathesis

# Load your OpenAPI spec
schema = schemathesis.from_uri("http://localhost:8080/openapi.json")

# Schemathesis auto-generates test cases from your spec
# and verifies all responses match the documented schema

@schema.parametrize()
def test_api_schema_compliance(case):
    """
    For every endpoint in the OpenAPI spec:
    - Generate valid request data
    - Send the request
    - Verify the response matches the declared response schema
    """
    response = case.call()
    case.validate_response(response)

# This tests:
# - Every endpoint returns the documented status codes
# - Response bodies match documented schemas (no undocumented fields)
# - Required response fields are always present
# - Field types match (string stays string, integer stays integer)

# Run: pytest test_schema.py -v
# If your API returns a field with wrong type: schemathesis catches it
```

## 7. Senior Engineer Perspective

**Bruno is the right choice for new projects:**

Postman collections stored in the cloud are disconnected from your codebase — when someone updates the API, the Postman collection might not be updated. Bruno's file-based format solves this: collections live in the same repo as the code. A PR that changes an API endpoint can include updates to the API test collection in the same commit. Review, version, and audit them together.

**API tests are valuable for: documenting expected behavior, regression testing after changes, and onboarding new engineers. They're NOT a replacement for integration tests:** API tests test the HTTP layer; integration tests test the logic underneath. You need both. An API test verifies the response format; an integration test verifies the data was actually saved to the database correctly.
