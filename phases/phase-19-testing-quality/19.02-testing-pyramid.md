## 1. The Testing Pyramid (Theory)

The testing pyramid (Mike Cohn, 2009) is a model for how to distribute tests across different levels. Tests at the base of the pyramid are many, fast, and cheap. Tests at the top are few, slow, and expensive.

```
              ┌─────────────────┐
              │   E2E / Browser  │  (few, slow, expensive)
              │                  │
              │  Selenium /      │
              │  Playwright      │
              └────────┬─────────┘
                       │
              ┌────────┴─────────┐
              │    Integration   │  (moderate, seconds, real dependencies)
              │                  │
              │  Real database,  │
              │  real HTTP calls │
              └────────┬─────────┘
                       │
              ┌────────┴─────────┐
              │     Unit Tests   │  (many, milliseconds, isolated)
              │                  │
              │  Mocked          │
              │  dependencies    │
              └──────────────────┘

IDEAL DISTRIBUTION:
  Unit tests:        70% of test suite
  Integration tests: 20%
  E2E tests:         10%
```

## 2. Unit Tests

Unit tests test a single function, method, or class in isolation. External dependencies are replaced with mocks or stubs.

```python
# Unit test example: testing a pricing function

def calculate_order_total(items: list[dict], discount_code: str = None) -> float:
    """
    items: [{"price": 10.00, "quantity": 2}, ...]
    discount_code: optional coupon code
    """
    subtotal = sum(item["price"] * item["quantity"] for item in items)
    
    discount = 0.0
    if discount_code == "SAVE10":
        discount = subtotal * 0.10
    elif discount_code == "SAVE20":
        discount = subtotal * 0.20
    
    return round(subtotal - discount, 2)

# TEST (no database, no HTTP, no external dependencies):
class TestCalculateOrderTotal:
    def test_basic_calculation(self):
        items = [{"price": 10.00, "quantity": 2}]
        assert calculate_order_total(items) == 20.00
    
    def test_multiple_items(self):
        items = [
            {"price": 10.00, "quantity": 1},
            {"price": 5.00, "quantity": 3},
        ]
        assert calculate_order_total(items) == 25.00
    
    def test_discount_10_percent(self):
        items = [{"price": 100.00, "quantity": 1}]
        assert calculate_order_total(items, "SAVE10") == 90.00
    
    def test_discount_20_percent(self):
        items = [{"price": 100.00, "quantity": 1}]
        assert calculate_order_total(items, "SAVE20") == 80.00
    
    def test_invalid_discount_code(self):
        items = [{"price": 100.00, "quantity": 1}]
        assert calculate_order_total(items, "INVALID") == 100.00
    
    def test_empty_order(self):
        assert calculate_order_total([]) == 0.00

# CHARACTERISTICS:
# Runs in < 1ms per test
# No database, no file system, no network
# Deterministic: same input always same output
# Can run 10,000 of these in 2 seconds
```

## 3. Integration Tests

Integration tests verify that your code works with real external dependencies: a real database, a real Redis instance, real HTTP endpoints.

```python
# Integration test example: testing a repository with a real database
# Uses pytest-asyncio + real PostgreSQL (from Docker or test env)

import pytest
import asyncpg
from your_app.repositories import UserRepository

@pytest.fixture
async def db_pool():
    """Create a real database pool for testing."""
    pool = await asyncpg.create_pool(
        "postgresql://test:test@localhost:5432/testdb"
    )
    yield pool
    await pool.close()

@pytest.fixture
async def clean_db(db_pool):
    """Clean database before each test."""
    async with db_pool.acquire() as conn:
        await conn.execute("TRUNCATE users CASCADE")
    yield

class TestUserRepository:
    @pytest.mark.asyncio
    async def test_create_user(self, db_pool, clean_db):
        repo = UserRepository(db_pool)
        user = await repo.create(email="alice@example.com", name="Alice")
        
        assert user.id is not None
        assert user.email == "alice@example.com"
        
        # Verify it actually persisted
        fetched = await repo.get_by_id(user.id)
        assert fetched.email == "alice@example.com"
    
    @pytest.mark.asyncio
    async def test_unique_email_constraint(self, db_pool, clean_db):
        repo = UserRepository(db_pool)
        await repo.create(email="alice@example.com", name="Alice")
        
        with pytest.raises(UniqueViolationError):
            await repo.create(email="alice@example.com", name="Alice 2")
    
    @pytest.mark.asyncio
    async def test_get_by_email(self, db_pool, clean_db):
        repo = UserRepository(db_pool)
        await repo.create(email="alice@example.com", name="Alice")
        
        user = await repo.get_by_email("alice@example.com")
        assert user.name == "Alice"

# CHARACTERISTICS:
# Runs in 50-200ms (real DB calls)
# Tests that your SQL is correct
# Tests that your schema matches your code
# Can't be done with mocks (mocks don't test real DB behavior)
```

## 4. End-to-End (E2E) Tests

E2E tests test the complete user journey through the real system. They're the most realistic but also the slowest and most brittle.

```python
# E2E test with Playwright (browser automation)

import pytest
from playwright.sync_api import Page

@pytest.mark.e2e
def test_user_can_checkout(page: Page):
    """Test the full checkout flow."""
    
    # Step 1: Log in
    page.goto("https://staging.yourapp.com/login")
    page.fill('[data-testid="email"]', "test@example.com")
    page.fill('[data-testid="password"]', "password123")
    page.click('[data-testid="submit"]')
    
    # Step 2: Add item to cart
    page.goto("https://staging.yourapp.com/products/widget-pro")
    page.click('[data-testid="add-to-cart"]')
    
    # Step 3: Checkout
    page.click('[data-testid="cart-icon"]')
    page.click('[data-testid="checkout"]')
    
    # Verify: order confirmation page
    page.wait_for_url("**/order-confirmation/**")
    assert page.locator('[data-testid="confirmation-number"]').is_visible()
    
# CHARACTERISTICS:
# Runs in 20-60 seconds
# Tests the complete user experience
# Catches integration failures (frontend + backend + DB)
# Most brittle: fails from network issues, timing, UI changes
# Use sparingly for critical user journeys only
```

## 5. The "Testing Trophy" Alternative

Kent C. Dodds proposed a modified model: the "testing trophy," which puts more weight on integration tests:

```
        ┌────────────────────────────────┐
        │         E2E (few)              │
        └────────────────────────────────┘
        ┌────────────────────────────────────────┐
        │         Integration (most)             │  ← most valuable
        └────────────────────────────────────────┘
        ┌──────────────────────────────────┐
        │          Unit (some)             │
        └──────────────────────────────────┘
        ┌──────────────────────────────┐
        │      Static analysis         │  (TypeScript, mypy, linting)
        └──────────────────────────────┘

ARGUMENT:
  Integration tests give the best confidence per test written.
  Unit tests can give false confidence (everything mocked might not reflect reality).
  The boundary between "unit" and "integration" is blurry anyway.
  
  In practice: both pyramids lead to similar test suites.
  The principle is the same: more small tests, fewer big ones.
```

## 6. Senior Engineer Perspective

**The pyramid is a guide, not a rule:**

The exact ratio depends on your application. A pure business logic library (no I/O) benefits heavily from unit tests. A data pipeline that's mostly I/O benefits heavily from integration tests. Don't force your tests into the pyramid — let the pyramid inform your distribution.

**The most dangerous anti-pattern is the "ice cream cone":**

```
      (wide at top)
    ┌─────────────────────────────────────────┐
    │         E2E (too many!)                  │  ← expensive, brittle
    └─────────────────────────────────────────┘
         ┌──────────────────────────────┐
         │      Integration (some)      │
         └──────────────────────────────┘
                  ┌──────────────┐
                  │  Unit (few)  │  ← few
                  └──────────────┘
```

Teams that only write E2E tests end up with a test suite that takes 2 hours to run, fails intermittently for unrelated reasons (network flakiness, timing issues), and catches bugs only at the system level. These tests become a burden rather than a safety net.
