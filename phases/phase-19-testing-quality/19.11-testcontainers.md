## 1. What Are Testcontainers? (Theory)

Testcontainers is a library that runs Docker containers during tests — giving you real databases, message queues, and other services without permanent local installations or shared test environments.

The problem it solves: integration tests need real services (PostgreSQL, Redis, Kafka), but those services shouldn't be shared across developers or CI environments — shared state causes flaky tests and "works on my machine" failures.

```
WITHOUT TESTCONTAINERS:
  Developer A runs tests → modifies shared test database
  Developer B runs tests simultaneously → data conflict, test failures
  
  Or: each developer installs and manages their own PostgreSQL
      → "I can't run tests, PostgreSQL isn't running"
      → Different versions, different configurations
  
WITH TESTCONTAINERS:
  Each test session starts a fresh Docker container.
  Container is isolated: no shared state.
  Container matches production version exactly.
  Container is destroyed after tests complete.
  Any developer with Docker can run tests immediately.
```

## 2. Python Testcontainers

```python
# pip install testcontainers

import pytest
import asyncpg
from testcontainers.postgres import PostgresContainer
from testcontainers.redis import RedisContainer

@pytest.fixture(scope="session")
def postgres():
    """Start PostgreSQL container for the test session."""
    with PostgresContainer("postgres:16-alpine") as pg:
        yield pg
    # Container automatically stopped and removed after yield

@pytest.fixture(scope="session")
def redis():
    """Start Redis container for the test session."""
    with RedisContainer("redis:7-alpine") as redis:
        yield redis

@pytest.fixture(scope="session")
async def db_pool(postgres):
    """Connection pool to the containerized database."""
    pool = await asyncpg.create_pool(postgres.get_connection_url())
    
    # Run schema migrations
    async with pool.acquire() as conn:
        await conn.execute("""
            CREATE TABLE IF NOT EXISTS users (
                id BIGSERIAL PRIMARY KEY,
                email TEXT UNIQUE NOT NULL,
                name TEXT NOT NULL,
                created_at TIMESTAMPTZ DEFAULT NOW()
            );
            CREATE TABLE IF NOT EXISTS orders (
                id BIGSERIAL PRIMARY KEY,
                user_id BIGINT REFERENCES users(id),
                total NUMERIC(10,2) NOT NULL,
                created_at TIMESTAMPTZ DEFAULT NOW()
            );
        """)
    
    yield pool
    await pool.close()

@pytest.fixture(autouse=True)
async def clean_database(db_pool):
    """Truncate all tables before each test."""
    async with db_pool.acquire() as conn:
        await conn.execute("TRUNCATE users, orders CASCADE")
    yield

# Integration tests that use real PostgreSQL:
class TestUserRepository:
    @pytest.mark.asyncio
    async def test_create_user(self, db_pool):
        repo = UserRepository(db_pool)
        user = await repo.create("alice@example.com", "Alice")
        
        assert user.id is not None
        assert user.email == "alice@example.com"
    
    @pytest.mark.asyncio
    async def test_unique_constraint(self, db_pool):
        repo = UserRepository(db_pool)
        await repo.create("alice@example.com", "Alice")
        
        with pytest.raises(asyncpg.UniqueViolationError):
            await repo.create("alice@example.com", "Alice 2")
```

## 3. Java/Kotlin Testcontainers

```java
// Testcontainers for Java/Spring Boot

import org.testcontainers.containers.PostgreSQLContainer;
import org.testcontainers.junit.jupiter.Container;
import org.testcontainers.junit.jupiter.Testcontainers;

@Testcontainers  // annotation activates Testcontainers support
class UserRepositoryTest {
    
    @Container  // starts/stops automatically per test class
    static PostgreSQLContainer<?> postgres = new PostgreSQLContainer<>("postgres:16-alpine")
        .withDatabaseName("testdb")
        .withUsername("test")
        .withPassword("test");
    
    @DynamicPropertySource
    static void configureProperties(DynamicPropertyRegistry registry) {
        // Override Spring datasource with container's connection details
        registry.add("spring.datasource.url", postgres::getJdbcUrl);
        registry.add("spring.datasource.username", postgres::getUsername);
        registry.add("spring.datasource.password", postgres::getPassword);
    }
    
    @Autowired
    private UserRepository userRepository;
    
    @Test
    void createsUserSuccessfully() {
        User user = new User("alice@example.com", "Alice");
        User saved = userRepository.save(user);
        
        assertThat(saved.getId()).isNotNull();
        
        Optional<User> found = userRepository.findByEmail("alice@example.com");
        assertThat(found).isPresent();
        assertThat(found.get().getName()).isEqualTo("Alice");
    }
    
    @Test
    void rejectssDuplicateEmail() {
        userRepository.save(new User("alice@example.com", "Alice"));
        
        assertThatThrownBy(() -> 
            userRepository.save(new User("alice@example.com", "Alice 2"))
        ).isInstanceOf(DataIntegrityViolationException.class);
    }
}
```

## 4. Node.js Testcontainers

```typescript
// @testcontainers/testcontainers

import { PostgreSqlContainer } from '@testcontainers/postgresql';
import { GenericContainer } from 'testcontainers';

describe('UserRepository', () => {
    let pgContainer: PostgreSqlContainer;
    let pool: Pool;
    
    beforeAll(async () => {
        // Start PostgreSQL container
        pgContainer = await new PostgreSqlContainer('postgres:16-alpine')
            .withDatabase('testdb')
            .start();
        
        // Create connection pool
        pool = new Pool({
            host: pgContainer.getHost(),
            port: pgContainer.getMappedPort(5432),
            database: pgContainer.getDatabase(),
            user: pgContainer.getUsername(),
            password: pgContainer.getPassword(),
        });
        
        // Run migrations
        await pool.query(`
            CREATE TABLE users (
                id BIGSERIAL PRIMARY KEY,
                email TEXT UNIQUE NOT NULL,
                name TEXT NOT NULL
            )
        `);
    });
    
    afterAll(async () => {
        await pool.end();
        await pgContainer.stop();
    });
    
    beforeEach(async () => {
        await pool.query('TRUNCATE users');
    });
    
    test('creates and retrieves user', async () => {
        const repo = new UserRepository(pool);
        const created = await repo.create({ email: 'alice@example.com', name: 'Alice' });
        
        const found = await repo.findById(created.id);
        expect(found?.email).toBe('alice@example.com');
    });
});
```

## 5. Performance Optimization

```python
# SLOW: start container per test (too expensive)
@pytest.fixture  # function scope = per test
def postgres():
    with PostgresContainer("postgres:16") as pg:
        yield pg

# FAST: start container once per session, clean data per test
@pytest.fixture(scope="session")   # one container, all tests
def postgres():
    with PostgresContainer("postgres:16") as pg:
        yield pg

@pytest.fixture(autouse=True)      # clean data, not container
async def clean_db(db_pool):
    await db_pool.execute("TRUNCATE ALL")
    yield

# BENCHMARK:
# Container startup: 3-8 seconds
# Table truncation: 5-50 milliseconds
# 
# 100 tests, session-scoped container:
#   Startup: 5s + 100 × 20ms = 7 seconds total
# 
# 100 tests, function-scoped container:
#   Startup: 100 × 5s = 500 seconds total
# 
# Session scope is 70x faster for 100 tests.
```

## 6. Senior Engineer Perspective

**Testcontainers end the "works on my machine" problem for databases:**

Before Testcontainers, integration tests required every developer to install and configure the exact right database version. CI environments needed separate database management. Test environments accumulated state. Testcontainers eliminates all of this: the test brings its own infrastructure, isolated per run, identical everywhere.

This is one of the highest-value testing investments you can make. A few hours setting up Testcontainers saves hundreds of hours of "why does this test fail on CI but not locally?" debugging over time.

**Use real databases for the tests that matter:**

The argument for mocking the database is speed. The argument for Testcontainers is correctness. With session-scoped containers and table truncation between tests, the speed difference is minimal (under 30 seconds for most test suites) while the confidence is dramatically higher. SQL behavior, index performance, constraint enforcement — all tested as they will actually behave in production.
