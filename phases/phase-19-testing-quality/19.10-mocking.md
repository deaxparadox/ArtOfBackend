## 1. Test Doubles (Theory)

"Mock" is often used to mean any replacement for a real dependency in tests. But there are distinct types of test doubles, each with a specific purpose. Knowing the difference helps you choose the right tool.

**The four types of test doubles:**

```
STUB:
  Returns predefined responses to specific calls.
  Purpose: control what a dependency returns.
  Doesn't verify how it was called.
  
  "Whenever getUser('u-123') is called, return this User object"
  
MOCK:
  Like a stub, but ALSO verifies that it was called correctly.
  Purpose: verify your code interacts with a dependency as expected.
  
  "Charge was called once with amount=9999"
  
FAKE:
  A simplified working implementation of the real dependency.
  Purpose: test behavior that requires state tracking.
  
  "An in-memory database implementation"
  
SPY:
  Wraps a real implementation and records calls.
  Purpose: observe how real code is called without replacing it.
  
  "The real EmailSender, but we're recording how many times it was called"
```

## 2. Stubs: Control Dependencies

```python
from unittest.mock import AsyncMock

# STUB: control what a dependency returns
async def test_user_not_found():
    mock_db = AsyncMock()
    mock_db.get_user.return_value = None  # stub: always returns None
    
    service = UserService(db=mock_db)
    
    with pytest.raises(UserNotFoundError):
        await service.process_payment(user_id="u-999", amount=99.99)
    
    # We don't care HOW mock_db was called, just that the behavior is correct
    # when get_user returns None.
```

## 3. Mocks: Verify Interactions

```python
from unittest.mock import AsyncMock

# MOCK: verify how code interacts with dependencies
async def test_payment_creates_stripe_charge():
    mock_stripe = AsyncMock()
    mock_stripe.create_charge.return_value = {"id": "ch_abc123"}
    
    mock_db = AsyncMock()
    mock_db.get_user.return_value = User(id="u-123", stripe_id="cus_abc")
    mock_db.save_payment.return_value = None
    
    service = PaymentService(stripe=mock_stripe, db=mock_db)
    await service.charge(user_id="u-123", amount=99.99)
    
    # VERIFICATION: how was the mock called?
    mock_stripe.create_charge.assert_called_once_with(
        amount=9999,            # amount in cents
        customer="cus_abc",
        description="Order payment",
    )
    mock_db.save_payment.assert_called_once()
    
    # Verify NOT called (security check: stripe should not be called twice)
    assert mock_stripe.create_charge.call_count == 1
```

## 4. Fakes: Working Implementations

```python
# FAKE: in-memory database for testing
class FakeUserRepository:
    """In-memory implementation of UserRepository interface."""
    
    def __init__(self):
        self._users: dict[str, User] = {}
        self._next_id = 1
    
    async def create(self, email: str, name: str) -> User:
        for user in self._users.values():
            if user.email == email:
                raise DuplicateEmailError(email)
        
        user_id = str(self._next_id)
        self._next_id += 1
        user = User(id=user_id, email=email, name=name)
        self._users[user_id] = user
        return user
    
    async def get_by_id(self, user_id: str) -> User | None:
        return self._users.get(user_id)
    
    async def get_by_email(self, email: str) -> User | None:
        for user in self._users.values():
            if user.email == email:
                return user
        return None
    
    async def list_all(self) -> list[User]:
        return list(self._users.values())

# USING THE FAKE:
class TestUserService:
    async def test_duplicate_email_rejected(self):
        repo = FakeUserRepository()  # works like a real repo, but in memory
        service = UserService(repo)
        
        await service.create_user("alice@example.com", "Alice")
        
        with pytest.raises(DuplicateEmailError):
            await service.create_user("alice@example.com", "Alice 2")
    
    async def test_list_returns_all_users(self):
        repo = FakeUserRepository()
        service = UserService(repo)
        
        await service.create_user("alice@example.com", "Alice")
        await service.create_user("bob@example.com", "Bob")
        
        users = await service.list_users()
        assert len(users) == 2

# WHEN TO USE FAKES:
# Tests that need to maintain state across multiple operations
# The dependency has complex behavior (caching, uniqueness constraints)
# You want tests to be fast and independent from real databases
# Many tests share the same fake (it becomes an investment)
```

## 5. Spies: Observe Without Replacing

```python
from unittest.mock import patch, call

# SPY: wraps real implementation, records calls
class TestEmailDelivery:
    def test_sends_exactly_one_welcome_email(self):
        real_email_service = EmailService()
        
        # spy: wrap the real send method, record calls
        with patch.object(real_email_service, 'send', wraps=real_email_service.send) as spy:
            user_service = UserService(email=real_email_service)
            user_service.create_user("alice@example.com", "Alice")
            
            # verify it was called once (real email was actually sent)
            spy.assert_called_once()
            assert spy.call_args.args[0] == "alice@example.com"
```

## 6. The Mock/Don't-Mock Decision Tree

```
SHOULD I MOCK THIS?

Does it call a real external service (HTTP, database, message queue)?
  YES → Mock at the boundary (don't hit real services in unit tests)
  NO  → Don't mock, test directly

Is it deterministic?
  NO (time.now(), random) → Mock or inject
  YES → Don't mock

Would a test failure with this mock tell me something useful?
  YES → Use a mock
  NO  → Question whether this test is valuable

Am I testing my code or testing the mock?
  Testing the mock → Remove the test
  Testing my code  → Keep the mock

RULES OF THUMB:
  ✓ Mock external systems (Stripe, AWS, third-party APIs)
  ✓ Mock the clock (datetime.now()) for time-dependent tests
  ✓ Mock random number generators for deterministic tests
  ✗ Don't mock your own code (that makes tests internal)
  ✗ Don't mock things just to make a test pass
  ✗ Don't mock things your test doesn't care about (use real objects)
```

## 7. Senior Engineer Perspective

**"Don't mock what you don't own" (principle by Steve Freeman, Nat Pryce):**

Don't mock third-party libraries directly. Instead, wrap them in your own interface and mock that wrapper. If you mock the Stripe SDK directly, you're coupled to its internal structure — any SDK version change might break your mocks. If you create a `PaymentGateway` interface and mock that, your mocks are stable even when the underlying SDK changes.

```python
# DON'T:
mock_stripe_sdk.Charge.create.return_value = {...}
# Now your test knows about Stripe's internal SDK structure

# DO:
class PaymentGateway:
    def charge(self, customer_id: str, amount: int) -> ChargeResult: ...

mock_gateway = AsyncMock(spec=PaymentGateway)
mock_gateway.charge.return_value = ChargeResult(id="ch_123", status="succeeded")
# Test is decoupled from Stripe's SDK structure
```
