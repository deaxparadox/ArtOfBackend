## 1. Why Integration Tests? (Theory)

Unit tests verify logic in isolation. But real bugs often live in the interaction between your code and its dependencies: your SQL query is wrong, the ORM generates a query that works differently than expected, your Redis expiry logic is off by one, your HTTP client doesn't handle redirects correctly.

Integration tests exercise those boundaries with real implementations — real PostgreSQL, real Redis, real HTTP responses.

**The specific things integration tests catch that unit tests can't:**

```
DATABASE:
  ✓ Your SQL query actually returns the right rows
  ✓ Indexes work as expected for your queries
  ✓ Foreign key constraints prevent invalid data
  ✓ Transactions roll back correctly on failure
  ✓ Concurrent updates don't cause data corruption
  ✗ Mocked DB: none of this is testable

HTTP INTEGRATION:
  ✓ Your HTTP client serializes the request correctly
  ✓ The response deserialization handles edge cases
  ✓ Retry logic actually retries on 429
  ✓ Timeout handling works
  ✗ Mocked HTTP client: behavioral differences masked

CACHE:
  ✓ TTL expiry works as expected
  ✓ Your key serialization is correct
  ✓ Cache invalidation logic is correct
  ✗ Mocked cache: none of this validated
```

## 2. Testing with a Real Database

The recommended approach: Testcontainers (run a Docker container with a real database for each test session).

```python
# conftest.py — setup real PostgreSQL for tests

import pytest
import asyncio
import asyncpg
from testcontainers.postgres import PostgresContainer

@pytest.fixture(scope="session")
def postgres_container():
    """Start a real PostgreSQL container for the test session."""
    with PostgresContainer("postgres:16") as pg:
        yield pg

@pytest.fixture(scope="session")
async def db_pool(postgres_container):
    """Create a connection pool to the test database."""
    pool = await asyncpg.create_pool(postgres_container.get_connection_url())
    
    # Run migrations to set up schema
    async with pool.acquire() as conn:
        await conn.execute("""
            CREATE TABLE IF NOT EXISTS users (
                id BIGSERIAL PRIMARY KEY,
                email TEXT UNIQUE NOT NULL,
                name TEXT NOT NULL,
                created_at TIMESTAMPTZ DEFAULT NOW()
            )
        """)
    
    yield pool
    await pool.close()

@pytest.fixture(autouse=True)
async def clean_database(db_pool):
    """Clean all tables before each test."""
    async with db_pool.acquire() as conn:
        await conn.execute("TRUNCATE users CASCADE")
    yield  # test runs here
    # no cleanup after — next fixture invocation handles it

# NOW: write integration tests
class TestUserRepository:
    @pytest.mark.asyncio
    async def test_create_and_retrieve(self, db_pool):
        repo = UserRepository(db_pool)
        
        # Create a user
        user = await repo.create("alice@example.com", "Alice")
        
        # Retrieve it — actual database round-trip
        fetched = await repo.get_by_id(user.id)
        assert fetched.email == "alice@example.com"
        assert fetched.name == "Alice"
    
    @pytest.mark.asyncio
    async def test_email_uniqueness_enforced(self, db_pool):
        repo = UserRepository(db_pool)
        await repo.create("alice@example.com", "Alice")
        
        with pytest.raises(UniqueViolationError):
            await repo.create("alice@example.com", "Alice 2")
    
    @pytest.mark.asyncio
    async def test_list_users_ordered_by_created_at(self, db_pool):
        repo = UserRepository(db_pool)
        await repo.create("charlie@example.com", "Charlie")
        await asyncio.sleep(0.01)  # ensure different timestamps
        await repo.create("alice@example.com", "Alice")
        
        users = await repo.list(order_by="created_at", ascending=True)
        assert users[0].name == "Charlie"
        assert users[1].name == "Alice"
```

## 3. Testing HTTP Endpoints

Testing your FastAPI/Flask/Express endpoints with a test client is a form of integration testing — you're testing the full request/response cycle including routing, middleware, and serialization.

```python
# FastAPI endpoint integration test

from fastapi.testclient import TestClient
from your_app.main import app

@pytest.fixture
def client(db_pool):
    """Override database dependency with test database."""
    app.dependency_overrides[get_db] = lambda: db_pool
    with TestClient(app) as client:
        yield client
    app.dependency_overrides.clear()

class TestUserEndpoints:
    def test_create_user(self, client, clean_database):
        response = client.post("/api/users", json={
            "email": "alice@example.com",
            "name": "Alice",
        })
        
        assert response.status_code == 201
        data = response.json()
        assert data["email"] == "alice@example.com"
        assert "id" in data
        assert "created_at" in data
    
    def test_get_user_returns_404_for_missing_id(self, client):
        response = client.get("/api/users/99999")
        assert response.status_code == 404
        assert response.json()["error"] == "User not found"
    
    def test_create_user_validates_email(self, client):
        response = client.post("/api/users", json={
            "email": "not-an-email",
            "name": "Alice",
        })
        assert response.status_code == 422  # Unprocessable Entity
        errors = response.json()["detail"]
        assert any(e["loc"] == ["body", "email"] for e in errors)
    
    def test_list_users_pagination(self, client, clean_database):
        # Create 25 users
        for i in range(25):
            client.post("/api/users", json={"email": f"user{i}@test.com", "name": f"User {i}"})
        
        # First page
        response = client.get("/api/users?page=1&per_page=10")
        assert response.status_code == 200
        data = response.json()
        assert len(data["items"]) == 10
        assert data["total"] == 25
        assert data["pages"] == 3
```

## 4. Testing with Mocked External Services (Wiremock / httpretty)

For tests that call third-party APIs (Stripe, Sendgrid, etc.), you don't want to hit real APIs. But you also don't want to mock at the SDK level (which hides HTTP details). Mock at the HTTP level instead:

```python
# httpretty: mock HTTP at the socket level

import httpretty
import json
import pytest

@pytest.fixture
def mock_stripe():
    httpretty.enable()
    
    # Mock Stripe charge creation
    httpretty.register_uri(
        httpretty.POST,
        "https://api.stripe.com/v1/charges",
        body=json.dumps({
            "id": "ch_test123",
            "status": "succeeded",
            "amount": 9999,
        }),
        content_type="application/json",
    )
    
    yield
    httpretty.disable()
    httpretty.reset()

def test_payment_creates_stripe_charge(client, mock_stripe):
    response = client.post("/api/payments", json={
        "user_id": "u-123",
        "amount": 99.99,
    })
    
    assert response.status_code == 201
    assert response.json()["stripe_charge_id"] == "ch_test123"
    
    # Verify the request was actually made to Stripe
    last_request = httpretty.last_request()
    request_body = dict(urllib.parse.parse_qsl(last_request.body.decode()))
    assert request_body["amount"] == "9999"
```

## 5. Database Cleanup Strategies

```python
# STRATEGY 1: TRUNCATE before each test (simple, reliable)
@pytest.fixture(autouse=True)
async def clean_db(db_pool):
    async with db_pool.acquire() as conn:
        await conn.execute("TRUNCATE users, orders, payments CASCADE")
    yield

# STRATEGY 2: TRANSACTION ROLLBACK (faster, more complex)
# Each test runs inside a transaction that's rolled back after
@pytest.fixture
async def transactional_db(db_pool):
    conn = await db_pool.acquire()
    tr = conn.transaction()
    await tr.start()
    yield conn
    await tr.rollback()
    await db_pool.release(conn)

# STRATEGY 3: DATABASE PER TEST (slowest, most isolated)
# Create a new database for each test
# Used when tests must run in parallel and interfere with each other

# RECOMMENDED: Strategy 1 (TRUNCATE) for simplicity
# Strategy 2 (rollback) for speed when you have many tests
```

## 6. Senior Engineer Perspective

**"Don't mock the database" is usually correct:**

Mocking the database saves test setup time but validates nothing about your actual data access layer. The most insidious bugs — wrong SQL, ORM quirks, migration mismatches — are invisible to mocked tests. Integration tests with real databases catch these before production.

The counter-argument is speed: real DB tests are 10-100x slower than mocked tests. The resolution: use Testcontainers with session-scoped containers (start once per test session, reuse for all tests). A 1000-test suite with a shared container can run in 30-60 seconds — fast enough for CI.

**Test your migrations too:**

Your migration scripts can have bugs. Test them: run migration up, verify schema is correct, run migration down, verify rollback is clean. This prevents "the migration worked in dev, broke in prod" scenarios that are extremely painful to debug.
