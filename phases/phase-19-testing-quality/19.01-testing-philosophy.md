## 1. Why Test? (Theory)

Tests are automated verification that your code does what you think it does — and keeps doing it after changes. Without tests, every change to production code requires manual verification and carries the risk of silent regressions.

But tests are not free. They take time to write, time to run, and time to maintain. Bad tests can be worse than no tests: they give false confidence, slow down development, and break constantly for the wrong reasons.

The art of testing is knowing what to test, how to test it, and when to stop.

## 2. What to Test

```
TEST BEHAVIOR, NOT IMPLEMENTATION:

The question to ask: "What should this code DO?"
Not: "How does this code work?"

BAD TEST (testing implementation):
  def test_user_service():
      service = UserService()
      service._validate_email("user@example.com")    # private method
      assert service._email_regex == r"..."           # internal detail
      
  Problem: This test breaks when you refactor the private method.
           Refactoring is not a bug. Tests that break on refactoring
           create friction for improvement.

GOOD TEST (testing behavior):
  def test_user_service_rejects_invalid_email():
      service = UserService()
      with pytest.raises(ValidationError):
          service.create_user(email="not-an-email", name="Alice")
  
  def test_user_service_accepts_valid_email():
      service = UserService()
      user = service.create_user(email="alice@example.com", name="Alice")
      assert user.email == "alice@example.com"
      
  These tests verify WHAT the service does (rejects invalid emails, accepts valid ones).
  The internal regex implementation can change freely.
```

## 3. What NOT to Test

```
DON'T TEST:
  ✗ Third-party libraries (trust Django's ORM works)
  ✗ Language features (Python's list.sort() doesn't need a test)
  ✗ Simple getters/setters with no logic
  ✗ Framework configuration (your web framework's routing works)
  
  RULE: Test YOUR logic, not other people's code.

DIMINISHING RETURNS:
  The first tests you write have the highest value: they catch real bugs.
  After a certain point, writing more tests doesn't catch more bugs —
  it just slows down development.
  
  There's no universal "right" coverage number. 80% coverage is often
  cited but meaningless: 100% coverage can coexist with 0 useful assertions.

WHAT ACTUALLY MATTERS:
  ✓ Test the paths that would be painful to break in production
  ✓ Test the edge cases in business logic (empty input, max values, zero)
  ✓ Test the integration between your code and external systems
  ✓ Test security-critical paths (authentication, authorization)
  ✗ Don't test the happy path only and declare victory
```

## 4. The Testing Decision Framework

```
FOR EACH PIECE OF CODE, ASK:

1. "If this is wrong, how would I know?"
   If the answer is "a user would report it" or "production would break silently":
   → Write a test.
   
2. "How often does this code change?"
   High-churn code → tests prevent regressions.
   Code that never changes → tests less valuable.
   
3. "What are the consequences of a bug?"
   Payment processing, authentication, data mutations → test thoroughly.
   Admin dashboard display code → test less aggressively.

4. "Is this logic complex enough to need documentation?"
   Complex business rules → tests ARE documentation.
   A test that shows all the edge cases is worth more than a comment.

5. "Would I know immediately if I broke this?"
   Yes (obvious compile error, immediate exception) → test less important.
   No (subtle behavior change, silent failure) → test essential.
```

## 5. Tests as Documentation

```python
# Tests document how code is supposed to be used.
# They're the best documentation because they never go stale —
# if the test doesn't match the behavior, it fails.

class TestPasswordPolicy:
    """
    Password policy:
    - Minimum 8 characters
    - At least one uppercase, one lowercase, one digit
    - No common passwords
    - Cannot reuse last 5 passwords
    """
    
    def test_minimum_length(self):
        assert not is_valid_password("Short1")   # 6 chars
        assert is_valid_password("LongEnough1")  # 10 chars
    
    def test_requires_uppercase(self):
        assert not is_valid_password("lowercase1")
        assert is_valid_password("HasUpper1")
    
    def test_rejects_common_passwords(self):
        assert not is_valid_password("Password1")  # too common
    
    def test_cannot_reuse_recent_passwords(self):
        user = create_test_user()
        user.set_password("OldPassword1")
        with pytest.raises(PasswordReuseError):
            user.set_password("OldPassword1")
    
    # These tests ARE the specification.
    # A new engineer can read them and understand the requirements.
    # No separate document needed.
```

## 6. Senior Engineer Perspective

**Tests are insurance, not bureaucracy:**

Teams without tests ship fast in the beginning and slow down over time as they break things they didn't expect. Teams with good tests pay upfront for speed later — every refactor, every new feature, every dependency upgrade is less risky because the tests catch regressions.

The payoff isn't immediate. It's 6 months later when you can upgrade a major dependency and know within 5 minutes if anything broke, instead of spending a week manually testing everything.

**Write tests that would fail for the right reasons:**

A test that always passes regardless of implementation is worthless. Before committing a new test, ask: "How do I know this test would fail if the behavior broke?" Deliberately break the implementation, run the test — if it passes, the test is wrong. A test that can't fail can't protect you.
