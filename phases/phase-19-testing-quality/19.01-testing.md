## Testing Philosophy (19.1-19.2)

```
THE TESTING PYRAMID:

         E2E (few)
        /         \
       /  Integration \
      /      Tests     \
     /                  \
    /    Unit Tests       \
   /     (many, fast)      \

UNIT TESTS: test one function/class in isolation
  Fast (milliseconds), deterministic, no external dependencies
  Mock everything outside the function under test
  
INTEGRATION TESTS: test a component with its dependencies
  Test the database query actually works
  Test the HTTP endpoint returns the right response
  Slower (seconds), may need real DB/Redis
  
E2E TESTS: test the full user flow in a real environment
  Slow (minutes), fragile, expensive to maintain
  Cover the most critical user journeys only

TEST COVERAGE MISCONCEPTION:
  High coverage % doesn't mean good tests.
  A test that calls every line but makes no assertions → 100% coverage, 0 value.
  Write tests for: correctness, edge cases, regressions from real bugs.
  
"Test behavior, not implementation" — don't test private methods or internal state.
Test: given this input, do I get this output? Not: does this internal variable get set?
```

## Mocking Strategies (19.10)

```python
from unittest.mock import AsyncMock, MagicMock, patch
import pytest

# Mocking: replace dependencies with controlled fakes

# Test that a payment service call happens correctly
async def test_order_charges_payment(db_session):
    payment_mock = AsyncMock()
    payment_mock.charge.return_value = {"payment_id": "pay_123", "status": "succeeded"}

    service = OrderService(payment_gateway=payment_mock, ...)
    order = await service.create_order(user_id=1, amount_cents=5000)

    payment_mock.charge.assert_called_once_with(5000, user_id=1)
    assert order.payment_id == "pay_123"

# When to mock vs. when to use real implementations:
# MOCK: external APIs (Stripe, SendGrid, Twilio), time/random, slow operations
# DON'T MOCK: your own database (use test DB), your own business logic

# Testcontainers: real services in Docker for integration tests
from testcontainers.postgres import PostgresContainer

@pytest.fixture(scope="session")
def real_postgres():
    with PostgresContainer("postgres:16") as postgres:
        yield postgres.get_connection_url()
# Each test session gets a real Postgres DB in Docker
# Slower than mocking, but tests actual SQL/queries
```

## Property-Based Testing (19.12)

```python
# Hypothesis: generate many inputs automatically to find edge cases
from hypothesis import given, strategies as st
from hypothesis import settings

@given(
    amount_cents=st.integers(min_value=0, max_value=1_000_000),
    currency=st.sampled_from(["USD", "EUR", "GBP"]),
)
def test_money_addition_is_commutative(amount_cents, currency):
    """a + b == b + a for any valid Money values."""
    a = Money(amount_cents, currency)
    b = Money(amount_cents // 2, currency)
    assert a + b == b + a

@given(
    email=st.emails(),
    password=st.text(min_size=8, max_size=100),
)
def test_create_user_accepts_valid_inputs(email, password):
    """User creation should succeed with any valid email and password."""
    user = create_user(email, password)
    assert user.email == email.lower()

# Hypothesis will automatically find: empty strings, unicode, extremes, None
# catches bugs your hand-written tests missed
```

## The Test Data Problem (19.19)

```python
# Factory pattern: create test data programmatically
import factory
from factory.alchemy import SQLAlchemyModelFactory
from app.models import User, Order

class UserFactory(SQLAlchemyModelFactory):
    class Meta:
        model = User
        sqlalchemy_session = None  # injected per-test

    email = factory.Sequence(lambda n: f"user{n}@example.com")  # unique
    name = factory.Faker("name")
    is_active = True
    created_at = factory.Faker("date_time_this_year")

class OrderFactory(SQLAlchemyModelFactory):
    class Meta:
        model = Order

    user = factory.SubFactory(UserFactory)  # auto-creates user if not provided
    total_cents = factory.Faker("random_int", min=100, max=100000)
    status = "pending"

# Usage in tests:
user = UserFactory(email="specific@example.com")  # override email
order = OrderFactory(user=user, total_cents=5000)
high_value_orders = OrderFactory.create_batch(10, total_cents=50000)
```

## Code Coverage (19.18)

```bash
# Python: pytest-cov
pytest --cov=app --cov-report=html --cov-report=term-missing

# Shows:
# app/services/order.py   85%    45, 67-71 not covered

# Coverage gates in CI:
pytest --cov=app --cov-fail-under=80  # fail if coverage drops below 80%

# What coverage tells you: which lines of code your tests execute
# What coverage DOESN'T tell you: whether your tests are correct or meaningful

# More useful than coverage %: branch coverage
pytest --cov=app --cov-branch  # checks both sides of every if/else

# IMPORTANT: coverage is a tool to find untested code.
# 100% coverage ≠ no bugs. Tests can pass all lines with wrong assertions.
# Use coverage to find dead spots; use code review to find weak tests.
```
