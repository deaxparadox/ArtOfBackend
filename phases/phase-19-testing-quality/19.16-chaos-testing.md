## 1. What Is Chaos Engineering? (Theory)

Chaos engineering is the practice of intentionally introducing failures into a system to verify it can handle them gracefully. The principle: failures are inevitable in production — better to discover how your system responds in a controlled experiment than during an actual outage.

**The origin: Netflix Chaos Monkey:**

```
2011: Netflix moved to AWS and ran experiments.
"What happens if we randomly kill EC2 instances?"

Result: engineers discovered their system wasn't as resilient as they thought.
Services were failing completely when a dependency went down.

Solution: build resilience. Implement circuit breakers, retries, fallbacks.
Then verify the resilience works by... intentionally breaking things.

THE PRINCIPLE:
  If a failure can happen in production (hardware failure, network partition,
  slow disk, OOM kill), then you should be able to reproduce it in a
  controlled way to verify your system handles it correctly.
  
  If your system can't handle controlled failure: it can't handle real failure.
  Better to find out now than during Black Friday.
```

## 2. Chaos Experiments

```
TYPES OF CHAOS EXPERIMENTS:

INFRASTRUCTURE FAILURES:
  Kill random pods (Kubernetes)
  Kill random EC2 instances
  Fill disk to 95% capacity
  Exhaust CPU/memory on a host
  
NETWORK FAILURES:
  Introduce 500ms network latency between services
  Drop 10% of packets between services
  Partition network (services can't reach each other)
  
APPLICATION FAILURES:
  Kill a specific service
  Cause a service to return 503 for 30 seconds
  Make database queries 10x slower
  Exhaust database connection pool
  
DATA FAILURES:
  Inject corrupt data into the database
  Clear Redis cache suddenly (cold cache test)
  Fail specific API calls at a percentage

WHAT YOU VERIFY:
  ✓ Circuit breakers open when a dependency is slow/down
  ✓ Fallback behaviors work (serve cached data, degrade gracefully)
  ✓ Alerts fire when failures happen
  ✓ Autoscaling responds to load spikes
  ✓ Service restarts correctly after crash
  ✓ Users see a graceful error page, not a stack trace
```

## 3. Chaos Monkey and LitmusChaos

```bash
# LITMUSCHAO: Kubernetes-native chaos engineering
# Install LitmusChaos
kubectl apply -f https://litmuschaos.github.io/litmus/litmus-operator-v3.0.0.yaml

# CHAOS EXPERIMENT: kill pods randomly
cat <<EOF | kubectl apply -f -
apiVersion: litmuschaos.io/v1alpha1
kind: ChaosEngine
metadata:
  name: payment-service-chaos
spec:
  appinfo:
    appns: production
    applabel: "app=payment-service"
    appkind: deployment
  
  chaosServiceAccount: litmus-admin
  
  experiments:
    - name: pod-delete
      spec:
        components:
          env:
            - name: TOTAL_CHAOS_DURATION
              value: "60"              # run for 60 seconds
            - name: CHAOS_INTERVAL
              value: "10"             # kill a pod every 10 seconds
            - name: FORCE
              value: "false"          # graceful shutdown (SIGTERM first)
            - name: PODS_AFFECTED_PERC
              value: "33"             # kill 33% of pods
EOF

# WHILE THIS RUNS: verify your system behaves correctly
# - API error rate should stay < 1% (surviving pods handle requests)
# - Kubernetes restarts killed pods within 30 seconds
# - Users don't see errors (load balancer stops routing to killed pods)
```

## 4. Network Chaos with Toxiproxy

Toxiproxy (Shopify) simulates network conditions in your test environment:

```python
# Toxiproxy: inject network failures between services

import toxiproxy

client = toxiproxy.ApiClient(host="localhost", port=8474)

# CREATE PROXY: route traffic through toxiproxy
proxy = client.proxies.create({
    "name": "database",
    "listen": "0.0.0.0:5433",          # your app connects to this port
    "upstream": "postgres:5432",         # toxiproxy forwards to real DB
    "enabled": True,
})

# INJECT LATENCY: simulates slow database
latency_toxic = proxy.toxics.create({
    "name": "latency",
    "type": "latency",
    "toxicity": 1.0,  # 100% of requests affected
    "attributes": {
        "latency": 500,   # 500ms added to each request
        "jitter": 50,     # ±50ms jitter
    },
})

# Now run your load test to see how the system handles slow DB:
# - Circuit breakers should open after sustained 500ms latency
# - Timeout errors should be logged and alerted
# - System should degrade gracefully (serve cached data?)

# REMOVE THE TOXIC: back to normal
latency_toxic.destroy()

# INJECT CONNECTION RESET: simulates dropped connections
proxy.toxics.create({
    "name": "reset",
    "type": "reset_peer",
    "toxicity": 0.1,  # 10% of connections dropped
})

# Test: does your connection pool recover from dropped connections?
# Most well-written connection pools do — but it's good to verify.
```

## 5. Chaos Experiments in Practice

```
CHAOS EXPERIMENT TEMPLATE:

1. STEADY STATE HYPOTHESIS:
   "The payment service handles 100 req/s with p99 < 200ms and error rate < 0.1%"
   
   Measure this baseline BEFORE introducing chaos.

2. INTRODUCE CHAOS:
   "Kill 50% of payment service pods for 2 minutes"
   
3. MEASURE:
   "During the chaos, what was the error rate? p99 latency?"
   
4. VERIFY STEADY STATE:
   "After chaos ends, does the system return to normal within 5 minutes?"

5. CONCLUDE:
   If steady state maintained: system is resilient to pod failures.
   If degraded: fix the issue. Add circuit breaker, fix health checks, etc.
   
EXAMPLE OUTCOMES:
   "Killing 50% of pods: error rate 0.5%, p99 250ms → ACCEPTABLE"
   "Killing 50% of pods: error rate 15%, p99 5s → UNACCEPTABLE, investigate"
   
   "Database latency 500ms: payment-service circuit breaker opens,
    serves cached responses → ACCEPTABLE (designed behavior)"
   "Database latency 500ms: all requests timeout after 30s, queue fills up,
    server OOM killed → UNACCEPTABLE, fix timeout/circuit breaker"
```

## 6. Chaos Testing vs. Chaos Engineering

```
CHAOS TESTING (narrow):
  Run specific failure scenarios in pre-production.
  "What happens if we kill the database?"
  One-time tests in staging.
  
CHAOS ENGINEERING (broader discipline):
  Run continuous experiments in production.
  Gradually increase blast radius as confidence grows.
  Start in staging → gradually enable in production.
  Build tooling to run experiments safely.
  Netflix runs chaos experiments 24/7 in production.
  
FOR MOST TEAMS:
  Start with chaos testing in staging.
  Define your steady state and acceptance criteria.
  Run experiments before major releases to validate resilience.
  Progress to production chaos only when you have mature observability
  and automated rollback capabilities.
```

## 7. Senior Engineer Perspective

**Chaos testing reveals assumptions, not bugs:**

The most valuable finding from a chaos experiment is usually: "we assumed X would handle this failure mode, but it doesn't." A circuit breaker that was never tested in a failure scenario might have a configuration bug. A retry policy that was never tested with real network partitions might retry indefinitely instead of failing fast.

These aren't code bugs — they're architectural assumptions. Chaos engineering validates those assumptions before production validates them for you.

**Start small, with clear acceptance criteria:**

The biggest mistake in chaos engineering is starting with "let's kill random things and see what happens." Start with a specific, well-defined hypothesis: "If I kill one of three database replicas, the service continues to serve reads within 200ms." Define what "acceptable" means upfront. Measure both during and after the chaos. This disciplined approach yields actionable results.
