## 1. Go's Built-In Testing (Theory)

Go's philosophy: the standard library does testing. No third-party framework needed. `go test` discovers and runs test files, measures coverage, benchmarks performance, and runs example code as tests.

**Go's testing is deliberately simple:**

```go
// pricing_test.go

package pricing

import "testing"

// Test function: starts with "Test", takes *testing.T
func TestCalculateOrderTotal(t *testing.T) {
    items := []Item{{Price: 10.00, Quantity: 2}}
    got := CalculateOrderTotal(items, "")
    want := 20.00
    
    if got != want {
        t.Errorf("CalculateOrderTotal() = %v, want %v", got, want)
    }
}
```

```bash
# Run tests
go test ./...           # all packages
go test ./pkg/pricing/  # specific package
go test -v ./...        # verbose (show test names)
go test -run TestCalc   # run tests matching pattern
go test -count=1 ./...  # disable test caching (force re-run)
go test -race ./...     # enable race detector (ALWAYS use in CI)
```

## 2. Table-Driven Tests

The Go community's canonical pattern for testing multiple inputs: table-driven tests.

```go
func TestCalculateOrderTotal(t *testing.T) {
    tests := []struct {
        name         string
        items        []Item
        discountCode string
        want         float64
    }{
        {
            name:  "basic calculation",
            items: []Item{{Price: 10.00, Quantity: 2}},
            want:  20.00,
        },
        {
            name:         "10% discount",
            items:        []Item{{Price: 100.00, Quantity: 1}},
            discountCode: "SAVE10",
            want:         90.00,
        },
        {
            name:         "invalid discount code",
            items:        []Item{{Price: 100.00, Quantity: 1}},
            discountCode: "INVALID",
            want:         100.00,  // no discount applied
        },
        {
            name:  "empty cart",
            items: []Item{},
            want:  0.00,
        },
    }
    
    for _, tt := range tests {
        t.Run(tt.name, func(t *testing.T) {  // sub-test per case
            got := CalculateOrderTotal(tt.items, tt.discountCode)
            if got != tt.want {
                t.Errorf("CalculateOrderTotal() = %v, want %v", got, tt.want)
            }
        })
    }
}

// Run specific sub-test:
// go test -run TestCalculateOrderTotal/10%_discount
```

## 3. Interfaces for Testability

Go achieves testability through interfaces — define what a dependency does, then swap real for fake in tests.

```go
// Define an interface for the email sender
type EmailSender interface {
    Send(to, subject, body string) error
}

// Real implementation (SMTP)
type SMTPSender struct {
    host string
    port int
}

func (s *SMTPSender) Send(to, subject, body string) error {
    // ... real SMTP code
}

// Fake implementation for tests
type FakeEmailSender struct {
    SentEmails []EmailRecord  // exposed for test assertions
}

func (f *FakeEmailSender) Send(to, subject, body string) error {
    f.SentEmails = append(f.SentEmails, EmailRecord{To: to, Subject: subject})
    return nil
}

// Service uses the interface, not the concrete type
type UserService struct {
    repo   UserRepository
    email  EmailSender  // interface, not *SMTPSender
}

// TEST:
func TestUserService_WelcomeEmailSentOnSignup(t *testing.T) {
    fakeEmail := &FakeEmailSender{}
    service := NewUserService(fakeRepo, fakeEmail)
    
    user, err := service.CreateUser("alice@example.com", "Alice")
    if err != nil {
        t.Fatal(err)
    }
    
    // Verify email was sent
    if len(fakeEmail.SentEmails) != 1 {
        t.Errorf("expected 1 email sent, got %d", len(fakeEmail.SentEmails))
    }
    if fakeEmail.SentEmails[0].To != "alice@example.com" {
        t.Errorf("email sent to wrong address")
    }
}
```

## 4. Testing HTTP Handlers

```go
package handlers_test

import (
    "encoding/json"
    "net/http"
    "net/http/httptest"
    "testing"
    
    "github.com/yourapp/handlers"
)

func TestGetUser(t *testing.T) {
    // httptest: in-memory HTTP server — no network required
    req := httptest.NewRequest("GET", "/users/123", nil)
    req.Header.Set("Authorization", "Bearer test-token")
    
    rr := httptest.NewRecorder()  // captures response
    
    handler := handlers.NewUserHandler(fakeUserRepo)
    handler.GetUser(rr, req)
    
    // Assert status code
    if rr.Code != http.StatusOK {
        t.Errorf("expected 200, got %d", rr.Code)
    }
    
    // Assert response body
    var user handlers.UserResponse
    if err := json.Unmarshal(rr.Body.Bytes(), &user); err != nil {
        t.Fatalf("failed to decode response: %v", err)
    }
    if user.Email != "alice@example.com" {
        t.Errorf("expected alice@example.com, got %s", user.Email)
    }
}

func TestGetUser_NotFound(t *testing.T) {
    req := httptest.NewRequest("GET", "/users/99999", nil)
    rr := httptest.NewRecorder()
    
    handler := handlers.NewUserHandler(fakeUserRepo)
    handler.GetUser(rr, req)
    
    if rr.Code != http.StatusNotFound {
        t.Errorf("expected 404, got %d", rr.Code)
    }
}
```

## 6. testify: Popular Go Testing Library

```go
// github.com/stretchr/testify — adds assert helpers and mocking

import (
    "testing"
    "github.com/stretchr/testify/assert"
    "github.com/stretchr/testify/require"
    "github.com/stretchr/testify/mock"
)

func TestWithTestify(t *testing.T) {
    user, err := service.GetUser("123")
    
    // require: stops test on failure (like t.Fatal)
    require.NoError(t, err)
    require.NotNil(t, user)
    
    // assert: continues test on failure (like t.Error)
    assert.Equal(t, "alice@example.com", user.Email)
    assert.Equal(t, "Alice", user.Name)
    assert.WithinDuration(t, time.Now(), user.CreatedAt, time.Minute)
}

// testify/mock: mock expectations
type MockEmailSender struct {
    mock.Mock
}

func (m *MockEmailSender) Send(to, subject, body string) error {
    args := m.Called(to, subject, body)
    return args.Error(0)
}

func TestWithMock(t *testing.T) {
    emailSender := new(MockEmailSender)
    emailSender.On("Send", "alice@example.com", mock.AnythingOfType("string"), mock.Anything).Return(nil)
    
    service := NewUserService(fakeRepo, emailSender)
    service.CreateUser("alice@example.com", "Alice")
    
    emailSender.AssertExpectations(t)  // verify all expected calls were made
}
```

## 7. Senior Engineer Perspective

**The race detector is non-negotiable:**

`go test -race` enables Go's race detector. It instruments memory access to detect data races — concurrent reads/writes without synchronization. Data races are the hardest bugs in Go: they manifest intermittently, cause corruption, and are nearly impossible to reproduce. Running tests with `-race` in CI catches most race conditions before they reach production. The overhead is 2-10x slower test execution — worth it every time.

**Table-driven tests are the right default in Go:**

When you have more than 2 test cases, use table-driven tests. They force you to explicitly name each case (making failures self-documenting), they make adding new cases trivial, and they avoid code duplication. This pattern is so idiomatic in Go that code reviewers will suggest it if you don't use it.
