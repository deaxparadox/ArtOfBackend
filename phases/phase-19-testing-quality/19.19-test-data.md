## 1. The Test Data Problem (Theory)

Tests need data. Where does that data come from? How do you ensure it's consistent, isolated, and realistic?

```
THREE TEST DATA STRATEGIES:

1. INLINE: create data directly in each test
   pro: explicit, self-contained
   con: verbose, duplication across tests

2. FACTORIES: programmatic builders for test objects
   pro: reusable, minimal code per test, realistic data
   con: requires factory setup

3. FIXTURES/SEEDS: pre-defined datasets loaded before tests
   pro: fast, consistent
   con: can become stale, shared state issues

RULE: use factories for unit/integration tests,
       seeds for specific known-state scenarios.
```

## 2. Factory Boy: Python Test Factories

```python
# pip install factory-boy

import factory
from factory.django import DjangoModelFactory
from faker import Faker

fake = Faker()

class UserFactory(factory.Factory):
    class Meta:
        model = User
    
    # Auto-generates realistic fake data
    email = factory.Sequence(lambda n: f"user{n}@example.com")  # unique per call
    name = factory.Faker("name")                                  # "Alice Johnson"
    age = factory.Faker("random_int", min=18, max=80)
    created_at = factory.LazyFunction(datetime.utcnow)

class OrderFactory(factory.Factory):
    class Meta:
        model = Order
    
    user = factory.SubFactory(UserFactory)   # auto-creates related user
    total = factory.Faker("pydecimal", min_value=1, max_value=10000, right_digits=2)
    status = factory.Iterator(["pending", "processing", "completed"])

# USAGE:
def test_order_total_calculation():
    # Create one user with one order
    user = UserFactory()
    order = OrderFactory(user=user, total=99.99)
    
    # Create user with specific overrides
    admin = UserFactory(email="admin@company.com", role="admin")
    
    # Create batch of 10 users
    users = UserFactory.build_batch(10)   # not saved to DB
    users = UserFactory.create_batch(10)  # saved to DB (Django only)

# TRAITS: named configurations
class UserFactory(factory.Factory):
    class Meta:
        model = User
    
    email = factory.Sequence(lambda n: f"user{n}@example.com")
    name = factory.Faker("name")
    role = "user"
    is_active = True
    
    class Params:
        admin = factory.Trait(role="admin")
        inactive = factory.Trait(is_active=False)

# Usage with traits:
admin = UserFactory(admin=True)         # role="admin"
inactive = UserFactory(inactive=True)   # is_active=False
```

## 3. JavaScript Factory (fishery / factory-ts)

```typescript
import { Factory } from 'fishery';
import { faker } from '@faker-js/faker';

interface User {
    id: string;
    email: string;
    name: string;
    role: 'user' | 'admin';
    createdAt: Date;
}

const userFactory = Factory.define<User>(({ sequence }) => ({
    id: `user-${sequence}`,          // unique ID per call
    email: faker.internet.email(),
    name: faker.person.fullName(),
    role: 'user',
    createdAt: new Date(),
}));

// Usage:
const user = userFactory.build();
const admin = userFactory.build({ role: 'admin' });
const users = userFactory.buildList(5);

// With specific values:
const alice = userFactory.build({
    email: 'alice@example.com',
    name: 'Alice',
});
```

## 4. Database Seeding

```python
# Database seeding: create known state for specific test scenarios

# seeds/development.py
async def seed_development_database(pool):
    """Seed the development database with realistic data."""
    async with pool.acquire() as conn:
        # Users
        await conn.execute("""
            INSERT INTO users (id, email, name, role) VALUES
            ('u-admin', 'admin@company.com', 'Admin User', 'admin'),
            ('u-alice', 'alice@example.com', 'Alice Smith', 'user'),
            ('u-bob', 'bob@example.com', 'Bob Jones', 'user')
            ON CONFLICT (email) DO NOTHING
        """)
        
        # Orders for alice
        await conn.execute("""
            INSERT INTO orders (id, user_id, total, status) VALUES
            ('o-001', 'u-alice', 99.99, 'completed'),
            ('o-002', 'u-alice', 149.99, 'pending'),
            ('o-003', 'u-bob', 24.99, 'completed')
            ON CONFLICT (id) DO NOTHING
        """)

# seeds/test_scenarios.py
async def seed_checkout_flow_test(pool):
    """Seed state needed for checkout E2E test."""
    async with pool.acquire() as conn:
        await conn.execute("""
            INSERT INTO users (id, email, name, password_hash)
            VALUES ('test-user-checkout', 'checkout@test.com', 'Test User', $1)
            ON CONFLICT DO NOTHING
        """, hash_password("password123"))
        
        await conn.execute("""
            INSERT INTO products (id, name, price, inventory)
            VALUES ('widget-pro', 'Widget Pro', 99.99, 100)
            ON CONFLICT (id) DO UPDATE SET inventory = 100
        """)
```

## 5. Test Data Isolation Strategies

```
STRATEGY 1: TRUNCATE + FACTORY (recommended)
  Before each test: truncate all tables
  In each test: use factories to create exactly needed data
  
  Pro: perfect isolation, no shared state
  Con: slightly slower (truncation + data creation per test)

STRATEGY 2: TRANSACTION ROLLBACK
  Each test runs in a transaction, rolled back after
  
  Pro: faster than truncate (no data creation cost for clean state)
  Con: complex with nested transactions, doesn't test transaction behavior

STRATEGY 3: UNIQUE NAMESPACING
  Each test creates data with unique prefix/suffix
  "test-{uuid}-user@example.com"
  
  Pro: no cleanup needed
  Con: data accumulates, queries need to filter by namespace
  Use for: local development, not CI

STRATEGY 4: SEPARATE DATABASE PER TEST (slowest, most isolated)
  Create a new database schema for each test
  
  Pro: perfect isolation
  Con: 100x slower than truncation
  Use for: tests that must run in parallel on the same host
```

## 6. Data Privacy in Test Environments

```
PRODUCTION DATA IN TESTS:
  NEVER use real production data in tests:
  - GDPR/CCPA: personal data must be protected
  - Data exposure risk: test databases are less secured
  - Data accuracy: production data changes, tests become fragile

ANONYMIZATION FOR STAGING:
  If you need production-like data for realistic testing,
  anonymize it before loading to staging:
  
  # Anonymize user data
  UPDATE users SET
      email = CONCAT('user', id, '@anonymized.test'),
      name = 'Anonymized User ' || id,
      phone = '555-000-' || LPAD(id::text, 4, '0');
  
  # Delete financial data
  DELETE FROM credit_cards;
  DELETE FROM bank_accounts;

SYNTHETIC DATA:
  Best practice: generate entirely synthetic data that's 
  statistically similar to production (same cardinality, distributions)
  Tools: Faker (Python), @faker-js/faker (JS), Mimesis (Python)
```

## 7. Senior Engineer Perspective

**Factories are the highest-leverage test infrastructure investment:**

Good factories make writing tests fast and readable. Instead of 20 lines of setup code creating related objects, you get: `user = UserFactory()` or `order = OrderFactory(user=user, status='completed')`. When requirements change, update the factory once — not in every test.

The investment: 2-4 hours writing factories for your core models. The return: every test thereafter is written 3-5x faster, and tests are more readable because the setup noise is gone.

**Don't seed too much — use factories:**

Pre-seeded databases feel convenient but create hidden dependencies. Test A expects 3 users to exist; Test B adds users; Test A fails because there are now 4. Factories create exactly what the test needs, no more. Tests become self-describing: the data in a test tells you exactly what state matters for that test.
