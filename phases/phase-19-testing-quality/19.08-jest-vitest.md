## 1. Jest: The Standard Node.js Test Runner

Jest (Meta/Facebook, 2014) is the most widely used JavaScript test framework. It includes a test runner, assertion library, mocking framework, and code coverage tool — all in one package.

```typescript
// Jest test example

import { calculateOrderTotal } from '../src/pricing';

describe('calculateOrderTotal', () => {
    it('calculates simple order total', () => {
        const items = [{ price: 10.00, quantity: 2 }];
        expect(calculateOrderTotal(items)).toBe(20.00);
    });
    
    it('applies 10% discount with SAVE10 code', () => {
        const items = [{ price: 100.00, quantity: 1 }];
        expect(calculateOrderTotal(items, 'SAVE10')).toBe(90.00);
    });
    
    it('handles empty cart', () => {
        expect(calculateOrderTotal([])).toBe(0);
    });
});
```

```typescript
// Jest mocking

import { PaymentService } from '../src/payment';
import { StripeClient } from '../src/stripe';
import { UserRepository } from '../src/users';

// AUTO-MOCK: Jest replaces module with automatic mocks
jest.mock('../src/stripe');
jest.mock('../src/users');

describe('PaymentService', () => {
    let paymentService: PaymentService;
    let mockStripe: jest.Mocked<StripeClient>;
    let mockUserRepo: jest.Mocked<UserRepository>;
    
    beforeEach(() => {
        // Get the mocked instances
        mockStripe = new StripeClient() as jest.Mocked<StripeClient>;
        mockUserRepo = new UserRepository() as jest.Mocked<UserRepository>;
        paymentService = new PaymentService(mockStripe, mockUserRepo);
        
        // Reset all mocks between tests
        jest.clearAllMocks();
    });
    
    it('charges the correct amount', async () => {
        mockUserRepo.findById.mockResolvedValue({
            id: 'u-123',
            stripeCustomerId: 'cus_abc',
        });
        mockStripe.createCharge.mockResolvedValue({
            id: 'ch_abc',
            status: 'succeeded',
        });
        
        const result = await paymentService.charge('u-123', 99.99);
        
        expect(mockStripe.createCharge).toHaveBeenCalledWith({
            amount: 9999,
            customer: 'cus_abc',
        });
        expect(result.chargeId).toBe('ch_abc');
    });
    
    it('throws UserNotFoundError for unknown user', async () => {
        mockUserRepo.findById.mockResolvedValue(null);
        
        await expect(
            paymentService.charge('u-999', 99.99)
        ).rejects.toThrow('User not found');
    });
});
```

## 2. Vitest: The Modern Alternative

Vitest (2022) is Jest's main competitor, designed specifically for Vite projects. It's faster than Jest because it uses Vite's native ESM support and runs tests in the same process as Vite's dev server.

```typescript
// Vitest: near-identical API to Jest, but ESM-native and faster

import { describe, it, expect, vi } from 'vitest';

describe('User validation', () => {
    it('rejects invalid email', () => {
        expect(isValidEmail('not-email')).toBe(false);
    });
    
    it('accepts valid email', () => {
        expect(isValidEmail('alice@example.com')).toBe(true);
    });
});

// MOCKING IN VITEST
import { vi } from 'vitest';

const mockFetch = vi.fn();
global.fetch = mockFetch;

mockFetch.mockResolvedValue({
    ok: true,
    json: () => Promise.resolve({ id: '123', name: 'Alice' }),
});

it('fetches user data', async () => {
    const user = await fetchUser('123');
    expect(user.name).toBe('Alice');
    expect(mockFetch).toHaveBeenCalledWith('/api/users/123');
});
```

## 3. Testing Express/Node.js APIs

```typescript
// Testing Express endpoints with supertest

import request from 'supertest';
import express from 'express';
import { userRouter } from '../src/routes/users';

const app = express();
app.use(express.json());
app.use('/users', userRouter);

describe('User API', () => {
    it('POST /users creates a user', async () => {
        const response = await request(app)
            .post('/users')
            .send({ email: 'alice@example.com', name: 'Alice' })
            .expect(201);
        
        expect(response.body.email).toBe('alice@example.com');
        expect(response.body.id).toBeDefined();
    });
    
    it('GET /users/:id returns 404 for unknown user', async () => {
        const response = await request(app)
            .get('/users/99999')
            .expect(404);
        
        expect(response.body.error).toBe('User not found');
    });
    
    it('requires valid email', async () => {
        await request(app)
            .post('/users')
            .send({ email: 'invalid', name: 'Alice' })
            .expect(400);
    });
});
```

## 4. Snapshot Testing

```typescript
// Snapshot testing: verify output doesn't change unexpectedly

it('renders user profile correctly', () => {
    const user = { id: '123', name: 'Alice', email: 'alice@example.com' };
    const html = renderUserProfile(user);
    
    expect(html).toMatchSnapshot();
    // First run: creates snapshot file
    // Subsequent runs: compares against snapshot
    // On intentional change: run with --updateSnapshot flag
});

// INLINE SNAPSHOT: snapshot stored in test file
it('formats currency', () => {
    expect(formatCurrency(1234.56)).toMatchInlineSnapshot(`"$1,234.56"`);
});

// WHEN TO USE SNAPSHOTS:
// ✓ Complex output that would be tedious to verify field-by-field
// ✓ Rendered HTML, formatted text, complex JSON
// ✗ Don't snapshot everything: snapshots become noise
// ✗ Snapshots with business logic are usually better explicit tests
```

## 5. Jest vs. Vitest: When to Choose

```
JEST:
  ✓ Most widely used (huge ecosystem, many tutorials)
  ✓ Works with any bundler (Webpack, Rollup, etc.)
  ✓ Mature, stable (many years of production use)
  ✓ Best for CRA (Create React App) projects
  ✗ Slower startup (CommonJS module transformation)
  ✗ Configuration can be complex with ESM

VITEST:
  ✓ Native ESM support (no transformation overhead)
  ✓ Much faster startup (uses Vite's cache)
  ✓ API compatible with Jest (easy migration)
  ✓ Best for Vite projects (Vue, React with Vite, SvelteKit)
  ✓ Watch mode is faster
  ✗ Newer (less ecosystem, fewer tutorials)
  ✗ Some Jest plugins don't work with Vitest

RECOMMENDATION (2024):
  New Vite project: Vitest
  Existing Jest project: keep Jest unless you have performance problems
  Non-Vite project (raw Webpack, Next.js): Jest
```

## 6. Configuration

```typescript
// vite.config.ts (Vitest config lives here)
import { defineConfig } from 'vitest/config';

export default defineConfig({
    test: {
        globals: true,              // no need to import describe, it, expect
        environment: 'node',        // 'jsdom' for browser env
        coverage: {
            provider: 'istanbul',
            reporter: ['text', 'html'],
            exclude: ['**/node_modules/**', '**/dist/**'],
        },
        setupFiles: ['./tests/setup.ts'],
    },
});
```

```json
// package.json scripts
{
    "scripts": {
        "test": "vitest run",
        "test:watch": "vitest",
        "test:coverage": "vitest run --coverage",
        "test:ui": "vitest --ui"   // browser UI for test results
    }
}
```

## 7. Senior Engineer Perspective

**TypeScript makes test assertions more powerful:**

With TypeScript, your mocks are type-checked. If the real implementation changes (adds a required parameter, changes a return type), TypeScript in your tests immediately shows the mismatch. This prevents a class of bugs where you update the implementation but forget to update the tests — and the tests pass because the mocks don't reflect reality.

**Vitest's HMR-based watch mode is a productivity multiplier:**

Vitest's watch mode with Vite's hot module replacement only re-runs tests affected by your changes. If you edit `payment.ts`, only `payment.test.ts` re-runs — not your entire 500-test suite. This makes the TDD loop (red → green → refactor) practically instantaneous and genuinely enjoyable.
