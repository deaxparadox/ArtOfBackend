## 1. What Is Code Coverage? (Theory)

Code coverage measures what percentage of your code was executed during the test suite. It tells you: "these lines ran." It does NOT tell you: "the behavior was verified."

```
TYPES OF COVERAGE:

LINE COVERAGE: did this line execute?
  def process_payment(amount, card):
      if amount > 0:              # line executed
          charge = charge_card(card, amount)  # line executed
          return charge           # line executed
      return None                 # line NOT executed (no test with amount <= 0)
  
  Line coverage: 3/4 = 75%

BRANCH COVERAGE: did BOTH branches of every condition execute?
  if amount > 0:
    ✓ True branch (amount > 0) executed
    ✗ False branch (amount <= 0) NOT executed
  
  Branch coverage: 1/2 = 50%

STATEMENT COVERAGE: similar to line, counts statements not lines

FUNCTION COVERAGE: did this function get called at all?
```

## 2. The Fundamental Limitation

```python
# HIGH COVERAGE ≠ GOOD TESTS

# CODE:
def calculate_discount(price: float, rate: float) -> float:
    if rate > 0:
        return price * (1 - rate)
    return price

# TEST (100% line coverage, but TERRIBLE test):
def test_calculate_discount():
    result = calculate_discount(100, 0.1)
    # Every line was executed...
    assert result is not None  # but this assertion is USELESS
    # We could change the calculation to multiply by 100x
    # and this test would still pass!

# BETTER TEST (also 100% coverage, but MEANINGFUL):
def test_calculate_discount_applies_rate():
    assert calculate_discount(100, 0.1) == 90.0   # exact value verified
    assert calculate_discount(100, 0.5) == 50.0
    assert calculate_discount(100, 0) == 100.0    # no discount

# The coverage number is the same (100%) but the quality is completely different.
# This is why coverage is a NECESSARY but not SUFFICIENT quality indicator.
```

## 3. Generating Coverage Reports

```bash
# PYTHON: pytest-cov
pip install pytest-cov

pytest --cov=src --cov-report=html --cov-report=term-missing tests/

# Output:
# Name                     Stmts   Miss  Cover   Missing
# -------------------------------------------------------
# src/payment.py              42      5    88%   23, 45-48, 67
# src/users.py                31      2    94%   89, 91
# src/auth.py                 28      0   100%
# -------------------------------------------------------
# TOTAL                      101      7    93%

# HTML report: htmlcov/index.html
# Shows exactly which lines are uncovered (red highlighting)

# FAIL BUILD IF COVERAGE DROPS:
pytest --cov=src --cov-fail-under=85 tests/
# Exit code 2 if coverage falls below 85%
```

```yaml
# GitHub Actions: coverage gating
- name: Test with coverage
  run: |
    pytest --cov=src --cov-report=xml --cov-fail-under=80 tests/

- name: Upload coverage to Codecov
  uses: codecov/codecov-action@v3
  with:
    file: ./coverage.xml
    fail_ci_if_error: true
```

## 4. Coverage Metrics That Actually Matter

```
COMMON COVERAGE THRESHOLDS (industry practice):

< 60%: Your test suite is sparse. Focus here first.
60-80%: Reasonable for many projects.
80-90%: Good. This is what most mature projects target.
> 90%: Consider the cost/benefit — the last 10% is often
       configuration code or error handling that's hard to test.

BETTER METRICS THAN AGGREGATE COVERAGE:

1. COVERAGE OF CRITICAL PATHS:
   Payment processing: 95%+ required
   Authentication: 90%+ required
   Admin CRUD: 70%+ acceptable
   Display-only code: 50%+ acceptable
   
   Weighted by business criticality, not all code equally.

2. COVERAGE DELTA:
   "This PR added 200 lines but only 150 are covered"
   Trend: is coverage going up or down over time?
   
3. UNCOVERED CRITICAL PATHS:
   Which specific functions have zero coverage?
   Are any of those security-critical?

4. TEST EFFECTIVENESS (mutation score — see topic 19.13):
   "Tests execute these lines AND verify they're correct"
```

## 5. What Good Coverage Looks Like

```python
# Example: comprehensive coverage for a service class

class UserService:
    def create_user(self, email: str, name: str) -> User:
        # Validation
        if not email or '@' not in email:
            raise ValidationError("Invalid email")
        if not name or len(name) < 2:
            raise ValidationError("Name too short")
        
        # Business rule
        existing = self.repo.get_by_email(email)
        if existing:
            raise DuplicateEmailError(email)
        
        # Creation
        user = self.repo.create(email=email, name=name)
        self.email_service.send_welcome(email)
        return user

# TESTS NEEDED FOR MEANINGFUL COVERAGE:
class TestUserServiceCreate:
    def test_success_case(self):          # happy path
    def test_invalid_email(self):         # validation branch 1
    def test_invalid_name(self):          # validation branch 2  
    def test_duplicate_email(self):       # business rule branch
    def test_welcome_email_sent(self):    # side effect verified
    def test_returns_created_user(self):  # return value verified
    
    # Edge cases:
    def test_email_with_spaces(self):     # email='  alice@example.com  '
    def test_empty_strings(self):         # email='', name=''
    def test_name_exactly_two_chars(self): # boundary condition
```

## 6. Coverage Anti-Patterns

```python
# ANTI-PATTERN 1: COVERAGE WITHOUT ASSERTIONS
def test_process_order():
    service = OrderService()
    service.process(order_id="123")  # runs the code, no assertions
    assert True  # always passes

# ANTI-PATTERN 2: TESTING ONLY THE HAPPY PATH
class TestPaymentService:
    def test_successful_charge(self):
        # 70% of lines covered — but only success scenario
        # What about: card declined? Invalid amount? User not found?
    
# ANTI-PATTERN 3: GAMING THE COVERAGE NUMBER
# Add tests that just "touch" code without verifying behavior

# ANTI-PATTERN 4: SETTING COVERAGE GOALS IN STONE
# "We must maintain 85% coverage" → people add bad tests
# to maintain the number rather than to improve confidence
# Better: "coverage should be growing" or "critical paths must be ≥90%"
```

## 7. Senior Engineer Perspective

**Coverage is a floor, not a ceiling:**

Treat coverage as a minimum bar, not a goal. "We must have 80% coverage" prevents coverage from dropping; it doesn't ensure tests are good. The correct question is: "Are the things that matter verified?"

Use coverage to find gaps (uncovered lines in critical paths), not to compare projects. A startup at 70% coverage with excellent tests on the critical paths is in better shape than a company at 95% coverage with assertions like `assert result is not None`.

**The 100% coverage trap:**

Attempting 100% coverage forces you to test error handling paths, third-party library behavior, and edge cases that may never matter in practice. The cost of writing and maintaining those tests often outweighs the benefit. Most experienced teams settle on 80-90% and accept that some code (startup configuration, rare error handling) will be uncovered.
