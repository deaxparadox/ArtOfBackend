## 1. What Is Performance Testing? (Theory)

Performance testing verifies that your system behaves correctly under load. It answers: "How many users can we handle before things break?"

```
TYPES OF PERFORMANCE TESTS:

LOAD TEST:
  Gradually increase load to expected production levels.
  "Can the system handle our expected 500 users/minute?"
  Duration: 10-30 minutes
  Goal: verify performance at normal load

STRESS TEST:
  Push beyond normal load until the system breaks.
  "Where is the breaking point? What breaks first?"
  Duration: 30-60 minutes
  Goal: find the bottleneck

SPIKE TEST:
  Sudden 10x traffic increase.
  "What happens when we go viral? Can we autoscale in time?"
  Pattern: flat → sudden spike → flat
  Goal: verify behavior under sudden load

SOAK TEST (endurance):
  Sustained load over 12-24 hours.
  "Does memory grow over time? Are there resource leaks?"
  Goal: find slow memory leaks, connection exhaustion over time

CAPACITY PLANNING:
  "How much infrastructure do we need for 10x our current load?"
  Used for: infrastructure purchasing, scaling decisions
```

## 2. k6: Modern Load Testing

k6 (Grafana Labs) is the modern standard for developer-focused performance testing. Tests are JavaScript/TypeScript code, easy to version-control and integrate with CI.

```javascript
// k6 load test script: basic-load-test.js

import http from 'k6/http';
import { check, sleep } from 'k6';
import { Rate, Trend } from 'k6/metrics';

// Custom metrics
const errorRate = new Rate('errors');
const checkoutDuration = new Trend('checkout_duration');

// Test configuration
export const options = {
    stages: [
        { duration: '2m', target: 50 },   // ramp up to 50 users
        { duration: '5m', target: 50 },   // stay at 50 users
        { duration: '2m', target: 100 },  // ramp to 100 users
        { duration: '5m', target: 100 },  // stay at 100 users
        { duration: '2m', target: 0 },    // ramp down to 0
    ],
    thresholds: {
        // Test FAILS if these thresholds are exceeded
        'http_req_duration': ['p99<500'],    // p99 must be under 500ms
        'errors': ['rate<0.01'],             // error rate under 1%
        'http_req_failed': ['rate<0.01'],    // http failure rate under 1%
    },
};

const BASE_URL = __ENV.BASE_URL || 'https://staging.yourcompany.com';

export default function () {
    // VIRTUAL USER BEHAVIOR: what each simulated user does
    
    // Step 1: browse product
    const productRes = http.get(`${BASE_URL}/api/products/widget-pro`);
    check(productRes, {
        'product loaded': (r) => r.status === 200,
        'has price': (r) => r.json('price') > 0,
    });
    errorRate.add(productRes.status !== 200);
    
    sleep(1); // think time: users don't hammer instantly
    
    // Step 2: add to cart
    const cartRes = http.post(`${BASE_URL}/api/cart`, JSON.stringify({
        product_id: 'widget-pro',
        quantity: 1,
    }), {
        headers: { 'Content-Type': 'application/json' },
    });
    check(cartRes, { 'added to cart': (r) => r.status === 201 });
    
    sleep(2);
    
    // Step 3: checkout (most important to measure)
    const checkoutStart = Date.now();
    const checkoutRes = http.post(`${BASE_URL}/api/checkout`, JSON.stringify({
        cart_id: cartRes.json('cart_id'),
        payment: { card: '4242424242424242', exp: '12/25', cvc: '123' },
    }), {
        headers: { 'Content-Type': 'application/json' },
        timeout: '30s',
    });
    checkoutDuration.add(Date.now() - checkoutStart);
    check(checkoutRes, { 'checkout succeeded': (r) => r.status === 201 });
    
    sleep(1);
}
```

```bash
# Run the test
k6 run basic-load-test.js

# With environment variables
k6 run -e BASE_URL=https://staging.yourcompany.com basic-load-test.js

# Output:
#          ✓ product loaded...........100% ✓ 4832  ✗ 0
#          ✓ added to cart............100% ✓ 4820  ✗ 12
#          ✓ checkout succeeded.......99%  ✓ 4789  ✗ 43
#
# http_req_duration..............: avg=124ms min=45ms  med=95ms  max=2.1s p90=245ms p99=445ms
# checkout_duration..............: avg=890ms min=120ms med=780ms max=5.2s p90=1.8s  p99=3.1s ← SLOW
#
# Threshold failures:
# checkout_duration: p99 (3.1s) exceeds 500ms threshold → TEST FAILED
```

## 3. Locust: Python-Based Load Testing

Locust (Python) is excellent when your team is more comfortable with Python and you want programmable user behavior:

```python
# locustfile.py

from locust import HttpUser, TaskSet, task, between

class UserBehavior(TaskSet):
    
    def on_start(self):
        """Log in when user starts."""
        self.client.post("/auth/login", json={
            "email": "test@example.com",
            "password": "password123",
        })
    
    @task(3)   # weight 3: run 3x more often than checkout
    def browse_products(self):
        """Browse product catalog (most common user action)."""
        self.client.get("/api/products")
    
    @task(1)   # weight 1: checkout is less frequent
    def checkout(self):
        """Complete a purchase."""
        # Add to cart
        response = self.client.post("/api/cart", json={
            "product_id": "widget-pro",
            "quantity": 1,
        })
        cart_id = response.json()["cart_id"]
        
        # Checkout
        self.client.post("/api/checkout", json={
            "cart_id": cart_id,
            "payment": {"card": "4242424242424242"},
        })

class WebsiteUser(HttpUser):
    tasks = [UserBehavior]
    wait_time = between(1, 3)  # 1-3 seconds between actions

# Run:
# locust -f locustfile.py --host=https://staging.yourcompany.com
# Opens web UI at http://localhost:8089
# Set: 100 users, 10 users/second ramp rate
# Watch real-time charts of RPS, response time, failures
```

## 4. Reading Performance Test Results

```
KEY METRICS TO ANALYZE:

p50 (median): typical user experience
p95: 5% of users experience this or worse
p99: 1% of users experience this or worse
p999: 0.1% — rare worst case

WHAT BAD LOOKS LIKE:
  p50: 50ms (fast!)
  p99: 4500ms (users occasionally wait 4.5 seconds!)
  
  The gap between p50 and p99 indicates high variance.
  Usually means: some database queries are very slow,
  some requests hit a slow code path.

THROUGHPUT vs. LATENCY TRADEOFF:
  As you add more users:
  Phase 1 (below capacity): latency stays flat
  Phase 2 (at capacity): latency starts growing
  Phase 3 (overloaded): latency spikes, errors appear
  
  The transition point between phase 1 and 2 is your capacity limit.

WHAT TO DO WITH RESULTS:
  - p99 > 500ms: optimize the slow queries/operations causing it
  - Error rate > 1%: find the error source (check logs)
  - Latency keeps growing: connection pool exhaustion, thread pool, memory
  - Memory growing over time: memory leak (run soak test to confirm)
```

## 5. Performance Testing in CI

```yaml
# GitHub Actions: run k6 smoke test on every PR deploy

name: Performance Check
on:
  pull_request:
    branches: [main]

jobs:
  performance:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      
      - name: Run k6 smoke test
        uses: grafana/k6-action@v0.3.0
        with:
          filename: tests/performance/smoke.js
        env:
          BASE_URL: ${{ vars.STAGING_URL }}
      
      # Smoke test: 10 virtual users, 2 minutes
      # Thresholds: p99 < 300ms, error rate < 0.1%
      # If thresholds fail: PR cannot merge
```

## 6. Senior Engineer Perspective

**Performance testing reveals capacity, not just bugs:**

Function tests verify correctness: "does this return the right answer?" Performance tests verify scale: "does this return the right answer in 100ms when 1000 users ask simultaneously?" Both are necessary, but they're fundamentally different questions.

The insight from performance tests that you can't get any other way: where exactly does your system break? Is it the database connections? The application threads? The cache? Memory? CPU? Performance testing under realistic load makes the bottleneck visible.

**Run performance tests against staging, not production:**

Performance tests generate artificial load that can affect real users. Always run against a staging environment. Exceptions: smoke tests (minimal load, verifying production works) and chaos testing with proper safeguards.
