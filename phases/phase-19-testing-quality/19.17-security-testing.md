## 1. What Is Security Testing? (Theory)

Security testing finds vulnerabilities in your application before attackers do. It's not a one-time activity — it's integrated throughout the development lifecycle.

```
THREE CATEGORIES OF SECURITY TESTING:

SAST (Static Application Security Testing):
  Analyzes source code without running it.
  Finds: SQL injection, command injection, hardcoded secrets, 
         insecure cryptography, unsafe deserialization.
  When: during development, in CI on every commit.
  Tools: Bandit (Python), Semgrep, CodeQL, SonarQube.

DAST (Dynamic Application Security Testing):
  Tests the running application.
  Sends malicious inputs, analyzes responses.
  Finds: XSS, SQL injection, authentication flaws,
         sensitive data exposure, OWASP Top 10.
  When: against staging environment, pre-release.
  Tools: OWASP ZAP, Burp Suite.

DEPENDENCY SCANNING:
  Checks if your dependencies have known CVEs.
  Finds: outdated packages with security vulnerabilities.
  When: continuously, in CI.
  Tools: Dependabot, Snyk, OWASP Dependency Check.
```

## 2. SAST: Static Analysis

```bash
# BANDIT: Python security linter
pip install bandit

bandit -r src/
# Checks for:
# - SQL queries with string formatting (B608: SQL injection)
# - subprocess calls with shell=True (B602: shell injection)
# - hardcoded passwords (B105, B106)
# - use of assert for security checks (B101)
# - insecure hash functions (B303: MD5/SHA1)

# Example findings:
# >> Issue: [B608:hardcoded_sql_expressions] SQL query built with string formatting
#    Severity: High   Confidence: Medium
#    Location: src/repositories/user_repo.py:45
#    query = f"SELECT * FROM users WHERE email = '{email}'"
#
# >> Issue: [B105:hardcoded_password_string] Possible hardcoded password
#    Location: src/config.py:12
#    SECRET_KEY = "mysecretkey123"

# SEMGREP: more powerful, rule-based SAST
pip install semgrep

# Run OWASP rules
semgrep --config "p/owasp-top-ten" src/

# Custom rule: detect SQL injection pattern
# .semgrep/sql-injection.yaml:
rules:
  - id: sql-injection-format-string
    patterns:
      - pattern: f"SELECT ... {$VAR} ..."
      - pattern: "SELECT ... " + $VAR + " ..."
    message: Potential SQL injection via string formatting. Use parameterized queries.
    severity: ERROR
    languages: [python]
```

## 3. Dependency Scanning

```yaml
# GITHUB DEPENDABOT: automatic dependency vulnerability alerts
# .github/dependabot.yml

version: 2
updates:
  - package-ecosystem: "pip"
    directory: "/"
    schedule:
      interval: "weekly"
    reviewers:
      - "security-team"
    open-pull-requests-limit: 10
  
  - package-ecosystem: "npm"
    directory: "/"
    schedule:
      interval: "daily"  # daily for npm (more vulnerabilities)

# SNYK: more detailed vulnerability reports
# npm install -g snyk
snyk test        # check for vulnerabilities
snyk monitor     # continuous monitoring
snyk fix         # auto-apply patches where available

# OWASP DEPENDENCY CHECK: audit Python packages
pip install safety
safety check
# Checks installed packages against CVE database
# Output: 
# -> Vulnerability found in requests==2.20.0
#    CVE-2018-18074: ReDoS vulnerability
#    Fixed in: 2.20.1
```

## 4. DAST: Dynamic Testing with OWASP ZAP

```bash
# OWASP ZAP: the most widely used DAST tool

# Docker-based automated scan against staging
docker run -t owasp/zap2docker-stable zap-baseline.py \
    -t https://staging.yourcompany.com \
    -r zap_report.html

# Outputs:
# WARN: /auth/login: Missing X-Frame-Options header
# WARN: /api/users: Cookie missing SameSite attribute
# FAIL: /api/search: Reflected XSS vulnerability found
# INFO: /api/products: CSP header present

# FULL SCAN (more thorough, more aggressive):
docker run -t owasp/zap2docker-stable zap-full-scan.py \
    -t https://staging.yourcompany.com \
    -r zap_report.html \
    -l WARN

# ZAP automated scan tests for:
# - SQL injection
# - XSS (reflected and stored)
# - CSRF vulnerabilities
# - Missing security headers
# - Sensitive data exposure
# - Insecure session management
# - Path traversal
```

## 5. Integrating Security Testing in CI

```yaml
# .github/workflows/security.yml

name: Security Tests
on:
  push:
    branches: [main]
  pull_request:

jobs:
  sast:
    name: Static Analysis
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      
      - name: Run Bandit
        run: |
          pip install bandit
          bandit -r src/ -ll -ii  # only HIGH severity, HIGH confidence
          # -ll = medium level and above
          # -ii = medium confidence and above
      
      - name: Run Semgrep
        uses: returntocorp/semgrep-action@v1
        with:
          config: p/python.flask  # Flask-specific security rules
          
  dependency-scan:
    name: Dependency Vulnerabilities
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      
      - name: Check Python dependencies
        run: |
          pip install safety
          safety check --full-report
      
      - name: Check npm dependencies  
        run: npm audit --audit-level=high
  
  secrets-scan:
    name: Scan for Secrets
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      
      - name: Detect secrets
        uses: trufflesecurity/trufflehog@main
        with:
          path: ./
          base: ${{ github.event.repository.default_branch }}
          extra_args: --only-verified
```

## 6. Secret Scanning

```bash
# TRUFFLEHOG: scan for accidentally committed secrets
pip install trufflehog

# Scan entire git history
trufflehog git https://github.com/yourorg/yourrepo

# Scan local repository
trufflehog filesystem ./src/

# Finds: API keys, database passwords, private keys, 
#        JWT secrets, AWS credentials, Stripe API keys

# PRE-COMMIT HOOK: prevent secrets from being committed
pip install detect-secrets

# Initialize baseline (existing "false positives" to ignore)
detect-secrets scan > .secrets.baseline

# .pre-commit-config.yaml
repos:
  - repo: https://github.com/Yelp/detect-secrets
    rev: v1.4.0
    hooks:
      - id: detect-secrets
        args: ['--baseline', '.secrets.baseline']

# Now: git commit with a secret → pre-commit hook blocks the commit.
```

## 7. Senior Engineer Perspective

**Security testing should be boring and automated:**

The goal is to make security testing so integrated into the development workflow that it requires no special effort. SAST runs on every commit. Dependency scanning runs weekly. Secret scanning is a pre-commit hook. DAST runs against staging before every production deploy.

When security testing is integrated and automated, it finds problems early (when they're cheap to fix) rather than late (when they're embarrassing or expensive). A SQL injection found by Bandit during code review is a 5-minute fix. The same vulnerability found in production is a potential data breach, a post-mortem, and potential regulatory consequences.

**The most impactful tool is secret scanning with a pre-commit hook:**

Accidentally committed secrets (API keys, database passwords) are the most common and most immediately dangerous security issue. Once a secret is in git history — even if you delete it later — it's potentially compromised. A pre-commit hook that runs detect-secrets prevents this at the most effective intervention point.
