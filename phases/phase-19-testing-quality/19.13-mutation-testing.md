## 1. What Is Mutation Testing? (Theory)

Mutation testing answers the question: "If I introduce a bug into the code, would my tests catch it?"

It works by automatically making small code changes (mutations) — changing `>` to `>=`, swapping `+` to `-`, returning `None` instead of a value — then running the test suite. If the tests still pass after the mutation, the tests **failed to catch the bug**. The mutation "survived."

```
MUTATION TESTING PROCESS:

ORIGINAL CODE:
  def calculate_discount(price, rate):
      if rate > 0:          # ← mutation: change > to >=
          return price * (1 - rate)
      return price

MUTATION 1: rate >= 0
  Test suite runs...
  If tests still pass: SURVIVED (tests don't verify behavior at rate=0)
  This reveals: no test covers the case where rate == 0

MUTATION 2: return price * (1 + rate)  (changed - to +)
  Test suite runs...
  If tests still pass: SURVIVED (tests don't verify the calculation direction)
  This reveals: tests might only check "discount was applied" not "in the right direction"

MUTATION 3: return price (removed calculation)
  Test suite runs...
  If tests fail: KILLED (tests catch this mutation — tests are effective!)
```

## 2. Mutation Score

```
MUTATION SCORE = killed mutations / total mutations

Example:
  100 mutations generated
  85 killed (tests caught the introduced bug)
  15 survived (tests missed the bug)
  
  Mutation score = 85/100 = 85%

INTERPRETING THE SCORE:
  < 60%: tests have significant gaps
  60-80%: acceptable for many projects
  80-90%: good test suite
  > 90%: excellent, but diminishing returns
  
WHY MUTATION SCORE > CODE COVERAGE:
  Code coverage: "the line was executed"
  Mutation score: "the line was VERIFIED to be correct"
  
  You can have 100% code coverage with terrible tests:
    def test_calculate():
        calculate_discount(100, 0.1)  # runs the code but checks nothing
        assert True                    # always passes!
  
  Mutation testing catches this: the mutation survives because
  the test has no meaningful assertion.
```

## 3. Mutmut: Python Mutation Testing

```bash
# pip install mutmut

# Run mutation testing on a package
mutmut run --paths-to-mutate src/pricing.py

# Typical output:
# Running mutation testing...
# ⠦ 1/42 🎉 35  ⏰ 0  🤔 0  🙁 7  🔇 0
# 🎉 Killed: test caught the mutation
# 🙁 Survived: test missed the mutation!

# View survived mutations
mutmut results

# Investigate a specific surviving mutation
mutmut show 7
# Shows the mutation that survived:
# - if rate > 0:
# + if rate >= 0:
# (tests don't distinguish > vs >= for rate)
```

```python
# EXAMPLE: mutation testing reveals test gap

# CODE:
def calculate_discount(price: float, rate: float) -> float:
    if rate > 0:
        return price * (1 - rate)
    return price

# EXISTING TEST (weak):
def test_calculate_discount():
    result = calculate_discount(100, 0.1)
    assert result == 90  # only tests when rate > 0

# MUTATION: rate > 0 → rate >= 0
# RESULT: SURVIVED (test still passes because rate=0 not tested)

# AFTER MUTATION TESTING, ADD:
def test_zero_discount_returns_original_price():
    result = calculate_discount(100, 0)
    assert result == 100  # now tests the boundary

# MUTATION: rate > 0 → rate >= 0
# RESULT: NOW KILLED (test_zero_discount catches this!)
```

## 4. Stryker: Mutation Testing for JavaScript/TypeScript

```bash
# npm install --save-dev @stryker-mutator/core @stryker-mutator/typescript-checker

# stryker.config.json
{
    "packageManager": "npm",
    "reporters": ["html", "clear-text", "progress"],
    "testRunner": "jest",
    "coverageAnalysis": "perTest",
    "mutate": ["src/**/*.ts", "!src/**/*.spec.ts"]
}

# Run
npx stryker run

# Output:
# Mutation score: 76.92% (100/130)
# ✓ Killed: 100
# ✗ Survived: 30
# ⚰ Timed out: 0

# Opens HTML report showing each surviving mutation
```

## 5. Common Surviving Mutations and What They Mean

```
SURVIVING MUTATION               IMPLICATION

> changed to >=                  No test for boundary value (rate == 0)
return None instead of result    No test verifies return value is not None
condition deleted entirely       Tests don't verify the condition is needed
arithmetic + changed to -        Tests don't verify the direction of calculation
AND changed to OR in condition   Tests don't cover both conditions independently

WHAT TO DO WITH SURVIVING MUTATIONS:
  1. Is the mutation meaningful? Sometimes mutations are semantically equivalent
     (adding "" vs null check when both result in the same behavior)
  2. If meaningful: add a test that kills this mutation
  3. If not meaningful (equivalent mutation): document and ignore it
  
EQUIVALENT MUTATIONS (acceptable survivors):
  - i++ vs i+=1 (same behavior)
  - "abc" vs new String("abc") (same value)
  - x = x + 1 vs x += 1
  These are cosmetic changes — don't waste tests on them.
```

## 6. When to Use Mutation Testing

```
USE MUTATION TESTING FOR:
  ✓ Security-critical code (authentication, authorization, payment)
  ✓ Business logic with complex rules
  ✓ Code where bugs have high consequences
  ✓ After writing a new test suite (verify it's effective)
  ✓ Periodic audits of test quality

DON'T USE FOR:
  ✗ The entire codebase (too slow, thousands of mutations)
  ✗ Simple CRUD with no business logic
  ✗ Generated code
  ✗ Configuration code

PRACTICAL APPROACH:
  Run mutation testing on your most critical 20% of code.
  Set a mutation score gate (e.g., 80%) for those files.
  This catches the most important test gaps without being impractical.

PERFORMANCE:
  Mutation testing is slow: each mutation requires running the test suite.
  100 mutations × 30 second test suite = 50 minutes.
  Mitigation: run only on changed files in CI, parallelize, use coverage data
  to skip mutations in code not covered at all.
```

## 7. Senior Engineer Perspective

**Mutation testing is the test for your tests:**

Code coverage tells you lines were executed. Mutation testing tells you lines were verified. The difference is enormous. A team with 95% code coverage might have 60% mutation score — meaning 40% of their bugs would be missed by their test suite. Mutation testing reveals this.

Run mutation testing once a quarter on your critical code paths. Use it not to achieve a metric, but to find the actual gaps — the places where your tests look good on paper but don't verify the right things.
