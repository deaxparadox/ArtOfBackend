## 1. What Is Contract Testing? (Theory)

In a microservices architecture, services communicate over APIs. The challenge: Service A (consumer) calls Service B (provider). Who is responsible for making sure they stay compatible?

**Traditional approaches and their failures:**

```
APPROACH 1: E2E tests in staging
  Deploy all services, run E2E tests.
  Problem: slow feedback (requires deploying everything), 
           hard to isolate which service broke the contract.

APPROACH 2: Shared Postman/API collections
  Manually maintained documentation.
  Problem: documentation drifts from reality, no enforcement.

APPROACH 3: TRUST (the most common approach)
  Service B deploys a change, Service A breaks in production.
  "Oops, we didn't know Service A was using that field."
  This is the status quo at most companies.
```

**Contract testing's solution:**

The consumer (Service A) writes tests that define what it expects from the provider (Service B). These expectations become a contract. Service B's test suite verifies it can satisfy all consumer contracts before any deployment.

```
CONTRACT TESTING FLOW:

1. Service A (consumer) writes: 
   "When I call GET /users/{id}, I expect a response with:
    {id: integer, email: string, name: string}"

2. These expectations are saved as a "pact" (JSON file).

3. Service B (provider) runs this pact against itself:
   "Can I satisfy Service A's contract?"
   
4. If Service B tries to remove the 'email' field:
   → Service A's contract test FAILS
   → Service B cannot deploy until it either:
     a) Keeps the email field (backward compatible)
     b) Coordinates the removal with Service A first

WITHOUT PACT: Service B removes a field, Service A breaks in production.
WITH PACT: The break is caught before either service deploys.
```

## 2. Pact: Consumer-Driven Contract Testing

Pact is the most widely used contract testing framework. It supports multiple languages and uses a central Pact Broker to share contracts between teams.

```python
# Service A (consumer) — defines the contract

from pact import Consumer, Provider

pact = Consumer('order-service').has_pact_with(
    Provider('user-service'),
    pact_dir='./pacts',
)

# Test: what does order-service expect from user-service?
def test_get_user_by_id():
    pact.given("User u-123 exists").upon_receiving(
        "A request to get user u-123"
    ).with_request(
        method="GET",
        path="/users/u-123",
        headers={"Authorization": "Bearer test-token"},
    ).will_respond_with(
        status=200,
        headers={"Content-Type": "application/json"},
        body={
            "id": "u-123",
            "email": "alice@example.com",
            "name": "Alice",
        },
    )
    
    with pact:
        # This runs against a Pact mock server (not the real user-service)
        client = UserServiceClient(base_url=pact.uri)
        user = client.get_user("u-123")
        
        # Verify our client handles the response correctly
        assert user.id == "u-123"
        assert user.email == "alice@example.com"
    
    # After the test, pact saves the contract to:
    # ./pacts/order-service-user-service.json
```

```python
# Service B (provider) — verifies it can satisfy the contract

from pact import Verifier

def test_provider_satisfies_order_service_contract():
    verifier = Verifier(
        provider="user-service",
        provider_base_url="http://localhost:8080",
    )
    
    # Provider states: set up the right data before verification
    provider_states = {
        "User u-123 exists": setup_user_u123,
    }
    
    output, _ = verifier.verify_pacts(
        "./pacts/order-service-user-service.json",
        provider_states_setup_url="http://localhost:8080/_pact/provider_states",
    )
    
    assert output == 0  # 0 = all contracts satisfied
    
def setup_user_u123():
    """Create the test data state."""
    db.execute("INSERT INTO users (id, email, name) VALUES ('u-123', 'alice@example.com', 'Alice')")
```

## 3. The Pact Broker

The Pact Broker is a central registry for contracts. When consumer teams publish contracts and provider teams verify against them, the broker tracks compatibility:

```
PACT BROKER WORKFLOW:

Consumer team:
  1. Write tests defining expectations
  2. Run tests → generates pact file
  3. Publish pact to broker:
     $ pact-broker publish ./pacts/ \
       --broker-base-url https://your-broker.pactflow.io \
       --consumer-app-version $(git rev-parse HEAD) \
       --branch main

Provider team:
  1. CI pipeline fetches latest contracts from broker
  2. Runs verification against each contract
  3. Reports results to broker
  
  $ pact-broker can-i-deploy \
    --pacticipant user-service \
    --version $(git rev-parse HEAD) \
    --to-environment production
  → Answer: "Yes, all consumers verified" or "No, order-service contract fails"

DEPLOYMENT GATE:
  Before deploying to production:
  can-i-deploy verifies all consumer contracts pass.
  If a contract fails: deployment blocked.
  
  This prevents breaking changes from reaching production
  without coordination between teams.
```

## 4. What Contract Testing Catches

```
CONTRACT TESTS CATCH:
  ✓ Field renamed (email → email_address)
  ✓ Field removed (id field deleted from response)
  ✓ Type changed (id changed from string to integer)
  ✓ Status code changed (200 → 204)
  ✓ New required request field added (now requires Authorization header)

CONTRACT TESTS DON'T CATCH:
  ✗ Business logic bugs (user service returns wrong data but correct format)
  ✗ Performance issues
  ✗ Security vulnerabilities
  ✗ Protocol-level issues (these are still E2E tests)
```

## 5. When Contract Testing Is Worth It

```
USE CONTRACT TESTING WHEN:
  ✓ Multiple teams own different services
  ✓ Services deploy independently (high deployment frequency)
  ✓ API breaking changes have caused production incidents before
  ✓ You have 5+ microservices with interdependencies
  ✓ You want to reduce E2E test suite size

SKIP CONTRACT TESTING WHEN:
  ✗ Monolith (no service-to-service API contracts)
  ✗ One team owns all services (just test directly)
  ✗ Services rarely change
  ✗ Small team where verbal coordination works
  
CONTRACT TESTING OVERHEAD:
  Initial setup: 1-3 days to set up Pact Broker + CI integration
  Per-service: 2-4 hours to write initial consumer contracts
  Ongoing: maintaining contracts as APIs evolve (moderate)
  
  Worth it when: you have broken API contracts in production more than twice.
```

## 6. Senior Engineer Perspective

**Consumer-driven contracts invert the responsibility:**

Traditional API design is provider-driven: "here's our API, consumers figure it out." Consumer-driven contracts flip this: consumers specify what they need, providers verify they deliver it. This catches breaking changes before deployment and creates a documented, tested contract between teams.

The cultural shift is significant. Teams with good contract testing have explicit, automated conversations about API contracts. Instead of "we changed this field, hope it doesn't break anyone" — you get "the pact broker shows 3 consumers use this field, coordinate with them before removing it."
