## Quarkus (4.18)

Quarkus (2019, Red Hat) is "Supersonic, Subatomic Java" — a Java framework built from the ground up for containers, GraalVM native compilation, and Kubernetes.

```java
// Quarkus: near-identical programming model to Spring, but optimized for cloud
@Path("/users")
@Produces(MediaType.APPLICATION_JSON)
@Consumes(MediaType.APPLICATION_JSON)
public class UserResource {

    @Inject
    UserService userService;

    @GET
    @Path("/{id}")
    public Response getUser(@PathParam("id") Long id) {
        return userService.findById(id)
            .map(user -> Response.ok(user).build())
            .orElse(Response.status(404).build());
    }

    @POST
    public Response createUser(@Valid CreateUserRequest request) {
        User user = userService.create(request);
        return Response.status(201).entity(user).build();
    }
}
```

```bash
# Build GraalVM native binary (fast startup, low memory)
./mvnw package -Pnative

# Result: ~40ms startup vs Spring's 5-15 seconds
# Memory: 50MB vs Spring's 200-400MB
# Perfect for: Lambda, Kubernetes with rapid scaling
```

**When Quarkus > Spring Boot:**
- Serverless (Lambda, Cloud Run)
- Kubernetes with aggressive HPA (frequent scale-up/down)
- Resource-constrained environments
- When your team knows JEE/Spring but needs cloud-native performance

## Micronaut (4.19)

Micronaut (2018, Object Computing) compiles dependency injection at build time (not runtime like Spring). This means:
- Fast startup (no runtime classpath scanning)
- Low memory (no reflection-based proxy generation)
- AOT compilation compatible

```java
@Controller("/users")
public class UserController {

    @Inject  // resolved at compile time
    private UserService userService;

    @Get("/{id}")
    public HttpResponse<UserDto> getUser(Long id) {
        return userService.findById(id)
            .map(HttpResponse::ok)
            .orElse(HttpResponse.notFound());
    }
}
```

**Micronaut vs. Quarkus:** Similar goals (fast startup, low memory), different approaches. Quarkus has more momentum and better ecosystem in 2024. Either is a valid Spring alternative for cloud-native Java.

## Ktor (4.20)

Ktor (2018, JetBrains) is Kotlin's native, coroutine-based web framework. Unlike Spring (imperative model adapted for coroutines), Ktor is designed from the ground up for Kotlin coroutines.

```kotlin
import io.ktor.server.application.*
import io.ktor.server.engine.*
import io.ktor.server.netty.*
import io.ktor.server.routing.*
import io.ktor.server.response.*
import io.ktor.server.request.*

fun main() {
    embeddedServer(Netty, port = 8080, module = Application::module).start(wait = true)
}

fun Application.module() {
    install(ContentNegotiation) { json() }
    install(Authentication) {
        bearer("auth") {
            authenticate { tokenCredential ->
                verifyToken(tokenCredential.token)
            }
        }
    }

    routing {
        route("/api/v1") {
            authenticate("auth") {
                get("/users/{id}") {
                    val id = call.parameters["id"]?.toLongOrNull()
                        ?: return@get call.respond(HttpStatusCode.BadRequest)

                    val user = userService.findById(id)
                        ?: return@get call.respond(HttpStatusCode.NotFound)

                    call.respond(user)
                }

                post("/users") {
                    val request = call.receive<CreateUserRequest>()
                    val user = userService.create(request)
                    call.respond(HttpStatusCode.Created, user)
                }
            }
        }
    }
}
```

**Ktor vs. Spring Boot for Kotlin:**
- Ktor: lightweight, coroutine-native, less "magic", better for microservices
- Spring Boot: more features, larger ecosystem, more hiring pool
- For new Kotlin microservices: Ktor is increasingly the preferred choice
- For large applications or teams already on Spring: Spring Boot with Kotlin
