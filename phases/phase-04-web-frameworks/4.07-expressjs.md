## 1. What Is Express.js?

Express.js (2010, TJ Holowaychuk) is the minimal, unopinionated Node.js web framework that became the standard for Node.js web development. It provides HTTP request routing and a middleware model — and nothing else. Everything else (ORM, validation, auth) you add yourself.

Express's philosophy mirrors Flask: small, composable, no magic. The result: an ecosystem of thousands of Express middleware packages.

**By the numbers:** Express remains the most installed npm package for web frameworks, with billions of weekly downloads. Most Node.js backend code in the world is running on Express.

## 2. Express Core

```javascript
import express from 'express';
import { z } from 'zod';

const app = express();
app.use(express.json());         // parse JSON bodies
app.use(express.urlencoded({ extended: true }));  // parse form bodies

// Route handlers
app.get('/users/:id', async (req, res) => {
    const { id } = req.params;
    try {
        const user = await db.findUser(parseInt(id));
        if (!user) return res.status(404).json({ error: 'Not found' });
        res.json(user);
    } catch (error) {
        next(error);  // pass to error handler
    }
});

// Middleware-based validation
const createUserSchema = z.object({
    email: z.string().email(),
    name: z.string().min(1).max(100),
});

function validate(schema) {
    return (req, res, next) => {
        const result = schema.safeParse(req.body);
        if (!result.success) {
            return res.status(422).json({
                error: 'validation_failed',
                details: result.error.flatten(),
            });
        }
        req.validatedBody = result.data;
        next();
    };
}

app.post('/users', validate(createUserSchema), async (req, res, next) => {
    try {
        const user = await db.createUser(req.validatedBody);
        res.status(201).json(user);
    } catch (error) {
        next(error);
    }
});

// Error handling middleware (must have 4 params)
app.use((err, req, res, next) => {
    console.error(err.stack);
    res.status(500).json({ error: 'Internal server error' });
});

app.listen(3000, () => console.log('Server running on port 3000'));
```

## 4. Express Middleware Chain

Express's architecture is a linear chain of middleware functions. Each middleware can:
- Process the request
- Send a response
- Call `next()` to pass to the next middleware

```javascript
// Request flows through middleware in order:
app.use(requestIdMiddleware)   // adds X-Request-Id
app.use(loggerMiddleware)      // logs the request
app.use(rateLimitMiddleware)   // checks rate limits
app.use(corsMiddleware)        // adds CORS headers
app.use(authMiddleware)        // validates JWT (for protected routes)
app.use('/api', router)        // route handlers
app.use(errorMiddleware)       // catches any errors

// Middleware signature:
function myMiddleware(req, res, next) {
    // do work...
    next();        // pass to next middleware
    // or:
    next(error);   // pass error to error handler
    // or:
    res.json(...)  // send response (middleware chain ends)
}
```

## 5. Express vs. Fastify

```
EXPRESS:
  ✓ Mature, huge ecosystem
  ✓ Simple, minimal API
  ✓ Every Node.js dev knows it
  ✓ Best documentation coverage

  ✗ Slower than Fastify (no schema-based serialization)
  ✗ No built-in TypeScript support
  ✗ No JSON schema validation
  ✗ Async error handling is awkward (must use try/catch + next)

FASTIFY:
  ✓ 2-3x faster than Express for JSON APIs
  ✓ Schema-based JSON serialization (faster, smaller responses)
  ✓ Excellent TypeScript support
  ✓ Async/await native (no try/catch + next needed)
  ✓ Plugin system is cleaner than middleware

  ✗ Smaller ecosystem
  ✗ Different paradigm (plugins vs. middleware)

RECOMMENDATION:
  New TypeScript API project: Fastify
  Team already knows Express: Express is fine, no urgent reason to switch
  Legacy codebase: stay on Express
```

## 6. Senior Engineer Perspective

**What a junior gets wrong:**

Not handling async errors properly. Express doesn't automatically catch rejected Promises. In Express 4.x:

```javascript
// This will CRASH the server (unhandled promise rejection):
app.get('/users', async (req, res) => {
    const users = await db.getUsers();  // if this throws, crash
    res.json(users);
});

// RIGHT: wrap in try/catch + pass to next
app.get('/users', async (req, res, next) => {
    try {
        const users = await db.getUsers();
        res.json(users);
    } catch (error) {
        next(error);  // passes to error handler
    }
});

// OR: use express-async-errors npm package (patches Express to handle async)
import 'express-async-errors';
app.get('/users', async (req, res) => {
    const users = await db.getUsers();  // errors now auto-forwarded to next()
    res.json(users);
});
```

Express 5 (currently RC) handles async errors natively. Until then, `express-async-errors` is essential for async Express code.
