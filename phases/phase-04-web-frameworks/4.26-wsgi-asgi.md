## 1. What Are WSGI and ASGI?

WSGI and ASGI are interface specifications that define how Python web servers (Gunicorn, Uvicorn) communicate with Python web applications (Django, Flask, FastAPI).

Without a standard interface: every web server would need to know the internals of every web framework. The interface standard decouples servers from applications.

---

## 2. WSGI (Web Server Gateway Interface, PEP 3333)

WSGI (2010) is the synchronous interface specification. A WSGI application is a callable that takes `(environ, start_response)` and returns an iterable of response bytes.

```python
# Minimal WSGI application (what Django/Flask apps are at their core)
def application(environ: dict, start_response) -> list[bytes]:
    # environ: HTTP request as a dict
    # start_response: callable to set status/headers
    
    method = environ["REQUEST_METHOD"]   # "GET", "POST", etc.
    path = environ["PATH_INFO"]          # "/users/123"
    body = environ["wsgi.input"].read()  # request body as bytes

    # ... process request ...
    
    status = "200 OK"
    headers = [("Content-Type", "application/json")]
    start_response(status, headers)
    
    return [b'{"hello": "world"}']

# How servers run WSGI apps:
# gunicorn --workers 4 myapp:application
# uwsgi --wsgi-file myapp.py --callable application
```

**WSGI's limitation:** Each request occupies a thread until fully complete. No concurrency within a single worker. To handle 1000 concurrent requests: need 1000 threads (or processes). Resource-intensive.

---

## 3. ASGI (Asynchronous Server Gateway Interface, PEP 3333 successor)

ASGI (2018, Django Channels) extends WSGI to support async, WebSockets, and HTTP/2. An ASGI application is an async callable with a different signature.

```python
# Minimal ASGI application (what FastAPI/Starlette/Django 4.1+ apps are)
async def application(scope: dict, receive, send) -> None:
    # scope: connection info (type, path, headers, etc.)
    # receive: async callable to read incoming messages
    # send: async callable to send outgoing messages
    
    if scope["type"] == "http":
        # Read request body
        body = b""
        while True:
            message = await receive()
            body += message.get("body", b"")
            if not message.get("more_body"):
                break
        
        # Send response
        await send({
            "type": "http.response.start",
            "status": 200,
            "headers": [[b"content-type", b"application/json"]],
        })
        await send({
            "type": "http.response.body",
            "body": b'{"hello": "world"}',
        })
    
    elif scope["type"] == "websocket":
        # WebSocket handling
        await send({"type": "websocket.accept"})
        while True:
            message = await receive()
            if message["type"] == "websocket.disconnect":
                break
            await send({"type": "websocket.send", "text": message["text"]})

# How servers run ASGI apps:
# uvicorn myapp:application --workers 4
# gunicorn myapp:application -k uvicorn.workers.UvicornWorker -w 4
```

## 4. WSGI vs. ASGI Comparison

| | WSGI | ASGI |
|---|---|---|
| Programming model | Synchronous | Async-native |
| Concurrency | One request per thread/process | Many requests per thread (event loop) |
| WebSockets | No | Yes |
| Server-Sent Events | Limited | Yes |
| HTTP/2 server push | No | Yes |
| Frameworks | Django, Flask, Bottle | FastAPI, Starlette, Django 4.1+ |
| Servers | Gunicorn, uWSGI, mod_wsgi | Uvicorn, Daphne, Hypercorn |

## 5. Production Server Configuration

```bash
# WSGI: Gunicorn with sync workers
gunicorn myapp.wsgi:application \
    --workers $(( 2 * $(nproc) + 1 )) \   # 2*CPU + 1
    --bind 0.0.0.0:8000 \
    --timeout 30

# ASGI: Gunicorn + Uvicorn workers (recommended for FastAPI)
gunicorn myapp.main:app \
    --workers 4 \                           # typically 2-4× CPU cores
    --worker-class uvicorn.workers.UvicornWorker \
    --bind 0.0.0.0:8000 \
    --timeout 30 \
    --graceful-timeout 30

# ASGI: Pure Uvicorn (simpler, good for containers)
uvicorn myapp.main:app \
    --host 0.0.0.0 \
    --port 8000 \
    --workers 4 \          # Python 3.11+: multiprocessing with uvicorn
    --no-access-log        # use your own logging middleware
```

**Gunicorn + Uvicorn vs. pure Uvicorn:**
- Gunicorn + UvicornWorker: process management (restarts crashed workers, graceful reloads)
- Pure Uvicorn: simpler, let Kubernetes/systemd handle process management
- In containers: pure Uvicorn is often better (K8s replaces crashed pods)

## 6. Django's Migration to ASGI

Django has supported ASGI since 3.0 (2019), with channels-style consumers since earlier. Django 4.1+ added async ORM support. The migration path:

```python
# settings.py: ASGI application
ASGI_APPLICATION = "myproject.asgi.application"

# asgi.py (replaces wsgi.py)
import os
from django.core.asgi import get_asgi_application

os.environ.setdefault("DJANGO_SETTINGS_MODULE", "myproject.settings")
application = get_asgi_application()  # ASGI-compatible Django app

# Now deploy with: uvicorn myproject.asgi:application
```

For async Django views (4.1+):

```python
from django.views import View
from django.http import JsonResponse

class UserView(View):
    async def get(self, request, user_id):
        user = await User.objects.aget(pk=user_id)  # async ORM
        return JsonResponse({"id": user.id, "email": user.email})
```

## 7. Senior Engineer Perspective

**The practical difference:**

For most APIs, the WSGI vs. ASGI choice is a consequence of framework choice:
- Django/Flask: WSGI (or ASGI if you opt in)
- FastAPI/Starlette: ASGI (required)

The performance difference only matters at high concurrency with I/O-bound work. ASGI can handle 10,000+ concurrent requests with a single thread (waiting for I/O). WSGI would need 10,000 threads. For most services: WSGI + a pool of 4-8 worker processes handles the load fine.

**When ASGI's concurrency matters:**

WebSockets (persistent connections), long-polling, SSE (Server-Sent Events), streaming responses — any case where connections are held open for a long time. WSGI can't efficiently handle 1,000 concurrent WebSocket connections. ASGI can.
