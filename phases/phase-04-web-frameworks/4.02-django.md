## 1. What Is Django?

Django (2005, Adrian Holovaty and Simon Willison) is a "batteries-included" Python web framework that provides everything needed for a full web application out of the box: ORM, admin interface, authentication, forms, templating, URL routing, and more.

**Django's philosophy:**
- **Don't Repeat Yourself (DRY)** — common patterns built in
- **Convention over configuration** — sensible defaults, less setup
- **Loosely coupled, tightly cohesive** — components work together but stay independent
- **Security by default** — CSRF protection, SQL injection prevention, XSS protection built in

**Used by:** Instagram (scaled to 1B+ users on Django), Pinterest, Disqus, Bitbucket, Mozilla.

---

## 2. Django's Architecture

```
Django follows MTV (Model-Template-View) — similar to MVC:
  Model:    Defines data structure + database interaction (Django ORM)
  Template: HTML templates with Django template language
  View:     Business logic — receives requests, returns responses

For APIs (no templates needed):
  Model:    Django ORM models
  Serializer: DRF serializers (replaces templates for API output)
  View/ViewSet: DRF views (replaces HTML views)
```

## 3. The Django ORM — Models Define Everything

```python
# models.py
from django.db import models
from django.contrib.auth.models import AbstractUser

class User(AbstractUser):  # extends Django's built-in user
    phone = models.CharField(max_length=20, blank=True)
    avatar_url = models.URLField(blank=True)
    created_at = models.DateTimeField(auto_now_add=True)

    class Meta:
        indexes = [
            models.Index(fields=["email"]),
        ]

class Order(models.Model):
    STATUS_CHOICES = [
        ("pending", "Pending"),
        ("paid", "Paid"),
        ("shipped", "Shipped"),
        ("delivered", "Delivered"),
    ]

    user = models.ForeignKey(User, on_delete=models.CASCADE, related_name="orders")
    status = models.CharField(max_length=20, choices=STATUS_CHOICES, default="pending")
    total_cents = models.IntegerField()
    created_at = models.DateTimeField(auto_now_add=True)
    updated_at = models.DateTimeField(auto_now=True)

    class Meta:
        ordering = ["-created_at"]
        indexes = [
            models.Index(fields=["user", "status"]),
        ]

# Django ORM queries (extremely expressive)
# All return lazy QuerySets (evaluated when iterated)
orders = Order.objects.filter(
    user=user,
    status="pending",
    created_at__gte=thirty_days_ago,
).select_related("user")  # JOIN — prevents N+1

# Aggregations
from django.db.models import Sum, Count, Avg
stats = Order.objects.filter(status="paid").aggregate(
    total=Sum("total_cents"),
    count=Count("id"),
    average=Avg("total_cents"),
)
```

## 4. Django REST Framework (DRF)

For APIs, Django is almost always used with DRF:

```python
# serializers.py
from rest_framework import serializers
from .models import Order, User

class UserSerializer(serializers.ModelSerializer):
    class Meta:
        model = User
        fields = ["id", "email", "first_name", "last_name"]  # explicit fields
        read_only_fields = ["id"]

class OrderSerializer(serializers.ModelSerializer):
    user = UserSerializer(read_only=True)  # nested serializer

    class Meta:
        model = Order
        fields = ["id", "user", "status", "total_cents", "created_at"]

# views.py
from rest_framework import viewsets, permissions, filters
from rest_framework.decorators import action
from rest_framework.response import Response
from django_filters.rest_framework import DjangoFilterBackend

class OrderViewSet(viewsets.ModelViewSet):
    serializer_class = OrderSerializer
    permission_classes = [permissions.IsAuthenticated]
    filter_backends = [DjangoFilterBackend, filters.OrderingFilter]
    filterset_fields = ["status"]
    ordering_fields = ["created_at", "total_cents"]

    def get_queryset(self):
        # Users only see their own orders
        return Order.objects.filter(user=self.request.user).select_related("user")

    def perform_create(self, serializer):
        serializer.save(user=self.request.user)

    # Custom action: POST /orders/{id}/cancel/
    @action(detail=True, methods=["post"])
    def cancel(self, request, pk=None):
        order = self.get_object()
        if order.status not in ["pending"]:
            return Response({"error": "Can only cancel pending orders"}, status=400)
        order.status = "cancelled"
        order.save()
        return Response({"status": "cancelled"})

# urls.py
from rest_framework.routers import DefaultRouter
from .views import OrderViewSet

router = DefaultRouter()
router.register("orders", OrderViewSet, basename="order")
# Automatically creates: GET/POST /orders/, GET/PUT/DELETE /orders/{id}/,
#                        POST /orders/{id}/cancel/
```

## 5. Django Admin — The Killer Feature

```python
# admin.py
from django.contrib import admin
from .models import Order, User

@admin.register(Order)
class OrderAdmin(admin.ModelAdmin):
    list_display = ["id", "user", "status", "total_cents", "created_at"]
    list_filter = ["status", "created_at"]
    search_fields = ["user__email", "id"]
    readonly_fields = ["created_at", "updated_at"]
    ordering = ["-created_at"]

    actions = ["mark_as_shipped"]

    def mark_as_shipped(self, request, queryset):
        count = queryset.filter(status="paid").update(status="shipped")
        self.message_user(request, f"{count} orders marked as shipped.")
    mark_as_shipped.short_description = "Mark selected orders as shipped"
```

Visit `/admin/` — Django generates a full CRUD admin panel with search, filtering, bulk actions, and custom actions. **Without writing a single line of frontend code.** This is Django's most underrated feature for internal tools and content management.

## 6. Migrations

```bash
# Generate a migration from model changes
python manage.py makemigrations

# Apply migrations (runs SQL)
python manage.py migrate

# Show migration status
python manage.py showmigrations

# Squash migrations (optimize old ones)
python manage.py squashmigrations myapp 0001 0050
```

```python
# migrations/0002_add_phone_to_user.py (generated automatically)
from django.db import migrations, models

class Migration(migrations.Migration):
    dependencies = [("myapp", "0001_initial")]
    operations = [
        migrations.AddField(
            model_name="user",
            name="phone",
            field=models.CharField(blank=True, max_length=20),
        ),
    ]
```

## 7. When to Use Django vs. When It's Overkill

```
DJANGO IS THE RIGHT CHOICE WHEN:
  ✓ You need admin panel for content/data management
  ✓ Building a full-stack web application (with templates)
  ✓ Teams already familiar with Django's conventions
  ✓ Rapid development — built-in auth, ORM, admin out of the box
  ✓ CMS-style applications (Django CMS, Wagtail)
  ✓ E-commerce (django-oscar, django-shop)
  ✓ Projects where the ORM's maturity matters (complex queries, migrations)

DJANGO IS OVERKILL WHEN:
  ✗ Building a pure JSON API microservice (FastAPI is faster to develop with)
  ✗ The admin is not needed (you're paying for it even if unused)
  ✗ Async-first from day one (Django's async support is improving but not native)
  ✗ Simple REST API with < 10 endpoints (Flask + SQLAlchemy is simpler)

THE N+1 PROBLEM IN DJANGO:
  # WRONG: N+1 queries
  for order in Order.objects.all():
      print(order.user.email)  # each access hits DB

  # RIGHT: select_related for ForeignKey, prefetch_related for ManyToMany
  for order in Order.objects.select_related("user").all():
      print(order.user.email)  # single JOIN query

  # Always check query count in development:
  from django.test.utils import override_settings
  from django.db import connection
  # pip install django-debug-toolbar (shows queries in browser)
```

## 8. Senior Engineer Perspective

**What a junior gets wrong:**

Not using `select_related`/`prefetch_related` in DRF serializers. DRF will serialize nested objects, triggering database queries per object. With 100 orders, each with a user serializer, that's 100+ queries. Always define `queryset` in ViewSets with appropriate select_related, and profile with `django-debug-toolbar` to see actual query counts.

**What Django does better than anything else:**

The Django admin, combined with migrations, makes content management and data operations extremely fast to implement. In 30 minutes of Django model + admin configuration, you can have a functional backoffice that would take weeks to build from scratch. This is why Instagram, Pinterest, and countless startups chose Django early — not for performance, but for development speed.
