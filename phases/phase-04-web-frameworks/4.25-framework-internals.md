## 1. Routing Trees: How URLs Map to Handlers

All web frameworks maintain a routing tree that matches URLs to handlers. Understanding the implementation explains performance and behavioral differences.

**Radix tree (trie-based routing):**

```
Routes registered:
  GET  /users
  GET  /users/:id
  POST /users
  GET  /users/:id/orders
  GET  /users/:id/orders/:order_id

Internal tree structure:
  /
  └── users
      ├── (GET) → listUsers
      ├── (POST) → createUser
      └── /:id
          ├── (GET) → getUser
          └── /orders
              ├── (GET) → listOrders
              └── /:order_id
                  └── (GET) → getOrder

Lookup: GET /users/123/orders
  → traverse: /users → /:id (capture "123") → /orders → match GET handler
  → O(k) where k = depth of path, regardless of number of routes
```

Gin, Chi, and Echo all use radix trees. Express uses a linear search (O(n) routes). At > 1000 routes, Express routing becomes measurably slower.

## 2. Middleware Chain: The Execution Pipeline

Every framework implements middleware as a chain — functions that wrap the handler.

```
Request → Middleware 1 → Middleware 2 → Middleware 3 → Handler → ... → Response

Each middleware can:
  1. Process request before calling next
  2. Call next() / await next() / call_next(request)
  3. Process response after next returns
  4. Short-circuit (return response without calling next)
```

**Implementation in pseudocode:**

```python
# How Starlette/FastAPI implements middleware
class BaseHTTPMiddleware:
    def __init__(self, app, dispatch=None):
        self.app = app  # next middleware or handler
        self.dispatch_func = dispatch or self.dispatch

    async def __call__(self, scope, receive, send):
        # Wraps each request in a Request/Response object
        request = Request(scope, receive)
        response = await self.dispatch_func(request, self.call_next)
        await response(scope, receive, send)

    async def call_next(self, request):
        # Calls the next layer (self.app) with raw ASGI interface
        response_started = asyncio.Event()
        # ... ASGI protocol details ...
        return response
```

**The important detail:** Middleware ordering matters. First registered = outermost wrapper = runs first on request, last on response.

```python
# Request flows: A → B → C → Handler
# Response flows: Handler → C → B → A

app.add_middleware(A)  # outermost
app.add_middleware(B)
app.add_middleware(C)  # innermost (closest to handler)
```

## 3. Request Lifecycle in FastAPI

Tracing a single request through FastAPI's internals:

```
1. Uvicorn (ASGI server) receives HTTP bytes → parses → creates ASGI scope dict
2. Starlette middleware chain begins (outermost first)
3. FastAPI routing middleware: matches path + method to route
4. FastAPI dependency injection: resolves all Depends() in the route
   - Each Depends() is called (with its own dependencies resolved first)
   - yield-based deps run up to their yield point
5. Pydantic validation: validates path params, query params, body
   - If invalid: raises RequestValidationError → caught by exception handler → 422
6. Route handler executes (your async def function)
7. Return value → Pydantic serialization via response_model (if set)
8. Starlette Response object constructed
9. BackgroundTasks scheduled (if any)
10. Response flows back up through middleware chain
11. BackgroundTasks execute after response is sent
12. yield-based Depends() resume after their yield (cleanup)
```

## 4. What Makes a Framework "Fast"

```
FRAMEWORK OVERHEAD SOURCES:
  - Route matching (radix tree vs. linear scan)
  - Middleware count and complexity
  - Request/response object allocation
  - JSON parsing (request body deserialization)
  - JSON serialization (response body)
  - Validation overhead (schema parsing, type coercion)

FASTAPI OVERHEAD:
  - Pydantic v2 (Rust-based): ~0.1ms for typical model
  - Route matching: ~0.01ms
  - DI resolution: ~0.01ms per dependency
  - Total framework overhead: ~0.5ms for typical request

WHY FRAMEWORK OVERHEAD IS USUALLY IRRELEVANT:
  DB query:  5-50ms
  Network:   2-10ms
  Framework: 0.5ms

0.5ms is 1-10% of total request time. Optimizing framework overhead
while database queries take 20ms is premature optimization.
```

## 5. Senior Engineer Perspective

**What understanding internals enables:**

When FastAPI says "dependency was called but the response was already sent, so BackgroundTasks can't be scheduled" — you understand this because you know background tasks run after the response and dependencies clean up after background tasks. The error message makes sense.

When Gin's middleware doesn't seem to run in the right order — you understand that `gin.Use()` adds to the global chain and `group.Use()` adds to the group chain, and groups inherit globals but run their middleware first for routes in the group.

Framework internals explain the "why" behind counterintuitive behaviors. The hour spent reading your framework's source code pays dividends across years of debugging.
