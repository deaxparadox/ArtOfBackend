## 1. What Is CORS and Why Does It Exist?

CORS (Cross-Origin Resource Sharing) is a browser security mechanism that controls which websites can make HTTP requests to your API.

**The Same-Origin Policy:** Browsers enforce a rule: JavaScript from `https://app.example.com` can only make requests to `https://app.example.com`. Requests to a different origin — different scheme, domain, or port — are blocked by default.

```
SAME ORIGIN (allowed):
  https://app.example.com → https://app.example.com/api  ✓

CROSS ORIGIN (blocked by default):
  https://app.example.com → https://api.example.com      ✗ (different subdomain)
  https://app.example.com → http://app.example.com       ✗ (different scheme)
  https://app.example.com → https://app.example.com:8080 ✗ (different port)
  https://app.example.com → https://evil.com             ✗ (different domain)
```

**Why it exists:** Without Same-Origin Policy, any website could silently make authenticated API calls using the visitor's credentials. You browse `evil.com`, and its JavaScript reads your bank account, posts messages on your behalf, or exfiltrates your data — all using your session cookies.

**CORS is how you SELECTIVELY allow cross-origin requests.** It doesn't replace authentication — it's a browser-enforced access control layer.

## 2. How CORS Actually Works

### Simple Requests (No Preflight)

"Simple" requests (GET, POST with form content types, limited headers) are sent directly. The browser checks the response headers to decide if the JavaScript is allowed to read the response.

```
SIMPLE CORS REQUEST:

JS at https://app.example.com:
  fetch("https://api.example.com/users")

Browser sends:
  GET /users HTTP/1.1
  Host: api.example.com
  Origin: https://app.example.com   ← browser adds this

Server must respond with:
  Access-Control-Allow-Origin: https://app.example.com  (or *)

Browser checks:
  Response header present AND origin matches?
    YES → JavaScript receives the response ✓
    NO  → Browser blocks the response, JS gets a network error ✗
         (The request WAS made to the server, just JS can't read it)
```

### Preflight Requests (OPTIONS)

"Complex" requests (PUT, DELETE, PATCH, custom headers like Authorization, JSON body) trigger a preflight: the browser first sends an OPTIONS request to ask "are you willing to accept this request?"

```
PREFLIGHT REQUEST (automatic, before your actual request):

Browser sends:
  OPTIONS /users HTTP/1.1
  Host: api.example.com
  Origin: https://app.example.com
  Access-Control-Request-Method: POST
  Access-Control-Request-Headers: Content-Type, Authorization

Server must respond:
  HTTP/1.1 204 No Content
  Access-Control-Allow-Origin: https://app.example.com
  Access-Control-Allow-Methods: GET, POST, PUT, DELETE, PATCH, OPTIONS
  Access-Control-Allow-Headers: Content-Type, Authorization
  Access-Control-Max-Age: 86400   ← cache preflight for 24 hours

IF server allows:
  Browser sends actual request:
  POST /users HTTP/1.1
  Authorization: Bearer eyJ...
  Content-Type: application/json
  ...

IF server doesn't respond with correct CORS headers:
  Browser blocks the actual request entirely.
  JS sees: "TypeError: Failed to fetch"
```

**The preflight is invisible to most developers** — it happens automatically before your fetch/axios call. When you see a mysterious "preflight request failed" error, your server is not responding to OPTIONS correctly.

## 3. CORS Headers Reference

```
RESPONSE HEADERS (what your server returns):

Access-Control-Allow-Origin: https://app.example.com
  Which origin is allowed. Use specific origins, not * for authenticated requests.
  Cannot be * if credentials (cookies/auth headers) are involved.

Access-Control-Allow-Origin: *
  Allow all origins. ONLY for public APIs with no authentication.
  Cannot be combined with Access-Control-Allow-Credentials: true.

Access-Control-Allow-Methods: GET, POST, PUT, DELETE, PATCH, OPTIONS
  Which HTTP methods are allowed.

Access-Control-Allow-Headers: Content-Type, Authorization, X-Request-Id
  Which request headers the client is allowed to send.

Access-Control-Expose-Headers: X-Request-Id, X-Total-Count
  Which response headers JavaScript is allowed to read.
  By default, only "safe" headers (Content-Type etc.) are exposed.
  Custom headers (X-Request-Id) must be explicitly exposed.

Access-Control-Max-Age: 86400
  How long to cache the preflight response (seconds).
  Reduces preflight overhead significantly.
  Browser caps this (Chrome: 7200s max, Firefox: 86400s max).

Access-Control-Allow-Credentials: true
  Allow cookies, Authorization headers, TLS client certificates.
  ONLY use with specific origins (not *).
  Required when your frontend sends cookies to your API.
```

## 4. Common CORS Configurations

```python
from fastapi import FastAPI
from fastapi.middleware.cors import CORSMiddleware

app = FastAPI()

# CONFIGURATION 1: Single-page app on different domain
app.add_middleware(
    CORSMiddleware,
    allow_origins=["https://app.example.com"],  # specific origin only
    allow_credentials=True,                      # allow cookies/auth headers
    allow_methods=["GET", "POST", "PUT", "DELETE", "PATCH", "OPTIONS"],
    allow_headers=["Authorization", "Content-Type", "X-Request-Id"],
    expose_headers=["X-Request-Id", "X-Total-Count"],
    max_age=86400,
)

# CONFIGURATION 2: Public API (no auth, anyone can use)
app.add_middleware(
    CORSMiddleware,
    allow_origins=["*"],     # any origin
    allow_credentials=False, # no cookies
    allow_methods=["GET"],   # read-only public API
    allow_headers=["*"],
)

# CONFIGURATION 3: Multiple allowed origins (dev + staging + prod)
import os

ALLOWED_ORIGINS = [
    "https://app.example.com",
    "https://staging.example.com",
]
if os.getenv("ENV") == "development":
    ALLOWED_ORIGINS.append("http://localhost:3000")
    ALLOWED_ORIGINS.append("http://localhost:5173")

app.add_middleware(
    CORSMiddleware,
    allow_origins=ALLOWED_ORIGINS,
    allow_credentials=True,
    allow_methods=["*"],
    allow_headers=["*"],
    max_age=3600,
)
```

## 5. CORS in Nginx (for non-framework config)

```nginx
server {
    location /api/ {
        # Handle preflight
        if ($request_method = OPTIONS) {
            add_header Access-Control-Allow-Origin "https://app.example.com" always;
            add_header Access-Control-Allow-Methods "GET, POST, PUT, DELETE, PATCH, OPTIONS" always;
            add_header Access-Control-Allow-Headers "Authorization, Content-Type" always;
            add_header Access-Control-Max-Age 86400;
            add_header Content-Length 0;
            return 204;
        }

        # Add CORS headers to all responses
        add_header Access-Control-Allow-Origin "https://app.example.com" always;
        add_header Access-Control-Allow-Credentials true always;

        proxy_pass http://backend;
    }
}
```

## 6. CORS Is Not a Security Mechanism for Your API

**Critical misconception:** CORS does not prevent non-browser clients from calling your API. `curl`, `requests`, Postman, mobile apps, server-to-server calls — none of them send an `Origin` header or enforce the Same-Origin Policy.

CORS only protects against **browser-based cross-origin JavaScript**. It is a browser feature, not a server security feature.

```
CORS protects against: evil.com's JS making requests using your user's browser session
CORS does NOT protect against: curl, Postman, mobile apps, server-to-server calls

For API security, you still need:
  ✓ Authentication (JWT, API keys)
  ✓ Authorization (RBAC)
  ✓ Rate limiting
  ✓ Input validation
```

## 7. Debugging CORS Errors

```
"Access to fetch at 'https://api.example.com' from origin 
 'https://app.example.com' has been blocked by CORS policy"

DEBUGGING STEPS:

1. Open browser DevTools → Network tab
2. Find the failing request
3. Check if there's an OPTIONS preflight before it
4. Look at the response headers of the OPTIONS request:
   - Is Access-Control-Allow-Origin present?
   - Does it match the requesting origin exactly (https, domain, port)?

COMMON MISTAKES:
  ✗ "https://app.example.com " (trailing space in config)
  ✗ "http://app.example.com"   (wrong scheme — http vs https)
  ✗ "app.example.com"          (missing scheme entirely)
  ✗ Using * with credentials: browser blocks this
  ✗ Server returns 401 on OPTIONS preflight (auth middleware runs before CORS)
     → CORS middleware must run BEFORE auth middleware

5. Check server logs — the OPTIONS request should reach your server.
   If it doesn't: firewall or load balancer is blocking OPTIONS method.
```

## 8. Senior Engineer Perspective

**What a junior gets wrong:**

Setting `allow_origins=["*"]` with `allow_credentials=True`. Browsers reject this combination — the spec prohibits it. You must list specific origins when credentials are involved.

The other classic mistake: putting authentication middleware BEFORE CORS middleware. The preflight OPTIONS request has no Authorization header (browsers never add it to preflight). If auth middleware runs first, it rejects OPTIONS with 401, the browser never sees the CORS headers, and JavaScript sees a network error. **CORS middleware must run first.**

**What a senior thinks about:**

CORS is a boundary of trust. Adding an origin to your allowlist means "I trust JavaScript from this origin to make authenticated requests on behalf of my users." Only add origins you control. Never use `*` for authenticated APIs.

For internal APIs (only called by your own backend services), disable CORS entirely — server-to-server calls don't need it and it adds unnecessary configuration complexity.
