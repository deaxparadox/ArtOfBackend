## 1. The Differences That Matter

| | TLS 1.2 | TLS 1.3 |
|---|---|---|
| Handshake RTTs | 2 RTTs | 1 RTT |
| 0-RTT resumption | No | Yes (with trade-offs) |
| Cipher suites | 37 (many weak) | 5 (all strong) |
| Forward secrecy | Optional | Mandatory |
| Static RSA key exchange | Allowed | Removed |
| Client sends data after | 2nd RTT | 1st RTT |
| Vulnerabilities | BEAST, POODLE, ROBOT | None known (2024) |
| Adoption | Universal (fallback) | ~95% of new TLS connections |

**You must support TLS 1.3. You should disable TLS 1.0 and 1.1. You may keep TLS 1.2 as fallback for old clients.**

---

## 2. Why TLS 1.3 Is Faster

```
TLS 1.2 HANDSHAKE (2 RTTs):
  → ClientHello
  ← ServerHello, Certificate, ServerKeyExchange, ServerHelloDone
  → ClientKeyExchange, ChangeCipherSpec, Finished
  ← ChangeCipherSpec, Finished
  → [First HTTP request]    ← 2 full RTTs before data

TLS 1.3 HANDSHAKE (1 RTT):
  → ClientHello + KeyShare  (client sends key exchange upfront)
  ← ServerHello + KeyShare + Certificate + CertificateVerify + Finished
  → Finished + [First HTTP request]  ← 1 RTT only
  ← [HTTP response]

On a 100ms RTT connection:
  TLS 1.2: 200ms before first byte
  TLS 1.3: 100ms before first byte
  Savings: 100ms per new connection
```

## 3. Forward Secrecy

**The most important security improvement in TLS 1.3.**

Without forward secrecy (TLS 1.2 with RSA key exchange):
```
Attacker records encrypted traffic today.
Server's private key is compromised later (breach, legal compulsion).
Attacker decrypts ALL previously recorded traffic retroactively.
```

With forward secrecy (all TLS 1.3, TLS 1.2 with ECDHE):
```
Each session generates a fresh ephemeral key pair (Diffie-Hellman).
Session key derived from ephemeral keys + not stored anywhere.
Server private key compromise does NOT decrypt past sessions.
Past sessions are permanently protected.
```

TLS 1.3 mandates forward secrecy. No configuration needed — you get it automatically.

## 4. What to Disable

```nginx
# CORRECT: TLS 1.2 minimum, prefer 1.3
ssl_protocols TLSv1.2 TLSv1.3;

# TLS 1.2 cipher suites (only strong ones)
ssl_ciphers ECDHE-RSA-AES128-GCM-SHA256:ECDHE-RSA-AES256-GCM-SHA384:ECDHE-RSA-CHACHA20-POLY1305;

# WRONG: never enable these
# ssl_protocols TLSv1 TLSv1.1;  ← known vulnerabilities, deprecated
# ssl_ciphers RC4:3DES:NULL;    ← cryptographically broken
```

```bash
# Test your TLS configuration
# Online: https://www.ssllabs.com/ssltest/
# Locally:
nmap --script ssl-enum-ciphers -p 443 api.example.com
openssl s_client -connect api.example.com:443 -tls1_2  # force TLS 1.2
openssl s_client -connect api.example.com:443 -tls1_3  # force TLS 1.3
```

## 5. Senior Engineer Perspective

**The check every senior runs before launching:**

```bash
# SSL Labs grade should be A or A+
curl "https://api.ssllabs.com/api/v3/analyze?host=api.example.com&publish=off" | \
  python3 -c "import sys,json; d=json.load(sys.stdin); print(d['endpoints'][0]['grade'])"
# A+
```

**What you must fix if your grade is below A:**
- B: TLS 1.0/1.1 enabled → disable them
- C: Weak cipher suites → remove RC4, 3DES, export ciphers
- F: Certificate issues or very old TLS → immediate action required

The configuration shown in topic 1.10 achieves A+ on SSL Labs. Use it as your template.
