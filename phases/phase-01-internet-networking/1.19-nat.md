## 1. What Is NAT?

NAT (Network Address Translation) translates IP addresses as packets cross a network boundary. The most common use: allowing many devices with private IP addresses to share one public IP address for internet access.

**Why it exists:** IPv4 has ~4.3 billion addresses. There are 8+ billion internet-connected devices. The math doesn't work without NAT.

---

## 2. How NAT Works

```
HOME NETWORK EXAMPLE:

Private network: 192.168.1.0/24
  Your laptop:  192.168.1.10
  Your phone:   192.168.1.11
  Your TV:      192.168.1.12

Router public IP: 203.0.113.5 (your ISP-assigned IP)

Your laptop sends:
  Source IP:   192.168.1.10:54321  (private, not internet-routable)
  Dest IP:     142.250.191.46:443  (Google)

Router NAT table:
  Maps 192.168.1.10:54321 → 203.0.113.5:40001 (assigned port)

Router rewrites and forwards:
  Source IP:   203.0.113.5:40001  (now the router's public IP)
  Dest IP:     142.250.191.46:443

Google responds to:
  203.0.113.5:40001

Router sees response, looks up NAT table:
  203.0.113.5:40001 → 192.168.1.10:54321

Router translates and forwards to laptop:
  Your laptop receives the response.
```

The NAT table tracks every active connection so responses get correctly routed back. This is **stateful NAT** — the router maintains state per connection.

## 3. Types of NAT

```
SNAT (Source NAT):
  Rewrites source IP/port of outgoing packets.
  Most common: home routers, AWS NAT Gateway.
  Use: private → internet (outbound traffic)

DNAT (Destination NAT):
  Rewrites destination IP/port of incoming packets.
  Use case: port forwarding, load balancing.
  Example: router exposes port 80 → internal server at 192.168.1.100:8080

PAT (Port Address Translation) / NAPT:
  Maps many private (IP:port) → one public IP with different ports.
  What home routers actually do. Often called "NAT" colloquially.

1:1 NAT:
  Maps one private IP to one public IP (no port rewriting).
  AWS Elastic IP assigned directly to a private instance.
```

## 4. NAT in AWS

```
PRIVATE SUBNET → INTERNET:
  EC2 instances in private subnets have no public IP.
  NAT Gateway in public subnet.
  Private → NAT Gateway → Internet Gateway → Internet
  
  Cost: ~$0.045/hour + $0.045/GB processed (per AZ)
  Use: give private instances outbound internet access (download packages, call APIs)

PUBLIC SUBNET (direct internet):
  EC2 instance with Elastic IP or public IP assigned.
  Instance → Internet Gateway → Internet
  Inbound internet requests reach the instance directly.
  Use: load balancers, bastion hosts, services needing inbound internet access

NAT GATEWAY IS ONE-WAY:
  Outbound: Private instances can reach internet ✓
  Inbound: Internet CANNOT initiate connections to private instances ✓
  This is a security property — private instances are protected.
```

## 5. NAT Limitations

```
PROBLEM 1: Breaks peer-to-peer connections
  Both devices behind NAT want to connect directly.
  Neither has a public IP. Neither can be reached from outside.
  Solution: STUN/TURN servers (used by WebRTC, gaming, VoIP)
  STUN: helps peers discover each other's NAT-translated address
  TURN: relay server when direct connection is impossible

PROBLEM 2: Connection tracking table exhaustion
  Each NAT translation uses an entry in the state table.
  High-traffic NAT devices can exhaust this table.
  Symptom: "Cannot assign requested address" errors at scale.

PROBLEM 3: IPv6 makes NAT unnecessary
  Every device gets a globally unique IPv6 address.
  No address sharing needed. Direct connections everywhere.
  This is why IPv6 proponents see NAT as a "necessary evil" to eliminate.

PROBLEM 4: Some protocols embed IP addresses in their payload
  FTP, SIP (VoIP) include the client's IP inside the application data.
  NAT only rewrites the IP header, not the payload.
  The private IP in the payload doesn't match the translated IP.
  Solution: ALG (Application Layer Gateway) patches these protocols.
```

## 6. Senior Engineer Perspective

**What a junior gets wrong:**

Assuming NAT Gateway is free. At AWS, NAT Gateway costs $0.045/hour per AZ (~$32/month) + $0.045/GB processed. A service processing 10TB/month of outbound traffic: $450/month just in NAT data processing fees. This adds up fast. Review your NAT Gateway usage. If your service primarily talks to other AWS services in the same region, use VPC endpoints (free) instead of going through NAT.

**What a senior optimizes:**

VPC endpoints for S3, DynamoDB, and other AWS services bypass NAT entirely — traffic stays on the AWS network (faster, free). For services in the same VPC, use private IP addresses directly — no NAT involved at all.
