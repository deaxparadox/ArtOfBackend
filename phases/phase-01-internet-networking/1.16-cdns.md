## 1. What Is a CDN?

A CDN (Content Delivery Network) is a globally distributed network of servers that caches and serves content from locations close to users, reducing latency and offloading traffic from your origin server.

**The physics problem CDNs solve:** Data cannot travel faster than light. A user in Mumbai requesting content from a server in Virginia waits ~160ms for the round trip — before any processing. A CDN edge node in Mumbai serves the same content in ~5ms.

**The real-world analogy:** Amazon doesn't ship every package from one warehouse. They have regional fulfillment centers so packages arrive in 1–2 days instead of weeks. CDNs do the same for internet content.

---

## 2. How CDNs Work

```
WITHOUT CDN:
  User (Mumbai) ──────────────────────────────► Origin (Virginia)
  RTT: ~160ms × 2 = 320ms + server processing

WITH CDN:
  User (Mumbai) ──► CDN Edge (Mumbai) ──────────► Origin (Virginia)
                    RTT: ~5ms                     Only on cache miss

  First user requesting /logo.png:
    Edge → miss → fetches from origin → caches → returns to user
  
  All subsequent users:
    Edge → hit → returns from cache (~5ms, no origin involved)
```

### PoPs (Points of Presence)

CDN providers operate physical server clusters worldwide called PoPs:

```
CLOUDFLARE: 300+ PoPs across 100+ countries
AWS CloudFront: 400+ edge locations
Fastly: 95+ PoPs
Akamai: 4000+ PoPs (largest)

A PoP contains:
  - High-performance servers with large SSDs (TBs of cache)
  - Multiple 10/100 Gbps network links
  - Direct peering with local ISPs (avoiding public internet routing)
  - Anycast routing (all PoPs share the same IP; nearest PoP answers)
```

### Anycast Routing

CDNs use **anycast** — multiple PoPs advertise the same IP address via BGP. The internet routes each user to the nearest one automatically.

```
Cloudflare's IP 1.1.1.1 is advertised from 300+ locations.
When you ping 1.1.1.1:
  User in Tokyo → hits Tokyo PoP (~2ms)
  User in London → hits London PoP (~5ms)
  User in São Paulo → hits São Paulo PoP (~3ms)

No DNS tricks needed. Routing handles it automatically.
```

## 3. What CDNs Cache

### Static Assets (Always Cache)

```
Images, CSS, JS, fonts, videos, PDFs:
  Cache-Control: public, max-age=31536000, immutable
  Immutable = "this URL's content will never change"
  Common pattern: content-hashed filenames
    /static/bundle.a3f9d2.js  → hash changes if content changes
    → Can cache forever safely
```

### Dynamic API Responses (Cache Selectively)

```
Public data that doesn't change often:
  Cache-Control: public, max-age=60, stale-while-revalidate=30
  "Cache for 60s. If stale but <90s old, serve stale + refresh in background."
  Use for: product listings, public content, weather data

User-specific data:
  Cache-Control: private, no-store
  CDN never caches. Goes to origin every time.
  Use for: user dashboards, account pages, personalized content
```

### Streaming and Large Files

CDNs cache video chunks, large file downloads, software packages. They handle byte-range requests so video scrubbing works.

## 4. Cache Invalidation at the CDN

The hardest CDN problem: you deployed new code, but the CDN is still serving old cached responses.

```
STRATEGIES:

1. Cache-busting via URL versioning (best for static assets):
   Old: /static/app.js          → old content cached at edge
   New: /static/app.a3f9d2.js   → new URL, no cached entry, fetches fresh
   Downside: requires updating all references to the URL

2. Explicit purge/invalidation API:
   Cloudflare: POST /zones/{zone_id}/purge_cache
   AWS CloudFront: create invalidation for path patterns
   Cost: CloudFront charges per invalidation path after first 1000/month
   Downside: propagation takes 1-30 seconds

3. Short TTL + stale-while-revalidate:
   Cache-Control: public, max-age=30, stale-while-revalidate=300
   Content is "stale" after 30s, refreshed in background.
   Maximum staleness: 330s. Simple, no invalidation needed.
   Use for: frequently updated content where brief staleness is OK

4. Surrogate-Cache-Control / Cache-Tag:
   Cloudflare supports Cache-Tag for fine-grained invalidation.
   Tag responses: Cache-Tag: user-123, product-456
   Purge by tag: invalidate everything tagged product-456
   Use for: purging all pages containing a specific product after edit
```

## 6. Edge Functions — CDNs That Compute

Modern CDNs run code at the edge — reducing latency for dynamic operations:

```
CLOUDFLARE WORKERS:
  JavaScript (V8 isolates) at 300+ PoPs.
  Cold start: ~0ms (isolates, not containers).
  Use for: auth at the edge, A/B testing, request transformation,
           geo-routing, bot detection.

CLOUDFLARE D1 / KV / R2:
  D1: SQLite database at the edge
  KV: Global key-value store
  R2: Object storage (S3-compatible, no egress fees)
  Together: full application stack at the edge.

VERCEL EDGE FUNCTIONS / NETLIFY EDGE:
  Run Next.js middleware at the edge.
  Personalization without round trip to origin.

USE EDGE COMPUTE WHEN:
  - Auth/session validation (50ms → 5ms)
  - Geo-based redirects (language, region)
  - A/B testing (no origin needed)
  - Lightweight API responses from KV/D1
```

## 7. Code Examples

### Setting correct CDN headers

```python
from fastapi import FastAPI, Response
import hashlib, json

app = FastAPI()

@app.get("/api/products")
async def get_products(response: Response):
    products = await db.get_all_products()

    # Public endpoint — CDN can cache
    response.headers["Cache-Control"] = "public, max-age=60, stale-while-revalidate=300"
    response.headers["Vary"] = "Accept-Encoding"  # cache separate gzip/non-gzip versions

    # Add ETag for conditional requests
    content_hash = hashlib.md5(json.dumps(products, sort_keys=True).encode()).hexdigest()
    response.headers["ETag"] = f'"{content_hash}"'

    return products

@app.get("/api/users/me")
async def get_my_profile(current_user = Depends(get_current_user), response: Response):
    # Private — never cache at CDN
    response.headers["Cache-Control"] = "private, no-store"
    return current_user
```

### Purging Cloudflare cache programmatically

```python
import httpx

async def purge_cloudflare_cache(zone_id: str, api_token: str, urls: list[str]):
    """Purge specific URLs from Cloudflare cache after content update."""
    async with httpx.AsyncClient() as client:
        response = await client.post(
            f"https://api.cloudflare.com/client/v4/zones/{zone_id}/purge_cache",
            headers={"Authorization": f"Bearer {api_token}"},
            json={"files": urls},  # up to 30 URLs per request
        )
        result = response.json()
        if not result["success"]:
            raise Exception(f"Cache purge failed: {result['errors']}")
        return result

# Call after updating products
await purge_cloudflare_cache(
    zone_id="abc123",
    api_token=os.environ["CF_API_TOKEN"],
    urls=["https://example.com/api/products", "https://example.com/products"]
)
```

## 8. Senior Engineer Perspective

**What a junior gets wrong:**

Two opposite mistakes: caching everything with long TTLs (users see stale data after deploys) or caching nothing (origin overloaded, slow responses, CDN unused). The right approach is per-endpoint cache strategy: public static data gets long TTLs with versioned URLs, public dynamic data gets short TTLs with stale-while-revalidate, user-specific data never touches CDN cache.

**What a senior thinks about:**

CDN as the first line of defense against traffic spikes. A news site that goes viral can receive 100x normal traffic in minutes. Without CDN, the origin falls over. With CDN, 99% of requests are cache hits that never reach the origin. Origin traffic stays stable regardless of CDN traffic.

Also: CDN as a DDoS mitigation layer. Cloudflare absorbs terabits of attack traffic before it reaches your origin. This is why essentially every production service should sit behind a CDN, even if you're not optimizing for performance.
