## 1. What Is It? (Theory)

TCP (Transmission Control Protocol) is the transport protocol that makes the internet reliable. It turns the unreliable, best-effort IP layer into a **reliable, ordered, error-checked byte stream** between two endpoints.

Without TCP: packets arrive out of order, duplicated, or not at all. Your browser would occasionally download corrupted files, your API responses would contain garbage bytes, your database writes would silently fail.

With TCP: the application layer sees a clean stream of bytes, in order, with guaranteed delivery. TCP handles all the messiness of packet loss, reordering, and duplication invisibly.

**The real-world analogy:**

IP is like the postal system — best effort, no guarantees. A letter might arrive, might not, might arrive before an earlier letter. TCP is like a professional courier service with delivery confirmation, tracking numbers, and reordering — they guarantee your packages arrive in sequence, ask you to sign for each one, and resend anything that gets lost.

**The cost:** All that reliability requires round trips, acknowledgements, and state tracking. TCP is inherently slower than UDP for scenarios where some loss is acceptable (streaming, gaming). For everything else — HTTP, database connections, SSH — TCP is the right choice.

## 2. How It Actually Works (Deep Mechanics)

### The Three-Way Handshake

Before any data is exchanged, TCP establishes a **connection** using a three-way handshake. This ensures both parties are reachable and synchronizes initial sequence numbers.

```
CLIENT                              SERVER
  │                                   │
  │──── SYN (seq=x) ─────────────────►│  "I want to connect.
  │                                   │   My sequence starts at x."
  │                                   │
  │◄─── SYN-ACK (seq=y, ack=x+1) ────│  "OK, I'm here.
  │                                   │   My sequence starts at y.
  │                                   │   I received up to x."
  │                                   │
  │──── ACK (ack=y+1) ───────────────►│  "Got it. Let's go."
  │                                   │
  │◄═══════════ DATA ════════════════►│  (application data flows)
  │

Cost of handshake: 1 full RTT before any data can be sent.
NYC → London RTT: ~80ms → 80ms before first byte.
```

**SYN = Synchronize.** SYN packets establish the initial sequence number — a random number that identifies the starting point of the byte stream. This prevents old, delayed packets from being mistaken for new data.

**Why is the sequence number random?** To prevent TCP session hijacking. If sequence numbers were predictable, an attacker could forge packets with the correct sequence number and inject data into the stream.

---

### Sequence Numbers and Acknowledgements

TCP is a byte-stream protocol. Every byte has a sequence number. The receiver tracks which bytes it has received and sends **ACK** (acknowledgement) packets telling the sender.

```
Sender sends:
  Segment 1: bytes  1–1460  (seq=1)
  Segment 2: bytes 1461–2920 (seq=1461)
  Segment 3: bytes 2921–4380 (seq=2921)

Receiver ACKs:
  ACK 1461  (received segment 1, expecting next byte at 1461)
  ACK 2921  (received segment 2)
  ACK 4381  (received segment 3, all received)

If segment 2 is lost:
  Receiver sends ACK 1461 again (duplicate ACK)
  Sender detects duplicate ACKs → retransmits segment 2
  Receiver sends ACK 4381 after receiving 2 (already has 3 buffered)
```

This is **cumulative acknowledgement** — the ACK number says "I have received everything up to this byte." If bytes 1–1460 and 2921–4380 arrive but 1461–2920 is lost, the receiver sends ACK 1461 (not ACK 4381) — it won't ACK beyond the gap.

---

### The TCP Window: Flow Control

TCP uses a **receive window** to prevent the sender from overwhelming the receiver's buffer. The receiver advertises how much buffer space it has. The sender cannot transmit more than the window size of unacknowledged data.

```
Receive window = 64KB (receiver has 64KB of buffer space)

Sender can have up to 64KB of unacknowledged data "in flight":
  Sent but not ACK'd: [segment 1][segment 2][segment 3]...[segment N]
  Total unacked: ≤ 64KB

When receiver processes data, its buffer frees up → sends window update
Sender can send more.

If receiver is slow (overloaded), window shrinks → sender slows down.
If receiver is fast, window grows → sender speeds up.
```

**Window size and throughput:**

Recall from [0.5]: Bandwidth Delay Product = Bandwidth × RTT

On a 1 Gbps link with 100ms RTT:
- BDP = 1 Gbps × 0.1s = 100 Mbit = ~12 MB
- TCP needs a 12 MB window to fully utilize the link
- Default TCP window: 65 KB → **uses only 0.5% of available bandwidth!**
- TCP Window Scaling (RFC 1323): enables up to 1 GB windows → must be enabled for high-BDP paths

Modern OS kernels have TCP window scaling enabled by default, but it's worth verifying on production servers.

---

### Congestion Control

Flow control prevents overwhelming the **receiver**. Congestion control prevents overwhelming the **network** (routers, links between sender and receiver).

TCP infers network congestion from **packet loss** (if a packet is dropped, a router somewhere was overloaded). When it detects loss, it backs off.

**The four phases:**

```
SLOW START:
  Start with 1 MSS (Max Segment Size, ~1460 bytes).
  Double the congestion window every RTT.
  1 → 2 → 4 → 8 → 16 → 32 MSS...
  (exponential growth)
  Until: loss OR slow start threshold reached

CONGESTION AVOIDANCE:
  Grow window by 1 MSS per RTT (linear growth).
  10 → 11 → 12 → 13 MSS...
  Until: loss

ON LOSS DETECTED (classic Reno):
  Set threshold = window/2
  Reset window to 1 MSS
  Restart slow start
  (drastic reduction!)

ON DUPLICATE ACKS (Fast Recovery, modern):
  Set threshold = window/2
  Set window = threshold (less drastic)
  Continue in congestion avoidance
```

This is why a new TCP connection to a distant server starts slow and takes several RTTs to "warm up" — it's in the slow start phase, probing the network capacity.

**Practical implication:** Long-lived connections (connection pooling, HTTP keep-alive) avoid slow start and perform much better than short-lived ones. This is one reason database connection pools exist.

### The Four-Way Teardown

Closing a TCP connection requires four packets (because each direction must be closed independently):

```
CLIENT                              SERVER
  │                                   │
  │──── FIN ─────────────────────────►│  "I'm done sending."
  │◄─── ACK ──────────────────────────│  "Got your FIN."
  │                                   │  (server may still be sending...)
  │◄─── FIN ──────────────────────────│  "I'm done sending too."
  │──── ACK ─────────────────────────►│  "Got your FIN."
  │                                   │
                TIME_WAIT
```

**TIME_WAIT state:** After sending the final ACK, the client waits for **2×MSL (Maximum Segment Lifetime, ~60–120 seconds)** before fully closing. This ensures the final ACK (which might be lost) can be resent if the server sends another FIN.

**The TIME_WAIT problem:** High-traffic servers can accumulate thousands of connections in TIME_WAIT state. Since TIME_WAIT holds the local port + remote IP + remote port combination, you can exhaust the ephemeral port range (49152–65535 = 16,383 ports) if you create and destroy connections faster than they exit TIME_WAIT.

**Solutions:**
- `SO_REUSEADDR` socket option
- `tcp_tw_reuse` kernel parameter
- Most importantly: **use connection pooling** — reuse connections instead of creating new ones

### TCP States (The Full State Machine)

```
States you'll see in netstat / ss output:

LISTEN       Server waiting for connections
SYN_SENT     Client sent SYN, waiting for SYN-ACK
SYN_RECEIVED Server received SYN, sent SYN-ACK
ESTABLISHED  Connection active, data flowing  ← normal operation
FIN_WAIT_1   Sent FIN, waiting for ACK
FIN_WAIT_2   Received ACK for FIN, waiting for server FIN
CLOSE_WAIT   Received FIN, waiting for app to close
LAST_ACK     Sent FIN in response, waiting for final ACK
TIME_WAIT    Waiting 2×MSL before fully closing
CLOSED       No connection

Diagnosing a server:
  Many ESTABLISHED:  healthy active connections
  Many TIME_WAIT:    high connection churn (need connection pooling)
  Many CLOSE_WAIT:   app not closing connections properly (bug/leak)
  Many SYN_RECEIVED: possible SYN flood attack
```

### TCP vs. the Application Layer

TCP delivers a **byte stream** — not messages, not packets, not requests. It guarantees order and delivery of bytes, but it says nothing about message boundaries.

```
You send: "Hello World" (11 bytes)

TCP might deliver it as:
  Chunk 1: "Hello " (6 bytes)
  Chunk 2: "World"  (5 bytes)

Or as:
  Chunk 1: "Hello World" (11 bytes)

Or as:
  Chunk 1: "Hel" (3 bytes)
  Chunk 2: "lo " (3 bytes)
  Chunk 3: "World" (5 bytes)

The application layer (HTTP, your protocol) must handle reassembly.
This is why HTTP uses Content-Length or chunked transfer encoding —
to tell the receiver when a message ends.
```

This is the **framing problem**. Every application protocol built on TCP must solve it. Common solutions:
- **Length prefix:** send the message length first, then the message
- **Delimiter:** use a special character (e.g., `\r\n\r\n` in HTTP headers)
- **Fixed size:** every message is exactly N bytes

## 3. Real-World Context

### The SYN flood attack

A SYN flood is a DoS attack exploiting the three-way handshake:
1. Attacker sends thousands of SYN packets with spoofed source IPs
2. Server allocates state for each half-open connection (waits for ACK that never comes)
3. Server's connection table fills up
4. Legitimate connections are rejected (no table space)

**SYN cookies** (RFC 4843) are the defense: the server doesn't allocate state on SYN. Instead, it encodes connection info into the SYN-ACK's sequence number. Only after the ACK arrives (proving the client is real) does it allocate state.

```bash
# Enable SYN cookies on Linux (should be default on modern kernels)
sysctl net.ipv4.tcp_syncookies=1
```

### Database connection pool and TCP

Every database connection is a TCP connection. Opening a new TCP connection costs:
- 3-way handshake: 1 RTT
- TLS handshake (if SSL): 1–2 RTTs
- Database auth: 1 RTT
- Total cold start: 3–4 RTTs × 1ms (same datacenter) = 3–4ms per new connection

With 1000 requests/second each needing a new connection: 1000 × 3ms = 3 seconds of connection overhead per second — more than the server can handle.

Connection pooling reuses established TCP connections, eliminating this overhead. Without a pool, your database server spends more time opening connections than executing queries at scale.

### HTTP persistent connections (keep-alive)

HTTP/1.1 introduced `Connection: keep-alive` — reusing the same TCP connection for multiple HTTP requests. This avoids the 3-way handshake overhead for every request.

```
WITHOUT keep-alive (HTTP/1.0):
  [TCP handshake]→[HTTP request]→[response]→[TCP close]
  [TCP handshake]→[HTTP request]→[response]→[TCP close]
  [TCP handshake]→[HTTP request]→[response]→[TCP close]
  
WITH keep-alive (HTTP/1.1):
  [TCP handshake]
  → [HTTP request]→[response]
  → [HTTP request]→[response]   (same connection, no handshake)
  → [HTTP request]→[response]
  [TCP close]
```

In production, set `Connection: keep-alive` timeouts carefully. Too long: server holds idle connections. Too short: clients reconnect too often. Nginx default: 75 seconds.

## 4. Code Examples

### Working with TCP sockets in Python

```python
import socket

# TCP SERVER
def run_server(host: str = "0.0.0.0", port: int = 8888):
    # AF_INET = IPv4, SOCK_STREAM = TCP
    server = socket.socket(socket.AF_INET, socket.SOCK_STREAM)
    server.setsockopt(socket.SOL_SOCKET, socket.SO_REUSEADDR, 1)  # avoid TIME_WAIT issues
    server.bind((host, port))
    server.listen(128)  # backlog: max queued connections
    print(f"Listening on {host}:{port}")

    while True:
        conn, addr = server.accept()  # blocks until client connects
        print(f"Connection from {addr}")
        try:
            while True:
                data = conn.recv(4096)  # receive up to 4096 bytes
                if not data:            # empty = client closed connection
                    break
                print(f"Received: {data.decode()}")
                conn.send(b"ACK: " + data)  # echo back
        finally:
            conn.close()

# TCP CLIENT
def run_client(host: str = "127.0.0.1", port: int = 8888):
    client = socket.socket(socket.AF_INET, socket.SOCK_STREAM)
    client.connect((host, port))  # 3-way handshake happens here

    client.send(b"Hello, TCP!")
    response = client.recv(4096)
    print(f"Response: {response.decode()}")
    client.close()  # 4-way teardown happens here
```

### Inspecting TCP connections on your server

```bash
# Show all TCP connections with state
ss -tnp

# Count by state
ss -tn | awk 'NR>1 {print $1}' | sort | uniq -c | sort -rn
# Output:
#  1823 ESTAB      ← established connections (healthy)
#   245 TIME-WAIT  ← closing connections waiting to expire
#     8 CLOSE-WAIT ← app not closing properly (investigate!)
#     2 LISTEN     ← servers listening

# Show connection to a specific port (e.g., Postgres on 5432)
ss -tnp | grep :5432

# Watch TCP connections in real time
watch -n 1 'ss -tn | awk "NR>1 {print \$1}" | sort | uniq -c'

# Kernel TCP tuning (check current settings)
sysctl net.ipv4.tcp_max_syn_backlog     # max queued SYN packets
sysctl net.core.somaxconn               # max listen backlog
sysctl net.ipv4.tcp_fin_timeout         # TIME_WAIT duration (default: 60s)
sysctl net.ipv4.tcp_tw_reuse            # reuse TIME_WAIT sockets (careful!)
```

### Setting timeouts correctly

```python
import socket
import httpx

# ALWAYS set timeouts on TCP connections
# Default is often "None" (wait forever) — will cause thread pool exhaustion

# Raw socket timeout
sock = socket.socket(socket.AF_INET, socket.SOCK_STREAM)
sock.settimeout(5.0)  # 5 second timeout for connect + read

# httpx (correct timeout structure)
timeout = httpx.Timeout(
    connect=2.0,   # TCP handshake + TLS
    read=10.0,     # waiting for response bytes
    write=5.0,     # sending request
    pool=1.0,      # waiting for a connection from pool
)
async with httpx.AsyncClient(timeout=timeout) as client:
    response = await client.get("https://api.example.com/data")

# requests (synchronous)
import requests
response = requests.get(
    "https://api.example.com/data",
    timeout=(2.0, 10.0)  # (connect_timeout, read_timeout)
)
```

## 6. Trade-offs

| Feature | TCP | UDP |
|---|---|---|
| Reliability | Yes (retransmits) | No |
| Ordering | Yes (sequence numbers) | No |
| Connection | Required (3-way HS) | None |
| Overhead | ~20 byte header + state | ~8 byte header |
| Latency | Higher (ACKs, handshake) | Lower |
| Throughput | Self-regulating (congestion control) | Unlimited (until network drops) |
| Use case | HTTP, DB, SSH, email | DNS, video, gaming, QUIC |

## 7. Senior Engineer Perspective

**What a junior gets wrong:**

Not setting timeouts. The default socket timeout in Python is `None` — wait forever. One slow external API, one hung database query, and the thread waits forever. With a thread pool of 20 threads, 20 slow requests = 0 capacity for new requests. The service appears down.

Every outgoing network call must have a timeout. No exceptions.

**What a senior thinks about:**

1. **Connection reuse:** Every new TCP connection costs 1 RTT (plus TLS). Use connection pools for databases, HTTP keep-alive for API calls, persistent gRPC channels for service-to-service.

2. **Window sizing on high-latency links:** Cross-region data transfer on a misconfigured TCP stack. `tcp_window_scaling` must be enabled (it is by default on Linux 2.6+), and for very high-BDP paths, check that the socket buffer sizes aren't artificially limited.

3. **TIME_WAIT and port exhaustion:** High-throughput services that create and destroy connections rapidly (short-lived HTTP calls without keep-alive) can exhaust the 16K ephemeral port range. Symptom: "connect: Cannot assign requested address". Fix: connection pooling, or `tcp_tw_reuse`.

4. **SYN backlog under traffic spikes:** The `listen()` backlog determines how many unaccepted connections can queue before the OS starts dropping SYN packets. On high-traffic servers, set `net.core.somaxconn` and `net.ipv4.tcp_max_syn_backlog` to values matching your traffic profile.

**The one mental model to carry:**

> **TCP is not free.** It trades latency (handshake, ACKs) and CPU (checksum verification, retransmit logic) for reliability. Every optimization on top of TCP — connection pools, keep-alive, multiplexing (HTTP/2) — is amortizing that cost across multiple operations.
