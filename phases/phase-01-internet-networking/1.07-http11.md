## 1. What Is It?

HTTP/1.1 (1997, revised 2014 in RFC 7230–7235) fixed HTTP/1.0's biggest problems and is still the baseline protocol that every backend engineer works with daily. Most "HTTP" you interact with is HTTP/1.1.

**What HTTP/1.1 added over HTTP/1.0:**
1. **Persistent connections** — reuse one TCP connection for multiple requests
2. **Pipelining** — send multiple requests without waiting for each response (theoretical; broken in practice)
3. **Chunked transfer encoding** — stream responses of unknown size
4. **Host header** — required, enables virtual hosting (one IP, many domains)
5. **Range requests** — download part of a file
6. **Cache-Control** — fine-grained caching directives
7. **More methods** — PUT, DELETE, OPTIONS, TRACE, CONNECT

## 2. How It Actually Works

### Persistent Connections (Keep-Alive)

HTTP/1.1 keeps the TCP connection open after a request/response cycle. Subsequent requests reuse the same connection.

```
HTTP/1.1 — loading a web page with 3 resources:

[TCP handshake + TLS]
  → GET /index.html → [response]   ← same connection
  → GET /style.css  → [response]   ← same connection
  → GET /script.js  → [response]   ← same connection
[TCP close]

Total: 1 TCP handshake + 3 content transfers.
Without keep-alive: 3 TCP handshakes + 3 content transfers.
```

**Configuration:** `Connection: keep-alive` is the default in HTTP/1.1. To close: send `Connection: close`. Servers also set a timeout after which idle connections are closed:

```nginx
# Nginx keep-alive config
keepalive_timeout 65;    # close idle connections after 65s
keepalive_requests 100;  # close after 100 requests on same connection
```

**The problem that remains:** Even with keep-alive, HTTP/1.1 is **serial** — requests on one connection must complete before the next begins. To fetch 6 resources in parallel, browsers open 6 TCP connections (up to 6 per domain, configurable). This is wasteful (6× TCP overhead) but necessary given HTTP/1.1's serial nature.

---

### The Host Header — Mandatory in HTTP/1.1

HTTP/1.0 had no `Host` header. A server received a request and could only serve one domain (because requests contained only the path, not the domain).

HTTP/1.1 made `Host` mandatory. This enables **virtual hosting** — one IP, multiple domains:

```
Request arrives at IP 54.23.167.89:

GET /users HTTP/1.1
Host: api.example.com       ← server uses this to route to the right application

vs.

GET /users HTTP/1.1
Host: api.other-company.com  ← same IP, different application
```

Every web server (Nginx, Apache, your framework) uses the `Host` header to route requests to the correct virtual host. Without it, you couldn't run multiple domains on one server. Today, tools like SNI (Server Name Indication, the TLS equivalent) extend this concept to HTTPS.

---

### Chunked Transfer Encoding

HTTP/1.0 required knowing `Content-Length` before sending. If you're generating a large response dynamically (streaming database results, generating a report), you don't know the total size upfront.

HTTP/1.1's **chunked transfer encoding** solves this: send the body in chunks, each prefixed with its size in hex, terminated with a zero-length chunk.

```
HTTP/1.1 200 OK
Transfer-Encoding: chunked
Content-Type: application/json

7\r\n               ← chunk size in hex (7 = 7 bytes)
{"id":1}\r\n         ← chunk data
6\r\n               ← next chunk size (6 bytes)
{"id":2}\r\n         ← chunk data
0\r\n               ← zero-length chunk = end of response
\r\n
```

**Use in practice:**
- Server-Sent Events (SSE)
- Streaming API responses
- Large file downloads
- Long-running operations that send progress updates

```python
# FastAPI streaming response
from fastapi import FastAPI
from fastapi.responses import StreamingResponse

app = FastAPI()

@app.get("/stream")
async def stream_data():
    async def generate():
        for i in range(100):
            yield f"data: row {i}\n\n"  # SSE format
            await asyncio.sleep(0.1)    # simulate work

    return StreamingResponse(generate(), media_type="text/event-stream")
```

### HTTP/1.1 Caching Headers

HTTP/1.1 introduced rich cache-control semantics:

```
RESPONSE HEADERS (server → cache → client):

Cache-Control: max-age=3600
  Cache this response for 3600 seconds. After that, re-validate with server.

Cache-Control: no-store
  Never cache this response (sensitive data).

Cache-Control: no-cache
  You CAN cache it, but always re-validate with server before using it.
  (Confusingly named — "no-cache" doesn't mean "don't cache")

Cache-Control: private
  Only the browser can cache this (not a shared CDN/proxy).
  Use for user-specific responses.

Cache-Control: public
  Any cache (CDN, proxy, browser) can cache this.
  Use for shared, non-user-specific responses.

Cache-Control: must-revalidate
  After max-age expires, MUST re-validate (cannot use stale copy even if offline).

ETag: "abc123def456"
  A fingerprint/hash of the response. Changes when the content changes.

Last-Modified: Tue, 15 Jun 2024 14:30:00 GMT
  When the resource was last changed.
```

**Conditional requests (cache validation):**

```
Client has a cached response with ETag: "abc123"

Client sends:
  GET /users/123 HTTP/1.1
  If-None-Match: "abc123"

Server checks: has the resource changed?
  If NO → 304 Not Modified (no body — save bandwidth)
  If YES → 200 OK with new body and new ETag
```

### HTTP/1.1 in 2024 — Still Everywhere

Despite HTTP/2 and HTTP/3 existing, HTTP/1.1 remains:
- The fallback for all servers (universal support)
- Used for internal service-to-service calls in many organizations
- The protocol most developers debug most often
- Required understanding to comprehend why HTTP/2 exists

## 3. Real-World Context

### The 6-connection-per-domain browser hack

Because HTTP/1.1 is serial per connection, browsers open multiple parallel connections. Chrome opens up to 6 TCP connections per origin. Loading a page with 30 assets: the browser maintains 6 connections and queues 5 requests per connection.

This led to the "domain sharding" optimization: serve assets from `static1.example.com`, `static2.example.com`, `static3.example.com` — tripling the browser's parallel connections to 18.

HTTP/2 made domain sharding unnecessary (multiplexing eliminated the serial limitation) and actually harmful (multiple TLS handshakes hurt). Never domain-shard for HTTP/2 traffic.

## 4. Code Examples

### Setting correct cache headers in FastAPI

```python
from fastapi import FastAPI, Response
from datetime import datetime, timedelta, timezone
import hashlib, json

app = FastAPI()

@app.get("/users/{user_id}")
async def get_user(user_id: int, response: Response):
    user = {"id": user_id, "name": "Alice"}

    # Generate ETag from content
    content = json.dumps(user, sort_keys=True)
    etag = f'"{hashlib.md5(content.encode()).hexdigest()}"'

    response.headers["ETag"] = etag
    response.headers["Cache-Control"] = "private, max-age=60"  # cache 1 min, user-specific
    response.headers["Last-Modified"] = "Tue, 15 Jun 2024 14:30:00 GMT"

    return user

@app.get("/products")
async def get_products(response: Response):
    products = [{"id": 1, "name": "Widget"}]

    response.headers["Cache-Control"] = "public, max-age=3600"  # CDN can cache 1 hour
    response.headers["Vary"] = "Accept-Encoding"  # different cache for gzip vs plain

    return products

@app.get("/dashboard")
async def get_dashboard(response: Response):
    # Never cache sensitive dashboards
    response.headers["Cache-Control"] = "no-store"
    response.headers["Pragma"] = "no-cache"  # HTTP/1.0 backward compat

    return {"data": "sensitive"}
```

### Connection keep-alive configuration (Nginx)

```nginx
http {
    # How long to keep idle connections open
    keepalive_timeout 65s;

    # Max requests per connection (prevent one client monopolizing)
    keepalive_requests 1000;

    upstream backend {
        server 10.0.1.1:8000;
        server 10.0.1.2:8000;

        # Keep persistent connections to upstream (crucial for performance)
        keepalive 32;  # maintain 32 idle upstream connections
    }

    server {
        location / {
            proxy_pass http://backend;
            proxy_http_version 1.1;
            proxy_set_header Connection "";  # enable keep-alive to upstream
        }
    }
}
```

## 5. Senior Engineer Perspective

**What a junior gets wrong:**

`Cache-Control: no-cache` means "always validate with server before using." It does NOT mean "don't cache." Countless production bugs come from misunderstanding this. If you want to truly prevent caching: `Cache-Control: no-store`.

Also: forgetting the `Vary` header. If your API returns different content based on `Accept-Language` or `Accept-Encoding`, you MUST include `Vary: Accept-Language` or `Vary: Accept-Encoding`. Otherwise a CDN caches the French version and serves it to English users.

**What a senior thinks about:**

Cache headers are a contract between your server and every cache in the chain (browser, CDN, proxy). Getting them wrong leads to stale data served to users (too aggressive caching) or slow, expensive responses (no caching when you could). Design cache strategy per endpoint: user-specific data → `Cache-Control: private`. Shared data → `Cache-Control: public, max-age=N`. Sensitive data → `Cache-Control: no-store`.
