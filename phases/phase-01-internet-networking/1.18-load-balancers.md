## 1. What Is a Load Balancer?

A load balancer distributes incoming traffic across multiple backend servers, ensuring no single server becomes a bottleneck, and providing high availability (if one server fails, the others continue serving).

**Two benefits:**
1. **Scalability:** Distribute load across N servers → N× capacity
2. **Availability:** Remove unhealthy servers from rotation → zero-downtime when a server dies

## 2. L4 vs L7 Load Balancers

The most important distinction — covered in [0.4 OSI model](../phase-00-mental-models/0.04-osi-model.md):

### L4 (Transport Layer)

```
SEES: TCP/UDP headers only (source/dest IP + port)
CANNOT SEE: HTTP headers, URL path, cookies, body

ROUTING BASED ON: IP address, port, TCP session
CANNOT ROUTE ON: URL path, hostname, cookie value

OPERATION: Routes entire TCP connections (or UDP flows) to a backend
           Client TCP connection → backend TCP connection (NAT or proxy)

EXAMPLES: AWS NLB, HAProxy (TCP mode), hardware load balancers

USE WHEN:
  ✓ Non-HTTP protocols (database connections, gRPC, custom TCP/UDP)
  ✓ Maximum throughput, minimum latency (no HTTP parsing overhead)
  ✓ TLS passthrough (don't decrypt, just route the TCP connection)
  ✓ Static IP needed for backend (NLB provides stable IPs)
```

### L7 (Application Layer)

```
SEES: Full HTTP request (headers, URL, body, cookies)
CAN SEE: Everything in the HTTP request

ROUTING BASED ON: URL path, Host header, cookie, any HTTP header

OPERATION: Terminates client connection, inspects HTTP, opens new connection to backend
           HTTP/2 multiplexing means many HTTP requests → fewer backend connections

EXAMPLES: AWS ALB, Nginx, Traefik, Envoy, GCP HTTPS Load Balancer

USE WHEN:
  ✓ HTTP/HTTPS traffic (almost always)
  ✓ Path-based routing (/api/* → service A, /static/* → service B)
  ✓ Host-based routing (api.example.com → API, app.example.com → frontend)
  ✓ SSL termination (handle HTTPS at LB, HTTP internally)
  ✓ Sticky sessions (route user to same backend based on cookie)
  ✓ A/B testing (route % of traffic to new version)
  ✓ Request/response manipulation (add headers, rewrite URLs)
```

## 3. Load Balancing Algorithms

```
ROUND ROBIN (default):
  Request 1 → Server A
  Request 2 → Server B
  Request 3 → Server C
  Request 4 → Server A (cycle repeats)
  
  Best when: servers are identical, requests take similar time.
  Problem: a slow request on Server A doesn't affect its turn in rotation.

WEIGHTED ROUND ROBIN:
  Server A weight=3, Server B weight=1 → A gets 3× the traffic
  Use when: servers have different capacity (different instance sizes)

LEAST CONNECTIONS:
  New request → server with fewest active connections
  Best when: requests have varying duration (some fast, some slow)
  Prevents "slow server gets more requests" problem of round robin

LEAST RESPONSE TIME:
  New request → server with lowest average response time
  Best for: heterogeneous servers, variable request complexity

IP HASH (sticky by client IP):
  Hash client IP → always routes to same server
  Use for: stateful applications that store session in-memory (not recommended;
           use Redis for sessions instead)
  Problem: if server dies, all its "hashed" clients go to different server

RANDOM:
  Random server selection
  Statistically approaches round robin at scale, simpler implementation

CONSISTENT HASH:
  Hash request key (user ID, URL) → stable assignment even as servers added/removed
  Only ~1/N sessions remapped when adding server N+1
  Use for: cache sharding, stateful routing
```

## 4. Health Checks

A load balancer continuously checks backend health and removes unhealthy instances from rotation.

```
ACTIVE HEALTH CHECK (load balancer pings backends):
  Every 10 seconds, LB sends GET /health to each backend.
  
  Healthy threshold: 2 consecutive successes → mark healthy
  Unhealthy threshold: 3 consecutive failures → mark unhealthy, stop routing
  
  Response requirements:
    Status: 200
    Response time: < 2s
    Optional: body contains {"status": "healthy"}

PASSIVE HEALTH CHECK (monitor real traffic):
  If backend returns 5xx or times out on real requests:
    After 5 failures → mark unhealthy, stop routing
  Advantages: no extra health check traffic
  Disadvantages: real users see errors before backend is removed

YOUR HEALTH ENDPOINT MUST CHECK DEPENDENCIES:
  ✗ Shallow (useless): return 200 always
  ✓ Deep (useful): check DB connection, Redis connection, disk space
  
  If DB is down: return 503 → LB removes this backend from rotation
  Other backends still have healthy DB connections → traffic rerouted
  
  But: don't make health checks too deep — they should be fast (<500ms)
  and shouldn't overload the DB on every check.
```

## 5. AWS Load Balancers

```
ALB (Application Load Balancer):
  L7. HTTP/HTTPS/WebSocket. Path and host routing. SSL termination.
  Target types: EC2, Lambda, IP addresses, ECS tasks
  Use for: almost all web applications

NLB (Network Load Balancer):
  L4. TCP/UDP/TLS. Static IP addresses. Ultra-low latency.
  Best for: non-HTTP protocols, high throughput, static IPs
  Use for: databases, gaming, gRPC at very high scale

CLB (Classic Load Balancer):
  Old. Being deprecated. Don't use for new workloads.

GATEWAY LOAD BALANCER:
  For deploying third-party virtual appliances (firewalls, IDS)
  Niche use case; most teams never need it.
```

## 6. Code Examples

### Health check endpoint

```python
from fastapi import FastAPI
from fastapi.responses import JSONResponse
import asyncpg, redis.asyncio as aioredis, time

app = FastAPI()
start_time = time.time()

@app.get("/health")
async def health_check():
    """
    Comprehensive health check for load balancer.
    Returns 200 if healthy, 503 if degraded.
    """
    checks = {}
    healthy = True

    # Check database
    try:
        conn = await asyncpg.connect(dsn=DATABASE_URL, timeout=2)
        await conn.fetchval("SELECT 1")
        await conn.close()
        checks["database"] = "ok"
    except Exception as e:
        checks["database"] = f"error: {str(e)}"
        healthy = False

    # Check Redis
    try:
        r = aioredis.from_url(REDIS_URL, socket_timeout=2)
        await r.ping()
        await r.aclose()
        checks["redis"] = "ok"
    except Exception as e:
        checks["redis"] = f"error: {str(e)}"
        healthy = False  # or False depending on Redis being required

    status = "healthy" if healthy else "unhealthy"
    http_status = 200 if healthy else 503

    return JSONResponse(
        status_code=http_status,
        content={
            "status": status,
            "uptime_seconds": int(time.time() - start_time),
            "checks": checks,
        }
    )

# Kubernetes readiness probe (stop routing when not ready)
@app.get("/ready")
async def readiness():
    if not app.state.initialized:
        return JSONResponse(status_code=503, content={"ready": False})
    return {"ready": True}

# Kubernetes liveness probe (restart if dead)
@app.get("/live")
async def liveness():
    return {"alive": True}  # always 200 unless process is completely broken
```

### Nginx load balancing config

```nginx
upstream api_backends {
    # Load balancing algorithm
    least_conn;   # or: round-robin (default), ip_hash, random

    server 10.0.1.45:8000 weight=2;  # gets 2x traffic (larger instance)
    server 10.0.1.46:8000 weight=1;
    server 10.0.1.47:8000 weight=1 backup;  # only used if others fail

    # Active health checks (Nginx Plus feature; use passive for open-source)
    keepalive 32;  # persistent connections to backends
}

server {
    listen 443 ssl http2;

    location / {
        proxy_pass http://api_backends;
        proxy_http_version 1.1;
        proxy_set_header Connection "";

        # Passive health: retry on errors
        proxy_next_upstream error timeout http_500 http_502 http_503;
        proxy_next_upstream_tries 2;  # try up to 2 backends before failing
    }
}
```

## 7. Senior Engineer Perspective

**What a junior gets wrong:**

Using IP hash or cookie-based sticky sessions because "the app stores session in memory." The real fix is to move sessions to Redis — then any backend can serve any user, and you get true horizontal scaling. Sticky sessions are a workaround for a stateful application, not a design goal.

**What a senior thinks about:**

Connection draining. When you deploy a new version and want to remove old backends from rotation, you don't just immediately stop routing traffic — in-flight requests would fail. Connection draining lets the LB stop sending new requests to a backend while existing connections complete (typically 30–60s). Every deployment pipeline should use connection draining.

The health check depth decision: shallow checks (always 200) mean you might route traffic to a backend that can't reach the DB. Deep checks (check DB, Redis, disk) are more accurate but add load and can cause cascading failures (DB is slow → health checks fail → all backends removed → 0% capacity). The right balance: check critical dependencies with timeouts, but degrade gracefully on non-critical ones.
