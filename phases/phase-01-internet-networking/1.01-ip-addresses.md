## 1. What Is It? (Theory)

Every device on the internet has an address. IP (Internet Protocol) addresses are those addresses — unique identifiers that allow packets to be routed from source to destination across any network.

Understanding IP addresses is not optional for backend engineers. You configure them in every cloud resource, security group, firewall rule, load balancer, and DNS record you create. Getting them wrong means your service is either unreachable or exposed to the wrong network.

**The real-world analogy:**

An IP address is like a postal address. Your house number and street identify you within a city (local network). Your city and country identify you globally (internet). The postal system (routing) uses the address hierarchy to deliver mail without needing to know every individual address — it just knows the country, then the city, then the street.

IPv4 is running out of addresses (like running out of house numbers in a finite city). IPv6 is the new, vastly larger addressing scheme.

## 2. How It Actually Works (Deep Mechanics)

### IPv4

IPv4 addresses are **32-bit integers**, written in **dotted-decimal notation** — four groups of 8 bits (octets), each 0–255, separated by dots.

```
IPv4 ADDRESS:  192.168.1.100

In binary:
  192     .  168     .  1       .  100
  11000000   10101000   00000001   01100100

All 32 bits: 11000000101010000000000101100100
As hex:      0xC0A80164

Range: 0.0.0.0 (0x00000000) to 255.255.255.255 (0xFFFFFFFF)
Total addresses: 2^32 = 4,294,967,296 (~4.3 billion)
```

**4.3 billion sounds like a lot. It isn't.** There are 8 billion people, each with multiple devices. IPv4 addresses were exhausted in 2011. The internet survives today through two techniques: NAT (Network Address Translation) and the slow migration to IPv6.

---

### IPv4 Address Classes (Historical, but you'll see these)

Originally, IPv4 was divided into classes based on the first octet:

```
CLASS A:  1.0.0.0   – 126.0.0.0    (first bit: 0)
          /8 network prefix
          16 million hosts per network
          Used by: large corporations, ISPs

CLASS B:  128.0.0.0 – 191.255.0.0  (first bits: 10)
          /16 network prefix
          65,536 hosts per network

CLASS C:  192.0.0.0 – 223.255.255.0 (first bits: 110)
          /24 network prefix
          254 hosts per network
          Most common for small networks

CLASS D:  224.0.0.0 – 239.255.255.255 (multicast)
CLASS E:  240.0.0.0 – 255.255.255.255 (reserved/experimental)
```

Classes are largely obsolete (replaced by CIDR), but you'll see references to "class C network" meaning /24 in older documentation.

---

### CIDR (Classless Inter-Domain Routing)

CIDR replaced the class system. It uses a **prefix length** (the `/` notation) to define which bits are the network portion and which are the host portion.

```
CIDR NOTATION: address / prefix_length

192.168.1.0/24

Network:  192.168.1  (first 24 bits — fixed, identifies the network)
Host:              .0–.255 (last 8 bits — varies per device)

Total addresses: 2^(32-24) = 2^8 = 256
Usable hosts: 254 (subtract .0 network address and .255 broadcast address)
```

**Reading CIDR notation:**

```
CIDR      Network Bits   Host Bits   Addresses   Typical Use
──────────────────────────────────────────────────────────────
/8        8              24          16,777,216  Large ISP backbone
/16       16             16          65,536      Corporate network
/24       24             8           256         Small office, VPC subnet
/25       25             7           128         Half of /24
/26       26             6           64          Quarter of /24
/27       27             5           32          Small subnet
/28       28             4           16          Tiny subnet
/30       30             2           4           Point-to-point link
/32       32             0           1           Single host, firewall rule
```

**Subnet mask:** The old way to express CIDR. /24 = 255.255.255.0. /16 = 255.255.0.0. Still widely used in configuration files.

```
/24 = 255.255.255.0
  11111111.11111111.11111111.00000000
  (24 ones, 8 zeros)

/22 = 255.255.252.0
  11111111.11111111.11111100.00000000
  (22 ones, 10 zeros → 2^10 = 1024 addresses)
```

**Calculating network and broadcast addresses:**

```
Network: 10.0.4.0/22

Prefix length: 22
Network address: 10.0.4.0   (all host bits = 0)
Broadcast:       10.0.7.255  (all host bits = 1)
  → 10.0.4.0 in binary: 00001010 00000000 00000100 00000000
  → Set last 10 bits to all 1: 00001010 00000000 00000111 11111111
  → = 10.0.7.255
Usable hosts: 10.0.4.1 – 10.0.7.254 (1022 hosts)
```

---

### Private vs. Public IP Addresses

Not all IPv4 addresses are routable on the public internet. Three ranges are **reserved for private networks** (RFC 1918):

```
PRIVATE RANGES (not routable on public internet):
  10.0.0.0/8       → 10.0.0.0   – 10.255.255.255   (16M addresses)
  172.16.0.0/12    → 172.16.0.0 – 172.31.255.255   (1M addresses)
  192.168.0.0/16   → 192.168.0.0– 192.168.255.255  (65K addresses)

SPECIAL PURPOSE:
  127.0.0.0/8      → Loopback (127.0.0.1 = "this machine")
  169.254.0.0/16   → Link-local (APIPA — assigned when DHCP fails)
  0.0.0.0          → "All interfaces" (used in server bind addresses)
  255.255.255.255  → Broadcast to all hosts on local network
```

**In AWS, every EC2 instance gets:**
- A private IP from your VPC range (e.g., `10.0.1.45`) — for internal communication
- Optionally a public IP (e.g., `54.23.167.89`) — for internet access

Traffic between EC2 instances in the same VPC uses private IPs (free, fast). Traffic to the internet goes through the public IP via NAT Gateway (costs money).

---

### Special Addresses You'll Configure Constantly

```
0.0.0.0         Means "all network interfaces" when binding a server.
                `uvicorn --host 0.0.0.0` → listen on all interfaces
                vs. `--host 127.0.0.1` → listen only on loopback (local only)

127.0.0.1       Loopback — always refers to "this machine"
                Your dev database: localhost = 127.0.0.1

::1             IPv6 loopback

0.0.0.0/0       In routing tables: "default route" — send everything here
                In security groups: "allow from any IP"
```

### IPv6

IPv6 addresses are **128-bit integers**, written as **eight groups of four hex digits** separated by colons:

```
2001:0db8:85a3:0000:0000:8a2e:0370:7334

Shorthand rules:
  1. Leading zeros in each group can be omitted:
     2001:db8:85a3:0:0:8a2e:370:7334

  2. One sequence of consecutive all-zero groups can be replaced with ::
     2001:db8:85a3::8a2e:370:7334

  3. :: can only appear once (otherwise ambiguous)

Special addresses:
  ::1             → loopback (equivalent to 127.0.0.1)
  ::              → unspecified (equivalent to 0.0.0.0)
  fe80::/10       → link-local (equivalent to 169.254.x.x)
  fc00::/7        → unique local (equivalent to 10.x.x.x / 192.168.x.x)
  2001:db8::/32   → documentation prefix (use in examples, never real traffic)
```

**Why IPv6 solves the address problem:**
- 128 bits = 2^128 = 340,282,366,920,938,463,463,374,607,431,768,211,456 addresses
- Every grain of sand on Earth could have billions of addresses
- Every device gets a globally unique address — NAT becomes unnecessary

**IPv6 adoption:** As of 2024, ~45% of global internet traffic is IPv6. Google, Cloudflare, all major cloud providers are fully IPv6-capable. New services should support both IPv4 and IPv6 (dual-stack).

---

### How Routers Use IP Addresses

Every router maintains a **routing table** — a list of network prefixes and where to send traffic for each:

```
ip route show (Linux):

default via 10.0.0.1 dev eth0    ← everything not matched → gateway
10.0.0.0/24 dev eth0 proto kernel ← directly connected network
172.17.0.0/16 dev docker0         ← Docker bridge network
```

When a packet arrives, the router finds the **most specific matching prefix** (longest prefix match):

```
Destination: 10.0.1.45

Routes in table:
  0.0.0.0/0    → default gateway      (matches: 0 bits)
  10.0.0.0/8   → internal router      (matches: 8 bits)
  10.0.1.0/24  → subnet A gateway     (matches: 24 bits) ← WINNER (most specific)

Router sends to subnet A gateway.
```

## 3. Real-World Context

### AWS VPC design

A typical AWS VPC uses 10.0.0.0/16 (65,536 addresses), divided into subnets:

```
VPC: 10.0.0.0/16

Public subnets (internet-accessible, load balancers, NAT gateways):
  10.0.1.0/24  us-east-1a  (254 hosts)
  10.0.2.0/24  us-east-1b  (254 hosts)
  10.0.3.0/24  us-east-1c  (254 hosts)

Private subnets (application servers, not directly internet-accessible):
  10.0.11.0/24 us-east-1a  (254 hosts)
  10.0.12.0/24 us-east-1b  (254 hosts)
  10.0.13.0/24 us-east-1c  (254 hosts)

Database subnets (databases, most restricted):
  10.0.21.0/24 us-east-1a  (254 hosts)
  10.0.22.0/24 us-east-1b  (254 hosts)
```

Security group rules reference CIDR blocks:
- Allow 443 from `0.0.0.0/0` → public HTTPS access
- Allow 5432 from `10.0.11.0/24` → only app servers can reach Postgres
- Allow 22 from `10.0.0.0/8` → SSH only from internal network

### Kubernetes pod networking

Each Kubernetes pod gets its own IP address from the cluster's pod CIDR (often `10.244.0.0/16`). Services get IPs from the service CIDR (often `10.96.0.0/12`). Understanding subnets and CIDR is required to debug pod-to-pod or pod-to-service connectivity issues.

## 4. Code Examples

### IP address operations in Python

```python
import ipaddress

# Parse and inspect IPv4 addresses
ip = ipaddress.ip_address("192.168.1.100")
print(ip.version)          # 4
print(ip.is_private)       # True
print(ip.is_loopback)      # False
print(int(ip))             # 3232235876 (as integer)

# Work with networks (subnets)
network = ipaddress.ip_network("10.0.1.0/24")
print(network.network_address)   # 10.0.1.0
print(network.broadcast_address) # 10.0.1.255
print(network.num_addresses)     # 256
print(network.prefixlen)         # 24

# Check if IP is in a subnet
ip = ipaddress.ip_address("10.0.1.45")
print(ip in network)  # True

ip2 = ipaddress.ip_address("10.0.2.1")
print(ip2 in network)  # False

# Iterate over hosts
for host in list(network.hosts())[:5]:
    print(host)
# 10.0.1.1
# 10.0.1.2
# 10.0.1.3
# 10.0.1.4
# 10.0.1.5

# Subnetting: split /24 into four /26s
network = ipaddress.ip_network("10.0.1.0/24")
for subnet in network.subnets(prefixlen_diff=2):  # /24 + 2 = /26
    print(subnet)
# 10.0.1.0/26   (64 addresses)
# 10.0.1.64/26  (64 addresses)
# 10.0.1.128/26 (64 addresses)
# 10.0.1.192/26 (64 addresses)

# IPv6
ipv6 = ipaddress.ip_address("2001:db8::1")
print(ipv6.version)          # 6
print(ipv6.is_global)        # False (documentation prefix)
print(ipv6.exploded)         # 2001:0db8:0000:0000:0000:0000:0000:0001
```

### Security group validation (FastAPI example)

```python
import ipaddress
from fastapi import Request, HTTPException

ALLOWED_INTERNAL_CIDR = ipaddress.ip_network("10.0.0.0/8")

def require_internal_ip(request: Request):
    """Only allow requests from internal network."""
    client_ip_str = request.client.host
    try:
        client_ip = ipaddress.ip_address(client_ip_str)
    except ValueError:
        raise HTTPException(status_code=400, detail="Invalid IP")

    if client_ip not in ALLOWED_INTERNAL_CIDR:
        raise HTTPException(
            status_code=403,
            detail=f"Access denied from {client_ip_str}"
        )
```

**Language-agnostic pattern — understanding the components of a network config:**

```
DECODING A NETWORK CONFIGURATION:

Server IP:      10.0.1.45
Subnet mask:    255.255.255.0  (= /24)
Network:        10.0.1.0/24   (all devices sharing this subnet)
Broadcast:      10.0.1.255    (send to all devices in subnet)
Default gateway:10.0.1.1      (packets outside subnet go here)
DNS:            8.8.8.8       (resolve hostnames)

What you can reach directly: 10.0.1.1 – 10.0.1.254 (same /24)
What must go through gateway: everything else
```

## 5. Tools

```bash
# Show your IP addresses and network interfaces
ip addr show           # Linux
ifconfig               # macOS/older Linux

# Show routing table
```

## 6. Trade-offs

| | IPv4 | IPv6 |
|---|---|---|
| Address space | 4.3 billion (exhausted) | 340 undecillion |
| Format | Easy to read/remember | 128-bit hex (harder) |
| NAT required? | Yes (at scale) | No |
| Adoption | Universal | ~45% of internet traffic |
| Header size | 20 bytes | 40 bytes (but simpler) |
| Dual-stack cost | — | Extra config complexity |

## 7. Senior Engineer Perspective

**What a junior gets wrong:**

Treating IP addresses as opaque strings. When you see `10.0.1.45/24` in a config, you need to immediately understand: this is a private address, it's in the 10.0.1.0/24 subnet, it can reach 10.0.1.1–10.0.1.254 directly, and everything else goes through the gateway.

Also: confusing "0.0.0.0" as a bind address (listen on all interfaces — correct) with "0.0.0.0/0" as a firewall rule (allow from anywhere — VERY different intent, verify before using).

**What a senior thinks about in infrastructure:**

Every security group rule, VPC subnet, and routing decision involves CIDR. The most common production security mistake is `0.0.0.0/0` on a port that should be internal-only. SSH (port 22) open to the world: every bot on the internet will hammer your server within minutes of launch. Database port (5432) open to the world: data breach waiting to happen.

The correct approach: databases only accept connections from the application subnet. Application servers only accept connections from the load balancer. Load balancers accept 80/443 from the world. Everything else is denied by default.
