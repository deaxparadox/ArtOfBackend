## 1. What Is a VPN?

A VPN (Virtual Private Network) creates an encrypted tunnel between two network endpoints, allowing traffic to flow securely over an untrusted network (the internet) as if it were on a private network.

**Two common uses that backend engineers encounter:**

1. **Remote access VPN:** Developers connect to production infrastructure from home. Traffic goes through the VPN so it appears to originate from within the company network.

2. **Site-to-site VPN:** Two networks (your AWS VPC + your office network) are connected as if they share a single private network. No public internet exposure for cross-network traffic.

## 2. How VPN Tunneling Works

```
WITHOUT VPN (traffic visible):
  Your laptop ──── internet (unencrypted) ──── Database server

WITH VPN:
  Your laptop → [encrypt + encapsulate] → VPN server → [decrypt] → Database server
                └───── VPN tunnel ────────┘
                       (looks like one packet to the internet,
                        nobody can read the inner packet)

ENCAPSULATION:
  Original packet: [IP header: your IP → DB IP][TCP][data]
  After VPN wrapping:
    [IP header: your IP → VPN server][UDP/ESP][encrypted original packet]
  
  The internet only sees: traffic between you and the VPN server.
  The original destination and content are hidden inside.
```

## 3. VPN Protocols

```
IPSEC (Internet Protocol Security):
  Industry standard. Works at L3 (IP layer).
  Site-to-site VPNs. AWS VPN uses IPSec.
  Strong but complex to configure.
  
  Two modes:
  - Transport mode: encrypts only payload (IP header exposed)
  - Tunnel mode: encrypts entire original packet + new IP header added

WIREGUARD:
  Modern, minimal, fast. Uses state-of-the-art cryptography.
  ~4000 lines of code (vs ~400,000 for OpenVPN).
  Simple config. Used by Tailscale, many corporate VPNs.
  Becoming the standard for new deployments.

OPENVPN:
  Open source, widely deployed, TLS-based.
  Runs over UDP (port 1194) or TCP (port 443).
  Mature ecosystem, many clients available.

SSL/TLS VPN:
  VPN over HTTPS (port 443). Difficult to block (looks like normal HTTPS).
  Cisco AnyConnect, OpenVPN Access Server.
  Used for corporate remote access VPN.
```

## 4. VPN in AWS

### AWS Client VPN (Remote Access)

```
Engineers connecting to AWS VPC resources:
  Engineer laptop → AWS Client VPN endpoint → Private subnet resources
  
  Engineer gets an IP in the VPN CIDR (e.g., 10.1.0.0/16)
  Can access: RDS on 10.0.21.5, Redis on 10.0.22.10, etc.
  Without VPN: those resources have no public IP — unreachable from internet
```

### AWS Site-to-Site VPN

```
Office network ←→ AWS VPC via encrypted IPSec tunnel
  
  Office: 192.168.0.0/16
  VPC: 10.0.0.0/16
  
  Route in office network: 10.0.0.0/16 → VPN gateway
  Route in VPC: 192.168.0.0/16 → Virtual Private Gateway
  
  Traffic between office and VPC flows encrypted over internet.
  Appears as if office and VPC are on the same private network.
  
  Cost: ~$36/month per VPN connection + data transfer
  Bandwidth: up to 1.25 Gbps per tunnel
  
  Alternative (higher bandwidth): AWS Direct Connect (physical fiber)
```

## 5. Tailscale — Modern VPN for Engineers

Tailscale is WireGuard-based VPN that's dramatically easier to operate:

```
Traditional VPN problems:
  - Requires central VPN server (single point of failure)
  - Complex certificate management
  - All traffic routed through central server (bottleneck)

Tailscale solution (mesh VPN):
  - Devices connect directly to each other (peer-to-peer) when possible
  - Falls back to relay (DERP servers) only when P2P impossible (both behind NAT)
  - No central server to manage
  - Auth via Google/GitHub/Okta SSO
  
Use cases:
  - Engineers SSH into production servers without exposing port 22
  - Access internal tools (Grafana, Kibana) without public exposure
  - Connect developer machines to staging databases
  
Setup (minutes, not days):
  curl -fsSL https://tailscale.com/install.sh | sh
  tailscale up
  # Done. Access any other tailscale device by its tailscale IP (100.x.x.x)
```

## 6. SSH Tunneling (Poor Man's VPN)

Before VPNs were easy to set up, SSH tunneling was the standard way to securely access remote services:

```bash
# Local port forwarding: access remote DB on your local machine
ssh -L 5432:database-host:5432 bastion.example.com -N

# Now connect your DB client to localhost:5432
# Traffic: localhost:5432 → SSH tunnel → bastion → database-host:5432

# Dynamic port forwarding (SOCKS proxy — routes all traffic)
ssh -D 8080 bastion.example.com -N
# Configure browser to use SOCKS proxy localhost:8080
# All browser traffic goes through bastion

# Remote port forwarding: expose local service to remote server
ssh -R 8080:localhost:3000 server.example.com -N
# Now: server.example.com:8080 → your local machine:3000
# Use case: test webhooks from Stripe against your local server
```

SSH tunneling is still useful for quick one-off access to a database or service. For regular access by a team, use a proper VPN or Tailscale.

---

## 7. Senior Engineer Perspective

**What a junior gets wrong:**

Opening database ports (5432, 3306, 6379) to the public internet so developers can connect from their laptops. Bots scan the entire internet for open ports constantly. An exposed PostgreSQL port gets brute-force attacks within minutes. The correct approach: databases in private subnets, access via VPN or SSH tunnel only.

**What a senior uses:**

Tailscale or AWS Client VPN for developer access to internal resources. Zero public ports on databases, caches, or internal services. All developer access goes through the VPN. This eliminates an entire class of attack vectors without significant operational overhead.
