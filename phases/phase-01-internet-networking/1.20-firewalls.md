## 1. What Is a Firewall?

A firewall filters network traffic based on rules. It decides which packets are allowed through and which are dropped.

Every production backend runs behind multiple firewall layers. Understanding them means you can diagnose "why can't my service reach the database?" and "why is my port blocked?" without spending an hour with the ops team.

---

## 2. Stateless vs. Stateful Firewalls

### Stateless

Examines each packet independently. No memory of previous packets.

```
Rule: ALLOW TCP port 443 from any source
Rule: DENY all others

Packet 1: SYN to port 443 → ALLOWED
Packet 2: SYN-ACK from server → evaluates as standalone packet
  Does it match "TCP port 443 from any source"? 
  SYN-ACK is FROM the server, not TO port 443.
  In a naive stateless firewall: might be DENIED.
  
Problem: must explicitly allow responses, which is complex to manage.
Use: older hardware firewalls, simple ACLs on routers.
```

### Stateful (most common today)

Tracks TCP connections. Knows which connections are established.

```
Rule: ALLOW TCP port 443 inbound
(No explicit rule for response traffic needed)

Client connects to port 443:
  SYN → firewall creates state table entry: (client:port ↔ server:443, state=SYN_SENT)
  SYN-ACK → firewall sees it belongs to existing connection → ALLOWED
  Established → all packets matching the connection state → ALLOWED

Unsolicited packet from internet to your server port 52341:
  Not in state table → no established connection → DENIED
```

Stateful firewalls protect against:
- Port scans (unsolicited SYN packets to random ports)
- Connection hijacking (forged packets not matching connection state)
- Many DoS attack variations

## 3. AWS Security Groups (Stateful Firewall)

Security groups are AWS's per-resource stateful firewall. Every EC2 instance, RDS database, Lambda function, and load balancer has associated security groups.

```
INBOUND RULES (who can reach this resource):
  Type       Protocol   Port   Source
  HTTPS      TCP        443    0.0.0.0/0      ← internet can reach HTTPS
  HTTP       TCP        80     0.0.0.0/0      ← internet can reach HTTP
  PostgreSQL TCP        5432   sg-0abc123     ← only instances in this security group
  SSH        TCP        22     10.0.0.0/8     ← only from internal network

OUTBOUND RULES (what this resource can reach):
  All traffic  All  All  0.0.0.0/0  ← allow all outbound (typical default)

KEY PROPERTIES:
  - Stateful: response traffic automatically allowed (no return rules needed)
  - Default deny: anything not explicitly allowed is blocked
  - Security group references: "allow from sg-0abc123" means allow from any
    resource associated with that security group (regardless of IP)
  - No explicit DENY rules (unlike NACLs) — only ALLOW rules
```

## 4. AWS Network ACLs (Stateless Firewall)

NACLs apply to an entire subnet. They are stateless — you must explicitly allow response traffic.

```
NACLs are the subnet-level line of defense.
Security Groups are the instance-level line of defense.

NACL INBOUND:
  Rule 100: ALLOW TCP 443 from 0.0.0.0/0
  Rule 200: ALLOW TCP 1024-65535 from 0.0.0.0/0  ← responses to outbound
  Rule *:   DENY all

NACL OUTBOUND:
  Rule 100: ALLOW TCP 443 to 0.0.0.0/0
  Rule 200: ALLOW TCP 1024-65535 to 0.0.0.0/0   ← responses to inbound
  Rule *:   DENY all

Rules evaluated in order (lowest rule number first).
First match wins.

Use NACLs for: subnet-level blocking (block a bad IP range for the whole subnet).
Use Security Groups for: instance-level control (most fine-grained).
```

## 5. WAF (Web Application Firewall)

A WAF operates at Layer 7 (HTTP), unlike traditional firewalls (L3/L4).

```
L3/L4 Firewall:
  Can block: IP addresses, port numbers, protocols
  Cannot see: HTTP URL path, request body, SQL injection patterns

WAF:
  Can inspect: full HTTP request (URL, headers, body, query params)
  Can block:
    - SQL injection patterns ("' OR 1=1 --")
    - XSS payloads ("<script>alert(1)</script>")
    - Path traversal ("../../etc/passwd")
    - Known exploit signatures (Log4Shell, etc.)
    - Rate limits per IP, per user
    - Bot detection (user-agent, request patterns)

Providers:
  AWS WAF: integrated with CloudFront, ALB, API Gateway
  Cloudflare WAF: managed rules + custom rules
  ModSecurity: open-source, runs in Nginx/Apache
```

## 6. The Defense-in-Depth Architecture

```
Internet
  │
  ▼
CDN + DDoS Protection (Cloudflare, AWS Shield)
  │  Absorbs volumetric attacks before they reach your infrastructure
  │
  ▼
WAF (L7 Firewall)
  │  Blocks malicious HTTP (SQLi, XSS, known exploits)
  │
  ▼
Load Balancer (ALB/NLB)
  │  Behind Security Group: only 443/80 allowed from internet
  │
  ▼
Security Group (App Servers)
  │  Only allow traffic FROM load balancer security group
  │  Allow 22/SSH only from bastion host security group
  │
  ▼
Security Group (Database)
  │  Only allow 5432 FROM app server security group
  │  No inbound from internet at all
  │
  ▼
Database
```

Each layer provides independent protection. An attacker who bypasses one layer faces the next.

## 7. Code Examples

### Checking security group rules via AWS CLI

```bash
# Show all inbound rules for a security group
aws ec2 describe-security-groups \
  --group-ids sg-0abc123def456 \
  --query 'SecurityGroups[0].IpPermissions' \
  --output table

# Find all security groups with port 22 open to the internet (0.0.0.0/0)
aws ec2 describe-security-groups \
  --query 'SecurityGroups[?IpPermissions[?ToPort==`22` && IpRanges[?CidrIp==`0.0.0.0/0`]]].{Name:GroupName,ID:GroupId}' \
  --output table
# Any results here are a security risk — SSH open to internet

# Audit: find instances with public IPs (potential attack surface)
aws ec2 describe-instances \
  --query 'Reservations[*].Instances[*].[InstanceId,PublicIpAddress,Tags[?Key==`Name`].Value|[0]]' \
  --output table
```

## 8. Senior Engineer Perspective

**What a junior gets wrong:**

Opening SSH (port 22) to `0.0.0.0/0` in a security group. Within minutes of an instance getting a public IP, SSH bots from around the world attempt to log in. The correct approach: SSH only from a bastion host (which itself has SSH limited to your office IP range), or use AWS Systems Manager Session Manager (no port 22 needed at all).

**What a senior designs:**

No instance in a private subnet should ever need a public IP. Public subnets host only load balancers, NAT gateways, and bastion hosts. Databases and application servers live in private subnets. The only inbound path from the internet is through the load balancer. The only inbound SSH path is through the bastion. Everything else is blocked at the subnet and security group level.
