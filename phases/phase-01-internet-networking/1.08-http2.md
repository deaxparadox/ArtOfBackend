## 1. What Is It?

HTTP/2 (2015, RFC 7540) is a major revision of HTTP that fixes HTTP/1.1's performance problems while keeping the same semantics. Your existing HTTP verbs, status codes, headers, and URLs all work identically. HTTP/2 changes only how the data is transmitted, not what it means.

**The three headline features:**

1. **Multiplexing** — multiple requests and responses over one TCP connection simultaneously (no more serialization)
2. **Header compression (HPACK)** — headers are compressed, reducing overhead for repeated headers
3. **Binary protocol** — framed binary messages instead of plaintext (faster to parse, less error-prone)

HTTP/2 is now deployed on ~65% of websites (2024). All major cloud providers, CDNs, and load balancers support it. Your backend server should support it.

## 2. How It Actually Works

### From Text to Frames

HTTP/1.1 is text. HTTP/2 is binary. Every HTTP/2 message is broken into **frames** — small, fixed-structure binary units with a type, flags, stream ID, and payload.

```
HTTP/2 FRAME STRUCTURE:

┌─────────────────────────────────────────────────────┐
│  Length (24 bits)  │  Type (8 bits)  │  Flags (8 bits) │
├─────────────────────────────────────────────────────┤
│  R(1)  │  Stream Identifier (31 bits)                 │
├─────────────────────────────────────────────────────┤
│  Frame Payload (variable)                            │
└─────────────────────────────────────────────────────┘

Frame types:
  DATA        (0x0)  — request/response body
  HEADERS     (0x1)  — request/response headers
  PRIORITY    (0x2)  — stream priority
  RST_STREAM  (0x3)  — cancel a stream
  SETTINGS    (0x4)  — connection configuration
  PUSH_PROMISE(0x5)  — server push announcement
  PING        (0x6)  — keepalive
  GOAWAY      (0x7)  — graceful shutdown
  WINDOW_UPDATE(0x8) — flow control
  CONTINUATION(0x9)  — continued headers
```

You never deal with frames directly — the HTTP/2 library handles them. But knowing they exist explains how multiplexing works.

---

### Multiplexing: The Core Innovation

HTTP/1.1 is serial: send request, wait for response, then send next request. On one connection, requests queue up.

HTTP/2 introduces **streams** — independent, bidirectional byte sequences within one TCP connection. Each request gets its own stream ID. Frames from different streams are interleaved freely.

```
HTTP/1.1 — ONE connection, SERIAL:
Connection:  [Request A] → [Response A] → [Request B] → [Response B]
             ├────────────────────────────────────────────────────────►
             Request B must WAIT for Response A to complete.

HTTP/2 — ONE connection, PARALLEL (multiplexed):
Connection:  [A headers][B headers][C headers][A data][B data][C data]
             ├────────────────────────────────────────────────────────►
             Stream 1:  [headers]──────────────────[data]
             Stream 3:       [headers]────────[data]
             Stream 5:            [headers]──────────[data]

All three requests in flight simultaneously. No queuing.
No need for 6 TCP connections per domain.
```

**Practical impact:**
- Pages load faster (no request queuing)
- No domain sharding needed (one connection per origin)
- No connection count limits

### HPACK: Header Compression

HTTP headers are repetitive. Every request to `api.example.com` sends:
```
:authority: api.example.com
:method: GET
:scheme: https
authorization: Bearer eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...  (200+ bytes)
accept: application/json
user-agent: MyApp/1.0
```

On 100 requests, that's 100× the same headers. In HTTP/1.1: 100% of header bytes sent every time. In HTTP/2 with HPACK:

```
HPACK COMPRESSION:

Both sides maintain a "header table" (a shared dictionary of seen headers).

First request:
  Send: ":method: GET" (11 bytes)
  Both sides add it to table at index 1.

Second request:
  Send: index reference "1" (1 byte!)
  Receiver looks up table[1] → ":method: GET"

Static table (61 predefined entries):
  Table index 2  →  ":method: GET"
  Table index 3  →  ":method: POST"
  Table index 4  →  ":path: /"
  Table index 7  →  ":scheme: https"
  ... etc.

Result: 85–95% reduction in header size for typical API calls.
A 500-byte header set becomes ~20 bytes after the first request.
```

### Server Push

HTTP/2 allows servers to proactively push resources the client will need before it asks. A browser requests `index.html`; the server can push `style.css` and `script.js` immediately, anticipating the browser will request them.

```
Server Push:
  Client:  GET /index.html
  Server:  PUSH_PROMISE (stream 2) → /style.css
  Server:  PUSH_PROMISE (stream 4) → /script.js
  Server:  HEADERS (stream 1) + DATA → index.html
  Server:  HEADERS (stream 2) + DATA → style.css
  Server:  HEADERS (stream 4) + DATA → script.js

Browser receives CSS and JS without sending separate requests.
Saves 1 RTT per pushed resource.
```

**Reality check:** Server push is largely deprecated in HTTP/3 and many implementations. The browser already has good prediction and caching. Push often wastes bandwidth by sending resources the browser already has cached. Use `103 Early Hints` instead (a simpler modern alternative).

---

### HTTP/2 and TLS

HTTP/2 is technically possible without TLS (`h2c` — HTTP/2 cleartext), but in practice:
- All major browsers require TLS for HTTP/2
- Most servers only offer HTTP/2 over TLS
- HTTP/2 uses ALPN (Application-Layer Protocol Negotiation) during TLS handshake to negotiate h2

**Result:** Deploying HTTP/2 means deploying HTTPS. This is the right approach anyway.

## 3. Real-World Context

### HTTP/2 and gRPC

gRPC uses HTTP/2 as its transport. Every gRPC call is an HTTP/2 stream. This gives gRPC:
- Multiplexed streams for multiple concurrent RPCs over one connection
- Bidirectional streaming (client streaming, server streaming, full duplex)
- Header compression for gRPC metadata
- Flow control per stream

HTTP/2's multiplexing is what makes gRPC's streaming models possible and efficient.

### The head-of-line blocking that HTTP/2 didn't fully solve

HTTP/2 solved HTTP-level head-of-line blocking (requests queuing behind each other). But it still uses TCP, which has its own head-of-line blocking at the TCP level: if one TCP segment is lost, ALL HTTP/2 streams on that connection freeze until the segment is retransmitted.

On lossy networks (mobile, congested links), a 1% packet loss rate causes significant performance degradation for HTTP/2 — worse than HTTP/1.1 with multiple connections, because HTTP/1.1 can keep making progress on other connections while one connection is stalled.

This is the primary motivation for HTTP/3 / QUIC — replace TCP with a UDP-based protocol that has per-stream reliability without cross-stream head-of-line blocking.

## 4. Code Examples

### Checking if a server supports HTTP/2

```bash
# curl shows the protocol used
curl -v --http2 https://api.example.com/health 2>&1 | grep "< HTTP/"
# < HTTP/2 200   ← HTTP/2 in use

# Check with nghttp2 (more detail)
nghttp -v https://api.example.com/

# OpenSSL: see ALPN negotiation during TLS handshake
openssl s_client -connect api.example.com:443 -alpn h2 </dev/null 2>&1 | grep ALPN
# ALPN protocol: h2   ← HTTP/2 negotiated
```

### Configuring Nginx for HTTP/2

```nginx
server {
    listen 443 ssl http2;    # ← enable HTTP/2
    server_name api.example.com;

    ssl_certificate /etc/ssl/certs/example.com.crt;
    ssl_certificate_key /etc/ssl/private/example.com.key;

    # TLS settings for HTTP/2 compatibility
    ssl_protocols TLSv1.2 TLSv1.3;
    ssl_ciphers ECDHE-RSA-AES128-GCM-SHA256:ECDHE-RSA-AES256-GCM-SHA384;

    location / {
        proxy_pass http://backend;
        proxy_http_version 1.1;  # HTTP/2 from client; HTTP/1.1 to backend
        proxy_set_header Host $host;
    }
}
```

### httpx: force HTTP/2

```python
import httpx
import asyncio

async def make_http2_request():
    # httpx supports HTTP/2 with http2=True
    async with httpx.AsyncClient(http2=True) as client:
        # Make multiple requests concurrently — all over one HTTP/2 connection
        responses = await asyncio.gather(
            client.get("https://api.example.com/users"),
            client.get("https://api.example.com/products"),
            client.get("https://api.example.com/orders"),
        )

    for r in responses:
        print(f"{r.url}: {r.http_version} {r.status_code}")
    # https://api.example.com/users: HTTP/2 200
    # https://api.example.com/products: HTTP/2 200
    # https://api.example.com/orders: HTTP/2 200

asyncio.run(make_http2_request())
```

## 5. Trade-offs

| | HTTP/1.1 | HTTP/2 |
|---|---|---|
| Connections | 6 per domain | 1 per domain |
| Parallelism | Multiple connections | Multiplexed streams |
| Header overhead | Full text each request | HPACK compressed |
| Protocol | Text | Binary frames |
| TLS required? | No (but strongly advised) | Effectively yes |
| Server complexity | Low | Higher |
| TCP HOL blocking | Per-connection (some parallelism) | Affects all streams |
| Best for | Simple services, internal APIs | High-traffic, many requests/page |

## 6. Senior Engineer Perspective

**What a junior gets wrong:**

Thinking HTTP/2 is a "drop-in upgrade" with zero changes. The binary framing means old HTTP/1.1 debugging tricks (reading raw packet dumps with `strings`) don't work. Monitoring tools must support HTTP/2. Load balancers must be configured for HTTP/2 — Nginx's `listen 443 ssl http2` is not default. Frame-level errors produce cryptic messages if you're not looking at the right layer.

**What a senior thinks about:**

HTTP/2 multiplexing eliminates the need for multiple connections per domain — which means connection pool sizing changes. An HTTP/2-aware load balancer can send all traffic through fewer upstream connections while maintaining the same request throughput. This reduces server-side connection overhead significantly.

Also: HTTP/2 flow control is per-stream, not just per-connection (unlike TCP). A slow consumer on stream 3 doesn't slow stream 5. This matters for mixed-priority traffic (images vs. API calls in the same connection).
