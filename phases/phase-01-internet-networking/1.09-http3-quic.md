## 1. What Is It?

HTTP/3 (2022, RFC 9114) is the third major version of HTTP. It keeps the same HTTP semantics but replaces TCP+TLS as the transport with **QUIC** — a new transport protocol built on UDP.

**The problem HTTP/3 solves:** HTTP/2 over TCP has TCP-level head-of-line blocking. When one TCP packet is lost, all streams freeze until it's retransmitted. On mobile networks where 1–2% packet loss is normal, this is significant.

**The solution:** Replace TCP with QUIC, which provides per-stream reliability. A lost packet only blocks the stream it belongs to, not all streams.

## 2. How QUIC Works

```
HTTP/2 STACK:            HTTP/3 STACK:
┌──────────┐             ┌──────────┐
│  HTTP/2  │             │  HTTP/3  │
├──────────┤             ├──────────┤
│  TLS 1.3 │             │  QUIC    │ ← combines transport + security
├──────────┤             │ (TLS 1.3 │
│   TCP    │             │ built-in)│
├──────────┤             ├──────────┤
│   IP     │             │   UDP    │
└──────────┘             └──────────┘
                                │
                            Uses UDP as
                            the datagram
                            substrate
```

**QUIC reimplements what TCP does, but better:**
- Reliable delivery: yes (per stream)
- Ordering: yes (per stream, not global)
- Congestion control: yes (improved algorithms)
- Encryption: built-in (TLS 1.3, not optional)
- Multiplexing: yes, with independent stream reliability

---

### 0-RTT Connection Establishment

HTTP/2 over TLS 1.3: minimum 1 RTT for TCP + 1 RTT for TLS = **2 RTTs** before first byte.

HTTP/3 with QUIC: **1 RTT** for new connections, **0 RTT** for resumed connections.

```
QUIC NEW CONNECTION (1 RTT):
  Client → Initial (QUIC handshake + TLS ClientHello)
  Server → Initial + Handshake (TLS ServerHello + cert + keys)
  Client → Handshake (TLS Finished) + HTTP request   ← 1 RTT only
  Server → HTTP response

QUIC RESUMED CONNECTION (0-RTT):
  Client stores session ticket from previous connection.
  Client → Initial + 0-RTT data (HTTP request embedded!)
  Server → Response
  
  No RTT wasted before data starts flowing.
  0-RTT has security trade-offs (replay attacks) — used carefully.
```

---

### Connection Migration

QUIC connections are identified by a **Connection ID**, not by the IP:port tuple like TCP.

```
TCP connection = (client-IP, client-port, server-IP, server-port)
  If your phone switches from WiFi to 4G: your IP changes → TCP connection dies.
  
QUIC connection = Connection ID (a random token)
  If your phone switches from WiFi to 4G: 
  QUIC sends a path migration packet → same Connection ID continues.
  No reconnection. No dropped HTTP request. Seamless.
```

This is enormously valuable for mobile users who frequently switch between networks.

## 4. Should Your Backend Use HTTP/3?

**For most backend-to-backend service calls: No — HTTP/2 is fine.**

HTTP/3 shines at the edge (CDN, load balancer) where you're serving mobile users over lossy networks. Internal service-to-service calls are on stable data center networks with near-zero packet loss — HTTP/2 is better here (simpler, more widely supported, less CPU overhead for encryption).

**Where HTTP/3 adds value:**
- CDN to browser (especially mobile users)
- Global API endpoints with geographically distributed users
- Video streaming (YouTube, Netflix)

**Where HTTP/3 adds no value:**
- Internal microservices on the same VPC
- Stable wired connections
- Already-fast connections (data center networks)

## 5. Code Examples

```bash
# Test if a server supports HTTP/3
curl --http3 https://cloudflare.com -I 2>&1 | head -5
# HTTP/3 200

# Check Alt-Svc header (server advertises HTTP/3 support)
curl -I https://api.example.com 2>&1 | grep alt-svc
# alt-svc: h3=":443"; ma=86400   ← "I support HTTP/3 on port 443 for 86400s"
```

```nginx
# Nginx HTTP/3 configuration (Nginx 1.25+)
server {
    listen 443 ssl http2;
    listen 443 quic reuseport;   # HTTP/3 over QUIC

    ssl_certificate /etc/ssl/certs/example.crt;
    ssl_certificate_key /etc/ssl/private/example.key;

    add_header Alt-Svc 'h3=":443"; ma=86400';  # advertise HTTP/3
}
```

## 6. Senior Engineer Perspective

**What a senior thinks about:**

HTTP/3 at the CDN layer is largely transparent — enable it on Cloudflare/CloudFront and users benefit automatically. The backend doesn't change. The value proposition is clear: better mobile performance, 0-RTT resumption, connection migration.

For internal services, HTTP/3 adds complexity (different UDP-based tooling for debugging, more CPU overhead per connection due to per-packet encryption). The improvement for intra-datacenter calls is negligible. Stick with HTTP/2 for internal calls; use HTTP/3 at the edge.
