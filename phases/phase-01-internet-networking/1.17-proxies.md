## 1. The Difference

| | Forward Proxy | Reverse Proxy |
|---|---|---|
| Sits in front of | Clients | Servers |
| Hides | Clients from servers | Servers from clients |
| Direction | Client → Proxy → Internet | Internet → Proxy → Servers |
| Who configures it | Client (browser, OS) | Server operator |
| Examples | VPNs, Squid, corporate proxies | Nginx, Caddy, Traefik, HAProxy |
| Use case | Privacy, filtering, bypassing restrictions | Load balancing, SSL termination, caching |

---

## 2. Forward Proxy

A forward proxy sits between clients and the internet. Clients send all their requests to the proxy, which forwards them on the clients' behalf.

```
WITHOUT PROXY:
  Client → api.example.com (client IP visible to server)

WITH FORWARD PROXY:
  Client → Forward Proxy → api.example.com
  Server sees the proxy's IP, not the client's.
```

**Uses:**
- **Corporate network filtering:** Block access to social media, log all traffic
- **VPN:** Route traffic through a proxy in another country
- **Caching:** ISP-level caching proxy reduces bandwidth costs
- **Anonymization:** Tor is a multi-hop forward proxy network

**As a backend engineer:** You encounter forward proxies when configuring outbound HTTP calls. If your server is behind a corporate proxy, you need to set `HTTP_PROXY` / `HTTPS_PROXY` environment variables so your Python/Node code routes through it.

```python
import httpx
import os

# Respect proxy settings from environment
proxies = {}
if os.getenv("HTTPS_PROXY"):
    proxies["https://"] = os.getenv("HTTPS_PROXY")

async with httpx.AsyncClient(proxies=proxies) as client:
    response = await client.get("https://external-api.com/data")
```

## 3. Reverse Proxy

A reverse proxy sits in front of your servers. Clients connect to the proxy, which forwards requests to the appropriate backend server. The client doesn't know (and doesn't care) how many servers are behind the proxy.

```
Internet → Reverse Proxy → Backend Server A
                         → Backend Server B
                         → Backend Server C

Client connects to: api.example.com (the proxy's IP)
Client never knows about: 10.0.1.45, 10.0.1.46, 10.0.1.47
```

**Why every production backend uses a reverse proxy:**

```
1. SSL TERMINATION:
   Handle HTTPS at the proxy layer.
   Backend servers receive plain HTTP (simpler, faster).
   One place to manage certificates.

2. LOAD BALANCING:
   Distribute requests across backend instances.
   Remove a backend from rotation if it's unhealthy.

3. CACHING:
   Cache responses at the proxy (closer to clients, faster).

4. COMPRESSION:
   gzip/brotli compress all responses at the proxy.
   Backend sends plain text; proxy compresses before sending to client.

5. SECURITY:
   Hide backend server IPs (can't be attacked directly).
   Rate limiting, IP blocking, WAF rules.
   DDoS protection layer.

6. A/B TESTING AND ROUTING:
   Route 10% of traffic to new backend version.
   Route /api/* to API service, /* to static file server.

7. HTTP/2 TERMINATION:
   Proxy speaks HTTP/2 to clients, HTTP/1.1 to simpler backends.
   Backends don't need HTTP/2 support.
```

## 4. Nginx as Reverse Proxy

Nginx is the most widely deployed reverse proxy in the world.

```nginx
# Minimal reverse proxy
server {
    listen 80;
    server_name api.example.com;

    location / {
        proxy_pass http://127.0.0.1:8000;  # forward to backend

        # Pass original request information to backend
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;

        # Timeouts (critical — must match your app's response time)
        proxy_connect_timeout 5s;
        proxy_read_timeout 30s;
        proxy_send_timeout 30s;
    }
}

# SSL termination + reverse proxy
server {
    listen 443 ssl http2;
    server_name api.example.com;

    ssl_certificate /etc/letsencrypt/live/api.example.com/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/api.example.com/privkey.pem;

    # Multiple backends (load balancing — see topic 1.18)
    upstream backend {
        server 10.0.1.45:8000;
        server 10.0.1.46:8000;
        server 10.0.1.47:8000;
        keepalive 32;  # persistent connections to backends
    }

    location / {
        proxy_pass http://backend;
        proxy_http_version 1.1;
        proxy_set_header Connection "";  # enable keepalive to upstream
        proxy_set_header Host $host;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto https;
    }
}
```

## 5. Other Reverse Proxies

```
CADDY:
  Automatic HTTPS (fetches Let's Encrypt certs automatically).
  Simple config language.
  HTTP/3 support by default.
  Best for: smaller deployments, developers who want zero-config HTTPS.

TRAEFIK:
  Cloud-native. Auto-discovers services from Docker/Kubernetes labels.
  No manual config changes when adding services.
  Best for: microservices on Docker/Kubernetes.

HAPROXY:
  Extremely high performance. TCP and HTTP load balancing.
  Battle-tested for millions of connections.
  Best for: high-traffic L4/L7 load balancing.

ENVOY:
  Programmable, modern proxy. Built for service meshes.
  Used by Istio, Lyft, Google.
  Best for: microservices with service mesh requirements.
```

## 6. Senior Engineer Perspective

**What a junior gets wrong:**

Not setting the `X-Forwarded-For` header and then trying to read the client IP from the socket connection in the application. The socket IP is the proxy's IP (127.0.0.1 or the load balancer's IP), not the client's. The real client IP is in `X-Forwarded-For`. Always configure your reverse proxy to pass this header.

**What a senior always does:**

Configures timeouts explicitly. `proxy_read_timeout` default in Nginx is 60 seconds — if your backend hangs, Nginx holds the connection for 60 seconds. With 100 concurrent hung requests, Nginx has 100 open connections to a dead backend. Set timeouts to match your SLO, not to be maximally permissive.
