## 1. The Five Classes

```
1xx — Informational    Processing, continue, upgrade
2xx — Success          Request received, understood, accepted
3xx — Redirection      Further action needed to complete request
4xx — Client Error     Request has an error (your fault)
5xx — Server Error     Request is valid but server failed (our fault)
```

The class is more important than the exact code. When you see a 4xx, the client did something wrong. When you see a 5xx, your server has a bug or is overloaded. This matters for alerting: alert on 5xx (your problem), track 4xx (client behavior signal).

## 2. 1xx — Informational

```
100 Continue
  Client sent request headers with Expect: 100-continue.
  Server says "send the body." Used for large uploads — check headers first
  before transmitting megabytes of data.

101 Switching Protocols
  Server is switching to a different protocol.
  Used for WebSocket upgrade:
    Client: Connection: Upgrade, Upgrade: websocket
    Server: 101 Switching Protocols → connection is now WebSocket

103 Early Hints
  Server sends preliminary response headers before the final response.
  Used for resource hints (Link: </style.css>; rel=preload).
  Browser starts loading CSS before the HTML response arrives.
  Modern replacement for HTTP/2 Server Push.
```

## 3. 2xx — Success

```
200 OK
  The standard success. GET, PUT, PATCH responses.
  Body contains the requested/updated resource.

201 Created
  Resource was created. POST requests that create a resource.
  ALWAYS include Location: /users/456 pointing to the new resource.
  Body should contain the created resource.

202 Accepted
  Request accepted but not yet processed. For async operations.
  "We received your request. Processing happens in the background."
  Include a polling URL or webhook instructions in the response.
  Use for: long-running jobs, email sends, report generation.

204 No Content
  Success but no body. DELETE requests, successful actions with nothing to return.
  NEVER send a body with 204. Some clients break if you do.

206 Partial Content
  Response to a Range: request. Used for resumable downloads, video streaming.
  Includes Content-Range: bytes 0-1023/4096 header.

207 Multi-Status
  Multiple independent operations, each with its own status.
  Used in WebDAV and batch APIs.
  Body contains a list of individual results.
```

## 4. 3xx — Redirection

```
301 Moved Permanently
  Resource has permanently moved. Update bookmarks, links.
  Response is cached indefinitely by browsers (or per Cache-Control).
  Use for: domain migrations, URL restructuring.
  ⚠️ Browser changes POST to GET after 301 (historical quirk). Use 308 if you
     need to preserve the method.

302 Found (Temporary Redirect)
  Resource has temporarily moved. Don't update bookmarks.
  Not cached by default.
  Use for: temporary maintenance pages, A/B testing redirects.
  ⚠️ Same POST→GET issue. Use 307 for method-preserving temporary redirect.

303 See Other
  Response to a POST: go look at this other resource with GET.
  The POST-Redirect-GET pattern. Prevents form resubmission on back button.
    1. User submits form (POST /checkout)
    2. Server processes payment → responds 303 → Location: /order/456
    3. Browser GETs /order/456
    4. User can press back/refresh without re-submitting the form.

304 Not Modified
  Response to conditional request (If-None-Match or If-Modified-Since).
  "Your cached copy is still valid. Use it."
  No body. Saves bandwidth. Always respect this in clients.

307 Temporary Redirect (method preserved)
  Like 302 but preserves the HTTP method. POST stays POST.
  Use when redirecting form submissions temporarily.

308 Permanent Redirect (method preserved)
  Like 301 but preserves the HTTP method.
  Use when permanently moving a POST endpoint.
```

## 5. 4xx — Client Errors

```
400 Bad Request
  The request is malformed. Missing required field, wrong type, invalid JSON.
  Response body MUST explain what's wrong. Useless 400s are a developer experience
  failure.
  
  Good 400:  {"error": "validation_failed", "field": "email",
               "message": "must be a valid email address"}
  Bad 400:   {"error": "bad request"}  ← useless

401 Unauthorized (actually means "unauthenticated")
  No credentials provided, or credentials are invalid/expired.
  Should include WWW-Authenticate header suggesting auth scheme.
  Client must authenticate before retrying.
  
  Use when: JWT is missing, expired, or has invalid signature.
  
  Don't confuse with 403:
    401 = "Who are you?"
    403 = "I know who you are. You can't do this."

403 Forbidden
  Authenticated but not authorized. You're logged in but don't have permission.
  Use when: user doesn't have the role, resource belongs to another user.
  
  Security note: sometimes returning 404 instead of 403 is better
  (don't reveal that the resource exists to unauthorized users).

404 Not Found
  Resource doesn't exist at this URL.
  Use for: missing resources, deleted resources, invalid IDs.
  
  Also use for 403 when you want to hide resource existence from unauthorized users.
  
  Don't use for: "the feature you requested doesn't exist" (that's 501),
  "no results found" (return 200 with empty list).

405 Method Not Allowed
  HTTP method not supported on this endpoint.
  MUST include Allow: GET, POST header listing supported methods.
  Example: DELETE to a read-only endpoint.

408 Request Timeout
  Client took too long to send the request.
  Client should retry (possibly immediately).

409 Conflict
  Request conflicts with current state of the resource.
  
  Use for:
    - Duplicate registration (email already exists)
    - Optimistic locking conflict (someone else modified this record)
    - State transition conflict (can't cancel an already-shipped order)
  
  Body should explain the conflict and what state would allow the operation.

410 Gone
  Resource existed but has been permanently deleted.
  Unlike 404 (never existed), 410 means "it was here, now it's gone forever."
  Signals to search engines: remove from index. Rarely used but semantically correct
  for deleted content that you want to officially decommission.

413 Content Too Large (formerly Payload Too Large)
  Request body exceeds server's limit.
  Response should include the limit: "Max upload size: 10MB"

415 Unsupported Media Type
  Content-Type of the request is not supported.
  Client sent XML but server only accepts JSON.

422 Unprocessable Entity
  Request is syntactically valid but semantically wrong.
  Preferred over 400 for validation errors in REST APIs.
  
  The distinction:
    400 = "I can't parse this" (bad JSON, missing Content-Type)
    422 = "I parsed it, but the values are invalid" (invalid email format, 
          negative quantity, date in the past)
  
  FastAPI uses 422 for Pydantic validation failures by default.

423 Locked
  Resource is locked (WebDAV). Used occasionally in APIs for optimistic locking.

429 Too Many Requests
  Rate limit exceeded. MUST include:
    Retry-After: 60         (seconds until limit resets)
    X-RateLimit-Limit: 100  (requests per window)
    X-RateLimit-Remaining: 0
    X-RateLimit-Reset: 1718450200  (Unix timestamp of reset)
  
  Client MUST implement exponential backoff when receiving 429.
  Clients that ignore 429 and hammer the API get IP-banned.

451 Unavailable For Legal Reasons
  Content blocked for legal reasons (copyright, court order, GDPR).
  Named after Ray Bradbury's Fahrenheit 451.
```

## 6. 5xx — Server Errors

```
500 Internal Server Error
  Something went wrong on the server. Catch-all for unhandled exceptions.
  
  Your error handling middleware should catch all unhandled exceptions
  and return 500 with a sanitized error message.
  
  NEVER leak internal details in 500 responses:
    Bad: {"error": "NullPointerException at UserService.java:342"}
    Good: {"error": "internal_error", "request_id": "abc123"}
  
  Log the full stack trace internally, return only request_id to client.

501 Not Implemented
  The server does not support the functionality required.
  Use for: endpoints you plan to build but haven't yet, features not yet supported.

502 Bad Gateway
  The server (acting as a proxy/gateway) received an invalid response from upstream.
  
  Common causes:
    - Backend server crashed or returned garbage
    - Backend started returning HTML error pages instead of JSON
    - Protocol mismatch (proxy expects HTTP/1.1, backend speaks something else)
  
  You see this from Nginx when your app server is down or returning errors.

503 Service Unavailable
  Server is temporarily unable to handle requests.
  
  Use for:
    - Planned maintenance (include Retry-After header)
    - Server overloaded (circuit breaker tripped)
    - Dependency unavailable (database down)
  
  Response should include Retry-After if you know when you'll be back.
  
  Health check endpoints return 503 when the service is not ready
  (starting up, dependency failure) — Kubernetes uses this to stop routing traffic.

504 Gateway Timeout
  The proxy/gateway timed out waiting for an upstream response.
  
  Common causes:
    - Backend is too slow (query taking longer than proxy timeout)
    - Backend is down and the proxy waited its full timeout
    - Network partition between proxy and backend
  
  You see this from Nginx when your app takes too long to respond.
  
  Fix: optimize the slow operation, increase timeout (short-term), or add async
  processing so the response returns before the timeout.

507 Insufficient Storage
  The server has run out of disk space. Uncommon in web APIs, common in
  file upload services and storage backends.

509 Bandwidth Limit Exceeded
  Non-standard (used by some hosting providers). You've exceeded your
  monthly bandwidth quota.
```

## 7. Correct Usage Patterns

```python
from fastapi import FastAPI, HTTPException
from fastapi.responses import JSONResponse

app = FastAPI()

# 201 Created with Location header
@app.post("/users", status_code=201)
async def create_user(data: UserCreate, response: Response):
    user = await db.create_user(data)
    response.headers["Location"] = f"/users/{user.id}"
    return user

# 202 Accepted for async operations
@app.post("/reports/generate", status_code=202)
async def generate_report(params: ReportParams):
    job_id = await queue.enqueue_report(params)
    return {
        "job_id": job_id,
        "status_url": f"/reports/jobs/{job_id}",
        "message": "Report generation started. Check status_url for progress."
    }

# 204 No Content for DELETE
@app.delete("/users/{user_id}", status_code=204)
async def delete_user(user_id: int):
    await db.delete_user(user_id)
    # Return nothing — 204 means no body

# 409 Conflict with helpful message
@app.post("/users/register")
async def register(data: RegisterRequest):
    if await db.email_exists(data.email):
        raise HTTPException(
            status_code=409,
            detail={
                "error": "email_already_registered",
                "message": f"{data.email} is already associated with an account",
                "action": "Try logging in, or use password reset if you forgot your password"
            }
        )
    ...

# 429 Too Many Requests with Retry-After
@app.get("/api/data")
async def get_data(request: Request, response: Response):
    client_ip = request.client.host
    remaining = await rate_limiter.check(client_ip)
    if remaining <= 0:
        reset_time = await rate_limiter.get_reset_time(client_ip)
        raise HTTPException(
            status_code=429,
            detail="Rate limit exceeded",
            headers={
                "Retry-After": str(int(reset_time - time.time())),
                "X-RateLimit-Limit": "100",
                "X-RateLimit-Remaining": "0",
                "X-RateLimit-Reset": str(int(reset_time)),
            }
        )
    ...

# 503 for health checks
@app.get("/health")
async def health_check():
    try:
        await db.ping()
        await redis.ping()
        return {"status": "healthy"}
    except Exception as e:
        # Return 503 — Kubernetes stops routing traffic here
        return JSONResponse(
            status_code=503,
            content={"status": "unhealthy", "reason": str(e)}
        )
```

## 8. Senior Engineer Perspective

**What a junior gets wrong:**

Using 200 for everything and putting the "real" status in the body: `{"success": false, "error": "not found"}`. HTTP clients, monitoring tools, CDNs, and load balancers all look at the status code. A 200 with an error body is invisible to the entire infrastructure layer. It also defeats HTTP caching (200 responses get cached; 404 responses don't).

**The status code decision tree:**

```
Request received and parsed correctly?
  No → 400 (bad syntax) or 422 (bad values)
  
Is the client authenticated?
  No → 401

Is the client authorized?
  No → 403 (or 404 to hide existence)

Does the resource exist?
  No → 404

Does the operation conflict with current state?
  Yes → 409

Did something go wrong on our server?
  Yes → 500 (log details internally, return request_id only)

Did our dependency (DB, external API) fail?
  Yes → 502 (their fault) or 503 (we're degraded)

Did our dependency time out?
  Yes → 504

Everything OK?
  Created something → 201
  Async action started → 202
  Action with no return value → 204
  Normal response → 200
```

**The monitoring rule:**

Alert on: 5xx rate > threshold (YOUR bugs), 4xx rate spikes (attack or bad deployment).
Track but don't alert on: normal 4xx (404s, 401s) — they're client behavior signals.
