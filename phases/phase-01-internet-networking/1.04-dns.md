## 1. What Is It? (Theory)

DNS (Domain Name System) is the distributed, hierarchical database that translates human-readable domain names (`api.example.com`) into machine-routable IP addresses (`54.23.167.89`).

It is one of the most critical pieces of internet infrastructure. If DNS fails, the internet does not "go down" — it becomes unreachable by name. You can still reach services by IP address, but everything built on domain names (every website, every API, every email) stops working.

**The real-world analogy:**

DNS is the phone book of the internet. Instead of memorizing every contact's phone number, you look them up by name. The phone book is distributed (no single copy) and hierarchical (country → region → local). When you look up "api.example.com", you're consulting a chain of phone books, each one knowing who to ask for the next level of detail.

## 2. How It Actually Works (Deep Mechanics)

### DNS Records — The Building Blocks

DNS is not just "name → IP." It is a general-purpose naming system with multiple record types:

```
RECORD TYPE  PURPOSE                           EXAMPLE
─────────────────────────────────────────────────────────────────────
A            IPv4 address                      api.example.com → 54.23.167.89
AAAA         IPv6 address                      api.example.com → 2001:db8::1
CNAME        Canonical name (alias)            www → api.example.com
MX           Mail exchange (email routing)     example.com → mail.example.com (priority 10)
TXT          Arbitrary text                    SPF records, DKIM, domain verification
NS           Name server for the domain        example.com → ns1.cloudflare.com
SOA          Start of authority                Zone metadata, serial number, TTL defaults
PTR          Reverse lookup (IP → name)        54.23.167.89 → api.example.com
SRV          Service location (host + port)    _http._tcp → host:port weight priority
CAA          Certificate authority auth        Who can issue SSL certs for this domain
ALIAS/ANAME  Like CNAME but at zone apex       example.com → loadbalancer.aws.com
```

**A Record — Most Common**

```
Query:  What is the IPv4 address for api.example.com?
Answer: api.example.com. 300 IN A 54.23.167.89
                         │      │ │
                         TTL   Class  IP address
                         (seconds)
```

**CNAME — Aliasing**

```
www.example.com → CNAME → api.example.com → A → 54.23.167.89

CNAME says "this name is just another name for that name."
The resolver follows the chain until it finds an A record.

CRITICAL RULE: You cannot have a CNAME at your zone apex (root domain).
  example.com CANNOT have a CNAME (would break MX and NS records).
  www.example.com CAN have a CNAME.

Modern DNS providers offer ALIAS or ANAME records that behave like CNAME
at the zone apex (flattened to A record at query time).
```

**MX Record — Email Routing**

```
dig MX gmail.com

gmail.com. 3600 IN MX 5  gmail-smtp-in.l.google.com.
gmail.com. 3600 IN MX 10 alt1.gmail-smtp-in.l.google.com.
gmail.com. 3600 IN MX 20 alt2.gmail-smtp-in.l.google.com.

Lower priority number = preferred server.
Multiple MX records = redundancy (try 5 first, then 10, then 20).
```

**TXT Record — Everything Else**

```
SPF (Sender Policy Framework — authorize email senders):
  example.com TXT "v=spf1 include:_spf.google.com ~all"

DKIM (domain key — sign outgoing emails):
  selector._domainkey.example.com TXT "v=DKIM1; k=rsa; p=MIGf..."

Domain ownership verification (Google, AWS, GitHub):
  _github-challenge-myorg.example.com TXT "abc123verify"

SRV record example (for service discovery):
  _mongodb._tcp.example.com SRV 0 5 27017 mongo.example.com.
```

### TTL (Time To Live) — The Most Important Concept

TTL is the number of seconds a DNS record can be cached by resolvers. It controls **propagation speed** vs. **resolver load**.

```
LOW TTL (60–300 seconds):
  → Changes propagate quickly (minutes)
  → More resolver queries (higher DNS server load)
  → Use when: before migrations, during incidents, for frequently-changed records

HIGH TTL (3600–86400 seconds):
  → Changes propagate slowly (hours to days)
  → Fewer resolver queries (lower load, faster responses from cache)
  → Use when: stable records, high-traffic domains

PRODUCTION PATTERN:
  Normal: TTL = 3600 (1 hour)
  Before a migration:
    1. Lower TTL to 60 (one week before migration)
    2. Wait for existing cache to expire (1 hour)
    3. Make the change
    4. Verify
    5. Raise TTL back to 3600

  If you change DNS with TTL=86400 without lowering first:
  Some users will see the OLD IP for up to 24 hours.
  You cannot force resolvers to flush their cache.
```

### DNS Lookup Flow (What Happens When You Type a URL)

```
1. BROWSER CACHE
   Checked first. Chrome/Firefox cache DNS results in-process.
   Even shorter than the DNS TTL sometimes.

2. OS CACHE (/etc/nsswitch.conf, Windows DNS cache)
   OS maintains its own DNS cache.
   Linux: `resolvectl status` or `systemd-resolve --statistics`
   macOS: `sudo dscacheutil -flushcache`

3. STUB RESOLVER (/etc/resolv.conf)
   The OS sends the query to the configured DNS resolver.
   Typically: your ISP's resolver, 8.8.8.8, 1.1.1.1, or in-cluster DNS.

4. RECURSIVE RESOLVER (8.8.8.8 / 1.1.1.1 / your ISP)
   If cached: returns immediately.
   If not cached: performs the full recursive resolution.
   See Topic 1.5 for the full recursive process.

5. ANSWER RETURNED
   Resolver returns the answer with the TTL.
   OS caches it. Browser caches it. You get the IP.
```

### /etc/hosts — Pre-DNS Override

Before DNS existed, `/etc/hosts` was how name-to-IP mapping worked. It still takes precedence over DNS on every Unix/Linux/macOS system:

```
/etc/hosts:

127.0.0.1       localhost
::1             localhost
192.168.1.100   mydb.local
10.0.1.45       api.internal
```

This is how Docker and Kubernetes inject names for containers/services, and how developers override DNS for local testing.

### DNS in Production Systems

**Your backend service and DNS:**

```python
import socket
import time

# DNS resolution happens implicitly on every connection
# httpx, requests, psycopg2, redis — all call getaddrinfo() before connecting

# The first call may take 50–100ms (cache miss)
# Subsequent calls: ~0ms (OS cache hit)

# PROBLEM: What if you cache the IP yourself and the server changes?
# Connection pools cache the IP. If the IP changes (failover, scaling),
# existing connections still work, but new connections will resolve the new IP.
# TTL expiry is when new IPs get picked up.

# GOOD PRACTICE: Use short connection TTLs in your pool
# so that IP changes are reflected quickly after DNS TTL expires.
```

**Health check with DNS-based load balancing:**

```python
import socket

def resolve_all_ips(hostname: str) -> list[str]:
    """
    DNS round-robin returns multiple A records.
    getaddrinfo() returns all of them.
    Most HTTP clients only use the first one — missing the round-robin intent.
    """
    results = socket.getaddrinfo(hostname, None, socket.AF_INET)
    return list({r[4][0] for r in results})  # unique IPs

ips = resolve_all_ips("api.example.com")
print(ips)  # ['54.23.167.89', '54.23.167.90', '54.23.167.91']
# DNS round-robin: each query returns IPs in different order
```

## 3. Real-World Context

### The GitLab DNS incident (2017)

GitLab's database recovery incident was partly complicated by DNS. After restoring from a backup, the old IPs pointed to non-existent services. DNS TTLs were high (1 hour), so some clients continued trying to reach the old endpoints long after they were gone.

**Lesson:** Always lower TTL before any significant infrastructure change. Don't wait until the migration — wait for the old TTL to expire first.

### Cloudflare 1.1.1.1 vs 8.8.8.8

```
8.8.8.8  (Google Public DNS):
  Latency: ~20ms average globally
  Privacy: Google may log queries
  Speed: Excellent

1.1.1.1  (Cloudflare):
  Latency: ~11ms average globally (fastest public DNS)
  Privacy: No query logging (audited by KPMG)
  Speed: Fastest in most regions
  Supports: DNS over HTTPS (DoH), DNS over TLS (DoT)

For production workloads: use your cloud provider's resolver
  AWS: 169.254.169.253 (VPC DNS, resolves .internal names)
  GCP: 169.254.169.254 (metadata server, also DNS)
  Kubernetes: CoreDNS (resolves service names within cluster)
```

### DNS-based service discovery in Kubernetes

```
Service name: postgres-svc in namespace default

Kubernetes CoreDNS creates:
  postgres-svc.default.svc.cluster.local → ClusterIP

Pods in same namespace can use:
  postgres-svc  (short form)
  postgres-svc.default  (with namespace)
  postgres-svc.default.svc.cluster.local  (fully qualified)

This is how your backend finds the database:
  DB_HOST=postgres-svc  ← resolves via CoreDNS to the service IP
```

## 4. Code Examples

### Using dnspython for production DNS work

```python
import dns.resolver
import dns.rdatatype

def inspect_domain(domain: str) -> dict:
    """Comprehensive DNS inspection for a domain."""
    result = {}
    resolver = dns.resolver.Resolver()
    resolver.timeout = 5
    resolver.lifetime = 10

    record_types = ["A", "AAAA", "MX", "TXT", "NS", "CNAME"]

    for rtype in record_types:
        try:
            answers = resolver.resolve(domain, rtype)
            result[rtype] = {
                "records": [str(r) for r in answers],
                "ttl": answers.rrset.ttl,
            }
        except (dns.resolver.NXDOMAIN, dns.resolver.NoAnswer,
                dns.resolver.NoNameservers):
            pass

    return result

# Check domain before a migration
info = inspect_domain("api.example.com")
for rtype, data in info.items():
    print(f"{rtype} (TTL={data['ttl']}s): {data['records']}")
# A (TTL=3600s): ['54.23.167.89']  ← high TTL! Lower before migration.
```

### DNS monitoring in production

```python
import dns.resolver
import time
from datetime import datetime

def monitor_dns_propagation(domain: str, expected_ip: str, interval: int = 30):
    """
    After a DNS change, monitor propagation across resolvers.
    Use after lowering TTL and making the change.
    """
    resolvers = {
        "Cloudflare": "1.1.1.1",
        "Google":     "8.8.8.8",
        "OpenDNS":    "208.67.222.222",
    }

    print(f"Monitoring {domain} → expecting {expected_ip}")
    print(f"Checking every {interval}s...")

    while True:
        all_updated = True
        timestamp = datetime.utcnow().strftime("%H:%M:%S")

        for name, ns_ip in resolvers.items():
            resolver = dns.resolver.Resolver(configure=False)
            resolver.nameservers = [ns_ip]
            resolver.timeout = 3

            try:
                answers = resolver.resolve(domain, "A")
                current_ip = str(answers[0])
                status = "✓" if current_ip == expected_ip else "✗"
                if current_ip != expected_ip:
                    all_updated = False
                print(f"  [{timestamp}] {name}: {current_ip} {status}")
            except Exception as e:
                all_updated = False
                print(f"  [{timestamp}] {name}: ERROR ({e})")

        if all_updated:
            print("\n✓ All resolvers updated! DNS propagation complete.")
            break

        print()
        time.sleep(interval)
```

dig +trace api.example.com

cat /etc/resolv.conf

nslookup api.example.com

## 6. Senior Engineer Perspective

**What a junior gets wrong:**

Not lowering TTL before DNS changes. The TTL on a record is a promise to resolvers: "This value is good for TTL seconds." Breaking that promise — changing the IP before the TTL expires — means some clients see the old IP for hours. Always lower TTL to 60 seconds at least one "old TTL" period before making the change.

**What a senior thinks about:**

1. **DNS as a failure point:** DNS is not DNS is not DNS. Your cloud provider's DNS resolver (`169.254.169.253` in AWS) resolves internal service names. Public resolvers (8.8.8.8) don't know about them. In production Kubernetes, CoreDNS resolves service names. Configure DNS properly from the start — retrofitting it is painful.

2. **SOA serial numbers:** When managing your own DNS zones, increment the SOA serial number on every zone change. Secondary nameservers use this to detect updates. A non-incremented serial means your changes don't propagate to secondary DNS.

3. **Negative caching (NXDOMAIN TTL):** Trying to resolve a non-existent domain gets you a NXDOMAIN response. This is also cached for the SOA's negative TTL (typically 1–24 hours). If your application makes DNS queries for hostnames that don't exist (typos, wrong config), those failures are cached and persist long after you fix the typo. Monitor for NXDOMAIN responses.

**The production checklist for DNS changes:**

```
□ Current TTL: ___ seconds
□ Lower TTL to 60 seconds (now)
□ Wait [current TTL] seconds for existing caches to expire
□ Make DNS change
□ Verify change propagated with: dig +short @1.1.1.1 @8.8.8.8 domain.com
□ Monitor for errors during propagation window
□ After 24h: raise TTL back to 3600
```
