## 1. What Is It?

HTTPS is HTTP over TLS (Transport Layer Security). TLS is the protocol that provides:
- **Confidentiality** — nobody can read your data in transit
- **Integrity** — nobody can tamper with your data in transit
- **Authentication** — you're actually talking to the server you think you are

Without HTTPS, every ISP, router, and attacker between client and server can read and modify all traffic. Passwords, tokens, personal data — all visible in plaintext. In 2024, HTTPS is mandatory for any production system.

**The real-world analogy:**

HTTP is like sending a postcard — everyone who handles it can read it. HTTPS is like a sealed, tamper-evident envelope that only the recipient can open, and the recipient's name is certified by a trusted notary.

## 2. How It Actually Works

### The TLS Handshake (TLS 1.3, simplified)

```
CLIENT                                          SERVER
  │                                               │
  │──── ClientHello ─────────────────────────────►│
  │  TLS version, cipher suites supported,        │
  │  client random, supported groups (curves)     │
  │                                               │
  │◄─── ServerHello ──────────────────────────────│
  │  Chosen cipher suite, server random,          │
  │  server's key share (Diffie-Hellman public)   │
  │                                               │
  │◄─── Certificate ──────────────────────────────│
  │  Server's public key certificate chain        │
  │  Client verifies: is this cert valid?         │
  │    - Signed by a trusted CA?                  │
  │    - Not expired?                             │
  │    - Matches the hostname (api.example.com)?  │
  │                                               │
  │◄─── CertificateVerify ────────────────────────│
  │  Proof server owns the private key            │
  │                                               │
  │◄─── Finished ─────────────────────────────────│
  │  Handshake complete from server side          │
  │                                               │
  │──── Finished ────────────────────────────────►│
  │  Handshake complete from client side          │
  │                                               │
  │◄══════════════ Encrypted Data ════════════════►│
  │  HTTP traffic now encrypted with session key  │
```

**TLS 1.3 is 1 RTT** (one round trip before data flows). TLS 1.2 was 2 RTTs. The improvement comes from TLS 1.3 sending the key exchange in the first message.

### Certificates: The Trust Chain

A TLS certificate is a public key + identity information + a digital signature from a Certificate Authority (CA).

```
CERTIFICATE STRUCTURE:
  Subject:     api.example.com
  Issuer:      Let's Encrypt Authority X3
  Public Key:  [2048-bit RSA key]
  Valid From:  2024-06-01
  Valid To:    2024-09-01   (90 days — Let's Encrypt default)
  Signature:   [signed by Let's Encrypt's private key]

TRUST CHAIN:
  api.example.com cert
    └── signed by Let's Encrypt Intermediate CA
         └── signed by Let's Encrypt Root CA
              └── trusted by all major OS/browsers (pre-installed)

Browser trusts root CA → trusts intermediate CA → trusts api.example.com
```

**Certificate Authorities (CAs):** Organizations that verify you own a domain and issue certificates. Browsers ship with ~150 trusted root CA certificates pre-installed. Let's Encrypt made free, automated certificates mainstream.

### How Certificate Validation Works

```python
# Python: what happens when you verify an SSL certificate
import ssl
import socket

context = ssl.create_default_context()  # loads system CA bundle

with socket.create_connection(("api.example.com", 443)) as sock:
    with context.wrap_socket(sock, server_hostname="api.example.com") as ssock:
        cert = ssock.getpeercert()
        print(f"Subject: {cert['subject']}")
        print(f"Issuer: {cert['issuer']}")
        print(f"Expires: {cert['notAfter']}")
        print(f"Cipher: {ssock.cipher()}")
        print(f"TLS version: {ssock.version()}")
```

**Certificate errors in production:**
```
ssl.SSLCertVerificationError: CERTIFICATE_VERIFY_FAILED

Common causes:
  1. Certificate expired (check: openssl s_client -connect host:443 | grep dates)
  2. Hostname mismatch (cert says api.example.com, you're hitting www.example.com)
  3. Self-signed cert (no trusted CA in chain) — common in internal services
  4. Intermediate CA missing from chain (server not sending full chain)
  5. System CA bundle out of date (update ca-certificates package)
  6. Clock wrong on server or client (cert appears expired/future)
```

### HSTS (HTTP Strict Transport Security)

Once a user visits your HTTPS site, the server can instruct browsers to NEVER use HTTP again:

```
Strict-Transport-Security: max-age=31536000; includeSubDomains; preload

max-age=31536000  → remember this for 1 year
includeSubDomains → apply to all subdomains too
preload           → submit to browsers' preload list (hardcoded HTTPS)
```

**HSTS preload list:** Browsers ship with a hardcoded list of domains that must always use HTTPS. Once you're on this list, HTTP is impossible for that domain — even on a fresh browser install. `https://hstspreload.org/` manages submissions.

### Certificate Lifecycle and Let's Encrypt

```bash
# Let's Encrypt with Certbot — free, automated certificates
# Install cert for Nginx:
certbot --nginx -d api.example.com -d www.example.com

# Renew automatically (add to cron or systemd timer):
certbot renew --quiet

# With Caddy — automatic HTTPS, no setup needed:
# Caddy fetches and renews Let's Encrypt certs automatically
# Just configure your domain in Caddyfile.

# Manual inspection of a certificate:
openssl s_client -connect api.example.com:443 -servername api.example.com </dev/null | \
  openssl x509 -noout -dates -subject -issuer
# notBefore=Jun  1 00:00:00 2024 GMT
# notAfter=Sep  1 00:00:00 2024 GMT
# subject=CN = api.example.com
# issuer=C = US, O = Let's Encrypt, CN = R3
```

## 3. Real-World Context

### The cost of a TLS certificate expiration

When Let's Encrypt changed their root certificate in 2021, millions of devices with old root CA bundles (old iPhones, old Android, old embedded systems) suddenly could not verify Let's Encrypt certificates. Engineers had to scramble to add cross-signed certificates or fall back to paid CA certificates.

**The lesson:** Certificate expiration is always fatal. Set up monitoring alerts for certificates expiring within 30 days, 14 days, 7 days. Automate renewal. Test renewal before the old cert expires.

### mTLS (Mutual TLS) for service-to-service auth

Standard TLS: only the server presents a certificate (client authenticates server). mTLS: both sides present certificates (server also authenticates client).

```
Use mTLS for:
  Service-to-service internal calls
  Zero-trust network architectures
  API endpoints that should only be called by known clients

Tools: Istio, Linkerd (automatic mTLS in Kubernetes)
       Let's Encrypt + client certs (manual)
       AWS API Gateway + client certificates
```

## 4. Code Examples

### Configuring TLS correctly in FastAPI (production)

```python
# Production: TLS terminated at Nginx/load balancer, not at the app.
# App sees plain HTTP internally.

# Development: run with TLS directly
import uvicorn
from fastapi import FastAPI

app = FastAPI()

if __name__ == "__main__":
    uvicorn.run(
        app,
        host="0.0.0.0",
        port=8443,
        ssl_keyfile="/etc/ssl/private/example.key",
        ssl_certfile="/etc/ssl/certs/example.crt",
    )
```

### Nginx TLS best practices configuration

```nginx
server {
    listen 443 ssl http2;
    server_name api.example.com;

    ssl_certificate     /etc/letsencrypt/live/api.example.com/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/api.example.com/privkey.pem;

    # Only allow TLS 1.2 and 1.3
    ssl_protocols TLSv1.2 TLSv1.3;

    # Strong cipher suites (TLS 1.2 fallback)
    ssl_ciphers ECDHE-RSA-AES128-GCM-SHA256:ECDHE-RSA-AES256-GCM-SHA384;
    ssl_prefer_server_ciphers off;  # TLS 1.3 ignores this anyway

    # HSTS
    add_header Strict-Transport-Security "max-age=31536000; includeSubDomains" always;

    # OCSP stapling (proves cert not revoked, without client contacting CA)
    ssl_stapling on;
    ssl_stapling_verify on;
    resolver 8.8.8.8 valid=300s;

    location / {
        proxy_pass http://backend:8000;  # HTTP internally
    }
}

# Redirect HTTP to HTTPS
server {
    listen 80;
    server_name api.example.com;
    return 301 https://$host$request_uri;
}
```

### Certificate monitoring script

```python
import ssl, socket
from datetime import datetime, timezone

def check_cert_expiry(hostname: str, port: int = 443) -> dict:
    """Check days until certificate expires."""
    context = ssl.create_default_context()
    with socket.create_connection((hostname, port), timeout=10) as sock:
        with context.wrap_socket(sock, server_hostname=hostname) as ssock:
            cert = ssock.getpeercert()

    expiry_str = cert["notAfter"]
    expiry = datetime.strptime(expiry_str, "%b %d %H:%M:%S %Y %Z")
    expiry = expiry.replace(tzinfo=timezone.utc)
    days_left = (expiry - datetime.now(timezone.utc)).days

    return {
        "hostname": hostname,
        "expires": expiry_str,
        "days_left": days_left,
        "issuer": dict(cert["issuer"][0]),
        "critical": days_left < 14,
        "warning": days_left < 30,
    }

# Run for your domains
for domain in ["api.example.com", "www.example.com"]:
    info = check_cert_expiry(domain)
    status = "🔴 CRITICAL" if info["critical"] else ("🟡 WARNING" if info["warning"] else "✅ OK")
    print(f"{status} {domain}: {info['days_left']} days left")
```

## 5. Senior Engineer Perspective

**What a junior gets wrong:**

Disabling certificate verification to "fix" SSL errors in development: `verify=False` in requests, `ssl_verify: false` in configs. This makes SSL completely pointless — you're encrypting data but accepting any certificate, including attacker-injected ones. Never do this in production. Fix the underlying certificate issue.

**What a senior thinks about:**

1. **Certificate pinning vs. rotation:** Pinning a specific cert hash breaks every time the cert rotates (every 90 days for Let's Encrypt). Pin the public key instead, or better: trust the CA chain properly.

2. **Internal services and self-signed certs:** Internal microservices using self-signed certs create maintenance burden (add to every container's trust store) and security theater (often `verify=False` ends up in the code). Use a private internal CA (cfssl, Vault PKI) instead — issue certs from it, install the internal CA root in all services.

3. **TLS termination placement:** TLS should be terminated at the load balancer / API gateway. Traffic inside the data center can be plaintext (reduces CPU overhead) unless you're in a zero-trust environment. Know your threat model.
