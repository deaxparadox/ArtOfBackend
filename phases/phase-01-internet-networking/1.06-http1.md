## 1. What Is It?

HTTP (HyperText Transfer Protocol) is the application-layer protocol that powers the web. Every API call, every web page, every REST request uses HTTP. Understanding HTTP is not optional — it is the vocabulary of backend engineering.

HTTP/1.0 (1996) was the first formalized version. It is simple to the point of being minimal: a client sends a text-based request, the server sends a text-based response, the connection closes. That's it.

HTTP/1.0 is mostly obsolete in production (HTTP/1.1, HTTP/2, HTTP/3 replaced it), but understanding its design reveals exactly why the upgrades were needed.

## 2. How It Actually Works

### HTTP Is Text

This is the most important thing to understand. HTTP/1.x is a human-readable text protocol. You can literally type it by hand.

```
HTTP/1.0 Request:

GET /users/123 HTTP/1.0\r\n
Host: api.example.com\r\n
Accept: application/json\r\n
\r\n
(blank line = end of headers)

HTTP/1.0 Response:

HTTP/1.0 200 OK\r\n
Content-Type: application/json\r\n
Content-Length: 52\r\n
\r\n
{"id": 123, "name": "Alice", "email": "alice@example.com"}
```

Every header is `Key: Value\r\n`. The blank line (`\r\n\r\n`) separates headers from the body. The body length is specified in `Content-Length`. Simple, readable, debuggable with any text tool.

---

### The Request Line

```
METHOD SPACE URI SPACE HTTP-VERSION CRLF

GET /users/123 HTTP/1.0
│    │         │
│    │         └── Protocol version
│    └── Request target (path + query string)
└── HTTP method
```

**HTTP Methods in HTTP/1.0:** GET, POST, HEAD (the original three). PUT, DELETE, PATCH came later.

**Request URI:**
```
/users/123                    → path only
/search?q=backend&page=2      → path + query string
http://api.example.com/users  → absolute URI (used with proxies)
*                             → applies to server (OPTIONS *)
```

---

### The Response Status Line

```
HTTP-VERSION SPACE STATUS-CODE SPACE REASON-PHRASE CRLF

HTTP/1.0 200 OK
HTTP/1.0 404 Not Found
HTTP/1.0 500 Internal Server Error
```

Status codes are the language servers use to communicate what happened:

```
1xx — Informational (rarely used in HTTP/1.0)
2xx — Success
  200 OK
  201 Created
  204 No Content

3xx — Redirection
  301 Moved Permanently   (cacheable, update bookmarks)
  302 Found               (temporary redirect)
  304 Not Modified        (cached response is still valid)

4xx — Client Error (you did something wrong)
  400 Bad Request
  401 Unauthorized        (not authenticated)
  403 Forbidden           (authenticated but not allowed)
  404 Not Found
  405 Method Not Allowed
  409 Conflict
  422 Unprocessable Entity
  429 Too Many Requests

5xx — Server Error (we did something wrong)
  500 Internal Server Error
  502 Bad Gateway         (proxy received bad response from upstream)
  503 Service Unavailable (overloaded, down for maintenance)
  504 Gateway Timeout     (proxy timed out waiting for upstream)
```

---

### HTTP/1.0's Fatal Flaw: One Request Per Connection

HTTP/1.0 opens a **new TCP connection for every request**. After the response, the connection closes.

```
HTTP/1.0 — loading a web page with 3 resources:

[TCP handshake] → GET /index.html → [response] → [TCP close]   2 RTTs + content
[TCP handshake] → GET /style.css  → [response] → [TCP close]   2 RTTs + content
[TCP handshake] → GET /script.js  → [response] → [TCP close]   2 RTTs + content

Total: 6 RTTs + 3 content transfers.
With 80ms RTT (cross-continental): 480ms just in TCP overhead.
```

This was the primary driver for HTTP/1.1's persistent connections and HTTP/2's multiplexing.

---

### HTTP Headers: The Metadata Layer

Headers carry metadata about the request and response. They are the extensibility mechanism of HTTP.

**Common request headers:**
```
Host: api.example.com          REQUIRED in HTTP/1.1+, optional in 1.0
                                Needed because one IP can serve multiple domains
Accept: application/json        "I want JSON"
Accept-Encoding: gzip, deflate  "I can handle compressed responses"
Content-Type: application/json  "I'm sending JSON in the body"
Content-Length: 52              "The body is 52 bytes"
Authorization: Bearer eyJ...    "Here is my token"
User-Agent: MyApp/1.0           "Who I am"
Cache-Control: no-cache         "Don't return cached response"
Connection: keep-alive          "Please reuse this connection (HTTP/1.1)"
```

**Common response headers:**
```
Content-Type: application/json; charset=utf-8
Content-Length: 84
Content-Encoding: gzip          "Response body is gzip-compressed"
Cache-Control: max-age=3600     "Cache this for 1 hour"
ETag: "abc123"                  "Version identifier for this resource"
Last-Modified: Tue, 15 Jun 2024 14:30:00 GMT
Location: /users/456            "Redirect to this URL (with 3xx responses)"
Set-Cookie: session=abc; HttpOnly; Secure
X-Request-Id: 550e8400-e29b     "Unique ID for this request (for tracing)"
```

## 3. Code Examples

### Making a raw HTTP/1.0 request manually

```python
import socket

def raw_http_get(host: str, path: str, port: int = 80) -> str:
    """
    Make an HTTP/1.0 GET request using only a raw TCP socket.
    This is exactly what every HTTP library does under the hood.
    """
    sock = socket.socket(socket.AF_INET, socket.SOCK_STREAM)
    sock.settimeout(10)
    sock.connect((host, port))

    # HTTP/1.0 request (bare minimum)
    request = (
        f"GET {path} HTTP/1.0\r\n"
        f"Host: {host}\r\n"
        f"\r\n"  # blank line = end of headers
    )
    sock.send(request.encode())

    # Read entire response (connection closes when done)
    response = b""
    while chunk := sock.recv(4096):
        response += chunk

    sock.close()

    # Split into status line + headers + body
    raw = response.decode("utf-8", errors="replace")
    header_section, _, body = raw.partition("\r\n\r\n")
    headers = header_section.split("\r\n")
    status_line = headers[0]

    print(f"Status: {status_line}")
    print(f"Body: {body[:200]}")
    return body

raw_http_get("httpbin.org", "/get")
# Status: HTTP/1.1 200 OK   (httpbin uses 1.1 but the request is 1.0-style)
# Body: {"args": {}, "headers": {...}, "origin": "...", "url": "..."}
```

### Parsing HTTP responses

```python
from http.client import HTTPResponse
from io import BytesIO

def parse_http_response(raw_bytes: bytes) -> dict:
    """Parse a raw HTTP response (useful for understanding the format)."""

    class FakeSocket:
        def __init__(self, data): self._data = data
        def makefile(self, *args, **kwargs): return BytesIO(self._data)

    response = HTTPResponse(FakeSocket(raw_bytes))
    response.begin()

    return {
        "status": response.status,
        "reason": response.reason,
        "headers": dict(response.getheaders()),
        "body": response.read().decode("utf-8", errors="replace"),
    }

# Test it
raw = b"HTTP/1.1 200 OK\r\nContent-Type: text/plain\r\nContent-Length: 5\r\n\r\nHello"
parsed = parse_http_response(raw)
print(parsed)
# {'status': 200, 'reason': 'OK', 'headers': {'Content-Type': 'text/plain', ...}, 'body': 'Hello'}
```

## 4. Senior Engineer Perspective

**What a junior gets wrong:**

Treating HTTP as a magic box handled by the framework. Understanding that HTTP is text means you can always fall back to `curl -v` or `telnet` to debug any HTTP issue. The framework hides nothing that a text editor couldn't reveal.

**The one thing that matters:**

The `Content-Type` header. If your server returns `Content-Type: application/json` but actually sends HTML (an error page), every client will fail to parse it and produce cryptic errors. Always match `Content-Type` to the actual body format. Always.
