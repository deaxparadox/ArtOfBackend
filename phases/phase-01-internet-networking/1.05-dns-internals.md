## 1. What Is It?

DNS is not one server — it is a distributed, hierarchical system of millions of servers organized into three roles: **recursive resolvers**, **root nameservers**, and **authoritative nameservers**.

Understanding this hierarchy explains why DNS is resilient, why changes take time to propagate, and why different users might see different DNS answers at the same time.

---

## 2. The Three Roles

### Recursive Resolver (The Workhorse)

The recursive resolver is the DNS server your OS queries first. It does the full work of tracking down an answer by querying the hierarchy.

```
Who is a recursive resolver?
  Your ISP's DNS server (assigned via DHCP)
  Google: 8.8.8.8 and 8.8.4.4
  Cloudflare: 1.1.1.1 and 1.0.0.1
  OpenDNS: 208.67.222.222
  AWS VPC DNS: 169.254.169.253
  Kubernetes CoreDNS: 10.96.0.10 (typical cluster IP)
```

The recursive resolver:
1. Receives a query from your OS
2. Checks its cache → if hit, returns immediately
3. If miss → walks the DNS hierarchy to find the answer
4. Caches the result for TTL seconds
5. Returns the answer

### Root Nameservers (The Starting Point)

There are exactly **13 root nameserver addresses** (A through M), operated by 12 different organizations. Despite being 13 IP addresses, there are actually ~1,500 physical servers worldwide using **anycast routing** — packets go to the nearest instance.

```
Root nameservers:
  a.root-servers.net  → Verisign (USA)
  b.root-servers.net  → USC-ISI (USA)
  c.root-servers.net  → Cogent (USA)
  d.root-servers.net  → University of Maryland (USA)
  e.root-servers.net  → NASA (USA)
  f.root-servers.net  → Internet Systems Consortium
  g.root-servers.net  → US DoD
  h.root-servers.net  → US Army Research Lab
  i.root-servers.net  → Netnod (Sweden)
  j.root-servers.net  → Verisign
  k.root-servers.net  → RIPE NCC (Netherlands)
  l.root-servers.net  → ICANN
  m.root-servers.net  → WIDE Project (Japan)
```

Root servers don't know the IP of `api.example.com`. They know which TLD servers are authoritative for `.com`, `.net`, `.org`, `.io`, etc. They respond to queries with a referral: "I don't know the answer, but ask this TLD server."

### TLD Nameservers

TLD (Top-Level Domain) servers are authoritative for a specific TLD. Verisign operates the `.com` and `.net` TLD servers. ICANN operates `.org`.

TLD servers don't know `api.example.com`'s IP either. They know which nameserver is authoritative for `example.com`. They respond: "Ask ns1.cloudflare.com."

### Authoritative Nameservers (The Source of Truth)

Authoritative nameservers hold the actual DNS records for a domain. When you buy a domain and configure DNS records in Route 53, Cloudflare, or GoDaddy, those are your authoritative nameservers.

Only authoritative nameservers can answer queries definitively. All other servers are just caching or referring.

## 3. The Full Resolution Walk (Cold Cache)

```
Browser → api.example.com

STEP 1: OS stub resolver → recursive resolver (8.8.8.8)
  "What is the IP for api.example.com?"

STEP 2: Recursive resolver → root server (a.root-servers.net)
  "What is the IP for api.example.com?"
  Root responds: "I don't know. For .com, ask:
    a.gtld-servers.net (192.5.6.30)
    b.gtld-servers.net (192.33.14.30)
    ..."

STEP 3: Recursive resolver → .com TLD server (a.gtld-servers.net)
  "What is the IP for api.example.com?"
  TLD responds: "I don't know. For example.com, ask:
    ns1.cloudflare.com (173.245.58.51)
    ns2.cloudflare.com (173.245.59.51)"

STEP 4: Recursive resolver → authoritative server (ns1.cloudflare.com)
  "What is the IP for api.example.com?"
  Authoritative responds: "api.example.com. 300 IN A 54.23.167.89"
                          ← This is the definitive answer.

STEP 5: Recursive resolver → caches result for 300 seconds → returns to OS
STEP 6: OS → browser: 54.23.167.89
STEP 7: Browser opens TCP connection to 54.23.167.89:443

Total time for a cold DNS resolution: 50–150ms (4 round trips)
Warm (cached at resolver): < 1ms
```

## 4. DNS Caching at Every Level

DNS results are cached at multiple levels simultaneously:

```
LEVEL                CACHE SIZE      INVALIDATION
────────────────────────────────────────────────────
Browser              per-tab         TTL or tab close
OS resolver          OS-managed      TTL or flush
Recursive resolver   large           TTL
Application code     manual          your logic
```

**This is why DNS changes take time to propagate.** Each level caches independently, and you can't force external resolvers to flush their cache. All you can do is:
1. Lower TTL before the change
2. Wait for the old TTL to expire everywhere
3. Make the change
4. Accept that the longest-cached copy expires when its TTL runs out

## 5. DNSSEC — Securing DNS

Standard DNS has no authentication. An attacker on the network between your client and the resolver can inject forged DNS responses (DNS spoofing / cache poisoning). DNSSEC adds cryptographic signatures to DNS records, allowing resolvers to verify authenticity.

```
WITHOUT DNSSEC:
  Query: api.example.com?
  Attacker intercepts, injects: api.example.com → evil-server-ip
  Browser connects to attacker's server.
  SSL certificate check fails (no valid cert for api.example.com).
  → But many users click through cert warnings.

WITH DNSSEC:
  Response includes a digital signature.
  Resolver verifies signature using public key from parent zone.
  Chain of trust: root → .com TLD → example.com
  Forged responses fail signature verification.
```

DNSSEC adoption is ~30–40% of top domains (2024). It's mandatory for .gov domains in the US. Enable it if your DNS provider supports it.

## 6. Modern DNS: DoH and DoT

Traditional DNS sends queries in plaintext over UDP port 53. An ISP, network admin, or attacker on the network can see every domain you resolve.

```
DNS over HTTPS (DoH):
  Sends DNS queries inside HTTPS (port 443).
  Indistinguishable from regular HTTPS traffic.
  Supported by: Firefox, Chrome, 1.1.1.1, 8.8.8.8
  Privacy: ISP cannot see your DNS queries.
  Downside: Bypasses corporate DNS filtering (a headache for IT).

DNS over TLS (DoT):
  Sends DNS queries over a dedicated TLS channel (port 853).
  More easily blocked/monitored by network admins.
  Supported by: 1.1.1.1, 8.8.8.8, Android 9+

For backend services: usually irrelevant (VPC DNS is internal).
For user privacy: increasingly important.
```

## 7. Code Examples

### Tracing the full DNS resolution path

```bash
# See every step of the recursive resolution
dig +trace api.example.com

# Output shows:
# . 518400 IN NS a.root-servers.net.        ← root servers
# com. 172800 IN NS a.gtld-servers.net.     ← .com TLD servers
# example.com. 172800 IN NS ns1.cloudflare.com. ← authoritative
# api.example.com. 300 IN A 54.23.167.89   ← final answer
```

### Checking authoritative vs. cached responses

```bash
# Query the authoritative nameserver directly (bypasses resolver cache)
# First find the authoritative server:
dig NS example.com +short
# ns1.cloudflare.com.
# ns2.cloudflare.com.

# Then query it directly:
dig @ns1.cloudflare.com api.example.com A
# This bypasses ALL caching — you get the true current value.
# Use this to verify a DNS change was made correctly.

# Compare with cached resolver answer:
dig @8.8.8.8 api.example.com A
# May differ if TTL hasn't expired yet after a recent change.
```

### Python: check if a DNS change has propagated

```python
import dns.resolver

def check_propagation(domain: str, expected_ip: str) -> dict:
    """
    Check DNS propagation across multiple resolvers.
    Use after making a DNS change.
    """
    resolvers = {
        "Cloudflare":  "1.1.1.1",
        "Google":      "8.8.8.8",
        "OpenDNS":     "208.67.222.222",
        "Quad9":       "9.9.9.9",
    }
    results = {}
    for name, server in resolvers.items():
        r = dns.resolver.Resolver(configure=False)
        r.nameservers = [server]
        r.timeout = 3
        try:
            answers = r.resolve(domain, "A")
            current = [str(a) for a in answers]
            results[name] = {
                "ips": current,
                "propagated": expected_ip in current,
            }
        except Exception as e:
            results[name] = {"error": str(e), "propagated": False}
    return results
```

## 8. Senior Engineer Perspective

**What a junior gets wrong:**

Thinking "DNS changed = DNS updated everywhere." DNS propagation is not instant. The concept of "DNS propagation" is often misunderstood — you're not waiting for a wave to wash across the internet. You're waiting for cached answers (at different TTLs on different resolvers) to expire. Different clients in different locations see different answers during the transition window.

**The one thing that matters most:**

Every production service should document its DNS TTL settings and have a documented pre-migration TTL lowering procedure. This is basic operational hygiene that prevents "why is half our traffic still going to the old server?" incidents.
