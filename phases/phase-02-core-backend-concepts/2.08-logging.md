## 1. What Is It?

Logging is the practice of recording a timestamped record of what your application did. Logs are the primary tool for debugging production issues, auditing security events, and understanding system behavior over time.

The shift from old-style logging to modern structured logging is the most important change in production logging practices of the last decade.

---

## 2. Log Levels

```
DEBUG   → Fine-grained diagnostic info. Verbose. Dev and staging only.
          "Executing query: SELECT * FROM users WHERE id=123"

INFO    → Significant events in normal operation.
          "User 456 logged in", "Order 789 created", "Payment processed"

WARNING → Something unexpected happened but the operation succeeded.
          "Cache miss — falling back to database"
          "API rate limit at 80% capacity"
          "Retry attempt 2 of 3 for external API"

ERROR   → An operation failed. Requires attention.
          "Failed to process payment for order 789"
          "Database connection failed — using fallback"

CRITICAL → System is in a dangerous state. Immediate action required.
           "Cannot connect to database — all requests failing"
           "Disk at 99% capacity"

PRODUCTION LEVEL GUIDANCE:
  LOG_LEVEL=debug   → development only (too noisy for production)
  LOG_LEVEL=info    → production default (significant events)
  LOG_LEVEL=warning → high-traffic production (reduce volume)
  LOG_LEVEL=error   → minimal logging (only failures)
```

## 3. Structured Logging vs. Plain Text

**The problem with plain text logs:**

```
# Plain text — unstructured, hard to query
2024-06-15 14:30:00 - User 123 logged in from 192.168.1.1
2024-06-15 14:30:01 - POST /orders took 234ms
2024-06-15 14:30:02 - Payment failed: insufficient funds for order 456
```

To find all orders that took > 200ms in the last hour, you need regex and grep. You can't aggregate by field. You can't query in your log platform.

**Structured logging — JSON, queryable:**

```json
{"timestamp": "2024-06-15T14:30:00Z", "level": "info", "event": "user_login", "user_id": 123, "ip": "192.168.1.1", "request_id": "abc123"}
{"timestamp": "2024-06-15T14:30:01Z", "level": "info", "event": "request_completed", "method": "POST", "path": "/orders", "duration_ms": 234, "status": 201, "request_id": "def456"}
{"timestamp": "2024-06-15T14:30:02Z", "level": "error", "event": "payment_failed", "order_id": 456, "reason": "insufficient_funds", "request_id": "ghi789"}
```

Now in your log platform (Elasticsearch, Loki, Datadog):
```
event="request_completed" AND duration_ms > 200 AND timestamp > now-1h
→ Returns all slow requests in the last hour
```

## 4. Structured Logging in Python

```python
import logging
import json
import sys
from datetime import datetime, timezone

class JSONFormatter(logging.Formatter):
    """Format log records as JSON for structured logging."""

    def format(self, record: logging.LogRecord) -> str:
        log_data = {
            "timestamp": datetime.now(timezone.utc).isoformat(),
            "level": record.levelname.lower(),
            "logger": record.name,
            "message": record.getMessage(),
        }

        # Include extra fields added via the extra= parameter
        for key, value in record.__dict__.items():
            if key not in logging.LogRecord.__dict__ and not key.startswith("_"):
                if key not in ("args", "msg", "levelname", "levelno", "pathname",
                               "filename", "module", "funcName", "lineno",
                               "created", "msecs", "relativeCreated", "thread",
                               "threadName", "processName", "process", "name",
                               "message"):
                    log_data[key] = value

        if record.exc_info:
            log_data["exception"] = self.formatException(record.exc_info)

        return json.dumps(log_data, default=str)

def get_logger(name: str) -> logging.Logger:
    logger = logging.getLogger(name)
    if not logger.handlers:
        handler = logging.StreamHandler(sys.stdout)
        handler.setFormatter(JSONFormatter())
        logger.addHandler(handler)
        logger.setLevel(logging.INFO)
    return logger

# Usage
logger = get_logger(__name__)

# Log with structured context
logger.info(
    "payment_processed",
    extra={
        "request_id": "abc123",
        "order_id": 789,
        "amount_cents": 5000,
        "currency": "usd",
        "user_id": 42,
        "duration_ms": 234,
    }
)
# Output:
# {"timestamp": "2024-06-15T14:30:00Z", "level": "info", "message": "payment_processed",
#  "request_id": "abc123", "order_id": 789, "amount_cents": 5000, ...}
```

### Using structlog (the best Python structured logging library)

```python
import structlog

structlog.configure(
    processors=[
        structlog.contextvars.merge_contextvars,
        structlog.processors.add_log_level,
        structlog.processors.StackInfoRenderer(),
        structlog.dev.set_exc_info,
        structlog.processors.TimeStamper(fmt="iso", utc=True),
        structlog.processors.JSONRenderer(),  # production
        # structlog.dev.ConsoleRenderer(),  # development (colored, readable)
    ],
    wrapper_class=structlog.make_filtering_bound_logger(logging.INFO),
    context_class=dict,
    logger_factory=structlog.PrintLoggerFactory(),
)

log = structlog.get_logger()

# Bind context that applies to all log calls in this request
structlog.contextvars.bind_contextvars(
    request_id="abc123",
    user_id=42,
)

log.info("payment_processed", order_id=789, amount_cents=5000)
# All entries automatically include request_id and user_id
```

## 5. What to Log

```
✓ ALWAYS LOG:
  Every request: method, path, status code, duration_ms, request_id
  Authentication events: login (success + fail), logout, token refresh
  Authorization failures: who tried to access what they shouldn't
  Business events: order created, payment processed, account deleted
  External API calls: service, method, status, duration
  Errors and exceptions: full stack trace, request context
  Significant state changes: user role changed, config updated
  Background job start/complete/fail: job type, ID, duration

✓ LOG WHEN DEBUGGING (DEBUG level or temporary):
  Query execution: SQL, params, duration
  Cache operations: key, hit/miss
  Detailed request/response bodies (during incident investigation)
```

## 6. What NOT to Log

```
✗ NEVER LOG:
  Passwords (obvious)
  Full credit card numbers (PCI DSS violation)
  Social Security Numbers (PII regulation)
  Auth tokens, API keys, session IDs (attackers can use these from logs)
  Encryption keys or secrets
  Full request body if it contains sensitive fields
  PII beyond what's needed (full addresses, medical info)
  
  SPECIFICALLY:
    ✗ logger.info(f"User logged in with password {password}")
    ✗ logger.debug(f"Auth header: {request.headers['authorization']}")
    ✗ logger.info(f"Connected to {DATABASE_URL}")  ← contains password
    ✗ logger.info(f"Request body: {await request.json()}")  ← may have CC numbers
  
  INSTEAD:
    ✓ logger.info("user_login", user_id=user.id, ip=ip_address)
    ✓ logger.info("auth_token_validated", token_type="bearer", user_id=user.id)
    ✓ logger.info("db_connected", host=db_host, database=db_name)  ← no credentials
    ✓ logger.info("request_received", path=path, content_length=length)
```

## 7. Request-Level Log Context

Every log entry from within a request should carry the request ID. Use middleware to bind it:

```python
from starlette.middleware.base import BaseHTTPMiddleware
import structlog

class LoggingMiddleware(BaseHTTPMiddleware):
    async def dispatch(self, request, call_next):
        request_id = request.headers.get("x-request-id", str(uuid.uuid4()))

        # Bind request context to all logs in this request's scope
        with structlog.contextvars.bound_contextvars(
            request_id=request_id,
            method=request.method,
            path=request.url.path,
            client_ip=request.headers.get("x-forwarded-for", ""),
        ):
            start = time.perf_counter()
            response = await call_next(request)
            duration_ms = (time.perf_counter() - start) * 1000

            log.info(
                "request_completed",
                status=response.status_code,
                duration_ms=round(duration_ms, 2),
            )
            response.headers["x-request-id"] = request_id
            return response
```

## 8. Senior Engineer Perspective

**What a junior gets wrong:**

Logging strings instead of structured data. `logger.info(f"User {user_id} created order {order_id}")` is queryable by regex but not by field. In production, you want to query "all orders created by users with failed payment in the last hour" — this requires `user_id` and `order_id` as queryable fields, not embedded in a string.

**The log volume trap:**

Excessive logging at DEBUG level in production creates three problems: high storage costs, noisy signal (hard to find real issues), and performance overhead (logging to stdout is not free under load). Log only what you'll actually use for debugging or monitoring. If you've never searched for a log message in 6 months, remove it.

**The one log every endpoint must emit:**

```
request_completed | request_id | method | path | status | duration_ms | user_id
```

This single log line, for every request, enables: latency monitoring, error rate tracking, per-user request history, and SLO calculations — from one structured field.
