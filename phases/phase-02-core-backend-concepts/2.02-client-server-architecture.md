## 1. What Is It?

Client-server architecture is the dominant model for networked applications. A **server** is a process that waits for and responds to requests. A **client** is any process that makes requests to a server.

What seems obvious becomes nuanced when you realize:
- The same service can be both client and server simultaneously
- "The client" is not always a browser — it can be a mobile app, a CLI, another server, or a cron job
- The server's responsibilities are clearly defined and should not leak into the client

---

## 2. What the Server Is Responsible For

```
SERVER RESPONSIBILITIES:
  ✓ Persistent data storage and integrity
  ✓ Business rules and validation (the authoritative source)
  ✓ Authentication and authorization
  ✓ Security enforcement
  ✓ Consistency across multiple clients
  ✓ Rate limiting and abuse prevention

CLIENT RESPONSIBILITIES:
  ✓ Presentation and UI
  ✓ User input collection
  ✓ Caching for performance (with server as truth)
  ✓ Offline capability (with sync on reconnect)

NEVER delegate server responsibilities to the client:
  ✗ "Validate on the frontend, skip server-side validation"
     → Every API call can be made with curl. Frontend doesn't protect the backend.
  
  ✗ "The frontend hides the Delete button, so we don't need to check permissions"
     → Anyone can call DELETE /users/123 directly.
  
  ✗ "Store the price in the cart on the frontend, send it to the server"
     → Client can send any price. Always recalculate on the server.
```

## 3. Architectural Variants

```
2-TIER (Client ↔ Server):
  Browser → Web Server + DB
  Simple, fine for small applications.
  Problem: hard to scale, frontend directly coupled to DB logic.

3-TIER (Client ↔ Application Server ↔ Database):
  Browser → API Server → Database
  The standard for most web applications.
  Separation of concerns: presentation / logic / data.

N-TIER (Microservices):
  Browser → API Gateway → Service A → Service B → Database A
                                    → Service C → Database B
  Maximum flexibility, maximum complexity.
  See Phase 14 for microservices deep dive.

PEER-TO-PEER (P2P):
  Every node is both client and server.
  No central authority.
  Examples: BitTorrent, WebRTC, Blockchain nodes.
  Backend engineers rarely build pure P2P but often build
  signaling servers that coordinate P2P connections.
```

## 4. The Server's Three Jobs (Recap from 0.2)

```python
# Every server endpoint does exactly three things:

@app.post("/orders")
async def create_order(data: OrderRequest, user = Depends(auth)):

    # JOB 1: Validate input
    if data.quantity <= 0:
        raise HTTPException(422, "Quantity must be positive")
    if not await user_can_order(user.id):
        raise HTTPException(403, "Account not eligible to order")

    # JOB 2: Execute business logic against state
    async with db.transaction():
        product = await db.get_product(data.product_id, for_update=True)
        if product.stock < data.quantity:
            raise HTTPException(409, "Insufficient stock")
        product.stock -= data.quantity
        order = await db.create_order(user.id, data)

    # JOB 3: Return response + trigger async side effects
    await email_queue.enqueue({"type": "order_confirmation", "order_id": order.id})
    return order
```

## 5. Senior Engineer Perspective

**What a junior gets wrong:**

Business logic in the client. "The app checks if the user has enough balance before letting them submit" — the server must also check. Clients are untrusted. Always.

**What a senior designs around:**

The server as the single source of truth. Clients can display cached or predicted state, but the server always validates and the server's verdict is final. This is what "server-authoritative" means in gaming (the server decides if your move is valid, not the client).
