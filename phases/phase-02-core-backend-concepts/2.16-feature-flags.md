## 1. What Is It?

A feature flag (also called feature toggle or feature switch) is a conditional in your code that controls whether a feature is active for a given request, user, or environment — without deploying new code.

**What this enables:**

- **Deploy code without activating features** — the new code is in production, disabled
- **Gradual rollout** — enable for 1% of users, then 10%, then 100%
- **A/B testing** — 50% see version A, 50% see version B
- **Kill switch** — instantly disable a bad feature without a deployment
- **Beta access** — enable for specific users or groups only

---

## 2. Types of Feature Flags

```
RELEASE FLAGS:
  Controls whether a new feature is live.
  Typically short-lived — removed once feature is fully rolled out.
  Example: new_checkout_flow = true/false

EXPERIMENT FLAGS:
  A/B tests. Split traffic to compare outcomes.
  Example: checkout_cta_text = "Buy Now" vs "Complete Purchase"
  Requires analytics to measure impact.

OPS FLAGS (kill switches):
  Emergency disablement of a feature.
  Example: disable_heavy_analytics = true  (during high load)
  
PERMISSION FLAGS:
  Enable features for specific users/groups.
  Example: early_access_users = [user_id_list]
           beta_features_enabled = true (for premium accounts)
```

## 3. Simple Redis-Backed Feature Flags

```python
import redis.asyncio as aioredis
import json
from typing import Any

redis = aioredis.from_url(settings.redis_url)

class FeatureFlags:
    def __init__(self, redis_client):
        self.redis = redis_client

    async def is_enabled(self, flag: str, user_id: int | None = None) -> bool:
        """Check if a feature flag is enabled."""
        data = await self.redis.get(f"feature:{flag}")
        if not data:
            return False  # default: disabled

        config = json.loads(data)

        # Boolean flag
        if isinstance(config, bool):
            return config

        # Percentage rollout
        if "percentage" in config:
            if user_id is None:
                return False
            # Deterministic: same user always gets same result
            import hashlib
            user_hash = int(hashlib.md5(f"{flag}:{user_id}".encode()).hexdigest(), 16)
            return (user_hash % 100) < config["percentage"]

        # User list
        if "users" in config:
            return user_id in config["users"]

        return False

flags = FeatureFlags(redis)

# Usage in route handler
@app.post("/checkout")
async def checkout(data: CheckoutData, user = Depends(get_current_user)):
    if await flags.is_enabled("new_checkout_flow", user.id):
        return await new_checkout_service.process(data)
    else:
        return await legacy_checkout_service.process(data)
```

## 5. Feature Flag Anti-Patterns

```
✗ LONG-LIVED FLAGS:
  Flags that stay in code for years accumulate "flag debt".
  Code becomes: if flag_A and not flag_B or (flag_C and user.premium):
  Unmaintainable. Set an expiry date for release flags.

✗ FLAGS FOR CONFIGURATION:
  Using flags for values that should be environment variables.
  feature_flag("max_upload_size") → use an env var instead.
  Flags are for binary on/off or user targeting, not configuration values.

✗ TESTING ONLY THE "FLAG ON" PATH:
  Both the enabled and disabled code paths must be tested.
  Otherwise: removing the flag breaks the app for all users who were in the "off" path.

✓ BEST PRACTICE: Flag lifecycle
  1. Create flag (disabled in production)
  2. Deploy code with flag (both paths tested)
  3. Enable for 1% → 10% → 50% → 100%
  4. When at 100% and stable: remove the flag and dead code
  5. Deploy cleanup (no flag logic needed)
```

---

## 6. Senior Engineer Perspective

**What a junior gets wrong:**

Not cleaning up flags. A codebase with 50 stale flags is unmaintainable — you never know which code paths are actually reachable. Treat flags like debt: minimize them. Every flag should have an owner and an expected removal date.

**What a senior uses flags for:**

Risky database migrations. New code that uses a new schema column is deployed behind a flag. The migration runs. Data is backfilled. Flag enabled for 1% — verify it works. Gradually increase. If a bug is found: instantly flip the flag off without a deployment. When fully rolled out: remove flag and old code path in the next deploy.
