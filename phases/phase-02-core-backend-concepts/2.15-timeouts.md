## 1. The Most Dangerous Default in Backend Engineering

`requests.get("https://external-api.com/data")` — no timeout.

Python's `requests` library has a default timeout of **None** (wait forever). Most HTTP clients, database drivers, and socket libraries share this dangerous default.

One hung external API call, and a thread waits forever. With a thread pool of 20 threads, 20 concurrent hung calls exhaust the pool. New requests can't get a thread. Your service appears down — because of one unresponsive third-party API.

**The rule:**

> **Every network call must have an explicit timeout. No exceptions.**

## 2. Types of Timeouts

```
CONNECTION TIMEOUT:
  How long to wait for the TCP connection to be established.
  If the server is down, this determines how long you wait to learn that.
  Typical value: 2-5 seconds.
  
  What it covers:
    - DNS resolution
    - TCP 3-way handshake
    - TLS handshake

READ TIMEOUT (response timeout):
  How long to wait for the server to send response data.
  Starts after the connection is established.
  If the server accepts the connection but hangs processing:
  this timeout fires.
  Typical value: 5-30 seconds (depends on expected operation time).

WRITE TIMEOUT:
  How long to wait to finish sending the request body.
  Relevant for large file uploads.
  Typical value: same as read or longer for uploads.

POOL TIMEOUT:
  How long to wait for an available connection from the connection pool.
  If all connections are busy, requests queue here.
  Typical value: 1-5 seconds.

TOTAL / DEADLINE TIMEOUT:
  Maximum total time for the entire operation.
  Useful as a hard ceiling regardless of other timeout settings.
```

## 3. Configuring Timeouts Correctly

```python
import httpx

# WRONG: no timeout — will hang forever
response = requests.get("https://external-api.com/data")

# WRONG: timeout as a single value — Python requests treats this as read timeout
# (no connection timeout)
response = requests.get("https://external-api.com/data", timeout=5)

# RIGHT: explicit timeout for each phase (httpx)
timeout = httpx.Timeout(
    connect=3.0,    # TCP + TLS must complete in 3s
    read=10.0,      # server must respond in 10s
    write=5.0,      # request body must send in 5s
    pool=2.0,       # wait max 2s for a connection from pool
)
async with httpx.AsyncClient(timeout=timeout) as client:
    response = await client.get("https://external-api.com/data")

# RIGHT: requests library (connection, read)
response = requests.get(
    "https://external-api.com/data",
    timeout=(3.0, 10.0)  # (connect_timeout, read_timeout)
)
```

```python
# Database timeouts
import asyncpg

pool = await asyncpg.create_pool(
    dsn=DATABASE_URL,
    command_timeout=30,       # individual query timeout
    max_inactive_connection_lifetime=60,
    timeout=10,               # wait max 10s to acquire a connection from pool
)

# Redis timeouts
import redis.asyncio as aioredis

r = aioredis.from_url(
    REDIS_URL,
    socket_connect_timeout=2,   # TCP connect timeout
    socket_timeout=5,           # read/write timeout per operation
    retry_on_timeout=True,
)
```

## 4. Setting the Right Timeout Value

```
FACTORS TO CONSIDER:
  1. Your SLO: if your API must respond in 200ms, all external calls must be < 200ms
  2. Normal operation time: what does this call take in the p99 case?
  3. User experience: how long is acceptable to wait?
  4. Retry budget: if you retry 3 times, total = 3 × timeout

TIMEOUT GUIDELINES:
  Same-region HTTP service:    connect: 1s,  read: 5s
  Cross-region HTTP service:   connect: 3s,  read: 15s
  Third-party external API:    connect: 3s,  read: 10s  (often set lower for SLO)
  Database query (simple):     pool: 2s,     command: 5s
  Database query (analytics):  pool: 5s,     command: 30-120s
  Redis operation:             connect: 1s,  read: 1s   (should be fast)
  File upload:                 connect: 5s,  write: 60s (large bodies)

CONTEXT MATTERS:
  Real-time user-facing endpoint:  aggressive timeouts (2-5s total)
  Background batch job:            relaxed timeouts (minutes)
  Analytics query:                 very relaxed (minutes to hours)
```

## 5. Propagating Deadlines (Context-Based Timeouts)

In a microservices chain, each service should know about the overall deadline:

```python
# Order service: user's request has a 5-second SLO
@app.post("/orders")
async def create_order(request: Request, data: OrderRequest):
    # Set a deadline: this entire handler must complete in 4.5s
    # (leaving 0.5s buffer for the framework)
    deadline = 4.5

    # Pass remaining deadline to each downstream call
    async with asyncio.timeout(deadline):  # Python 3.11+
        inventory_task = asyncio.create_task(
            check_inventory(data.product_id, timeout=1.5)
        )
        payment_task = asyncio.create_task(
            process_payment(data.amount, timeout=2.0)
        )
        inventory, payment = await asyncio.gather(inventory_task, payment_task)

    return create_order_record(inventory, payment)
```

In Go/gRPC contexts, this is called **context propagation** — each function receives a context with a deadline, and any I/O using that context inherits the deadline automatically.

## 6. Senior Engineer Perspective

**What a junior gets wrong:**

Setting the same timeout for every operation. A Redis cache lookup should time out in 500ms — if Redis is slow, fall back to the database, don't wait 30 seconds. A database analytics query legitimately takes 2 minutes. One timeout value for everything is always wrong for at least one use case.

**The timeout audit every senior runs before shipping:**

```bash
# Find all HTTP calls without explicit timeouts
grep -rn "requests.get\|requests.post\|httpx.get" --include="*.py" | \
  grep -v "timeout"
# Any hits need timeouts added

# Find database connections without command_timeout
grep -rn "create_pool\|connect(" --include="*.py" | \
  grep -v "timeout\|command_timeout"
```

**The production incident pattern:**

"Everything was fine, then External-API-X started timing out at 2am. Our service stopped responding within 3 minutes."

Translation: External-API-X has no timeout in your code. At 2am, all workers got stuck waiting for it. Fixed with: `timeout=(3, 10)` on that one HTTP call.
