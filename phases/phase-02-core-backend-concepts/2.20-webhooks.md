## 1. What Are Webhooks?

Webhooks are HTTP callbacks — instead of your application polling an external service for updates ("has anything changed?"), the external service calls your application when something happens ("something changed, here's what").

**Push vs. Pull:**

```
POLLING (pull — inefficient):
  Your app → GET /payment/status every 5 seconds → 12 requests/minute
  Most requests: "no change" (wasted)
  Latency: up to 5 seconds to detect a change
  
WEBHOOKS (push — efficient):
  Payment event occurs → provider → POST /your-webhook (immediate)
  Zero wasted requests
  Latency: seconds (as fast as the provider sends it)
```

**Real examples:**
- Stripe: sends a webhook when a payment succeeds, fails, or is disputed
- GitHub: sends webhooks on push, PR, issue events
- Twilio: sends webhook when SMS is delivered or replied to
- Shopify: sends webhooks on order creation, fulfillment, cancellation

## 2. Receiving Webhooks

### The Basic Endpoint

```python
from fastapi import FastAPI, Request, HTTPException, Header
import hashlib, hmac, json

app = FastAPI()

STRIPE_WEBHOOK_SECRET = settings.stripe_webhook_secret

@app.post("/webhooks/stripe")
async def stripe_webhook(
    request: Request,
    stripe_signature: str = Header(alias="stripe-signature"),
):
    # STEP 1: Read raw body (must be raw bytes for signature verification)
    body = await request.body()

    # STEP 2: Verify signature (CRITICAL — see next section)
    if not verify_stripe_signature(body, stripe_signature, STRIPE_WEBHOOK_SECRET):
        raise HTTPException(status_code=400, detail="Invalid signature")

    # STEP 3: Parse the event
    event = json.loads(body)
    event_type = event["type"]
    event_id = event["id"]

    # STEP 4: Acknowledge immediately (return 200 before processing)
    # Stripe resends if no 200 within 30 seconds.
    # Heavy processing must be async — respond first, process after.
    await job_queue.enqueue("process_stripe_event", {
        "event_id": event_id,
        "event_type": event_type,
        "data": event["data"],
    })

    return {"received": True}
```

### Signature Verification — Never Skip This

Without signature verification, anyone on the internet can POST fake webhook events to your endpoint. A fake "payment_succeeded" event could credit a user's account without any actual payment.

```python
def verify_stripe_signature(
    payload: bytes,
    signature_header: str,
    secret: str,
) -> bool:
    """
    Stripe signs webhooks with HMAC-SHA256.
    Format: t=timestamp,v1=signature
    """
    try:
        # Parse the signature header
        parts = {k: v for k, v in (p.split("=", 1) for p in signature_header.split(","))}
        timestamp = parts.get("t")
        received_sig = parts.get("v1")

        if not timestamp or not received_sig:
            return False

        # Prevent replay attacks: reject events older than 5 minutes
        import time
        if abs(time.time() - int(timestamp)) > 300:
            return False

        # Compute expected signature
        signed_payload = f"{timestamp}.{payload.decode()}"
        expected_sig = hmac.new(
            secret.encode(),
            signed_payload.encode(),
            hashlib.sha256,
        ).hexdigest()

        # Compare using constant-time comparison (prevents timing attacks)
        return hmac.compare_digest(expected_sig, received_sig)

    except Exception:
        return False

# GitHub webhooks use X-Hub-Signature-256
def verify_github_signature(payload: bytes, signature_header: str, secret: str) -> bool:
    expected = "sha256=" + hmac.new(
        secret.encode(), payload, hashlib.sha256
    ).hexdigest()
    return hmac.compare_digest(expected, signature_header)
```

## 3. Idempotency for Webhook Processors

Webhook providers deliver "at least once" — they may deliver the same event multiple times (on retry, after timeout). Your processor MUST be idempotent.

```python
async def process_stripe_event(event_id: str, event_type: str, data: dict):
    """Process Stripe webhook event — must be idempotent."""

    # Check if already processed
    already_processed = await db.fetchval(
        "SELECT 1 FROM processed_webhook_events WHERE event_id = $1",
        event_id
    )
    if already_processed:
        logger.info("webhook_already_processed", event_id=event_id)
        return  # Skip — already handled

    # Process the event
    if event_type == "payment_intent.succeeded":
        order_id = data["object"]["metadata"]["order_id"]
        await db.execute(
            "UPDATE orders SET status='paid', paid_at=NOW() WHERE id=$1",
            order_id
        )

    elif event_type == "payment_intent.payment_failed":
        order_id = data["object"]["metadata"]["order_id"]
        await db.execute(
            "UPDATE orders SET status='payment_failed' WHERE id=$1",
            order_id
        )

    # Mark as processed AFTER successful processing
    # INSERT ... ON CONFLICT DO NOTHING handles concurrent duplicate processing
    await db.execute(
        "INSERT INTO processed_webhook_events (event_id, processed_at) "
        "VALUES ($1, NOW()) ON CONFLICT (event_id) DO NOTHING",
        event_id
    )

    logger.info("webhook_processed", event_id=event_id, event_type=event_type)
```

## 4. Sending Webhooks (Being the Webhook Provider)

When YOUR service sends webhooks to customers:

```python
import httpx, hashlib, hmac, time, asyncio

async def deliver_webhook(
    endpoint_url: str,
    secret: str,
    event_type: str,
    payload: dict,
    max_retries: int = 5,
):
    """
    Deliver a webhook to a customer endpoint with retries.
    At-least-once delivery guarantee.
    """
    body = json.dumps(payload).encode()
    timestamp = str(int(time.time()))

    # Sign the payload
    sig = hmac.new(
        secret.encode(),
        f"{timestamp}.{body.decode()}".encode(),
        hashlib.sha256,
    ).hexdigest()

    headers = {
        "Content-Type": "application/json",
        "X-Webhook-Signature": f"t={timestamp},v1={sig}",
        "X-Webhook-Event": event_type,
        "X-Delivery-Id": str(uuid.uuid4()),
    }

    for attempt in range(max_retries):
        try:
            async with httpx.AsyncClient(timeout=10.0) as client:
                response = await client.post(
                    endpoint_url, content=body, headers=headers
                )
                if response.status_code < 500:
                    # 2xx = success, 4xx = permanent failure (don't retry)
                    return {"status": "delivered", "status_code": response.status_code}
        except (httpx.TimeoutException, httpx.NetworkError) as e:
            logger.warning("webhook_delivery_failed", attempt=attempt, error=str(e))

        if attempt < max_retries - 1:
            wait = min(2 ** attempt, 3600)  # exponential, max 1 hour
            await asyncio.sleep(wait)

    # All retries failed
    await db.execute(
        "UPDATE webhook_deliveries SET status='failed', last_error=$1 WHERE ...",
        "Max retries exceeded"
    )
    return {"status": "failed"}
```

## 5. Senior Engineer Perspective

**What a junior gets wrong:**

Processing webhook events synchronously in the handler and returning 200 only after processing. If processing takes >30 seconds (Stripe's timeout), Stripe thinks delivery failed and retries. Now you have the same event delivered twice — and if the code isn't idempotent, you process it twice.

**Always:**
1. Verify signature
2. Return 200 immediately (acknowledge receipt)
3. Process asynchronously in a background job
4. Make processing idempotent (use event ID)
5. Store event ID to detect and skip duplicates

**What a senior builds for a webhook system:**

- Webhook delivery log: every attempt, status code, response body, timestamp
- Automatic retry with exponential backoff for 5xx responses
- Manual retry UI for failed deliveries
- Webhook event history for customers to debug
- Alert when a customer endpoint is consistently failing (their server is down)

This is exactly what Stripe, GitHub, and Twilio provide — and it's what you should build when your platform sends webhooks to third parties.
