## 1. What Is It?

Configuration is anything that varies between environments (development, staging, production). Database URLs, API keys, feature flags, port numbers, log levels — these must NOT be hardcoded in your source code.

Environment variables are the standard mechanism: key-value pairs injected into a process from the operating environment. The same codebase can behave differently in dev vs. prod without changing a single line of code.

**The 12-Factor App (Factor III: Config):**
> *"Store config in the environment. An app's config is everything that is likely to vary between deploys (staging, production, developer environments, etc.)."*

## 2. What Belongs in Environment Variables

```
✓ IN ENVIRONMENT VARIABLES:
  DATABASE_URL=postgresql://user:pass@db-host:5432/mydb
  REDIS_URL=redis://redis-host:6379
  SECRET_KEY=supersecretrandomstring
  STRIPE_API_KEY=sk_live_abc123
  AWS_ACCESS_KEY_ID=AKIAIOSFODNN7EXAMPLE
  LOG_LEVEL=info
  PORT=8000
  DEBUG=false
  ALLOWED_ORIGINS=https://app.example.com

✗ HARDCODED IN CODE (wrong):
  db_url = "postgresql://admin:password123@prod-db.example.com:5432/app"
  api_key = "sk_live_abc123"

✗ IN VERSION CONTROL (catastrophic):
  Never commit .env files with real credentials to git.
  This is how most credential leaks happen.
  GitHub scans for secrets — but the damage is done the moment they're pushed.
```

## 3. Loading and Validating Config in Python

### python-dotenv (development)

```python
# .env file (never commit this to git)
DATABASE_URL=postgresql://localhost:5432/mydb_dev
REDIS_URL=redis://localhost:6379
SECRET_KEY=dev-secret-not-for-production
LOG_LEVEL=debug
DEBUG=true

# .env.example (DO commit — shows required variables without values)
DATABASE_URL=
REDIS_URL=
SECRET_KEY=
LOG_LEVEL=info
DEBUG=false
```

```python
# load_dotenv loads .env in development
from dotenv import load_dotenv
load_dotenv()  # loads .env file into os.environ

import os
db_url = os.getenv("DATABASE_URL")  # returns None if not set
```

### Pydantic Settings (production-grade)

The right approach for any serious application: validate configuration at startup, fail fast with clear error messages if required variables are missing.

```python
from pydantic_settings import BaseSettings, SettingsConfigDict
from pydantic import PostgresDsn, RedisDsn, field_validator
from typing import Literal

class Settings(BaseSettings):
    model_config = SettingsConfigDict(
        env_file=".env",           # load from .env in development
        env_file_encoding="utf-8",
        case_sensitive=False,
    )

    # Required (no default = startup fails if missing)
    database_url: PostgresDsn
    secret_key: str

    # Optional with sensible defaults
    redis_url: RedisDsn = "redis://localhost:6379"
    log_level: Literal["debug", "info", "warning", "error"] = "info"
    port: int = 8000
    debug: bool = False
    allowed_origins: list[str] = ["http://localhost:3000"]

    @field_validator("secret_key")
    @classmethod
    def secret_key_must_be_strong(cls, v: str) -> str:
        if len(v) < 32:
            raise ValueError("SECRET_KEY must be at least 32 characters")
        if v == "dev-secret-not-for-production" and not os.getenv("ALLOW_WEAK_SECRET"):
            raise ValueError("Replace default SECRET_KEY in production")
        return v

# Singleton pattern — load once at startup
settings = Settings()

# Usage throughout the app
from config import settings

async def get_db():
    return await asyncpg.connect(str(settings.database_url))
```

**Why validate at startup:** If `DATABASE_URL` is missing, your app should fail immediately with a clear error — not fail cryptically on the first database call 10 minutes after deployment. "Missing required environment variable: DATABASE_URL" is infinitely more debuggable than a NullPointerException deep in SQLAlchemy.

## 4. Configuration in Different Environments

```
DEVELOPMENT:
  .env file loaded by python-dotenv
  Weak credentials, local services (localhost)
  DEBUG=true, LOG_LEVEL=debug

TESTING / CI:
  Environment variables set in CI pipeline (GitHub Actions secrets)
  Test database (separate from dev/prod)
  Mocked external services

STAGING:
  Variables set in deploy config (K8s ConfigMaps/Secrets, ECS task definition)
  Real external services but separate credentials
  Debug logging for easier troubleshooting

PRODUCTION:
  Variables injected by orchestrator (Kubernetes Secrets, AWS Parameter Store)
  Strongest credentials, prod databases
  LOG_LEVEL=info or warning (not debug)
  DEBUG=false always

RULE: Never share credentials across environments.
      Staging should have its own Stripe test key, its own DB, its own Redis.
```

## 5. Node.js Equivalent

```javascript
// Using dotenv + zod for validation (Node.js)
import { z } from 'zod';
import dotenv from 'dotenv';

dotenv.config();

const schema = z.object({
    DATABASE_URL: z.string().url(),
    SECRET_KEY: z.string().min(32),
    PORT: z.coerce.number().default(3000),
    NODE_ENV: z.enum(['development', 'test', 'production']).default('development'),
    LOG_LEVEL: z.enum(['debug', 'info', 'warn', 'error']).default('info'),
});

const parsed = schema.safeParse(process.env);
if (!parsed.success) {
    console.error('Invalid environment variables:', parsed.error.format());
    process.exit(1);  // fail fast
}

export const config = parsed.data;
```

## 6. The .gitignore Non-Negotiable

```
# .gitignore — these must ALWAYS be present
.env
.env.local
.env.*.local
.env.production

# Provide a template instead
# .env.example → committed (shows variables needed, no values)
# .env → not committed (contains actual values)
```

A pre-commit hook that prevents `.env` files from being committed is worth setting up on every team:

```bash
# .git/hooks/pre-commit or using pre-commit framework
if git diff --cached --name-only | grep -qE '^\.env$|\.env\.'; then
    echo "ERROR: Attempting to commit .env file. Remove it from staging."
    exit 1
fi
```

## 7. Senior Engineer Perspective

**What a junior gets wrong:**

Committing API keys in source code "just for now." There is no "just for now." Once a key is in git history, it must be rotated — even if you delete the file. GitHub's secret scanning will find it and alert you. Attackers scrape public repos for keys constantly. A leaked AWS key has resulted in five-figure cloud bills overnight.

**What a senior enforces:**

1. `.env` in `.gitignore` from day one of every project
2. Configuration validation at startup (fail fast)
3. Separate credentials per environment (no shared prod/staging keys)
4. Rotation procedure documented for every secret

The rule: **if a credential is exposed, assume it's compromised immediately.** Rotate it before investigating further.
