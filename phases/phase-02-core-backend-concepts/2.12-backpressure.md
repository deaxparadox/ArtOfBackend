## 1. What Is It?

Backpressure is a mechanism that allows a system to signal to its producers that it is overwhelmed and cannot process more work at the current rate. Rather than accepting all work and collapsing, a system with backpressure slows or rejects the input to match its actual processing capacity.

**The real-world analogy:**

A garden hose attached to a fire hydrant without a flow regulator — the hose explodes. Backpressure is the pressure regulator that limits flow to what the hose can handle.

---

## 2. Why Systems Without Backpressure Fail

```
SCENARIO: Traffic spike — system receives 3× its normal load

WITHOUT BACKPRESSURE:
  1. All 3× requests are accepted
  2. Request queue grows → memory fills up
  3. Latency increases (requests wait longer)
  4. DB connection pool exhausted
  5. Requests time out waiting for DB connections
  6. Error rate rises
  7. More retries from clients → even more load
  8. Positive feedback loop → complete collapse

WITH BACKPRESSURE:
  1. System detects it's at capacity (queue full, pool exhausted)
  2. New requests rejected immediately with 429 or 503
  3. System processes existing requests at sustainable rate
  4. Latency stays low for accepted requests
  5. Clients back off (respect Retry-After header)
  6. Load normalizes when traffic subsides
```

## 3. Backpressure Mechanisms

### Thread Pool / Async Concurrency Limit

```python
import asyncio
from fastapi import FastAPI, HTTPException

app = FastAPI()

# Limit concurrent requests to prevent overload
request_semaphore = asyncio.Semaphore(100)  # max 100 concurrent requests

@app.get("/process")
async def process_request():
    # Try to acquire without waiting
    if not request_semaphore.locked():
        pass  # semaphore has capacity

    acquired = request_semaphore._value > 0
    if not acquired:
        raise HTTPException(
            status_code=503,
            detail="Server at capacity. Try again later.",
            headers={"Retry-After": "1"}
        )

    async with request_semaphore:
        # Only 100 requests can execute this block concurrently
        result = await do_heavy_work()
        return result
```

### Message Queue as Buffer (Natural Backpressure)

```python
# Producer (API) never blocks — just enqueues work
@app.post("/process-video")
async def submit_video(video: UploadFile):
    video_id = str(uuid.uuid4())
    await storage.save(video_id, await video.read())

    # Queue has backpressure built in:
    # If queue is full → reject with informative error
    try:
        await job_queue.enqueue(
            "process_video",
            {"video_id": video_id},
            timeout=300
        )
    except QueueFullError:
        raise HTTPException(
            status_code=503,
            detail="Processing queue is full. Please try again in a few minutes.",
            headers={"Retry-After": "60"}
        )

    return {"job_id": video_id, "status": "queued"}

# Consumer processes at its own pace — natural backpressure
# Queue depth is the pressure gauge
```

### Connection Pool as Backpressure

```python
import asyncpg

# Connection pool IS a backpressure mechanism
pool = await asyncpg.create_pool(
    dsn=DATABASE_URL,
    min_size=5,
    max_size=20,
    max_inactive_connection_lifetime=60,
    command_timeout=30,
    # If pool is exhausted: wait up to this long for a connection
    timeout=10.0,  # after 10s waiting: raise asyncpg.TooManyConnectionsError
)

@app.get("/users")
async def get_users():
    try:
        async with pool.acquire(timeout=5.0) as conn:  # wait max 5s
            return await conn.fetch("SELECT * FROM users LIMIT 100")
    except asyncpg.TooManyConnectionsError:
        raise HTTPException(
            status_code=503,
            detail="Database at capacity. Please retry.",
            headers={"Retry-After": "2"}
        )
```

## 4. The Correct Response to Backpressure: 429 vs 503

```
503 Service Unavailable:
  "We are temporarily unable to handle requests due to overload or maintenance."
  Use when: entire service is at capacity.
  Include: Retry-After header.

429 Too Many Requests:
  "You specifically have sent too many requests."
  Use when: per-client rate limit exceeded.
  Include: Retry-After, X-RateLimit-* headers.
```

Clients MUST implement exponential backoff when they receive 503 or 429. A client that ignores these and immediately retries makes the overload worse.

## 5. Senior Engineer Perspective

**What a junior gets wrong:**

Building systems that accept unlimited work. "We'll handle load by scaling horizontally" — but scaling takes time. During a spike, you need backpressure to protect the system while scaling happens. Also: accepting all work and queuing it in memory → OOM when the queue grows unbounded.

**The Little's Law check:**

Before setting concurrency limits:
```
Your service: 50ms average latency, 4 CPU cores with thread pool of 50

Throughput capacity = Concurrency / Latency = 50 / 0.05s = 1000 req/s

Set your backpressure threshold at 80% of capacity: 800 req/s
Above that: start rejecting with 503.
This protects you from the 20% buffer needed for request variance.
```
