## 1. What Is It?

Input validation is the process of verifying that data entering your system is well-formed, makes business sense, and doesn't contain malicious payloads — before using it in any operation.

**The cardinal rule:**

> **Trust nothing that comes from outside your system. Validate at every boundary.**

"Outside your system" includes: HTTP request bodies, query parameters, headers, cookies, uploaded files, URL path parameters, data from third-party APIs, data read from message queues, data read from files.

---

## 2. The Three Layers of Validation

```
LAYER 1 — STRUCTURAL (format)
  Is this valid JSON? Does it have the required fields?
  Is the type correct (integer, string, boolean)?
  Is the string within the max length?
  Handled by: Pydantic, Joi, Zod, JSON Schema

LAYER 2 — SEMANTIC (meaning)
  Is this email address syntactically valid?
  Is the quantity a positive integer?
  Is the date in the future?
  Does this enum value exist?
  Handled by: Pydantic validators, custom business logic

LAYER 3 — BUSINESS RULES (context)
  Does this user own the resource they're modifying?
  Is the product in stock?
  Has the promo code expired?
  Handled by: authorization checks + business logic
```

All three layers must run on the server. Clients can validate for UX (immediate feedback), but server validation is the only one that matters for security.

## 3. Validation with Pydantic (Python)

```python
from pydantic import BaseModel, EmailStr, field_validator, model_validator
from typing import Annotated
from pydantic import Field
import re

class CreateUserRequest(BaseModel):
    email: EmailStr                          # validates email format
    username: Annotated[str, Field(
        min_length=3,
        max_length=50,
        pattern=r'^[a-zA-Z0-9_-]+$'         # alphanumeric + _ and -
    )]
    password: Annotated[str, Field(min_length=8, max_length=128)]
    age: Annotated[int, Field(ge=13, le=120)]  # ge=greater equal, le=less equal
    role: Literal["user", "admin"] = "user"

    @field_validator("password")
    @classmethod
    def password_complexity(cls, v: str) -> str:
        if not re.search(r"[A-Z]", v):
            raise ValueError("Password must contain an uppercase letter")
        if not re.search(r"[0-9]", v):
            raise ValueError("Password must contain a digit")
        return v

    @model_validator(mode="after")
    def username_not_email(self) -> "CreateUserRequest":
        if self.username and self.email:
            if self.username.lower() == self.email.lower():
                raise ValueError("Username cannot be the same as email")
        return self

class UpdateOrderRequest(BaseModel):
    quantity: Annotated[int, Field(gt=0, le=1000)]  # 1 to 1000
    shipping_address_id: int
    notes: Annotated[str | None, Field(max_length=500)] = None

# FastAPI uses Pydantic models automatically
@app.post("/users", status_code=201)
async def create_user(data: CreateUserRequest):
    # If request body fails validation: FastAPI returns 422 with details
    # {"detail": [{"loc": ["body", "email"], "msg": "value is not a valid email", "type": "value_error.email"}]}
    user = await db.create_user(data)
    return user
```

## 4. Validation with Zod (Node.js/TypeScript)

```typescript
import { z } from 'zod';

const CreateUserSchema = z.object({
    email: z.string().email(),
    username: z.string()
        .min(3)
        .max(50)
        .regex(/^[a-zA-Z0-9_-]+$/, "Only alphanumeric, underscore, hyphen"),
    password: z.string()
        .min(8)
        .max(128)
        .regex(/[A-Z]/, "Must contain uppercase")
        .regex(/[0-9]/, "Must contain digit"),
    age: z.number().int().min(13).max(120),
    role: z.enum(["user", "admin"]).default("user"),
});

type CreateUserRequest = z.infer<typeof CreateUserSchema>;

// In Express handler:
app.post('/users', (req, res) => {
    const result = CreateUserSchema.safeParse(req.body);
    if (!result.success) {
        return res.status(422).json({
            error: 'validation_failed',
            details: result.error.flatten()
        });
    }
    const data: CreateUserRequest = result.data;
    // data is now type-safe and validated
});
```

## 5. What to Validate

```
STRING INPUTS:
  ✓ Maximum length (prevent DoS via huge inputs)
  ✓ Format (email, URL, UUID, phone number)
  ✓ Allowed characters (prevent injection if used in commands)
  ✓ Not just whitespace
  ✓ Encoding (valid UTF-8)

NUMERIC INPUTS:
  ✓ Type (integer, not float for quantities)
  ✓ Range (min/max boundaries)
  ✓ Not NaN, not Infinity

DATE/TIME INPUTS:
  ✓ Valid date format
  ✓ Within reasonable range (not year 9999)
  ✓ Timezone specified or assumed

FILE UPLOADS:
  ✓ MIME type check (don't trust Content-Type header — check file magic bytes)
  ✓ File size limit
  ✓ Filename sanitization (strip path traversal: ../../../etc/passwd)
  ✓ Scan for malware (for user-uploaded content)

IDs / REFERENCES:
  ✓ Correct format (UUID, integer, etc.)
  ✓ Exists in database (before performing operations on it)
  ✓ Belongs to the authenticated user (authorization, not just validation)
```

## 6. Common Validation Mistakes

```python
# MISTAKE 1: Trusting Content-Type for file type
@app.post("/upload")
async def upload(file: UploadFile):
    # WRONG: attacker sends image.jpg with Content-Type: image/jpeg
    # but file is actually a PHP script or executable
    if file.content_type != "image/jpeg":
        raise HTTPException(400, "Not a JPEG")

    # RIGHT: check magic bytes (first bytes of file identify its format)
    content = await file.read(12)
    if content[:3] != b'\xff\xd8\xff':  # JPEG magic bytes
        raise HTTPException(400, "Not a valid JPEG")
    await file.seek(0)  # reset for actual processing

# MISTAKE 2: Validating on create but not on update
class CreateProduct(BaseModel):
    name: str = Field(min_length=1, max_length=200)
    price: float = Field(gt=0)

class UpdateProduct(BaseModel):
    name: str | None = None    # ← no min_length! empty string allowed through
    price: float | None = None # ← no gt=0! negative price allowed through

# RIGHT: share validators
class ProductBase(BaseModel):
    name: Annotated[str, Field(min_length=1, max_length=200)]
    price: Annotated[float, Field(gt=0)]

class CreateProduct(ProductBase):
    pass

class UpdateProduct(ProductBase.model_rebuild()):
    name: Annotated[str, Field(min_length=1, max_length=200)] | None = None
    price: Annotated[float, Field(gt=0)] | None = None

# MISTAKE 3: Validating format but not existence
@app.delete("/users/{user_id}")
async def delete_user(user_id: int):  # validates it's an int
    await db.delete_user(user_id)     # what if user_id doesn't exist?
    # RIGHT: check existence first and return 404 if not found

# MISTAKE 4: Not sanitizing file paths
filename = request.form.get("filename")
path = f"/uploads/{filename}"
# attacker sends filename = "../../../etc/passwd"
# path becomes /uploads/../../../etc/passwd = /etc/passwd

# RIGHT:
import os
filename = os.path.basename(filename)  # strip path components
# or use pathlib:
from pathlib import Path, PurePath
safe_name = PurePath(filename).name  # just the filename, no directory
```

## 7. Senior Engineer Perspective

**What a junior gets wrong:**

Skipping server-side validation because "the frontend already validates." The frontend is not your security boundary. Every API endpoint is publicly accessible. Tools like Postman, curl, and Burp Suite allow anyone to send arbitrary requests to your API. Server-side validation is not optional.

**The validation philosophy:**

Validate at the boundary. Data inside your system (between your own services, from your own database) can be trusted. Data entering from outside (user input, third-party webhooks, message queues from other teams) must be validated. The boundary is the entry point into your system.

**The error message balance:**

Validation error messages should be helpful to developers and clients but not informative to attackers:
- Good: `{"field": "email", "error": "must be a valid email address"}`
- Too revealing: `{"error": "user already exists with email alice@example.com"}`
- The second tells attackers which emails are registered — a user enumeration vulnerability
