## 1. What Is It?

Graceful shutdown is the process of stopping a server in an orderly way — completing in-flight requests, closing connections cleanly, and flushing pending work — rather than abruptly killing the process.

**Why it matters:** In Kubernetes, every rolling deployment sends SIGTERM to old pods before starting new ones. Without graceful shutdown, active requests are killed mid-processing. Users see 502 errors. Database transactions are abandoned. Background jobs are interrupted. This happens on every deployment.

---

## 2. What Happens on a Non-Graceful Shutdown

```
DEPLOYMENT WITHOUT GRACEFUL SHUTDOWN:
  1. Kubernetes sends SIGTERM to pod
  2. Process dies immediately
  3. In-flight: 12 HTTP requests → all receive TCP RST (connection reset)
  4. In-flight: 3 database transactions → abandoned mid-write
  5. User sees: "Failed to fetch" or "502 Bad Gateway"
  6. Database: potential partial writes, broken state

DEPLOYMENT WITH GRACEFUL SHUTDOWN:
  1. Kubernetes sends SIGTERM to pod
  2. Load balancer stops routing NEW requests to this pod
     (Kubernetes removes pod from Endpoints — LB stops sending traffic)
  3. Process: stop accepting new connections
  4. Process: wait for in-flight requests to complete (up to N seconds)
  5. Process: close database connections cleanly
  6. Process: flush log buffers
  7. Process exits cleanly
  8. Users: zero errors during deployment
```

## 3. Implementing Graceful Shutdown

```python
import asyncio
import signal
import logging
from fastapi import FastAPI
import uvicorn

logger = logging.getLogger(__name__)
app = FastAPI()

# FastAPI/Uvicorn handles SIGTERM gracefully when configured correctly

# Option 1: Uvicorn handles it (simplest)
if __name__ == "__main__":
    uvicorn.run(
        app,
        host="0.0.0.0",
        port=8000,
        timeout_graceful_shutdown=30,  # wait up to 30s for requests to complete
    )

# Option 2: Manual signal handling
class GracefulServer:
    def __init__(self):
        self.shutdown_event = asyncio.Event()
        self._active_requests = 0

    def handle_sigterm(self, signum, frame):
        logger.info("SIGTERM received — initiating graceful shutdown")
        # Signal: stop accepting new requests, finish existing ones
        asyncio.get_event_loop().call_soon_threadsafe(self.shutdown_event.set)

    async def shutdown(self, db_pool, redis):
        """Cleanup sequence on shutdown."""
        logger.info("Waiting for in-flight requests to complete...")
        # Wait for request count to reach 0 (with timeout)
        timeout = 30
        for _ in range(timeout * 10):
            if self._active_requests == 0:
                break
            await asyncio.sleep(0.1)
        else:
            logger.warning(f"Shutdown timeout reached with {self._active_requests} active requests")

        logger.info("Closing database connections...")
        await db_pool.close()

        logger.info("Closing Redis connections...")
        await redis.aclose()

        logger.info("Flushing log buffers...")
        logging.shutdown()

        logger.info("Graceful shutdown complete")

# FastAPI lifespan event (modern approach)
from contextlib import asynccontextmanager

@asynccontextmanager
async def lifespan(app: FastAPI):
    # STARTUP: initialize resources
    logger.info("Application starting up")
    app.state.db_pool = await create_db_pool()
    app.state.redis = await create_redis_client()

    yield  # application runs here

    # SHUTDOWN: cleanup (called on SIGTERM/SIGINT)
    logger.info("Application shutting down")
    await app.state.db_pool.close()
    await app.state.redis.aclose()
    logger.info("Shutdown complete")

app = FastAPI(lifespan=lifespan)
```

## 4. Kubernetes Graceful Shutdown Config

```yaml
# The complete configuration for zero-downtime deployments in Kubernetes

apiVersion: apps/v1
kind: Deployment
spec:
  template:
    spec:
      terminationGracePeriodSeconds: 60  # how long K8s waits before SIGKILL

      containers:
      - name: api
        lifecycle:
          preStop:
            exec:
              # Critical: sleep before handling SIGTERM
              # K8s removes pod from endpoints, but load balancer may still
              # send traffic for a few seconds. This sleep absorbs that gap.
              command: ["/bin/sh", "-c", "sleep 5"]

        # Resource limits prevent OOM kills during shutdown
        resources:
          limits:
            memory: "512Mi"
```

**The preStop sleep is critical.** Without it:
1. K8s sends SIGTERM
2. App starts shutting down (stops accepting requests)
3. BUT load balancer hasn't yet learned pod is terminating
4. LB sends requests to the shutting-down pod → 503 errors

With 5s sleep: the endpoint removal propagates to the LB before SIGTERM processing begins.

## 5. Senior Engineer Perspective

**What a junior gets wrong:**

Not implementing graceful shutdown and then wondering why there are 502 errors on every deployment. Or implementing SIGTERM handling but forgetting the `preStop` sleep — so there's still a brief window of 502s during rollout.

**The zero-downtime deployment checklist:**

```
□ terminationGracePeriodSeconds > request timeout (default 30s is often not enough)
□ preStop sleep: 5-10 seconds (allows LB to drain)
□ Application handles SIGTERM: stop accepting new requests
□ Application waits for in-flight requests: up to (grace period - preStop) seconds
□ Application closes DB connections cleanly on exit
□ Application flushes async log buffers on exit
□ Background jobs: checkpoint state before exiting
```
