## 1. What Is It?

Health checks are endpoints your infrastructure uses to determine if an application instance is functioning correctly. The infrastructure uses this to decide whether to route traffic to the instance (load balancer, Kubernetes) or restart it (Kubernetes).

**Two distinct concepts in Kubernetes:**

- **Liveness probe:** Is this container alive? If liveness fails → Kubernetes restarts the container.
- **Readiness probe:** Is this container ready to receive traffic? If readiness fails → Kubernetes removes it from the load balancer rotation (no restart, just no traffic).

Getting these wrong causes either unnecessary restarts (bad liveness) or traffic sent to broken instances (bad readiness).

---

## 2. Liveness vs. Readiness

```
LIVENESS PROBE:
  Question: "Is this process still alive and not deadlocked?"
  Failure response: RESTART the container
  
  Should check:
    ✓ Process is running and not deadlocked
    ✗ NOT external dependencies (if database is down, don't restart the app —
       restarting won't fix the database)
  
  Implementation: simple "is the process responding?" check
  Endpoint: GET /live → always 200 unless completely broken

READINESS PROBE:
  Question: "Is this instance ready to serve traffic?"
  Failure response: REMOVE from load balancer rotation (no restart)
  
  Should check:
    ✓ Database is reachable and responsive
    ✓ Redis/cache is available
    ✓ Required config is loaded
    ✓ Application is fully initialized
    ✗ Not non-critical dependencies
  
  Implementation: check required dependencies
  Endpoint: GET /ready → 200 if ready, 503 if not

STARTUP PROBE (Kubernetes 1.18+):
  Question: "Has the container finished starting up?"
  Used for slow-starting containers to avoid liveness killing them during init.
  Once startup succeeds: liveness and readiness take over.
```

## 3. Implementation

```python
from fastapi import FastAPI
from fastapi.responses import JSONResponse
import asyncpg
import redis.asyncio as aioredis
import time

app = FastAPI()
start_time = time.time()

# LIVENESS — minimal check, always fast
@app.get("/live")
async def liveness():
    """
    Liveness probe: is this process alive?
    Never check external dependencies here.
    If this fails → process is restarted.
    """
    return {"status": "alive", "uptime_seconds": int(time.time() - start_time)}

# READINESS — check dependencies
@app.get("/ready")
async def readiness():
    """
    Readiness probe: is this instance ready to receive traffic?
    Check required dependencies.
    If this fails → removed from load balancer rotation.
    """
    checks = {}
    ready = True

    # Check required: database
    try:
        conn = await asyncpg.connect(dsn=settings.database_url, timeout=2)
        await conn.fetchval("SELECT 1")
        await conn.close()
        checks["database"] = "ok"
    except Exception as e:
        checks["database"] = f"error: {type(e).__name__}"
        ready = False  # database is required

    # Check required: Redis
    try:
        r = aioredis.from_url(settings.redis_url, socket_timeout=2)
        await r.ping()
        await r.aclose()
        checks["redis"] = "ok"
    except Exception as e:
        checks["redis"] = f"error: {type(e).__name__}"
        ready = False  # Redis is required

    # Check optional: background job queue (not required for basic operation)
    try:
        await job_queue.ping()
        checks["job_queue"] = "ok"
    except Exception as e:
        checks["job_queue"] = f"degraded: {type(e).__name__}"
        # Not required — don't set ready = False

    status_code = 200 if ready else 503
    return JSONResponse(
        status_code=status_code,
        content={
            "status": "ready" if ready else "not_ready",
            "checks": checks,
        }
    )

# HEALTH (combined summary, for humans and monitoring dashboards)
@app.get("/health")
async def health():
    """Human-readable health summary for dashboards."""
    return {
        "status": "healthy",
        "version": settings.app_version,
        "uptime_seconds": int(time.time() - start_time),
    }
```

## 4. Kubernetes Probe Configuration

```yaml
apiVersion: apps/v1
kind: Deployment
spec:
  template:
    spec:
      containers:
      - name: api
        livenessProbe:
          httpGet:
            path: /live
            port: 8000
          initialDelaySeconds: 10    # wait before first check (startup time)
          periodSeconds: 10          # check every 10s
          failureThreshold: 3        # restart after 3 consecutive failures
          timeoutSeconds: 5          # probe must respond in 5s

        readinessProbe:
          httpGet:
            path: /ready
            port: 8000
          initialDelaySeconds: 5     # start checking sooner than liveness
          periodSeconds: 5           # check every 5s
          failureThreshold: 3        # stop routing after 3 failures
          successThreshold: 2        # require 2 successes to re-add to rotation
          timeoutSeconds: 3          # readiness probe must be fast

        startupProbe:
          httpGet:
            path: /live
            port: 8000
          failureThreshold: 30       # give 300s to start (30 * 10s)
          periodSeconds: 10
```

## 5. Senior Engineer Perspective

**What a junior gets wrong:**

Checking external dependencies in the liveness probe. If your database goes down, liveness fails → all pods restart → pods come back up → database still down → liveness fails again → pods restart in a loop → complete service outage instead of graceful degradation.

**The rule:**

Liveness = "can the process handle requests at all?" (never check external deps)
Readiness = "is this instance fit to receive traffic?" (check what it needs)

If the database is down: readiness fails → all instances removed from rotation → service returns 503 to load balancer → users see 503 (service unavailable). When database recovers: readiness passes → instances re-added → traffic resumes. No restarts. No thrashing. Graceful degradation.
