## 1. What Is It?

Retry logic is the practice of automatically re-attempting a failed operation, with the expectation that transient failures (temporary network blips, momentary service unavailability) resolve themselves.

The challenge: retrying incorrectly causes more harm than the original failure.

---

## 2. When to Retry vs. When NOT to

```
SAFE TO RETRY:
  ✓ Network timeout (operation may have failed to reach server)
  ✓ 429 Too Many Requests (rate limited — wait and retry)
  ✓ 503 Service Unavailable (server overloaded — retry after Retry-After)
  ✓ 502 Bad Gateway (upstream error — transient)
  ✓ Connection refused (service restarting — transient)
  ✓ Idempotent operations (GET, PUT with idempotency key, DELETE)

NEVER RETRY:
  ✗ 400 Bad Request — your request is wrong, retrying won't fix it
  ✗ 401 Unauthorized — your token is invalid, retrying won't authenticate you
  ✗ 403 Forbidden — you don't have permission, retrying won't grant it
  ✗ 404 Not Found — the resource doesn't exist, retrying won't create it
  ✗ 422 Unprocessable Entity — validation failed, same request same failure
  ✗ Non-idempotent operations without idempotency keys
    (POST /payments without idempotency key → could charge twice)

THE RULE: Retry on transient failures. Never retry on permanent failures.
```

---

## 3. Exponential Backoff

Immediately retrying a failed request often hits the same problem. The server needs time to recover. Exponential backoff increases the wait time between retries:

```
FIXED DELAY (wrong):
  Try 1: fail. Wait 1s.
  Try 2: fail. Wait 1s.
  Try 3: fail.
  → All clients retry simultaneously at the same interval → thundering herd

EXPONENTIAL BACKOFF:
  Try 1: fail. Wait 1s.
  Try 2: fail. Wait 2s.
  Try 3: fail. Wait 4s.
  Try 4: fail. Wait 8s.
  
  Formula: wait = base * (2 ^ attempt)
  Cap at a maximum (e.g., 60s) to avoid indefinitely long waits.
```

---

## 4. Jitter — Preventing the Thundering Herd

If 1,000 clients all get a 503 and all wait exactly 2 seconds and retry simultaneously, they create a 1,000-request spike — potentially causing another 503. **Jitter** adds randomness to spread the retry load:

```python
import random
import time

def exponential_backoff_with_jitter(
    attempt: int,
    base_delay: float = 1.0,
    max_delay: float = 60.0,
    jitter_factor: float = 0.5,
) -> float:
    """
    Calculate backoff delay with full jitter.
    Full jitter: random between 0 and the computed delay.
    Recommended by AWS for retry scenarios.
    """
    delay = min(base_delay * (2 ** attempt), max_delay)
    # Full jitter: random between 0 and delay
    return random.uniform(0, delay)

# Without jitter: 1000 clients all retry at exactly t=2s → spike
# With jitter: 1000 clients retry randomly between t=0s and t=2s → smooth load
```

## 5. Production-Grade Retry with tenacity

```python
from tenacity import (
    retry,
    stop_after_attempt,
    wait_exponential_jitter,
    retry_if_exception_type,
    before_sleep_log,
    RetryError,
)
import httpx
import logging

logger = logging.getLogger(__name__)

@retry(
    # Stop after 4 total attempts (1 original + 3 retries)
    stop=stop_after_attempt(4),
    # Exponential backoff: 1s → 2s → 4s, with jitter
    wait=wait_exponential_jitter(initial=1, max=30, jitter=2),
    # Only retry on these exceptions
    retry=retry_if_exception_type((
        httpx.TimeoutException,
        httpx.NetworkError,
        httpx.HTTPStatusError,  # but see retry_if_result for status code filtering
    )),
    # Log before each retry
    before_sleep=before_sleep_log(logger, logging.WARNING),
    # Reraise the last exception (not RetryError wrapper)
    reraise=True,
)
async def call_payment_service(order_id: int, amount: int) -> dict:
    async with httpx.AsyncClient(timeout=5.0) as client:
        response = await client.post(
            "http://payment-svc/charge",
            json={"order_id": order_id, "amount": amount},
            headers={"Idempotency-Key": f"order-{order_id}"},  # REQUIRED for safe retries
        )
        # Only retry 5xx errors, not 4xx
        if response.status_code >= 500:
            response.raise_for_status()  # raises HTTPStatusError → triggers retry
        elif response.status_code == 429:
            retry_after = int(response.headers.get("retry-after", "5"))
            raise httpx.TimeoutException(f"Rate limited, retry after {retry_after}s")
        return response.json()

# Usage
try:
    result = await call_payment_service(order_id=123, amount=5000)
except httpx.HTTPStatusError as e:
    # All retries exhausted — payment service is truly down
    raise HTTPException(503, "Payment service unavailable")
```

## 6. Retry Budget — Preventing Retry Storms

At scale, retries amplify load during outages. If 10% of requests fail and all retry 3 times, your effective load becomes 1.3× normal. If 50% fail and all retry 3 times: 2× normal load — making recovery harder.

**The retry budget:** limit the fraction of traffic that can be retries.

```python
import time
import threading

class RetryBudget:
    """
    Allow at most X% of total requests to be retries.
    Prevents retry storms during partial outages.
    """
    def __init__(self, max_retry_ratio: float = 0.1, window_seconds: int = 60):
        self.max_ratio = max_retry_ratio
        self.window = window_seconds
        self._total = 0
        self._retries = 0
        self._lock = threading.Lock()

    def can_retry(self) -> bool:
        with self._lock:
            if self._total == 0:
                return True
            current_ratio = self._retries / self._total
            return current_ratio < self.max_ratio

    def record_request(self, is_retry: bool):
        with self._lock:
            self._total += 1
            if is_retry:
                self._retries += 1
```

## 7. Senior Engineer Perspective

**What a junior gets wrong:**

Retrying non-idempotent operations without idempotency keys. Retrying `POST /payments` without an idempotency key → double charge. Always use idempotency keys for payment and other write operations that must not be duplicated.

**The retry rule applied to HTTP methods:**

```
GET:   Idempotent → safe to retry, no key needed
HEAD:  Idempotent → safe to retry
PUT:   Idempotent (replaces resource) → safe to retry
DELETE: Idempotent (deleting twice = deleted) → safe to retry
POST:  NOT idempotent by default → REQUIRES idempotency key to retry safely
PATCH: Depends on implementation → use idempotency key to be safe
```

**The combination pattern every senior uses:**

```
Timeout (2-5s) 
  + Retry with exponential backoff + jitter (max 3 retries) 
  + Circuit breaker (open after 5 failures) 
  + Idempotency key (for write operations)
```

These four together handle 99% of transient failures without amplifying load or causing duplicates.
