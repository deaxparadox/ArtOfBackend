## 1. What Is It?

Secrets are credentials that grant access to protected resources: database passwords, API keys, encryption keys, JWT signing secrets, TLS private keys. Unlike configuration (log level, port), secrets are sensitive — their exposure is a security incident.

**The three rules of secrets:**
1. Never hardcode in source code
2. Never commit to version control
3. Always encrypt at rest and in transit

Environment variables (from .env files) are a step up from hardcoding, but for production systems, you need a **secrets manager** — a dedicated service that stores secrets encrypted, controls access, and logs every retrieval.

---

## 2. Why .env Files Aren't Enough for Production

```
.env FILE PROBLEMS IN PRODUCTION:
  - Must be copied to every server (error-prone, not scalable)
  - If server is compromised: entire .env file exposed at once
  - No audit log of who accessed which secret when
  - No automatic rotation
  - Hard to manage across many services and environments
  - CI/CD must have access to the file somehow

SECRETS MANAGER BENEFITS:
  - Centralized: all secrets in one place
  - Encrypted at rest (AES-256)
  - Fine-grained access control (service A can only read its own secrets)
  - Audit log: who accessed what, when
  - Automatic rotation (rotate DB passwords without downtime)
  - Version history (roll back to previous secret value)
  - SDK-based access (no files to copy)
```

## 3. Major Secrets Managers

### AWS Secrets Manager

```python
import boto3
import json
from functools import lru_cache

@lru_cache(maxsize=None)  # cache in memory after first fetch
def get_secret(secret_name: str, region: str = "us-east-1") -> dict:
    """Fetch secret from AWS Secrets Manager."""
    client = boto3.client("secretsmanager", region_name=region)
    response = client.get_secret_value(SecretId=secret_name)

    if "SecretString" in response:
        return json.loads(response["SecretString"])
    else:
        # Binary secret
        return {"binary": response["SecretBinary"]}

# Usage
db_creds = get_secret("prod/myapp/database")
DATABASE_URL = (
    f"postgresql://{db_creds['username']}:{db_creds['password']}"
    f"@{db_creds['host']}:{db_creds['port']}/{db_creds['dbname']}"
)
```

**AWS Secrets Manager pricing:** ~$0.40/secret/month + $0.05 per 10,000 API calls. For 20 secrets: ~$8/month — far cheaper than a security incident.

### AWS SSM Parameter Store

```python
import boto3

ssm = boto3.client("ssm", region_name="us-east-1")

def get_parameter(name: str, decrypt: bool = True) -> str:
    """Fetch parameter from SSM Parameter Store."""
    response = ssm.get_parameter(Name=name, WithDecryption=decrypt)
    return response["Parameter"]["Value"]

# SecureString parameters are encrypted with KMS
api_key = get_parameter("/prod/myapp/stripe_api_key")
```

**SSM vs Secrets Manager:**
- SSM Parameter Store: cheaper (free tier for standard params), simpler, no automatic rotation
- Secrets Manager: automatic rotation, richer API, slightly more expensive
- Use SSM for non-rotating secrets, Secrets Manager for database credentials that rotate

### HashiCorp Vault

```python
import hvac

client = hvac.Client(url="https://vault.internal:8200")
client.auth.aws.iam_login(role="myapp-prod")  # or token auth

# Read a secret
secret = client.secrets.kv.v2.read_secret_version(
    path="myapp/database",
    mount_point="secret"
)
db_password = secret["data"]["data"]["password"]
```

Vault is self-hosted — more complex to operate but no vendor lock-in. Used by large organizations with strict data residency requirements.

## 4. Kubernetes Secrets

```yaml
# Create a secret
kubectl create secret generic db-credentials \
  --from-literal=username=myapp \
  --from-literal=password=secretpassword

# Or from a file:
kubectl create secret generic tls-cert \
  --from-file=tls.crt=./server.crt \
  --from-file=tls.key=./server.key
```

```yaml
# Use in pod spec
apiVersion: apps/v1
kind: Deployment
spec:
  template:
    spec:
      containers:
      - name: myapp
        env:
        - name: DB_PASSWORD
          valueFrom:
            secretKeyRef:
              name: db-credentials
              key: password
        # Or mount as file:
        volumeMounts:
        - name: secrets-volume
          mountPath: /etc/secrets
          readOnly: true
      volumes:
      - name: secrets-volume
        secret:
          secretName: db-credentials
```

**⚠️ Kubernetes Secrets are base64-encoded by default, NOT encrypted.** Use `etcd` encryption at rest or integrate with external secrets (External Secrets Operator → AWS Secrets Manager / Vault).

## 5. Secret Rotation

The practice of periodically changing credentials — especially after potential exposure.

```
MANUAL ROTATION (bad):
  DBA changes DB password → updates .env on all servers → restarts services
  Error-prone, causes downtime, requires coordination.

AUTOMATIC ROTATION (AWS Secrets Manager):
  1. Secrets Manager creates new password in the database
  2. Verifies new password works
  3. Updates the secret value
  4. Old password still works briefly (dual-password period)
  5. Old password invalidated
  
  Application fetches secret on startup (or on connection error).
  Zero downtime, no human involvement.

ROTATION ON EXPOSURE:
  If a secret is exposed (committed to git, seen in logs, etc.):
  1. Rotate immediately (before investigation)
  2. Check audit logs for unauthorized access
  3. Investigate how it was exposed
  4. Fix the root cause
```

## 6. Secret Anti-Patterns to Avoid

```
✗ Secrets in environment variables baked into Docker images
  docker run myapp -e DB_PASSWORD=...  ← visible in process list, docker inspect
  
✗ Secrets in application logs
  logger.info(f"Connecting to {DATABASE_URL}")  ← password in URL → in logs
  Fix: log only the host/database, not the full URL with credentials

✗ Secrets in error messages returned to clients
  raise Exception(f"Failed to connect: {DATABASE_URL}")  ← returned in 500 response

✗ Secrets in Slack/email (to "debug an issue")
  DM-ing database credentials is a credential exposure.

✗ Long-lived secrets that never rotate
  Rotate all production secrets at least quarterly.
  Rotate immediately on any suspected exposure.
```

## 7. Senior Engineer Perspective

**What a junior gets wrong:**

Treating secrets like regular configuration. Database URLs with passwords in docker-compose.yml checked into git. API keys in GitHub Actions workflow files in plaintext. These are extremely common and extremely dangerous.

**The production audit:**

```bash
# Find any secrets accidentally committed to git history
git log --all --full-history -- "*.env" "*.pem" "*.key"

# Detect secrets in current files (truffleHog, detect-secrets, gitleaks)
gitleaks detect --source . --verbose

# If found: rotate the secret immediately, then clean git history with:
git filter-branch --force --index-filter \
  "git rm --cached --ignore-unmatch .env" HEAD

# Better: use git-filter-repo (safer than filter-branch)
pip install git-filter-repo
git filter-repo --path .env --invert-paths
```

**The architecture principle:**

Treat secrets as infrastructure, not configuration. They have a lifecycle (create, rotate, expire, revoke), access controls (who can read them), and audit requirements (log every access). A secrets manager is the right tool for this lifecycle. Environment variables in a `.env` file are appropriate for development. They are not appropriate for production.
