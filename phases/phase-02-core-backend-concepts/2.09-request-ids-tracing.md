## 1. What Is It?

A **request ID** is a unique identifier attached to every request that flows through your system. It is included in every log line, every error response, and every downstream call triggered by that request.

**The problem it solves:**

```
WITHOUT REQUEST IDs:
  LOG STREAM:
  14:30:00  INFO  user_login user_id=42
  14:30:00  INFO  user_login user_id=99
  14:30:01  ERROR payment_failed order_id=456
  14:30:01  INFO  order_created order_id=789
  
  Which user triggered the payment failure?
  Did order_id=456 belong to user 42 or 99?
  Which log lines are from the same request?
  → Impossible to tell.

WITH REQUEST IDs:
  14:30:00  INFO  user_login user_id=42  request_id=abc
  14:30:00  INFO  user_login user_id=99  request_id=def
  14:30:01  ERROR payment_failed order_id=456  request_id=abc
  14:30:01  INFO  order_created order_id=789   request_id=def
  
  Grep for request_id=abc → see the complete story of that request.
```

## 2. Request ID Implementation

```python
import uuid
from starlette.middleware.base import BaseHTTPMiddleware
import structlog

log = structlog.get_logger()

class RequestIdMiddleware(BaseHTTPMiddleware):
    async def dispatch(self, request, call_next):
        # Accept from upstream (if behind load balancer / API gateway)
        # Generate new one if not present
        request_id = (
            request.headers.get("x-request-id")
            or request.headers.get("x-correlation-id")
            or str(uuid.uuid4())
        )

        # Bind to all log calls in this async context
        structlog.contextvars.bind_contextvars(request_id=request_id)

        # Store on request state for handlers to access
        request.state.request_id = request_id

        response = await call_next(request)

        # Echo back in response so clients can reference it in support tickets
        response.headers["x-request-id"] = request_id

        # Clear context vars (important! async contexts can bleed)
        structlog.contextvars.clear_contextvars()

        return response
```

## 3. Propagating Request IDs to Downstream Services

In a microservices architecture, one user request triggers calls to multiple services. The request ID must be propagated to all of them.

```python
import httpx
from fastapi import Request

async def call_payment_service(request: Request, order_id: int) -> dict:
    """Forward request ID to payment service."""
    request_id = request.state.request_id

    async with httpx.AsyncClient() as client:
        response = await client.post(
            "http://payment-service/charge",
            json={"order_id": order_id},
            headers={
                "x-request-id": request_id,       # propagate
                "x-source-service": "order-service",  # identify caller
            },
            timeout=5.0,
        )
    return response.json()

# Payment service receives the same request_id
# All its log lines include request_id=abc
# You can trace the full flow across services by searching for request_id=abc
```

## 4. W3C Trace Context — The Standard

`X-Request-Id` is informal. For proper distributed tracing, use the **W3C Trace Context** standard:

```
traceparent: 00-4bf92f3577b34da6a3ce929d0e0e4736-00f067aa0ba902b7-01
             │  │                                │                │
             │  └── trace-id (16 bytes / 32 hex)  │                └── flags (01=sampled)
             │                                    └── parent-span-id (8 bytes / 16 hex)
             └── version (always 00)

tracestate: vendor-specific key=value pairs
```

**trace-id:** The unique ID for the entire request chain. Same across all services for one user action.

**span-id (parent-span-id):** Each hop creates its own span. The parent span is the one that called this service.

```python
from opentelemetry import trace
from opentelemetry.sdk.trace import TracerProvider
from opentelemetry.sdk.trace.export import BatchSpanProcessor
from opentelemetry.exporter.otlp.proto.grpc.trace_exporter import OTLPSpanExporter
from opentelemetry.instrumentation.fastapi import FastAPIInstrumentor
from opentelemetry.instrumentation.httpx import HTTPXClientInstrumentor
from opentelemetry.instrumentation.asyncpg import AsyncPGInstrumentor

# Setup (once at startup)
provider = TracerProvider()
provider.add_span_processor(
    BatchSpanProcessor(OTLPSpanExporter(endpoint="http://otel-collector:4317"))
)
trace.set_tracer_provider(provider)

# Auto-instrument FastAPI, httpx, asyncpg
FastAPIInstrumentor.instrument_app(app)   # wraps every request in a span
HTTPXClientInstrumentor().instrument()    # adds traceparent to outgoing calls
AsyncPGInstrumentor().instrument()        # records DB queries as spans

# Now every request automatically:
# 1. Creates a trace
# 2. Propagates traceparent header to downstream HTTP calls
# 3. Records DB queries as child spans
# 4. Sends to your tracing backend (Jaeger, Tempo, Zipkin, Datadog)
```

## 5. What You Get from Distributed Tracing

```
USER ACTION: "Place order" (trace_id=abc)

Waterfall view in Jaeger/Tempo:
  [0ms]    order-service: POST /orders                    [234ms total]
  [2ms]      auth-service: validate_token                 [5ms]
  [10ms]     inventory-service: check_stock               [15ms]
  [30ms]     payment-service: charge_card                 [150ms]
  [50ms]       stripe-api: POST /charges                  [120ms]
  [185ms]    order-service: INSERT INTO orders            [15ms]
  [205ms]    notification-service: queue_confirmation     [5ms]

→ Immediately visible: Stripe API call takes 120ms out of 234ms total.
  That's where to optimize.
```

## 6. Senior Engineer Perspective

**What a junior gets wrong:**

Not propagating the request ID when making outbound calls. The request ID is useful within one service, but if the downstream service doesn't receive it, you can't trace a failure that originates in the downstream service. Propagation is critical.

**What a senior considers:**

Sampling. In production at high traffic, tracing every request is expensive. Most tracing systems support sampling — trace 1% of requests normally, 100% of requests that return errors. This keeps costs manageable while ensuring you always capture failures.

**When distributed tracing outpays itself:**

The first time your service calls 5 other services and you can't determine which one is causing the 500ms latency degradation — distributed tracing shows you in 30 seconds what would otherwise take hours of log correlation.
