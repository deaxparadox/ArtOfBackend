## 1. What Is It?

Rate limiting controls how many requests a client can make in a given time period. It protects your API from:
- Accidental abuse (buggy client sending requests in a loop)
- Intentional abuse (scraping, brute force, DoS)
- Resource exhaustion (one user consuming all capacity)
- Cost overruns (expensive operations called too frequently)

## 2. The Three Algorithms

### Fixed Window Counter

```
FIXED WINDOW:
  Window: 60 seconds, Limit: 100 requests
  
  Window 1 (00:00–01:00): 100 requests allowed
  Window 2 (01:00–02:00): 100 requests allowed (counter reset)
  
  PROBLEM: Burst attack at window boundary.
    00:59: 100 requests  (window 1, fully consumed)
    01:00: 100 requests  (window 2, just started)
    → 200 requests in 2 seconds! Limit was 100/minute but 200 slipped through.
  
  USE: Simple, easy to implement. Acceptable when burst isn't a concern.
```

### Sliding Window Log

```
SLIDING WINDOW LOG:
  Keep a timestamped log of every request.
  On each new request: count requests in the last 60 seconds.
  If count > limit: reject.
  
  ACCURATE: No boundary burst problem.
  EXPENSIVE: Store one entry per request. High memory at scale.
  
  USE: When accuracy matters more than efficiency.
```

### Sliding Window Counter (Hybrid)

```
SLIDING WINDOW COUNTER:
  Approximates sliding window using two fixed windows.
  
  current_count = (
    previous_window_count × (remaining_fraction_of_previous_window)
    + current_window_count
  )
  
  Example at 01:15 (25% into window 2):
    previous_count = 80 (window 1: 00:00–01:00)
    current_count = 30  (window 2: 01:00–02:00)
    remaining = 75% of window 1 overlaps with the last 60s
    
    rate = 80 × 0.75 + 30 = 90
    
  ACCURATE ENOUGH: Low error rate in practice.
  EFFICIENT: Only 2 counters per key.
  USE: Production standard for most APIs.
```

### Token Bucket

```
TOKEN BUCKET:
  Bucket with capacity N. Refills at rate R tokens/second.
  Each request consumes 1 token. If bucket empty: reject.
  
  Example: bucket capacity=10, refill=2/second
    Burst of 10 requests: all allowed (bucket was full)
    Next 5 requests in 1s: allowed only if 2 tokens refilled
    
  BENEFIT: Allows controlled bursts (important for good UX)
  USE: When you want to allow short bursts above sustained rate.
       GitHub, Stripe use token bucket.
```

## 3. Redis-Backed Rate Limiting

```python
import redis.asyncio as aioredis
import time
from fastapi import Request, HTTPException

redis = aioredis.from_url(settings.redis_url)

async def sliding_window_rate_limit(
    key: str,
    limit: int,
    window_seconds: int,
) -> tuple[bool, dict]:
    """
    Sliding window counter using Redis.
    Returns (is_allowed, headers_dict).
    """
    now = int(time.time())
    window_start = now - window_seconds

    pipe = redis.pipeline()

    # Key for current window
    current_key = f"rate:{key}:{now // window_seconds}"
    # Key for previous window
    prev_key = f"rate:{key}:{(now // window_seconds) - 1}"

    pipe.incr(current_key)
    pipe.expire(current_key, window_seconds * 2)
    pipe.get(prev_key)
    results = await pipe.execute()

    current_count = results[0]
    prev_count = int(results[2] or 0)

    # How far into the current window are we? (0.0 to 1.0)
    window_fraction = (now % window_seconds) / window_seconds
    # Approximate count: weighted average of previous + current
    estimated_count = int(prev_count * (1 - window_fraction) + current_count)

    remaining = max(0, limit - estimated_count)
    reset_at = (now // window_seconds + 1) * window_seconds

    headers = {
        "X-RateLimit-Limit": str(limit),
        "X-RateLimit-Remaining": str(remaining),
        "X-RateLimit-Reset": str(reset_at),
    }

    if estimated_count > limit:
        headers["Retry-After"] = str(reset_at - now)
        return False, headers

    return True, headers


# FastAPI middleware
from starlette.middleware.base import BaseHTTPMiddleware

class RateLimitMiddleware(BaseHTTPMiddleware):
    async def dispatch(self, request: Request, call_next):
        # Rate limit key: per-IP for anonymous, per-user for authenticated
        client_ip = request.headers.get("x-forwarded-for", request.client.host).split(",")[0]
        auth = request.headers.get("authorization", "")

        if auth.startswith("Bearer "):
            # Authenticated: rate limit by user token (more generous)
            key = f"user:{auth[7:20]}"  # use prefix of token as key
            limit, window = 1000, 60  # 1000/min for authenticated
        else:
            # Anonymous: rate limit by IP (stricter)
            key = f"ip:{client_ip}"
            limit, window = 60, 60    # 60/min for anonymous

        allowed, headers = await sliding_window_rate_limit(key, limit, window)

        if not allowed:
            from fastapi.responses import JSONResponse
            return JSONResponse(
                status_code=429,
                content={"error": "rate_limit_exceeded", "message": "Too many requests"},
                headers=headers,
            )

        response = await call_next(request)
        for key, value in headers.items():
            response.headers[key] = value
        return response
```

## 4. Tiered Rate Limits

```
PRODUCTION PATTERNS:

Global rate limit:   10,000 req/min across all clients (protect total capacity)
Per-IP:              60 req/min for anonymous users
Per-API-key (free):  100 req/min
Per-API-key (pro):   1,000 req/min
Per-API-key (enterprise): custom

Per-endpoint limits (expensive operations get stricter limits):
  GET /users:          1,000 req/min
  POST /ai/generate:   10 req/min (expensive)
  POST /reports:       5 req/min (very expensive)
  POST /exports:       1 req/min (generates massive data)
```

## 5. Senior Engineer Perspective

**What a junior gets wrong:**

Rate limiting by IP address only. Behind corporate NAT, hundreds of users share one IP — your rate limit blocks the entire organization if any one user is aggressive. Rate limit by authenticated user ID for logged-in users. Use IP as a fallback for unauthenticated requests.

**What a senior uses:**

Tiered limits with clear headers. When clients hit the rate limit, they should know: how many requests do I have left? When does my limit reset? How long should I wait? The `X-RateLimit-*` headers answer these. A client with good headers can implement smart backoff; a client with no information just has to guess and often hammers the API harder.
