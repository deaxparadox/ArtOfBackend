## 1. What Is It?

**Stateless server:** Every request contains all information needed to fulfill it. The server keeps no memory of previous requests from the same client.

**Stateful server:** The server stores client state between requests. It "remembers" you from one request to the next.

This distinction is the most consequential architectural decision for scalability. Stateless servers scale horizontally (add more instances, route any request to any instance). Stateful servers are much harder to scale.

---

## 2. The Difference in Practice

```
STATEFUL (in-memory sessions):
  Request 1: POST /login → server stores session in RAM → returns session_id=abc
  Request 2: GET /profile (session_id=abc) → must go to SAME server (has the session)
  
  Problem: User A is always tied to Server 1.
           If Server 1 dies → User A loses session, gets logged out.
           Adding Server 2 doesn't help User A.

STATELESS (externalized sessions):
  Request 1: POST /login → stores session in Redis → returns session_id=abc
  Request 2: GET /profile (session_id=abc) → ANY server looks up Redis → same result
  
  Benefit: Any server can handle any request.
           If Server 1 dies → Server 2 picks up without disruption.
           Add 10 more servers → immediate capacity increase, zero config.
```

## 3. How to Make a Stateful App Stateless

The key: **move state out of application memory into a shared store.**

```python
# STATEFUL (don't do this in production):
app_sessions = {}  # in-process dict — dies with the process, not shareable

@app.post("/login")
async def login(credentials: LoginRequest):
    user = authenticate(credentials)
    session_id = secrets.token_urlsafe(32)
    app_sessions[session_id] = {"user_id": user.id}  # stored in RAM
    return {"session_id": session_id}

# STATELESS (production-correct):
import redis.asyncio as aioredis

redis = aioredis.from_url("redis://redis-cluster:6379")

@app.post("/login")
async def login(credentials: LoginRequest):
    user = await authenticate(credentials)
    session_id = secrets.token_urlsafe(32)
    await redis.setex(
        f"session:{session_id}",
        86400,  # TTL: 24 hours
        json.dumps({"user_id": user.id, "email": user.email})
    )
    return {"session_id": session_id}
# Any server can look up any session_id in Redis.
# Add 10 servers → all can serve any user.
```

## 4. Where State Belongs

```
APPLICATION SERVERS:  STATELESS (business logic only, no client state)
DATABASE:             STATEFUL  (persistent truth — PostgreSQL, MongoDB)
CACHE:                STATEFUL  (short-lived, losable — Redis, Memcached)
MESSAGE QUEUE:        STATEFUL  (durable messages — Kafka, RabbitMQ, SQS)
CDN:                  STATEFUL  (cached content — but losable)
JWT TOKENS:           STATELESS (client carries the state in the token)
```

## 5. JWT: Stateless Authentication

JWT (JSON Web Token) is the stateless approach to auth. Instead of storing sessions server-side, the server signs a token containing user info and sends it to the client. The client includes it in every request.

```
STATELESS JWT FLOW:
  Login → server creates JWT {user_id: 123, exp: ...} signed with secret
  Client stores JWT (localStorage, cookie)
  Every request: client sends JWT
  Server: verify signature → extract user_id (NO database/Redis lookup needed)

BENEFIT: Any server can verify any JWT (they all share the signing secret/public key)
COST:
  - Tokens cannot be invalidated before expiry (no logout blacklist without state)
  - Token contains all necessary info → tokens can be large
  - Secret must be rotated carefully

STATEFUL SESSION (Redis) vs STATELESS JWT:
  Sessions:  Server can revoke instantly (delete from Redis). Requires Redis lookup per request.
  JWT:       Cannot revoke without adding state (blacklist). No lookup needed.
  
  Production pattern: short-lived JWT (15min) + long-lived refresh token (in Redis).
  Revoke = delete refresh token. JWT expires naturally in 15min max.
```

## 6. Senior Engineer Perspective

**What a junior gets wrong:**

Building stateful app servers and then trying to "fix" scaling by using load balancer sticky sessions. Sticky sessions mean load balancer always routes a user to the same server — you've just made the load balancer stateful. If that server dies, those users lose sessions. The right fix: make the app server truly stateless.

**The 12-factor principle:**

[12-factor app](https://12factor.net/) Factor VI: **Processes** — "Execute the app as one or more stateless processes. Any data that needs to persist must be stored in a stateful backing service." This is the canonical statement of stateless app servers.
