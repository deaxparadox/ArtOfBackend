## 1. What Is It?

A circuit breaker is a pattern that detects when a downstream dependency is failing and **stops calling it for a period of time**, rather than letting every request fail slowly while waiting for timeouts.

Named after the electrical circuit breaker: when a circuit is overloaded, the breaker trips to prevent damage. It can be reset once the problem is resolved.

**The problem it solves:**

```
WITHOUT CIRCUIT BREAKER:
  Payment service is down.
  Every order request → calls payment service → waits 30s timeout → fails.
  100 requests in flight = 100 threads waiting 30s = thread pool exhausted.
  Order service is now also "down" — cascading failure.

WITH CIRCUIT BREAKER:
  Payment service starts failing.
  After N failures: circuit breaker OPENS — stop calling payment service.
  Subsequent requests: fail immediately (no 30s wait) with "Service unavailable".
  Thread pool: never exhausted.
  Order service: still functional for operations not requiring payment.
  After T seconds: circuit breaker tries one request (HALF-OPEN state).
  If it succeeds: circuit CLOSES, normal operation resumes.
```

## 2. Circuit Breaker States

```
CLOSED (normal operation):
  All requests pass through.
  Track failure rate.
  If failure rate > threshold: trip → OPEN

OPEN (dependency is failing, stop calling it):
  Reject all requests immediately (no actual call made).
  Wait for recovery timeout (e.g., 30 seconds).
  After timeout: → HALF-OPEN

HALF-OPEN (testing if dependency recovered):
  Allow one request through.
  Success → CLOSE (resume normal operation)
  Failure → OPEN again (back to waiting)

STATE MACHINE:
  CLOSED ──(failure threshold exceeded)──► OPEN
  OPEN ──(recovery timeout)──► HALF-OPEN
  HALF-OPEN ──(success)──► CLOSED
  HALF-OPEN ──(failure)──► OPEN
```

## 3. Implementation with pybreaker

```python
import pybreaker
import httpx
from fastapi import HTTPException

# Define the circuit breaker
payment_breaker = pybreaker.CircuitBreaker(
    fail_max=5,           # open after 5 consecutive failures
    reset_timeout=30,     # try again after 30 seconds
    name="payment_service",
)

class PaymentService:
    @payment_breaker
    async def charge(self, amount: int, token: str) -> dict:
        """Will not be called when circuit is OPEN."""
        async with httpx.AsyncClient(timeout=5.0) as client:
            response = await client.post(
                "http://payment-service/charge",
                json={"amount": amount, "token": token},
            )
            response.raise_for_status()
            return response.json()

payment_svc = PaymentService()

@app.post("/orders")
async def create_order(data: OrderRequest):
    try:
        payment = await payment_svc.charge(data.amount, data.payment_token)
    except pybreaker.CircuitBreakerError:
        # Circuit is OPEN — fail fast
        raise HTTPException(
            status_code=503,
            detail="Payment service temporarily unavailable. Please retry in 30 seconds.",
            headers={"Retry-After": "30"}
        )
    except httpx.HTTPError as e:
        # Actual payment failure (will count toward breaker threshold)
        raise HTTPException(status_code=402, detail="Payment failed")

    return create_order_in_db(data, payment)
```

## 4. Circuit Breakers in Practice

```
WHEN TO USE CIRCUIT BREAKERS:
  ✓ External HTTP API calls (Stripe, Twilio, AWS services)
  ✓ Calls to other microservices
  ✓ Database connections (with caution)
  ✓ Any dependency that can be unavailable

CIRCUIT BREAKER CONFIGURATION GUIDE:
  fail_max:         How many failures before opening?
                    Higher = more tolerance. Lower = faster response.
                    For expensive operations: 3-5 failures.
                    For cheap idempotent operations: 10-20 failures.
  
  reset_timeout:    How long to wait before testing recovery?
                    Should be longer than your dependency's typical recovery time.
                    For services that restart: 10-30 seconds.
                    For databases: 30-60 seconds.
  
  error_threshold:  Some breakers use % failure rate (not count).
                    50% failure rate over 60s is more accurate at high traffic.

FALLBACK STRATEGIES WHEN CIRCUIT IS OPEN:
  1. Return cached/stale data     (for read operations)
  2. Return a default response    (product is "available" unless we know otherwise)
  3. Queue for later processing   (payment → queue → retry when service recovers)
  4. Fail fast with 503           (for operations that cannot be delayed)
```

## 5. Senior Engineer Perspective

**What a junior gets wrong:**

Not combining circuit breakers with timeouts. A circuit breaker on a call with no timeout will wait indefinitely on the first failures before the breaker knows to open. The correct pattern: short timeout (2-5s) + circuit breaker. The timeout determines how long a single failed request blocks; the circuit breaker determines when to stop trying at all.

**Circuit breakers vs. retries:**

- Retries: "try again — it might work this time"
- Circuit breaker: "stop trying — it definitely won't work right now"

These are complementary. Retry transient errors (occasional 503). Circuit break persistent failures (service is down). Using only retries during an outage amplifies the load on the failing service. Circuit breakers protect both your service and the downstream service.
