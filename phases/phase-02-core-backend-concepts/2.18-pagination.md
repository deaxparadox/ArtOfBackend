## 1. Why Pagination Exists

Returning all records from a database table is almost always wrong:
- 10 million users table → 10M rows → gigabytes of JSON → OOM, timeout, terrible UX
- Every request must return a bounded amount of data
- Users rarely need "everything" — they need the current page

Pagination is how you return a manageable slice of a large dataset.

---

## 2. Three Approaches

### Offset Pagination (Most Common, Often Wrong at Scale)

```sql
-- Page 1: first 20 records
SELECT * FROM orders ORDER BY created_at DESC LIMIT 20 OFFSET 0;

-- Page 2: next 20 records
SELECT * FROM orders ORDER BY created_at DESC LIMIT 20 OFFSET 20;

-- Page N:
SELECT * FROM orders ORDER BY created_at DESC LIMIT 20 OFFSET (N-1)*20;
```

```
API: GET /orders?page=5&limit=20
     GET /orders?offset=80&limit=20

PROS:
  ✓ Jump to any page
  ✓ Simple to implement
  ✓ Familiar to users ("page 5 of 12")

CONS:
  ✗ OFFSET N requires database to scan and discard the first N rows
    OFFSET 10000 = database reads 10,020 rows, throws away 10,000
    At large offsets: extremely slow even with indexes
  
  ✗ Data drift: if a row is inserted/deleted between pages:
    Page 1 returns items 1-20
    New item inserted at position 5
    Page 2 returns old items 21-40 (skipped item 20, which moved to position 21)
    Users see duplicates or missing items
  
  USE WHEN: small datasets, admin UIs where exact page navigation is needed,
            datasets that rarely change
```

### Cursor Pagination

```
CURSOR: an opaque token encoding the position in the result set.
        Usually base64-encoded metadata (last seen ID, timestamp, etc.).
        Client sends cursor to get the next page.

API: GET /orders                    → returns first page + cursor
     GET /orders?cursor=eyJ...      → returns next page + new cursor
     GET /orders?cursor=eyJ...      → returns next page + new cursor
     (no cursor in response means: no more pages)
```

```python
import base64, json

def encode_cursor(order_id: int, created_at: str) -> str:
    data = {"id": order_id, "created_at": created_at}
    return base64.urlsafe_b64encode(json.dumps(data).encode()).decode()

def decode_cursor(cursor: str) -> dict:
    return json.loads(base64.urlsafe_b64decode(cursor.encode()))

@app.get("/orders")
async def list_orders(cursor: str | None = None, limit: int = 20):
    if cursor:
        pos = decode_cursor(cursor)
        # Keyset condition: fetch records AFTER the cursor position
        rows = await db.fetch("""
            SELECT * FROM orders
            WHERE (created_at, id) < ($1, $2)
            ORDER BY created_at DESC, id DESC
            LIMIT $3
        """, pos["created_at"], pos["id"], limit + 1)  # +1 to detect if more pages
    else:
        rows = await db.fetch("""
            SELECT * FROM orders ORDER BY created_at DESC, id DESC LIMIT $1
        """, limit + 1)

    has_more = len(rows) > limit
    items = rows[:limit]

    next_cursor = None
    if has_more:
        last = items[-1]
        next_cursor = encode_cursor(last["id"], str(last["created_at"]))

    return {
        "items": items,
        "next_cursor": next_cursor,
        "has_more": has_more,
    }
```

```
CURSOR PROS:
  ✓ Consistent: new inserts/deletes don't affect ongoing pagination
  ✓ Fast: no OFFSET scan — uses indexed keyset condition
  ✓ Works for infinite scroll (social media feeds, chat history)

CURSOR CONS:
  ✗ Cannot jump to arbitrary page
  ✗ Client must track cursor state
  ✗ Cursor becomes invalid if sort order changes
  
  USE WHEN: large datasets, feeds, APIs with frequent data changes,
            mobile infinite-scroll UIs
```

### Keyset Pagination

Keyset is the underlying technique behind cursor pagination — using the values of the last row as the WHERE condition:

```sql
-- First page
SELECT * FROM orders ORDER BY id ASC LIMIT 20;
-- last row: id=20

-- Next page (WHERE id > last_seen_id)
SELECT * FROM orders WHERE id > 20 ORDER BY id ASC LIMIT 20;
-- last row: id=40

-- This is O(1) per query regardless of page depth.
-- OFFSET 1000 = scan 1000 rows. Keyset = use index to jump directly to id=1000.
```

## 3. Response Format

```json
// Offset pagination response
{
  "items": [...],
  "total": 1532,
  "page": 5,
  "limit": 20,
  "total_pages": 77,
  "has_next": true,
  "has_prev": true
}

// Cursor pagination response (preferred)
{
  "items": [...],
  "next_cursor": "eyJpZCI6IDIwMCwgImNyZWF0ZWRfYXQiOiAiMjAyNC0wNi0xNVQxNDozMDowMFoifQ==",
  "has_more": true
}
// No "total" — don't compute COUNT(*) unless you need it.
// COUNT(*) on a large table is expensive. Avoid it if possible.
```

## 4. Performance Comparison

```
Dataset: 10 million orders table. Fetch page 500 (items 9,981–10,000).

OFFSET PAGINATION:
  SELECT * FROM orders ORDER BY created_at DESC LIMIT 20 OFFSET 9980;
  DB must: read and discard 9,980 rows → find the next 20
  Time: ~800ms (even with index — still scans 10,000 rows)

KEYSET PAGINATION:
  SELECT * FROM orders WHERE (created_at, id) < ($1, $2)
  ORDER BY created_at DESC, id DESC LIMIT 20;
  DB must: use index to jump to position → read exactly 20 rows
  Time: ~2ms (same speed regardless of how deep in the dataset)

400× performance difference at page 500.
At page 5000: offset = 30+ seconds. Keyset = still 2ms.
```

## 5. Senior Engineer Perspective

**What a junior gets wrong:**

Using OFFSET pagination for everything because it's simpler. It works fine for small datasets. At scale, it's a performance disaster and the cause of many production slowdowns. Switch to cursor/keyset pagination when your tables have more than ~100,000 rows and users need to paginate deeply.

**What a senior designs:**

Cursor pagination for all list endpoints that could grow large. Also: never expose `SELECT COUNT(*)` for large tables in the response — `COUNT(*)` is expensive and often unnecessary. "Showing 1-20 of 1,532,471 results" requires a full table scan just for the count. Show "showing next 20" without the total, or use approximate counts.
