## 1. Why API Versioning Exists

APIs evolve. You add fields, rename endpoints, change response shapes, remove deprecated functionality. If clients depend on your API and you break backwards compatibility, they break.

Versioning gives you a mechanism to evolve your API while giving clients time to migrate.

**The golden rule:**

> Never make breaking changes to an existing API version. Add a new version for breaking changes.

**What is a breaking change:**
- Removing a field from a response
- Renaming a field
- Changing a field's type
- Removing an endpoint
- Changing endpoint semantics (same URL, different behavior)
- Making an optional parameter required

**What is NOT a breaking change:**
- Adding new optional fields to responses
- Adding new optional request parameters
- Adding new endpoints
- Fixing bugs (behavior becomes more correct)

## 2. Three Strategies

### URL Path Versioning (Most Common)

```
/api/v1/users
/api/v2/users

PROS:
  ✓ Visible in URLs — easy to debug with curl/logs
  ✓ Can be bookmarked, cached by CDN per version
  ✓ No ambiguity — version is explicit in every URL
  ✓ Easy routing: /v1/* → v1 handlers, /v2/* → v2 handlers

CONS:
  ✗ "Violates REST" (Roy Fielding disagrees with versioning in URLs)
  ✗ Forces clients to update all hardcoded URLs
  ✗ Proliferation of routes as versions accumulate

USED BY: Twitter, Stripe, GitHub v3, most public APIs
```

```python
from fastapi import APIRouter, FastAPI

app = FastAPI()

v1_router = APIRouter(prefix="/api/v1")
v2_router = APIRouter(prefix="/api/v2")

@v1_router.get("/users/{user_id}")
async def get_user_v1(user_id: int):
    user = await db.get_user(user_id)
    return {"id": user.id, "name": user.name, "email": user.email}

@v2_router.get("/users/{user_id}")
async def get_user_v2(user_id: int):
    user = await db.get_user(user_id)
    return {
        "id": user.id,
        "display_name": user.name,    # renamed: "name" → "display_name"
        "contact": {                   # nested: email moved into "contact" object
            "email": user.email,
            "phone": user.phone,
        }
    }

app.include_router(v1_router)
app.include_router(v2_router)
```

### Header Versioning

```
GET /users/123
Accept: application/vnd.myapi.v2+json

or:

GET /users/123
API-Version: 2024-06-15

PROS:
  ✓ Clean URLs (not cluttered with version)
  ✓ Version is metadata, not part of the resource identifier

CONS:
  ✗ Invisible — hard to test with browser, curl needs headers
  ✗ CDNs don't automatically vary cache by custom headers
    (must add Vary: API-Version)
  ✗ Clients easily miss the header requirement

USED BY: GitHub API (uses Accept header), Stripe (uses Stripe-Version header)
```

### Query Parameter Versioning

```
GET /users/123?version=2
GET /users/123?api_version=v2

PROS:
  ✓ Easy to test in browser URL bar
  ✓ Optional parameter easy to add to existing clients

CONS:
  ✗ Pollutes the URL semantics (version is not a resource filter)
  ✗ Easy to forget
  ✗ CDN caching: must include version in cache key

USED BY: Some Google APIs, AWS query-string versioning
```

## 3. Stripe's Date-Based Versioning (Industry Best Practice)

Stripe uses a date-based version in a header:

```
Stripe-Version: 2024-06-15
```

Each version represents a snapshot of the API behavior on that date. When Stripe makes a breaking change:
1. They add a new date version
2. Existing customers keep their pinned version forever
3. New customers get the latest version by default
4. Old version maintained until a sunset date with 12+ months notice

This is the most mature versioning approach but requires significant infrastructure to maintain multiple parallel versions.

## 4. Version Lifecycle

```
VERSIONING LIFECYCLE:

1. Launch: /api/v1 is the only version
2. Breaking change needed: launch /api/v2 alongside v1
3. Communicate: document what changed, provide migration guide
4. Support both: maintain v1 and v2 simultaneously
5. Deprecate v1: mark as deprecated in docs and response headers
   Deprecation-Notice: This API version will be removed on 2025-01-01
6. Sunset v1: remove after sufficient notice (typically 6-12 months)

DEPRECATION HEADER (add to all v1 responses):
  Deprecation: Sat, 01 Jan 2025 00:00:00 GMT
  Sunset: Sat, 01 Jan 2025 00:00:00 GMT
  Link: <https://api.example.com/docs/migration/v2>; rel="successor-version"
```

## 5. Internal vs. Public APIs

```
INTERNAL APIS (only your own frontend/services consume):
  Version loosely or not at all.
  You control all consumers — coordinate changes.
  
  Strategy: consumer-driven contracts (Pact testing).
  Change API and update all consumers in the same deploy.

PUBLIC APIS (third-party developers, mobile apps):
  Version strictly. You don't control consumers.
  Breaking change = new version.
  Mobile app versions can live in the wild for YEARS (users don't update).
  
  Strategy: URL versioning with 12-month deprecation windows.
```

## 6. Senior Engineer Perspective

**What a junior gets wrong:**

Versioning too early. A v1 that immediately has a v2 "because we might change things" adds complexity with no benefit. The decision to version should be driven by an actual breaking change, not anticipation of future changes. Build v1 with backwards-compatible design in mind; version when you actually need to break something.

**What a senior thinks about:**

Additive-first design. Most "versioning needs" can be avoided by designing the API to be extensible:
- Return objects with named fields (adding new fields is never breaking)
- Make all parameters optional with sensible defaults
- Use enums that can be extended
- Avoid semantic changes to existing endpoints

If you can go 2 years without a breaking change, you've saved yourself a versioning headache and your clients have saved themselves a migration. Design for extensibility first; version as a last resort.
