## Caching Fundamentals (7.1)

Caching is the practice of storing copies of expensive-to-compute or expensive-to-fetch results in a faster store so that future requests can be served from the cache.

**Why it exists:** Moving data "up the memory hierarchy" (see topic 0.6). A database query at 10ms that runs 1000 times = 10 seconds. The same data served from Redis at 0.5ms = 0.5 seconds. The improvement is independent of your application code.

**The cache decision framework:**

```
IS IT WORTH CACHING?
  How expensive is generating this value?     → High: cache it
  How often is it requested?                  → High: cache it
  How often does it change?                   → High: shorter TTL or no cache
  What is the cost of stale data?            → High: cache carefully
  
FORMULA:
  Cache benefit = (request_frequency × computation_cost) - cache_overhead
  Cache worthwhile when: benefit > 0
```

## Cache Eviction Policies (7.2)

When cache is full, something must be evicted:

```
LRU (Least Recently Used):
  Evict the item not accessed for the longest time.
  Best for: general-purpose caching. Most common.
  Redis default: LRU with approximation (sampling, not exact)

LFU (Least Frequently Used):
  Evict the item accessed least often.
  Better than LRU when some items are always popular.
  Redis: allkeys-lfu, volatile-lfu

FIFO (First In First Out):
  Evict oldest inserted item.
  Simple but ignores access patterns.

TTL (Time To Live):
  Evict items after a configured time.
  Not an eviction policy per se — TTL expires items regardless of access.

RANDOM:
  Evict a random item.
  Surprisingly effective, very cheap to implement.

LRU vs LFU EXAMPLE:
  Cache size: 3. Access pattern: A B C A B C A B C ... D
  With LRU: C gets evicted when D arrives (C was least recently used)
  With LFU: D gets evicted immediately (D accessed only once vs A,B,C many times)
  LFU is better here — A,B,C are hot items; D is a one-time access.
```

## Cache Invalidation (7.3): The Hard Problem

Phil Karlton's famous quote: *"There are only two hard things in Computer Science: cache invalidation and naming things."*

```
INVALIDATION STRATEGIES:

TTL (Time-based expiry):
  Cache expires after N seconds, regardless of whether data changed.
  Pro: Simple, no coordination needed
  Con: Stale data for up to TTL seconds
  Use: Data that changes infrequently, stale-for-N-seconds is acceptable

EVENT-BASED INVALIDATION:
  When data changes, explicitly delete or update the cache key.
  Pro: Cache is always fresh
  Con: Must coordinate between write and cache systems
  Use: User profiles, product prices, anything that MUST be current

WRITE-THROUGH (update cache on write):
  When writing data, update the cache at the same time.
  Pro: Cache is always fresh
  Con: All writes pay the cache update cost
  Use: Data that is read far more than written

VERSION-BASED (cache-busting):
  Embed a version number in the cache key: user:123:v2
  When data changes, increment the version.
  Old cache key simply expires (no explicit deletion needed).
  Use: Complex objects where partial invalidation is hard

TAG-BASED INVALIDATION:
  Tag cache entries with related identifiers.
  When "product 456" changes, invalidate all cache entries tagged "product:456"
  Use: Multiple cache entries depend on the same underlying data.

THE CONSISTENCY PROBLEM:
  Between "update database" and "invalidate cache":
  another request may read the stale value from cache.
  
  Solution: invalidate cache BEFORE or DURING the database write
  (not after — the window for stale reads is larger if you invalidate after).
  
  For critical consistency: use transactions to write DB + delete cache atomically
  (using Lua scripts in Redis or Redis transactions).
```

## Redis as a Cache (7.5)

```python
import redis.asyncio as aioredis
import json
from functools import wraps

cache = aioredis.from_url("redis://localhost:6379", decode_responses=True)

# Basic cache-aside pattern
async def get_user(user_id: int) -> dict:
    cache_key = f"user:{user_id}"

    # Try cache first
    cached = await cache.get(cache_key)
    if cached is not None:
        return json.loads(cached)

    # Cache miss — fetch from DB
    user = await db.fetchrow("SELECT * FROM users WHERE id=$1", user_id)
    if not user:
        return None

    # Store in cache with 5 minute TTL
    await cache.setex(cache_key, 300, json.dumps(dict(user)))
    return dict(user)

# Decorator-based caching
def cached(ttl: int = 300, key_fn=None):
    def decorator(func):
        @wraps(func)
        async def wrapper(*args, **kwargs):
            cache_key = key_fn(*args, **kwargs) if key_fn else f"{func.__name__}:{args}:{kwargs}"
            result = await cache.get(cache_key)
            if result is not None:
                return json.loads(result)
            result = await func(*args, **kwargs)
            if result is not None:
                await cache.setex(cache_key, ttl, json.dumps(result))
            return result
        return wrapper
    return decorator

@cached(ttl=60, key_fn=lambda user_id: f"user_orders:{user_id}")
async def get_user_orders(user_id: int):
    return await db.fetch("SELECT * FROM orders WHERE user_id=$1 LIMIT 20", user_id)
```

## Cache Patterns (7.7-7.10)

### Cache-Aside (Lazy Loading) — Most Common

```python
# Application controls cache population
# Read: check cache → miss → load from DB → populate cache → return
# Write: update DB → (optionally) invalidate or update cache
```

### Write-Through

```python
# Write to cache AND database simultaneously
async def update_user(user_id: int, data: dict):
    # Write to both at the same time
    await db.execute("UPDATE users SET name=$1 WHERE id=$2", data["name"], user_id)
    await cache.setex(f"user:{user_id}", 300, json.dumps(data))
    # Cache always matches DB — reads never miss
```

### Write-Behind (Write-Back)

```python
# Write to cache, persist to DB asynchronously
# Higher performance, risk of data loss if cache fails before DB write
async def update_metrics(key: str, value: int):
    await cache.incr(key)  # immediate cache update
    # Background job flushes to DB every 30s
    # Risk: if Redis dies before flush, last 30s of metrics lost
```

### Read-Through

The cache itself is responsible for loading from DB on miss (the cache is an intermediary layer, not the application):

```python
# Used with caching frameworks that support this pattern
# Application: cache.get(key)
# Cache: on miss, automatically calls db.get(key) internally
# Less common in Python; more common with dedicated cache proxies
```

## Cache Stampede (7.11): The Thundering Herd

```
PROBLEM:
  Popular cache key expires.
  100 concurrent requests all find cache miss.
  All 100 go to database simultaneously.
  Database is overwhelmed.

SOLUTIONS:

1. Probabilistic Early Expiry (PER):
   Start regenerating cache BEFORE it expires.
   With probability = f(time_remaining, regeneration_cost):
   if random() < probability: regenerate early

2. Mutex / Locking (most common):
   First request to find cache miss acquires a lock.
   Other requests wait for the lock.
   Lock holder regenerates cache, releases lock.
   Others read the new value.
```

```python
import asyncio

async def get_with_stampede_protection(key: str, fetch_fn, ttl: int = 300):
    """Prevent cache stampede using Redis locks."""
    # Try cache first
    value = await cache.get(key)
    if value:
        return json.loads(value)

    # Try to acquire lock (only one caller regenerates)
    lock_key = f"lock:{key}"
    acquired = await cache.set(lock_key, "1", nx=True, ex=5)  # NX = set only if not exists

    if acquired:
        try:
            # This caller won the lock — regenerate
            value = await fetch_fn()
            await cache.setex(key, ttl, json.dumps(value))
            return value
        finally:
            await cache.delete(lock_key)
    else:
        # Another caller is regenerating — wait and retry
        for _ in range(10):
            await asyncio.sleep(0.2)
            value = await cache.get(key)
            if value:
                return json.loads(value)
        # Fallback: go to DB if cache never populated
        return await fetch_fn()
```

## HTTP Caching (7.14)

```python
# FastAPI: set HTTP cache headers
from fastapi import FastAPI, Response
from datetime import datetime, timezone
import hashlib, json

app = FastAPI()

@app.get("/products/{product_id}")
async def get_product(product_id: int, response: Response):
    product = await db.get_product(product_id)

    # ETag: fingerprint of the response
    etag = f'"{hashlib.md5(json.dumps(product, sort_keys=True).encode()).hexdigest()}"'
    response.headers["ETag"] = etag
    response.headers["Cache-Control"] = "public, max-age=300, stale-while-revalidate=60"
    # public: CDN can cache
    # max-age=300: fresh for 5 minutes
    # stale-while-revalidate=60: serve stale for 1 min while refreshing in background

    return product
```

## CDN Caching (7.15)

CDN caches are the outermost cache layer — closest to users, farthest from your servers.

```
CDN CACHING STRATEGY:

ALWAYS CACHE (long TTL with versioned URLs):
  /static/app.a3f9d2.js          → Cache-Control: public, max-age=31536000, immutable
  /static/logo.png               → Same
  
CACHE WITH SHORT TTL:
  /api/products                  → Cache-Control: public, max-age=60, stale-while-revalidate=300
  
NEVER CACHE:
  /api/users/me                  → Cache-Control: private, no-store
  /api/checkout                  → Cache-Control: no-store
  Any authenticated API call    → Cache-Control: private, no-store

CDN INVALIDATION AFTER DEPLOY:
  1. Bump version in URLs (content-hashing): no invalidation needed
  2. For non-versioned paths: call CDN API to purge
     Cloudflare: POST /zones/{zone}/purge_cache with file list
     CloudFront: create InvalidationBatch via API
```

## Bloom Filters (7.19)

A probabilistic data structure that efficiently answers "is this item in the set?" with possible false positives but no false negatives.

```python
# Redis has a built-in Bloom filter (ReBloom module)
# or use pybloom-live

from pybloom_live import BloomFilter

# Create with expected 1M items, 0.1% false positive rate
bloom = BloomFilter(capacity=1_000_000, error_rate=0.001)

# Add items
bloom.add("user:123")
bloom.add("user:456")

# Check membership
"user:123" in bloom  # True (definitely in set)
"user:789" in bloom  # False (definitely NOT in set)
"user:000" in bloom  # Possibly False or True (0.1% chance of false positive)

# USE CASE: Prevent cache miss stampede to DB
# "Has user 999999 ever existed?" — check bloom filter first
# If bloom says NO: user definitely doesn't exist (skip DB)
# If bloom says YES: check DB (might be false positive but not frequent)

# Another use case: URL deduplication (web crawler)
# "Have we visited this URL before?" — cheap O(1) check before DB lookup
```

## Caching in Production: The Rules

```
RULE 1: Measure before caching
  Don't cache everything. Profile first, find the hot paths, cache those.
  Unused caches add complexity and operational overhead.

RULE 2: Cache at the highest level that makes sense
  In-process (dict): fastest, but not shared across servers
  Redis: shared across all servers, fast enough for most cases
  CDN: prevents requests from reaching your servers at all

RULE 3: Know your TTL
  Too short: cache misses frequent, little benefit
  Too long: stale data, user-visible bugs
  Use stale-while-revalidate for the middle ground

RULE 4: Design for cache failure
  What happens when Redis is down?
  Answer should be: "we go to the database, it's slower but works"
  Never make cache availability a hard dependency for basic functionality

RULE 5: Monitor cache hit rate
  Hit rate < 80%: cache is barely helping. Review what you're caching.
  Hit rate > 95%: optimal. Cache is doing its job.
  Sudden drop in hit rate: something changed (new access pattern, keys changed).
```
