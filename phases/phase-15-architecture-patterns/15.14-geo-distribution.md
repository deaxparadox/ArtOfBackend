## 1. Geo-Distribution (15.14)

### What Is It?

Geo-distribution means running your application in multiple geographic regions to: reduce latency for users worldwide, satisfy data residency requirements, and provide disaster recovery.

```
THE LATENCY MOTIVATION:

User in Tokyo connecting to a server in Virginia:
  Round-trip time: ~150ms
  
  For a modern web app requiring 10 API calls:
  10 × 150ms = 1.5 seconds of pure network latency
  Before your server even starts thinking.

WITH GEO-DISTRIBUTION:
  Tokyo user → Tokyo region server: ~5ms RTT
  10 × 5ms = 50ms of network latency
  30x improvement in round-trip overhead
```

### Multi-Region Architecture Patterns

```
ACTIVE-PASSIVE (one primary region):
  Primary region: handles all writes
  Passive regions: read-only replicas
  
  User writes → route to primary region
  User reads → route to nearest region
  
  Trade-off: writes still go to primary (latency for write-heavy users far from primary)
  Use when: most users are close to primary, writes are less frequent

ACTIVE-ACTIVE (multiple primary regions):
  Multiple regions: each can handle reads AND writes
  Replication: synchronous or asynchronous between regions
  
  User writes → route to nearest region
  User reads → route to nearest region
  
  Challenge: how do you handle conflicting writes?
    User A in Tokyo and User B in Virginia modify the same record simultaneously?
    
  Solutions:
    Last-write-wins (LWW): timestamp determines winner. Risk: clock skew.
    Vector clocks: detect conflicts, resolve manually or by merge.
    Conflict-free data types (CRDTs): design data structures where merges are always valid.
    Geo-pinning: user always writes to their home region (no conflicts, but fails on region outage).

DATA RESIDENCY REQUIREMENTS:
  GDPR (EU): EU users' personal data must stay in the EU.
  Data sovereignty (China): data about Chinese citizens must stay in China.
  Healthcare (US HIPAA): protected health information must stay in US.
  
  Solution: shard data by user's country/region.
  EU users → EU database.
  US users → US database.
  Route all requests based on user's registered country.
```

### AWS Multi-Region Pattern

```python
# Route requests to nearest region
import boto3

class RegionalRouter:
    REGION_MAP = {
        "NA": "us-east-1",
        "EU": "eu-west-1",
        "APAC": "ap-southeast-1",
    }
    
    def get_db_connection(self, user_country: str) -> str:
        """Return the database endpoint for a user's region."""
        if user_country in ("US", "CA", "MX", "BR"):
            region = "NA"
        elif user_country in ("GB", "DE", "FR", "SE", "NL"):
            region = "EU"
        else:
            region = "APAC"
        
        return f"db.{self.REGION_MAP[region]}.myapp.com"

# Route 53 latency-based routing:
# Route 53 automatically routes users to the lowest-latency region.
# No application code needed — DNS handles it.
# Each region's load balancer is registered with its region's latency.
```

## 2. Database Sharding (15.15)

### What Is It?

Sharding (horizontal partitioning) splits a database across multiple servers, where each server holds a subset of the data. This allows scaling writes beyond what a single server can handle.

```
WITHOUT SHARDING (vertical scaling limit):
  One database server.
  Add CPU, RAM, disk — up to a physical limit.
  Eventually: one server can't handle the write throughput.

WITH SHARDING (horizontal scaling):
  Shard 0: rows where user_id % 4 == 0 → Server A
  Shard 1: rows where user_id % 4 == 1 → Server B
  Shard 2: rows where user_id % 4 == 2 → Server C
  Shard 3: rows where user_id % 4 == 3 → Server D
  
  4x the write capacity.
  Each server holds 25% of the data.
  
  Scale to 8 shards: 8x write capacity.
  Virtually unlimited horizontal scale.
```

### Sharding Strategies

```python
RANGE SHARDING:
  Shard by ranges of a key.
  Shard 0: user_id 0-1,000,000
  Shard 1: user_id 1,000,001-2,000,000
  
  Pros: range queries easy ("all users 500k-1M")
  Cons: "hot shards" (new users always go to the latest shard)

HASH SHARDING:
  Hash the shard key → assign to shard.
  shard_number = hash(user_id) % num_shards
  
  Pros: even distribution, no hot shards
  Cons: range queries require all shards, resharding is painful

DIRECTORY SHARDING:
  A lookup table tells you which shard holds which key.
  
  ShardDirectory:
  user_123 → Shard 2
  user_456 → Shard 0
  
  Pros: maximum flexibility, can rebalance by updating directory
  Cons: directory is a bottleneck, must be highly available

THE RESHARDING PROBLEM:
  You start with 4 shards. Now you need 8.
  hash(user_id) % 4 ≠ hash(user_id) % 8
  
  EVERY ROW needs to potentially move to a new shard.
  Migration while serving live traffic is extremely difficult.
  
  SOLUTION: Consistent Hashing (see topic 15.16) minimizes resharding impact.
```

## 3. Consistent Hashing (15.16)

### What Is It?

Consistent hashing is an algorithm that distributes data across nodes such that when nodes are added or removed, only a minimal portion of data needs to be moved — not all of it.

**The problem with simple modulo hashing:**

```
4 servers: hash(key) % 4
  User 123 → 123 % 4 = 3 → Server 3
  User 456 → 456 % 4 = 0 → Server 0

ADD 1 server (now 5 servers): hash(key) % 5
  User 123 → 123 % 5 = 3 → Server 3 (same — lucky)
  User 456 → 456 % 5 = 1 → Server 1 (DIFFERENT — was Server 0!)
  
  78% of keys move to different servers!
  All that data needs to be migrated.
  Cache misses everywhere. Database overload.
```

**Consistent hashing solution:**

```
CONSISTENT HASHING RING:

Imagine a circle (0 to 2^32-1 in value space).
Place servers at positions on the ring (by hashing their names).

  Server A at position 10
  Server B at position 30
  Server C at position 60
  Server D at position 80
  
  The ring:
  0 ──── [Server A: 10] ──── [Server B: 30] ──── [Server C: 60] ──── [Server D: 80] ──── 100

For a key: hash it, find its position on the ring.
Route to the FIRST server clockwise from the key's position.

  Key at position 15 → first server clockwise → Server B (30)
  Key at position 45 → first server clockwise → Server C (60)
  Key at position 95 → first server clockwise → Server A (10) (wraps around)

ADDING A NEW SERVER:
  Add Server E at position 50:
  
  0 ──── [A: 10] ──── [B: 30] ──── [NEW E: 50] ──── [C: 60] ──── [D: 80] ──── 100
  
  Only keys between positions 30 and 50 need to move (from C to E).
  That's 20% of the ring (one section), not all of it.
  
  MINIMAL DATA MOVEMENT: only 1/N fraction moves when adding 1 of N servers.

VIRTUAL NODES (for better distribution):
  Problem: servers might cluster on the ring → uneven distribution.
  Solution: each server has multiple "virtual nodes" (replica positions on the ring).
  
  Server A: positions 10, 25, 70, 90
  Server B: positions 15, 40, 65, 85
  
  Each server handles 25% of the ring on average.
  More virtual nodes = more even distribution.
  
  Cassandra, Redis Cluster, and Riak all use virtual nodes.
```

```python
import hashlib
import bisect

class ConsistentHashRing:
    """
    Consistent hash ring for distributing keys across nodes.
    Uses virtual nodes for even distribution.
    """
    
    def __init__(self, replicas: int = 150):
        self.replicas = replicas  # virtual nodes per physical node
        self.ring: dict[int, str] = {}
        self.sorted_keys: list[int] = []
    
    def _hash(self, key: str) -> int:
        return int(hashlib.md5(key.encode()).hexdigest(), 16)
    
    def add_node(self, node: str) -> None:
        """Add a node with `replicas` virtual positions on the ring."""
        for i in range(self.replicas):
            virtual_key = self._hash(f"{node}:vnode:{i}")
            self.ring[virtual_key] = node
            bisect.insort(self.sorted_keys, virtual_key)
    
    def remove_node(self, node: str) -> None:
        """Remove a node and all its virtual positions."""
        for i in range(self.replicas):
            virtual_key = self._hash(f"{node}:vnode:{i}")
            del self.ring[virtual_key]
            self.sorted_keys.remove(virtual_key)
    
    def get_node(self, key: str) -> str:
        """Find which node a key maps to."""
        if not self.ring:
            raise ValueError("Ring is empty")
        
        key_hash = self._hash(key)
        # Find first node clockwise from this key's position
        idx = bisect.bisect(self.sorted_keys, key_hash)
        if idx == len(self.sorted_keys):
            idx = 0  # wrap around
        return self.ring[self.sorted_keys[idx]]

# USAGE: distribute cache keys across Redis nodes
ring = ConsistentHashRing(replicas=150)
for server in ["redis-1", "redis-2", "redis-3"]:
    ring.add_node(server)

# Route a key
node = ring.get_node("user:12345:session")
print(f"user:12345:session → {node}")  # redis-2

# Add a 4th server
ring.add_node("redis-4")
# Only ~25% of keys move to redis-4, not 100%

# REAL-WORLD USE:
# Redis Cluster uses consistent hashing (with 16,384 slots)
# Cassandra uses consistent hashing for partition key → node mapping
# CDN providers use consistent hashing for URL → edge node mapping
```

## 4. Senior Engineer Perspective

**Sharding is a last resort:**

Before sharding, exhaust all vertical scaling options, read replicas, caching, and query optimization. Sharding creates enormous operational complexity: cross-shard queries become very difficult, cross-shard transactions require sagas, and resharding is painful.

Teams that shard too early spend enormous engineering effort on sharding infrastructure that was never actually needed. Only shard when you've confirmed that a single server's capacity is genuinely insufficient.

**Consistent hashing for caching:**

Even if you don't shard your database, consistent hashing is valuable for cache clusters. When you add a Redis node to your cluster, consistent hashing ensures only a fraction of cache keys become misses (hitting the database). Without it: all cache keys become misses simultaneously → cache stampede → database overload → outage.
