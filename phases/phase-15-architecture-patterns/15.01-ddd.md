## 1. What Is It? (Theory)

Domain-Driven Design (DDD) is a software development approach by Eric Evans (2003, "Domain-Driven Design: Tackling Complexity in the Heart of Software") that centers the design of software around the **business domain** — the area of activity, knowledge, and expertise that the software supports.

DDD is not a technology. It's a way of thinking about software that puts business problems at the center, rather than technical solutions. The result: software that closely models the real world it's meant to serve, making it easier to change as the business evolves.

**The problem DDD solves:**

```
TRADITIONAL APPROACH:
  Engineers talk to business stakeholders.
  Business says: "We need to manage customer accounts."
  Engineer hears: "I need a `customers` table and a CRUD API."
  
  Business says: "Accounts need to handle credit limits."
  Engineer hears: "I need a credit_limit column."
  
  Business says: "We do 'account holds' for fraud."
  Engineer hears: "I need a hold table."
  
  Over time: the code accumulates business logic scattered across
  data structures the business doesn't recognize.
  "Which table has the current balance?" requires understanding
  10 tables and their relationships.
  
  The software and the business speak DIFFERENT LANGUAGES.

DDD APPROACH:
  Business and engineers develop a SHARED LANGUAGE.
  Code uses the EXACT words the business uses.
  "Account", "Hold", "CreditLimit" — these mean the same thing
  in the business and in the code.
  
  When the business changes, engineers understand the change.
  When engineers discuss code, business understands the discussion.
```

**The real-world analogy:**

Building a medical system without medical domain knowledge leads to code that stores "patient states" and "appointment slots" but doesn't capture the clinical logic correctly. A doctor explains their workflow using medical terminology. DDD says: **use those exact terms in the code**. Don't translate "episode of care" to `treatment_session` — call it an `EpisodeOfCare` in the code. The code should read like a doctor wrote it.

## 2. How It Actually Works (Deep Mechanics)

### The Ubiquitous Language

The most important DDD concept: a shared, carefully crafted language used by BOTH technical and business team members, in conversations AND in code.

```
BEFORE UBIQUITOUS LANGUAGE:
  
  Business says: "When a customer places an order, we need to check 
                  if they're in good standing."
  
  Engineer asks: "Do you mean check if they have a VALID USER STATUS 
                  in the ACCOUNT TABLE?"
  
  → Constant translation needed. Bugs emerge from translation errors.
  → Code: user.status == "active" — what does "active" mean?
     Does "active" include accounts under review? What about suspended?

AFTER UBIQUITOUS LANGUAGE:
  
  Team agrees: a "Customer" has a "CreditStatus" which can be:
  GOOD_STANDING, UNDER_REVIEW, SUSPENDED, CLOSED
  
  "Place order" → requires Customer.creditStatus == GOOD_STANDING
  
  Code reflects this exactly:
  if customer.credit_status != CreditStatus.GOOD_STANDING:
      raise OrderRejected("Customer not in good standing")
  
  → Business can read this code and verify it's correct.
  → Engineers can discuss it with business without translation.
  → The test says: "Customer in good standing can place order"
    Not: "Customer with status='active' can insert order record"

HOW TO BUILD UBIQUITOUS LANGUAGE:
  1. Identify: what are the core business concepts?
     (For an e-commerce: Product, Order, Customer, Inventory, Shipment)
  
  2. Agree on precise definitions:
     - "Customer" vs "User"? (In this system: a Customer has placed an order, a User has an account but hasn't ordered)
     - "Order" vs "Purchase"? (Order = intent, Purchase = completed payment)
     - "Available" vs "In Stock"? (Available = can be purchased, In Stock = physically present)
  
  3. Use these exact terms everywhere:
     → In class/method names
     → In database table/column names
     → In API endpoint names
     → In meeting discussions
     → In documentation
  
  4. Continuously refine:
     When business uses a new term: add it to the language.
     When engineers use a technical term: translate it to business language.
```

### Bounded Contexts in Depth

We covered bounded contexts in [14.11 — Service decomposition](../phase-14-distributed-microservices/14.11-service-decomposition.md). Here's the DDD perspective:

```
A BOUNDED CONTEXT is an explicit boundary within which a domain model applies.

  Inside the boundary:
  → The ubiquitous language is consistent and precise.
  → Every term has exactly one meaning.
  → The model serves the needs of that context.
  
  At the boundary:
  → Explicit translation/mapping between models.
  → Each context may have different "User" objects with different attributes.

REAL EXAMPLE: "Order" in an e-commerce company

  Sales Context:
  Order = {id, customer_id, line_items, total, created_at, status}
  "Order" represents a customer's purchase intent.
  Status: pending → confirmed → shipped → delivered
  
  Finance Context:
  Order = {id, invoice_id, payment_method, tax_amount, refund_status}
  "Order" represents a financial transaction to be recorded.
  Finance doesn't care about line_items — they care about totals and taxes.
  
  Logistics Context:
  Order = {id, weight_kg, dimensions, warehouse_id, delivery_address}
  "Order" represents a physical shipment to be prepared.
  Logistics doesn't care about payment — they care about what to pack and where to send it.
  
  SAME WORD, THREE DIFFERENT MODELS.
  Don't fight this — embrace it.
  Each context owns its Order model.
  Context boundaries map to service/module boundaries.
```

## 3. The Strategic DDD Patterns

### Domain Classification

```
CORE DOMAIN:
  The area where your business has its competitive advantage.
  The code that makes your product unique.
  
  E-commerce: recommendation engine, dynamic pricing, fraud detection
  Banking: credit scoring model, risk management
  Health: clinical decision support
  
  → Invest your best engineers here.
  → Write it cleanly, test it thoroughly.
  → This is where DDD's complexity pays off most.

SUPPORTING SUBDOMAIN:
  Important but not your competitive differentiator.
  Other companies also need this. It's necessary, not unique.
  
  E-commerce: order management, inventory tracking, shipping calculations
  Banking: transaction processing, account management
  
  → Good enough engineering.
  → Could use off-the-shelf components with customization.
  → Consider open-source solutions.

GENERIC SUBDOMAIN:
  Commodity functionality that's identical across many companies.
  
  E-commerce: PDF invoice generation, email delivery, payment processing
  Banking: regulatory reporting, audit logging, identity verification
  
  → Buy, don't build.
  → Stripe for payments. DocRaptor for PDFs. SendGrid for email.
  → Investing engineering here is waste — it's not your advantage.
```

### Context Mapping: How Bounded Contexts Relate

```
CONTEXT RELATIONSHIPS:

PARTNERSHIP:
  Two teams coordinate their work closely.
  Changes in one affect the other — they plan together.
  Used when: teams have strong mutual dependency and work together.

SHARED KERNEL:
  Two contexts share a small portion of the domain model.
  Changes to the shared part require coordination.
  Used when: duplication would be worse than coordination overhead.

CUSTOMER-SUPPLIER:
  Upstream (supplier) provides API for downstream (customer).
  Downstream depends on upstream's schedule.
  Used when: one team controls what another team depends on.

CONFORMIST:
  Downstream adopts upstream's model wholesale.
  No translation layer — downstream conforms to upstream.
  Used when: upstream is too large/inflexible to negotiate with.
  Example: integrating with a legacy ERP — you conform to its data model.

ANTI-CORRUPTION LAYER (ACL):
  Downstream translates upstream's model into its own model.
  Protects the downstream domain from being corrupted by upstream's concepts.
  Used when: upstream model is poor, different, or from a third party.
  See topic 15.10.

OPEN HOST SERVICE:
  A service provides a well-defined API for multiple consumers.
  The API is stable, versioned, public.
  Used when: a service is shared by many contexts.

PUBLISHED LANGUAGE:
  A well-documented data model used for communication.
  Consumers don't need to understand internals.
  Often combined with Open Host Service.
  Example: an event schema published by an order service.
```

## 4. The Tactical DDD Patterns

Strategic DDD (bounded contexts, ubiquitous language) answers "how should I structure the system?" Tactical DDD answers "how should I write the code within a bounded context?"

```
ENTITY:
  An object with a unique identity that persists over time.
  The same entity can change its attributes — it's still the same entity.
  
  class Order:
      id: str  # identity — this is what makes it "the same order"
      status: OrderStatus  # can change without changing identity
      total_cents: int  # can change (price adjustments)
  
  Two orders with the same id → same order (even if attributes differ)
  Two orders with different ids → different orders (even if identical)

VALUE OBJECT:
  An object defined by its attributes, not an identity.
  Immutable — if you change it, you create a new one.
  Two value objects with the same attributes are equal.
  
  @dataclass(frozen=True)  # immutable
  class Money:
      amount_cents: int
      currency: str
  
  a = Money(1000, "USD")
  b = Money(1000, "USD")
  a == b  # True (value equality)
  # a is b  # False (different objects, but equal by value)
  
  Value objects are great for: money, addresses, coordinates, ranges, dates.

AGGREGATE:
  A cluster of entities and value objects treated as a single unit.
  One entity is the AGGREGATE ROOT — the only entry point.
  All modifications must go through the root.
  
  class Order:  # Aggregate Root
      id: str
      items: list[OrderItem]  # inside the aggregate
      
      def add_item(self, product_id, quantity, price):
          # Only Order can add items — enforces business rules
          if self.status != OrderStatus.PENDING:
              raise OrderFrozen()
          self.items.append(OrderItem(product_id, quantity, price))
  
  # CORRECT: go through the aggregate root
  order.add_item(product_id, quantity, price)
  
  # WRONG: bypass the aggregate root
  order.items.append(OrderItem(product_id, quantity, price))  # NO!
  # This bypasses validation — breaks business invariants

DOMAIN SERVICE:
  Business logic that doesn't naturally belong to any entity.
  
  class PricingService:
      def calculate_order_total(self, order: Order, coupon: Coupon) -> Money:
          # Pricing logic doesn't belong to Order (it references external discount rules)
          # Doesn't belong to Coupon (it references the order)
          # Belongs to a Domain Service
          ...

DOMAIN EVENT:
  A record of something significant that happened in the domain.
  Named in past tense. Immutable.
  
  @dataclass(frozen=True)
  class OrderPlaced:
      order_id: str
      user_id: str
      total_cents: int
      occurred_at: datetime
  
  # Domain events are the "what happened" record.
  # Other parts of the system react to them.
```

## 5. Real-World Context

### How Uber Applied DDD

Uber's engineering blog describes their migration from a monolith to domain-driven microservices. They identified their core domain (the matching of drivers to riders — their competitive advantage), supporting domains (payments, maps, notifications), and generic domains (authentication, internal tools).

The ubiquitous language was critical: the term "Trip" meant very different things to the dispatch system (live ride), the billing system (financial transaction), and the analytics system (data point). They established explicit bounded contexts for each, with clear models within each context and translation at the boundaries.

### Eric Evans on DDD Maturity

Evans himself has noted that DDD is often applied at the wrong level: teams learn the tactical patterns (entities, value objects, repositories) but miss the strategic patterns (bounded contexts, ubiquitous language). The tactical patterns without the strategic patterns produce over-engineered code without better alignment with the business.

The most valuable parts of DDD are: ubiquitous language (always valuable regardless of scale) and bounded contexts (invaluable for any system with multiple teams or significant complexity).

## 6. Senior Engineer Perspective

**The common DDD mistake:**

Learning DDD and immediately applying all tactical patterns everywhere — aggregates, value objects, domain services — even for simple CRUD operations. This produces extremely complex code for simple problems.

DDD complexity is justified for your **core domain**. For a simple admin CRUD panel: just use the ORM. For the pricing engine that determines your margins: invest in DDD design.

**The ubiquitous language ROI:**

The cheapest, most valuable part of DDD is the ubiquitous language — it costs nothing beyond a few meetings to agree on terms. The payoff: engineers and product managers discuss the same concepts using the same words. Features are built correctly because the misunderstanding gap is eliminated.

Start here. Build the glossary. Enforce it in code. That alone is worth more than any tactical pattern.

**The bounded context as a deployment boundary:**

The cleanest microservices align exactly with DDD bounded contexts. When a bounded context becomes a service, the context's public interface becomes the service's API, the context's aggregate roots become the service's API resources, and the context's domain events become the service's published events.

If your microservices don't align with bounded contexts: you likely have wrong service boundaries (splitting things that should be together, or lumping things that should be separate).
