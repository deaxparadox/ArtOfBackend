## 1. Hexagonal Architecture (Ports & Adapters) (15.2)

### What Is It?

Hexagonal Architecture (Alistair Cockburn, 2005), also called "Ports and Adapters," organizes code so that the **core business logic is completely isolated from infrastructure** (databases, HTTP, message brokers, external APIs).

The name "hexagonal" is arbitrary — the hexagon represents the application core, with each side representing a port (interface point). The shape doesn't matter; the principle does.

**The problem it solves:**

```
TRADITIONAL LAYERED ARCHITECTURE:
  HTTP request → Controller → Service → Repository → Database
  
  The service layer knows about the database (imports ORM models).
  The service layer knows about HTTP (imports request/response objects).
  
  PROBLEMS:
  - Can't test business logic without a database running
  - Can't swap the database without rewriting services
  - Business logic is coupled to infrastructure choices
  - If you want to trigger business logic via a CLI: can't reuse services
    (they're coupled to HTTP request/response)

HEXAGONAL ARCHITECTURE:
  
  Core (business logic):
    Pure Python/Java/Go — no imports of frameworks, ORMs, or HTTP libraries.
    Only domain models and interfaces.
  
  Ports (interfaces):
    Abstract contracts the core expects from the outside world.
    "I need something that can store and retrieve orders."
    (NOT: "I need SQLAlchemy and a PostgreSQL connection")
  
  Adapters (implementations):
    Concrete implementations of ports using specific technologies.
    PostgreSQLOrderRepository implements OrderRepository (port).
    InMemoryOrderRepository also implements OrderRepository (for tests).
    
  The core never directly imports adapters.
  The core only knows about ports (interfaces).
```

### The Architecture in Code

```python
# ─── DOMAIN (core business logic, no infrastructure) ─────────────────────────

from abc import ABC, abstractmethod
from dataclasses import dataclass
from datetime import datetime

@dataclass
class Order:
    id: str
    user_id: str
    total_cents: int
    status: str

# PORT: what the core needs from storage infrastructure
class OrderRepository(ABC):
    @abstractmethod
    async def save(self, order: Order) -> None: ...
    
    @abstractmethod
    async def find_by_id(self, order_id: str) -> Order | None: ...
    
    @abstractmethod
    async def find_by_user(self, user_id: str) -> list[Order]: ...

# PORT: what the core needs from payment infrastructure
class PaymentGateway(ABC):
    @abstractmethod
    async def charge(self, user_id: str, amount_cents: int) -> str: ...  # returns transaction_id

# PORT: what the core needs from event publishing
class EventPublisher(ABC):
    @abstractmethod
    async def publish(self, event_type: str, payload: dict) -> None: ...

# DOMAIN SERVICE: pure business logic, only depends on ports
class OrderService:
    def __init__(
        self,
        order_repo: OrderRepository,       # port, not implementation
        payment_gateway: PaymentGateway,    # port, not implementation
        event_publisher: EventPublisher,    # port, not implementation
    ):
        self.order_repo = order_repo
        self.payment_gateway = payment_gateway
        self.event_publisher = event_publisher
    
    async def place_order(self, user_id: str, amount_cents: int) -> Order:
        # Pure business logic — no HTTP, no SQL, no Kafka
        if amount_cents <= 0:
            raise ValueError("Order amount must be positive")
        
        transaction_id = await self.payment_gateway.charge(user_id, amount_cents)
        
        order = Order(
            id=generate_id(),
            user_id=user_id,
            total_cents=amount_cents,
            status="confirmed",
        )
        await self.order_repo.save(order)
        await self.event_publisher.publish("order.placed", {"order_id": order.id})
        
        return order

# ─── ADAPTERS (infrastructure implementations of ports) ──────────────────────

# DATABASE ADAPTER
import asyncpg

class PostgreSQLOrderRepository(OrderRepository):
    def __init__(self, pool: asyncpg.Pool):
        self.pool = pool
    
    async def save(self, order: Order) -> None:
        async with self.pool.acquire() as conn:
            await conn.execute(
                "INSERT INTO orders (id, user_id, total_cents, status) VALUES ($1,$2,$3,$4)",
                order.id, order.user_id, order.total_cents, order.status
            )
    
    async def find_by_id(self, order_id: str) -> Order | None:
        async with self.pool.acquire() as conn:
            row = await conn.fetchrow("SELECT * FROM orders WHERE id=$1", order_id)
        return Order(**row) if row else None

# PAYMENT ADAPTER
class StripePaymentGateway(PaymentGateway):
    async def charge(self, user_id: str, amount_cents: int) -> str:
        return await stripe_client.charge(user_id, amount_cents)

# IN-MEMORY ADAPTER (for tests — no database needed)
class InMemoryOrderRepository(OrderRepository):
    def __init__(self): self.orders: dict[str, Order] = {}
    async def save(self, order: Order): self.orders[order.id] = order
    async def find_by_id(self, order_id): return self.orders.get(order_id)

# HTTP ADAPTER
from fastapi import FastAPI

app = FastAPI()

async def get_order_service():
    # Dependency injection — wire the ports to their adapters
    return OrderService(
        order_repo=PostgreSQLOrderRepository(db_pool),
        payment_gateway=StripePaymentGateway(),
        event_publisher=KafkaEventPublisher(kafka_producer),
    )

@app.post("/orders")
async def create_order(request: CreateOrderRequest, 
                        svc: OrderService = Depends(get_order_service)):
    order = await svc.place_order(request.user_id, request.amount_cents)
    return {"order_id": order.id}
```

**The key insight:** OrderService has no imports from FastAPI, asyncpg, Stripe, or Kafka. It only imports abstract interfaces. This means:
- Tests inject `InMemoryOrderRepository` and `FakePaymentGateway` — no real database or Stripe needed
- Swap PostgreSQL for MongoDB: write a new adapter, change the wiring
- Add CLI trigger: wire OrderService to a CLI adapter, reuse all business logic

## 2. Clean Architecture (15.3)

Clean Architecture (Robert C. Martin, 2017) is a generalization of Hexagonal Architecture, Onion Architecture, and similar patterns. It defines layers where **dependencies point only inward** — inner layers know nothing about outer layers.

```
CLEAN ARCHITECTURE LAYERS:

  ┌─────────────────────────────────────────────────────┐
  │  Frameworks & Drivers                               │
  │  (FastAPI, SQLAlchemy, Kafka, Stripe SDK)           │
  │  ┌───────────────────────────────────────────────┐  │
  │  │  Interface Adapters                           │  │
  │  │  (Controllers, Repositories, Presenters)      │  │
  │  │  ┌─────────────────────────────────────────┐  │  │
  │  │  │  Application Business Rules             │  │  │
  │  │  │  (Use Cases / Application Services)     │  │  │
  │  │  │  ┌───────────────────────────────────┐  │  │  │
  │  │  │  │  Enterprise Business Rules        │  │  │  │
  │  │  │  │  (Entities, Value Objects,        │  │  │  │
  │  │  │  │   Domain Services)                │  │  │  │
  │  │  │  └───────────────────────────────────┘  │  │  │
  │  │  └─────────────────────────────────────────┘  │  │
  │  └───────────────────────────────────────────────┘  │
  └─────────────────────────────────────────────────────┘

DEPENDENCY RULE: dependencies point INWARD ONLY.
  Entities: depend on nothing external
  Use Cases: depend on Entities only
  Adapters: depend on Use Cases and Entities
  Frameworks: depend on Adapters

The inner layers are stable and can be reused.
The outer layers change frequently (frameworks, databases change).
```

### Clean Architecture vs. Hexagonal Architecture

```
HEXAGONAL ARCHITECTURE:
  Binary: inside vs. outside the hexagon
  Inside: domain logic
  Outside: everything that communicates with it (ports & adapters)
  
  Simpler conceptually. Good starting point.

CLEAN ARCHITECTURE:
  More layers inside: Entity layer + Use Case layer + Interface Adapter layer
  More granular: separates "what the business does" (use cases)
  from "what the business is" (entities)
  
  More overhead. Justified for complex, large codebases.
  Often overkill for smaller services.

WHEN EACH IS APPROPRIATE:
  Hexagonal: most applications. Isolates business logic from infrastructure.
  Clean Architecture: large teams, complex domains where the extra layer separation helps.
  
  Both share the same fundamental principle: business logic at the center,
  infrastructure at the edges. Implementation details vary.
```

## 3. Real-World Context

### FastAPI + Hexagonal Architecture in Practice

The pattern used by many Python backend teams:

```
app/
├── domain/                     ← core business logic
│   ├── models.py               ← Order, User, Product (entities)
│   ├── repositories.py         ← abstract repository interfaces (ports)
│   ├── services.py             ← OrderService, PaymentService
│   └── events.py               ← domain events
│
├── adapters/                   ← infrastructure implementations
│   ├── repositories/           ← concrete repository adapters
│   │   ├── postgres.py         ← PostgreSQLOrderRepository
│   │   └── redis.py            ← RedisCacheRepository
│   ├── gateways/               ← external service adapters
│   │   ├── stripe.py           ← StripePaymentGateway
│   │   └── sendgrid.py         ← SendGridEmailAdapter
│   └── messaging/              ← event publishing adapters
│       └── kafka.py            ← KafkaEventPublisher
│
├── api/                        ← HTTP adapter
│   ├── routers/
│   └── dependencies.py         ← dependency injection wiring
│
└── tests/
    ├── domain/                 ← pure unit tests (no infrastructure)
    └── integration/            ← tests with real DB/Redis
```

## 4. Senior Engineer Perspective

**The testability argument:**

The best reason to adopt hexagonal architecture is testability. When your OrderService imports directly from SQLAlchemy and Stripe, testing it requires:
- A running PostgreSQL database
- Valid Stripe test credentials
- Network access to Stripe's test API
- Careful setup/teardown of test data

With hexagonal architecture, you inject in-memory fakes:

```python
async def test_order_placement_charges_card():
    fake_payment = FakePaymentGateway()
    fake_repo = InMemoryOrderRepository()
    service = OrderService(fake_repo, fake_payment, FakeEventPublisher())
    
    order = await service.place_order("user-123", 5000)
    
    assert order.status == "confirmed"
    assert fake_payment.was_charged("user-123", 5000)
    assert await fake_repo.find_by_id(order.id) is not None
```

This test runs in milliseconds, needs no infrastructure, and tests exactly the business logic.

**The over-application warning:**

Not every service needs hexagonal architecture. A simple CRUD service with no business logic (just reading/writing to a database) gets no benefit from the pattern — it adds boilerplate without improving anything. Apply where business logic is complex enough to be worth isolating.
