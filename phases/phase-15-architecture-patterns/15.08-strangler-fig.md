## 1. Strangler Fig Pattern (15.8)

### What Is It?

The Strangler Fig pattern (Martin Fowler, 2004) is a strategy for incrementally replacing a legacy system with a new one, without a big-bang rewrite. New functionality is built in a new system while old functionality continues to work in the legacy system. Gradually, the new system "strangles" the legacy system.

Named after the strangler fig tree — a parasitic plant that grows around another tree, eventually replacing it.

```
THE PROBLEM WITH BIG-BANG REWRITES:
  "Let's rewrite everything from scratch in [modern technology]."
  
  Reality:
  - Takes 12-24 months
  - Legacy keeps getting bugs fixed while new system is built
  - Feature parity is a moving target
  - When you "flip the switch": many edge cases discovered
  - High risk: all or nothing
  
  Most big-bang rewrites fail or are abandoned.
  The legacy system lives on (often renamed "the old system that we're replacing soon").

THE STRANGLER FIG APPROACH:
  
  Phase 1: New system exists alongside legacy.
    New features: built in new system.
    Old features: still served by legacy.
    Router/proxy: directs traffic to appropriate system.
  
  Phase 2: Migrate specific functionality.
    Pick one module or feature.
    Build it in the new system.
    Route that feature's traffic to the new system.
    Legacy still handles everything else.
  
  Phase 3: Repeat until legacy handles nothing.
    Each migration is independent.
    Each migration delivers value immediately.
    Rollback per feature is possible.
  
  Phase 4: Decommission the legacy.
    Once all traffic is routed to the new system.
    Legacy becomes a backup, then is shut down.
```

### Implementation with HTTP Proxy

```python
# STRANGLER FIG PROXY:
# New routes go to new service.
# Old routes proxy to legacy system.
# Gradually move routes from legacy to new service.

from fastapi import FastAPI
import httpx

app = FastAPI()

LEGACY_BASE_URL = "http://legacy-system:8080"
NEW_SERVICE_URL = "http://new-order-service:8000"

# NEW: orders endpoint migrated to new service
@app.api_route("/api/orders/{path:path}", methods=["GET", "POST", "PUT"])
async def orders_proxy(request: Request, path: str):
    return await proxy_to(NEW_SERVICE_URL, request)

# LEGACY: users still in old system (not yet migrated)
@app.api_route("/api/users/{path:path}", methods=["GET", "POST", "PUT"])
async def users_proxy(request: Request, path: str):
    return await proxy_to(LEGACY_BASE_URL, request)

# LEGACY: products still in old system
@app.api_route("/api/products/{path:path}", methods=["GET", "POST", "PUT"])
async def products_proxy(request: Request, path: str):
    return await proxy_to(LEGACY_BASE_URL, request)

async def proxy_to(base_url: str, request: Request):
    async with httpx.AsyncClient() as client:
        url = f"{base_url}{request.url.path}"
        resp = await client.request(
            method=request.method,
            url=url,
            headers=dict(request.headers),
            content=await request.body(),
        )
        return Response(
            content=resp.content,
            status_code=resp.status_code,
            headers=dict(resp.headers),
        )

# Migration progress:
# Week 1: orders migrated → new service handles /api/orders/*
# Week 4: products migrated → update products_proxy to new URL
# Week 8: users migrated → update users_proxy to new URL
# Week 12: legacy decommissioned → remove proxy entirely
```

## 2. Anti-Corruption Layer (15.10)

The Anti-Corruption Layer (ACL) is a translation layer that protects your domain model from being "corrupted" by an external system's model.

```
THE CORRUPTION PROBLEM:

Your domain: 
  Order, Customer, Product — clean, well-designed model

External ERP (legacy):
  invoice, custno, itemcode — poorly named, different semantics
  "custno" = your Customer.id + extra data
  "itemcode" = your Product.sku + different encoding
  
  If you use the ERP's concepts directly in your code:
  → Your code becomes polluted with ERP terminology
  → Business rules get mixed with ERP-specific translations
  → Changing ERP means changing your domain code

WITH ANTI-CORRUPTION LAYER:
  ERP → ACL (translation) → Your Domain
  
  class ERPAdapter:  # the anti-corruption layer
      async def create_invoice(self, order: Order) -> None:
          """Translates your Order into ERP invoice format."""
          erp_data = {
              "custno": self._translate_customer_id(order.customer_id),
              "itemcode": self._translate_product_code(order.product_id),
              "qty": order.quantity,
              "price": self._translate_price(order.unit_price_cents),
          }
          await self.erp_client.create_invoice(erp_data)
      
      def _translate_customer_id(self, customer_id: str) -> str:
          # ERP has different ID format
          return f"C{customer_id.replace('-', '')[:8]}"
  
  Your domain knows nothing about ERP formats.
  The ACL handles all the messy translation.
  Changing ERP: only update the ACL, not your domain.
```

**When to use ACL:**
- Integrating with legacy systems that have poor models
- Integrating with third-party systems (Salesforce, SAP, etc.)
- Any time an external model would "corrupt" your clean domain model
- When the external model changes frequently

## 3. Shared Kernel (15.11)

The Shared Kernel is a small, explicitly agreed-upon part of the domain model shared between two bounded contexts.

```
SHARED KERNEL:
  Two bounded contexts that are independent but need to share something.
  The shared part is explicitly designated and versioned.
  Changes require agreement from both teams.

EXAMPLE:
  Order Context and Fulfillment Context both need:
  - Address value object
  - Currency/money concepts
  - Product ID format
  
  These go in a shared kernel:
  
  shared/
  ├── money.py      ← Money value object (used by both contexts)
  ├── address.py    ← Address value object
  └── product_id.py ← ProductId type and validation
  
  Order Context imports from shared/
  Fulfillment Context imports from shared/
  
  CONSTRAINT: changes to shared kernel require coordination between both teams.
  The shared kernel must remain small.
  If it grows large: it becomes a source of coupling, not reduction.
  
SHARED KERNEL vs SHARED DATABASE:
  Shared kernel: a small set of value objects, explicitly agreed upon.
  Shared database: entire database schemas shared → anti-pattern in microservices.
  
  Use shared kernel for: common types, common validation logic
  Never use shared database for: service-level isolation
```

## 4. Twelve-Factor App Methodology (15.12)

The Twelve-Factor App (Heroku, Adam Wiggins, 2011) is a methodology for building software-as-a-service apps that are:
- Maximally portable across environments
- Deployable on modern cloud platforms
- Scalable without significant changes to architecture or development

```
THE TWELVE FACTORS:

I.   CODEBASE:
     One codebase tracked in version control. Many deploys.
     Never: different codebases for dev vs prod.
     
II.  DEPENDENCIES:
     Explicitly declare and isolate dependencies.
     Never assume a system-wide tool (like imagemagick).
     Use requirements.txt, package.json, go.mod.
     
III. CONFIG:
     Store config in the environment.
     Everything that varies between deploys: DATABASE_URL, API_KEY, LOG_LEVEL.
     Never: hardcoded in code or committed to version control.
     
IV.  BACKING SERVICES:
     Treat backing services as attached resources.
     Database, Redis, S3, SMTP server: all attached via URL in config.
     Can swap dev MySQL for prod MySQL without code changes.
     
V.   BUILD, RELEASE, RUN:
     Strictly separate build stage, release stage, run stage.
     Build: convert code into an executable bundle.
     Release: combine build with config. Releases are immutable.
     Run: execute the app in the execution environment.
     Never: modify code at runtime.
     
VI.  PROCESSES:
     Execute as stateless processes.
     Share nothing. Any data that needs to persist: external state store.
     Never: assume in-memory session lives between requests.
     
VII. PORT BINDING:
     Export services via port binding.
     App is self-contained. Web server is bundled in the app.
     Never: inject a web server container. Use embedded server (uvicorn, etc.).
     
VIII.CONCURRENCY:
     Scale out via the process model.
     Different process types for web, worker, clock.
     Scale web processes for more throughput, worker processes for more jobs.
     
IX.  DISPOSABILITY:
     Maximize robustness with fast startup and graceful shutdown.
     Start up in seconds. Handle SIGTERM gracefully.
     Crash-only software: processes can die at any moment.
     
X.   DEV/PROD PARITY:
     Keep development, staging, production as similar as possible.
     Same backing services (PostgreSQL in dev, not SQLite).
     Same environment variables pattern.
     Deploy frequently (minutes, not months).
     
XI.  LOGS:
     Treat logs as event streams.
     Never write to log files from the app.
     Write to stdout. Let the execution environment route and store.
     
XII. ADMIN PROCESSES:
     Run admin/management tasks as one-off processes.
     Database migrations, one-time scripts: run as one-off processes.
     Never: run via HTTP endpoint in a request handler.
     Use the same codebase and config as the production processes.

MOST VIOLATED IN PRACTICE:
  III (config): credentials hardcoded or in .env files committed to git
  VI (stateless): sessions or state stored in-process memory
  X (dev/prod parity): developers use SQLite, production uses PostgreSQL
  XI (logs): apps write to files instead of stdout
```

## 5. Senior Engineer Perspective

**The Strangler Fig is the only safe way to migrate:**

Big-bang rewrites fail at a stunning rate. The Strangler Fig pattern's power is that it allows continuous delivery of value during the migration. Each step is independently valuable, independently reversible, and independently testable. No "day 1 where everything switches at once and many things break."

**The Twelve-Factor App is a checklist, not a religion:**

Some factors (config in environment, stateless processes, logs to stdout) are universally correct and should always be followed. Others (one codebase, port binding) may have legitimate exceptions in specific architectures. Treat it as a checklist of common mistakes to avoid, not a dogmatic set of rules.
