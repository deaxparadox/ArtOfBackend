## 1. Repository Pattern (15.4)

### What Is It?

The Repository pattern abstracts data access behind a collection-like interface. From the domain's perspective, a repository looks like a collection of domain objects that you can add to, remove from, and query — without knowing whether they're stored in PostgreSQL, MongoDB, or memory.

**The real-world analogy:**

A library has a collection of books. You don't care whether they're organized by Dewey Decimal, stored in a warehouse, or digitized. You interact with the library through simple operations: "find me all books by Tolkien," "add this new book," "remove this damaged copy." The library's internal organization is hidden.

The repository is the library. Your code is the patron.

```
WITHOUT REPOSITORY (direct ORM dependency):
  
  class OrderService:
      async def get_user_orders(self, user_id: int):
          # Directly uses SQLAlchemy — coupled to PostgreSQL
          return await db.execute(
              select(Order).where(Order.user_id == user_id)
          )
  
  Problems:
  - Tests require a real database
  - Changing to MongoDB requires rewriting OrderService
  - Business logic mixed with data access syntax

WITH REPOSITORY:
  
  class OrderService:
      def __init__(self, orders: OrderRepository):
          self.orders = orders
      
      async def get_user_orders(self, user_id: int):
          return await self.orders.find_by_user(user_id)
  
  Benefits:
  - Tests use InMemoryOrderRepository (no database)
  - Swap PostgreSQL for MongoDB: only change the repository implementation
  - Business logic is clean — reads like business intent
```

### Implementation

```python
from abc import ABC, abstractmethod

# REPOSITORY INTERFACE (lives in domain layer)
class OrderRepository(ABC):
    
    @abstractmethod
    async def find_by_id(self, order_id: str) -> Order | None:
        """Find order by ID. Returns None if not found."""
        ...
    
    @abstractmethod
    async def find_by_user(self, user_id: str) -> list[Order]:
        """Find all orders for a user."""
        ...
    
    @abstractmethod
    async def find_by_status(self, status: str) -> list[Order]:
        """Find orders with a specific status."""
        ...
    
    @abstractmethod
    async def save(self, order: Order) -> None:
        """Persist an order (insert or update)."""
        ...
    
    @abstractmethod
    async def delete(self, order_id: str) -> None:
        """Remove an order."""
        ...

# CONCRETE IMPLEMENTATION (lives in adapter/infrastructure layer)
import asyncpg

class PostgreSQLOrderRepository(OrderRepository):
    def __init__(self, pool: asyncpg.Pool):
        self._pool = pool
    
    async def find_by_id(self, order_id: str) -> Order | None:
        async with self._pool.acquire() as conn:
            row = await conn.fetchrow(
                "SELECT * FROM orders WHERE id=$1",
                order_id
            )
        return self._map_row(row) if row else None
    
    async def find_by_user(self, user_id: str) -> list[Order]:
        async with self._pool.acquire() as conn:
            rows = await conn.fetch(
                "SELECT * FROM orders WHERE user_id=$1 ORDER BY created_at DESC",
                user_id
            )
        return [self._map_row(row) for row in rows]
    
    async def save(self, order: Order) -> None:
        async with self._pool.acquire() as conn:
            await conn.execute("""
                INSERT INTO orders (id, user_id, total_cents, status, created_at)
                VALUES ($1, $2, $3, $4, $5)
                ON CONFLICT (id) DO UPDATE SET
                    status = EXCLUDED.status,
                    total_cents = EXCLUDED.total_cents
            """, order.id, order.user_id, order.total_cents, order.status, order.created_at)
    
    def _map_row(self, row) -> Order:
        return Order(
            id=row["id"],
            user_id=row["user_id"],
            total_cents=row["total_cents"],
            status=row["status"],
        )

# IN-MEMORY IMPLEMENTATION (for tests)
class InMemoryOrderRepository(OrderRepository):
    def __init__(self):
        self._orders: dict[str, Order] = {}
    
    async def find_by_id(self, order_id: str) -> Order | None:
        return self._orders.get(order_id)
    
    async def find_by_user(self, user_id: str) -> list[Order]:
        return [o for o in self._orders.values() if o.user_id == user_id]
    
    async def save(self, order: Order) -> None:
        self._orders[order.id] = order
    
    async def delete(self, order_id: str) -> None:
        self._orders.pop(order_id, None)

# TEST EXAMPLE (no database needed):
async def test_place_order():
    repo = InMemoryOrderRepository()
    service = OrderService(repo, FakePaymentGateway())
    
    order = await service.place_order("user-123", 5000)
    
    found = await repo.find_by_id(order.id)
    assert found is not None
    assert found.status == "confirmed"
```

## 2. Unit of Work Pattern (15.5)

### What Is It?

The Unit of Work pattern tracks changes to domain objects in a business operation and commits them atomically as a single database transaction at the end.

The problem it solves: when a service operation modifies multiple aggregates, you need to ensure all changes commit together or all roll back. Without Unit of Work, you'd explicitly manage transactions everywhere.

```
WITHOUT UNIT OF WORK:
  async def transfer_funds(from_account_id, to_account_id, amount):
      async with db.transaction():
          source = await account_repo.find_by_id(from_account_id)
          target = await account_repo.find_by_id(to_account_id)
          
          source.debit(amount)
          target.credit(amount)
          
          # Manual save for each modified aggregate
          await account_repo.save(source)
          await account_repo.save(target)
      # Transaction management is explicit everywhere

WITH UNIT OF WORK:
  async def transfer_funds(from_account_id, to_account_id, amount):
      async with uow:  # starts transaction automatically
          source = await uow.accounts.find_by_id(from_account_id)
          target = await uow.accounts.find_by_id(to_account_id)
          
          source.debit(amount)
          target.credit(amount)
          # No explicit save — Unit of Work tracks changes
      # Context manager exit commits everything atomically
```

```python
from contextlib import asynccontextmanager

class SqlAlchemyUnitOfWork:
    """
    Unit of Work backed by SQLAlchemy async session.
    Tracks all changes within a transaction boundary.
    """
    
    def __init__(self, session_factory):
        self.session_factory = session_factory
    
    async def __aenter__(self):
        self.session = self.session_factory()
        self.orders = SqlAlchemyOrderRepository(self.session)
        self.accounts = SqlAlchemyAccountRepository(self.session)
        return self
    
    async def __aexit__(self, exc_type, exc_val, exc_tb):
        if exc_type:
            await self.rollback()
        else:
            await self.commit()
        await self.session.close()
    
    async def commit(self):
        await self.session.commit()
        # Also collect and publish domain events from all modified aggregates
        await self._publish_domain_events()
    
    async def rollback(self):
        await self.session.rollback()
    
    async def _publish_domain_events(self):
        """Collect domain events from all aggregates and publish them."""
        for aggregate in self.session.new | self.session.dirty:
            if hasattr(aggregate, '_pending_events'):
                for event in aggregate._pending_events:
                    await event_publisher.publish(event)
                aggregate._pending_events.clear()

# USAGE:
class FundsTransferService:
    def __init__(self, uow_factory):
        self.uow_factory = uow_factory
    
    async def transfer(self, from_id: str, to_id: str, amount: int):
        async with self.uow_factory() as uow:
            source = await uow.accounts.find_by_id(from_id)
            target = await uow.accounts.find_by_id(to_id)
            
            if source.balance < amount:
                raise InsufficientFunds()
            
            source.debit(amount)
            target.credit(amount)
            # UoW commits both changes atomically when context exits
```

## 3. Senior Engineer Perspective

**Repository vs. ORM:**

The Repository pattern doesn't replace ORMs — it wraps them. Your PostgreSQLOrderRepository uses SQLAlchemy internally. The repository is the interface; the ORM is the implementation detail.

This matters because: you can swap ORMs, or switch from ORM to raw SQL for specific performance-critical queries, without changing any code that uses the repository.

**Unit of Work and domain events:**

The Unit of Work is the natural place to collect and publish domain events. Each aggregate tracks its own pending events. The UoW, on commit, collects events from all modified aggregates and publishes them. This ensures: events are published only if the transaction succeeds, and all events from one business operation are published together.
