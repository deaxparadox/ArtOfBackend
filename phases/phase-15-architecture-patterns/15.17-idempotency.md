## 1. Idempotent APIs and At-Least-Once Delivery (15.17)

**Covered in depth in:**
- [0.10 — Idempotency](../phase-00-mental-models/0.10-idempotency.md) — the concept, why it exists, how to implement it
- [12.3 — Delivery guarantees](../phase-12-messaging-events/12.03-delivery-guarantees.md) — at-least-once vs. exactly-once in messaging

**The architectural pattern:**

```
IDEMPOTENCY KEY PATTERN (Stripe's approach):

API Design:
  Every non-idempotent operation accepts an idempotency key.
  Same key + same request → same result (no duplicate effect).
  
  POST /charges
  Headers:
    Idempotency-Key: a4e3d2b1-...  ← client-generated, unique per operation attempt
  
  Body: {amount: 5000, currency: "usd", ...}

Server Implementation:
  1. Check if idempotency_key already processed.
  2. If YES: return cached result (no new charge).
  3. If NO: process, store result with key, return result.
  
  Storage: Redis with TTL (24h) or database table with index on idempotency_key.

IDEMPOTENCY IN THE ARCHITECTURE:
  Every service-to-service call that has side effects → idempotency keys.
  Every message consumer → deduplication via message ID.
  Every webhook delivery → idempotency key in header.
  
  Design principle: the entire system is safe to retry at every level.
```

## 2. Outbox Pattern (15.18)

**Covered in depth in [12.20 — Outbox pattern](../phase-12-messaging-events/12.20-outbox.md)**

**The architectural significance:**

The Outbox pattern solves the dual-write problem — a fundamental challenge in event-driven architectures. Without it, you have a choice between:
- Write to DB + publish event (two separate operations that can partially fail)
- Distribute transactions across DB and message broker (2PC, fragile)

The Outbox pattern achieves reliability through:
```
SINGLE TRANSACTION:
  BEGIN;
    INSERT INTO orders (id, ...) VALUES (...);         ← business write
    INSERT INTO outbox (event_type, payload) VALUES (...); ← event write
  COMMIT;
  
  EITHER BOTH SUCCEED OR NEITHER DOES.
  
  Background process:
    Read unpublished outbox events.
    Publish to broker.
    Mark as published.

RESULT: reliable event publishing with no distributed transactions.
```

## 3. Change Data Capture (15.19)

**Covered in [12.10 — Kafka Connect and CDC](../phase-12-messaging-events/12.10-kafka-connect.md)**

**The architectural significance:**

CDC is the alternative to the Outbox pattern for existing systems. Instead of adding outbox writes to application code, CDC taps the database's replication log directly:

```
CDC ARCHITECTURE:

PostgreSQL WAL (write-ahead log)
    ↓ Debezium reads continuously
Kafka (event stream)
    ↓ Consumers subscribe
Elasticsearch (search index sync)
Redis (cache invalidation)
Analytics DB (data warehouse)
Audit Log (compliance)

ALL FROM THE SAME DATABASE CHANGES.
No application code changes for existing tables.

WHEN CDC > OUTBOX:
  ✓ Adding event-driven to an existing system without code changes
  ✓ Multiple systems need the same database changes
  ✓ Complex data models where outbox would need many table entries

WHEN OUTBOX > CDC:
  ✓ Full control over event schema (CDC events reflect DB schema, not business events)
  ✓ You want business-level events ("OrderPlaced") not DB-level events ("INSERT into orders")
  ✓ New system being built from scratch
```

## 4. Read-Through Architecture with Write-Behind (15.20)

This pattern combines two cache strategies to maximize both read performance and write performance:

```
READ-THROUGH:
  On cache miss: cache layer fetches from database, populates cache, returns to caller.
  Application never directly touches the database for reads.
  
  Application → Cache → (miss) → Database → Cache populated → Response
  
  Application code is simpler: always call cache, never database.
  Cache handles the database interaction.

WRITE-BEHIND (Write-Back):
  On write: update cache immediately, write to database asynchronously.
  
  Application → Cache (immediate, synchronous)
              → Database (async, background, batch possible)
  
  PROS: writes return fast (no database round-trip)
  CONS: if cache fails before DB write: data lost
  
  Use only when: some data loss is acceptable (metrics, counters, session data)
  Never use for: financial transactions, inventory counts, critical data

COMBINED PATTERN:
  
  class WriteBackCache:
      def __init__(self, cache: Redis, db: PostgreSQL, flush_interval: int = 5):
          self.cache = cache
          self.db = db
          self.dirty_keys: set = set()
          
          # Background task flushes dirty cache to DB every 5 seconds
          asyncio.create_task(self._flush_periodically(flush_interval))
      
      async def get(self, key: str) -> Any:
          """Read-through: cache → DB if miss."""
          value = await self.cache.get(key)
          if value is None:
              value = await self.db.get(key)
              await self.cache.set(key, value, ttl=300)
          return value
      
      async def set(self, key: str, value: Any) -> None:
          """Write-behind: cache immediately, DB async."""
          await self.cache.set(key, value, ttl=300)
          self.dirty_keys.add(key)  # mark for later flush
      
      async def _flush_periodically(self, interval: int):
          while True:
              await asyncio.sleep(interval)
              await self._flush_dirty_keys()
      
      async def _flush_dirty_keys(self):
          if not self.dirty_keys:
              return
          keys_to_flush = self.dirty_keys.copy()
          self.dirty_keys.clear()
          for key in keys_to_flush:
              value = await self.cache.get(key)
              if value:
                  await self.db.set(key, value)

WHEN THIS PATTERN APPLIES:
  ✓ High read-to-write ratio (reads 100x more common than writes)
  ✓ Read performance critical, some write latency acceptable
  ✓ Reads can tolerate eventual consistency (few hundred ms stale)
  ✓ Write-behind acceptable (some loss tolerable OR data can be reconstructed)
  
  Example: User activity counters (page views, click counts)
    Write-behind: update counter in Redis immediately (fast UI response)
    Flush to PostgreSQL: every 30 seconds (analytics doesn't need real-time)
    If Redis fails: lose last 30 seconds of counters (acceptable)
```
