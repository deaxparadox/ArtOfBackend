## 1. CQRS + DDD: The Natural Combination

CQRS and DDD work together naturally:

```
DDD AGGREGATE → COMMAND SIDE:
  Commands target specific aggregates.
  Aggregates enforce business invariants.
  Aggregates emit domain events on state change.
  
  PlaceOrderCommand → OrderAggregate.place()
    → Validates: sufficient inventory, valid user
    → Emits: OrderPlacedEvent
    → Returns: nothing (or the aggregate ID)

DOMAIN EVENTS → EVENT BUS:
  OrderPlacedEvent published to event bus
  
  Multiple read model projections subscribe:
  - UserOrderHistoryProjection
  - ProductSalesProjection
  - DailyRevenueProjection
  - SearchIndexProjection (Elasticsearch)

QUERY SIDE → READ MODELS:
  User opens orders page → query UserOrderHistoryProjection
  Admin opens dashboard → query DailyRevenueProjection
  User searches → query SearchIndexProjection

DEPENDENCY DIRECTION:
  Read models depend on domain events (event schema).
  Domain events depend on the domain model.
  Command side is independent of read models.
  → Adding new read model: zero changes to command side.
```

## 2. Event Sourcing + CQRS: The Combined Pattern

```
WRITE SIDE (Event Store):
  All state changes stored as immutable events.
  No UPDATE or DELETE — only APPEND.
  
  event_store table:
  | aggregate_type | aggregate_id | version | event_type    | payload |
  | Order          | ord-123      | 1       | OrderPlaced   | {...}   |
  | Order          | ord-123      | 2       | PaymentConfirmed | {...} |
  | Order          | ord-123      | 3       | OrderShipped  | {...}   |

READ SIDE (Projections/Views):
  Projections subscribe to events and build optimized read models.
  Stored separately (PostgreSQL table, Redis, Elasticsearch).
  
  order_summaries table (read model):
  | order_id | user_id | status   | total | last_updated |
  | ord-123  | usr-456 | shipped  | 5000  | 2024-06-15   |

THE POWER: REBUILD READ MODELS FROM SCRATCH.
  Projection has a bug? Fix it.
  Need a new type of analytics? Create a new projection.
  Both: replay all historical events through the new/fixed projection.
  Historical data never lost — events are permanent.
```

## 3. When to Use (and Not Use) These Patterns

```
USE CQRS WHEN:
  ✓ Read and write loads differ significantly (100:1 ratio or more)
  ✓ Multiple screens need the same data in radically different shapes
  ✓ Read performance needs denormalization that would complicate writes
  ✓ Using event sourcing (they pair naturally)

DON'T USE CQRS WHEN:
  ✗ Simple CRUD (adds complexity for no gain)
  ✗ Strong read-your-writes consistency required immediately everywhere
  ✗ Small team without bandwidth for eventual consistency handling

USE EVENT SOURCING WHEN:
  ✓ Complete audit trail required (finance, healthcare, legal)
  ✓ Time-travel debugging required ("what was the state at time T?")
  ✓ Replay capability needed (new projections, bug fixes)
  ✓ GDPR right to erasure needed (crypto-shredding approach)

DON'T USE EVENT SOURCING WHEN:
  ✗ Simple CRUD (massive over-engineering)
  ✗ Data changes very frequently (millions of events per aggregate per day = slow replay)
  ✗ Team unfamiliar with the paradigm
  ✗ Schema evolution isn't handled yet in your organization
```
