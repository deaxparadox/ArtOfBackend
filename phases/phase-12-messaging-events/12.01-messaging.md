## Why Messaging? (12.1-12.2)

```
SYNCHRONOUS (direct API calls):
  Service A calls Service B synchronously
  + Simple, immediate feedback
  - Service A is coupled to Service B's availability
  - If B is slow: A is slow
  - If B is down: A fails

ASYNCHRONOUS (messaging):
  Service A publishes event to queue
  Service B consumes from queue at its own pace
  + Temporal decoupling: A doesn't care if B is slow or down
  + Scale independently: add more B consumers without changing A
  + Replay: replay events from the beginning
  - More complex: need to handle partial failures, ordering, deduplication

MESSAGE QUEUE vs EVENT STREAM:
  Queue (SQS, RabbitMQ): message consumed and deleted by one consumer
  → Work distribution: job queue (any worker processes the job)
  
  Stream (Kafka, Kinesis): messages retained, consumers track their position
  → Event log: multiple consumer groups each process all events
  → Can replay from any point in time
```

## Delivery Guarantees (12.3)

```
AT-MOST-ONCE: fire and forget
  Producer sends, message may or may not arrive
  Use: metrics, logs (losing a few is acceptable)

AT-LEAST-ONCE: guaranteed delivery, possible duplicates
  Producer retries until acknowledged
  Consumer may receive same message twice
  Use: most production workloads (make consumers idempotent)

EXACTLY-ONCE: guaranteed, no duplicates
  Very hard in distributed systems
  Kafka transactions + idempotent producers achieve "effectively exactly-once"
  Use: financial transactions (deduplication via idempotency key)
```

## RabbitMQ (12.5-12.6)

```python
import pika
import json

# Producer
connection = pika.BlockingConnection(pika.URLParameters("amqp://rabbitmq:5672"))
channel = connection.channel()

# Exchange types: direct, topic, fanout, headers
channel.exchange_declare(exchange="events", exchange_type="topic", durable=True)

# Publish order event
channel.basic_publish(
    exchange="events",
    routing_key="order.created",   # topic: consumers bind with patterns like "order.*"
    body=json.dumps({"order_id": 123, "user_id": 456}),
    properties=pika.BasicProperties(
        delivery_mode=2,           # persistent (survives RabbitMQ restart)
        content_type="application/json",
    ),
)

# Consumer
channel.queue_declare(queue="order_processor", durable=True)
channel.queue_bind("order_processor", "events", routing_key="order.*")

def callback(ch, method, properties, body):
    event = json.loads(body)
    try:
        process_order(event["order_id"])
        ch.basic_ack(delivery_tag=method.delivery_tag)  # acknowledge: remove from queue
    except Exception as e:
        ch.basic_nack(delivery_tag=method.delivery_tag, requeue=True)  # retry

channel.basic_qos(prefetch_count=1)  # only one unacked message per worker
channel.basic_consume("order_processor", callback)
channel.start_consuming()
```

## Kafka (12.7-12.9)

```python
from confluent_kafka import Producer, Consumer

# KAFKA CONCEPTS:
# Topic: named stream of events (like a table in a database)
# Partition: parallel units within a topic (events in one partition are ordered)
# Offset: position within a partition (consumer tracks its offset)
# Consumer group: set of consumers sharing the load (each partition → one consumer)

# Producer
producer = Producer({"bootstrap.servers": "kafka:9092"})

def delivery_callback(err, msg):
    if err:
        logger.error("Delivery failed", error=str(err))

producer.produce(
    topic="order-events",
    key=str(user_id).encode(),     # partitioning key (same user → same partition → ordered)
    value=json.dumps({
        "event_type": "order.created",
        "order_id": 123,
    }).encode(),
    callback=delivery_callback,
)
producer.flush()  # wait for all messages to be delivered

# Consumer
consumer = Consumer({
    "bootstrap.servers": "kafka:9092",
    "group.id": "order-processor",      # consumer group ID
    "auto.offset.reset": "earliest",    # start from beginning if no committed offset
    "enable.auto.commit": False,        # manual commit (at-least-once delivery)
})

consumer.subscribe(["order-events"])

while True:
    msg = consumer.poll(timeout=1.0)
    if msg is None: continue
    if msg.error(): continue

    event = json.loads(msg.value().decode())
    try:
        process_event(event)
        consumer.commit(message=msg)  # mark as processed
    except Exception as e:
        logger.error("Processing failed", error=str(e))
        # Don't commit — message will be redelivered
```

**Kafka vs. RabbitMQ:**

```
KAFKA:
  Log-based: messages retained (default 7 days, configurable forever)
  High throughput: millions of messages/second
  Consumer tracks its own offset (replay possible)
  Best for: event sourcing, analytics pipelines, audit logs, data streaming

RABBITMQ:
  Message-based: messages deleted after consumption
  Complex routing: exchanges + bindings (topic, fanout, direct, headers)
  Simpler semantics: just queues with routing rules
  Best for: task queues, job distribution, RPC patterns
  
RULE: Use Kafka when you need event replay or very high throughput.
      Use RabbitMQ for most task queue and messaging patterns.
```

## Celery (Python Task Queue) (12.24)

```python
# celery_app.py
from celery import Celery
import os

app = Celery("myapp",
    broker=os.environ["REDIS_URL"],
    backend=os.environ["REDIS_URL"],
)

app.conf.update(
    task_serializer="json",
    result_serializer="json",
    timezone="UTC",
    task_acks_late=True,         # ack after completion (not before = at-least-once)
    task_reject_on_worker_lost=True,  # re-queue if worker dies
    worker_prefetch_multiplier=1,     # one task at a time per worker (fairer)
)

# Define tasks
@app.task(
    bind=True,
    max_retries=3,
    default_retry_delay=60,  # seconds
)
def send_email(self, user_id: int, template: str, context: dict):
    try:
        user = User.objects.get(id=user_id)
        EmailService.send(user.email, template, context)
    except EmailServiceError as exc:
        raise self.retry(exc=exc, countdown=2 ** self.request.retries)  # exponential backoff

@app.task
def process_image(image_id: int):
    """CPU-bound task — should use separate queue with process pool workers."""
    image = Image.objects.get(id=image_id)
    image.generate_thumbnail()
    image.save()

# Calling tasks
# Async (recommended)
send_email.delay(user_id=123, template="welcome", context={"name": "Alice"})

# With options
send_email.apply_async(
    args=[123, "welcome", {"name": "Alice"}],
    countdown=60,      # delay 60 seconds
    expires=3600,      # expire if not processed in 1 hour
    queue="emails",    # specific queue
)
```

```bash
# Start Celery worker
celery -A myapp worker \
  --loglevel=info \
  --concurrency=4 \
  --queues=default,emails \
  --pool=prefork      # for CPU-bound tasks (use gevent for I/O-bound)

# Celery Beat (scheduler)
celery -A myapp beat --loglevel=info

# Monitor with Flower
celery -A myapp flower --port=5555
```

## Event Sourcing (12.19)

```python
# Event sourcing: store state as a sequence of events, not current state
# Current state = replay all events from beginning

# Events table (append-only log)
CREATE TABLE order_events (
    id         BIGSERIAL PRIMARY KEY,
    order_id   UUID NOT NULL,
    event_type VARCHAR(50) NOT NULL,  -- 'order_placed', 'payment_received', 'shipped'
    payload    JSONB NOT NULL,
    occurred_at TIMESTAMPTZ DEFAULT NOW()
);

# Append event (never UPDATE or DELETE)
async def place_order(order_id: str, user_id: int, items: list):
    await db.execute(
        "INSERT INTO order_events (order_id, event_type, payload) VALUES ($1, $2, $3)",
        order_id, "order_placed", json.dumps({"user_id": user_id, "items": items})
    )

# Reconstruct current state by replaying events
async def get_order_state(order_id: str) -> dict:
    events = await db.fetch(
        "SELECT * FROM order_events WHERE order_id=$1 ORDER BY id",
        order_id
    )
    state = {}
    for event in events:
        if event["event_type"] == "order_placed":
            state = {"status": "placed", **event["payload"]}
        elif event["event_type"] == "payment_received":
            state["status"] = "paid"
            state["payment"] = event["payload"]
        elif event["event_type"] == "shipped":
            state["status"] = "shipped"
            state["tracking"] = event["payload"]["tracking_number"]
    return state
```

**Event sourcing benefits:** Complete audit trail, replay to any point in time, easy to add new projections from existing events. **Cost:** More complex than CRUD, eventual consistency for read models, snapshots needed for long event histories.

## Outbox Pattern (12.20)

```python
# Problem: writing to DB and publishing event atomically
# Solution: write event to "outbox" table in same transaction as DB write
# Separate process reads outbox and publishes to message broker

# In the same transaction:
async with db.transaction():
    order = await db.create_order(data)
    # Write to outbox (same transaction — guaranteed to commit together)
    await db.execute(
        "INSERT INTO outbox_events (event_type, payload, published) VALUES ($1, $2, false)",
        "order.created", json.dumps({"order_id": order.id})
    )

# Background worker (separate process/job):
async def publish_outbox_events():
    events = await db.fetch(
        "SELECT * FROM outbox_events WHERE published=false ORDER BY id LIMIT 100"
    )
    for event in events:
        await kafka.produce(event["event_type"], event["payload"])
        await db.execute("UPDATE outbox_events SET published=true WHERE id=$1", event["id"])
```
