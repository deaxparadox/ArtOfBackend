## 1. What Is It? (Theory)

CQRS (Command Query Responsibility Segregation) is a pattern where **the model for writing (commands) is separate from the model for reading (queries)**. Instead of one unified data model that must serve both read and write concerns, you have two: one optimized for writing, one optimized for reading.

This sounds like a simple separation, but it has profound implications for scalability, performance, and system design.

**The real-world analogy:**

A library has two systems:
- The **acquisition system** (writing): handles purchasing books, tracking donations, managing budgets. Needs accuracy, consistency, audit trails. Complex relationships.
- The **catalog system** (reading): helps patrons find books. Needs speed, full-text search, recommendation, browsing. Reads need to be fast and flexible.

These systems have fundamentally different requirements. Trying to build one database that serves both would mean compromising both. CQRS separates them: write to one, project to the other.

---

## 2. The Problem CQRS Solves

```
THE SINGLE MODEL PROBLEM:

You have an Order model backed by an orders table.

WRITE requirements:
  Enforce business rules (can't order negative quantities)
  Maintain consistency (stock deduction and order creation must be atomic)
  Audit trail (who changed what, when)
  Validate relationships (product must exist, user must be verified)
  ACID transactions across multiple tables

READ requirements:
  "Show me the user's last 10 orders with product names and images"
  → Joins: orders × order_items × products × users
  → Returns rich, denormalized view
  
  "Show the order dashboard: revenue by region, product mix, time trends"
  → Complex aggregations
  → Large result sets
  → May not need consistency (yesterday's numbers are fine)
  
  "Search orders by customer name, status, date range, amount"
  → Full-text search
  → Fuzzy matching
  → Multiple filter combinations

ONE MODEL CAN'T SERVE BOTH:
  Normalized for writes → complex joins for reads → slow
  Denormalized for reads → multiple places to update → write anomalies
  Strong consistency for reads → blocks writes
  Fast reads without consistency → stale data inconsistencies
```

## 3. How It Actually Works (Deep Mechanics)

### The Two Sides of CQRS

```
COMMAND SIDE (write model):
  Receives commands (create, update, delete)
  Enforces business rules and invariants
  Writes to the write database (normalized, ACID)
  Emits domain events when state changes
  
  NOT concerned with: how data is queried, what views need to show
  
QUERY SIDE (read model):
  Listens to domain events from the command side
  Updates read models (denormalized, optimized views)
  Serves queries from the read database
  
  NOT concerned with: business rules, consistency, transactions
```

### Building and Maintaining Read Models

```python
# READ MODEL: "user_order_summary" — optimized for the orders page

# COMMAND SIDE: business logic
class OrderCommandHandler:
    async def handle_place_order(self, user_id: int, product_id: int, qty: int):
        # Business validation
        product = await db.get_product_for_update(product_id)
        if product.stock < qty:
            raise InsufficientStock()
        
        # Write to normalized tables (the "source of truth")
        async with db.transaction():
            order = await db.create(Order(user_id=user_id, status="pending"))
            item = await db.create(OrderItem(order_id=order.id, product_id=product_id, qty=qty))
            await db.update_stock(product_id, -qty)
        
        # Emit event (async, doesn't block the response)
        await event_bus.publish("order.placed", {
            "order_id": order.id,
            "user_id": user_id,
            "product_id": product_id,
            "product_name": product.name,   # denormalized now, for read model
            "product_image": product.image_url,
            "quantity": qty,
            "unit_price": product.price_cents,
            "total": product.price_cents * qty,
        })
        
        return order.id

# QUERY SIDE: event handler that maintains the read model
class UserOrderSummaryProjection:
    
    @event_handler("order.placed")
    async def on_order_placed(self, event: dict):
        # Denormalize all necessary data into a read-optimized table
        # No joins needed at query time — everything pre-computed
        await read_db.upsert("user_order_summaries", {
            "order_id":      event["order_id"],
            "user_id":       event["user_id"],
            "product_name":  event["product_name"],   # denormalized
            "product_image": event["product_image"],  # denormalized
            "quantity":      event["quantity"],
            "total_cents":   event["total"],
            "status":        "pending",
            "placed_at":     event["occurred_at"],
        })
    
    @event_handler("order.shipped")
    async def on_order_shipped(self, event: dict):
        await read_db.update("user_order_summaries",
            where={"order_id": event["order_id"]},
            set={"status": "shipped", "shipped_at": event["occurred_at"]}
        )
    
    async def get_user_orders(self, user_id: int, page: int, limit: int) -> list:
        # Single fast query, no joins, pre-denormalized
        return await read_db.fetch(
            "SELECT * FROM user_order_summaries WHERE user_id=$1 "
            "ORDER BY placed_at DESC LIMIT $2 OFFSET $3",
            user_id, limit, (page - 1) * limit
        )
```

## 4. Multiple Read Models from the Same Events

The power of CQRS with event-driven projections: one event can update multiple read models.

```python
# "order.placed" event drives multiple independent read models:

@event_handler("order.placed")
async def update_user_order_history(event):
    """For: user's orders page"""
    ...

@event_handler("order.placed")
async def update_product_sales_stats(event):
    """For: product analytics page"""
    await read_db.upsert("product_sales", {
        "product_id": event["product_id"],
        "total_sold": ...,  # increment
        "revenue_cents": ...,  # increment
    })

@event_handler("order.placed")
async def update_daily_revenue(event):
    """For: admin revenue dashboard"""
    date = event["occurred_at"][:10]  # "2024-06-15"
    await read_db.upsert("daily_revenue", {
        "date": date,
        "order_count": ...,  # increment
        "revenue_cents": ...,  # increment
    })

@event_handler("order.placed")
async def index_for_search(event):
    """For: order search functionality"""
    await elasticsearch.index("orders", {
        "id": event["order_id"],
        "user_id": event["user_id"],
        "product_name": event["product_name"],
        "status": "pending",
        "placed_at": event["occurred_at"],
    })
```

Each read model is independently maintained, independently scaled, and optimized for its specific query pattern. The command side doesn't know or care how many read models consume its events.

## 5. Eventual Consistency: The Trade-off

```
CQRS WITH ASYNC READ MODELS = EVENTUAL CONSISTENCY

Timeline:
  t=0:   User places order → command side writes → event published
  t=50ms: Event received by read model projection
  t=51ms: Read model updated ("user_order_summaries" has new order)
  
  Between t=0 and t=51ms:
  User's browser immediately redirects to "Your orders"
  User sees their orders list
  The new order MAY NOT APPEAR YET (read model not yet updated)
  
HANDLING EVENTUAL CONSISTENCY IN UI:
  Option 1: Wait for read model (poll until order appears)
  Option 2: Optimistically show the order (update UI before confirmation)
  Option 3: Direct read from write DB for immediate confirmation
             (defeats CQRS performance benefit, but acceptable for critical paths)
  
READ-YOUR-WRITES CONSISTENCY:
  A common pattern: after a command, read from the write database
  for the FIRST request from the same user.
  Subsequent reads: read model (fast, cached).
  
  Implementation: track "last write timestamp per user" in Redis.
  If user's last write was < 200ms ago: query write DB.
  Otherwise: query read model.
```

## 6. When to Use CQRS

```
USE CQRS WHEN:
  ✓ Read and write workloads have fundamentally different requirements
  ✓ Multiple different views/screens need the same data in different shapes
  ✓ Read scaling is a bottleneck (reads 100x more frequent than writes)
  ✓ Some reads can tolerate eventual consistency
  ✓ Complex reporting/analytics need denormalized views
  ✓ Using event sourcing (CQRS + event sourcing are often paired)

DON'T USE CQRS WHEN:
  ✗ Simple CRUD applications (adds complexity with no benefit)
  ✗ Strong consistency required for reads immediately after writes
  ✗ Small team without bandwidth for additional complexity
  ✗ The read and write patterns are similar (one model works fine)

THE GRADIENT:
  Full CQRS with separate databases: maximum complexity, maximum benefit
  CQRS with read replicas: moderate complexity, significant read scaling benefit
  CQRS with materialized views: low-moderate complexity, good for specific views
  Single model with indexes: lowest complexity, works for most applications
  
  Start simple. Move to CQRS when the pain of not having it is real and measured.
```

## 7. Senior Engineer Perspective

**CQRS is not the default:**

Junior engineers who learn CQRS often want to apply it everywhere. This is a mistake. CQRS adds significant complexity: two databases to manage, eventual consistency to reason about, read model projection bugs to debug, and event schema evolution to handle. These costs are real.

Apply CQRS when the benefit is concrete and measured: "our order listing endpoint takes 800ms because it joins 5 tables, and we can bring that to 5ms with a pre-computed read model." Not "CQRS is a best practice so we should use it."

**The projection bug is a real operational concern:**

When a read model projection has a bug (wrong calculation, missing field), you fix the bug and need to **rebuild the projection** — replay all historical events through the corrected projection code. This requires Kafka or another event stream with retention. With an SQS-based event system (messages deleted after processing), projection bugs mean manual data repair. This is why CQRS almost always pairs with Kafka's event retention.
