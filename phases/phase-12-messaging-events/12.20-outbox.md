## 1. What Is It? (Theory)

The Outbox Pattern is a technique for reliably publishing events from a service that also writes to a database — ensuring that either both the database write AND the event publication happen, or neither does.

Without this pattern, there is a fundamental reliability gap between database writes and event publishing that can cause subtle data inconsistency bugs.

**The problem it solves:**

```
THE DUAL-WRITE PROBLEM:

Attempt 1: Write to DB, then publish event.
  
  Database: ORDER CREATED (success)
  Message broker: [CRASH] event not published
  
  Result: Database has the order. No event. Downstream services never know.
  Payment service never charges. Email service never sends.
  Order exists but is in limbo.

Attempt 2: Publish event, then write to DB.
  
  Message broker: ORDER_PLACED event published (success)
  Database: [CRASH] order not created
  
  Result: Event published. No order in DB.
  Payment service processes payment. Order doesn't exist.
  Money charged, no order. Very bad.

BOTH APPROACHES ARE WRONG.
The database write and event publication are not atomic.
One can succeed while the other fails.
```

## 2. How It Actually Works (Deep Mechanics)

### The Outbox Solution

```
OUTBOX PATTERN:

STEP 1: Write to the database AND to an "outbox" table IN THE SAME TRANSACTION.

  BEGIN TRANSACTION;
    INSERT INTO orders (id, user_id, amount, status) VALUES (...);
    INSERT INTO outbox_events (event_type, payload, published) 
    VALUES ('order.placed', '{"order_id": 1, ...}', false);
  COMMIT;
  
  ATOMICITY GUARANTEE:
  Either both the order AND the outbox event are created.
  Or neither is created (transaction rolled back).
  No split state is possible.

STEP 2: A background process reads unpublished outbox events and publishes them.

  SELECT * FROM outbox_events WHERE published = false ORDER BY created_at LIMIT 100;
  
  For each event:
    Publish to message broker (Kafka/SQS/RabbitMQ).
    If publish succeeds: mark event as published.
    If publish fails: leave as unpublished, retry next cycle.

STEP 3: Broker delivers to consumers as normal.

GUARANTEES:
  If broker is down: outbox accumulates events, published when broker recovers.
  If DB is down: transaction fails (order not created, no outbox event).
  If process crashes after publish but before marking: event re-published (duplicate).
  → Consumers must be idempotent (at-least-once delivery).
```

## 3. Implementation

```python
import asyncio
import asyncpg
import json
import uuid
from datetime import datetime

# WRITE SIDE: business logic + outbox
async def place_order(
    conn: asyncpg.Connection,
    user_id: int,
    product_id: int,
    quantity: int,
) -> str:
    """
    Create an order and queue the event atomically.
    If anything fails, both the order and event are rolled back.
    """
    order_id = str(uuid.uuid4())
    
    async with conn.transaction():
        # Business logic: create the order
        await conn.execute("""
            INSERT INTO orders (id, user_id, product_id, quantity, status)
            VALUES ($1, $2, $3, $4, 'pending')
        """, order_id, user_id, product_id, quantity)
        
        # Deduct inventory atomically
        affected = await conn.execute("""
            UPDATE products 
            SET stock = stock - $1 
            WHERE id = $2 AND stock >= $1
        """, quantity, product_id)
        
        if affected == "UPDATE 0":
            raise InsufficientStock("Not enough stock")
        
        # OUTBOX WRITE: same transaction guarantees atomicity
        await conn.execute("""
            INSERT INTO outbox_events (
                id, event_type, aggregate_id, payload, 
                created_at, published
            ) VALUES ($1, $2, $3, $4, $5, false)
        """,
            str(uuid.uuid4()),          # unique event ID
            "order.placed",             # event type
            order_id,                   # which entity this is about
            json.dumps({                # the event data
                "order_id": order_id,
                "user_id": user_id,
                "product_id": product_id,
                "quantity": quantity,
            }),
            datetime.utcnow(),
        )
        # Transaction committed: BOTH order AND outbox event exist.
    
    return order_id

# PUBLISHER: background worker that reads and publishes outbox events
async def outbox_publisher(pool: asyncpg.Pool, kafka_producer):
    """
    Runs continuously. Reads unpublished events and publishes them.
    Idempotent: re-runs after failure safely.
    """
    while True:
        async with pool.acquire() as conn:
            # Claim a batch of unpublished events
            # FOR UPDATE SKIP LOCKED: concurrent workers don't duplicate-process
            events = await conn.fetch("""
                SELECT id, event_type, aggregate_id, payload
                FROM outbox_events
                WHERE published = false
                  AND created_at > NOW() - INTERVAL '7 days'  -- don't retry ancient failures
                ORDER BY created_at ASC
                LIMIT 100
                FOR UPDATE SKIP LOCKED
            """)
            
            if not events:
                await asyncio.sleep(0.5)  # no work: brief pause
                continue
            
            for event in events:
                try:
                    # Publish to the message broker
                    await kafka_producer.publish(
                        topic=event["event_type"].replace(".", "-"),
                        key=event["aggregate_id"],
                        value=json.loads(event["payload"]),
                        # Include event ID for consumer deduplication
                        headers={"event-id": event["id"]},
                    )
                    
                    # Mark as published (only after successful broker publish)
                    await conn.execute(
                        "UPDATE outbox_events SET published = true, published_at = NOW() WHERE id = $1",
                        event["id"]
                    )
                    
                except Exception as e:
                    # Log failure — event remains unpublished, will be retried
                    logger.error("Failed to publish outbox event",
                                event_id=event["id"],
                                error=str(e))
        
        # Small sleep to avoid hammering the DB when queue is empty
        await asyncio.sleep(0.1)
```

## 4. Outbox Table Schema

```sql
CREATE TABLE outbox_events (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    event_type      VARCHAR(100) NOT NULL,       -- "order.placed", "user.registered"
    aggregate_id    VARCHAR(100) NOT NULL,       -- "order-123", "user-456"
    payload         JSONB NOT NULL,              -- the event data
    created_at      TIMESTAMPTZ DEFAULT NOW(),
    published       BOOLEAN DEFAULT FALSE,
    published_at    TIMESTAMPTZ,
    attempts        INTEGER DEFAULT 0,           -- for debugging
    last_error      TEXT,                        -- for debugging failed events
);

-- Index for the publisher query (fast lookup of unpublished events)
CREATE INDEX idx_outbox_unpublished ON outbox_events (created_at)
WHERE published = false;

-- Cleanup old published events (run nightly)
-- Don't keep published events forever — disk will fill up
DELETE FROM outbox_events 
WHERE published = true 
  AND published_at < NOW() - INTERVAL '7 days';
```

## 5. Alternative: Change Data Capture (CDC)

Instead of writing to an outbox table, you can use Debezium (see topic 12.10) to watch the database's WAL and publish events:

```
CDC APPROACH (no application code for outbox):
  1. Application writes to database normally.
  2. Debezium watches the PostgreSQL WAL (write-ahead log).
  3. Every database change → Debezium publishes to Kafka.
  4. Other services consume from Kafka.

ADVANTAGES over application outbox:
  ✓ No application code needed for event publishing
  ✓ Works with existing code that doesn't publish events
  ✓ Low-latency (milliseconds from DB write to Kafka event)

DISADVANTAGES:
  ✗ Events tied to database schema (not business events)
  ✗ Requires Debezium infrastructure
  ✗ Less control over event schema
  ✗ Tables must be designed for CDC (consistent columns)

WHICH TO USE:
  New application with event-driven design: application outbox pattern
  Existing application adding events without code changes: CDC with Debezium
  Both: use outbox for new code, CDC for legacy services
```

## 6. Senior Engineer Perspective

**The missed detail that causes subtle bugs:**

Teams implement the outbox but forget to run the publisher process as a separate worker. The events accumulate in the outbox table but are never published. Consumers have a growing lag they don't notice because their queue is empty. Hours or days later: "why are orders being created but not processed?"

Always deploy the outbox publisher as a separate process (or scheduled task) from your main application. Monitor the outbox table depth: if unpublished events are older than 5 minutes, something is wrong with the publisher.

**The deduplication requirement:**

Because the publisher uses at-least-once semantics (might publish twice if it crashes between publish and mark-as-published), consumers MUST be idempotent. Include the outbox event ID in the published message. Consumers check this ID against a "processed events" table before processing. This combination — outbox + consumer idempotency — gives you effectively-exactly-once end-to-end.
