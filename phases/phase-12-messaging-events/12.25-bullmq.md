## 1. What Is It? (Theory)

BullMQ is the dominant background job library for Node.js. It's backed by Redis and provides a rich TypeScript API with built-in support for retries, rate limiting, priorities, repeatable jobs, parent-child job dependencies, and a comprehensive monitoring dashboard.

**BullMQ vs. Bull:**

BullMQ is the successor to the original `bull` library. It's a complete rewrite with TypeScript-first design, improved architecture, and better support for complex workflows. New projects should use BullMQ; Bull remains maintained for existing projects.

## 2. How It Actually Works (Deep Mechanics)

### Core Concepts

```
QUEUE: A named channel for a category of jobs.
  const emailQueue = new Queue('emails', { connection });

JOB: A unit of work placed in a queue.
  { data: any, opts: { attempts, delay, priority, removeOnComplete } }

WORKER: A process that consumes jobs from a queue.
  const worker = new Worker('emails', async (job) => { ... });

FLOW PRODUCER: Creates parent-child job dependencies.
  Parent job doesn't complete until all children complete.
```

### Production Setup

```typescript
import { Queue, Worker, Job, QueueEvents, FlowProducer } from 'bullmq';
import Redis from 'ioredis';

const connection = new Redis({
    host: process.env.REDIS_HOST,
    port: parseInt(process.env.REDIS_PORT!),
    maxRetriesPerRequest: null,  // required by BullMQ
});

// QUEUE DEFINITION
const emailQueue = new Queue('emails', {
    connection,
    defaultJobOptions: {
        attempts: 3,                    // retry up to 3 times
        backoff: {
            type: 'exponential',
            delay: 2000,                // 2s, 4s, 8s between retries
        },
        removeOnComplete: { count: 100 },  // keep last 100 completed
        removeOnFail: { age: 7 * 24 * 3600 },  // keep failed for 7 days
    },
});

// ADD JOBS
await emailQueue.add('welcome-email', {
    userId: 123,
    template: 'welcome',
}, {
    delay: 5000,        // wait 5 seconds before processing
    priority: 1,        // lower number = higher priority (1=urgent, 10=low)
});

// SCHEDULED / REPEATABLE JOBS
await emailQueue.add('weekly-digest', {}, {
    repeat: {
        pattern: '0 9 * * 1',    // 9am every Monday (cron syntax)
    },
    jobId: 'weekly-digest',      // deduplication: only one instance running
});

// WORKER DEFINITION
const emailWorker = new Worker<EmailJobData, EmailJobResult>(
    'emails',
    async (job: Job) => {
        const { userId, template } = job.data;
        
        // Update progress (visible in Bull Board dashboard)
        await job.updateProgress(10);
        
        const user = await db.getUser(userId);
        await job.updateProgress(40);
        
        await emailService.send({
            to: user.email,
            template,
            context: { name: user.name },
        });
        
        await job.updateProgress(100);
        
        // Return value stored as job result (if removeOnComplete not set)
        return { sent: true, email: user.email, sentAt: new Date().toISOString() };
    },
    {
        connection,
        concurrency: 10,               // process up to 10 jobs in parallel
        limiter: {
            max: 100,                  // rate limit: max 100 jobs
            duration: 60_000,          // per 60 seconds
        },
    }
);

// WORKER EVENT HANDLING
emailWorker.on('completed', (job: Job) => {
    console.log(`✓ Job ${job.id} completed:`, job.returnvalue);
});

emailWorker.on('failed', (job: Job | undefined, error: Error) => {
    console.error(`✗ Job ${job?.id} failed:`, error.message);
    
    if (job?.attemptsMade === job?.opts.attempts) {
        // Final failure — alert the team
        alertService.send(`Email job permanently failed: ${job?.id}`);
    }
});

emailWorker.on('progress', (job: Job, progress: number) => {
    console.log(`Job ${job.id} is ${progress}% complete`);
});

// QUEUE EVENTS: monitor from a separate process
const queueEvents = new QueueEvents('emails', { connection });
queueEvents.on('completed', ({ jobId }) => {
    console.log(`Job ${jobId} done`);
});
queueEvents.on('failed', ({ jobId, failedReason }) => {
    console.error(`Job ${jobId} failed: ${failedReason}`);
});
```

## 3. Parent-Child Job Dependencies (Flows)

BullMQ's flow feature lets you define parent jobs that only complete when all children complete — without custom coordination:

```typescript
const flow = new FlowProducer({ connection });

await flow.add({
    name: 'process-order',       // parent job
    queueName: 'orders',
    data: { orderId: '123' },
    children: [
        {
            name: 'charge-payment',
            queueName: 'payments',
            data: { orderId: '123', amount: 5000 },
        },
        {
            name: 'reserve-inventory',
            queueName: 'inventory',
            data: { orderId: '123', productId: '456', qty: 2 },
        },
        {
            name: 'send-confirmation',
            queueName: 'emails',
            data: { orderId: '123', userId: '789' },
        },
    ],
});

// Parent "process-order" job runs only after ALL children complete.
// Children run in parallel.
// If any child fails (after retries): parent is also marked failed.
```

## 4. Bull Board: Monitoring Dashboard

```typescript
import { createBullBoard } from '@bull-board/api';
import { BullMQAdapter } from '@bull-board/api/bullMQAdapter';
import { ExpressAdapter } from '@bull-board/express';
import express from 'express';

const serverAdapter = new ExpressAdapter();
serverAdapter.setBasePath('/admin/queues');

createBullBoard({
    queues: [
        new BullMQAdapter(emailQueue),
        new BullMQAdapter(paymentQueue),
        new BullMQAdapter(reportQueue),
    ],
    serverAdapter,
});

const app = express();
app.use('/admin/queues', serverAdapter.getRouter());

// Now at http://localhost:3000/admin/queues:
// - See all queues with job counts (active, waiting, completed, failed)
// - View individual job details and logs
// - Retry failed jobs manually
// - Clean up old jobs
// - Pause/resume queues
```

## 5. Senior Engineer Perspective

**The `maxRetriesPerRequest: null` requirement:**

BullMQ requires Redis connections to have `maxRetriesPerRequest: null` set. This tells ioredis to keep retrying connection to Redis indefinitely rather than throwing after N attempts. Without this, BullMQ workers will crash when Redis has a brief connectivity blip.

**Using Queues vs. Events for background work:**

BullMQ is a task queue, not an event bus. Use it when:
- Work is done exactly once (one worker processes each job)
- Work needs retries, timeouts, rate limiting
- Work has priorities
- You need job progress tracking

Use Kafka/RabbitMQ/SNS when:
- Multiple services need to react to the same event
- Event history and replay are important
- Pub/sub fan-out is needed

Many Node.js applications use both: BullMQ for background jobs, Kafka for event streaming.
