## 1. What Is It? (Theory)

Apache Kafka (LinkedIn, 2011) is a distributed event streaming platform. It is an append-only, distributed log — a sequence of records where each record is immutable and persists for a configured retention period, readable by multiple independent consumer groups.

Kafka is the infrastructure behind event-driven architectures at companies handling millions of events per second: LinkedIn, Netflix, Uber, Airbnb, Cloudflare. When you need an event bus that multiple teams can subscribe to independently, that can handle millions of events per second, and that retains history for replay — Kafka is the answer.

**The mental model:**

Imagine a very fast, very durable, append-only log file — like a database's write-ahead log. Every event is appended to the end. Readers can start at any position and read forward. Multiple readers can read independently without interfering with each other. The log is retained indefinitely (or by time/size policy).

Now distribute that log across a cluster for fault tolerance and horizontal scale. That's Kafka.

## 2. How It Actually Works (Deep Mechanics)

### Topics: Named Event Streams

A **topic** is a named category of events. Think of it as a logical grouping:

```
Topics in an e-commerce system:
  "user-events"       ← all user actions (login, logout, profile update)
  "order-events"      ← all order lifecycle events
  "payment-events"    ← all payment events
  "inventory-events"  ← all inventory changes

Topics are the unit of data organization in Kafka.
Services subscribe to topics, not individual queues.
```

### Partitions: The Unit of Parallelism

Each topic is divided into **partitions** — independent, ordered sub-logs stored on different brokers.

```
Topic: "order-events" with 6 partitions

Partition 0: [event_0][event_1][event_2][event_3]...
Partition 1: [event_0][event_1][event_2]...
Partition 2: [event_0][event_1][event_2][event_3][event_4]...
Partition 3: [event_0][event_1]...
Partition 4: [event_0][event_1][event_2][event_3]...
Partition 5: [event_0][event_1][event_2][event_3][event_4][event_5]...

WHY PARTITION?

1. PARALLELISM:
   Each partition is consumed by ONE consumer in a consumer group.
   6 partitions → up to 6 consumers can work in parallel.
   Without partitions: one consumer reads everything (can't parallelize).

2. ORDERING WITHIN A PARTITION:
   Events within ONE partition are strictly ordered.
   Events ACROSS partitions have no ordering guarantee.
   
   If order events for the same user must be processed in order:
   → Route all events for user_id=123 to the same partition.
   → Use user_id as the partition key.

3. DISTRIBUTION:
   Partitions live on different brokers in the cluster.
   6 partitions across 3 brokers → 2 partitions per broker.
   Balanced load. No single broker handles everything.
```

### The Partition Key: Controlling Where Events Land

```
PRODUCER SENDS: event with key="user-123"
KAFKA COMPUTES: hash("user-123") % num_partitions → partition 2
RESULT: all events with key="user-123" go to partition 2

This guarantees:
  - All events for user-123 are in ONE partition
  - Events for user-123 are ordered (partition is ordered)
  - A single consumer handles all user-123 events

WITHOUT A KEY: round-robin across partitions
  Event 1 → partition 0
  Event 2 → partition 1
  Event 3 → partition 2
  (load balanced, but no ordering guarantee across events)

CHOOSING A PARTITION KEY:
  "What events MUST be processed in order?"
  That's your partition key.
  
  Order events: partition by order_id (order's events stay ordered)
  User events: partition by user_id (user's events stay ordered)
  Payment events: partition by payment_id
  Log events: no key needed (ordering across log lines is fine to lose)
```

### Offsets: Your Position in the Log

Each event in a partition has an **offset** — a unique, sequential integer:

```
Partition 0:
  Offset 0: {"event_type": "order.placed", "order_id": 100}
  Offset 1: {"event_type": "order.placed", "order_id": 101}
  Offset 2: {"event_type": "payment.completed", "order_id": 100}
  Offset 3: {"event_type": "order.shipped", "order_id": 101}
  Offset 4: {"event_type": "order.delivered", "order_id": 100}
  (current end)

Consumer Group "payment-service" has committed offset 3 for this partition.
  → It has processed offsets 0, 1, 2, 3
  → Next poll returns offset 4
  → If it crashes: restarts at offset 3 (already committed)

Consumer Group "analytics-service" has committed offset 1 for this partition.
  → It has processed offsets 0, 1
  → Currently 3 events behind
  → Next poll returns offset 2, 3, 4 (catching up)

THE CRITICAL INSIGHT:
  Each consumer group tracks its own offset independently.
  One consumer group being behind does NOT affect other consumer groups.
  Adding a new consumer group: starts at any offset (earliest = day 0, latest = now).
```

### Consumer Groups: Parallelism AND Independence

```
CONSUMER GROUP "payment-service" (3 consumers, 6 partitions):

  Partition assignment:
    Consumer A: partitions 0, 1
    Consumer B: partitions 2, 3
    Consumer C: partitions 4, 5
  
  Parallelism: 3 consumers process events simultaneously
  Each partition: assigned to exactly ONE consumer in the group
  Scale: add Consumer D → rebalance → each consumer gets 1-2 partitions

MULTIPLE CONSUMER GROUPS (all reading the same topic):

  Topic "order-events" (6 partitions)
    ↓
  Consumer Group "payment-service"    → processes all order events for payments
  Consumer Group "analytics-service"  → processes all order events for analytics
  Consumer Group "fraud-detection"    → processes all order events for fraud
  Consumer Group "warehouse-service"  → processes order events for shipping

  Each group independently consumes ALL events.
  Adding a new group doesn't affect existing groups.
  Groups can be at different positions in the log (different offsets).
```

## 3. Real-World Context

### LinkedIn's Origin Story

LinkedIn built Kafka in 2011 because they had over 100 different metrics and log streams that needed to flow to multiple destinations (Hadoop, Splunk, monitoring systems). Copying data manually between systems was error-prone and created tight coupling. Kafka's "one topic, many consumer groups" model solved this: each producer writes once to a topic; each destination system subscribes independently.

Today LinkedIn processes over 7 trillion messages per day through Kafka.

### The Partition Count Decision Is Permanent (Almost)

Increasing partition count later is painful: messages with the same key may route to different partitions, breaking per-key ordering. Choose partition count upfront based on expected peak consumers.

```
PARTITION COUNT GUIDELINES:
  Low traffic, strict ordering: 3-6 partitions
  Medium traffic (100-10,000 events/sec): 6-24 partitions
  High traffic (10,000+ events/sec): 12-48 partitions
  Very high (millions/sec per topic): 100+ partitions
  
  RULE: partition count ≥ peak consumer count
  (more partitions than consumers = some consumers idle)
  (more consumers than partitions = excess consumers idle)
  
  Common choice: start with 12-24 partitions
  (divisible by 1, 2, 3, 4, 6, 12 → flexible team scaling)
```

## 4. Code Examples

```python
from confluent_kafka import Producer, Consumer
import json

# PRODUCER
producer = Producer({
    "bootstrap.servers": "kafka:9092",
    "enable.idempotence": True,   # prevent duplicate sends on retry
    "acks": "all",                 # wait for all in-sync replicas
    "compression.type": "snappy", # compress for lower bandwidth
})

def delivery_report(err, msg):
    if err:
        print(f"DELIVERY FAILED: {err}")
    # else: successful delivery — can log offset, partition, etc.

# Send with a partition key
producer.produce(
    topic="order-events",
    key="order-123".encode(),    # ensures all order-123 events → same partition
    value=json.dumps({
        "event_type": "order.placed",
        "order_id": "order-123",
        "user_id": "user-456",
        "amount_cents": 5000,
    }).encode(),
    callback=delivery_report,
)
producer.flush()  # wait for all pending messages to be delivered

# CONSUMER
consumer = Consumer({
    "bootstrap.servers": "kafka:9092",
    "group.id": "payment-service",       # consumer group
    "auto.offset.reset": "earliest",     # start from beginning if no committed offset
    "enable.auto.commit": False,         # manual offset commits (safer)
    "max.poll.interval.ms": 300000,      # 5 min max between poll() calls
})
consumer.subscribe(["order-events"])

try:
    while True:
        message = consumer.poll(timeout=1.0)
        
        if message is None:
            continue  # no message this poll cycle
        if message.error():
            print(f"Consumer error: {message.error()}")
            continue
        
        # Process the event
        event = json.loads(message.value().decode())
        
        try:
            if event["event_type"] == "order.placed":
                process_payment(event["order_id"], event["amount_cents"])
            
            # Commit offset AFTER successful processing
            # This is the "acknowledgment" in Kafka
            consumer.commit(message=message)
        
        except Exception as e:
            print(f"Processing failed: {e}")
            # Don't commit → message will be redelivered after consumer restart
            # For permanent failures: use a DLQ

finally:
    consumer.close()  # graceful shutdown: commit pending offsets
```

## 5. Kafka vs. RabbitMQ: The Definitive Comparison

```
QUESTION: "Should I use Kafka or RabbitMQ?"

ANSWER: "What do you need to do with messages after the first consumer processes them?"

KEEP MESSAGES? → Kafka
  - Multiple systems need the same event
  - Analytics needs history
  - Replay capability required
  - Audit trail required
  
DELETE MESSAGES? → RabbitMQ (or SQS)
  - One system processes each message
  - Flexible routing (topic/fanout exchanges)
  - Request-reply patterns
  - Task queue with priorities

THROUGHPUT:
  Kafka: millions of events/second per topic
  RabbitMQ: ~50,000-200,000 messages/second
  → For very high throughput: Kafka

OPERATIONAL COMPLEXITY:
  Kafka: higher (ZooKeeper or KRaft, cluster management, partition rebalancing)
  RabbitMQ: lower (single node works well for most uses)
  → For simpler operations: RabbitMQ

THE COMMON ARCHITECTURE:
  Use BOTH.
  Kafka for: business events (order.placed, user.registered) — streams
  RabbitMQ/SQS for: task queues (send email, resize image) — queues
```

## 6. Senior Engineer Perspective

**What a junior gets wrong:**

Using Kafka when SQS/RabbitMQ would be sufficient. Kafka requires cluster management, ZooKeeper (or KRaft), careful partition planning, and consumer group monitoring. For a service that just needs to queue background jobs, Kafka is massive overkill. RabbitMQ or SQS handles 99% of task queue needs with a fraction of the operational complexity.

Kafka shines when you have these specific requirements:
- Multiple independent systems consuming the same events
- Historical replay capability
- Very high throughput (>100K events/sec)

If none of these apply: use a simpler solution.

**The rebalance awareness:**

When a consumer joins or leaves a consumer group, Kafka triggers a rebalance — partition assignments are redistributed. During a rebalance, the group pauses processing for 30-120 seconds. For high-throughput systems, frequent rebalances (caused by consumer crashes or deployments) are disruptive. Use `cooperative-sticky` partition assignment strategy to minimize rebalance impact.
