## 1. What Is It? (Theory)

Celery is the de facto standard distributed task queue for Python. It lets you define Python functions as asynchronous tasks, queue them from anywhere in your application, and execute them in separate worker processes that can run on any number of machines.

**What Celery provides:**

```
TASK DEFINITION:    Decorate any Python function with @app.task
QUEUING:            .delay(), .apply_async() to queue from your code
EXECUTION:          Worker processes execute tasks from the queue
RETRIES:            Automatic retry with configurable backoff
SCHEDULING:         Periodic tasks via Celery Beat
MONITORING:         Flower dashboard for real-time visibility
ROUTING:            Send different tasks to different queues/workers
RESULTS:            Optional result backend to check task outcomes
```

## 2. How It Actually Works (Deep Mechanics)

### The Celery Architecture

```
                    Redis (broker)
                         │
Web Server               │          Workers
  │                      │             │
  ├── order created       │         Worker 1 ─── sends welcome email
  ├── .delay()  ─────────►│──────────►Worker 2 ─── resizes image
  │                       │         Worker 3 ─── generates report
                         ▲│
                          │
                     Celery Beat
                  (schedules periodic tasks)
```

### Production Configuration

```python
# celery_app.py
from celery import Celery
from celery.schedules import crontab
import os

app = Celery("myapp")

app.conf.update(
    # Broker: Redis is the recommended choice
    broker_url=os.environ["REDIS_URL"],
    result_backend=os.environ["REDIS_URL"],  # or None if results not needed
    
    # Serialization
    task_serializer="json",
    result_serializer="json",
    accept_content=["json"],
    timezone="UTC",
    enable_utc=True,
    
    # RELIABILITY: acknowledge AFTER execution, not before
    # Without this: if worker crashes mid-task, message is lost
    task_acks_late=True,
    task_reject_on_worker_lost=True,
    
    # CONCURRENCY: only take 1 unacknowledged task at a time per worker
    # Without this: worker takes many tasks, crashes, all lost
    worker_prefetch_multiplier=1,
    
    # TIMEOUTS: don't let tasks run forever
    task_soft_time_limit=300,    # SIGTERM after 5 minutes (handle gracefully)
    task_time_limit=360,         # SIGKILL after 6 minutes (force kill)
    
    # ROUTING: different task types → different queues
    task_routes={
        "myapp.tasks.send_email": {"queue": "emails"},
        "myapp.tasks.process_payment": {"queue": "payments"},
        "myapp.tasks.generate_report": {"queue": "reports"},
    },
    
    # PERIODIC TASKS (replaces cron)
    beat_schedule={
        "send-daily-digest": {
            "task": "myapp.tasks.send_daily_digest",
            "schedule": crontab(hour=8, minute=0),  # 8:00 AM daily
        },
        "cleanup-expired-sessions": {
            "task": "myapp.tasks.cleanup_sessions",
            "schedule": 3600.0,  # every hour
        },
    },
)
```

### Defining Tasks

```python
from celery_app import app
from celery.utils.log import get_task_logger

logger = get_task_logger(__name__)

# SIMPLE TASK
@app.task(
    name="myapp.tasks.send_welcome_email",
    bind=True,            # bind=True gives access to self (the task instance)
    max_retries=3,        # retry up to 3 times
    default_retry_delay=60,  # 1 minute between retries
)
def send_welcome_email(self, user_id: int):
    """
    Send a welcome email to a new user.
    Retries on transient failures.
    """
    from myapp.models import User
    from myapp.services import EmailService
    
    try:
        user = User.objects.get(id=user_id)
        EmailService.send_welcome(user.email, user.name)
        logger.info("Welcome email sent", user_id=user_id)
        
    except User.DoesNotExist:
        # Permanent failure — don't retry
        logger.error("User not found", user_id=user_id)
        return  # Exit without raising — task succeeds (no more retries)
    
    except (ConnectionError, TimeoutError) as exc:
        # Transient failure — retry with exponential backoff
        retry_in = 60 * (2 ** self.request.retries)  # 60s, 120s, 240s
        logger.warning("Email failed, retrying", retry_in=retry_in)
        raise self.retry(exc=exc, countdown=retry_in)

# TASK WITH ACKNOWLEDGMENT-SAFE DESIGN
@app.task(
    name="myapp.tasks.process_payment",
    bind=True,
    max_retries=5,
    acks_late=True,       # per-task override of global setting
)
def process_payment(self, order_id: str):
    """Process a payment — must be idempotent."""
    # Idempotency: check if already processed
    from myapp.models import Order
    
    order = Order.objects.get(id=order_id)
    if order.payment_status in ("completed", "failed"):
        logger.info("Payment already processed, skipping", order_id=order_id)
        return {"status": "skipped", "payment_status": order.payment_status}
    
    # Process payment
    result = payment_gateway.charge(order.user_id, order.total_cents)
    order.payment_status = "completed" if result.success else "failed"
    order.save()
    
    return {"status": "processed", "result": result.transaction_id}
```

### Calling Tasks

```python
# DELAY: simplest, queue immediately
send_welcome_email.delay(user_id=123)

# APPLY_ASYNC: full control
send_welcome_email.apply_async(
    args=[],
    kwargs={"user_id": 123},
    countdown=300,           # wait 5 minutes before executing
    expires=3600,            # discard if not started within 1 hour
    queue="emails",          # specific queue
    retry=True,
    priority=9,              # highest priority (0-9)
)

# CHAIN: execute tasks sequentially (output of one → input of next)
from celery import chain
chain(
    validate_order.s(order_id),
    reserve_inventory.s(),
    process_payment.s(),
    send_confirmation.s(),
)()

# GROUP: execute tasks in parallel
from celery import group
group(
    send_email.s(user_id=1),
    send_email.s(user_id=2),
    send_email.s(user_id=3),
)()

# CHORD: parallel tasks then a callback when all finish
from celery import chord
chord(
    [process_batch.s(batch) for batch in batches],
    aggregate_results.s(),  # called with list of all results
)()
```

## 3. Running Workers

```bash
# Start a worker (prefork pool — best for CPU-bound or blocking I/O tasks)
celery -A myapp.celery_app worker \
  --loglevel=info \
  --concurrency=4 \           # 4 parallel worker processes
  --queues=emails,default \   # only consume these queues
  --pool=prefork \            # default: fork processes (good for most tasks)
  --hostname=worker1@%h

# For I/O-bound tasks: use gevent/eventlet for much higher concurrency
pip install gevent
celery -A myapp.celery_app worker \
  --pool=gevent \
  --concurrency=100 \         # 100 concurrent coroutines in one process
  --queues=emails

# Start beat scheduler (for periodic tasks)
celery -A myapp.celery_app beat --loglevel=info

# Monitor with Flower (web UI at http://localhost:5555)
pip install flower
celery -A myapp.celery_app flower --port=5555

# Check queue stats from CLI
celery -A myapp.celery_app inspect active      # running tasks
celery -A myapp.celery_app inspect reserved    # queued, not started
celery -A myapp.celery_app inspect stats       # worker statistics
```

## 4. Senior Engineer Perspective

**The `task_acks_late=True` requirement:**

Without `task_acks_late=True` (the default is False), Celery acknowledges a task the moment a worker picks it up, before executing it. If the worker crashes during execution, the task is gone — never retried, never executed.

With `task_acks_late=True`, the task is acknowledged only after successful execution. Crashes cause the task to be requeued and retried. This requires tasks to be idempotent (safe to run twice), but that's always true for production-grade tasks anyway.

**Always set `worker_prefetch_multiplier=1`:**

By default, Celery fetches multiple tasks per worker at once ("prefetch"). If a worker crashes, all prefetched tasks are lost (until the visibility timeout expires). With `worker_prefetch_multiplier=1`, workers take one task at a time. Slightly lower throughput, dramatically safer under failure conditions.

**Celery Beat is a single point of failure:**

Celery Beat is a single process that schedules periodic tasks. If it crashes, periodic tasks stop until it's restarted. For production: monitor Celery Beat health explicitly, use a supervisor (systemd, Kubernetes), and alert if Beat stops sending tasks to the queue.
