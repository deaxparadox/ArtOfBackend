## 1. What Is It? (Theory)

Event-Driven Architecture (EDA) is a software design paradigm where components communicate by **producing and consuming events**, rather than making direct API calls. Services are decoupled: they don't know who (if anyone) is interested in their events. They just publish facts about what happened.

The vocabulary of EDA matters. Using the wrong terms leads to misunderstandings and incorrect system design.

**The real-world analogy:**

In a company, there are three ways to communicate:

- **Commands:** "Alice, please process this invoice." Alice is the intended recipient. You expect her to do something.
- **Events:** "The board announces: Q3 earnings released." It's broadcast information. Anyone who cares can act on it. The board doesn't know or care who reads it.
- **Queries:** "What is the current stock price?" You want information returned, but nothing changes.

These three communication modes have fundamentally different semantics, and mixing them up creates poorly designed systems.

## 2. The Vocabulary: Commands, Events, Queries

```
COMMAND:
  An instruction to DO something.
  Has ONE intended recipient.
  Can be rejected (the recipient may decline).
  Imperative tense: "PlaceOrder", "ProcessPayment", "SendEmail"
  
  Examples:
  "PlaceOrder" → sent to OrderService
  "ProcessPayment" → sent to PaymentService
  "SendWelcomeEmail" → sent to EmailService
  
  Properties:
  - Directed: has a specific target
  - Stateful: expects something to change
  - Can fail: recipient validates and may reject
  
EVENT:
  A notification that something DID happen.
  No intended recipient (published to anyone who cares).
  Cannot be rejected (the past cannot be changed).
  Past tense: "OrderPlaced", "PaymentProcessed", "EmailSent"
  
  Examples:
  "OrderPlaced" → published to the event bus
  "PaymentProcessed" → published to the event bus
  "UserRegistered" → published to the event bus
  
  Properties:
  - Broadcast: no specific target
  - Immutable: represents a fact
  - Cannot fail: it already happened
  
QUERY:
  A request for information.
  No side effects, no state changes.
  Gets information, changes nothing.
  
  Examples:
  "GetOrder" → returns order data
  "ListUsers" → returns user list
  "GetProductPrice" → returns price
```

## 3. Why the Distinction Matters

```
DESIGNING WITH COMMANDS (creates coupling):
  OrderService → sends "ProcessPayment" command → PaymentService
  
  OrderService KNOWS:
    - PaymentService exists
    - PaymentService's API for receiving payment commands
    - PaymentService's location (URL or queue name)
  
  Problems:
  - Adding InventoryService: change OrderService to also call it
  - PaymentService is down: OrderService request fails
  - Adding analytics: modify every service that needs to record analytics

DESIGNING WITH EVENTS (creates decoupling):
  OrderService → publishes "OrderPlaced" event → message broker
  
  OrderService KNOWS:
    - Nothing about consumers. Zero.
  
  Benefits:
  - PaymentService subscribes independently
  - InventoryService subscribes independently
  - Analytics subscribes independently
  - Adding a new consumer: zero changes to OrderService
  - PaymentService is down: event waits in queue, processed when it recovers
```

## 4. Event Design: What Makes a Good Event

```
GOOD EVENT:
  Immutable fact about the business domain.
  Self-contained (doesn't require fetching more data to understand).
  Named in business language (not technical language).
  
  Good example:
  {
    "event_type": "order.placed",
    "event_id": "evt_abc123",
    "occurred_at": "2024-06-15T14:30:00Z",
    "order_id": "ord_456",
    "user_id": "usr_789",
    "product_id": "prd_101",
    "quantity": 2,
    "unit_price_cents": 2500,
    "total_cents": 5000,
    "currency": "USD",
  }
  → Self-contained: payment service can process without fetching order details
  → Immutable: captures the state at the time the order was placed
  → Named for business domain: "order.placed" not "insert_into_orders_table"
  
BAD EVENTS (anti-patterns):
  
  Too thin (requires lookups):
  {"event_type": "order.placed", "order_id": "ord_456"}
  → Consumer must call GET /orders/ord_456 to get needed data
  → Tight coupling to Order Service's API
  → N+1 problem at the event level
  
  Technical names instead of business names:
  {"event_type": "orders_table_row_inserted", "row_id": 456}
  → Exposes implementation detail (it's a database row)
  → Consumers shouldn't know/care about the database schema
  
  Commands disguised as events:
  {"event_type": "please_send_confirmation_email", "order_id": 456}
  → This is a command, not an event
  → Should be: {"event_type": "order.placed"} → Email Service reacts by sending email
```

## 5. Event Schema and Versioning

```
EVENT ENVELOPE PATTERN:
  Every event shares a common envelope:
  {
    "event_id": "evt_unique_id",         ← for deduplication
    "event_type": "order.placed",        ← what happened
    "event_version": "1.0",              ← schema version
    "occurred_at": "2024-06-15T14:30:00Z", ← when it happened
    "aggregate_type": "order",           ← what entity this is about
    "aggregate_id": "ord_456",           ← which specific entity
    "producer_service": "order-service", ← who published it
    "correlation_id": "req_abc123",      ← traces back to original request
    "payload": { ... }                   ← event-specific data
  }

SCHEMA EVOLUTION RULES:
  SAFE (backward compatible):
    Add new optional fields to payload → existing consumers ignore unknown fields
    Rename event_type with new version → old consumers ignore new type
  
  BREAKING (requires consumer migration):
    Remove existing fields → consumers that read them will break
    Change field type (string → integer) → consumers will fail
    Rename existing fields → consumers can't find the data
  
  VERSION STRATEGY:
    "order.placed.v2" for breaking changes
    Consumers must explicitly upgrade to handle v2
    Maintain v1 support until all consumers migrated
    Then deprecate and remove v1
```

## 6. Real-World Context

### Amazon's SOA Mandate

Jeff Bezos's famous 2002 API mandate (discussed in topic 12.1) required services to communicate through APIs and service interfaces — no direct database access, no sharing of memory. The enforcement mechanism: messaging and events. Each team owned its events and contracts. This architecture is what allowed AWS to be built: each service (S3, EC2, DynamoDB) exposes events that other services can react to, without needing to know each other's internals.

### How Stripe Processes Payments via Events

Stripe publishes rich events (payment_intent.succeeded, payment_intent.payment_failed, charge.refunded) both internally and externally (via webhooks). Their internal event pipeline allows dozens of teams to react to payment events without calling each other directly: the fraud team subscribes to analyze patterns, the billing team subscribes to update invoices, the notification team subscribes to alert customers. Each team is autonomous and independently deployable.

## 7. Senior Engineer Perspective

**The event-driven trap: distributed monolith:**

Teams adopt event-driven architecture but still design services that need to coordinate tightly. Service A publishes an event. Service B receives it and publishes a response event. Service A waits for that response. This is just synchronous RPC reimplemented over events — with all the coupling of synchronous calls plus the complexity of async messaging.

Genuine event-driven means: when A publishes "order.placed," A's job is done. A doesn't care what B, C, or D do with it. If you find yourself designing a workflow where Service A must wait for Service B's response event before continuing, reconsider whether commands (RPC) would be more appropriate for that interaction.

**The event-driven mental shift:**

The shift from RPC thinking ("call this service, get a result") to event thinking ("announce this happened, let interested parties react") is subtle but profound. It requires asking not "what do I need to call?" but "what facts am I responsible for announcing?" That reframing changes your entire service design philosophy.
