## 1. What Is It? (Theory)

Apache Pulsar (Yahoo, 2016; Apache 2018) is a distributed messaging and streaming platform designed to address Kafka's operational pain points. It separates compute (brokers) from storage (Apache BookKeeper), enabling independent scaling and operational features that are difficult in Kafka.

---

## 2. What Makes Pulsar Different

### Compute-Storage Separation

```
KAFKA ARCHITECTURE:
  Brokers own both compute AND storage.
  Each broker stores partitions on local disk.
  
  Scaling storage: must add brokers (even if compute is sufficient).
  Adding partitions: rebalancing moves data between brokers (slow, disruptive).
  Partition count: effectively fixed post-creation (painful to change).

PULSAR ARCHITECTURE:
  Brokers: stateless compute layer (receive messages, serve consumers)
  BookKeeper: distributed storage layer (persists messages across nodes)
  
  Add more broker nodes: instant, no data movement.
  Add more BookKeeper nodes: adds storage capacity, data rebalances gradually.
  Change topic configurations: no data movement required.
  
BENEFIT:
  Scale compute when you need more throughput.
  Scale storage when you need more retention.
  They scale independently.
```

### Multi-Tenancy Built-In

```
PULSAR TOPIC HIERARCHY:
  persistent://tenant/namespace/topic
  
  Example:
  persistent://payments/production/order-events
  persistent://payments/staging/order-events
  persistent://analytics/production/clickstream
  
  Tenant: top-level organizational unit (team, company, customer)
  Namespace: grouping within a tenant (environment, service)
  Topic: the actual event stream
  
  Each tenant can have its own:
    - Authentication (separate JWT or mTLS)
    - Authorization rules
    - Rate limits
    - Retention policies
    - Storage quotas
  
WHY THIS MATTERS FOR SAAS PLATFORMS:
  One Pulsar cluster can serve 100 different customers.
  Customer A: 100GB retention, 10MB/s throughput
  Customer B: 7 days retention, 100MB/s throughput
  
  With Kafka: would need separate clusters per customer (complex, expensive).
  With Pulsar: one cluster, namespace-level isolation.
```

### Tiered Storage: Automatic Cold Data Offload

```
KAFKA TIERED STORAGE:
  All data on broker disks (expensive per GB).
  Old data you rarely need: still on expensive SSD.
  
PULSAR TIERED STORAGE:
  Hot data: recent events in BookKeeper (fast SSD storage).
  Cold data: older events automatically offloaded to S3/GCS/HDFS.
  
  Policy: after 7 days → offload to S3 (cost: $0.023/GB → $0.00099/GB deep archive)
  Reads: Pulsar fetches from S3 transparently (slower for cold data, instant for hot)
  
  COST IMPACT at 1TB/day:
  Kafka (all on SSD): $0.023 × 30 days × 1TB = $690/month
  Pulsar (7d hot, 23d cold on S3): $0.023 × 7d × 1TB + $0.023 × 23d × 1TB (S3 Standard)
  → Similar cost, but Pulsar can keep 1 year+ of history cost-effectively
  → Or: $0.00099 × 23d × 1TB on S3 Glacier = ~$22/month for cold storage
```

## 3. When to Choose Pulsar over Kafka

```
CHOOSE PULSAR WHEN:
  ✓ Multi-tenant SaaS platform (built-in tenant isolation)
  ✓ Need to keep months/years of data cost-effectively (tiered storage)
  ✓ Active-active geo-replication is a primary requirement
  ✓ Need to add/modify topics without partition count constraints
  ✓ Building from scratch with no existing Kafka expertise
  ✓ Pulsar Functions (lightweight compute, like Kafka Streams but polyglot)

STAY WITH KAFKA WHEN:
  ✓ Team has Kafka expertise
  ✓ Large existing Kafka ecosystem (Kafka Connect, Streams, ksqlDB)
  ✓ Kafka-compatible tools are important (Debezium, Flink, etc.)
  ✓ Managed Kafka (Confluent Cloud, MSK) reduces operational burden
  ✓ Kafka's operational challenges are manageable for your scale
```

---

## 4. Senior Engineer Perspective

**The honest Pulsar vs. Kafka comparison in 2024:**

Pulsar has a smaller ecosystem and community than Kafka. Kafka-compatible tools (Debezium for CDC, extensive Kafka Connect connectors, Flink Kafka source) have years of production hardening that Pulsar alternatives lack.

Pulsar's technical advantages (tiered storage, multi-tenancy, compute-storage separation) are real and meaningful for specific use cases — particularly large-scale SaaS platforms and data platforms requiring long retention. For most teams, Kafka (especially managed Kafka on Confluent Cloud or MSK) is the pragmatic choice.
