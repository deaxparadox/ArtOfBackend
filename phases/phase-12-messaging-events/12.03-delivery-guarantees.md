## 1. What Is It? (Theory)

Every messaging system must answer a fundamental question: what do you guarantee about message delivery? How many times will a message be delivered? What happens if a consumer crashes mid-processing?

There are three possible answers — three "delivery semantics" — and they represent different tradeoffs between reliability, performance, and implementation complexity.

**The real-world analogy:**

Imagine you hire a courier to deliver a package.

- **At-most-once:** The courier tries once. If they can't deliver, the package is lost. No retry. Fast, cheap, unreliable.
- **At-least-once:** The courier keeps trying until they get confirmation. You might receive the package twice if the first delivery wasn't confirmed. Reliable but possible duplicates.
- **Exactly-once:** The courier guarantees delivery precisely once. Complex coordination, expensive, requires the recipient to verify they haven't received it before.

For software: most business-critical messaging uses at-least-once delivery with idempotent consumers — which achieves the same result as exactly-once at lower complexity cost.

## 2. How It Actually Works (Deep Mechanics)

### At-Most-Once: Fire and Forget

```
FLOW:
  Producer sends message.
  Broker receives (or not — we don't check).
  Consumer receives (or not — we don't check).
  No acknowledgment. No retry.
  
  Message may be delivered:
    0 times (broker was down when producer sent)
    1 time (normal delivery)
    Never more than 1 time (no retry = no duplicate)
  
IMPLEMENTATION:
  Producer: send without waiting for ack
  Consumer: process without acknowledging
  Broker: store in memory only, no durability

WHEN MESSAGES ARE LOST:
  - Broker crashes before forwarding to consumer
  - Network partition between producer and broker
  - Consumer crashes after receiving but before processing
  
WHY WOULD YOU USE THIS?
  Speed. No acknowledgment overhead. No retry overhead.
  Use when losing some messages is acceptable.
  
REAL USE CASES:
  - Metrics (losing 0.01% of metrics doesn't materially affect dashboards)
  - Logs (losing a few log lines is rarely catastrophic)
  - Game state updates (the next update will correct any missed one)
  - StatsD (UDP-based metrics — fire and forget by design)
```

### At-Least-Once: Guaranteed Delivery, Possible Duplicates

```
FLOW:
  Producer sends message → waits for broker acknowledgment.
  If no ack within timeout → producer retries.
  
  Consumer processes message → sends acknowledgment to broker.
  Broker only removes message after acknowledgment.
  If consumer crashes before acking → broker redelivers.
  
  Message is delivered:
    1 time (normal flow)
    2+ times (consumer crashed after processing but before acking)
  
WHEN DUPLICATES OCCUR:
  
  Consumer receives message.
  Consumer processes payment. ($50 charge goes through)
  Consumer crashes before sending ack.
  Broker: "no ack received, must have failed" → redelivers.
  Another consumer instance picks it up.
  Consumer processes payment AGAIN. ($50 charged AGAIN!)
  
  This is the duplicate processing problem.
  It WILL happen in production. Design for it.

SOLUTION: Idempotent Consumers

  Idempotent = processing the same message twice produces the same result as once.
  
  IDEMPOTENT CONSUMER PATTERN:
  
  async def process_payment(message: dict):
      payment_id = message["payment_id"]
      amount = message["amount"]
      
      # Check if already processed (deduplication)
      already_done = await db.fetchval(
          "SELECT 1 FROM processed_payments WHERE payment_id = $1",
          payment_id
      )
      if already_done:
          return  # Skip — already processed. Safe to do nothing.
      
      # Process atomically: charge AND record as processed
      async with db.transaction():
          await payment_gateway.charge(amount)
          await db.execute(
              "INSERT INTO processed_payments (payment_id) VALUES ($1)",
              payment_id
          )
          # If we crash here, the transaction rolls back.
          # The payment was charged but not recorded.
          # Next delivery: gateway might reject duplicate, or we check idempotency key there too.

WHY AT-LEAST-ONCE IS THE PRODUCTION DEFAULT:
  - Simpler to implement than exactly-once
  - Works correctly with idempotent consumers
  - Supported by all major brokers (Kafka, SQS, RabbitMQ) by default
```

### Exactly-Once: The Holy Grail

```
DEFINITION: Each message is processed exactly once, regardless of failures.
  No duplicates. No missed messages.

WHY IT'S HARD:
  Distributed systems can't achieve exactly-once without coordination.
  
  The fundamental problem: "Two Generals Problem"
  Producer and Consumer can never be 100% certain the message was delivered.
  
  After sending: did the broker receive it?
  After processing: was the ack delivered?
  
  There's always a failure scenario where you can't tell.

HOW KAFKA ACHIEVES "EFFECTIVELY EXACTLY-ONCE":

  Producer side: idempotent producer
    Each message has a sequence number per producer-partition pair.
    If broker receives duplicate (retry): broker rejects it (sequence already seen).
    Producer config: enable.idempotence=true
  
  Consumer side: transactional read-process-write
    Consumer reads from Kafka.
    Processes message (writes to DB or another Kafka topic).
    Commits Kafka offset AND writes result in ONE atomic transaction.
    If crash: transaction rolls back. Both offset and result rolled back.
    On restart: offset is at previous position. Message redelivered.
    But the result write was also rolled back, so re-processing is safe.
  
  Together: exactly-once semantics within the Kafka ecosystem.

REAL EXACTLY-ONCE COSTS:
  - 30-60% throughput reduction vs. at-least-once
  - Only works within Kafka (not to external systems like databases)
  - For external systems: still need idempotency
  
THE PRACTICAL TRUTH:
  "Exactly-once" in real systems usually means:
  at-least-once + idempotent consumers + deduplication
  = functionally equivalent to exactly-once, but achievable.
```

## 3. The Acknowledgment Protocol

At-least-once delivery depends on acknowledgment. Understanding exactly when to acknowledge is critical:

```
SQS VISIBILITY TIMEOUT MODEL:

  Consumer receives message → message becomes "invisible" for N seconds
  Consumer processes successfully → calls DeleteMessage (explicit ack)
  
  If consumer crashes before DeleteMessage:
    Visibility timeout expires → message reappears in queue
    Another consumer instance picks it up → processed again
    
  If consumer processes but DeleteMessage call fails:
    Message reappears → processed again (duplicate!)
    Solution: make processing idempotent

  VISIBILITY TIMEOUT SIZING:
  Set to 3× expected processing time.
  
  Processing takes 10s average → set visibility timeout to 30s.
  
  Too short: message reappears while still being processed (duplicates)
  Too long: if consumer crashes, message is "stuck" invisible for too long
            (delayed redelivery → delayed retry → delayed failure recovery)

KAFKA MANUAL COMMIT MODEL:

  Consumer polls messages.
  Consumer processes them.
  Consumer manually commits offset to Kafka.
  
  If crash before commit: offset not advanced. Messages redelivered.
  If crash after commit: offset advanced. No redelivery.
  
  enable.auto.commit=false: safest, manual control
  enable.auto.commit=true: Kafka commits periodically (default 5s)
    Risk: processed 1000 messages, committed at 500, crash
    → messages 501-1000 redelivered (processed twice)
    → require idempotent processing to handle this
```

## 4. Code Examples

```python
# SQS consumer: at-least-once with explicit acknowledgment
import boto3
import json

sqs = boto3.client('sqs')
QUEUE_URL = os.environ['QUEUE_URL']

def consume_with_at_least_once():
    """Receive messages, process them, acknowledge or let them redeliver."""
    
    response = sqs.receive_message(
        QueueUrl=QUEUE_URL,
        MaxNumberOfMessages=10,
        WaitTimeSeconds=20,           # long polling: wait up to 20s for messages
        VisibilityTimeout=60,         # message invisible for 60s while we process
    )
    
    for message in response.get('Messages', []):
        message_id = message['MessageId']
        body = json.loads(message['Body'])
        
        try:
            # Process the message (must be idempotent!)
            process_order(body, message_id=message_id)
            
            # Acknowledge: tell SQS "I processed this, delete it"
            sqs.delete_message(
                QueueUrl=QUEUE_URL,
                ReceiptHandle=message['ReceiptHandle']
            )
        
        except TransientError:
            # Don't delete: message returns to queue after VisibilityTimeout
            # SQS will redeliver to another consumer
            logger.warning("Transient error, will retry", message_id=message_id)
        
        except PermanentError:
            # Message can never be processed (bad data, etc.)
            # After maxReceiveCount retries: SQS moves to DLQ automatically
            # Delete manually to fail fast instead of waiting for max retries
            sqs.delete_message(QueueUrl=QUEUE_URL, ReceiptHandle=message['ReceiptHandle'])
            logger.error("Permanent failure, sending to DLQ manually", message_id=message_id)


def process_order(data: dict, message_id: str):
    """Idempotent order processor."""
    
    # Deduplication: check if already processed
    if db.exists("SELECT 1 FROM processed_messages WHERE id = %s", message_id):
        logger.info("Duplicate message, skipping", message_id=message_id)
        return
    
    # Process and mark as done in one transaction
    with db.transaction():
        order = Order.create(
            user_id=data['user_id'],
            product_id=data['product_id'],
            amount=data['amount'],
        )
        db.insert("processed_messages", {"id": message_id, "processed_at": now()})
    
    return order
```

from confluent_kafka import Consumer

consumer = Consumer({

while True:
    msg = consumer.poll(timeout=1.0)
    if msg is None or msg.error():

    try:
        event = json.loads(msg.value())

## 5. Comparison Table

| | At-Most-Once | At-Least-Once | Exactly-Once |
|---|---|---|---|
| **Guarantee** | 0 or 1 delivery | 1+ deliveries | Exactly 1 delivery |
| **Duplicates** | Never | Possible | Never |
| **Data loss** | Possible | Never | Never |
| **Complexity** | Very low | Low | High |
| **Performance** | Fastest | Fast | Slowest (30-60% overhead) |
| **Consumer requirement** | None | Idempotent | Transactional |
| **Best for** | Metrics, logs | Most production workloads | Financial ledgers in Kafka |

## 6. Senior Engineer Perspective

**The myth of exactly-once:**

Teams often ask "how do we achieve exactly-once delivery?" The right question is "how do we ensure each message is processed once effectively?" The answer is almost always: at-least-once delivery + idempotent consumer + deduplication key stored in the database. This achieves the same observable outcome as exactly-once but with simpler infrastructure and better performance.

**The deduplication key design:**

The deduplication key is the most important design decision for at-least-once consumers. It must be:
- **Unique per logical operation** (not per message delivery — redeliveries have the same ID)
- **Stable** (same operation → same key, even if the message arrives differently formatted)
- **Stored transactionally** with the business operation (if the transaction rolls back, the dedup record must also roll back)

Usually: use the message's ID field provided by the broker, or a business-meaningful ID (order_id, payment_id) embedded in the message.
