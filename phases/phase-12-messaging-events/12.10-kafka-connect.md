## 1. What Is It? (Theory)

Kafka Connect is a framework for streaming data between Kafka and external systems (databases, file systems, search engines, cloud services) without writing custom consumer/producer code.

Before Kafka Connect, every team building a pipeline had to write:
- A Kafka consumer that reads events
- Code to transform the data format
- A writer that pushes into the destination (PostgreSQL, Elasticsearch, S3, etc.)

Kafka Connect standardizes this into a connector ecosystem. Confluent Hub has 200+ production-ready connectors. Most data pipeline needs are covered — you configure, not code.

**The two connector types:**

```
SOURCE CONNECTOR: External system → Kafka
  Database → Kafka (via CDC: Debezium)
  File system → Kafka
  REST API → Kafka
  S3 → Kafka

SINK CONNECTOR: Kafka → External system
  Kafka → Elasticsearch (indexing)
  Kafka → PostgreSQL (data warehouse loading)
  Kafka → S3 (data lake archival)
  Kafka → MongoDB
  Kafka → Slack (notifications)
```

## 2. How It Actually Works (Deep Mechanics)

```
KAFKA CONNECT ARCHITECTURE:

  Workers (Connect worker processes):
    Distributed mode: multiple workers in a cluster (production)
    Standalone mode: single worker (development, small scale)
  
  Connectors: configuration for a specific integration
    Each connector = one data pipeline definition
    
  Tasks: units of parallelism within a connector
    One connector may spawn N tasks (N partitions worth of parallelism)
  
  The workers execute the tasks, handling:
    - Offset tracking (where in the source we've read to)
    - Retries and error handling
    - Schema registry integration
    - Load balancing across tasks
```

### Change Data Capture with Debezium (Most Important Source Connector)

**Debezium** reads the PostgreSQL write-ahead log (WAL) and publishes every database change as a Kafka event. This is the most valuable Kafka Connect pattern: zero-downtime database streaming.

```
DEBEZIUM CDC FLOW:

PostgreSQL WAL:
  BEGIN
  INSERT INTO orders (id, user_id, amount) VALUES (123, 456, 5000)
  UPDATE users SET order_count = order_count + 1 WHERE id = 456
  COMMIT

Debezium reads WAL → publishes to Kafka:
  Topic "postgres.public.orders":
    {
      "op": "c",          ← c=create, u=update, d=delete, r=read (snapshot)
      "before": null,     ← state before (null for INSERT)
      "after": {          ← state after (the new row)
        "id": 123,
        "user_id": 456,
        "amount": 5000
      },
      "source": {
        "table": "orders",
        "ts_ms": 1718450200000,
        "lsn": 23456789    ← WAL position (for exactly-once processing)
      }
    }

Consumers:
  Elasticsearch sink: indexes order for search
  Redis sink: updates order cache
  Analytics: reads into data warehouse
  Notification service: triggers order confirmation

ADVANTAGES:
  Zero application code change for CDC
  Millisecond latency from DB write to Kafka event
  Exactly-once via WAL LSN tracking
  Automatic schema change detection
```

## 3. Common Sink Connectors

```
ELASTICSEARCH SINK:
  Kafka events → Elasticsearch index
  Use: real-time search indexing
  
  {
    "connector.class": "io.confluent.connect.elasticsearch.ElasticsearchSinkConnector",
    "topics": "orders",
    "connection.url": "http://elasticsearch:9200",
    "type.name": "_doc",
    "key.ignore": "false",
    "schema.ignore": "true"
  }

S3 SINK (data lake archival):
  Kafka events → S3 files (Parquet, Avro, JSON)
  Use: cheap long-term storage, analytics, compliance
  Files partitioned by date: s3://my-bucket/orders/year=2024/month=06/day=15/
  
  {
    "connector.class": "io.confluent.connect.s3.S3SinkConnector",
    "topics": "orders,users",
    "s3.region": "us-east-1",
    "s3.bucket.name": "my-data-lake",
    "s3.part.size": "5242880",
    "flush.size": "1000",            ← batch 1000 messages per file
    "rotate.interval.ms": "3600000", ← rotate file every hour
    "storage.class": "io.confluent.connect.s3.storage.S3Storage",
    "format.class": "io.confluent.connect.s3.format.parquet.ParquetFormat"
  }

JDBC SINK (into a database):
  Kafka events → PostgreSQL/MySQL table rows
  Use: event-driven ETL into data warehouse
```

## 4. Senior Engineer Perspective

**When to use Kafka Connect vs. custom consumer:**

Use Kafka Connect for standard integrations (DB → Kafka, Kafka → Elasticsearch). The connectors handle schema evolution, error handling, offset management, and fault tolerance that would take weeks to build correctly from scratch.

Write a custom consumer when: your transformation logic is complex business logic (not just format conversion), you need tight control over processing semantics, or no connector covers your specific case.

**Debezium is often the "missing piece" in event-driven architecture:**

Teams that can't easily migrate their applications to publish events can use Debezium to tap the database directly. The database becomes the implicit event source. Every row change is an event. This allows moving toward event-driven architecture incrementally, without requiring all applications to publish events explicitly.
