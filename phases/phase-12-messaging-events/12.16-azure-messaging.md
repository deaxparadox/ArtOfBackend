## 1. Azure Service Bus: Enterprise Messaging

Azure Service Bus is Microsoft's managed enterprise message broker. It provides queues (point-to-point) and topics/subscriptions (pub/sub) similar to SQS+SNS, but with more enterprise-focused features.

```
SERVICE BUS TIERS:
  Standard:  Queues + Topics, up to 256KB messages
  Premium:   Dedicated capacity, up to 100MB messages, VNet integration

KEY FEATURES:
  ✓ Message sessions: ordered processing per session ID (like Kafka partition keys)
  ✓ Scheduled messages: deliver at a specific future time
  ✓ Message deferral: defer specific messages for later processing
  ✓ Dead letter queues: built-in, per-queue
  ✓ Duplicate detection: exactly-once within time window
  ✓ Message lock (like SQS visibility timeout)
  ✓ Topics with subscription filters (SQL-like expressions)
```

```python
from azure.servicebus import ServiceBusClient, ServiceBusMessage
import os, json

CONN_STR = os.environ["AZURE_SERVICEBUS_CONNECTION_STRING"]

# PRODUCER
with ServiceBusClient.from_connection_string(CONN_STR) as client:
    with client.get_queue_sender("orders") as sender:
        message = ServiceBusMessage(
            json.dumps({"order_id": 123, "amount": 5000}),
            content_type="application/json",
            session_id="user-456",          # session: ordered per user
            scheduled_enqueue_time_utc=..., # optional: deliver at specific time
        )
        sender.send_messages(message)

# CONSUMER
with ServiceBusClient.from_connection_string(CONN_STR) as client:
    with client.get_queue_receiver("orders", max_wait_time=10) as receiver:
        for message in receiver:
            body = json.loads(str(message))
            try:
                process(body)
                receiver.complete_message(message)   # ack: delete from queue
            except Exception:
                receiver.abandon_message(message)    # nack: redeliver
                # After max_delivery_count: auto-moved to dead letter queue
```

## 2. Azure Event Hubs: Kafka-Compatible Streaming

Azure Event Hubs is Microsoft's event streaming service — and crucially, it supports the **Kafka protocol**. Kafka clients can connect to Event Hubs without code changes, just by changing the bootstrap servers configuration.

```python
# Your existing Kafka producer code → change connection string
from confluent_kafka import Producer

producer = Producer({
    # ← Kafka: 'bootstrap.servers': 'kafka:9092'
    "bootstrap.servers": "my-namespace.servicebus.windows.net:9093",
    "security.protocol": "SASL_SSL",
    "sasl.mechanism": "PLAIN",
    "sasl.username": "$ConnectionString",
    "sasl.password": os.environ["EVENT_HUB_CONNECTION_STRING"],
})

# Everything else: identical to Kafka code
producer.produce("orders", json.dumps({"order_id": 123}).encode())
producer.flush()
```

**Why Event Hubs + Kafka compatibility matters:**

Teams with existing Kafka consumers can migrate to Event Hubs by changing two lines of configuration. Azure handles the cluster management. You keep your Kafka expertise, tooling, and code.

## 3. Service Bus vs. Event Hubs

```
SERVICE BUS:
  ✓ Enterprise messaging patterns (sessions, deferrals, scheduled messages)
  ✓ Topics/subscriptions with SQL filter rules
  ✓ At-most-once, at-least-once, exactly-once delivery
  ✓ Small messages, complex routing
  ✗ Not designed for high-throughput streaming
  Best for: enterprise application integration, workflow coordination

EVENT HUBS:
  ✓ High-throughput event streaming (millions/sec)
  ✓ Kafka-compatible API
  ✓ Event retention and replay
  ✓ Integration with Azure Stream Analytics, HDInsight (Spark)
  ✗ No message filtering (consume everything or nothing)
  ✗ No dead letter queues
  Best for: IoT telemetry, log streaming, event sourcing, Kafka migration
```
