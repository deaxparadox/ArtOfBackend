## 1. What Is It? (Theory)

AWS SQS (Simple Queue Service) and SNS (Simple Notification Service) are AWS's managed messaging primitives. They require zero cluster management, auto-scale to virtually unlimited throughput, and integrate deeply with the AWS ecosystem.

**The pattern:**

- **SQS** alone: point-to-point queue (one producer → one consumer group)
- **SNS** alone: fan-out to multiple endpoints (one → many)
- **SNS → SQS**: fan-out with durability (the production standard)

## 2. SQS Deep Dive

### Standard vs. FIFO Queues

```
SQS STANDARD:
  - At-least-once delivery (duplicates possible)
  - Best-effort ordering (may arrive out of order)
  - Unlimited throughput
  - Use for: most workloads where ordering isn't critical
  
SQS FIFO:
  - Exactly-once processing (deduplication within 5-minute window)
  - First-in, first-out ordering (guaranteed, within message group)
  - Up to 3,000 messages/second with batching (300 without)
  - MessageGroupId: ordering enforced within a group (like Kafka partition key)
  - MessageDeduplicationId: prevents duplicates
  - Use for: financial transactions, order processing where sequence matters
```

### The Visibility Timeout Explained

```
VISIBILITY TIMEOUT BEHAVIOR:

Consumer calls ReceiveMessage → message returned AND hidden from queue.
Other consumers cannot see it during visibility timeout period.

Normal flow:
  Consumer processes message (within visibility timeout).
  Consumer calls DeleteMessage.
  Message removed from queue.

Failure flow:
  Consumer crashes (or processing takes too long).
  Visibility timeout expires.
  Message becomes visible again.
  Another consumer can now receive it.
  → At-least-once semantics

SIZING VISIBILITY TIMEOUT:
  Too short: message reappears while still being processed → duplicate processing
  Too long: if consumer crashes, message "stuck" invisible for too long
  
  Rule: set to 3× expected max processing time.
  For a task that takes 30s max: visibility_timeout = 90s

EXTENDING VISIBILITY TIMEOUT:
  If processing takes longer than expected:
  Call ChangeMessageVisibility before timeout expires.
  Extend by another N seconds.
  Use for: long-running tasks that need more time but are making progress.
```

### SQS Best Practices

```python
import boto3, json, time

sqs = boto3.client('sqs', region_name='us-east-1')

# LONG POLLING: wait up to 20s for a message (reduces empty responses)
# Without: poll constantly, most calls return nothing (expensive, wasteful)
# With: one call blocks until message arrives (or 20s passes)
response = sqs.receive_message(
    QueueUrl=QUEUE_URL,
    MaxNumberOfMessages=10,        # batch: get up to 10 at once
    WaitTimeSeconds=20,            # long polling
    VisibilityTimeout=90,          # hide message for 90s while processing
    AttributeNames=['All'],        # include metadata
    MessageAttributeNames=['All'], # include custom attributes
)

for message in response.get('Messages', []):
    receipt = message['ReceiptHandle']
    receive_count = int(message['Attributes']['ApproximateReceiveCount'])
    body = json.loads(message['Body'])
    
    # Extend timeout if processing is slow
    if is_slow_task(body):
        sqs.change_message_visibility(
            QueueUrl=QUEUE_URL,
            ReceiptHandle=receipt,
            VisibilityTimeout=300,  # extend to 5 minutes
        )
    
    try:
        process(body)
        sqs.delete_message(QueueUrl=QUEUE_URL, ReceiptHandle=receipt)
    except PermanentError:
        # Don't retry — let it go to DLQ after maxReceiveCount
        sqs.delete_message(QueueUrl=QUEUE_URL, ReceiptHandle=receipt)

# BATCH DELETE: delete multiple messages at once (10x API efficiency)
entries = [
    {'Id': str(i), 'ReceiptHandle': messages[i]['ReceiptHandle']}
    for i in range(len(messages))
]
sqs.delete_message_batch(QueueUrl=QUEUE_URL, Entries=entries)
```

## 3. The SNS → SQS Fan-Out Pattern

```
ARCHITECTURE: one event → many durably processed by multiple systems

           ┌─ SQS Queue (payment-service)      → Payment Worker
           │    DLQ: payment-service-dlq
SNS Topic ─┼─ SQS Queue (inventory-service)    → Inventory Worker
           │    DLQ: inventory-service-dlq
"orders"   ├─ SQS Queue (email-service)         → Email Worker
           │    DLQ: email-service-dlq
           └─ Lambda (fraud-detection)           → Real-time fraud check
              (direct Lambda subscription — no queue needed for fast operations)

WHY SNS→SQS INSTEAD OF SNS DIRECT?
  
  SNS can deliver directly to: HTTP endpoints, Lambda, SQS, email, SMS.
  
  PROBLEM with SNS → HTTP endpoint directly:
    If your HTTP service is down → SNS retries 3 times over 23 days → then DROPS.
    No DLQ. No visibility into failures. Messages lost.
  
  PROBLEM with SNS → Lambda directly:
    Lambda handles retries (3 times) then drops to async event DLQ.
    Limited retry control.
  
  SOLUTION: SNS → SQS → your service
    SQS retains messages indefinitely.
    Your service processes at its own pace.
    Full DLQ support with configurable maxReceiveCount.
    Visibility timeout for processing control.
    Cost: SQS adds ~$0.40 per million messages.
```

## 4. SNS Message Filtering

```python
# SNS message attributes enable filtering at the subscription level
# Subscribers only receive messages matching their filter policy

# Publisher adds attributes
sns.publish(
    TopicArn="arn:aws:sns:us-east-1:123:orders",
    Message=json.dumps({"order_id": 123, "amount": 5000}),
    MessageAttributes={
        "event_type": {
            "DataType": "String",
            "StringValue": "order.placed",
        },
        "country": {
            "DataType": "String",
            "StringValue": "US",
        },
        "amount_cents": {
            "DataType": "Number",
            "StringValue": "5000",
        },
    }
)

# Subscription filter: US high-value orders only
# Set on the SQS subscription:
{
    "event_type": ["order.placed"],
    "country": ["US", "CA"],
    "amount_cents": [{"numeric": [">=", 10000]}]  # $100+
}
# This SQS queue only receives US/CA orders over $100
# Other subscribers get different subsets

# Benefit: consumers only receive relevant events
# No need to filter in application code
# Reduces SQS costs for high-volume topics
```

## 5. Senior Engineer Perspective

**SQS is the right default for AWS teams:**

For most teams building on AWS, SQS should be the first messaging tool you reach for. It's genuinely serverless (no cluster to manage), scales without configuration, integrates with Lambda, ECS, and every other AWS service. The operational overhead is close to zero. Compare this to running a RabbitMQ cluster or even MSK (managed Kafka) — both require ongoing operational attention.

The trade-offs: SQS has higher latency than RabbitMQ (50-100ms vs 1-5ms) and lacks Kafka's event streaming capabilities. For most business-level messaging (email jobs, order processing, webhook delivery): SQS is sufficient and significantly simpler.

**The messageGroupId in FIFO queues is the Kafka partition key equivalent:**

Events with the same messageGroupId are processed in order, with one consumer handling all events for that group at a time. Use it like a Kafka partition key: set it to the entity ID (order_id, user_id) that requires ordered processing.
