## 1. What Is It? (Theory)

RabbitMQ (2007, Pivotal/VMware) is a message broker implementing the AMQP (Advanced Message Queuing Protocol). It receives messages from producers, routes them through exchanges based on configurable rules, stores them in queues, and delivers them to consumers.

RabbitMQ's power lies in its **routing flexibility** — it can do sophisticated message routing that would require complex application-level logic elsewhere. The right routing configuration means services are cleanly decoupled.

**The real-world analogy:**

The postal system has:
- **Post boxes** (producers drop mail here)
- **Sorting facilities** (exchanges — route mail based on address)
- **Mailboxes** (queues — where specific recipients' mail waits)
- **Recipients** (consumers — collect from their mailbox)

The magic is in the sorting facility. It knows the routing rules (bindings) for how to direct each piece of mail to the correct mailbox.

## 2. How It Actually Works (Deep Mechanics)

### The Four Core Components

```
COMPONENT           RESPONSIBILITY                      ANALOGY
──────────────────────────────────────────────────────────────────
Exchange            Receives messages from producers     Sorting facility
                    Routes to queues based on bindings

Queue               Stores messages until consumed       Mailbox
                    Consumers subscribe to queues

Binding             Routing rule: exchange → queue       Address label
                    Defines which messages go where

Message             The data unit being transmitted      Letter
                    Has: body, routing key, headers, properties
```

### What Happens When a Producer Sends a Message

```
FLOW:
  1. Producer → publishes message to Exchange
     specifies: exchange name + routing key
     
  2. Exchange → examines routing key
     checks: which queues am I bound to, and with what routing keys?
     
  3. Exchange → routes message to matching queues
     (may go to 0 queues, 1 queue, or many queues depending on exchange type)
  
  4. Queue → stores message (persistent or transient)
  
  5. Consumer → receives message from its queue
     processes it
     sends acknowledgment

IMPORTANT: Exchanges don't store messages.
  If there's no queue bound to receive a message, it's discarded.
  If the queue doesn't exist: the message is dropped.
  Always ensure queues and bindings are set up before publishing.
```

### Message Properties

```python
# A message is more than just a body:
import aio_pika

message = aio_pika.Message(
    body=json.dumps({"order_id": 123}).encode(),
    
    # DELIVERY MODE: persistence
    delivery_mode=aio_pika.DeliveryMode.PERSISTENT,
    # PERSISTENT: message survives broker restart (written to disk)
    # TRANSIENT:  message lost if broker restarts (faster, for non-critical data)
    
    # CONTENT TYPE: tells consumer how to decode the body
    content_type="application/json",
    
    # CORRELATION ID: for request-reply patterns (match reply to original request)
    correlation_id=str(uuid.uuid4()),
    
    # REPLY TO: where the consumer should send the reply (for RPC patterns)
    reply_to="my-reply-queue",
    
    # EXPIRATION: message TTL in milliseconds (discard if not consumed in time)
    expiration="60000",  # 60 seconds
    
    # PRIORITY: 0-9, higher = processed first (if queue is priority queue)
    priority=5,
    
    # HEADERS: arbitrary key-value pairs (used by headers exchange for routing)
    headers={"environment": "production", "version": "v2"},
)
```

## 3. Durability: Surviving Restarts

This is a critical production concern. Three things must ALL be durable for messages to survive a broker restart:

```
DURABILITY CHECKLIST:

1. DURABLE EXCHANGE:
   exchange.declare(name="orders", durable=True)
   → Exchange definition persists after broker restart

2. DURABLE QUEUE:
   queue.declare(name="order-processor", durable=True)
   → Queue and its messages persist after broker restart

3. PERSISTENT MESSAGES:
   message = Message(body, delivery_mode=DeliveryMode.PERSISTENT)
   → This specific message is written to disk, not just held in RAM

MISSING ANY ONE = MESSAGES LOST ON RESTART:
  Durable exchange + durable queue + transient message → messages lost
  Durable exchange + non-durable queue + persistent message → queue lost on restart
  Non-durable exchange + durable queue → exchange gone on restart (no routing)
  
  ALL THREE MUST BE DURABLE for complete persistence.

PERFORMANCE TRADEOFF:
  Persistent messages are written to disk (fsync).
  This limits throughput: ~20,000 persistent messages/sec on typical hardware.
  vs: ~500,000 transient messages/sec.
  
  For most production use cases: durability is worth the cost.
  For metrics, real-time analytics where loss is acceptable: use transient.
```

## 4. Acknowledgment and Prefetch

```python
import aio_pika

async def consume_orders(channel: aio_pika.Channel):
    """
    Reliable consumer with manual acknowledgment and prefetch.
    """
    # PREFETCH: don't send me more than N unacknowledged messages at once
    # Without this: broker dumps all queued messages on the consumer
    # (can exhaust memory if queue is large)
    await channel.set_qos(prefetch_count=10)
    
    queue = await channel.declare_queue("order-processor", durable=True)
    
    async with queue.iterator() as queue_iter:
        async for message in queue_iter:
            # context manager: acks on success, nacks on exception
            async with message.process(requeue=False):
                try:
                    body = json.loads(message.body)
                    
                    # Your processing here
                    await process_order(body["order_id"])
                    
                    # Context manager sends ACK here (message deleted from queue)
                    
                except PermanentError as e:
                    # This message can never be processed
                    # requeue=False means: don't put back, go to DLQ
                    logger.error("Permanent failure", error=str(e))
                    raise  # context manager sends NACK, message → DLQ
                
                except TransientError as e:
                    # Transient error: requeue for retry
                    # Set requeue=True on the context manager for this case
                    logger.warning("Transient failure, requeuing", error=str(e))
                    raise  # requeue=False → nack without requeue → DLQ
                    # For requeue: use message.nack(requeue=True) explicitly
```

**Prefetch count guidance:**

```
PREFETCH COUNT DECISION:

prefetch_count=0:    No limit (default if not set)
                     → Broker dumps ALL messages to consumer
                     → With 100,000 queued messages: 100,000 in consumer memory
                     → Consumer crash = all 100,000 messages lost from consumer
                     → NEVER use this in production

prefetch_count=1:    One message at a time
                     → Perfect fairness (slow consumers don't hog messages)
                     → Lower throughput (can't pipeline)
                     → Use for: slow, stateful processing

prefetch_count=10-100: Sweet spot for most use cases
                     → Good throughput (prefetch keeps pipeline full)
                     → Bounded memory usage
                     → Use for: typical background jobs

prefetch_count=1000+: For fast, lightweight processing
                     → Maximum throughput
                     → Higher memory usage
                     → Use for: simple transformations, fan-out
```

## 5. Production Setup

```python
# Complete production RabbitMQ setup with aio_pika
import aio_pika
import json
import os
from contextlib import asynccontextmanager

RABBITMQ_URL = os.environ["RABBITMQ_URL"]

@asynccontextmanager
async def get_channel():
    """Get a channel with a robust connection (auto-reconnects)."""
    connection = await aio_pika.connect_robust(
        RABBITMQ_URL,
        reconnect_interval=5.0,   # retry connection every 5s
        heartbeat=60,             # keep-alive heartbeat interval
    )
    async with connection:
        channel = await connection.channel()
        yield channel

async def setup_infrastructure():
    """
    Declare all exchanges, queues, and bindings.
    Call at application startup (idempotent — safe to run multiple times).
    """
    async with get_channel() as channel:
        # Exchange
        events_exchange = await channel.declare_exchange(
            "events",
            aio_pika.ExchangeType.TOPIC,
            durable=True,
        )
        
        # Dead Letter Exchange + Queue
        dlx = await channel.declare_exchange(
            "events.dlx",
            aio_pika.ExchangeType.DIRECT,
            durable=True,
        )
        dlq = await channel.declare_queue(
            "order-processor.dlq",
            durable=True,
        )
        await dlq.bind(dlx, routing_key="order-processor")
        
        # Main queue with DLQ config
        order_queue = await channel.declare_queue(
            "order-processor",
            durable=True,
            arguments={
                "x-dead-letter-exchange": "events.dlx",
                "x-dead-letter-routing-key": "order-processor",
                "x-message-ttl": 86400000,   # 24h: messages expire if not consumed
            }
        )
        
        # Binding: route all order.* events to this queue
        await order_queue.bind(events_exchange, routing_key="order.#")

async def publish_event(event_type: str, payload: dict):
    """Publish an event to the events exchange."""
    async with get_channel() as channel:
        exchange = await channel.declare_exchange(
            "events",
            aio_pika.ExchangeType.TOPIC,
            durable=True,
        )
        await exchange.publish(
            aio_pika.Message(
                body=json.dumps(payload).encode(),
                delivery_mode=aio_pika.DeliveryMode.PERSISTENT,
                content_type="application/json",
            ),
            routing_key=event_type,  # e.g., "order.placed", "order.cancelled"
        )
```

## 6. Senior Engineer Perspective

**The declaration idempotency rule:**

Always declare exchanges, queues, and bindings in your application startup, not just once manually. Declarations are idempotent (calling twice with the same parameters does nothing). This means your service self-heals: if RabbitMQ is restarted and loses its configuration (non-durable setup), your service recreates everything it needs when it reconnects.

**The prefetch_count omission is the most common production bug:**

Without `set_qos(prefetch_count=N)`, RabbitMQ pushes all queued messages to your consumer the moment it connects. If you have 50,000 queued messages and 2 consumers: one consumer gets 30,000 messages in its local buffer, the other gets 20,000. Now one consumer is overwhelmed; the other is idle. Worse: if the overwhelmed consumer crashes, those 30,000 messages are redelivered, and another consumer gets them all. Always set prefetch.
