## 1. What Is It? (Theory)

The exchange type is the most important RabbitMQ design decision. It determines the routing logic — how messages get from producers to queues. Choosing the wrong exchange type means building routing logic in application code that should live in the broker.

Each exchange type implements a different routing pattern. Master all four and you can implement virtually any messaging architecture.

## 2. How It Actually Works (Deep Mechanics)

### Direct Exchange: One-to-One Exact Match

```
ROUTING RULE: Message goes to queues with binding key exactly matching the routing key.

EXAMPLE SETUP:
  Exchange: "tasks" (type: direct)
  
  Binding 1: routing_key="email"   → queue: "email-worker"
  Binding 2: routing_key="sms"     → queue: "sms-worker"
  Binding 3: routing_key="push"    → queue: "push-worker"

MESSAGE ROUTING:
  Producer sends: routing_key="email"
  → Only "email-worker" queue receives it
  
  Producer sends: routing_key="sms"
  → Only "sms-worker" queue receives it
  
  Producer sends: routing_key="unknown"
  → No queue matches. Message discarded.

MULTIPLE QUEUES WITH SAME BINDING KEY:
  Binding 1: routing_key="email" → queue: "email-worker-1"
  Binding 2: routing_key="email" → queue: "email-worker-2"
  
  Producer sends: routing_key="email"
  → BOTH queues receive the message (fan-out within direct exchange)
  → Workers compete for messages within each queue separately

USE WHEN:
  Routing key maps deterministically to one specific worker type.
  Simple: "this type of work → that set of workers"
  
  Example: task queue routing by task type
  routing_key="resize-image" → image-resize-workers queue
  routing_key="send-email" → email-sender-workers queue
```

### Topic Exchange: Pattern-Based Routing

```
ROUTING RULE: Routing keys use dot notation (log.error, user.created).
              Bindings use wildcards:
              * = exactly one word segment
              # = zero or more word segments

EXAMPLE SETUP:
  Exchange: "events" (type: topic)
  
  Binding 1: pattern="order.*"     → queue: "order-service"
  Binding 2: pattern="payment.*"   → queue: "payment-service"
  Binding 3: pattern="*.created"   → queue: "audit-log"
  Binding 4: pattern="user.#"      → queue: "user-pipeline"
  Binding 5: pattern="#"           → queue: "full-event-log"

MESSAGE ROUTING:
  Producer sends: routing_key="order.created"
  
  order.*    matches! → "order-service" receives it
  *.created  matches! → "audit-log" receives it
  #          matches! → "full-event-log" receives it
  payment.*  no match
  user.#     no match (doesn't start with "user")
  
  Message delivered to: "order-service", "audit-log", "full-event-log"
  
  Producer sends: routing_key="user.profile.updated"
  
  user.#     matches! (# = zero or more words)
  #          matches!
  *.created  no match (doesn't end in "created")
  order.*    no match

WILDCARD RULES:
  "user.*"      matches: user.created, user.deleted  (one word after "user.")
  "user.*"      no match: user.profile.updated       (two words after "user.")
  "user.#"      matches: user.created, user.profile.updated, user.x.y.z
  "*.*.created" matches: user.profile.created        (exactly 3 segments)

USE WHEN:
  Services subscribe to categories of events they care about.
  Adding new event types: no reconfiguration needed (patterns cover them).
  
  MOST FLEXIBLE and most commonly used exchange type in production.
  When in doubt, use topic exchange.
```

### Fanout Exchange: Broadcast to All

```
ROUTING RULE: Ignores routing key entirely. Sends to ALL bound queues.

EXAMPLE SETUP:
  Exchange: "announcements" (type: fanout)
  
  Binding 1: → queue: "notification-service"
  Binding 2: → queue: "analytics-service"
  Binding 3: → queue: "audit-service"
  Binding 4: → queue: "cache-invalidation-service"

MESSAGE ROUTING:
  Producer sends: routing_key="anything" (routing key ignored)
  → ALL 4 queues receive a copy of the message

  Adding a new subscriber queue:
  Just bind it to the fanout exchange.
  No changes to producers. No changes to existing consumers.

USE WHEN:
  Every subscriber must receive every message.
  Number of subscribers may change (fanout handles this automatically).
  
  Example: user settings changed → invalidate every relevant cache
           system-wide maintenance announcement → all service queues
           price list updated → all regional services must refresh

NOTE: Fanout exchange with lots of bound queues = high message multiplication.
  1 message published → N messages created (one per bound queue).
  Be mindful of storage if N is large.
```

### Headers Exchange: Route by Message Headers

```
ROUTING RULE: Routes based on message header values (not routing key).
              Binding specifies headers that must match.
              x-match: "all" = all headers must match
              x-match: "any" = at least one header must match

EXAMPLE SETUP:
  Exchange: "events" (type: headers)
  
  Binding 1: headers={priority: "high", env: "prod"}  x-match: "all"
             → queue: "critical-processor"
  
  Binding 2: headers={priority: "low"}                x-match: "all"
             → queue: "batch-processor"
  
  Binding 3: headers={env: "dev"}                     x-match: "any"
             → queue: "dev-debug-sink"

MESSAGE ROUTING:
  Message headers: {priority: "high", env: "prod"}
  → Matches binding 1 (all match)
  → Routes to "critical-processor"
  
  Message headers: {priority: "low", env: "prod"}
  → Matches binding 2 (priority=low matches)
  → Routes to "batch-processor"
  
  Message headers: {priority: "high", env: "dev"}
  → Matches binding 1? No (env must be "prod")
  → Matches binding 3? Yes (env=dev matches with x-match: "any")

USE WHEN:
  Routing criteria doesn't fit neatly in a string routing key.
  Multiple independent dimensions of routing needed.
  
  In practice: rarely used. Topic exchange handles most cases.
  Headers exchange shines for: multi-dimensional routing that would
  require complex routing key encoding otherwise.
```

## 3. Exchange Comparison

```
EXCHANGE     ROUTING BASED ON          BEST FOR
──────────────────────────────────────────────────────────────
Direct       Exact routing key match   Simple work queues, task distribution
Topic        Pattern wildcards (*, #)  Event-driven systems, services subscribing to categories
Fanout       Ignores routing key       Broadcasts, cache invalidation, announcements
Headers      Message header values     Multi-dimensional routing (rare)
```

## 4. Real-World Design: E-commerce Event Bus

```python
# Design: topic exchange as the backbone for an e-commerce event bus

TOPIC EXCHANGE: "ecommerce.events"

ROUTING KEYS (from producers):
  order.placed
  order.payment.completed
  order.payment.failed
  order.shipped
  order.delivered
  order.cancelled
  user.registered
  user.profile.updated
  inventory.updated
  inventory.low_stock

CONSUMER QUEUE BINDINGS:
  "payment-service":     order.placed
                         (only processes new orders)
  
  "inventory-service":   order.placed
                         order.cancelled
                         (reserve stock on order, release on cancel)
  
  "email-service":       order.*
                         user.registered
                         (all order events + new user welcome email)
  
  "analytics-service":   #
                         (all events — analytics sees everything)
  
  "fraud-detection":     order.placed
                         order.payment.*
                         (check fraud on payment events)
  
  "warehouse-service":   order.payment.completed
                         order.cancelled
                         (start fulfillment after payment, cancel it if order cancelled)

ADDING A NEW SERVICE (e.g., SMS notifications):
  Create queue: "sms-service"
  Add binding: order.shipped → "sms-service"
  
  No changes to any existing service.
  The exchange routes to the new queue automatically.

# Python setup
async def setup_ecommerce_bus(channel):
    exchange = await channel.declare_exchange(
        "ecommerce.events", aio_pika.ExchangeType.TOPIC, durable=True
    )
    
    queues = {
        "payment-service":   ["order.placed"],
        "inventory-service": ["order.placed", "order.cancelled"],
        "email-service":     ["order.*", "user.registered"],
        "analytics-service": ["#"],
        "fraud-detection":   ["order.placed", "order.payment.*"],
        "warehouse-service": ["order.payment.completed", "order.cancelled"],
    }
    
    for queue_name, patterns in queues.items():
        queue = await channel.declare_queue(queue_name, durable=True)
        for pattern in patterns:
            await queue.bind(exchange, routing_key=pattern)
```

## 5. Senior Engineer Perspective

**The default exchange (a hidden direct exchange):**

RabbitMQ has a built-in "default exchange" — a direct exchange with an empty name (`""`). Every queue is automatically bound to it with its own name as the routing key. So publishing to `exchange=""` with `routing_key="my-queue"` sends directly to the queue named "my-queue".

This is useful for simple setups and testing, but don't use it in production. It bypasses the exchange routing system and tightly couples producers to specific queue names.

**Topic exchange is almost always the right choice:**

When in doubt, use a topic exchange. It's the most flexible — you can always start with specific routing keys (`order.placed`) and expand patterns as needed (`order.*`, `#`). It handles fan-out (multiple queues bound to the same pattern), selective consumption (subscribe to only what you need), and future extensibility (new event types routed automatically by existing patterns).

The only reasons to choose another type:
- Direct: slightly lower overhead when routing is always exact match (rarely matters)
- Fanout: every single subscriber must always get every message (simpler, no routing key needed)
- Headers: complex multi-attribute routing that can't be expressed as a string pattern
