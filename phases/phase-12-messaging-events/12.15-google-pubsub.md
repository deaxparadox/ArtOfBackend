## 1. What Is It? (Theory)

Google Cloud Pub/Sub is GCP's fully managed, globally scalable message service. Like SQS+SNS combined into one service, it provides both pub/sub fan-out AND durable message queuing. It scales to millions of messages per second without configuration and integrates deeply with GCP's data services (BigQuery, Dataflow, Cloud Run).

---

## 2. Core Concepts

```
PUB/SUB MODEL:

  Publisher → publishes to TOPIC
  
  Topic → multiple subscriptions
    Subscription A: "payment-processor"  → payment service consumes
    Subscription B: "analytics"          → analytics service consumes
    Subscription C: "bigquery-sink"      → streams directly to BigQuery
  
  Each subscription independently receives ALL messages from the topic.
  Messages retained until ALL subscriptions acknowledge (or TTL expires).
  
PULL vs PUSH SUBSCRIPTIONS:
  
  PULL (your service polls):
    subscriber.subscribe(subscription_path, callback)
    Your service pulls messages when ready.
    Good for: services with variable load, batch processing.
  
  PUSH (Pub/Sub delivers to your endpoint):
    Pub/Sub sends HTTP POST to your configured endpoint.
    Your service must respond 2xx within ack_deadline_seconds.
    Good for: serverless (Cloud Run, Cloud Functions), simple webhooks.
    
  BIGQUERY SUBSCRIPTION (no code needed):
    Messages streamed directly into BigQuery table.
    Configure schema, Pub/Sub handles the rest.
    Zero-code pipeline from any publisher to BigQuery.
```

## 3. Key Production Configuration

```python
from google.cloud import pubsub_v1
from google.cloud.pubsub_v1.types import FlowControl
import json

# SUBSCRIBER with flow control (prevent overwhelming your service)
subscriber = pubsub_v1.SubscriberClient()
subscription_path = subscriber.subscription_path("my-project", "orders-payment-sub")

def callback(message: pubsub_v1.types.ReceivedMessage) -> None:
    body = json.loads(message.data.decode())
    
    try:
        process_payment(body)
        message.ack()           # success: remove from subscription
    except TransientError:
        message.nack()          # failure: redeliver after ack_deadline
    except PermanentError:
        # Pub/Sub will move to dead letter topic after max_delivery_attempts
        message.ack()           # ack it to prevent more retries
        log_to_dead_letter_manually(body)

# Flow control: limit outstanding messages to prevent memory exhaustion
flow_control = FlowControl(
    max_messages=100,               # at most 100 unacked messages
    max_bytes=10 * 1024 * 1024,    # at most 10MB unacked
)

future = subscriber.subscribe(
    subscription_path,
    callback,
    flow_control=flow_control,
)

# PUBLISHER with ordering keys (like Kafka partition key)
publisher = pubsub_v1.PublisherClient(
    publisher_options=pubsub_v1.types.PublisherOptions(
        enable_message_ordering=True,  # required for ordered delivery
    )
)
topic_path = publisher.topic_path("my-project", "orders")

publisher.publish(
    topic_path,
    json.dumps({"event": "order.placed", "order_id": 123}).encode(),
    ordering_key="order-123",    # all messages with same ordering_key → in order
)
```

## 4. Dead Letter Topics

```python
# Pub/Sub dead letter configuration (via Terraform)
resource "google_pubsub_subscription" "orders_payment" {
    name  = "orders-payment-sub"
    topic = google_pubsub_topic.orders.name
    
    dead_letter_policy {
        dead_letter_topic     = google_pubsub_topic.orders_dlq.id
        max_delivery_attempts = 5   # after 5 failures → dead letter topic
    }
    
    ack_deadline_seconds = 60       # must ack within 60s or redelivered
    
    retry_policy {
        minimum_backoff = "10s"
        maximum_backoff = "600s"    # exponential backoff: 10s → 600s
    }
}
```

## 5. Pub/Sub vs. SQS/SNS

```
GOOGLE PUB/SUB:
  ✓ Global by default (replicated across 3+ regions automatically)
  ✓ Single service replaces both SNS and SQS
  ✓ Native BigQuery integration (stream to BQ without code)
  ✓ Push subscriptions for serverless
  ✓ Message ordering with ordering keys
  ✗ Slightly more complex subscription model than SQS
  ✗ GCP-specific (no AWS equivalent)
  Best for: GCP-native workloads, streaming to BigQuery, global multi-region

AWS SQS + SNS:
  ✓ Simpler mental model (queue = SQS, fan-out = SNS+SQS)
  ✓ Deep AWS ecosystem integration
  ✓ SQS FIFO for strict ordering
  ✓ Better Lambda integration
  ✗ Two services to manage (SNS + SQS) for fan-out patterns
  Best for: AWS-native workloads
```
