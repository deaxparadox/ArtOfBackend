## 1. What Is It? (Theory)

A **Dead Letter Queue (DLQ)** is a special secondary queue where messages that cannot be successfully processed are automatically moved after N failed delivery attempts.

Without a DLQ, there are only two outcomes for a message that repeatedly fails:
1. **Keep retrying forever** — the "poison message" blocks the queue (or consumer resources) indefinitely, potentially preventing other messages from being processed
2. **Discard silently** — the message and its data are lost with no record of what went wrong

Both outcomes are bad. The DLQ is the correct third option: move the problematic message somewhere safe, stop retrying it, alert a human, and continue processing other messages normally.

**The real-world analogy:**

In a physical mail system, mail that cannot be delivered (wrong address, recipient moved, no postage) goes to a "dead letter office" — a special facility where humans inspect undeliverable mail and decide what to do: return to sender, investigate, or discard. The DLQ is the software equivalent.

## 2. How It Actually Works (Deep Mechanics)

### What Makes a Message "Dead"

Messages end up in the DLQ for specific, configured reasons:

```
REASONS A MESSAGE GOES TO THE DLQ:

1. MAX RECEIVE COUNT EXCEEDED:
   Message received and unacknowledged N times.
   Example: received 5 times, processing failed each time → DLQ.
   
2. MESSAGE EXPIRED (TTL):
   Message stayed in the queue longer than the configured TTL.
   Message was never picked up (no consumers running) → DLQ.
   
3. QUEUE CAPACITY EXCEEDED:
   Source queue reached its maximum size.
   Some brokers redirect overflow to DLQ instead of dropping.
   
4. CONSUMER EXPLICITLY REJECTS (some brokers):
   Consumer determines the message is unprocessable.
   Sends explicit negative acknowledgment (nack without requeue).
   Message goes directly to DLQ.
```

### The Poison Message Problem

A "poison message" is a message that always causes consumer failures. Without protection, it creates a catastrophic feedback loop:

```
POISON MESSAGE SCENARIO (no DLQ):

Queue: [Good msg][Good msg][POISON msg][Good msg][Good msg]

Consumer receives POISON msg.
Consumer crashes (unhandled exception, OOM, assertion failure).
Message visibility timeout expires.
POISON msg is redelivered.
Consumer crashes again.
...repeats forever...

Meanwhile: good messages sitting behind the poison message are delayed.
Other consumers may also pick up the poison message and crash.
System can become completely stuck on one bad message.

WITH DLQ:
Queue: [Good msg][Good msg][POISON msg][Good msg][Good msg]

Consumer receives POISON msg.
Consumer crashes.
Message returned to queue. Receive count: 2.
Consumer receives POISON msg again.
Consumer crashes.
Receive count: 3.
...
Receive count: 5 (maxReceiveCount reached).
SQS/broker automatically moves POISON msg → DLQ.

Original queue: [Good msg][Good msg][Good msg][Good msg]
Consumers process these normally.
DLQ: [POISON msg] ← waiting for human investigation.
Alert fires: "DLQ has 1 message"
```

## 3. Configuring DLQs in Production

### AWS SQS

```hcl
# Terraform: SQS with DLQ
resource "aws_sqs_queue" "order_processing" {
    name                        = "order-processing"
    visibility_timeout_seconds  = 60      # must be >= max processing time
    message_retention_seconds   = 86400   # 1 day
    
    redrive_policy = jsonencode({
        deadLetterTargetArn = aws_sqs_queue.order_processing_dlq.arn
        maxReceiveCount     = 5  # move to DLQ after 5 failed deliveries
    })
}

resource "aws_sqs_queue" "order_processing_dlq" {
    name                       = "order-processing-dlq"
    message_retention_seconds  = 1209600  # 14 days — time to investigate
}

# CRITICAL: Alert when DLQ receives any message
resource "aws_cloudwatch_metric_alarm" "dlq_has_messages" {
    alarm_name          = "order-processing-dlq-not-empty"
    comparison_operator = "GreaterThanThreshold"
    threshold           = 0
    evaluation_periods  = 1
    period              = 60
    statistic           = "Sum"
    metric_name         = "ApproximateNumberOfMessagesVisible"
    namespace           = "AWS/SQS"
    dimensions = {
        QueueName = aws_sqs_queue.order_processing_dlq.name
    }
    alarm_actions = [var.pagerduty_sns_topic_arn]
}
```

### RabbitMQ

```python
import aio_pika

async def setup_queue_with_dlq(channel):
    """Configure a queue with dead letter routing."""
    
    # First: declare the DLQ (dead letter exchange and queue)
    dlq_exchange = await channel.declare_exchange(
        "dlx",  # dead letter exchange
        aio_pika.ExchangeType.DIRECT,
        durable=True,
    )
    dlq = await channel.declare_queue(
        "order-processing.dlq",
        durable=True,
    )
    await dlq.bind(dlq_exchange, routing_key="order-processing")
    
    # Then: declare the main queue with DLQ settings
    main_queue = await channel.declare_queue(
        "order-processing",
        durable=True,
        arguments={
            "x-dead-letter-exchange": "dlx",
            "x-dead-letter-routing-key": "order-processing",
            "x-message-ttl": 86400000,   # 24 hours: discard if unprocessed
        }
    )
    
    return main_queue, dlq
```

## 4. What To Do with DLQ Messages

A DLQ is not a trash can — it's a hospital waiting room. Messages there need attention.

```
DLQ INVESTIGATION WORKFLOW:

STEP 1: RECEIVE THE ALERT
  "order-processing-dlq has 3 messages"
  
STEP 2: LOOK AT THE MESSAGE
  aws sqs receive-message --queue-url $DLQ_URL
  
  Typical DLQ message categories:
  
  Category A: Application Bug
    Error: "KeyError: 'discount_code'" 
    Message has a new field your code doesn't handle.
    Fix: update your code to handle the new field.
    Resolution: fix bug, deploy, redrive DLQ → source queue.
  
  Category B: Data Corruption / Bad Input
    Error: "Invalid UUID format in user_id field"
    Message was malformed when published.
    Fix: fix the producer to validate before publishing.
    Resolution: possibly process manually, discard the bad message.
  
  Category C: Dependency Failure
    Error: "ConnectionRefused: cannot connect to payment-service"
    Your code was correct, but a dependency was down.
    The message failed because of transient infrastructure issue.
    Resolution: simply redrive DLQ → source queue. Will succeed now.
  
  Category D: Business Logic Error
    Error: "Order #123 already cancelled, cannot process payment"
    The message arrived out of order or represents an impossible state.
    Fix: add proper state validation, decide if this is a bug or expected.
    Resolution: depends on the business — might be safe to discard.

STEP 3: REDRIVE (if appropriate)
  Move messages from DLQ back to the original queue for reprocessing.
  
  AWS SQS: Start DLQ redrive in console or via API
  Custom: read from DLQ, send to original queue, delete from DLQ

STEP 4: NEVER DELETE MESSAGES FROM DLQ WITHOUT UNDERSTANDING THEM
  Each DLQ message represents a real event that didn't get processed.
  Deleting without investigation hides bugs and causes data loss.
```

## 5. Code Example: DLQ Consumer for Investigation

```python
import boto3, json

sqs = boto3.client('sqs')
DLQ_URL = os.environ['DLQ_URL']
SOURCE_URL = os.environ['SOURCE_QUEUE_URL']

def investigate_dlq():
    """
    Read from DLQ for investigation. Don't delete messages here.
    Only delete/redrive after understanding the problem.
    """
    response = sqs.receive_message(
        QueueUrl=DLQ_URL,
        MaxNumberOfMessages=10,
        AttributeNames=['All'],           # include metadata (receive count, etc.)
        MessageAttributeNames=['All'],
    )
    
    for message in response.get('Messages', []):
        receive_count = int(message['Attributes'].get('ApproximateReceiveCount', 0))
        first_received = message['Attributes'].get('ApproximateFirstReceiveTimestamp')
        
        body = json.loads(message['Body'])
        
        print(f"Message ID: {message['MessageId']}")
        print(f"Receive attempts: {receive_count}")
        print(f"First received: {first_received}")
        print(f"Body: {json.dumps(body, indent=2)}")
        print("---")
    
    # Don't delete messages — leave them for further investigation

def redrive_dlq_to_source():
    """
    After fixing the bug: move all DLQ messages back to the source queue.
    """
    redriven = 0
    while True:
        response = sqs.receive_message(
            QueueUrl=DLQ_URL,
            MaxNumberOfMessages=10,
        )
        messages = response.get('Messages', [])
        if not messages:
            break
        
        for message in messages:
            # Send to original queue
            sqs.send_message(
                QueueUrl=SOURCE_URL,
                MessageBody=message['Body'],
            )
            # Delete from DLQ
            sqs.delete_message(
                QueueUrl=DLQ_URL,
                ReceiptHandle=message['ReceiptHandle'],
            )
            redriven += 1
    
    print(f"Redriven {redriven} messages from DLQ to source queue")
```

## 6. DLQ Best Practices

```
ALWAYS:
  □ Configure DLQ on every production queue (no exceptions)
  □ Set alert on DLQ message count > 0
  □ Set DLQ retention to 14 days (maximum investigation time)
  □ Treat DLQ messages as production incidents

MAXRECEIVECOUNT GUIDANCE:
  3-5 retries: appropriate for most transient failures
  1 retry: for non-idempotent operations that already partially succeeded
  10+ retries: only if you expect many transient failures (network-heavy)

VISIBILITY TIMEOUT vs MAXRECEIVECOUNT:
  Visibility timeout: time between delivery attempts
  maxReceiveCount: number of delivery attempts before DLQ
  
  Too short visibility timeout + high maxReceiveCount:
  = many rapid retries = hammers a failing dependency
  
  Solution: use exponential backoff between retries
  (visibility timeout is constant in SQS, handle in application code)

NEVER:
  ✗ Ignore DLQ messages ("they'll go away")
  ✗ Delete DLQ messages without understanding them
  ✗ Use DLQ as a discard bin
  ✗ Set maxReceiveCount to 1 for retryable operations
```

## 7. Senior Engineer Perspective

**The incident that teaches respect for DLQs:**

A team builds an order processing system. No DLQ configured (they'll add it later). A bad deployment introduces a bug that causes 1% of orders to fail processing. Those 1% of orders keep retrying. The queue grows. Consumers spend 50% of their time retrying the bad orders, slowing down the other 99%. After 6 hours, the team discovers 10,000 messages stuck in the queue. No DLQ means no visibility into which orders failed. Manual investigation of 10,000 SQS messages to find the bad ones. Data loss for customers whose orders failed silently.

With a DLQ: the 100 bad orders go to the DLQ immediately after 5 retries. The alert fires. The other 9,900 orders process normally. The bug is identified in the DLQ in 10 minutes.

**The maxReceiveCount is also a bug detector:**

If your DLQ regularly has messages in it, your system has a class of inputs it can't handle. This is valuable signal — not just an operational nuisance. Every DLQ message represents a production failure that succeeded in reaching your system but not your business logic. Investigate every class of DLQ failure until you either fix the code or explicitly accept that failure mode.
