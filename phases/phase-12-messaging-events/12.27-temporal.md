## 1. What Is It? (Theory)

Temporal is a workflow execution platform that makes long-running, multi-step processes **durable** — they survive server crashes, network failures, and deployments. When you write a workflow in Temporal, each step is automatically checkpointed. If the server crashes mid-workflow, Temporal resumes exactly where it left off.

**The gap Temporal fills:**

Celery and BullMQ are great for single, isolated tasks. But what about multi-step workflows that span minutes, hours, or days? A standard job queue has no memory between steps. If your process crashes between step 3 and step 4 of a 10-step workflow, you have to figure out what got done, what didn't, and how to resume — manually.

Temporal does this automatically.

## 2. How It Actually Works (Deep Mechanics)

### The Durability Model

```
HOW TEMPORAL ACHIEVES DURABILITY:

Every await in a workflow is a checkpoint.
Temporal records the result of each activity in its event history.

Workflow:
  step 1: charge_card()         ← recorded: "charge succeeded, result=$100"
  step 2: reserve_inventory()   ← recorded: "reservation succeeded"
  step 3: send_confirmation()   ← server CRASHES HERE
  step 4: update_order_status() ← never ran

On restart:
  Temporal has the event history.
  It replays the workflow code deterministically.
  When it hits step 1: result is already in history → return stored result (no re-execution)
  When it hits step 2: result is already in history → return stored result (no re-execution)
  When it hits step 3: no result in history → ACTUALLY execute send_confirmation()
  When it hits step 4: execute update_order_status()
  
  Result: workflow resumes exactly at step 3.
  Steps 1 and 2 were NOT re-executed (no double-charge, no double-reservation).
```

### Python SDK Example

```python
from temporalio import workflow, activity
from temporalio.client import Client
from temporalio.worker import Worker
from temporalio.common import RetryPolicy
from datetime import timedelta

# ACTIVITIES: the actual work (side effects, external calls)
@activity.defn
async def charge_card(order_id: str, amount_cents: int) -> str:
    """Call payment provider. Returns transaction_id."""
    result = await payment_gateway.charge(order_id, amount_cents)
    if not result.success:
        raise ValueError(f"Payment failed: {result.error}")
    return result.transaction_id

@activity.defn
async def reserve_inventory(order_id: str, product_id: str, quantity: int) -> bool:
    """Reserve stock. Returns True if successful."""
    return await inventory_service.reserve(order_id, product_id, quantity)

@activity.defn
async def send_order_confirmation(order_id: str, user_email: str) -> None:
    """Send confirmation email."""
    await email_service.send_confirmation(order_id, user_email)

@activity.defn
async def release_inventory(order_id: str) -> None:
    """Compensating transaction: release reserved stock."""
    await inventory_service.release(order_id)

@activity.defn
async def issue_refund(transaction_id: str) -> None:
    """Compensating transaction: refund the charge."""
    await payment_gateway.refund(transaction_id)

# WORKFLOW: the coordination logic
@workflow.defn
class PlaceOrderWorkflow:
    @workflow.run
    async def run(self, order_id: str, amount_cents: int,
                  product_id: str, quantity: int, user_email: str) -> str:
        
        # Step 1: Charge the card
        # RetryPolicy: retry up to 3 times, wait 2s between retries
        transaction_id = await workflow.execute_activity(
            charge_card,
            args=[order_id, amount_cents],
            start_to_close_timeout=timedelta(minutes=2),
            retry_policy=RetryPolicy(max_attempts=3, initial_interval=timedelta(seconds=2)),
        )
        
        # Step 2: Reserve inventory
        # If this fails: compensate by refunding the charge
        try:
            reserved = await workflow.execute_activity(
                reserve_inventory,
                args=[order_id, product_id, quantity],
                start_to_close_timeout=timedelta(minutes=1),
            )
            if not reserved:
                raise ValueError("Inventory not available")
        except Exception:
            # Compensate: refund the already-processed charge
            await workflow.execute_activity(
                issue_refund,
                args=[transaction_id],
                start_to_close_timeout=timedelta(minutes=2),
            )
            raise
        
        # Step 3: Send confirmation email (fire and forget — don't block on this)
        await workflow.execute_activity(
            send_order_confirmation,
            args=[order_id, user_email],
            start_to_close_timeout=timedelta(minutes=1),
            retry_policy=RetryPolicy(max_attempts=5),  # email delivery can be slow
        )
        
        return transaction_id

# WORKER: connects to Temporal server and executes workflows/activities
async def start_worker():
    client = await Client.connect("localhost:7233")
    
    worker = Worker(
        client,
        task_queue="orders",
        workflows=[PlaceOrderWorkflow],
        activities=[charge_card, reserve_inventory, send_order_confirmation,
                    release_inventory, issue_refund],
    )
    await worker.run()

# CLIENT: starts workflows from your application
async def place_order_api(order_data: dict):
    client = await Client.connect("localhost:7233")
    
    handle = await client.start_workflow(
        PlaceOrderWorkflow.run,
        args=[
            order_data["order_id"],
            order_data["amount_cents"],
            order_data["product_id"],
            order_data["quantity"],
            order_data["user_email"],
        ],
        id=f"place-order-{order_data['order_id']}",  # workflow ID (unique)
        task_queue="orders",
    )
    
    # Option 1: get the result (blocks until workflow completes)
    result = await handle.result()
    
    # Option 2: return immediately, check status later
    return {"workflow_id": handle.id, "run_id": handle.first_execution_run_id}
```

## 3. Temporal vs. Celery

```
CELERY:
  ✓ Simple: define a function, call .delay()
  ✓ Mature ecosystem, well-known
  ✓ Good for: isolated single-step tasks
  ✗ No native multi-step workflow coordination
  ✗ Saga requires manual implementation
  ✗ Crash between steps = manual recovery
  ✗ Long-running workflows require custom state management

TEMPORAL:
  ✓ Built for multi-step, durable workflows
  ✓ Crash recovery is automatic (replay from checkpoint)
  ✓ Saga with compensation is natural
  ✓ Long-running workflows (minutes, hours, days) are first-class
  ✓ Built-in retry, timeout, cancellation per activity
  ✗ More complex to set up (Temporal server + worker)
  ✗ Learning curve (determinism requirements for workflow code)
  ✗ Heavier infrastructure

WHEN TO CHOOSE TEMPORAL:
  ✓ Multi-step workflows requiring saga compensation
  ✓ Workflows that span minutes to days
  ✓ Workflows requiring human approval steps
  ✓ You're implementing distributed sagas and want durability without custom code

WHEN TO CHOOSE CELERY/BULLMQ:
  ✓ Independent, single-step background jobs
  ✓ Simple retry logic
  ✓ Simpler infrastructure requirements
  ✓ Team unfamiliar with Temporal's programming model
```

## 4. Senior Engineer Perspective

**The determinism requirement:**

Temporal replays workflow code to restore state. For this to work correctly, workflow code must be **deterministic**: the same inputs must always produce the same sequence of execution. This means:

- Never use `datetime.now()` in workflow code (use `workflow.now()` which returns deterministic time)
- Never use `random()` in workflow code
- Never access external resources directly in workflow functions (only in activities)
- Never use non-deterministic data structures

Activities are where side effects live (database calls, HTTP calls, random values). Workflows are pure coordination logic. This separation makes workflows testable and replayable.

**Temporal Cloud:**

Self-hosting Temporal requires operational expertise. For most teams: **Temporal Cloud** (managed Temporal) is the right choice. It's expensive relative to running your own Temporal, but eliminates cluster management entirely.
