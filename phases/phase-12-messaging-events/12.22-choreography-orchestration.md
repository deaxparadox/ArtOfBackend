## 1. What Is It? (Theory)

Both choreography and orchestration are coordination styles for sagas and multi-service workflows. They answer the question: "Who decides what happens next?"

**Choreography:** No coordinator. Each service knows its role and reacts to events independently. The overall workflow emerges from the sum of independent reactions.

**Orchestration:** A central coordinator (orchestrator) explicitly directs each service — "you do X, then you do Y."

**The real-world analogy:**

A jazz band is **choreography**: each musician listens to the others and reacts appropriately. There's no conductor telling each person what to play. The music emerges from their collective responses.

An orchestra is **orchestration**: a conductor explicitly signals each section — "strings now, brass after bar 32." The conductor coordinates the whole.

Both produce music. The choice depends on the complexity of coordination needed.

## 2. How It Actually Works (Deep Mechanics)

### Choreography: Decentralized Event Reactions

```
CHOREOGRAPHY FLOW — "Place Order Saga":

  Services react independently to events.
  No service knows the full workflow.
  
  [OrderService]         [InventoryService]      [PaymentService]
       |                        |                       |
  Creates order                 |                       |
  Publishes:                    |                       |
  "order.created" ─────────────►|                       |
                          Receives event                |
                          Reserves stock                |
                          Publishes:                    |
                          "inventory.reserved" ─────────►|
                                                  Receives event
                                                  Charges card
                                                  Publishes:
                                                  "payment.completed"
       |◄──────────────────────────────────────────────|
  Receives event
  Updates to CONFIRMED
```

**Choreography properties:**

```
PROS:
  ✓ No single point of failure (no orchestrator can fail)
  ✓ Services are fully decoupled (Order Service doesn't know about Payment Service)
  ✓ Easy to add new participants (subscribe to existing events)
  ✓ Natural event-driven model

CONS:
  ✗ Hard to understand the full workflow (distributed across services)
  ✗ "Where is this order in the process?" requires querying all services
  ✗ Circular dependencies possible (Service A reacts to B's event, B reacts to A's)
  ✗ Testing the full workflow requires all services running
  ✗ Adding a new step may require changing multiple services' event handlers
```

### Orchestration: Centralized Coordination

```
ORCHESTRATION FLOW — "Place Order Saga":

  A central "OrderSaga" orchestrator explicitly directs each step.
  
  [OrderSaga Orchestrator]      [InventoryService]      [PaymentService]
           |                            |                       |
     Starts saga                        |                       |
     Creates order                      |                       |
     Sends command:                     |                       |
     "ReserveInventory" ───────────────►|                       |
                                 Reserves stock                 |
                                 Replies: "InventoryReserved"   |
           |◄──────────────────────────|                       |
     Receives reply                     |                       |
     Records state: "INVENTORY_RESERVED"|                       |
     Sends command:                     |                       |
     "ProcessPayment" ──────────────────────────────────────────►|
                                                          Charges card
                                                          Replies: "PaymentProcessed"
           |◄──────────────────────────────────────────────────|
     Receives reply
     Updates order to CONFIRMED
     Saga complete
     
     IF PaymentFailed:
     Sends command:
     "ReleaseInventory" ───────────────►|
                                 Releases reservation
                                 Replies: "InventoryReleased"
           |◄──────────────────────────|
     Marks order as FAILED
```

**Orchestration properties:**

```
PROS:
  ✓ Central visibility: "where is this order?" → query the orchestrator
  ✓ Easy to understand the complete workflow (all logic in one place)
  ✓ Easy to add, remove, or reorder steps (change only the orchestrator)
  ✓ Clear error handling and compensation logic

CONS:
  ✗ Orchestrator knows about all participants (coupling)
  ✗ Orchestrator is a potential bottleneck and single point of failure
  ✗ Temptation to put business logic in the orchestrator (God Object anti-pattern)
  ✗ More infrastructure (the orchestrator process must run)
```

## 3. Practical Decision Guide

```
USE CHOREOGRAPHY WHEN:
  ✓ Simple workflows (3-4 steps)
  ✓ Services must remain truly decoupled
  ✓ New services frequently subscribe to existing events
  ✓ Team values autonomy of individual services
  ✓ The workflow rarely needs debugging ("it just works")

USE ORCHESTRATION WHEN:
  ✓ Complex workflows (5+ steps, conditional branches)
  ✓ Workflow visibility and debugging are important
  ✓ Business logic requires coordination across many services
  ✓ Workflow can change frequently (one place to update)
  ✓ Temporal.io or similar tools available (reduces orchestrator burden)

THE HYBRID APPROACH (most real systems use this):
  Core business saga: orchestrated (visible, debuggable)
  Side effects: choreographed (email notifications, analytics)
  
  OrderSaga orchestrates: Inventory + Payment (critical path)
  "order.confirmed" event triggers (choreographed): Email, Analytics, Fraud review
  → Critical path has visibility; side effects are decoupled
```

## 4. Temporal: The Modern Orchestration Platform

**Temporal** (see topic 12.27) is the state-of-the-art solution for saga orchestration. It makes durable workflows simple:

```python
from temporalio import workflow, activity
from datetime import timedelta

@workflow.defn
class PlaceOrderWorkflow:
    @workflow.run
    async def run(self, order_data: dict) -> str:
        # Temporal automatically handles: retries, crashes, timeouts
        # State is durably checkpointed after each step
        
        order_id = await workflow.execute_activity(
            create_order, order_data,
            start_to_close_timeout=timedelta(seconds=10),
        )
        
        try:
            await workflow.execute_activity(
                reserve_inventory, {"order_id": order_id},
                start_to_close_timeout=timedelta(minutes=1),
                retry_policy=RetryPolicy(max_attempts=3),
            )
        except ActivityError:
            await workflow.execute_activity(cancel_order, order_id)
            raise
        
        try:
            await workflow.execute_activity(
                process_payment, {"order_id": order_id},
                start_to_close_timeout=timedelta(minutes=2),
            )
        except ActivityError:
            # Compensate: release inventory
            await workflow.execute_activity(release_inventory, order_id)
            await workflow.execute_activity(cancel_order, order_id)
            raise
        
        await workflow.execute_activity(confirm_order, order_id)
        return order_id
# If the server crashes mid-workflow: Temporal resumes exactly where it left off.
# No manual saga state tracking needed — Temporal handles it.
```

## 5. Senior Engineer Perspective

**The choreography visibility problem:**

The biggest practical issue with choreography is the "where is this order?" problem. When a customer calls support about their stuck order, you need to know: did the inventory get reserved? Did payment process? Choreography distributes this knowledge across services. You need to query each service individually, or maintain a separate "saga state" tracker anyway — at which point you've essentially built an orchestrator.

**The orchestration God Object problem:**

The risk with orchestration is that the orchestrator accumulates business logic. "The orchestrator should only coordinate. Business rules belong in the services." When the orchestrator starts making business decisions (should we retry? which payment method is preferred? what discount applies?) it becomes a tightly coupled monolith that defeats the purpose of microservices.

The rule: orchestrators coordinate sequencing and error handling. They don't know about business rules within each step. Business rules belong in the services.
