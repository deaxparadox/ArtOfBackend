## 1. What Is It? (Theory)

Event sourcing is a persistence pattern where **you store every change to state as an immutable event**, rather than storing the current state. The current state of any entity is derived by replaying all its events in sequence.

This inverts the traditional database model: instead of "UPDATE accounts SET balance = 500 WHERE id = 1", you append "MoneyWithdrawn: $100 from account 1 at time T". The account balance is computed by summing all deposits and withdrawals.

**The bank account analogy (the canonical one):**

Your bank doesn't store "your current balance: $1,247.50." Your bank stores every transaction: deposits, withdrawals, fees, interest payments. Your current balance is computed by summing all transactions. This is event sourcing.

Why does the bank do this? Because the transaction history IS the valuable data. The balance is just a derived view.

## 2. How It Actually Works (Deep Mechanics)

### The Event Store

```
TRADITIONAL (current state):
  Table: bank_accounts
  | account_id | balance | last_updated        |
  | acc-001    | 1247.50 | 2024-06-15 14:30:00 |
  
  When user withdraws $100:
  UPDATE bank_accounts SET balance = 1147.50 WHERE account_id = 'acc-001'
  
  History: gone. You only know the current balance.
  "Why is the balance $1,247.50?" → can't answer from the DB.

EVENT SOURCING:
  Table: account_events
  | event_id | account_id | event_type        | amount  | timestamp           |
  | evt-001  | acc-001    | AccountOpened     | 0       | 2024-01-01 09:00:00 |
  | evt-002  | acc-001    | MoneyDeposited    | 1500    | 2024-01-15 10:30:00 |
  | evt-003  | acc-001    | MoneyWithdrawn    | 200     | 2024-01-20 14:00:00 |
  | evt-004  | acc-001    | MoneyDeposited    | 300     | 2024-02-01 11:00:00 |
  | evt-005  | acc-001    | MoneyWithdrawn    | 352.50  | 2024-06-15 14:30:00 |
  
  Current balance = 0 + 1500 - 200 + 300 - 352.50 = $1,247.50
  
  "Why is the balance $1,247.50?" → Read the events. Complete answer.
  "What was the balance on February 15?" → Replay events up to that date.
  "How many withdrawals this month?" → Query events by type and date.
  All questions answerable from the immutable event log.
```

### The Aggregate Root Pattern

```python
from dataclasses import dataclass, field
from typing import Optional
from datetime import datetime
import uuid

@dataclass
class DomainEvent:
    event_id: str = field(default_factory=lambda: str(uuid.uuid4()))
    event_type: str = ""
    aggregate_id: str = ""
    payload: dict = field(default_factory=dict)
    version: int = 0
    occurred_at: datetime = field(default_factory=datetime.utcnow)

class BankAccount:
    """
    Aggregate root: entity whose state is built from events.
    """
    
    def __init__(self, account_id: str):
        self.account_id = account_id
        self.balance_cents = 0
        self.status = "closed"
        self.version = 0
        self._pending_events: list[DomainEvent] = []
    
    # ── COMMANDS: validate then emit events ─────────────────
    
    def open(self, owner_id: str, initial_deposit: int = 0):
        if self.status != "closed":
            raise ValueError("Account already open")
        self._emit(DomainEvent(
            event_type="AccountOpened",
            aggregate_id=self.account_id,
            payload={"owner_id": owner_id, "initial_balance": initial_deposit},
        ))
    
    def deposit(self, amount_cents: int):
        if self.status != "open":
            raise ValueError("Account is not open")
        if amount_cents <= 0:
            raise ValueError("Deposit amount must be positive")
        self._emit(DomainEvent(
            event_type="MoneyDeposited",
            aggregate_id=self.account_id,
            payload={"amount_cents": amount_cents},
        ))
    
    def withdraw(self, amount_cents: int):
        if self.status != "open":
            raise ValueError("Account is not open")
        if amount_cents > self.balance_cents:
            raise ValueError(f"Insufficient funds: balance={self.balance_cents}, requested={amount_cents}")
        self._emit(DomainEvent(
            event_type="MoneyWithdrawn",
            aggregate_id=self.account_id,
            payload={"amount_cents": amount_cents},
        ))
    
    # ── STATE CHANGES: applied from events ──────────────────
    # These are pure functions — no business logic, no validation
    # Only called via _emit() or _reconstitute()
    
    def _apply_AccountOpened(self, event: DomainEvent):
        self.status = "open"
        self.balance_cents = event.payload["initial_balance"]
    
    def _apply_MoneyDeposited(self, event: DomainEvent):
        self.balance_cents += event.payload["amount_cents"]
    
    def _apply_MoneyWithdrawn(self, event: DomainEvent):
        self.balance_cents -= event.payload["amount_cents"]
    
    # ── INTERNAL MECHANICS ────────────────────────────────────
    
    def _emit(self, event: DomainEvent):
        """Record and apply a new event."""
        event.version = self.version + 1
        self._apply(event)
        self._pending_events.append(event)
    
    def _apply(self, event: DomainEvent):
        """Apply an event to update state. Used both for new events and replay."""
        handler = getattr(self, f"_apply_{event.event_type}", None)
        if handler:
            handler(event)
        self.version = event.version
    
    @classmethod
    def reconstitute(cls, account_id: str, events: list[DomainEvent]) -> "BankAccount":
        """Rebuild the aggregate by replaying all its events."""
        account = cls(account_id)
        for event in events:
            account._apply(event)
        return account
```

### The Repository: Saving and Loading Events

```python
class BankAccountRepository:
    
    async def save(self, account: BankAccount):
        """Append new events to the event store."""
        for event in account._pending_events:
            await db.execute("""
                INSERT INTO account_events
                    (event_id, account_id, event_type, payload, version, occurred_at)
                VALUES ($1, $2, $3, $4, $5, $6)
                ON CONFLICT (account_id, version) DO NOTHING
            """,
                event.event_id,
                event.aggregate_id,
                event.event_type,
                json.dumps(event.payload),
                event.version,
                event.occurred_at,
            )
        account._pending_events.clear()
    
    async def load(self, account_id: str) -> BankAccount:
        """Rebuild account state by loading and replaying all events."""
        rows = await db.fetch(
            "SELECT * FROM account_events WHERE account_id=$1 ORDER BY version ASC",
            account_id
        )
        if not rows:
            raise AccountNotFound(account_id)
        
        events = [
            DomainEvent(
                event_id=row["event_id"],
                event_type=row["event_type"],
                aggregate_id=row["account_id"],
                payload=json.loads(row["payload"]),
                version=row["version"],
                occurred_at=row["occurred_at"],
            )
            for row in rows
        ]
        
        return BankAccount.reconstitute(account_id, events)
```

## 3. Snapshots: Optimization for Long-Lived Aggregates

Replaying 10,000 events on every load is expensive. Snapshots solve this:

```python
async def load_with_snapshot(account_id: str) -> BankAccount:
    """Load account using snapshot if available."""
    
    # Check for recent snapshot
    snapshot = await db.fetchrow(
        "SELECT * FROM account_snapshots WHERE account_id=$1 ORDER BY version DESC LIMIT 1",
        account_id
    )
    
    if snapshot:
        # Restore from snapshot
        account = BankAccount.from_snapshot(snapshot)
        start_version = snapshot["version"]
    else:
        account = BankAccount(account_id)
        start_version = 0
    
    # Replay only events AFTER the snapshot
    events = await db.fetch(
        "SELECT * FROM account_events WHERE account_id=$1 AND version > $2 ORDER BY version",
        account_id, start_version
    )
    for event in events:
        account._apply(DomainEvent(**event))
    
    return account

async def save_snapshot(account: BankAccount):
    """Save current state as a snapshot every N events."""
    await db.execute("""
        INSERT INTO account_snapshots (account_id, version, state)
        VALUES ($1, $2, $3)
    """,
        account.account_id,
        account.version,
        json.dumps({"balance_cents": account.balance_cents, "status": account.status})
    )
```

## 4. When to Use Event Sourcing

```
USE EVENT SOURCING WHEN:
  ✓ Complete audit trail required by the domain (finance, healthcare, legal)
  ✓ Time-travel debugging: "what was the state at time T?"
  ✓ Event replay for new features: "analyze 2 years of orders with new algorithm"
  ✓ Business requirement for full history (regulatory compliance)
  ✓ Complex domain with rich behavior (not just CRUD)
  ✓ Already using CQRS (the combination is natural)

DON'T USE EVENT SOURCING WHEN:
  ✗ Simple CRUD application (massive over-engineering)
  ✗ Team unfamiliar with the pattern (high learning curve)
  ✗ No requirement for history or audit trail
  ✗ Data privacy requirements conflict with immutable history
    (GDPR "right to be forgotten" is hard with immutable event log)
  ✗ Reporting needs don't require historical accuracy

EVENT SOURCING COSTS:
  Complex: queries require projections (not direct SELECT)
  Schema evolution: past events are immutable, schema changes are hard
  Storage: storing events grows unboundedly (snapshots help but add complexity)
  Debugging: "what's the current state?" requires replaying events (use projections)
  Learning curve: different mental model from CRUD
  
  RULE: event sourcing should be driven by a business requirement, not by technology fashion.
```

## 5. Senior Engineer Perspective

**The GDPR "right to be forgotten" problem:**

Event sourcing stores immutable history. GDPR requires the ability to delete user personal data. These are in direct conflict. Solutions:
1. Crypto-shredding: encrypt PII in events with per-user keys. "Delete" = delete the key. Events remain but PII is unreadable.
2. Redaction: replace PII in historical events with placeholder values (requires event store that supports mutation — complex).
3. Separate PII: don't put PII in events. Store PII separately (users table). Events reference user IDs. GDPR deletion removes the user record; events retain user ID but no PII.

**The schema evolution lesson:**

Events are immutable. If you wrote `{"amount": 100}` 6 months ago and now your schema says `{"amount_cents": 10000}`, you have a problem. Old events have the old schema. New code expects the new schema. Solutions:
- Upcasting: transform old events to new format when reading (adds complexity)
- Versioning: keep both schema versions in the consumer code
- New event type: deprecate old event type, create new one

The lesson: **event schemas are as permanent as database schemas — think carefully before publishing.** Once an event is in the store, you're committed to handling it forever.
