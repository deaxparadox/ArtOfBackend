## 1. What Is It? (Theory)

Sidekiq is the dominant background job library for Ruby and Rails. Its innovation over older Ruby job libraries (like Delayed::Job and Resque): it uses threads instead of processes, allowing a single Sidekiq process to handle hundreds of concurrent jobs without the memory overhead of hundreds of processes.

## 2. Core Usage

```ruby
# Define a worker
class WelcomeEmailWorker
  include Sidekiq::Worker
  
  sidekiq_options(
    queue: :emails,           # queue name
    retry: 5,                 # retry 5 times before giving up
    backtrace: true,          # store backtrace in job data for debugging
    dead: true,               # move to dead set after retries exhausted
  )
  
  def perform(user_id)
    user = User.find(user_id)
    UserMailer.welcome_email(user).deliver_now
  rescue ActiveRecord::RecordNotFound
    logger.error "User #{user_id} not found"
    # Don't raise — record not found is permanent, no point retrying
  end
end

# Enqueue from anywhere in your code
WelcomeEmailWorker.perform_async(user.id)               # enqueue immediately
WelcomeEmailWorker.perform_in(1.hour, user.id)          # enqueue after 1 hour
WelcomeEmailWorker.perform_at(Time.now + 24.hours, user.id)  # specific time

# Bulk enqueue
Sidekiq::Client.push_bulk(
    'class' => WelcomeEmailWorker,
    'args' => user_ids.map { |id| [id] }
)
```

## 3. Configuration

```yaml
# config/sidekiq.yml
:concurrency: 25              # 25 threads per Sidekiq process
:queues:
  - [critical, 3]             # checked 3x more often
  - [default, 2]
  - [emails, 1]
  - [low_priority, 1]

:max_retries: 5

# Retry schedule: attempts at:
# 1st retry: 15 seconds
# 2nd retry: 15 + 2^1 * 60 = 135 seconds
# 3rd retry: 15 + 2^2 * 60 = 255 seconds
# etc. — exponential backoff automatically
```

```ruby
# config/initializers/sidekiq.rb
Sidekiq.configure_server do |config|
    config.redis = { url: ENV['REDIS_URL'], network_timeout: 5 }
    
    config.on(:startup) do
        Rails.logger.info "Sidekiq starting"
    end
    
    config.on(:shutdown) do
        Rails.logger.info "Sidekiq shutting down"
    end
end

Sidekiq.configure_client do |config|
    config.redis = { url: ENV['REDIS_URL'] }
end
```

## 4. Scheduled Jobs with Sidekiq-Cron

```ruby
# Gemfile
gem 'sidekiq-cron'

# config/schedule.yml
daily_digest:
    cron: "0 8 * * *"     # 8am daily
    class: DailyDigestWorker
    queue: default

cleanup_sessions:
    cron: "0 * * * *"     # every hour
    class: CleanupSessionsWorker
    queue: low_priority

# Load the schedule
Sidekiq::Cron::Job.load_from_hash YAML.load_file("config/schedule.yml")
```

## 5. Web UI

Sidekiq includes a built-in web dashboard:

```ruby
# config/routes.rb
require 'sidekiq/web'

# Mount with authentication
authenticate :admin do
    mount Sidekiq::Web => '/sidekiq'
end

# Or with HTTP basic auth in development
Sidekiq::Web.use(Rack::Auth::Basic) do |username, password|
    ActiveSupport::SecurityUtils.secure_compare(username, ENV['SIDEKIQ_USER']) &&
    ActiveSupport::SecurityUtils.secure_compare(password, ENV['SIDEKIQ_PASSWORD'])
end
```

The dashboard shows: busy workers, queue stats, failed jobs with full stacktraces, scheduled jobs, and allows retrying or deleting failed jobs manually.

## 6. Senior Engineer Perspective

**Sidekiq's threading model vs. Celery's process model:**

Sidekiq uses threads (25 by default), not processes. This means 25 concurrent jobs in one OS process. Ruby's GIL limits CPU parallelism, but for I/O-bound jobs (database queries, HTTP calls, email delivery), threads are highly efficient — they yield during I/O, allowing other threads to run.

Celery's prefork model spawns separate processes — heavier memory usage but true parallelism for CPU-bound work.

**The `dead` set — don't ignore it:**

Failed jobs that exhaust retries go to Sidekiq's "Dead" set. They're kept for 6 months. Sidekiq Web shows these prominently. A growing dead set means a systematic failure pattern that needs investigation. Never ignore it.
