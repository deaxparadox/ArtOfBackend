## 1. What Is It? (Theory)

Inngest is a serverless platform for durable workflows and background jobs. Its key innovation: your functions run as HTTP handlers in your existing server — no separate worker processes, no Redis to manage, no Celery workers to deploy.

**How Inngest differs from Celery/BullMQ:**

```
CELERY/BULLMQ:
  → Your code + message broker (Redis) + worker processes
  → Workers are separate deployments
  → Need to manage, scale, and monitor workers separately
  → Works on any infrastructure

INNGEST:
  → Your code + Inngest cloud (event bus + queue)
  → Your API server IS the worker (functions run as HTTP handlers)
  → No separate worker infrastructure
  → Managed event bus handles routing, retries, scheduling
  → Works best with serverless (Vercel, AWS Lambda, etc.)
```

## 2. How It Actually Works

```
INNGEST EXECUTION MODEL:

  User action → POST /api/inngest/events
                      ↓
              Inngest Cloud (event bus)
                      ↓
              POST to your function endpoint
              with retry/backoff managed by Inngest
                      ↓
              Your function runs (on your server)
              Each step.* call is checkpointed
```

### Code Example

```typescript
import { Inngest } from 'inngest';
import { serve } from 'inngest/next';  // or express, fastify, etc.

const inngest = new Inngest({ id: 'my-e-commerce-app' });

// FUNCTION DEFINITION: like a Temporal workflow, but simpler
const processNewOrder = inngest.createFunction(
    {
        id: 'process-new-order',
        retries: 3,                   // retry entire function up to 3 times
    },
    { event: 'order/placed' },        // trigger: when this event is received
    async ({ event, step }) => {
        const { orderId, userId, amount } = event.data;
        
        // step.run: durable checkpoint — runs once, result cached on retry
        const user = await step.run('fetch-user', async () => {
            return db.getUser(userId);
        });
        
        // step.run: charge the card (only runs once even on retry)
        const payment = await step.run('charge-card', async () => {
            return paymentGateway.charge(orderId, amount);
        });
        
        if (!payment.success) {
            throw new Error(`Payment failed: ${payment.error}`);
        }
        
        // step.sleep: wait 3 days (Inngest handles this efficiently)
        await step.sleep('wait-for-delivery-window', '3d');
        
        // This runs 3 days later, all previous steps' results are preserved
        await step.run('send-delivery-reminder', async () => {
            return emailService.sendDeliveryUpdate(user.email, orderId);
        });
        
        // step.waitForEvent: pause until another event arrives
        const deliveryConfirmed = await step.waitForEvent(
            'wait-for-delivery',
            {
                event: 'delivery/confirmed',
                match: 'data.orderId',       // match on event.data.orderId
                timeout: '7d',               // wait up to 7 days
            }
        );
        
        if (deliveryConfirmed) {
            await step.run('send-review-request', async () => {
                return emailService.sendReviewRequest(user.email, orderId);
            });
        } else {
            await step.run('escalate-to-support', async () => {
                return supportService.createTicket(orderId, 'Delivery not confirmed');
            });
        }
    }
);

// TRIGGER: send an event from your API handler
// This is all you do in your POST /orders handler:
await inngest.send({
    name: 'order/placed',
    data: { orderId: order.id, userId: user.id, amount: order.total },
});

// SERVE: register your functions with Inngest
// This HTTP endpoint is called by Inngest to run your functions
export default serve(inngest, [processNewOrder]);
// Mount at POST /api/inngest
```

## 3. Inngest vs. Temporal vs. Celery

```
INNGEST:
  ✓ No infrastructure: no workers, no Redis, no Temporal server
  ✓ Functions run in your existing server
  ✓ Great for Next.js, Vercel, serverless
  ✓ step.waitForEvent: powerful human-in-the-loop patterns
  ✓ step.sleep: efficient waiting (no polling)
  ✗ Requires Inngest cloud (managed service, not self-hosted)
  ✗ Not ideal for very high throughput (thousands of concurrent workflows)
  ✗ Vendor lock-in to Inngest platform

TEMPORAL:
  ✓ Most powerful durability guarantees
  ✓ Can be self-hosted
  ✓ Any scale
  ✗ Requires separate Temporal server and worker fleet
  ✗ Steeper learning curve

CELERY/BULLMQ:
  ✓ Simple, well-understood
  ✓ Self-hosted
  ✓ High throughput
  ✗ No native multi-step workflow support
  ✗ Manual saga implementation

CHOOSE INNGEST WHEN:
  ✓ You're on Vercel, Next.js, or serverless
  ✓ You want durable workflows without managing infrastructure
  ✓ You need step.waitForEvent for human-in-the-loop flows
  ✓ Small-to-medium scale

CHOOSE TEMPORAL WHEN:
  ✓ Very complex workflows
  ✓ Self-hosting required
  ✓ Any language (Go, Java, Python, TypeScript)
  ✓ High scale

CHOOSE CELERY/BULLMQ WHEN:
  ✓ Simple background jobs
  ✓ Your team knows the tool
  ✓ High throughput task processing
```

## 4. Senior Engineer Perspective

**The step.waitForEvent pattern is genuinely novel:**

Being able to pause a workflow mid-execution waiting for a specific external event — and resume when it arrives — is something that's very hard to implement correctly in traditional job queues. You'd need to store state externally, set up a separate event listener, and manually resume the workflow. Inngest makes this trivial:

```typescript
// Pause the function for up to 7 days until the user verifies their email
const verified = await step.waitForEvent('email-verification', {
    event: 'user/email-verified',
    match: 'data.userId',   // event.data.userId must match current event.data.userId
    timeout: '7d',
});

if (!verified) {
    // User didn't verify in 7 days — take action
    await step.run('send-reminder', async () => { ... });
}
```

This pattern naturally implements email verification flows, payment confirmation flows, approval workflows — any case where you need to pause and wait for human action.
