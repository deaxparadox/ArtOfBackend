## 1. What Is It? (Theory)

Kafka consumers are the reading side of the streaming system. How you configure them — the consumer group strategy, offset commit behavior, and partition assignment — determines whether you get exactly-once semantics, how quickly you recover from failures, and whether one slow consumer disrupts others.

The consumer group protocol is Kafka's mechanism for distributing work across multiple processes. Understanding it prevents the most common Kafka production issues: rebalances that pause processing, lag that grows silently, and commits that lose messages.

## 2. How It Actually Works (Deep Mechanics)

### Consumer Group Rebalancing

A rebalance happens when the consumer group membership changes. During rebalancing, ALL consumers in the group pause processing — this is the most disruptive event in Kafka consumer operations.

```
TRIGGERS FOR REBALANCING:
  1. New consumer joins the group (scale up)
  2. Consumer leaves (scale down, deployment, crash)
  3. Consumer heartbeat timeout (consumer is "stuck" — slow processing)
  4. Topic partition count changes
  5. Consumer subscribes to a new topic pattern that matches a new topic

REBALANCING IMPACT:
  "Stop the world" for the consumer group.
  No messages processed during rebalancing.
  Duration: typically 30-120 seconds (depends on cluster size and consumer count)
  
  At 100,000 events/second → 30s rebalance = 3,000,000 events queued but unprocessed.
  Consumer lag spikes by 3M during the rebalance.
  
HEARTBEAT TIMEOUT (the hidden rebalance trigger):
  Each consumer sends heartbeats to the Kafka group coordinator.
  If no heartbeat within session.timeout.ms → consumer declared dead → rebalance.
  
  CRITICAL: if your message processing takes longer than max.poll.interval.ms
  Kafka declares the consumer dead (stopped polling) → rebalance.
  
  Example of the trap:
  max.poll.interval.ms = 300,000 (5 minutes, default)
  Your consumer processes a message in 6 minutes (slow external API call)
  → Kafka thinks consumer is dead → rebalance
  → Another consumer gets the same message
  → DUPLICATE PROCESSING even without a crash!
  
  FIX: set max.poll.interval.ms > your maximum message processing time
       OR make processing faster
       OR process in a separate thread, poll() in the main thread
```

### Partition Assignment Strategies

```
EAGER REBALANCING (default "range" or "roundrobin"):
  On any consumer change:
  1. ALL consumers revoke all their partitions
  2. Group coordinator assigns partitions fresh
  3. ALL consumers receive new partition assignments
  
  Cost: every rebalance = full processing pause + partition reassignment overhead.
  
COOPERATIVE STICKY REBALANCING (recommended for production):
  On consumer change:
  1. Only the AFFECTED partitions are reassigned
  2. Consumers keep their unaffected partitions
  3. Only the consumer gaining/losing partitions briefly pauses
  
  Cost: minimal disruption, other consumers continue processing.
  
  Enable with:
  partition.assignment.strategy = cooperative-sticky
  
  This reduces rebalance impact from "entire group pauses" 
  to "only affected consumers briefly pause".
```

### Offset Management: The Guarantee

```python
OFFSET COMMIT STRATEGIES:

# STRATEGY 1: Auto-commit (easier, less reliable)
consumer = Consumer({
    "enable.auto.commit": True,
    "auto.commit.interval.ms": 5000,  # commit every 5 seconds
})

# Risk:
# Consumer processes messages at offsets 100-200.
# At t=3s: processes message 150.
# At t=5s: auto-commit fires → commits offset 150.
# At t=4s: consumer crashes BEFORE auto-commit.
# On restart: resumes from offset 100 (last committed).
# Messages 100-149: REPROCESSED → duplicates.
# Must be idempotent!

# Worse scenario:
# Consumer processes messages 100-200.
# At t=4.9s: commit about to fire.
# At t=4.5s: consumer crashes AFTER processing 100-200 but BEFORE commit.
# On restart: resumes from offset 100.
# Messages 100-200: REPROCESSED again.

# STRATEGY 2: Manual commit after processing (recommended)
consumer = Consumer({
    "enable.auto.commit": False,
})

def process_batch(messages: list) -> list[failed_offsets]:
    """Process a batch and return any failed messages."""
    ...

while True:
    message = consumer.poll(1.0)
    if message is None: continue
    
    try:
        process(message)
        # Commit AFTER successful processing
        # Message redelivered if we crash before this
        consumer.commit(message=message)
    except Exception:
        # Don't commit → message redelivered on restart
        pass

# STRATEGY 3: Async commit (higher throughput, slightly risky)
consumer.commit(asynchronous=True)
# Doesn't block → higher throughput
# Risk: commit callback may indicate failure AFTER you've moved on
# Handle commit failures explicitly in the callback

# STRATEGY 4: Commit at batch boundaries (best throughput for bulk processing)
messages = []
for _ in range(100):  # collect 100 messages
    msg = consumer.poll(0.1)
    if msg: messages.append(msg)

if messages:
    process_batch(messages)
    # Commit the LAST offset in the batch
    consumer.commit(message=messages[-1])
    # If crash after processing but before commit:
    # Entire batch reprocessed on restart (must be idempotent)
```

## 3. Consumer Lag: The Most Important Kafka Health Metric

**Consumer lag** is the difference between the latest offset in a partition and the consumer group's committed offset. It's the number of events the consumer is behind.

```
CONSUMER LAG MEANING:

  Topic "orders", Partition 0:
  Latest offset: 10,500 (most recent event produced)
  Consumer group "payment-service" committed: 10,492
  
  Lag = 10,500 - 10,492 = 8
  "Payment service is 8 events behind the latest"
  
  This is healthy if 8 events = < 1 second of data.
  This is concerning if 8 events = 30 minutes of data (events arrive slowly).

WHAT GROWING LAG MEANS:
  Lag that grows over time = consumer processing slower than events arrive.
  
  Events arriving at: 1,000/second
  Consumer processing: 800/second
  Lag grows at: 200/second
  After 1 hour: lag = 720,000 events
  
  FIX:
  - Add more consumer instances (if partitions allow)
  - Optimize per-message processing time
  - Use batch processing (process N messages per DB write, not 1-per-write)
  - Scale the processing infrastructure

WHAT SUDDEN LAG SPIKE MEANS:
  Lag was 0, suddenly jumps to 500,000.
  
  Possible causes:
  - Consumer group rebalancing (processing paused 30-120s)
  - Consumer crashed, not yet restarted
  - Slow message in the batch causing the consumer to stall
  - External dependency (DB, API) slowed down

MONITORING:
  Track: lag per topic, per partition, per consumer group.
  Alert when: lag > threshold AND lag is growing (not just temporarily high).
  
  "High lag, not growing" = consumer is catching up = fine.
  "High lag, growing" = consumer can't keep up = problem.
```

```bash
# CLI: check consumer lag
kafka-consumer-groups.sh \
  --bootstrap-server kafka:9092 \
  --describe \
  --group payment-service

# Output:
# GROUP          TOPIC        PARTITION  CURRENT-OFFSET  LOG-END-OFFSET  LAG
# payment-service  orders     0          10492           10500           8
# payment-service  orders     1          9821            9821            0
# payment-service  orders     2          11203           11210           7

# Python: check lag programmatically
from confluent_kafka import Consumer, TopicPartition

consumer = Consumer({
    "bootstrap.servers": "kafka:9092",
    "group.id": "lag-checker",  # temporary group, won't affect production
})

def check_consumer_lag(group_id: str, topic: str) -> dict:
    """Check lag for a consumer group on a topic."""
    # Get committed offsets
    partitions = consumer.list_topics(topic).topics[topic].partitions
    
    lag_by_partition = {}
    for partition_id in partitions:
        tp = TopicPartition(topic, partition_id)
        
        # Get consumer group's committed offset
        committed = consumer.committed([tp])[0]
        committed_offset = committed.offset if committed.offset >= 0 else 0
        
        # Get latest offset in partition
        lo, hi = consumer.get_watermark_offsets(tp)
        latest_offset = hi
        
        lag_by_partition[partition_id] = {
            "committed": committed_offset,
            "latest": latest_offset,
            "lag": latest_offset - committed_offset,
        }
    
    return lag_by_partition
```

## 4. The max.poll.interval.ms Trap in Detail

```
THE MOST COMMON KAFKA CONSUMER BUG:

DEFAULT: max.poll.interval.ms = 300000 (5 minutes)

Kafka says: "If you don't call poll() within 5 minutes of the last poll,
            I'll assume you're dead and trigger a rebalance."

SCENARIO:
  Consumer polls → gets message M
  Consumer processes M (calls external API that takes 6 minutes)
  At 5 minutes: Kafka thinks consumer is dead → rebalance
  Partitions reassigned to other consumers
  They receive message M again → duplicate processing

  Original consumer finishes at 6 minutes:
  Tries to commit offset → fails (no longer assigned to this partition)
  Consumer rejoins group → gets partitions back
  → Another rebalance!

  With 10 consumers, each occasionally processing slow messages:
  Constant rebalancing, constant lag spikes, duplicate processing.

FIXES:
  Option 1: Increase max.poll.interval.ms
    max.poll.interval.ms = 1800000  # 30 minutes
    Set it longer than your WORST-CASE processing time.
    
  Option 2: Process asynchronously (poll thread separate from process thread)
    Poll thread: continuously calls consumer.poll()
    Processing thread: does the actual work
    Commit: only after processing thread confirms completion
    
  Option 3: Speed up processing
    Don't do slow external calls inline.
    Use asyncio or batch DB writes to reduce per-message time.

RULE: max.poll.interval.ms > max expected processing time per message × 3
```

## 5. Senior Engineer Perspective

**The "committed but not processed" danger with auto-commit:**

Auto-commit is seductive (set it and forget it) but creates a subtle data loss scenario. The consumer fetches messages into a local buffer. Before processing them, the 5-second auto-commit fires and commits the latest offset. The consumer then crashes. On restart: the offset is committed past the messages in the local buffer — those messages are lost, never processed.

Manual commit after processing (with idempotent consumers) is the correct production approach. Yes, it means some messages are reprocessed on restart. With idempotency, that's harmless. Without idempotency but with auto-commit, you risk silently losing messages.

**Scaling consumer groups is bounded by partition count:**

If you have 6 partitions and add a 7th consumer to the group, the 7th consumer gets no partitions and processes nothing. It's idle. This is why partition count must be chosen with your maximum consumer count in mind.

Scale beyond partitions requires: either increasing partition count (disruptive, breaks key ordering) or using separate consumer groups where each group independently processes all events (different use case — multiple systems consuming, not work distribution).
