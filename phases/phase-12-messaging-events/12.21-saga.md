## 1. What Is It? (Theory)

In a microservices architecture, a single business operation often spans multiple services. Traditional ACID transactions work within one database, but across services, you can't use a distributed transaction (Two-Phase Commit) without creating tight coupling and availability problems.

A **Saga** is the pattern for managing consistency across multiple services. It's a sequence of local transactions, each in its own service, coordinated so that either all succeed or compensating actions undo the work of the ones that did succeed.

**The real-world analogy:**

Think of booking a trip: you book a flight, book a hotel, and rent a car. Each booking is in a separate system. If the car rental fails, you need to cancel the flight and hotel — those are the compensating transactions.

The saga is the management of: "I need all three to succeed. If any fails, undo the ones that worked."

## 2. The Two-Phase Commit Problem (Why Sagas Exist)

```
2PC (Two-Phase Commit) across services:
  Phase 1: "Are you ready to commit?"
    Order Service: "Yes"
    Payment Service: "Yes"
    Inventory Service: "Yes"
  
  Phase 2: "Commit now."
    All services commit.
  
PROBLEMS:
  1. AVAILABILITY: all participating services must be available simultaneously
     → One service down = entire transaction stalls
  
  2. LOCKING: all services must hold locks during both phases
     → Locks held while waiting for all participants → reduces throughput
  
  3. COORDINATOR FAILURE: if the coordinator crashes between phase 1 and 2
     → Participants are stuck in "prepared" state (holding locks) indefinitely
  
  4. LATENCY: must wait for slowest participant in each phase
     → N services × 2 round trips = high latency for the entire operation
  
CONCLUSION: 2PC doesn't scale in microservices. Sagas are the alternative.
```

## 3. How It Actually Works (Deep Mechanics)

### The Saga Sequence

```
PLACE ORDER SAGA (ordering service, inventory, payment):

HAPPY PATH:
  1. OrderService: create order (status: PENDING)
     → publishes: OrderCreated

  2. InventoryService: receives OrderCreated
     → checks and reserves stock
     → publishes: InventoryReserved  (or: InsufficientStock)

  3. PaymentService: receives InventoryReserved
     → charges card
     → publishes: PaymentProcessed  (or: PaymentFailed)

  4. OrderService: receives PaymentProcessed
     → updates order status to CONFIRMED
     → publishes: OrderConfirmed

FAILURE PATH (payment fails after inventory reserved):
  3. PaymentService: receives InventoryReserved
     → card declined
     → publishes: PaymentFailed

  Compensating transactions triggered by PaymentFailed:
  3a. InventoryService: receives PaymentFailed
      → releases stock reservation (compensating transaction)
      → publishes: InventoryReleased
  
  3b. OrderService: receives PaymentFailed + InventoryReleased
      → marks order as FAILED

COMPENSATING TRANSACTION = the undo action for a completed step.
  InventoryReserved ← reversed by → InventoryReleased
  PaymentProcessed  ← reversed by → PaymentRefunded
  
  NOT ALL STEPS HAVE COMPENSATING TRANSACTIONS:
  Some steps are "pivot transactions" — irreversible by nature.
  Sending an email: you can't un-send it. (Acceptable: send "order cancelled" email instead)
```

## 4. Choreography (Events) — The Decentralized Saga

In a choreography-based saga, each service listens for events and decides what to do next. There's no central coordinator.

```python
# ORDER SERVICE: starts the saga
@app.post("/orders")
async def place_order(data: OrderRequest):
    order = await db.create_order(data, status="PENDING")
    await event_bus.publish("order.created", {
        "order_id": order.id,
        "user_id": data.user_id,
        "product_id": data.product_id,
        "quantity": data.quantity,
        "amount_cents": data.amount_cents,
    })
    return {"order_id": order.id, "status": "PENDING"}

# INVENTORY SERVICE: reacts to OrderCreated
@event_handler("order.created")
async def handle_order_created(event: dict):
    order_id = event["order_id"]
    
    reserved = await try_reserve_inventory(event["product_id"], event["quantity"])
    
    if reserved:
        await event_bus.publish("inventory.reserved", {"order_id": order_id})
    else:
        await event_bus.publish("inventory.reservation_failed", {
            "order_id": order_id,
            "reason": "insufficient_stock",
        })

# PAYMENT SERVICE: reacts to InventoryReserved
@event_handler("inventory.reserved")
async def handle_inventory_reserved(event: dict):
    order_id = event["order_id"]
    order = await fetch_order(order_id)
    
    charged = await try_charge_card(order.user_id, order.amount_cents)
    
    if charged:
        await event_bus.publish("payment.completed", {"order_id": order_id})
    else:
        await event_bus.publish("payment.failed", {
            "order_id": order_id,
            "reason": "card_declined",
        })

# INVENTORY SERVICE: compensating transaction for payment failure
@event_handler("payment.failed")
async def handle_payment_failed(event: dict):
    await release_inventory_reservation(event["order_id"])
    await event_bus.publish("inventory.released", {"order_id": event["order_id"]})

# ORDER SERVICE: finalize
@event_handler("payment.completed")
async def handle_payment_completed(event: dict):
    await db.update_order(event["order_id"], status="CONFIRMED")
    await event_bus.publish("order.confirmed", {"order_id": event["order_id"]})

@event_handler("payment.failed")
async def handle_payment_failed_order(event: dict):
    await db.update_order(event["order_id"], status="PAYMENT_FAILED")
```

## 5. Saga State and Idempotency

```
SAGA STATE MANAGEMENT:
  Track what step the saga is currently on.
  Necessary for: debugging, support, compensating correctly.
  
  Table: saga_instances
  | saga_id  | order_id | current_step    | status    | updated_at |
  | saga-001 | ord-456  | AWAITING_PAYMENT | IN_PROGRESS | 2024-06-15 |

IDEMPOTENCY IN SAGAS:
  Each step must be idempotent.
  Events may be delivered more than once (at-least-once messaging).
  Receiving "inventory.reserved" twice should not reserve twice.
  
  Pattern:
  async def handle_inventory_reserved(event: dict):
      order_id = event["order_id"]
      
      # Idempotency check: was this step already completed?
      if await was_already_processed("inventory.reserved", order_id):
          return  # skip — already done
      
      # Process
      ...
      
      # Mark as processed
      await mark_processed("inventory.reserved", order_id)
```

## 6. When to Use Sagas

```
USE SAGAS WHEN:
  ✓ Business operation spans multiple microservices
  ✓ Each service has its own database
  ✓ Eventual consistency is acceptable
  ✓ Each step has a meaningful compensating transaction

SAGA IS NOT APPROPRIATE WHEN:
  ✗ You need ACID isolation (use a single database transaction)
  ✗ Compensation is not possible or meaningful for your domain
  ✗ Immediate consistency is required (use a monolith/single DB)
  ✗ The overhead of multiple messages and compensations isn't justified

THE COST OF SAGAS:
  Complexity: multiple events, compensating transactions, error paths
  Eventual consistency: order may be PENDING for seconds before CONFIRMED
  Debugging: "where in the saga is this order?" requires saga state tracking
  Testing: all failure paths must be tested
  
  Worth it when: you genuinely have distributed transactions across services.
  Not worth it when: you could just use a single database.
```

## 7. Senior Engineer Perspective

**Sagas are not a replacement for transactions — they're a response to distributed systems:**

If you find yourself implementing sagas for operations that could be done in a single database transaction, you've over-engineered your architecture. The correct response is to reconsider whether those operations should be in separate services at all.

Sagas exist because sometimes you genuinely need distributed consistency — payment, inventory, and fulfillment legitimately live in different services with different databases. For those cases, sagas are the right tool.

**The compensating transaction that isn't:**

Some actions in a saga are "pivotal" — they can't be undone, only mitigated. Sending an email can't be un-sent. Charging a credit card can technically be reversed but with a different API call (refund, not un-charge). "Compensating" these actions means: do the logical opposite (send a cancellation email, issue a refund), not literally undo the operation. Design your compensating transactions before building the happy path.
