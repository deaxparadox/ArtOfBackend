## 1. What Is It? (Theory)

Apache Spark (UC Berkeley, 2009; Apache 2013) is the dominant distributed computing framework for large-scale data processing. It can process petabytes of data across a cluster of hundreds of machines, making it possible to do in hours what would take days on a single server.

Spark replaced Hadoop MapReduce for most use cases by being 10-100x faster — primarily because Spark keeps data in memory between processing steps, while MapReduce wrote to disk after every step.

**The scale where Spark makes sense:**

```
SINGLE SERVER LIMITS:
  8-core server with 32GB RAM:
  - Process 1GB CSV: done in seconds (Python Pandas works fine)
  - Process 100GB CSV: hours (barely fits in RAM, swapping to disk)
  - Process 1TB CSV: days (impossible without large server)

SPARK CLUSTER:
  10 × 8-core servers with 32GB each:
  - 1TB CSV split into 10 chunks → 10 servers process in parallel
  - What took days: takes hours
  - What takes hours: takes minutes
  
  100 × 8-core servers:
  - 1TB → 100 chunks → processes in minutes
  
  Spark enables horizontal scaling of data processing.
  You're buying wall-clock time with hardware.
```

## 2. How It Actually Works (Deep Mechanics)

### The RDD and DataFrame Model

```
RDD (Resilient Distributed Dataset):
  The original Spark abstraction (Spark 1.x).
  A distributed collection of data partitioned across the cluster.
  Operations are lazy — computed only when needed.
  
DataFrame (Spark 2+):
  A distributed table — like a SQL table, but distributed.
  Much more optimized than RDD (Catalyst query optimizer, Tungsten execution engine).
  Used in PySpark, Spark SQL, and most modern Spark code.
  
DATASET (Scala/Java):
  Type-safe version of DataFrame.
  Compile-time type checking.

USE DataFrame/Dataset IN PRODUCTION. RDDs are the low-level API.
```

### The Execution Model

```
SPARK JOB EXECUTION:

  Your PySpark code → Spark Driver (your process)
  
  Driver:
    1. Creates a logical execution plan (DAG of transformations)
    2. Optimizes the plan (Catalyst optimizer)
    3. Splits into stages (where shuffles occur)
    4. Distributes tasks to executor processes
  
  Executors (worker nodes):
    5. Read data partitions from source (S3, HDFS, disk)
    6. Execute task on partition
    7. (If shuffle needed) Write intermediate results to disk
    8. Read shuffled data, execute next stage
    9. Write final output

TRANSFORMATIONS (lazy — no computation yet):
  filter(), select(), join(), groupBy(), withColumn(), map()
  
ACTIONS (trigger computation):
  show(), count(), collect(), save(), write.parquet()
  
EXAMPLE:
  df = spark.read.parquet("s3://data/orders/")  ← reads nothing yet
  
  filtered = df.filter(df.status == "paid")      ← records the filter
  grouped = filtered.groupBy("country").agg(     ← records the aggregation
      sum("amount_cents").alias("total")
  )
  
  grouped.write.parquet("s3://output/revenue/")  ← TRIGGERS EXECUTION
  # Now Spark reads data, applies filter, aggregates, writes output
```

### PySpark in Practice

```python
from pyspark.sql import SparkSession
from pyspark.sql import functions as F
from pyspark.sql.window import Window

# Create Spark session (local for development, cluster config for production)
spark = SparkSession.builder \
    .appName("OrderAnalysis") \
    .config("spark.sql.adaptive.enabled", "true") \  # auto-optimize
    .getOrCreate()

# Read data from S3 (Parquet is the standard format)
orders = spark.read.parquet("s3://data-lake/orders/year=2024/")
users = spark.read.parquet("s3://data-lake/users/")

# TRANSFORMATION 1: Filter paid orders
paid_orders = orders.filter(F.col("status") == "paid")

# TRANSFORMATION 2: Join with users
orders_with_users = paid_orders.join(
    users.select("user_id", "country", "plan"),
    on="user_id",
    how="left"
)

# TRANSFORMATION 3: Aggregate
revenue_by_country = orders_with_users \
    .groupBy("country", "plan") \
    .agg(
        F.sum("amount_cents").alias("total_revenue_cents"),
        F.count("*").alias("order_count"),
        F.countDistinct("user_id").alias("unique_customers"),
        F.avg("amount_cents").alias("avg_order_cents"),
    ) \
    .orderBy(F.col("total_revenue_cents").desc())

# TRANSFORMATION 4: Add rank within each plan tier
window = Window.partitionBy("plan").orderBy(F.col("total_revenue_cents").desc())
ranked = revenue_by_country.withColumn("rank_in_plan", F.rank().over(window))

# ACTION: Write output (triggers all computations above)
ranked.write \
    .mode("overwrite") \
    .partitionBy("plan") \            # partition output files by plan
    .parquet("s3://output/revenue/")

# Stop the session
spark.stop()
```

## 3. Spark Architecture Components

```
SPARK CLUSTER:

  ┌─────────────────────────────────────────────┐
  │  Driver Program                             │
  │  SparkContext / SparkSession                │
  │  DAG Scheduler, Task Scheduler              │
  │  (your PySpark code runs here)              │
  └─────────────────┬───────────────────────────┘
                    │ coordinates
  ┌─────────────────▼───────────────────────────┐
  │  Cluster Manager                            │
  │  (YARN, Kubernetes, Standalone, Mesos)       │
  └─────────────────┬───────────────────────────┘
                    │ allocates
  ┌────────────────────────────────────────────────┐
  │  Executor (per worker node)                    │
  │  - Runs tasks                                  │
  │  - Stores data in cache                        │
  │  - Reports results back to Driver              │
  └────────────────────────────────────────────────┘

DATA PARTITIONING:
  A 1TB dataset split into 1000 partitions of 1GB each.
  Each partition processed by one task on one executor.
  1000 tasks distributed across N executors.
  Spark tracks lineage: if a partition is lost, recompute it.
  
SHUFFLE (the expensive operation):
  groupBy, join, repartition → requires data to move between nodes.
  All 100GB of "country=US" rows must end up on the same nodes.
  Shuffle: write to disk on sender, read from disk on receiver.
  
  MINIMIZE SHUFFLES for performance:
  - Filter before joining (reduce data volume first)
  - Broadcast small tables (broadcast join avoids shuffle)
  - Pre-partition by join key if joining multiple times on same key
```

## 4. Spark Deployment Options

```
LOCAL (development):
  spark = SparkSession.builder.master("local[*]").getOrCreate()
  Single machine, all cores. No cluster needed. For development and testing.

YARN (Hadoop clusters):
  --master yarn --deploy-mode cluster
  Traditional on-premise Hadoop clusters.

KUBERNETES:
  --master k8s://https://kubernetes:6443
  Modern cloud-native Spark deployment.
  Each Spark job spawns Kubernetes pods for executors.
  Pods auto-clean up after job completion.
  Best for: cloud-native, elastic scaling.

MANAGED SERVICES (recommended for most teams):
  AWS EMR: managed Spark on AWS. Configure cluster size, submit jobs.
  Google Dataproc: managed Spark on GCP.
  Azure HDInsight/Synapse: managed Spark on Azure.
  Databricks: managed Spark with enhanced features (Delta Lake, MLflow, Unity Catalog).
             Databricks is the commercial Spark company and significantly improves UX.

DATABRICKS VS OPEN SOURCE SPARK:
  Open source: full control, operational burden
  Databricks: Unity Catalog, auto-optimized clusters, collaborative notebooks
              Significantly easier to use, costs 20-50% premium over raw cloud
              Best for: teams that want Spark without operational burden
```

## 5. When Spark vs. SQL in the Warehouse

```
USE SPARK WHEN:
  ✓ Data too large for the warehouse to handle cost-effectively
  ✓ Complex transformations not expressible in SQL (ML feature engineering, graph processing)
  ✓ Processing data from multiple sources (join S3 files with database tables)
  ✓ Writing ML pipelines (Spark MLlib, feature engineering)
  ✓ Custom business logic in Python/Scala that doesn't fit in SQL UDFs
  ✓ Streaming (Spark Structured Streaming)

USE SQL IN WAREHOUSE (BigQuery/Snowflake/Redshift + dbt) WHEN:
  ✓ Data fits within warehouse capacity (up to ~100TB for most warehouses)
  ✓ Transformations are expressible in SQL
  ✓ You want data analysts (not just engineers) to own transformations
  ✓ Simpler operations (no cluster to manage)
  ✓ Modern warehouses handle massive SQL queries efficiently (no Spark needed)

THE TREND:
  Modern cloud warehouses (BigQuery, Snowflake) are significantly closing the gap.
  Many teams that used Spark 5 years ago now use BigQuery + dbt.
  Spark's advantage: complex Python logic, ML pipelines, very large scale (PB+).
  Spark's disadvantage: operational complexity, no SQL-only workflow.
```

## 6. Senior Engineer Perspective

**The partition size matters enormously:**

The most common Spark performance problem: wrong partition count. Too few partitions → some executors idle, parallelism wasted. Too many partitions → overhead of managing thousands of tiny tasks. Rule of thumb: 2-4 partitions per CPU core, each partition 100MB-1GB.

```python
# Check current partition count
print(df.rdd.getNumPartitions())

# Repartition for better parallelism (more partitions, causes shuffle)
df = df.repartition(200)

# Coalesce to fewer partitions (reduces partitions, no shuffle)
df = df.coalesce(10)

# Repartition by a column (for join performance - pre-partition by join key)
df = df.repartition("country")
```

**Broadcast joins — the most impactful single optimization:**

When joining a large table with a small table (< 10MB), Spark can "broadcast" the small table to every executor, avoiding a shuffle entirely:

```python
from pyspark.sql.functions import broadcast

# Force broadcast join for small dimension table
result = large_orders.join(
    broadcast(small_country_codes),  # broadcast the small table
    on="country_code"
)
# No shuffle needed — country_codes copied to every executor
# Can be 10x faster than a regular join
```
