## 1. Apache Flink (16.4): Stateful Stream Processing

### What Is It?

Apache Flink (TU Berlin, 2014; Apache 2014) is a distributed stream processing framework designed for high-throughput, low-latency, stateful stream processing. While Spark Streaming uses micro-batches, Flink processes truly one event at a time.

**What makes Flink distinctive:**

```
FLINK'S CORE STRENGTHS:

1. TRUE STREAMING:
   Events processed immediately as they arrive.
   Latency: 10ms-1s (vs. Spark Streaming's 5-30s micro-batch)
   
2. STATEFUL PROCESSING:
   Rich state management — each event processed in context of history.
   "Has this IP address sent 100+ requests in the last minute?"
   → Flink maintains per-IP counters in state, updated per event.
   
3. EVENT TIME + WATERMARKS:
   First-class support for event time (when event occurred) vs. processing time.
   Handles late-arriving events correctly.
   
4. EXACTLY-ONCE SEMANTICS:
   With Kafka + Flink checkpointing: each event processed exactly once.
   Even if Flink crashes and restarts mid-processing.
   
5. UNIFIED BATCH + STREAMING:
   Same API for both. Same code runs as batch (bounded) or stream (unbounded).
```

### Flink Concepts

```
DATASTREAM API:
  Processing events one at a time.
  
KEYED STREAMS:
  Partition stream by a key. Each key's events processed sequentially.
  Flink routes all events with key="user-123" to the same slot.
  State is per-key — user-123's state doesn't mix with user-456's state.

WINDOWS:
  Aggregate events over a time period.
  
  TUMBLING WINDOWS: non-overlapping, fixed-size
  10:00-10:05 window → 10:05-10:10 window → (each event in exactly one window)
  
  SLIDING WINDOWS: overlapping
  "last 10 minutes" recalculated every 1 minute
  (each event appears in multiple windows)
  
  SESSION WINDOWS: grouped by activity gap
  Events within 30 min of each other → same session
  
STATE BACKENDS:
  MemoryStateBackend: state in JVM heap (fast, limited by heap size)
  FsStateBackend: state in heap, checkpoints to filesystem (HDFS/S3)
  RocksDBStateBackend: state in RocksDB (slow disk-based, huge state possible)

CHECKPOINTING:
  Periodic snapshots of ALL state to persistent storage (S3, HDFS).
  On failure: restore from last checkpoint, replay events from there.
  Exactly-once guarantee: Flink + Kafka commit offset only after checkpoint.
```

### Flink Code Example

```java
// Apache Flink - Java (Flink's primary language)
import org.apache.flink.streaming.api.environment.StreamExecutionEnvironment;
import org.apache.flink.streaming.api.datastream.DataStream;
import org.apache.flink.streaming.connectors.kafka.FlinkKafkaConsumer;
import org.apache.flink.streaming.api.windowing.time.Time;

StreamExecutionEnvironment env = StreamExecutionEnvironment.getExecutionEnvironment();

// Enable checkpointing every 60 seconds (for exactly-once)
env.enableCheckpointing(60_000);
env.getCheckpointConfig().setCheckpointStorage("s3://my-bucket/checkpoints");

// Read from Kafka
DataStream<PaymentEvent> payments = env.addSource(
    new FlinkKafkaConsumer<>("payments", new PaymentEventSchema(), kafkaProps)
);

// Stateful processing: count fraud indicators per user per 10-minute window
DataStream<FraudScore> fraudScores = payments
    .keyBy(event -> event.getUserId())           // partition by user
    .window(TumblingEventTimeWindows.of(Time.minutes(10)))
    .aggregate(new FraudScoringAggregator());    // stateful aggregation

// Write results to another Kafka topic
fraudScores.addSink(new FlinkKafkaProducer<>("fraud-scores", new FraudScoreSchema(), kafkaProps));

env.execute("Fraud Detection Pipeline");
```

### When to Use Flink vs. Spark Streaming

```
USE FLINK WHEN:
  ✓ True sub-second latency required
  ✓ Complex stateful processing (complex event patterns, running totals)
  ✓ Event time processing is critical (out-of-order events are common)
  ✓ Exactly-once semantics required (financial transactions)
  ✓ Complex windowing logic
  
  Use cases: fraud detection, real-time recommendations, IoT processing,
             financial tick data, gaming event processing

USE SPARK STREAMING (micro-batch) WHEN:
  ✓ Your team knows Spark (reuse existing skills and infrastructure)
  ✓ 10-30 second latency is acceptable
  ✓ You want batch + streaming in one unified codebase
  ✓ Event time processing isn't needed (processing time is fine)
  
  Use cases: near-real-time dashboards, ETL with moderate freshness,
             ML feature updates every few minutes

MANAGED FLINK OPTIONS:
  Amazon Managed Service for Apache Flink (formerly Kinesis Analytics)
  Ververica Platform (commercial Flink from the original creators)
  Confluent Cloud (Flink-based processing on Confluent)
```

## 2. Apache Beam (16.5): Unified Batch and Streaming

### What Is It?

Apache Beam (Google, 2016) is a **unified programming model** for defining both batch and streaming data processing pipelines. You write one Beam pipeline, and it can run on multiple execution engines (runners): Apache Flink, Apache Spark, Google Dataflow, or others.

**The problem Beam solves:**

```
WITHOUT BEAM:
  Write Spark job → runs only on Spark
  Write Flink job → runs only on Flink
  
  If you want to switch from Spark to Flink:
  Rewrite all your data processing code.
  Different APIs, different abstractions, different deployment models.

WITH BEAM:
  Write one Beam pipeline → runs on any runner (Spark, Flink, Dataflow)
  
  Switching runners: change one line of configuration.
  Code stays the same.
```

### Beam's Model

```python
# Apache Beam Python SDK example
import apache_beam as beam
from apache_beam.options.pipeline_options import PipelineOptions

# Define the pipeline (same code for batch and streaming)
with beam.Pipeline(options=PipelineOptions()) as pipeline:
    
    # READ: from file (batch) or PubSub (streaming) — same code
    lines = pipeline | "ReadOrders" >> beam.io.ReadFromText("gs://data/orders.csv")
    
    # TRANSFORM: Python functions applied to each element
    amounts = (
        lines
        | "ParseCSV" >> beam.Map(parse_order_csv)
        | "FilterPaid" >> beam.Filter(lambda order: order["status"] == "paid")
        | "ExtractAmount" >> beam.Map(lambda order: (order["country"], order["amount_cents"]))
    )
    
    # AGGREGATE: group and sum
    totals = (
        amounts
        | "GroupByCountry" >> beam.GroupByKey()
        | "SumRevenue" >> beam.Map(lambda kv: (kv[0], sum(kv[1])))
    )
    
    # WRITE: to file or to database
    totals | "WriteResults" >> beam.io.WriteToText("gs://output/revenue")

# Run on Dataflow (change PipelineOptions for different runners)
# options = PipelineOptions(runner="DataflowRunner", project="my-project", ...)
# OR run on Spark: options = PipelineOptions(runner="SparkRunner", ...)
# OR run on Flink: options = PipelineOptions(runner="FlinkRunner", ...)
```

### When Beam Makes Sense

```
USE BEAM WHEN:
  ✓ You use Google Cloud (Dataflow is GCP's managed Beam service)
  ✓ You want portability across runners
  ✓ You need unified batch + streaming in one API
  ✓ Your organization has diverse infrastructure (some Spark, some Flink teams)
  
BEAM'S LIMITATIONS:
  - Python SDK lags behind Java SDK in features
  - Some runner-specific optimizations lost with abstraction
  - Not always the best choice for pure Spark or pure Flink workloads
  - Less common than Spark/Flink in most organizations
  
GOOGLE DATAFLOW (managed Beam on GCP):
  Fully serverless: no cluster to manage
  Auto-scaling: adds workers automatically during peak load
  Excellent for GCP-native workloads
  Pricing: per vCPU-hour consumed
```

## 3. Senior Engineer Perspective

**The data processing ecosystem choice:**

For most backend engineers, the relevant question is: "What does our data engineering team use?" Your job as a backend engineer is usually to:
1. Make your data sources pipeline-friendly (stable schemas, timestamps, soft deletes)
2. Understand enough about Spark/Flink to debug issues when your data pipeline breaks
3. Know when to involve a data engineer vs. building something yourself

**When to build vs. buy streaming:**

Building a real-time streaming pipeline with Flink is complex. Before doing so:
- Can you achieve your goal with a simpler micro-batch approach (Spark Streaming, scheduled queries)?
- Can you use a managed service (Kinesis, Dataflow, Confluent) to reduce operational burden?
- Does the business actually need true real-time, or is "within 5 minutes" actually fine?

True streaming is worth its complexity for: fraud detection, real-time inventory, live leaderboards, financial tick data processing. For most analytics: batch or micro-batch is sufficient.
