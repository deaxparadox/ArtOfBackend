## 1. Schema Evolution (16.13)

### What Is It?

Schema evolution is the challenge of changing the schema of data over time — adding columns, removing columns, changing types — while maintaining compatibility with existing code, pipelines, and stored data that were written with the old schema.

**The real-world problem:**

```
YOU HAVE:
  orders table: {id, user_id, amount_cents, status, created_at}
  
  Stored in S3 as Parquet files (petabytes of historical data).
  Kafka topic receiving new orders.
  Flink job reading from Kafka.
  dbt models transforming the data.
  Downstream ML model using this data.
  
  ALL OF THESE understand the old schema.

YOUR TEAM WANTS TO CHANGE:
  - Add: currency_code column (new requirement)
  - Add: discount_cents column (new feature)
  - Rename: amount_cents → amount (business prefers this)
  - Remove: internal_notes column (contains PII, must be deleted)
  - Change type: status from STRING → ENUM

EVERY CHANGE HAS COMPATIBILITY IMPLICATIONS.
Get them wrong: pipelines break, ML models fail, historical data becomes unusable.
```

### The Three Compatibility Categories

```
BACKWARD COMPATIBLE (new code reads old data):
  New code can read data written with the old schema.
  
  SAFE CHANGES:
  ✓ Add new optional field with a default value
    Old data: no currency_code field → new code reads it, uses default "USD"
  
  UNSAFE CHANGES:
  ✗ Remove a field old data has → new code tries to read it → null
  ✗ Change a field type (INT → FLOAT is usually OK, STRING → INT may fail)
  ✗ Rename a field → old field name gone, new code can't find it

FORWARD COMPATIBLE (old code reads new data):
  Old code can read data written with the new schema.
  
  SAFE CHANGES:
  ✓ Remove an optional field
    New data: no internal_notes → old code reads it → null (old code handles null)
  
  UNSAFE CHANGES:
  ✗ Add a required field → old code doesn't know to include it → missing

FULL COMPATIBLE (both):
  Both old and new code can read both old and new data.
  The safest. Achieved by:
  - Adding optional fields with defaults (backward safe)
  - Not removing fields (forward safe)
  - Not changing field types or semantics
  
  Achieved by: Apache Avro Schema Registry, Protobuf, careful API design.
```

### Schema Evolution Strategies

```
STRATEGY 1: EXPAND-CONTRACT (safest for relational databases)
  
  Phase 1 (Expand): ADD new column alongside old
    Add column: amount_usd (new) alongside amount_cents (old)
    Old code: reads amount_cents (unchanged)
    New code: reads amount_usd (new)
    
  Phase 2 (Migrate): backfill new column with data
    UPDATE orders SET amount_usd = amount_cents / 100.0 WHERE amount_usd IS NULL;
    
  Phase 3 (Contract): remove old column once all code is updated
    ALTER TABLE orders DROP COLUMN amount_cents;
    
  Takes weeks/months. Safe throughout. No big-bang.

STRATEGY 2: SCHEMA REGISTRY + VERSIONING (for streaming systems)
  Maintain a schema registry.
  Each schema version has a number.
  Messages include version number.
  
  Consumer: "I received a version 3 message. My code handles versions 1-3."
  Consumer: "I received a version 5 message. I don't understand it → DLQ."
  
  Tools: Confluent Schema Registry, AWS Glue Schema Registry.

STRATEGY 3: FILE FORMAT VERSIONING (for data lakes)
  Old Parquet files: schema v1 (no currency_code)
  New Parquet files: schema v2 (with currency_code)
  
  Apache Iceberg: handles this automatically.
  Reads old files → currency_code is NULL (handles missing columns).
  Reads new files → currency_code is present.
  
  Query layer merges both seamlessly.
```

## 2. Apache Avro (16.14)

### What Is It?

Apache Avro is a data serialization framework that includes the schema in the data — and provides a **Schema Registry** that enables schema evolution while maintaining compatibility.

```
AVRO KEY PROPERTIES:
  - Binary serialization (compact, fast)
  - Schema embedded in data OR referenced from schema registry
  - Rich schema evolution support
  - Widely used with Kafka (Confluent Schema Registry)

AVRO SCHEMA DEFINITION:
  {
    "type": "record",
    "name": "Order",
    "namespace": "com.example.orders",
    "fields": [
      {"name": "order_id",    "type": "string"},
      {"name": "user_id",     "type": "long"},
      {"name": "amount_cents","type": "int"},
      {"name": "status",      "type": "string"},
      {"name": "created_at",  "type": "long",   "logicalType": "timestamp-millis"},
      
      // Optional field added in schema v2 (backward compatible)
      {
        "name": "currency_code",
        "type": ["null", "string"],  // union type: null OR string
        "default": null              // default: null (safe for backward compatibility)
      }
    ]
  }
```

### Schema Registry Pattern

```python
# Confluent Schema Registry with Python

from confluent_kafka.avro import AvroProducer, AvroConsumer
from confluent_kafka.avro.serializer import MessageSerializer

SCHEMA_REGISTRY_URL = "http://schema-registry:8081"

# PRODUCER: serialize with schema
producer = AvroProducer(
    {
        "bootstrap.servers": "kafka:9092",
        "schema.registry.url": SCHEMA_REGISTRY_URL,
    },
    default_value_schema=value_schema,
)

# Avro serializes with schema ID: [magic_byte][schema_id][avro_data]
# schema_id stored in Schema Registry, not repeated in every message
producer.produce(
    topic="orders",
    value={"order_id": "123", "user_id": 456, "amount_cents": 5000, "status": "paid"},
)

# CONSUMER: deserialize with schema validation
consumer = AvroConsumer(
    {
        "bootstrap.servers": "kafka:9092",
        "group.id": "order-processor",
        "schema.registry.url": SCHEMA_REGISTRY_URL,
    }
)
consumer.subscribe(["orders"])

msg = consumer.poll(1.0)
# Avro automatically looks up schema by ID, validates against writer schema
# Can evolve to new schema: new fields with defaults are handled automatically
order = msg.value()  # dict with order data
```

## 3. Columnar File Formats: Parquet and ORC (16.15)

### What Is It?

Columnar file formats are the standard for storing data in data lakes and passing data between processing steps. They are dramatically more efficient than row-based formats (CSV, JSON) for analytical workloads.

**Why columnar is better for analytics:**

```
CSV (ROW FORMAT):
  Row 1: [order_id=1][user_id=456][amount=5000][status=paid][country=US]
  Row 2: [order_id=2][user_id=789][amount=2500][status=shipped][country=EU]
  
  Query: SELECT country, SUM(amount) FROM orders GROUP BY country
  Must read: EVERY column of EVERY row (even unused columns)
  
  For 1TB CSV: read 1TB to compute one aggregation.

PARQUET (COLUMNAR FORMAT):
  country column: [US][EU][US][US][EU][EU][US]...   (compressed together)
  amount column:  [5000][2500][3000][4500][1500]...  (compressed together)
  order_id column: (not read for this query)
  status column:   (not read for this query)
  
  Query: SELECT country, SUM(amount) FROM orders GROUP BY country
  Must read: ONLY country + amount columns
  
  For 1TB Parquet: read only 100GB (10% of data if 2 of 10 columns needed)
  → 10x less I/O → 10x faster → 10x cheaper query cost

COMPRESSION BONUS:
  Columnar data compresses much better (all same type, predictable patterns)
  country: [US][EU][US][US][EU] → run-length encode: US×3, EU×2 → 2 numbers
  CSV: can't compress as aggressively (mixed types per row)
  
  Typical Parquet compression: 3-10x over CSV
  Combined with column pruning: 10-100x improvement in query cost
```

### Parquet vs. ORC

```
PARQUET:
  Origin: Twitter + Cloudera (2013)
  Used by: Spark (default), Flink, BigQuery, Snowflake, Athena, dbt
  Format: nested schema support (good for JSON-like data), Dremel encoding
  Compression: Snappy, GZIP, LZ4, ZSTD
  Best with: Spark, the Hadoop ecosystem, cloud data lakes
  
ORC (Optimized Row Columnar):
  Origin: Hortonworks (2013, for Hive optimization)
  Used by: Hive, Hive-based systems
  Format: ACID support (used in Hive transactional tables)
  Compression: Zlib, Snappy, LZ4
  Best with: Hive, older Hadoop environments
  
CHOOSING:
  Parquet: for everything new. Universal support across all modern tools.
  ORC: only when Hive compatibility required or existing Hive infrastructure.
  
  When in doubt: Parquet.

PARQUET READING IN PYTHON:
  import pyarrow.parquet as pq
  import pandas as pd
  
  # Read entire file
  df = pd.read_parquet("orders.parquet")
  
  # Read only specific columns (column pruning)
  df = pd.read_parquet("orders.parquet", columns=["country", "amount_cents"])
  
  # Filter at file level (row group pruning)
  df = pd.read_parquet("orders.parquet",
      filters=[("status", "=", "paid"), ("country", "=", "US")])
  # Parquet's min/max statistics allow skipping row groups that can't match the filter
  # Further reduces I/O beyond column pruning
```
