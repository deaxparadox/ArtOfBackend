## 1. The Data Integration Problem

Before the data team can analyze anything, they need data in the warehouse. Data comes from dozens of sources: PostgreSQL, MySQL, Salesforce, HubSpot, Stripe, Shopify, Google Analytics, GitHub, Zendesk...

Each source has a different API, different authentication, different schema, different rate limits, different change detection mechanisms. Building and maintaining extractors for each source is enormous ongoing work.

**The problem solved by Airbyte/Fivetran:**

```
WITHOUT AN INTEGRATION PLATFORM:
  Data engineer writes custom Python extractor for PostgreSQL → 2 weeks
  Data engineer writes custom extractor for Salesforce → 3 weeks
  Data engineer maintains them as APIs change → ongoing, painful
  Add Stripe → 2 more weeks
  Add Google Analytics → 3 more weeks
  
  Entire team could spend months just building extractors.
  No time for actual data analysis.

WITH AIRBYTE OR FIVETRAN:
  Select PostgreSQL from 300+ available connectors.
  Configure: host, port, username, password, tables to sync.
  Done. Data flows to warehouse.
  
  Data engineer: 1 hour setup, zero ongoing maintenance.
  Team: all energy goes to data modeling (dbt) and analysis.
```

## 2. Airbyte (16.7): Open-Source Data Integration

### What Is It?

Airbyte (2020) is an open-source data integration platform with 300+ pre-built connectors. You self-host it (or use Airbyte Cloud), configure sources and destinations, and it handles the rest.

```
AIRBYTE ARCHITECTURE:

  Sources:
    PostgreSQL, MySQL, MongoDB, Salesforce, Stripe, Shopify,
    Google Analytics, HubSpot, GitHub, Zendesk, S3, ...
         ↓ pull data via connectors
  Airbyte Server (orchestrates syncs)
         ↓ push data
  Destinations:
    Snowflake, BigQuery, Redshift, S3, PostgreSQL, ...

SYNC MODES:
  Full Refresh (Overwrite): delete and recreate the table every sync
    Use: small tables, no incremental support in source
    
  Full Refresh (Append): append all current data each sync (grows forever)
    Use: audit history, when you want all historical snapshots
    
  Incremental (Append): only sync new/changed rows since last sync
    Use: large tables with a cursor field (updated_at, created_at)
    Most efficient: only transfers delta
    
  Incremental (Deduped): sync changes, deduplicate by primary key in destination
    Use: replicate exact current state of source table

CURSOR FIELD:
  For incremental sync: Airbyte tracks the max value of a cursor field.
  "Give me all rows where updated_at > last_synced_at"
  
  YOUR APPLICATION MUST HAVE: updated_at column on every table.
  Without it: incremental sync is impossible → must use Full Refresh → slow and expensive.
```

### Configuring a PostgreSQL Source

```python
# Airbyte API (Python SDK or REST)
import airbyte as ab

client = ab.get_client(api_key="your-api-key")

# Create a PostgreSQL source
source = client.sources.create(
    name="production-postgres",
    source_type="postgres",
    config={
        "host": "db.example.com",
        "port": 5432,
        "database": "myapp",
        "username": "airbyte_readonly",
        "password": "...",  # use secrets manager in production
        "ssl_mode": {"mode": "require"},
        "replication_method": {
            "method": "CDC",  # Change Data Capture via PostgreSQL logical replication
            # OR: "Standard" (SELECT-based, simpler but slower for large tables)
        }
    }
)

# Create a BigQuery destination
destination = client.destinations.create(
    name="bigquery-warehouse",
    destination_type="bigquery",
    config={
        "project_id": "my-project",
        "dataset_id": "raw",
        "credentials_json": "...",  # service account JSON
    }
)

# Create a connection (source → destination)
connection = client.connections.create(
    source_id=source.source_id,
    destination_id=destination.destination_id,
    streams=[
        {"name": "orders", "sync_mode": "incremental", "cursor_field": "updated_at"},
        {"name": "users", "sync_mode": "incremental", "cursor_field": "updated_at"},
        {"name": "products", "sync_mode": "full_refresh"},  # no cursor field
    ],
    schedule={"schedule_type": "cron", "cron_expression": "0 * * * *"},  # hourly
)
```

### CDC with Airbyte (for Real-Time-ish Sync)

```
CDC (Change Data Capture) via PostgreSQL logical replication:
  Airbyte reads the PostgreSQL WAL (write-ahead log).
  Every INSERT/UPDATE/DELETE → captured and replicated to warehouse.
  Latency: minutes (vs. hourly for standard cursor-based sync).
  
SETUP REQUIREMENTS:
  1. Enable logical replication in PostgreSQL:
     wal_level = logical  (in postgresql.conf)
  
  2. Create a replication user:
     CREATE ROLE airbyte_replication REPLICATION LOGIN;
     GRANT SELECT ON ALL TABLES IN SCHEMA public TO airbyte_replication;
  
  3. Ensure database is accessible from Airbyte server.
  
RESULT:
  Orders table in PostgreSQL changes → Airbyte captures via WAL → writes to BigQuery raw layer
  Delay: 2-5 minutes typically (vs. 1 hour with cursor-based sync)
```

## 3. Fivetran (16.8): Managed Data Pipelines

### What Is It?

Fivetran (2012) is the commercial, fully managed alternative to Airbyte. No self-hosting, no connector maintenance. You configure in a UI, Fivetran handles everything.

```
FIVETRAN vs. AIRBYTE:

FIVETRAN:
  ✓ Zero infrastructure to manage (fully managed SaaS)
  ✓ More reliable connectors (commercial, battle-tested)
  ✓ Better enterprise features (SOC 2, HIPAA, RBAC)
  ✓ Better support (SLA-backed)
  ✗ Expensive: ~$1 per 1000 "Monthly Active Rows" (can get very expensive)
  ✗ Vendor lock-in (proprietary platform)
  ✗ Less flexibility (can't customize connector behavior)

AIRBYTE:
  ✓ Open source (Apache 2.0)
  ✓ Self-hosted (lower cost, data stays in your infrastructure)
  ✓ Customizable (write your own connector)
  ✓ Active community, growing connector list (300+)
  ✗ Self-hosting has operational overhead
  ✗ Connectors can be less reliable than Fivetran's
  ✗ Less enterprise polish

CHOOSING:
  Startup, cost-sensitive, technical team: Airbyte (self-hosted)
  Enterprise, want managed service, willing to pay: Fivetran
  Want managed Airbyte: Airbyte Cloud
  
  The decision often comes down to: "Is the engineering time to self-host
  Airbyte worth more or less than Fivetran's pricing difference?"
  At small scale: self-hosting is worthwhile.
  At large scale or with compliance requirements: Fivetran often wins.
```

## 4. The Full Modern Data Stack

With Airbyte/Fivetran + dbt + BigQuery/Snowflake, you have the core of a "Modern Data Stack":

```
MODERN DATA STACK:

  Operational Systems           Data Stack
  ─────────────────            ──────────────────────────────────────
  PostgreSQL (app DB)    ──┐
  MySQL (legacy DB)      ──┤  EXTRACT + LOAD (Airbyte/Fivetran)
  Salesforce (CRM)       ──┼──────────────────────────────────────►
  Stripe (payments)      ──┤  Hourly/daily sync to raw layer
  Google Analytics       ──┤
  HubSpot (marketing)    ──┘
                              ↓
                         RAW LAYER (BigQuery/Snowflake)
                         raw.orders, raw.users, raw.events, etc.
                              ↓
                         TRANSFORM (dbt)
                         staging/ → intermediate/ → marts/
                              ↓
                         ANALYTICS LAYER
                         dim_customers, fct_orders, monthly_revenue
                              ↓
                         BI / SELF-SERVICE
                         Looker, Tableau, Metabase, Superset
                              ↓
                         ANALYSTS + BUSINESS STAKEHOLDERS

WHAT THIS STACK GIVES:
  ✓ All company data in one place (warehouse)
  ✓ Automated, maintained pipelines (Airbyte/Fivetran)
  ✓ Version-controlled, tested transformations (dbt)
  ✓ Business-friendly data models (marts layer)
  ✓ Self-serve analytics for non-engineers (BI tools)
```

## 5. Senior Engineer Perspective

**Making your application database pipeline-friendly:**

The most impactful things a backend engineer can do for the data team:

1. **`updated_at` on every table** (required for incremental sync)
   ```sql
   ALTER TABLE orders ADD COLUMN updated_at TIMESTAMPTZ DEFAULT NOW();
   -- Trigger to keep it updated on every change
   ```

2. **Soft deletes** (`deleted_at`) not hard deletes
   Data pipelines don't capture deletes (the row is gone). With soft deletes:
   - Airbyte syncs the row with `deleted_at` set
   - dbt can handle "hide deleted items from reports"

3. **Stable primary keys** (UUID or sequential integer — don't change)
   Fivetran/Airbyte use primary keys for deduplication in incremental sync

4. **Logical replication enabled** if you want low-latency CDC
   ```sql
   -- Check current wal_level (need 'logical' for CDC)
   SHOW wal_level;
   -- If not 'logical': needs postgresql.conf change + restart
   ```

5. **A read-only database user** for Airbyte/Fivetran
   Never give data pipelines write access to production databases
