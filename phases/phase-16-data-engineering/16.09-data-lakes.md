## 1. Data Lakes (16.9)

### What Is It?

A data lake is a centralized repository that stores all your data — structured, semi-structured, and unstructured — at any scale, in its raw format. Unlike a data warehouse (which requires predefined schema), a data lake stores everything first and applies schema when reading.

**The core concept:**

```
DATA WAREHOUSE PHILOSOPHY: Schema on write
  "Define the schema before loading data."
  Data is clean, typed, structured when it arrives.
  Queries are fast. Data quality is enforced.
  
DATA LAKE PHILOSOPHY: Schema on read
  "Load everything first, figure out schema later."
  Raw CSVs, JSON files, images, logs, parquet files — all stored as-is.
  Schema applied at query time by whoever needs the data.
  
THE DATA LAKE PROMISE:
  "Store everything, it might be valuable later."
  Your web server access logs from 2015 → store them.
  Raw JSON events from mobile app → store them.
  Product images → store them.
  
  Cost: S3 at $0.02/GB/month = ~$20/TB/month
  vs. Redshift at ~$1,000/TB/month
  → 50x cheaper for long-term retention
```

### Implementation: S3 as a Data Lake

```
TYPICAL DATA LAKE STRUCTURE ON S3:

s3://my-company-data-lake/

├── raw/                     ← as-extracted from sources
│   ├── orders/
│   │   └── year=2024/month=06/day=15/  ← partitioned by date
│   │       ├── orders_20240615_000000.jsonl
│   │       └── orders_20240615_060000.jsonl
│   ├── users/
│   └── events/
│
├── processed/               ← cleaned and validated
│   ├── orders.parquet/
│   └── events.parquet/
│
├── analytics/               ← business-ready aggregations
│   ├── daily_revenue/
│   └── user_segments/
│
└── archive/                 ← old data, infrequently accessed
    └── ...  (Glacier storage class: $0.004/GB/month)

FILE FORMATS IN THE DATA LAKE:
  JSON/JSONL: raw events, flexible schema, larger size
  CSV: simple exports, human-readable, no schema enforcement
  Parquet: columnar, compressed, fast for analytics (preferred for processed)
  ORC: similar to Parquet, preferred in Hive ecosystem
  Avro: row-based, good for streaming, schema embedded
```

### Data Lake Challenges

```
THE DATA SWAMP PROBLEM:
  Data lake without governance → data swamp.
  
  "We stored everything!" → 5 years later:
  Nobody knows what's in it.
  Nobody knows the schema of that 3TB file from 2019.
  Data quality is unknown.
  Analysts can't find what they need.
  
SOLUTIONS:
  DATA CATALOG (AWS Glue, Apache Atlas, DataHub):
    Inventory of all datasets, schemas, descriptions, owners.
    "What tables exist? What do the columns mean? Who owns this?"
  
  SCHEMA REGISTRY:
    Track and version schemas for datasets.
    Alert when schema changes in a breaking way.
  
  DATA QUALITY TOOLS (Great Expectations, Monte Carlo):
    Run checks on data quality automatically.
    "Are there suddenly null values in order_id? Alert the team."
  
  DATA LINEAGE:
    Track where data came from and where it goes.
    "This table feeds these 5 dashboards and this ML model."

DATA LAKEHOUSE (the modern evolution):
  Add Iceberg/Delta Lake to S3-based data lake:
  → ACID transactions (update/delete without full rewrite)
  → Schema enforcement (like a warehouse, but in the lake)
  → Time travel (query as-of any past timestamp)
  → Better performance (file pruning, Z-ordering)
  
  See topic 13.20 for full coverage.
```

## 2. Apache Hadoop (16.10)

### What Is It? (Brief Coverage)

Apache Hadoop (Yahoo, 2006) was the original distributed data processing framework. It consists of:

```
HDFS (Hadoop Distributed File System):
  Distributed storage across commodity servers.
  Large files split into 128MB blocks, replicated 3× across nodes.
  Designed for: large sequential reads, not random access.
  
MapReduce (Hadoop's processing model):
  Map: apply function to each record → emit (key, value) pairs
  Shuffle: group all records by key
  Reduce: aggregate all values for each key
  
  Every Map and Reduce step writes to disk.
  Very slow by modern standards (Spark is 10-100x faster).

HADOOP ECOSYSTEM:
  HDFS: distributed storage
  YARN: resource management
  MapReduce: batch processing (largely replaced by Spark)
  Hive: SQL on HDFS (topic 16.11)
  HBase: distributed database on HDFS
  Pig: scripting for Hadoop (rarely used now)
  Sqoop: import data from relational DBs to HDFS

HADOOP TODAY:
  Still widely deployed in enterprises with existing Hadoop clusters.
  New deployments increasingly favor cloud object storage (S3/GCS) + Spark.
  HDFS is being replaced by S3 + cloud storage for most new workloads.
  
  "Do you need to know Hadoop in 2024?"
  → If maintaining existing Hadoop clusters: yes.
  → If starting new: focus on cloud-native approach (S3 + Spark/Flink).
  → For interviews: understand the concepts, know Spark replaced MapReduce.
```

## 3. Apache Hive (16.11)

### What Is It?

Apache Hive (Facebook, 2008; Apache 2010) provides a SQL-like interface (HiveQL) for querying data stored in HDFS or S3. It was transformative for Hadoop: instead of writing Java MapReduce jobs, analysts could write SQL.

```
HIVE ARCHITECTURE:

  SQL query (HiveQL)
        ↓
  Hive Metastore (catalog: table definitions, schemas, locations)
        ↓
  Query Engine (translates SQL to execution plan)
        ↓
  Execution Engine (MapReduce or Tez or Spark)
        ↓
  HDFS or S3 (actual data files)

HIVE METASTORE:
  The key piece: a relational database (usually MySQL/PostgreSQL) that
  stores metadata about Hive tables:
  - Table name, column names, data types
  - Location: "this table's data is in s3://bucket/path/"
  - Partition information
  
  Other tools (Spark, Presto, Athena) can also use the Hive Metastore.
  It's become the de facto catalog for the Hadoop/data lake ecosystem.

EXAMPLE HIVE QUERY:
  -- Create external table pointing to S3 data
  CREATE EXTERNAL TABLE orders (
      order_id  BIGINT,
      user_id   BIGINT,
      amount    DECIMAL(10,2),
      status    STRING,
      created_at TIMESTAMP
  )
  PARTITIONED BY (year INT, month INT, day INT)
  STORED AS PARQUET
  LOCATION 's3://my-data-lake/processed/orders/';
  
  -- Query it
  SELECT
      country,
      SUM(amount) AS total_revenue
  FROM orders
  WHERE year = 2024 AND month = 6  -- partition pruning: reads only June 2024 files
  GROUP BY country;

HIVE VS. PRESTO/TRINO VS. SPARK SQL:
  Hive (original): slow (MapReduce/Tez execution), widely deployed
  Presto/Trino: fast interactive queries on S3/Hive data (Facebook, 2012)
  Spark SQL: Spark-based, fast, works with Hive Metastore
  
  Modern choice: Trino or Spark SQL for new workloads.
  Hive: maintain existing, or just use its Metastore with a faster engine.
  
  Athena (AWS): managed Presto/Trino on top of S3 + Glue Catalog.
               Pay per query. Zero infrastructure.
               The most common way to query data lakes on AWS today.
```

## 4. Senior Engineer Perspective

**The data lake's value for backend engineers:**

As a backend engineer, understanding data lakes helps you:

1. **Long-term log retention**: dump your application logs to S3 cheaply ($20/TB/month). Use Athena to query them when you need to debug something from 2 years ago — without paying for warehouse storage.

2. **Event archival**: raw events to S3 (cheap, infinite retention) → Parquet conversion → Athena queries for historical analysis

3. **ML training data**: raw data for ML model training lives in data lakes. When the ML team asks for historical data exports, they want it in S3 as Parquet or JSON.

**Hadoop's legacy:**

Hadoop pioneered the idea of storing and processing data across commodity hardware at massive scale — this was revolutionary in 2006 when enterprise storage was extremely expensive. The concepts (distributed storage, data locality, fault tolerance through replication) influenced every data system that came after it, including Spark, Kafka, and cloud object storage design.

Understanding Hadoop's design explains why Spark exists (faster Hadoop), why S3 is used as a data lake (Hadoop filesystem semantics at cloud scale), and why the Hive Metastore became a universal catalog standard.
