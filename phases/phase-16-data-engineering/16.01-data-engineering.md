## ETL vs. ELT (16.1)

```
ETL (Extract, Transform, Load):
  1. Extract: pull data from sources (databases, APIs, files)
  2. Transform: clean, enrich, aggregate in a processing engine
  3. Load: write transformed data to destination (data warehouse)
  
  Old pattern: processing power was expensive → transform before loading
  Tools: Apache Spark, custom scripts

ELT (Extract, Load, Transform):
  1. Extract: pull raw data
  2. Load: store raw data in destination (data lake or warehouse)
  3. Transform: use the destination's compute power (SQL in BigQuery/Snowflake)
  
  New pattern: modern warehouses (BigQuery, Snowflake) have massive compute
  → Load raw first, transform later with SQL (dbt)
  → More flexible: rerun transformations without re-extracting data
  
  Tools: Fivetran/Airbyte (Extract+Load), dbt (Transform)
```

## Batch vs. Stream Processing (16.2)

```
BATCH PROCESSING:
  Process large amounts of historical data at once.
  Example: compute daily revenue report at midnight.
  Tools: Apache Spark, SQL in BigQuery/Snowflake
  Latency: minutes to hours
  Use when: freshness requirements are hours or days

STREAM PROCESSING:
  Process events as they arrive, continuously.
  Example: detect fraudulent transactions within 100ms.
  Tools: Apache Flink, Kafka Streams, Spark Streaming
  Latency: milliseconds to seconds
  Use when: need near-real-time results

MICRO-BATCH (Spark Streaming):
  Process small batches every few seconds.
  Middle ground: not quite streaming, much fresher than daily batch.
  Latency: 1-30 seconds
```

## Apache Spark (16.3)

```python
# PySpark: Python API for Apache Spark (distributed batch processing)
from pyspark.sql import SparkSession
from pyspark.sql import functions as F

spark = SparkSession.builder \
    .appName("OrderAnalysis") \
    .config("spark.sql.adaptive.enabled", "true") \
    .getOrCreate()

# Read from multiple sources
orders = spark.read.parquet("s3://data-lake/orders/")
users = spark.read.jdbc(url=POSTGRES_URL, table="users", properties=DB_PROPS)

# Distributed transformations (lazy — executed only when needed)
revenue_by_region = (
    orders
    .filter(F.col("status") == "paid")
    .join(users, "user_id")
    .groupBy("region", F.date_trunc("month", "created_at").alias("month"))
    .agg(
        F.sum("total_cents").alias("revenue_cents"),
        F.count("*").alias("order_count"),
        F.countDistinct("user_id").alias("unique_customers"),
    )
    .orderBy("month", "region")
)

# Write results
revenue_by_region.write \
    .mode("overwrite") \
    .partitionBy("month") \
    .parquet("s3://data-warehouse/revenue-by-region/")

spark.stop()
```

## dbt (16.6): SQL-Based Data Transformations

```sql
-- dbt: transform raw data in your warehouse using SQL
-- models/orders/order_summary.sql

{{ config(
    materialized='table',   -- or 'view', 'incremental'
    schema='analytics',
    post_hook=['GRANT SELECT ON {{ this }} TO analytics_users']
) }}

WITH raw_orders AS (
    SELECT * FROM {{ ref('stg_orders') }}   -- reference to another dbt model
),
with_user_info AS (
    SELECT
        o.*,
        u.email,
        u.region,
        u.created_at AS user_created_at
    FROM raw_orders o
    LEFT JOIN {{ ref('stg_users') }} u ON o.user_id = u.id
)
SELECT
    DATE_TRUNC('month', created_at) AS order_month,
    region,
    COUNT(*)              AS order_count,
    SUM(total_cents)      AS revenue_cents,
    COUNT(DISTINCT user_id) AS unique_customers,
    AVG(total_cents)      AS avg_order_value_cents
FROM with_user_info
WHERE status = 'paid'
GROUP BY 1, 2

-- Run: dbt run --models orders.order_summary
-- Test: dbt test  (run schema tests: not_null, unique, relationships)
-- Docs: dbt docs generate && dbt docs serve
```

## Apache Airflow (16.12): Workflow Orchestration

```python
# Airflow DAG: define a data pipeline as a Directed Acyclic Graph
from airflow import DAG
from airflow.operators.python import PythonOperator
from airflow.providers.postgres.operators.postgres import PostgresOperator
from datetime import datetime, timedelta

with DAG(
    dag_id="daily_revenue_report",
    start_date=datetime(2024, 1, 1),
    schedule="0 6 * * *",   # every day at 6am UTC
    catchup=False,           # don't run historical dates on first deploy
    default_args={
        "retries": 2,
        "retry_delay": timedelta(minutes=5),
        "email_on_failure": True,
        "email": ["data-team@example.com"],
    },
) as dag:

    # Task 1: extract from production DB
    extract = PostgresOperator(
        task_id="extract_orders",
        postgres_conn_id="prod_db",
        sql="""
            COPY (
                SELECT * FROM orders WHERE DATE(created_at) = CURRENT_DATE - 1
            ) TO STDOUT WITH CSV HEADER
        """,
    )

    # Task 2: transform
    def transform(**context):
        ds = context["ds"]  # execution date: YYYY-MM-DD
        # ... process the extracted data ...

    transform_task = PythonOperator(
        task_id="transform_data",
        python_callable=transform,
    )

    # Task 3: load to warehouse
    load = PythonOperator(
```

## Schema Evolution (16.13)

```python
# When data formats change over time:

# Apache Avro: schema-based serialization with schema registry
# Schemas evolve while maintaining backward/forward compatibility

# COMPATIBILITY RULES:
# Backward compatible: new schema can read data written with old schema
#   → Safe to upgrade consumers before producers
#   → You CAN: add optional fields with defaults, remove optional fields
#   → You CANNOT: rename fields, add required fields without defaults

# Forward compatible: old schema can read data written with new schema
#   → Safe to upgrade producers before consumers

# Full compatible: both backward AND forward compatible
#   → Most conservative, safest for long-lived schemas

# Avro schema (before):
schema_v1 = {
    "type": "record",
    "name": "Order",
    "fields": [
        {"name": "id", "type": "string"},
        {"name": "amount_cents", "type": "int"},
    ]
}

# Avro schema (after - backward compatible: new field has default):
schema_v2 = {
    "type": "record",
    "name": "Order",
    "fields": [
        {"name": "id", "type": "string"},
        {"name": "amount_cents", "type": "int"},
        {"name": "currency", "type": "string", "default": "USD"},  # OK: has default
    ]
}
```
