## 1. What Is It? (Theory)

dbt (data build tool, dbt Labs, 2016) is the tool that brought software engineering best practices to data transformation. Before dbt, analytical transformations were a mess: SQL scripts living in ad-hoc files, transformations hidden inside BI tools, no version control, no tests, no documentation, no way to understand dependencies.

dbt changes this by treating SQL transformations as **software artifacts**: versioned in git, testable, self-documenting, and with explicit dependency management.

**The dbt promise in one sentence:**

> Write SQL SELECT statements. dbt takes care of creating the tables, managing dependencies, running tests, and generating documentation.

## 2. How It Actually Works (Deep Mechanics)

### The Core Concept: Models Are SELECT Statements

```sql
-- models/orders/paid_orders.sql
-- dbt turns this SELECT statement into a table or view in your warehouse

SELECT
    id AS order_id,
    user_id,
    amount_cents,
    currency_code,
    created_at,
    status
FROM {{ source('raw', 'orders') }}   -- {{ }} = Jinja templating
WHERE status = 'paid'

-- dbt generates and runs:
-- CREATE TABLE analytics.paid_orders AS (
--   SELECT id AS order_id, ...
--   FROM raw.orders
--   WHERE status = 'paid'
-- )
```

### The Dependency Graph (DAG)

dbt automatically discovers dependencies between models using `{{ ref() }}`:

```sql
-- models/orders/order_summary.sql
-- References paid_orders (above) and users

SELECT
    o.order_id,
    o.user_id,
    u.email,
    u.country,
    o.amount_cents,
    o.created_at,
    -- window function: running total per user
    SUM(o.amount_cents) OVER (
        PARTITION BY o.user_id 
        ORDER BY o.created_at
    ) AS lifetime_value_cents
FROM {{ ref('paid_orders') }} o          -- depends on paid_orders model
JOIN {{ ref('stg_users') }} u ON o.user_id = u.user_id  -- depends on stg_users

-- dbt knows: order_summary depends on paid_orders AND stg_users
-- dbt runs: raw tables → paid_orders, stg_users → order_summary
-- In the right order, respecting dependencies.
```

### Tests: Built-In Data Quality

```yaml
# models/marts/core/schema.yml
version: 2

models:
  - name: fct_orders
    description: "One row per completed order"
    columns:
      - name: order_id
        description: "Unique identifier for each order"
        tests:
          - unique           # no duplicate order_ids
          - not_null         # every row has an order_id
      
      - name: user_id
        tests:
          - not_null
          - relationships:   # referential integrity
              to: ref('dim_customers')
              field: customer_id
      
      - name: status
        tests:
          - accepted_values:  # only these values allowed
              values: ['placed', 'confirmed', 'shipped', 'delivered', 'cancelled']
      
      - name: amount_cents
        tests:
          - not_null
          # Custom test: all amounts must be positive
          - dbt_utils.expression_is_true:
              expression: "> 0"
```

```bash
# Run all tests
dbt test

# Output:
# Running 12 tests...
# ✓ unique order_id                 (0 failures)
# ✓ not_null order_id               (0 failures)
# ✗ relationships user_id → dim_customers  (23 failures!)
#   23 orders reference user_ids not in customers table → orphaned records
```

### Running dbt

```bash
# Run all models (creates/replaces tables in the warehouse)
dbt run

# Run specific model and all its dependencies
dbt run --select order_summary+  # + includes downstream models

# Run only changed models (incremental workflow)
dbt run --select state:modified+

# Test data quality
dbt test

# Generate documentation site
dbt docs generate
dbt docs serve    # serves interactive lineage DAG at localhost:8080

# Check what would change before running
dbt compile       # generates SQL without executing

# Incremental models (only process new data)
dbt run --full-refresh  # full refresh of incremental models
```

### Incremental Models: Efficient Processing

For large tables, dbt supports incremental materialization — only process new/changed records:

```sql
-- models/marts/fct_events.sql
{{
    config(
        materialized='incremental',
        unique_key='event_id',
        on_schema_change='fail'  # fail if source schema changes unexpectedly
    )
}}

SELECT
    event_id,
    user_id,
    event_type,
    properties,
    occurred_at
FROM {{ source('raw', 'events') }}

{% if is_incremental() %}
    -- On incremental run: only process new events since last run
    WHERE occurred_at > (SELECT MAX(occurred_at) FROM {{ this }})
{% endif %}

-- First run: processes ALL events
-- Subsequent runs: processes only NEW events (fast, cheap)
```

## 3. Real-World Context

### The dbt + Snowflake/BigQuery Revolution

Before dbt, a typical data transformation looked like:
- Python script in a Jupyter notebook (not version-controlled)
- Schedule via cron (no visibility)
- No tests (don't know if data quality degraded)
- No documentation (analysts don't know what tables mean)
- No lineage (don't know what breaks if you change a source table)

After dbt:
- SQL in git repository (version control, pull requests, code review)
- Scheduled via dbt Cloud or Airflow (visible, monitored)
- Tests run automatically (data quality catches issues before analysts see broken data)
- Auto-generated documentation (analysts understand every table, column description)
- Lineage DAG (immediately see "if I change stg_orders, which downstream models break?")

Thousands of companies switched to dbt-based architectures between 2018-2022. It's now the de facto standard for data transformation in modern data stacks.

## 4. Senior Engineer Perspective

**dbt as the interface between backend and data engineering:**

As a backend engineer, you own the source tables that dbt reads from (`source('raw', 'orders')` → your application's orders table replicated to the warehouse). When you change your schema:

1. **Communicate changes in advance**: dbt models will break if you rename a column without warning the data team
2. **Use backward-compatible changes where possible**: add new columns (safe), don't rename existing ones (breaking)
3. **The raw layer**: Airbyte/Fivetran replicate your tables exactly — dbt models reference these. Schema changes in your app DB propagate to the raw layer automatically (if the connector handles it) and must be handled in dbt staging models

**dbt tests as a contract:**

`not_null order_id` in dbt is a contract between you (backend) and the data team: "order_id will never be null." If your application starts inserting null order_ids, dbt tests fail and alert the data team. These tests are a communication mechanism across team boundaries.
