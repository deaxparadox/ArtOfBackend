## 1. What Is It? (Theory)

Workflow orchestration is the coordination, scheduling, monitoring, and management of data pipelines — sequences of tasks that depend on each other, run on a schedule, and must complete reliably.

The key insight: **a data pipeline is not just a script you run once. It's a production system that must:**
- Run on a schedule (hourly, daily, triggered by upstream completion)
- Retry failed steps automatically
- Alert when things break
- Be observable (which step is running? which failed? what's the output?)
- Handle dependencies (run step B only after step A completes successfully)
- Allow partial reruns (re-run only the failed step, not the whole pipeline)

Workflow orchestration tools provide all of this.

**The real-world analogy:**

Managing data pipelines without an orchestrator is like managing a complex project with sticky notes. You know what needs to happen, but tracking what's running, what finished, what failed, and what to do when something goes wrong requires constant manual attention. An orchestrator is the project management software that tracks all of it automatically.

## 2. The Three Major Tools

### Apache Airflow (16.12a)

Apache Airflow (Airbnb, 2014; Apache 2019) is the most widely-deployed workflow orchestrator. Pipelines are defined as Python code (DAGs — Directed Acyclic Graphs).

```python
# Apache Airflow DAG definition

from datetime import datetime, timedelta
from airflow import DAG
from airflow.operators.python import PythonOperator
from airflow.providers.postgres.operators.postgres import PostgresOperator
from airflow.providers.amazon.aws.operators.s3 import S3CreateObjectOperator

# DAG configuration
default_args = {
    "owner": "data-team",
    "depends_on_past": False,
    "email_on_failure": True,
    "email": ["data-alerts@example.com"],
    "retries": 2,
    "retry_delay": timedelta(minutes=5),
    "retry_exponential_backoff": True,
}

with DAG(
    dag_id="daily_revenue_pipeline",
    default_args=default_args,
    description="Extract orders → transform → load to warehouse → refresh dashboard",
    schedule="0 6 * * *",     # every day at 6am UTC
    start_date=datetime(2024, 1, 1),
    catchup=False,             # don't backfill missed runs
    tags=["revenue", "daily"],
) as dag:
    
    # Task 1: extract orders from production DB to S3
    extract_orders = PostgresOperator(
        task_id="extract_orders",
        postgres_conn_id="production_db",
        sql="""
            COPY (
                SELECT * FROM orders WHERE DATE(created_at) = '{{ ds }}'
            ) TO STDOUT WITH CSV HEADER
        """,
    )
    
    # Task 2: transform with Spark (run a spark job)
    transform_data = SparkSubmitOperator(
        task_id="transform_data",
        application="s3://code/transform_orders.py",
        application_args=["--date", "{{ ds }}"],  # {{ ds }} = execution date
    )
    
    # Task 3: load to data warehouse
    def load_to_warehouse(execution_date, **kwargs):
        # Python function called by Airflow
        load_parquet_to_bigquery(
            source_path=f"s3://data/transformed/orders/{execution_date}",
            destination_table="analytics.orders",
        )
    
    load_warehouse = PythonOperator(
        task_id="load_to_warehouse",
        python_callable=load_to_warehouse,
        provide_context=True,
    )
    
    # Task 4: refresh dashboard
    refresh_dashboard = HttpOperator(
        task_id="refresh_dashboard",
        endpoint="/api/dashboards/revenue/refresh",
        method="POST",
        http_conn_id="looker_api",
    )
    
    # DEPENDENCIES: define task execution order
    extract_orders >> transform_data >> load_warehouse >> refresh_dashboard
    # extract_orders runs first
    # transform_data runs after extract_orders succeeds
    # load_warehouse runs after transform_data succeeds
    # refresh_dashboard runs after load_warehouse succeeds
```

```
AIRFLOW KEY CONCEPTS:

DAG (Directed Acyclic Graph):
  The pipeline definition. A Python file that defines tasks and their dependencies.
  
TASK:
  A single unit of work within a DAG.
  
OPERATOR:
  The type of task (Python function, SQL query, Spark job, HTTP call, etc.)
  Airflow has 100+ built-in operators for common services.
  
SENSOR:
  A special operator that waits for a condition.
  "Wait until this S3 file exists, then proceed."
  
SCHEDULER:
  Background process that triggers DAG runs based on schedule.
  
WORKER:
  Executes the actual tasks.
  
METADATA DATABASE (PostgreSQL):
  Stores DAG definitions, task states, execution history.
  
AIRFLOW UI:
  Web interface showing: all DAGs, their status, task logs, Gantt charts.
```

```
AIRFLOW STRENGTHS:
  ✓ Most widely deployed — huge ecosystem, many operators, much documentation
  ✓ Rich UI: Gantt charts, dependency graphs, log viewer, retry management
  ✓ Python code — familiar to engineers, version-controlled in git
  ✓ Mature: handles complex dependency patterns
  ✓ Extensive operator library (all cloud providers, all databases)

AIRFLOW WEAKNESSES:
  ✗ Operational complexity: separate scheduler, workers, metadata DB
  ✗ DAG parsing performance: 1000+ DAGs can make the scheduler slow
  ✗ Dynamic workflows: task counts defined at DAG-parse time (not dynamic)
  ✗ Testing: hard to unit test DAGs locally
  ✗ No first-class data awareness: Airflow orchestrates tasks, not data
```

### Prefect (16.12b)

Prefect (2020) is a modern alternative to Airflow that addresses Airflow's rough edges:

```python
# Prefect flow definition (modern Python, much simpler)

from prefect import flow, task
from prefect.tasks import task_input_hash
from datetime import timedelta

@task(
    retries=3,
    retry_delay_seconds=60,
    cache_key_fn=task_input_hash,
    cache_expiration=timedelta(hours=1),
)
def extract_orders(date: str) -> list[dict]:
    """Extract orders from production DB."""
    return db.fetch_orders_for_date(date)

@task(retries=2)
def transform_orders(orders: list[dict]) -> list[dict]:
    """Apply business transformations."""
    return [
        {**order, "amount_usd": convert_to_usd(order["amount_cents"], order["currency"])}
        for order in orders
        if order["status"] == "paid"
    ]

@task
def load_to_warehouse(data: list[dict], date: str) -> None:
    """Load to BigQuery."""
    bigquery.load(data, f"analytics.orders__{date}")

@task
def refresh_dashboard() -> None:
    """Notify dashboard to refresh."""
    requests.post("https://looker.example.com/api/refresh")

@flow(name="Daily Revenue Pipeline")
def daily_revenue_pipeline(date: str = None):
    """Main pipeline orchestrating all steps."""
    date = date or str(date.today())
    
    # Tasks run in dependency order automatically
    orders = extract_orders(date)
    transformed = transform_orders(orders)
    load_to_warehouse(transformed, date)
    refresh_dashboard()

# Run locally
if __name__ == "__main__":
    daily_revenue_pipeline(date="2024-06-15")

# Deploy to Prefect Cloud or self-hosted
# prefect deploy --name daily-revenue-pipeline --cron "0 6 * * *"
```

```
PREFECT ADVANTAGES OVER AIRFLOW:
  ✓ Python-native: feels like regular Python code, easy to test locally
  ✓ Dynamic workflows: task counts can be determined at runtime
  ✓ Better developer experience: run flows locally during development
  ✓ Prefect Cloud: managed observability (no self-hosting required)
  ✓ Automatic retries, notifications built-in to decorators
  
PREFECT DISADVANTAGES:
  ✗ Smaller ecosystem than Airflow (fewer native integrations)
  ✗ Smaller community (less Stack Overflow, less documentation)
  ✗ Relatively newer (less battle-tested at extreme scale)
```

### Dagster (16.12c)

Dagster (2018) introduces a different philosophy: **orchestrate data, not just tasks**. Dagster is aware of the data assets your pipelines produce and consume.

```python
# Dagster - asset-based orchestration
from dagster import asset, Output, AssetIn, Definitions, ScheduleDefinition, build_schedule_from_partitioned_job

@asset
def raw_orders(context) -> pd.DataFrame:
    """Extract raw orders from production DB."""
    return pd.read_sql("SELECT * FROM orders", production_db)

@asset(ins={"raw_orders": AssetIn()})
def paid_orders(raw_orders: pd.DataFrame) -> pd.DataFrame:
    """Filter to only paid orders."""
    return raw_orders[raw_orders["status"] == "paid"]

@asset(ins={"paid_orders": AssetIn()})
def revenue_by_country(paid_orders: pd.DataFrame) -> pd.DataFrame:
    """Aggregate revenue by country."""
    return paid_orders.groupby("country")["amount_cents"].sum().reset_index()

# Dagster generates a LINEAGE GRAPH:
# raw_orders → paid_orders → revenue_by_country

# Schedule and sensor definitions
defs = Definitions(
    assets=[raw_orders, paid_orders, revenue_by_country],
    schedules=[ScheduleDefinition(job=..., cron_schedule="0 6 * * *")],
)
```

```
DAGSTER'S UNIQUE VALUE:
  ✓ Asset-centric: understands the data assets (tables, files) your code produces
  ✓ Data lineage: visible graph of how data flows through your pipeline
  ✓ Observability: knows when each asset was last updated, its freshness
  ✓ Partitioned assets: handles daily/monthly partitions natively
  ✓ Type checking: assets have explicit types, caught at development time
  ✓ Best for: teams that care about data products, data mesh architectures
  
DAGSTER DISADVANTAGES:
  ✗ Steeper learning curve than Prefect
  ✗ Different paradigm from task-centric tools (requires retraining)
```

## 3. Choosing the Right Orchestrator

```
CHOOSE AIRFLOW WHEN:
  ✓ Your organization already uses Airflow (migration cost is high)
  ✓ You need specific operators for legacy or niche systems
  ✓ You have large teams familiar with Airflow
  ✓ Managed Airflow (Astronomer, MWAA) reduces operational burden

CHOOSE PREFECT WHEN:
  ✓ New project, engineer-first philosophy
  ✓ You want easy local testing
  ✓ Dynamic pipelines (task counts unknown until runtime)
  ✓ Prefect Cloud removes operational burden

CHOOSE DAGSTER WHEN:
  ✓ Data mesh / data products-oriented organization
  ✓ Data lineage visibility is critical
  ✓ You want to model "what data assets exist" not just "what tasks run"
  ✓ Building a modern data platform from scratch

MANAGED OPTIONS (avoid self-hosting):
  Astronomer (managed Airflow): best for teams committed to Airflow
  Prefect Cloud: best developer experience, generous free tier
  Google Cloud Composer (managed Airflow): good for GCP teams
  AWS MWAA (managed Airflow): good for AWS teams with Airflow expertise
```

## 4. Senior Engineer Perspective

**Orchestration vs. application scheduling:**

Backend engineers often reach for Celery Beat or cron for scheduled data jobs. For simple cases, this is fine. But as pipelines grow:
- "The export job failed at step 3 of 7 — can we re-run just step 3?" → not possible with cron
- "Which dashboards are broken because of this pipeline failure?" → not visible with cron
- "The daily revenue report was 2 hours late today — why?" → no observability with cron

Orchestration tools provide these capabilities. For any data pipeline that's more than 1-2 steps and runs on a schedule, an orchestrator is worth the investment.

**The orchestrator is infrastructure, not part of your application:**

Data orchestrators should be managed by the data team, not the backend team. Backend engineers contribute by:
1. Making APIs and databases orchestrators can call (stable endpoints, read-only DB users)
2. Alerting the data team to schema changes that break pipelines
3. Understanding what the orchestrator does so you can debug when data is stale
