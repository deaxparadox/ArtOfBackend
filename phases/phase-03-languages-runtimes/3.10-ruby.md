## 1. Ruby's Place in Backend History

Ruby on Rails (2004) was one of the most influential frameworks in web development history. DHH (David Heinemeier Hansson) built Basecamp in Rails and open-sourced it. Rails popularized:

- **Convention over configuration** — sensible defaults reduce boilerplate
- **DRY (Don't Repeat Yourself)** — the principle became mainstream
- **MVC web framework** — the template every other framework copied
- **Database migrations** — version-controlled schema changes
- **ActiveRecord ORM** — the pattern every ORM copied
- **REST conventions** — scaffolding built on REST from day one

GitHub, Shopify, Airbnb, Basecamp, and Twitter were all built on Rails.

## 2. Ruby's Characteristics

```ruby
# Ruby is extremely expressive — reads like English
class User
  attr_accessor :name, :email

  def initialize(name, email)
    @name = name
    @email = email
  end

  def adult?  # methods can end in ? (returns boolean by convention)
    age >= 18
  end

  def to_s
    "#{name} <#{email}>"
  end
end

# Blocks, procs, lambdas — Ruby's functional features
[1, 2, 3, 4, 5]
  .select { |n| n.odd? }
  .map    { |n| n * 2 }
  .reduce { |sum, n| sum + n }
# → 18

# Rails in action — the famous 15-minute blog demo concept
class PostsController < ApplicationController
  def index
    @posts = Post.recent.published.paginate(page: params[:page])
  end

  def create
    @post = current_user.posts.build(post_params)
    if @post.save
      redirect_to @post, notice: "Post created!"
    else
      render :new, status: :unprocessable_entity
    end
  end
end
```

## 3. Ruby's Concurrency Model

Ruby has the GIL (like Python) in its reference implementation (MRI/CRuby). This limits CPU parallelism with threads.

**Common solutions:**
- **Puma** (Rails default): multi-threaded server. GIL limits CPU work, but I/O is released.
- **Unicorn**: multi-process (no GIL issue, but more memory)
- **Sidekiq**: background jobs in threads (I/O-bound work, GIL is fine)
- **JRuby**: Ruby on JVM — no GIL, true thread parallelism

---

## 4. Modern Ruby (3.x) and Ractors

Ruby 3 introduced **Ractors** — an actor-model concurrency primitive that allows true parallelism by bypassing the GIL:

```ruby
# Ractors run in parallel (true CPU parallelism)
ractor1 = Ractor.new { expensive_computation(1) }
ractor2 = Ractor.new { expensive_computation(2) }

result1 = ractor1.take
result2 = ractor2.take
```

Ractors are still maturing (experimental in 3.0, improving in 3.x) but represent Ruby's path to removing the GIL limitation.

## 5. When to Choose Ruby/Rails

```
RUBY/RAILS IS EXCELLENT FOR:
  ✓ Rapid prototyping and MVPs (fastest time to working product)
  ✓ CRUD-heavy applications (Rails scaffolding is unmatched)
  ✓ Teams already using Rails (ecosystem inertia is real)
  ✓ Admin dashboards and internal tools
  ✓ E-commerce (Shopify is the world's most successful Rails app)

RUBY/RAILS STRUGGLES WITH:
  ✗ High-concurrency, CPU-bound workloads
  ✗ Services that need to scale to millions of users per server
  ✗ Real-time features (though ActionCable helps)
  ✗ Hiring (Python/JS/Go have larger talent pools)

SHOPIFY'S SCALE ON RAILS (2023):
  Handles $5.6 billion in sales on Black Friday on Rails.
  Proof that Rails can scale — with significant engineering investment.
  They've extracted microservices for the hottest paths,
  but the core is still Rails.
```

## 6. Senior Engineer Perspective

**The Rails legacy:** If you join a company with an existing Rails codebase, you're joining a rich tradition of conventions. The Rails Way, when followed, is genuinely productive. When violated (fat controllers, no service layer, callbacks everywhere), it becomes a maintenance nightmare.

**Ruby's future:** Ruby 3 is 3x faster than Ruby 2 (the "3x3" goal, achieved). YJIT (in-process JIT compiler, adopted from Shopify) continues to push performance. Ruby is not dying — Shopify's massive investment ensures its continued relevance. But its market share vs. Python, TypeScript, and Go has declined.
