## 1. The Three Memory Management Strategies

Every language must answer: "When is it safe to free memory?"

### Manual Memory Management (C, C++)

The programmer explicitly allocates and frees memory. Fast and predictable. Dangerous.

```c
// C: manual allocation
char* buffer = malloc(1024);    // allocate 1KB
if (buffer == NULL) { /* handle error */ }

// use buffer...
strcpy(buffer, "hello world");

free(buffer);    // must free manually
buffer = NULL;   // good practice: prevent use-after-free

// BUGS if you don't:
// Memory leak: forget to call free() → memory grows until OOM
// Use-after-free: access buffer after free() → undefined behavior, security vuln
// Double-free: call free() twice → heap corruption
// Buffer overflow: write past the end → stack smashing, security vuln
```

These bugs are the root cause of most historical CVEs (security vulnerabilities). Buffer overflow exploits have been used in attacks for 30+ years.

### Garbage Collection (Python, Java, Go, C#, JavaScript)

The runtime automatically tracks which memory is reachable and frees unreachable memory.

```
APPROACHES:
  Reference Counting (Python CPython):
    Each object tracks how many references point to it.
    Refcount drops to 0 → free immediately.
    Fast and predictable. Can't handle cycles (needs supplemental cycle collector).

  Mark-and-Sweep (Java G1, Go, V8):
    Periodically: trace all reachable objects from roots (globals, stack).
    Anything not reached = garbage → sweep and free.
    Handles cycles. Requires pause ("stop-the-world") or concurrent marking.

  Generational GC (Java, .NET, Python partial):
    Most objects die young. Collect young generation frequently (cheap).
    Long-lived objects promoted to older generation (collected rarely).
    Reduces GC pause frequency for typical allocation patterns.

TRADE-OFFS:
  GC PROS: no manual memory management, no leaks, no use-after-free
  GC CONS: GC pauses (even sub-ms pauses add to p99 latency)
           Memory overhead (GC needs headroom to work efficiently)
           Non-deterministic timing
```

### Ownership System (Rust)

Rust's ownership system is the third approach: compile-time memory management. The compiler tracks every memory region's lifetime and inserts free calls automatically — without a GC.

```rust
fn main() {
    let s1 = String::from("hello");  // allocated on heap, owned by s1
    let s2 = s1;                     // s1's ownership MOVED to s2
    // s1 is no longer valid (compiler enforces this)
    // println!("{}", s1);            // COMPILE ERROR

    {
        let s3 = String::from("world");  // allocated
    }  // s3's scope ends → automatically freed here (no GC needed)

    // s2 is freed when main() returns
}

// RESULT:
// No GC pauses. No memory leaks. No use-after-free.
// Memory is freed exactly when it's no longer needed.
// All verified at compile time.
```

## 2. GC Tuning in Production

```
JAVA GC TUNING:
  # G1GC (default): good for most workloads
  java -Xms2g -Xmx4g -XX:+UseG1GC \
       -XX:MaxGCPauseMillis=200 \
       -XX:G1HeapRegionSize=16m \
       -jar myapp.jar

  # ZGC: for latency-sensitive (<1ms pauses)
  java -Xms4g -Xmx4g -XX:+UseZGC -jar myapp.jar

  # GraalVM Native: no GC pauses at all (AOT compiled)

PYTHON GC TUNING:
  import gc
  # Disable automatic GC for performance-critical sections
  gc.disable()
  # ... do your work ...
  gc.collect()  # manual collection at safe point
  gc.enable()

  # Tune thresholds (default: 700, 10, 10)
  gc.set_threshold(1000, 15, 15)

GO GC TUNING:
  # GOGC: target heap growth ratio (default 100 = GC when heap doubles)
  GOGC=200 ./myservice    # less frequent GC (higher memory, less CPU)
  GOGC=50 ./myservice     # more frequent GC (lower memory, more CPU)

  # GOMEMLIMIT (Go 1.19+): hard memory limit
  GOMEMLIMIT=512MiB ./myservice  # GC aggressively to stay under 512MB
```

## 3. Memory Leaks in GC Languages

GC prevents *accidental* memory leaks (forgetting to free). But logical leaks — holding references longer than needed — still happen:

```python
# Python memory leak: holding references in a long-lived container
class CacheService:
    def __init__(self):
        self._cache = {}  # grows unbounded if keys are never removed!

    def cache(self, key: str, value: any):
        self._cache[key] = value  # no eviction = memory leak

    # Fix: use functools.lru_cache, cachetools.TTLCache, or Redis

# Python memory leak: event listeners not removed
class EventBus:
    listeners = {}

    @classmethod
    def subscribe(cls, event, handler):
        cls.listeners.setdefault(event, []).append(handler)
        # If handler is a method of a short-lived object:
        # the EventBus holds a reference to the object via handler
        # → object is never GC'd → memory leak

    # Fix: use weakref.WeakMethod or explicitly unsubscribe

# Node.js memory leak: global variables accumulating data
const cache = {};  // global
app.get('/items', (req, res) => {
    const item = processRequest(req);
    cache[item.id] = item;  // grows forever
    // Fix: use a size-limited cache (lru-cache npm package)
});
```

## 4. Senior Engineer Perspective

**What a junior gets wrong:**

Assuming GC languages can't have memory leaks. They can — just different kinds. In Python: circular references with `__del__` methods, unclosed file handles, growing global caches. In Java: static collections holding objects, thread-local storage not cleaned up. GC handles the "forgot to free" case; engineers must handle the "holding reference unnecessarily" case.

**Monitoring memory in production:**

Every backend service should track heap/RSS memory over time. A gradual upward trend over hours/days is a memory leak. A sudden spike is a bursty allocation. Normal: memory fluctuates around a stable baseline. Abnormal: continuous growth without plateau.

```bash
# Linux: process memory over time
watch -n 5 'cat /proc/<PID>/status | grep -E "VmRSS|VmHeap"'

# Python: tracemalloc in production
import tracemalloc
tracemalloc.start()
# ... periodic snapshots and diff to find leak source ...

# Go: pprof heap profile
go tool pprof http://localhost:6060/debug/pprof/heap
```
