## 1. The Goroutine Scheduler (M:N Scheduling)

Go uses **M:N scheduling** — multiplexing M goroutines onto N OS threads.

```
M goroutines (could be millions)
    ↓ scheduled by Go runtime
N OS threads (typically = GOMAXPROCS = number of CPU cores)
    ↓ scheduled by OS
P CPU cores
```

```
SCHEDULER COMPONENTS:
  G (Goroutine):  the lightweight concurrent unit
  M (Machine):    an OS thread
  P (Processor):  a logical processor, holds run queue

  Each P has a local run queue of goroutines.
  M picks goroutines from P's run queue and executes them.
  If P's queue is empty: M steals goroutines from other Ps (work stealing).

DEFAULT: GOMAXPROCS = number of CPU cores
  4-core machine → 4 Ps → 4 goroutines run truly in parallel
  But millions of goroutines can exist (most sleeping/waiting for I/O)
```

**Why goroutines can sleep without wasting an OS thread:**

When a goroutine does a network I/O, Go's netpoller (backed by epoll/kqueue) registers the I/O with the OS and parks the goroutine. The M (OS thread) is freed to run other goroutines. When I/O completes, the goroutine is re-added to a run queue. This is how Go achieves massive concurrency without massive thread counts.

## 2. The Go Garbage Collector

Go uses a **tri-color mark-and-sweep concurrent GC**.

```
KEY PROPERTIES:
  - Concurrent: GC runs alongside the application (not stop-the-world for most work)
  - Low pause times: stop-the-world pauses typically < 1ms
  - Non-moving: objects don't move in memory (no compaction)
  - Target: GOGC environment variable controls aggressiveness
    GOGC=100 (default): GC when heap doubles
    GOGC=50: GC more frequently (lower memory, more CPU)
    GOGC=200: GC less frequently (higher memory, less CPU)

PHASES:
  1. Mark Setup:     brief STW, enables write barrier
  2. Marking:        concurrent, traces live objects (tri-color)
  3. Mark Termination: brief STW, finalizes marking
  4. Sweep:          concurrent, frees dead objects
```

**Practical implications:**

```go
// Memory allocation triggers GC. Reduce allocations for lower GC pressure.

// BAD: allocates new string on each call
func formatID(id int) string {
    return fmt.Sprintf("user-%d", id)  // new string every call
}

// BETTER: pre-allocate buffer
var buf = make([]byte, 0, 20)
func formatID(id int) string {
    buf = buf[:0]  // reset length, keep capacity
    return string(strconv.AppendInt(buf, int64(id), 10))
}

// BEST for hot paths: use sync.Pool to reuse objects
var pool = sync.Pool{
    New: func() any { return new(bytes.Buffer) },
}

func formatID(id int) string {
    buf := pool.Get().(*bytes.Buffer)
    defer pool.Put(buf)
    buf.Reset()
    fmt.Fprintf(buf, "user-%d", id)
    return buf.String()
}
```

## 3. Escape Analysis: Stack vs. Heap Allocation

The Go compiler decides at compile time whether a variable lives on the stack (cheap, no GC) or escapes to the heap (more expensive, GC managed).

```go
// This stays on the stack (does not escape):
func add(a, b int) int {
    return a + b  // result returned by value, no heap allocation
}

// This escapes to the heap:
func newUser(name string) *User {
    u := User{Name: name}  // u is taken by address → must outlive this function
    return &u              // address taken → escapes to heap
}

// Check escape analysis:
// go build -gcflags="-m" ./...
// Output: "./main.go:5:2: moved to heap: u"
```

**When to care about escape analysis:**

In hot paths (e.g., a function called 10,000 times per second), heap allocations add GC pressure. Tools:

```bash
# See all escape decisions
go build -gcflags="-m -m" ./...

# Benchmark with allocation stats
go test -bench=. -benchmem ./...
# BenchmarkProcess-8  1000000  1234 ns/op  256 B/op  4 allocs/op
#                                               ↑            ↑
#                               bytes allocated per op  allocs per op
```

## 4. Race Detector

Go includes a built-in data race detector:

```bash
go test -race ./...    # run tests with race detection
go run -race main.go   # run with race detection

# Output on race detected:
# WARNING: DATA RACE
# Write at 0x00c0000b4008 by goroutine 7:
#   main.main.func1()
#       /tmp/main.go:14 +0x30
# Previous read at 0x00c0000b4008 by goroutine 6:
#   main.main.func2()
#       /tmp/main.go:9 +0x28
```

Run tests with `-race` in CI. Always. Race conditions are notoriously hard to reproduce in production.

## 5. Senior Engineer Perspective

**What a junior gets wrong:**

Creating goroutines without a way to wait for them or handle their errors. A goroutine that panics without recovery crashes the entire program. Use `sync.WaitGroup` to wait for goroutines to complete, and always recover from panics in goroutine entry points.

**The performance insight:**

Go's GC stop-the-world pauses are typically sub-millisecond. But at high allocation rates, GC CPU overhead can be significant. The key optimization: allocate less. Use value types instead of pointers where possible, reuse objects with `sync.Pool`, preallocate slices with `make([]T, 0, expectedLen)`. The `go tool pprof` command with the `allocs` profile reveals exactly where allocations happen.
