## 1. Coroutines: The Foundation

A **coroutine** is a function that can pause its execution and yield control back to the event loop, resuming later from where it left off. In Python, any function defined with `async def` is a coroutine function.

```python
import asyncio

# A coroutine function (returns a coroutine object when called)
async def fetch_user(user_id: int) -> dict:
    # await: pause here, give control to event loop until this completes
    result = await database.fetchrow(
        "SELECT * FROM users WHERE id = $1", user_id
    )
    return dict(result)

# Calling it returns a coroutine object (not a result)
coro = fetch_user(123)  # nothing has run yet
print(type(coro))       # <class 'coroutine'>

# To actually run it: await it or schedule it
result = await fetch_user(123)  # only works inside async def
# or:
result = asyncio.run(fetch_user(123))  # from synchronous code
```

## 2. The Event Loop: The Engine

The event loop is a loop that:
1. Picks the next ready coroutine
2. Runs it until it hits an `await`
3. While it waits for I/O, runs another coroutine
4. When I/O completes, marks the original coroutine as ready
5. Repeat

```python
import asyncio

async def main():
    # Sequential: each await blocks this coroutine until completion
    # But other coroutines can run while this one is awaiting
    user = await fetch_user(123)     # event loop: run other tasks while DB responds
    orders = await fetch_orders(123) # event loop: same
    return user, orders

# Concurrent: both DB queries run simultaneously
async def main_concurrent():
    user, orders = await asyncio.gather(
        fetch_user(123),    # both started "at the same time"
        fetch_orders(123),  # both running while each other waits for DB
    )
    return user, orders

# asyncio.run() creates an event loop, runs main(), closes the loop
asyncio.run(main_concurrent())
```

## 3. Tasks vs. Coroutines

A **Task** wraps a coroutine and schedules it to run on the event loop immediately. Unlike awaiting a coroutine directly, tasks can run concurrently.

```python
async def demo():
    # SEQUENTIAL (total time = sum of both):
    result1 = await slow_operation(1)   # wait 2s
    result2 = await slow_operation(2)   # wait 2s
    # Total: 4s

    # CONCURRENT with gather (total time = max of both):
    result1, result2 = await asyncio.gather(
        slow_operation(1),
        slow_operation(2),
    )
    # Total: ~2s

    # CONCURRENT with create_task (more control):
    task1 = asyncio.create_task(slow_operation(1), name="op1")
    task2 = asyncio.create_task(slow_operation(2), name="op2")
    # Tasks start immediately when created
    # Do other work here while they run...
    result1 = await task1
    result2 = await task2
    # Total: ~2s

    # Cancel a task
    task1.cancel()
    try:
        await task1
    except asyncio.CancelledError:
        pass  # expected
```

## 4. asyncio.gather vs. asyncio.TaskGroup (3.11+)

```python
# asyncio.gather — older, still widely used
results = await asyncio.gather(
    fetch_user(1),
    fetch_user(2),
    fetch_user(3),
    return_exceptions=True,  # don't raise on individual failures
)

# asyncio.TaskGroup — Python 3.11+, preferred for structured concurrency
async def fetch_multiple():
    async with asyncio.TaskGroup() as tg:
        task1 = tg.create_task(fetch_user(1))
        task2 = tg.create_task(fetch_user(2))
        task3 = tg.create_task(fetch_user(3))
    # All tasks completed (or one failed and the group is cancelled)
    return task1.result(), task2.result(), task3.result()
```

**Why TaskGroup is better:**
- If one task raises, all other tasks are cancelled automatically
- `gather(return_exceptions=True)` silently swallows exceptions
- TaskGroup makes exception handling explicit and correct

## 5. Common Async Mistakes

```python
# MISTAKE 1: Calling a sync blocking function in async context
@app.get("/users")
async def get_users():
    # This BLOCKS the event loop — no other requests can be handled
    import time
    time.sleep(1)           # WRONG: sync sleep blocks entire event loop
    return {"users": []}

# RIGHT:
@app.get("/users")
async def get_users():
    await asyncio.sleep(1)  # async sleep — yields to event loop

# MISTAKE 2: Using synchronous I/O in async handlers
@app.get("/file")
async def read_file():
    with open("large_file.txt") as f:  # WRONG: sync file read blocks event loop
        content = f.read()
    return content

# RIGHT: use aiofiles or asyncio.to_thread
import aiofiles

@app.get("/file")
async def read_file():
    async with aiofiles.open("large_file.txt") as f:
        content = await f.read()  # async, non-blocking
    return content

# MISTAKE 3: Using a sync database library in async code
@app.get("/users")
async def get_users():
    import psycopg2  # sync library
    conn = psycopg2.connect(...)  # WRONG: blocks event loop
    # ...

# RIGHT: use an async driver
import asyncpg

@app.get("/users")
async def get_users():
    async with pool.acquire() as conn:
        users = await conn.fetch("SELECT * FROM users")  # async
    return users

# MISTAKE 4: Not handling exceptions from Tasks
task = asyncio.create_task(risky_operation())
# If risky_operation raises and you never await task:
# Exception is silently swallowed, logged as "Task exception was never retrieved"

# RIGHT: always await tasks or add error callbacks
task = asyncio.create_task(risky_operation())
task.add_done_callback(lambda t: t.exception())  # handle exceptions
# or use TaskGroup which handles this automatically
```

## 6. Running Sync Code in Async Context

Sometimes you need to call a synchronous library from async code:

```python
import asyncio

def cpu_heavy_work(data: list) -> int:
    """A synchronous, CPU-bound function."""
    return sum(x**2 for x in data)

async def async_handler():
    data = list(range(1_000_000))

    # asyncio.to_thread: runs sync function in a thread pool
    # Doesn't block the event loop
    result = await asyncio.to_thread(cpu_heavy_work, data)
    return result

# For CPU-bound: use ProcessPoolExecutor (bypasses GIL)
from concurrent.futures import ProcessPoolExecutor

executor = ProcessPoolExecutor(max_workers=4)

async def async_cpu_work():
    loop = asyncio.get_event_loop()
    result = await loop.run_in_executor(executor, cpu_heavy_work, data)
    return result
```

## 7. asyncio in FastAPI: The Full Picture

```python
from fastapi import FastAPI
import asyncio, httpx, asyncpg

app = FastAPI()

# FastAPI runs each request handler as a coroutine
# The event loop manages concurrency — thousands of requests with one thread

@app.get("/dashboard/{user_id}")
async def get_dashboard(user_id: int):
    # Run 3 DB queries concurrently (not sequentially)
    async with asyncpg.create_pool(dsn=DATABASE_URL) as pool:
        user, orders, notifications = await asyncio.gather(
            pool.fetchrow("SELECT * FROM users WHERE id=$1", user_id),
            pool.fetch("SELECT * FROM orders WHERE user_id=$1 LIMIT 10", user_id),
            pool.fetch("SELECT * FROM notifications WHERE user_id=$1 LIMIT 5", user_id),
        )

    # Call an external API concurrently with other work
    async with httpx.AsyncClient() as client:
        analytics = await client.get(f"http://analytics-svc/users/{user_id}/summary")

    return {
        "user": dict(user),
        "orders": [dict(o) for o in orders],
        "notifications": [dict(n) for n in notifications],
        "analytics": analytics.json(),
    }

# For CPU-bound work in a FastAPI route: use to_thread
```

## 8. Senior Engineer Perspective

**What a junior gets wrong:**

Thinking `async def` makes everything automatically concurrent. `async def` makes a function a coroutine — it can yield control. But if it never hits an `await`, it never yields. If it hits a blocking `await` (sync library called with `await`), it still blocks. Concurrency comes from properly using `await` with actual async operations.

**The mental model that makes async click:**

A coroutine is a state machine. Each `await` is a "save my state here and let someone else run." The event loop is the traffic controller that decides who runs when. If nobody yields (no `await`), the traffic controller can't switch — the loop is blocked.

**The production rule:**

In a FastAPI application, every function that does I/O (database, HTTP, Redis, file) must be `async` and use an async-capable library. One synchronous I/O call in one handler blocks ALL handlers for its duration.
