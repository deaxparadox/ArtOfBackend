## 1. The Event Loop Phases

The Node.js event loop has 6 phases. Each phase has a queue of callbacks to execute. The loop processes all callbacks in one phase before moving to the next.

```
┌──────────────────────────────────────────────┐
│                   timers                      │  Phase 1: setTimeout, setInterval callbacks
└─────────────────────┬────────────────────────┘
                      │
┌─────────────────────▼────────────────────────┐
│              pending callbacks                │  Phase 2: I/O errors from previous tick
└─────────────────────┬────────────────────────┘
                      │
┌─────────────────────▼────────────────────────┐
│                    idle, prepare              │  Phase 3: internal use only
└─────────────────────┬────────────────────────┘
                      │
┌─────────────────────▼────────────────────────┐
│                      poll                     │  Phase 4: ← where most I/O callbacks run
│  Wait for I/O events. Execute I/O callbacks.  │     Retrieve new I/O events
│  If timer is due, advance to timers phase.    │     Blocks here if queue empty
└─────────────────────┬────────────────────────┘
                      │
┌─────────────────────▼────────────────────────┐
│                    check                      │  Phase 5: setImmediate callbacks
└─────────────────────┬────────────────────────┘
                      │
┌─────────────────────▼────────────────────────┐
│               close callbacks                 │  Phase 6: socket.on('close') etc.
└─────────────────────┬────────────────────────┘
                      │
                      └──────(loop back to timers)
```

**Between every phase:** microtasks (Promises, queueMicrotask) are processed.

## 2. Microtasks: The Queue That Jumps Ahead

Microtasks run **between** event loop phases and between callbacks within a phase. They always run before the next event loop tick.

```javascript
setTimeout(() => console.log('setTimeout'), 0);    // Phase 1 (timers)
setImmediate(() => console.log('setImmediate'));    // Phase 5 (check)

Promise.resolve().then(() => console.log('Promise microtask'));
process.nextTick(() => console.log('nextTick'));   // highest priority microtask

// Output order:
// nextTick            ← runs immediately before any other async
// Promise microtask   ← runs before event loop phases
// setTimeout          ← timers phase
// setImmediate        ← check phase
```

**Priority order:**
1. `process.nextTick()` — highest priority, runs before any I/O
2. `Promise.then()` / `queueMicrotask()` — microtask queue
3. `setImmediate()` — check phase
4. `setTimeout(fn, 0)` — timers phase (minimum ~1ms delay)

**The danger of `process.nextTick`:**

```javascript
function recursiveNextTick() {
    process.nextTick(recursiveNextTick);  // ← THIS STARVES THE EVENT LOOP
}
recursiveNextTick();
// I/O callbacks NEVER run — nextTick is always ahead of them
// Use setImmediate() for recursive scheduling instead
```

## 3. The libuv Thread Pool

Some operations can't be done asynchronously at the OS level. libuv uses a thread pool (default: 4 threads) for these:

```
OPERATIONS THAT USE THE THREAD POOL:
  - File system operations (fs.readFile, fs.writeFile, etc.)
  - DNS resolution (dns.lookup)
  - CPU-bound crypto operations (bcrypt, pbkdf2)
  - User-defined C++ addon async operations

OPERATIONS THAT DO NOT USE THE THREAD POOL (truly async via OS):
  - TCP/UDP network I/O
  - Pipes
  - Timers

IMPLICATION: If you read 5 files simultaneously in Node.js,
  4 run in the thread pool immediately. The 5th waits.
  UV_THREADPOOL_SIZE=8 environment variable increases pool size.
  For bcrypt hashing at high concurrency: increase this.
```

## 4. async/await Under the Hood

```javascript
// What you write:
async function fetchUser(id) {
    const user = await db.getUser(id);
    const orders = await db.getOrders(id);
    return { user, orders };
}

// What JavaScript actually compiles it to (conceptually):
function fetchUser(id) {
    return new Promise((resolve, reject) => {
        db.getUser(id)
            .then(user => {
                return db.getOrders(id)
                    .then(orders => ({ user, orders }));
            })
            .then(resolve)
            .catch(reject);
    });
}

// The async/await is syntactic sugar over Promises.
// Each `await` is a `.then()` callback scheduled as a microtask.
// The function suspends and resumes as microtasks are processed.
```

## 5. Worker Threads (CPU-Bound Parallelism)

For CPU-bound tasks, Node.js provides `worker_threads`:

```javascript
// main.js
import { Worker, isMainThread, parentPort, workerData } from 'worker_threads';

if (isMainThread) {
    // Main thread: spin up worker for CPU-bound work
    function runWorker(data) {
        return new Promise((resolve, reject) => {
            const worker = new Worker('./worker.js', { workerData: data });
            worker.on('message', resolve);
            worker.on('error', reject);
        });
    }

    // Run 4 workers in parallel (true parallelism via OS threads)
    const results = await Promise.all([
        runWorker({ start: 0,         end: 250_000 }),
        runWorker({ start: 250_000,   end: 500_000 }),
        runWorker({ start: 500_000,   end: 750_000 }),
        runWorker({ start: 750_000,   end: 1_000_000 }),
    ]);
} else {
    // worker.js: runs in separate thread
    const { start, end } = workerData;
    let sum = 0;
    for (let i = start; i < end; i++) sum += i * i;
    parentPort.postMessage(sum);
}
```

Worker threads share memory via `SharedArrayBuffer` and `Atomics` — but this requires careful synchronization. Usually better to just send data via `postMessage`.

## 6. Senior Engineer Perspective

**What a junior gets wrong:**

Using `setTimeout(fn, 0)` to defer work, thinking it will run "immediately after." It will run in the timers phase of the next event loop iteration. All microtasks (Promises) between the current point and that timer fire first. For truly immediate deferral: `setImmediate` or `queueMicrotask`.

**The event loop mastery test:**

If you can predict the exact output order of a piece of code involving `process.nextTick`, `Promise.then`, `setTimeout(fn, 0)`, and `setImmediate` — you understand the event loop deeply enough to debug any async timing issue.

**The production implication:**

Logging frameworks, error handlers, and APM agents use `process.nextTick` extensively. If you see strange ordering in your logs vs. your actual operations, the microtask queue is almost certainly involved.
