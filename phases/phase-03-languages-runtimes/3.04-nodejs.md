## 1. What Is Node.js?

Node.js is a JavaScript runtime built on **V8** (Google's JavaScript engine) that allows JavaScript to run outside the browser — on servers, CLI tools, build pipelines, and more.

**Why it matters for backend:** Node popularized the **non-blocking I/O + event loop** model for web servers in 2009. Before Node, the dominant model was thread-per-request (Apache, Rails). Node demonstrated that a single-threaded event loop could handle thousands of concurrent connections more efficiently.

**Node's defining characteristics:**
- Single-threaded JavaScript execution (like Python's GIL, but by design)
- Non-blocking I/O everywhere
- npm ecosystem (2+ million packages — largest in the world)
- JavaScript on both client and server (same language, same developers)
- Excellent for real-time applications (chat, streaming, collaborative tools)

## 2. The Three Pillars of Node.js

### V8 — The JavaScript Engine

V8 is Google's open-source JavaScript engine, written in C++. It:
- Compiles JavaScript to machine code using JIT (Just-In-Time compilation)
- Manages memory allocation and garbage collection
- Is also used in Chrome, Edge, Deno, and Bun

```
YOUR JS CODE
  │
  ▼ V8: Parse → AST → Bytecode (Ignition interpreter)
  │               ↓ (hot code detected)
  ▼ V8: JIT compile to machine code (TurboFan optimizer)
MACHINE EXECUTION

Hot code paths get compiled to optimized machine code.
Cold paths run as bytecode. Adaptive optimization.
```

### libuv — The Async I/O Library

libuv is a C library that provides:
- The event loop
- Async I/O for files, DNS, sockets
- A thread pool for operations the OS can't do asynchronously (file I/O on Linux, DNS)
- Cross-platform abstraction (epoll on Linux, kqueue on macOS, IOCP on Windows)

```
Node.js:   [ JavaScript (V8) ] ←→ [ libuv (C) ] ←→ [ OS / Hardware ]
           └── event loop ──────────────────────────────────────────┘
```

### The Event Loop (libuv + V8 integration)

The event loop coordinates between the single JavaScript thread and the async I/O operations managed by libuv.

## 3. Node.js Strengths and Weaknesses

```
STRENGTHS:
  ✓ Non-blocking by default — I/O-bound performance is excellent
  ✓ Huge npm ecosystem — a package for everything
  ✓ JavaScript everywhere — share code between frontend and backend
  ✓ Real-time applications — WebSockets, SSE, streaming
  ✓ JSON-native — no serialization overhead for REST APIs
  ✓ Fast startup (great for serverless/Lambda)
  ✓ Massive community and job market

WEAKNESSES:
  ✗ Single-threaded — CPU-bound tasks block the event loop
  ✗ Callback hell legacy (mitigated by async/await, but ecosystem still has callbacks)
  ✗ Error handling can be inconsistent across the npm ecosystem
  ✗ TypeScript required for any serious project (JS is too dynamic for large codebases)
  ✗ npm dependency hell (node_modules can be enormous)
  ✗ Weak at: image processing, heavy computation, scientific computing

SWEET SPOT:
  ✓ REST APIs and GraphQL servers
  ✓ Real-time applications (chat, notifications, gaming)
  ✓ API gateways and BFF (Backend for Frontend)
  ✓ Serverless functions (Lambda, Vercel, Cloudflare Workers)
  ✓ Build tools and CLI applications
```

## 4. The Modern Node.js Stack

```
RUNTIME: Node.js 20+ LTS (or Bun for performance-critical workloads)
LANGUAGE: TypeScript 5.x (always — JavaScript for production is unwise)

WEB FRAMEWORKS:
  Express.js   — mature, minimal, huge ecosystem
  Fastify      — faster than Express, JSON schema validation
  NestJS       — opinionated, modular, Angular-inspired
  Hono         — ultra-fast, runs on Node/Edge/Bun/Deno

ORM / QUERY BUILDER:
  Prisma       — type-safe, code-first schema, migrations
  Drizzle ORM  — SQL-first, lightweight, TypeScript-native
  TypeORM      — decorator-based, Active Record/Data Mapper

DATABASE DRIVERS:
  postgres / pg  — PostgreSQL
  mysql2        — MySQL/MariaDB
  mongodb       — MongoDB driver

VALIDATION: Zod (TypeScript-first, runtime + compile-time safety)
HTTP CLIENT: axios, got, or native fetch (Node 18+)
CACHING: ioredis
QUEUE: BullMQ (Redis-backed), bull
TESTING: Jest, Vitest, supertest
PACKAGE MANAGER: pnpm (faster, disk-efficient) or npm
```

## 5. TypeScript: Non-Negotiable for Production

```typescript
// JavaScript (don't do this for backend)
function createUser(data) {
    return db.insert(data);  // what's in data? who knows
}

// TypeScript (production standard)
interface CreateUserInput {
    email: string;
    name: string;
    age: number;
}

interface User extends CreateUserInput {
    id: number;
    createdAt: Date;
}

async function createUser(data: CreateUserInput): Promise<User> {
    return db.insert(data);  // compiler knows what goes in and comes out
}

// tsconfig.json (strict mode — always)
{
    "compilerOptions": {
        "target": "ES2022",
        "module": "Node16",
        "strict": true,              // enables all strict type checks
        "noImplicitAny": true,       // no implicit 'any' type
        "strictNullChecks": true,    // null and undefined must be explicit
        "outDir": "./dist",
        "rootDir": "./src"
    }
}
```

## 6. Bun: The Node.js Replacement?

Bun (2023) is a new JavaScript runtime built from scratch, aiming to replace Node.js:

```
BUN vs NODE:
  Speed: Bun is 2-4x faster for most benchmarks
         (JavaScriptCore engine vs V8, C++ vs Rust internals)
  
  Built-in: bundler, transpiler, test runner, package manager
  (replaces webpack/esbuild + babel + jest + npm)
  
  Compatibility: runs most Node.js code (CommonJS + ESM support)
  
  Usage (2024): growing, not production-ready for all workloads yet
  Best for: new greenfield projects, Cloudflare Workers ecosystem
```

## 7. Code Example: FastAPI vs Express vs Fastify

```typescript
// Express.js (most widely deployed, 2013-era API)
import express from 'express';
const app = express();
app.use(express.json());

app.get('/users/:id', async (req, res) => {
    const user = await db.findUser(parseInt(req.params.id));
    if (!user) return res.status(404).json({ error: 'Not found' });
    res.json(user);
});

app.listen(3000);

// Fastify (modern, schema-validated, type-safe)
import Fastify from 'fastify';
const fastify = Fastify({ logger: true });

fastify.get<{
    Params: { id: string };
    Reply: { id: number; name: string } | { error: string };
}>('/users/:id', {
    schema: {
        params: { type: 'object', properties: { id: { type: 'string' } }, required: ['id'] },
        response: {
            200: { type: 'object', properties: { id: { type: 'number' }, name: { type: 'string' } } },
            404: { type: 'object', properties: { error: { type: 'string' } } },
        },
    },
}, async (request, reply) => {
    const user = await db.findUser(parseInt(request.params.id));
    if (!user) return reply.status(404).send({ error: 'Not found' });
    return user;
});

await fastify.listen({ port: 3000 });
```

## 8. Senior Engineer Perspective

**What a junior gets wrong:**

Running CPU-heavy work in a Node.js handler. Video encoding, PDF generation, ML inference — these block the single JavaScript thread. All other requests wait. The fix: offload to a worker thread (`worker_threads`), a child process, or a separate microservice.

**The correct mental model:**

Node.js is a single-threaded concierge that orchestrates I/O operations. Excellent at: "call the database, call this API, stream the result." Terrible at: "calculate something complex for 2 seconds."

For teams already strong in JavaScript/TypeScript, Node.js is a legitimate first choice for backend APIs. For teams coming from Python/Java, the async model is conceptually the same — the syntax just differs.
