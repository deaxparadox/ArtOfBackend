## 1. The Two Axes of Type Systems

Type systems are often described on two independent axes:

### Static vs. Dynamic Typing

**Static:** Types are checked at compile time. Errors caught before execution.
**Dynamic:** Types are checked at runtime. Errors caught during execution.

```python
# Dynamic (Python): type errors at runtime
def add(a, b):
    return a + b

add(1, "2")      # TypeError at RUNTIME (when you call it)
# No warning until someone actually runs this code path

# Static (Go): type errors at compile time
func add(a int, b int) int {
    return a + b
}
// add(1, "2")   ← COMPILE ERROR before program runs
// Cannot use "2" (type string) as type int
```

### Strong vs. Weak Typing

**Strong:** Type system rarely allows implicit conversions. Type mismatches are errors.
**Weak:** Implicit type coercions are common. Types can be used interchangeably.

```javascript
// Weak typing (JavaScript): implicit coercions
"5" + 3      // "53" (string + number = string concatenation)
"5" - 3      // 2   (string - number = numeric subtraction!)
null + 1     // 1   (null coerced to 0)
undefined + 1 // NaN (undefined coerced to NaN)
[] == false  // true (array coerced to empty string, which is falsy)
```

```python
# Strong typing (Python): explicit conversion required
"5" + 3      # TypeError: can only concatenate str to str
"5" + str(3) # "53" (explicit conversion — OK)
int("5") + 3 # 8    (explicit conversion — OK)
```

## 3. Structural vs. Nominal Typing

A third dimension that matters for large codebases:

**Nominal:** Types are compatible only if they have the same name (declaration-based).
**Structural:** Types are compatible if they have the same shape (duck typing).

```typescript
// TypeScript: STRUCTURAL (Go is also structural)
interface Serializable {
    serialize(): string;
}

class User {
    id: number;
    name: string;
    serialize() { return JSON.stringify(this); }  // has the method
}

class Product {
    id: number;
    title: string;
    serialize() { return JSON.stringify(this); }  // also has the method
}

function save(item: Serializable) { /* ... */ }
save(new User());    // OK — User has serialize()
save(new Product()); // OK — Product has serialize() (no declaration needed)
```

```java
// Java: NOMINAL
interface Serializable {
    String serialize();
}

class User implements Serializable {  // explicit declaration required
    public String serialize() { return "..."; }
}

class Product {
    public String serialize() { return "..."; }  // has the method but...
}

void save(Serializable item) { /* ... */ }
save(new User());     // OK — explicitly implements Serializable
save(new Product());  // COMPILE ERROR — Product doesn't implement Serializable
```

## 4. Gradual Typing: Python's Solution

Python added type hints (PEP 484, 2014) — optional static typing annotations checked by external tools (mypy, pyright), not the interpreter itself.

```python
# Untyped (still valid Python):
def process(data):
    return data["result"]

# Typed (same behavior, but statically checkable):
def process(data: dict[str, any]) -> any:
    return data["result"]

# Running mypy:
# mypy myapp.py
# myapp.py:3: error: Item "str" of "dict[str, Any]" has incompatible type
# Found 1 error in 1 file

# TypeScript is JavaScript's equivalent: gradual typing
// .js file: no types, valid JavaScript
function add(a, b) { return a + b; }

// .ts file: types, caught by tsc compiler
function add(a: number, b: number): number { return a + b; }
```

## 5. What Type Systems Mean for Backend Engineers

```
WHY STATIC TYPING MATTERS AT SCALE:

1. REFACTORING SAFETY:
   Change a function signature → compiler shows every call site to update.
   Dynamic typing: you find out about breakages at runtime (or in tests, if lucky).

2. IDE SUPPORT:
   Static types enable autocomplete, go-to-definition, find-all-references.
   Dynamic typing: IDE guesses (often wrong for complex objects).

3. API CONTRACTS:
   TypeScript + Zod: your API input/output types are enforced by the compiler.
   Pydantic: runtime validation, but statically type-checked with mypy.
   Missing field in request body: caught at the validation layer.

4. TEAM SCALING:
   Static types are documentation that can't go stale.
   You understand a function by looking at its signature, not its implementation.
   Critical for large codebases (millions of lines) with many contributors.

TRADE-OFF:
  More type annotations = more verbose code = slower initial development.
  More type annotations = fewer runtime surprises = faster debugging.
  The larger the codebase and team, the more static typing pays off.

RULE OF THUMB:
  < 10 engineers, < 100k LOC: dynamic typing often fine (Python, Ruby, JS)
  > 10 engineers, > 100k LOC: static typing or gradual typing (TypeScript, mypy)
```

## 6. Senior Engineer Perspective

**What a junior gets wrong:**

Fighting the type checker instead of working with it. When mypy or TypeScript reports an error, the instinct is to add `# type: ignore` or `as any` to suppress it. These are debt. The type error is often pointing to a real design problem — a function that should return `User | None` but you assumed is always `User`. Working with the type system finds bugs before production.

**The one investment that pays off:**

Add strict mypy or pyright to your Python project's CI from day one. Enable `strict=true` in TypeScript. Yes, it takes a day to fix the initial batch of errors. No, you won't regret it when you're doing a large refactor 18 months later and the compiler tells you every file that broke.
