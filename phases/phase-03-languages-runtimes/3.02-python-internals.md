## 1. CPython: The Reference Implementation

Python is a language specification. **CPython** is the dominant implementation (the one you install from python.org). There are others:

```
CPython:  Written in C. The reference. What 99% of developers use.
PyPy:     JIT-compiled Python. 5-10x faster for CPU-bound code. Different GIL behavior.
Jython:   Python on the JVM. Can use Java libraries.
IronPython: Python on .NET CLR.
GraalPython: GraalVM-based, experimental.
MicroPython: For microcontrollers (ESP32, Raspberry Pi Pico).
```

When backend engineers say "Python," they mean CPython. When they hit performance problems, PyPy is the first optimization to try.

## 2. How Python Executes Your Code

```
YOUR SOURCE CODE (.py file)
  │
  ▼ Python parser
ABSTRACT SYNTAX TREE (AST)
  │
  ▼ Compiler (compile())
BYTECODE (.pyc file in __pycache__)
  │
  ▼ CPython virtual machine (ceval.c)
MACHINE EXECUTION

Example:
  Source:   result = a + b
  Bytecode: LOAD_FAST a
            LOAD_FAST b
            BINARY_OP +
            STORE_FAST result

The bytecode interpreter (ceval.c) is a C function that runs in a loop,
executing one bytecode instruction per iteration.
This is why Python is "slow" compared to compiled languages —
each Python operation goes through multiple C function calls.
```

## 3. The GIL: Deep Dive

The **Global Interpreter Lock** is a mutex in CPython that ensures only one thread executes Python bytecode at a time.

**Why it exists:**

CPython uses **reference counting** for memory management. Every Python object has a `ob_refcnt` field — how many variables point to it. When refcount hits 0, the object is freed.

```c
// Simplified: every assignment increments refcount
x = some_object   // ob_refcnt++
// Every scope exit or reassignment decrements refcount
del x             // ob_refcnt--; if == 0: free memory
```

Without the GIL: two threads could simultaneously increment/decrement the same object's refcount → race condition → refcount corrupted → double-free or use-after-free → crash. The GIL serializes all refcount operations.

**The GIL's behavior:**

```
Thread A executing Python code → holds GIL
Thread B wants to run → waits for GIL

Every 5ms (sys.getswitchinterval()): GIL released voluntarily
Thread B acquires GIL → runs for 5ms
Thread A waits...

I/O operation (socket read, file read):
Thread A: initiates I/O → releases GIL → wait for OS
Thread B: acquires GIL → runs Python code
Thread A: I/O completes → waits to reacquire GIL → continues
```

**Python 3.13's free-threaded mode (experimental):**

Python 3.13 added `--disable-gil` build option. This allows true multi-threaded execution at the cost of slower single-threaded performance (reference counting becomes atomic, which is slower). As of 2024, it's experimental and not production-ready for most workloads.

## 4. Memory Management

### Reference Counting

```python
import sys

x = []          # refcount = 1
y = x           # refcount = 2
del x           # refcount = 1
del y           # refcount = 0 → freed immediately

print(sys.getrefcount([1,2,3]))  # temporary reference in getrefcount adds 1
```

### Garbage Collector (Cycle Detection)

Reference counting can't handle circular references:

```python
a = {}
b = {}
a["other"] = b   # b's refcount = 2 (b variable + a["other"])
b["other"] = a   # a's refcount = 2 (a variable + b["other"])

del a  # a's refcount = 1 (only b["other"] points to it)
del b  # b's refcount = 1 (only a["other"] points to it)

# Neither a nor b reaches refcount 0!
# CPython's cycle detector (gc module) finds and frees these.
# gc.collect() runs periodically or can be triggered manually.
```

```python
import gc

# Disable GC for performance-critical code (if you know there are no cycles)
gc.disable()
# ... performance-critical section ...
gc.enable()
gc.collect()  # manually collect at a safe point
```

### Memory Pools (pymalloc)

CPython doesn't allocate small objects directly from the OS. It maintains its own memory pools:

```
Object sizes 1-512 bytes: allocated from CPython's pool (fast)
Object sizes > 512 bytes: allocated from the OS (malloc/free)

The pool is organized into:
  arenas (256KB each) → pools (4KB each) → blocks (8-512 bytes)

Benefit: Avoids OS allocation overhead for tiny objects (ints, tuples, strings).
Cost: Memory returned to the pool is not necessarily returned to the OS.
      Python process memory appears to grow and not shrink.
```

### int Interning (The `is` vs `==` Trap)

```python
# Small integers (-5 to 256) are cached in CPython
a = 100
b = 100
print(a is b)   # True (same object, cached)

a = 1000
b = 1000
print(a is b)   # False (different objects, not cached)
print(a == b)   # True (same value)

# NEVER use `is` to compare values — use `==`
# `is` checks object identity (same memory address)
# `==` checks value equality
```

## 5. Python Object Overhead

Understanding memory cost of Python objects is critical for high-throughput systems:

```python
import sys

sys.getsizeof(42)          # 28 bytes (int)
sys.getsizeof("hello")     # 54 bytes (str)
sys.getsizeof([1, 2, 3])   # 88 bytes (list of 3 elements)
sys.getsizeof({})          # 64 bytes (empty dict)
sys.getsizeof({"a": 1})    # 232 bytes (dict with one entry)

# Compare to C: int = 4 bytes. Python int = 28 bytes. 7x overhead.
# A Python list of 1M ints: ~28MB just for the ints + ~8MB for list pointers
# A C array of 1M ints: 4MB

# For numeric computation: use NumPy arrays (C-level storage)
import numpy as np
arr = np.array([1, 2, 3], dtype=np.int32)
arr.itemsize   # 4 bytes per element (C int)
```

## 6. Profiling Python Programs

```python
# cProfile: deterministic profiling (call counts and time per function)
import cProfile
cProfile.run("my_slow_function()")

# More readable with pstats:
import cProfile, pstats, io
pr = cProfile.Profile()
pr.enable()
my_slow_function()
pr.disable()
s = io.StringIO()
ps = pstats.Stats(pr, stream=s).sort_stats("cumulative")
ps.print_stats(20)  # top 20 functions by cumulative time
print(s.getvalue())

# py-spy: sampling profiler, works on running processes (production-safe)
# pip install py-spy
# py-spy top --pid <PID>                    # live view
# py-spy record -o profile.svg --pid <PID>  # flame graph

# memory_profiler: line-by-line memory usage
# pip install memory_profiler
from memory_profiler import profile

@profile
def memory_hungry():
    a = [i for i in range(1_000_000)]  # shows memory at each line
    b = {i: i for i in range(1_000_000)}
    return a, b
```

## 7. Senior Engineer Perspective

**What a junior gets wrong:**

Using `del` to free memory expecting immediate effect. `del x` decrements the reference count. If other references exist (e.g., in a list, another variable), the object is NOT freed. To free memory, you must remove ALL references. The GC handles cycles but not retained references.

**The performance insight:**

Python is "slow" for CPU-bound code: ~50-100ns per bytecode instruction vs ~1ns for machine code. But for I/O-bound web APIs, the bottleneck is:
1. Database query: 5ms
2. Network: 2ms
3. Python execution: 0.1ms

The 0.1ms Python execution is irrelevant compared to the 7ms I/O. This is why FastAPI can serve 10,000 req/s — it's not bottlenecked by Python execution.

**When to care about Python performance:**

When Python execution time exceeds ~1ms per request (visible in profiler). Typical culprits: JSON serialization of large objects, nested loops over large datasets, excessive object creation. Fix with: Pydantic v2 (Rust-based serialization), Polars (Rust-based DataFrames), or moving the computation to a compiled extension.
