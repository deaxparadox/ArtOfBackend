## 1. The Three Execution Models

### Compiled (Ahead-Of-Time / AOT)

Source code → machine code at compile time. The binary runs directly on the CPU.

```
SOURCE CODE (.c, .go, .rs)
  │
  ▼ Compiler (gcc, go build, rustc)
MACHINE CODE BINARY (.exe, ELF binary)
  │
  ▼ OS loads and executes directly
CPU runs your code at full hardware speed

Examples: C, C++, Go, Rust, Swift, Zig
Performance: fastest possible (direct CPU instructions)
Startup: near-instant (no runtime initialization needed for most)
Portability: binary is platform-specific (x86-64 Linux vs ARM macOS)
```

### Interpreted

Source code → executed line by line by an interpreter at runtime.

```
SOURCE CODE (.py, old .rb, early .php)
  │
  ▼ Interpreter reads and executes
INTERPRETER EXECUTES each statement directly

Examples: early Python (still CPython internally uses bytecode), Bash
Performance: slowest (interpreter overhead on every operation)
Startup: fast (no compilation step)
Portability: runs anywhere the interpreter is installed
```

**Modern "interpreted" languages actually compile to bytecode first:**

```python
# Python: source → bytecode → interpreter
your_code.py
  │
  ▼ Python compiler (at runtime or cached)
__pycache__/your_code.cpython-311.pyc  (bytecode)
  │
  ▼ CPython virtual machine interprets bytecode
EXECUTION
```

### JIT (Just-In-Time) Compilation

Bytecode → machine code at runtime, only for "hot" code paths.

```
SOURCE CODE
  │
  ▼ Compile to intermediate form (bytecode / IR)
BYTECODE (JVM .class, .NET CIL, PyPy bytecode)
  │
  ▼ JIT compiler (runs at runtime, triggered by "hot" code)
  │  First execution:   interpret bytecode (slow)
  │  After N executions: compile to machine code (fast!)
NATIVE MACHINE CODE for hot paths

Examples: JVM (Java, Kotlin, Scala), .NET CLR, PyPy, V8 (JavaScript)

ADVANTAGE over pure interpretation: eventually as fast as compiled code
ADVANTAGE over pure compilation: can optimize based on actual runtime behavior
  (e.g., inline virtual method calls after profiling which class is actually used)
DISADVANTAGE: warmup time before reaching peak performance
```

## 2. The Performance Hierarchy

```
THEORETICAL PEAK PERFORMANCE:

1. Native compiled (C, Rust, Go)         100%
2. JIT-compiled at warmup (JVM, .NET)    85-100%
3. Bytecode interpreted (CPython)        5-20%
4. Pure interpreted (bash scripts)       1-5%

IN PRACTICE FOR WEB APIS:
  The database query takes 5ms.
  Python processes the result in 0.1ms.
  Go processes it in 0.01ms.
  
  The 10x difference between Python and Go on CPU work
  is irrelevant when the DB dominates.
  
  When it matters: CPU-heavy middleware, data transformation,
  serialization of large payloads, cryptographic operations.
```

## 3. JIT Warmup in Practice

```
JAVA WARMUP BEHAVIOR:
  Request 1-100:   interpreted bytecode → slow (~200ms per request)
  Request 101-1000: C1 JIT compilation → medium (~50ms per request)
  Request 1001+:   C2 JIT compilation → fast (~5ms per request)

  → Java performance graphs show a clear "warming up" curve.
  → Load testing must run long enough to hit steady state.
  → Lambda/serverless cold starts don't benefit from JIT warmup.

V8 (JavaScript) TIERED COMPILATION:
  Tier 0: Ignition interpreter (immediate execution)
  Tier 1: Sparkplug (fast baseline compiler, minimal optimization)
  Tier 2: Maglev (mid-tier optimizer, new in Chrome 116)
  Tier 3: TurboFan (aggressive optimizer, for hot loops)
  
  Hot code continuously recompiled to better machine code.
  V8 can also DEOPTIMIZE: if a function's type changes (x was int, now string),
  V8 reverts to bytecode interpretation and eventually re-optimizes.
```

## 4. Practical Implications

```
FOR BACKEND PERFORMANCE TUNING:

CPython (Python):
  - Profile to find CPU-bound hot paths
  - Replace hot paths with C extensions (Cython, ctypes, cffi)
  - Use PyPy for CPU-bound server code (drop-in, 5-10x faster)
  - Use Rust extensions via PyO3 for maximum speed

JVM (Java/Kotlin):
  - Benchmark with enough warmup requests before measuring
  - Use JVM profiler (async-profiler, JFR) after warmup
  - GraalVM native for serverless (no warmup cost)
  - Consider ZGC for latency-sensitive services

V8 (Node.js):
  - Avoid type polymorphism in hot functions (helps TurboFan)
  - Use --prof and node --prof-process for flame graphs
  - Worker threads for CPU-bound work (avoids deoptimization in main thread)

Go/Rust (compiled):
  - pprof for Go, perf/flamegraph for Rust
  - Memory allocation is the main bottleneck (not interpretation overhead)
```

## 5. Senior Engineer Perspective

**The mental model that matters:**

For web API backends, "which language is faster" is almost always the wrong question. The right question is: "where is the time being spent?" If 95% of request time is a database query, the language's execution model is irrelevant. Measure first. The "slow" Python server is often faster than the "fast" Go server in production because the Python developer spent time optimizing the SQL query.

**When execution model genuinely matters:**

- Streaming data processing: parsing GB of logs, transforming large CSVs
- Cryptography: password hashing (bcrypt), key derivation (PBKDF2)
- Serialization/deserialization at very high rates
- ML inference (though GPU makes CPU language irrelevant for heavy models)
- Tight calculation loops (pricing engines, physics simulations)

In these cases: Python with Rust extensions (via PyO3) or pure Go/Rust is the correct answer.
